Глава про исключения почти позади: ты умеешь их бросать, ловить, заворачивать с `cause`, читать stack trace и знаешь, что снимок стека делается в момент создания объекта, а не при `throw`. Осталась последняя тема — и она не про синтаксис, а про **гигиену**: как исключение обрабатывать и как его обрабатывать **нельзя**. Звучит как «пара правил код-стайла», но именно здесь живёт самый дорогой баг всей главы, и платят за него уже в проде.

![[Pasted image 20260707175630.png]]

Сценарий. Воркер крутит фоновую работу в цикле: тянет задачи, между итерациями `Thread.sleep`. Приложение надо погасить — ты говоришь потоку «стоп». Но `sleep` объявлен как `throws InterruptedException`, это checked-исключение: компилятор не пустит сборку, пока ты его не «обработаешь». Самый быстрый путь к зелёному билду:

```java
while (running) {
    Task task = queue.take();
    process(task);
    try {
        Thread.sleep(1000);
    } catch (InterruptedException e) {
        // компилятор доволен
    }
}
```

Собралось. Уехало. А потом приложение перестаёт гаситься чисто: ты дал команду «стоп», поток её **проглотил** и крутится дальше, деплой висит, оркестратор через таймаут добивает под `SIGKILL`. Две пустые строки в `catch` стоили тебе graceful shutdown. Этот урок — про то, почему так вышло, и про всё семейство ошибок, к которому этот баг принадлежит.

**Mental model.** Каждый антипаттерн обработки исключения — это способ **выбросить информацию, которую исключение несёт**. Пустой `catch` теряет провал, широкий `catch` прячет твои же баги, заворот без `cause` теряет первопричину. `InterruptedException` — острейший случай: он несёт не «что-то сломалось», а «тебя просят остановиться», и проглотить его — значит убить саму способность потока завершиться.

### Часть 1. Контракт прерывания

#### Прерывание — это кооперативный сигнал отмены

Чтобы понять, почему две пустые строки так дорого обошлись, надо увидеть, что такое прерывание.

В Java один поток **не может силой убить** другой. Исторический `Thread.stop` бросал `ThreadDeath` в произвольной точке и оставлял объекты в битом состоянии; он давно deprecated, а на JDK 20+ вообще обезврежен — вызов бросает `UnsupportedOperationException` (проверено на JDK 21). Убить нельзя — можно только **попросить**. Отмена в Java **кооперативная**: поток A выставляет потоку B флаг «тебя попросили остановиться», а B обязан сам это заметить и отреагировать.

Кто эту просьбу замечает? Блокирующие методы — `Thread.sleep`, `Object.wait`, `Thread.join`, `BlockingQueue.take`, `Future.get`. Они следят за флагом: если он выставлен, метод не продолжает блокироваться, а **немедленно бросает `InterruptedException`**. То есть само исключение и есть сигнал, дошедший до твоего кода: «кто-то попросил тебя остановиться, прямо сейчас».

Теперь видно, что натворил пустой `catch`: ты поймал сигнал отмены и уронил его на пол. Наверх не дошло ничего, B продолжает крутиться как ни в чём не бывало.

> [!tip] 💡 Из C#  
> Это в точности дух `CancellationToken`. Ты прокидываешь токен, и отменяемая сторона **обязана** проверять `IsCancellationRequested` либо реагировать на `OperationCanceledException` — отмена это просьба, которую нельзя проигнорировать, а не force-kill. Буквальный аналог `Thread.Interrupt` / `ThreadInterruptedException` в .NET тоже есть, но живая идиома сегодня — именно `CancellationToken`. Само прерывание для тебя новое, так что пока держим на уровне «что это и зачем» — механику опроса флага и кооперативную остановку в цикле разберём в главе 23.

#### Бросок сбрасывает флаг

Самый важный кусок механики: что происходит с флагом **в момент броска**. Воркер спит. Поток A зовёт `worker.interrupt()` — выставляет флаг. `sleep` видит флаг и бросает `InterruptedException`. Вопрос: а флаг после этого какой? Интуиция кричит `true` — нас же прервали. Проверяем (прогонял на JDK 21):

```java
public class Flag {
    public static void main(String[] args) throws Exception {
        Thread worker = new Thread(() -> {
            System.out.println("до прерывания: flag = " + Thread.currentThread().isInterrupted());
            try {
                Thread.sleep(5000);
            } catch (InterruptedException e) {
                System.out.println("внутри catch: flag = " + Thread.currentThread().isInterrupted());
            }
        });
        worker.start();
        Thread.sleep(200);      // дать воркеру заснуть
        worker.interrupt();     // A выставляет флаг B
        worker.join();
    }
}
```

Вывод:

```
до прерывания: flag = false
внутри catch:  flag = false
```

Внутри `catch` флаг уже `false`. Блокирующий метод, бросая `InterruptedException`, **в том же движении гасит флаг**. Логика: бросок — это и есть доставка сигнала. Метод «израсходовал» прерывание, переведя его из формата «флаг» в формат «исключение», и передал тебе полную ответственность.

Отсюда растёт смертельность пустого `catch`. После броска флага уже нет — **единственный носитель сигнала отмены это объект исключения у тебя в руках**. Проглотил его — информация не «потерялась в логах», она испарилась полностью. Никто, ни выше по стеку, ни через повторную проверку флага, уже не узнает, что отмену запрашивали.

#### Два законных хода

Раз пойманное исключение — последний живой носитель сигнала, контракт очевиден: **что законно сделать, поймав `InterruptedException`?** Правильных ответов ровно два, и оба сохраняют сигнал живым.

**Ход первый — пробросить.** Если твой метод сам по себе блокирующий или отменяемый, объяви `throws InterruptedException` и не лови вовсе. Ты прозрачное звено: сигнал прошёл сквозь тебя наверх, и решение примет вызывающий — он знает контекст лучше.

**Ход второй — восстановить флаг и выйти.** Иногда пробросить нельзя: ты внутри `Runnable.run()`, а его сигнатура — `void run()` без `throws`, checked-исключение из неё не выпустишь. Тогда ловишь, ставишь флаг обратно через `Thread.currentThread().interrupt()` и **выходишь** из работы (`break`/`return`). Восстановление говорит: «я исключение употребил, но сигнал вернул — кто проверит флаг следующим (другой блокирующий вызов, пул потоков), увидит, что нас прерывали».

Три пути — на схеме выше: `throws` и «вернуть флаг + выйти» держат сигнал живым, пустой `catch` (правый красный ящик) убивает его. Вот паттерн «восстановил и вышел» внутри `Runnable`, где пробросить нельзя (скомпилировано и погашено чисто на JDK 21):

```java
Runnable worker = () -> {
    while (true) {
        try {
            String task = queue.take();          // блокирующий вызов
            process(task);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();  // вернули флаг
            break;                                // и завершаем работу
        }
    }
};
```

> [!warning] ⚠️ Подводный камень  
> Восстановить флаг — это не «поставил `interrupt()` и забыл». Если ты вернул флаг, но **не вышел** из цикла, следующий `take()` мгновенно увидит флаг и снова бросит `InterruptedException` — ты крутишься на пустом месте, сжигая CPU. Восстановление это **сообщение, а не остановка**; останавливаешься ты сам через `break`/`return`. Связка всегда парная: вернул флаг **и** завершил работу.

Насколько дорого стоит нарушение — видно на зеркальном эксперименте. Тот же воркер, но с пустым `catch` вместо restore+break, после `interrupt()` + `join(1000)` (проверено на JDK 21):

```
worker alive after interrupt+join? true
```

Поток проглотил команду «стоп» и остался жив — ровно сломанный shutdown из опенера. Тот же код с restore+break даёт `alive = false`: поток гаснет чисто.

### Часть 2. Галерея антипаттернов

Прерывание — острейший случай, но это часть семейства. Пройдёмся по галерее грехов, и начнём с тех, где ты **ловишь слишком много** или **делаешь слишком мало**.

#### `catch (Throwable)` / `catch (Error)` — ловишь несжигаемое

`Throwable` — корень всего, под ним и `Exception`, и `Error`. Поймав `Throwable` (или прямо `Error`), ты хватаешь то, что ловить **не предназначено**: `OutOfMemoryError`, `StackOverflowError`, `NoClassDefFoundError`. Это JVM сообщает «я умираю», а не «вот recoverable-проблема, разрули». Таким `catch` ты делаешь вид, что справился, тогда как куча кончилась или класс не загрузился — и маскируешь фатальное состояние. Это прямое нарушение того, ради чего `Error` вообще отделён от `Exception` в иерархии (6.1): `Error` — для условий, которые приложение обрабатывать не должно.

#### `catch (Exception)` — слишком широкий, ловит твои же баги

Подвох тоньше, покажу на коде (гонял на JDK 21):

```java
static int parseAge(String input) {
    try {
        return Integer.parseInt(input.trim());
    } catch (Exception e) {       // хотели поймать «не парсится»
        return -1;
    }
}
```

Замысел: вернуть `-1`, если строка не число. Но `catch (Exception)` ловит **всё**, включая `RuntimeException`-баги. Передали `null` → `input.trim()` бросает `NullPointerException` → он тоже летит в этот `catch` → метод возвращает `-1`, как будто просто «не распарсилось». А на деле в вызывающем коде баг: кто-то прислал `null`, и ты его только что **спрятал**. Проверено:

```
broad, 'abc' -> -1
broad, null  -> -1          // NPE замаскирован под «невалидный ввод»
narrow, null -> честно упал: NullPointerException
```

Лечится сужением до конкретного типа: `catch (NumberFormatException e)`. Тогда `NPE` пролетает мимо и честно падает, указывая на настоящий баг.

> [!tip] 💡 Из C#  
> Ровно тот же грех — `catch (Exception)`, проглатывающий `NullReferenceException` или `ArgumentException`, которые были багами. Универсально для обоих языков: лови самый узкий тип, который реально умеешь обработать.

#### log-and-swallow и `printStackTrace()` — логирование вместо обработки

```java
} catch (IOException e) {
    log.error("не смог прочитать конфиг", e);
    // ...и молча идём дальше с полупустым конфигом
}
```

Залогировать — **не значит обработать**. Лог фиксирует, что провал случился; он не возвращает систему в рабочее состояние. Если восстановиться нельзя (нет валидного конфига — работать нельзя), лог + продолжение даёт объект в наполовину собранном состоянии и отложенный взрыв где-то дальше, уже без связи с причиной. Не можешь восстановиться → пробрось.

А `e.printStackTrace()` как «обработка» в проде — отдельная беда: он сыплет в `stderr` **мимо твоего логгера**. Ни уровня (`ERROR`/`WARN`), ни `MDC` (request-id, user-id), ни попадания в агрегатор логов. В проде это строки в никуда. `printStackTrace()` — отладочный мусор из IDE, не обработка.

> [!tip] 💡 Из C#  
> `Console.WriteLine(ex)` вместо нормального `ILogger.LogError(ex, ...)` — тот же грех, тот же обход инфраструктуры логирования.

#### Заворот без `cause` — теряешь первопричину

```java
} catch (SQLException e) {
    throw new ServiceException("не удалось сохранить заказ");  // e выброшена на пол
}
```

У «плохого» варианта `getCause()` возвращает `null` (проверено). Исходное исключение — с сообщением `connection refused`, со stack trace, указывающим на **реальную** строку сбоя, — испарилось. Твой `ServiceException` начинает stack trace с точки заворота, а откуда растёт проблема, не узнает никто. Дебажащий видит «не удалось сохранить заказ» и упирается в стену.

Лечится одним аргументом — передать `cause`: `new ServiceException("...", e)`. Это ровно wrap-and-rethrow из 6.6–6.8: тип ты транслируешь (наружу не должен течь `SQLException` — это утечка слоя БД), но цепочку причин сохраняешь. Тогда в логе видно обоих через `Caused by:` (проверено):

```
ServiceException: не удалось сохранить заказ
    at Cause.good(Cause.java:16)
Caused by: java.sql.SQLException: connection refused
    at Cause.save(Cause.java:8)      // настоящая строка сбоя
```

#### Исключения для control flow — дорого и не по назначению

```java
try {
    int i = 0;
    while (true) {
        process(array[i++]);          // ждём ArrayIndexOutOfBounds как «конец»
    }
} catch (ArrayIndexOutOfBoundsException e) {
    // дошли до конца — продолжаем
}
```

Два провала разом. **Цена:** конструктор исключения снимает snapshot стека прямо при создании (это разбирали в 6.9 — дорогая часть именно `fillInStackTrace`, а не `throw`), так что это на порядки медленнее обычного условия цикла. **Смысл:** исключения — для исключительных ситуаций, а конец массива это норма, не аномалия (Effective Java, Item 69). Плюс этот `catch` проглотит и настоящий `AIOOBE`-баг внутри `process`. Цикл `for (int i = 0; i < array.length; i++)` бесплатен и честен.

> [!tip] 💡 Из C#  
> Идентичная заповедь — «не используй исключения для потока управления», и в .NET они тоже дорогие. Нюанс из 6.9: C# снимает стек при `throw`, Java — при создании объекта; но вывод для обоих один.

### Замыкающий контраст: почему пустой `catch` на `InterruptedException` — болезнь именно Java

Каждый антипаттерн — способ выбросить информацию исключения: пустой `catch` глотает провал, `catch (Throwable)` хватает невосстановимое, широкий `catch` прячет твои баги, заворот без `cause` теряет корень, исключения-как-`if` ломают назначение механизма. `InterruptedException` — острейший, потому что несёт не диагностику, а **команду на отмену**, и бросок уже сбросил флаг.

Последний поворот — почему пустой `catch` именно на нём это болезнь конкретно **Java**. В C# нет checked-исключений: компилятор никогда не заставляет тебя ничего обрабатывать, и давления «заткни компилятор» не существует. В Java `sleep`/`wait`/`take` объявлены `throws InterruptedException` — checked, — и сборка не пройдёт, пока ты с ним что-то не сделаешь. Самый ленивый способ «сделать» — пустой `catch`. Тот самый checked-контракт, добавленный ради безопасности, и порождает худший антипаттерн: фича, заставляющая обрабатывать ошибки, создаёт соблазн **притвориться**, что обработал.

> [!example] 🔬 Глубже / на синьора  
> **`Error` ловится языком, но не конвенцией.** Синтаксически `catch (OutOfMemoryError e)` компилируется и даже поймает `OOM` — язык не запрещает. Запрет чисто конвенциональный (Javadoc `Error`: «указывает на серьёзные проблемы, которые разумное приложение ловить не должно»). Единственное оправданное место — **последний рубеж**: top-level handler в веб-сервере/фреймворке, который логирует и даёт запросу/потоку умереть, а не притворяется, что бизнес-логика отработала. И даже там `Error` часто пробрасывают дальше или гасят весь процесс. Граница фреймворка ≠ твой бизнес-`catch`.
> 
> **Почему `interrupt()` не мгновенный на CPU-bound коде.** Флаг замечают только блокирующие методы и явные проверки `isInterrupted()`. Поток, крутящий чистый расчёт без единого блокирующего вызова, `interrupt()` не заметит никогда — флаг выставлен, но некому его прочитать. Отсюда правило кооперативной отмены: в длинных вычислениях надо самому периодически звать `Thread.interrupted()` / `isInterrupted()`. Это уже механика главы 23 — здесь важно понимать, что `InterruptedException` прилетает не «в любой момент», а строго на блокирующей точке.
> 
> **`Thread.interrupted()` vs `isInterrupted()`.** Первый — статический, **читает и сбрасывает** флаг текущего потока; второй — инстансный, читает **без** сброса. Перепутать легко, и цена ошибки — случайно проглоченный флаг. Деталь для главы 23, но на собесе по прерыванию всплывает.
> 
> **Почему `fillInStackTrace` так дорог.** Он проходит весь стек вызовов и материализует массив `StackTraceElement` — это и делает исключения непригодными для control flow (Item 69). Можно подавить, переопределив `fillInStackTrace()` пустышкой (так делают некоторые библиотеки для control-flow-исключений типа сигналов) — но это редкая оптимизация, не паттерн на каждый день.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Пустой `catch (InterruptedException)`** — грех №1. Флаг уже сброшен броском, исключение проглочено — носителей сигнала не осталось; поток не гасится (EJ Item 77).
> 2. **Восстановил флаг, но не вышел из цикла** — следующий блокирующий вызов мгновенно снова бросит `IE`; busy-loop на пустом месте. Связка `interrupt()` + `break` парная.
> 3. **`catch (Exception)` вокруг парсинга/валидации** — глотает `NPE`/`IllegalArgumentException`-баги под видом «невалидный ввод». Сужай до конкретного типа.
> 4. **`catch (Throwable)` «на всякий случай»** — хватает `Error` (`OOM`, `StackOverflow`), маскируя умирающую JVM. Только на последнем рубеже, и то с осторожностью.
> 5. **Заворот без передачи `cause`** — `getCause()` вернёт `null`, первопричина и её stack trace потеряны. Всегда `new XException(msg, e)`.
> 6. **`printStackTrace()` в проде** — идёт в `stderr` мимо логгера: без уровня, без `MDC`, без агрегатора.

#### На собесе

> [!question]- 1. Что напечатает этот код? _(senior-любимый)_
> 
> java
> 
> ```java
> Thread t = new Thread(() -> {
>     try {
>         Thread.sleep(10_000);
>     } catch (InterruptedException e) {
>         System.out.println(Thread.currentThread().isInterrupted());
>     }
> });
> t.start();
> Thread.sleep(100);
> t.interrupt();
> ```
> 
> **Ответ:** `false`. Интуиция кричит `true` — «мы же только что прервали поток!». Но `sleep`, бросая `InterruptedException`, в тот же момент **сбросил** флаг. Внутри `catch` поток уже «не прерван».
> 
> _Follow-up:_ «Как тогда сообщить наверх, что нас прерывали?» — `Thread.currentThread().interrupt()`, восстановить флаг вручную. _Trap:_ «А если восстановил, но из цикла не вышел?» — следующий блокирующий вызов мгновенно снова бросит `IE`; восстановление это сигнал, останавливаешься ты сам через `break`/`return`.

> [!question]- 2. Метод ловит `Exception`, чтобы вернуть `-1` на невалидном вводе. Что не так?  
> **Ответ:** `catch (Exception)` ловит не только ожидаемое (`NumberFormatException`), но и `RuntimeException`-баги — `NullPointerException`, `IllegalArgumentException`. Передали `null`, получили `-1` вместо громкого падения, и баг вызывающего кода ты спрятал. `-1` для дебажащего неотличимо от «честно невалидный ввод». Лечится сужением до `catch (NumberFormatException e)`.
> 
> _Follow-up:_ «Но разве вернуть `-1` — это не defensive programming?» — нет, это маскировка. Defensive — провалидировать `null` явно и упасть/вернуть осмысленную ошибку, а не глотать любой сбой под одну ветку.
> 
> _Слабый ответ:_ «`catch (Exception)` — норма, я же логирую» — не назван ключевой водораздел: широкий `catch` ловит твои же баги.

> [!question]- 3. Почему нельзя `catch (Throwable)` «на всякий случай»?  
> **Ответ:** `Throwable` накрывает и `Error` — `OutOfMemoryError`, `StackOverflowError`, `NoClassDefFoundError`. Это невосстановимые состояния; поймав их, ты делаешь вид, что справился, и маскируешь факт, что JVM при смерти.
> 
> _Follow-up (с подвохом):_ «Но top-level хендлер в Spring/сервере же ловит `Throwable` — почему ему можно?» — потому что это **последний рубеж**: он логирует и даёт запросу/потоку умереть, а не притворяется, что бизнес-логика отработала. И даже там `Error` часто пробрасывают дальше. Граница фреймворка ≠ твой бизнес-`catch`.

> [!question]- 4. Логирование в `catch` — это обработка исключения?  
> **Ответ:** нет. Лог фиксирует, что провал был; он не возвращает систему в рабочее состояние. Не можешь восстановиться — пробрось, а не «залогируй и иди дальше с битым состоянием».
> 
> _Follow-up:_ «А когда логировать как раз правильно?» — один раз, на границе, которая реально обрабатывает или транслирует исключение. `log.error(e)` + `throw` на каждом слое даёт дублирующий лог одной ошибки в пяти местах — ищи потом, где настоящая.

> [!question]- 5. Поймал `InterruptedException` внутри `Runnable.run()`. Почему нельзя просто `throws`, и что делать?  
> **Ответ:** сигнатура `void run()` не объявляет `throws`, checked-исключение из неё не выпустишь — компилятор не даст. Значит: восстановить флаг через `Thread.currentThread().interrupt()` и выйти из работы (`break`/`return`).
> 
> _Follow-up:_ «Зачем восстанавливать флаг, если всё равно выходишь?» — чтобы код выше (пул потоков, обёртка, проверка статуса потока) увидел, что прерывание было. Ты сохраняешь сигнал, а не глушишь его на своём уровне.

> [!question]- 6. Почему пустой `catch` на `InterruptedException` встречается именно в Java?  
> **Ответ:** `InterruptedException` — checked, и компилятор не пускает сборку, пока ты его не «обработаешь». Самый ленивый способ заткнуть компилятор — пустые скобки. В C# checked-исключений нет, такого давления не существует.
> 
> _Follow-up:_ «Назови другое checked-исключение с той же болезнью» — `IOException`, любое checked: тот же соблазн проглотить ради зелёной сборки. _Follow-up (философский):_ «Значит, checked-исключения — ошибка дизайна?» — спорный вопрос. Bloch (EJ) советует применять их скупо, многие современные API их избегают (Spring заворачивает в unchecked), но изначальный замысел — заставить вызывающего осознанно отреагировать. Зрелый ответ — «инструмент, который легко применить не там».

#### Практика — меню, не контракт

Бери что зацепит, не обязательно всё.

1. **Увидеть сброс флага руками.** Поток, который спит; прерви его, внутри `catch` напечатай `isInterrupted()`. Убедись, что `false`. Потом добавь восстановление и убедись, что `true`. _~15 мин, только JDK._ Цель — поверить механике глазами.
2. **Отменяемый воркер.** `Runnable`, тянущий задачи из `BlockingQueue` в цикле; сделай чистую остановку на `interrupt()` через «восстановил флаг + `break`». Потом нарочно сломай пустым `catch` и посмотри, как поток отказывается умирать на `shutdown`. _~30–40 мин, только JDK._ Цель — почувствовать разницу между graceful shutdown и зависшим потоком.
3. **Охота на спрятанный баг.** Возьми `parseAge` с `catch (Exception)`, напиши тест с `null` — увидишь `-1` вместо падения. Сузь до `catch (NumberFormatException)` и смотри, как тест честно валится с `NPE`. _~20 мин, JUnit по желанию._ Цель — вбить рефлекс «узкий `catch` вытаскивает баги наружу».
4. **Цепочка причин.** Собери 2–3 слоя вызовов (repo → service), заверни нижнее исключение один раз без `cause`, один раз с `cause`. Распечатай stack trace, сравни секцию `Caused by:`. _~30 мин, только JDK._ Цель — увидеть, что именно `cause` даёт дебажащему.
5. **(Со звёздочкой) Замерить цену.** Пройди по массиву двумя способами — `for` против ловли `ArrayIndexOutOfBounds` как «конца» — и засеки время на миллионах итераций через `System.nanoTime`. _~45–60 мин, JMH по желанию._ Оговорка: наивный замер шумный, но разрыв тут виден и так. Цель — оцифровать «исключения дорогие».

#### Итоговый ответ по теме _(RU/EN)_

**RU:** Прерывание в Java — кооперативный сигнал отмены: поток нельзя убить силой, только выставить ему флаг. Блокирующие методы (`sleep`, `wait`, `join`, `take`) этот флаг видят и бросают `InterruptedException`, **сбрасывая флаг тем же движением** — после броска единственный носитель сигнала это само исключение. Поэтому поймав его, есть ровно два законных хода: пробросить (`throws`), если метод сам отменяемый, либо восстановить флаг через `Thread.currentThread().interrupt()` и выйти, если пробросить нельзя (например, внутри `run()`). Проглотить пустым `catch` нельзя — сигнал отмены умирает, поток не гасится. Это острейший случай общего принципа: любой антипаттерн обработки — способ выбросить информацию исключения. `catch (Exception)` ловит и твои баги — сужай тип; `catch (Throwable)` хватает невосстановимый `Error`; логирование — не обработка; заворачиваешь — всегда с `cause`; исключения не для control flow (дорог snapshot стека при создании). _(senior add-on)_ Пустой `catch` именно на `InterruptedException` — болезнь конкретно Java: checked-контракт не пускает сборку, и пустые скобки — ленивый способ заткнуть компилятор; в C# checked-исключений нет.

**EN:** In Java, interruption is a cooperative cancellation signal: you can't force-kill a thread, only raise a flag on it. Blocking methods (`sleep`, `wait`, `join`, `take`) observe that flag and throw `InterruptedException`, **clearing the flag in the same move** — after the throw, the only carrier of the signal is the exception object itself. So when you catch it, there are exactly two legal moves: rethrow (`throws`) if your method is itself cancellable, or restore the flag via `Thread.currentThread().interrupt()` and exit if you can't rethrow (e.g. inside `run()`). Swallowing it with an empty `catch` kills the cancellation signal and the thread never stops. This is the sharpest case of a general principle: every handling antipattern is a way to throw away the information an exception carries. `catch (Exception)` also catches your own bugs — narrow the type; `catch (Throwable)` grabs unrecoverable `Error`; logging is not handling; when you wrap, always pass the `cause`; and don't use exceptions for control flow (the stack snapshot on construction is expensive). _(senior add-on)_ An empty `catch` specifically on `InterruptedException` is a Java-specific ailment: the checked contract blocks the build, and empty braces are the lazy way to silence the compiler — C# has no checked exceptions and no such pressure.
