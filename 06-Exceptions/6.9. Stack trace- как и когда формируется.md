Представь профайлер на живом сервисе: парсер дат в горячем пути, на каждой невалидной строке — `throw new ParseException()`, и таких строк десятки тысяч в секунду. Профиль показывает, что время горит не там, где ты его искал: не в `catch`, который ты переписывал, а в `new`, на который даже не смотрел. Отдельная, вроде бы никак не связанная загадка из ops: в проде сыплется `NullPointerException` вообще без стектрейса и без сообщения, хотя логгер настроен правильно и трейс никто не проглатывал. Обе истории растут из одного механизма — из того, **в какой момент** Java снимает stack trace. Разберём этот момент, и обе объяснятся сами.

**Mental model.** В Java снимок стека снимается в момент `new` — внутри конструктора `Throwable`, — а не в момент `throw`. Всё остальное в теме (трейс «врёт» о месте броска, цена исключений, стэклес-трюки, пустой `NPE` в проде) — прямые следствия этого одного факта.

### Исключение — это объект, а не событие

`RuntimeException`, `IllegalArgumentException` — это **классы**, такие же, как `String` или `ArrayList`. Значит, исключение можно создать обычным `new` и просто положить в переменную — и при этом **ничего не падает**. Объект спокойно лежит, программа идёт дальше. Авария начинается только на отдельной команде `throw`.

```java
public class ExceptionIsObject {
    public static void main(String[] args) {
        // Шаг 1: просто СОЗДАЁМ объект. Ничего не падает.
        RuntimeException e = new RuntimeException("boom");
        System.out.println("Объект создан. Программа живёт.");
        System.out.println("Внутри объекта: " + e.getMessage());
        System.out.println("Всё ещё живём, ничего не упало...");

        // Шаг 2: только ТЕПЕРЬ решаем его бросить.
        throw e;
    }
}
```

Вывод (проверено на JDK 21):

```
Объект создан. Программа живёт.
Внутри объекта: boom
Всё ещё живём, ничего не упало...
Exception in thread "main" java.lang.RuntimeException: boom
	at ExceptionIsObject.main(ExceptionIsObject.java:4)
```

Первые три строки напечатались спокойно — создание объекта-исключения мирное, как `new ArrayList()`. Две разные операции: `new RuntimeException(...)` — **смастерить** объект (тихо); `throw e` — **швырнуть** его (вот тут грохот). Обычно их пишут слитно (`throw new ...`), поэтому в голове они слиплись в одно — но между ними может пройти время, объект можно передать, вернуть, сохранить.

Теперь главный вопрос ложится ровно: у нас два момента — `new` и `throw`. В какой снимается фото? Ответ уже в выводе. `new` стоит на **строке 4**, `throw e` — на **строке 10**. А трейс показал строку **4, а не 10**. Между ними прошли три `println`, объект полежал в переменной — и всё равно трейс держится за место `new`. Бросок фотографию не переснял.

> [!tip] 💡 Из C#  
> Это не Java-фокус — в C# ровно так же: `var e = new Exception("boom");` создаёт объект и ничего не роняет, `throw e;` — отдельно. Ты это делал, просто, скорее всего, никогда не разносил два шага в уме, потому что писал одной строкой. Расхождение — не в этом, а в том, **в каком из двух моментов** снимается стек. Смотри диаграмму.

### Кто снимает — `fillInStackTrace()`

Возьмём объект, который вообще **никогда не бросали**, и спросим снимок напрямую:

```java
public class FillInDemo {
    static Throwable makeIt() {
        return new RuntimeException("создали, но не бросали");  // строка 3 — тут new
    }
    public static void main(String[] args) {
        Throwable t = makeIt();                          // создали, не бросаем
        StackTraceElement[] frames = t.getStackTrace();  // спросим снимок напрямую
        System.out.println("Кадров в снимке: " + frames.length);
        for (StackTraceElement f : frames) System.out.println("  " + f);
        System.out.println("Дошли до конца, ничего не бросали.");
    }
}
```

Вывод:

```
Кадров в снимке: 7
  FillInDemo.makeIt(FillInDemo.java:3)
  FillInDemo.main(FillInDemo.java:6)
  ... 5 служебных кадров лончера java File.java (Method.invoke, launcher) ...
Дошли до конца, ничего не бросали.
```

Объект **не брошен**, а снимок внутри уже лежит — полный, готовый; `getStackTrace()` его просто достал. Наши кадры — верхние два (`makeIt` строка 3 = место `new`, и `main`); ниже — служебный шум от запуска через `java File.java`, не пугайся его. Значит фотограф отработал ещё в конструкторе.

Этого фотографа зовут `fillInStackTrace()`. Механика: любой `new SomeException(...)` через цепочку `super(...)` доходит до конструктора `Throwable`, а тот в конце вызывает `fillInStackTrace()`. Это `native`-метод — он уходит в потроха JVM, проходит по кадрам стека текущего потока и складывает их в объект. Происходит это **всегда** при конструировании, для любого исключения, хочешь ты трейс или нет (задокументировано в Javadoc 21 на `Throwable.fillInStackTrace()`).

### `throw e` vs обёртка — что происходит с трейсом при перебросе

Два разных действия с разным эффектом. `throw e;` бросает **тот же** объект — снимок в нём снят давно, при его `new`, и бросок ничего не переснимает (ты это видел: создан на строке 4, брошен на 10, трейс про 4). А `throw new RuntimeException("...", e)` мастерит **новый** объект: у него снимается свой снимок здесь, плюс старое исключение цепляется как `cause`.

```java
public class RethrowDemo {
    static void low() {
        throw new IllegalStateException("низкоуровневая беда");   // создано — строка 3
    }
    static void high() {
        try { low(); }
        catch (IllegalStateException e) {
            throw new RuntimeException("обёртка сверху", e);      // создано — строка 8
        }
    }
    public static void main(String[] a) { high(); }
}
```

Вывод:

```
Exception in thread "main" java.lang.RuntimeException: обёртка сверху
	at RethrowDemo.high(RethrowDemo.java:8)        <- снимок обёртки: место ЕЁ new
	at RethrowDemo.main(RethrowDemo.java:12)
Caused by: java.lang.IllegalStateException: низкоуровневая беда
	at RethrowDemo.low(RethrowDemo.java:3)         <- снимок причины: место ЕГО new
	at RethrowDemo.high(RethrowDemo.java:6)
	at RethrowDemo.main(RethrowDemo.java:12)
```

**Каждое исключение в цепочке `cause` несёт свой собственный снимок, снятый при его собственном `new`.** Обёртка показывает строку 8 (где её сделали), причина в `Caused by:` — строку 3 (где сделали её). Два независимых снимка, склеенные в одну распечатку через `Caused by:`. Это прямое следствие ядра: снимок принадлежит объекту, а цепочка — просто два объекта, ссылающихся друг на друга (механика `cause`/`getCause` — урок 6.6).

> [!tip] 💡 Из C#  
> Теперь полный контраст встаёт на место. Раз в .NET трейс снимается при `throw`, у тебя там есть выбор в точке переброса:
> 
> - `throw;` (без операнда) — **дописывает** текущие кадры к уже накопленным, сохраняя «откуда пришло».
> - `throw ex;` (с операндом) — **сбрасывает** трейс на текущую точку; исходное место беды теряется. Классический баг новичков в C#.
> 
> В Java этого выбора **физически нет**: трейс зашит при `new`, и переписать его «текущей точкой» броском нельзя в принципе. `throw e;` всегда несёт место создания. Хочешь добавить «где я это перехватил» — только оборачиванием (`throw new X(e)`), и тогда получаешь новый кадр **плюс** `cause`, а не затирание старого. Java-механика лишает тебя возможности случайно потерять исходный трейс — но взамен заставляет платить за снимок на каждом `new`.

### Практический вывод: дорогая часть — это `new`, а не `throw`/`catch`

Раз `fillInStackTrace` проходит по **каждому** кадру стека, цена снимка линейна по глубине: чем длиннее стопка вызовов в момент `new`, тем дороже. Сам бросок и поимка (`throw`/`catch` — JVM находит обработчик и разматывает стек) стоят копейки рядом с проходом по кадрам. Люди интуитивно винят `try/catch` («он же стек разматывает»), а тормозит создание объекта.

Вот и ответ на опенер про парсер: на каждой неудачной попытке `new ParseException()` честно фотографировал стек — а в реальном приложении он не на 5 кадров, а на десятки (сервлет, фильтры, фреймворк). Десятки тысяч раз в секунду фотографировать глубокий стек — вот где сгорало время. Не в `catch`, а в `new`. Отсюда старое правило — **исключения для исключительных ситуаций, не для control flow** — но не «потому что некрасиво», а потому что у каждого `new` исключения есть эта линейная по глубине цена. Для частого ожидаемого исхода возвращай значение.

В человекочитаемый массив он раскрывается **лениво** — только когда ты реально просишь (`getStackTrace()` / `printStackTrace()`). Это две разные по цене операции:

#### Бесстековые исключения — выключить снимок

Раз вся цена в снимке — выключим снимок, и исключение станет почти бесплатным. Два легальных способа, оба живут в реальных библиотеках.

java

```java
// Способ 1 — переопределить fillInStackTrace() пустышкой
static class NoStack1 extends RuntimeException {
    NoStack1(String msg) { super(msg); }
    @Override public Throwable fillInStackTrace() { return this; }   // ничего не снимаем
}

// Способ 2 — protected-конструктор Throwable с writableStackTrace=false (Java 7+)
static class NoStack2 extends RuntimeException {
    NoStack2(String msg) {
        super(msg, null, false, false); // cause, enableSuppression, writableStackTrace
    }
}
```

Замер обоих против обычного на глубине 500 — замеряемый цикл создаёт исключение и держит его живым **без** раскрытия трейса, то есть меряется чистый проход по стеку:

```
Normal (с трейсом):         ~23 000 нс,  кадров в трейсе: 507
override fillInStackTrace:  ~ 2 000 нс,  кадров в трейсе: 0
writableStackTrace=false:   ~ 2 000 нс,  кадров в трейсе: 0
```

Бесстековые — раз в десять с лишним быстрее на этой глубине (а чем глубже стек, тем больше разрыв: обычному надо обойти все 500 кадров, этим — ноль). Обрати внимание на правую колонку — **0 кадров**: цена ушла вместе с трейсом. Приём оправдан только там, где ты сознательно меняешь отладку на скорость: высокочастотные сигнальные исключения, у которых место броска и так фиксировано и известно (классика — исключения-сигналы внутри парсеров и стейт-машин, которые ловятся на соседней строке и никогда не всплывают наружу).

> [!tip] 💡 Из C#  
> Прямого аналога нет, и понятно почему: в C# ты за снимок при `new` вообще не платишь (он снимается при `throw`), так что и «отключать снятие на конструировании» там незачем. Вся ветка стэклес-трюков — чисто Java-следствие того, что фотограф работает на `new`.

#### JIT fast-throw — пустой `NPE` из ниоткуда

А это то же самое, но делает за тебя **JVM**, без твоего ведома, на самых обычных `NullPointerException`. Сценарий: горячий метод кидает `NPE` (или `ArithmeticException` от деления на ноль, или `ArrayIndexOutOfBounds`) снова и снова. HotSpot после прогрева решает: «зачем каждый раз фотографировать один и тот же стек ради исключения, которое тут всё равно ловят» — и начинает кидать **заранее заготовленный** объект без трейса и без сообщения.

```java
public class FastThrow {
    static String s = null;
    public static void main(String[] a) {
        int firstEmpty = -1;
        for (int i = 0; i < 200_000; i++) {
            try { s.length(); }                       // неявный NPE в горячем пути
            catch (NullPointerException e) {
                if (e.getStackTrace().length == 0 && firstEmpty == -1) {
                    firstEmpty = i;
                    System.out.println("message при пустом трейсе: " + e.getMessage());
                }
            }
        }
        System.out.println("Первая итерация с ПУСТЫМ трейсом: " + firstEmpty);
    }
}
```

Вывод:

```
=== Дефолт (fast-throw включён) ===
message при пустом трейсе: null
Первая итерация с ПУСТЫМ трейсом: 40960

=== java -XX:-OmitStackTraceInFastThrow FastThrow.java ===
Первая итерация с ПУСТЫМ трейсом: -1   (никогда)
```

Первые ~40k раз `NPE` прилетает нормальный — с трейсом и (по умолчанию с Java 15, `JEP 358` про helpful-сообщения; в Java 14 это было под флагом `-XX:+ShowCodeDetailsInExceptionMessages`) с подробным «Cannot invoke … because … is null». А после того как JIT докомпилировал горячий метод — с 40961-й итерации `NPE` идёт пустой: ноль кадров, `message == null`. Флаг `-XX:-OmitStackTraceInFastThrow` отключает обнуление — трейс остаётся всегда.

Вот ответ на классический ops-вопрос: **почему в логах прода `NPE` иногда без стека и без сообщения.** Это не баг и не «логи проглотили трейс» — это оптимизация. Характерная примета: в начале лога несколько нормальных `NPE` с полным трейсом, а дальше по тому же месту сыплются пустые.

> [!warning] ⚠️  
> Fast-throw работает **только** для неявных исключений, которые JVM генерит сама (`NullPointerException`, `ArithmeticException`, `ArrayIndexOutOfBoundsException`, `ArrayStoreException`, `ClassCastException`) — у неё для них есть преаллоцированные экземпляры. Твои собственные `throw new MyException()` под fast-throw не попадают, их JVM не подменяет.

#### `StackWalker` вместо фейкового исключения (`JEP 259`)

Если нужно эффективно **обойти стек** не ради исключения, а чтобы, например, фреймворку узнать своего вызывающего — раньше делали грязный трюк `new Throwable().getStackTrace()` (дорого: снимает и полностью раскрывает весь массив). С Java 9 для этого есть `StackWalker` (`JEP 259`) — ленивый обход, который не материализует полный трейс и берёт ровно столько кадров, сколько просишь. Глубоко не лезем (детали — глава 12), просто знай: это правильный инструмент вместо фейкового исключения.

> [!warning] ⚠️ Подводные камни
> 
> 1. **«Создал здесь — бросил там».** Если создать исключение заранее, а бросить позже/в другом месте — трейс покажет место **создания**, не броска. Обычно `new` и `throw` на одной строке, поэтому совпадает; но объект, переданный через несколько уровней, будет «врать» о точке броска.
> 2. **Бесстековое исключение в проде = ноль строк «откуда».** Прилетит `NoStack: x` без единой строки трейса — будешь гадать, какой из десятка путей его кинул. Для нормальных «что-то сломалось» так делать нельзя, только для сигнальных, что ловятся рядом.
> 3. **Пустой `NPE` в проде — это fast-throw, а не потеря логов.** Не крути логгер: посмотри **первые** вхождения (до компиляции трейс ещё полный) или подними `-XX:-OmitStackTraceInFastThrow`.
> 4. **Исключения как control flow в горячем цикле.** Цена ∝ глубине, а реальный стек — не 5 кадров, а десятки (фильтры, фреймворк). Десятки тысяч бросков в секунду на глубоком стеке — вот где сгорает время.
> 5. **Двойной `printStackTrace()` НЕ переснимает стек.** Снимок взят один раз на `new`; `printStackTrace`/`getStackTrace` только **читают** его (раскрывая `backtrace`). Сколько ни печатай — кадры те же (проверено: два `getStackTrace()` подряд дают идентичный массив).

#### На собесе

Заходи с одной фразы и держись за неё: «В Java снимок стека снимается при конструировании исключения, в `new`, а не при `throw`. Это делает нативный `fillInStackTrace`, который конструктор `Throwable` зовёт автоматически». Из этого выводи всё остальное на ходу, не зубри списком: следствие про **отладку** (трейс показывает место создания), про **перформанс** (дорог `new`, не `throw`/`catch`, цена ∝ глубине), про **прод** (пустой `NPE` = fast-throw). Контраст с C# держи как козырь — он показывает, что ты понимаешь механику, а не зазубрил.

> [!question]- В какой момент Java снимает stack trace — при `new` или при `throw`?  
> **RU:** При `new`, в конструкторе `Throwable` (через нативный `fillInStackTrace`). Снимок зашивается в объект на конструировании; `throw` лишь несёт уже готовый объект наверх — поэтому если создать исключение в одном месте, а бросить в другом, трейс покажет место **создания**, и может «врать» о точке броска.
> 
> **EN:** At `new`, inside the `Throwable` constructor (via the native `fillInStackTrace`). The snapshot is baked into the object at construction; `throw` only carries the ready object up — so if you create an exception in one place and throw it in another, the trace points to the **creation** site and can misreport where it was thrown.
> 
> _Follow-up:_ «А в C#?» — наоборот, при `throw`; отсюда в C# `throw;` против `throw ex;`, а в Java такого выбора нет в принципе. _Слабый ответ:_ «при `throw`» — путает Java с C#.

> [!question]- Что в исключениях реально дорого — `throw`, `catch` или что-то ещё?  
> **RU:** Проход по стеку в `fillInStackTrace` на `new`. `throw`/`catch` (поиск обработчика, размотка) — копейки рядом: фотограф обходит **каждый** кадр, цена линейна по глубине стека.
> 
> **EN:** The stack walk in `fillInStackTrace` at `new`. `throw`/`catch` (locating the handler, unwinding) are cheap next to it: the snapshot visits **every** frame, so the cost is linear in stack depth.
> 
> _Follow-up:_ «Почему тогда исключения-как-control-flow тормозят?» — в горячем цикле каждый `new` фотографирует глубокий стек фреймворка. _Follow-up:_ «Насколько дорого?» — на сотнях кадров это десятки-сотни микросекунд против единиц наносекунд у обычного `new`.

> [!question]- В проде `NullPointerException` без стектрейса и без сообщения. Логи проглотили трейс?  
> Нет — это JIT fast-throw: HotSpot после прогрева горячего пути кидает преаллоцированный объект без трейса и `message` (`OmitStackTraceInFastThrow`). Работает только для неявных исключений, что генерит сама JVM (`NPE`, `ArithmeticException` и т.п.).
> 
> _Follow-up:_ «Как вернуть трейс?» — флаг `-XX:-OmitStackTraceInFastThrow`, либо смотреть первые вхождения в логе. _Follow-up:_ «Это работает для моих `throw new MyException()`?» — нет, только для JVM-генерируемых.

> [!question]- Как сделать почти бесплатное исключение для control-flow, и почему это обычно плохо?  
> `@Override fillInStackTrace() { return this; }` **или** `protected`-конструктор `Throwable(..., writableStackTrace=false)`. Плохо тем, что трейс становится пустым — теряешь отлаживаемость.
> 
> _Follow-up:_ «Где оправдано?» — высокочастотные сигнальные исключения, что ловятся рядом и не всплывают наружу (парсеры, стейт-машины). _Follow-up:_ «А в C# такой приём есть?» — нет смысла: там за снимок на `new` не платишь.

> [!question]- Чем по трейсу `throw e;` отличается от `throw new RuntimeException("...", e)`?  
> `throw e;` несёт исходный снимок (место создания `e`); обёртка снимает **новый** снимок на своём `new` и цепляет `e` как `cause`. Снимок принадлежит объекту, а в выводе два снимка склеиваются через `Caused by:`.
> 
> _Follow-up:_ «Сколько трейсов в цепочке из трёх исключений?» — три, каждый снят при своём конструировании.

> [!question]- (с подвохом) Вызов `e.printStackTrace()` дважды переснимет стек?  
> Нет. Снимок снят один раз на `new`; `printStackTrace`/`getStackTrace` только **читают** его (раскрывая ленивый `backtrace` в `StackTraceElement[]`). Сколько ни печатай — кадры те же.
