Представь слой данных: `UserRepository` внутри дёргает JDBC, и при сбое наружу летит `SQLException` — потеря коннекта, нарушение констрейнта, `ORA-00942`. Это исключение про **детали хранилища**. Сервис уровнем выше живёт в терминах домена («не смог загрузить пользователя») и знать про SQL не должен: если `SQLException` протечёт наверх, ты намертво прибил весь верхний код к тому, что под ним именно JDBC — переедешь на Mongo, и всё, что ловило `SQLException`, придётся переписывать. Напрашивается перевод на границе: поймать низкоуровневое и бросить своё. Но тут вторая ловушка — если просто выбросить новое исключение и **потерять** оригинал, в 3 ночи ты увидишь голое «не смог загрузить» и ни слова о том, что реально упало. Этот урок — про механизм, который решает обе проблемы разом: `cause` (сохранить первопричину), свой тип исключения (транспорт под неё) и паттерн `wrap-and-rethrow` (само движение на границе).

**Mental model.** Исключение может нести внутри себя другое — «причину» (`cause`); они образуют линейную цепочку высокоуровневое → `Caused by:` → низкоуровневое, которую `getCause()` проходит вниз до корня. Заворачивание на границе слоёв меняет **тип** исключения на язык слоя, не теряя первопричину.

### Бит A — механизм `cause`

**How it works.** У любого `Throwable` (корень иерархии — от него наследуют и `Error`, и `Exception`) есть слот под причину. Передаёшь оригинал вторым аргументом конструктора — он садится внутрь как `cause`; достаёшь через `getCause()`.

```java
import java.sql.SQLException;

public class CauseDemo {
    static void jdbc() throws SQLException {
        throw new SQLException("ORA-00942: table or view does not exist");
    }
    static void loadUser() {
        try {
            jdbc();
        } catch (SQLException low) {
            // заворачиваем: low уходит внутрь как cause
            throw new RuntimeException("Не удалось загрузить пользователя id=42", low);
        }
    }
    public static void main(String[] args) { loadUser(); }
}
```

Прогнал на JDK 21 (скомпилированный класс, не single-file launcher — это важно ниже):

```
Exception in thread "main" java.lang.RuntimeException: Не удалось загрузить пользователя id=42
	at CauseDemo.loadUser(CauseDemo.java:11)
	at CauseDemo.main(CauseDemo.java:15)
Caused by: java.sql.SQLException: ORA-00942: table or view does not exist
	at CauseDemo.jdbc(CauseDemo.java:5)
	at CauseDemo.loadUser(CauseDemo.java:9)
	... 1 more
```

Две части. Сверху — твоё доменное «не смог загрузить». Ниже JVM сама печатает блок **`Caused by:`** и разворачивает оригинальный `SQLException` с его трейсом. Ничего не потеряно, а `getCause()` вернёт тот самый объект `SQLException`, если он нужен программно.

Про хвост **`... 1 more`**: JVM не печатает у причины кадры, **общие** с уже напечатанным трейсом-обёрткой. Здесь общий кадр один — `main`, поэтому `... 1 more`. (В двухслойном примере ниже увидишь `... 2 more`.) Кстати, если запустить это через `java CauseDemo.java` (single-file launcher), в хвосте будет `... 6 more` — лишние кадры добавляет рефлексивный запуск лаунчера; в реальной скомпилированной программе их нет.

> [!tip] 💡 Из C#  
> Один в один твой `InnerException`. `new RuntimeException(msg, low)` = `new Exception(msg, inner)`; `getCause()` = `.InnerException`. Блок `Caused by:` в трейсе — это то же, что `---> System.X` внутри `ToString()`, когда C#-эксепшен разворачивает внутренние. Механизм идентичный, нейминг другой.

#### `initCause()` — легаси-чёрный-ход

Причину можно поставить не только конструктором. У `Throwable` есть отдельный сеттер `initCause(Throwable)`. Логичный вопрос: зачем он, если конструктор уже умеет? Ответ — совместимость. Цепочки причин завезли в **Java 1.4** (2002, `@since 1.4` прямо в исходнике). До этого куча исключений в JDK и библиотеках уже имела конструкторы без `cause`-параметра — перелопатить их все нельзя без слома. `initCause()` даёт таким исключениям дослать причину после создания. Прогнал все граничные случаи на JDK 21:

```
1) getCause без установки: null
2) после initCause: java.lang.IllegalStateException: boom
3) повторный initCause -> IllegalStateException: Can't overwrite cause with ...: again
4) cause=self -> IllegalArgumentException: Self-causation not permitted
5) initCause после ctor-with-cause -> IllegalStateException: Can't overwrite cause with ...: late
```

> [!warning] ⚠️ Подводный камень  
> `initCause()` зовётся **не более одного раза** и только если причина ещё не стоит. Второй вызов → `IllegalStateException` («Can't overwrite cause»). Если причину уже поставил конструктор `(message, cause)` — `initCause()` тоже кинет `IllegalStateException` (слот занят). Назначить исключение причиной самому себе → `IllegalArgumentException` («Self-causation not permitted»), иначе `getCause().getCause()...` зациклится на себя. **Практический вывод:** в новом коде всегда конструктор `(message, cause)`; `initCause()` — только когда заворачиваешь тип, чей конструктор причину не принимает.

> [!tip] 💡 Из C#  
> Поле `cause` = `InnerException`, `getCause()` = чтение `.InnerException`. А вот прямого аналога `initCause()` в C# **нет** — там `InnerException` ставится только конструктором и дальше read-only. Само существование `initCause()` — звоночек, что это штука с историей.

#### Цепочка вниз — и почему `suppressed` цепляется вбок

Цепочка `cause` линейна и направлена вниз: `getCause()` → следующее → его `getCause()` → … до корня, где `getCause()` вернёт `null`. Когда исключение долетает наверх неперехваченным, JVM проходит эту цепочку и печатает каждое звено блоком `Caused by:`.

А `suppressed` (из урока 6.5) цепляется принципиально иначе. Диаграмма показывает разницу, которую проза легко проскакивает:

![[Pasted image 20260707165532.png]]

Слева вертикаль — цепочка `cause`: причинность вниз через `getCause()`, печатается `Caused by:`. Справа, вбок — `suppressed`: исключение, возникшее **дополнительно** при закрытии ресурса в `try-with-resources`, пока уже летело первичное. Оно не причина — оно побочный эффект: своё поле (`suppressed`, это массив), своё API (`getSuppressed()`), свой блок `Suppressed:` в трейсе. Прогнал `try-with-resources`, где `use()` бросает основное, а `close()` — вторичное:

```
Поймали первичное: сбой при работе
getCause: null
getSuppressed: IllegalStateException - сбой при close
--- полный трейс ---
java.lang.RuntimeException: сбой при работе
	at SuppressedDemo$Resource.use(SuppressedDemo.java:3)
	at SuppressedDemo.main(SuppressedDemo.java:9)
	Suppressed: java.lang.IllegalStateException: сбой при close
		at SuppressedDemo$Resource.close(SuppressedDemo.java:4)
		at SuppressedDemo.main(SuppressedDemo.java:8)
```

`getCause()` тут `null` — close-исключение **не** стало причиной, оно ушло в `getSuppressed()`. Разница на пальцах: **`cause` отвечает «из-за чего это случилось», `suppressed` — «что ещё пошло не так попутно, пока мы убирались за собой».**

> [!tip] 💡 Из C#  
> `suppressed` в C# нет как концепции. Там `using` при исключении в `Dispose()` просто теряет/перетирает первичное — Java сознательно решила ничего не терять и сложила вторичное в отдельный карман. Чисто джавовая механика без аналога.

> [!example] 🔬 Глубже / на синьора  
> **Один сентинел объясняет всё поведение `initCause`.** В `java.lang.Throwable` (JDK 21) поле причины инициализировано **самим объектом**, а не `null`:
> 
> java
> 
> ```java
> private Throwable cause = this;                 // сентинел: «причина ещё не установлена»
> 
> public synchronized Throwable getCause() {
>     return (cause == this ? null : cause);       // this ⇒ наружу отдаём null
> }
> public synchronized Throwable initCause(Throwable cause) {
>     if (this.cause != this)                      // уже стоит (ctor ИЛИ прошлый initCause)
>         throw new IllegalStateException("Can't overwrite cause ...");
>     if (cause == this)                           // сам себе причина
>         throw new IllegalArgumentException("Self-causation not permitted");
>     this.cause = cause;
>     return this;
> }
> ```
> 
> Отсюда разом три вещи из урока: `getCause()` даёт `null` до установки (`cause == this`); повторный `initCause()` и `initCause()` после конструктора-с-cause ловят одну и ту же проверку `this.cause != this` (конструкторы `(msg, cause)`/`(cause)` пишут `this.cause = cause` напрямую, минуя сеттер); а self-causation — отдельная проверка `cause == this` для аргумента. Одно поле-сентинел — весь контракт.
> 
> **`... N more` формально:** JVM сравнивает трейс причины с трейсом обёртки с конца и печатает только уникальный префикс, заменяя `N` общих хвостовых кадров на `... N more`. Поэтому число зависит от глубины общего стека, а не от «важности» исключения.

### Бит B — свой тип исключения

Чтобы заворачивать, твой тип обязан **уметь принять `cause`** — иначе цепочку не протянуть. Отсюда каноничная форма.

**How it works.** Твоё исключение зеркалит четыре конструктора `Throwable`/`Exception`. Скомпилировал и прогнал на JDK 21:

```java
import java.io.Serial;

public class RepositoryException extends RuntimeException {
    @Serial private static final long serialVersionUID = 1L;

    public RepositoryException() { super(); }
    public RepositoryException(String message) { super(message); }
    public RepositoryException(String message, Throwable cause) { super(message, cause); }
    public RepositoryException(Throwable cause) { super(cause); }
}
```

Зачем именно четыре — они дают **точки входа** под разный вызывающий код: `()` — маркер без сообщения; `(String)` — самый частый, описание проблемы на своём уровне; `(String, Throwable)` — **главный для заворачивания** (своё сообщение + оригинал); `(Throwable)` — когда у причины уже хорошее сообщение (к тому же `super(cause)` подтянет `cause.toString()` в message — проверено, message стал `java.sql.SQLException: connection reset`). Каждый просто делегирует в `super(...)`.

> [!warning] ⚠️ Подводный камень  
> Если объявить `(String, Throwable)`-конструктор с телом `super(message)` (забыв `cause`), исключение **молча проглотит причину** — `getCause()` вернёт `null`. Наличие параметра `Throwable cause` в сигнатуре ничего не гарантирует; причину ставит только `super(message, cause)`. Проверил тремя конструкторами разом:
> 
> ```
> 1) корректный wrap, getCause: java.sql.SQLException: connection reset
> 2) super(cause), message: java.sql.SQLException: connection reset
> 3) сломанный конструктор (super(msg) без cause), getCause: null
> ```
> 
> Делегирование в `super` — обязательная часть, не декоративная.

> [!tip] 💡 Из C#  
> Та же конвенция: в .NET свой эксепшен по гайдлайнам делает `()`, `(string)`, `(string, Exception)` — это правило анализатора `CA1032`. Разница вылезет ниже — на развилке checked/unchecked, которой в C# нет.

#### Решение 1: `extends Exception` или `extends RuntimeException`

Это развилка checked/unchecked из 6.1–6.2, только теперь ты её принимаешь для своего типа. Правило `Effective Java` (Item 70): **checked** (`extends Exception`) — для условий, из которых вызывающий может осмысленно восстановиться («файл занят — попробуй другой»); компилятор заставит поймать или объявить. **Unchecked** (`extends RuntimeException`) — для багов и невосстановимого здесь (невалидный аргумент, упавшая БД).

Но индустрия сильно качнулась в unchecked даже для прикладных исключений — и это спрашивают. Канон — Spring: его `DataAccessException` (обёртка над `SQLException`) намеренно `RuntimeException`. Логика: 99% кода, словив ошибку БД, ничего полезного не сделает — только пробросит выше; сделай её checked — и `throws SQLException` засорит весь стек вызовов, каждый промежуточный метод тащит в сигнатуре чужую деталь. **Дефолт для своего прикладного типа в современном коде — `extends RuntimeException`**, если нет конкретного сценария «вызывающий обязан обработать здесь».

> [!tip] 💡 Из C#  
> Этой развилки в C# **нет вообще** — все исключения unchecked, компилятор никогда не заставляет ловить или объявлять. Пишешь `extends RuntimeException` — воспроизводишь дефолт C#. `extends Exception` — вес, которого в .NET ты не нёс. На собесах в EU/US это любят именно из-за слепого пятна у .NET-разработчиков.

> [!example] 🔬 Глубже / на синьора  
> Почему уклон именно в unchecked, а не «checked строже — значит безопаснее»: checked-исключение — это часть **сигнатуры** метода, то есть часть его публичного контракта. Пробрасывая checked-тип через N слоёв, ты вписываешь деталь нижнего слоя в контракт каждого промежуточного — и любое изменение источника ошибки рябью идёт по всем `throws`. Unchecked снимает этот налог: кто хочет — ловит, кто не хочет — не обязан, а контракт слоя остаётся чистым. Обратная сторона — компилятор больше не напоминает про обработку, поэтому checked осознанно оставляют там, где обработка на вызывающем **обязательна и возможна**.

#### Решение 2: зачем строка `serialVersionUID`

`Throwable implements Serializable`, значит **любое** исключение сериализуемо (исторически — чтобы долетать через RMI/распределённые вызовы). У сериализуемого класса есть «версия формата» — `serialVersionUID`. Не объявишь явно — компилятор сгенерит число из структуры класса; добавил поле → число поехало → при чтении старого сериализованного экземпляра в новую версию класса прилетит `InvalidClassException`.

> [!warning] ⚠️ Подводный камень  
> Объявляй `serialVersionUID` руками и контролируй сам. На практике до проблем с исключениями доходит редко, но линтеры орут, и это дешёвая гигиена (плюс ты не контролируешь, сериализует ли их фреймворк под капотом). С **Java 14** есть аннотация `@Serial` — вешаешь на это поле, и компилятор проверит, что serialization-члены объявлены правильно. Глубже в механику `Serializable` — урок 19.

#### Решение 3: а может, свой тип не нужен

Самая частая ошибка — плодить классы исключений на каждый чих. `Effective Java` Item 72 прямым текстом: **предпочитай стандартные**. JDK уже даёт готовые, которые все знают наизусть: `IllegalArgumentException` (аргумент невалиден), `IllegalStateException` (объект в неподходящем состоянии), `NullPointerException` (через `Objects.requireNonNull`), `UnsupportedOperationException` (операция не поддерживается), `IndexOutOfBoundsException`, `ConcurrentModificationException`. Их знает каждый, трейс читается мгновенно, не плодишь классы.

Свой тип оправдан ровно в двух случаях. Первый — нужен **отдельный ловибельный тип**: `catch (InsufficientFundsException e)` → показать «пополнить», где `catch (Exception e)` не годится. Второй — нужны **диагностические поля** (Item 75): код ошибки, ID сущности, проблемное значение — чтобы обработчик реагировал программно, а не парсил текст. Прогнал на JDK 21:

```java
import java.io.Serial;

public class OrderProcessingException extends RuntimeException {
    @Serial private static final long serialVersionUID = 1L;

    private final long orderId;
    private final int errorCode;

    public OrderProcessingException(String message, long orderId, int errorCode, Throwable cause) {
        super(message, cause);            // cause-конструктор всё равно на месте
        this.orderId = orderId;
        this.errorCode = errorCode;
    }
    public long getOrderId() { return orderId; }
    public int getErrorCode() { return errorCode; }
}
```

Обработчик выше читает `e.getErrorCode()` и ветвится — `retry` / `alert` / вернуть юзеру код (`order=42 code=503 -> retry`). Текст — для логов, поля — для логики.

> [!tip] 💡 Из C#  
> Item 72 — это твои `ArgumentException` / `InvalidOperationException` / `ArgumentNullException`. Маппинг почти дословный: `IllegalArgumentException` ≈ `ArgumentException`, `IllegalStateException` ≈ `InvalidOperationException`. Привычка не плодить свои типы там, где есть стандартный, — та же.

### Бит C — wrap-and-rethrow

Всё, что мы собрали, существует ради одного движения на границе слоёв.

**How it works.** Паттерн `Effective Java` Item 73 — «exception translation» (частный случай с сохранением `cause` — «exception chaining»):

```java
public User findById(long id) {
    try {
        return jdbcQuery(id);                          // тут вылетает SQLException
    } catch (SQLException low) {
        throw new RepositoryException(
            "Не удалось загрузить пользователя id=" + id, low);   // перевод + сохранение cause
    }
}
```

Прогнал двухслойный сценарий (сервис зовёт репозиторий, репозиторий заворачивает, сервис пробрасывает как есть):

```
Exception in thread "main" RepositoryException: Не удалось загрузить пользователя id=42
	at UserRepository.findById(WrapDemo.java:14)
	at UserService.load(WrapDemo.java:22)
	at WrapDemo.main(WrapDemo.java:30)
Caused by: java.sql.SQLException: connection reset
	at UserRepository.findById(WrapDemo.java:12)
	... 2 more
```

Сверху — доменное «не смог», снизу — техническое «коннект отвалился». В 3 ночи у тебя оба конца, а `... 2 more` — это общие кадры `load` + `main`.

![[Pasted image 20260707165557.png]]

Два мотива, оба обязательны. **Не протекать деталями реализации:** `SQLException` кричит «я работаю через JDBC»; вылетел наружу — весь код выше прибит к этому. `RepositoryException` говорит на языке слоя данных, и под капотом завтра хоть Mongo — тип и сигнатура не меняются. **Не терять первопричину:** передаёшь `low` как `cause` — в трейсе остаётся весь хвост `Caused by:` до корня.

> [!warning] ⚠️ Подводный камень  
> Заворот **без** передачи cause — `throw new RepositoryException("не смог")` без `low` — грех: тип сменил, первопричину выкинул, в трейсе голое «не смог» и ни слова про `connection reset`. Дебажить нечем. (Это один из антипаттернов обработки; остальные — в 6.10–6.11, здесь флагаю только этот, потому что он ломает ровно то, что мы строим.)

#### Когда НЕ заворачивать

Перевод осмыслен только на **настоящих границах абстракции**, где нижний тип реально чужероден слою выше. Внутри одного слоя — незачем. И обратное: **если нижнее исключение само уместно — пробрасывай как есть**. Классика — валидация: метод бросил `IllegalArgumentException`, это и есть правильный, всем понятный тип; заворачивать его в свой `MyValidationException` — лишний слой, прячущий читаемое стандартное исключение. Item 72 (бери стандартное) и Item 73 (переводи только через границу) работают в паре, а не в конфликте.

> [!tip] 💡 Из C#  
> Дословный аналог — `catch (SqlException ex) { throw new RepositoryException("...", ex); }`, где `ex` уходит в `InnerException`. Тот же exception translation, та же дисциплина «не оборачивай ради оборачивания». Единственная джава-специфика — выбирая тип-обёртку, ты ещё решаешь checked/unchecked, чего в C# нет.

> try { throw new SQLException("db down"); }
> catch (SQLException e) { throw new RepositoryException("fail", e); }
> ```
> 
> **Ответ:** `null` (проверил запуском). Параметр `Throwable cause` в сигнатуре есть, но в `super(msg)` не передан — причину ставит только `super(message, cause)` или `initCause()`.
> 
> _Follow-up:_ «как починить?» → `super(msg, cause)`. _Follow-up:_ «а если `super` уже вызван с одним аргументом?» → `this.initCause(cause)` в теле, ровно один раз, но это уже костыль — правильнее сразу `super(msg, cause)`.
> 
> _Слабый ответ:_ «вернёт `SQLException`, ведь мы его передали» — путает наличие параметра с фактом установки причины.

> [!question]- 2. «Чем `cause` отличается от `suppressed`?» _(классика)_  
> **Model answer (RU):** `cause` — причинность, «X случилось из-за Y»: линейная цепочка вниз, читается `getCause()`, печатается `Caused by:`. `suppressed` — вторичные исключения, возникшие попутно (типично при закрытии ресурса в `try-with-resources`, пока уже летело первичное): массив, читается `getSuppressed()`, печатается `Suppressed:`. Разные поля, разный API, разный смысл.
> 
> **Model answer (EN):** `cause` is causality — "X happened because of Y": a linear chain downward, read via `getCause()`, printed as `Caused by:`. `suppressed` holds secondary exceptions that arose alongside — typically while closing a resource in try-with-resources as the primary was already propagating: an array, read via `getSuppressed()`, printed as `Suppressed:`. Different fields, different API, different meaning.
> 
> _Follow-up:_ «откуда берётся suppressed?» → и тело `try`, и `close()` бросили — Java оставляет первичным то, что из тела, а из `close()` цепляет как suppressed, чтобы не потерять. _Follow-up:_ «в обычном catch работает?» → нет, автозаполнение только в try-with-resources; руками — `addSuppressed()`.

> [!question]- 3. «Своё исключение — `extends Exception` или `RuntimeException`?»  
> **Model answer (RU):** Зависит от восстановимости: восстановим вызывающим → checked (`Exception`); баг или «здесь не починить» → unchecked (`RuntimeException`). Но для прикладных исключений современный уклон — unchecked, чтобы не тащить `throws` через весь стек; поэтому Spring `DataAccessException` — `RuntimeException`.
> 
> **Model answer (EN):** It depends on recoverability: recoverable by the caller → checked (`Exception`); a bug or "can't fix it here" → unchecked (`RuntimeException`). But for application exceptions the modern lean is unchecked, to avoid dragging `throws` through the whole stack — which is why Spring's `DataAccessException` is a `RuntimeException`.
> 
> _Follow-up (ловушка):_ «то есть checked вообще не нужны?» → нужны точечно, когда вызывающий обязан и может среагировать; не «всё unchecked», а «unchecked по умолчанию, checked осознанно». _Follow-up:_ «а в C#?» → выбора нет, всё unchecked — повод объяснить, зачем Java завезла checked (контракт компилятора на ожидаемые сбои).

> [!question]- 4. «Сколько раз можно звать `initCause()` и зачем он, если есть конструктор?»  
> **Ответ:** Не более одного раза и только если cause ещё не стоит — иначе `IllegalStateException`. Существует ради легаси: цепочки причин появились в Java 1.4, а исключения со старыми конструкторами (без cause-параметра) уже были — `initCause()` даёт им дослать причину. Ограничение — потому что cause логически финален (в исходнике поле-сентинел `cause = this`, перезапись = потеря исходной).
> 
> _Follow-up:_ «уже передал cause в конструктор, потом дёрну initCause?» → `IllegalStateException`, слот занят (проверял). _Follow-up:_ «cause = самому себе?» → `IllegalArgumentException`, иначе `getCause()` зациклится.

> [!question]- 5. «Зачем исключению `serialVersionUID`?»  
> **Ответ:** `Throwable implements Serializable`, значит любое исключение сериализуемо (исторически — чтобы долетать через RMI/распределённые вызовы). `serialVersionUID` — версия формата; объявляешь явно, чтобы контролировать совместимость при десериализации. Не объявишь — компилятор сгенерит число из структуры класса, и оно поедет при любом изменении полей → `InvalidClassException` между версиями.
> 
> _Follow-up:_ «а если я никогда не сериализую исключения?» → линтеры всё равно орут, дешёвая гигиена; плюс не контролируешь, сериализует ли их фреймворк. _Follow-up:_ «что такое `@Serial`?» → аннотация с Java 14, помечает serialization-члены для проверки компилятором.

> [!question]- 6. «Поймал `IllegalArgumentException` из валидатора. Завернуть в свой `ValidationException`?»  
> **Ответ:** Нет. `IllegalArgumentException` — стандартный, всем понятный, уместный тип; заворачивать — лишний слой, прячущий читаемое исключение (Item 72 + 73). Перевод осмыслен только на границе абстракции, где нижний тип чужероден слою (`SQLException` в репозитории — да); внутри слоя или когда тип уже подходит — пробрасывай как есть.
> 
> _Follow-up:_ «когда свой тип оправдан?» → отдельный `catch`-тип (вызывающий ловит именно его) или доп. поля для диагностики (код ошибки, ID). _Follow-up:_ «не противоречит ли wrap-and-rethrow?» → нет: оборачивай на границе, не оборачивай ради оборачивания. Item 72 и 73 — пара, а не конфликт.

> [!tip] Устная шпаргалка  
> **Сохранить первопричину:** `new RepositoryException(msg, original)`, достать — `getCause()`. Не передал — потерял корень.  
> **Цепочка причин:** исключение несёт внутри другое; `getCause()` вниз до корня, JVM печатает `Caused by:`. Прямой аналог `InnerException`.  
> **cause vs suppressed:** cause — «из-за чего», вниз, `Caused by:`. suppressed — «что упало попутно при закрытии ресурса», массив, `Suppressed:`.  
> **initCause:** один раз, только если cause не стоит; для легаси без cause-конструктора. В новом коде — всегда конструктор.  
> **Свой тип checked/unchecked:** по умолчанию `RuntimeException` (не засорять `throws`); checked — осознанно, когда вызывающий обязан и может восстановиться.  
> **Четыре конструктора:** зеркалят `Throwable` — `()`, `(String)`, `(String, Throwable)`, `(Throwable)`; последние два несут cause; каждый делегирует в `super`.  
> **wrap-and-rethrow:** на границе слоёв ловлю чужеродное низкоуровневое, бросаю своё высокоуровневое, оригинал передаю как cause — не протекаю деталями и не теряю первопричину.
