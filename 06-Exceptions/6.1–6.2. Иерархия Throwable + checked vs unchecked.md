
Пишешь метод, который читает файл через `Files.readAllLines`, — и `javac` наотрез отказывается его компилировать, пока ты что-нибудь не сделаешь с `IOException`. Пишешь соседний метод, который дёргает `s.length()` на строке, вполне способной прилететь `null`, — и компилятор молчит, хотя в проде это рванёт `NullPointerException`. Оба случая — исключение, объект, летящий вверх по стеку; но к одному компилятор придирается насмерть, а на другое ему плевать. Этот урок — про то, где именно в языке проходит граница между «заставлю обработать» и «пропущу молча» и почему её провели так, а не иначе. Практический выход простой: перестаёшь гадать, какое исключение компилятор потребует обработать, и читаешь это прямо по дереву классов.

**Mental model.** Позиция класса в дереве `Throwable` — это закон, по которому компилятор решает: принудить тебя обработать исключение или пропустить молча. Всё остальное в теме разворачивается из этого одного факта.

> [!tip] 💡 Из C#  
> Сама модель исключений переносится один-в-один и переучивать её не надо: объект бросается `throw`, летит вверх по стеку, либо ловится подходящим `catch`, либо доходит до верха и роняет поток. `try/catch/finally` почти идентичны.
> 
> Новое ровно одно, и в C# этого **нет в принципе**: часть исключений компилятор Java _заставляет_ обработать. «Обработать» = либо поймать в `try/catch`, либо расписаться в сигнатуре — «отсюда может вылететь вот это» — клаузой `throws`. В C# любое исключение можно молча проигнорировать на уровне компилятора; клаузы `throws` не существует, а `/// <exception>` в XML-доке — это текст для людей, компилятор его не проверяет.

Два легальных способа «обработать» на примере того самого метода — поймать или объявить:

```java
// вариант 1: поймать здесь
List<String> read(Path path) {
    try { return Files.readAllLines(path); }
    catch (IOException e) { return List.of(); }
}
// вариант 2: расписаться и спихнуть выше
List<String> read(Path path) throws IOException {   // компилируется
    return Files.readAllLines(path);
}
```

`throws IOException` в сигнатуре — это **не** «бросаю», а объявление «могу бросить, разбирайтесь выше». Третьего компилятор не даёт: без одного из этих двух `javac` просто не соберёт код.

### Дерево `Throwable`

Всё, что можно `throw` и поймать в `catch`, обязано наследоваться от `java.lang.Throwable` — это корень. У него ровно два прямых потомка: `Error` и `Exception`. И вот первый соблазн — решить, что граница checked/unchecked проходит здесь, между `Error` и `Exception`. Нет. Она режет `Exception` пополам.

![[Pasted image 20260707163516.png]]

Читаем дерево. **Unchecked** — это `Error` и `RuntimeException` со всеми потомками; на них компилятор не реагирует никак, можешь ловить, можешь игнорировать. **Checked** — всё остальное под `Exception`, кроме ветки `RuntimeException`. Именно на checked компилятор и надевает наручники.

### Контракт компилятора

Правило чисто **compile-time** (JLS §11.2): если из метода может вылететь checked-исключение, ты обязан _либо_ поймать его в `try/catch` прямо здесь, _либо_ объявить в сигнатуре через `throws` и спихнуть на вызывающего. Третьего `javac` не даёт. На unchecked правило не распространяется — `NPE` летит откуда угодно, декларировать его никто не заставляет.

> [!tip] 💡 Из C#  
> В C# дерево тоже есть — `System.Exception` в корне, ниже `SystemException` и прочее, — но там **всё unchecked поголовно**: ни клаузы `throws`, ни compile-time принуждения. `throws` в Java — это тот же смысл, что XML-док `/// <exception>` в C#, но с зубами: компилятор реально откажет в сборке, а не оставит подсказку для людей.

### Почему границу провели именно так — три корзины

Линия checked/unchecked — не каприз. За ней один вопрос: **может ли вызывающий разумно среагировать на сбой?** Bloch раскладывает это на три корзины (Effective Java, Item 70).

`Error` — проблемы уровня JVM: кончилась куча (`OutOfMemoryError`), стек переполнился рекурсией (`StackOverflowError`), класс не загрузился. Реагировать в коде нечем, приложение по сути уже мертво → не лови, декларировать незачем → unchecked.

`RuntimeException` — **баг программиста**, нарушенный контракт. `NullPointerException` — не проверил `null`. `IndexOutOfBoundsException` — полез за границу. `IllegalArgumentException` — передал в метод мусор, который тот запретил. Это не «внешний мир подвёл», это ты написал неправильный код. Заставлять писать `throws NullPointerException` на каждом методе, который теоретически разыменовывает ссылку, — абсурд: пришлось бы обмазать `throws` вообще всё → unchecked.

`Checked` (`IOException`, `SQLException`, …) — **восстановимый сбой по внешней причине**: файл не нашёлся, сеть отвалилась, БД недоступна. Ты сделал всё верно, подвёл внешний мир — и вызывающий может разумно среагировать: ретрай, фолбэк, сообщение пользователю. Компилятор силой напоминает: «этот сценарий реален, обработай осознанно» → checked.

> [!example] 🔬 Глубже / на синьора  
> **В байткоде разницы между checked и unchecked нет вообще.** Деление — чистая фикция `javac`, в рантайме его не существует, JVM про него не знает. Вот два метода, скомпилированные на JDK 21 (`javap -v`, проверено):
> 
> ```
> void checkedThrow() throws java.io.IOException;
>     Code:
>       stack=3, locals=1, args_size=1
>          0: new           #7   // class java/io/IOException
>          3: dup
>          4: ldc           #9   // String io
>          6: invokespecial #11  // IOException."<init>"
>          9: athrow
>     Exceptions:                 <- отдельный атрибут, СНАРУЖИ Code
>       throws java.io.IOException
> 
> void uncheckedThrow();          <- атрибута Exceptions просто НЕТ
>     Code:
>          0: new … 3: dup … 4: ldc … 6: invokespecial … 9: athrow
> ```
> 
> Тело обоих — байт-в-байт одно и то же: `new → dup → ldc → invokespecial → athrow`. Клауза `throws` живёт в атрибуте `Exceptions`, который лежит _рядом_ с `Code`, но **не внутри** него, и при исполнении JVM в него **не заглядывает**. Атрибут нужен ровно двум читателям: (1) компилятору, когда тот компилирует _твоего вызывающего_ и требует обработки, и (2) рефлексии (`Method.getExceptionTypes()`). Сам `try/catch` в байткоде — это `exception table`, структурно одинаковая для любого типа; различается лишь колонка с классом из `catch`.
> 
> Прямое следствие: раз проверка — compile-time, её можно обойти. Через дженерик-дыру (`sneaky throws`) метод бросает checked, не объявив его, и JVM спокойно проглатывает (проверено: `IOException` вылетел из метода без `throws`). Механизм и боль с лямбдами в Stream API — урок 9.6; сейчас важно, что дыра существует _потому что_ в рантайме проверять некому.

### Честная калибровка: а это вообще хорошая идея?

На собесе про checked спрашивают не только «что это», но и «а правильно ли так?» — и хотят обе стороны.

Checked-исключения — **спорное решение, и индустрия в основном проголосовала против**. Kotlin убрал их полностью (всё unchecked), Scala никогда не имел, C#, как ты знаешь, тоже. Spring оборачивает проверяемый `SQLException` в unchecked-иерархию `DataAccessException` — чтобы не обмазывать `throws`/`try-catch` каждый репозиторий. Даже сам Bloch, автор кучи checked-API в JDK, в Item 71 пишет: «избегай _ненужных_ checked» — плоди их только когда сбой реально восстановим И вызывающий реально что-то сделает.

Аргументы **за**: компилятор не даёт молча проигнорировать восстановимый сбой (забыть физически нельзя); сигнатура с `throws` документирует контракт машинно-проверяемо, а не комментарием, который протухает.

Аргументы **против**: ломается композиция — лямбды и стримы не пробрасывают checked, начинается боль с обёртками; провоцируют антипаттерн «проглотил» — уставший разработчик пишет `catch (IOException e) {}` пустым телом, лишь бы компилировалось (хуже, чем если бы исключение летело дальше); провоцируют `throws Exception` на всю широту — формально компилируется, а контракт убит; ломают версионирование — добавил в публичный метод новый `throws SomeCheckedException` и сломал сборку всем вызывающим.

Позиция на собесе: не вставай ни на одну сторону категорично. «Идея — машинная гарантия обработки восстановимых сбоев — здравая; на практике плохо масштабируется (композиция, версионирование), поэтому современная Java их минимизирует, а Spring заворачивает в unchecked; Bloch, Item 71, советует то же». Звучит как инженер, который думал, а не зубрил.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`Error` — unchecked, хотя «серьёзно».** Легко на автомате решить «`Error` — это критично, значит checked». Наоборот: он настолько фатален (`OutOfMemoryError`, `StackOverflowError`), что заставлять его декларировать бессмысленно — ловить его ты и так не должен.
> 2. **Override может только _сужать_ `throws`, не расширять** (JLS §8.4.8.3). Кто-то держит ссылку типа `Parent`, видит в контракте `throws IOException` и готов ловить только это; если сабкласс объявит `throws Exception`, он нарушит ожидания. Проверено на JDK 21:
>     
>     java
>     
>     ```java
>     class Parent { void read() throws IOException {} }
>     class NarrowOk  extends Parent { void read() throws FileNotFoundException {} } // OK: подтип
>     class DropOk    extends Parent { void read() {} }                             // OK: убрал вовсе
>     class WidenBad  extends Parent { void read() throws Exception {} }            // ОШИБКА компиляции
>     ```
>     
>     `javac`: `read() in WidenBad cannot override read() in Parent — overridden method does not throw Exception`. На unchecked правило не действует — любые `RuntimeException` при override добавлять можно.
> 3. **`throws Exception` компилируется, но убивает контракт.** Это говорит вызывающему «могу бросить вообще что угодно» — то есть не говорит ничего. Он вынужден ловить `Exception` целиком или прокидывать `throws Exception` дальше, и зараза расползается по стеку. Объявляй конкретный тип.
> 4. **`catch (Throwable)` / `catch (Error)` — почти всегда баг.** Поймав `Throwable`, ты заодно ловишь `OutOfMemoryError` и `StackOverflowError` — фатальные состояния, где приложению надо дать упасть. Маскировка фатального хуже самого падения. (Полный разбор антипаттернов — урок 6.11.)
> 5. **Checked внутри лямбды не компилируется.** `list.forEach(s -> { throw new IOException(s); })` → `javac`: `unreported exception IOException`. Причина прямо из всего выше: лямбда реализует метод функционального интерфейса (`Consumer.accept`), который `throws` не объявляет — значит и лямбда не имеет права выпустить checked. Отсюда вся боль с checked в Stream API. Глубоко — урок 9.6.
> 6. **`return` в `finally` проглатывает исключение целиком.** `try { throw new IllegalStateException(); } finally { return 42; }` → метод вернёт `42`, исключение наружу не вылетит (проверено). Классический антипаттерн: молча теряешь исключение. Детали `finally` — урок 6.4.

#### На собесе

> [!question]- Где в дереве `Throwable` проходит граница checked/unchecked?  
> Не между `Error` и `Exception`, как кажется. Unchecked — это `Error` и `RuntimeException` со всеми потомками; checked — остальные потомки `Exception`. Граница режет `Exception` пополам. Критерий — не «насколько серьёзно», а «может ли вызывающий разумно восстановиться».
> 
> **Model answer (RU):** Корень — `Throwable`, единственное, что можно `throw`/`catch`; под ним `Error` и `Exception`. Граница проходит не между ними, а внутри `Exception`: unchecked — это `Error` и `RuntimeException` с потомками, checked — прочие потомки `Exception`. Checked обязан быть либо пойман, либо объявлен в `throws` — это правило компиляции, не рантайма.
> 
> **Model answer (EN):** The root is `Throwable`, the only thing you can `throw`/`catch`; below it sit `Error` and `Exception`. The line doesn't run between them — it runs inside `Exception`: unchecked is `Error` and `RuntimeException` with their subtypes, checked is the rest of `Exception`'s subtypes. A checked exception must be either caught or declared with `throws` — a compile-time rule, not a runtime one.
> 
> _Follow-up:_ «`Error` — checked или unchecked?» → unchecked, контринтуитивно: настолько фатален, что ловить не надо. _Follow-up:_ «Почему тогда `RuntimeException` unchecked, если он под `Exception`?» → это баги программиста (`NPE`, выход за границу), они возможны на каждой строке; декларировать каждый — сделать `throws` бессмысленным шумом.
> 
> _Слабые ответы:_ «граница между `Error` и `Exception`» — фатально, самая частая ошибка. «`Error` — checked, потому что серьёзно» — нет, unchecked.

> [!question]- Есть ли в байткоде разница между checked и unchecked? _(сильный подвох)_  
> Нет. Это compile-time фикция `javac`. `athrow` идентичен, `exception table` структурно одинакова. Клауза `throws` живёт в отдельном атрибуте class-файла `Exceptions`, который JVM при исполнении игнорирует.
> 
> _Follow-up:_ «Тогда можно бросить checked, не объявив?» → да, через `sneaky throws` (дженерик-дыра) — JVM проглотит, проверять в рантайме некому. _Follow-up:_ «Зачем тогда атрибут `Exceptions` хранится?» → для `javac` при компиляции _вызывающего_ и для рефлексии (`getExceptionTypes`).

> [!question]- В C# нет checked, в Java есть. Кто прав?  
> Дай обе стороны, не вставай категорично. **За:** машинно-проверяемая гарантия, что восстановимый сбой не проигнорируют молча. **Против:** ломает композицию (лямбды/стримы), провоцирует проглатывание и `throws Exception`, ломает версионирование API. Kotlin/Scala/C# обошлись без checked; Spring заворачивает их в unchecked `DataAccessException`; Bloch (Item 71) советует минимизировать.
> 
> **Model answer (RU):** В C# все исключения unchecked, клаузы `throws` нет вовсе. Идея checked здравая — компилятор гарантирует, что восстановимый сбой обработают осознанно, — но масштабируется плохо: не пробрасывается через лямбды, провоцирует пустые `catch` и `throws Exception`, ломает совместимость API при добавлении нового checked. Поэтому индустрия в основном от них ушла, а современная Java их минимизирует.
> 
> **Model answer (EN):** In C# every exception is unchecked — there's no `throws` clause at all. The idea behind checked is sound — the compiler guarantees a recoverable failure is handled deliberately — but it scales poorly: it doesn't propagate through lambdas, it invites empty `catch` blocks and `throws Exception`, and it breaks API compatibility when a new checked type is added. That's why the industry largely moved away from them and modern Java keeps them to a minimum.
> 
> _Follow-up:_ «А ты бы добавлял checked в свой API?» → редко: только если сбой реально восстановим И вызывающий реально что-то сделает; иначе unchecked.

> [!question]- Может ли переопределённый метод бросать более широкое checked, чем родитель?  
> Нет. `throws` при override можно только сузить или убрать, не расширить (JLS §8.4.8.3). Вызывающий по ссылке родительского типа готов ловить лишь то, что объявлено в родителе; расширение нарушило бы контракт.
> 
> _Follow-up:_ «А unchecked можно добавить при override?» → да, любые `RuntimeException`, потому что они и так не декларируются и не проверяются.

> [!question]- Стоит ли ловить `Error` или `Throwable`?  
> Нет. `OutOfMemoryError`, `StackOverflowError` — состояние, из которого осмысленно не восстановиться, приложению надо дать упасть. Поймав, ты маскируешь фатальное и оставляешь JVM в неопределённом состоянии. `catch (Throwable)` — тем более нет, он включает `Error`. Изредка оправдан на самой границе фреймворка (логнуть и пробросить), но не в обычном коде.

> [!question]- Что напечатает этот код?
> 
> java
> 
> ```java
> static int f() {
>     try { throw new IllegalStateException(); }
>     finally { return 42; }
> }
> ```
> 
> `42`. `return` в `finally` перекрывает начатую раскрутку исключения — оно не вылетит наружу. _Follow-up:_ «Это хорошо или плохо?» → плохо, классический антипаттерн: молча теряешь исключение (детали — урок 6.4).

> [!question]- Устная шпаргалка — стержень темы  
> **Позиция класса в дереве `Throwable` — это закон, по которому компилятор решает, принуждать тебя обрабатывать исключение или нет.** Отсюда всё:  
> — Иерархия: `Throwable` в корне → `Error` и `Exception`; `Exception` делится на `RuntimeException` и остальное.  
> — Граница: unchecked = `Error` + `RuntimeException` с потомками; checked = прочие потомки `Exception`. Режет `Exception` пополам, не проходит между `Error` и `Exception`.  
> — Три корзины: `Error` — сбой JVM, реагировать нечем; `RuntimeException` — баг программиста; checked — восстановимый внешний сбой.  
> — Байткод: разницы нет, `athrow` одинаков, `throws` сидит в атрибуте `Exceptions` мимо рантайма.  
> — C#: checked нет вовсе; решение спорное, индустрия минимизирует, Spring заворачивает в unchecked.  
> — Override: `throws` только сужать или убирать, не расширять.  
> — `Error`/`Throwable` не лови — дай приложению упасть.
