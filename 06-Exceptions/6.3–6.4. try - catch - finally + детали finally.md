Представь типичный метод в backend-сервисе: взял соединение из пула, отправил запрос, начал читать ответ — и посередине `parseResponse(...)` падает, потому что прилетел кривой JSON. Что с соединением? Если не вернёшь его в пул — оно протекло; сделай так десять тысяч раз под нагрузкой, и пул пуст, сервис висит, дежурного будят в три ночи. Тебе нужен блок, который выполнится **при любом исходе** метода: вернулись штатно — освободи, прилетело исключение — всё равно освободи, кто-то сделал `return` в середине — снова освободи. Не «обычно», а гарантированно. Эту гарантию даёт `finally`, и весь урок — про то, как именно она устроена и где в ней спрятаны грабли, на которых сыплются даже опытные.

**Mental model.** `finally` — это не «код в конце метода», а обязательство JVM, привязанное к **каждому ребру выхода** из `try`. Именно поэтому он потом сможет перехватывать и `return`, и летящее исключение.

Базовый конструкт:

```java
try {
    // защищаемый блок: тут может полететь исключение
} catch (SomeException e) {
    // ловим, если тип летящего исключения подходит
} finally {
    // выполняется ВСЕГДА, на любом пути выхода из try
}
```

`catch` ловит летящее вверх по стеку исключение, если оно подходит по типу — помнишь дерево `Throwable` из 6.1–6.2: `catch (IOException e)` поймает и `FileNotFoundException`, потому что та — подтип. `finally` JVM обязуется выполнить перед тем, как управление покинет `try`, каким бы способом оно его ни покидало.

> [!tip] 💡 Из C#  
> Синтаксис у тебя уже в руках — `try`/`catch`/`finally` выглядят практически один-в-один и семантика «finally всегда» та же, так что «что это вообще» проскакиваем. Вся ценность — в Java-специфике: `multi-catch` (`catch (A | B e)`), жёсткие правила порядка `catch`-блоков и — самое интересное — что `finally` творит, когда внутри него `return` или `throw`. Вот там Java расходится с C# радикально, и именно это любят на собесах.

### Три пути выхода — и все через `finally`

Из `try` есть ровно три способа выйти, и каждый проходит через `finally`. Это сердце конструкции: `finally` сидит не «в конце», а на каждом пути, которым управление может покинуть блок.

![[Pasted image 20260707163609.png]]

На конкретике: `fetch()` бросает, `catch (ParseException e)` ловит только `ParseException`. Если летит `IllegalStateException` — этот `catch` пропускается целиком, как будто его нет, но `finally` всё равно срабатывает, `pool.release(conn)` выполняется, и **уже после** исключение продолжает лететь вверх. Метод завершился с исключением, но ресурс не протёк — вот весь смысл `finally`.

> [!tip] 💡 Из C#  
> Три пути и гарантия — один-в-один. Перенос почти бесплатный; расхождения начнутся ниже, на `multi-catch` и на том, что Java разрешает делать **внутри** `finally`.

### Порядок `catch`: от частного к общему

Когда `catch`-блоков несколько, JVM проверяет их **сверху вниз** и берёт **первый подходящий по типу**. А раз `catch (IOException e)` ловит и потомков (включая `FileNotFoundException`), то широкий тип, стоящий выше узкого, «съедает» исключения, которые ты хотел поймать узким `catch` ниже. Java ловит это **на компиляции** (проверено на JDK 21):

```java
try { new FileInputStream("x"); }
catch (IOException e) {}             // широкий — ловит и FileNotFoundException
catch (FileNotFoundException e) {}   // узкий — сюда уже ничего не дойдёт
```

```
error: exception FileNotFoundException has already been caught
```

Это не рантайм-сюрприз, а статическая проверка достижимости (JLS §11.2.3): компилятор видит, что во второй `catch` нет ни одного пути, и отказывается собирать класс. Правило железное — **узкие типы выше, широкие ниже**.

> [!tip] 💡 Из C#  
> Порядок `catch` от частного к общему тот же, и недостижимый `catch` — тоже compile error (CS0160). Но «ловлю checked, которое не может полететь» в C# невозможно как класс ошибок — там checked-исключений нет, любой `catch (SomeException)` легален. В Java же `catch` на checked-исключение, которое `try` в принципе бросить не может, — снова compile error: `exception X is never thrown in body of corresponding try statement`. Это прямое следствие checked-контракта из 6.1–6.2: компилятор статически знает множество того, что может полететь. С `RuntimeException` такого нет — их можно ловить всегда.

### `multi-catch`: один блок на несколько типов

Частая ситуация — два разных исключения, реакция одна. С Java 7 (часть Project Coin, отдельного `JEP` нет) есть `multi-catch` — несколько типов через `|`:

```java
try { process(); }
catch (IOException | SQLException e) {
    log.error("failed", e);
    throw new ServiceException(e);
}
```

Две тонкости про `e`, обе проверены. Её статический тип — **общий супертип** (least upper bound) перечисленных классов; для `IOException | SQLException` это `Exception`, и `javac` это подтверждает буквально — попытка вызвать `e.getSQLState()` даёт `location: variable e of type Exception`. Значит специфичные методы подтипа недоступны — нужен отдельный `catch`. И `e` **implicitly final**: `e = new IOException()` внутри блока → `error: multi-catch parameter e may not be assigned`.

> [!tip] 💡 Из C#  
> В C# нет `catch (A | B)`. Эквивалент делают через exception filter: `catch (Exception e) when (e is IOException or DbException)`. Это другой механизм — `when` исполняется как предикат на стадии раскрутки стека и может содержать произвольное условие. В Java фильтров `when` нет вообще, только тип в `catch`. Java-way для «один обработчик на несколько типов» — это `|`, и ничего богаче язык не даёт.

### `return` и `throw` в `finally` — перехват выхода

Раз `finally` сидит на каждом ребре выхода, у этого есть следствие, на котором сыплются даже опытные: `finally` может **перехватить и подменить** то, что выходит из метода. Разбираем на проверенных запусках.

```java
static int f() { try { return 1; } finally { return 2; } }
```

`f()` возвращает `2`, не `1` (проверено: `returnVsReturn = 2`). `try` дошёл до `return 1` — значение готово к выходу, но выход обязан пройти через `finally`, а там свой `return 2` — новый выход, замещающий старый. `return 1` просто исчезает.

`finally` глотает **исключение** ровно так же:

```java
static int g() { try { throw new RuntimeException("boom"); } finally { return 2; } }
```

Возвращает `2`, исключение `boom` **проглочено** без всякого стектрейса (`throwVsReturn = 2`). Это худшее, что можно сделать в блоке очистки: написал `return` — и потерял реальную ошибку, летевшую из `try`. То же с `throw`: если `try` бросал `IOException`, а `finally` бросает `RuntimeException`, наружу уйдёт `RuntimeException`, а оригинал **исчезнет бесследно** — проверил, `getSuppressed()` пуст, никакой автоматической записи в suppressed тут нет (это появится только в `try-with-resources`, урок 6.5).

Есть и более тонкая ловушка — **значение `return` вычисляется ДО `finally`**:

```java
static int h() { int x = 10; try { return x; } finally { x++; } }
```

Интуиция кричит «11», ответ — **`10`** (`mutatePrimitive = 10`). `return expr` — это два шага: сначала вычислить `expr` и положить в слот возврата, потом выполнить `finally`, потом отдать запомненное. `finally` стоит **между** шагами и видит уже снятый снимок. `x++` меняет переменную, но не зафиксированный слот.

Ключевой нюанс — для примитива снимок это копия значения, а для **ссылки** снимок это копия ссылки, но объект-то один:

```java
static List<Integer> bug() {
    List<Integer> list = new ArrayList<>();
    try { return list; } finally { list.add(99); }   // объект тот же — мутация ВИДНА
}
```

Вернётся `[99]` (`mutateRef = [99]`). Снимается значение слота (для ссылки — сама ссылка), а не глубокая копия, поэтому мутация того же объекта в `finally` видна вызывающему.

> [!example] 🔬 Глубже / на синьора  
> **`multi-catch` — не сахар из двух скопированных блоков.** В байткоде это **один** handler, на который указывают **несколько записей** в exception table. Проверено `javap -c`:
> 
> ```
> static void handle();
>    Code:
>       0: invokestatic  #7    // risky()
>       3: goto          15
>       6: astore_0            // ← единственный обработчик, смещение 6
>       7: getstatic … ldc "caught" … println
>      15: return
>    Exception table:
>       from  to  target type
>          0   3      6   Class java/io/IOException
>          0   3      6   Class java/sql/SQLException   // ← тот же target 6
> ```
> 
> Код обработчика в единственном экземпляре, а в таблице две записи, обе ведут в точку `6`. До Java 7 два отдельных `catch` с одинаковым телом давали два разных handler-блока с дублированным байткодом — `multi-catch` это схлопывает.
> 
> **У самого `finally` в JVM инструкции нет.** `javac` берёт код `finally` и **физически дублирует** его в каждый путь выхода, плюс вешает catch-all обработчик на защищаемый регион. Проверено на `try { print("try"); } finally { print("finally"); }`:
> 
> ```
>  8: … println   // ← КОПИЯ #1 finally (нормальный путь)
> 16: goto 30
> 19: astore_0    // ← catch-all: ловим ЛЮБОЕ исключение
> 20: … println   // ← КОПИЯ #2 finally (путь исключения)
> 28: aload_0
> 29: athrow      // и перебрасываем пойманное дальше
> Exception table:
>    from  to  target type
>       0   8     19   any    // ← "any" = catch-all на весь try
> ```
> 
> `println("finally")` встречается **дважды** (смещения 8 и 20). Если в `try` что-то полетело, оно попадает в handler `19`, там отрабатывает **копия** `finally`, а `athrow` перебрасывает исходное исключение дальше. Так `finally` и оказывается на каждом ребре — он буквально вклеен на каждый выход.
> 
> **💡 Из C#:** исторически (до Java 6) `finally` собирался через `jsr`/`ret` (jump-to-subroutine — одна копия кода, прыжки в неё), но от этого отказались: `jsr`/`ret` ломали верификатор байткода. Теперь только дублирование. В CLR (.NET) `finally` — first-class конструкция в IL: protected region с `endfinally`, обработчик **не** дублируется. Разные стратегии: JVM дублирует на уровне `javac`, CLR держит `finally` в IL.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Никогда не клади `return` или `throw` в `finally`.** Они перебивают всё, что выходило из `try`, включая летящее исключение — оно молча теряется, а `throw` вдобавок затирает оригинал без suppression. `finally` — только для очистки (`release`, `close`, `unlock`). Линтеры (IDEA, SonarQube, ErrorProne) на это ругаются — слушай их.
> 2. **Снимок возврата снимается до `finally`.** Мутация примитива в `finally` на результат не влияет (`10`, не `11`), а вот мутация объекта по возвращаемой ссылке — видна (`[99]`). Сильный follow-up: «а если возвращать не `int`, а `List`?».
> 3. **Порядок `catch`: узкий выше широкого**, иначе `already been caught` на компиляции. И `catch` на checked, которое `try` бросить не может, — тоже compile error (`never thrown in body`).
> 4. **`finally` не выполнится, если JVM до него не доживёт:** `System.exit(n)` или `Runtime.getRuntime().halt(n)` в `try` (проверено: `finally` пропущен, но shutdown hooks при `exit` отрабатывают, при `halt` — нет), краш JVM (`kill -9`, фатальный `OutOfMemoryError`), бесконечный цикл или вечная блокировка в `try`, демон-поток, убитый при выходе JVM.
> 5. **А вот обычный `return` в `try` — `finally` НЕ отменяет** (проверено). Это ловушка на невнимательность: `return` — штатный выход (путь 1 на схеме), `finally` пропускают только аварийные сценарии, где умирает сам процесс/поток.

#### На собесе

> [!question]- Что вернёт этот метод?
> 
> java
> 
> ```java
> static int f() { try { return 1; } finally { return 2; } }
> ```
> 
> `2`. `return 1` готовит значение к выходу, но выход обязан пройти через `finally`, а там `return 2` замещает выход целиком — `return 1` теряется.
> 
> _Follow-up 1:_ `try { int x = 10; return x; } finally { x++; }` → `10`. Значение `return` снимается ДО `finally`; инкремент меняет переменную, но не зафиксированный слот. _Follow-up 2:_ `try { throw new RuntimeException(); } finally { return 2; }` → `2`, **исключение проглочено**. Тут добавь сам, не дожидаясь: «поэтому `return`/`throw` в `finally` — баг, теряется реальная ошибка». _Follow-up 3:_ «а если возвращать `List` и мутировать его в `finally`?» → мутация видна (`[99]`), потому что снимок — это копия ссылки, объект один.

> [!question]- Когда `finally` НЕ выполнится?  
> Когда JVM не доживает до него: `System.exit()` / `Runtime.halt()` в `try`, краш JVM (`kill -9`, фатальный `OutOfMemoryError`), бесконечный цикл или вечная блокировка в `try`, демон-поток, убитый при выходе JVM. `finally` привязан к нормальному потоку управления внутри живого потока — перечисленное это смерть процесса/потока, а не выход из блока.
> 
> _Follow-up:_ «а если в `try` стоит `return` — `finally` выполнится?» → **да**. `return` — штатный выход, он `finally` не отменяет. Ловушка на невнимательность.
> 
> **Model answer (RU):** `finally` не выполнится, только если JVM или поток умирают, не дойдя до него: `System.exit`, `Runtime.halt`, краш JVM, бесконечный цикл в `try`, убитый демон-поток. Любой нормальный выход из `try` — включая `return` — через `finally` проходит обязательно.
> 
> **Model answer (EN):** `finally` is skipped only if the JVM or the thread dies before reaching it: `System.exit`, `Runtime.halt`, a JVM crash, an infinite loop in `try`, or a killed daemon thread. Any normal exit from `try` — including `return` — always passes through `finally`.

> [!question]- Широкий `catch` стоит выше узкого — что будет?  
> **Compile error** — `exception FileNotFoundException has already been caught`. Не рантайм, а статическая проверка достижимости (JLS §11.2.3): во второй `catch` нет ни одного пути, потому что `IOException` уже ловит всех потомков.
> 
> _Follow-up:_ «почему компилятор это вообще видит?» → `catch` матчится по типу, а компилятор статически знает иерархию `Throwable` и порядок проверки сверху вниз. _Follow-up 2:_ «а `catch` для checked, которое `try` не бросает?» → тоже compile error (`never thrown in body`) — следствие checked-контракта.

> [!question]- `multi-catch`: какого типа `e` и можно ли переприсвоить?  
> Тип — **общий супертип** (least upper bound); для `IOException | SQLException` это `Exception`, поэтому внутри доступны только методы `Exception`. И `e` **implicitly final** — переприсвоить нельзя (`may not be assigned`).
> 
> _Follow-up:_ «чем `multi-catch` отличается в байткоде от двух `catch` с одинаковым телом?» → один handler-блок, на который указывают несколько записей в exception table с одним `target`. Не дублированный код, как было бы до Java 7.
> 
> **Model answer (RU):** `multi-catch` (`catch (A | B e)`, с Java 7) объединяет несколько типов в один обработчик; тип `e` — их общий супертип, и `e` неявно `final`. В байткоде это один handler с несколькими записями в exception table, а не два скопированных блока.
> 
> **Model answer (EN):** `multi-catch` (`catch (A | B e)`, since Java 7) merges several types into one handler; `e`'s type is their least upper bound, and `e` is implicitly `final`. In bytecode it's a single handler with multiple exception-table rows, not two duplicated blocks.

> [!question]- Как `finally` реализован в байткоде?  
> Инструкции «finally» в JVM нет. `javac` **дублирует** код `finally` в каждый путь выхода и вешает catch-all обработчик (тип `any` в exception table) на защищаемый регион; тот выполняет копию `finally` и перебрасывает исходное исключение через `athrow`. Дублирование, а не одна копия, потому что исторический механизм `jsr`/`ret` (одна копия, прыжки в неё) убрали — он ломал верификацию байткода.
> 
> _Follow-up:_ «почему это важно знать?» → объясняет, почему `finally` срабатывает на каждом выходе (он буквально вклеен на каждый) и почему может перехватить `return`/исключение. Механика прямо объясняет поведение.

> [!question]- Почему в C# `return` в `finally` — ошибка компиляции, а в Java компилируется?  
> Дизайн-решение. C# запрещает `return`/`break`/`continue`/`goto` из `finally` на компиляции (CS0157, «Control cannot leave the body of a finally clause»), потому что это почти всегда баг. Java разрешает — код собирается, ворчит только линтер, — оставляя ответственность на разработчике.
> 
> Формулируй на собесе именно так: «в Java компилируется и глотает исключение, в C# это CS0157 на компиляции». Классический контраст философий: C# защищает от выстрела в ногу, Java даёт выстрелить. Сильный сигнал для Middle+, что ты понимаешь не только Java, но и где она расходится с другими языками.

> [!question]- Устная шпаргалка  
> _«Как работают try/catch/finally?»_ — `try` защищаемый блок; `catch` ловит летящее исключение, если тип подходит, проверяя блоки сверху вниз и беря первый подходящий; `finally` выполняется на любом пути выхода — штатном, через `catch`, через `return`, через непойманное исключение. Освобождение ресурсов — его работа.
> 
> _«Что опасного в finally?»_ — `return`/`throw` внутри перебивают всё, что выходило из `try`, включая летящее исключение — оно молча теряется. Только очистка, никаких `return`/`throw`. И: значение `return` снимается ДО `finally`, поэтому мутация примитива не влияет, а мутация объекта по ссылке — видна.
> 
> _«Когда finally не сработает?»_ — `System.exit`, `Runtime.halt`, краш JVM, бесконечный цикл в `try`, убитый демон. Обычный `return` в `try` его не отменяет.
> 
> _«multi-catch?»_ — `catch (A | B e)` с Java 7, тип — общий супертип, `e` implicitly final, в байткоде один обработчик с несколькими записями в exception table. Широкий тип нельзя выше узкого — compile error.
> 
> _«Чем Java отличается от C# тут?»_ — синтаксис почти идентичен, но Java разрешает `return` в `finally` (только линтер), C# запрещает на компиляции (CS0157). И `multi-catch` в Java через `|`, а в C# через фильтр `when`, которого в Java нет.
