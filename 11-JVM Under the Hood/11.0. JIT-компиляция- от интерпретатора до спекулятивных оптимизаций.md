Знакомая сцена: после деплоя первые запросы к сервису отвечают за 800 мс, через минуту — за 15. Код тот же, данные те же. Или наивный бенчмарк, где Java «в десять раз медленнее C» на первой секунде и внезапно быстрее на десятой. Оба явления — работа JIT: JVM исполняет твой код _по-разному в разные моменты жизни процесса_, и понимание этого конвейера объясняет и warmup, и почему секундомерные бенчмарки врут, и за счёт чего управляемый язык на длинной дистанции соревнуется с компилируемыми.

> [!abstract] Ментальная модель  
> JVM начинает с интерпретации байткода, считает частоту вызовов, и горячие методы прогоняет через два компилятора: быстрый C1 (собирает профиль) и агрессивный C2 (оптимизирует на основе профиля — в том числе спекулятивно, с правом отката через деоптимизацию). Профиль реального исполнения — то преимущество, которого нет у AOT-компилятора.

### Зачем интерпретатор и компилятор одновременно

Интерпретация — медленно, но мгновенно: код исполняется с первого байта. Компиляция — быстро в исполнении, но дорого в подготовке: компилировать _всё_ на старте — значит долго ждать и оптимизировать вслепую, без знания, как код ведёт себя на реальных данных. HotSpot (само имя — про это) выбирает третий путь: исполнять всё интерпретатором, замерять, и тратить усилия компилятора только на «горячие точки» — в типичном приложении подавляющая часть времени проходит в единицах процентов кода. Остальные тысячи методов, вызванных по разу на старте, компилировать просто невыгодно.

### Tiered compilation: конвейер из пяти уровней

![[Pasted image 20260716164912.png]]

Уровней формально пять. **Уровень 0** — интерпретатор: исполняет и ведёт два счётчика на метод — вызовов (invocation counter) и обратных переходов в циклах (backedge counter, чтобы горячим считался и метод, вызванный однажды, но крутящий миллиард итераций). Перешагнул порог — метод уходит в очередь компиляции. **Уровни 1–3** — компилятор **C1**: быстрый, без глубоких оптимизаций; главный режим — уровень 3, где в машинный код вшиты датчики профиля: какие ветки берутся, какие типы реально приходят в виртуальные вызовы, бывают ли `null`. **Уровень 4** — **C2**: медленный, прожорливый, но производящий код, ради которого всё затевалось — и топливом ему служит профиль, собранный тройкой. (Уровни 1–2 — служебные: C1 без профилирования для тривиальных методов, которым C2 не поможет.) Результаты складываются в code cache из урока 11.2.

Весь конвейер виден вживую. Метод `square` и цикл в `main`, `java -XX:+PrintCompilation` (JDK 21):

```
35    7 %     3       Hot::main @ 4 (32 bytes)
36    6       2       Hot::square (4 bytes)
37    9 %     4       Hot::main @ 4 (32 bytes)
38    7 %     3       Hot::main @ 4 (32 bytes)   made not entrant
39   10       4       Hot::square (4 bytes)
40    6       2       Hot::square (4 bytes)   made not entrant
```

Колонки: время (мс), id задачи, флаги, **уровень**, метод. Тут вся биография: `main` скомпилирован на уровне 3, потом на 4, и версия-3 помечена `made not entrant` — «новые вызовы сюда не входят», код устарел. Знак `%` — это **OSR** (on-stack replacement): цикл в `main` горячий, но сам `main` вызван один раз — ждать его следующего вызова бессмысленно, и JVM подменяет исполнение _прямо посреди работающего цикла_, на лету перепрыгивая из интерпретируемой версии в скомпилированную (`@ 4` — байткод-индекс точки входа).

### Инлайнинг — мать всех оптимизаций

Самая важная оптимизация JIT — банальная на вид: подставить тело вызываемого метода в вызывающий. Прямая выгода (нет стоимости вызова) — мелочь; настоящая причина в том, что инлайнинг **открывает дорогу всем остальным оптимизациям**: пока `square(i)` — вызов, C2 видит чёрный ящик; когда его тело встроено, умножение попадает в общий поток данных цикла — и его можно векторизовать, вынести, свернуть. Escape analysis из следующего раздела тоже работает только сквозь инлайн. Решения инлайнера видны через `-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining` — в нашем прогоне: `Hot::square (4 bytes) inline (hot)`.

Инлайнингу мешают две вещи. **Размер**: тела крупнее порогов (в байткодах!) не встраиваются — ещё один довод за маленькие методы. И **виртуальность**: чтобы встроить вызов интерфейсного метода, надо знать, чьё тело встраивать. Тут JIT снова опирается на профиль: если в конкретной точке вызова по факту приходил один тип (**мономорфный** call site) — C2 встраивает его реализацию, поставив дешёвый guard-чек типа; два типа (**биморфный**) — встраивает оба с ветвлением; три и больше (**мегаморфный**) — сдаётся, честный виртуальный вызов. Да, «все методы в Java виртуальны по умолчанию» (глава 4.3) — но на горячем пути JIT девиртуализует подавляющее большинство из них.

### Деоптимизация: право на ошибку

А что если профиль соврал? Вот верифицированный сценарий — метод, всю жизнь видевший только котов:

```java
static int len(Speaker s) { return s.sound().length(); }

Speaker cat = new Cat();
for (int i = 0; i < 300_000; i++) sum += len(cat);  // мир, где есть только коты

Speaker dog = new Dog();                            // внезапно собака
for (int i = 0; i < 300_000; i++) sum += len(dog);
```

`-XX:+PrintCompilation`:

```
36   10       4       Deopt::len (10 bytes)
42   10       4       Deopt::len (10 bytes)   made not entrant
42   15       4       Deopt::len (10 bytes)
```

C2 скомпилировал `len` в оптимистичном предположении «`Speaker` — это всегда `Cat`» и встроил `Cat.sound()`. Появление `Dog` провалило guard — сработал **uncommon trap**: исполнение аварийно вернулось в интерпретатор (состояние регистров аккуратно разложено обратно в интерпретаторский фрейм), оптимистичная версия помечена `made not entrant`, профиль дособран, метод перекомпилирован (id 15) уже с честной биморфной диспетчеризацией. Программа при этом не заметила ничего — только пару микросекунд.

Это и есть ответ на вопрос «как JIT может обгонять AOT»: C2 позволяет себе спекуляции, за которые статический компилятор не возьмётся — девиртуализация по фактическому профилю, выбрасывание веток, которые ни разу не исполнялись, оптимизация под реальные данные, — потому что у него всегда есть страховка отката. AOT обязан быть прав всегда; JIT — только пока прав.

### Escape analysis: «объекты всегда в heap» — миф

Перед аллокацией C2 задаёт вопрос: **убегает** ли объект — может ли ссылка на него пережить компилируемый фрагмент (уйти в поле, в параметр невстроенного вызова, в другой поток)? Если нет, объект можно не создавать вовсе: **scalar replacement** разбирает его на поля-скаляры, и `p.x` живёт в регистре CPU, а не в Eden. Проверим — 200 миллионов «объектов» `Point`, не покидающих итерацию (JDK 21, Serial GC на 1 CPU):

```java
for (int i = 0; i < 200_000_000; i++) {
    Point p = new Point(i, i);          // объект не покидает итерацию
    sum += p.x() + p.y();
}
```

```
$ java -Xlog:gc Boxes                          # EA включён (дефолт)
sum=39999999800000000, 91 ms                   # GC-пауз: ноль

$ java -XX:-DoEscapeAnalysis -Xlog:gc Boxes    # EA выключен
sum=39999999800000000, 570 ms                  # GC-пауз: 357
```

С выключенным EA — 357 сборок мусора и шестикратное замедление; с включённым — **ни одной аллокации**: двести миллионов объектов существовали только как пара регистров. Бонусом EA даёт **lock elision**: `synchronized` на неубегающем объекте — синхронизация, которую никто не может увидеть, — выбрасывается целиком. Важная точность формулировок: это не «аллокация на стеке» как фича языка — это оптимизация конкретной скомпилированной версии, работающая только когда C2 _доказал_ неубегание (а доказать он может лишь то, что видит, — то есть сквозь инлайн).

> [!tip] 💡 Из C#  
> Современный .NET сошёлся с HotSpot почти поточечно: tiered compilation (с .NET Core 3.0: Tier 0 quick JIT → Tier 1), OSR (.NET 7), Dynamic PGO с девиртуализацией по профилю и деоптимизационными guard'ами (дефолт с .NET 8). Так что твоя интуиция переносится хорошо; различия в акцентах: у HotSpot профилировка глубже и история спекуляций длиннее, у .NET сильнее развит AOT-фланг (ReadyToRun, NativeAOT) — джавовый аналог которого GraalVM Native Image, а стандартное решение проблемы старта в JVM-мире — CDS и грядущий Project Leyden. И заметная разница: escape analysis в RyuJIT до недавнего времени был точечным (stack allocation объектов появляется постепенно), тогда как в HotSpot скалярная замена — рабочая лошадь уже лет пятнадцать.

> [!example] 🔬 Глубже / на синьора  
> **Анатомия строки `PrintCompilation`.** `38 7 % 3 Hot::main @ 4 (32 bytes) made not entrant`: миллисекунды от старта VM; id компиляционной задачи; флаги (`%` — OSR, `s` — synchronized, `!` — есть обработчики исключений, `n` — native wrapper); уровень 0–4; метод; для OSR — `@ bci` точки входа; размер в _байткодах_ (именно он сравнивается с порогами инлайнинга); статус. `made not entrant` — в устаревший код не входят новые вызовы (текущие дорабатывают); следующая стадия — `made zombie`, когда в него не осталось живых фреймов и его можно выкинуть из code cache.
> 
> **Интринзики: методы, которые не исполняют свой байткод.** Для избранных методов (`Math.sqrt`, `System.arraycopy`, `String.indexOf`, CRC32, весь `VarHandle`-слой) JIT игнорирует Java-реализацию и подставляет заранее написанный вручную машинный код под конкретный CPU — с SIMD-инструкциями, которые из байткода не выразить. В JDK они помечены `@IntrinsicCandidate`. Отсюда практическое правило: «переписать `System.arraycopy` циклом, чтобы было быстрее» — гарантированная пессимизация.
> 
> **Пороги и очередь.** Дефолтный ориентир горячести для C2 — порядка 10 000 срабатываний (`-XX:CompileThreshold`, в tiered-режиме реальная формула сложнее и учитывает оба счётчика). Компиляция асинхронна: метод стоит в очереди, а пока работает его предыдущая версия — поэтому на машине с 1 CPU (как тестовая среда этого урока) JIT-потоки конкурируют с приложением, и warmup заметно дольше.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Секундомерный бенчмарк измеряет что угодно, кроме кода.** Первые итерации — интерпретатор, середина — C1, конец — C2; а если результат вычисления никуда не уходит, EA + dead code elimination могут выбросить измеряемый код _целиком_ (в эксперименте выше убери чтение `sum` — и цикл исчезнет). Поэтому JMH: прогрев, изоляция, `Blackhole` против DCE. «Мой бенчмарк показал 0 нс» — это не скорость, это элиминация.
> 2. **Warmup — продовая проблема, не теоретическая.** Пускать полный трафик на свежий под — значит отдать первые сотни запросов интерпретатору. Смягчения: прогревающие запросы до readiness probe, плавный ввод трафика; системно — AppCDS (Java 10+, JEP 310) и AOT-кеш Project Leyden (JEP 483 в JDK 24) для переноса прогрева между запусками.
> 3. **Переполненный code cache молча выключает JIT.** Одно предупреждение `CodeCache is full. Compiler has been disabled` — и приложение до рестарта живёт на интерпретаторе с деградацией в разы. Типичный виновник — обильная генерация классов/лямбд; лечится `-XX:ReservedCodeCacheSize`.
> 4. **Мегаморфный вызов в горячем цикле — потолок для инлайнера.** Больше двух реальных типов в точке вызова — и девиртуализация невозможна: честный dispatch на каждой итерации плюс закрытое окно для дальнейших оптимизаций. Это не повод бояться полиморфизма — это повод не гонять цикл-миллионник через точку, куда приходят пять реализаций по очереди.
> 5. **Загрязнение профиля.** Профиль привязан к _точке вызова_: если общий утилитный метод на старте прогнали с экзотическими типами, спекуляции для него навсегда осторожнее. Из этого же семейства — «прогрев» бенчмарка данными, не похожими на боевые: измеришь код, скомпилированный под чужой профиль.

### На собесе

> [!question]- Как JVM исполняет код? Зачем интерпретатор, если есть компилятор?  
> **Частота:** очень высокая — вход в тему.
> 
> **RU:** Смешанно. Старт — интерпретация байткода: мгновенный запуск плюс сбор статистики. Методы, перешагнувшие порог горячести (по счётчикам вызовов и итераций циклов), компилируются в машинный код: сначала быстрым C1 с датчиками профиля, затем — если метод действительно горяч — агрессивным C2 на основе собранного профиля. Компилировать всё подряд невыгодно: большинство методов исполняется слишком редко, чтобы окупить компиляцию, а без профиля оптимизации были бы слепыми.
> 
> **EN:** In mixed mode. It starts by interpreting bytecode — instant startup plus statistics gathering. Methods that cross a hotness threshold (invocation and loop-backedge counters) get compiled: first by the fast C1 with profiling hooks, then — if genuinely hot — by the aggressive C2 using the collected profile. Compiling everything wouldn't pay: most methods run too rarely to justify it, and without a profile the optimizations would be blind.
> 
> **Follow-up:** «Что такое HotSpot?» → Название реализации JVM — буквально про эту стратегию: усилия тратятся на горячие точки.

> [!question]- Расскажите про tiered compilation.  
> **Частота:** высокая на mid+.
> 
> **RU:** Пять уровней: 0 — интерпретатор со счётчиками; 1–3 — C1 (основной — третий: быстрый код с встроенным профилированием веток и типов); 4 — C2 с полными оптимизациями по профилю. Метод мигрирует по уровням по мере горячести, старые версии помечаются made not entrant. Для горячих циклов есть OSR — подмена кода прямо посреди работающего цикла, в логе компиляции помечается знаком `%`.
> 
> **EN:** Five tiers: 0 — the interpreter with counters; 1–3 — C1 (mainly tier 3: fast code with embedded branch/type profiling); 4 — C2 with full profile-guided optimizations. A method migrates up as it gets hotter; stale versions are marked made not entrant. Hot loops get OSR — on-stack replacement, swapping code mid-loop, marked `%` in the compilation log.
> 
> **Follow-up:** «Зачем два компилятора, а не сразу C2?» → C2 дорог и требует профиля; C1 даёт быстрое ускорение и собирает этот профиль. Убрать C1 — получить дольше warmup и хуже финальный код.

> [!question]- Почему JIT может генерировать код быстрее, чем AOT-компилятор?  
> **Частота:** средняя-высокая; любимый «философский» вопрос.
> 
> **RU:** У JIT есть профиль реального исполнения и право на откат. Он видит фактические типы в виртуальных вызовах (девиртуализация и инлайнинг), фактические ветки, реальный CPU под ногами — и оптимизирует спекулятивно: предполагает мономорфность, выбрасывает неисполнявшиеся ветки. Ошибся — деоптимизация вернёт метод в интерпретатор и перекомпилирует с новым знанием. AOT обязан быть корректен для всех возможных сценариев сразу, поэтому вынужден быть консервативнее. Цена JIT — warmup и расход CPU/памяти на саму компиляцию.
> 
> **EN:** The JIT has a real execution profile and the right to bail out. It sees the actual types at virtual call sites (devirtualization and inlining), the actual branches taken, the actual CPU — and optimizes speculatively: assumes monomorphism, drops never-taken branches. If a guess fails, deoptimization drops back to the interpreter and recompiles with updated knowledge. AOT must be correct for every possible scenario upfront, so it has to be more conservative. The JIT's price is warmup and the CPU/memory cost of compiling at runtime.
> 
> **Follow-up:** «А GraalVM Native Image тогда зачем?» → Другая точка на кривой: мгновенный старт и маленький footprint ценой пиковой производительности и динамики — идеально для serverless/CLI.

> [!question]- Что такое деоптимизация и когда она происходит?  
> **Частота:** средняя; хороший маркер глубины.
> 
> **RU:** Откат из оптимизированного кода в интерпретатор, когда нарушилось предположение, под которое код скомпилирован: в мономорфную точку вызова пришёл новый тип, загрузился класс, сломавший иерархический анализ, исполнилась ветка, считавшаяся мёртвой. Срабатывает uncommon trap: состояние аккуратно переносится в интерпретаторский фрейм, версия помечается made not entrant, метод перекомпилируется с уточнённым профилем. Для программы это прозрачно — только микроскопическая задержка.
> 
> **EN:** A bailout from optimized code back to the interpreter when an assumption baked into the compilation breaks: a new type reaches a monomorphic call site, a freshly loaded class invalidates hierarchy analysis, a branch presumed dead executes. An uncommon trap fires: state is carefully mapped back into an interpreter frame, the version is marked made not entrant, and the method recompiles with the refined profile. It's transparent to the program — just a microscopic hiccup.
> 
> **Ловушка:** «Деоптимизация — это баг/ошибка?» → Нет, это спроектированный механизм, _делающий возможными_ спекулятивные оптимизации.

> [!question]- Почему инлайнинг называют самой важной оптимизацией? Что ему мешает?  
> **Частота:** средняя.
> 
> **RU:** Не из-за экономии на вызове — из-за того, что он открывает остальные оптимизации: встроенное тело попадает в общий анализ потока данных, и только тогда работают устранение проверок, векторизация, escape analysis. Мешают размер тела (пороги в байткодах — довод за маленькие методы) и полиморфизм точки вызова: один-два фактических типа JIT встраивает с guard'ами, мегаморфную точку — не может.
> 
> **EN:** Not because of call overhead — because it unlocks everything else: an inlined body joins the caller's dataflow analysis, and only then do check elimination, vectorization, and escape analysis apply. It's blocked by body size (thresholds counted in bytecodes — an argument for small methods) and call-site polymorphism: one or two observed types get inlined behind guards; a megamorphic site can't be.
> 
> **Follow-up:** «Все методы в Java виртуальные — это же дорого?» → На горячем пути JIT девиртуализует по профилю подавляющее большинство вызовов, так что дефолтная виртуальность почти бесплатна.

> [!question]- Всегда ли объекты создаются в heap?  
> **Частота:** средняя; вопрос-ловушка.
> 
> **RU:** Как правило языка — да; как факт исполнения — нет. Escape analysis в C2 проверяет, убегает ли объект из компилируемого фрагмента; неубегающий объект подвергается scalar replacement — разбирается на поля, живущие в регистрах, аллокации не происходит вовсе. Проверяется экспериментом: цикл с временным объектом при включённом EA не производит ни одной GC-паузы, при `-XX:-DoEscapeAnalysis` — сотни. Бонус — lock elision: `synchronized` на неубегающем объекте выбрасывается. Оговорка: это оптимизация, а не гарантия — работает только там, где C2 доказал неубегание, обычно сквозь инлайнинг.
> 
> **EN:** As a language rule — yes; as an execution fact — no. C2's escape analysis checks whether an object escapes the compiled fragment; a non-escaping object gets scalar-replaced — decomposed into fields living in registers, with no allocation at all. It's demonstrable: a loop with a temporary object produces zero GC pauses with EA on and hundreds with `-XX:-DoEscapeAnalysis`. A bonus is lock elision: `synchronized` on a non-escaping object is removed. Caveat: it's an optimization, not a guarantee — it applies only where C2 proves non-escape, usually through inlining.

> [!question]- Почему нельзя мерить производительность простым секундомером? Зачем JMH?  
> **Частота:** высокая в командах, где перформанс — тема.
> 
> **RU:** Потому что измеряется движущаяся мишень: начало прогона — интерпретатор, конец — C2, и среднее бессмысленно. Хуже: если результат не используется, EA и dead code elimination могут удалить измеряемый код целиком — «0 наносекунд». Плюс OSR-версии циклов отличаются от обычных, а профиль зависит от разогревочных данных. JMH решает это институционально: warmup-итерации, форки процессов, `Blackhole` против элиминации, статистика вместо одного замера.
> 
> **EN:** Because the target moves: the run starts in the interpreter and ends in C2 — the average is meaningless. Worse, if the result isn't consumed, EA plus dead code elimination can remove the measured code entirely — "0 nanoseconds". Add OSR-compiled loop versions differing from regular ones, and profiles shaped by warmup data. JMH addresses this institutionally: warmup iterations, process forks, `Blackhole` against elimination, statistics instead of a single reading.
> 
> **Senior add-on:** проверять, _что_ именно скомпилировалось, можно `-XX:+PrintCompilation`/`-XX:+PrintInlining` — иногда бенчмарк сравнивает не два алгоритма, а инлайн против не-инлайна.

> [!question]- Что такое warmup и как с ним борются в проде?  
> **Частота:** средняя, растёт с сеньорностью позиции.
> 
> **RU:** Период после старта, пока горячий код не прошёл через C2: первые запросы обслуживает интерпретатор — латентность в разы хуже стационарной. Тактически: прогревающие запросы до readiness probe, постепенный ввод трафика в балансировку. Системно: CDS/AppCDS (быстрее загрузка классов), GraalVM Native Image (нет warmup вообще, ценой пиковой скорости), и магистральное направление — Project Leyden с AOT-кешем профилей и кода между запусками (JEP 483, JDK 24).
> 
> **EN:** The post-startup period before hot code reaches C2: early requests run interpreted, with latency several times worse than steady state. Tactically: warmup requests before the readiness probe, gradual traffic ramp-up. Systemically: CDS/AppCDS (faster class loading), GraalVM Native Image (no warmup at all, trading peak speed), and the mainline direction — Project Leyden, caching AOT-compiled code and profiles across runs (JEP 483, JDK 24).
