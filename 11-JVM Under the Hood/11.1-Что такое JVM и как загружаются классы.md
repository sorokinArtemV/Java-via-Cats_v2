Ты собрал jar, задеплоил на сервер — и приложение упало на старте с `NoClassDefFoundError`, хотя локально всё компилировалось и работало. Или зависимость притащила «тот же» класс второй раз, и в логах появилось абсурдное `CatFood cannot be cast to CatFood`. Обе истории — не про твой код, а про механизм, который стоит между `.class`-файлом и работающим объектом: загрузку классов. На собеседованиях это одна из любимых тем «на понимание рантайма», потому что она отделяет тех, кто писал на Java, от тех, кто понимает, как Java исполняется.

> [!abstract] Ментальная модель  
> JVM — это спецификация виртуальной машины, исполняющей байткод; HotSpot и другие — её реализации. Классы попадают в JVM не все сразу, а лениво, по первому реальному использованию, через иерархию загрузчиков, где каждый сначала делегирует запрос родителю и лишь потом ищет сам.

### JVM — спецификация, а не программа

Когда говорят «JVM», обычно имеют в виду конкретный процесс `java` на машине. Но формально JVM — это документ, _JVM Specification_: он описывает, что виртуальная машина обязана уметь (формат `.class`, набор инструкций байткода, модель памяти), но не описывает, _как_ именно. Поэтому реализаций несколько:

- **HotSpot** — реализация в OpenJDK, стандарт де-факто. Всё, что ты запускаешь через Temurin, Corretto, Zulu или Oracle JDK — это сборки OpenJDK с HotSpot внутри, различия между вендорами в лицензии и поддержке, не в машине.
- **Eclipse OpenJ9** — альтернатива от IBM/Eclipse, фокус на низком потреблении памяти и быстром старте (популярна в контейнерах).
- **GraalVM** — HotSpot с JIT-компилятором, написанным на самой Java, плюс режим Native Image: AOT-компиляция в нативный бинарник без JVM вообще (ценой части динамики — reflection требует конфигурации).

Дальше в главе всё «под капотом» — про HotSpot, потому что в 99% вакансий подразумевается он.

> [!tip] 💡 Из C#  
> В .NET реализация рантайма фактически одна — CoreCLR от Microsoft (Mono остался в нишах вроде Unity). В Java-мире спецификация и реализация разделены намеренно, отсюда зоопарк: OpenJDK — открытый исходный код, вендоры собирают из него дистрибутивы. Вопрос «чем Oracle JDK отличается от OpenJDK» — это вопрос про лицензию, а не про технологии.

### Байткод: что на самом деле лежит в `.class`

`javac` не производит машинный код. Он производит байткод — инструкции для стековой виртуальной машины, один `.class`-файл на каждый класс (включая вложенные: `Outer$Inner.class`). Посмотреть на него можно штатным `javap`:

```java
public class Purr {
    public static int add(int a, int b) {
        return a + b;
    }
}
```

```
$ javap -c Purr
  public static int add(int, int);
    Code:
       0: iload_0      // положить a на стек операндов
       1: iload_1      // положить b
       2: iadd         // снять оба, сложить, положить результат
       3: ireturn      // вернуть вершину стека
```

Никаких регистров — только стек операндов. Этот байткод одинаков на Linux, macOS и Windows; платформенно-зависимая часть (интерпретация и JIT-компиляция в машинный код) — работа JVM, и её мы разберём в уроке 11.4. Пока достаточно зафиксировать: `.class` — это переносимый полуфабрикат, а не исполняемый код.

> [!tip] 💡 Из C#  
> Прямой аналог — IL (MSIL): `javac` ≈ `csc`, байткод ≈ IL, `javap` ≈ `ildasm`. Обе машины стековые. Заметная разница в упаковке: .NET компилирует в assembly (одна DLL — много типов + манифест), Java — в россыпь `.class`-файлов, которые jar просто складывает в zip без общего манифеста типов. Отсюда и разное поведение при «нехватке» типа в рантайме — увидим ниже.

### Кто приносит классы в JVM

JVM на старте не загружает весь classpath. Классы приносят по одному специальные объекты — загрузчики (`ClassLoader`), и их три штатных, выстроенных в иерархию:

```java
import java.net.http.HttpClient;

public class WhoLoadsCats {
    public static void main(String[] args) {
        System.out.println("String        -> " + String.class.getClassLoader());
        System.out.println("HttpClient    -> " + HttpClient.class.getClassLoader());
        System.out.println("WhoLoadsCats  -> " + WhoLoadsCats.class.getClassLoader());

        ClassLoader app = WhoLoadsCats.class.getClassLoader();
        System.out.println("parent        -> " + app.getParent());
        System.out.println("grandparent   -> " + app.getParent().getParent());
    }
}
```

Вывод (JDK 21):

```
String        -> null
HttpClient    -> jdk.internal.loader.ClassLoaders$PlatformClassLoader@45ee12a7
WhoLoadsCats  -> jdk.internal.loader.ClassLoaders$AppClassLoader@639fee48
parent        -> jdk.internal.loader.ClassLoaders$PlatformClassLoader@45ee12a7
grandparent   -> null
```

Три уровня:

- **Bootstrap** — загружает ядро платформы (`java.base`: `String`, `Object`, коллекции). Написан на C++ внутри самой JVM, поэтому Java-объекта для него нет — `getClassLoader()` возвращает `null`. Это не ошибка, это опознавательный знак.
- **Platform loader** — остальные модули JDK (`java.sql`, `java.net.http` и т.д.). До Java 9 на этом месте был extension loader с механизмом `ext`-директорий — его убрали вместе с введением модулей (JPMS, глава 8.8).
- **Application (system) loader** — всё с classpath: твой код и твои зависимости.

Иерархия — это не наследование классов, а связь parent через поле: у application-загрузчика родитель — platform, у platform — bootstrap (`null`).

### Parent delegation: сначала спроси родителя

Ключевое правило, ради которого иерархия и существует: получив запрос на класс, загрузчик **сначала делегирует его родителю** и только если вся цепочка выше класс не нашла — ищет сам.

![[Pasted image 20260716164536.png]]

Зачем такая церемония? Две причины, и обе стоят упоминания на собеседовании:

1. **Безопасность.** Никакой код с classpath не может подсунуть свой `java.lang.String`: запрос на это имя всегда доедет до bootstrap первым, и он вернёт настоящий `String` из `java.base`. Твой файл-самозванец даже не будет рассмотрен.
2. **Единственность.** Один и тот же класс в пределах цепочки загружается ровно один раз — все, кто ниже, получают ту же самую `Class`-инстанцию, а значит `static`-состояние и результаты `instanceof` консистентны.

> [!tip] 💡 Из C#  
> Ближайший аналог `ClassLoader` — `AssemblyLoadContext` (в старом .NET Framework — `AppDomain`). Но принципиальная разница в дефолте: в .NET сборки резолвятся в основном плоско в default-контексте, и изоляция через `AssemblyLoadContext` — осознанный шаг для плагинов. В Java иерархия с делегацией — это дефолт, на ней стоят Tomcat, OSGi и вся плагинная экосистема.

### Классы грузятся лениво

JVM не читает classpath наперёд. Класс проходит инициализацию только при первом _активном_ использовании — и это легко увидеть:

```java
class Whiskers {
    static { System.out.println(">> Whiskers: класс инициализирован"); }
    static void meow() { System.out.println("мяу"); }
}

public class LazyCats {
    public static void main(String[] args) {
        System.out.println("1. main() стартовал");
        System.out.println("2. объявляю переменную типа Whiskers");
        Whiskers cat = null;                  // тип упомянут — но класс не нужен
        System.out.println("3. а теперь реально трогаю класс");
        Whiskers.meow();                      // вот здесь происходит инициализация
    }
}
```

```
1. main() стартовал
2. объявляю переменную типа Whiskers
3. а теперь реально трогаю класс
>> Whiskers: класс инициализирован
мяу
```

Объявление переменной и даже `Whiskers cat = null` класс не трогают. Инициализацию (выполнение `static`-блоков и присваивание `static`-полей) запускают: `new`, вызов `static`-метода, чтение/запись `static`-поля (кроме compile-time констант — см. подводные камни), `Class.forName`, а также инициализация наследника (сначала инициализируется родитель). Из этой ленивости, кстати, растёт легальный идиоматический singleton через holder-класс — встретим его в главе 25.

### `ClassNotFoundException` vs `NoClassDefFoundError`

Два способа «класса нет», которые путают постоянно. Разница — в том, _кто_ и _когда_ просил класс:

```java
class Persian {
    static String breed() { return "Persian"; }
}

public class Kennel {
    public static void main(String[] args) throws Exception {
        try {
            Class.forName("com.example.SphynxCat");   // класса не было никогда
        } catch (ClassNotFoundException e) {
            System.out.println("CNFE: " + e.getMessage());
        }
        System.out.println(Persian.breed());           // класс был при компиляции
    }
}
```

Скомпилируем, убедимся, что работает, а потом **удалим `Persian.class`** и запустим снова — смоделировав ситуацию «в jar на проде не доехала зависимость»:

```
$ java Kennel                      # всё на месте
CNFE: com.example.SphynxCat
Persian

$ rm Persian.class && java Kennel  # Persian исчез после компиляции
CNFE: com.example.SphynxCat
Exception in thread "main" java.lang.NoClassDefFoundError: Persian
        at Kennel.main(Kennel.java:14)
Caused by: java.lang.ClassNotFoundException: Persian
```

- **`ClassNotFoundException`** — checked-исключение. Возникает, когда класс запрашивают динамически, строкой (`Class.forName`, `loadClass`), и его нет. Компилятор о классе ничего не знал — это штатная, ожидаемая ситуация, поэтому исключение и checked.
- **`NoClassDefFoundError`** — `Error`. Компилятор класс _видел_ (ссылка зашита в байткод), а рантайм его найти не смог. Это сломанное окружение: неполный classpath, конфликт версий зависимостей, «толстый» jar собран криво. Обрати внимание на `Caused by` — под капотом это тот же неудачный `loadClass`, обёрнутый в `Error`, потому что чинить тут нечего — деплой битый.

Практическое правило для дебага: `CNFE` — смотри на код, который строит имя класса; `NCDFE` — смотри на classpath и на дерево зависимостей (`mvn dependency:tree`).

> [!tip] 💡 Из C#  
> В .NET сборка — атомарная единица, поэтому аналог один: `FileNotFoundException`/`TypeLoadException` на уровне assembly. В Java гранулярность — отдельный класс, поэтому jar может быть _частично_ битым: половина классов грузится, вторая падает. Отсюда типично «джавовые» истории про два jar-а разных версий одной библиотеки в одном classpath.

> [!example] 🔬 Глубже / на синьора  
> **Загрузка — это три фазы, а не одна.** По JVMS (§5.3–5.5) класс проходит: **loading** (читаем байты, создаём `Class`-объект) → **linking**, которое само делится на _verification_ (проверка корректности байткода — то, что делает невозможным битый `.class`), _preparation_ (выделение памяти под `static`-поля с дефолтными значениями) и _resolution_ (превращение символических ссылок в прямые — HotSpot делает это лениво, при первом использовании ссылки) → **initialization** (выполнение `<clinit>` — склеенных `static`-блоков и инициализаторов полей; в `javap -c` его видно как метод `static {}`). Ленивая инициализация из примера выше — это именно про последнюю фазу: загрузить класс можно и раньше, чем инициализировать.
> 
> **Идентичность класса = (полное имя, загрузчик).** Одно имя — не значит один класс. Проверяется в пять строк:
> 
> java
> 
> ```java
> var loaderA = new URLClassLoader(urls, null);   // parent = null: минуя app loader
> var loaderB = new URLClassLoader(urls, null);
> 
> Class<?> a = loaderA.loadClass("Purr");
> Class<?> b = loaderB.loadClass("Purr");
> 
> System.out.println(a.getName().equals(b.getName()));  // true
> System.out.println(a == b);                           // false
> System.out.println(a.isAssignableFrom(b));            // false!
> ```
> 
> Два разных `Class`-объекта, взаимно несовместимых: каст между ними даст `ClassCastException` с издевательским сообщением «`Purr` cannot be cast to `Purr`». Это не баг, а фундамент изоляции: Tomcat даёт каждому веб-приложению свой загрузчик, и два приложения могут возить разные версии одной библиотеки, не мешая друг другу. Обрати внимание на второй аргумент `URLClassLoader` — parent: передав `null`, мы сознательно сломали делегацию до app loader (до bootstrap она всё равно дойдёт, `Object` возьмётся оттуда).
> 
> **`loadClass` vs `Class.forName`.** Оба загружают, но `loadClass` останавливается до инициализации, а `Class.forName(name)` по умолчанию ещё и инициализирует (`<clinit>` выполняется). Проверено:
> 
> java
> 
> ```java
> cl.loadClass("Sleepy");     // ">> Sleepy initialized" НЕ печатается
> Class.forName("Sleepy");    // а вот теперь печатается
> ```
> 
> Есть и перегрузка `Class.forName(name, initialize, loader)` с явным контролем. На этом различии строятся фреймворки, которым нужно посканировать классы, не запуская их побочные эффекты.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Упавший `static`-блок отравляет класс навсегда.** Первое касание бросит `ExceptionInInitializerError` с настоящей причиной внутри, но класс помечается как failed, и _второе_ касание даст уже голый `NoClassDefFoundError: Could not initialize class X` — без причины. Проверено: в логах длинно живущего приложения ты почти всегда видишь второй вариант, а настоящая ошибка была одна и давно уползла вверх по логу. Ищи самый первый `ExceptionInInitializerError`.
> 2. **`Class.forName("com.mysql.Driver")` — карго-культ.** Со времён JDBC 4.0 (Java 6) драйверы находятся через `ServiceLoader` автоматически. Если видишь это в туториале — туториалу лет пятнадцать.
> 3. **`getClassLoader()` может вернуть `null`** — для классов из `java.base` это норма (bootstrap). Код вида `clazz.getClassLoader().getResourceAsStream(...)` словит NPE ровно в тот момент, когда `clazz` окажется, скажем, `String.class`. Безопаснее `ClassLoader.getSystemClassLoader()` или загрузчик _своего_ класса.
> 4. **Compile-time константы инлайнятся и не трогают класс.** Чтение `static final int MAX = 9;` из другого класса не вызывает инициализацию владельца — значение зашито в байткод потребителя на этапе компиляции. Обратная сторона: обновил константу в библиотеке, не перекомпилировал потребителя — он продолжит жить со старым значением.
> 5. **Один интерфейс, две копии в разных загрузчиках** — классика плагинных систем: плагин реализует `Plugin`, но свой `Plugin`, загруженный его собственным загрузчиком. `instanceof` вернёт `false`, каст упадёт. Правило: общий контракт (API-интерфейсы) должен жить в _родительском_ загрузчике, чтобы через делегацию все видели одну копию.

### На собесе

> [!question]- Чем отличаются JVM, JRE и JDK? И что такое OpenJDK vs Oracle JDK?  
> **Частота:** почти гарантированный разогревочный вопрос на мидл-позицию.
> 
> **RU:** JVM — виртуальная машина, исполняющая байткод; формально это спецификация, самая распространённая реализация — HotSpot. JRE — JVM плюс стандартная библиотека, минимум для запуска (как отдельный дистрибутив с Java 11 больше не поставляется). JDK — JRE плюс инструменты разработки: `javac`, `javap`, `jcmd`. OpenJDK — открытый референсный исходный код; Oracle JDK, Temurin, Corretto — сборки из него, различаются лицензией и поддержкой, а не технологиями.
> 
> **EN:** The JVM executes bytecode — technically it's a specification, with HotSpot being the dominant implementation. The JRE is the JVM plus the standard library — just enough to run applications (no longer shipped separately since Java 11). The JDK adds development tools: `javac`, `javap`, `jcmd`. OpenJDK is the open-source reference; Oracle JDK, Temurin, and Corretto are builds of it, differing in licensing and support, not technology.
> 
> **Follow-up:** «А что случилось с JRE после Java 11?» → Модульная система позволяет собирать урезанный рантайм под приложение через `jlink`, отдельный универсальный JRE стал не нужен.
> 
> **Слабый ответ:** «JVM — это то, что запускает Java» без слов «спецификация/реализация» — формально верно, но не отличает мидла от джуна.

> [!question]- Расскажите про иерархию classloader'ов.  
> **Частота:** высокая; стандартный вход в тему рантайма.
> 
> **RU:** Три штатных: bootstrap (ядро платформы, `java.base`; реализован нативно, поэтому `getClassLoader()` для `String` возвращает `null`), platform (остальные модули JDK — `java.sql`, HTTP-клиент), application (classpath — код приложения и зависимости). Связаны через parent-ссылку, и запрос на загрузку сначала делегируется вверх.
> 
> **EN:** Three built-in loaders: bootstrap (core platform classes from `java.base`; implemented natively, so `getClassLoader()` returns `null` for `String`), platform (remaining JDK modules like `java.sql`), and application (the classpath — your code and dependencies). They're linked via parent references, and load requests delegate upward first.
> 
> **Follow-up:** «Почему для `String.class.getClassLoader()` возвращается `null`?» → Bootstrap встроен в JVM (C++), у него нет Java-объекта. **Ловушка:** ответ «потому что класс ещё не загружен» — неверен и мгновенно выдаёт незнание темы.
> 
> **Senior add-on:** До Java 9 вместо platform был extension loader с `ext`-директориями; убрали вместе с JPMS.

> [!question]- Что такое parent delegation model и зачем она нужна?  
> **Частота:** высокая, почти всегда идёт следом за предыдущим.
> 
> **RU:** Загрузчик, получив запрос, сначала передаёт его родителю и ищет сам только если вся цепочка выше не нашла класс. Это даёт безопасность — нельзя подменить `java.lang.String`, запрос всегда дойдёт до bootstrap первым — и единственность: класс в пределах цепочки загружен один раз, `static`-состояние консистентно.
> 
> **EN:** A loader first delegates every request to its parent and only searches itself if the whole chain above fails. This gives security — you can't spoof `java.lang.String`, since bootstrap always sees the request first — and uniqueness: a class is loaded once per chain, keeping static state consistent.
> 
> **Follow-up:** «Кто нарушает эту модель и зачем?» → Контейнеры вроде Tomcat: загрузчик веб-приложения сначала ищет в WEB-INF, чтобы приложение могло возить свои версии библиотек, и только потом делегирует. Знать это — уверенный плюс.

> [!question]- В чём разница между ClassNotFoundException и NoClassDefFoundError?  
> **Частота:** очень высокая; любимый вопрос «на отличение практика».
> 
> **RU:** `ClassNotFoundException` — checked, возникает при явной динамической загрузке по строковому имени (`Class.forName`, `loadClass`): компилятор про класс не знал, ситуация штатная. `NoClassDefFoundError` — `Error`: ссылка на класс была в байткоде, компиляция его видела, а рантайм не нашёл — битый classpath или конфликт зависимостей. Диагностика: CNFE — смотрю код, формирующий имя; NCDFE — смотрю classpath и `dependency:tree`.
> 
> **EN:** `ClassNotFoundException` is a checked exception thrown on explicit dynamic loading by name (`Class.forName`, `loadClass`) — the compiler never knew about the class. `NoClassDefFoundError` is an `Error`: the reference was compiled into bytecode, but the runtime couldn't find the class — broken classpath or dependency conflict. Debugging: CNFE — inspect the code building the name; NCDFE — inspect the classpath and dependency tree.
> 
> **Follow-up (ловушка):** «А может ли NCDFE возникнуть, когда класс физически на месте?» → Да: если его `static`-инициализатор однажды упал. Первое касание — `ExceptionInInitializerError`, все последующие — `NoClassDefFoundError: Could not initialize class` без причины внутри. Не знать это — нормально для мидла; знать — заявка на мид+.

> [!question]- Когда класс загружается и инициализируется?  
> **Частота:** средняя, но следом почти всегда идёт вопрос про singleton.
> 
> **RU:** Лениво. Инициализация (`static`-блоки, `static`-поля) происходит при первом активном использовании: `new`, вызов `static`-метода, обращение к `static`-полю (кроме compile-time констант — они инлайнятся), `Class.forName`, инициализация наследника. Объявление переменной типа или загрузка без инициализации (`loadClass`) `static`-блоки не запускают. JVM гарантирует, что инициализация выполнится ровно один раз и потокобезопасно.
> 
> **EN:** Lazily. Initialization (static blocks and fields) happens on first active use: `new`, a static method call, static field access (except compile-time constants, which are inlined), `Class.forName`, or initialization of a subclass. Merely declaring a variable of the type, or loading via `loadClass`, doesn't trigger it. The JVM guarantees initialization runs exactly once, thread-safely.
> 
> **Follow-up:** «Как это используется на практике?» → Holder-идиома для ленивого singleton: вложенный класс с полем инициализируется только при первом обращении, и потокобезопасность даёт сама JVM — без `synchronized` и volatile.

> [!question]- Можно ли подменить java.lang.String своей реализацией?  
> **Частота:** средняя; проверка понимания делегации «с другого конца».
> 
> **RU:** Через classpath — нет: parent delegation доведёт запрос до bootstrap, и он вернёт настоящий `String`. Дополнительно JVM явно запрещает пользовательским загрузчикам определять классы в пакетах `java.*` — даже кастомный загрузчик, сломавший делегацию, получит `SecurityException`. Легальный способ трогать содержимое `java.base` — только `--patch-module`, и это инструмент отладки, не продакшена.
> 
> **EN:** Not via the classpath: parent delegation carries the request to bootstrap, which returns the real `String`. Additionally, the JVM explicitly forbids user-defined loaders from defining classes in `java.*` packages — even a custom loader that breaks delegation gets a `SecurityException`. The only sanctioned way to alter `java.base` is `--patch-module`, a debugging tool, not production practice.
> 
> **Слабый ответ:** «Нет, потому что String — final» — путает наследование с подменой класса; вопрос вообще не про `final`.

> [!question]- Что определяет идентичность класса в JVM? Может ли один класс быть загружен дважды?  
> **Частота:** ниже среднего; маркер сеньорности, любят в компаниях с плагинными системами.
> 
> **RU:** Идентичность — пара (полное имя, загрузчик). Один и тот же `.class`, загруженный двумя независимыми загрузчиками, даёт два разных `Class`-объекта: `==` — false, каст между ними — `ClassCastException` вида «X cannot be cast to X». На этом строится изоляция: сервлет-контейнеры дают каждому приложению свой загрузчик, и разные версии одной библиотеки сосуществуют. Отсюда правило для плагинов: общие API-интерфейсы должны жить в родительском загрузчике.
> 
> **EN:** Class identity is the pair (fully qualified name, defining loader). The same `.class` file loaded by two independent loaders yields two distinct `Class` objects: `==` is false, and casting between them throws `ClassCastException` — the infamous "X cannot be cast to X". This underpins isolation: servlet containers give each app its own loader so different library versions coexist. Hence the plugin rule: shared API interfaces must live in the parent loader.
> 
> **Follow-up:** «Как такое чинится, когда встречается в проде?» → Найти, из каких двух мест грузится класс (`-verbose:class` или `clazz.getClassLoader()` + `getProtectionDomain().getCodeSource()`), и вынести общий контракт выше по иерархии либо убрать дубль из classpath.

> [!question]- Чем отличаются loadClass и Class.forName?  
> **Частота:** ниже среднего, но отличный «добивающий» после вопроса про ленивость.
> 
> **RU:** Оба загружают класс, но `loadClass` останавливается до инициализации — `static`-блоки не выполняются; `Class.forName(name)` по умолчанию загружает _и_ инициализирует. Есть перегрузка `forName(name, initialize, loader)` с явным флагом. Разница важна для фреймворков, сканирующих классы: инициализация чужого класса — это выполнение чужого кода с побочными эффектами.
> 
> **EN:** Both load a class, but `loadClass` stops before initialization — static blocks don't run; `Class.forName(name)` loads _and_ initializes by default. The `forName(name, initialize, loader)` overload gives explicit control. This matters for frameworks scanning classes: initializing someone else's class means executing arbitrary side effects.
