![[Pasted image 20260716164621.png]]Пройдёмся по областям — с флагами и дефолтами, снятыми с JDK 21 через `java -XX:+PrintFlagsFinal -version`.

**Heap** — единственная область, где живут объекты, общая для всех потоков, единственная под управлением GC. `-Xms` — стартовый размер, `-Xmx` — потолок. Без явных флагов JVM берёт эргономику от машины: по умолчанию `MaxRAMPercentage = 25` — четверть доступной памяти (на моей тестовой машине с ~4 GB получился ровно 1 GB). Про «доступной» будет отдельный подводный камень: в контейнере JVM с Java 10+ видит лимит cgroup, а не память хоста.

**Стеки потоков** — по одному на каждый поток, вне heap. Хранят фреймы вызовов: локальные переменные (примитивы — значением, объекты — ссылкой; сами объекты всё равно в heap), операндный стек, адрес возврата. Размер — `-Xss`, дефолт на Linux x64 — `ThreadStackSize = 1024` KB, то есть 1 MB. Влияние размера на глубину рекурсии проверяется за минуту:

```java
public class DeepPurr {
    static int depth = 0;
    static void dig() { depth++; dig(); }
    public static void main(String[] args) {
        try { dig(); } catch (StackOverflowError e) {
            System.out.println("StackOverflowError на глубине: " + depth);
        }
    }
}
```

```
$ java DeepPurr            # дефолтный 1m
StackOverflowError на глубине: 23306
$ java -Xss256k DeepPurr
StackOverflowError на глубине: 1795
$ java -Xss8m DeepPurr
StackOverflowError на глубине: 206806
```

Точные числа плавают между запусками (JIT меняет размер фреймов), но порядок и пропорция стабильны. Заодно видно, почему тысячи platform-потоков — дорого: тысяча потоков — гигабайт только на стеки. Это одна из мотиваций virtual threads (глава 24.13).

**Metaspace** — метаданные загруженных классов: структура, методы, байткод, constant pool. Живёт в native-памяти, не в heap. Дефолтный потолок — его нет: `MaxMetaspaceSize = 2^64−1`, то есть «сколько даст ОС». Ограничивается `-XX:MaxMetaspaceSize`.

**Code cache** — машинный код, который произвёл JIT (глава 11.4). Обычно про него вспоминают только когда он переполнен и JIT молча перестаёт компилировать.

**Direct memory** — буферы `ByteBuffer.allocateDirect()`, которыми NIO и сетевые библиотеки (Netty!) обмениваются с ОС без копирования в heap. Лимит — `-XX:MaxDirectMemorySize`, по умолчанию равен `-Xmx`.

> [!tip] 💡 Из C#  
> Общая схема совпадает: managed heap + стек на поток + native-память процесса. Аналог Metaspace размазан по внутренним loader heaps CLR — отдельного именованного региона с флагом нет, и это одна из причин, почему в Java-мире про Metaspace спрашивают, а в .NET-мире про метаданные — нет. Direct memory ≈ unmanaged-память через `Marshal.AllocHGlobal` / pinned buffers. А вот дефолт «взять 25% RAM под heap» — джавизм; CLR-heap растёт по запросу без верхней планки в стиле `-Xmx`.

### PermGen → Metaspace: что случилось в Java 8

До Java 8 метаданные классов лежали в **PermGen** — области _внутри heap_ с жёстким лимитом (дефолт 64–85 MB). Каждый, кто застал те времена, помнит `OutOfMemoryError: PermGen space` при горячем редеплое на Tomcat: старые классы не выгружались, лимит кончался. В Java 8 (JEP 122) PermGen удалили, метаданные переехали в native-память и стали Metaspace: растёт по потребности, освобождается при выгрузке классов. Проблему это не убило, а переименовало — утечка загрузчиков классов теперь ест native-память до бесконечности вместо того, чтобы упереться в лимит. Вопрос «что случилось с PermGen» — вечнозелёный на собеседованиях, потому что мгновенно показывает, с какой версии Java человек в теме.

### Устройство heap: поколения

GC-алгоритмы (глава 11.3) стоят на эмпирике, которую называют **generational hypothesis**: подавляющее большинство объектов умирает молодыми. Обёртки, итераторы, строки-времянки — родились в методе, метод вернулся, объекты мусор. Поэтому heap делят на поколения и молодёжь обрабатывают отдельно и часто, а старожилов — редко.

![[Pasted image 20260716164709.png]]

Проследим биографию одного объекта. `new Cat("Whiskers")` рождается в **Eden** — области, куда попадают почти все новые объекты. Eden заполнился — случается **minor GC**: всё живое из Eden и из активного survivor-пространства _копируется_ в пустое survivor-пространство, а Eden объявляется чистым целиком (мёртвых никто не трогает — в этом фокус copying-подхода, детали в 11.3). Наш кот выжил — он теперь в survivor с возрастом 1. Каждый пережитый minor GC — переезд между `S0` и `S1` и `age++`. Дожил до порога — **promotion**: кот признан старожилом и переезжает в **Old generation**, где его будут беспокоить только редкие и дорогие major/full GC. Утилитарные же объекты — коробки, из которых кот уже вылез — не переживают и первого minor GC и исчезают бесплатно.

Числа с JDK 21 по умолчанию:

- `NewRatio = 2` — Old вдвое больше Young (Young ≈ треть heap);
- `SurvivorRatio = 8` — Eden в 8 раз больше каждого survivor-пространства;
- `MaxTenuringThreshold = 15` — максимальный возраст до promotion. Именно **максимальный**: реальный порог JVM подбирает динамически по заполненности survivor'ов, а 15 — жёсткий потолок (почему именно 15 — в 🔬-блоке).

Два survivor'а вместо одного — не расточительность: копирование «всех живых в заведомо пустое место» позволяет укладывать объекты плотно, без дыр, и полностью снимает проблему фрагментации в Young. Один из survivor'ов всегда пустой и ждёт следующего GC.

Одна честная оговорка: непрерывные области, как на схеме, — это картина Serial/Parallel GC. Дефолтный в JDK 21 G1 нарезает heap на регионы (~2048 штук), и Eden/Survivor/Old там — не сплошные куски, а _множества регионов_, состав которых меняется. Но роли и путь объекта — ровно те же, поэтому схема остаётся правильной ментальной моделью; регионы разберём в 11.3.

> [!tip] 💡 Из C#  
> Прямая параллель: gen 0 ≈ Eden + survivor, gen 1 ≈ переходное звено, gen 2 ≈ Old. Гипотеза поколений и copying-подход те же. Заметные отличия: в .NET есть выделенный LOH (Large Object Heap) — объекты ≥ 85 KB рождаются сразу там и не копируются; в Java близкий механизм есть только у G1 (humongous-регионы для объектов больше половины региона), у Serial/Parallel большие объекты просто аллоцируются сразу в Old. И в .NET нет публичного «возраста» объекта с порогом — promotion привязан к номеру пережитой сборки поколения.

### Зоопарк OutOfMemoryError

`OutOfMemoryError` — не одна ошибка, а семейство: текст после двоеточия говорит, _какая именно_ область кончилась, и от него зависит, куда смотреть. Два самых частых воспроизводятся тривиально (оба проверены):

```java
List<byte[]> baskets = new ArrayList<>();
while (true) baskets.add(new byte[1_000_000]);
```

```
$ java -Xmx64m Hoarder
Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
```

```java
List<ByteBuffer> buffers = new ArrayList<>();
while (true) buffers.add(ByteBuffer.allocateDirect(1_000_000));
```

```
$ java -Xmx64m -XX:MaxDirectMemorySize=16m DirectHoarder
Exception in thread "main" java.lang.OutOfMemoryError:
    Cannot reserve 1000000 bytes of direct buffer memory (allocated: 16000000, limit: 16777216)
```

Шпаргалка по сообщениям:

|Сообщение|Что кончилось|Куда смотреть|
|---|---|---|
|`Java heap space`|Heap|`-Xmx`, утечка (глава 11.5), heap dump|
|`GC overhead limit exceeded`|Формально heap ещё есть, но >98% времени уходит на GC, освобождающий <2%|Та же утечка, поймана раньше|
|`Metaspace`|Metaspace (при заданном `MaxMetaspaceSize`)|Утечка загрузчиков классов, динамическая генерация классов|
|`Cannot reserve ... direct buffer memory`|Direct memory|`MaxDirectMemorySize`, кто держит буферы (Netty и др.)|
|`unable to create native thread`|Память/лимиты ОС под стек нового потока|Число потоков (`ulimit`, thread dump), `-Xss`|

Отдельно запомни: `GC overhead limit exceeded` — это _милосердие_, а не жестокость. JVM признаёт агонию (процесс жив, но всё время молотит GC впустую) и падает с диагнозом, пока ещё можно снять heap dump.

> [!example] 🔬 Глубже / на синьора  
> **TLAB — почему аллокация в Java почти бесплатна.** Наивная модель «heap общий → на `new` нужна синхронизация потоков» неверна. Каждому потоку выдаётся личный кусочек Eden — **TLAB** (Thread-Local Allocation Buffer, флаг `UseTLAB = true` по умолчанию), внутри которого аллокация — это bump-the-pointer: сдвинуть указатель на размер объекта, и всё, порядка десятка машинных инструкций без единого лока. Кончился TLAB — поток берёт следующий (вот тут короткая синхронизация). Итог: аллокация в Java обычно _дешевле_, чем `malloc` в C. Это же объясняет, почему «переиспользовать объекты ради экономии» в современной Java почти всегда пессимизация.
> 
> **CompressedOops и магия числа 32 GB.** На 64-битной JVM ссылка должна бы занимать 8 байт, но по умолчанию включён `UseCompressedOops`: ссылки хранятся 32-битными. Трюк — выравнивание: объекты лежат по адресам, кратным 8, значит младшие 3 бита адреса всегда нули, и 32-битное число со сдвигом на 3 адресует 2³² × 8 = 32 GB. Проверено на JDK 21:
> 
> ```
> $ java -Xmx31g ... | grep UseCompressedOops   →  true   {ergonomic}
> $ java -Xmx32g ... | grep UseCompressedOops   →  false
> ```
> 
> Отсюда контринтуитивный факт для собеседования: **heap в 33 GB может вмещать меньше объектов, чем heap в 30 GB** — все ссылки внезапно удвоились. Практическое правило: либо держись ниже ~30–31 GB, либо прыгай сразу сильно выше (~45+ GB), чтобы окупить распухание.
> 
> **Возраст объекта живёт в заголовке.** У каждого объекта есть header: mark word (64 бита: identity hashCode, биты локов — и 4 бита под возраст, отсюда потолок `MaxTenuringThreshold = 15`) плюс class word — сжатый указатель на класс (те же compressed pointers, `UseCompressedClassPointers`). Итого 12 байт заголовка при включённом сжатии, поэтому «пустой» объект с выравниванием занимает 16 байт. Проект Lilliput ужимает заголовок до 8 байт — Compact Object Headers, JEP 450 (экспериментально в JDK 24); на JDK 21 его ещё нет.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`-Xmx` = лимит контейнера — гарантированный `OOMKilled`.** Процессу нужны Metaspace, стеки, code cache, direct memory и внутренние структуры GC _поверх_ heap. Правило большого пальца: heap ≤ 70–75% лимита контейнера, либо `-XX:MaxRAMPercentage=70` вместо абсолютного `-Xmx`. И помни: дефолт — всего 25% от лимита, то есть без флагов в контейнере на 4 GB ты получишь heap в 1 GB.
> 2. **`OOMKilled` ищут не в тех логах.** Убийца — ядро ОС (cgroup), в логах JVM тишина: ни `OutOfMemoryError`, ни heap dump, процесс просто исчез. Смотри `dmesg` / события пода, а не application log.
> 3. **Metaspace без лимита умирает молча.** `MaxMetaspaceSize` по умолчанию не ограничен, и утечка загрузчиков (типично: горячий редеплой, динамическая генерация классов прокси) не даст `OutOfMemoryError: Metaspace` — она будет тихо раздувать процесс, пока его не убьёт ОС. В контейнерах лимит стоит выставлять явно, чтобы получить диагностируемое падение вместо загадочного.
> 4. **Direct-буферы освобождает GC — с задержкой.** Нативная память direct-буфера возвращается ОС, только когда GC соберёт его heap-обёртку. Если heap просторный и GC редкий, буферы копятся: «direct memory кончилась, а heap полупустой». Отсюда мигающие баги, которые «лечатся» вызовом full GC.
> 5. **`StackOverflowError` — это не всегда бесконечная рекурсия.** Глубокий, но конечный обход (несбалансированное дерево, длинная цепочка `CompletableFuture`, рекурсивный парсер на вложенном JSON) честно упирается в ~20–30 тысяч фреймов дефолтного 1 MB стека. Иногда правильный фикс — `-Xss4m`, а не переписывание на итерацию; но для входных данных неограниченной глубины — только итерация.
> 6. **«Примитивы на стеке, объекты в heap» — верно лишь как первое приближение.** Локальный примитив — да, в фрейме; но примитивное _поле объекта_ — в heap вместе с объектом. А JIT через escape analysis умеет вообще не аллоцировать невыбравшийся наружу объект (глава 11.4). На собеседовании формулируй аккуратно.

### На собесе

> [!question]- Из каких областей состоит память JVM?  
> **Частота:** очень высокая — стандартное открытие темы JVM.
> 
> **RU:** Heap — все объекты, общий для потоков, под управлением GC (`-Xms`/`-Xmx`). Стек на каждый поток — фреймы вызовов с локальными переменными (`-Xss`, дефолт 1 MB). Metaspace — метаданные классов в native-памяти, с Java 8 вместо PermGen. Code cache — машинный код от JIT. Direct memory — NIO-буферы вне heap. Плюс служебная native-память самой JVM, поэтому footprint процесса всегда заметно больше `-Xmx`.
> 
> **EN:** The heap holds all objects, shared across threads, managed by GC (`-Xms`/`-Xmx`). Each thread has its own stack with call frames and locals (`-Xss`, 1 MB default). Metaspace stores class metadata in native memory — it replaced PermGen in Java 8. The code cache holds JIT-compiled machine code, and direct memory backs NIO buffers outside the heap. Plus the JVM's own native overhead — so the process footprint always exceeds `-Xmx` noticeably.
> 
> **Follow-up:** «Почему процесс ест больше, чем `-Xmx`?» → Всё перечисленное живёт _поверх_ heap; `-Xmx` ограничивает только одну область. Слабый ответ — «утечка памяти»: чаще это просто арифметика областей.

> [!question]- Что хранится в стеке, а что в heap?  
> **Частота:** высокая, любят на скрининге.
> 
> **RU:** В фрейме стека — локальные примитивы значением и локальные _ссылки_ на объекты; сами объекты — всегда в heap, включая их примитивные поля. Стек — приватный для потока и освобождается выходом из метода; heap — общий и под GC. С оговоркой: JIT через escape analysis может вообще не аллоцировать объект, не покидающий метод, — но это оптимизация, а не правило языка.
> 
> **EN:** A stack frame holds local primitives by value and local _references_ to objects; the objects themselves always live in the heap, including their primitive fields. The stack is thread-private and freed by returning; the heap is shared and GC-managed. One caveat: the JIT's escape analysis may avoid allocating an object that never escapes the method — but that's an optimization, not a language rule.
> 
> **Ловушка follow-up:** «Где лежит поле `int` у объекта?» → В heap, внутри объекта. Ответ «примитивы на стеке» без этого уточнения — минус.

> [!question]- Что случилось с PermGen? Чем Metaspace отличается?  
> **Частота:** высокая; лакмус на актуальность знаний.
> 
> **RU:** В Java 8 (JEP 122) PermGen удалили. PermGen был областью внутри heap с жёстким маленьким лимитом — отсюда классический `OutOfMemoryError: PermGen space` при редеплоях. Metaspace хранит те же метаданные классов, но в native-памяти, растёт по потребности и по умолчанию не ограничен (`-XX:MaxMetaspaceSize` — по желанию). Проблема утечки загрузчиков классов не исчезла — она теперь проявляется как рост native-памяти процесса.
> 
> **EN:** Java 8 (JEP 122) removed PermGen. It was a fixed-size region inside the heap — hence the classic `OutOfMemoryError: PermGen space` on redeploys. Metaspace holds the same class metadata but in native memory, grows on demand, and is unlimited by default (`-XX:MaxMetaspaceSize` is optional). Classloader leaks didn't disappear — they now show up as native memory growth of the process.
> 
> **Follow-up:** «Зачем тогда ограничивать Metaspace?» → Чтобы утечка дала диагностируемый `OutOfMemoryError: Metaspace` вместо молчаливого `OOMKilled` от ОС.

> [!question]- Какие виды OutOfMemoryError вы встречали и о чём они говорят?  
> **Частота:** высокая на позициях с продовой ответственностью.
> 
> **RU:** `Java heap space` — кончился heap: лимит или утечка, снимаю dump. `GC overhead limit exceeded` — heap формально есть, но >98% времени уходит на GC с мизерным результатом; та же утечка, пойманная раньше. `Metaspace` — классы: генерация прокси, утечка загрузчиков. `Cannot reserve ... direct buffer memory` — direct-буферы, смотрю на Netty и лимит `MaxDirectMemorySize`. `Unable to create native thread` — упёрлись в лимиты ОС по потокам. Отдельно держу в голове, что `OOMKilled` в Kubernetes — не `OutOfMemoryError`: это ядро убило процесс за суммарный footprint.
> 
> **EN:** `Java heap space` — heap exhausted: sizing or a leak, take a dump. `GC overhead limit exceeded` — technically heap remains, but 98%+ of time goes to GC reclaiming almost nothing; same leak caught earlier. `Metaspace` — class metadata: proxy generation, classloader leaks. `Cannot reserve ... direct buffer memory` — direct buffers, check Netty and `MaxDirectMemorySize`. `Unable to create native thread` — OS thread limits. And `OOMKilled` in Kubernetes is _not_ an `OutOfMemoryError` — the kernel killed the process for total footprint.
> 
> **Senior add-on:** `-XX:+HeapDumpOnOutOfMemoryError` должен стоять в проде заранее — после смерти процесса dump снимать не с чего (глава 11.5).

> [!question]- Как устроен heap? Зачем два survivor-пространства?  
> **Частота:** высокая; проверка понимания GC до вопросов про GC.
> 
> **RU:** Heap поделён по generational hypothesis — большинство объектов умирает молодыми. Young = Eden + два survivor'а, Old — для долгожителей. Объект рождается в Eden; при minor GC живые _копируются_ в пустой survivor, возраст растёт с каждым пережитым GC, при достижении порога (динамический, максимум 15) — promotion в Old. Два survivor'а нужны для copying-схемы: живое всегда копируется в заведомо пустое пространство подряд, без дыр — фрагментация в Young невозможна, а мёртвые объекты не стоят вообще ничего.
> 
> **EN:** The heap follows the generational hypothesis — most objects die young. Young = Eden plus two survivor spaces; Old holds long-lived objects. Objects are born in Eden; on minor GC, live ones are _copied_ into the empty survivor, aging with each GC survived, and get promoted to Old at a threshold (dynamic, capped at 15). Two survivors enable the copying scheme: live objects always move into a known-empty space, packed densely — no fragmentation in Young, and dead objects cost literally nothing.
> 
> **Follow-up:** «А в G1 тоже так?» → Роли те же, но Eden/Survivor/Old — это множества регионов (~2048 на heap), а не сплошные области. Знать — плюс к сеньорности.

> [!question]- От чего зависит StackOverflowError и как на него влиять?  
> **Частота:** средняя.
> 
> **RU:** Стек потока фиксирован (`-Xss`, дефолт 1 MB на Linux x64) и переполняется, когда фреймов больше, чем влезает — на дефолте это порядка 20–30 тысяч простых фреймов. Причина — не только бесконечная рекурсия: глубокий конечный обход (несбалансированное дерево, вложенный JSON) тоже упирается. Варианты: увеличить `-Xss` для предсказуемо глубоких, но конечных случаев; переписать на явный стек/итерацию, если глубина не ограничена входными данными.
> 
> **EN:** Each thread's stack is fixed (`-Xss`, 1 MB default on Linux x64) and overflows when frames exceed capacity — roughly 20–30 thousand simple frames at the default. It's not only infinite recursion: deep but finite traversals (unbalanced trees, nested JSON) hit it too. Options: raise `-Xss` for predictably deep, bounded cases; rewrite with an explicit stack/iteration when depth is input-controlled.
> 
> **Follow-up:** «Почему `StackOverflowError` — это `Error`, но его иногда ловят?» → Формально ловить можно (стек уже размотан, место есть), но состояние программы после него не гарантировано; допустимо на границе изоляции (обработчик запроса), не как штатная логика.

> [!question]- Что такое direct memory и зачем ByteBuffer.allocateDirect?  
> **Частота:** средняя; почти обязательная, если в стеке Netty/Kafka.
> 
> **RU:** Direct-буфер живёт в native-памяти вне heap, поэтому ОС может читать/писать в него напрямую (сокеты, файлы) без промежуточного копирования между heap и native — GC не двигает эту память. Цена: аллокация дороже, чем в heap, а освобождение привязано к сборке heap-обёртки — отсюда ситуации «direct memory кончилась при полупустом heap». Лимит — `-XX:MaxDirectMemorySize`, по умолчанию равен `-Xmx`.
> 
> **EN:** A direct buffer lives in native memory outside the heap, so the OS can read/write it directly (sockets, files) without copying between heap and native — GC never moves it. The cost: allocation is more expensive than heap allocation, and the native memory is only released when GC collects the heap wrapper — hence "out of direct memory with a half-empty heap". The limit is `-XX:MaxDirectMemorySize`, defaulting to `-Xmx`.
> 
> **Follow-up:** «Почему GC не управляет этой памятью напрямую?» → GC работает только с heap; для direct-буфера в heap лежит лишь маленькая обёртка, а нативный кусок освобождается её `Cleaner`'ом.

> [!question]- Почему heap в 33 GB может работать хуже, чем в 31 GB?  
> **Частота:** редкая, сеньорский маркер — но задают в компаниях с большими heap'ами.
> 
> **RU:** Из-за CompressedOops: до ~32 GB ссылки на 64-битной JVM хранятся 32-битными (объекты выровнены по 8 байт, младшие биты адреса нулевые — 32 бита со сдвигом адресуют 32 GB). Выше границы флаг выключается, каждая ссылка становится 8-байтной, объекты и кеши распухают, полезная ёмкость heap падает, давление на кеши CPU растёт. Правило: либо ≤ ~30–31 GB, либо сразу сильно больше, чтобы компенсировать распухание.
> 
> **EN:** Because of CompressedOops: below ~32 GB, references on a 64-bit JVM are stored as 32 bits (objects are 8-byte aligned, so the low bits are zero — 32 bits shifted address 32 GB). Past the boundary the flag turns off, every reference doubles to 8 bytes, objects and caches bloat, effective heap capacity drops and CPU cache pressure rises. Rule of thumb: stay at ≤ ~30–31 GB, or jump well beyond to outweigh the bloat.
> 
> **Follow-up:** «Как проверить, включён ли он?» → `java -Xmx... -XX:+PrintFlagsFinal -version | grep UseCompressedOops` — на 31g `true (ergonomic)`, на 32g уже `false`.
