График heap в Grafana рисует пилу, у которой зубцы ползут вверх: каждый GC освобождает чуть меньше, чем в прошлый раз. Full GC учащаются, паузы растут, и через трое суток аптайма сервис умирает с `OutOfMemoryError` — а после рестарта цикл начинается заново. GC исправен, он честно собирает всё, что может. Проблема в том, _что именно_ он не имеет права собрать — и этот урок про то, как найти виновника по горячим следам: гистограмма, дамп, dominator tree, а для проблем с CPU — профилировщик, который не врёт.

> [!abstract] Ментальная модель  
> Утечка в Java — это не потерянная память, а ровно наоборот: сохранившийся путь по ссылкам от GC root к объекту, который логически уже мёртв. GC обязан хранить всё достижимое, поэтому поиск утечки — это всегда поиск нежелательного пути до корня, и инструменты (histo, heap dump, MAT) отвечают именно на вопрос «кто держит».

### Как течёт память при исправном GC

Вспомним урок 11.3: GC собирает _недостижимое_. Значит, утечь может только достижимое — объект, к которому сохранился путь от GC root, хотя программа им больше никогда не воспользуется. GC не телепат: отличить «нужен» от «достижим» он не может по определению.

![[Pasted image 20260716164958.png]]

Классические виновники, каждый — вариация одной схемы «долгоживущий держит короткоживущее»:

1. **Static-коллекции и кеши без вытеснения** — герой диаграммы. `static Map` живёт вечно (корень — класс), и всё положенное в него — тоже. Кеш без ограничения размера/TTL — это утечка с отложенным взрывателем; правильный кеш — Caffeine с явной политикой eviction.
2. **Неотписанные слушатели и колбэки.** Объект зарегистрировался в долгоживущем `EventBus`/реестре и «умер» для логики — но реестр держит ссылку, а через слушателя жив и весь его граф. Лечение — симметричный unregister (идеально — через try-with-resources), либо weak-ссылки в реестре.
3. **`ThreadLocal` в пуле потоков.** Значение живёт, пока жив поток — а потоки пула живут вечно. Забыл `remove()` в конце обработки запроса — значение (и его classloader, привет Metaspace из 11.2) висит до конца дней пода. Механика — в главе 23.13, здесь фиксируем как топ-источник продовых утечек.
4. **Внутренние нестатические классы и лямбды, захватившие `this`.** Inner class держит неявную ссылку на внешний объект (глава 3.12): маленький `Runnable`, отданный в долгоживущий планировщик, тащит за собой «внешний» объект на сотни мегабайт.

> [!tip] 💡 Из C#  
> Всё переносится один в один: static-коллекции, event handlers (легендарная .NET-утечка через невыписанный `+=`), замыкания, захватившие `this`. Даже инструментарий рифмуется: dominator tree есть и в dotMemory/PerfView. Различие в акценте: в .NET чемпион — события (языковая конструкция!), в Java — `ThreadLocal` в пулах и classloader-утечки, которых в CLR-мире практически нет из-за атомарности сборок.

### Первый уровень диагностики: гистограмма классов

Прежде чем тащить с прода многогигабайтный дамп — быстрый снимок «кто ест heap». Вот живой утекающий процесс (тот самый `static Map` c 10-килобайтными массивами), `jmap -histo:live <pid>` (то же самое — `jcmd <pid> GC.class_histogram`):

```
 num     #instances         #bytes  class name (module)
-------------------------------------------------------
   1:         16148       41926568  [B (java.base@21.0.11)
   2:         12027         288648  java.lang.String (java.base)
   4:          5057         161824  java.util.HashMap$Node (java.base)
```

`[B` — это JVM-нотация `byte[]` (знать её обязательно: `[I` — `int[]`, `[Ljava.lang.String;` — `String[]`) — 42 MB и растёт. Модификатор `:live` важен: он форсирует full GC перед подсчётом, отсекая ещё-не-собранный мусор. Один снимок мало что доказывает — **сравнивай два с интервалом**: у утечки счётчики монотонно растут. Гистограмма отвечает на «_что_ копится», но не на «_кто держит_» — для этого нужен дамп.

### Heap dump: снятие и анализ

**Снятие.** Современный способ — `jcmd` (реальный вывод с того же процесса):

```
$ jcmd 523 GC.heap_dump /path/leak.hprof
Dumping heap to /path/leak.hprof ...
Heap dump file created [48317731 bytes in 0.302 secs]
```

Легаси-синоним — `jmap -dump:live,format=b,file=...`. И обязательный продовый флаг — автодамп в момент аварии, когда он ценнее всего (проверено на программе, съевшей heap):

```
$ java -Xmx48m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/dumps/oom.hprof Hoarder
java.lang.OutOfMemoryError: Java heap space
Dumping heap to /dumps/oom.hprof ...
Heap dump file created [49480010 bytes in 0.092 secs]
```

Помни физику: дамп — это STW на всё время записи и файл размером порядка живого heap'а (наш 64-мегабайтный heap дал 48 MB; 30-гигабайтный даст десятки GB и десятки секунд паузы).

**Анализ.** Стандарт — **Eclipse MAT**; для дампов поменьше хватает VisualBM/IntelliJ. (Если в старой статье встретишь `jhat` — он удалён ещё в JDK 9; на JDK 21 проверено: `команда не найдена`.) В MAT рабочих приёма два:

- **Dominator tree** — главный экран. Объект A доминирует над B, если _все_ пути от корней к B проходят через A — то есть удаление A освободит B. Дерево доминаторов сортирует объекты по **retained size** (сколько станет мусором, если убрать этот объект) — и утечка обычно видна первой строкой: наш `HashMap` с retained size в десятки мегабайт.
- **Path to GC roots** (с опцией «exclude weak/soft references») — для найденного толстяка показывает ту самую цепочку с диаграммы: `byte[] ← HashMap$Node ← HashMap ← static Leaky.CACHE`. Это уже готовый диагноз с именем поля и класса.

### Профилирование в проде

Утечки — не единственный жанр; «куда уходит CPU» и «почему латентность» требуют профилировщика, и здесь два инструмента де-факто.

**JFR (Java Flight Recorder)** — событийный самописец, встроенный в JVM (открыт с JDK 11, JEP 328), с накладными расходами ~1% в дефолтном профиле — его _можно и нужно_ держать включённым в проде постоянно, как чёрный ящик в самолёте. Запись реальна и проверена:

```
$ java -XX:StartFlightRecording=duration=4s,filename=rec.jfr ... 
$ jfr summary rec.jfr
 Event Type                              Count  Size (bytes)
 jdk.BooleanFlag                           505         15365
 ...
$ jfr print --events jdk.GCHeapSummary rec.jfr
jdk.GCHeapSummary {
  gcId = 1
  when = "Before GC"
  heapUsed = 20.8 MB
  ...
```

Внутри — сотни типов событий: аллокации (с стектрейсами — часто утечку видно и без дампа), contention на локах, GC-фазы, файловый и сетевой I/O, исключения. Управление на лету — `jcmd <pid> JFR.start/dump/stop`, GUI-анализ — JDK Mission Control (JMC).

**async-profiler** — сэмплирующий CPU/alloc-профилировщик, стандарт для flame graphs. Его ключевое преимущество — он **не страдает safepoint bias**. Вот обещанная связка с уроком 11.3: наивные сэмплеры (всё, что построено на `jstack`/JVMTI `GetAllStackTraces`, включая старые GUI-профилировщики) умеют снимать стек _только в safepoint'е_. А safepoint'ы стоят на выходах из методов и рёбрах циклов — значит, горячий код _между_ ними для такого сэмплера невидим, и профиль систематически приписывает время не тем строкам. async-profiler снимает стеки асинхронно (через `AsyncGetCallTrace` + perf_events), в произвольных точках исполнения, и заодно видит нативные фреймы — JIT-компилятор, GC-потоки, JNI. Правило: CPU-профиль, снятый safepoint-based инструментом, годится для «в каком подсистеме беда», но не для «какая строка горячая».

> [!example] 🔬 Глубже / на синьора  
> **Shallow vs retained size — формально.** Shallow — собственные байты объекта (заголовок + поля); у `HashMap` он смешные ~48 байт. Retained — суммарный размер всего, что станет недостижимым при его удалении, то есть размер его поддерева в дереве доминаторов. Утечки ищут _только_ по retained: чемпионы по shallow — почти всегда невинные `byte[]`, интересен тот, кто их удерживает. Тонкость доминирования: объект, на который есть _два_ независимых пути, не доминируется ни одним из держателей — его retained «размазывается» вверх до общего доминатора; поэтому shared-кеши в dominator tree выглядят скромнее, чем весят.
> 
> **Куда смотреть, когда heap стабилен, а RSS растёт.** Финальный босс диагностики — утечка _вне_ heap (вспоминай карту из 11.2). Инструмент — **NMT** (Native Memory Tracking): запуск с `-XX:NativeMemoryTracking=summary` (налог 5–10%, поэтому не «всегда включён»), снятие — `jcmd <pid> VM.native_memory summary`. Реальный вывод:
> 
> ```
> Total: reserved=1657872KB, committed=90720KB
> -  Java Heap (reserved=262144KB, committed=65536KB)
> -      Class (reserved=1048665KB, committed=217KB)
> -     Thread (...)  -  Code (...)  -  GC (...)
> ```
> 
> Разбивка по категориям сразу сужает круг: растёт `Class` — утечка классов/загрузчиков; `Thread` — кто-то плодит потоки; ничего не растёт, а RSS пухнет — значит, память утекает мимо учёта JVM (нативная библиотека, direct-буферы под `Other`, аллокатор glibc) — там уже `jemalloc`-профилирование и `pmap`.
> 
> **Epsilon GC как инструмент.** `-XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC` (JEP 318) — коллектор, который _не собирает ничего_: heap кончился — процесс умер. Звучит как шутка, но это точный прибор: замер истинной скорости аллокации приложения, тесты «уложимся ли в бюджет памяти без GC», изоляция GC-эффектов в бенчмарках.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Heap dump на боевом узле под нагрузкой** — STW на секунды-минуты для больших heap'ов плюс запись десятков гигабайт на диск. Порядок действий: вывести узел из балансировки → снять дамп → вернуть. Исключение — автодамп по OOM: процессу уже всё равно.
> 2. **`HeapDumpPath` в никуда.** Дамп по OOM в контейнере, пишущийся в эфемерную файловую систему, умирает вместе с подом; в переполненный диск — не пишется вовсе. Путь должен указывать на persistent volume с местом порядка `-Xmx`.
> 3. **Гистограмма без `:live` и без второго снимка.** Несобранный мусор в снимке неотличим от утечки; и даже «живые» 500 MB в кеше — возможно, штатный прогретый кеш. Диагноз ставится по _динамике_: два-три снимка с интервалом, монотонный рост одних и тех же классов.
> 4. **Паника по RSS.** JVM неохотно возвращает память ОС: heap, выросший в пиковый час до `-Xmx`, снаружи выглядит «утечкой» навсегда (G1 умеет отдавать при простое только с JEP 346, Java 12, и не агрессивно). Рост RSS при стабильном used heap — повод для NMT, а не для рестарта по крону.
> 5. **CPU-профиль через `jstack` в цикле или старый сэмплер** — safepoint bias: время приписывается границам методов, горячие участки между safepoint'ами невидимы. Для строк и flame graphs — async-profiler или JFR; jstack-серия годится только для «где всё зависло» (и незаменима для deadlock'ов — глава 23.11).
> 6. **JFR включают после пожара.** Инцидент кончился — данных нет; запись «после» показывает здоровую систему. Правильный паттерн — continuous recording с кольцевым буфером (`maxage`/`maxsize`) всегда, и `JFR.dump` в момент инцидента достаёт последние N минут истории.

### На собесе

> [!question]- Как возможна утечка памяти в языке с GC? Приведите примеры.  
> **Частота:** максимальная — визитная карточка темы.
> 
> **RU:** GC собирает недостижимое, а утечка — это достижимый, но логически мёртвый объект: сохранился путь по ссылкам от GC root. Классика: static-коллекции и кеши без вытеснения; неотписанные слушатели в долгоживущих реестрах; `ThreadLocal` без `remove()` в пуле потоков — значение живёт, пока жив поток пула, то есть вечно; внутренние классы и лямбды, неявно захватившие `this` и уехавшие в долгоживущую структуру.
> 
> **EN:** GC collects the unreachable; a leak is a reachable but logically dead object — a reference path from a GC root survived. Classics: static collections and caches without eviction; listeners never unregistered from long-lived registries; `ThreadLocal` without `remove()` on pooled threads — the value lives as long as the pool thread, i.e. forever; inner classes and lambdas implicitly capturing `this` and escaping into long-lived structures.
> 
> **Follow-up:** «А циклические ссылки?» → Не утечка: трассирующий GC собирает недостижимый цикл целиком (в отличие от reference counting).

> [!question]- Течёт память в проде — ваши действия по шагам?  
> **Частота:** очень высокая на позициях с он-коллом; проверяют процесс, а не эрудицию.
> 
> **RU:** (1) Подтверждаю по метрикам: растёт ли used heap _после_ full GC — только этот тренд отличает утечку от штатного роста до `-Xmx`. (2) Дешёвая разведка: `jcmd GC.class_histogram` дважды с интервалом — какие классы монотонно растут. (3) Heap dump (`jcmd GC.heap_dump`; узел — из балансировки; при OOM дамп уже есть, если стоял `HeapDumpOnOutOfMemoryError`). (4) MAT: dominator tree по retained size → path to GC roots с исключением weak/soft — получаю конкретное поле-держателя. (5) Фикс + подтверждение теми же метриками.
> 
> **EN:** (1) Confirm via metrics: does used heap grow _after_ full GCs — only that trend distinguishes a leak from normal growth toward `-Xmx`. (2) Cheap recon: `jcmd GC.class_histogram` twice with an interval — which classes grow monotonically. (3) Heap dump (`jcmd GC.heap_dump`; node out of the balancer first; on OOM the dump already exists if `HeapDumpOnOutOfMemoryError` was set). (4) MAT: dominator tree by retained size → path to GC roots excluding weak/soft — yielding the exact holding field. (5) Fix and re-verify with the same metrics.
> 
> **Слабый ответ:** «поставлю больше `-Xmx`» — откладывает OOM, увеличивая паузы; интервьюер ждёт слова «dump» и «кто держит».

> [!question]- Чем shallow size отличается от retained size?  
> **Частота:** средняя, но решающая — по ней видно, открывал ли человек MAT.
> 
> **RU:** Shallow — собственный размер объекта: заголовок плюс поля (у коллекции — копейки). Retained — всё, что станет недостижимым при удалении объекта: размер его поддерева в дереве доминаторов. Утечки ищут по retained: по shallow всегда лидируют безликие `byte[]` и `char[]`, а виновник — маленький объект с гигантским retained, который их удерживает.
> 
> **EN:** Shallow is the object's own size — header plus fields (tiny for a collection). Retained is everything that would become unreachable if the object were removed — the size of its subtree in the dominator tree. Leak hunting uses retained: by shallow size, anonymous `byte[]`/`char[]` always dominate, while the culprit is a small object with a huge retained size holding them.
> 
> **Follow-up:** «Что такое dominator tree?» → A доминирует над B, если все пути от корней к B идут через A; дерево доминаторов и даёт retained за O(1) на объект.

> [!question]- Чем профилировать CPU в проде? Почему не серия thread dump'ов?  
> **Частота:** средняя-высокая.
> 
> **RU:** JFR — постоянно включённый событийный самописец с ~1% оверхеда, и async-profiler — сэмплирующий, для flame graphs. Серия jstack'ов страдает safepoint bias: стек снимается только в safepoint'ах (выходы из методов, рёбра циклов), поэтому время систематически приписывается не тем строкам, а код между safepoint'ами невидим. async-profiler сэмплирует асинхронно в произвольных точках и видит нативные фреймы — JIT, GC, JNI. jstack остаётся инструментом для «что сейчас висит» и для диагностики deadlock'ов.
> 
> **EN:** JFR — an always-on event recorder at ~1% overhead — and async-profiler for sampling and flame graphs. A series of jstack dumps suffers from safepoint bias: stacks are only captured at safepoints (method exits, loop back-edges), so time gets systematically attributed to the wrong lines, and code between safepoints is invisible. async-profiler samples asynchronously at arbitrary points and sees native frames — JIT, GC, JNI. jstack remains the tool for "what's stuck right now" and deadlock diagnosis.
> 
> **Senior add-on:** механика bias — прямое следствие устройства safepoint'ов из темы GC: остановиться поток может только там, где JVM знает раскладку его ссылок.

> [!question]- Что такое JFR и можно ли его держать включённым в проде?  
> **Частота:** средняя, растёт вместе с сеньорностью.
> 
> **RU:** Java Flight Recorder — встроенный в JVM событийный рекордер (открыт с JDK 11, JEP 328): аллокации со стектрейсами, contention локов, GC-фазы, I/O, исключения. Дефолтный профиль стоит около 1% — поэтому правильный паттерн именно «включён всегда»: continuous recording в кольцевой буфер (`maxage`/`maxsize`), а при инциденте `jcmd JFR.dump` достаёт последние минуты истории. Анализ — JDK Mission Control или CLI `jfr print`/`summary`.
> 
> **EN:** Java Flight Recorder — the JVM's built-in event recorder (free since JDK 11, JEP 328): allocations with stack traces, lock contention, GC phases, I/O, exceptions. The default profile costs about 1%, so the right pattern is exactly "always on": continuous recording into a ring buffer (`maxage`/`maxsize`), with `jcmd JFR.dump` extracting the last minutes of history during an incident. Analysis — JDK Mission Control or the `jfr print`/`summary` CLI.
>

> [!question]- Механика утечки через ThreadLocal в пуле потоков?  
> **Частота:** средняя; любимый вопрос там, где есть свои пулы или Tomcat.
> 
> **RU:** Значение `ThreadLocal` физически лежит в `ThreadLocalMap` самого потока — значит, время его жизни привязано к потоку, а потоки пула не умирают. Не вызвал `remove()` в конце задачи — значение достижимо от GC root (объект `Thread`) навсегда. Отягчающее: через значение часто достижим его classloader — в контейнерах это блокирует выгрузку целых приложений (утечка Metaspace). Терапия: `try/finally` с `remove()`; сигнатура ссылок в MAT: `Thread → ThreadLocalMap → Entry → value`.
> 
> **EN:** A `ThreadLocal` value physically lives in the thread's own `ThreadLocalMap`, so its lifetime is tied to the thread — and pool threads never die. Skip `remove()` at task end, and the value stays reachable from a GC root (the `Thread` object) forever. Aggravating factor: the value often keeps its classloader reachable — in containers this blocks unloading whole applications (a Metaspace leak). Treatment: `try/finally` with `remove()`; the MAT signature: `Thread → ThreadLocalMap → Entry → value`.
> 
> **Follow-up:** «Entry там weak — почему не спасает?» → Weak — только ключ (сам `ThreadLocal`); значение держится сильно, пока map жива. Детали — глава 23.13.

> [!question]- Память процесса растёт, а used heap стабилен. Куда смотреть?  
> **Частота:** ниже среднего, сеньорский финальный босс — и отличный способ показать системность.
> 
> **RU:** Значит, течёт вне heap — по карте памяти: Metaspace (генерация классов, утечка загрузчиков), direct-буферы (Netty; лимит `MaxDirectMemorySize`), стеки (растёт число потоков — thread dump), code cache, либо чисто нативные аллокации (JNI-библиотеки, glibc). Инструмент — NMT: перезапуск с `-XX:NativeMemoryTracking=summary`, затем `jcmd VM.native_memory summary` и сравнение снимков по категориям — какая растёт, туда и копать. Если NMT ничего не показывает, а RSS растёт — утечка мимо учёта JVM: `pmap`, jemalloc-профилирование нативной библиотеки.
> 
> **EN:** Then the leak is outside the heap — walk the memory map: Metaspace (class generation, classloader leaks), direct buffers (Netty; `MaxDirectMemorySize`), thread stacks (growing thread count — take a thread dump), code cache, or purely native allocations (JNI libraries, glibc). The tool is NMT: restart with `-XX:NativeMemoryTracking=summary`, then `jcmd VM.native_memory summary` and diff snapshots by category — whichever grows is where to dig. If NMT shows nothing while RSS climbs, it's leaking past JVM accounting: `pmap`, jemalloc profiling of the native library.
> 
> **Follow-up:** «А почему просто не смотреть на RSS?» → RSS не разложен по причинам; и JVM неохотно возвращает память ОС, так что высокий RSS после пика — норма, а не утечка.
