Первый рабочий день в IDEA почти у всех выглядит одинаково: клонируешь репозиторий, ждёшь индексацию — и половина импортов подчёркнута красным, `cannot resolve symbol`, хотя файлы лежат вот они, рядом. В Rider такого не бывает: SDK-style `.csproj` неявно собирает все `.cs` под собой, папка автоматически часть проекта. IDEA так не делает — она не считает папку исходниками, пока ей это явно не сказали (или пока это не вывел Maven/Gradle из build-файла).

Это и есть главный сдвиг в голове при переезде: **IDEA относится к структуре проекта гораздо более явно, чем .NET-тулинг**. Где Rider додумывает за тебя, Java-тулинг требует пометки. Отсюда практическое следствие, ради которого стоит прочитать главу: 90% «странного поведения» первой недели — это не баги IDE, а непомеченные roots, не выставленный SDK или забытый reload после правки `pom.xml`. Разберёшься с этими механизмами один раз — перестанешь терять по полчаса на пустом месте.

Идём по фичам ровно в том порядке, в каком они мешают работать: структура проекта → запуск и дебаг → генерация кода → Maven/Gradle → хоткеи. В конце — что из этого реально всплывает на собесе (спойлер: не хоткеи).

### 2.1. Project structure: `Solution/.csproj` → `Project/Module`

В Rider у тебя двухуровневая модель: `Solution (.sln)` — контейнер, внутри `Projects (.csproj)`, каждый компилируется в свою сборку. В IDEA модель называется иначе, но мапится почти один-в-один.

**Mental model:** `Project` в IDEA — это весь воркспейс (аналог `Solution`), `Module` — единица со своими исходниками и зависимостями (аналог `.csproj`). Для обычного Maven-проекта это один `Project` + один `Module`; для multi-module Maven — один `Project` + N модулей.

- `Project` (IDEA) ≈ `Solution` (Rider) — корень, один открытый проект = одно окно IDEA.
- `Module` (IDEA) ≈ `.csproj` (Rider) — собственные исходники, зависимости и (опционально) свой SDK.

**Content roots — то, чего в Rider нет.** Внутри модуля каждая папка имеет тип, и ты видишь его по цвету в дереве проекта: `Sources root` (синий, обычный код), `Test sources root` (зелёный — отдельный тип, чтобы тесты не попадали в артефакт и видели `test`-scope зависимости), `Resources root` / `Test resources root` (не-код, в classpath как есть), `Excluded` (IDEA игнорирует при индексации — `target/`, `build/`). При импорте Maven/Gradle-проекта IDEA расставляет пометки сама из конвенции; руками трогаешь только при нестандартном layout или «голом» проекте без системы сборки.

![[Pasted image 20260702182711.png]]

> 💡 **Из C#:** в SDK-style `.csproj` принадлежность файла проекту выводится автоматически (implicit glob по папке). В IDEA это либо явная пометка папки, либо вывод из `pom.xml`/`build.gradle`. Поэтому красный класс при живом файле почти всегда = папка не помечена как sources root, либо ты её случайно исключил.

**Где живёт SDK — и почему это не `TargetFramework`.** Открывается через `Project Structure` (`⌘;` на Mac / `Ctrl+Alt+Shift+S` на Win/Linux). Есть **Project SDK** (дефолт для всего проекта) и **Module SDK** (модуль может переопределить). Ключевое отличие от .NET: это **не «таргет»** вроде `TargetFramework`, а ссылка на реальную установку JDK на диске. Твой Temurin лежит где-то вроде `/Library/Java/JavaVirtualMachines/temurin-21.jdk/...` — этот путь и выставляешь как Project SDK.

> ⚠️ **Подводный камень — language level ≠ SDK.** У модуля есть отдельная настройка `Language level` (`Project Structure → Modules → вкладка Sources`). Можно иметь установленный JDK 21, но language level, выставленный в 8 — и тогда `record`, `sealed` или pattern matching for `switch` IDEA подсветит как ошибку, хотя реальный `javac` из JDK 21 их бы собрал. Для Maven-проектов language level IDEA берёт из `<maven.compiler.release>`. Если IDE ругается на современный синтаксис при правильном JDK — первым делом проверь language level, а не SDK.

> 🔬 **Глубже / на синьора.**
> 
> - **Где физически хранится модель проекта.** IDEA держит конфигурацию в папке `.idea/` (и, для старого формата, в `*.iml`-файлах модулей): content roots, зависимости, language level, SDK. Но для Maven/Gradle-проекта это **производная**, а не источник истины — при reload IDEA перечитывает `pom.xml`/`build.gradle` и перегенерирует модель. Правка `.iml` руками при живом Maven бессмысленна: следующий reload её затрёт. Отсюда правило — структуру проекта менять через build-файл, не через диалоги IDE.
> - **Delegate build to Maven/Gradle.** В `Settings → Build Tools → Maven → Runner` есть переключатель: компилировать встроенным компилятором IDEA или делегировать сборку самому Maven/Gradle. Делегирование медленнее на инкрементальных сборках, но даёт байт-в-байт тот же результат, что CI (те же плагины, тот же порядок фаз). Расхождение «у меня собирается, на CI падает» часто лечится именно включением делегирования.
> - **Invalidate Caches.** Если пометки на месте, а класс всё равно красный — `File → Invalidate Caches / Restart`. Это сброс индексов (тяжёлая артиллерия), помогает при битом кеше. Не первый инструмент, а последний.

### 2.2. Run Configurations и дебаггер

В Rider run-конфиги создаются автоматически под проекты — выбрал startup project, нажал play. В IDEA механика похожая, но первую неделю путает пара мелочей.

**Контекстный запуск.** Ставишь курсор в `main()` или в тест, жмёшь зелёную стрелку в gutter — IDEA создаёт **временную** run-конфигурацию. С клавиатуры: `⌃R` (run) / `⌃D` (debug) на Mac; на Win/Linux контекстный запуск — `Ctrl+Shift+F10`. Временный конфиг становится постоянным, если сохранить его явно (`Edit Configurations` в выпадашке на тулбаре). В постоянном задаёшь main class, program arguments, VM options, env variables, working directory, classpath модуля — те же поля, что в Rider, собранные в один диалог.

**Conditional breakpoints.** Правый клик по точке останова → поле `Condition`, пишешь boolean-выражение на Java (`order.getId() == 42`, `items.isEmpty()`) — брейк срабатывает, только когда `true`. Механика идентична Rider. Управлять всеми брейками сразу — `⌘⇧F8` / `Ctrl+Shift+F8` (`View Breakpoints`): условия, count-фильтры (сработать только на N-й раз), логирование вместо остановки.

**Evaluate Expression — самый важный хоткей дебага.** `⌥F8` / `Alt+F8`. Это твой Rider Immediate Window + QuickWatch в одном окне: на паузе пишешь любой Java-код — вызываешь методы, читаешь поля, даже мутируешь состояние объекта прямо в работающей JVM. Быстрый вариант «глянуть выражение под курсором» — `⌘⌥F8` / `Ctrl+Alt+F8` (`Quick Evaluate`). Заучи `⌥F8` в первый день. `Watches` живут в окне Debug (`⌘5` / `Alt+5`), панель `Variables`.

**Exception breakpoints.** `⌘⇧F8` → `+` → `Java Exception Breakpoints` → выбираешь класс исключения (например `NullPointerException`), можно заскоупить на caught/uncaught. Встаёшь ровно в точке, где исключение бросают, ещё **до раскрутки стека** — прямой аналог риддеровского «break on exception». Незаменимо, когда падает в глубине и стектрейс не говорит, откуда прилетело.

**HotSwap и почему он такой ограниченный.** Правишь тело метода во время дебаг-сессии, делаешь Build (`⌘F9` / `Ctrl+F9`) — IDEA подменяет новый байткод в живой JVM без рестарта. Но ограничение жёсткое, и идёт оно **от JVM, а не от IDEA**: стандартный HotSwap умеет менять **только тела методов**. Нельзя добавлять/удалять методы и поля, менять сигнатуру, менять иерархию, трогать static-инициализаторы. Попробуешь — IDEA скажет «hot swap finished but N classes were not reloaded» и предложит рестарт.

> 💡 **Из C#:** Edit-and-Continue в VS/Rider ограничен примерно так же и по той же причине — CLR тоже не любит структурные изменения на лету. Ощущение знакомое, просто пороги чуть другие.

> 🔬 **Глубже / на синьора.**
> 
> - **Почему только тела методов.** За HotSwap отвечает `JVMTI`-операция `RedefineClasses`. Она разрешает менять байткод метода, но запрещает всё, что меняет «схему» класса: набор и порядок полей (сломало бы уже размещённые в heap объекты), таблицу методов / vtable (сломало бы виртуальную диспетчеризацию), иерархию. Добавить метод = изменить method table → стандартный JVM это не делает. Расширенную переопределяемость дают DCEVM/HotswapAgent или коммерческий JRebel — в дефолтной поставке JDK их нет.
> - **Как дебаггер подключается к JVM.** Под капотом — `JDWP` (Java Debug Wire Protocol) поверх `JVMTI`. JVM запускается с агентом `-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005`. Тот же протокол — для **remote debug**: приложение крутится в контейнере/на стенде с этими флагами, а ты подключаешься run-конфигом `Remote JVM Debug`. Стандартный способ дебажить то, что уже запущено в докере.
> - **Drop Frame.** На паузе можно «откатить» выполнение к началу текущего метода и пройти его заново — откатывается только стек-фрейм, не heap. Полезно, когда проскочил интересное место на шаг.

**Мелочи, бьющие по мышечной памяти после Rider** (шаги дебага; `F7`/`F8`/`⇧F8` совпадают на всех ОС): `F8` — Step Over, `F7` — Step Into, `⇧F8` — Step Out (в VS/Rider это `F10`/`F11`/`⇧F11`). Resume — `⌘⌥R` / `F9`. Run to Cursor — `⌥F9` / `Alt+F9`. Inline-значения переменных на паузе включены по умолчанию, как в Rider.

### 2.3. Code generation: `⌘N` (Win/Linux — `Alt+Insert`)

Сразу поправка к популярной формулировке: «Alt+Insert» — это Windows/Linux keymap. На Mac меню `Generate` висит на `⌘N`. Внутри тела класса получаешь меню: `Constructor`, `Getter`/`Setter`, `equals() and hashCode()`, `toString()`, `Override Methods` (`⌃O`), `Implement Methods` (`⌃I`), `Delegate Methods`.

Почему это генерят руками, а не даёт язык, как в C#? Потому что в Java нет auto-properties, нет компиляторного value-equality для обычных классов и нет `record` по умолчанию. Boilerplate, который в C# материализует язык или ReSharper, тут превращается в реальные строки в файле. Минус — визуальный шум; плюс — всё явно и дебажится как обычный код (по сгенерированному `equals` можно поставить брейк).

**`equals()` and `hashCode()`: два template'а — здесь копают на собесе.** Визард спрашивает: какие поля включить и какой **template**. Template определяет проверку типа:

- **`getClass()`** — `equals` вернёт `true` только для точно того же класса. Безопасно при наследовании (симметрия не ломается), но подкласс никогда не равен суперклассу, даже если по данным совпадают.
- **`instanceof`** — подкласс может равняться суперклассу. Гибче, но легко словить нарушение симметрии (`a.equals(b) != b.equals(a)`), если у обоих в иерархии свой `equals`.

Дефолтный template в IntelliJ — `getClass()`. В списке есть вариант «`java.util.Objects.equals` and `hashCode`» — он чище (использует `Objects.equals` / `Objects.hash` вместо ручных null-проверок). Бери его.

> 💡 **Из C#:** в C# `record` даёт value equality из коробки, и выбор «getClass vs instanceof» сделан за тебя. В Java до records ты его делаешь сам — и это ровно тот момент, где интервьюер копает в сторону контракта `equals` (полный разбор — глава 3.8).

Сгенерированный «`Objects`»-вариант для простого value-класса (проверено на JDK 21):

java

```java
import java.util.Objects;

public final class Money {
    private final long amountMinor;
    private final String currency;

    public Money(long amountMinor, String currency) {
        this.amountMinor = amountMinor;
        this.currency = currency;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Money money = (Money) o;
        return amountMinor == money.amountMinor
                && Objects.equals(currency, money.currency);
    }

    @Override
    public int hashCode() {
        return Objects.hash(amountMinor, currency);
    }
}
```

**Когда вместо генерёжки — `record`.** Если класс — чистый неизменяемый носитель данных (DTO, value object, без идентичности и мутабельного состояния), бери `record` (Java 16+, [JEP 395](https://openjdk.org/jeps/395)). Он бесплатно даёт `equals`/`hashCode`/`toString`/аксессоры — Java-аналог C# `record`:

java

```java
public record Money(long amountMinor, String currency) {}
```

Тот же контракт, что у класса выше, но без 20 строк boilerplate. Автосгенерированный `toString()` печатает `Money[amountMinor=1050, currency=EUR]` (проверено на JDK 21). Руками генеришь, когда нужна **мутабельность**, **наследование** или это **JPA-entity**: Hibernate плохо дружит с records как с сущностями — ему нужен no-arg конструктор и сеттеры, а `record` их не даёт. (Records целиком — глава 17.)

### 2.4. Maven/Gradle integration

С NuGet ты привык, что пакеты «просто есть»: добавил `PackageReference`, оно восстановилось, транзитивные зависимости спрятаны, конфликты разрулились тихо. В Java-тулинге то же по сути, но IDE заметно прозрачнее показывает кишки — и это к лучшему, потому что в Maven конфликты версий разбирать руками приходится чаще.

**Tool window.** Справа на gutter иконка Maven (слоник) или Gradle — открывает дерево: lifecycle-фазы, плагины, зависимости.

**Source of truth — build-файл.** `pom.xml` (Maven) или `build.gradle(.kts)` (Gradle). Ключевое отличие от ощущений NuGet: правка build-файла **не обновляет модель IDE автоматически**.

> ⚠️ **Reload ≠ Rebuild — главная ловушка первой недели.** **Reload** (плавающая кнопка «Load Maven Changes» / `⌘⇧I`) перечитывает build-файл: заново резолвит зависимости, пересинхронивает модель модуля (content roots, deps, language level). **Rebuild** (`⌘⇧F9` / `Ctrl+Shift+F9`) перекомпилирует `.class`. Reload — после правки **build-файла**; rebuild — после правки **кода**. Самая частая ошибка новичка: добавил зависимость в `pom.xml`, импорт красный, паника — а ты просто не нажал reload.

**Транзитивные зависимости.** Вот где Maven честнее NuGet: ты реально лезешь в дерево (`Maven tool window → Dependencies` или `mvn dependency:tree`). Когда две библиотеки тянут разные версии третьей, Maven разрешает конфликт по своему правилу, а ты в рантайме можешь словить `NoSuchMethodError` — и идёшь читать это дерево.

> 💡 **Из C#:** NuGet обычно «просто работает» и прячет транзитивщину. В Java посмотреть `mvn dependency:tree` и добавить `<exclusion>` — рутина, а не экзотика.

Полная механика (как Maven выбирает версию, scopes, `<exclusion>`, BOM, multi-module наследование) — тема **главы 8**. Здесь важно одно: IDE не резолвит зависимости сама и не прячет конфликты — она их показывает, а разбираешь ты.

> 🔬 **Про поиск зависимостей — почему в Java нет «NuGet-подобного» поиска.** Раньше в IDEA был живой поиск по Maven Central прямо из `pom.xml` (сервис **Package Search**). **Веб-сервис Package Search отключён 1 апреля 2025, плагин снят с поддержки, версии для IDEA 2025.1+ нет.** Что осталось: автодополнение координат в build-файле на основе локального `~/.m2/repository` (только то, что ты уже качал), плюс можно вручную скачать индекс Maven Central (`Settings → Maven → Repositories → Update`) — индекс здоровый, качается долго. И даже с индексом это поиск **по координатам / фрагменту имени артефакта**, а не по смыслу: в NuGet пакеты названы как продукты (`Newtonsoft.Json`), в Maven — reverse-DNS (`com.fasterxml.jackson.core:jackson-databind`). Нет канонического «Entity Framework»: сначала ты должен **знать**, что ORM — это Hibernate или Spring Data JPA, и только потом искать `hibernate-core`. Нормальный воркфлоу для незнакомой либы — **гугл → mvnrepository.com / доки → скопировать координаты → в `pom.xml` → reload**. Это реальное положение дел в экосистеме на 2026 год, а не пробел в навыках.

**Два must-have плагина:** **Key Promoter X** (при клике мышкой показывает хоткей этого действия — мышечная память набивается за неделю) и опционально **GitToolBox** (inline-blame, статус веток). Остальное в Ultimate встроено.

### Что потрогать руками

1. **Сломай и почини roots** (~10 мин). В `Project Structure` сними у `src/main/java` пометку sources root, посмотри, как всё покраснело, верни. Прочувствуй связь «красный класс ↔ непомеченная папка».
2. **Conditional breakpoint + evaluate** (~15 мин). Цикл на 100 итераций, брейк с условием `i == 73`, на паузе через `⌥F8` вызови метод на переменной и поменяй ей значение. Затем сделай то же count-фильтром.
3. **equals: два template'а + record** (~20 мин). Сгенери `equals`/`hashCode` с `getClass`, потом с `Objects` — сравни диффом. Перепиши класс как `record` — посмотри, сколько кода исчезло.
4. **Reload vs rebuild** (~15 мин). Добавь зависимость в `pom.xml`, НЕ делай reload — убедись, что импорт красный. Reload — ожил. Запусти `mvn dependency:tree`, найди транзитивную зависимость, которую явно не подключал.
5. **Exception breakpoint + чтение стека** (~10 мин). Брейк на `NullPointerException`, спровоцируй NPE через обёртку, пройди стек, найди настоящий `Caused by`.

### На собесе

Хоткеи наизусть никто не гоняет. Тулинг всплывает в двух местах. **Live-coding:** дадут пописать в IDEA, и сеньор смотрит — тормозишь ли на навигации, умеешь ли поставить брейк и сделать `evaluate`, или сидишь в `System.out.println`. **Устные вопросы про дебаг и окружение** — ниже.

**Q: «Как ловить баг, который воспроизводится раз в сто запусков?»**

- Слабый ответ: «добавлю логов и буду ждать» — работает, но медленно.
- Модель (RU): «Conditional breakpoint по условию, при котором баг проявляется, либо count-фильтр на N-й итерации. Если это исключение — exception breakpoint на конкретный класс, встану ровно в точке броска до раскрутки стека».
- Model (EN): "A conditional breakpoint on the state that triggers the bug, or a hit-count filter. If it surfaces as an exception, an exception breakpoint on that class stops me right where it's thrown, before the stack unwinds."

**Q: «Прилетел `NoSuchMethodError` в проде — что делаешь?»**

- Follow-up: «почему это вообще возможно, если компилировалось?» Ловушка: это **не** ошибка компиляции — код собран против одной версии, а в рантайме на classpath оказалась другая (конфликт транзитивных версий).
- Модель (RU): «Смотрю `mvn dependency:tree`, ищу, кто тянет конфликтующие версии одной библиотеки. Maven оставляет одну по своему правилу, в рантайме грузится не тот класс — отсюда отсутствующий метод. Фикс — привести версии через `<exclusion>` или BOM».
- Model (EN): "Run `mvn dependency:tree` and find who pulls conflicting versions of the same library. Maven keeps one by its resolution rule; at runtime the wrong class loads, so the method is missing. Fix by aligning versions via an `<exclusion>` or a BOM."

**Q (спрашивают чаще, чем кажется): «Как читаешь стектрейс?»**

- Ключевое: сверху вниз до **первого своего класса**, и обязательно докручиваешь `Caused by:` до самого глубокого. **Последний** `Caused by` — корневая причина; верхнее исключение часто обёртка (exception translation — глава 6.7).
- Модель (RU): «Верх стека — где всплыло, а не где сломалось. Иду по `Caused by` до последнего — там реальная причина; выше обёртки. В своём коде ищу первый свой фрейм».
- Model (EN): "The top of the trace is where it surfaced, not where it broke. I follow the `Caused by` chain to the last one — that's the root cause; the outer ones are wrappers. I look for the first frame in my own code."
- Почему сигнал: «мне нужен последний `Caused by`, остальное — обёртки» звучит как человек, который чинил прод, а не учил по книжке.
