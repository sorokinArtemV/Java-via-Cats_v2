Граница между массивом и списком пересекается в каждом Java-проекте по многу раз на дню: API отдаёт `List`, легаси-метод хочет массив, тестовые данные удобно объявить литералом, а поле класса нужно защитить от внешних мутаций. Для всего этого JDK предлагает с виду взаимозаменяемые инструменты — `Arrays.asList`, `List.of`, `Collections.unmodifiableList`, `List.copyOf`, — которые на самом деле дают три _разных_ уровня гарантий. Перепутаешь уровень — получишь либо `UnsupportedOperationException` в проде, либо тихую утечку инкапсуляции, когда чужой код мутирует твои внутренности. Именно на этой границе живёт целая обойма классических вопросов собеса, и в этом уроке мы её пройдём насквозь: конверсии в обе стороны и лестница гарантий от «нельзя менять через эту ссылку» до «не изменится никогда».

**Ментальная модель.** Конверсии массив↔список спотыкаются о два факта из прошлого урока: примитивы не лезут в дженерики, а стёртый тип не восстановить. Защита данных — лестница из трёх ступеней: _view_ защищает одну ссылку, _immutable_ защищает контейнер целиком, _defensive copy_ разрывает связь с источником.

### Из `List` в массив: почему `toArray()` отдаёт `Object[]`

```java
List<String> list = new ArrayList<>(List.of("x", "y", "z"));

Object[] objs = list.toArray();               // Object[] — не String[]!
String[] bad = (String[]) list.toArray();     // ClassCastException в рантайме
String[] strs = list.toArray(new String[0]);  // ✅ String[]
```

Первая строка — прямое следствие стирания типов из `1.7a`: в рантайме `ArrayList` не знает, что его `T` — это `String`, внутри у него `Object[]`, и создать `String[]` ему просто не из чего. Каст во второй строке не спасает — объект _реально_ имеет класс `[Ljava.lang.Object;`, и `checkcast` его honestно отвергает (проверено на JDK 21: `ClassCastException`).

Рабочая идиома — перегрузка с массивом-образцом: `toArray(new String[0])`. Тип элементов приезжает в рантайм через сам объект массива (массив-то reified — он свой тип помнит), и метод через рефлексию создаёт массив нужного класса.

> [!tip] 💡 Из C#
> этой проблемы не существует — `list.ToArray()` возвращает честный `string[]`, потому что дженерики reified и `T` доступен в рантайме. Идиома «передай пустой массив-образец» — чисто джавовый костыль под erasure.

### Из массива в `List`: капкан с примитивами

Обратное направление, эталонная ловушка собеса:

```java
int[] nums = {1, 2, 3};
List<?> trap = Arrays.asList(nums);
System.out.println(trap.size());                  // 1 (!), элемент имеет тип int[]

Integer[] boxes = {1, 2, 3};
System.out.println(Arrays.asList(boxes).size());  // 3
```

Сигнатура — `Arrays.asList(T... a)`, varargs. Типовой параметр `T` не может быть примитивом (дженерики работают только со ссылочными типами — снова erasure), поэтому `int[]` не раскладывается на элементы: компилятор выводит `T = int[]` и упаковывает весь массив как **один** элемент. Результат — `List<int[]>` размера 1, без единого warning. С `Integer[]` всё ожидаемо: `T = Integer`, три элемента.

Правильная конверсия примитивного массива — через примитивный стрим с боксингом:

```java
List<Integer> ok = Arrays.stream(nums).boxed().toList();   // [1, 2, 3], Java 16+
```

### `Arrays.asList` — полумутабельная обёртка с общим хранилищем

Даже с ссылочными типами `Arrays.asList` возвращает не тот список, который ожидает интуиция. Проверим его класс и поведение:

```java
String[] arr = {"a", "b", "c"};
List<String> list = Arrays.asList(arr);

System.out.println(list.getClass().getName());  // java.util.Arrays$ArrayList

list.set(0, "z");        // ✅ работает
System.out.println(arr[0]);       // z — записалось в исходный массив!

arr[1] = "Q";
System.out.println(list.get(1));  // Q — и в обратную сторону тоже

list.add("d");           // ❌ UnsupportedOperationException
list.remove(0);          // ❌ UnsupportedOperationException
```

Это `java.util.Arrays$ArrayList` — приватный вложенный класс, **не** родственник `java.util.ArrayList`, только тёзка. По Javadoc он «returns a fixed-size list backed by the specified array»: обёртка фиксированного размера прямо _поверх_ переданного массива, без копирования.

Отсюда оба свойства. `set` работает — он пишет в ячейку существующего массива, размер не трогает. `add`/`remove` меняют размер — а массив нерастяжим, поэтому `UnsupportedOperationException`. И хранилище **общее**: изменения протекают в обе стороны, список и массив — два окна в одну память. Такой список «полумутабельный»: содержимое ячеек менять можно, структуру — нет.

Когда нужен нормальный изменяемый список — копируй: `new ArrayList<>(Arrays.asList(arr))`.

### `List.of` — настоящая иммутабельность (Java 9, JEP 269)

`List.of(...)` из JEP 269 «Convenience Factory Methods for Collections» — другой зверь: **структурно иммутабельный** список с собственным хранилищем, на которое снаружи нет ни одной ссылки.

```java
List<String> of = List.of("a", "b");

of.set(0, "z");          // ❌ UnsupportedOperationException — даже set
of.add("c");             // ❌ UnsupportedOperationException

List.of("a", null);      // ❌ NullPointerException — null запрещён при создании
List.of("a", "a");       // ✅ [a, a] — дубликаты в List легальны
Set.of("a", "a");        // ❌ IllegalArgumentException — а вот в Set нет
```

Три отличия от `Arrays.asList`, каждое проверено: запрещён даже `set` (не только структурные изменения); `null`-элементы отвергаются на создании; хранилище создаётся внутри и наружу не отдаётся — данные не изменятся _никогда_, а не «пока никто не трогает исходный массив».

Враждебность к `null` идёт дальше, чем кажется, — она распространяется и на **запросы**:

```java
List.of("a").contains(null);   // ❌ NullPointerException — не false!
```

Проверено на JDK 21: невинный `contains(null)` — например, внутри чужого кода, проверяющего «есть ли null в коллекции», — роняет программу. Javadoc `List.of` это честно оговаривает; на практике об этом узнают из стектрейса.

### `Collections.unmodifiableList` — только view

Третий инструмент решает другую задачу и даёт самую слабую гарантию:

```java
List<String> mutable = new ArrayList<>(List.of("a", "b"));
List<String> view = Collections.unmodifiableList(mutable);

view.add("c");            // ❌ UnsupportedOperationException
mutable.add("c");         // ✅ оригинал живой
System.out.println(view); // [a, b, c] — мутация протекла через view
```

`unmodifiableList` возвращает **представление** (`view`) поверх живого списка: обёртку, которая делегирует чтения и отвергает записи. Она защищает от изменений _через эту конкретную ссылку_ — и только. Backing-список продолжает жить своей жизнью, и всё, что с ним происходит, видно сквозь view. Гарантии «данные не изменятся» здесь нет вовсе — есть гарантия «ты не изменишь их вот этой ссылкой».

### Три уровня на одной лестнице

Сведём: три инструмента — три разных обещания.

![[Pasted image 20260702125803.png]]

Третья ступень — `defensive copy` — это не класс, а паттерн: принимая коллекцию в конструктор или возвращая её из геттера, ты отдаёшь/сохраняешь _копию_, чтобы внешний код не остался со ссылкой на твои внутренности. С Java 10 для этого есть штатный однострочник:

```java
public final class Order {
    private final List<String> items;

    public Order(List<String> items) {
        this.items = List.copyOf(items);   // копия при приёме: снаружи её не изменят
    }

    public List<String> items() {
        return items;                      // уже иммутабельный — отдаём как есть
    }
}
```

`List.copyOf(source)` делает иммутабельную копию, отвязанную от источника. С умной оптимизацией (проверено на JDK 21): если на вход пришёл список, _уже_ созданный через `List.of`/`copyOf`, метод возвращает **его же**, без копирования — иммутабельное копировать незачем. А вот `unmodifiableList`-view он за иммутабельный не признаёт (гарантий-то у view нет) и честно копирует.

> [!tip] 💡 Из C#
> лестница отображается почти один в один. `Collections.unmodifiableList` ≈ `list.AsReadOnly()` — та же view-обёртка (`ReadOnlyCollection<T>`) с мутабельным backing. `List.of` / `List.copyOf` ≈ `ImmutableList<T>` из `System.Collections.Immutable` — настоящая иммутабельность с собственным хранилищем. Разница культурная: в .NET иммутабельные коллекции — отдельный NuGet-пакет с builder-паттерном, в Java с 9-й версии они встроены в сами интерфейсы коллекций как статические фабрики.

Последний штрих границы: какой изменяемый `List` брать, когда нужна настоящая коллекция. Короткий ответ — `ArrayList`, это правильный дефолт в подавляющем большинстве случаев; `LinkedList` и `CopyOnWriteArrayList` существуют под узкие сценарии. Почему `ArrayList` почти всегда выигрывает и как он устроен внутри — глава 13.

> [!example] 🔬 Глубже / на синьора
> **Внутри `Arrays$ArrayList`.** Класс хранит переданный массив как единственное поле `private final E[] a` — ноль копирования, `size()` возвращает `a.length` навсегда. `set`/`get` реализованы напрямую поверх массива, а вот `add`/`remove` он _не реализует вообще_: они наследуются от `AbstractList`, чья дефолтная реализация — `throw new UnsupportedOperationException()`. То есть «fixed-size» — не специальная защита, а честное «я не стал переопределять».
> 
> **Внутри `ImmutableCollections`.** `List.of` возвращает специализации по размеру (проверено: `getClass()`): `List12` для 1–2 элементов — хранит их в двух полях `e0`/`e1`, массива нет вообще, экономия на заголовке; `ListN` — для остальных, с внутренним массивом. Обе — подклассы в `java.util.ImmutableCollections`, финальные, без единого мутирующего пути. `null`-враждебность зашита в конструкторы _и_ в методы запросов — отсюда `NPE` на `contains(null)` и `indexOf(null)`.
> 
> **Порядок в `Set.of` намеренно нестабилен.** Внутри `ImmutableCollections` есть поле `SALT`, инициализируемое случайно при старте JVM, — оно перемешивает порядок итерации `Set.of`/`Map.of` от запуска к запуску. Это дизайн-решение: чтобы никто не завязался на недокументированный порядок (у `List.of` порядок, разумеется, стабилен — он часть контракта списка).
> 
> **`toArray(new String[0])` против пред-размеренного.** Интуиция подсказывает передать `new String[list.size()]`, чтобы «сэкономить аллокацию». Бенчмарки Алексея Шипилёва («Arrays of Wisdom of the Ancients») показали обратное: вариант с пустым массивом стабильно быстрее или не хуже. Механизм: получив массив нулевой длины, метод сам создаёт результат нужного размера через `Arrays.copyOf` — JIT видит аллокацию и заполнение как одно целое и не зануляет память дважды; с пред-размеренным массивом JVM обязана сначала занулить его при создании, а потом перезаписать. Плюс в конкурентной среде `size()` между вызовом и заполнением может устареть. Идиома `new String[0]` — правильная и по скорости, и по корректности. С Java 11 есть ещё `list.toArray(String[]::new)` — та же семантика через ссылку на конструктор.
> 
> **`Stream.toList()` vs `Collectors.toList()`.** `stream.toList()` (Java 16, прямо в интерфейсе `Stream`) возвращает _неизменяемый_ список (проверено: `add` → `UnsupportedOperationException`), но, в отличие от `List.of`, **терпимый к `null`-элементам** (проверено: `[a, null]` создаётся спокойно) — это третья точка в пространстве гарантий. `Collectors.toList()` по спецификации не гарантирует вообще ничего — ни тип, ни мутабельность (фактически сегодня `ArrayList`, но контракт этого не обещает); когда нужна гарантированная мутабельность — `Collectors.toCollection(ArrayList::new)`.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`Arrays.asList(int[])` — список из одного элемента.** Varargs + запрет примитивов в дженериках = весь массив пакуется как единственный элемент, `List<int[]>` размера 1, ни ошибки, ни warning. Чинится боксингом: `Arrays.stream(nums).boxed().toList()`.
> 2. **Общий backing у `Arrays.asList`.** Мутации протекают в обе стороны: `list.set` меняет исходный массив, запись в массив видна списку. Отдал такой список наружу, продолжая держать массив, — отдал и право менять список.
> 3. **`contains(null)` на `List.of` — `NPE`, не `false`.** Null-враждебность распространяется на запросы, не только на создание. Код вида «проверим, нет ли null в пришедшей коллекции» падает на самой проверке. Особенно коварно, когда `List.of` создан глубоко в чужом слое.
> 4. **Unmodifiable ≠ immutable, и оба — неглубокие.** View протекает при мутации backing-списка — это разобрали. Но и `List.of` иммутабелен лишь _структурно_: `List.of(new StringBuilder("x"))` не защищает от `list.get(0).append("!")` — контейнер заморожен, элементы живые. Настоящая глубокая иммутабельность требует иммутабельных элементов.
> 5. **`(String[]) list.toArray()` — `ClassCastException`.** Возвращённый объект реально имеет класс `Object[]`, каст его не переделает. Только `toArray(new String[0])` или `toArray(String[]::new)`.
> 6. **`Collectors.toList()` ничего не обещает.** Полагаться на его мутабельность — ставка на деталь реализации; спецификация разрешает вернуть что угодно. Нужна мутабельность — `toCollection(ArrayList::new)`; нужна иммутабельность — `stream.toList()` или `Collectors.toUnmodifiableList()`.

### На собесе

> [!question]- Чем отличаются Arrays.asList, List.of и new ArrayList<>()? _(частота: очень высокая, уровень: mid; вопрос-визитка темы)_  
> **Модельный ответ:** `new ArrayList<>()` — полноценный мутабельный список со своим хранилищем. `Arrays.asList` — обёртка фиксированного размера поверх переданного массива: `set` можно, `add`/`remove` — `UnsupportedOperationException`, хранилище общее с массивом, мутации видны в обе стороны. `List.of` (Java 9, JEP 269) — структурно иммутабельный список с собственным недоступным снаружи хранилищем: запрещён даже `set`, `null` отвергается.
> 
> **EN:** `new ArrayList<>()` is a fully mutable list with its own storage. `Arrays.asList` is a fixed-size wrapper over the given array — `set` works, `add`/`remove` throw, and the backing array is shared both ways. `List.of` (Java 9, JEP 269) is structurally immutable with private storage: even `set` throws, and nulls are rejected.
> 
> → _Follow-up:_ «Как из `Arrays.asList` получить нормальный список?» — скопировать: `new ArrayList<>(Arrays.asList(...))`.
> 
> → _Типичный слабый ответ:_ «`Arrays.asList` иммутабельный» — нет, он полумутабельный: `set` работает и пишет в исходный массив. Ответ выдаёт, что человек ни разу не звал на нём `set`.

> [!question]- List.of vs Collections.unmodifiableList — в чём разница? _(частота: высокая, уровень: mid+)_  
> **Модельный ответ:** разные гарантии. `unmodifiableList` — view поверх живого списка: защищает от записи через эту ссылку, но backing мутируется снаружи, и изменения видны сквозь view. `List.of` — собственное хранилище без внешних ссылок: данные не изменятся никогда. Первое — «нельзя менять этой ссылкой», второе — «нельзя изменить вообще».
> 
> **EN:** Different guarantees. `unmodifiableList` is a view over a live list — it blocks writes through that reference, but the backing list can still be mutated externally and changes show through. `List.of` owns its storage with no outside references — the data can never change.
> 
> → _Follow-up:_ «Покажи, как сломать первое» — держим ссылку на оригинальный `ArrayList`, мутируем его, view показывает новое содержимое. С `List.of` такого вектора нет.
> 
> → _Follow-up 2:_ «А `List.copyOf`?» — иммутабельная копия, отвязанная от источника; если источник уже создан через `List.of`/`copyOf` — возвращает его же без копирования.

> [!question]- Что напечатает Arrays.asList(new int[]{1,2,3}).size()? _(частота: высокая, уровень: mid; ловушка-фаворит)_  
> **Модельный ответ:** `1`. Сигнатура — `asList(T...)`, а `T` не может быть примитивом, поэтому `int[]` не рассыпается на элементы — компилятор выводит `T = int[]` и упаковывает весь массив одним элементом: `List<int[]>`. С `Integer[]` было бы 3.
> 
> **EN:** It prints `1`. `asList` takes varargs `T...`, and `T` can't be a primitive — so the whole `int[]` becomes a single element, yielding a `List<int[]>` of size 1. With `Integer[]` you'd get 3.
> 
> → _Follow-up:_ «Как получить `[1, 2, 3]`?» — `Arrays.stream(nums).boxed().toList()` либо изначально `Integer[]`.

> [!question]- Почему list.toArray() возвращает Object[], а не типизированный массив? _(частота: средняя, уровень: mid+; мост к дженерикам)_  
> **Модельный ответ:** стирание типов — в рантайме список не знает свой `T`, создать `T[]` не из чего. Каст `(String[])` не поможет: реальный класс объекта — `Object[]`, будет `ClassCastException`. Типизированный массив получают через перегрузку с образцом `toArray(new String[0])`: массив reified и привозит тип элементов в рантайм.
> 
> **EN:** Type erasure — at runtime the list doesn't know its `T`, so it can't create a `T[]`. Casting fails with `ClassCastException` because the object's actual class is `Object[]`. Use `toArray(new String[0])`: the sample array is reified and carries the element type into runtime.
> 
> → _Follow-up (на синьора):_ «Пустой образец или пред-размеренный?» — пустой: быстрее (JIT не зануляет память дважды — бенчмарки Шипилёва) и корректнее при конкурентном изменении размера. С Java 11 — `toArray(String[]::new)`.

> [!question]- List.of("a").contains(null) — что вернёт? _(частота: ниже средней, уровень: mid+→senior; отделяет читавших Javadoc)_  
> **Модельный ответ:** не `false`, а `NullPointerException`. Null-враждебность `List.of` зашита не только в создание, но и в методы запросов (`contains`, `indexOf`) — так специфицировано в Javadoc. Практическое следствие: универсальный код, проверяющий коллекции на `null`-элементы, падает на иммутабельных фабричных списках.
> 
> **EN:** It throws `NullPointerException`, not `false`. The null-hostility of `List.of` covers query methods too — `contains(null)` and `indexOf(null)` throw, per the Javadoc. Generic code that probes collections for nulls will blow up on factory lists.
> 
> → _Follow-up:_ «Где ещё в этом семействе сюрприз?» — `Set.of` с дубликатами кидает `IllegalArgumentException` на создании; порядок итерации `Set.of` намеренно рандомизирован между запусками JVM.

> [!question]- (бонус, на синьора) Stream.toList() vs Collectors.toList() — что выбирать? _(частота: растёт, уровень: mid+→senior)_  
> **Модельный ответ:** `stream.toList()` (Java 16) возвращает неизменяемый список, при этом терпимый к `null`-элементам — в отличие от `List.of`. `Collectors.toList()` по контракту не гарантирует ни тип, ни мутабельность возвращённого списка — сегодня это `ArrayList`, но это деталь реализации. Нужна гарантированная мутабельность — `Collectors.toCollection(ArrayList::new)`; иммутабельность — `toList()` или `Collectors.toUnmodifiableList()`.
> 
> **EN:** `stream.toList()` (Java 16) returns an unmodifiable, null-tolerant list. `Collectors.toList()` guarantees nothing about type or mutability by spec — it happens to be an `ArrayList` today. Need guaranteed mutability — `Collectors.toCollection(ArrayList::new)`; immutability — `toList()` or `Collectors.toUnmodifiableList()`.
> 
> → _Follow-up:_ «Чем его иммутабельный список отличается от `List.of`?» — допускает `null`-элементы; это третья точка в пространстве гарантий: неизменяемый, но не null-враждебный.
