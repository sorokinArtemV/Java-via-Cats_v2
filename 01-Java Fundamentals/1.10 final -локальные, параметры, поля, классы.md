`final` — одно из самых перегруженных слов в языке: его можно поставить примерно в десяти разных местах, и почти в каждом оно значит чуть-чуть другое. За ним тянутся, когда хотят зафиксировать значение — защитить поле от подмены, объявить константу, запретить наследование. Практическая польза тройная: читаемость (видно, что переменная привязана к одному значению), корректность (лямбды и immutable-объекты стоят на нём) и потокобезопасность (`final`-поле безопасно публикуется между потоками). А главное, что нужно уложить в голову сразу: `final` фиксирует **ячейку**, а не **объект** в ней — на этом зазоре между именем и механикой держится половина собеседных вопросов по теме.

### Ментальная модель

`final` = «эту ячейку присвоить можно ровно один раз». Ячейка — это переменная, поле или параметр; что лежит по ссылке (объект в heap), `final` вообще не трогает.

### Ссылка vs объект — фундамент

Разведём две сущности, которые слово «final» в голове склеивает. **Ячейка** хранит значение: для примитива это само число, для reference-типа — ссылку (адрес в куче). **Объект** — то, что лежит в heap. `final` запрещает менять ячейку и молчит про объект:

```java
final List<String> list = new ArrayList<>();
list.add("hello");        // ✅ меняем объект в heap
list = new ArrayList<>(); // ❌ error: cannot assign a value to final variable list
```

Я это прогнал — `list.add(...)` спокойно доводит размер до нужного, а строка с переприсваиванием не компилируется. Для примитива разделения нет: `final int x = 5` — значение и есть содержимое ячейки, так что тут `final` просто «нельзя переприсвоить», и всё совпадает с интуицией.

Тот же капкан с массивами: `final int[] arr = {1,2,3}` — `arr[0] = 99` проходит (проверено, печатает `99`), а `arr = new int[5]` нет. Массив — это объект, его содержимое не под защитой.

> 💡 **Из C#:** это ровно `readonly`. `readonly List<string> list` — переприсвоить поле нельзя, а `list.Add(...)` работает. Семантика для reference-типов один в один, интуицию из C# переносишь без правок — Java тут не придумала ничего нового, просто слово другое.

### `final` для локальных переменных и параметров

Здесь `final` значит: переменная инициализируется **ровно один раз** и больше не присваивается.

```java
void process(final String input) {  // параметр переприсвоить нельзя
    input = input.trim();           // ❌ error: final parameter input may not be assigned
    final int size;                 // объявили без значения
    size = input.length();          // ✅ первое и единственное присваивание
    size = 0;                       // ❌ error: cannot assign a value to final variable size
}
```

Два термина, которые надо назвать явно. **Blank final** — это `final`-переменная (локальная или поле), объявленная **без** инициализатора. `final int size;` выше — blank final: присвоить можно позже, но строго один раз и до первого чтения. **Definite assignment** (определённое присваивание, JLS §16) — правило компилятора: на каждом пути выполнения до момента чтения переменная должна быть гарантированно присвоена, а для `final` — присвоена **не более** одного раза. Компилятор доказывает это статически:

```java
final int x;
if (cond) { x = 1; }        // если cond == false, x не присвоен
System.out.println(x);      // ❌ error: variable x might not have been initialized
```

Прогнал — сообщение дословно `variable x might not have been initialized`. Чтобы прошло, присвой на **всех** ветках (`else { x = 2; }`). Практический смысл `final` на локалках — не производительность (JIT и так всё видит), а читаемость и защита от случайной переинициализации: это документация в коде «переменная привязана к одному значению, не ищи, где она меняется».

> 💡 **Из C#:** прямого аналога `final` на локальной переменной или параметре в C# нет (`in`-параметры — про другое, про передачу по readonly-ссылке). Конструкция для тебя новая по форме, хоть и тривиальная по смыслу.

### Effectively final и почему Java копирует, а не делит

С Java 8 появился термин **effectively final** (фактически финальная): переменная, которую ты **не** пометил `final`, но которая после инициализации ни разу не меняется. Компилятор вычисляет это сам. Понадобилось оно из-за **лямбд и анонимных классов** — они могут захватывать (capture) локальные переменные окружающего метода, но только `final` или effectively final:

```java
int counter = 0;
Runnable r = () -> System.out.println(counter);
counter++;   // ❌ local variables referenced from a lambda expression must be final or effectively final
```

Дословно этот текст ошибки я и получил. Теперь — **почему** такое ограничение, и вот здесь принципиальное расхождение с C#.

Когда лямбда в Java захватывает локальную переменную, она захватывает её **по значению**: компилятор копирует текущее значение в скрытое поле объекта-лямбды. Логика простая — локальная переменная живёт на стеке метода, а метод может завершиться раньше, чем выполнится лямбда (её отдали в другой поток, в коллбэк, в `executor.submit`). Стек уже размотан, оригинальной ячейки нет — поэтому Java копирует значение в кучу вместе с лямбдой. А раз это копия — что будет, если разрешить менять оригинал? Появятся две независимые сущности: твой `counter` на стеке и копия внутри лямбды. Поменял одну — другая не изменилась, источник тонких багов. Поэтому Java просто запрещает захватывать то, что меняется.

C# решает ту же проблему **наоборот**: компилятор при захвате «поднимает» (hoisting) локальную переменную в скрытый объект-замыкание, и внешний код, и лямбда работают с **одним и тем же** полем этого объекта. Захват по ссылке — изменения видны с обеих сторон, поэтому C# не требует никакого `final`/effectively final.

> ⚠️ **Подводный камень — цикл:** классический вопрос. В C-style цикле переменная меняется, значит не effectively final:
> 
> ```java
> for (int i = 0; i < 3; i++) {
>     tasks.add(() -> System.out.println(i)); // ❌ i не effectively final
> }
> ```
> 
> А в enhanced-for **каждая итерация** создаёт новую переменную — она effectively final, и это компилируется (проверил оба случая):
> 
> ```java
> for (String name : List.of("a", "b", "c")) {
>     tasks.add(() -> System.out.println(name)); // ✅ name eff. final на каждой итерации
> }
> ```
> 
> Обход для C-style — скопировать в локальную: `int captured = i;` и захватывать `captured`.

### Карта всех позиций `final` — одно слово, десять мест

Прежде чем нырять в поля и классы, полезно разом увидеть, куда `final` вообще легально ставится. В большинстве позиций смысл один — «присвоить один раз», — но есть места, где он навязан **неявно**, и это ловушки.

Явные позиции (ставишь ты): локальная переменная, параметр метода, переменная for-each, ресурс в try-with-resources, параметр catch, параметр лямбды, instance-поле, static-поле, класс, метод.

```java
for (final String s : list) { ... }                    // for-each
try (final var r = Files.newBufferedReader(path)) { }  // ресурс
catch (final IOException e) { ... }                    // catch-параметр
Runnable r = (final var x) -> use(x);                  // параметр лямбды (var в лямбде — Java 11+)
```

Неявные позиции (навязаны компилятором) — вот где спотыкаются:

**Ресурс в try-with-resources и так неявно final.** Переприсвоить `r` внутри блока нельзя даже без слова `final` (`error: auto-closeable resource r may not be assigned`). С Java 9 можно вообще передать уже существующую effectively-final переменную: `try (r) { ... }` (проверил — работает).

**Параметр multi-catch неявно final.** В `catch (IOException | SQLException e)` присвоить `e` нельзя — `error: multi-catch parameter e may not be assigned`. Причина: тип `e` — общий супертип, переприсваивание сломало бы типизацию. В одиночном `catch (IOException e)` присвоить можно (если не пометил `final`).

**Компоненты `record` — всегда final.**

```java
record Point(int x, int y) {}   // x и y — неявно private final
```

Record даёт immutable data-класс из коробки: все компоненты `final` + сгенерированные `equals`/`hashCode`/`toString`. Но та же ловушка — если компонент это `List`, сам record не защищает содержимое, нужен defensive copy в compact-конструкторе (детали — глава 17).

**Enum неявно final… но не всегда.** Контринтуитивный момент, который я проверил на JDK 21:

```java
enum Suit { CLUBS, HEARTS }                        // Suit — неявно final: true
enum Op {
    PLUS  { int apply(int a,int b){ return a+b; } },  // тело → анонимный подкласс
    MINUS { int apply(int a,int b){ return a-b; } };
    abstract int apply(int a,int b);
}                                                  // Op — final: FALSE
```

Как только у хотя бы одной константы появляется тело (constant-specific body), enum **перестаёт быть final** — такая константа становится анонимным подклассом (JLS §8.9). `Op.PLUS.getClass()` вернул `Op$1`, а не `Op`; `Suit.CLUBS.getClass()` вернул сам `Suit`. Это сюрприз при рефлексии и в `switch`.

### `final`-поля и безопасная публикация

Поле-`final` инициализируется в одном из трёх мест: в декларации, в конструкторе, или в instance-инициализаторе `{ ... }`. После конструктора — заморожено. И тут неочевидная, но важная гарантия из модели памяти.

Введу термин: **безопасная публикация** (safe publication) — это когда другой поток, увидев ссылку на объект, гарантированно видит его **полностью и корректно проинициализированным**, а не «наполовину собранным». С Java 5 (модель памяти JSR-133, закреплено в JLS §17.5 «final Field Semantics») действует правило: если объект **корректно сконструирован** — ссылка на него не «утекла» наружу до конца конструктора, — то любой поток, который потом увидит объект, гарантированно увидит правильные значения его `final`-полей **без всякой синхронизации**. Ни `synchronized`, ни `volatile` — сам факт `final` даёт барьер видимости.

Зачем практически: именно поэтому **immutable-объекты с `final`-полями безопасно шарить между потоками без блокировок**. Это фундамент идеи «immutable = thread-safe by default». Без `final` рискуешь, что другой поток увидит поле с дефолтом (`0`/`null`) вместо реального значения. Глубже в модель памяти не лезем — это глава 23; запомни связку: **`final`-поле → безопасная публикация → immutable-объект безопасен в многопоточке**.

> ⚠️ **Подводный камень:** гарантия рушится, если ссылка утекла из конструктора раньше времени — например, `SomeRegistry.register(this)` в середине конструктора, когда `final`-поля ещё не присвоены. Правило: не публикуй `this` из конструктора.

### `final` ≠ immutable: как это делает `String`

Самое частое заблуждение, его на собесе проверяют в лоб. **`final` сам по себе не делает объект неизменяемым** — мы это видели на `final List` в самом начале. Immutable-класс — это набор условий, а не один модификатор: все поля `private final`; нет сеттеров и любых мутирующих методов; для **mutable**-полей (коллекции, массивы, `Date`) — defensive copy и на входе, и на выходе; желательно сам класс `final`.

Вот почему `final`-поля мало, если поле — изменяемая коллекция:

```java
public final class Allowlist {
    private final List<String> hosts;        // ссылка final, но список сам mutable

    public Allowlist(List<String> hosts) {
        this.hosts = List.copyOf(hosts);     // defensive copy на входе
    }
    public List<String> getHosts() {
        return hosts;                        // List.copyOf вернул immutable — отдавать копию не нужно
    }
}
```

Без `List.copyOf` в конструкторе твой `final` бесполезен: снаружи остаётся ссылка на тот же `ArrayList`, и `originalList.add(...)` пробьёт «иммутабельность» насквозь. Я проверил `List.copyOf`: он и делает снимок (после копирования исходный `src.add(4)` в копии не виден), и возвращает immutable-список — `copy.add(...)` кидает `UnsupportedOperationException`.

Как это сделано в `String` (проверил рефлексией на JDK 21): `String` is final: `true`; поля — `private final byte[] value` и `private final byte coder`. Три слоя защиты: класс `final` (нельзя подсунуть мутирующий наследник), массив `private final` (снаружи не дотянуться и ссылку не переприсвоить), ноль мутаторов (`substring`/`replace` возвращают новый `String`). Массив инкапсулирован — наружу не отдаётся, поэтому defensive copy не нужен. Кстати, поле стало `byte[]` (а не `char[]`) с Java 9 — это **JEP 254 Compact Strings**: латиница хранится в один байт вместо двух (детали — глава 14).

> 💡 **Из C#:** прямого immutable-паттерна как идиомы в C# меньше (есть `record` с `init`-сеттерами, `readonly struct`), но `string` в C# точно так же sealed + неизменяемый. И в Java 16+ `record` даёт все поля `final` из коробки — короткий путь к immutable data-классу.

### Когда `final`-поле реально получает значение

Кажется, что `final int x = 42` значит «`x` всегда `42`». Но во время **конструирования** объекта есть окно, где `final`-поле держит дефолт, и его можно увидеть, если вызвать переопределяемый метод из конструктора родителя.

```java
class Base { Base() { onInit(); }  void onInit() {} }

class Sub extends Base {
    final int x;
    Sub(int v) { x = v; }
    @Override void onInit() { System.out.println("x = " + x); }  // печатает 0, а не v!
}
// new Sub(42) → выведет "x = 0"
```

Прогнал — печатает `x = 0`. Порядок конструирования (JLS §12.5): сначала отрабатывает `super()` (то есть `Base()`), `Base()` вызывает `onInit()`, переопределённый в `Sub`, — но инициализаторы полей `Sub` ещё не выполнились, они идут **после** возврата из `super()`. В этот момент `x` физически `0`.

![[Pasted image 20260702170652.png]]

> 🔬 **Глубже — почему в исходной версии этого примера печаталось `42`.** Если написать `final int x = 42` (константа прямо в декларации), пример **сломается и напечатает `42`**, хотя окно с нулём никуда не делось. Причина: `final int x = 42` — это _constant variable_ (JLS §4.12.4), и `javac` инлайнит `42` прямо в место чтения. Я посмотрел байткод: `System.out.println("x = " + x)` свернулось в `ldc "x = 42"` — там уже нет обращения к полю, там литерал. То есть пример читает не поле, а вшитую константу. Чтобы окно с нулём было видно, поле **не должно быть константой**: blank final из конструктора (`x = v`) или `= compute()`. Это тот же механизм constant-variable, который сейчас всплывёт в reflection.

### `final` класс и `final` метод

`final class` — нельзя наследовать. `final` метод — нельзя переопределить.

```java
public final class Money { /* extends Money → ошибка */ }
public class Base { public final void criticalLogic() { } }  // наследник не переопределит
```

Зачем — две причины, и вторая слабее, чем думают.

**Защита инвариантов.** Если класс immutable, но не `final`, невнимательный наследник может добавить mutable-состояние или переопределить метод так, что сломает контракт. `final class` это закрывает. Это главная, дизайновая причина.

> 🔬 **Глубже — миф о производительности.** Введу термин: **девиртуализация** (devirtualization) — когда JIT, зная, что у метода нет переопределений, превращает виртуальный вызов (через таблицу методов) в прямой и дальше **инлайнит** его. Кажется, `final` тут помогает — переопределений быть не может. Но на практике выигрыш почти всегда нулевой: HotSpot девиртуализирует и **без** `final`, опираясь на анализ иерархии классов в рантайме (Class Hierarchy Analysis, CHA) — если на этапе исполнения класс по факту единственный, вызов девиртуализируется; появится наследник — JIT сделает деоптимизацию и перекомпилирует. Так что `final` на методе ставят ради дизайна (запрет override, защита инвариантов), а не ради микросекунд.

> 💡 **Из C#:** `final class` = `sealed class`. С методом — внимание на разворот: в C# методы по умолчанию **не** виртуальны, виртуальность включается через `virtual`, а `sealed` закрывает уже виртуальный метод. В Java **наоборот** — все нестатические методы виртуальны по умолчанию, а `final` их «запечатывает». Так что `final` метод в Java по эффекту ближе к «убрать `virtual`», а не к C#-шному `sealed override`.

> ⚠️ **Подводный камень:** `final abstract` — ошибка компиляции (`illegal combination of modifiers: abstract and final`): `abstract` требует переопределения, `final` его запрещает. А `private final` метод — `final` избыточен: `private`-методы и так не наследуются (IDE подсветит как redundant; `javac` молчит даже с `-Xlint:all` — проверял).

### 🔬 `final` + reflection + compile-time constant

Вопрос «можно ли изменить `final`-поле через reflection» интервьюер задаёт, чтобы проверить, понимаешь ли ты разницу между уровнями. Правильный ответ — **зависит от того, какое поле**. Я прогнал все случаи на JDK 21:

**Instance (нестатическое) `final`-поле — да, можно** (если модульная система не запретит доступ): `setAccessible(true)` + `set(...)` меняет значение слота. По Javadoc `Field.set`, запись в `final` разрешена только когда `setAccessible(true)` прошёл, поле **нестатическое**, класс не hidden и **не `record`**.

**Static `final` — нет.** `Field.set` на статическом `final`-поле кидает `IllegalAccessException` — на JDK 21 это верно для **любого** статического финала, и для примитивной константы, и для вычисляемого объекта (`static final Integer X = valueOf(...)`). Тут я поправил формулировку сырья: настраивается только instance-поле.

**`record`-поля — нет** (с Java 16): `IllegalAccessException`.

И главный гвоздь — **compile-time constant** (константа времени компиляции). Это `static final` (или даже instance final) поле примитива или `String`, инициализированное **константным выражением** — JLS зовёт такие _constant variable_ (§4.12.4). Компилятор **инлайнит** их: подставляет значение прямо в байткод во всех местах использования (JLS §13.4.9). Я проверил это точечно на instance-константе:

```java
class Box { final int value = 42; }   // constant variable
// после f.set(box, 99) через reflection:
box.value      // → 42   (в байткоде это литерал, не чтение поля)
f.get(box)     // → 99   (рефлексия читает реальный слот)
```

Прямое чтение `box.value` даёт `42`, рефлексивное `f.get(box)` — `99`. Reflection меняет слот, но **уже заинлайненные чтения** остаются со старым литералом — reflection их «не догоняет».

> ⚠️ **Подводный камень билда (binary compatibility):** из инлайна следует знаменитая засада. Поменял `public static final int VERSION = 1;` в библиотеке, пересобрал **только её** — а зависимый код всё ещё помнит `1`, зашитое в его байткод. Я это воспроизвёл: `App` печатал `VERSION=1` после пересборки только `Lib` (со значением `2`), и `2` появилось лишь после пересборки самого `App`. Поэтому константы, которые могут меняться между релизами, лучше отдавать через метод-геттер, а не через `public static final`.

> 💡 **Из C#:** `static final` compile-time constant ≈ C#-шный `const` (тоже инлайнится в потребителя, с той же binary-compat проблемой). А `static final` с **не**-константным инициализатором (`static final Logger LOG = LoggerFactory.getLogger(...)`) — ближе к `static readonly`: не инлайнится, вычисляется в рантайме при загрузке класса. Ключевое отличие: в C# ты **выбираешь** `const` vs `readonly` ключевым словом; в Java пишешь `final` в обоих случаях, а классифицирует компилятор по виду инициализатора.

> ⚠️ Старый трюк «убрать `final` через рефлексию над полем `Field.modifiers`» в современных JDK **мёртв**: я запросил `Field.class.getDeclaredField("modifiers")` — `NoSuchFieldException`, поле отфильтровано из рефлексии. Остаётся только грязь через `VarHandle`/`Unsafe`, что само по себе сигнал «делаешь что-то не то».

### ⚠️ Сводка подводных камней

1. `final` защищает ссылку, не объект — `final List` мутируется через `.add()`, `final int[]` через `arr[0]=x`.
2. Blank final должен быть присвоен на **всех** путях и ровно один раз.
3. `final`-поле держит дефолт (`0`/`null`) во время конструирования → не зови переопределяемые методы из конструктора.
4. …но если поле — constant variable (`final int x = 42`), чтение инлайнится, и «окно с нулём» этот конкретный пример не покажет (напечатает `42`).
5. Правило «effectively final» — только про локалки и параметры. **Поля** в лямбде менять можно: лямбда захватывает `this`, а через `this.count` поле мутируется (проверил — `() -> count++` компилируется и работает).
6. C-style `for` ломает захват (`i` меняется), enhanced-for — нет (новая ячейка на итерацию).
7. `static final` примитив/`String` с константой инлайнится → binary-compat ловушка при смене значения без пересборки зависимых.
8. Reflection: instance `final` — можно; `static final` (любой) — `IllegalAccessException`; `record`-поля — нельзя.
9. `final abstract` — ошибка компиляции; `private final` метод — избыточно.
10. Ресурс в try-with-resources и параметр multi-catch — неявно final.
11. Enum неявно final, **кроме** случая с constant-specific body (тогда `getClass()` → `Op$1`).
12. `record`-компоненты неявно final, но mutable содержимое всё равно требует defensive copy.

### На собесе

**Q1 (частый). Что напечатает?**

```java
final List<Integer> list = new ArrayList<>(List.of(1, 2, 3));
list.add(4);
System.out.println(list.size());
```

→ `4`. `final` запрещает переприсвоить `list`, но `add` меняет объект в heap. **Follow-up:** «Как сделать, чтобы `add` бросал исключение?» → `List.copyOf(...)`/`List.of(...)` дают immutable-список, `add` кинет `UnsupportedOperationException`. **Trap:** «А `final` тут гарантирует потокобезопасность?» → нет; `final` на ссылке даёт видимость самой ссылки при безопасной публикации, но `ArrayList` внутри не потокобезопасен.

- RU: «`final` — это про ячейку, не про объект. `.add()` меняет объект, поэтому размер 4. Иммутабельность даёт `List.copyOf`, а не `final`.»
- EN: “`final` locks the reference, not the object. `.add()` mutates the object, so size is 4. Immutability comes from `List.copyOf`, not from `final`.”

**Q2 (частый). Почему не компилируется и как починить?**

```java
for (int i = 0; i < 3; i++) executor.submit(() -> System.out.println(i));
```

→ `i` меняется в цикле, не effectively final, лямбда не может захватить. **Починка:** `int captured = i;` внутри тела. **Follow-up:** «Почему в `for (String s : list)` проблемы нет?» → в enhanced-for переменная создаётся заново каждую итерацию. **Trap:** «В C# тот же код работает — почему?» → C# захватывает по ссылке через hoisting в closure-объект; до C# 5 это как раз давало багу с `foreach` (все лямбды видели последнее значение), её потом специально пофиксили.

- RU: «Java захватывает локалки по значению (копией), поэтому меняющуюся переменную захватить нельзя. Копируем в отдельную локальную.»
- EN: “Java captures locals by value (a copy), so a mutating variable can't be captured. Copy it into a separate local.”

**Q3 (частый). `final` делает объект неизменяемым?**  
→ Нет — только запрещает переприсвоить переменную/поле. **Follow-up:** «Что нужно для immutable-класса?» → `private final` поля + нет сеттеров + defensive copy для mutable-полей + `final` класс. **Follow-up:** «Где именно `final` поля мало?» → если поле `List`, без `List.copyOf` в конструкторе внешняя ссылка позволит мутировать его снаружи.

- RU: «`final` ≠ immutable. Immutable — это набор условий; `final` только один из них.»
- EN: “`final` ≠ immutable. Immutability is a set of conditions; `final` is just one of them.”

**Q4 (mid+/senior). Можно ли изменить `final`-поле через reflection?**  
→ Instance `final` — да (`setAccessible(true)` + `set`); `static final` — нет (`IllegalAccessException`); `record`-поля — нет (Java 16+). **Follow-up:** «Если поменял instance final — везде увижу новое значение?» → для compile-time constant (`static final` или instance `final` примитив/`String` с константой) — нет, оно заинлайнено в байткод потребителей, reflection не догонит. **Follow-up:** «Почему это плохая идея?» → JIT мог оптимизировать в расчёте на неизменность; поведение становится непредсказуемым.

- RU: «Instance — да, static — нет. Но константу reflection не догонит: она вшита литералом в байткод в местах использования.»
- EN: “Instance yes, static no. But a compile-time constant won't budge: it's inlined as a literal into consumers' bytecode.”

**Q5 (частый). Зачем `String` сделали `final`-классом?**  
→ Гарантировать неизменяемость и безопасность инвариантов: наследник мог бы добавить мутирующее поведение и сломать предположения (строки — в проверках доступа, ключах `HashMap`, путях файлов). Плюс это включает безопасную публикацию и кеширование `hashCode`. **Follow-up:** «`hashCode` у `String` кешируется?» → да, в immutable-объекте это безопасно, считается лениво один раз. **Follow-up:** «Связано с String pool?» → да, неизменяемость — необходимое условие, чтобы безопасно шарить один объект-строку между многими ссылками в пуле.

- RU: «`final` + immutable → объект безопасно шарить между потоками без синхронизации и кешировать `hashCode`; для строк это ещё и security.»
- EN: “`final` + immutable → the object is safe to share across threads without synchronization and to cache `hashCode`; for strings it's also a security property.”

**Q6 (trap). `final` метод быстрее обычного?**  
→ На практике почти никогда. HotSpot девиртуализирует и инлайнит и без `final` (CHA в рантайме), появится наследник — деоптимизирует. `final` ставят ради дизайна, не ради скорости. **Follow-up:** «Тогда зачем `final` методы?» → контракт «этот метод нельзя менять в наследниках» — template method, критичная логика.

- RU: «Perf-аргумент слабый: девиртуализация работает и без `final`. `final` — это про дизайн и запрет override.»
- EN: “The perf argument is weak: devirtualization works without `final`. `final` is about design and forbidding override.”
