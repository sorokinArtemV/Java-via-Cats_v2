Правила видимости в Java выглядят «как везде», пока не упрёшься в две конкретные асимметрии, которые компилятор трактует строго и которые любят на собеседовании. Первая: локальную переменную нельзя прочитать до присваивания — это ошибка компиляции, а поле с тем же «непроинициализированным» состоянием читается и отдаёт дефолт. Вторая: `this` внутри лямбды и внутри анонимного класса указывает на _разные_ объекты, и от этого зависит, что напечатает `this.getClass()`. Обе выводятся из механики, а не заучиваются — этим и займёмся.

**Ментальная модель.** Область видимости локальной переменной — от объявления до конца охватывающего блока `{}`; переобъявить её в перекрывающемся scope нельзя, а затенить поле — можно. Локальную нельзя читать, пока компилятор не докажет присваивание по всем путям (definite assignment); поле такого требования не имеет, потому что память объекта зануляется при аллокации. `this` — получатель текущего вызова; лямбда своего `this` не вводит, анонимный класс вводит.

### Block scope: re-declaration против shadowing

Область видимости локальной переменной (JLS §6.3) — от точки объявления до конца охватывающего блока. За скобкой её нет. Внутри перекрывающегося scope переобъявить переменную нельзя:

```java
int x = 1;
{
    int x = 2;   // error: variable x is already defined in method main
}
```

Но локальная переменная **может затенить поле** — это shadowing, и он разрешён (проверено):

```java
class Counter {
    int value = 1;
    void m() {
        int value = 2;                   // OK: локальная затеняет поле
        System.out.println(value);       // 2  — локальная
        System.out.println(this.value);  // 1  — поле через this
    }
}
```

Различай термины: re-declaration (локальная поверх локальной в общем scope) — запрещён; shadowing (локальная поверх поля) — разрешён, доступ к полю через `this.`. Отдельные, не пересекающиеся блоки могут переиспользовать имя свободно — два `for (int i …)` подряд не конфликтуют, потому что их scope не перекрываются.

> 💡 **Из C#:** здесь ощутимое расхождение. C# запрещает _и_ shadowing поля локальной переменной в некоторых конфигурациях мягче, но главное отличие в другом: в C# нельзя объявить локальную, если имя уже занято переменной из объемлющего scope, даже в _непересекающихся_ дочерних блоках одного метода (правило «declaration space»). Java строже привязана именно к перекрытию scope: непересекающиеся блоки одного метода спокойно переиспользуют имя. Интуиция из .NET «раз я уже использовал `i` в этом методе, второй раз нельзя» — в Java неверна.

### Definite assignment: почему локальная и поле ведут себя по-разному

Термин: **definite assignment** (JLS §16) — правило, по которому локальную переменную нельзя _прочитать_, пока компилятор не докажет, что ей присвоено значение по всем путям выполнения. Проверено:

```java
int local;
System.out.println(local);   // error: variable local might not have been initialized
```

А поле читается без явного присваивания и даёт **значение по умолчанию** (`0`/`false`/`null`):

```java
class Box {
    int field;                                   // не инициализировано явно
    void show() { System.out.println(field); }   // печатает 0
}
```

Причина разницы — где живёт память. Память под объект в куче зануляется при аллокации, поэтому у полей есть гарантированный дефолт. Локальные живут в кадре стека, где гарантированного нуля нет; Java требует доказуемого присваивания и ловит чтение мусора ещё на компиляции. Это сознательный дизайн: потенциальный баг превращается в compile error.

Ключевой нюанс: definite assignment — это **поток-анализ**, а не «присвоено ли где-то выше по тексту». Оба варианта ниже прогнаны:

```java
int n;
if (cond) n = 1; else n = 2;
System.out.println(n);   // OK: присвоено на всех путях
```

Убери `else` — и `return n;`/`print(n)` дадут `error: variable n might not have been initialized`, потому что появляется путь, где `n` не присвоена. Компилятор рассуждает по ветвям управления, а не по порядку строк.

> 💡 **Из C#:** definite assignment есть и в C# — там он тоже compile error (`Use of unassigned local variable`), и логика та же. Отличие в полях: в C# для полей `struct` есть свои правила инициализации, но для полей `class` поведение совпадает с Java — дефолт при аллокации. Так что здесь C# — не «пол», а точный аналог; переносится один в один.

#### 🔬 Глубже / на синьера: blank final и flow scoping паттернов

Definite assignment — не только про «прочитать». На нём же держится **blank final** — поле или локальная, объявленные `final` без инициализатора: компилятор через тот же поток-анализ проверяет, что присваивание происходит ровно один раз по всем путям (детали `final` — урок 1.10). Тот же механизм.

Отдельный современный кейс — **flow scoping** переменной паттерна (`instanceof String s`, JEP 394, Java 16): область видимости `s` определяется не фигурными скобками, а потоком управления — `s` видна только там, где компилятор доказал, что `instanceof` истинен. Это definite assignment, распространённый на _область видимости_, а не только на «прочитать/нельзя». Пример прогнан: внутри `if (o instanceof String s)` переменная `s` в scope, за `if` — нет. Разбор pattern matching — глава 17; здесь важна связь: и flow scoping, и blank final — проявления одного поток-анализа.

### `this`: получатель, qualified this, static

`this` — ссылка на текущий экземпляр (получатель вызова). Обязателен при shadowing, когда параметр прячет поле:

```java
class Point {
    int x;
    Point(int x) {
        this.x = x;   // без this.: x = x присвоит параметр сам себе, поле останется 0
    }
}
```

`this(...)` — вызов другого конструктора этого же класса (constructor chaining); прогон подтверждает делегацию (`no-arg` вызывает `this(42)`, поле становится `42`). До Java 21 включительно `this(...)`/`super(...)` обязаны быть **первой** инструкцией конструктора — иначе `error: call to this must be first statement in constructor` (проверено). Глубокий разбор конструкторов и порядка инициализации — глава 3/4; здесь только обозначаю правило.

**Qualified this — `Outer.this`.** Во вложенном (non-static) классе `this` означает экземпляр _внутреннего_ класса. Чтобы достать охватывающий объект, пишут `Outer.this` (прогнано):

```java
class Outer {
    int x = 1;
    class Inner {
        int x = 2;
        int innerX() { return this.x; }        // 2 — поле Inner
        int outerX() { return Outer.this.x; }  // 1 — поле Outer
    }
}
```

`this` **не существует в static-контексте** — у статического метода нет получателя, использование `this` там это `error: non-static variable this cannot be referenced from a static context` (проверено).

> 💡 **Из C#:** `Outer.this` — то, чему в C# нет чистого аналога. Вложенные классы в C# по умолчанию _не_ держат ссылку на экземпляр внешнего класса (они как Java `static nested`), поэтому квалифицировать нечего. Java `inner` (non-static) класс, наоборот, несёт неявную ссылку на охватывающий экземпляр — отсюда и синтаксис `Outer.this`, и целый класс утечек памяти через эту ссылку. Разбор inner-классов и утечки — глава 3.12.

### ⚠️ `this` в лямбде против анонимного класса

Это самый частый «что напечатает» на тему `this`. Лямбда **не вводит свой `this`** — внутри неё `this` указывает на охватывающий экземпляр. В анонимном классе наоборот — `this` это сам анонимный объект. Прогон даёт наглядный результат:

```java
class ThisLambda {
    void run() {
        Runnable viaAnon = new Runnable() {
            public void run() { System.out.println(this.getClass().getName()); }
        };
        Runnable viaLambda = () -> System.out.println(this.getClass().getName());
        viaAnon.run();     // ThisLambda$1   ← сам анонимный объект
        viaLambda.run();   // ThisLambda     ← охватывающий экземпляр
    }
}
```

`ThisLambda$1` — это сгенерированное имя анонимного класса; `this` внутри него — новый объект. В лямбде `this` — тот же `ThisLambda`, в чьём методе лямбда написана. Механически: лямбда не создаёт нового scope для `this`/`super`, а анонимный класс создаёт (это отдельный объект отдельного класса). Отсюда практический вывод: если внутри анонимного `Runnable` нужен _охватывающий_ объект, пишут `ThisLambda.this`; в лямбде достаточно голого `this`.

> 💡 **Из C#:** прямого расхождения нет — C# лямбды/анонимные делегаты тоже захватывают `this` охватывающего типа, и там нет «анонимного класса» как отдельной сущности со своим `this`. Особенность именно в том, что в Java лямбда и анонимный класс _выглядят_ взаимозаменяемо, но по `this` ведут себя по-разному — а в C# такой пары «почти одно и то же, но с разным `this`» просто нет.

### ⚠️ Подводные камни

**Захват в лямбде требует effectively final.** Лямбда захватывает локальную переменную только если та `final` или effectively final (не переприсваивается после инициализации). Иначе — `error: local variables referenced from a lambda expression must be final or effectively final` (проверено). Это следствие того же pass-by-value: захватывается _значение_ переменной на момент создания лямбды, и Java запрещает захват изменяемой локальной, чтобы не создавать иллюзию общей ячейки. Разбор захвата и отличий от C#-замыканий (которые захватывают саму переменную, а не значение) — глава 9.5.

**`x = x` без `this` — тихий баг.** В конструкторе `Point(int x) { x = x; }` без `this.` присваивает параметр сам себе; поле остаётся `0`. Компилятор выдаёт warning `self-assignment` при `-Xlint`, но не ошибку — легко пропустить.

**Shadowing поля в методе меняет смысл голого имени.** После `int value = 2;` голое `value` — это локальная, а поле доступно только через `this.value`. В длинном методе это источник путаницы «почему поле не меняется» — меняется-то локальная.

**Переменная цикла живёт только в цикле.** `for (int i …)` — `i` в scope только внутри цикла и его заголовка. Попытка использовать `i` после цикла — compile error. Два цикла подряд с `int i` не конфликтуют.

**`this` нельзя использовать до завершения `super()`.** Пока конструктор родителя не отработал, объект не считается полностью инициализированным, и обращение к `this` (кроме как для `this()`/`super()`) ограничено. Полный разбор порядка инициализации — глава 4.

### На собесе

**1. «Скомпилируется ли `int local; print(local);`? А для поля `int field;`?»**  
Ответ: локальная — compile error (`might not have been initialized`, definite assignment, JLS §16); поле — OK, печатает дефолт (`0`/`false`/`null`). _Follow-up:_ «Почему разница?» → Память объекта зануляется при аллокации, у полей гарантированный дефолт; локальные на стеке без гарантированного нуля, поэтому требуется доказуемое присваивание. _Follow-up 2:_ «`int n; if(cond) n=1; else n=2; print(n)` — ок?» → Да, присвоено по всем путям. Убери `else` — ошибка: появился путь без присваивания. _Слабый ответ:_ «локальную надо всегда инициализировать» — неточно; нужно присваивание до _чтения_ по всем путям, объявление без инициализации допустимо.

**2. «Чему равен `this` внутри лямбды и внутри анонимного класса?»**  
Ответ: в лямбде `this` — охватывающий экземпляр (лямбда своего `this` не вводит); в анонимном классе `this` — сам анонимный объект. _Follow-up:_ «Что напечатает `this.getClass().getName()` в каждом?» → В анонимном `Outer$1`, в лямбде `Outer`. _Follow-up 2:_ «Зачем так?» → Лямбда не создаёт новый scope для `this`/`super`, это упрощает захват и делает её прозрачной для охватывающего контекста; анонимный класс — отдельный объект отдельного класса. _Follow-up 3:_ «Как из анонимного класса достать охватывающий объект?» → `Outer.this`.

**3. «Разница между shadowing и re-declaration?»**  
Ответ: re-declaration (локальная поверх локальной в перекрывающемся scope) — запрещён; shadowing (локальная поверх поля) — разрешён, поле доступно через `this.`. _Follow-up:_ «Два `for (int i …)` подряд — ошибка?» → Нет, scope не перекрываются. _Follow-up 2:_ «`Point(int x){ x = x; }` — что с полем?» → Останется `0`: присвоили параметр сам себе, нужен `this.x = x`.

**4. «Что такое `Outer.this` и когда нужен?»**  
Ответ: qualified this — доступ к экземпляру охватывающего класса из non-static вложенного, где голый `this` означает внутренний объект. _Follow-up:_ «Почему в C# такого нет?» → Вложенные классы C# не держат ссылку на внешний экземпляр (аналог Java `static nested`); Java `inner` держит неявную ссылку — отсюда синтаксис и класс утечек памяти. _Follow-up 2:_ «Есть ли `this` в static-методе?» → Нет, у него нет получателя — compile error.

**5. «Почему лямбда не может захватить изменяемую локальную?»**  
Ответ: требуется effectively final; захватывается значение на момент создания, изменяемая ячейка создала бы иллюзию общего состояния. _Follow-up:_ «Как обойти, если нужно накапливать?» → Через контейнер (`AtomicInteger`, массив, поле объекта) — мутируешь объект, а не переприсваиваешь локальную. _Follow-up 2 (senior):_ «Чем отличается от C#?» → C#-замыкания захватывают саму переменную (можно менять снаружи и видеть внутри); Java захватывает значение и потому требует неизменности.

**Модель-ответ (RU):** «Область видимости локальной — от объявления до конца блока. Переобъявить в перекрывающемся scope нельзя, затенить поле — можно, поле тогда через `this.`. Локальную нельзя прочитать до definite assignment по всем путям — compile error по JLS §16; поле читается и даёт дефолт, потому что память объекта зануляется при аллокации. `this` в лямбде — охватывающий экземпляр, лямбда своего `this` не вводит; в анонимном классе `this` — сам анонимный объект, а охватывающий достаётся через `Outer.this`. В static-контексте `this` нет.»

**Модель-ответ (EN):** "A local variable's scope runs from its declaration to the end of the enclosing block. You can't re-declare it in an overlapping scope, but a local may shadow a field — the field is then reached via `this.`. A local can't be read until it's definitely assigned on every path — that's a compile error per JLS §16; a field reads fine and yields its default, because object memory is zeroed on allocation. Inside a lambda `this` is the enclosing instance — a lambda introduces no `this` of its own; inside an anonymous class `this` is the anonymous object itself, and the enclosing instance is reached via `Outer.this`. There's no `this` in a static context."
