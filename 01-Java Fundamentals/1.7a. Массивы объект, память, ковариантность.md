Массив — это фундамент, на котором стоит вся Java: внутри `ArrayList` лежит `Object[]`, внутри `HashMap` — массив бакетов, внутри `String` — `byte[]`. Даже если ты редко пишешь `new int[n]` руками, ты постоянно работаешь с массивами опосредованно, и их свойства протекают наверх: почему `ArrayList<Integer>` ест в пять раз больше памяти, чем мог бы; почему у коллекций есть перформанс-обрывы, которых нет в аналогичном коде на C#. А ещё массив — носитель единственной официально признанной дыры в системе типов Java, и вопрос про неё («почему массивы ковариантны, а дженерики нет?») — один из самых частых на собесах уровня mid+. В этом уроке разберём массив как объект: что он знает о себе в рантайме и что из этого следует.

**Ментальная модель.** Массив в Java — полноценный объект на куче, который _помнит свой настоящий тип элементов в рантайме_. Из этой памяти следует и его сила (JVM проверяет каждую запись), и его дыра (ковариантность, узаконенная в 1995-м).

### Массив — объект, а не блок памяти

Проверим, что массив — объект, напрямую: спросим у него класс и посмотрим, что он унаследовал от `Object`.


```java
int[] a = {1, 2, 3};
String[] s = {"x"};

System.out.println(a.getClass());   // class [I
System.out.println(s.getClass());   // class [Ljava.lang.String;
System.out.println(a.length);       // 3
System.out.println(a);              // [I@27ce24aa
```

У массива есть настоящий `Class`-объект — с криптографичным JVM-именем (`[I` = «одномерный массив `int`», `[Ljava.lang.String;` = «массив ссылок на `String`», `[[I` = `int[][]`). Есть заголовок объекта (`object header`), как у любого объекта на куче. И есть `length` — причём это `public final` **поле**, а не метод (JLS §10.7). Отсюда вечная путаница у новичков: `arr.length` без скобок, `str.length()` со скобками, `list.size()` — третий вариант. Это три разные сущности на уровне языка, а не непоследовательность API.

Последняя строка — `[I@27ce24aa` — уже подсказывает вторую особенность: массив ничего не переопределяет из `Object`. Его `toString` печатает имя класса и identity-хеш, его `equals` сравнивает ссылки. К этому вернёмся в подводных камнях.

> 💡 **Из C#:** там массив тоже объект (наследник `System.Array`), и `Length` — тоже не метод, а свойство. Но в C# свойство маскирует разницу синтаксически — `arr.Length` и `str.Length` выглядят одинаково. В Java свойств нет, поэтому шов «поле vs метод» торчит наружу.

### `int[]` vs `Integer[]` — две разные структуры данных

Ключевое различие, которое определяет перформанс-профиль массивов и через них — всех коллекций. Замерим на JDK 21 миллион элементов:

```java
int[] prims = new int[1_000_000];          // ~4 МБ
Integer[] boxed = new Integer[1_000_000];
for (int i = 0; i < 1_000_000; i++) boxed[i] = i + 1000;   // ~20 МБ
```

Замер через `Runtime` даёт ~3.3 МБ против ~20 МБ. Пятикратная разница — не случайность, а прямое следствие раскладки в памяти:

![[Pasted image 20260702125112.png]]

Читаем диаграмму сверху вниз. `int[]` — непрерывный блок четырёхбайтовых значений сразу после заголовка. Чтобы взять `arr[5]`, JVM вычисляет `base + смещение_заголовка + 5*4` и читает значение — одна операция, соседние элементы уже в кэш-линии процессора.

`Integer[]` — блок **ссылок** (по 4 байта при включённом `compressed oops` — а он включён по умолчанию на хипах до 32 ГБ), каждая из которых указывает на отдельный объект `Integer` где-то ещё на куче. Чтение `arr[5]` — это две операции: взять ссылку, прыгнуть по ней, прочитать поле `value`. Прыжок — в непредсказуемое место, то есть почти гарантированный промах кэша. Плюс каждый бокс несёт собственный заголовок: ~12 байт header + 4 байта `value` = 16 байт на объект, который хранит 4 байта полезных данных. Отсюда и наши замеры: `4 МБ ссылок + 16 МБ боксов = 20 МБ` против `4 МБ` у `int[]`. И это ещё не всё — миллион мелких объектов означает миллион единиц работы для GC при каждой сборке.

> 💡 **Из C#:** в .NET `int` — value type, и `List<int>` держит внутри настоящий `int[]` без боксинга — верхняя строка диаграммы даром. В Java аналога **нет**: дженерики не работают с примитивами, `List<Integer>` всегда боксит. Это реальная причина, почему в Java для горячих числовых путей берут голый `int[]` или примитивные коллекции (`fastutil`, Eclipse Collections), а не `List<Integer>`. Проект Valhalla (value types для JVM) должен закрыть разрыв, но на Java 21 его нет.

Ещё одно следствие раскладки — дефолтные значения. JVM зануляет память массива при создании, и «ноль» для каждого типа свой:

```java
int[] prims = new int[3];
Integer[] boxes = new Integer[3];

System.out.println(prims[0]);   // 0
System.out.println(boxes[0]);   // null — ссылки занулены, боксов нет
int x = boxes[0];               // NullPointerException — unboxing null
```

`new Integer[1000]` не создаёт тысячу объектов `Integer` — он создаёт тысячу `null`-ссылок. Боксы появятся только когда ты их туда положишь.

### Многомерный массив — это массив массивов

`int[3][4]` в Java — не прямоугольный блок 3×4, как двумерный массив в C. Это массив из трёх ссылок, каждая указывает на независимый `int[]`. Поэтому «строки» легально имеют разную длину:

```java
int[][] jagged = new int[3][];       // три null-ссылки
jagged[0] = new int[]{1, 2};
jagged[1] = new int[]{1, 2, 3, 4, 5};
// jagged[2] так и остался null

System.out.println(jagged[0].length);              // 2
System.out.println(jagged[1].length);              // 5
System.out.println(Arrays.deepToString(jagged));   // [[1, 2], [1, 2, 3, 4, 5], null]
```

Такой массив называют `jagged` (рваный). Обратная сторона гибкости — та же индирекция, что у `Integer[]`: строки разбросаны по куче, обход «по столбцам» бьёт мимо кэша. Когда нужен настоящий прямоугольный блок под производительность, делают одномерный `int[rows * cols]` и индексируют вручную: `arr[row * cols + col]`.

> 💡 **Из C#:** в C# есть **оба** вида — `int[,]` (настоящий прямоугольный, один блок памяти) и `int[][]` (jagged). В Java существует только jagged; синтаксис `new int[3][4]` — это сахар, который создаёт внешний массив и четыре внутренних за тебя, но структура всё равно «массив ссылок на массивы».

### Ковариантность: что массив разрешает и как за это платит

Массивы в Java ковариантны (JLS §4.10.3): раз `String` — подтип `Object`, то и `String[]` — подтип `Object[]`. Присваивание `Object[] arr = new String[3]` легально. Посмотрим, куда это ведёт:

```java
static void poison(Object[] arr) {
    arr[0] = 42;                       // компилируется без единого warning
}

public static void main(String[] args) {
    String[] names = {"Anna", "Boris"};
    poison(names);                     // ArrayStoreException: java.lang.Integer
}
```

Разберём по ролям. Компилятор внутри `poison` видит статический тип `Object[]` — положить туда `Integer` формально корректно, претензий нет. Но реальный объект, пришедший в параметр, — `String[]`, и он _помнит_ об этом: настоящий тип элементов записан в заголовке объекта. На каждой записи ссылки в массив JVM исполняет байткод-инструкцию `aastore`, которая сверяет реальный тип значения с реальным типом элементов массива. `Integer` не является `String` — рантайм кидает `ArrayStoreException` (JLS §10.5).

![[Pasted image 20260702125629.png]]

Это и называется дырой в системе типов: статическая типизация пообещала, что скомпилированный без предупреждений код типобезопасен, — и не сдержала слово. Ошибка типа переехала из compile time в runtime.

Свойство «массив помнит свой настоящий тип в рантайме» называется _reified_ — тип овеществлён, физически присутствует в работающей программе. Именно реификация делает ковариантность массивов _терпимой_: дыра есть, но у неё есть страховочная сетка — `aastore` поймает нарушение в момент записи, а не даст данным тихо испортиться.

Зачем дыру вообще открыли? Исторический компромисс. В Java 1.0 не было дженериков, а обобщённые методы были нужны: без ковариантности невозможно написать `Arrays.sort(Object[])`, работающий с любым массивом ссылок. Ковариантность была единственным способом получить полиморфизм по массивам — и её вшили в язык, а убрать теперь нельзя из-за обратной совместимости.

Важная граница дыры: она существует **только для массивов ссылочных типов**. `int[]` не ковариантен к `Object[]` — он вообще не `Object[]` (хотя сам по себе `Object`):

```java
int[] prim = {1, 2};
Object o = prim;         // ✅ массив — объект
Object[] oa = prim;      // ❌ error: incompatible types: int[] cannot be converted to Object[]
```

Внутри `int[]` лежат голые значения, а не ссылки, — подсунуть туда объект чужого типа физически невозможно, поэтому и `aastore`-проверки для примитивных массивов нет: компилятор фиксирует тип намертво.

> 💡 **Из C#:** дыра унаследована один в один. `string[]` присваивается `object[]`, запись чужого типа даёт `ArrayTypeMismatchException` — буквальный близнец `ArrayStoreException`. Оба языка сегодня считают это дизайнерской ошибкой эпохи, которую нельзя исправить.

### Мост вперёд: почему с `List` этот трюк не проходит

Логично ожидать симметрии: раз `String[]` → `Object[]` можно, то и `List<String>` → `List<Object>`... Компилятор отвечает жёстко:

```java
List<String> strings = new ArrayList<>();
Object[] arr = new String[3];        // ✅ ковариантность массивов
List<Object> objects = strings;      // ❌ error: incompatible types:
                                     //    List<String> cannot be converted to List<Object>
```

Дженерики в Java **инвариантны**: `List<String>` не является подтипом `List<Object>`, это два несвязанных типа. Причина упирается в то самое различие reified/erased. Дженерики в Java — _erased_: компилятор проверяет типовые параметры и **стирает** их, в рантайме `List<String>` — это просто `List`. А значит, у дженериков нет страховочной сетки: JVM физически не может проверить на записи «а точно ли этот `Integer` подходит списку строк?» — типа-то уже нет. Разреши язык ковариантность дженериков — чужое значение попало бы в список **тихо**, без исключения, и взорвалось бы в произвольном месте программы гораздо позже. Поэтому компилятор рубит несовместимость сразу, на присваивании.

Формула для запоминания: **массивы reified и ковариантны — дыра есть, но ловится в рантайме; дженерики erased и инвариантны — дыры нет, всё ловится на компиляции.** Что такое стирание в деталях, почему инвариантность мутабельных контейнеров — это отдельный закон, не зависящий от erasure, и как wildcards (`? extends` / `? super`) возвращают дженерикам управляемую дозу вариантности — всё это в главе 7 про generics.

### Утилиты `Arrays` и identity-семантика

Мы уже видели, что массив не переопределяет методы `Object`. Разворачиваем последствия:

```java
int[] a = {1, 2, 3};
int[] b = {1, 2, 3};

System.out.println(a);                    // [I@27ce24aa — имя класса + identity-хеш
System.out.println(a.equals(b));          // false — сравнение ссылок, как ==
System.out.println(Arrays.equals(a, b));  // true — сравнение содержимого
System.out.println(Arrays.toString(a));   // [1, 2, 3]
```

`equals` у массива — это identity, тот же `==`. Для содержимого — статические хелперы `Arrays.equals` / `Arrays.toString`, а для вложенных массивов их «глубокие» версии `Arrays.deepEquals` / `Arrays.deepToString` (обычный `Arrays.toString(int[][])` напечатает внутренние массивы как `[I@...` по ссылкам).

Копирование — три инструмента:

```java
int[] src = {1, 2, 3, 4, 5};
int[] grown = Arrays.copyOf(src, 7);          // [1, 2, 3, 4, 5, 0, 0] — добивает дефолтами
int[] slice = Arrays.copyOfRange(src, 1, 4);  // [2, 3, 4] — границы [from, to)
int[] full  = src.clone();                    // полная копия
```

Все три — поверхностные (`shallow`). Для `int[]` это неотличимо от глубокой копии — копируются сами значения. Для `Object[]` копируются **ссылки**: новый массив указывает на те же объекты.

```java
StringBuilder[] sbs = { new StringBuilder("x") };
StringBuilder[] copy = sbs.clone();
copy[0].append("!");
System.out.println(sbs[0]);   // x! — оригинал видит мутацию через копию
```

И рабочая пара сортировка/поиск:

```java
int[] nums = {5, 1, 4, 2, 3};
Arrays.sort(nums);                          // in-place, dual-pivot quicksort для примитивов
System.out.println(Arrays.binarySearch(nums, 4));   // 3
System.out.println(Arrays.binarySearch(nums, 6));   // -6 = -(insertion point) - 1
```

Отрицательный результат `binarySearch` — не мусор, а закодированная точка вставки: `-(insertionPoint) - 1`. Элемента `6` нет, вставлять его в индекс 5, ответ `-6`.

> 💡 **Из C#:** роль LINQ-методов (`SequenceEqual`, копирование) в мире массивов Java играет статический класс-хелпер `Arrays` — у самого массива методов нет. `Array.Copy` ≈ `System.arraycopy`, `Array.BinarySearch` — тот же контракт с отрицательной точкой вставки.

### 🔬 Глубже / на синьора

**Байткод: `aastore` vs `iastore`.** Скомпилируем два метода и посмотрим `javap -c` (прогнано на JDK 21):

```java
static void poison(Object[] arr) { arr[0] = Integer.valueOf(42); }
static void primStore(int[] arr) { arr[0] = 42; }
```

```
static void poison(java.lang.Object[]);
    0: aload_0
    1: iconst_0
    2: bipush        42
    4: invokestatic  #7    // Integer.valueOf:(I)Ljava/lang/Integer;
    7: aastore              // ← store с проверкой типа
    8: return

static void primStore(int[]);
    0: aload_0
    1: iconst_0
    2: bipush        42
    4: iastore              // ← store без проверки
    5: return
```

У каждой категории массивов своя store-инструкция: `iastore` для `int[]`, `lastore`, `dastore` и т.д. — и только `aastore` (array-of-references store) несёт рантайм-проверку типа. Это и есть цена ковариантности: каждая запись ссылки в массив — потенциальная проверка. JIT умеет её убирать, когда доказывает тип массива статически (например, массив создан в том же методе), но в общем случае — например, в `poison(Object[])`, куда может прийти что угодно, — проверка остаётся.

**Заголовки и `compressed oops`.** На 64-битном HotSpot с `compressed oops` (дефолт до ~32 ГБ хипа): заголовок обычного объекта — 12 байт (8 mark word + 4 сжатый указатель на класс), у массива ещё +4 байта на `length`, итого 16; всё выравнивается по 8 байт. `Integer` = 12 байт header + 4 байта `value` = 16 байт ровно. Ссылка — 4 байта сжатая (8 без сжатия, флаг `-XX:-UseCompressedOops`). Отсюда наша арифметика 20 МБ на миллион `Integer`. VERIFY при желании копнуть глубже: точную раскладку смотрят через JOL (`jol-core`), в контейнере его не было — цифры сошлись с замером через `Runtime`.

**Имена классов массивов** — JVM-дескрипторы: `[I`, `[J` (long), `[Ljava.lang.String;`, `[[I`. Полезно узнавать их в стектрейсах, хипдампах и выводе профилировщиков: строка `[B` в топе хипдампа — это byte-массивы (чаще всего внутренности строк), а не экзотика.

**Ковариантность как прото-дженерики.** До Java 5 сигнатура `Arrays.sort(Object[])` или `Collections` API были единственным способом писать обобщённый код. Ковариантность массивов — это «дженерики бедного человека» эпохи 1.0: полиморфизм получили, типобезопасность отложили в рантайм. Когда в Java 5 появились настоящие дженерики, у дизайнеров уже был этот шрам — и дженерики сознательно сделали инвариантными, а вариантность отдали в руки программиста через wildcards (глава 7).

### ⚠️ Подводные камни

**1. `println(arr)` и `arr.equals(b)` — молчаливый identity.** Массив печатается как `[I@hash`, сравнивается по ссылке. Компилятор не предупредит (IDE — предупредит). Всегда: `Arrays.toString` / `Arrays.equals`, для 2D — `deep`-версии. Особо коварно в тестах: `assertEquals(expected, actualArray)` на массивах сравнит ссылки и провалится с загадочным сообщением.

**2. `clone()` у `Object[]` — поверхностный.** Копия массива ≠ копия объектов: мутация элемента через копию видна оригиналу. Для настоящей изоляции — глубокая копия каждого элемента. С `int[]` проблемы нет, значения копируются как есть.

**3. `Integer[]` инициализируется `null`-ами, не нулями.** `new int[n]` — нули, `new Integer[n]` — `null`-ссылки. Прочитал бокс до записи и заанбоксил — `NullPointerException` посреди арифметики.

**4. `Arrays.binarySearch` на неотсортированном массиве.** Контракт требует отсортированный вход, но нарушителя никто не бьёт по рукам: метод не падает, а возвращает произвольный результат. На `{9, 1, 8, 2, 7}` поиск `1` вернул `-1` («не найден»), хотя элемент в массиве есть. Ни исключения, ни warning — тихо неверный ответ.

**5. `int[3][4]` — это 4 объекта, а не один.** Внешний массив плюс три внутренних, каждый со своим заголовком, разбросаны по куче. Аллокация в горячем цикле и обход по столбцам стоят заметно дороже, чем подсказывает интуиция «это же один блок 3×4». Плюс строка может быть `null`, если массив создавали как `new int[3][]` — NPE при обходе.

### На собесе

**1. «Что выведет?» — самый частый заход в тему** _(частота: очень высокая, уровень: mid)_

```java
Object[] arr = new Integer[3];
arr[0] = "hello";
```

Модельный ответ: компилируется чисто, в рантайме — `ArrayStoreException`. Статический тип `arr` — `Object[]`, поэтому компилятор разрешает запись любого `Object`. Но реальный тип объекта — `Integer[]`, он записан в заголовке, и инструкция `aastore` на записи сверяет: `String` не является `Integer` → исключение.  
EN: _It compiles cleanly but throws `ArrayStoreException` at runtime. The static type is `Object[]`, so the compiler allows storing any object, but the actual runtime type is `Integer[]` — arrays are reified, and the `aastore` bytecode instruction checks every reference store against the array's real component type._  
→ Follow-up: _«А если `int[]` присвоить `Object[]`?»_ — ловушка: не скомпилируется. `int[]` — это `Object`, но не `Object[]`; ковариантность работает только между массивами ссылочных типов.  
→ Типичный слабый ответ: «упадёт ClassCastException» — неверный класс исключения выдаёт, что человек не понимает механизм (`CCE` — про casts/`checkcast`, `ASE` — про записи/`aastore`).

**2. «Почему массивы ковариантны, а дженерики инвариантны?»** _(частота: очень высокая, уровень: mid+; главный вопрос темы)_

Модельный ответ: массивы reified — помнят тип элементов в рантайме, поэтому ковариантность подстрахована проверкой `aastore`: дыра есть, но нарушение ловится в момент записи. Дженерики erased — после компиляции типовой параметр стёрт, рантайм-проверка невозможна в принципе. Ковариантность дженериков означала бы тихую порчу данных без исключения, поэтому их сделали инвариантными: несовместимость режется на компиляции. Управляемую вариантность дженерикам возвращают wildcards.  
EN: _Arrays are reified — the JVM knows the real component type at runtime, so covariance is backed by a store check (`ArrayStoreException`). Generics are erased — there's nothing left to check at runtime, so covariant generics would corrupt data silently. Hence generics are invariant, and controlled variance is reintroduced via wildcards (`? extends` / `? super`)._  
→ Follow-up: _«Зачем тогда массивы вообще сделали ковариантными?»_ — до Java 5 не было дженериков; без ковариантности нельзя написать `Arrays.sort(Object[])` для любых массивов. Компромисс 1995 года, закреплённый обратной совместимостью.  
→ Follow-up 2 (на синьора): _«Инвариантность дженериков — следствие erasure?»_ — тонкая ловушка. Строго — нет: мутабельный контейнер обязан быть инвариантным в любой системе типов (в .NET дженерики reified, а `List<T>` всё равно инвариантен). Erasure лишь отнимает страховочную сетку, делая запрет единственной линией обороны. Полный разбор — тема дженериков.

**3. «Чем `int[]` отличается от `Integer[]` по памяти и скорости?»** _(частота: высокая, уровень: mid+)_

Модельный ответ: `int[]` — непрерывный блок значений, 4 байта на элемент, чтение — одна операция с хорошей локальностью кэша. `Integer[]` — блок ссылок на отдельные объекты: ~4 байта ссылка + ~16 байт бокс = впятеро больше памяти, плюс индирекция с промахом кэша на каждом чтении, плюс нагрузка на GC. Миллион элементов: ~4 МБ против ~20 МБ.  
EN: _`int[]` is a contiguous block of values — 4 bytes each, cache-friendly. `Integer[]` is a block of references to separately allocated boxes: ~20 bytes per element, an extra dereference per read, and GC pressure. That's roughly 4 MB vs 20 MB per million elements._  
→ Follow-up: _«Почему тогда не `List<int>`?»_ — дженерики Java не работают с примитивами (erasure сводит всё к `Object`), `List<Integer>` всегда боксит. Для горячих путей — `int[]` или примитивные коллекции; Valhalla когда-нибудь.

**4. «`a.equals(b)` для двух массивов с одинаковым содержимым — что вернёт?»** _(частота: средняя, уровень: junior→mid, но валит многих)_

Модельный ответ: `false`. Массив не переопределяет `equals`, наследует identity-сравнение от `Object` — это фактически `==`. Содержимое сравнивает `Arrays.equals`, вложенные массивы — `Arrays.deepEquals`.  
EN: _`false` — arrays don't override `equals`, so it's reference identity. Use `Arrays.equals` for content, `Arrays.deepEquals` for nested arrays._  
→ Follow-up: _«А почему `Arrays.toString(int[][])` печатает мусор для строк?»_ — печатает внутренние массивы через их `toString`, то есть `[I@hash`; нужен `deepToString`.

**5. «`int[3][4]` — сколько объектов создаётся?»** _(частота: средняя, уровень: mid)_

Модельный ответ: четыре — внешний массив ссылок и три внутренних `int[4]`. Многомерных массивов в JVM нет, есть массивы массивов; строки независимы, могут быть разной длины и `null`.  
EN: _Four objects: the outer reference array plus three inner `int[4]` arrays. The JVM has no true multidimensional arrays — only arrays of arrays; rows are independent and may differ in length or be null._  
→ Follow-up: _«Как получить настоящий прямоугольный блок?»_ — одномерный `int[rows * cols]` с ручной индексацией `row * cols + col` — так делают в перформанс-критичном коде ради локальности.

**6. (бонус, на синьора) «Во что компилируется запись в массив и что там проверяется?»** _(частота: низкая, уровень: senior)_

Модельный ответ: для каждой категории — своя инструкция: `iastore`, `lastore` и т.д. для примитивов — без проверок; `aastore` для ссылок — с рантайм-сверкой типа значения против реального типа элементов массива, отсюда `ArrayStoreException`. Это постоянная цена ковариантности; JIT элиминирует проверку, когда может доказать тип массива статически.  
EN: _Each element category has its own store opcode; only `aastore` (reference stores) performs the runtime component-type check that backs array covariance. The JIT can eliminate the check when the array's type is statically provable._
