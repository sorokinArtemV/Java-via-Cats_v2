Ты переносишь сервис с C# на Java и попадаешь в самую коварную зону миграции: конструкции, которые _выглядят_ идентично, но ведут себя по-разному. `if`, `switch`, циклы — это код, который ты пишешь на автопилоте, не задумываясь. И именно автопилот здесь подводит: C#-привычки в 90% случаев работают, а в оставшихся 10% дают баг, который компилируется, проходит часть тестов и всплывает в проде. Эта глава — карта этих 10%: где Java-семантика расходится с C#-интуицией, почему язык так спроектирован и что из этого спрашивают на собеседованиях. Заодно разберём то, чего в C# нет вовсе: два поколения `switch` (классический из 1995-го и современный из Java 14/21) и labeled `break`.

**Ментальная модель.** В Java два разных `switch`, живущих под одним ключевым словом: старый _statement_ с fall-through (наследие C — управление проваливается сквозь ветки, пока не встретит `break`) и новый _expression_ со стрелками (Java 14+) — изолированные ветки, возвращаемое значение, проверка полноты покрытия компилятором. Всё остальное в control flow почти совпадает с C#, кроме точечных мин, которые разберём по одной.

### `if`: одна настоящая мина

Ты, возможно, ждёшь, что главное отличие — «условие строго `boolean`, нет неявного `int → bool`». Но это различие C vs (Java **и** C#): в C# `if (5)` тоже не компилируется. Тут переучиваться не надо, и тернарник `cond ? a : b` идентичен.

Единственная реальная Java-специфика — обёртка `Boolean` в условии:

```java
static Boolean computeFlag() { return null; }

Boolean flag = computeFlag();
if (flag) {                    // NullPointerException — здесь
    doSomething();
}
```

Проверено на JDK 21, сообщение говорит само за себя:

```
java.lang.NullPointerException: Cannot invoke "java.lang.Boolean.booleanValue()"
```

`if` требует `boolean`, у тебя `Boolean` — компилятор молча вставляет auto-unboxing, то есть вызов `flag.booleanValue()`. При `flag == null` это `NPE` ровно на строке `if`, хотя визуально там «просто проверка условия».

> [!tip] 💡 Из C#  
> с `bool?` такое невозможно — `if (nullableBool)` не компилируется, C# заставляет писать `if (flag == true)` или `.Value`. Java проверку на тебя не перекладывает — она перекладывает на тебя _падение_. Это та же ловушка, что `Integer == Integer` из 1.4: обёртка выглядит как примитив, пока не оказывается `null`.

### Классический `switch`: fall-through по умолчанию

Синтаксически классический `switch` совпадает с C# посимвольно. Семантически — противоположен в ключевой точке. Выведем правило из кода (запускалось, вывод реальный):

```java
int x = 2;
switch (x) {
    case 1:  System.out.println("one");
    case 2:  System.out.println("two");
    case 3:  System.out.println("three");
    default: System.out.println("default");
}
```

Печатает **три строки**: `two`, `three`, `default`. Совпал `case 2` — и дальше управление просто _течёт вниз_ сквозь `case 3` и `default`, потому что ни одна ветка не сказала `break`. Это fall-through: ветка классического `switch` — не изолированная коробка, а точка входа в общий поток. `break` — не «завершение ветки», а явный выход из всего `switch`.

Fall-through — не баг, а инструмент группировки:

```java
switch (level) {
    case 1:
    case 2:
        System.out.println("low");    // и для 1, и для 2 — намеренно
        break;
    case 3:
        System.out.println("mid");    // нет break — провалится в default: почти всегда баг
    default:
        System.out.println("fallback");
}
```

`case 1:` без тела — легальный идиоматичный способ сказать «1 и 2 обрабатываются одинаково». А вот непустая ветка без `break` — в 99% случаев забытый `break`, и компилятор молчит.

> [!tip] 💡 Из C#  
> там этот класс багов уничтожен на уровне языка — каждая непустая `case`-ветка обязана заканчиваться `break`/`return`/`throw`/`goto case`, иначе ошибка компиляции. Группировка пустыми метками (`case 1: case 2:`) разрешена и там. То есть твоя интуиция «ветки герметичны» — это гарантия C#-компилятора, а не свойство синтаксиса. В Java этой гарантии нет; она появится только в стрелочном `switch` ниже.

#### Какие типы можно свитчить

Селектор классического `switch` принимает: `byte`, `short`, `char`, `int`, их обёртки (`Byte`, `Short`, `Character`, `Integer` — через unboxing, со всеми `null`-последствиями), `String` (с Java 7), `enum`. Нельзя: `long`, `float`, `double`, `boolean`. Проверено:

```java
long v = 1L;
switch (v) { ... }
// error: selector type long is not allowed
```

Почему именно так — не произвол, а механика байткода: инструкции `tableswitch`/`lookupswitch` оперируют 32-битным `int`-индексом, `long`/`float`/`double` в него не помещаются. `boolean` запрещён по другой причине: для двух значений есть `if`, свитчить нечего. Точная семантика — [JLS §14.11](https://docs.oracle.com/javase/specs/jls/se21/html/jls-14.html#jls-14.11).

#### `null` в классическом `switch`

```java
String s = null;
switch (s) {
    case "A": System.out.println("A"); break;
    default:  System.out.println("default");   // НЕ спасает
}
```

`NullPointerException` — до входа в какую-либо ветку. Реальное сообщение с JDK 21 выдаёт механику:

```
NullPointerException: Cannot invoke "String.hashCode()" because "<local2>" is null
```

Для `String` под капотом первым делом дёргается `s.hashCode()`, для `enum` — `c.ordinal()` (тоже проверено: `Cannot invoke "Color.ordinal()"`). `default` не ловит `null`, потому что его контракт — «значение есть, но не совпало ни с одной меткой», а не «значения нет».

> [!tip] 💡 Из C#  
> там `switch (null)` спокойно уходит в `case null` или `default`. Это принципиальное расхождение, его надо запомнить намертво: **классический Java-`switch` по ссылочному селектору с `null` — всегда `NPE`**. Единственный способ поймать `null` внутри `switch` появится в pattern-режиме Java 21 (ниже).

> [!example] 🔬 Глубже / на синьора  
> `switch` компилируется тремя разными стратегиями в зависимости от типа селектора — и это любимая тема для «а почему так?» на собеседованиях. Всё ниже проверено через `javap -c -p` на JDK 21.
> 
> ![[Pasted image 20260702122157.png]]
> 
> **`int` и друзья.** Компилятор выбирает между двумя инструкциями по плотности ключей. Плотные (`case 0,1,2`) → `tableswitch`: непрерывная таблица переходов, вычисление адреса за O(1) — «ключ минус смещение, прыжок». Разреженные (`case 1, 1000, 50000`) → `lookupswitch`: отсортированный список пар «ключ → адрес», бинарный поиск O(log n). Выбор автоматический; на моём тестовом классе `javap` показал ровно это — `tableswitch { // 0 to 2` для плотных и `lookupswitch { // 3` для разреженных.
> 
> **`String`.** В таблицу переходов строку не положишь — там только `int`. Поэтому javac строит двухуровневую схему: сначала `lookupswitch` по `s.hashCode()`, потом внутри выбранной ветки `s.equals("literal")` для отсева hash-коллизий (у разных строк хеши могут совпадать). В байткоде видны оба вызова: `invokevirtual String.hashCode` перед `lookupswitch` и `invokevirtual String.equals` внутри веток. Отсюда и `NPE` на `null` — `hashCode()` дёргается раньше всего остального.
> 
> **`enum` — с нюансом, который сырые конспекты обычно упрощают.** Проверил оба случая на JDK 21:
> 
> - Если enum объявлен **в том же файле** (той же compilation unit), что и `switch`, javac свитчит прямо по `c.ordinal()` — без всяких прослоек. Они компилируются вместе, рассинхрон между ними невозможен.
> - Если enum **из другого файла** — javac генерит синтетический вложенный класс (`MyClass$1`) с массивом `$SwitchMap$Color` и свитчит по `$SwitchMap[c.ordinal()]`.
> 
> Зачем прослойка? Бинарная совместимость. `ordinal()` — это позиция константы в объявлении. Если кто-то вставит константу в начало enum и перекомпилирует _только_ enum, все ordinal съедут. `switch`, прошитый на старые ordinal напрямую, молча заматчил бы не те ветки. А `$SwitchMap` заполняется в статическом инициализаторе синтетического класса **по именам констант при загрузке** — то есть пересчитывается под актуальную версию enum. Я это проверил живым экспериментом: после вставки новой константы в начало enum и перекомпиляции только его, старые ветки продолжили матчиться правильно.

### `switch`-выражение — Java 14, [JEP 361](https://openjdk.org/jeps/361)

Классический `switch` — это _statement_: он делает, но не возвращает. Чтобы вычислить значение, приходилось заводить переменную, присваивать в каждой ветке и не забыть `break` — три места для ошибки на ровном месте. C# решил это давно (`switch`-выражение с `=>`), Java догнала в 14-й версии, и решение получилось из двух частей.

Первая — **стрелочный синтаксис** `case X ->`. Справа от стрелки одно выражение или блок; ветки полностью изолированы, fall-through не существует, `break` не нужен и невозможен. Вторая — `switch` теперь может быть **выражением**, то есть возвращать значение:

```java
int numLetters = switch (day) {
    case MONDAY, FRIDAY, SUNDAY -> 6;
    case TUESDAY                -> 7;
    case THURSDAY, SATURDAY     -> 8;
    case WEDNESDAY              -> 9;
};
```

Несколько меток через запятую, никаких `break`, каждая ветка герметична — та самая C#-семантика, к которой ты привык. Если ветке нужен блок из нескольких строк, значение из блока «возвращается» ключевым словом `yield`:

```java
int result = switch (token) {
    case "A" -> 1;
    case "B" -> {
        int computed = expensiveCalc();
        yield computed;          // значение этой ветки switch
    }
    default -> 0;
};
```

`yield` — это «верни вот это как результат данной ветки». Три вещи, с которыми его нельзя путать: `return` (выходит из всего метода, а не из ветки), `break` (в выражении запрещён) и C#-овский `yield return` (тот вообще про итераторы-генераторы — совсем другая опера, совпадение слова случайное).

> [!warning] Подводный камень
> смешивать стрелки и двоеточия в одном `switch` нельзя. Проверено: `error: different case kinds used in the switch`. Либо весь `switch` на `->`, либо весь на `:` (старый стиль с двоеточием технически тоже умеет быть выражением через `case X: yield ...;`, но в новом коде так не пишут).

#### Exhaustiveness: компилятор проверяет полноту

Раз выражение обязано вернуть значение, компилятор требует покрыть **все** возможные входы. Для произвольного `int` или `String` это значит «обязателен `default`». Но для `enum` есть тонкость: если явно перечислены все константы, `default` не нужен — компилятор сам видит, что выходов не осталось:

```java
enum Direction { NORTH, SOUTH, EAST, WEST }

String label = switch (dir) {
    case NORTH -> "up";
    case SOUTH -> "down";
    case EAST  -> "right";
    case WEST  -> "left";
    // default не нужен — все 4 покрыты
};
```

> [!warning] Подводный камень
> — рассинхрон версий. А что если после компиляции этого класса кто-то добавил в enum константу `UP` и перекомпилировал _только_ enum? Компилятор эту ситуацию видеть не может, поэтому javac страхуется: вставляет в байткод **скрытую default-ветку**, кидающую исключение. Я воспроизвёл это живьём на JDK 21: скомпилировал полный `switch`, добавил константу, перекомпилировал только enum, запустил —
> 
> ```
> Exception in thread "main" java.lang.MatchException
> ```
> 
> При этом старые константы продолжили работать корректно (спасибо name-based `$SwitchMap`). До Java 21 в этом сценарии летел `IncompatibleClassChangeError`; замену на `MatchException` зафиксировал [JEP 441](https://openjdk.org/jeps/441). Вывод для собеса: exhaustiveness — это compile-time гарантия _для текущей сборки_; рассинхрон бинарников ловится на рантайме явным исключением, а не тихим проскоком мимо всех веток.

### Pattern matching в `switch` — Java 21, [JEP 441](https://openjdk.org/jeps/441)

До сих пор `switch` сравнивал значение с константами. JEP 441 (финал в Java 21) научил его матчить по **типу** и связывать переменную прямо в метке — то, что в C# давно живёт под именем pattern matching в `switch`. Здесь — ровно та часть, которая относится к самому `switch`; полный разбор паттернов (record patterns, деконструкция, ADT) — в главе 17.

**Type pattern** — `case Тип имя`: если значение этого типа, оно уже скастовано и доступно под именем. **Guarded pattern** — добавка `when условие`: ветка срабатывает, только если и тип подошёл, и условие истинно; иначе матчинг идёт дальше. **`case null`** — впервые `null` можно поймать веткой. Всё вместе (запускалось, все семь входов дали ожидаемое):

```java
static String classify(Object obj) {
    return switch (obj) {
        case Integer i when i < 0  -> "negative int";
        case Integer i when i == 0 -> "zero";
        case Integer i             -> "positive int: " + (i * 2);
        case String s              -> "string of length " + s.length();
        case int[] arr             -> "int array of " + arr.length;
        case null                  -> "null value";
        default                    -> "something else";
    };
}
```

> [!tip] 💡 Из C#  
> ключевое слово guard-условия — тоже `when`, интуиция переносится один в один. (В preview-версиях Java был `&&`, в финале — `when`.)

Про `null` — важная симметрия со старым поведением: **без** явного `case null` pattern-`switch` на `null` по-прежнему кидает `NPE` (проверено — летит из `Objects.requireNonNull`). То есть поведение обратно совместимо, а ловля `null` — осознанный opt-in. Частая идиома — совместить: `case null, default -> ...` (тоже проверено, работает).

> [!warning] Подводный камень
> — **dominance**. Порядок веток важен: более общий паттерн не может стоять раньше более частного, иначе частный недостижим — и это ошибка компиляции, не warning:
> 
> ```java
> switch (obj) {
>     case Object o -> "any";        // ловит всё
>     case String s -> "string";     // error: this case label is dominated
> }                                  //        by a preceding case label
> ```
> 
> Логика ровно как с порядком `catch`-блоков: частное раньше общего.

И кульминация exhaustiveness — **`sealed`-типы**. Для обычного `Object` компилятор требует `default` (подтипов бесконечно много). Но если тип запечатан и список наследников конечен и известен, `default` не нужен:

```java
sealed interface Shape permits Circle, Square {}
record Circle(double r) implements Shape {}
record Square(double side) implements Shape {}

double area = switch (shape) {
    case Circle c -> Math.PI * c.r() * c.r();
    case Square s -> s.side() * s.side();
    // default не нужен — sealed, оба наследника покрыты
};
```

Убери ветку `Square` — получишь (проверено) `error: the switch expression does not cover all possible input values`. В этом весь смысл связки `sealed` + pattern-`switch`: добавишь завтра `record Triangle implements Shape` — каждый такой `switch` в кодовой базе **перестанет компилироваться**, пока не обработаешь новый случай. Проверку «не забыл ли вариант» делает компилятор, а не код-ревью. Цепочка `if/else if` такой гарантии не даёт в принципе. Детали `sealed` — в главе 17.

### Циклы: только различия от C#

Базовый синтаксис `for` / `while` / `do-while` идентичен. Реальные различия умещаются в таблицу:

|Конструкция|C#|Java|Что важно|
|---|---|---|---|
|C-style `for`|`for (int i=0; i<n; i++)`|то же|идентично|
|foreach|`foreach (var x in coll)`|`for (var x : coll)`|двоеточие, слова `foreach` нет|
|что итерируемо|любой тип с `GetEnumerator` (duck typing)|только `Iterable<T>` или массив|Java строже: нужен интерфейс|
|`while` / `do-while`|идентично|идентично|—|
|прыжок через уровни|`goto label`|`break label` / `continue label`|разные механизмы, ниже|

> [!warning] Подводный камень
> — модификация коллекции в enhanced `for`. И он двухголовый, причём вторая голова опаснее первой. Оба сценария запускались:
> 
> ```java
> List<String> list = new ArrayList<>(List.of("a", "b", "c"));
> for (String s : list) {
>     if (s.equals("a")) list.remove(s);   // ConcurrentModificationException
> }
> ```
> 
> Это ожидаемая голова: enhanced `for` под капотом — `Iterator`, итераторы большинства коллекций fail-fast и при структурной модификации кидают `CME` (несмотря на название — в одном потоке, без всякой многопоточности). А теперь вторая:
> 
> ```java
> List<String> list = new ArrayList<>(List.of("a", "b", "c"));
> for (String s : list) {
>     if (s.equals("b")) list.remove(s);   // НЕТ исключения!
> }
> // цикл завершился, "c" никогда не посещался, list = [a, c]
> ```
> 
> Удаление **предпоследнего** элемента не кидает ничего: после удаления размер стал 2, курсор итератора тоже 2, `hasNext()` вернул `false` — цикл тихо закончился, а последний элемент просто не был посещён. Проверка `checkForComodification` живёт в `next()`, до которого дело не дошло. Тихий пропуск элемента хуже исключения: `CME` хотя бы орёт. Правильные способы удаления — `Iterator.remove()` или `list.removeIf(...)`. Механика `modCount` и fail-fast — в главе 13, здесь достаточно знать, откуда `CME` (и его отсутствие) берутся в безобидном однопоточном цикле.

### Labeled `break` / `continue` — и почему это не `goto`

В Java на цикл можно навесить метку и выходить через уровни вложенности (запускалось — из точки `[1][1]` вышло сразу из обоих циклов):

```java
outer:
for (int i = 0; i < matrix.length; i++) {
    for (int j = 0; j < matrix[i].length; j++) {
        if (matrix[i][j] == target) {
            System.out.println("found at " + i + "," + j);
            break outer;       // выход из ОБОИХ циклов
        }
    }
}
```

`break outer` выходит из внешнего цикла целиком; `continue outer` бросает остаток внутреннего и переходит к следующей итерации внешнего. Без метки оба действуют на ближайший цикл — как в C#. Метку технически можно навесить на любой statement, но осмысленна она практически только на циклах.

> [!tip] 💡 Из C#  
> аналога нет — там для выхода через уровни используют `goto label` или флаг с проверкой. Java пошла другим путём: слово `goto` зарезервировано, но не используется — намеренное решение против «спагетти»-переходов. Labeled `break`/`continue` — структурированный огрызок `goto`: прыгать можно только **наружу** из вложенной конструкции, нельзя в произвольную точку, нельзя вперёд через код, нельзя построить цикл из переходов. Поэтому поток остаётся анализируемым. C#-шный `goto` мощнее (включая `goto case` внутри `switch`) — и ровно настолько же опаснее.

Когда оправдан: выход из вложенных циклов по найденному условию — и практически только он. Если меток больше одной или поток скачет — это запах: внутренний цикл просится в отдельный метод с обычным `return`.

### Вопросы с собеса

> [!question]- 1. Что напечатает?
> ```java
> int x = 2;
> switch (x) {
>     case 1: System.out.println("one");
>     case 2: System.out.println("two");
>     case 3: System.out.println("three");
>     default: System.out.println("default");
> }
> ```
> 
> Разбор: `two`, `three`, `default` — три строки. Ни одного `break`, с совпавшего `case 2` управление проваливается до конца (fall-through — поведение языка по умолчанию, не баг примера). Частый неверный ответ — «two», от C#-интуиции герметичных веток.  
> Follow-up: «Перепиши, чтобы печаталось только `two` — без `break`» → стрелочный `switch`: `case 2 -> System.out.println("two");` — там проваливания нет by design.  
> Follow-up глубже: «Чем `case 2 ->` отличается от `case 2:` семантически, не синтаксически?» → двоеточие — точка входа в общий поток, текущий дальше до `break`/конца; стрелка — изолированная ветка.  
> Модельный ответ (EN): _"Classic Java switch falls through by default — without `break`, control flows into subsequent branches, so this prints three lines starting from the matched case. Arrow-style switch from Java 14 removes fall-through entirely."_

> [!question]- 2. По каким типам можно свитчить и почему нельзя `long` и `boolean`?
> Разбор: `byte/short/char/int` + обёртки, `String` (Java 7), `enum`; с Java 21 — любой ссылочный тип в pattern-режиме. `long/float/double` нельзя: байткод-инструкции `tableswitch`/`lookupswitch` индексируются 32-битным `int`. `boolean` — потому что для двух значений есть `if`.  
> Follow-up: «`String` — не `int`, как он свитчится?» → двухуровнево: `lookupswitch` по `hashCode()` + `equals()` внутри ветки для отсева коллизий.  
> Follow-up-trap: «Значит, `switch` по строке может кинуть `NPE`?» → да, при `null`: `hashCode()` дёргается до выбора ветки, и `default` не спасает.  
> Модельный ответ (EN): _"Selectors are limited to int-family types, their wrappers, String, and enums, because the underlying `tableswitch`/`lookupswitch` bytecodes index by 32-bit int. Strings compile to a two-level hashCode-plus-equals scheme, which is also why a null selector throws NPE before any branch."_

> [!question]- 3. Что произойдёт?
> ```java
> Color c = null;
> switch (c) {          // классический switch по enum
>     case RED: break;
>     default:  break;
> }
> ```
> 
> Разбор: `NPE`, не уход в `default`. Дёргается `c.ordinal()` (напрямую или через `$SwitchMap`), а `null.ordinal()` невозможен. `default` означает «значение не совпало», а не «значения нет». Неверный ответ «уйдёт в default» — прямой перенос C#-семантики.  
> Follow-up: «Как в Java 21 обработать `null` внутри `switch`?» → pattern-`switch` с явной веткой `case null ->` или `case null, default ->`. Без неё pattern-`switch` тоже кидает `NPE` — совместимость со старым поведением.  
> Модельный ответ (EN): _"Classic switch over a reference type NPEs on null before selecting any branch — `default` doesn't catch it. Only a Java 21 pattern switch with an explicit `case null` label can handle null inside the switch."_

> [!question]- 4. Почему не компилируется?
> ```java
> double area = switch (shape) {     // shape: sealed Shape permits Circle, Square
>     case Circle c -> Math.PI * c.r() * c.r();
> };
> ```
> 
> Разбор: не покрыт `Square`; `switch`-выражение обязано быть exhaustive, тип `sealed` — компилятор знает оба варианта и требует оба (или `default`). Реальная ошибка: `the switch expression does not cover all possible input values`.  
> Follow-up: «А если бы `shape` был `Object`?» → обязателен `default`: подтипов бесконечно много, полноту иначе не доказать.  
> Follow-up-trap: «Практическая польза того, что оно не компилируется?» → добавишь `record Triangle implements Shape` — все такие `switch` сломаются на компиляции; невозможно забыть обработать новый вариант. Compile-time гарантия, которой нет у `if/else`.  
> Модельный ответ (EN): _"Switch expressions must be exhaustive. For a sealed type the compiler knows the full set of subtypes and requires all of them covered — that's the whole point of sealed-plus-switch: add a subtype and every affected switch fails to compile."_

> [!question]- 5. Чем `break label` отличается от `goto`?
> Разбор: `break label`/`continue label` прыгают только **наружу** из вложенной конструкции; `goto` (в C#) умеет в произвольную метку, вперёд и внутрь. Java намеренно не дала `goto` (слово зарезервировано, не используется): ограниченные прыжки не превращаются в неструктурированный поток, код остаётся анализируемым.  
> Follow-up: «Когда labeled `break` оправдан?» → выход из вложенных циклов по условию; в остальных случаях — сигнал вынести внутренний цикл в метод и сделать `return`.  
> Модельный ответ (EN): _"Labeled break and continue only jump outward from a nested construct — you can't jump to arbitrary points like goto. Java reserved but never implemented goto deliberately; labels are its structured, analyzable substitute."_

> [!question]- 6. (trap после правильного ответа про exhaustiveness) «Ты сказал, при полном покрытии enum `default` не нужен. А что выполнится, если в enum добавили константу, перекомпилировали только его, и старый класс со `switch` получил новое значение?»
> Разбор: javac вставил скрытую default-ветку — на JDK 21 она кидает `MatchException` (до 21 — `IncompatibleClassChangeError`; замена зафиксирована в JEP 441). Тихого проскока мимо всех веток не будет — упадёт явно. При этом старые константы продолжают матчиться корректно (диспетчеризация переживает пересортировку). Exhaustiveness — гарантия для конкретной сборки; рассинхрон бинарников ловится рантайм-исключением.  
> Модельный ответ (EN): _"The compiler emits a hidden default branch for that scenario. On Java 21 it throws `MatchException` at runtime — previously `IncompatibleClassChangeError`, changed by JEP 441 — so binary drift fails loudly instead of silently falling through."_

### Практика

**Декомпиляция `switch`** (~25 мин, нужен только JDK). Три метода: `switch` по плотному `int`, по разреженному `int`, по `String`, плюс два по `enum` — один с enum в том же файле, один с enum в отдельном. `javac`, затем `javap -c -p`. Найди `tableswitch` vs `lookupswitch`, пару `hashCode`+`equals` для строк, прямой `ordinal()` для локального enum и `$SwitchMap` (в синтетическом классе `YourClass$1`) для внешнего. Это закрепит механику намертво.

**Рассинхрон enum руками** (~15 мин). Собери enum и класс со `switch`-выражением без `default` в отдельных файлах. Добавь константу в начало enum, перекомпилируй только его, запусти со старой и с новой константой. Увидишь: старые работают, новая даёт `MatchException`.

**Рефакторинг statement → expression** (~15 мин). Возьми большой `switch`-statement с накоплением результата в переменную и перепиши на выражение со стрелками. Прочувствуй, где `yield` обязателен (блочная ветка), а где нет.

**Exhaustiveness + dominance** (~20 мин, Java 21). `sealed interface` с двумя `record`-наследниками, `switch`-выражение без `default` — компилируется. Добавь третий `record` — прочитай ошибку. Затем в pattern-`switch` поставь `case Object o` перед `case String s` — поймай `dominated by a preceding case label`.

**Два лица удаления в enhanced `for`** (~15 мин). На списке `[a, b, c]` воспроизведи оба сценария: удаление `"a"` (получишь `CME`) и удаление `"b"` (тихий выход, `"c"` не посещён). Второй прогони под дебаггером и посмотри на курсор итератора после удаления — станет ясно, почему `hasNext()` соврал.

### На собесе

Главный тезис, с которого начинай про `switch`: «В Java классический `switch` — fall-through по умолчанию, в отличие от C#: без `break` управление проваливается в следующую ветку. Стрелочный `switch` из Java 14 это убрал — ветки изолированы, и он может быть выражением.»

Про типы селектора: «`byte/short/char/int` и обёртки, `String` с Java 7, `enum`; с Java 21 — любой ссылочный тип через pattern matching. Нельзя `long/float/double/boolean`: байткод-инструкции работают по 32-битному `int`, а для `boolean` есть `if`.»

Про механику, если копают: «`int` компилируется в `tableswitch` (плотные ключи, O(1)) или `lookupswitch` (разреженные, O(log n)). `String` — двухуровнево: `switch` по `hashCode`, потом `equals` для коллизий, поэтому `null` даёт `NPE` до веток. `enum` из другого файла — через синтетический `$SwitchMap` по `ordinal`, ради бинарной совместимости при пересортировке констант.»

Про `null`: «В классическом `switch` `null` — всегда `NPE`, `default` не ловит. Поймать `null` умеет только pattern-`switch` через явный `case null`.»

Про новый синтаксис: «`switch`-выражение возвращает значение, ветки через `->`, многострочная ветка отдаёт значение через `yield`. Не путать ни с `return`, ни с C#-овским `yield return`. Смешивать `->` и `:` в одном `switch` нельзя.»

Про exhaustiveness: «`switch`-выражение обязано покрывать все входы. Для enum с полным покрытием или для `sealed`-типа `default` не нужен — компилятор знает все варианты. Добавишь вариант — `switch` перестанет компилироваться. При рассинхроне бинарников на рантайме летит `MatchException` (JEP 441), не тихий проскок.»

Про guard и порядок: «Условие в ветке — `when`, как в C#. Порядок веток проверяется: общий паттерн раньше частного — ошибка компиляции (dominance), логика как у `catch`-блоков.»

Про labeled `break`: «Метка на цикле плюс `break label`/`continue label` — структурированная замена `goto`, которого в Java нет намеренно. Прыжок только наружу из вложенности; если меток много — выноси цикл в метод.»

Отличие `case X:` от `case X ->` одной фразой: «двоеточие — точка входа в общий поток с fall-through; стрелка — изолированная ветка без проваливания».
