Любой Java-проект больше пары файлов сталкивается с двумя вопросами: как не ловить конфликт имён, когда в разных библиотеках встречается класс `User` или `List`, и как писать `List` вместо `java.util.List` по сто раз на файл. Пакеты решают первое — дают каждому типу глобально уникальное имя (`java.util.List` против `java.awt.List`). Импорты решают второе — позволяют внутри файла обращаться к типу коротким именем. Звучит как рутина из C#, но у Java здесь три собственных механики, на которых стабильно спотыкаются на собеседовании: жёсткая привязка пакета к файловой системе, неявный `java.lang`, и то, что `import` — это вообще не то, чем кажется человеку с C-бэкграундом.

### Mental model

Пакет — это часть полного имени класса и одновременно путь в файловой системе; они обязаны совпадать. `import` — это чисто компиляторный алиас: он не тащит код, не влияет на байткод, а лишь разрешает писать короткое имя вместо полного.

### Как это работает

#### package: имя + путь, связаны жёстко

`package` — первая инструкция в файле (до неё только комментарии). Она делает две вещи сразу: задаёт префикс полного имени класса и указывает, в каком каталоге должен лежать `.class`.

```java
package com.shop.order;
public class Order { }        // полное имя: com.shop.order.Order
```

Скомпилированный `Order.class` обязан лежать в `com/shop/order/`. Это не соглашение по стилю — это проверяет рантайм. Если утащить класс из его каталога, JVM откажется его грузить:

```
Error: Could not find or load main class com.demo.App
Caused by: java.lang.NoClassDefFoundError: App (wrong name: com/demo/App)
```

> [!tip] 💡 Из C#
> namespace ≈ package по роли (уникальность имён, логическая группировка), но C#-namespace **не привязан к файлам** — можно объявить `namespace Whatever` в любом файле любого каталога. В Java пакет = путь, всегда. Это первое, что цепляет при переходе. И второе, важнее: C#-namespace **не создаёт границу доступа**, а Java-пакет создаёт (package-private) — но это тема главы 3.5, здесь только держи в голове, что пакет в Java «весомее», чем namespace.

Мелкий, но частый на практике момент: классы **из своего же пакета импортировать не нужно** — они видны по короткому имени автоматически. Импорт нужен только для типов из _других_ пакетов.

#### java.lang импортируется неявно

Почему `String`, `Object`, `Integer`, `System`, `Math`, `Thread` доступны без единого `import`? Потому что каждый файл автоматически получает `import java.lang.*` — это вшито в язык (JLS 7.3). Всё остальное — `java.util.List`, `java.time.LocalDate`, `java.util.Map` — импортируешь явно.

```java
public class NoImport {
    public static void main(String[] args) {
        String s = "hi";                       // java.lang.String — без import
        System.out.println(Math.max(s.length(), 10));  // java.lang.System, java.lang.Math
    }
}
```

Проверено: компилируется и печатает `10` без единого импорта.

> [!tip] 💡 Из C#
> близкий аналог — `global using` / _implicit usings_ (C# 10+), где `System` и компания подключаются автоматически. Разница в том, что в Java набор неявных импортов фиксирован языком (ровно `java.lang`), а в C# implicit usings настраиваются проектом.

#### import — это алиас, а не #include

Вот ядро урока. `import java.util.List;` не «подключает» и не «вставляет» ничего. Это исключительно указание компилятору: «когда встретишь короткое `List`, подставляй `java.util.List`». В байткод сам `import` не попадает вообще — там всегда лежат полные имена (fully-qualified names, FQN).

Скомпилируем класс с импортом и посмотрим constant pool через `javap -v`:

```
#7  = Class    java/util/ArrayList
#13 = Class    java/util/List
#19 = Class    java/lang/System
```

Строки `import` там нет — только FQN. Более того: программа, написанная с `import java.util.List; List xs`, и та же программа с `java.util.List xs` без импорта дают **идентичные** ссылки в constant pool. Импорт — чистый синтаксический сахар на этапе компиляции.

▼ Схема ниже показывает этот путь: короткое имя в исходнике → `javac` резолвит его → в `.class` только полное имя.

> [!warning] ⚠️ `import` — это не `#include` из C
> `#include` физически вставляет содержимое заголовка в единицу трансляции. `import` не вставляет ничего и на рантайм не влияет: удалил неиспользуемый импорт — байткод не изменился ни на байт. Поэтому «лишние импорты замедляют программу» — миф; они не стоят ничего, кроме шума в глазах.

> [!tip] 💡 Из C#
> прямой аналог — `using System.Collections.Generic;`. Тоже алиас времени компиляции, тоже не влияет на IL. Механика один в один — этот мост честный.

#### import static: тащим член, а не тип

Обычный `import` подключает _тип_. `import static` подключает _статический член_ (метод или константу), чтобы писать его без префикса класса:

```java
import static java.lang.Math.max;   // max(a, b) вместо Math.max(a, b)
import static java.lang.Math.PI;     // PI вместо Math.PI
```

Внутри это тоже алиас: `max(3, 7)` компилируется в обычный `invokestatic java/lang/Math.max` — как если бы ты написал `Math.max(3, 7)`. Никакой магии, только короче в исходнике.

> [!tip] 💡 Из C#
> это `using static System.Math;` — введён в C# 6, работает так же.

Правило применения (Bloch, _Effective Java_, Item 3rd-ed. про static import): **используй редко, только когда это заметно улучшает читаемость.** Хорошо — когда член из одного класса используется массово и контекст очевиден:

```java
import static org.junit.jupiter.api.Assertions.*;   // классика в тестах
import static java.lang.Math.*;                      // в численном коде
```

Плохо — когда короткое имя заставляет читателя гадать об источнике: голый `of(...)`, `max(...)`, `RED` без префикса непонятно откуда пришли. `Color.RED` яснее, чем импортированный `RED`.

> [!example] 🔬 Глубже / на синьора
> 
> **Порядок разрешения имени.** Когда компилятор видит короткое `List`, он ищет в строгом порядке: (1) тип из текущей единицы компиляции / текущего пакета, (2) single-type import (`import java.util.List`), (3) type-import-on-demand (`import java.util.*`), (4) неявный `java.lang.*`. Single-type import **перекрывает** wildcard — это разрешает классический конфликт:
> 
> ```java
> import java.util.*;
> import java.awt.*;
> import java.util.List;   // явный single-type import побеждает оба wildcard
> // теперь List — это java.util.List, без неоднозначности
> ```
> 
> Проверено: без третьей строки `List x;` не компилируется — `reference to List is ambiguous` (в `java.util` и `java.awt` оба есть `List`). С ней — резолвится в `java.util.List`.
> 
> **Куда язык движется (Java 25, за пределами нашего baseline 21).** Тема импортов активно эволюционирует, полезно знать траекторию — но это **не Java 21**, в проде 2026 ты это почти не встретишь:
> 
> - **Module imports — JEP 511 (финализирован в Java 25; preview в 23/24 — JEP 476/494).** Одна строка `import module java.base;` втягивает _все_ пакеты, экспортируемые модулем — `List`, `Map`, `Path` и сотни других без поштучных импортов. Неоднозначности (например, `Date` есть и в `java.util`, и в `java.sql`) разрешаются добавлением обычного single-type import, который перекрывает модульный. Мощно для скриптов и обучения; в production поштучные импорты честнее — видно, откуда что.
> - **`java.lang.IO` — JEP 512 (финализирован в Java 25).** Класс для базового консольного I/O переехал из `java.io` в `java.lang`, поэтому сам класс доступен без импорта: `IO.println("hi")`. Тонкость: его статические методы в compact source files **не** импортируются неявно — писать надо `IO.println(...)` с префиксом, либо явно `import static java.lang.IO.*;`, чтобы получить голый `println(...)`. Это сознательное решение убрать «магию» невидимых методов.
> 
> Обе фичи компилируются только на Java 25+ (на 23/24 — под `--enable-preview`). Упоминаю ради понимания направления, не для использования на 21.
> 
> **Крупный уровень инкапсуляции — JPMS.** Помимо пакета, в Java есть модули (`module-info.java` с `exports`), которые решают, какие пакеты вообще видны снаружи модуля. Это отдельный, «сборочный» уровень видимости. Детали — глава 8.8; в обычных не-модульных проектах его почти не трогают.

> [!warning] ⚠️ Подводные камни и edge cases
> 
> **Неиспользуемый импорт ≠ рантайм-эффект.** IDE подсвечивает их как warning ради чистоты, но на производительность и байткод они не влияют вообще. Не путай «убрать для аккуратности» с «убрать ради скорости».
> 
> **Коллизия `import static` бьёт в точке вызова, а не импорта.** Два одноимённых статических метода из разных классов импортировать _можно_ — ошибка возникнет, только если реально вызвать голое имя:
> 
> ```java
> import static java.lang.Math.max;
> import static java.lang.Integer.max;   // компилируется, пока max() не вызван
> ...
> max(1, 2);   // ← вот здесь: reference to max is ambiguous
> ```
> 
> Проверено: файл с обоими импортами без вызова `max()` компилируется; с вызовом — падает. Лечится префиксом (`Math.max`) или отказом от одного static import.
> 
> **Подпакет — это ДРУГОЙ пакет.** Пакеты в Java плоские, не иерархические. `com.shop.order` и `com.shop.order.internal` — два _разных_ пакета, между ними нет никаких особых отношений по видимости или импорту. «Пакет и все его подпакеты» как единая область — такого в Java нет. (Следствия для package-private доступа — глава 3.5; здесь важно само то, что это отдельные пакеты.)
> 
> **Unnamed package.** Файл без `package`-декларации попадает в _безымянный_ пакет. Такой класс **нельзя** импортировать из класса, лежащего в именованном пакете. Это одна из причин, почему любой реальный код всегда объявляет пакет — классы в дефолтном пакете практически недостижимы для остального проекта.
> 
> **`import static` из enum vs `switch` — не путать.** В `switch` по enum ты и так пишешь голую константу (`case RED ->`) — это правило самого `switch`, оно не требует никакого импорта. `import static ...Color.RED` нужен только для использования константы _вне_ switch, например `if (c == RED)`. Люди иногда решают, что static import «включает» короткие константы в switch — нет, это независимые вещи.
> 
> **Wildcard-импорт двух пакетов с одноимённым типом.** `import java.util.*` + `import java.awt.*` и затем `List` → `ambiguous`. Классическая ловушка на legacy Swing-коде. Решение — single-type import нужного `List`.

### На собесе

> [!question]- Q1. Что делает `import` в Java? Влияет ли он на рантайм? _(junior/mid, очень частый)_
> Модель-ответ (RU): «`import` — это алиас времени компиляции, позволяющий писать короткое имя вместо полного. Он не подключает код и не влияет на байткод — в constant pool всегда лежат полные имена. Удаление неиспользуемого импорта на рантайм не влияет никак.»  
> Model answer (EN): "`import` is a compile-time alias that lets you use a short name instead of the fully-qualified one. It doesn't include or bundle any code and has zero runtime effect — the bytecode always stores fully-qualified names. Removing unused imports changes nothing at runtime."
> 
> - Провальный ответ: «`import` подключает библиотеку / загружает класс». Путаница с `#include` или с classpath. Класс грузит ClassLoader по FQN во время исполнения, импорт тут ни при чём.
> - Follow-up trap: «Тогда как класс вообще попадает в программу?» → через classpath и ClassLoader по полному имени, а не через `import`.

> [!question]- Q2. Почему `String` доступен без импорта? _(junior/mid)_
> RU: «`java.lang` неявно импортируется в каждый файл — это часть спецификации языка. Поэтому `String`, `Object`, `System`, `Math` и прочее из `java.lang` доступны по короткому имени без `import`.»  
> EN: "`java.lang` is implicitly imported into every source file by the language spec, so its types are available unqualified."

> [!question]- Q3. Обязан ли `.java`-файл лежать в каталоге, соответствующем его пакету? _(mid)_
> RU: «`.class` — обязан: рантайм грузит класс по пути, совпадающему с пакетом, иначе `NoClassDefFoundError: wrong name`. Для `.java` это де-факто требование сборки (Maven ожидает `src/main/java/com/shop/order/Order.java`), хотя одиночный файл через `java File.java` компилятор запустит и без совпадения.»
> 
> - Follow-up: «А в C#?» → там namespace от расположения файла не зависит вообще; это ключевое расхождение.

> [!question]- Q4. Когда оправдан `import static`? _(mid+)_
> RU: «Редко — когда член из одного класса используется массово и источник очевиден: `Assertions.*` в тестах, `Math.*` в численном коде. Плохо — для общеназванных `of`/`max`/констант enum, где префикс класса важен для читаемости. Правило Блоха: только когда заметно улучшает читаемость.»
> 
> - Trap-вопрос интервьюера: «Что напечатает код с `import static Math.max` и `import static Integer.max` и вызовом `max(1,2)`?» → не напечатает: ошибка компиляции `reference to max is ambiguous` в точке вызова (сами импорты валидны).

> [!question]- Q5. `import java.util.*` — это плохо? _(mid)_
> RU: «Функционально безвредно (не медленнее поштучных — импорт ничего не стоит), но снижает читаемость и создаёт риск коллизий имён между пакетами (`java.util.List` vs `java.awt.List`). Индустриальный стиль (и Google/Oracle guidelines) — single-type imports. Разрешение коллизии — явный single-type import, он перекрывает wildcard.»

> [!question]- Q6. `com.shop.order` и `com.shop.order.internal` — один пакет или разные? _(mid+, ловушка)_
> RU: «Разные. Пакеты в Java плоские, отношения "родитель-подпакет" по доступу не существует. Package-private класс из одного для другого невидим, несмотря на вложенное имя.»
> 
> - Follow-up (уводит в 3.5): «Тогда как спрятать реализацию внутри фичи?» → package-private + организация по фичам, детали в теме модификаторов доступа.

Границу урока держал жёстко по нашему плану из Turn 1 — package-private, `internal`-мост и package-by-feature ушли указателями в 3.5, JPMS — в 8.8. В сыром черновике это занимало ~60% объёма и расфокусировало урок про импорты в урок про инкапсуляцию. Здесь остались только те следствия пакетной механики, что реально про _пакеты и импорты_ (плоскость пакетов, unnamed package).

Java 25 (JEP 511 module imports, JEP 512 `java.lang.IO`) вынес в 🔬 с явной пометкой «за пределами baseline 21» — сырьё подавало их почти как норму, но на 21 они не компилируются. Оба факта сверил по OpenJDK JEP-страницам; заявленная в сырье тонкость про «статические методы `IO` не импортируются неявно в compact source files» — верна, подтвердил.

Всё, что помечено «проверено», прогнано на JDK 21: неявный `java.lang`, FQN в constant pool через `javap`, коллизия `import static` в точке вызова, «подпакет — другой пакет» через `wrong name`/`not public`, wildcard-неоднозначность и приоритет single-type import.
