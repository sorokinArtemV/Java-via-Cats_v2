Ты сравниваешь два `Long`-идентификатора через `==` — скажем, `orderId` из двух разных DTO. На тестовых данных, где id маленькие, всё зелёное. На проде, где id перевалили за пару сотен, равные значения внезапно «не равны», и баг воспроизводится только на реальных данных. Причина — в том, как Java сшивает два своих мира: примитивы и объекты. Компилятор строит между ними молчаливый мост, а в стандартной библиотеке живёт кеш маленьких значений, который делает поведение `==` зависящим от **величины числа**. Этот урок — про устройство моста и оба места, где он проваливается.

**Ментальная модель.** Примитивы и объекты — разные миры; компилятор молча соединяет их вызовами `Integer.valueOf` (туда) и `intValue()` (обратно). Мост не бесплатный и дырявый в двух местах: кеш `-128..127` ломает `==` на обёртках, а `null` ломает unboxing.

### Зачем вообще существуют обёртки

Восемь примитивов из урока 1.2 — быстрые и компактные, но у них фатальный недостаток: они не объекты. Это бьёт в трёх местах:

1. **Generics не принимают примитивы.** `List<int>` не компилируется — только `List<Integer>`. Причина — type erasure: в рантайме `List<T>` работает через `Object`, а `int` в `Object` не превращается без коробки. Подробно — 7.4.
2. **Примитив не бывает `null`.** `int` — всегда какое-то число. А «значения может не быть» — это половина реальных данных: nullable-колонка в БД, необязательное поле в JSON.
3. **У примитива нет методов.** Распарсить строку, получить `hashCode`, положить в `HashMap` — всё требует объекта.

Отсюда восемь wrapper-классов: `Integer`, `Long`, `Short`, `Byte`, `Character`, `Float`, `Double`, `Boolean`. Каждый — обычный immutable объект с одним полем соответствующего примитива. Из 1.3 ты уже знаешь его цену: 16 байт в heap плюс 4 байта ссылка — против 4 байт у голого `int`.

> [!tip] 💡 Из C#
> в .NET `Int32` — это `struct`, то есть value type, и boxing происходит только при явном приведении к `object`/интерфейсу — редкое событие, о котором ты обычно знаешь. В Java разделение жёстче: `int` и `Integer` — два **разных типа** с разной семантикой, и переходы между ними компилятор вставляет постоянно и молча. И главное расхождение впереди: в .NET две коробки одного числа — всегда разные объекты; в Java для маленьких чисел — один и тот же. Прямого аналога `Integer cache` в .NET нет.

### Автобоксинг: что компилятор делает на самом деле

До Java 5 мост строили руками: `list.add(Integer.valueOf(42))`, обратно — `.intValue()`. Автобоксинг (Java 5, 2004) — синтаксический сахар над теми же вызовами. Проверим, а не поверим. Компилируем:

```java
public class Boxing {
    public static void main(String[] args) {
        Integer a = 42;
        int b = a + 1;
    }
}
```

`javap -c Boxing` на JDK 21 выдаёт:

```
 0: bipush        42
 2: invokestatic  #7   // Method java/lang/Integer.valueOf:(I)Ljava/lang/Integer;
 5: astore_1
 6: aload_1
 7: invokevirtual #13  // Method java/lang/Integer.intValue:()I
10: iconst_1
11: iadd
12: istore_2
```

Никакой магии: `Integer a = 42` — это `Integer.valueOf(42)`, а `a + 1` — это `a.intValue() + 1`. Обычные вызовы методов, вставленные компилятором. Формально это boxing conversion и unboxing conversion. Где они срабатывают:

- **Boxing**: присваивание примитива в переменную обёртки; передача примитива в параметр-обёртку; возврат примитива из метода с reference-типом возврата.
- **Unboxing**: симметрично, плюс — арифметика (`+ - * / %`) на обёртках распаковывает оба операнда; `if (boxedBool)` распаковывает условие; сравнения `< > <= >=` распаковывают оба операнда.

Каждая такая точка — это реальный вызов метода, который может аллоцировать объект или упасть с NPE. Обе беды разберём по очереди.

### `Integer cache`: почему `==` зависит от величины числа

Ключ в том, что `Integer.valueOf(int)` — не `new Integer(int)`, а фабричный метод с оптимизацией. Упрощённо из исходников JDK:

```java
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

`IntegerCache` — статический вложенный класс, при инициализации создающий массив объектов для диапазона по умолчанию `-128..127`. Все `valueOf` в диапазоне возвращают **один и тот же объект** из массива; вне диапазона — каждый раз новый.

![[Pasted image 20260702103518.png]]

Проверено на JDK 21:

```java
Integer a = 127, b = 127, c = 128, d = 128;
System.out.println(a == b);   // true  — один объект из кеша
System.out.println(c == d);   // false — два объекта в heap

Integer e = -128, f = -128, g = -129, h = -129;
System.out.println(e == f);   // true  — нижняя граница включительно
System.out.println(g == h);   // false — уже за границей
```

Почему именно `-128..127`? Это диапазон `byte` — маленькие числа (счётчики, индексы, коды) статистически самый частый груз для боксинга, и плодить под них объекты расточительно. Важно: этот диапазон — не оптимизация конкретной JVM, а **гарантия спецификации**. JLS §5.1.7 требует, чтобы boxing одного и того же значения из этого диапазона (а также `boolean`, `byte` целиком и `char` в `0..127`) всегда давал одну и ту же ссылку.

Границы на практике (проверено экспериментальным прощупыванием от -300 до 3000): по умолчанию `-128..127`; с флагом `-XX:AutoBoxCacheMax=2000` — `-128..2000`. Верхняя граница расширяется флагом, нижняя зашита. Кто ещё кеширует (всё проверено на JDK 21):

|Тип|Кеш|Проверка `==` на JDK 21|
|---|---|---|
|`Integer`, `Long`, `Short`, `Byte`|`-128..127` (у `Byte` это весь диапазон)|`127L == 127L` → `true`, `128L == 128L` → `false`|
|`Character`|`0..127`|`(char)127` → `true`, `(char)128` → `false`|
|`Boolean`|оба значения — константы `TRUE`/`FALSE`|всегда `true`|
|`Float`, `Double`|**кеша нет**|`1.0 == 1.0` в обёртках → `false`|

А что с `new Integer(1)`? Конструктор помечен `@Deprecated(since="9", forRemoval=true)`, и на JDK 21 он **всё ещё компилируется** — с warning'ом `[removal]` (проверено: `new Integer(1) == Integer.valueOf(1)` даёт `false` — конструктор всегда мимо кеша). В интервью и легаси-коде он ещё встречается, в новом коде — никогда: он и плодит лишние объекты, и однажды действительно будет удалён.

### `==` на обёртках: три режима одного оператора

Теперь можно сформулировать точно, потому что режим зависит от **типов операндов** (проверено):

```java
Integer big1 = 1000, big2 = 1000;
int prim = 1000;

big1 == big2   // false — два reference-типа: сравнение ссылок (JLS §15.21.3)
big1 == prim   // true  — примитив рядом: unboxing big1, числовое сравнение (§15.21.1)
```

Если оба операнда — обёртки, работает reference equality из 1.3: «тот же объект?» — и ответ зависит от кеша. Если хотя бы один — примитив, компилятор распаковывает обёртку и сравнивает числа — корректно, но с риском NPE (`null == prim` падает, проверено). Смысловое сравнение двух обёрток — это `equals` (у `Integer` он сравнивает значения, а `hashCode` равен самому значению — проверено: `Integer.valueOf(200).hashCode()` → `200`), либо null-безопасный `Objects.equals(a, b)`.

Правило для кода: **на обёртках — только `equals`/`Objects.equals`; `==` — только когда оба операнда гарантированно примитивы.** IDEA подчёркивает `==` на обёртках жёлтым — это тот warning, который надо чинить, а не глушить.

### NPE при unboxing: где компилятор подставляет

Каждая точка, где компилятор вставил `.intValue()`, — потенциальный NPE. Зоопарк проверенных случаев:

```java
Integer a = null;
Integer b = 5;
a == b;                          // false — ссылки, БЕЗ unboxing, NPE нет
a > b;                           // NPE — сравнение требует чисел, unboxing обоих

Map<String, Integer> counts = new HashMap<>();
int n = counts.get("missing");   // NPE — get вернул null, unboxing в int
counts.getOrDefault("missing", 0);  // 0 — лекарство

Integer wrapped = null;
int raw = 100;
Integer result = true ? wrapped : raw;  // NPE — см. ниже
```

Самая частая продовая версия — `Map.get` в примитив: `get` возвращает `null` для отсутствующего ключа, и молчаливый unboxing превращает «ключа нет» в падение. Helpful NPE из 1.3 здесь называет виновника прямым текстом (реальное сообщение с JDK 21): `Cannot invoke "java.lang.Integer.intValue()" because the return value of "java.util.Map.get(Object)" is null`.

Тернарник — отдельная собесная классика. Условие `true`, берём `wrapped`, он `Integer` — казалось бы, передавай как есть. Но тип тернарного выражения компилятор вычисляет **статически** по обеим веткам: если одна ветка — обёртка, а другая — соответствующий примитив, общий тип — примитив, значит обе ветки проходят через unboxing. Неактивность ветки в рантайме роли не играет — тип уже выбран на компиляции, и `wrapped.intValue()` на `null` падает.

И последний штрих — во что разворачивается инкремент обёртки (проверено `javap -c`):

```java
Integer counter = 0;
counter++;
// байткод: intValue() → iadd → Integer.valueOf(...)
// то есть: counter = Integer.valueOf(counter.intValue() + 1)
```

Каждый `++` на обёртке — распаковка, сложение и **новый объект** (как только значение вышло за кеш). Обёртка immutable, «увеличить на месте» её нельзя в принципе. Что это стоит — сейчас, с цифрами.

> [!example] 🔬 Глубже / на синьора
> **Реальная цена, измеренная честно.** Сумма 200 миллионов чисел в аккумуляторе `long` против `Long`, с прогревом, JDK 21:
> 
> ```java
> long sum = 0;                    // 72 ms
> Long boxedSum = 0L;              // 100 ms  (?!)
> ```
> 
> Стоп — где обещанный «порядок медленнее»? Его съел escape analysis: JIT (C2) доказал, что коробки не покидают метод, и применил scalar replacement — разложил `Long` по регистрам, не аллоцируя (механика — в 1.3 и 11.9). Теперь выключим EA флагами `-XX:-DoEscapeAnalysis -XX:-EliminateAllocations` и увидим цену без страховки:
> 
> ```java
> long sum:  71 ms
> Long sum:  361 ms   // 5x, и это ещё простой цикл
> ```
> 
> Параллельно `-Xlog:gc` насчитал 1071 young-паузу без EA против 535 с ним — сотни миллионов короткоживущих `Long` в Eden против нуля аллокаций у примитива. Отсюда два честных вывода вместо одного лозунга. Первый: наивный «босканный» цикл на современном JIT **может** быть почти бесплатным. Второй, важный: полагаться на это нельзя — EA сдаётся от малейшего усложнения (объект ушёл в поле, в коллекцию, в невинлайненный вызов, в лямбду), и в проде ты не управляешь тем, что JIT решит доказать. Правильная привычка — hot path пишется на примитивах сразу; EA — подушка безопасности, не контракт.
> 
> **Память.** Из замера в 1.3: `Integer[]` — 20 байт на элемент против 4 у `int[]`, пятикратно, плюс убитая cache locality (каждый элемент — отдельный объект где-то в heap) и миллион объектов на учёте у GC. Для больших числовых наборов существуют primitive collections: в стандартной библиотеке — `IntStream`/`LongStream` (переход из `Stream<Integer>` через `mapToInt`, обратно через `boxed()` — детали в 10.7), сторонние — Eclipse Collections (`IntIntHashMap`, `IntArrayList`) и fastutil. Брать их стоит при >100k элементов в hot path; для сотни элементов `List<Integer>` — нормально, читаемость дороже.
> 
> **Траектория.** Project Valhalla (value classes: типы без identity и header) целится ровно в этот разрыв — `Integer` без коробки. На JDK 21–25 его нет; в проде живём с обёртками и кешем.

> [!warning] ⚠️ Подводные камни
> 
> **`==` на обёртках работает «иногда» — и это худший режим отказа.** Тесты на маленьких числах зелёные (кеш), прод на больших — красный. Поведение, зависящее от величины значения, почти невозможно поймать ревью. Только `equals`/`Objects.equals`.
> 
> **`Map.get` в примитив.** `int n = map.get(key)` — молчаливый NPE на любом отсутствующем ключе. `getOrDefault`, явная проверка на `null`, или оставить `Integer` и обработать отсутствие осознанно.
> 
> **Тернарник со смешанными ветками.** `cond ? wrapperExpr : primitiveExpr` распаковывает обёртку независимо от того, какая ветка выполнится. Если ветки — `Integer` и `Long`, будет unboxing обоих плюс widening к общему числовому типу. Смешал обёртку с примитивом в тернарнике — получил риск NPE.
> 
> **`equals` сам по себе не null-безопасен.** `a.equals(b)` при `a == null` — NPE ещё до всякого сравнения. `Objects.equals(a, b)` (проверено: спокойно возвращает `false` при одном `null`) или константа слева: `Integer.valueOf(5).equals(a)`.
> 
> **Аккумулятор-обёртка в цикле.** `Long sum` в горячем цикле — скрытая фабрика объектов: unbox, сложение, box в новый объект на каждой итерации. Аккумуляторы, счётчики, индексы — всегда примитивы.
> 
> **`Float`/`Double` без кеша — и вообще другой мир сравнения.** `Double x = 1.0; Double y = 1.0; x == y` → `false` всегда (проверено). А смысловое сравнение float-значений имеет свои ловушки (`NaN`, `-0.0`, `equals` vs `==` расходятся) — это территория 15.3.

### Вопросы с собеса (с разбором)

> [!question]- 1. Что напечатает? _(вопрос номер один по этой теме, встречается почти на каждом Java-собесе)_
> ```java
> Integer a = 1000, b = 1000;
> Integer c = 100, d = 100;
> System.out.println(a == b);
> System.out.println(c == d);
> ```
> 
> Ответ: `false`, `true`. Оба присваивания идут через `Integer.valueOf`; `100` — в кеше `-128..127`, обе ссылки указывают на один объект; `1000` — вне кеша, два разных объекта.
> 
> _Follow-up #1_: «Это оптимизация JVM или гарантия языка?» — Гарантия: JLS §5.1.7 требует, чтобы boxing значений `-128..127` возвращал одну и ту же ссылку. Реализация вправе кешировать больше (флаг `-XX:AutoBoxCacheMax=N` поднимает верхнюю границу; нижняя фиксирована), но минимум — обязана.
> 
> _Follow-up #2_: «А с `Long` вместо `Integer`?» — То же самое: `Long.valueOf` кеширует `-128..127` (проверено). Кешируют все целочисленные обёртки, `Character` (`0..127`) и `Boolean`; `Float`/`Double` — нет.
> 
> _Follow-up #3 (ловушка)_: «А `new Integer(100) == Integer.valueOf(100)`?» — `false`: конструктор всегда создаёт новый объект мимо кеша. Он `@Deprecated(since="9", forRemoval=true)`; на JDK 21 ещё компилируется с removal-warning'ом, но в новом коде его быть не должно.

> [!question]- 2. Что напечатает?
> ```java
> static Integer find() { return null; }
> // ...
> int x = find();
> ```
> 
> Ответ: `NullPointerException`. Возврат `Integer` присваивается в `int` — компилятор вставил `.intValue()`, на `null` он падает.
> 
> _Follow-up_: «Где именно падает — в `find` или на присваивании?» — На присваивании: место NPE — точка unboxing, не точка `return null`. `Integer y = find()` прошёл бы без ошибки. Stack trace укажет на строку с `int x = find()`, и helpful NPE (JEP 358) прямо напишет, что null — возвращённое значение `find()`.

> [!question]- 3. Что напечатает?
> ```java
> Integer a = null;
> Integer b = 5;
> System.out.println(a == b);
> System.out.println(a > b);
> ```
> 
> Ответ: `false`, затем NPE. `==` для двух обёрток — сравнение ссылок, unboxing не нужен, `null`-ссылка спокойно не равна ненулевой. А `>` определён только для чисел — unboxing обоих операндов, `a.intValue()` падает.
> 
> _Follow-up #1_: «А `a == 5` (примитив справа)?» — NPE: когда один операнд примитив, `==` становится числовым сравнением (JLS §15.21.1) и распаковывает обёртку. Режим `==` зависит от статических типов операндов.
> 
> _Follow-up #2_: «Как сравнить null-безопасно по значению?» — `Objects.equals(a, b)`. Голый `a.equals(b)` при `a == null` — NPE до всякого сравнения.

> [!question]- 4. Что напечатает и почему это не про рантайм?
> ```java
> Integer wrapped = null;
> Integer result = true ? wrapped : 100;
> ```
> 
> Ответ: NPE. Тип тернарника вычисляется статически по обеим веткам: обёртка + примитив → общий тип примитив (JLS §15.25) → unboxing обеих веток вставлен на компиляции. То, что правая ветка никогда не выполнится, значения не имеет — `wrapped.intValue()` уже в байткоде.
> 
> _Follow-up_: «Как починить, не меняя логики?» — Сделать обе ветки обёртками: `true ? wrapped : Integer.valueOf(100)` — общий тип станет `Integer`, unboxing исчезнет, `result` получит `null` без падения.

> [!question]- 5. Почему версия B медленнее и насколько?
> ```java
> long sum = 0;                                        // A
> for (int i = 0; i < N; i++) sum += i;
> Long sum = 0L;                                       // B
> for (int i = 0; i < N; i++) sum += i;
> ```
> 
> Ответ: в B каждая итерация — unboxing, сложение и boxing в **новый** объект (значения давно вне кеша, а `Long` immutable — мутировать нечего): миллионы аллокаций в Eden и работа для GC против одного регистра в A.
> 
> _Follow-up (на синьора)_: «Замерь — а там разница всего процентов сорок. Почему?» — Escape analysis: JIT доказал, что коробки не покидают метод, и заменил их скалярами (мой замер на JDK 21: 72 vs 100 мс; с выключенным EA — 71 vs 361 мс и вдвое больше young-пауз GC). Сильный ответ разводит два утверждения: EA может спасти простой цикл — и полагаться на это нельзя, потому что EA отваливается от любого усложнения, а управлять им из кода невозможно. Hot path пишется на примитивах.
> 
> _Follow-up #2_: «Когда `Long` в коде оправдан?» — Когда нужен `null` (отсутствие значения), generic-контекст (`List<Long>`, `Map<Long, ...>`), API, требующее объект. В арифметике и аккумуляторах — никогда.

> [!question]- 6. `HashMap<Integer, String>` — поиск ключа идёт по `==` или по `equals`? Кеш влияет?
> Ответ: по `hashCode` (выбор бакета), затем `equals` (подтверждение). У `Integer` оба переопределены по значению (`hashCode` равен самому числу — проверено), поэтому `map.get(1000)` находит ключ независимо от того, из кеша объект или нет. Кеш влияет только на `==`, которым `HashMap` ключи не сравнивает как основным механизмом.
> 
> _Follow-up_: «Тогда зачем вообще знать про кеш, если коллекции работают правильно?» — Затем, что `==` пишут руками: сравнение id, `stream().filter(x -> x == target)`, assert'ы в тестах. Там кеш превращается в баг, зависящий от данных. Плюс это вопрос-фильтр на понимание разницы identity/equality — контракт `equals`/`hashCode` целиком в 3.8.

### Практика

**1. Прощупать границы кеша.** Цикл от `-300` до `300`: `Integer x = i, y = i;` — печатай `i`, где `x != y`. Запусти как есть, потом с `-XX:AutoBoxCacheMax=500`. Убедись: верхняя граница уехала, нижняя — нет.

_Сложность: низкая. Время: 15–20 минут. Что даёт: спецификация JLS §5.1.7, потроганная руками._

**2. Карта боксинга по байткоду.** Класс с пятью случаями: присваивание в обёртку, арифметика на обёртках, тернарник обёртка/примитив, передача `int` в параметр `Integer`, `counter++`. Через `javap -c` найди все `valueOf`/`intValue` и сопоставь каждой строке кода.

_Сложность: низкая. Время: 30 минут. Что даёт: рефлекс видеть невидимые вызовы — включая тернарник, где unboxing обеих веток виден прямо в байткоде._

**3. Цена коробки с EA и без.** Сумма 200M чисел в `long` и `Long` с прогревом. Запусти обычно, потом с `-XX:-DoEscapeAnalysis -XX:-EliminateAllocations`, в обоих случаях добавь `-Xlog:gc` и сравни число young-пауз. Сверь с моими цифрами (72/100 и 71/361 мс).

_Сложность: средняя. Время: 45–60 минут. Что даёт: эмпирика «EA спасает, но не всегда» вместо лозунга; чтение GC-лога. Для честных микробенчмарков дальше — JMH, но и наивный замер с прогревом уже показывает картину._

**4. Зоопарк NPE.** Воспроизведи все точки падения из урока: `Map.get` в `int`, `a > b` с null, тернарник, возврат из метода. Для каждой прочитай helpful-NPE-сообщение и почини своим способом (`getOrDefault`, `Objects.equals`, выравнивание типов веток).

_Сложность: низкая. Время: 20–30 минут. Что даёт: чек-лист опасных мест, прошитый в пальцы._

**5. Primitive collections на вкус.** Подключи `org.eclipse.collections:eclipse-collections`, заполни `IntIntHashMap` и `HashMap<Integer, Integer>` миллионом пар, сравни память (методика замера — из практики 1.3). Ожидаемая разница — в разы.

_Сложность: средняя. Время: ~1 час. Что даёт: понимание, когда тянуть стороннюю библиотеку оправдано._

### На собесе

> [!question]- «В чём разница между `int` и `Integer`?»
> `int` — примитив: 4 байта значения, не бывает `null`, не имеет методов и identity. `Integer` — immutable объект-обёртка в heap (16 байт плюс ссылка): бывает `null`, живёт в generics и коллекциях. Мост между ними — автобоксинг: компилятор молча вставляет `Integer.valueOf` и `intValue()`, и оба направления не бесплатны.
> 
> EN: `int` is a primitive value; `Integer` is an immutable wrapper object with identity that can be `null` and can live in generics. The compiler bridges them silently via `valueOf`/`intValue` — autoboxing and unboxing.

> [!question]- «Что такое `Integer cache`?»
> `Integer.valueOf` для `-128..127` возвращает объекты из заранее созданного массива — одна ссылка на одно значение. Это гарантия JLS §5.1.7, не оптимизация JVM; верхняя граница расширяется флагом `-XX:AutoBoxCacheMax`. Кешируют все целочисленные обёртки, `Character` (0..127) и `Boolean`; `Float`/`Double` — нет. Следствие: `==` на обёртках даёт `true` для маленьких чисел и `false` для больших.
> 
> EN: `Integer.valueOf` returns shared pre-built objects for -128..127 — mandated by JLS §5.1.7, extendable upward via `-XX:AutoBoxCacheMax`. That's why `==` on wrappers "works" for small values and silently breaks for large ones.

> [!question]- «Почему `==` на обёртках — это баг?»
> Для двух обёрток `==` сравнивает ссылки, и результат зависит от кеша — то есть от величины числа. Тесты на маленьких значениях проходят, прод на больших падает. Правильно — `equals` или null-безопасный `Objects.equals`; `==` — только между примитивами. Нюанс: если один операнд примитив, `==` распаковывает обёртку и сравнивает числа — корректно, но с риском NPE.
> 
> EN: With two wrappers `==` compares references, so the result depends on the cache — value-dependent behavior. Use `equals`/`Objects.equals`. With a primitive operand `==` unboxes and compares numerically — correct but NPE-prone.

> [!question]- «Где вылетает NPE при unboxing?»
> В каждой точке, где компилятор вставил `intValue()`: присваивание обёртки в примитив, арифметика и сравнения `<`/`>` на обёртках, тернарник со смешанными ветками (unboxing обеих независимо от условия — тип выражения статический, JLS §15.25), возврат из метода. Главный продовый случай — `Map.get` сразу в `int`: отсутствующий ключ превращается в падение. Лекарства: `getOrDefault`, явная проверка, `Objects.equals`.
> 
> EN: Anywhere the compiler inserts `intValue()`: wrapper-to-primitive assignment, arithmetic and relational operators, mixed-type ternaries (both branches unbox — the expression type is static), and `Map.get` assigned straight to a primitive.

> [!question]- «Что боксинг стоит по перфомансу?»
> Каждый boxing вне кеша — аллокация; обёртка immutable, поэтому аккумулятор-обёртка в цикле создаёт объект на каждой итерации. Мой замер на JDK 21: цикл на `Long` — 361 мс против 71 мс на `long` при выключенном escape analysis; с включённым JIT почти спасает простой цикл (100 мс), но EA — оптимизация, а не контракт, и от усложнения кода отваливается. Hot path — на примитивах; большие числовые коллекции — `IntStream` или primitive collections (Eclipse Collections, fastutil): 4 байта на элемент против 20.
> 
> EN: Each out-of-cache boxing is an allocation, and wrapper accumulators allocate every iteration. Escape analysis may rescue simple loops, but it's an optimization, not a contract — hot paths belong on primitives, large numeric datasets on primitive collections.
