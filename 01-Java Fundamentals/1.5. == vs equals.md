Ты пишешь проверку значения из HTTP-запроса: `if (input == "admin")`. Компилятор молчит, тесты зелёные, ревью пропустило. В проде проверка срабатывает через раз, и локально баг не воспроизводится: результат зависит от того, **каким путём строка попала в программу**. Это худший режим отказа — код, который иногда случайно прав. Причина в том, что `==` в Java значит не то, что в C#, а «иногда true» обеспечивает механизм переиспользования строк внутри JVM. В этом уроке — что именно сравнивает `==` для примитивов и для ссылок, откуда берётся смысловое равенство `equals`, и почему у строк и `Double` на этих операциях живут главные «что напечатает»-вопросы.

**Ментальная модель.** `==` в Java — одна и та же зашитая операция без исключений: для примитивов — сравнение битов значений, для ссылок — сравнение самих ссылок («тот же объект?»). Перегрузки операторов в языке нет, поэтому смысловое равенство всегда выражается методом `equals`.

### `==` для примитивов: биты, и два сюрприза IEEE 754

Для примитивов `==` сравнивает битовые представления значений — ни объектов, ни ссылок:

```java
int a = 5, b = 5;
a == b;   // true — биты совпали
```

Скучно — ровно до floating-point. Проверено на JDK 21:

```java
double nan = 0.0 / 0.0;
nan == nan;        // false (!) — NaN не равен ничему, включая себя
Double.isNaN(nan); // true — культурная проверка (или идиома x != x)

0.0 == -0.0;       // true — а это РАЗНЫЕ битовые паттерны (знаковый бит)
```

Оба сюрприза — не причуды Java, а требования стандарта IEEE 754, которому подчиняются `float`/`double`: `NaN` несравним ни с чем, положительный и отрицательный нули считаются равными оператором, хотя биты различаются. Запомни пару `NaN`/`-0.0` — ниже она перевернётся ровно наоборот в `equals`, и это готовый собесный вопрос.

💡 Из C#: для примитивов мост переносится без изменений — IEEE 754 один на всех, `NaN != NaN` и в .NET. Расхождения начинаются со следующего раздела.

### `==` для ссылок: identity, всегда, без исключений

Из 1.3 ты знаешь: переменная reference-типа хранит ссылку. `==` для двух таких переменных спрашивает одно: «это ссылки на **один и тот же объект**?» Не «одинаковое ли содержимое» — а буквально «тот же ли адресат». Это называется **identity comparison**, в противовес **value comparison**:

```java
String x = new String("hi");
String y = new String("hi");
x == y;        // false — два объекта в heap, две разные ссылки
x.equals(y);   // true  — содержимое одинаковое
```

И вот «всегда»: `==` для ссылок невозможно переопределить, перенастроить или перегрузить. Это операция уровня JVM (в байткоде — инструкция `if_acmpeq`, «compare references»), а не метод.

💡 Из C#: здесь мост ломается жёстче всего, и это ошибка номер один у перешедших с .NET. В C# `==` — перегружаемый оператор: `string` перегружает его на сравнение значений, твой класс может перегрузить как угодно. В Java **перегрузки операторов нет в языке вообще** — привычка «`==` для строк сравнивает текст» превращается в прямой источник багов. Маппинг методов: C# `Equals()` → `equals()`, `GetHashCode()` → `hashCode()`, `ReferenceEquals(a, b)` → в Java это просто `a == b`.

### `equals`: от identity к value

`equals` объявлен в `Object` — корне всех классов, и его дефолтная реализация — буквально `return this == obj`. То есть класс, не переопределивший `equals`, сравнивается по identity, как `==`. Весь смысл переопределения — заменить identity на value: «эти два объекта представляют одно и то же значение?» У `String`, обёрток, `record`-типов это сделано за тебя; у своих классов-значений — твоя работа, и она регулируется контрактом из пяти правил, за нарушение которого молча ломаются коллекции. Полный разбор контракта, `hashCode` и ловушка наследования — глава 3.8; здесь достаточно рабочего правила: **смысловое равенство — только `equals`**.

Практическая деталь, которая понадобится немедленно, — null-безопасность (проверено):

```java
String input = null;
input.equals("admin");            // NPE — метод на null-ссылке
"admin".equals(input);            // false — идиома «константа слева»
Objects.equals(input, "admin");   // false — универсальный null-safe вариант
```

`Objects.equals(a, b)` внутри — `(a == b) || (a != null && a.equals(b))`: сначала identity (заодно ловит «оба null» → `true`), потом защищённая делегация.

### String pool: почему `==` на строках иногда true

Теперь центральный механизм. Прогоняем набор случаев (JDK 21, всё в одном классе):

```java
static final String FIN = "ja";
// ...
String a = "java";
String b = "ja" + "va";        // конкатенация двух констант
String c = new String("java");
String d = c.intern();
String pre = "ja";             // обычная переменная
String e = pre + "va";         // конкатенация с переменной
String f = FIN + "va";         // конкатенация с final-константой

a == b   // true  (!)
a == c   // false
a == d   // true
a == e   // false (!)
a == f   // true  (!)
```

Если `==` — всегда сравнение ссылок, то `true` возможен только в одном случае: переменные указывают на **один объект**. Так и есть, и это не случайность, а **string pool** — внутренняя таблица JVM, где живут уникальные экземпляры строк. Строки иммутабельны (почему — глава 14.1), поэтому два литерала `"java"` в разных местах программы не нуждаются в двух объектах: JVM держит один и раздаёт ссылку всем. Механика:

1. Компилятор складывает строковые литералы в constant pool класс-файла.
2. При первом исполнении инструкции `ldc` (load constant) для литерала JVM ищет строку в пуле: нет — создаёт и регистрирует, есть — возвращает существующую ссылку.
3. Каждый следующий литерал с тем же содержимым получает **ту же ссылку**.

Это ровно та же идея, что `Integer cache` из 1.4 — переиспользование иммутабельных объектов, из-за которого identity «случайно» совпадает, — только без границы 127: пул растёт по мере надобности.

Теперь объясняем каждый случай. `b = "ja" + "va"` — выражение целиком из compile-time констант, и компилятор обязан свернуть его в константу `"java"` ещё при компиляции (constant folding, JLS §15.29). Улика в байткоде (снято `javap -v`): `a` и `b` загружаются **одной и той же** инструкцией `ldc #7 // String java` — к рантайму от конкатенации не остаётся следа.

`e = pre + "va"` — `pre` обычная переменная, её значение в общем случае известно только в рантайме. Компилятор сворачивать не вправе; в байткоде вместо `ldc` стоит `invokedynamic makeConcatWithConstants` — конкатенация выполняется при исполнении (механизм — JEP 280, детали в 14.3) и создаёт **новый** объект мимо пула. Отсюда `false`.

`f = FIN + "va"` — а вот это снова `true`, и это любимый поворот интервьюеров. `FIN` объявлена `static final` с константным инициализатором — по JLS §4.12.4 это **constant variable**, компилятор инлайнит её значение и сворачивает выражение. В класс-файле у `FIN` стоит атрибут `ConstantValue: String ja`, а `f` грузится тем же `ldc #7`. Разница между `e` и `f` — одно слово `final`, а `==` перещёлкивается между `false` и `true`.

`c = new String("java")` — явный приказ создать объект: свежий экземпляр в heap со своей ссылкой, `false` против любого литерала. А `d = c.intern()` — ручной вход в пул: «верни пуловую версию этой строки (добавив её туда, если не было)». `"java"` в пуле есть (литерал `a` позаботился) — `d` получает пуловую ссылку, `true`.

![[Pasted image 20260702103742.png]]

Заметь тонкость с диаграммы: `new String("java")` не обходит пул стороной полностью — литерал-аргумент `"java"` в пуле есть или появится, — но **возвращает** ссылку на свежий объект. Объектов два, `==` их честно различает.

Зачем `intern()` вообще существует? Исторически — дедупликация: парсишь гигантский файл, где `"ACTIVE"` встречается миллион раз, — `intern()` схлопывает миллион копий в одну. Но в современном коде это почти всегда антипаттерн: ручное интернирование ради `==`-сравнений хрупко (одна неинтернированная строка в цепочке — и `==` молча врёт), сам вызов не бесплатен (нативная хеш-таблица `StringTable` внутри JVM), а задачу дедупликации решает GC-флаг `-XX:+UseStringDeduplication` (G1): он схлопывает **массивы символов** под дубликатами строк автоматически, не меняя identity объектов — то есть не ломая семантику `==`.

💡 Из C#: interning и constant folding в .NET есть точно так же (`String.Intern`, компилятор сворачивает `"a" + "b"`). Но ты их никогда не замечал — `==` для `string` в C# сравнивает значения, и интернирована строка или нет, наблюдаемое поведение одинаково. В Java тот же внутренний оптимизатор **протекает наружу** через `==`. Один механизм — разная наблюдаемость; поэтому в Java он источник багов и собесных вопросов, а в C# — сноска.

### `Double`: три семантики сравнения

Обещанный переворот пары `NaN`/`-0.0`. У `Double` три способа сравнить — и на этих двух значениях они дают **разные ответы** (всё проверено на JDK 21):

|Пара|`==` (IEEE 754)|`Double.equals` (биты)|`Double.compare` (порядок)|
|---|---|---|---|
|`NaN`, `NaN`|`false`|`true`|`0` (равны)|
|`0.0`, `-0.0`|`true`|`false`|`1` (`0.0` больше)|

`==` следует IEEE 754 — семантика вычислений. `Double.equals` сравнивает через `doubleToLongBits`: все `NaN` канонизируются в один битовый паттерн (равны), а нули различаются знаковым битом (не равны). `Double.compare` строит **total ordering** — полный порядок, где каждое значение сравнимо с каждым: `NaN` объявлен больше всего (включая `POSITIVE_INFINITY` — проверено, `compare` даёт `1`), а `-0.0 < +0.0`.

Почему три, а не одна? Потому что у них разные заказчики. IEEE-семантика нужна арифметике. А коллекциям нужна детерминированность: если бы `NaN.equals(NaN)` был `false`, ты не смог бы достать из `HashMap` значение по ключу `NaN` (проверено: `HashSet` с `Double.NaN` находит его через `contains` — благодаря `equals`); если бы `compare` следовал IEEE и отказывался упорядочивать `NaN`, сортировка списка с `NaN` была бы недетерминированной. Java выбрала согласовать `compare` с `equals`, а не с `==` — это прямо оговорено в Javadoc `Double.compare`. Наглядное следствие (проверено): `TreeSet` после `add(0.0)` и `add(-0.0)` содержит **оба** элемента — `[-0.0, 0.0]`, для порядка это разные значения.

Практические правила из этого. Проверка на `NaN` — `Double.isNaN(x)`. Сравнение полей-`double` в `compareTo`/`Comparator` — только `Double.compare(a, b)`: популярный трюк `(int)(a - b)` теряет дробную часть и переполняется. А `==` для `double` в бизнес-логике не используется вовсе — даже не из-за `NaN`, а из-за ошибок округления, о которых глава 15.3. Машинерия `Comparable`/`Comparator` целиком — в главе 13.

💡 Из C#: `Double.compare(a, b)` ≈ `a.CompareTo(b)`; `Comparator<T>`/`Comparable<T>` — это `IComparer<T>`/`IComparable<T>` под другими именами, контракт знака идентичен. И казус тот же: `double.NaN.Equals(double.NaN)` в C# — тоже `true` при `NaN != NaN`. Просто в .NET-коде ты редко на это наступал — а на Java-собесе наступишь обязательно.

### 🔬 Глубже / на синьора

**Где живёт пул и почему об этом спрашивают.** До Java 7 string pool физически жил в PermGen — области фиксированного размера для метаданных; агрессивный `intern()` ронял JVM с `OutOfMemoryError: PermGen space`. В Java 7 пул переехал в обычный heap (JDK-6962931): интернированные строки стали собираться GC как все, а лимит пула стал лимитом heap. В Java 8 PermGen удалили целиком, заменив Metaspace (JEP 122) — но пула это уже не касалось. Ответ «в heap с Java 7; до того — PermGen, которого с Java 8 нет» показывает понимание эволюции, а не заученный факт.

**`intern()` с Java 7 не копирует.** Раз пул — в heap, `intern()` для строки, которой в пуле ещё нет, регистрирует **сам этот объект**, а не копию. Следствие проверяемо и контринтуитивно (JDK 21):

```java
String x = new StringBuilder("ур").append("ок").toString();
System.out.println(x.intern() == x);   // true — пул запомнил сам x
System.out.println("урок" == x);       // true (!) — литерал разрешился... в x
```

Вторая строка — вишенка: литералы резолвятся лениво, при первом исполнении `ldc`. К этому моменту `"урок"` в пуле уже есть — это объект `x`, и литерал получает его ссылку. С уже занятым словом трюк не работает: если литерал `"hello"` встретился раньше, `y.intern() == y` для рантайм-собранного `"hello"` даст `false` (проверено) — пул вернёт старожила. Классический вопрос на снятие «зазубрил или понимает».

**Улики в байткоде** (снято `javap -v` на JDK 21). Constant folding: `a`, `b` и `f` — три `ldc` одной записи `#7 = String "java"`; у `final`-переменной в класс-файле атрибут `ConstantValue` — маркер constant variable из JLS §4.12.4. Runtime-конкатенация: `invokedynamic makeConcatWithConstants` через `StringConcatFactory` (JEP 280, Java 9) — не `StringBuilder`-цепочка, как до Java 9; что это меняет — в 14.3. Сам `==` для ссылок — инструкция `if_acmpeq`: одна операция сравнения двух слов, никакого кода не вызывается — потому его и невозможно «переопределить».

**Цена `intern()`.** `StringTable` — нативная хеш-таблица JVM с фиксированным числом бакетов (`-XX:StringTableSize`, по умолчанию 65536); миллионы интернированных строк — деградация до цепочек + давление на GC-паузы (таблица сканируется). Диагностика — `-XX:+PrintStringTableStatistics` на выходе JVM. Ещё одна причина, почему массовый `intern()` — антипаттерн, а дедупликация отдана G1.

### ⚠️ Подводные камни

**`==` на строках компилируется молча и зависит от данных.** Литералы и свёрнутые константы — `true`, всё пришедшее извне (HTTP, файл, БД, `substring`, конкатенация с переменной) — `false`. Тесты на литералах зелёные, прод на реальных данных красный. Только `equals`; IDEA подсвечивает `==` на строках — это чинится, не глушится.

**`equals` на возможном null слева — NPE до сравнения.** `input.equals("admin")` падает при `input == null`. Идиомы: константа слева — `"admin".equals(input)`, или `Objects.equals(input, expected)` когда обе стороны могут быть null.

**Одно слово `final` меняет результат `==`.** `pre + "va"` — новый объект; `final String pre = "ja"` превращает выражение в compile-time константу и возвращает пуловую ссылку. Код, чья identity-семантика зависит от модификатора, — мина; ещё одна причина не завязываться на `==`.

**Полагаться на «я всё интернировал» нельзя.** Одна строка, пришедшая в цепочку сравнения мимо `intern()`, — и `==` молча врёт `false` на равном содержимом. Инвариант «все строки интернированы» не проверяется компилятором и не переживает рефакторинг.

**`NaN` ломает наивные проверки.** `if (value == expected)` никогда не сработает, если оба `NaN`; `values.contains(Double.NaN)` для `List` — сработает (там `equals`). Проверка на `NaN` — только `Double.isNaN`; фильтрация некорректных значений через `x == Double.NaN` — классический мёртвый код.

**Смешение трёх семантик `Double` в одной структуре.** `TreeSet`/`TreeMap` живут по `compare` (`-0.0` и `0.0` — разные элементы), `HashSet`/`HashMap` — по `equals` (тоже разные!), а твой код проверяет `==` (равные). Данные с нулями обоих знаков ведут себя по-разному в разных коллекциях — нормализуй (`+ 0.0` схлопывает `-0.0` в `0.0`) или знай, где какая семантика.

### Вопросы с собеса (с разбором)

**1. Что напечатает?** — самый частый string-вопрос, встречается от джуна до мидла.

```java
String a = "java";
String b = "ja" + "va";
String c = new String("java");
String d = c.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(a == d);
```

Ответ: `true`, `false`, `true`. `b` — конкатенация двух compile-time констант, свёрнута компилятором в литерал `"java"` (JLS §15.29) — та же пуловая ссылка. `c` — явный новый объект. `d` — пуловая версия, а в пуле уже лежит объект литерала `a`.

_Follow-up #1_: «А `String b = pre + "va"`, где `pre = "ja"`?» — `false`: `pre` — переменная, свёртка невозможна, конкатенация уходит в рантайм (`invokedynamic`) и создаёт новый объект. С `final String pre` — снова `true`: constant variable (JLS §4.12.4) инлайнится.

_Follow-up #2_: «`new String("java")` — сколько объектов создаётся?» — Аккуратный ответ: выражение гарантированно создаёт **один** новый объект (сам `new`); литерал `"java"` при этом должен быть в пуле, но был ли он создан именно этой строкой кода — зависит от того, встречался ли раньше. Ответ «всегда два» — та самая зазубренная неточность, которую follow-up и ловит.

---

**2. Почему `==` для строк в Java нельзя «починить», как в C#?**

Ответ: в Java нет перегрузки операторов — в принципе, как фичи языка. `==` для ссылок — это инструкция JVM (`if_acmpeq`), сравнивающая две ссылки; она не диспетчеризуется ни в какой метод, и повлиять на неё из кода невозможно. Смысловое равенство в Java существует только в форме метода — `equals`.

_Follow-up_: «Это осознанное решение или ограничение?» — Осознанное: дизайнеры Java исключили перегрузку операторов ради читаемости (у `+` на пользовательских типах непредсказуемая семантика). Цена — вербозность (`a.equals(b)`, `a.add(b)` для `BigDecimal`) и вот этот класс багов у пришедших из языков с перегрузкой.

---

**3. Что напечатает?**

```java
System.out.println(Double.NaN == Double.NaN);
System.out.println(Double.valueOf(Double.NaN).equals(Double.NaN));
System.out.println(0.0 == -0.0);
System.out.println(Double.valueOf(0.0).equals(-0.0));
```

Ответ: `false`, `true`, `true`, `false` — пара `NaN`/`-0.0` перевёрнута между `==` и `equals`. `==` следует IEEE 754; `Double.equals` сравнивает канонизированные биты (`doubleToLongBits`): все `NaN` равны, нули различны.

_Follow-up_: «Зачем `equals` сделали несовместимым с `==`?» — Чтобы `Double` работал ключом хеш-коллекций: с IEEE-семантикой `NaN.equals(NaN) == false` положенное по ключу `NaN` значение было бы невозможно достать. Javadoc `Double.equals` оговаривает отступление явно. Проверяемо: `HashSet` находит добавленный `Double.NaN`.

---

**4. Чем `Double.compare` отличается от обоих, и что он вернёт для `(0.0, -0.0)` и `(NaN, +∞)`?**

Ответ: `1` и `1`. `compare` реализует total ordering — каждое значение сравнимо с каждым, иначе сортировка и `TreeMap` недетерминированы. Для этого приняты два волевых решения: `NaN` больше всего, `-0.0 < +0.0`. Согласован `compare` с `equals` (`compare(a,b) == 0` ⟺ `equals`), не с `==`.

_Follow-up_: «Как правильно сравнить поле `double` в `compareTo`?» — `Double.compare(this.price, other.price)`. Антипаттерн `(int)(a - b)` теряет дробную часть (0.5 − 0.1 → 0 — «равны») и переполняется на больших разницах.

---

**5. Где живёт string pool и почему это менялось?**

Ответ: с Java 7 — в heap; до этого — в PermGen, области фиксированного размера, где массовый `intern()` вызывал `OutOfMemoryError: PermGen space`. После переезда интернированные строки собираются GC как обычные объекты. В Java 8 PermGen удалён целиком в пользу Metaspace (JEP 122) — но пул к тому моменту уже жил в heap.

_Follow-up (на синьора)_: «Что вернёт `x.intern() == x`, если `x` собрана в рантайме?» — Зависит от того, была ли строка в пуле: не была — с Java 7 пул регистрирует сам объект `x`, и это `true`; была (например, есть такой литерал) — вернётся старожил, `false`. Проверено на JDK 21, включая эффектное следствие: литерал, исполненный после такого `intern()`, резолвится в объект `x`.

---

**6. Сравни `Objects.equals(a, b)` и `a.equals(b)` — когда что?**

Ответ: `a.equals(b)` падает NPE при `a == null`. `Objects.equals` null-безопасен: сначала `a == b` (ловит «оба null» → `true` и «тот же объект»), затем `a != null && a.equals(b)`. Для сравнения с константой достаточно идиомы «константа слева»: `"admin".equals(input)`.

_Follow-up_: «А `Objects.equals(null, null)` — это `true` семантически осмысленно?» — Для сравнения полей в `equals` классов-значений — да: два объекта с полем `name == null` в остальном равные должны быть равны. Именно поэтому `Objects.equals` — стандартный кирпич внутри переопределённых `equals` (увидишь в 3.8).

### На собесе

**«Что сравнивает `==`?»**

> Для примитивов — биты значений, с двумя IEEE-оговорками: `NaN != NaN` и `0.0 == -0.0`. Для ссылок — всегда identity: «тот же ли это объект», без исключений и без возможности переопределить — перегрузки операторов в Java нет. Смысловое равенство — только `equals`; для строк `==` — это баг, даже когда он случайно даёт `true`.

> EN: For primitives `==` compares value bits (with IEEE quirks: `NaN != NaN`, `0.0 == -0.0`). For references it's always identity — same object or not. Java has no operator overloading, so semantic equality is always `equals`.

**«Почему `==` на строках иногда true?»**

> Из-за string pool: литералы интернируются автоматически, равные литералы дают одну ссылку. Компилятор сворачивает константные выражения — `"ja" + "va"` становится тем же пуловым `"java"` (JLS §15.29), и `final`-переменная с константным инициализатором участвует в свёртке. Конкатенация с обычной переменной уходит в рантайм и создаёт новый объект; `new String` — всегда новый объект. То есть `==` на строках отвечает не «равны ли», а «одним ли путём созданы».

> EN: String literals are pooled, and the compiler folds constant expressions into pooled literals — so `==` reflects how the string was created, not what it contains. Runtime concatenation and `new String` produce fresh objects.

**«Что такое `intern()` и когда его использовать?»**

> Ручной вход в пул: возвращает пуловую версию строки, регистрируя её при отсутствии (с Java 7 — сам объект, без копии). Почти всегда антипаттерн: `==`-сравнения на интернированных строках хрупки, `StringTable` не бесплатна. Дедупликацию памяти решает `-XX:+UseStringDeduplication` на G1 — прозрачно и без изменения identity. Пул живёт в heap с Java 7; до того — PermGen, удалённый в Java 8 (JEP 122).

> EN: `intern()` returns the pooled instance (since Java 7 it may register the object itself, no copy). It's mostly an antipattern — G1's string deduplication solves the memory problem transparently. The pool lives in the heap since Java 7; PermGen was removed in Java 8 (JEP 122).

**«`==`, `equals`, `compare` для `Double` — в чём разница?»**

> Три семантики, расходящиеся на `NaN` и нулях. `==` — IEEE 754: `NaN != NaN`, нули равны. `equals` — канонизированные биты: все `NaN` равны, нули различны — иначе `Double` не работал бы ключом `HashMap`. `compare` — total ordering для сортировки и `TreeMap`: `NaN` больше всего, `-0.0 < +0.0`; согласован с `equals`, не с `==`. В `compareTo` для `double`-полей — всегда `Double.compare`, никогда `(int)(a - b)`.

> EN: Three semantics diverging on `NaN` and zeros: `==` follows IEEE 754, `equals` compares canonical bits (all `NaN`s equal — required for hash keys), `compare` builds a total ordering (`NaN` greatest, `-0.0 < +0.0`), consistent with `equals`, not `==`.

**«Как сравнивать строки null-безопасно?»**

> Константа слева — `"admin".equals(input)` — или `Objects.equals(a, b)`, который сначала проверяет `a == b` (ловя «оба null»), потом делегирует с null-защитой. Голый `a.equals(b)` — NPE при `a == null` ещё до сравнения.

> EN: Put the constant on the left — `"admin".equals(input)` — or use `Objects.equals(a, b)`; a bare `a.equals(b)` throws NPE when `a` is null.
