
Первые дни на Java после .NET проходят спокойно ровно до момента, когда ты по привычке пишешь `42.toString()` или объявляешь `List<int>` — и компилятор отказывает в обоих случаях. Причина не в синтаксисе, а в устройстве системы типов: Java делит все типы на два несмешивающихся мира, и `int` живёт не в том, где живут методы, коллекции и `null`. Этот урок — про то, как устроен мир примитивов, чем он отличается от знакомых тебе value types из .NET, и где эта разница будет цеплять тебя в реальном коде.

**Мысленная модель:** примитив в Java — это голое значение фиксированного размера, зашитое в спецификацию языка и живущее _вне_ иерархии `Object`: без методов, без `null`, без наследования. Это не «облегчённый объект» и не `struct` — это отдельная категория сущностей в системе типов.

### Два мира: примитивы и ссылочные типы

В Java ровно восемь примитивных типов, и они определены в самой спецификации языка, а не в стандартной библиотеке. У примитива нет класса, нет суперкласса, нет методов. Переменная примитивного типа хранит само значение, а не ссылку на объект в heap.

Проще всего увидеть границу между мирами через код, который компилятор отвергает. Всё проверено на JDK 21:

```java
int x = 42;
String s = x.toString();
// error: int cannot be dereferenced

List<int> list = null;
// error: unexpected type
//   required: reference
//   found:    int
```

`int cannot be dereferenced` — буквально «у `int` нельзя разыменовать ссылку», потому что ссылки нет: `x` — это 4 байта со значением 42, а не указатель на объект. А дженерики (`List<T>`) работают только с reference-типами — компилятор прямо говорит `required: reference`. Почему так — увидишь в главе 7 (type erasure).

При этом вот это компилируется:

```java
Object obj = 42;
System.out.println(obj.getClass());   // class java.lang.Integer
```

Никакой магии: компилятор молча вставил autoboxing — превратил `int` в объект класса-обёртки `Integer`. Для каждого примитива обёртка есть: `Integer`, `Long`, `Short`, `Byte`, `Character`, `Float`, `Double`, `Boolean`. Они нужны везде, где требуется объект: коллекции, дженерики, рефлексия. Как autoboxing работает внутри и где его главная ловушка (`Integer` cache) — целиком урок 1.4; пока достаточно знать, что мост между мирами существует и что он не бесплатный.

Пара деталей к схеме. `Number` — абстрактный класс (не интерфейс), задающий `intValue()`, `longValue()`, `doubleValue()` и т.д.; его же наследуют `BigInteger` и `BigDecimal` — на схеме их нет, потому что у них нет парного примитива. `Character` и `Boolean` наследуют `Object` напрямую: `char` технически хранит число, но семантически это не число, а `boolean` вообще не про арифметику.

💡 **Из C#:** в .NET единая иерархия — `int` это alias для `System.Int32`, `struct`, наследует `ValueType` → `Object`, реализует `IComparable<int>`, и `42.ToString()` — легальный код. Разделение value/reference там про _семантику копирования и размещения_, но оба мира живут внутри одной ОО-системы. В Java разделение жёстче: примитивы полностью _вне_ ОО-системы. Никакого общего корня, никакого `ValueType`, никаких интерфейсов у `int`. Wrapper-классы — это не «`int` и есть объект», а отдельные объекты-коробки, придуманные, чтобы примитивы могли хотя бы через посредника попадать в ОО-мир.

⚠️ **Подводный камень:** привычка из C# «всё есть объект» будет подсовывать тебе `Integer` там, где нужен `int` — в полях DTO, в сигнатурах методов. Это не эквивалентная замена: `Integer` может быть `null`, живёт в heap с накладными расходами и сравнивается через `==` не так, как ты ожидаешь. Правило до урока 1.4: примитив по умолчанию, обёртка — только когда без объекта нельзя.

### Восемь типов: размеры, диапазоны, дефолты

Каноническая таблица — она живёт в этом уроке, дальше по книге я буду просто на неё ссылаться. Все границы проверены через `MIN_VALUE`/`MAX_VALUE` на JDK 21:

|Тип|Размер|Диапазон|Default (поля)|
|---|---|---|---|
|`byte`|8 бит|−128 … 127|`0`|
|`short`|16 бит|−32 768 … 32 767|`0`|
|`int`|32 бита|−2³¹ … 2³¹−1 (≈ ±2,1 млрд)|`0`|
|`long`|64 бита|−2⁶³ … 2⁶³−1|`0L`|
|`float`|32 бита|IEEE 754, ~7 значащих цифр|`0.0f`|
|`double`|64 бита|IEEE 754, ~15–16 значащих цифр|`0.0d`|
|`char`|16 бит|0 … 65 535 (беззнаковый)|`'\u0000'`|
|`boolean`|не задан JLS|`true` / `false`|`false`|

Три структурных факта, которые важнее самих цифр.

**Все целочисленные типы — знаковые.** `byte` — это −128…127, а не 0…255. Беззнаковых типов в Java нет — это осознанное решение Гослинга ещё из 1995 года: он считал unsigned-арифметику источником багов. Единственное исключение — `char` (о нём ниже). С Java 8 появились методы для _трактовки_ знакового `int`/`long` как беззнакового на уровне операций — `Integer.toUnsignedString()`, `Integer.divideUnsigned()`, `Integer.compareUnsigned()` — но отдельного типа нет.

**Размеры фиксированы спецификацией.** `int` — это 32 бита на любой платформе, всегда. Никаких платформозависимых размеров как у `nint` в C# или `int` в C.

**`boolean` — единственный без гарантированного размера.** JLS намеренно оставляет представление на усмотрение JVM; на практике HotSpot держит `boolean` как 32-битный слот в локальных переменных и как байт в массивах. Детали — в 🔬-блоке.

#### Литералы и суффиксы

Компилятор трактует целочисленный литерал как `int`, а дробный — как `double`, и отсюда два обязательных суффикса:

```java
long ok    = 3_000_000_000L;   // суффикс L — литерал становится long
long wrong = 3_000_000_000;
// error: integer number too large  ← литерал парсится как int и не влезает

float f  = 3.14f;              // суффикс f
float f2 = 3.14;
// error: incompatible types: possible lossy conversion from double to float
```

Пиши `L` заглавной: строчная `l` в большинстве шрифтов неотличима от `1`. Подчёркивания в литералах (`3_000_000_000L`, `0xFF_FF`) легальны с Java 7 и сильно помогают читать большие числа.

Есть одно послабление, которое стоит понять сразу, потому что оно выглядит как нарушение правил. Присваивание «меньшему» типу обычно требует явного каста, но для _констант времени компиляции_, влезающих в диапазон, компилятор делает narrowing сам:

```java
byte b = 100;      // ок: константа 100 влезает в byte
char c = 66;       // ок: 66 влезает в char → 'B'

byte b2 = 130;     // error: possible lossy conversion from int to byte
char c2 = 70000;   // error: possible lossy conversion from int to char
```

Компилятор вычисляет константу, проверяет диапазон — влезает, значит безопасно. Работает только с литералами и `final`-константами, известными на этапе компиляции; с обычной переменной `int` тот же код не соберётся.

💡 **Из C#:** таблица размеров почти совпадает с .NET, но три расхождения будут бить по рукам при портировании. Первое: `byte` в C# беззнаковый (0…255), а знаковый аналог — `sbyte`; в Java `byte` знаковый, и код разбора бинарных протоколов, где C#-версия спокойно писала `byte checksum = 200`, в Java потребует переосмысления (обычно — работы через `int` с маской `& 0xFF`). Второе: семейства `uint`/`ulong`/`ushort` нет совсем — беззнаковый 32-битный счётчик из протокола ты хранишь в `long` или трактуешь `int` через `*Unsigned`-методы. Третье: `decimal` в Java не существует как тип — для точной десятичной арифметики есть класс `BigDecimal`, многословный и без перегруженных операторов. Почему `double` не годится для денег и как жить с `BigDecimal` — глава 15.

Поведенческий слой чисел — переполнение, `Math.addExact`, IEEE 754, widening/narrowing при операциях и numeric promotion — намеренно не здесь: это глава 15. Тут запомни только категориальный факт: переполнение `int` в Java _молчаливое_, по кольцу two's complement, без исключений.

### `char` — самый странный из восьми

В .NET ты относился к `char` как к символу, у которого случайно есть числовой код. В Java честнее думать наоборот: `char` — это **16-битное беззнаковое целое** (`JLS ` относит его к integral types), к которому приклеена семантика символа. Это единственный беззнаковый тип в языке — диапазон 0…65 535, потому что отрицательных кодов у символов не бывает.

Интегральная природа означает, что арифметика работает прямо на `char`, без конверсий:

```java
char c = 'A';
int code = c;                     // 65 — неявное расширение, каст не нужен

char next = (char) (c + 1);       // 'B' — а вот обратно каст нужен
System.out.println('a' + 1);      // 98 — int, не строка "a1"

char digit = '7';
int value = digit - '0';          // 7 — классический трюк парсинга цифры
```

Обрати внимание на вторую строку: результат `c + 1` — это `int`, и присвоить его обратно в `char` без каста нельзя:

```java
char a = 'a', b = 'b';
char sum = a + b;
// error: incompatible types: possible lossy conversion from int to char
int sum2 = a + b;   // 195 — ок
```

Любая арифметика над `char` (как и над `byte`/`short`) выполняется в `int` — это binary numeric promotion, механика которого целиком разобрана в главе 15. Здесь важен только видимый эффект: `char + char` даёт `int`.

#### `char` — это code unit, не символ

`char` хранит одну единицу UTF-16 (code unit). Символы базовой многоязыковой плоскости — латиница, кириллица, большинство иероглифов — умещаются в один `char`. Эмодзи и редкие символы за пределами BMP — нет: они кодируются суррогатной парой из двух `char`. Прямое следствие:

```java
String emoji = "😀";
System.out.println(emoji.length());                          // 2
System.out.println(emoji.codePointCount(0, emoji.length())); // 1
```

`length()` считает code units, и `charAt(i)` вернёт тебе _половину_ эмодзи — сломанный суррогат. Для работы с настоящими символами есть `codePointAt()` и стрим `codePoints()`. Глубокая работа с Unicode в строках — глава 14; на этом уроке достаточно держать в голове само различие «code unit ≠ символ».

Статические утилиты живут в обёртке `Character`, и они знают весь Unicode, не только ASCII:

```java
Character.isDigit('7');            // true
Character.isLetter('я');           // true — кириллица тоже буква
Character.isDigit('٧');            // true — арабская цифра семь
Character.toUpperCase('a');        // 'A'
Character.getNumericValue('7');    // 7
```

💡 **Из C#:** технически `char` в .NET устроен так же — 16 бит, UTF-16 code unit, беззнаковый, `"😀".Length == 2`, та же проблема суррогатных пар. Разница культурная: в C# `char` — это `struct` с методами и статикой (`char.IsLetter('A')`), и как число его используют редко. В Java `char` — примитив без методов (утилиты — в `Character`), и его числовая природа используется в коде постоянно: индексация, сдвиги, парсинг цифр через `- '0'`.

### Дефолтные значения: поля — да, локальные — нет

Правило простое, но на нём стабильно ловят: автоматическую инициализацию получают только **поля** (instance и static). Локальные переменные не инициализируются вообще, и компилятор запрещает чтение до присваивания:

```java
class Counter {
    int count;        // 0
    boolean active;   // false
    char marker;      // '\u0000'
    String name;      // null — reference-тип

    void method() {
        int local;
        System.out.println(local);
        // error: variable local might not have been initialized
    }
}
```

Полный список дефолтов задан в `JLS §4.12.5`: все целочисленные (включая `char`) — ноль, `float`/`double` — `+0.0`, `boolean` — `false`, любой reference-тип — `null`. Для `char` «ноль» — это `'\u0000'`, null character: при печати он невидим (не пробел!), и проверка `if (c == ' ')` его не поймает — я прогнал `System.out.println("[" + defChar + "]")` на JDK 21 и получил скобки с невидимым байтом `\u0000` между ними.

💡 **Из C#:** здесь миры совпадают — поля класса в .NET тоже получают `default(T)`, а definite assignment для локальных переменных работает так же. Одно из немногих мест урока, где интуицию переносить можно без правок.

### 🔬 Глубже / на синьора

**Реальный размер `boolean`.** JLS определяет для `boolean` только множество значений, но не представление. В HotSpot локальная переменная `boolean` занимает полный 32-битный слот стека (JVM оперирует `boolean` через `int`-инструкции — в байткоде ты увидишь `iconst_0`/`iconst_1` и `iload`), а элемент `boolean[]` — один байт (инструкции `baload`/`bastore` общие с `byte[]`). То есть `boolean[1_000_000]` — это мегабайт, а не 125 КБ. Если нужна настоящая битовая упаковка — `java.util.BitSet`.

**`strictfp` мёртв.** Исторически промежуточные float-вычисления на x87 FPU могли идти в 80-битной точности, и результаты чуть плавали между платформами; `strictfp` принуждал к строгому IEEE 754. С Java 17  _все_ операции с плавающей точкой всегда strict, ключевое слово стало no-op и оставлено для совместимости. На собеседовании достаточно одной фразы: «знаю, что было; с 17-й всегда strict».

**Project Valhalla.** Многолетняя попытка добавить в JVM настоящие value-типы — «value classes», объекты без идентичности, которые JVM сможет расплющивать в память как примитивы. Это закрыло бы главный гэп с .NET-`struct` и убрало бы цену боксинга в коллекциях. К Java 21 (и на момент написания — к Java 25) до стандарта не доехало. Для собеседования хватает одного предложения: «Valhalla — value types для JVM, ещё не в релизе».

**Почему у примитивов вообще нет методов.** Это не лень дизайнеров, а прямое следствие модели вызова: вызов метода в JVM диспетчеризуется через заголовок объекта (class pointer), а у примитива заголовка нет — он и есть только биты значения. Синтаксис вида `42.toString()` потребовал бы либо автобоксинга на каждый вызов (скрытая аллокация), либо спецобработки в компиляторе для каждого примитива. Java выбрала честную границу: хочешь методы — бери объект.

### ⚠️ Подводные камни

**Невидимый `'\u0000'`.** Дефолтный `char` в неинициализированном поле печатается как «ничего» и не равен пробелу. Классический симптом: конкатенация с таким полем даёт строку, которая _выглядит_ нормально, но `length()` и `equals()` ведут себя «необъяснимо».

**`long`-литерал без `L`.** `long x = 3_000_000_000;` не компилируется — литерал сначала парсится как `int`. Ошибка ловится компилятором, но регулярно съедает время у людей после C#, где `long y = 3_000_000_000;` работает (компилятор C# сам подбирает тип литерала по контексту).

**Знаковый `byte` при портировании.** Байт `0xC8` (200 в C#) в Java — это `-56`. Любой код, читающий бинарные данные, — checksums, сетевые пакеты, кодировки — при прямом переносе с .NET тихо ломается на значениях > 127. Идиома: расширить до `int` через `b & 0xFF`.

**Арифметика над `char`/`byte`/`short` даёт `int`.** `char sum = a + b` и `byte s = b1 + b2` не компилируются, даже когда «очевидно влезает». Механика (numeric promotion и скрытый каст в `+=`) — глава 15, но спотыкаться об это ты начнёшь уже сейчас.

**`float f = 0.5;` не компилируется, а `double d = 0.5f;` — да.** Литерал по умолчанию `double`; сужение требует суффикса `f`, расширение `float → double` проходит молча. Симметрии нет — направление конверсии решает всё.

### Практика

**1. Восемь дефолтов** — easy, ~10 мин. Класс с полями всех восьми примитивов плюс `String`, ничего не инициализируй, распечатай все поля. Отдельно распечатай `char`-поле в скобках `"[" + c + "]"` и сравни его с `' '` и с `'\u0000'`. Ничего кроме этого урока не нужно.

**2. Коллекция ошибок компилятора** — easy, ~15 мин. Собери в один файл и по очереди раскомментируй: `long` без `L` на 3 миллиардах, `float f = 3.14;`, `byte b = 130;`, `char c = 70000;`, `int x = 42; x.toString();`, чтение неинициализированной локальной. Запиши формулировку каждой ошибки — на ревью и в собес-задачках «что не скомпилируется» ты будешь узнавать их по первым словам.

**3. Шифр Цезаря на `char`-арифметике** — easy, ~20 мин. Метод, сдвигающий каждую латинскую букву строки на N позиций с заворотом (`'z' + 1 → 'a'`). Используй `char`-арифметику и трюк `c - 'a'`. Понадобятся `String.charAt()` и `StringBuilder`.

**4. Code units против code points** — easy-medium, ~20 мин. Возьми строку `"Hi 👋 мир 🌍"`. Сравни `length()` и `codePointCount()`. Пройдись `chars()` и `codePoints()` стримами, печатая каждое значение в hex, и найди суррогатные пары глазами (диапазон `0xD800–0xDFFF`).

**5. Граница миров на ощупь** — medium, ~25 мин. Напиши код, демонстрирующий границу примитив/объект: `((Object) 42).getClass()`, `((Object) 'x').getClass()`, попытка `List<int>` (в комментарии — текст ошибки), `List<Integer>` с добавлением `int`-литерала. Затем проверь `Character.isLetter()` / `isDigit()` на `'я'`, `'٧'`, `'7'`, `'㍿'` и сформулируй в две строки, почему утилиты живут в `Character`, а не на самом `char`.

### На собесе

Примитивы — разогревочная тема: через неё проверяют, понимаешь ли ты систему типов Java вообще, и быстро уходят на follow-up в autoboxing, числа или строки. Вопросы ниже — в порядке частоты.

**Q1. Чем примитивные типы отличаются от ссылочных?** — самый частый, открывает почти любой Java-собес.  
Модель ответа (RU): «Примитивов восемь, они зашиты в спецификацию языка: хранятся по значению, не являются объектами, не имеют методов, не могут быть `null` и не входят в иерархию `Object`. Reference-типы — объекты в heap, переменная хранит ссылку, копируется при передаче именно ссылка».  
Model answer (EN): "Java has exactly eight primitives defined by the language spec: they're stored by value, aren't objects, have no methods, can never be `null`, and live outside the `Object` hierarchy. Reference types are heap objects; variables hold references, and it's the reference that gets copied around."  
Follow-up: «Почему нельзя `List<int>`?» → дженерики работают только с reference-типами из-за type erasure (глава 7); мостом служат обёртки и autoboxing. Ошибка слабых ответов — «примитивы лежат на стеке, объекты в куче»: примитивное _поле объекта_ лежит в heap внутри этого объекта, привязка «примитив = стек» неверна.

**Q2. `int` vs `Integer` — в чём разница?**  
RU: «`int` — примитив: 4 байта значения, дефолт `0`, не бывает `null`. `Integer` — класс-обёртка: объект в heap с заголовком, может быть `null`, имеет методы, работает в дженериках. Между ними — autoboxing/unboxing».  
EN: "`int` is a primitive — four bytes of raw value, defaults to `0`, can't be `null`. `Integer` is a wrapper class — a heap object with a header, nullable, has methods, works with generics. Autoboxing/unboxing converts between them."  
Follow-up trap: «`Integer x = null; int y = x;` — что произойдёт?» → `NullPointerException` при unboxing в рантайме, компилятор молчит. Классическая боль legacy-кода с непроинициализированными `Integer`-полями. Дальше почти наверняка спросят про `Integer` cache и `==` — это коронный вопрос урока 1.4, там же follow-up-дерево.

**Q3. Сколько примитивов в Java? Перечисли.**  
RU/EN: восемь — `byte`, `short`, `int`, `long`, `float`, `double`, `char`, `boolean`. Звучит тривиально, но кандидаты стабильно забывают `char` или называют `String`. Назвал `String` — собес по типам закончился, не начавшись: это reference-тип.

**Q4. Есть ли в Java беззнаковые целые?**  
RU: «Как отдельных типов — нет, все целочисленные знаковые. Единственное беззнаковое — `char`, 16 бит, 0…65535. С Java 8 есть методы для беззнаковой трактовки знаковых: `Integer.toUnsignedString`, `divideUnsigned`, `compareUnsigned`».  
EN: "No unsigned integer types — all integral types are signed. The only unsigned one is `char`: 16 bits, 0 to 65535. Since Java 8 there are methods to treat signed values as unsigned: `Integer.toUnsignedString`, `divideUnsigned`, `compareUnsigned`."  
Follow-up: «А `byte` — какой у него диапазон?» — ловят пришедших из C/C#: −128…127, не 0…255.

**Q5. Дефолтное значение `boolean`-поля? А локальной переменной?**  
Trap-вопрос целиком. RU: «Поле — `false` автоматически. У локальной переменной дефолта нет: чтение до инициализации — ошибка компиляции». EN: "A field defaults to `false`. A local variable has no default — reading it before assignment is a compile error." Кто отвечает «`false`» на обе части — попался; интервьюер обычно уточняет специально.

**Q6. Что особенного в `char`?**  
RU: «`char` — интегральный примитив: 16-битный беззнаковый, единственный беззнаковый тип в Java, арифметика работает напрямую, `'A' + 1` даёт `int` 66. Хранит UTF-16 code unit, а не символ: эмодзи занимает два `char`, поэтому `"😀".length() == 2`».  
EN: "`char` is an integral primitive: 16-bit unsigned — the only unsigned type in Java — and supports arithmetic directly; `'A' + 1` yields `int` 66. It stores a UTF-16 code unit, not a character: an emoji takes two `char`s, so `"😀".length()` is 2."  
Follow-up: «Как тогда правильно посчитать символы?» → `codePointCount()` / `codePoints()`. Senior-надбавка: упомянуть, что арифметика над `char` идёт в `int` из-за numeric promotion.

**Устная шпаргалка (30 секунд):** в Java два мира типов. Восемь примитивов — голые значения фиксированного размера вне иерархии `Object`: без методов, без `null`, все целые знаковые, единственный беззнаковый — `char`, он же интегральный UTF-16 code unit. Поля получают дефолты (нули, `false`, `null`), локальные переменные — нет. Мост в мир объектов — wrapper-классы и autoboxing; настоящих value types нет, `decimal` нет, unsigned-семейства нет — это три главных отличия от .NET.
