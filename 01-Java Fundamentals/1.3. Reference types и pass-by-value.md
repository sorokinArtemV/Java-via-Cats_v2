Ты пишешь сервисный метод: принимаешь `List<Order>`, обогащаешь заказы данными, что-то фильтруешь. Иногда изменения видны вызывающему коду, иногда — бесследно исчезают, и по коду не сразу понятно, почему. А на собесе тебя в первые десять минут спросят: «Java передаёт аргументы по значению или по ссылке?» — и это вопрос-фильтр: по ответу видно, понимает ли человек, что вообще лежит в переменной. Этот урок закрывает модель раз и навсегда: что такое ссылка, чем она отличается от объекта, и что именно копируется при присваивании и вызове метода.

**Ментальная модель.** Переменная в Java — это ячейка, в которой лежит либо примитив, либо ссылка на объект в heap. Всё, что Java делает с переменными — присваивание, передача в метод, возврат — это копирование содержимого ячейки. Объект не копируется и не передаётся никогда; передаётся только ссылка на него.

### Два мира: что лежит в переменной

Из урока 1.2 ты знаешь восемь примитивов — они живут как голые значения: `int x = 42` означает, что в ячейке `x` лежат буквально эти 4 байта. Всё остальное — `String`, `ArrayList`, массивы, твой класс `Order` — reference types. Объект такого типа всегда живёт в heap, а в переменной лежит только ссылка на него:

```java
int x = 42;                      // в ячейке — само значение
List<String> items = new ArrayList<>();  // в ячейке — ссылка; объект в heap
```

Ссылка — это, по сути, указатель: машинный адрес объекта (или его сжатое представление — об этом в 🔬). Но указатель кастрированный, и это осознанное дизайн-решение: нельзя сделать арифметику над ссылкой, нельзя взять адрес переменной, нельзя получить ссылку «внутрь» объекта. GC свободно перемещает объекты по heap и прозрачно обновляет все ссылки — именно потому, что язык не даёт тебе увидеть «сырой» адрес.

💡 Из C#: интуиция value type vs reference type переносится напрямую — «значение само по себе» против «ссылка на объект в куче». Два отличия. Первое: в C# ты можешь объявить свой value type через `struct`; в Java список value-типов — это восемь примитивов, hard-coded, точка (Project Valhalla обещает value classes, но в проде их нет). Второе: в Java нет `unsafe`, нет `fixed`, нет интерьерных указателей — ссылку нельзя «пришпилить» и нельзя разглядывать как число.

### Ссылка ≠ объект: aliasing

Ключевое следствие: присваивание копирует ссылку, а не объект. Две переменные легко указывают на один и тот же объект — это называется aliasing:

```java
List<String> items = new ArrayList<>(List.of("book"));
Order a = new Order("A-1", items);
Order b = a;                    // скопировали ссылку; объект один

System.out.println(a == b);     // true — одна и та же ссылка

items.add("pen");               // мутируем через ТРЕТЬЮ ссылку
System.out.println(a.items());  // [book, pen] — видно через все ссылки
```

Проверено на JDK 21: `true`, потом `[book, pen]`. Никакого копирования не произошло ни на одном шаге — `Order b = a` стоит ровно столько же, сколько `int y = x`: копирование одной ячейки.

![[Pasted image 20260702103159.png]]

Сразу зафиксируем следствие для сравнения: `==` для reference-типов сравнивает ссылки — «это буквально один объект?». Смысловое равенство — это `equals`:

```java
Order c = new Order("A-1", a.items());
System.out.println(a == c);        // false — другой объект в heap
System.out.println(a.equals(c));   // true — record сравнивает по значениям
```

Пока держим только это правило: `==` на объектах — идентичность, `equals` — смысл. Почему со `String` оператор `==` «иногда работает» и почему это ловушка — целиком урок 1.5; контракт `equals`/`hashCode` — 3.8.

### null: ссылка «в никуда»

Раз в ячейке лежит ссылка, у неё есть особое значение — `null`: «не указывает ни на что». Правила, где `null` появляется сам:

```java
public class Defaults {
    static String staticRef;                // поле: дефолт null
    String instanceRef;                     // поле: дефолт null

    public static void main(String[] args) {
        String[] arr = new String[3];
        System.out.println(arr[0]);         // null — ячейки массива тоже
        String local;
        // System.out.println(local);       // не скомпилируется:
        // error: variable local might not have been initialized
    }
}
```

Поля и ячейки массивов молча инициализируются `null`. Локальные переменные — нет: компилятор требует definite assignment и не даст прочитать неинициализированную локальную (проверено — ошибка компиляции). Это несимметрично, и это частый источник NPE «издалека»: поле забыли проинициализировать в конструкторе, а взорвалось через три вызова в другом классе.

С Java 15 JVM по умолчанию говорит, **какое именно звено** было `null` (helpful NullPointerException, JEP 358). Проверено на JDK 21:

```java
User u = new User(new Address(null));
u.address().city().name().length();
// Exception in thread "main" java.lang.NullPointerException:
//   Cannot invoke "City.name()" because the return value of "Address.city()" is null
```

До Java 15 ты получал голое `NullPointerException` со строкой — и гадал, что из четырёх разыменований в цепочке было null. Теперь JVM разбирает байткод падения и называет виновника. Отключается флагом `-XX:-ShowCodeDetailsInExceptionMessages` (по умолчанию включено).

💡 Из C#: `null` и `NullReferenceException` — прямой аналог. Но в C# 8+ есть nullable reference types (`string?`) и компилятор ворчит на разыменование потенциального null. В Java **ничего подобного на уровне языка нет**: `String` всегда может быть `null`, и компилятор молчит. Индустрия компенсирует аннотациями `@Nullable`/`@NonNull` и статическим анализом — подробно в 20.11.

### Pass-by-value: ядро урока

Теперь главный вопрос. Выводим правило из кода, который проверен на JDK 21:

```java
static void increment(int x)          { x = x + 1; }

static void mutate(List<Integer> list) {
    list.add(42);                     // мутируем объект по ссылке
    list = new ArrayList<>();         // переприсваиваем ЛОКАЛЬНУЮ копию
    list.add(999);
}

public static void main(String[] args) {
    int n = 10;
    increment(n);
    System.out.println(n);            // 10

    List<Integer> outer = new ArrayList<>();
    mutate(outer);
    System.out.println(outer);        // [42]
}
```

Вывод: `10`, потом `[42]`. Разбираем, что произошло, — и правило соберётся само.

С `increment` всё очевидно: в фрейм метода скопировалось значение `10`, метод увеличил свою копию, копия умерла вместе с фреймом. С `mutate` интереснее. При вызове в фрейм метода скопировалось содержимое переменной `outer` — а это **ссылка**. Теперь `outer` и `list` — две ссылки на один `ArrayList` (тот самый aliasing). Поэтому `list.add(42)` виден снаружи: объект-то один. А вот `list = new ArrayList<>()` перезаписал **локальную копию ссылки** — `outer` в фрейме `main` как указывал на первый список, так и указывает. `999` ушёл в объект, на который после возврата из метода не указывает никто, — и его заберёт GC.

![[Pasted image 20260702103220.png]]

Отсюда правило, которое ты произносишь на собесе дословно: **Java всегда передаёт по значению**. Просто «значение» переменной reference-типа — это сама ссылка. Копируется она; через копию можно дотянуться до того же объекта и мутировать его, но переприсвоить переменную вызывающего кода — нельзя.

Лакмусовая бумажка, отличающая настоящий pass-by-reference от «pass-reference-by-value» — классический `swap`:

```java
static void swap(StringBuilder x, StringBuilder y) {
    StringBuilder tmp = x;
    x = y;
    y = tmp;
}

StringBuilder a = new StringBuilder("A");
StringBuilder b = new StringBuilder("B");
swap(a, b);
System.out.println(a + " " + b);   // A B — ничего не поменялось
```

В языке с настоящей передачей по ссылке (C# с `ref`, C++ с `&`) `swap` бы сработал: метод получил бы доступ к самим переменным вызывающего кода. В Java он получает копии двух ссылок, тасует копии — и они умирают вместе с фреймом. Проверено: `A B`.

Интервьюер часто формулирует ловушку мягко: «ну, объекты-то передаются по ссылке?» Соглашаться нельзя. Объекты не передаются вообще — они лежат в heap и никуда не двигаются; передаётся копия ссылки. Формулировка «pass by reference» означает конкретную семантику (доступ к переменной вызывающего), и в Java её нет.

💡 Из C#: без модификаторов C# ведёт себя ровно так же — копия значения для value type, копия ссылки для reference type. Разница в том, что в C# есть `ref`/`out`/`in`, включающие настоящую передачу по ссылке, — а в Java **нет никакого аналога**. Если метод должен вернуть несколько значений — возвращаешь `record` (аналог ad-hoc кортежа; про records — глава 17) или массив. Паттерн «out-параметр» в Java не существует как язык — только как костыль с мутабельным контейнером, и это считается плохим стилем.

### final для ссылочной переменной

`final` запрещает переприсвоить **переменную** — и не говорит ничего про объект:

```java
static void clearAndFill(final List<String> list) {
    list.clear();
    list.add("replaced");          // ок: мутируем объект
    // list = new ArrayList<>();   // не скомпилируется
}

List<String> data = new ArrayList<>(List.of("one", "two"));
clearAndFill(data);
System.out.println(data);          // [replaced]
```

`final` прибивает гвоздём стрелку, а не то, на что она указывает. Иммутабельность объекта — совсем другой механизм (3.7); неизменяемые коллекции — 13.11.

💡 Из C#: для полей это ровно `readonly` — ссылку не переприсвоишь, объект мутируется. Для локальных переменных и параметров в C# аналога нет вовсе (предложение `let`/`readonly` locals так и не доехало).

### 🔬 Глубже / на синьора

**Сколько весит ссылка.** На 64-битной HotSpot ссылка — это 8 байт... если бы не compressed oops (compressed ordinary object pointers). JVM хранит 32-битное смещение вместо 64-битного адреса и умножает на 8 при разыменовании — работает, потому что объекты выровнены по 8 байт (`ObjectAlignmentInBytes = 8`). Покрывает heap до ~32 GB. Проверено на JDK 21: `UseCompressedOops = true {ergonomic}` — включено по умолчанию. Итого ссылка стоит 4 байта.

**Сколько весит объект.** Минимальный объект — заголовок плюс поля плюс выравнивание:

![[Pasted image 20260702103239.png]]

Это не теория — я замерил на JDK 21: массив `Integer[]` на миллион элементов (значения вне кеша, реальные объекты) занял **20 000 280 байт** — ровно 20 байт на элемент: 4 байта ссылка в массиве + 16 байт объект. `int[]` на миллион — **4 000 016 байт**. Пятикратная разница, и она вся — цена «объектности»: заголовок, ссылка, выравнивание. Что это делает с cache locality и GC — разберём в 1.4 и главе 11. Траектория: JEP 450 (экспериментально в JDK 24) и JEP 519 (продуктовая опция в JDK 25) сжимают заголовок до 8 байт — но на baseline JDK 21 считай 12–16.

**Ссылки в байткоде.** JVM различает примитивы и ссылки на уровне системы команд. Проверено через `javap -c`:

```java
static int primitives(int a, long b) {   // iload_0, lload_1, l2i, iadd,
    int c = a + (int) b;                 // istore_3, iload_3, ireturn
    return c;
}
static Object refs(String s, int[] arr) { // aload_0, astore_2,
    Object o = s;                         // aload_1, astore_2,
    o = arr; o = null;                    // aconst_null, astore_2,
    return o;                             // aload_2, areturn
}
```

Префикс инструкции — это тип: `i` для `int`, `l` для `long`, `a` для ссылок (address). `null` — отдельная инструкция `aconst_null`. Верификатор байткода не даст перепутать: положить ссылку через `istore` — невалидный класс-файл. То есть разделение «примитив/ссылка» — не соглашение компилятора, а физика виртуальной машины.

**Ссылка — не стабильный адрес.** Копирующие GC (глава 11) перемещают живые объекты между регионами heap на каждой сборке. Все ссылки при этом обновляются прозрачно. Следствие: `System.identityHashCode` — это не адрес (адрес меняется, hash — нет), и никакой код не может полагаться на расположение объекта. В C# ты можешь `fixed`-нуть объект для интеропа; в Java пиннинга нет в принципе (FFM API из JEP 454 решает интероп иначе — через off-heap память).

### ⚠️ Подводные камни

**Переприсваивание параметра — no-op для вызывающего.** Любой код вида `param = ...` внутри метода меняет только локальную копию. Классический симптом — «метод-инициализатор», который присваивает в параметр и удивляется, что снаружи всё по-старому. Если метод должен дать новый объект — он его **возвращает**.

**`final` ≠ иммутабельность.** `final List<String> list` спокойно позволяет `list.add(...)`. Прибита ссылка, не объект. Реально неизменяемое — `List.copyOf` и родня (13.11).

**Aliasing как дыра в инкапсуляции.** Геттер, возвращающий ссылку на внутренний мутабельный список, раздаёт наружу право мутировать твоё состояние: `order.getItems().clear()` — и инварианты класса растоптаны без единого сеттера. Лекарство — defensive copying и неизменяемые обёртки (20.4, 20.10).

**Поле молча инициализируется в `null` — и взрывается далеко от места ошибки.** Забыл присвоить поле в конструкторе — компилятор промолчал (в отличие от локальной переменной), NPE вылетит при первом разыменовании, возможно в другом классе через десять минут работы. Helpful NPE (JEP 358) хотя бы назовёт виновное звено.

**`==` на объектах сравнивает идентичность.** `a == c` для двух одинаковых по содержимому объектов — `false`. Со `String` из-за пула констант `==` «иногда работает» — это худший вид бага, потому что тесты на литералах зелёные, а прод на данных из сети падает. Полный разбор — 1.5.

**Мутация объекта, переданного в чужой код.** Ты отдал `List` в метод библиотеки/коллеги — он имеет полное право его мутировать (и наоборот: тебе передали список — мутируя его, ты меняешь состояние вызывающего). Контракт «кто владеет объектом» в сигнатуре Java не выражается — только документацией и копиями.

### Вопросы с собеса (с разбором)

**1. «Java передаёт аргументы по значению или по ссылке?»** — самый частый вопрос уровня screening.

Ответ: всегда по значению. Для reference-типов значением переменной является ссылка, и копируется она. Через копию можно мутировать тот же объект; переприсвоить переменную вызывающего — нельзя.

_Follow-up #1_: «Тогда почему `list.add()` внутри метода виден снаружи?» — Потому что обе ссылки (оригинал и копия) указывают на один объект в heap. Мутация объекта — не то же самое, что изменение переменной.

_Follow-up #2 (ловушка)_: «Значит, объекты передаются по ссылке?» — Нет. Объекты не передаются вообще — передаётся копия ссылки. «Pass by reference» — это конкретная семантика (доступ к самой переменной вызывающего, как `ref` в C#), и её в Java нет. Проверка — `swap`: в Java не работает, при настоящем pass-by-reference работал бы.

---

**2. Что напечатает?**

```java
static void mutate(List<Integer> list) {
    list.add(42);
    list = new ArrayList<>();
    list.add(999);
}
// ...
List<Integer> outer = new ArrayList<>();
mutate(outer);
System.out.println(outer);
```

Ответ: `[42]`. `add(42)` прошёл по ссылке в общий объект; `list = new ArrayList<>()` перезаписал локальную копию, `999` ушёл в объект-сироту, который заберёт GC.

_Follow-up_: «Что произойдёт со вторым `ArrayList` после возврата из метода?» — На него не останется ни одной ссылки, он станет unreachable и будет собран GC при следующей сборке. Утечки нет — наоборот, это ровно тот мусор, ради которого GC существует.

---

**3. Скомпилируется ли и что выведет?**

```java
static void fill(final List<String> list) {
    list.add("x");
    list = new ArrayList<>();
}
```

Ответ: не скомпилируется — но только из-за второй строки: `final` параметр нельзя переприсвоить. Убери её — компилируется, и `add("x")` виден снаружи: `final` фиксирует ссылку, не объект.

_Follow-up_: «Как сделать список действительно неизменяемым?» — `List.copyOf(list)` или `List.of(...)` — реально immutable коллекции, `add` кинет `UnsupportedOperationException`. `Collections.unmodifiableList` — только view поверх мутабельного оригинала. Разница — в 13.11.

---

**4. Где хранятся объекты, а где ссылки?**

Ответ: объекты — в heap, всегда. Ссылка живёт там, где живёт её переменная: локальная — в фрейме стека, поле объекта — внутри объекта в heap, статическое поле — в структурах класса.

_Follow-up (на синьора)_: «Всегда ли объект оказывается в heap?» — Концептуально да, но JIT через escape analysis может доказать, что объект не покидает метод, и применить scalar replacement — разложить поля по регистрам, не аллоцируя. Это оптимизация, не гарантия — полагаться на неё для корректности нельзя. Детали в 11.9.

---

**5. Что напечатает и почему?**

```java
StringBuilder a = new StringBuilder("A");
StringBuilder b = new StringBuilder("B");
swap(a, b);   // внутри: tmp = x; x = y; y = tmp;
System.out.println(a + " " + b);
```

Ответ: `A B`. Метод тасует свои копии ссылок; переменные `a` и `b` в вызывающем фрейме не тронуты. Это каноническое доказательство отсутствия pass-by-reference в Java.

_Follow-up_: «А как всё-таки поменять местами?» — Никак через метод с такой сигнатурой. Варианты: вернуть record с парой значений и переприсвоить у вызывающего; поменять **содержимое** объектов (`a.setLength(0); a.append(...)`) — но это мутация объектов, а не обмен ссылок, и с иммутабельными типами вроде `String` не сработает вовсе.

---

**6. Чем поле отличается от локальной переменной в плане инициализации?**

Ответ: поля (и ячейки массивов) получают дефолт автоматически — `null` для ссылок, нули для примитивов. Локальные переменные дефолта не имеют: чтение до присваивания — ошибка компиляции (definite assignment).

_Follow-up_: «Почему такая асимметрия?» — Для локальных переменных компилятор может доказать инициализацию анализом потока управления внутри метода — и требует этого. Для полей порядок записи/чтения зависит от порядка вызовов в рантайме (и от других потоков) — статически доказать нельзя, поэтому язык даёт предсказуемый дефолт вместо мусора из памяти.

_Follow-up #2_: «Как быстро найти, какое звено цепочки `a.b().c()` было null?» — С Java 15 helpful NPE (JEP 358) пишет это прямо в сообщении исключения: `...because the return value of "Address.city()" is null`.

### Практика

**1. Три эксперимента с передачей.** Напиши `increment(int)`, `mutate(List)`, `swap(StringBuilder, StringBuilder)` из урока, предскажи вывод до запуска, запусти, объясни каждый результат одним предложением через слово «копия».

_Сложность: низкая. Время: 15–20 минут. Что даёт: модель pass-by-value перестаёт быть заклинанием._

**2. Ссылки в байткоде.** Скомпилируй класс с методом на примитивах и методом на ссылках (включая `= null`), запусти `javap -c`. Найди `iload`/`istore` против `aload`/`astore`/`aconst_null`. Сопоставь каждую строку кода с инструкциями.

_Сложность: низкая. Время: 30 минут. Что даёт: понимание, что примитив/ссылка — различие уровня JVM, не компилятора._

**3. Взвесить ссылку и объект.** Через `Runtime.getRuntime().totalMemory() - freeMemory()` (с `System.gc()` до замера) сравни `int[1_000_000]` и `Integer[1_000_000]`, заполненный значениями от 128 (вне кеша). Посчитай байты на элемент, сверь с раскладкой 4 + 16.

_Сложность: средняя. Время: 30–40 минут. Что даёт: цифры 4 vs 20 байт из личного опыта, а не из урока._

**4. Helpful NPE руками.** Построй цепочку из трёх record'ов, положи `null` в середину, вызови `a.b().c().d()`. Прочитай сообщение. Перезапусти с `-XX:-ShowCodeDetailsInExceptionMessages` и почувствуй жизнь до Java 15.

_Сложность: низкая. Время: 15 минут. Что даёт: рефлекс читать сообщение NPE, а не только stack trace._

**5. Сломать инкапсуляцию через aliasing.** Класс `Order` с приватным `List<String> items` и геттером, возвращающим поле как есть. Снаружи через геттер очисти список и добавь мусор — инварианты мертвы без единого сеттера. Почини: геттер возвращает `List.copyOf(items)`. Замерь, что сломалось в «злом» коде.

_Сложность: низкая. Время: 30 минут. Что даёт: телесное понимание, зачем существует defensive copying (20.4)._

### На собесе

**«Java передаёт по значению или по ссылке?»**

> Всегда по значению. Значение переменной reference-типа — это сама ссылка, и копируется она. Через копию можно мутировать общий объект, но переприсвоить переменную вызывающего нельзя — `swap` в Java не работает. Аналога `ref`/`out` из C# нет; несколько значений возвращаются через record.

> EN: Java is always pass-by-value. For reference types, the value being copied is the reference itself — you can mutate the shared object through the copy, but reassigning the parameter never affects the caller.

**«Что лежит в переменной?»**

> Либо примитив, либо ссылка на объект в heap. Присваивание и передача в метод всегда копируют содержимое ячейки; объект не копируется никогда. Две переменные легко ссылаются на один объект — это aliasing, и мутация видна через любую из них.

> EN: A variable holds either a primitive value or a reference to a heap object. Assignment copies the cell's content, so two variables can alias the same object.

**«Что делает `final` на ссылочной переменной?»**

> Запрещает переприсвоить переменную — и всё. Объект остаётся мутабельным: `final List` спокойно принимает `add`. Иммутабельность — свойство объекта, не ссылки; для коллекций это `List.of`/`List.copyOf`.

> EN: `final` pins the reference, not the object — the object stays mutable. Immutability is a property of the object itself.

**«Где хранятся объекты и ссылки?»**

> Объекты — в heap; ссылки — там, где их переменные: локальные во фрейме стека, поля — внутри своих объектов. Ссылка — непрозрачный указатель: без арифметики, без пиннинга, GC свободно двигает объекты и обновляет ссылки. С compressed oops ссылка стоит 4 байта, минимальный объект — 16.

> EN: Objects live on the heap; references live wherever their variable lives. A reference is an opaque pointer — no arithmetic, no pinning, GC relocates objects freely.

**«Каков дефолт у ссылки?»**

> У полей и ячеек массивов — `null`; локальные переменные дефолта не имеют, чтение до присваивания — ошибка компиляции. Разыменование `null` — NPE, и с Java 15 (JEP 358) JVM называет конкретное null-звено в сообщении. Nullable-типов уровня языка, как `string?` в C#, в Java нет — только аннотации и статический анализ.

> EN: Fields and array cells default to `null`; locals must be definitely assigned before use. Since Java 15 (JEP 358), NPE messages pinpoint exactly which part of the expression was null.
