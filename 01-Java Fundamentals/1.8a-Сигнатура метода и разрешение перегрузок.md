Перегрузка — повседневность Java-API, а не экзотика: у `System.out.println` десять вариантов, у `List.of` — двенадцать, у `Math.max` — четыре. Каждый раз, когда ты пишешь вызов, компилятор обязан из всех кандидатов детерминированно выбрать ровно один — и делает это по формальным правилам. Знать эти правила стоит по двум причинам. Во-первых, они объясняют поведение стандартной библиотеки, включая пару знаменитых footgun'ов, на которых ловятся даже опытные разработчики. Во-вторых, «что напечатает `f(5)` при пяти перегрузках» — стандартный блок вопросов на mid+ собеседовании, и отвечать на него нужно не угадыванием, а прогоном по алгоритму.

**Ментальная модель.** Сигнатура метода — это имя плюс типы параметров, больше ничего. Выбор перегрузки происходит на компиляции в три фазы: сначала без боксинга и без varargs (widening разрешён), потом с автобоксингом, потом varargs. Компилятор берёт первую непустую фазу и выбирает в ней самый специфичный метод.

### Сигнатура: что в неё входит, а что нет

Формальное определение: сигнатура — это **имя метода и типы параметров в порядке объявления**. Не входят: возвращаемый тип, имена параметров, `throws`-клауза, модификаторы (`public`/`private`/`static`/`final`).

Проверим следствие на компиляторе — две «перегрузки», отличающиеся только return type:

```java
public class RetType {
    int  parse(String s) { return 0; }
    long parse(String s) { return 0L; }
}
```

JDK 21 отвечает:

```
RetType.java:3: error: method parse(String) is already defined in class RetType
```

Для компилятора это один и тот же метод, объявленный дважды. Причина дизайна — направление, в котором работает resolution: компилятор выбирает перегрузку **по аргументам вызова**, до того как известно, что будет с результатом. Возьми вызов-statement:

```java
parse("42");   // результат отброшен — по какому return type выбирать?
```

Здесь нет никакого контекста, из которого можно вывести желаемый возвращаемый тип. Чтобы выбор был всегда однозначным, Java вынесла return type за рамки сигнатуры на уровне языка.

> 💡 **Из C#:** здесь отличий нет — в C# перегрузка только по return type тоже запрещена, и по той же причине. Мозговые циклы на это не трать; расхождения начнутся в фазах resolution ниже.

#### 🔬 Глубже / на синьора: в байткоде return type есть

На уровне class-файла метод идентифицируется **дескриптором**, и return type в него входит. Реальный вывод `javap -s`:

```
int parse(java.lang.String);
    descriptor: (Ljava/lang/String;)I
```

`I` в конце — это `int`. JVM спокойно различает методы с одинаковым именем и параметрами, но разным return type — запрет существует только в языке. И компилятор сам этим пользуется для **covariant return types**: когда ты сужаешь возвращаемый тип при переопределении, он генерирует синтетический bridge-метод. Живой пример — переопределяем `Object clone()` на `Box clone()` и смотрим `javap -c -p`:

```
class Box implements java.lang.Cloneable {
  protected Box clone() throws CloneNotSupportedException;
  protected java.lang.Object clone() throws CloneNotSupportedException;
    Code:
       0: aload_0
       1: invokevirtual #13    // Method clone:()LBox;
       4: areturn
}
```

В class-файле физически **два** метода `clone` — твой и мостик `Object clone()`, который просто делегирует твоему. Сам объявить такую пару нельзя; это привилегия компилятора. Механика bridge-методов целиком развернётся в главе 7 (type erasure) — там они порождаются тем же механизмом. Если интервьюер копнёт «но в байткоде же можно?» — вот готовый ответ.

### Overload resolution: три фазы

Сначала термин: **widening primitive conversion** — безопасное расширяющее преобразование примитива без потери величины: `byte → short → int → long → float → double`, а также `char → int → ...`. Компилятор применяет его неявно.

Теперь сам алгоритм. Компилятор ищет применимые методы в **три фазы** и останавливается на первой, где нашёлся хотя бы один кандидат:

1. **strict** — применимость без боксинга и без varargs; widening (примитивный и ссылочный — подтип в супертип) разрешён;
2. **loose** — то же плюс autoboxing/unboxing; varargs всё ещё нет;
3. **varargs** — разрешается переменная арность.

Если в выбранной фазе кандидатов несколько — берётся **самый специфичный** (JLS §15.12.2.5): метод A специфичнее B, если любой аргумент, годный для A, годится и для B. `int` специфичнее `long`, `String` специфичнее `Object`.

![[Pasted image 20260702132746.png]]

Прогоним лестницу на живом компиляторе. Все пять перегрузок объявлены, из них по одной убираем сверху — каждый вывод ниже получен реальным запуском на JDK 21:

```java
static void f(int x)     { System.out.println("int"); }
static void f(long x)    { System.out.println("long"); }
static void f(Integer x) { System.out.println("Integer"); }
static void f(Object x)  { System.out.println("Object"); }
static void f(int... x)  { System.out.println("varargs"); }

f(5);   // ?
```

- **Все пять на месте** → фаза 1 находит `f(int)` (точное совпадение) и `f(long)` (widening). Самый специфичный — `f(int)`. Печатает `int`.
- **Убрали `f(int)`** → в фазе 1 остался `f(long)`. Печатает `long` — боксинг в `Integer` даже не рассматривался.
- **Убрали и `f(long)`** → фаза 1 пуста. Фаза 2: `f(Integer)` (boxing) и `f(Object)` (boxing + ссылочный widening). Специфичнее `Integer`. Печатает `Integer`.
- **Остались `f(Object)` и `f(int...)`** → фаза 2 находит `f(Object)`. Печатает `Object` — varargs проигрывает даже `Object`.
- **Только `f(int...)`** → лишь фаза 3. Печатает `varargs`.

Тот же прогон объясняет и соседние вызовы: `byte b = 5; f(b)` и `short`, и `char` — все уходят в `f(int)` через widening в фазе 1, а `f(5L)` — в `f(long)` точным совпадением.

> 💡 **Из C#:** фазовая модель с приоритетом точного совпадения над конверсиями есть и в C#, и там `f(5)` тоже выберет `int`. Расхождение — в жёстком порядке «widening → boxing → varargs» как отдельных фаз. Практическое следствие для интуиции из .NET: при наличии `f(long)` и `f(Integer)` вызов `f(5)` уходит в `long`, потому что widening живёт в фазе 1, а боксинг — только в фазе 2. Тот, кто привык думать «`Integer` — это же почти `int`», предсказывает неправильно.

### Фаза важнее «близости» типа

Ключевое следствие трёхфазности, на котором ловят даже тех, кто выучил лестницу: компилятор сравнивает специфичность **только внутри одной фазы**. Между фазами никакого взвешивания «что ближе» нет — побеждает более ранняя фаза, какой бы «дальней» ни казалась конверсия. Проверено на компиляторе:

```java
static void f(double x)  { System.out.println("double"); }
static void f(Integer x) { System.out.println("Integer"); }

f(5);   // печатает "double"
```

Интуитивно `Integer` выглядит «роднее» для `int`, чем `double` — но `int → double` это widening из фазы 1, а `int → Integer` — боксинг из фазы 2. До фазы 2 дело не доходит. Число молча конвертируется в `5.0`, и если внутри `f(double)` другая логика — привет, тихий баг без единого warning.

Самый известный производственный footgun на эту тему живёт прямо в `java.util.List`, у которого есть `remove(int index)` и `remove(Object o)`:

```java
List<Integer> list = new ArrayList<>(List.of(10, 20, 30, 40));
list.remove(2);                    // [10, 20, 40]  — удалён элемент с ИНДЕКСОМ 2
list.remove(Integer.valueOf(20));  // [10, 40]      — удалено ЗНАЧЕНИЕ 20
```

Литерал `2` — это `int`; `remove(int)` совпадает в фазе 1 без боксинга и побеждает. Чтобы удалить значение — боксишь явно. Для `List<Integer>` это регулярный источник багов: намерение «удали двойку» и код `remove(2)` означают разное.

### ⚠️ Подводные камни

**Неоднозначность двух varargs-перегрузок.** «Самый специфичный» не всегда существует. Реальная ошибка компиляции:

```java
static void g(String a, Object... rest) { }
static void g(Object a, String... rest) { }
g("x", "y");
// error: reference to g is ambiguous
//   both method g(String,Object...) and method g(Object,String...) match
```

У первого специфичнее первый параметр, у второго — тип элементов varargs. Ни один не «покрывает» другой — компилятор отказывается выбирать. Разрешается явным кастом аргумента или переименованием методов.

**`null` как аргумент.** `null` совместим с любым ссылочным типом, поэтому применимы все ссылочные перегрузки — дальше работает та же специфичность. Для `h(String)` и `h(Object)` вызов `h(null)` уходит в `String` (специфичнее). А вот для `h(String)` и `h(StringBuilder)` — типы не связаны иерархией, ни один не специфичнее, и это `reference to h is ambiguous` (проверено). Лечится кастом: `h((String) null)`.

**Narrowing для аргументов не работает — даже для констант.** `byte b = 5;` компилируется: в assignment context константное сужение разрешено. Но при вызове метода такого послабления нет:

```java
static void f(byte x) { }
f(5);   // error: possible lossy conversion from int to byte
```

Литерал `5` — это `int`, а invocation context допускает только widening. Нужен явный каст `f((byte) 5)`. Асимметрия «присваивать можно, передавать нельзя» — готовый вопрос-ловушка.

**Boxing и примитивный widening не сцепляются.** Компилятор умеет `int → long` (widening), умеет `int → Integer` (boxing), умеет `Integer → Object`/`Number` (boxing + ссылочный widening) — но не умеет `int → long → Long` и не умеет `int → Integer → Long`:

```java
static void f(Long x) { }
f(5);   // error: int cannot be converted to Long
```

Единственная перегрузка — и всё равно не скомпилируется. Цепочка «расширь примитив, потом упакуй» в конверсиях вызова запрещена. Пишут `f(5L)` или `f((long) 5)`.

**Фиксированная арность всегда бьёт varargs.** `f(int, int)` и `f(int...)` не конфликтуют: `f(1, 2)` уходит в фиксированную (фазы 1–2 раньше фазы 3), а `f(1)` и `f(1, 2, 3)` — в varargs, потому что фиксированная к ним неприменима. Именно поэтому в JDK у `List.of` есть перегрузки от 0 до 10 фиксированных параметров и только потом `of(E...)` — частые вызовы не платят за varargs (чем именно платят — в уроке 1.8b).

### Вопросы с собеса (с разбором)

**1. «Что напечатает `f(5)`?» (пять перегрузок из урока)**  
Ответ: `int` — точное совпадение в фазе 1. _Follow-up:_ «Убрали `f(int)`?» → `long`: widening остаётся в фазе 1, боксинг ждёт в фазе 2. _Follow-up 2:_ «Почему `long`, а не `Integer` — он же ближе к `int`?» → Между фазами «близость» не сравнивается; JLS §15.12.2 предписывает брать первую непустую фазу. Специфичность работает только внутри фазы. Слабый ответ, который завалит follow-up: «компилятор выбирает наиболее подходящий тип» — без механизма фаз это звучит как угадывание.

**2. «Почему нельзя объявить две перегрузки, отличающиеся только return type?»**  
Ответ: return type не входит в сигнатуру (имя + типы параметров, JLS §8.4.2). Resolution идёт по аргументам на компиляции, до использования результата — statement вида `parse("42");` не дал бы контекста для выбора. _Follow-up:_ «А в байткоде?» → В дескрипторе return type есть (`(Ljava/lang/String;)I`), JVM такие методы различает. На этом построены covariant returns: компилятор генерирует bridge-метод, в class-файле физически два метода. _Ловушка:_ ответить «JVM не может их различить» — ровно наоборот, не может язык, а JVM может.

**3. «`list.remove(2)` для `List<Integer>` — что удалит?»**  
Ответ: элемент с индексом 2 — `remove(int)` совпадает в фазе 1 и бьёт `remove(Object)`, которому нужен боксинг. _Follow-up:_ «Как удалить значение 2?» → `list.remove(Integer.valueOf(2))`. _Follow-up 2:_ «Это баг дизайна?» → Скорее наследие: `remove(int)` был в `List` до generics и автобоксинга (Java 1.2 против 5.0); после их появления вызовы стали неоднозначно читаться человеком, но менять API уже было нельзя.

**4. «Перегрузки `f(double)` и `f(Integer)`, вызов `f(5)` — что выполнится?»**  
Ответ: `f(double)` — widening `int → double` из фазы 1 побеждает боксинг из фазы 2, несмотря на кажущуюся «дальность» `double`. _Follow-up:_ «Чем это опасно на практике?» → Ошибок и warning нет, число молча превращается в `5.0`; при добавлении перегрузки с плавающей точкой в существующий API можно тихо перенаправить старые вызовы.

**5. «`h(null)` — что выберет компилятор?»**  
Ответ: зависит от иерархии перегрузок. `null` применим ко всем ссылочным типам; из применимых берётся самый специфичный: для `h(String)`/`h(Object)` — `String`. Если типы не связаны (`String`/`StringBuilder`) — ошибка `ambiguous`. _Follow-up:_ «Как разрулить?» → Явный каст: `h((String) null)`. _Follow-up 2:_ «А если перегрузка одна и примитивная, `h(int)`?» → `null` к примитиву неприменим вообще — compile error об incompatible types.

**6. «Единственный метод `f(Long x)`, вызов `f(5)` — скомпилируется?»**  
Ответ: нет — `int cannot be converted to Long`. Конверсии вызова не сцепляют widening с боксингом: ни `int → long → Long`, ни `int → Integer → Long`. _Follow-up:_ «Но `f(Object)` для `f(5)` работает — там же тоже цепочка?» → Работает связка «boxing, затем ссылочный widening» (`int → Integer → Object`) — она в фазе 2 разрешена явно. Запрещена именно связка с примитивным widening. _Follow-up 2:_ «Как починить вызов?» → `f(5L)` — литерал сразу `long`, дальше чистый boxing.

### На собесе

«Сигнатура в Java — имя метода плюс типы параметров. Return type, имена параметров, `throws` и модификаторы не входят. Поэтому перегрузить только по возвращаемому типу нельзя: резолюция идёт по аргументам на компиляции, до использования результата. В байткоде return type в дескрипторе есть, и JVM методы по нему различает — на этом построены covariant returns через bridge-методы, которые генерирует компилятор.»

«Разрешение перегрузок — три фазы по JLS §15.12.2: strict с widening, но без боксинга и varargs; loose с автобоксингом; и только потом varargs. Компилятор берёт первую непустую фазу и в ней самый специфичный метод. Специфичность сравнивается только внутри фазы — поэтому `f(5)` при паре `f(long)`/`f(Integer)` уходит в `long`, а при паре `f(double)`/`f(Integer)` — даже в `double`: widening всегда раньше боксинга.»

«Классика на эту тему — `list.remove(2)` удаляет по индексу, а не значение `2`: `remove(int)` совпадает в фазе 1 и бьёт `remove(Object)`. Для значения — `remove(Integer.valueOf(2))`.»

«Пограничные случаи: `null` применим ко всем ссылочным перегрузкам, берётся самая специфичная, а без иерархии между ними — ambiguous. Narrowing в вызовах запрещён даже для констант — `f(5)` не попадёт в `f(byte)`, хотя `byte b = 5` компилируется. Widening с боксингом не сцепляются: `f(Long)` не примет `int`. Фиксированная арность всегда бьёт varargs — поэтому у `List.of` десять фиксированных перегрузок перед varargs-версией.»

**EN, компактно:** "A method signature in Java is the name plus parameter types — return type is excluded, so you can't overload on it alone. Overload resolution runs at compile time in three phases per JLS §15.12.2: strict (widening allowed, no boxing, no varargs), then loose (boxing allowed), then varargs. The compiler takes the first non-empty phase and picks the most specific method within it — specificity is never compared across phases. That's why `f(5)` prefers `f(long)`, or even `f(double)`, over `f(Integer)`: widening beats boxing. The classic production consequence is `list.remove(2)` removing by index, not by value."
