В обычной работе ты пишешь `mvn package` и идёшь пить кофе — Maven сам вызывает `javac`, собирает jar с правильным манифестом и раскладывает classpath. А потом в проде падает `NoClassDefFoundError`, или библиотека внезапно старой версии, хотя в `pom.xml` новая, — и вот тут знание механики отделяет «починил за минуту» от «полдня копирую со StackOverflow вслепую». Это ровно та зона, где .NET-интуиция не поможет: в C# ты `-cp` руками не писал никогда, рантайм находил сборки сам.

**Ментальная модель.** `classpath` — это упорядоченный список мест (папки и jar-ы), где JVM ищет `.class`-файлы по имени класса; побеждает **первое** совпадение. А `.jar` — это буквально ZIP-архив с деревом `.class` и одним текстовым манифестом; «исполняемым» его делает единственная строка `Main-Class`.

#### Classpath: как JVM находит класс

Когда ты пишешь `java -cp out org.example.Main`, JVM берёт полное имя класса (FQCN — fully qualified class name), превращает точки в слэши, дописывает `.class` и ищет получившийся путь внутри каждой записи classpath по порядку:

![[Pasted image 20260702093915.png]]

Записи разделяются `:` на Linux/macOS и `;` на Windows:

```bash
java -cp out:libs/gson.jar:libs/slf4j.jar org.example.Main
```

**Порядок важен** — берётся первое совпадение. Проверяемо: два каталога `a/` и `b/`, в каждом свой `V.class`:

```
java -cp a:b V   → version A
java -cp b:a V   → version B
```

Отсюда растёт классический баг «в проде старая версия библиотеки, хотя в `pom.xml` новая»: где-то в classpath раньше затесался лишний jar со старым классом, и он выигрывает.

> ⚠️ **Подводный камень:** есть ещё переменная окружения `CLASSPATH`. Не используй её — это глобальное состояние, которое молча ломает сборки соседних проектов. Всегда задавай classpath явно через `-cp`. Если classpath не указан вообще, JVM смотрит только в текущую директорию (`.`).

> 💡 **Из C#:** прямого аналога `classpath` нет. Рантайм .NET находит сборки через `AssemblyLoadContext`, `.deps.json` и probing-пути — ты `-cp` никогда не писал. Ближайшее по духу — это `PATH` для исполняемых файлов в shell: список мест, просматриваемый по порядку.

#### `ClassNotFoundException` vs `NoClassDefFoundError`

Два симптома одной болезни («класс не в classpath»), но приходят по-разному, и на собесе любят проверять, понимаешь ли ты разницу.

`NoClassDefFoundError` (это `Error`) — класс **был** при компиляции, но пропал в рантайме. Компилируем `App`, зависящий от `Helper`, удаляем `Helper.class`, запускаем:

```
Exception in thread "main" java.lang.NoClassDefFoundError: Helper
        at App.main(App.java:3)
Caused by: java.lang.ClassNotFoundException: Helper
```

`ClassNotFoundException` (это checked `Exception`) — кто-то попросил класс **по имени** в рантайме, а его нет. Типично — `Class.forName`, рефлексия, загрузка плагинов:

```
Exception in thread "main" java.lang.ClassNotFoundException: com.does.not.Exist
        at ...BuiltinClassLoader.loadClass...
```

Вся суть — в разнице механизма запроса, а не «одного нет сильнее другого». Обрати внимание на `Caused by` в первом примере: `NoClassDefFoundError` **оборачивает** `ClassNotFoundException`. Линковка обратилась к загрузчику по имени, тот кинул `ClassNotFoundException`, а рантайм-линковка завернула его в `Error`, потому что для скомпилированного кода отсутствие уже слинкованного класса — фатально.

||`ClassNotFoundException`|`NoClassDefFoundError`|
|---|---|---|
|Тип|checked `Exception`|`Error`|
|Кто запросил|явно по имени (`Class.forName`, рефлексия)|неявно, при линковке скомпилированного кода|
|Класс при компиляции|не требовался|**присутствовал**, исчез в рантайме|
|Типичная причина|плагин/имя из конфига отсутствует|зависимость не попала в рантайм-classpath / деплой сломан|

> 🔬 **Глубже / на синьора:** `NoClassDefFoundError` бывает и коварнее — если класс **упал в статическом инициализаторе** при первой загрузке, JVM помечает его как `Erroneous`, и все последующие обращения кидают `NoClassDefFoundError` уже без «Caused by», хотя `.class` физически на месте. Тогда ищи не classpath, а исключение при инициализации, которое произошло раньше. Механика загрузки/линковки/инициализации классов — гл. 11–12.

#### JAR — это просто ZIP

`.jar` (Java ARchive) — ZIP-архив с деревом `.class` и служебной папкой `META-INF/`, где лежит `MANIFEST.MF`. Не веришь — переименуй `app.jar` в `app.zip` и распакуй, `unzip -l` показывает ровно это:

```
META-INF/MANIFEST.MF
org/example/Main.class
```

А сам манифест — обычный текст:

```
Manifest-Version: 1.0
Created-By: 21.0.11 (Ubuntu)
Main-Class: org.example.Main
```

![[Pasted image 20260702093933.png]]

Собирается это так (руками — чтобы понимать, что делает Maven):

```bash
javac -d out $(find src -name "*.java")
jar --create --file app.jar --main-class org.example.Main -C out .
```

Разница между библиотекой и приложением — ровно одна строка. Есть `Main-Class` в манифесте → это исполняемое приложение, `java -jar app.jar` сработает. Нет → это библиотека, её только подключают в classpath других проектов. Без `Main-Class` запуск падает:

```
no main manifest attribute, in lib.jar
```

> 💡 **Из C#:** аналогия с `.dll` тут хромает. В .NET `.dll` и `.exe` — разные форматы (PE), `.exe` запускает сама ОС. В Java формат один — всегда ZIP, и что библиотека, что приложение. Разница только в наличии строки `Main-Class`. И сам `.jar` ничего не «исполняет»: его всегда запускает JVM. Двойной клик работает лишь потому, что в ОС зарегистрирована ассоциация `.jar` → `java -jar`.

#### `java -jar` против `java -cp` — и главная ловушка

```bash
java -jar app.jar arg1        # точку входа берём из манифеста
java -cp out org.example.Main arg1   # точку входа задаём сами
```

При `-jar` флаг `-cp` **игнорируется полностью** — classpath берётся только из манифеста (`Class-Path:`). Проверяемо: даже с заведомо битым `-cp` jar запускается как ни в чём не бывало, потому что `-cp` просто не смотрят:

```
$ java -cp /nonexistent -jar app.jar Test
hello Test
```

Отсюда классические грабли: пишут `java -cp libs/ -jar app.jar` и не понимают, почему внешние библиотеки не находятся. Если jar-у нужны зависимости, они должны быть либо **внутри** jar (это **fat jar** / uber jar), либо явно прописаны в манифесте через `Class-Path:`. Именно поэтому Spring Boot делает fat jar: запихивает все зависимости внутрь одного файла на десятки мегабайт — запускаешь один `java -jar`, и всё на месте.

#### «Maven же всё сделает» — зачем тогда знать

Сделает: `mvn compile` → `javac`, `mvn package` → jar с манифестом, управление зависимостями и сборку classpath. Команд `javac`/`jar` ты в обычной работе не видишь. Но механику знать нужно ровно для четырёх ситуаций: дебаг `ClassNotFoundException`/`NoClassDefFoundError` (это всегда про classpath); собес («что происходит при `java -jar`» — стандартный вопрос); конфликты версий в classpath (кто выиграл при дубле); ручной запуск класса на сервере без Maven (миграционный скрипт, `java -cp`).

> #### 🔬 Глубже / на синьора
> 
> **`Class-Path:` в манифесте — относительный и хрупкий.** Пути в нём резолвятся относительно расположения самого jar, разделяются **пробелами** (не `:`!) и не поддерживают wildcards на произвольную вложенность. Поэтому в реальном мире зависимости либо кладут fat jar-ом, либо оставляют classpath на Maven/лончер, а не редактируют манифест руками.
> 
> **Fat jar не бесплатен.** Простое «распаковать все jar-ы в один» ломается на конфликтах: два jar-а с файлом `META-INF/services/...` (SPI) затрут друг друга. Плагины (`maven-shade-plugin`) поэтому не просто мёржат, а _реолокейтят_ пакеты и сливают service-файлы. Отсюда и «slim»-сборки: вынести зависимости наружу и подтянуть их classpath-ом, чтобы образ приложения был лёгким.
> 
> **Кто именно грузит твой класс.** `Main.class.getClassLoader()` в наших запусках печатает `AppClassLoader` — это загрузчик приложения, читающий твой classpath. Над ним platform- и bootstrap-загрузчики (грузят `java.base`, `java.lang.*`). Классы `java.*` не приходят из твоего classpath — их подтягивает bootstrap. Иерархия загрузчиков и делегирование — гл. 11–12.
> 
> **JPMS в стороне.** С Java 9 есть module path (`--module-path`) как альтернатива classpath, но в обычных приложениях по-прежнему доминирует classpath. Модули — гл. 8.

#### ⚠️ Подводные камни

- **`-cp` при `-jar` молча игнорируется.** Не «не работает с ошибкой», а именно молча — classpath берётся только из манифеста. Диагностировать тяжело, потому что симптом (`NoClassDefFoundError` на внешний класс) не указывает на причину.
- **Порядок classpath решает при дублях.** Два jar-а с одним классом → выигрывает первый. «Странный баг со старой версией» — почти всегда это.
- **`CLASSPATH` из окружения перебивает ожидания.** Заданная глобально переменная подмешивается и ломает изолированность. Не задавай её.
- **`NoClassDefFoundError` ≠ «класса нет».** Он может означать «класс есть, но упал в статической инициализации при первой загрузке». Проверяй логи _выше_ по времени.
- **`ClassNotFoundException` — checked.** Его нужно ловить или объявлять (`throws`) — компилятор заставит. `NoClassDefFoundError` — `Error`, его обычно не ловят.
- **jar без JVM — не бинарник.** На голой Linux-машине без установленной Java твой `.jar` — просто архив с непонятным содержимым. Никакой «нативной исполняемости» у него нет.

#### На собесе

**«Объясни, что происходит при `java -jar app.jar`.»** _(частый, Middle)_  
JVM стартует как процесс, открывает jar как ZIP, читает `META-INF/MANIFEST.MF`, берёт атрибут `Main-Class`. Через ClassLoader грузит этот класс, находит `public static void main(String[])`, вызывает его; аргументы командной строки попадают в `args`. Дальше — интерпретация байткода и JIT горячих методов.  
→ _follow-up:_ «А что такое ClassLoader?» → механизм загрузки `.class` в JVM; есть иерархия (bootstrap → platform → application), classpath — вход для application-загрузчика. Детали — гл. 11–12.

> _EN model answer:_ “The JVM starts as a process, opens the jar as a ZIP, reads `META-INF/MANIFEST.MF` and takes the `Main-Class` attribute. It loads that class via the ClassLoader, finds `public static void main(String[])` and invokes it with the command-line args, then interprets the bytecode and JIT-compiles hot methods.”

**«Чем `java -jar` отличается от `java -cp`?»** _(любимая ловушка)_  
При `-jar` точка входа и classpath берутся из манифеста, а флаг `-cp` **игнорируется**. При `-cp` ты сам указываешь classpath и FQCN точки входа.  
→ _трап:_ «Тогда как дать jar-у внешние зависимости?» → fat jar (всё внутри) или `Class-Path:` в манифесте — но не через `-cp`.

**«В чём разница между `ClassNotFoundException` и `NoClassDefFoundError`?»** _(проверка глубины)_  
`ClassNotFoundException` — checked-исключение, класс запрошен по имени в рантайме (`Class.forName`, рефлексия) и не найден. `NoClassDefFoundError` — `Error`: класс был при компиляции, но отсутствует при линковке в рантайме (или упал в статической инициализации).  
→ _follow-up:_ «Могут ли быть связаны?» → да: `NoClassDefFoundError` часто оборачивает `ClassNotFoundException` как `Caused by` — линковка обратилась к загрузчику по имени, тот не нашёл.

> _EN model answer:_ “`ClassNotFoundException` is a checked exception — something asked for a class by name at runtime (reflection, `Class.forName`) and it wasn’t on the classpath. `NoClassDefFoundError` is an `Error` — the class was present at compile time but missing at link time, or it failed in its static initializer. The former is often the `Caused by` of the latter.”

**«JAR — это бинарник?»**  
Нет, это ZIP-архив с `.class` и манифестом. Без JVM он не исполняется; «исполняемым» его делает лишь строка `Main-Class`, которую читает `java -jar`.

**Типичный слабый ответ, который проваливает:** «`-cp` и `-jar` — просто два способа запустить, разницы нет». Проваливает сразу же на follow-up про зависимости: не понимая, что `-jar` игнорирует `-cp`, кандидат не объяснит ни fat jar, ни почему его `java -cp libs/ -jar app.jar` не находит классы.
