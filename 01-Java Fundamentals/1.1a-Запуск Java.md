### Урок 1.1a — От исходника до запуска: `javac`, байткод, JVM, `main`

Ты пишешь `Hello.java`, а на сервере в проде лежит `Hello.class`, и запускается всё это командой `java`, которая на самом деле не «запускает файл», а стартует отдельный нативный процесс — JVM. Между этими тремя точками (`.java` → `.class` → выполнение) прячется вся разница между Java и .NET на уровне запуска, и первый же собес по Java почти гарантированно спросит «что происходит, когда ты пишешь `java`». В .NET ты этой цепочкой никогда не занимался руками — `dotnet run` и забыл. Здесь механику надо понимать, потому что из неё растут `ClassNotFoundException`, fat jar и половина вопросов про JVM.

**Ментальная модель.** Java-запуск — это две отдельные фазы: `javac` переводит исходник в платформонезависимый **байткод** (`.class`), а `java` стартует **JVM** — процесс, который этот байткод грузит, интерпретирует и на горячих участках дожимает JIT-ом в нативный код. Исходник компилируется один раз, байткод один и тот же на всех ОС, «нативным» его делает уже рантайм.

#### Java — это не язык, а три вещи

Когда говорят «Java», путают три разных слоя, и это стоит развести сразу.

**Java (язык)** — синтаксис, система типов, OOP-модель. Версии, которые считаются LTS: 8, 11, 17, 21 (и 25). Наш baseline — **Java 21**.

**JVM (Java Virtual Machine)** — рантайм, исполняющий байткод. Это спецификация, реализаций несколько: `HotSpot` (Oracle/OpenJDK, дефолт), `OpenJ9`, `GraalVM`. На JVM живёт не только Java — Kotlin, Scala, Groovy, Clojure компилируются в тот же байткод.

**JDK (Java Development Kit)** — тулкит разработчика: компилятор `javac`, лончер `java`, стандартная библиотека, утилиты (`javap`, `jar`, `jshell`). Есть ещё JRE (только рантайм, без `javac`), но сегодня почти всегда ставят JDK целиком.

> 💡 **Из C#:** прямой аналог — C# (язык) → CLR (рантайм) → .NET SDK (тулкит). Логика один-в-один. Отличие в деталях фаз: CLR тоже JIT-ит IL в нативный код, `HotSpot` делает это агрессивнее и с профилированием (об этом — гл. 11). Роль `HotSpot` в мире JVM ≈ роль `CoreCLR` в мире .NET: одна из реализаций, но фактический стандарт.

#### `javac` → байткод → `.class`

`javac` компилирует `.java` в `.class` — файл с байткодом JVM. Это ровно то же, что `csc`/`dotnet build`, выдающий IL в `.dll`.

```java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, Java");
    }
}
```

```bash
javac Hello.java     # → Hello.class
java Hello           # → Hello, Java
```

Байткод платформонезависим. Проверить, что внутри `.class` не машинный код, можно прямо через `file`:

```
Hello.class: compiled Java class data, version 65.0
```

`version 65.0` — это версия формата class-файла, и она жёстко привязана к версии Java: `52`=Java 8, `55`=Java 11, `61`=Java 17, **`65`=Java 21**. Отсюда классическая ошибка `UnsupportedClassVersionError` — когда `.class`, собранный новым `javac`, пытаются запустить на старой JVM.

> 💡 **Из C#:** концептуально `.class` ↔ `.dll` с MSIL. Но есть важная гранулярность: в .NET одна сборка (`.dll`) содержит много типов, а в Java **один top-level класс = один `.class`-файл**. Поэтому реальное Java-приложение — это всегда десятки и сотни `.class`-файлов, и упаковывают их отдельно (в `.jar` — про это урок 1.1b).

#### `main` по частям

JVM должна знать, с чего начать выполнение. Точка входа — метод со строго фиксированной сигнатурой:

```java
public static void main(String[] args)
```

|Часть|Зачем именно так|
|---|---|
|`public`|JVM вызывает метод «снаружи» класса — он должен быть доступен|
|`static`|вызов происходит **до** создания любого объекта: некому вызвать instance-метод|
|`void`|сам `main` ничего не возвращает; код выхода — через `System.exit(int)`|
|`String[] args`|аргументы командной строки; `java Hello a b` → `args = ["a","b"]`|

Ключевое здесь — `static`. JVM запускается, находит класс, и ей нужно вызвать `main` **до** того, как в программе появится хоть один объект. Нестатический метод вызывать не на чем — экземпляра ещё нет, и создавать его JVM не станет. Статический метод принадлежит классу, а не объекту, поэтому его можно вызвать напрямую.

> 💡 **Из C#:** в C# `Main` может быть `static int Main(string[] args)` и возвращать exit code прямо из сигнатуры. В Java `main` **всегда** `void`, exit code — только через `System.exit(int)`. Ещё в C# файл можно называть как угодно; в Java `public`-класс обязан лежать в одноимённом файле (`public class Hello` → `Hello.java`), иначе `javac` откажется компилировать.

#### Что реально происходит при `java Hello`

На уровне mid+ достаточно связной цепочки (детали каждого шага — в главах 11–12):

1. Стартует **JVM** — обычный нативный процесс ОС.
2. **ClassLoader** находит `Hello` через `classpath` (урок 1.1b) и грузит его байткод.
3. JVM ищет в классе `public static void main(String[])`, создаёт главный поток, вызывает `main`.
4. Байткод сначала **интерпретируется**; параллельно JVM собирает статистику вызовов.
5. Горячие методы **JIT** (`HotSpot`: C1 → C2, tiered) компилирует в нативный код.
6. Когда `main` возвращает управление (или вызван `System.exit`), отрабатывают shutdown hooks, и процесс завершается.

> 💡 **Из C#:** CLR делает ровно то же, только слои называются иначе: `AssemblyLoadContext` вместо `ClassLoader`, tiered JIT (`Tier 0 → Tier 1`) ↔ (`C1 → C2`). Если ты понимаешь запуск .NET-приложения, ты уже понимаешь запуск JVM на 80% — расходятся детали, не модель.

> Это ориентир на всю книгу, не тема для заучивания сейчас — каждая строка раскрывается в своей главе. Держи как компас.
> 
> **Работает так же (доверяй интуиции):**
> 
> - Процесс + heap/stack + область метаданных типов (`Metaspace` ↔ Loader Heap) + code cache.
> - Ленивая загрузка (`ClassLoader` ↔ `AssemblyLoadContext`) → детали в **гл. 11–12**.
> - Tiered JIT, generational GC со stop-the-world паузами → **гл. 11**.
> - Reflection, аннотации ↔ атрибуты → **гл. 12**. Модель потоков и happens-before (JMM ↔ ECMA) → **гл. 23**.
> 
> **Ломается — это настоящие архитектурные расхождения:**
> 
> |Тема|C#|Java|Где чиним|
> |---|---|---|---|
> |Value types|полноценные `struct` на стеке|только примитивы; свой `struct` нельзя (ждём Project Valhalla)|—|
> |Generics|reified, `typeof(T)`, перегрузка по `List<int>`/`List<string>`|**type erasure**, в рантайме `List` один|**гл. 7**|
> |Async|`async/await` в языке с 2012|`CompletableFuture`, затем **virtual threads** (JEP 444)|**гл. 24**|
> |Функции|`delegate` — конструкция языка|функциональные интерфейсы + лямбды на `invokedynamic`|**гл. 9**|
> |Свойства|`int Age { get; set; }`|`getAge()/setAge()` руками или `record`|**гл. 3, 17**|
> |Нативный код|`[DllImport]`|JNI, теперь Foreign Function & Memory API (JEP 454)|—|
> 
> Это твоё преимущество на собесе: большинство Java-кандидатов не видели, как те же концепции решены в CLR. Про .NET можно вслух не упоминать — но глубина понимания managed-рантаймов у тебя выше среднего.

> #### 🔬 Глубже / на синьора
> 
> **Байткод `main` под `javap`.** Дизассемблер показывает, что `println` вызывается инструкцией `invokevirtual` (виртуальная диспетчеризация), а сам `main` JVM зовёт через `invokestatic`:
> 
> ```
> public static void main(java.lang.String[]);
>   Code:
>      0: getstatic     #7   // System.out:PrintStream
>      3: ldc           #13  // String "Hello, Java"
>      5: invokevirtual #15  // PrintStream.println:(String)V
>      8: return
> ```
> 
> Отсюда прямой ответ на трюк-вопрос «можно ли `@Override` для `main`?» — нет. Статические методы не переопределяются, а **скрываются** (method hiding): `invokestatic` резолвится по типу, а не по объекту. Виртуальные вызовы и `invokevirtual` — тема гл. 4.
> 
> **Куда грузится класс — heap или нет.** Метаданные класса (байткод методов, constant pool, структура) живут в **`Metaspace`** — это нативная память вне heap (до Java 8 это был PermGen внутри heap, убрали в JEP 122). А вот объект-зеркало `Class<Hello>`, который возвращает `getClass()`, лежит в обычном heap. То есть «класс в Metaspace, но его `Class`-объект в heap» — обе части правды. Подробно — гл. 11.
> 
> **Скриптовый запуск без компиляции (JEP 330, Java 11+).** `java Hello.java` работает напрямую — JVM компилирует в памяти и запускает, `.class` на диск не пишется. Удобно для одноразовых скриптов и уроков; в проде так не собирают.
> 
> **Куда язык идёт: `main` без церемоний.** В Java 21 это preview (JEP 445 «Unnamed Classes and Instance Main Methods») — можно писать `void main()` без `public static` и без `class`, но только под `--enable-preview --source 21`:
> 
> ```
> $ java Unnamed.java
> error: unnamed classes are a preview feature and are disabled by default
> $ java --enable-preview --source 21 Unnamed.java
> unnamed + instance main
> ```
> 
> Финализировали это уже в **Java 25 (JEP 512, «Compact Source Files and Instance Main Methods»)**. Для baseline 21 держи в голове как форвард: «классический `public static void main` — не единственная форма, язык двигается к скриптовой точке входа».

#### ⚠️ Подводные камни

- **Имя `public`-класса ≠ имя файла.** `public class Different` в файле `Wrong.java` — не рантайм-ошибка, а отказ компилятора: `error: class Different is public, should be declared in a file named Different.java`. В C# такого ограничения нет вообще.
- **`main` не `static`.** Компилируется молча, но при запуске JVM падает с конкретным текстом (не «not found»!): `Error: Main method is not static in class …, please define the main method as: public static void main(String[] args)`.
- **`main` возвращает не `void`.** `public static int main(...)` компилируется, но на запуске: `Error: Main method must return a value of type void in class …`. Отдельная диагностика, отдельный текст.
- **`main` вообще отсутствует.** Вот **тут** будет `Error: Main method not found in class …`. Три разных симптома — три разных сообщения; не путай их (сырьё путало).
- **`args` — не `null`, а пустой массив.** При запуске без аргументов `args.length == 0`, а не `args == null`. `args[0]` без проверки длины → `ArrayIndexOutOfBoundsException`, не NPE.
- **`UnsupportedClassVersionError`.** `.class` собран новым `javac` (например, v65/Java 21) и запускается на старой JVM. Симптом деплоя, не кода: рантайм-Java старше compile-time-Java.

#### На собесе

**«Что происходит, когда ты запускаешь `java Hello`?»** _(частый, Middle)_  
Стартует JVM как нативный процесс → ClassLoader находит класс через classpath и грузит его байткод → JVM ищет `public static void main(String[])`, создаёт главный поток и вызывает его → байткод интерпретируется, горячие методы JIT-компилятор переводит в нативный код.  
→ _follow-up:_ «А байткод — это машинный код?» → **Нет**, это инструкции для JVM, платформонезависимые; машинным их делает JIT в рантайме. → _ловушка:_ «Значит `.class` можно запустить без Java?» → нет, без JVM это просто набор байтов.

> _EN model answer:_ “`java` starts the JVM as a native process. The ClassLoader locates the class on the classpath and loads its bytecode; the JVM finds `public static void main(String[])`, spins up the main thread and invokes it. Bytecode is interpreted first, and `HotSpot`’s JIT compiles hot methods to native code as the program runs.”

**«В чём разница между JDK, JRE и JVM?»** _(базовый, но спрашивают)_  
JVM — рантайм, исполняет байткод. JRE — JVM + стандартная библиотека, только для запуска. JDK — JRE + `javac` и инструменты, для разработки.  
→ _follow-up:_ «Зачем раньше ставили только JRE?» → экономия места на серверах, где компилятор не нужен; сегодня почти всегда ставят JDK.

**«Почему `main` обязан быть `static`?»** _(любят как проверку понимания)_  
JVM вызывает `main` до создания любого объекта — нестатический метод вызывать не на чем.  
→ _follow-up:_ «Можно `@Override` `main`?» → нет: статические методы не переопределяются, а скрываются (method hiding); вызов идёт через `invokestatic` по типу.

> _EN model answer:_ “`main` must be static because the JVM calls it before any instance of the class exists — there’s no object to invoke an instance method on. And no, you can’t override it: static methods are hidden, not overridden.”

**«Что такое байткод и зачем он нужен?»**  
Промежуточное платформонезависимое представление: `javac` его генерит, JVM исполняет. Один раз скомпилировал — запускаешь на любой ОС с совместимой JVM. Версия формата привязана к версии Java (v65 = Java 21), отсюда `UnsupportedClassVersionError`.

**Типичный слабый ответ, который проваливает:** «`java` запускает `.class`-файл». Формально мимо — `java` запускает **JVM-процесс**, который грузит и интерпретирует класс. Разница не педантичная: из неё следует, почему нужен classpath, почему jar — не бинарник и почему без JVM ничего не стартует (это уже мост в урок 1.1b).
