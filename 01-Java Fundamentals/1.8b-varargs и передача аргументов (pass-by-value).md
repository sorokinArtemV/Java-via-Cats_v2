Предыдущий урок был про то, _какой_ метод выберет компилятор. Этот — про то, что физически происходит с аргументами в точке вызова: как переменная арность (`String...`) превращается в массив и почему её злоупотребление бьёт по GC, и что именно копируется при передаче объекта в метод. Вторая половина — самый частый вопрос бывшему .NET-разработчику на собеседовании: «Java — pass-by-value или pass-by-reference?». Ответ односложный, но за ним стоит механика, которую половина инженеров объясняет неверно, и её нужно уметь развернуть на три follow-up вглубь.

**Ментальная модель.** varargs — это синтаксический сахар над массивом `T[]`, который компилятор аллоцирует на каждом вызове. А передача аргументов в Java — **всегда pass-by-value, без единого исключения**: для примитива копируется значение, для объекта — значение ссылки (не сам объект).

### varargs — это `T[]`, который лепит компилятор

Объявление `void log(String... args)` для JVM неотличимо от `void log(String[] args)` — в сигнатуре стоит массив. Сахар работает на стороне _вызова_: компилятор сам заворачивает переданные аргументы в новый массив. Посмотрим на байткод вызова `log("a", "b")` через `javap -c`:

```
0: iconst_2                    // размер массива = 2
1: anewarray  String           // ← создаём new String[2] прямо здесь
5: iconst_0
6: ldc        "a"
8: aastore                     // array[0] = "a"
10: iconst_1
11: ldc       "b"
13: aastore                    // array[1] = "b"
14: invokestatic log:([Ljava/lang/String;)V
```

Инструкция `anewarray` — это и есть тот массив, которого нет в исходнике. Отсюда два инженерных следствия, подтверждённых прогоном:

**Аллокация на каждый вызов.** Каждый `log(x, y)` создаёт новый `String[]` в куче (в Eden). В горячем цикле на миллион итераций — миллион короткоживущих массивов и лишнее давление на GC. Именно поэтому в JDK varargs-методы, вызываемые часто, продублированы фиксированными перегрузками. `List.of` имеет `of()`, `of(E)`, `of(E, E)` … до десяти элементов, и только потом `of(E...)` — чтобы типичные вызовы не платили за массив. Это ровно та причина «фиксированная бьёт varargs» из урока 1.8a, но с другой стороны: не про выбор, а про стоимость.

**Массив можно передать напрямую.** `log(arr)` не заворачивается повторно — внутри `args` окажется ровно `arr` (проверено: `length=3` для трёхэлементного массива). А `log()` без аргументов даёт **пустой** массив `length=0`, а не `null`.

> [!tip] 💡 Из C#
> прямой аналог — `params T[]`, механика идентична (компилятор так же аллоцирует массив на месте вызова). Отличие косметическое: в C# `params` пишется перед типом (`params object[] args`), в Java троеточие после типа (`Object... args`). Никаких концептуальных сюрпризов здесь нет — в отличие от следующего раздела.

> [!example] 🔬 Глубже / на синьора: heap pollution и `@SafeVarargs`
> 
> Проблема возникает, когда varargs-параметр _обобщённый_: `void f(List<String>... lists)`. Из-за стирания типов (глава 7) компилятор не может создать честный массив `List<String>[]` — массивы в Java reifiable, а `List<String>` нет. Он молча создаёт `List[]` (raw) и выдаёт unchecked-warning «possible heap pollution». Если внутри метода прочитать элементы как `List<String>`, а на деле туда через raw-ссылку положили `List<Integer>`, получишь `ClassCastException` в неожиданном месте. Аннотация `@SafeVarargs` на методе гасит этот warning — но это обещание разработчика, что метод только _читает_ varargs-массив и не даёт ему утечь наружу, а не проверка компилятора. Полностью эта тема разворачивается в главе 7; здесь достаточно знать, что дженерик + varargs — источник особого класса warning'ов.

> [!warning] ⚠️ Ловушка `null` + varargs
> 
> `f(null)` для `void f(Object... args)` неоднозначен для человека, но не для компилятора: `null` можно трактовать и как «весь массив равен `null`», и как «один элемент». Компилятор выбирает первое — `null` становится **самим массивом**. Прогон подтверждает все три варианта:
> 
> ```java
> static void f(Object... args) {
>     System.out.println(args == null ? "null array" : "length=" + args.length);
> }
> 
> f(null);            // "null array"   — null это массив
> f((Object) null);   // "length=1"     — null это единственный элемент
> f((Object[]) null); // "null array"   — явно: массив равен null
> ```
> 
> Причём `f(null)` компилируется **с предупреждением**, дословный текст которого стоит узнавать на слух (его цитируют на собесах):
> 
> ```
> warning: non-varargs call of varargs method with inexact argument type for last parameter;
>   cast to Object for a varargs call
>   cast to Object[] for a non-varargs call and to suppress this warning
> ```
> 
> Практическое следствие: если метод начнёт итерировать `args` не проверив на `null`, вызов `f(null)` даст `NullPointerException`. Каст к `(Object)` — способ передать одиночный `null`-элемент безопасно.

### pass-by-value: что копируется при передаче

Формулировку держи дословно: **Java — всегда pass-by-value, без исключений.** Копируется:

- для примитива — само значение (биты `int`, `double`, …);
- для объекта — **значение ссылки** (по сути handle/адрес), но не сам объект.

Параметр получает _копию_ ссылки. Копия и оригинал указывают на один объект в куче — отсюда иллюзия «pass-by-reference»: мутируешь объект через параметр, снаружи видно. Но передалась копия ссылки, а не переменная вызывающего.

![[Pasted image 20260702163044.png]]

Два сценария, между которыми — вся суть темы. Оба прогнаны:

```java
static void mutate(Account acc)   { acc.setBalance(0); }    // через ссылку — объект общий
static void reassign(Account acc) { acc = new Account(999); } // переприсваивание — двигает копию

Account a = new Account(100);
mutate(a);     // a.getBalance() == 0    ← изменили общий объект, видно снаружи
a.setBalance(100);
reassign(a);   // a.getBalance() == 100  ← не изменилось! a осталась на старом объекте
```

`mutate` вызывает `setBalance` на объекте, на который смотрит копия `acc` — а это тот же объект, что и `a`. Изменение общее. `reassign` присваивает `acc = new Account()` — переставляет _локальную копию_ ссылки на другой объект; переменная `a` вызывающего про это не знает.

Идеальный тест на понимание — попытка написать `swap`. Прогон подтверждает, что он не работает:

```java
static void swap(Object x, Object y) { Object t = x; x = y; y = t; }

String p = "left", q = "right";
swap(p, q);
// p, q по-прежнему "left", "right" — переставились копии ссылок внутри метода
```

`swap` двух переменных через метод в Java **невозможен в принципе** — и это прямое доказательство pass-by-value. Примитив ведёт себя так же очевидно: `incr(int x) { x++; }` не меняет переменную снаружи (`5` остаётся `5`).

> [!tip] 💡 Из C#
> здесь главное расхождение. В C# есть `ref`/`out` — настоящая передача переменной по ссылке, и `swap(ref a, ref b)` работает. **В Java аналога нет вообще** — это первое, что спросят у бывшего .NET-разработчика. Готовая формулировка: _"Java is always pass-by-value; references are passed by value too."_ Если нужно «вернуть» переприсвоенную ссылку — либо `return`, либо holder.

### Как обойти отсутствие `ref`/`out`

Когда переписать на `return` нельзя, применяют holder — одноэлементный массив или mutable-обёртку. Прогон подтверждает, что через holder «swap» работает, потому что мутируется общий объект-контейнер, а не переставляется ссылка:

```java
static void swapHolder(Object[] box) { Object t = box[0]; box[0] = box[1]; box[1] = t; }

Object[] box = {"left", "right"};
swapHolder(box);   // box[0]="right", box[1]="left" — сработало
```

Это тот же принцип, что и `mutate`: массив — общий объект, его содержимое меняется, изменение видно снаружи. Вместо `Object[]` можно взять `AtomicReference` или свой mutable-класс.

Но holder в идиоматичной Java избегают. **Чтобы вернуть несколько значений, возвращают `record`** — маленький именованный тип-результат (`record MinMax(int min, int max) {}`), это Java-идиома вместо кортежей и `out`-параметров. Для пары «ключ-значение» на месте иногда берут `Map.entry(k, v)`. Полный разбор records — глава 17; здесь достаточно указателя: holder — крайний случай, `record` — стандарт.

> [!warning] ⚠️ Подводные камни
> 
> **Строки усиливают иллюзию «by-reference наоборот».** `swap` со `String` не меняет переменные — но не из-за иммутабельности `String`, а из-за pass-by-value. Даже с mutable-объектами `swap` тела `x = y` не сработает. Не путай два независимых факта: `String` иммутабелен (нельзя изменить объект) и переменные передаются по значению (нельзя переставить чужую переменную).
> 
> **`final` на параметре ничего не говорит о вызывающем.** `void f(final Account acc)` запрещает _переприсвоить_ `acc` внутри метода, но никак не влияет на видимость мутаций снаружи и не делает передачу «by-reference». Это локальное ограничение на тело метода, не контракт с вызывающим.
> 
> **Мутация через параметр — источник неожиданных side-эффектов.** Метод, принимающий `List` и вызывающий `list.add(...)`, меняет коллекцию вызывающего — потому что ссылка общая. Это не баг pass-by-value, а его следствие; защита — defensive copy (глава 20), а не «Java by-reference».
> 
> **Автобоксинг + varargs = тихая аллокация в цикле.** `String.format("%d", n)` в горячем цикле аллоцирует и `Object[]`-массив под varargs, и `Integer` под боксинг `n`. Два источника мусора на вызов; в перф-критичном коде это заметно.
> 
> **`f(null)` при перегрузке varargs и не-varargs.** Если есть и `f(Object[])`, и `f(Object...)` — они конфликтуют так же, как обсуждалось в 1.8a, потому что для JVM это одна сигнатура `f(Object[])`; объявить обе нельзя.

### Вопросы с собеса (с разбором)

> [!question]- 1. «Java — pass-by-value или pass-by-reference?» _(частый, именно для .NET-бэкграунда)_
> Ответ: всегда pass-by-value. Для примитивов копируется значение, для объектов — значение ссылки; сам объект не копируется. _Follow-up:_ «Но я меняю объект в методе, и снаружи видно — разве не by-reference?» → Разделяй мутацию и переприсваивание. Мутация (`acc.setX()`) идёт через копию ссылки в _общий_ объект — видна. Переприсваивание (`acc = new...`) двигает только копию ссылки — не видно. _Follow-up 2:_ «Напиши `swap(a, b)`.» → Невозможно: переставишь копии ссылок. Аналога `ref`/`out` нет; либо `return`, либо holder. _Слабый ответ, который валит follow-up:_ «примитивы by-value, объекты by-reference» — это распространённая, но неверная формулировка; на первом же follow-up про `swap` или `reassign` она разваливается.

> [!question]- 2. «Что печатает `f(null)` для `void f(Object... args)`?»
> Ответ: `args == null` — `null` трактуется как весь массив, не как элемент. Итерирование → `NullPointerException`. Компилируется с warning. _Follow-up:_ «Как передать один `null`-элемент и как — null-массив явно?» → `f((Object) null)` даёт `length == 1`; `f((Object[]) null)` даёт `args == null` без warning. _Follow-up 2:_ «Почему вообще warning?» → Компилятор видит неоднозначность «массив или элемент» и просит разрешить её кастом.

> [!question]- 3. «Что происходит с varargs на уровне байткода?»
> Ответ: компилятор аллоцирует массив `T[]` на месте вызова (`anewarray`), заполняет `aastore` и передаёт как обычный массив-параметр. _Follow-up:_ «Чем это грозит?» → Аллокация на каждый вызов, давление на GC в горячем пути; поэтому в JDK частые varargs-методы дублированы фиксированными перегрузками (`List.of`). _Follow-up 2 (senior):_ «А если varargs дженерик?» → Heap pollution из-за стирания типов, unchecked-warning, гасится `@SafeVarargs` при условии, что метод только читает массив.

> [!question]- 4. «Меняет ли метод переменную вызывающего, если внутри делает `param = newValue`?»
> Ответ: нет — переприсваивается локальная копия ссылки (или значения). Снаружи не видно. _Follow-up:_ «А `param.setX(...)`?» → Да, видно: мутируется общий объект. _Follow-up 2:_ «`final` на параметре что-то меняет?» → Только запрещает переприсваивание внутри метода; на вызывающего не влияет.

> [!question]- 5. «Как вернуть несколько значений из метода?»
> Ответ: `record` — идиоматичный способ (Java 16+, глава 17); `Map.entry` для пары; holder/массив — крайний случай; `out`/`ref` в Java нет. _Follow-up:_ «Почему не holder по умолчанию?» → Мутабельный контейнер хуже читается и опаснее в конкурентном коде; `record` иммутабелен и самодокументируется именами полей.

### На собесе

«Java — pass-by-value всегда, без исключений. Для примитивов копируется значение, для объектов — значение ссылки. Сам объект не копируется, поэтому через переданную ссылку я мутирую общий объект, и это видно снаружи. Но если переприсвою параметр — `obj = new ...` — двигается только локальная копия ссылки, оригинал не трогается. `swap` двух переменных через метод в Java невозможен — это и есть доказательство pass-by-value. `ref`/`out` из C# здесь нет; несколько значений возвращаю `record`'ом.»

«varargs — это `T[]`, компилятор аллоцирует массив на каждом вызове, в байткоде это `anewarray`. Поэтому в горячем цикле varargs давит на GC, и частые методы в JDK дублированы фиксированными перегрузками — `List.of` до десяти элементов. `f(null)` для `Object...` — это null-массив, не элемент; чтобы передать один null, кастуй к `Object`. Дженерик + varargs — это heap pollution и `@SafeVarargs`.»

**EN, компактно:** "Java is strictly pass-by-value. For primitives the value is copied; for objects the reference value is copied, never the object itself — so mutating through the parameter is visible to the caller, but reassigning the parameter isn't, and a `swap` of two variables via a method is impossible. There's no `ref`/`out` as in C#; to return multiple values you return a `record`. varargs is sugar over a `T[]` the compiler allocates at each call site (`anewarray` in bytecode), which is why hot-path methods in the JDK ship fixed-arity overloads alongside the varargs one. `f(null)` for `Object...` binds `null` to the whole array, not to a single element."
