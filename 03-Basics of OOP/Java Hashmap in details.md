Хорошо, продолжаем — разберём, что происходит внутри `HashMap`, когда ты пишешь `map.get(key)`. Прошлый раз мы сравнили его раскладку с `Dictionary`, теперь копнём саму механику Java на уровне, который спрашивают на мидл-собесах.

---

Ты кладёшь в `HashMap` миллион записей, потом просишь `map.get(someKey)` — и ответ приходит мгновенно, будто записей там три, а не миллион. Как он находит один ключ среди миллиона, не перебирая их? Ответ в том, что он _не ищет_ — он **вычисляет**, где ключ обязан лежать, и идёт прямо туда. Поиск превращается в арифметику.

И вот тут прячется неприятный сюрприз: эта арифметика держится целиком на твоём `hashCode`. Пока он раскидывает ключи равномерно — всё O(1). Стоит ему начать выдавать одинаковые числа — и `HashMap` тихо деградирует. До Java 7 — до O(n), до скорости связного списка. С Java 8 — деградация мягче, до O(log n), потому что появилась защита, о которой половина кандидатов не знает. Разберём и нормальный режим, и обе поломки.

### Массив бакетов и ленивая инициализация

Внутри `HashMap` — массив `Node<K,V>[] table`. Каждая ячейка массива называется **бакетом** (bucket). `Node` хранит четыре вещи: закешированный `hash`, `key`, `value` и ссылку `next` на следующий узел в этом бакете.

Дефолтная начальная ёмкость (capacity, длина массива) — 16, но сам массив создаётся **лениво**, при первом `put`, а не в конструкторе. Это экономит память на мапах, которые создали и не наполнили. Capacity всегда степень двойки — даже если ты передал в конструктор `new HashMap<>(20)`, внутри он округлится вверх до 32. Почему именно степень двойки — увидишь через раздел.

> Источник: константы и поведение — `java.util.HashMap` (`DEFAULT_INITIAL_CAPACITY = 16`, `DEFAULT_LOAD_FACTOR = 0.75f`), Javadoc `HashMap`.

### Как ключ превращается в индекс

Чтобы найти бакет, `HashMap` делает два шага. Сначала зовёт `key.hashCode()`. Потом — и вот это неочевидно — **размешивает** результат, прежде чем брать индекс:

```java
// из java.util.HashMap, дословно
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
// индекс бакета = hash & (table.length - 1)
```

Индекс — это `hash & (n - 1)`, где `n` — длина таблицы. Раз `n` степень двойки, `n - 1` — это маска из одних единиц в младших битах (для `n=16` это `0b1111`). Битовое `&` с такой маской — то же самое, что «взять остаток от деления на `n`», только в разы дешевле деления. Вот зачем степень двойки: индексация превращается в одну битовую операцию.

Но у маски есть цена: она смотрит **только на младшие биты** хеша, старшие отбрасываются. Если у тебя ключи, чьи `hashCode` отличаются только в старших битах (а младшие совпадают), они все сядут в один бакет. Поэтому перед маской идёт `h ^ (h >>> 16)` — сдвигаем старшие 16 бит вниз и подмешиваем их XOR-ом в младшие. Теперь младшие биты несут «эхо» старших, и маска видит разнообразие.

![[Pasted image 20260605072959.png]]

> 💡 **Из C#:** прошлый раз видели — `Dictionary` решает ту же проблему распределения иначе: размер таблицы простое число и индекс через `% size`. Java выбрала «степень двойки + маска» (быстрее деления) и компенсирует потерю старших бит размешиванием. Два разных компромисса под одну задачу.

### Коллизии: цепочка, а потом дерево

Два разных ключа могут дать один индекс — это **коллизия**, и она нормальна (хешей всего 2³², ключей может быть больше). Бакет тогда держит не один `Node`, а цепочку, связанную через `next` — ту самую раскладку, что мы рисовали в прошлый раз.

`put` в бакет работает так: считаем индекс, идём по цепочке и для каждого узла сначала сравниваем закешированный `hash` (дешёвый `int`), и только при совпадении хеша зовём `equals` (дорого). Нашли равный ключ — заменяем `value`; дошли до конца — добавляем новый узел. `get` идёт по той же цепочке с тем же сравнением `hash` → `equals`. Вот где всплывает контракт из прошлого урока: `hashCode` выбирает бакет, `equals` сравнивает внутри. Закешированный в `Node` хеш — это оптимизация: `hashCode` ключа за всё время жизни узла зовётся один раз.

Теперь главная Java-специфика. Если цепочка в одном бакете дорастает до **8** узлов _и_ таблица при этом не меньше **64** ячеек — список конвертируется в **красно-чёрное дерево** (`TreeNode`). Поиск в бакете становится O(log n) вместо O(n). Это защита от плохого или враждебного `hashCode`: даже если кто-то специально подсунул ключи-коллизии (hash-flooding атака), бакет не выродится в линейный список. Если же таблица меньше 64 — вместо деревенения происходит `resize`: расширение перераспределит узлы и, скорее всего, укоротит цепочку дешевле, чем строить дерево.

> ⚠️ **Подводный камень:** дереву нужно как-то _упорядочивать_ ключи. Если ключ реализует `Comparable` — используется `compareTo`. Если нет — `HashMap` падает на запасной tie-break (сравнение по имени класса и `System.identityHashCode`). Дерево всё равно работает, но менее эффективно. Так что для ключей, которые могут коллизить пачками, `Comparable` — плюс. Пороги: деревенеет на 8, разворачивается обратно в список на **6** (`UNTREEIFY_THRESHOLD`), чтобы не дёргаться туда-сюда на границе.

> Источник: пороги `TREEIFY_THRESHOLD = 8`, `MIN_TREEIFY_CAPACITY = 64`, `UNTREEIFY_THRESHOLD = 6` — поля `java.util.HashMap`; деревенение и улучшенное хеширование добавлены в Java 8 (JDK-8023463).

### Resize и load factor

Бакетов фиксированное число, а ключей всё больше — цепочки удлиняются, O(1) тает. Чтобы этого не было, `HashMap` следит за заполненностью через **load factor** (коэффициент загрузки), по умолчанию `0.75`. Порог расширения = `capacity * loadFactor`, то есть для дефолтных 16 это 12. Как только `size` переваливает за порог — таблица удваивается (16 → 32 → 64 …), и узлы перераспределяются по новым бакетам.

0.75 — компромисс: меньше (0.5) — меньше коллизий, но больше пустой памяти и более частые resize; больше (1.0) — плотнее, но длиннее цепочки. Javadoc прямо называет 0.75 хорошим балансом времени и памяти.

Самое интересное — _как_ Java 8 перераспределяет. Наивно — пересчитать индекс каждому узлу. Java 8 заметила: при удвоении ёмкости новый индекс узла — это либо тот же индекс, либо «старый индекс + старая ёмкость», и решает это **ровно один бит** хеша (`hash & oldCap`). Узлы каждого бакета разделяются на две группы — «остаются» (lo) и «переезжают» (hi) — без полного пересчёта и с сохранением порядка:

![[Pasted image 20260605073014.png]]

`resize` сам по себе O(n) — нужно пройти все узлы. Это амортизируется (происходит редко, при удвоении), но если ты заранее знаешь размер — передай initial capacity в конструктор и сэкономь серию расширений: `new HashMap<>(expectedSize / 0.75 + 1)`.

> ⚠️ **Подводный камень:** в Java 7 при перераспределении узлы перецеплялись в _обратном_ порядке, и при одновременном resize из двух потоков цепочка могла свернуться в кольцо — поток уходил в бесконечный цикл на 100% CPU (классический «HashMap infinite loop»). Java 8 сохраняет порядок (lo/hi-разделение выше) и кольцо больше не образует. Но это **не сделало `HashMap` потокобезопасным** — конкурентный доступ всё равно теряет данные. Для многопоточки — `ConcurrentHashMap`.

### null, мутации и потокобезопасность

`HashMap` разрешает **один** `null`-ключ (его хеш принудительно 0, см. `hash()` выше — он лежит в бакете 0) и сколько угодно `null`-значений.

java

```java
Map<String, Integer> m = new HashMap<>();
m.put(null, 42);
System.out.println(m.get(null)); // 42 — легально
```

> 💡 **Из C#:** `Dictionary` на `null`-ключ кидает `ArgumentNullException`. И `ConcurrentHashMap` в Java тоже не терпит `null` ни в ключе, ни в значении — так что «`null`-ключ ок» верно именно для `HashMap`, не для всей иерархии.

Самый коварный баг — **мутабельный ключ**. Положил объект, потом изменил поле, входящее в его `hashCode` — и узел остался физически в старом бакете, а `get` ищет уже в новом. Ключ «теряется»: не находится, но из мапы не исчезает.

java

```java
Map<List<Integer>, String> map = new HashMap<>();
List<Integer> key = new ArrayList<>(List.of(1, 2));
map.put(key, "value");
key.add(3);                       // hashCode ключа изменился
System.out.println(map.get(key)); // null — не нашли
System.out.println(map.size());   // 1 — но он всё ещё внутри
```

Лечится одним правилом: **ключи должны быть иммутабельными** (или хотя бы не меняться, пока лежат в мапе). Поэтому `String`, числа, `enum`, `record` — идеальные ключи.

И последний edge case — **fail-fast** итератор. Структурно изменил мапу (put нового ключа / remove) во время обхода — получишь `ConcurrentModificationException`:

java

```java
for (String k : m.keySet()) {
    if (k.equals("a")) m.remove(k);   // → ConcurrentModificationException
}
// правильно: Iterator<String> it = ...; it.remove();
```

Это не гарантия, а best-effort защита (через счётчик `modCount`): она помогает ловить баги, но полагаться на неё как на «детектор конкурентности» нельзя.

### 🐱 На котиках

Кот Барсик на сортировке писем. Полок 16, и Барсик кладёт письмо на полку по _последней цифре_ индекса получателя. Приходят письма на индексы `10005`, `90005`, `40005` — последняя цифра у всех `5`, и весь поток валится на одну полку, остальные пустуют. Барсик в панике перебирает гору на пятой полке.

Приходит кошка-наставница: «Ты смотришь только на хвост индекса. Сложи-ка первую цифру с последней, тогда `10005` даст `6`, `90005` даст `14`, `40005` даст `9` — разные полки». Барсик так и делает — гора рассасывается. Это и есть `h ^ (h >>> 16)`: подмешать «голову» числа в «хвост», по которому считается полка, чтобы письма, похожие с конца, всё равно разъехались.

А когда на одну полку всё-таки набивается восемь писем, Барсик перестаёт раскладывать их стопкой и заводит на этой полке маленькую картотеку по алфавиту — искать в ней быстрее, чем перелопачивать стопку. Это treeify.

### Вопросы с собеса (с разбором)

**1. «Как `HashMap` достаёт значение за O(1)? И когда перестаёт?»**

Ответ: он не ищет перебором — вычисляет индекс бакета из хеша ключа и идёт прямо туда; внутри бакета сравнивает `hash`, потом `equals`. O(1), пока хеш раскидывает ключи равномерно. Ломается при плохом `hashCode`: много коллизий → длинные бакеты.

Follow-up: _«Что если `hashCode` всегда возвращает 0?»_ — Все ключи в бакете 0. До Java 8 — O(n) (список). С Java 8, когда наберётся ≥8 и таблица ≥64, бакет деревенеет → O(log n). Корректность не страдает, но это деградация. И если ключи не `Comparable`, дерево строится на запасном tie-break — работает, но слабее.

**2. «Зачем размешивать `h ^ (h >>> 16)`?»**

Ответ: индекс берётся как `hash & (n-1)` — это только младшие биты. Ключи, различающиеся лишь в старших битах, сядут в один бакет. Размешивание сдвигает старшие биты в младшие, чтобы маска их «увидела».

Follow-up: _«Почему не просто `% n`?»_ — `&`-маска по степени двойки дешевле деления. Java выбрала скорость индексации, расплата — потеря старших бит, которую и компенсирует размешивание. Отсюда же требование «capacity — степень двойки».

**3. «Что происходит при resize и почему Java 8 его переписала?»**

Ответ: таблица удваивается, узлы перераспределяются. Java 8 делит каждый бакет на две группы по одному биту (`hash & oldCap`): либо тот же индекс, либо `index + oldCap`. Без полного пересчёта, с сохранением порядка.

Follow-up: _«А почему в Java 7 был бесконечный цикл?»_ — Перецепление в обратном порядке при конкурентном resize сворачивало цепочку в кольцо. Java 8 это устранила сохранением порядка, но `HashMap` остался непотокобезопасным — для конкурентности `ConcurrentHashMap`.

**4. «Положил ключ, потом изменил его поле — что будет?»**

Ответ: если поле входит в `hashCode`/`equals`, узел остался в старом бакете, а `get` ищет в новом → `null`. Ключ не находится, но из мапы не пропал (виден в итерации, считается в `size`).

Follow-up: _«Как предотвратить?»_ — Иммутабельные ключи: `String`, числа, `enum`, `record`. Никогда не мутировать объект, лежащий ключом в мапе.

**5. «Когда цепочка становится деревом и что для этого нужно от ключей?»**

Ответ: при ≥8 узлах в бакете И таблице ≥64 (иначе вместо дерева — resize). Дерево даёт O(log n) и защищает от hash-flooding.

Follow-up: _«Что нужно от ключей?»_ — Желательно `Comparable` (дерево упорядочивает по `compareTo`); без него — запасной tie-break по классу и identity hash. Обратно в список — на пороге 6.

**6. «Сколько `null`-ключей можно?»**

Ответ: один (бакет 0), `null`-значений — сколько угодно.

Follow-up: _«А в `ConcurrentHashMap`?»_ — Там `null` запрещён и в ключе, и в значении (NPE), чтобы не путать «нет ключа» и «ключ есть, значение null» в конкурентном контексте.

### Практика (меню, выбери что зацепит)

1. **Свой мини-`HashMap`** (~1.5–2 ч). Массив бакетов + цепочки, `put`/`get`/`remove`, resize при load factor. Без деревьев. Прочувствуешь индексацию и коллизии руками.
2. **Спровоцируй treeify** (~20 мин). Класс-ключ с `hashCode()` возвращающим константу, положи 10+ штук в мапу, поставь брейкпоинт и посмотри в дебаггере, что узлы бакета стали `TreeNode`, а не `Node`.
3. **Цена resize** (~20 мин). Вставь 1М записей в `new HashMap<>()` и в `new HashMap<>(2_000_000)`, замерь время. Разница — это серия пропущенных удвоений.
4. **Потеря мутабельного ключа** (~10 мин). `ArrayList` как ключ, мутируй после `put`, убедись что `get` даёт `null`, а `size` остаётся 1. Потом попробуй найти его обходом `entrySet`.
5. **Поймай `ConcurrentModificationException`** (~10 мин). Удали элемент в for-each, получи исключение, почини через `Iterator.remove()`.

### На собесе

`HashMap` — это массив бакетов (`Node[] table`), индекс ключа считается как `hash & (n-1)`, где `n` — степень двойки, а `hash` — это `key.hashCode()`, предварительно размешанный через `h ^ (h >>> 16)`. Размешивание нужно, потому что маска `& (n-1)` берёт только младшие биты; без него ключи, различающиеся в старших битах, коллизили бы. Степень двойки выбрана ради дешёвой маски вместо деления.

Коллизии живут в бакете цепочкой (`Node.next`); поиск сравнивает сначала закешированный `hash`, потом `equals`. С Java 8, если в бакете набирается 8 узлов и таблица ≥64, список превращается в красно-чёрное дерево — O(log n) вместо O(n), защита от hash-flooding; обратно в список на 6. Дереву желательны `Comparable`-ключи.

Заполненность отслеживается через load factor 0.75: при `size > capacity * 0.75` таблица удваивается. Java 8 при resize делит бакет на две группы по одному биту (`hash & oldCap`) без полного пересчёта и с сохранением порядка — это заодно убрало бесконечный цикл Java 7 при конкурентном resize. Но `HashMap` всё равно не потокобезопасен — для многопоточки `ConcurrentHashMap`.

Граничные случаи держи наготове: один `null`-ключ разрешён (бакет 0), `null`-значений сколько угодно; мутабельный ключ — баг, узел теряется в старом бакете (ключи должны быть иммутабельны); итератор fail-fast, кидает `ConcurrentModificationException` при структурной модификации в обходе.

Если просят одной фразой про сложность: «амортизированно O(1) на `get`/`put` при хорошем хеше; худший случай в одном бакете — O(log n) с Java 8 благодаря деревенению, раньше был O(n)». Если спросят отличие от `Dictionary` в C# — цепочки через объекты-узлы и деревья против плоского `Entry[]` с индексной связностью, степень двойки против простого числа.

### Скорость операций `HashMap`

|Операция|Средняя|Худший случай (Java 8+)|Почему так|
|---|---|---|---|
|`get` / `put` / `remove` / `containsKey`|**O(1)**|**O(log n)**|амортизированно; худший — деревенелый бакет|
|`containsValue`|O(n)|O(n)|по значениям индекса нет — полный скан таблицы|
|`size`|O(1)|O(1)|кешированное поле, не считается обходом|
|итерация (`entrySet`/`keySet`)|O(capacity + size)|—|пробегает **и пустые бакеты** массива|
|`resize`|O(n)|O(n)|редко, амортизируется в `put`|

Три уточнения, на которых ловят:

«O(1) средняя» опирается на два допущения: `hashCode` раскидывает ключи равномерно, и амортизация съедает периодический resize. Сломай первое плохим хешем — деградируешь к O(log n) (а до Java 8 было к O(n), скорость списка).

`containsValue` — это **O(n)**, а не O(1). Мапа индексирует ключи, а не значения. Если тебе часто нужен поиск по значению — нужна вторая мапа в обратную сторону (или `BiMap` из Guava).

> ⚠️ **Подводный камень:** итерация — это **O(capacity + size)**, а не O(size). Ты пробегаешь весь массив бакетов, включая пустые. Создал `new HashMap<>(10_000_000)`, положил 5 элементов — обход будет ползти по десяти миллионам пустых ячеек. Это, кстати, преимущество `LinkedHashMap`: он итерируется за чистый O(size), потому что идёт по своему связному списку записей, а не по массиву бакетов.

### Реализации `Map` в Java

Да, выбор есть — но устроен он чище, чем в .NET. В .NET у тебя набор отдельных классов (`Dictionary`, `SortedDictionary`, `SortedList`, `ConcurrentDictionary`…). В Java один интерфейс `Map` и несколько реализаций под ним, а выбираешь по двум осям: **нужен ли порядок** и **нужна ли потокобезопасность** (плюс спец-случаи вроде enum-ключей).

|Реализация|Порядок обхода|get/put|Когда брать|Аналог в .NET|
|---|---|---|---|---|
|`HashMap`|никакого|O(1)|дефолт, порядок не важен|`Dictionary`|
|`LinkedHashMap`|вставки или доступа|O(1)|нужен предсказуемый порядок; LRU-кеш|— (нет прямого)|
|`TreeMap`|сортировка по ключу|O(log n)|порядок ключей + range-запросы|`SortedDictionary`|
|`ConcurrentHashMap`|никакого|O(1)|многопоточный доступ|`ConcurrentDictionary`|
|`EnumMap`|объявления enum|O(1)*|ключи — один `enum`|— (нет)|
|`Map.of` / `Map.copyOf`|не определён|O(1)|маленькая неизменяемая мапа|`ImmutableDictionary`|

_* O(1) с крошечной константой — внутри обычный массив, индекс = `ordinal()` ключа._

Что стоит за тремя главными:

**`LinkedHashMap`** — это `HashMap` плюс двусвязный список, прошивающий все записи. Операции остаются O(1), добавляется лишь обновление пары указателей. По умолчанию обход в порядке _вставки_. Но если создать с флагом `accessOrder=true` и переопределить `removeEldestEntry`, получаешь **LRU-кеш в десять строк** — записи переупорядочиваются при доступе, и самая старая выпадает первой. Это классический интервью-приём.

**`TreeMap`** — красно-чёрное дерево, ключи всегда отсортированы (по natural order или по переданному `Comparator`). Платишь O(log n) за каждую операцию, но получаешь то, чего хеш-мапа не умеет: реализует `NavigableMap` → `firstKey`/`lastKey`, `floorKey`/`ceilingKey` (ближайший снизу/сверху), `subMap`/`headMap`/`tailMap` (диапазоны). Берёшь, когда нужны упорядоченный обход или запросы «найди ближайший ключ ≤ X».

**`ConcurrentHashMap`** — потокобезопасный, но без грубой блокировки всей мапы: чтения почти без локов (CAS), записи лочат только отдельный бакет. Отсюда высокая параллельность. Не принимает `null` ни в ключе, ни в значении, итератор weakly-consistent (не кидает `ConcurrentModificationException`).

> 💡 **Из C#:** две Java-реализации не имеют чистого аналога в .NET. `LinkedHashMap` (предсказуемый порядок без сортировки) — в .NET обычно лепят `Dictionary` + `List` руками. `EnumMap` (массив по `ordinal`) — тоже отсутствует. Зато `SortedList<K,V>` из .NET (отсортированный, но на массиве, дешёвая память / дорогая вставка) в Java прямого аналога не имеет — `TreeMap` всегда на дереве. И ещё: Java _явно_ разделяет «хочу порядок → бери `LinkedHashMap`/`TreeMap`», тогда как `Dictionary` в .NET сохранял порядок вставки как недокументированную деталь реализации (и ломал её на `remove`) — на это полагаться было нельзя.

Нишевые, чтоб знать что есть: `Hashtable` — legacy, синхронизирован целиком, без `null`, не использовать (есть `HashMap`/`ConcurrentHashMap`). `IdentityHashMap` — сравнивает ключи по `==`, а не `equals` (нужен редко — например, при обходе графа объектов). `WeakHashMap` — ключи на слабых ссылках, GC их забирает; для кешей и канонизирующих мап.

Короткое правило выбора: по умолчанию `HashMap`; нужен стабильный порядок обхода — `LinkedHashMap`; нужен порядок _по ключу_ или range-запросы — `TreeMap` (и плати O(log n)); многопоточка — `ConcurrentHashMap`; ключи-`enum` — `EnumMap`.

#### Зачем нужен отдельный enum map?

Потому что для enum-ключей всё устройство `HashMap` — хеш, бакеты, узлы, разрешение коллизий — становится лишним. У enum фиксированный, заранее известный набор констант, и у каждой есть `ordinal()` — порядковый номер 0, 1, 2… в порядке объявления. А раз у тебя плотный диапазон целых от 0 до N−1, ключ можно использовать прямо как индекс в обычном массиве. Никакого хеширования не нужно.

`EnumMap` внутри — это плоский `Object[]` длиной во всё число констант enum-а:

java

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

EnumMap<Day, String> hours = new EnumMap<>(Day.class);
hours.put(Day.MON, "9–18");
hours.put(Day.SAT, "10–14");
hours.get(Day.MON);   // внутри ≈ vals[MON.ordinal()] → vals[0]
```

`put` — это `vals[key.ordinal()] = value`, `get` — `return vals[key.ordinal()]`. Один доступ к массиву по индексу, и всё.

Что это даёт против `HashMap`:

- **Скорость.** O(1) с минимально возможной константой: ни вызова `hashCode`, ни прохода по бакету, ни `equals`. Просто индексация массива.
- **Память.** Один массив на N элементов. Нет `Node`-объектов (в `HashMap` каждая запись — отдельный объект в куче с полями hash/key/value/next), нет запаса пустых бакетов под load factor, нет накладных расходов на хеш-таблицу. Очень компактно, и всё лежит подряд — хорошая cache locality.
- **Порядок обхода бесплатно.** Итерируешь массив по индексам — получаешь константы в порядке объявления, детерминированно. `LinkedHashMap` для этого не нужен.
- **Коллизий не бывает в принципе** — у каждого ключа свой персональный слот.

Главное «зачем» по Блоху (Effective Java, Item 37): индексировать массив по `ordinal()` ты мог бы и руками — `Object[] arr; arr[day.ordinal()] = x`. Но это хрупко и небезопасно: нет типобезопасности (можно сунуть любой `int`), нужен каст на чтении, и всё ломается, если кто-то переставит или добавит константу в enum. `EnumMap` — это ровно тот же приём «массив по ordinal», но инкапсулированный, типобезопасный и устойчивый к изменениям enum-а. Поэтому правило простое: ключи — `enum` → бери `EnumMap`, а не `HashMap` и не ручной массив.

Ограничения: все ключи одного enum-типа (поэтому в конструктор передаёшь `Day.class` — так он узнаёт размер вселенной ключей), не потокобезопасен, `null`-ключ запрещён (`null`-значения можно — внутри отличает «нет записи» от «значение null» через служебный маркер).

> 💡 **Из C#:** прямого аналога нет — ты бы взял `Dictionary<Day, string>` и платил за хеширование там, где Java читает по индексу. Заодно есть парный `EnumSet`: множество enum-значений на битовой маске (один `long` на ≤64 константы), где операции объединения/пересечения — это побитовые `|`/`&`. Та же идея «enum → компактная структура без хеша», только для `Set`.
