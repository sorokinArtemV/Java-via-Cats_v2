В твоём банковском бэкграунде объекты вроде `Money`, `Trade`, `AccountId` кочуют по всему коду: их кладут в мапы, шарят между потоками, кешируют, передают в чужие методы. Если такой объект можно случайно изменить после создания — начинается ад невоспроизводимых багов: значение в `HashMap` «уехало» из своего бакета, два потока увидели разное состояние, метод-приёмник тихо подпортил то, что ему передали для чтения. Иммутабельность — это дисциплина, которая делает подобные баги невозможными _по построению_, а не отлавливаемыми тестами. Ты платишь за неё аллокацией нового объекта на каждое «изменение» — и взамен получаешь потокобезопасность без единого лока, безопасное кеширование и объект, за состояние которого не надо переживать. В Java это стоит освоить плотно: половина стандартной библиотеки (`String`, `Integer`, `LocalDate`, `BigDecimal`) неизменяема именно ради этого, и на собесах требование «напиши immutable-класс» — почти гарантированный вопрос.

**Ментальная модель.** Immutable-объект — тот, чьё наблюдаемое состояние не меняется после выхода из конструктора. В Java это не одно ключевое слово, а комбинация: `final`-класс + `private final`-поля + отсутствие мутаторов + защитные копии на _обеих_ границах, если внутри лежит что-то изменяемое. Причём `final` на поле защищает ссылку, а не объект за ней — и на этом зазоре ломается большинство «почти иммутабельных» классов.

### Пять требований — и что ломается без каждого

Effective Java (Item 17, «Minimize mutability») даёт пять требований к immutable-классу. Ценность не в том, чтобы вызубрить список, а в том, чтобы понимать, _какой конкретно баг открывает пропуск каждого пункта_ — на собесе спрашивают именно это.

**1. Класс `final` (или `private` конструктор + статические фабрики).** Без этого наследник добавит изменяемое состояние или переопределит метод, нарушив инвариант, — и метод, принимающий твой тип, больше не может доверять неизменяемости:

```java
public class Money {                  // забыли final
    private final long amountMinor;
    public Money(long a) { this.amountMinor = a; }
    public long getAmountMinor() { return amountMinor; }
}
public class EvilMoney extends Money {
    private long extra;               // добавил mutable-состояние
    public EvilMoney(long a) { super(a); }
    public void setExtra(long x) { this.extra = x; }
}
```

`void process(Money m)` думает, что получил неизменяемый объект, а ему передали `EvilMoney`, меняющийся под ногами. Лечение: либо `final`-класс, либо `private` конструктор + фабрики `Money.of(...)` (так делает `BigInteger`, чтобы контролировать подклассы), либо `sealed`-иерархия (глава 17) с заведомо иммутабельными наследниками.

**2. Все поля `private final`.** `final` не даёт переприсвоить поле после конструктора; `private` не даёт записать его снаружи. Без `final` поле перезапишешь из любого метода класса, без `private` — напрямую извне.

**3. Никаких мутаторов.** Ни сеттеров, ни методов, тихо меняющих поле (`recalculate()`, мутирующий кэш). «Изменение» у immutable-класса возвращает _новый_ объект — как `String.toUpperCase()` отдаёт новую строку, а исходную не трогает.

**4. Защитная копия на входе (в конструкторе).** Если приняли _изменяемый_ объект (`List`, массив, `Date`) и сохранили ссылку напрямую — вызывающий держит ту же ссылку и меняет наше состояние снаружи.

**5. Защитная копия на выходе (в геттере).** Если вернули изменяемое поле напрямую — получатель меняет наше состояние через возвращённую ссылку.

Пункты 4 и 5 — самое частое место провала, поэтому разберём их на работающем коде.

### Две утечки — на коде

Защитная копия (defensive copy) — создание независимой копии изменяемого объекта на границе класса, чтобы внешний код и твоё поле не указывали на _один и тот же_ объект. Пока обе ссылки смотрят в один `ArrayList`, `.add()` с любой стороны портит обоих; копия разрывает связь.

**Утечка (а) — конструктор сохранил переданную ссылку:**

```java
final class Trade {
    private final List<String> tags;
    public Trade(List<String> tags) { this.tags = tags; }   // сохранили чужую ссылку
    public List<String> getTags()   { return tags; }
}

List<String> input = new ArrayList<>(List.of("fx", "spot"));
Trade trade = new Trade(input);
input.add("HACKED");                 // мутируем СВОЮ переменную
trade.getTags();                     // [fx, spot, HACKED] — объект изменился
```

Проверено: `[fx, spot, HACKED]`. `final` не спас, `private` не спас — снаружи осталась живая ссылка на тот самый список, что лежит в поле. Починка: скопировать на входе — `this.tags = List.copyOf(tags);`.

![[Pasted image 20260703151754.png]]

**Утечка (б) — геттер вернул поле напрямую:**

```java
final class Trade {
    private final List<String> tags;
    public Trade(List<String> tags) { this.tags = new ArrayList<>(tags); } // на входе скопировали
    public List<String> getTags()   { return tags; }                       // но отдали внутренний
}

Trade trade = new Trade(List.of("fx", "spot"));
trade.getTags().add("HACKED");       // получили внутренний список и мутировали
trade.getTags();                     // [fx, spot, HACKED]
```

Проверено: `[fx, spot, HACKED]`. Копия на входе была, но геттер вернул наружу живую ссылку на внутренний список. Починка: `return List.copyOf(tags);`, либо `Collections.unmodifiableList(tags)`, либо хранить уже неизменяемую коллекцию и возвращать её как есть.

Чем копировать разные типы:

|Что|Как скопировать|
|---|---|
|`List` / `Set`|`List.copyOf(src)`, `new ArrayList<>(src)`|
|массив|`arr.clone()`, `Arrays.copyOf(arr, arr.length)`|
|`Date` (legacy)|`new Date(date.getTime())`|

### `List.copyOf` vs `Collections.unmodifiableList` — копия против view

Это различие любят на собесе, и оно ловит даже опытных. `List.copyOf(src)` создаёт **независимую неизменяемую копию** — отдельный объект, источник на него больше не влияет. `Collections.unmodifiableList(src)` создаёт **обёртку-view** над тем же `src`: сама через обёртку не поменяешь, но если кто-то держит ссылку на исходный `src` и вызовет `src.add(...)` — изменение просочится сквозь view. Проверено на JDK 21:

```java
List<String> src  = new ArrayList<>(List.of("a"));
List<String> view = Collections.unmodifiableList(src);
List<String> copy = List.copyOf(src);
src.add("b");

view   // [a, b]  ← view видит изменение backing-листа
copy   // [a]     ← копия независима
// view.add("x") -> UnsupportedOperationException
```

Вывод для защитных копий: `unmodifiableList` защищает, только если backing-лист приватный и ссылка на него никуда не утекла. `List.copyOf` безопаснее — он одновременно копирует _и_ делает неизменяемым, разрывая связь с источником.

> [!tip] 💡 Из C#  
> ближайшая аналогия — `ReadOnlyCollection<T>` (обёртка-view, как `unmodifiableList` — видит изменения backing-коллекции) против `ImmutableList<T>` (независимая неизменяемая структура, как `List.copyOf`). Та же дихотомия «view над чужим состоянием vs самостоятельный снимок», те же следствия: view защищает лишь от мутации _через себя_, но не от мутации источника.

### `final` ≠ immutable

Ключевой зазор, на котором рушится половина «иммутабельных» классов: `final` на поле гарантирует, что **ссылку нельзя переприсвоить** после инициализации, и ничего не говорит про **объект**, на который она указывает.

```java
final List<String> list = new ArrayList<>();
list.add("работает");           // компилируется — final защищает ССЫЛКУ, не содержимое
// list = new ArrayList<>();     // а это уже не скомпилируется
```

Поэтому `final`-поля — необходимое, но не достаточное условие иммутабельности. Достаточное = `final`-поля **плюс** либо примитивы/уже-неизменяемые типы (`String`, `Integer`, `LocalDate`, `BigDecimal`), либо защитная копия для изменяемых компонентов. `private final List`, куда при конструировании лёг `List.copyOf(...)`, — неизменяем; `private final List`, куда сохранили чужой `ArrayList`, — нет, несмотря на `final`.

> [!tip] 💡 Из C#  
> `readonly`-поле ≈ Java `final`-поле — оба про «ссылку не переприсвоить», и ловушка идентична: `readonly List<string>` всё равно позволяет `.Add()`. Для настоящей неизменяемости в C# берут `ImmutableList<T>`, а не полагаются на `readonly`, — ровно как в Java берут `List.copyOf`, а не полагаются на `final`.

### Почему за иммутабельность стоит платить

Платишь аллокацией на каждое «изменение» — получаешь четыре свойства, каждое из которых руками воспроизводится дорого:

**Безопасно шарить и кешировать.** Раз объект не меняется, одну ссылку раздают куда угодно, кешируют, переиспользуют. На этом стоит String pool (интернирование безопасно только потому, что строки неизменяемы — глава 14) и Integer cache (`-128..127` работает, потому что `Integer` immutable — глава 1.4).

**Хороший ключ `HashMap`.** Ключ обязан иметь стабильный `hashCode`, пока лежит в мапе. Изменяемый ключ, поменянный после вставки, меняет `hashCode`, «уезжает» не в тот бакет — и ты больше не найдёшь его по тому же ключу (детали контракта — 3.8). У immutable-объекта этого класса багов нет по построению.

**Проще рассуждать.** Объект, переданный в метод, не изменится за время вызова — не надо держать в голове, кто его может мутировать. Ровно поэтому `String`, `Integer`, `LocalDate`, `BigDecimal` сделаны неизменяемыми: они кочуют по всему коду, и мутабельность превратила бы их в источник неуловимых багов.

**Потокобезопасность из коробки** — но с тонкостью, которую спрашивают отдельно; она в 🔬-блоке.

### record как инструмент иммутабельности (Java 16, JEP 395)

`record` — синтаксис для immutable value-класса, который сам генерирует то, что ты писал руками: `private final`-поля, аксессоры, `equals`/`hashCode`/`toString`, канонический конструктор. Класс рекорда неявно `final`. Детально рекорды — глава 17; здесь только как инструмент неизменяемости.

```java
record Money(long amountMinor, String currency) {}
// поля final, аксессоры amountMinor()/currency(), equals/hashCode/toString — даром
```

Проверено: `new Money(1050, "EUR").equals(new Money(1050, "EUR"))` → `true`, `toString` → `Money[amountMinor=1050, currency=EUR]`. Рекорд закрывает пункты 1–3 чеклиста автоматически (класс `final`, поля `private final`, нет сеттеров). Но **НЕ** закрывает 4–5: defensive copy рекорд за тебя не делает. Компонент-`List` или массив течёт ровно как в ручном классе:

```java
record Trade(String symbol, List<String> tags) {}

var t = new Trade("EURUSD", new ArrayList<>());
t.tags().add("HACKED");              // внутренний список изменился
t.tags();                            // [HACKED]
```

Проверено — течёт. Лечится compact constructor.

**Compact constructor** — конструктор рекорда без списка параметров: выполняется _до_ неявного присваивания полей, и в нём валидируешь/трансформируешь параметры перед тем, как они лягут в поля:

```java
record Trade(String symbol, List<String> tags) {
    Trade {                          // compact — без (String symbol, List<String> tags)
        tags = List.copyOf(tags);    // переприсваиваем ПЕРЕД неявным this.tags = tags
    }
}
```

Проверено: `List.copyOf` в compact constructor чинит _обе_ границы разом — вход скопирован (мутация внешней ссылки не достаёт), и в поле лёг неизменяемый список, поэтому аксессор `tags()` отдаёт неизменяемое (`.add()` через него → `UnsupportedOperationException`). Важный нюанс: если бы в compact constructor стояло `tags = new ArrayList<>(tags)` (изменяемая копия), вход бы защитили, а аксессор снова потёк бы — пришлось бы ещё и его переопределять. Поэтому для рекордов с коллекциями `List.copyOf` на входе — чистый путь.

> [!tip] 💡 Из C#  
> Java `record` ≈ C# `record` — оба генерируют value-equality и неизменяемость по умолчанию. И ловушка та же: C# `record Trade(string Symbol, List<string> Tags)` список не копирует — `Tags` шарится, `trade.Tags.Add(...)` работает. Лечится симметрично: `ImmutableList<string>` (≈ `List.copyOf`) вместо `List`. C# `init`-аксессоры ≈ компоненты Java-рекорда: задал в конструкторе, потом не меняешь.

> [!example] 🔬 Глубже / на синьора  
> **Safe publication и почему потокобезопасность требует именно `final`.** Утверждение «immutable-объект потокобезопасен» верно, но гарантию даёт не отсутствие сеттеров, а `final`-поля — и это разные вещи. По Java Memory Model (JSR-133) `final`-поля имеют свойство **safe publication**: после завершения конструктора любой поток видит их полностью проинициализированными _без всякой синхронизации_, даже если ссылку на объект опубликовали через гонку (например, присвоили в нефинальное shared-поле без `volatile`). У нефинальных полей такой гарантии нет — другой поток может увидеть частично сконструированный объект: поле ещё в дефолтном `0`/`null`, хотя конструктор уже «прошёл». Поэтому «я просто не вызываю сеттеры» memory-visibility не гарантирует, а `final` — гарантирует. Практический вывод для собеса: «immutable + все поля `final` = можно шарить между потоками без локов», и ключевое слово в этой фразе — `final`, а не «immutable».
> 
> **record с массивом-компонентом требует и compact ctor, и переопределённого аксессора.** `List.copyOf` для массива не существует, а массив неизменяемым не сделать — поэтому одного compact constructor мало: он закроет вход, но сгенерированный аксессор всё равно вернёт внутренний массив. Нужны _оба_ `clone()`:
> 
> ```java
> record Reading(String id, int[] values) {
>     Reading { values = values.clone(); }          // копия на входе
>     public int[] values() { return values.clone(); } // копия на выходе
> }
> ```
> 
> Проверено на JDK 21: `src[0]=99` снаружи и `r.values()[1]=88` через аксессор — оба не достигают поля, `values()` стабильно даёт `[1, 2, 3]`. Это единственный случай, где рекорд с коллекцией/массивом заставляет переопределять аксессор руками; для `List` через `List.copyOf` он не нужен.
> 
> **Cost model.** Иммутабельность меняет CPU на аллокации: каждое «изменение» — новый объект и работа для GC. Для мелких короткоживущих объектов это почти бесплатно (escape analysis + дешёвая сборка в young gen, главы 11), поэтому `String`-конкатенация в цикле дорога не из-за иммутабельности как таковой, а из-за квадратичного копирования — и решается `StringBuilder` (глава 14). Правило: иммутабельность по умолчанию, мутабельный buffer — точечно, где профайлер показал горячую точку.

> [!warning] ⚠️ Подводные камни
> 
> **1. `final` защищает ссылку, не объект.** `final List`, `final Date`, `final int[]` — все изменяемы через свои методы (`.add()`, `setTime()`, `arr[0]=...`). `final` — про привязку имени к объекту, а не про состояние объекта; это два независимых уровня.
> 
> **2. Утечка через конструктор.** Сохранил переданный `List`/массив/`Date` напрямую — вызывающий держит живую ссылку и меняет твоё состояние снаружи. Копируй на входе: `List.copyOf`, `clone()`, `new Date(d.getTime())`.
> 
> **3. Утечка через геттер.** Вернул внутренний изменяемый объект — получатель меняет твоё состояние через него. Возвращай неизменяемое или копию.
> 
> **4. `record` не делает defensive copy.** Рекорд с `List`/массивом течёт через обе границы так же, как ручной класс. Для коллекций — compact constructor с `List.copyOf`; для массивов — плюс переопределённый аксессор с `clone()`.
> 
> **5. Рассинхрон поля и геттера (класс ошибок, которого нет в C#).** Раз поле и геттер в Java не связаны языком (урок 3.6), copy-paste-баг вида `getWidth()`, возвращающий `height`, компилятор пропускает молча. В C# такого класса ошибок нет — property единая сущность; в Java ловит только тест или ревью.
> 
> **6. `unmodifiableList` над утёкшим backing-листом.** Обернул в `unmodifiableList`, но сохранил (или отдал) ссылку на исходный `src` — мутация `src` видна сквозь «неизменяемую» обёртку. View неизменяем только со своей стороны; источник он не замораживает. Для настоящей отвязки — `List.copyOf`.

### На собесе

> [!question]- 1. «Достаточно ли сделать все поля `final`, чтобы класс был immutable?» _(самый частый; проверяет понимание зазора `final`≠immutable)_  
> _Модельный ответ (RU):_ «Нет. `final` защищает ссылку от переприсваивания, но не объект по ссылке — `final List`, `final Date`, `final int[]` изменяемы через свои методы. Достаточное условие: `final`-поля плюс либо примитивы/уже-неизменяемые типы, либо защитные копии изменяемых компонентов на входе и на выходе.»
> 
> _(EN):_ "No. `final` protects the reference from reassignment, not the object it points to — a `final List`, `final Date`, `final int[]` are all mutable through their methods. The sufficient condition is `final` fields plus either primitives/already-immutable types, or defensive copies of mutable components on both the way in and the way out."
> 
> _Follow-up-trap:_ «А если поле `final List` и я его не меняю?» → поменяет вызывающий: через геттер (утечка б) или через сохранённую при конструировании ссылку (утечка а). Нужен `List.copyOf` на входе и хранить неизменяемое.
> 
> _Типичный слабый ответ:_ «да, `final` делает поля неизменяемыми» — путает неизменяемость ссылки с неизменяемостью объекта.

> [!question]- 2. «Что напечатает?» _(проверка на понимание обеих утечек сразу)_  
> ```java
> final class Box {
>     private final int[] data;
>     public Box(int[] data) { this.data = data; }
>     public int[] getData() { return data; }
> }
> int[] arr = {1, 2, 3};
> Box box = new Box(arr);
> box.getData()[0] = 99;
> arr[1] = 88;
> System.out.println(box.getData()[0] + " " + box.getData()[1]);
> ```
> 
> _Модельный ответ (RU):_ «`99 88`. Обе утечки разом: конструктор сохранил `arr` (мутация снаружи через `arr[1]=88`), геттер вернул внутренний массив (мутация через `getData()[0]=99`). Массивы всегда изменяемы, `final` тут ни при чём.»
> 
> _(EN):_ "`99 88`. Both leaks at once: the constructor kept `arr` (external mutation via `arr[1]=88`), the getter returned the internal array (mutation via `getData()[0]=99`). Arrays are always mutable; `final` is irrelevant here."
> 
> _Follow-up:_ «Почини.» → `this.data = data.clone();` в конструкторе и `return data.clone();` в геттере. Массив неизменяемым не сделать (нет `Array.of`), поэтому только `clone`. Проверено: после фикса — `1 2`.

> [!question]- 3. «Зачем immutable-класс делают `final`?»  
> _Модельный ответ (RU):_ «Чтобы наследник не добавил изменяемое состояние и не переопределил методы, нарушив инварианты. Иначе метод, принимающий базовый тип, не может доверять неизменяемости — ему могли подсунуть мутабельный наследник. Если `final` нельзя — `private` конструктор плюс статические фабрики (так у `BigInteger`) либо `sealed`-иерархия, где все наследники тоже immutable.»
> 
> _(EN):_ "So a subclass can't add mutable state or override methods and break invariants. Otherwise a method taking the base type can't trust immutability — it might be handed a mutable subclass. If `final` isn't an option: a `private` constructor plus static factories (as `BigInteger` does), or a `sealed` hierarchy where all subtypes are immutable too."

> [!question]- 4. «Immutable-объект потокобезопасен? Почему именно?» _(мид+ переходящий в синьора на follow-up)_  
> _Модельный ответ (RU):_ «Да — нет изменяемого состояния, нет гонок при чтении, локи не нужны. Но ключевое условие — все поля `final`: по JMM (JSR-133) `final`-поля дают safe publication, то есть после конструктора видны другим потокам полностью проинициализированными даже при публикации ссылки через гонку. У нефинального поля такой гарантии нет — другой поток может увидеть его в дефолте. Поэтому "я не меняю" потокобезопасности не даёт, её даёт `final`.»
> 
> _(EN):_ "Yes — no mutable state, no read races, no locks needed. But the key condition is that all fields are `final`: under the JMM (JSR-133) `final` fields have safe publication, meaning other threads see them fully initialized after the constructor even if the reference is published via a race. A non-final field has no such guarantee — another thread may observe it at its default. So 'I don't mutate it' doesn't give thread-safety; `final` does."

> [!question]- 5. «`record` полностью решает immutability?»  
> _Модельный ответ (RU):_ «Нет. Рекорд делает поля `final`, класс `final`, генерирует аксессоры — закрывает три пункта Item 17 из пяти. Но defensive copy изменяемых компонентов не делает: `record` с `List` или массивом течёт через обе границы. Для коллекции чинится compact constructor с `List.copyOf`; для массива — плюс переопределённый аксессор с `clone()`.»
> 
> _(EN):_ "No. A record makes fields `final`, the class `final`, and generates accessors — covering three of Item 17's five points. But it doesn't defensively copy mutable components: a record with a `List` or array leaks through both boundaries. For a collection, fix it with a compact constructor and `List.copyOf`; for an array, add an overridden accessor with `clone()`."
> 
> _Follow-up:_ «Покажи, как сломать.» → `record Trade(List<String> tags){}`, затем `new Trade(new ArrayList<>()).tags().add("x")` — внутренний список изменился.

> [!question]- 6. «Чем `List.copyOf` отличается от `Collections.unmodifiableList`?»  
> _Модельный ответ (RU):_ «`copyOf` — независимая неизменяемая копия, отдельный объект: источник на неё не влияет. `unmodifiableList` — обёртка-view над тем же списком: сама менять не даёт, но изменения backing-листа сквозь неё видны. Для защиты состояния `unmodifiableList` годится, только если backing приватный и ссылка не утекла; `copyOf` безопаснее — разрывает связь с источником.»
> 
> _(EN):_ "`copyOf` is an independent immutable copy — a separate object the source can't affect. `unmodifiableList` is a view over the same list: it blocks mutation through itself, but changes to the backing list show through it. For protecting state, `unmodifiableList` only works if the backing list is private and its reference hasn't escaped; `copyOf` is safer — it severs the link to the source."
> 
> _Follow-up-trap:_ «`view = unmodifiableList(src)`, потом `src.add(x)` — что в `view`?» → увидит `x`; view не копирует, а делегирует чтения в `src`. Проверено: `[a, b]`.
