Static-члены выглядят просто — «то, что принадлежит классу, а не объекту», — но за этой простотой прячется целый механизм жизненного цикла класса, который на собеседованиях разбирают до винтика. Когда именно выполняется `static`-блок? Почему один `static`-инициализатор кидает исключение при первом обращении к классу, а второе обращение проходит гладко? Почему чтение `static final int` не «будит» класс, а чтение `static final` результата вызова метода — будит? Ответы на всё это — в различении загрузки и инициализации класса и в правилах ленивого «пробуждения». Этот урок собирает воедино static-состояние (поля, методы, блоки), которое частично затрагивалось в 3.1 (порядок инициализации) и 3.2 (инлайн констант), и достраивает недостающее: жизненный цикл класса от загрузки до инициализации и два способа загрузить класс вручную.

**Ментальная модель.** Static-член живёт с классом, а не с объектом — одна копия на всю программу. Класс проходит три этапа: загрузка (нашли байткод, создали `Class`), связывание (проверка + выделение памяти под static-поля с дефолтами), инициализация (выполнение static-блоков и присваивание реальных значений). Инициализация **ленивая** — откладывается до первого активного использования класса, и происходит **ровно один раз** за жизнь программы.

### Static-поля и методы: одно на класс

Static-поле — одно на класс, общее для всех экземпляров (детально «где живёт» — в 3.2: Metaspace-метаданные + зеркало `Class` в куче). Static-метод привязан к классу, а не к объекту, и у него **нет `this`** — а значит, нет доступа к instance-членам:

```java
class SC {
    int instanceField = 5;
    static void m() {
        System.out.println(instanceField);
        // error: non-static variable instanceField cannot be referenced from a static context
    }
}
```

Проверено на JDK 21. Причина не в стиле, а в физике: static-метод вызывают без объекта, поэтому «над каким объектом читать `instanceField`» — вопрос без ответа. Из static-контекста доступны только другие static-члены. Обратное свободно: instance-метод видит и static-члены (у него есть и объект, и класс).

💡 **Из C#:** static-поля и методы концептуально те же, что в .NET, и то же ограничение «из static нет доступа к instance». Различия начнутся ниже — на static-блоках (у C# это статический конструктор с другими правилами триггера) и на инлайне констант.

### Static-блок: инициализатор уровня класса

`static {}` (static initializer, статический инициализатор) — код, выполняемый при инициализации класса, наравне со static-полями. Нужен там, где static-состояние не уместить в одну строку присваивания:

```java
class Lookup {
    static final Map<String, Integer> CODES;
    static {
        CODES = new HashMap<>();
        CODES.put("OK", 200);
        CODES.put("NotFound", 404);
    }
}
```

Ключевое правило порядка: **static-поля и static-блоки образуют единую последовательность, выполняемую строго в текстовом порядке.** Не «сначала все поля, потом блоки» — вперемешку, как написаны в файле. Проверено:

```java
static int a = trace("field a = 1", 1);   // 1-е
static { System.out.println("static block #1"); }  // 2-е
static int b = trace("field b = 2", 2);   // 3-е
static { System.out.println("static block #2"); }  // 4-е
```

Вывод — ровно `field a=1 → block #1 → field b=2 → block #2`. Это тот же принцип текстового порядка, что у instance-инициализаторов в 3.1, только на уровне класса.

### Static vs instance: однократно против «на каждый `new`»

Здесь удобно поставить static-блоки и instance-блоки рядом — их взаимный порядок и есть каркас всех «что напечатает» этого кластера. Проверенный прогон:

```java
class Parent {
    static { System.out.println("Parent: static block"); }
    { System.out.println("Parent: instance block"); }
    Parent() { System.out.println("Parent: constructor"); }
}
class Child extends Parent {
    static { System.out.println("Child: static block"); }
    { System.out.println("Child: instance block"); }
    Child() { System.out.println("Child: constructor"); }
}
// new Child(); new Child();
```

Вывод:

```
=== first new Child() ===
Parent: static block      ← инициализация классов: однократно, снизу вверх
Child: static block
Parent: instance block    ← создание объекта: на каждый new
Parent: constructor
Child: instance block
Child: constructor
=== second new Child() ===
Parent: instance block    ← второй new: static БОЛЬШЕ НЕ выполняется
Parent: constructor
Child: instance block
Child: constructor
```

Два разных ритма в одном выводе. **Static-блоки** — фаза инициализации класса: выполняются один раз за всю программу, по иерархии снизу вверх (сначала весь static родителя, потом потомка). При втором `new Child()` ни одного static в выводе нет — класс уже инициализирован. **Instance-блоки и конструкторы** — фаза создания объекта: отрабатывают на каждый `new` (полная механика с `super()` — в 3.1; полный мастеркласс порядка с наследованием и виртуальными вызовами — отдельно, после главы 4).

### Загрузка ≠ инициализация — жизненный цикл класса

Вот различие, ради которого урок и существует: класс проходит **три разных этапа** (JLS §12.2–12.4), и «загрузка» с «инициализацией» — не синонимы, а разнесённые во времени фазы.

**Loading (загрузка).** `ClassLoader` находит `.class`, читает байткод, создаёт объект `Class`. Происходит рано.

**Linking (связывание).** Верификация байткода → preparation (выделение памяти под static-поля и заполнение их **дефолтами** — `0`/`false`/`null`, не пользовательскими значениями) → резолюция символических ссылок.

**Initialization (инициализация).** Выполнение static-инициализаторов и присваивание static-полям их **реальных** значений. Происходит **лениво** — откладывается до первого _активного использования_ класса (JLS §12.4.1): создание экземпляра (`new`), вызов static-метода, чтение/запись non-constant static-поля, рефлексия.

Практическое следствие «лениво и однократно»: static-блок может кинуть исключение при первом обращении к классу (обёрнутое в `ExceptionInInitializerError`), а класс останется в состоянии «неинициализирован/сломан» — но это отдельная большая тема ошибок инициализации; здесь важно зафиксировать сам факт ленивости.

#### Контринтуитивный edge case: compile-time константа не будит класс

Прямое следствие правил активного использования, которое проверяли в 3.2 со стороны байткода, а теперь смотрим со стороны инициализации. Чтение `static final`-константы примитивного типа или `String`, инициализированной **compile-time константным выражением**, _не_ триггерит инициализацию класса — потому что значение инлайнится компилятором в место использования, и обращения к классу в рантайме нет вовсе. Проверенный прогон:

```java
class Holder {
    static final int CONST = 100;            // compile-time константа
    static final int RUNTIME = compute();     // НЕ константа (вызов метода)
    static { System.out.println(">>> Holder initialized <<<"); }
    static int compute() { System.out.println(">>> compute() called <<<"); return 42; }
}

Holder.CONST      // печатает "100" — и БОЛЬШЕ НИЧЕГО: класс не инициализирован
Holder.RUNTIME    // печатает ">>> compute() called <<<", ">>> Holder initialized <<<", 42
```

Проверено на JDK 21: обращение к `Holder.CONST` дало только `100` — ни `compute()`, ни static-блок не сработали, класс не «проснулся». А `Holder.RUNTIME` (инициализирован вызовом метода, не константа) запустил полную инициализацию. Разница — ровно в том, является ли значение compile-time константой: `100` вшивается в байткод клиента (в 3.2 видели `bipush 100` и `ldc` для строки), `compute()` — нет.

💡 **Из C#:** это тот же самый механизм, что различает `const` и `static readonly` в C#. `const` инлайнится в вызывающие сборки и не триггерит инициализацию типа — как Java-шная compile-time константа; `static readonly` вычисляется при инициализации типа — как `RUNTIME` выше. Совпадает и «капкан развёртывания»: поменял `const`/compile-time константу, не перекомпилировал клиентов — они видят старое вшитое значение (подробно разбирали в 3.2, 🔬).

### `Class.forName` vs `loadClass` — загрузить или разбудить

Два способа загрузить класс вручную по имени, и разница между ними — ровно на грани «загрузка ≠ инициализация»:

```java
ClassLoader cl = MyClass.class.getClassLoader();

cl.loadClass("Target");                    // ТОЛЬКО загрузка — static-блок НЕ выполнится
Class.forName("Target");                   // загрузка + ИНИЦИАЛИЗАЦИЯ — static-блок выполнится
Class.forName("Target2", false, cl);       // загрузка без инициализации (initialize=false)
```

Проверенный вывод подтверждает все три ветки: после `loadClass("Target")` static-блока `Target` в выводе нет (только загрузка); `Class.forName("Target")` печатает `>>> Target static block <<<` (инициализация состоялась); `Class.forName("Target2", false, cl)` снова без блока. Формула на собес: **`Class.forName` по умолчанию инициализирует класс, `loadClass` — только загружает; у `forName` есть трёхаргументная перегрузка с `initialize=false` для загрузки без пробуждения.**

Практический контекст: это не академическая деталь — на ней стоит загрузка JDBC-драйверов. Историческое `Class.forName("com.mysql.jdbc.Driver")` работало именно потому, что `forName` _инициализирует_ класс, а static-блок драйвера регистрировал его в `DriverManager`. `loadClass` тут не сработал бы — драйвер загрузился бы, но не зарегистрировался.

💡 **Из C#:** прямого аналога пары нет. В .NET загрузка типов управляется CLR неявно, и явного «загрузи, но не инициализируй» в повседневном API обычно не пишут; ближайшее по духу — `Type.GetType(name)` для получения `Type` по имени, но модель триггеров инициализации у CLR своя (`beforefieldinit`). Это место, где Java явно разделяет то, что .NET прячет в рантайме.

### 🔬 Глубже / на синьора

**Инлайн константы в байткоде — `ConstantValue` и `bipush`.** Со стороны class-файла compile-time константа хранится как атрибут `ConstantValue` у поля, но в _использующем_ коде её и вовсе нет — она вшита инструкцией-литералом. Реальный `javap` с JDK 21 для `static int useLimit() { return LIMIT; }` при `LIMIT = 100`:

```
static int useLimit();
   0: bipush        100      // литерал вшит, обращения к полю/классу нет
   2: ireturn
```

Ни `getstatic`, ни ссылки на класс — просто число. Именно поэтому чтение такой константы не может триггерить инициализацию: в байткоде нечему её триггерить. Контраст с non-constant static-полем (`getstatic` + инициализация класса) разбирали в 3.2.

**Потокобезопасность инициализации класса — бесплатный happens-before.** JVM гарантирует, что инициализация класса выполняется **под блокировкой и ровно один раз**, даже если к классу одновременно обратились несколько потоков (JLS §12.4.2): остальные ждут, пока инициализирующий поток закончит. Из этого растёт идиома **Initialization-on-demand holder** для потокобезопасного ленивого синглтона без единого `synchronized` в коде:

```java
class Service {
    private Service() {}
    private static class Holder { static final Service INSTANCE = new Service(); }
    static Service getInstance() { return Holder.INSTANCE; }  // Holder инициализируется лениво и потокобезопасно
}
```

`Holder` не инициализируется, пока не вызван `getInstance()` — а когда вызван, JVM гарантирует однократность и видимость. Вся синхронизация — на механизме инициализации класса, руками не написана ни строчки. Полный разбор синглтонов — глава 25.2; здесь важна связь с ленивой инициализацией.

**Preparation vs initialization для static-полей.** На этапе preparation static-поля получают дефолты (`static int x` → `0`), на этапе initialization — реальные значения (`static int x = 5` → `5`). Между этими фазами поле физически существует со значением `0`. Обычно зазор незаметен, но при циклических зависимостях классов в static-инициализаторах можно наблюдать поле-в-дефолте — редкий, но реальный источник багов инициализации.

### ⚠️ Подводные камни

**1. Доступ к instance-члену из static-контекста.** `non-static variable cannot be referenced from a static context` — из static-метода/блока не видно instance-полей и instance-методов: нет объекта, над которым работать. Обратное (instance → static) разрешено всегда.

**2. Порядок static-инициализации — строго текстовый.** Static-поля и static-блоки — единая последовательность по тексту файла. Static-блок, читающий static-поле, объявленное _ниже_, увидит его в дефолте (или не скомпилируется при прямой forward-ссылке — как в 3.1). Порядок объявления имеет значение.

**3. Исключение в static-блоке ломает класс.** Выброс из static-инициализатора оборачивается в `ExceptionInInitializerError`, и класс переходит в состояние «неинициализируемый» — повторные обращения дают `NoClassDefFoundError`. Первое обращение выглядит как одна ошибка, последующие — как другая, хотя причина одна.

**4. `loadClass` не инициализирует.** Ждёшь от загрузки класса срабатывания его static-блока (регистрация, side-effect) — а взял `loadClass` вместо `Class.forName` — блок не выполнится. Классически всплывало на JDBC-драйверах.

**5. Мутабельное static-поле = глобальное состояние.** Повтор из 3.2, но в контексте инициализации особенно коварно: static-поле инициализируется один раз, и если оно мутабельное и шарится между потоками — гонки гарантированы без синхронизации. Static-состояние оправдано для констант и stateless-утилит; изменяемое состояние — в объекты.

**6. Compile-time константа и устаревшее значение.** Публичная `static final`-константа примитива/`String` вшивается в байткод всех клиентов; смена значения без перекомпиляции зависимых модулей оставляет им старое (разбор — 3.2, 🔬). Всё, что может меняться, — не compile-time константа, а конфигурация или значение, вычисляемое в рантайме.

### На собесе

**1. «Когда выполняется static-блок? Сколько раз?»** _(базовый, почти гарантированный)_

_Модельный ответ (RU):_ «При инициализации класса — лениво, при первом активном использовании (создание экземпляра, вызов static-метода, доступ к non-constant static-полю, рефлексия), и ровно один раз за всю жизнь программы. Static-поля и static-блоки выполняются единой последовательностью в текстовом порядке. Инициализация происходит после загрузки и связывания — это разные фазы.»

_(EN):_ "During class initialization — lazily, on the first active use (instantiation, a static method call, access to a non-constant static field, reflection), and exactly once per program run. Static fields and static blocks execute as one sequence in textual order. Initialization comes after loading and linking — those are distinct phases."

_Follow-up-trap:_ «`new Child()` дважды — сколько раз сработает static-блок `Parent`?» → один раз, при первом; второй `new` static не трогает.

**2. «Чем загрузка класса отличается от инициализации?»** _(мид+, разделяющий вопрос)_

_Модельный ответ (RU):_ «Три фазы: загрузка — `ClassLoader` читает байткод и создаёт `Class`; связывание — верификация плюс preparation, где static-поля получают дефолты; инициализация — выполнение static-блоков и присваивание реальных значений, лениво, при первом активном использовании. Загрузка ранняя, инициализация отложенная — между ними static-поле физически существует в дефолтном значении.»

_(EN):_ "Three phases: loading — the `ClassLoader` reads bytecode and creates the `Class`; linking — verification plus preparation, where static fields get their defaults; initialization — running static blocks and assigning real values, lazily, on first active use. Loading is early, initialization deferred — between them a static field physically exists holding its default."

_Синьорский добавок:_ «Инициализация потокобезопасна и однократна по JLS §12.4.2 — на этом стоит holder-идиома ленивого синглтона без `synchronized`.»

**3. «Триггерит ли чтение `static final int X = 5` инициализацию класса?»** _(любимый trap на понимание констант)_

_Модельный ответ (RU):_ «Нет. Это compile-time константа — компилятор инлайнит её значение прямо в место использования (в байткоде литерал, не `getstatic`), обращения к классу в рантайме нет вообще, инициализировать нечего. А вот `static final int X = compute()` — уже не константа: читается через `getstatic` и инициализацию триггерит.»

_(EN):_ "No. It's a compile-time constant — the compiler inlines its value at the use site (a literal in bytecode, not `getstatic`), so there's no runtime reference to the class at all, nothing to initialize. But `static final int X = compute()` isn't a constant: it's read via `getstatic` and does trigger initialization."

_Follow-up:_ «А `static final String S = "x"`?» → тоже compile-time константа, тоже не триггерит. «А `static final Object O = new Object()`?» → не константа (не примитив/`String` и не константное выражение), триггерит.

**4. «`Class.forName` vs `classLoader.loadClass` — в чём разница?»**

_Модельный ответ (RU):_ «`Class.forName` по умолчанию **инициализирует** класс — выполняет static-блоки; `loadClass` только **загружает**, без инициализации. У `forName` есть перегрузка `forName(name, false, loader)` с `initialize=false` — загрузка без пробуждения. Классический кейс: старая регистрация JDBC-драйвера через `Class.forName` работала именно потому, что static-блок драйвера, регистрирующий его в `DriverManager`, выполняется только при инициализации.»

_(EN):_ "`Class.forName` initializes the class by default — runs its static blocks; `loadClass` only loads it, without initialization. `forName` has a `forName(name, false, loader)` overload with `initialize=false` for loading without waking the class. Classic case: the old JDBC driver registration via `Class.forName` worked precisely because the driver's static block, which registers it with `DriverManager`, runs only on initialization."

_Типичный слабый ответ:_ «оба загружают класс» — упускает, что `forName` ещё и инициализирует, а это и есть суть различия.

**5. «Почему из static-метода нельзя обратиться к instance-полю?»**

_Модельный ответ (RU):_ «Static-метод вызывается без объекта, у него нет `this` — нет того экземпляра, чьё instance-поле читать. Компилятор даёт `non-static variable cannot be referenced from a static context`. Обратное разрешено: instance-метод видит static-члены, потому что у него есть и объект, и класс.»

_(EN):_ "A static method is called without an object — it has no `this`, so there's no instance whose field to read. The compiler gives `non-static variable cannot be referenced from a static context`. The reverse is allowed: an instance method can see static members, since it has both an object and a class."

**6. «Как сделать потокобезопасный ленивый синглтон без `synchronized`?»** _(reach-вопрос, мид+ к синьору; мост к 25.2)_

_Модельный ответ (RU):_ «Initialization-on-demand holder: приватный вложенный static-класс с полем `static final INSTANCE`, доступ через `getInstance()`. Вложенный класс инициализируется лениво — только при первом обращении к `getInstance()`, — а JVM гарантирует, что инициализация класса потокобезопасна и однократна (JLS §12.4.2). Вся синхронизация — на механизме инициализации класса, руками ни строчки.»

_(EN):_ "Initialization-on-demand holder: a private nested static class with a `static final INSTANCE` field, accessed via `getInstance()`. The nested class initializes lazily — only on the first `getInstance()` call — and the JVM guarantees class initialization is thread-safe and happens once (JLS §12.4.2). All the synchronization rides on the class-init mechanism; not a line written by hand."

_Follow-up:_ «Чем лучше double-checked locking?» → нет `volatile`, нет явной блокировки, меньше места для ошибки; корректность гарантирована спецификацией, а не аккуратностью кода.
