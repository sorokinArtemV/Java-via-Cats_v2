В уроке 3.12 (A) мы увидели, что `inner`-объект несёт скрытую ссылку `this$0` на внешний объект. Эта ссылка невидима и почти бесплатна — до тех пор, пока `inner`-объект не переживёт того, кто по задумке должен был умереть первым. Тогда получается целый класс продовых OOM: ты отдаёшь крошечный listener или callback во что-то долгоживущее, а он молча удерживает весь внешний объект вместе с его тяжёлыми полями.

На JDK 21 у этой механики есть важное уточнение, которого не было в старых версиях: ссылка `this$0` создаётся, **только если `inner` реально обращается к внешнему экземпляру**. Поэтому утечка на самом деле про _захват_ внешнего объекта — и это ровно та же история, что и с лямбдами. Этот урок — механизм утечки, почему «переписать на лямбду» не спасает, и как найти такой баг в проде за пять минут.

### Ментальная модель

`inner`/`anonymous`/`local`-объект удерживает внешний объект через `this$0` — но на JDK 21 только когда действительно использует его состояние (иначе `javac` эту ссылку опускает). Утечка — это захватывающий объект, положенный во что-то долгоживущее, при том что внешний должен был умереть; достижимость транзитивна, поэтому живым остаётся весь внешний объект со всеми его полями.

### Как это работает

Паттерн утечки всегда один: ты отдаёшь `inner`-объект (listener, callback, `Runnable`, `Comparator`) во что-то **долгоживущее** — шина событий, статическая коллекция, кеш, пул потоков, реестр колбэков, — а внешний объект по задумке должен был умереть быстро. Не умирает, потому что его держит этот `inner` через `this$0`.

Серверный пример, который ты реально можешь написать:

```java
public class PriceService {
    private final Map<String, BigDecimal> cache = loadHugeCache();   // тяжёлый, мегабайты

    public void start(EventBus bus) {                                // bus — глобальный
        bus.subscribe(new PriceListener() {                          // анонимный = inner
            public void onTick(Tick t) {
                cache.put(t.symbol(), t.price());                    // трогает cache внешнего PriceService
            }
        });
    }
}
```

Если `PriceService` создаётся на запрос, а `bus` глобальный на всё приложение — каждый `PriceService` повисает навсегда. Listener крошечный, но через `this$0` тянет весь `PriceService`, а тот — свой `cache`. Вот цепочка, которую видит сборщик мусора:

Читай цепочку от корня: сборщик мусора собирает только то, до чего **нельзя дотянуться ни от одного живого корня**, и достижимость транзитивна. `EventBus` — живой корень (на него ссылается что-то долгоживущее). Значит жив `Listener`, значит через `this$0` жив `PriceService`, значит жив его `cache`. Крошечный listener пиннит граф на мегабайты. За неделю таких «мёртвых» сервисов набегает на OOM.

Проверим это runnable-тестом: подпишем listener на вечную шину, уроним ссылку на сервис, дёрнем GC и посмотрим через `WeakReference`, собрался ли он:

```
анонимный inner (трогает поле): жив после GC = true    ← течёт
лямбда с this.field           : жив после GC = true    ← течёт
лямбда без this                : жив после GC = false   ← собрана
```

Первые два держат внешний объект и не дают его собрать; третий — нет.

#### Уточнение JDK 18, которое всё меняет

Классическое утверждение «`inner` держит внешний объект **всегда**, даже если не трогает его поля» — верно только **до JDK 18**. Начиная с JDK 18 (JDK-8271717, release note JDK-8277604) неиспользуемые поля `this$` опускаются — поле генерируется только для inner-классов, которые обращаются к своему внешнему экземпляру. Проверено на JDK 21 через `javap`: у `inner`-класса, читающего поле `Outer`, есть `final Host this$0`, а у `inner`-класса, который поля не трогает, — этого поля нет вовсе. При этом форма конструктора не меняется — параметр внешнего экземпляра остаётся в сигнатуре, он просто игнорируется, а не сохраняется в поле. [Openjdk](https://bugs.openjdk.org:443/browse/JDK-8278181)[Openjdk](https://bugs.openjdk.org:443/browse/JDK-8278181)

Практическое следствие огромное: на JDK 21 **`inner`-объект удерживает внешний, только если реально его использует** — ровно как лямбда, захватывающая `this`. Мой тест это и показывает: анонимка, которая трогает поле, течёт; анонимка, которая не трогает, — собирается. Мотивация самой правки прямо про это — неиспользуемые поля `this$0` были источником утечек памяти, и их убрали. [Java](https://bugs.openjdk.java.net/browse/JDK-8277965?page=com.atlassian.jira.plugin.system.issuetabpanels:worklog-tabpanel)

> ⚠️ Исключение: для `Serializable`-классов, объявляющих `serialVersionUID`, оптимизация не применяется (JDK-8278181) — там `this$0` эмитится даже неиспользуемым, чтобы не сломать сериализацию. Такой сериализуемый `inner` потечёт, даже не трогая внешний.

#### Ловушка «перепишу на лямбду»

После разговоров про лямбды соблазн такой: «заменю анонимку на лямбду — `this$0` не будет». **Не спасёт**, если лямбда трогает состояние экземпляра:

```java
bus.subscribe(t -> cache.put(t.symbol(), t.price()));   // cache — это this.cache
```

Чтобы достать `cache`, лямбда захватывает `this` — тот же самый внешний объект. Утечка идентичная (в тесте: `жив = true`). А лямбда, которая `this` не трогает, — безопасна: её `javac` компилирует в `private static`-метод (проверено в уроке 3.12 A), держать внешний объект нечем.

Правило после JDK 18: течёт **захват внешнего экземпляра**, а не «анонимка против лямбды». Тронул поле или instance-метод `Outer` — захватил его целиком — рискуешь, чем бы это ни было написано. Не тронул — не течёт ни то, ни другое.

> [!tip] 💡 Из C#
> ты это знаешь как утечку через event handler. Подписал метод или лямбду на долгоживущий `event` через `+=` — подписчик жив, пока не сделаешь `-=`. Java-версия та же, только подписку прячет компилятор в `this$0` вместо явного `+=`, и отписываться так же надо руками.

#### Как чинить (по убыванию надёжности)

Самое надёжное — управлять жизненным циклом явно: сохрани ссылку на listener и сделай `bus.unsubscribe(listener)`, когда сервис больше не нужен, не уповая на GC. Если внешнее состояние listener'у не нужно — сделай его `static nested` и передай нужные данные в конструктор явно, чтобы он держал только их, а не весь сервис. Крайний вариант — `WeakReference<PriceService>` внутри listener'а: GC сможет собрать сервис, listener проверяет на `null` (костыльно, но это ровно история андроидных листенеров).

> [!example] 🔬 Глубже / на синьора
> **`javap`-доказательство элизии.** Для `class Host { class InnerUses { int f(){return used;} } class InnerEmpty { int f(){return 0;} } }` на JDK 21: `Host$InnerUses` содержит `final Host this$0`, а `Host$InnerEmpty` — нет; у обоих конструктор принимает `Host`. То есть на байткод-уровне видно: ссылка кладётся ровно тогда, когда используется.
> 
> **Захватывается весь `Outer`, а не поле.** `this$0` — это ссылка на _внешний объект целиком_. Стоит `inner` тронуть хоть одно поле `Outer` — и он пиннит весь граф внешнего объекта, включая тяжёлые поля, к которым он даже не обращается. Нельзя «захватить только `cache`» через `inner` — только весь `PriceService`. Именно поэтому лечение №2 (передать данные в конструктор `static nested`) работает: там ты держишь ровно то, что передал.
> 
> **Почему не-`this` лямбда безопасна.** `javac` компилирует её в `private static` synthetic-метод (`lambda$...$0`), а сам объект лямбды не содержит ссылки на внешний экземпляр — держать `Outer` нечем. Это и подтвердил тест: не-`this` лямбда собралась. Детали генерации (`invokedynamic`, `LambdaMetafactory`) — урок 9.8.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Тронул одно поле — захватил весь `Outer`.** `this$0` тянет внешний объект целиком, а не только использованное поле. Один `cache.put(...)` пиннит весь `PriceService`.
> 2. **Лямбда, трогающая `this`, течёт как анонимка** (проверено). Переписывание формы ничего не меняет — важен захват.
> 3. **Неочевидные долгоживущие держатели:** пулы потоков, статические кеши, `ThreadLocal`, реестры колбэков, списки слушателей. Отдал туда capturing-объект — риск.
> 4. **`Serializable` + `serialVersionUID`** отключает элизию `this$0` (JDK-8278181): такой `inner` держит внешний объект даже неиспользуемым. Ещё одна причина не делать `inner`-классы `Serializable`.
> 5. **Расчёт на GC вместо отписки.** GC не спасёт, пока цепочка достижима от живого корня. Надёжно — разорвать её самому (`unsubscribe`).

### На собесе

> [!question]- 1. «Как `inner`-класс приводит к утечке памяти?» _(частый, прод-ориентированный)_
> RU: Через `this$0` `inner`-объект держит внешний. Если `inner` отдали во что-то долгоживущее (шина, статика, пул), а внешний должен был умереть, — внешний остаётся достижим и не собирается, таща за собой все свои поля. На JDK 21 важная деталь: `inner` держит внешний, **только если реально использует** его состояние — `javac` с JDK 18 опускает неиспользуемую `this$0` (JDK-8271717).
> 
> EN: Through `this$0` an `inner` object holds its outer. If the `inner` is handed to something long-lived (an event bus, a static field, a pool) while the outer was meant to die, the outer stays reachable and isn't collected, dragging all its fields along. On JDK 21 there's a key detail: an `inner` holds the outer **only if it actually uses** its state — since JDK 18 `javac` omits an unused `this$0` (JDK-8271717).
> 
> Follow-up (главный): _«Я заменил анонимку на лямбду, теперь не течёт?»_ → Если лямбда трогает поле или instance-метод, она захватывает `this` — та же утечка. Течёт захват `this`, а не «анонимка vs лямбда». Не-`this` лямбда компилируется в static-метод и безопасна.
> 
> Добьёт: _«Как чинить?»_ → Явно отписать/дерегистрировать (надёжнее всего); либо `static nested` + передать только нужные данные; либо `WeakReference` как крайний вариант. Диагностика — heap dump и path to GC root.
> 
> Senior add-on: _«На JDK 21 всегда ли `inner` держит внешний?»_ → Нет, только при использовании внешнего экземпляра; blanket-утверждение «всегда» — pre-18.

> [!question]- 2. «Почему захватываемая локальная переменная должна быть effectively final?»
> RU: Захват в Java идёт **по значению** — копия значения кладётся в synthetic-поле захватчика. Будь переменная изменяемой, копия и оригинал разъехались бы, и было бы неясно, какое значение «настоящее». Java выбрала запретить, а не синхронизировать. Чтобы копить счётчик, захватывай не переменную, а изменяемый объект: `int[] sum = {0}; sum[0]++` или `AtomicInteger` — ссылка effectively final, а содержимое меняй.
> 
> EN: Capture in Java is **by value** — a copy of the value is stored in a synthetic field of the capturing object. If the variable were mutable, the copy and the original would diverge, and it'd be unclear which value is "real". Java chose to forbid rather than synchronize. To accumulate a counter, capture a mutable object instead of the variable: `int[] sum = {0}; sum[0]++` or `AtomicInteger` — the reference stays effectively final while the contents change.
> 
> Добьёт (C#-ловушка): _«В C# же можно менять захваченную переменную?»_ → Да, C# захватывает **по ссылке** через сгенерированный display-class: метод и лямбда видят одно поле. Java — по значению. Разный дизайн: Java этим исключает случайное общее изменяемое состояние между потоками из коробки.
