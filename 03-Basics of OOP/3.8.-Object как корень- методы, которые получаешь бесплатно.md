### Что это и зачем

В Java нет ни одного класса, который не был бы наследником `Object` — компилятор дописывает `extends Object` за тебя, даже когда ты этого не пишешь. Из-за этого любой твой класс с рождения умеет пять базовых вещей: сравниваться (`equals`), выдавать хеш (`hashCode`), превращаться в строку (`toString`), сообщать свой тип (`getClass`) и копироваться (`clone`). Все эти дефолтные реализации написаны в расчёте на одну-единственную вещь — _тождество экземпляра_, а не на смысл данных, которые ты в объект положил.

Знать, что именно они делают из коробки, важно по практической причине: добрая половина «странного» поведения новичка в Java — `почему две одинаковые записи не равны`, `почему в логе com.example.User@568bf312`, `почему изменение в копии протекло в оригинал` — это и есть незаметно сработавший дефолт `Object`. Этот урок разбирает весь набор, кроме тяжёлой пары `equals`/`hashCode` с её собственным контрактом — ей посвящён отдельный урок 3.9, потому что она ядовитее всех остальных вместе взятых.

### Ментальная модель

`Object` — общий предок каждого класса в Java; он бесплатно раздаёт всем базовый набор методов, но их дефолты отвечают на вопрос «это физически тот же самый объект в памяти?», тогда как тебе почти всегда нужен ответ на вопрос «это одинаковые по смыслу данные?».

> [!tip] 💡 Из C#
> один-в-один твой `System.Object`. В .NET тоже всё неявно наследует `object`, и набор совпадает: `Equals`, `GetHashCode`, `ToString`, `GetType`, `MemberwiseClone`. Дефолты совпадают и по духу (identity-based). Так что сама идея «корень с виртуальными методами» тебе знакома — новизна в деталях каждого метода, и они у Java свои.

### Как это работает

#### Три метода, заточенных под identity

`Object.equals` дословно (исходник JDK):

```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

То есть по умолчанию `equals` — это буквально `==`: «равны, только если это одна и та же ссылка». Мы это уже видели в уроке про `==` vs `equals` — вот источник того вывода.

`Object.hashCode` возвращает **identity hash code** — целое, которое JVM назначает конкретному экземпляру. Оно стабильно на протяжении жизни объекта, но у двух разных экземпляров (даже с идентичным содержимым) практически всегда разное.

`Object.toString` собирается по формуле `getClass().getName() + '@' + Integer.toHexString(hashCode())`, то есть `com.example.Box@568bf312`. Этот hex-хвост — как раз identity hash в шестнадцатеричном виде.

Проверим все три на JDK 21 на классе `Box` без единого переопределённого метода:

```java
Box a = new Box(1);
Box b = new Box(1);
System.out.println(a.equals(b));                  // false
System.out.println(a.equals(a));                  // true
System.out.println(a.hashCode() == b.hashCode()); // false
System.out.println(a);                            // Box@568bf312
```

Реальный вывод:

```
a.equals(b): false
a.equals(a): true
a.hashCode()==b.hashCode(): false
toString: Box@568bf312
```

`a` и `b` — «одинаковые» по смыслу, но дефолтный `equals` их не считает равными, а дефолтный `hashCode` даёт им разные числа. Строка `Box@568bf312` в точности совпала с ручной сборкой `getClass().getName() + "@" + Integer.toHexString(a.hashCode())`. Вывод: из коробки `Object` умеет сравнивать только идентичность — на большее у него нет информации.

#### `toString` — единственный из трёх, что почти всегда стоит переопределить

Где он срабатывает: логи, сообщения исключений, дебаггер, конкатенация `"" + obj`, `System.out.println`. Дефолтный `Box@568bf312` в логе бесполезен — по нему невозможно понять, _что_ за объект. Переопределяешь — включаешь значимые поля.

> [!warning] Подводный камень
> не клади в `toString` секреты — пароли, токены, номера карт, PII. Логи текут в системы агрегации, где их видит полкомпании; `toString` доменного объекта — один из самых частых каналов утечки. Консистентность с `equals` контрактом не требуется, а вот осмысленность и безопасность — на тебе.

IDE генерирует `toString` в два клика, а `record` даёт корректный из коробки (глава 17). Это Item 12 из Effective Java — «Always override `toString`».

#### `getClass` — точный рантайм-тип

Возвращает `Class<?>` рантайм-класса объекта — точный тип, а не тот, под которым лежит ссылка. Метод `final`, переопределить нельзя. Прямой аналог `GetType()` в C#. Используется в reflection (глава 12) и в `equals` при наследовании как строгая проверка типа (детали — урок 3.9).

Ключевое отличие от `instanceof`: `getClass()` даёт _ровно этот_ класс, `instanceof` — _этот или любой подкласс_. `subInstance.getClass() == Parent.class` вернёт `false`, а `subInstance instanceof Parent` — `true`.

#### `clone` / `Cloneable` — как делать НЕ надо (но знать обязательно)

Два термина, которые надо ввести явно:

**Marker interface** (интерфейс-маркер) — интерфейс без методов, который ничего не добавляет к API, а лишь _сигнализирует_ что-то рантайму или библиотекам. `Cloneable` — именно такой: он не объявляет метод `clone`.

**Shallow copy** (поверхностная) — копируются значения полей; поля-ссылки в копии указывают на _те же самые_ объекты, что в оригинале. **Deep copy** (глубокая) — рекурсивно копируются и объекты по ссылкам.

Как это устроено: `clone()` живёт в `Object`, он `protected` и `native`. `Cloneable` его _не объявляет_ — он лишь переключает поведение `Object.clone`: реализуешь `Cloneable` → `clone` вернёт побитовую копию полей; не реализуешь → `Object.clone` кинет `CloneNotSupportedException`. Проверим на классе `Person` с изменяемым полем-списком:

```java
Person original = new Person("Мурзик", new ArrayList<>(List.of("cat")));

Person shallow = original.clone();          // super.clone() -> поверхностная
shallow.tags.add("added-via-clone");
System.out.println(original.tags);          // список изменился и в оригинале!

Person deep = new Person(original);          // copy constructor -> глубокая
deep.tags.add("added-via-copy-ctor");
System.out.println(original.tags);          // оригинал не тронут
```

Реальный вывод на JDK 21:

```
original после мутации clone: [cat, added-via-clone]
original после мутации copy : [cat]
```

Мутация списка в `shallow`-клоне протекла в оригинал, потому что оба держат ссылку на _один и тот же_ `ArrayList`. `deep`-копия через copy constructor такого не делает — у неё свой список. Вот эта разница на объектном графе:

Почему `clone` считается сломанным (Effective Java, Item 13):

1. `Cloneable` — маркерный интерфейс, который _не объявляет_ `clone`, а лишь меняет поведение метода суперкласса. Интерфейс, переключающий семантику чужого метода, — кривой дизайн сам по себе.
2. Копия по умолчанию **поверхностная** — вложенные изменяемые объекты шарятся (мы это только что увидели на списке).
3. `clone()` **не вызывает конструктор** — объект создаётся в обход инвариантов, которые ты в конструкторе установил.
4. Плохо дружит с `final`-полями: их нельзя переприсвоить в теле метода клонирования.

Правильная замена — **copy constructor** или статическая **copy factory**:

```java
Person(Person other) {                 // copy constructor
    this.name = other.name;
    this.tags = new ArrayList<>(other.tags);  // новый список — глубокая копия
}
```

> [!tip] 💡 Из C#
> прямая параллель — `ICloneable` в .NET считается ошибкой дизайна по ровно той же причине: интерфейс не уточняет, копия глубокая или поверхностная, контракт размыт. А `MemberwiseClone()` — это буквально shallow copy, тот же побитовый механизм, что у `Object.clone`. Критика тебе уже знакома концептуально.

#### `wait` / `notify` / `notifyAll`

Эти три тоже живут на `Object` — потому что в Java любой объект может служить монитором для синхронизации. Но это механика многопоточности, разберём её в главе 23; здесь достаточно знать, _почему_ они на `Object`.

> [!example] 🔬 Глубже / на синьора
> **Identity hash code — это не адрес объекта.** Частый ответ на собесе «`hashCode` по умолчанию — адрес в памяти» неверен дважды. Во-первых, Javadoc `Object.hashCode` прямо снимает это требование: значение «обычно реализуется через преобразование внутреннего адреса в `int`, но эта техника языком не требуется». Во-вторых, глубже: сборщик мусора при компактизации _перемещает_ объекты в памяти, а identity hash обязан быть неизменным всю жизнь объекта (правило консистентности). Адрес меняется — хеш нет. Значит, хеш не может быть текущим адресом. В HotSpot он вычисляется лениво при первом обращении (`hashCode()` / `System.identityHashCode()`) и кешируется в заголовке объекта (mark word); до первого запроса его в объекте попросту нет.
> 
> **`Object.clone` под капотом** — `native`-метод, делающий побайтовое копирование полей минуя любой конструктор. Для примитивов копируется значение, для ссылок — сама ссылка (отсюда и поверхностность). Именно обход конструктора делает его опасным для классов, чьи инварианты держатся на конструкторной логике.
> 
> **`finalize()` мёртв.** Он тоже наследуется от `Object`, но помечен `@Deprecated(forRemoval)` начиная с Java 9 (окончательно — JEP 421, Java 18). Никогда не использовать: недетерминированное время вызова, риск воскрешения объекта, тормоза GC. Для освобождения ресурсов — try-with-resources или `java.lang.ref.Cleaner`.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Дефолтный `toString` в логах.** `log.info("saved {}", user)` без переопределённого `toString` даст `User@1b6d3586` — бесполезная строка. Переопределяй `toString` у всех доменных объектов.
> 2. **Миф «`hashCode` = адрес».** См. блок выше — на собесе это ловушка на понимание GC и mark word.
> 3. **Shallow-clone шарит изменяемое состояние.** Для `final`/immutable-полей поверхностной копии достаточно; для изменяемых (списки, мапы, массивы) — источник трудноуловимых багов, где мутация «копии» протекает в оригинал.
> 4. **`Cloneable` без переопределения `clone` бесполезен.** Даже реализовав `Cloneable`, снаружи ты не вызовешь `clone` — он унаследован от `Object` как `protected`. Нужно ещё переопределить его как `public`. Сам маркер публичного API не открывает.
> 5. **`getClass()` vs `instanceof` путаница.** `x.getClass() == Foo.class` — строгое равенство типа (`false` для подкласса); `x instanceof Foo` — тип-или-подкласс (`true`). Перепутать их — прямой путь сломать `equals` при наследовании (урок 3.9).

### На собесе

> [!question]- 1. «Почему `Object.equals` по умолчанию — это `==`?» _(частый, начальный)_
> RU: Потому что `Object` ничего не знает о полях конкретного класса. Единственное равенство, которое он может гарантированно проверить без знания семантики, — идентичность ссылок. Любое сравнение «по содержимому» требует знания, какие поля значимы, а этого у `Object` нет.
> 
> EN: Because `Object` has no knowledge of a class's fields. The only equality it can safely guarantee without class-specific semantics is reference identity; value equality requires knowing which fields matter, which `Object` doesn't have.
> 
> Follow-up: _«А `hashCode` по умолчанию — это адрес?»_ → Нет: identity hash, не адрес. GC двигает объекты, а хеш обязан быть стабильным, поэтому он вычисляется лениво и кешируется в заголовке объекта.

> [!question]- 2. «Дефолтный `hashCode` — это адрес объекта?» _(ловушка)_
> Частый неверный ответ: «да, адрес в памяти». Почему проваливается: Javadoc снимает это требование, и физически невозможно — GC перемещает объект, а identity hash обязан оставаться прежним.
> 
> RU: Нет. Это identity hash code — число, назначаемое экземпляру и стабильное всю его жизнь. Адресом быть не может, потому что GC при компактизации перемещает объекты, а хеш меняться не должен. В HotSpot он считается лениво и кешируется в mark word.
> 
> EN: No. It's an identity hash code — assigned per instance and stable for its lifetime. It can't be the address because the GC relocates objects during compaction while the hash must stay constant. In HotSpot it's computed lazily and cached in the object header (mark word).

> [!question]- 3. «Почему `clone()` считается сломанным и как копировать правильно?»
> RU: `Cloneable` — маркерный интерфейс, не объявляющий `clone`; копия поверхностная (общее изменяемое состояние); `clone` не вызывает конструктор, обходя инварианты; плохо работает с `final`-полями. Правильно — copy constructor или статическая copy factory.
> 
> EN: `Cloneable` is a marker interface that doesn't declare `clone`; the copy is shallow (shared mutable state); `clone` bypasses the constructor and its invariants; and it interacts badly with `final` fields. The right approach is a copy constructor or a static copy factory.
> 
> Follow-up: _«Чем именно плоха поверхностная копия?»_ → Вложенные изменяемые объекты (списки, мапы) шарятся между копией и оригиналом; мутация в одном видна в другом. Для immutable-полей проблемы нет.

> [!question]- 4. «Что напечатает дефолтный `toString` и из чего он состоит?»
> RU: `ClassName@hex`, точнее `getClass().getName() + '@' + Integer.toHexString(hashCode())`. Hex-хвост — это identity hash в шестнадцатеричном виде, а не адрес.
> 
> EN: `ClassName@hex`, precisely `getClass().getName() + '@' + Integer.toHexString(hashCode())`. The hex suffix is the identity hash in hexadecimal, not the memory address.

> [!question]- 5. «`getClass()` vs `instanceof` — когда что?» _(senior add-on)_
> RU: `getClass()` — точный рантайм-тип, строгая проверка (важна в `equals` при наследовании, чтобы `Point` и `ColorPoint` не считались равными). `instanceof` — тип-или-подкласс, полиморфная проверка (и заодно безопасна к `null`). Выбор между ними в `equals` — целый компромисс между симметрией и принципом подстановки Лисков, разбираем в уроке 3.9.
> 
> EN: `getClass()` gives the exact runtime type — a strict check (matters in `equals` under inheritance so a subtype isn't treated as equal to its parent). `instanceof` matches the type or any subtype — a polymorphic check that's also null-safe. Choosing between them in `equals` is a genuine trade-off between symmetry and the Liskov substitution principle, covered in lesson 3.9.
