В Java класс можно объявить прямо внутри другого класса. Смысл простой: некоторые классы имеют значение только в контексте другого — `Builder`, который собирает конкретный иммутабельный объект; итератор, который ходит по конкретному списку; одноразовый `Comparator` для одной сортировки. Плодить ради каждого отдельный файл и засорять пространство имён проекта — незачем, поэтому такой класс живёт внутри своего «хозяина».

Видов ровно четыре, и почти всё, что нужно про них знать, сводится к одному водоразделу: `static nested` против `inner`. Первый — самостоятельный класс, механически совпадающий с nested-классом в C#. Второй несёт скрытую ссылку на объект внешнего класса, и вот у _этого_ прямого аналога в C# нет — из него растут и итераторы, и (в бонус-уроке 3.12 B) целый класс утечек памяти.

### Ментальная модель

Один вопрос делит все четыре вида надвое: нужен ли классу доступ к состоянию _конкретного экземпляра_ внешнего класса? Нет — `static nested`, самостоятельный, ничего за собой не держит. Да — семейство `inner`/`local`/`anonymous`, каждый из которых несёт скрытую ссылку `this$0` на внешний объект.

### Как это работает

#### `static nested` — самостоятельный класс внутри другого

`static nested` берут, когда нужен класс-помощник, логически живущий рядом с внешним, но которому **не нужен его экземпляр**. Он один-в-один твой C#-nested: ничего нового в поведении. Самый частый случай в реальном коде — `Builder` для иммутабельного объекта:

```java
public class Account {
    private final String owner;
    private final long balanceMinor;

    private Account(Builder b) {            // приватный конструктор
        this.owner = b.owner;
        this.balanceMinor = b.balanceMinor;
    }

    public static class Builder {          // static nested
        private String owner;
        private long balanceMinor;
        public Builder owner(String owner) { this.owner = owner; return this; }
        public Builder balance(long minor) { this.balanceMinor = minor; return this; }
        public Account build()             { return new Account(this); }
    }
}
```

```java
Account acc = new Account.Builder()        // объекта Account ещё НЕТ
        .owner("Артём").balance(150_00).build();
```

Почему `Builder` обязан быть `static`, а не `inner`: он строит `Account`, которого _ещё не существует_. `inner`-классу нужно приклеиться к уже живому объекту внешнего класса — а его нет. Получился бы парадокс «чтобы создать `Account`, дай мне готовый `Account`». И обратная сторона границы: раз `static nested` ни к какому `Account` не приклеен, он и _не может_ лезть в его instance-поля — объекта-то нет. `Builder`-у это и не нужно, поэтому всё сходится. (Сам паттерн `Builder` во всей полноте — глава 25.4; здесь он лишь механический пример вложенности.)

> 💡 **Из C#:** для такого ты бы чаще взял object initializer (`new Account { Owner = ... }`) или `record` с `with`. В Java инициализаторов объекта нет, и классический ответ для иммутабельного объекта с валидацией — `static nested Builder`. То есть `static nested` = твой C#-nested, а `Builder` — место №1, где ты с ним столкнёшься.

#### `inner` — класс, приклеенный к объекту

`inner` — тот же вложенный класс, но **без** `static`. Он видит instance-поля внешнего объекта напрямую, потому что привязан к конкретному экземпляру. Самый честный пример, где он реально нужен, — итератор коллекции: ему надо ходить по данным _одного конкретного_ списка и помнить, где он сейчас.

```java
public class IntBox implements Iterable<Integer> {
    private int[] data = new int[10];
    private int size = 0;
    public void add(int v) { data[size++] = v; }

    @Override public Iterator<Integer> iterator() { return new Itr(); }

    private class Itr implements Iterator<Integer> {   // inner, без static
        private int cursor = 0;                        // своё состояние: докуда дошли
        @Override public boolean hasNext() { return cursor < size; }   // видит size напрямую
        @Override public Integer next()    { return data[cursor++]; }  // видит data напрямую
    }
}
```

Проверено на JDK 21 — `for (int x : box)` печатает `10 20 30`. Два момента. Первый: `Itr` лезет в `size` и `data` внешнего `IntBox` без всяких параметров — это его суперсила. Сделай его `static` — и он мгновенно перестанет их видеть, пришлось бы пихать массив и размер через конструктор руками. Второй: создаётся он строчкой `new Itr()` _внутри метода_ `IntBox`, и текущий `this` приклеивается автоматически. Странный синтаксис `box.new Itr()` нужен только чтобы создать `inner` _снаружи_ (проверено, компилируется и работает) — встречается редко, но узнавать в лицо надо.

Кто это реально использует: у `ArrayList`, `HashMap`, `HashSet` итераторы — `inner`-классы. Каждый твой `for (x : list)` под капотом зовёт `list.iterator()` и крутит приклеенный `inner`. А как `Itr` физически видит `size`, если это отдельный класс — Java при `new Itr()` втихую кладёт внутрь него ссылку на тот `IntBox`, при котором его создали. У этой ссылки есть имя в байткоде — `this$0`. Механику `this$0` и то, как она роняет прод по памяти, разберём в бонус-уроке 3.12 B.

> 💡 **Из C#:** это твой `IEnumerator` + `yield return`. Когда в C# пишешь `yield return`, компилятор генерит класс-машину состояний, помнящую позицию. Java заставляет написать этот класс-ходок руками (или берёт готовый из библиотеки), но идея та же: stateful-обходчик, привязанный к коллекции. Прямого аналога скрытой ссылки на внешний объект в C# нет.

#### `anonymous` — одноразовая реализация на месте (и откуда растут лямбды)

Иногда нужна **одноразовая** реализация интерфейса — здесь, один раз, больше нигде. Заводить ради этого именованный класс с файлом — перебор. Анонимный класс объявляет класс и тут же создаёт его объект одним выражением, без имени:

```java
names.sort(new Comparator<String>() {         // анонимный класс
    @Override public int compare(String a, String b) { return a.length() - b.length(); }
});
```

Нигде нет строчки `class SomethingComparator` — фигурные скобки с телом _и есть_ определение класса, созданного на месте. То же самое лямбдой:

```java
names.sort((a, b) -> a.length() - b.length());  // ровно то же, короче
```

**Лямбда — это стенография для анонимного класса, реализующего интерфейс с одним методом.** Отсюда сразу понятно, _почему_ лямбда работает только для интерфейсов с одним абстрактным методом (SAM): лямбда не называет, _какой_ метод реализует — Java понимает по тому, что он единственный. Будь методов два, было бы неоднозначно. Анонимный класс называет метод явно (`compare`), поэтому тянет любое их количество, а также состояние и наследование класса.

Один идиом ты точно напишешь — трюк Jackson, чтобы распарсить JSON в дженерик-коллекцию:

```java
var users = mapper.readValue(json, new TypeReference<List<User>>() {});
```

Вот этот `new TypeReference<List<User>>() {}` — анонимный класс ровно для одной цели, которую лямбда не умеет (он сохраняет generic-тип, который иначе стёрся бы; почему — type erasure, глава 7). Пока просто узнавай его в лицо.

> 💡 **Из C#:** анонимного класса в Java-смысле в C# почти нет. C#-шный anonymous type (`new { Name = "x" }`) — это мешок данных, а не «реализуй интерфейс на месте». C# эту задачу с самого начала решал делегатами и лямбдами; Java получила лямбды только в Java 8 (2014), и до этого анонимный класс был её громоздким способом сделать то, что C# делал лямбдой. Видишь в старом Java-коде здоровенную анонимку — мысленно переводи «в C# это была бы лямбда».

#### `local` — именованный класс внутри метода

`local class` объявлен прямо внутри метода, с именем, видимый только в этом методе. По сути — анонимный класс, которому дали имя, и имя нужно, когда объект надо создать в методе _несколько раз_, а снаружи он никому не интересен:

```java
void process(List<Transaction> txns) {
    long limit = 100_00;                       // effectively final
    class Flag {                               // local class
        final Transaction txn; final String reason;
        Flag(Transaction txn, String reason) { this.txn = txn; this.reason = reason; }
        void report() { System.out.println(txn.id() + ": " + reason); }
    }
    List<Flag> flags = new ArrayList<>();
    for (Transaction t : txns) {
        if (t.amountMinor() > limit) flags.add(new Flag(t, "превышен лимит"));
        if (t.amountMinor() < 0)     flags.add(new Flag(t, "отрицательная сумма"));
    }
    flags.forEach(Flag::report);
}
```

Сюда анонимка не лезет: `new Flag(...)` создаётся дважды, у него конструктор и состояние (анонимка — одно выражение = один объект). А выносить `Flag` в отдельный nested-класс не хочется — он часть логики только этого метода. Честная калибровка: на Java 16+ для таких **data-holder** случаев ты бы вообще объявил локальный `record` прямо в методе — `record Flag(Transaction txn, String reason) {}` — короче, и его тоже можно объявлять локально (проверено на JDK 21). Так что под data-кейсы голый local class съели локальные `record`, и поводов для него осталось совсем мало.

#### Захват переменных — коротко

`local` и `anonymous` (и лямбды) могут использовать локальные переменные метода, но переменная обязана быть **effectively final** — присвоена один раз и больше не менявшаяся. Причина в том, что захват в Java идёт **по значению**: копия значения кладётся в synthetic-поле класса-захватчика. Будь переменная изменяемой, копия и оригинал разъехались бы. Если нужно копить счётчик — захватывай не переменную, а изменяемый объект: `int[] acc = {0}; ... acc[0]++`, или `AtomicInteger`. Ссылка остаётся effectively final, а содержимое объекта меняй сколько угодно.

> 💡 **Из C#:** твой постоянный контраст — C# захватывает **по ссылке** через сгенерированный display-class, поэтому там захваченную переменную можно прямо менять, и метод с лямбдой видят одно поле. Java — по значению, отсюда требование effectively final. (Глубже про захват и внутренности лямбд — уроки 9.5 и 9.8.)

### 🔬 Глубже / на синьора

**Приватный доступ через nest-based access control (JEP 181, Java 11).** `inner` читает `private`-поле внешнего класса напрямую — но на уровне JVM у класса нет доступа к приватным членам _другого_ класса, а `Outer` и `Outer$Inner` после компиляции — разные классы. До Java 11 `javac` обходил это, генерируя synthetic bridge-методы (`access$000` и подобные) — лишние методы в байткоде. JEP 181 ввёл _nest_: `Outer` и все вложенные образуют «гнездо», и JVM разрешает членам одного гнезда лезть в приватные члены друг друга. Проверено на JDK 21 через `javap -v`: у `Outer` появляется атрибут `NestMembers`, у вложенного — `NestHost: class Outer`, а методов `access$000` в байткоде **нет**.

**`static`-члены в `inner` — только с Java 16.** До Java 16 в `inner`-классе можно было объявлять лишь `static final` константы; произвольные `static`-поля и методы запрещались (они концептуально конфликтуют с привязкой к экземпляру). Java 16 это ограничение сняло — проверено, `static int liveCount` в `inner` компилируется на JDK 21.

**`this` внутри анонимки ≠ `this` внутри лямбды.** В анонимном классе `this` — это _сам анонимный объект_ (отдельный класс `Outer$1`), поэтому `this.toString()` внутри анонимки вернёт её собственный. В лямбде своего `this` нет — лямбда не вводит новый scope для `this`, и `this` внутри неё — это _внешний_ экземпляр. Это следствие того, как они компилируются: анонимка — в отдельный `.class` (`Outer$1`), лямбда — через `invokedynamic`/`LambdaMetafactory` без отдельного класса (детали — урок 9.8). Проверено: лямбда без обращения к `this` компилируется в `private static` synthetic-метод, а трогающая `this` — в нестатический (тот самый захват `this`, из-за которого текут).

### ⚠️ Подводные камни

1. **`static nested` не видит instance-полей внешнего класса** — и это правильно: он не привязан к объекту. Если помощнику нужно состояние конкретного экземпляра — это сигнал, что нужен `inner`, а не `static nested`.
2. **`box.new Itr()` — неожиданный синтаксис** создания `inner` снаружи. Внутри метода внешнего класса пишешь `new Itr()` (текущий `this` приклеится сам), а снаружи — только `outer.new Inner()`. Многих на собесе это ставит в тупик.
3. **Лямбда только для SAM.** Интерфейс с двумя абстрактными методами лямбдой не реализуешь — нужен анонимный класс, называющий методы явно.
4. **`this` в анонимке — не внешний объект.** Обращение к `this` (или `MyClass.this` для внешнего) внутри анонимки путают постоянно; в лямбде такой ловушки нет.
5. **`inner` всегда несёт `this$0`** — даже если ни разу не обращается к полям внешнего. Ссылка проставлена безусловно, и именно из-за неё `inner`-объекты текут памятью (весь разбор — бонус-урок 3.12 B).

### На собесе

**1. «В чём разница между `static nested` и `inner`?»** (частый, ключевой водораздел)

RU: `inner` держит неявную ссылку на экземпляр внешнего класса (`this$0`), `static nested` — нет. Поэтому `inner` нельзя создать без объекта `Outer`, а `static nested` — можно, и именно `inner` течёт памятью. По умолчанию выбирай `static nested` — он легче и ничего за собой не держит; `inner` бери только когда реально нужен доступ к состоянию конкретного экземпляра `Outer`.

EN: An `inner` class holds an implicit reference to the outer instance (`this$0`); a `static nested` class doesn't. So an `inner` can't be created without an `Outer` object, while a `static nested` can, and it's the `inner` that leaks memory. Default to `static nested` — it's lighter and holds nothing; use `inner` only when you genuinely need access to a specific `Outer` instance's state.

Follow-up: _«А создать `inner` вообще без `Outer` можно?»_ → Нет. `this$0` должен на что-то указывать: либо изнутри метода `Outer` (`new Inner()`, `this` приклеится), либо снаружи через `outer.new Inner()`.

**2. «Чем анонимный класс отличается от лямбды?»**

RU: Три пункта. Первый — `this`: в анонимке это сам анонимный объект, в лямбде — внешний экземпляр (лямбда не вводит свой scope для `this`). Второй — компиляция: анонимка в отдельный `.class` (`Outer$1`), лямбда через `invokedynamic` без отдельного класса. Третий — возможности: анонимка может иметь состояние, несколько методов, наследовать класс; лямбда — только функциональный интерфейс с одним методом.

EN: Three points. First — `this`: in an anonymous class it's the anonymous object itself, in a lambda it's the enclosing instance (a lambda introduces no new `this` scope). Second — compilation: an anonymous class becomes a separate `.class` (`Outer$1`), a lambda goes through `invokedynamic` with no separate class. Third — capability: an anonymous class can hold state, have multiple methods, and extend a class; a lambda is limited to a single-method functional interface.

Follow-up: _«Что напечатает `this.toString()` внутри лямбды и внутри анонимки?»_ → В лямбде — `toString` внешнего класса, в анонимке — её собственный (она отдельный объект `Outer$1`).

**3. «Что напечатает?»** (подвох на захват)

```java
List<Runnable> tasks = new ArrayList<>();
for (int i = 0; i < 3; i++) { int captured = i; tasks.add(() -> System.out.print(captured)); }
tasks.forEach(Runnable::run);
```

RU: `012`. `captured` объявлена в теле цикла — на каждой итерации это _новая_ переменная, каждая лямбда захватила свою копию своего значения. Если захватить прямо `i` — не скомпилируется: `i` меняется (`i++`), не effectively final. Здесь C#-интуиция врёт: старый C#-код мог бы вывести `333`, Java просто не даст собрать.

EN: `012`. `captured` is declared inside the loop body — each iteration it's a _fresh_ variable, and each lambda captured its own copy of its own value. Capturing bare `i` won't compile: `i` mutates (`i++`), so it isn't effectively final. C# intuition misleads here — old C# code could print `333`; Java refuses to compile it.

**4. «На Java 21 — что генерит компилятор для доступа `inner` к приватному полю `Outer`?»** (senior, эволюция)

RU: На Java 21 — ничего особого. `Outer` и вложенные образуют _nest_, и nest-based access control (JEP 181, Java 11) разрешает им читать приватные члены друг друга напрямую; `javap -v` покажет `NestHost`/`NestMembers`. До Java 11 `javac` генерировал synthetic bridge-методы (`access$000`) — JEP 181 их убрал. И смежное: произвольные `static`-члены в `inner` разрешены только с Java 16, раньше — лишь `static final` константы.

EN: On Java 21, nothing special. `Outer` and its nested classes form a _nest_, and nest-based access control (JEP 181, Java 11) lets them read each other's private members directly; `javap -v` shows `NestHost`/`NestMembers`. Before Java 11, `javac` generated synthetic bridge methods (`access$000`) — JEP 181 removed them. Related: arbitrary `static` members in an `inner` class are allowed only since Java 16; before that, only `static final` constants.
