В C# доступ к состоянию — это properties: `public double Celsius { get; set; }`, снаружи выглядит как поле, внутри может быть логикой, и переход от автосвойства к вычисляемому не ломает вызывающий код. В Java такого синтаксиса нет вообще — есть только поля и методы. Поэтому «инкапсуляция состояния» здесь выражается парой обычных методов `getX()`/`setX()`, а вокруг их именования выстроена целая машинная конвенция (JavaBeans), на которую опирается пол-экосистемы: Jackson, JPA/Hibernate, старый Spring, шаблонизаторы. Понимать её надо не ради стиля, а потому что фреймворки читают состояние именно через эти имена.

**Ментальная модель.** Property в Java — не языковая сущность, а _соглашение об именах методов_: пара `getX()`/`setX()` (для `boolean` — `isX()`) образует «свойство `x`», даже если поля `x` нет. Инкапсуляция достигается тем, что поле `private`, а наружу торчат методы, которые можно переопределить, провалидировать или сделать вычисляемыми, не трогая клиентский код.

### Зачем вообще методы, если можно публичное поле

Публичное поле кажется короче. Проблема — в необратимости: как только `public double celsius;` попал в API, любой клиент пишет `t.celsius` напрямую, и в тот день, когда понадобится валидация, логирование, ленивое вычисление или смена внутреннего представления, менять придётся весь вызывающий код. Геттер/сеттер с самого начала оставляют это пространство:

```java
class Temperature {
    private double celsius;                          // состояние скрыто

    public double getCelsius()         { return celsius; }
    public void   setCelsius(double c) { this.celsius = c; }

    public double getFahrenheit()      { return celsius * 9 / 5 + 32; } // derived
    public boolean isFreezing()        { return celsius <= 0; }
}
```

Проверено на JDK 21: `setCelsius(25.0)` → `getFahrenheit()` возвращает `77.0`. Ключевое здесь — `getFahrenheit()` и `isFreezing()` **не имеют поля за собой**, это чисто вычисляемые свойства. Снаружи они неотличимы от геттеров реального поля — и в этом весь смысл: клиент не знает и не должен знать, что хранится, а что считается на лету.

💡 **Из C#:** это ровно то, что property решает синтаксически. Разница чисто в цене абстракции: в C# автосвойство `{ get; set; }` при необходимости превращается в `get => celsius * 9 / 5 + 32;` без единого изменения на стороне вызова, потому что синтаксис доступа (`t.Celsius`) одинаков для поля-подобного и вычисляемого. В Java «доступ» с самого начала — вызов метода `t.getCelsius()`, поэтому переход «поле → вычисление» бесплатен _по другой причине_: он и так всегда был методом. Вывод для дизайна одинаковый (не публикуй голое поле), механика — противоположная: C# прячет метод под синтаксис поля, Java просто не притворяется полем.

### JavaBeans-конвенция — и почему она машинно-значима

«Property» в Java формализована спецификацией JavaBeans через шаблон имён:

- геттер: `getX()` без аргументов, возвращает тип свойства;
- для `boolean` геттер обычно `isX()`;
- сеттер: `setX(T)`, один аргумент, `void`;
- имя свойства — `X` с маленькой буквы (`getCelsius` → свойство `celsius`).

Это не стилевая рекомендация — по этим именам фреймворки _находят_ свойства через рефлексию. Доказательство живьём: стандартный `java.beans.Introspector`, натравленный на класс выше, без всякой конфигурации выдал:

```
property: celsius     | read=getCelsius     | write=setCelsius
property: fahrenheit  | read=getFahrenheit  | write=-
property: freezing    | read=isFreezing     | write=-
```

Он увидел три свойства из одних имён методов. `fahrenheit` и `freezing` — read-only (сеттера нет), и поля `fahrenheit` в классе не существует вовсе — интроспектор оперирует методами, а не полями. Именно так Jackson сериализует объект в JSON `{"celsius":25.0,"fahrenheit":77.0,"freezing":false}`, а JPA маппит свойства на колонки: они вызывают `getX()`/`setX()`, найденные по шаблону. Нарушишь конвенцию (`temperature()` вместо `getCelsius()`) — фреймворк свойство не увидит.

💡 **Из C#:** аналог — рефлексия по `PropertyInfo`: `System.Text.Json` и EF Core тоже находят свойства автоматически, но опираются на _настоящие_ property-члены типа, которые есть в метаданных. В Java свойств в метаданных нет — есть только методы, поэтому «свойство» реконструируется эвристикой имён `get/set/is`. Практическое следствие: в Java геттер/сеттер, названный не по шаблону, для фреймворка не существует; в C# переименование метода не сломает биндинг property, потому что биндинг идёт по property, а не по имени метода.

### Когда голое поле всё-таки нормально

Догма «всё через геттеры» вредна не меньше публичных полей. Есть законные исключения:

- **`public static final`-константы** — `public static final int MAX = 100;` торчит полем намеренно, геттер к константе бессмысленен.
- **Приватные поля внутри пакета/класса** обслуживаются напрямую — инкапсуляция про _публичную границу_, а не про запрет трогать своё же поле.
- **`record`** (глава 17) — для иммутабельных носителей данных язык сам генерирует аксессоры, но по своей конвенции: `celsius()`, _без_ префикса `get`. Это осознанный разрыв с JavaBeans, и он иногда конфликтует со старыми фреймворками, ждущими `getX()`.

Effective Java (Item 16) формулирует так: публичные классы не должны выставлять изменяемые поля напрямую; но для package-private или private вложенных классов выставить поле бывает и чище.

### ⚠️ Подводные камни

**1. Сеттер, тихо роняющий значение (shadowing).** `setCelsius(double celsius) { celsius = celsius; }` — параметр присвоен сам себе, поле не тронуто, компилятор молчит. Разбирали в 3.2: слева всегда `this.`. В сеттерах эта ошибка особенно живуча, потому что имя параметра почти всегда совпадает с полем.

**2. `boolean` и `is`/`get`.** Фреймворки по-разному относятся к `isActive()` vs `getActive()`; для `Boolean` (обёртки) поведение исторически расходится с `boolean` (примитивом). Если сериализованный JSON недосчитался булева свойства — первый подозреваемый именно префикс. Держись `isX()` для примитивного `boolean`.

**3. Сеттер, отдающий наружу мутабельную ссылку.** `void setTags(List<String> t) { this.tags = t; }` сохраняет _чужой_ список — вызывающий продолжит его менять за спиной объекта. Симметрично `getTags()`, возвращающий внутренний список, отдаёт наружу право на мутацию. Лечение — защитные копии (глава 3.7); здесь просто фиксируем, что сеттер/геттер — típичное место утечки инкапсуляции.

**4. Геттер с побочным эффектом.** `getX()`, который лениво инициализирует, логирует или меняет состояние, ломает ожидания: и люди, и фреймворки зовут геттеры многократно и в произвольном порядке (тот же Jackson — при каждой сериализации). Геттер должен быть идемпотентным чтением.

**5. Бойлерплейт как симптом.** Класс, где на N полей ровно N тривиальных `getX`/`setX` и больше ничего, — обычно не объект с поведением, а структура данных. В современной Java это сигнал взять `record` (иммутабельно) — глава 17. Плодить геттеры/сеттеры руками там, где нужен носитель данных, — не «инкапсуляция», а шум.

### На собесе

**1. «Почему в Java геттеры/сеттеры, а не публичные поля?»** _(частый, базовый)_

_Модельный ответ (RU):_ «Публичное поле фиксирует прямой доступ `obj.field` в API навсегда — в момент, когда понадобится валидация, вычисление или смена представления, ломается весь клиентский код. Геттер/сеттер оставляют это пространство с самого начала: поле `private`, наружу — методы, которые можно переопределить, провалидировать или сделать вычисляемыми, не трогая вызывающих. Плюс на именах `getX/setX/isX` держится вся JavaBeans-экосистема — Jackson, JPA, шаблонизаторы находят свойства по этому шаблону.»

_(EN):_ "A public field bakes direct `obj.field` access into the API permanently — the moment you need validation, a computed value, or a representation change, every caller breaks. Getters/setters keep that room from day one: the field is `private`, exposed through methods you can override, validate, or make derived without touching callers. On top of that, the whole JavaBeans ecosystem — Jackson, JPA, templating — discovers properties by the `getX/setX/isX` naming pattern."

_Follow-up:_ «Всегда ли через геттер?» → нет: `public static final`-константы, package-private/private классы; для иммутабельных носителей данных — `record`.

**2. «Что такое property в Java?»** _(проверка на понимание, что языковой сущности нет)_

_Модельный ответ (RU):_ «Языковой конструкции property в Java нет — есть поля и методы. "Property" — это соглашение JavaBeans: пара `getX()`/`setX()` (для `boolean` — `isX()`) образует свойство `x`, причём поля `x` может и не быть — свойство может быть чисто вычисляемым. Фреймворки реконструируют свойства из имён методов через рефлексию/`Introspector`.»

_(EN):_ "Java has no `property` language construct — only fields and methods. A 'property' is a JavaBeans convention: a `getX()`/`setX()` pair (`isX()` for `boolean`) forms property `x`, and there may be no field `x` at all — the property can be purely computed. Frameworks reconstruct properties from the method names via reflection/`Introspector`."

_Типичный слабый ответ:_ «property — это приватное поле с геттером и сеттером». Пропускает главное: свойство определяется методами, а не полем, и бывает без поля вовсе (`getFahrenheit()`).

**3. «Как Jackson/JPA понимает, какие у объекта свойства?»**

_Модельный ответ (RU):_ «Через рефлексию по JavaBeans-шаблону: сканирует методы, находит `getX/isX/setX`, выводит имя свойства из имени метода (`getCelsius` → `celsius`). Поэтому геттер, названный не по конвенции, для фреймворка не существует, а вычисляемый геттер без поля — наоборот, попадёт в JSON как полноценное свойство.»

_(EN):_ "Via reflection over the JavaBeans pattern: it scans methods, matches `getX/isX/setX`, and derives the property name from the method name (`getCelsius` → `celsius`). So a getter named off-convention is invisible to the framework, while a computed getter with no backing field still shows up in JSON as a real property."

_Follow-up-trap:_ «Есть метод `getFahrenheit()`, но поля `fahrenheit` нет — что в JSON?» → появится `"fahrenheit"`: сериализатор идёт по геттерам, наличие поля ему безразлично.

**4. «`record` — это property?»** _(мост к главе 17)_

_Модельный ответ (RU):_ «`record` даёт аксессоры автоматически, но по своей конвенции — `celsius()`, без `get`. Формально это тоже "чтение свойства", но имена расходятся с классическим JavaBeans, поэтому старые фреймворки, жёстко ждущие `getX()`, с рекордами могут не работать без настройки. Современные версии Jackson рекорды понимают.»

_(EN):_ "A `record` generates accessors automatically, but by its own convention — `celsius()`, no `get`. It's still property reading conceptually, but the names diverge from classic JavaBeans, so older frameworks hard-wired to `getX()` may need configuration to work with records. Recent Jackson versions support records directly."
