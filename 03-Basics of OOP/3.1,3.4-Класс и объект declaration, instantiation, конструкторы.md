В C# создание объекта обвешано сахаром: object initializers, `init`-сеттеры, primary constructors — язык даёт несколько дверей к новому объекту. В Java дверь одна: конструктор. Вся дисциплина инициализации — обязательность полей, инварианты, иммутабельность — выражается только через него, поэтому конструкторная механика в Java несёт заметно больше нагрузки, чем ты привык. И ровно поэтому вокруг неё крутится целый пласт собесных вопросов: когда компилятор генерирует конструктор, а когда нет; в каком порядке отрабатывают инициализаторы; почему `this()` и `super()` обязаны стоять первой строкой. Этот урок закрывает фундамент: что физически происходит при `new`, и как устроена жизнь конструкторов в пределах одного класса. Наследование сюда не тащим — полный порядок инициализации с иерархией будет отдельным мастерклассом после главы 4.

**Ментальная модель.** Класс — чертёж: его метаданные живут один раз на программу. Объект — блок в куче, отштампованный по чертежу: `new` выделяет память, JVM зануляет её дефолтами, конструктор доводит до валидного состояния, наружу уходит ссылка. Объект не существует «наполовину созданным» для твоего кода — конструктор либо довёл его до конца, либо бросил исключение.

### Declaration и instantiation — где что живёт

Объявление класса — это описание: какие поля, какие методы, какие конструкторы. Само по себе оно не создаёт ни одного объекта. `new` — это создание экземпляра по этому описанию:

```java
class User {
    String name;
    int age;
}

User u = new User();   // instantiation: объект в heap, ссылка в u
```

Переменная `u` — не объект. Это ссылка (в C#-терминах — то же самое, что переменная reference type): маленькое значение на стеке или в другом объекте, указывающее на блок в куче. Присваивание `User v = u;` копирует ссылку, не объект — два имени, один блок памяти.

> [!tip] 💡 Из C#
> для `class` картина идентична .NET — ссылка отдельно, объект в куче отдельно. Ключевое отличие Java: здесь **нет** `struct` и пользовательских value types. Каждый `new` — семантически всегда объект в куче; варианта «разместить на стеке» язык не предоставляет. (JIT умеет иногда не аллоцировать через escape analysis — об этом в 🔬-блоке, но это оптимизация рантайма, а не семантика языка.)

### Что физически делает `new`

`new User()` — не одна операция, а конвейер из четырёх шагов:

1. **Аллокация** в куче: JVM выделяет блок под объект — в начале object header (служебный заголовок), дальше поля.
2. **Зануление**: вся память под поля заполняется дефолтами — `0` для числовых типов, `false` для `boolean`, `'\u0000'` для `char`, `null` для ссылок. Это происходит _до любого твоего кода_ — «грязной» памяти из-под предыдущего объекта ты не увидишь никогда.
3. **Инициализация**: инициализаторы полей, instance-блоки, тело конструктора — в порядке, который разберём ниже.
4. Наружу отдаётся **ссылка** на готовый объект.

![[Pasted image 20260703125211.png]]

Про header на этом уровне достаточно знать две вещи: в нём лежит служебное состояние объекта (identity hashcode, состояние блокировки, GC-биты — это mark word) и указатель на метаданные класса (class pointer) — через него работают `getClass()` и виртуальная диспетчеризация. Метаданные при этом одни на класс, сколько бы объектов ты ни создал. Детали устройства header — в 🔬-блоке.

### Дефолты полей vs локальные переменные

Из шага «зануление» следует правило, которое стоит зафиксировать сразу: **поля** объекта всегда имеют дефолтное значение ещё до конструктора. Проверено на JDK 21 — свежесозданные поля дают ровно это:

```java
int i;  long l;  double d;  boolean b;  char c;  String s;
// i=0, l=0, d=0.0, b=false, c='\u0000', s=null
```

**Локальные переменные** дефолта не получают. Компилятор требует доказуемого присваивания до первого чтения — правило definite assignment (JLS §16):

```java
void method() {
    int x;
    System.out.println(x);
    // error: variable x might not have been initialized
}
```

Почему такая асимметрия: для локальной переменной компилятор видит все пути внутри метода и может статически доказать «присвоено до чтения». Для поля порядков обращения слишком много — через методы, рефлексию, другие потоки — доказать ничего нельзя, поэтому язык вместо доказательства даёт гарантию: дефолт всегда.

> [!tip] 💡 Из C#
> здесь один в один — те же дефолты для полей, тот же definite assignment для локальных. Переучиваться нечего, просто знай, что правило и его имя спрашивают.

### Конструкторы: default, перегрузка

**Default constructor** — конструктор без аргументов, который компилятор генерирует сам. Ключевое правило: генерирует **только если ты не написал ни одного конструктора явно**. Написал хоть один — дефолтного больше нет:

```java
class User {
    User(String name) { }    // есть явный конструктор
}

new User();
// error: constructor User in class User cannot be applied to given types;
//   required: String, found: no arguments
```

Это не мелочь: добавление первого конструктора с параметрами — изменение, ломающее всех клиентов, которые звали `new User()`. Отсюда идиома: если no-arg конструктор нужен рядом с параметризованным (частое требование фреймворков сериализации/ORM) — пиши его явно.

**Перегрузка** конструкторов работает как перегрузка методов: несколько конструкторов с разными сигнатурами, выбор по аргументам в compile-time.

> [!tip] 💡 Из C#
> правило про default constructor идентично C# — «добавил свой ctor, потерял сгенерированный». Тут мост прямой.

### `this(...)` и `super(...)` — делегирование и цепочка

`this(...)` вызывает _другой конструктор этого же класса_, `super(...)` — конструктор _родителя_. Канонический паттерн — перегрузки делегируют одному «главному» конструктору, где сосредоточена вся логика и валидация:

```java
class Rectangle {
    private final int w;
    private final int h;

    Rectangle(int w, int h) {                       // главный ctor
        if (w <= 0 || h <= 0)
            throw new IllegalArgumentException("стороны > 0");
        this.w = w;
        this.h = h;
    }

    Rectangle(int side) {
        this(side, side);                           // делегируем
    }
}
```

Три правила, каждое проверено компилятором JDK 21:

**Первая строка.** И `this(...)`, и `super(...)` обязаны быть первым statement конструктора. Попытка поставить второй вызов после первого падает с `call to this must be first statement in constructor`. Рационал: к моменту выполнения тела конструктора объект должен быть в согласованном состоянии — родительская часть проинициализирована, либо управление полностью передано другому конструктору. На Java 21 LTS это жёсткое правило; послабление появилось позже — см. 🔬-блок про JEP 513.

**Один из двух.** В одном конструкторе либо `this()`, либо `super()` — не оба. Цепочка `this()` в итоге всё равно приводит к конструктору, который вызывает `super()` (явно или неявно).

**Без циклов.** `this()`-цепочка не может замкнуться сама на себя — компилятор ловит: `error: recursive constructor invocation`.

**Неявный `super()` — классическая ловушка.** Если ты не написал ни `this()`, ни `super()`, компилятор вставляет `super()` без аргументов первой строкой. И если у родителя нет no-arg конструктора — код не компилируется:

```java
class Base {
    Base(int x) { }        // только ctor с аргументом
}
class Derived extends Base {
    Derived() { }
    // error: constructor Base in class Base cannot be applied
    //   to given types; required: int, found: no arguments
}
```

Чинится явным `super(42)` первой строкой. Наследование мы разберём в главе 4, но эту механику надо знать уже сейчас — она чисто конструкторная, и её спрашивают в связке с default constructor.

> [!tip] 💡 Из C#
> семантика та же, синтаксис другой. C# выражает цепочку в сигнатуре — `: this(...)` / `: base(...)` после списка параметров, — и поэтому «первая строка» там обеспечена синтаксически, нарушить нельзя. Java пишет вызов внутри тела, и правило первой строки проверяет компилятор. `base` ↔ `super`, всё остальное совпадает, включая неявный вызов родительского no-arg конструктора и ту же ошибку при его отсутствии.

### Порядок инициализации внутри одного класса

Между «память занулена» и «тело конструктора выполнилось» есть ещё один слой: **инициализаторы полей** (`int a = ...;`) и **instance-блоки** (`{ ... }`). Вот проверенный на JDK 21 прогон:

```java
class Order {
    private int a = trace("field a");
    { System.out.println("instance block #1"); }
    private int b = trace("field b");
    { System.out.println("instance block #2"); }

    Order() {
        System.out.println("constructor body");
    }
    Order(String tag) {
        this();
        System.out.println("constructor(" + tag + ")");
    }
    static int trace(String s) { System.out.println(s); return 42; }
}

new Order();
new Order("copy");
```

```
field a              ← new Order()
instance block #1
field b
instance block #2
constructor body
---
field a              ← new Order("copy")
instance block #1
field b
instance block #2
constructor body     ← сначала this() целиком
constructor(copy)    ← потом тело внешнего ctor
```

Два вывода из этого прогона:

**Текстовый порядок.** Инициализаторы полей и instance-блоки — единая последовательность, выполняемая строго в порядке появления в файле, вперемешку. Не «сначала все поля, потом все блоки».

**Один раз на объект, даже при `this()`-цепочке.** Обрати внимание на второй `new`: инициализаторы отработали один раз, перед телом самого «глубокого» конструктора цепочки. Механика: компилятор вшивает инициализаторы только в конструкторы, начинающиеся с `super()` (явного или неявного); конструктор, начинающийся с `this()`, их не содержит — иначе поля инициализировались бы дважды.

![[Pasted image 20260703125249.png]]

Текстовый порядок охраняется компилятором — прямое чтение поля, объявленного ниже, не компилируется:

```java
class F {
    int a = b;      // error: illegal forward reference
    int b = 5;
}
```

Но проверка эта чисто синтаксическая, и через метод она обходится — и вот тут дефолты из шага «зануление» становятся наблюдаемыми:

```java
class F2 {
    int a = getB();     // компилируется!
    int b = 5;
    int getB() { return b; }
}
// new F2(): a == 0, b == 5   — getB() прочитал b ДО его инициализатора
```

Проверено: `a = 0, b = 5`. Никакого исключения, никакого предупреждения — тихий `0` (для ссылки был бы тихий `null`). Это одноклассовая версия той самой ловушки «конструктор видит недоинициализированное состояние»; её полномасштабный вариант с наследованием и виртуальными вызовами — тема будущего мастеркласса.

### `blank final` — обязательство перед компилятором

`final`-поле без инициализатора называется blank final. Компилятор требует, чтобы **каждый** путь через конструкторы присвоил его **ровно один раз** — то же definite assignment, но теперь в обе стороны: «точно присвоено» и «точно не дважды»:

```java
class Config {
    private final String env;              // blank final

    Config(String env) { this.env = env; } // ок

    Config(boolean prod) {
        if (prod) { env = "PROD"; }
        // error: variable env might not have been initialized
        //        (ветка else не присваивает)
    }
}
```

Это фундамент иммутабельности: `final`-поля + присваивание в конструкторе = объект, состояние которого зафиксировано в момент создания. Полный разбор иммутабельности — в 3.7, но механика живёт здесь.

> [!tip] 💡 Из C#
> аналог — `readonly`-поле: тоже присваивается в объявлении или конструкторе. Разница в строгости: C# позволяет `readonly`-полю остаться неприсвоенным (получит дефолт, компилятор максимум предупредит через анализаторы), Java же делает неприсвоенный blank final **ошибкой компиляции**. Java здесь жёстче — и это удобно: забыть инициализировать `final`-поле нельзя физически.

### Чего в Java нет: object initializer

Синтаксиса `new User { Name = "Artem", Age = 25 }` в Java не существует ни в каком виде. Единственный способ породить объект — конструктор; «присвоить после» — это уже обычные вызовы сеттеров на живом объекте. Чем закрывают потребность в читаемой сборке объекта с кучей опциональных полей — паттерн Builder, вынесен в appendix в конце урока. И одно предостережение наперёд: встретишь в чужом коде конструкцию `new ArrayList<>() {{ add("a"); }}` — это _не_ инициализатор, а анонимный подкласс с instance-блоком внутри; почему это антипаттерн, разберём в 3.12 про вложенные классы.

> [!example] 🔬 Глубже / на синьора
> **Байткод `new`.** Компилятор разворачивает `Object o = new Object();` в три инструкции — вот реальный `javap -c` с JDK 21:
> 
> ```
> 0: new           #2    // class java/lang/Object
> 3: dup
> 4: invokespecial #1    // Method java/lang/Object."<init>":()V
> 7: astore_1
> ```
> 
> `new` — только аллокация и зануление, объект после неё «сырой». `dup` дублирует ссылку на стеке операндов (одна копия уйдёт конструктору и будет им потреблена, вторая останется тебе). `invokespecial <init>` — вызов конструктора: на уровне байткода конструктор — это метод со специальным именем `<init>`, вызываемый невиртуально. Разделение на `new` + `invokespecial` — то самое место, где живёт «объект существует, но ещё не инициализирован»; верификатор байткода не даёт пользоваться ссылкой до вызова `<init>`.
> 
> **Object header подробнее.** В HotSpot header — это mark word (машинное слово: identity hashcode, биты возраста для GC, состояние блокировки для `synchronized`) плюс class pointer (compressed class pointer при включённых compressed oops — дефолт на heap до ~32 ГБ). Практическое следствие: у «пустого» объекта ненулевой размер (на 64-бит HotSpot с compressed oops — 16 байт с выравниванием), и миллион мелких объектов-обёрток — это десятки мегабайт чистых header'ов. Точные размеры зависят от флагов JVM — для точных замеров есть JOL (`jol-core`).
> 
> **Escape analysis.** JIT (C2) умеет доказать, что объект не «убегает» из метода (не сохраняется в поля, не передаётся наружу, не возвращается), и тогда применить scalar replacement — разложить поля объекта по регистрам, не аллоцируя в куче вовсе. Это оптимизация рантайма, не семантика: язык по-прежнему говорит «каждый `new` — объект в куче», просто JIT иногда может не платить за это. На собесе упоминай одной фразой и именно в такой рамке.
> 
> **JEP 513 — flexible constructor bodies.** Правило «`this()`/`super()` первой строкой» ослаблено: финализировано в JDK 25 (JEP 513, сентябрь 2025) после цепочки preview (JEP 447 в Java 22, 482 в 23, 492 в 24). Теперь конструктор делится на prologue (код _до_ вызова `super()`/`this()` — валидация аргументов, вычисления; трогать `this` нельзя, кроме присваивания собственным полям) и epilogue (код после, как раньше). Мотивация — убрать уродливый паттерн `super(validate(x))` со static-хелперами. **На baseline Java 21 LTS этого нет** — там правило первой строки жёсткое, и отвечать на собесе надо именно так, добавляя JEP 513 как «знаю, куда движется язык».

> [!warning] ⚠️ Подводные камни
> 
> **1. Первый явный конструктор ломает `new X()`.** Добавил в класс конструктор с параметрами — сгенерированный no-arg исчез, и всё, что звало `new X()` (включая фреймворки сериализации/ORM, которым нужен no-arg конструктор), перестало компилироваться или работать. Нужны оба — пиши no-arg явно.
> 
> **2. Неявный `super()` против родителя без no-arg конструктора.** Не написал `super(...)` — компилятор вставил `super()`; у родителя только конструкторы с параметрами — ошибка компиляции в _потомке_, хотя «сломан» с виду родитель. Особенно сбивает, когда родитель — чужой класс из библиотеки.
> 
> **3. Тихие дефолты через методы в инициализаторах.** Прямой forward reference компилятор ловит, но `int a = getB();` — нет: метод прочитает `b` до его инициализатора и вернёт `0`/`null` без единого предупреждения. Правило: инициализаторы полей не должны вызывать методы, читающие другие поля.
> 
> **4. `blank final` и ветвление.** Каждый путь каждого конструктора обязан присвоить blank final ровно один раз. Забытая ветка `else`, ранний `return`, путь через `catch` — и компиляция падает. Это фича, а не боль: компилятор бесплатно проверяет полноту инициализации.
> 
> **5. `this()`-цепочка и двойная работа.** Инициализаторы полей выполняются один раз (перед телом «глубокого» конструктора цепочки), а вот код, продублированный в телах нескольких конструкторов вместо делегирования, — столько раз, сколько написан. Валидацию и логику держи в одном главном конструкторе, остальные — тонкие обёртки через `this()`.
> 
> **6. Вызов переопределяемого метода из конструктора.** Даже в пределах знаний этого урока видно, почему это опасно: конструктор работает на объекте, чьи поля могут быть ещё в дефолтах. С наследованием это превращается в знаменитый trap с `null` — целиком разберём в мастерклассе по порядку инициализации, пока правило: из конструктора не зовём ничего переопределяемого (Effective Java, Item 19).

### На собесе

> [!question]- 1. «Что происходит при `new`?» _(частый, разогревочный)_
> 
> _Модельный ответ (RU):_ «`new` — это аллокация блока в куче (header + поля), зануление всех полей дефолтами до любого пользовательского кода, затем инициализация — инициализаторы полей и instance-блоки в текстовом порядке, тело конструктора, — и наружу уходит ссылка. Метаданные класса при этом не копируются — они одни на класс, объект держит на них указатель в header.»
> 
> _(EN):_ "`new` allocates a block on the heap — object header plus fields — zeroes all fields to defaults before any user code runs, then executes initialization: field initializers and instance blocks in textual order, then the constructor body. What you get back is a reference. Class metadata isn't copied per object — the header holds a pointer to it."
> 
> _Follow-up:_ «Может ли объект оказаться на стеке?» → семантически — нет, в Java нет value types; JIT может через escape analysis не аллоцировать объект, не убегающий из метода, но это оптимизация, а не гарантия языка. _Синьорский добавок:_ байткод — `new` + `dup` + `invokespecial <init>`; конструктор на уровне JVM — метод `<init>`, вызываемый невиртуально.
> 
> _Типичный слабый ответ:_ «`new` вызывает конструктор» — пропускает аллокацию и зануление, а на них держатся follow-up'ы про дефолты.

> [!question]- 2. «Когда компилятор генерирует конструктор по умолчанию?»
> 
> _Модельный ответ (RU):_ «Только когда в классе нет ни одного явного конструктора. Появился хоть один — default constructor не генерируется, и `new X()` перестаёт компилироваться. Если no-arg нужен рядом с параметризованным — например, для фреймворка — его пишут явно.»
> 
> _(EN):_ "Only if the class declares no constructors at all. Once you add any explicit constructor, the default one is gone and `new X()` no longer compiles — if you need a no-arg constructor alongside, you write it explicitly."
> 
> _Follow-up-trap:_ «А у default constructor какой модификатор доступа?» → тот же, что у класса (у `public` класса — `public`, у package-private — package-private).

> [!question]- 3. «Правила `this()` и `super()`»
> 
> _Модельный ответ (RU):_ «Оба обязаны быть первым statement конструктора, в одном конструкторе — только один из них, циклы `this()` запрещены. Нет явного вызова — компилятор вставляет `super()` без аргументов; если у родителя нет no-arg конструктора, потомок не скомпилируется, чинится явным `super(args)`. На Java 21 правило первой строки жёсткое; в Java 25 его ослабил JEP 513 — можно выполнять валидацию до `super()`.»
> 
> _(EN):_ "Both must be the first statement; a constructor may contain at most one of them; recursive `this()` chains are a compile error. Without an explicit call the compiler inserts a no-arg `super()` — if the parent has no no-arg constructor, the subclass fails to compile. That first-statement rule is strict on Java 21; Java 25 relaxed it with JEP 513, allowing validation before `super()`."
> 
> _Follow-up:_ «Зачем правило первой строки?» → к началу тела конструктора объект должен быть в согласованном состоянии: родительская часть построена либо управление делегировано целиком.

> [!question]- 4. «В каком порядке инициализируется объект (один класс)?»
> 
> _Модельный ответ (RU):_ «Зануление дефолтами, потом инициализаторы полей и instance-блоки — единой последовательностью строго в текстовом порядке, потом тело конструктора. При `this()`-цепочке инициализаторы выполняются один раз — перед телом самого глубокого конструктора: компилятор вшивает их только в конструкторы, начинающиеся с `super()`.»
> 
> _(EN):_ "Fields are zeroed to defaults, then field initializers and instance initializer blocks run as one sequence in textual order, then the constructor body. With a `this()` chain the initializers run exactly once — before the body of the deepest constructor — because the compiler emits them only into constructors that start with `super()`."
> 
> _Follow-up-trap:_ «`int a = getB(); int b = 5;` — что в `a`?» → `0`: прямой forward reference не компилируется, а через метод — компилируется и читает дефолт. _Follow-up:_ «А с наследованием?» → static родителя → static потомка (однократно), затем instance-часть родителя → потомка; полный разбор — отдельная тема.

> [!question]- 5. «Почему поля имеют дефолт, а локальные переменные — нет?»
> 
> _Модельный ответ (RU):_ «Для локальных переменных компилятор статически доказывает присваивание до чтения — definite assignment, JLS §16; неприсвоенная локальная — ошибка компиляции. Для полей доказать это невозможно — обращения идут через методы, рефлексию, другие потоки — поэтому язык вместо доказательства даёт гарантию: JVM зануляет поля при аллокации.»
> 
> _(EN):_ "For locals the compiler statically proves assignment-before-read — definite assignment, JLS §16 — so an unassigned local is a compile error. For fields that proof is impossible (access via methods, reflection, other threads), so instead of proving, the language guarantees: the JVM zeroes fields at allocation."

> [!question]- 6. «Что такое blank final?»
> 
> _Модельный ответ (RU):_ «`final`-поле без инициализатора. Каждый путь через конструкторы обязан присвоить его ровно один раз — иначе ошибка компиляции. Это механическая основа иммутабельных классов: состояние фиксируется в конструкторе, и компилятор гарантирует, что ни один путь его не пропустил.»
> 
> _(EN):_ "A `final` field with no initializer. Every constructor path must assign it exactly once, or the code doesn't compile. It's the mechanical backbone of immutable classes: state is fixed in the constructor, and the compiler guarantees no path skips it."
> 
> _Follow-up:_ «Отличие от C# `readonly`?» → семантика похожа, но C# допускает неприсвоенный `readonly` (останется дефолт), Java делает это ошибкой компиляции.

### ⏩ Appendix: Builder — продвинутая вставка

_Забегаем вперёд: канонически это глава 25.4 (плюс иммутабельность из 3.7), но паттерн — прямое продолжение конструкторной темы и нужен в текущей работе уже сейчас. Здесь — рабочий рецепт; в 25.4 вернёмся к нему в контексте остальных creational-паттернов._

Проблема, которую решает Builder: у объекта много полей, часть опциональна, и конструкторный подход разваливается. Телескопические конструкторы (`Cat(name, age)`, `Cat(name, age, color)`, `Cat(name, age, color, indoor)`…) дают комбинаторный взрыв, а на месте вызова `new Cat("Felix", 3, "ginger", false, null)` невозможно вспомнить, что значит четвёртый аргумент. Сеттеры (JavaBeans) решают читаемость, но ценой мутабельности: объект можно увидеть полусобранным и менять после. Builder даёт третий путь — именованную, читаемую сборку, на выходе которой иммутабельный и атомарно провалидированный объект.

#### Рецепт — пять шагов

1. **Целевой класс — иммутабельный:** все поля `final`, конструктор `private` и принимает билдер: `private Cat(Builder b)`. Единственная дверь к объекту — через билдер.
2. **Builder — `public static` вложенный класс** внутри целевого. (Почему именно `static` — тема 3.12; короткая версия: билдеру не нужен существующий объект `Cat`, он его создаёт.)
3. **В билдере дублируются поля:** обязательные — `final`, задаются в конструкторе _билдера_; опциональные — обычные поля с разумными дефолтами.
4. **На каждое опциональное поле — метод по имени поля, возвращающий `this`** — отсюда цепочка вызовов.
5. **`build()`** проверяет инварианты и возвращает `new Cat(this)`.

#### Пример (проверен на JDK 21)

```java
public final class Cat {
    private final String name;                  // обязательное
    private final int age;                      // обязательное
    private final String color;                 // опциональное
    private final boolean indoor;               // опциональное
    private final List<String> nicknames;       // мутабельный тип → защитная копия

    private Cat(Builder b) {
        this.name      = b.name;
        this.age       = b.age;
        this.color     = b.color;
        this.indoor    = b.indoor;
        this.nicknames = List.copyOf(b.nicknames);
    }

    public static final class Builder {
        private final String name;              // обязательные → ctor билдера
        private final int age;
        private String color = "unknown";       // опциональные → дефолты
        private boolean indoor = true;
        private final List<String> nicknames = new ArrayList<>();

        public Builder(String name, int age) {
            this.name = name;
            this.age  = age;
        }
        public Builder color(String c)       { this.color = c;  return this; }
        public Builder indoor(boolean v)     { this.indoor = v; return this; }
        public Builder addNickname(String n) { this.nicknames.add(n); return this; }

        public Cat build() {
            if (name == null || name.isBlank())
                throw new IllegalArgumentException("name обязателен");
            if (age < 0)
                throw new IllegalArgumentException("age >= 0");
            return new Cat(this);
        }
    }
}
```

Использование и фактический вывод прогона:

```java
Cat felix = new Cat.Builder("Felix", 3)
        .color("ginger").indoor(false).addNickname("Рыжий").build();
// Cat{name=Felix, age=3, color=ginger, indoor=false, nicknames=[Рыжий]}

new Cat.Builder("Whiskers", 1).build();
// Cat{name=Whiskers, age=1, color=unknown, indoor=true, nicknames=[]}

new Cat.Builder("", -5).build();
// IllegalArgumentException: name обязателен — недостроенный Cat наружу не вышел
```

#### Практики — с обоснованием

**Продукт иммутабелен, конструктор `private`.** Половина ценности билдера — на выходе замороженный, провалидированный объект: он потокобезопасен «бесплатно» и не бывает полусобранным. Публичный конструктор рядом с билдером открывает обход валидации и убивает эту гарантию.

**Обязательность — на уровне компилятора.** Обязательные параметры идут в конструктор билдера: `new Cat.Builder()` без `name`/`age` физически не компилируется. Это дешевле и надёжнее рантайм-проверки.

**Валидация — в `build()`.** Межполевые инварианты можно проверить только когда заполнены все поля — то есть в `build()`. Кидаешь `IllegalArgumentException` с внятным сообщением; объект до прохождения проверки не существует. Простые однополевые проверки допустимо ставить прямо в fluent-метод для fail-fast.

**Защитные копии для мутабельных типов.** Без `List.copyOf` вызвавший сохранит свою ссылку на `List` и поменяет «иммутабельного» кота за спиной. `List.copyOf` закрывает обе стороны: неизменяемая копия при сборке, и её же можно отдавать из аксессора как есть. Копию делай в конструкторе продукта, не в fluent-методе — валидировать надо то, что реально ляжет в объект.

**Когда билдер НЕ нужен.** Два-три обязательных поля без опциональных и без инвариантов — это конструктор (а если объект — чистый носитель данных, то `record`, глава 17): билдер там — пустая церемония. Ориентир из JDK: `HttpRequest` — иммутабельный, но строится билдером, потому что полей много и большинство опциональны.

> [!tip] 💡 Из C#
> билдер в Java закрывает то, что в C# 9+ делает связка object initializer + `init`-сеттеры: `new Cat { Color = "ginger" }` даёт и именованную читаемую сборку, и иммутабельность после создания. В Java `init`-сеттеров нет, поэтому эквивалент собирается руками. Практическое правило: ловишь себя на мысли «в C# тут был бы инициализатор» — в Java это сигнал «возможно, нужен билдер». В индустрии его часто генерят Lombok'ом (`@Builder`), но на собесе просят написать руками — рецепт выше и есть ответ.

**Частые ошибки:** оставить сеттеры или не-`final` поля на продукте (мутабельность убивает смысл); забыть защитную копию (объект «течёт» через мутабельную ссылку); публичный конструктор рядом с билдером (обход валидации); билдер на трёх обязательных полях (церемония без пользы).
