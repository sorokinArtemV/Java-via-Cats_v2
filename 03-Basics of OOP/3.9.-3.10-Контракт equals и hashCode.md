### Что это и зачем

Как только ты делаешь класс, объекты которого должны считаться одинаковыми _по содержимому_ — деньги, координаты, идентификаторы, ключи кеша, — ты попадаешь под договор между `equals` и `hashCode`. Договор держится на честном слове: компилятор его не проверяет, а `HashSet` и `HashMap` молча верят, что ты его соблюдаешь. Нарушение проявляется не исключением и не падением, а тихой поломкой хеш-коллекций: дубликаты в `Set`, `get` возвращает `null` для существующего ключа, `put` не перезаписывает, а создаёт второй entry.

Это самый частый «отсеивающий» блок на Java-собесах, потому что он за один вопрос проверяет три вещи: знаешь ли ты контракт дословно, понимаешь ли _почему_ он именно такой, и ловил ли руками баг, который он порождает. Разберём все три слоя.

### Ментальная модель

`equals` определяет, какие объекты считать одинаковыми по смыслу; `hashCode` раскладывает их по бакетам ещё _до_ того, как `equals` вообще получит шанс сработать. Отсюда и односторонний договор: равные объекты обязаны иметь равный хеш — иначе коллекция уведёт поиск не в тот бакет и потеряет объект, не дойдя до сравнения.

### Как это работает

#### Баг, ради которого существует весь контракт

Возьмём аккуратный value-класс `Point`: переопределим `equals` так, что два поинта с одинаковыми координатами равны. А `hashCode` «забудем».

```java
final class Point {
    private final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }
    @Override public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point p)) return false;
        return x == p.x && y == p.y;
    }
    // hashCode НЕ переопределён — остался Object.hashCode (identity)
}
```

Проверяем на JDK 21:

```java
Point a = new Point(1, 2);
Point b = new Point(1, 2);
Set<Point> set = new HashSet<>();
set.add(a);
```

Реальный вывод:

```
a.equals(b)               : true
a.hashCode()==b.hashCode(): false
set.contains(b)           : false
```

`b` равен `a` по `equals`, но `set.contains(b)` его не находит. Ничего не упало, исключения нет — просто дедупликация молча не работает. Причина в том, что поиск в хеш-коллекции идёт в две ступени, и на первой тебя уже увело не туда:

Главное из картинки: при разном `hashCode` `equals` вообще не получает шанса. Кажется, что раз `equals` правильный — всё должно работать, но `equals` это _вторая_ ступень поиска, а на первой (выбор бакета) тебя уже увело в пустой бакет.

#### Пять законов `equals`

Javadoc `Object.equals` задаёт пять свойств. Перечислить их мало — для каждого важно знать, _что именно ломается в реальной коллекции_.

**Рефлексивность:** `x.equals(x)` всегда `true`. Сломать трудно, но реально — если в `equals` сравниваешь `double`-поле, а в нём `NaN` (потому что `NaN != NaN`). Ломается `list.contains(x)` для объекта, _который лежит в самом списке_: `contains` внутри делает `x.equals(element)`, и объект себя не находит.

**Симметричность:** `x.equals(y) == y.equals(x)`. Классическое нарушение — подкласс добавляет поле (разберём ниже, это целый раздел). Ломается то, что поведение коллекции начинает зависеть от _порядка_ объектов: `contains` и `remove` дают разный результат в зависимости от того, кто слева.

**Транзитивность:** если `a.equals(b)` и `b.equals(c)`, то `a.equals(c)`. Нарушается «миксующими» реализациями между типами. Ломается детерминизм множеств и мап при ребалансировке.

**Консистентность:** повторные вызовы `x.equals(y)` дают один результат, пока объекты не менялись. Главное нарушение — `equals`, зависящий от _изменяемых_ полей: положил объект в `HashSet`, мутировал поле из `equals`/`hashCode` — и объект «потерялся» в своём же сете, потому что он теперь считается лежащим в другом бакете, чем тот, куда его физически положили.

**`x.equals(null) == false`** для любого ненулевого `x`. Поэтому в `equals` не нужен явный `if (o == null) return false` — проверка `instanceof` уже отсекает `null` (`null instanceof Anything` — это `false`).

> Источник: формулировки — Javadoc `Object.equals(Object)`; разбор последствий — Effective Java, Item 10.

#### Три правила `hashCode` и односторонняя стрелка

Javadoc `Object.hashCode` задаёт три правила:

1. **Консистентность в рамках запуска:** один объект (если поля из `equals` не менялись) при каждом вызове возвращает один `int`.
2. **`equals` равны ⇒ `hashCode` ОБЯЗАН совпасть.** На этом держится всё.
3. **`equals` не равны ⇒ `hashCode` совпадать НЕ обязан** (но желательно для распределения по бакетам).

Ключевой вопрос на собесе — _обратное_ к правилу (2): может ли быть два равных объекта с разным `hashCode`? Нет, это прямое нарушение. А два _неравных_ объекта с одинаковым хешем — пожалуйста, это **коллизия**, легальная и неизбежная вещь (`hashCode` — это `int`, а объектов может быть больше, чем 2³²). Стрелка работает в одну сторону: равенство ⇒ равные хеши; из равных хешей равенство не следует.

Отсюда правило без исключений: **переопределяешь `equals` — переопределяй `hashCode`** (Effective Java, Item 11). Зеркальная ошибка — `hashCode` без `equals` — коллекцию не ломает (контракт формально соблюдён), но делает бесполезной: `equals` остаётся identity-based, два «логически равных» объекта никогда не равны, дедупликации нет.

> Полное внутреннее устройство `HashMap` — бакеты, размешивание хеша, treeification, resize, load factor — вынесено в отдельный бонус-урок. Здесь нам нужен только контрактный механизм: сначала `hashCode` выбирает бакет, потом `equals` сравнивает внутри бакета.

#### `equals` и наследование — место, где нет хорошего ответа

Самый тонкий момент темы и любимая ловушка интервьюеров. Вопрос звучит безобидно: как написать `equals` в иерархии, где подкласс добавляет поле? Канонический пример из Effective Java (Item 10) — `Point` и `ColorPoint`.

**Подход 1 — `instanceof`.** `Point.equals` через `instanceof Point`, `ColorPoint.equals` добавляет сравнение цвета. Проверяем симметричность на JDK 21:

```java
Point p = new Point(1, 2);
ColorPoint cp = new ColorPoint(1, 2, "red");
```

```
p.equals(cp) = true
cp.equals(p) = false
```

Разбор: `p.equals(cp)` вызывает `Point.equals`, `cp instanceof Point` → `true` (`ColorPoint` _является_ `Point`), сравниваются только `x, y` → `true`. `cp.equals(p)` вызывает `ColorPoint.equals`, `p instanceof ColorPoint` → `false` (обычный `Point` не является `ColorPoint`) → `false`. Симметрия сломана.

**Попытка починить — сделать `ColorPoint.equals` снисходительным** к обычному `Point` (сравнивать без цвета, если справа не `ColorPoint`). Чинит симметрию, но ломает транзитивность:

```
red.equals(plain)  = true
plain.equals(blue) = true
red.equals(blue)   = false   ← транзитивность мертва
```

Два разноцветных поинта равны одному бесцветному, но не друг другу.

**Подход 2 — `getClass()` вместо `instanceof`.** `getClass()` возвращает точный рантайм-класс, поэтому `Point` и `ColorPoint` никогда не равны:

```
gp.equals(gcp) = false
gcp.equals(gp) = false   ← симметрия и транзитивность целы
```

Но здесь мы ломаем **принцип подстановки Лисков**: объект подкласса больше не может выступать в роли объекта суперкласса при сравнении. `ColorPoint` в `Set<Point>` перестаёт вести себя как `Point` — `contains` для эквивалентного `Point` вернёт `false`.

Вывод Блоха жёсткий: **нельзя расширить инстанцируемый класс новым value-компонентом, не нарушив контракт `equals`.** Это фундаментальное ограничение. Решения два: **композиция вместо наследования** (Item 18) — `ColorPoint` не наследует, а _содержит_ `Point` как поле; либо **запретить наследование** — сделать value-класс `final` (тогда подклассов нет, `instanceof` безопасен). Именно поэтому в первом примере с `HashSet` я писал `final class Point`.

> [!tip] 💡 Из C#
> та же боль и те же решения. В .NET `record` и рекомендованный паттерн для value-типов тоже тяготеют к запечатыванию (`sealed`) или композиции. Идея «не наследуй value-класс» универсальна для обоих языков.

#### Канонический шаблон `equals`

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;                // 1. быстрый путь: тот же объект
    if (!(o instanceof Point p)) return false; // 2. тип (+ отсекает null, + pattern binding)
    return x == p.x && y == p.y;               // 3. сравнение значимых полей
}
```

Шаг 1 — оптимизация: сравнение ссылок дешевле сравнения полей, а объект часто сравнивают сам с собой. Шаг 2 через `instanceof` с pattern binding (Java 16+, JEP 394) заодно закрывает `equals(null) == false` и убирает отдельный каст. Самые дешёвые поля сравнивай первыми — короткое замыкание `&&` сэкономит на дорогих.

#### `Objects.equals` и `Objects.hash`

`java.util.Objects` (с Java 7) даёт два хелпера. `Objects.equals(a, b)` — null-safe сравнение (оба `null` → `true`, один `null` → `false`, иначе `a.equals(b)`), аналог статического `object.Equals(a, b)` в C#. `Objects.hash(x, y, ...)` — генератор `hashCode` из набора полей:

```java
@Override public int hashCode() { return Objects.hash(x, y); }
```

> [!warning] Подводный камень
> `Objects.hash` принимает `varargs`, то есть на каждый вызов аллоцирует массив `Object[]` и автобоксит примитивы (`int` → `Integer`). Для холодного кода неважно. Для горячего пути (`hashCode` в цикле на миллионах элементов) — заметный мусор в Eden. Тогда пишут руками классический паттерн с множителем 31:

```java
@Override public int hashCode() {
    int result = Integer.hashCode(x);
    result = 31 * result + Integer.hashCode(y);
    return result;
}
```

Почему 31: нечётное простое число (чётное теряло бы биты при переполнении), и `31 * i` JIT умеет свернуть в `(i << 5) - i`. Тот же множитель, что в `String.hashCode` (Effective Java, Item 11).

#### `record` как современный выход

Для чистых data-классов в Java 21 ты часто не пишешь `equals`/`hashCode` руками вообще. `record` (финализирован в Java 16, JEP 395) генерит корректные `equals`, `hashCode`, `toString` из компонентов и **неявно `final`** — наследоваться от него нельзя, значит проблема симметрии из раздела про наследование в принципе не возникает. Детали — глава 17; здесь достаточно держать в голове, что для value-классов это дефолтный инструмент.

> [!example] 🔬 Глубже / на синьора
> **Консистентность и изменяемые ключи.** Самое коварное нарушение. Кладёшь объект в `HashSet`; в момент `add` вычисляется его хеш и выбирается бакет. Потом мутируешь поле, входящее в `hashCode` — но объект физически остаётся в старом бакете, никто его не перекладывает. Теперь `contains` того же объекта считает _новый_ хеш, идёт в _новый_ бакет — и не находит. Объект «потерян» в собственном сете. Практический вывод: ключи `HashMap` и элементы `HashSet` должны быть immutable по полям, участвующим в `equals`/`hashCode`.
> 
> **Константный `hashCode` — корректно, но смертельно для перформанса.** Если написать `return 42`, контракт формально соблюдён (равные объекты дают равный хеш), и коллекция _работает правильно_. Но все элементы валятся в один бакет, и `HashMap` деградирует до линейного поиска — O(n) вместо O(1). Это чёткая граница: контракт — про _корректность_, хорошее распределение — про _скорость_. Одно без другого возможно.
> 
> **`record` и `double`-компонент.** Сгенерированный `equals` использует не `==`, а `Double.compare` (точнее, `Double.doubleToLongBits`-семантику). Проверено на JDK 21:
> 
> ```
> NaN == NaN (примитив)  : false
> record NaN vs NaN      : true
> +0.0 == -0.0 (примитив): true
> record +0.0 vs -0.0    : false
> ```
> 
> То есть в `record` два `NaN` равны, а `+0.0` и `-0.0` — различны. Это ровно противоположно поведению `==` для примитива `double` (был отдельный разговор про IEEE 754). Компилятор делает «правильную для `equals`» вещь: полное отношение эквивалентности, где нет исключений вроде «значение не равно самому себе».

> [!warning] ⚠️ Подводные камни
> 
> 1. **`equals` без `hashCode`** — главный. «Равные» объекты получают разные identity-хеши, уходят в разные бакеты, `contains` промахивается, `equals` не зовётся. Симптомы: дубликаты в `Set`, `get` возвращает `null`, `put` не перезаписывает.
> 2. **`hashCode` без `equals`** — зеркальная ошибка. Коллекция _не_ ломается, но `equals` остаётся identity-based: «логически равные» объекты не схлопываются, дедупликации нет. Корректно, но бесполезно.
> 3. **`equals`/`hashCode` на изменяемых полях** — объект теряется в своём же `HashSet` после мутации (см. 🔬). Ключи хеш-коллекций держи immutable.
> 4. **`Objects.hash` в горячем коде** — varargs-массив + автобоксинг на каждый вызов. В hot path — ручной паттерн с 31.
> 5. **Симметрия через `instanceof` при наследовании** — как только подкласс добавляет value-поле, `x.equals(y) != y.equals(x)`. Лечится композицией или `final`, а не хитрым `equals`.

### На собесе

> [!question]- 1. «Переопределил `equals`, забыл `hashCode`. Что вернёт `set.contains(equalObject)`?» _(очень частый)_
> RU: Почти наверняка `false`. Дефолтный `hashCode` (identity) даёт «равным» объектам разные хеши → разные бакеты → `contains` ищет не в том бакете → `equals` даже не вызывается. Бакет выбирается первым, по хешу; `equals` сравнивает только внутри найденного.
> 
> EN: Almost certainly `false`. The default identity `hashCode` gives "equal" objects different hashes → different buckets → `contains` looks in the wrong bucket → `equals` is never called. The bucket is chosen first by hash; `equals` only compares within the located bucket.
> 
> Follow-up: _«А наоборот — `hashCode` без `equals`?»_ → Коллекция не ломается (контракт соблюдён), но `equals` остаётся identity-based, дедупликации нет. Корректно, но бесполезно.

> [!question]- 2. «Можно ли иметь два равных объекта с разным `hashCode`? А два неравных с одинаковым?»
> RU: Равные с разным хешем — нельзя, прямое нарушение правила 2. Неравные с одинаковым — можно и нормально, это коллизия, неизбежная (хеш — `int`, объектов больше 2³²). Стрелка односторонняя: равенство ⇒ равные хеши, обратное не выводится.
> 
> EN: Equal objects with different hashes — not allowed, a direct violation of rule 2. Unequal objects with the same hash — perfectly fine, that's a collision, and it's inevitable (a hash is an `int`, there can be more than 2³² objects). The implication is one-way: equality ⇒ equal hashes, not the reverse.
> 
> Follow-up: _«Тогда зачем правило про распределение?»_ → `return 42` контракт соблюдает и коллекция работает корректно, но всё валится в один бакет и `HashMap` деградирует до O(n). Распределение — про перформанс, не про корректность.

> [!question]- 3. «Что напечатает?» (`Point` через `instanceof`, `ColorPoint extends Point`)
> 
> ```java
> System.out.println(p.equals(cp) + " " + cp.equals(p));
> ```
> 
> RU: `true false`. `cp` является `Point` → сравниваются `x,y` → `true`. `p` не является `ColorPoint` → `false`. Нарушена симметричность. Починить хитрым `equals` нельзя: снисходительный вариант ломает транзитивность, `getClass` ломает Лисков. Правильно — композиция или `final`-класс (Effective Java Item 10).
> 
> EN: `true false`. `cp` is a `Point`, so only `x,y` are compared → `true`. `p` is not a `ColorPoint` → `false`. Symmetry is broken. You can't fix it with a cleverer `equals`: the lenient variant breaks transitivity, `getClass` breaks Liskov. The right fix is composition or a `final` class (Effective Java Item 10).

> [!question]- 4. «`hashCode`, возвращающий константу — это легально?»
> RU: Да, контракт формально соблюдён — равные объекты дают равный хеш. Коллекция работает _корректно_, но все элементы попадают в один бакет, и `HashMap` вырождается в линейный поиск O(n). Это законно, но убивает производительность. Контракт — про корректность, распределение — про скорость.
> 
> EN: Yes, the contract is formally satisfied — equal objects yield equal hashes. The collection stays _correct_, but every element lands in one bucket and `HashMap` degrades to O(n) linear search. It's legal but kills performance. The contract is about correctness; distribution is about speed.

> [!question]- 5. «`record` генерит `equals` — по `instanceof` или `getClass`?» _(senior add-on)_
> RU: По компонентам через сравнение значений, и `record` неявно `final` — наследоваться нельзя, поэтому проблема симметрии из вопроса 3 не возникает вовсе. Для `double`-компонента используется `Double.compare`, а не `==`: `NaN` равен `NaN`, `+0.0` и `-0.0` различны — противоположно поведению примитива.
> 
> EN: By value comparison of components, and a `record` is implicitly `final` — it can't be subclassed, so the symmetry problem from question 3 can't arise. For a `double` component it uses `Double.compare`, not `==`: `NaN` equals `NaN`, and `+0.0` differs from `-0.0` — the opposite of primitive behavior.
