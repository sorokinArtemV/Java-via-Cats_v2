### Что это и зачем

В уроке 3.9 мы опирались на формулу «`hashCode` выбирает бакет, `equals` сравнивает внутри» как на чёрный ящик — этого хватало, чтобы понять контракт. Здесь мы этот ящик вскрываем: как именно `hashCode` превращается в индекс, что происходит при коллизиях, откуда берётся амортизированная O(1) и почему `HashMap` перестраивает себя на ходу.

Тема стоит усилий по двум причинам. Во-первых, это второй по частоте вопрос про `HashMap` на собесе после самого контракта — им проверяют, понимаешь ли ты, что «O(1)» на самом деле амортизированная и при каких условиях ломается. Во-вторых, это самый резкий контраст с C#: `HashMap` и `Dictionary` снаружи выглядят близнецами, а внутри устроены принципиально по-разному, и это любят спрашивать на позициях, где важен и .NET-бэкграунд.

### Ментальная модель

`HashMap` — это массив бакетов; `hashCode` ключа (после размешивания и битовой маски) выбирает индекс бакета, а внутри бакета элементы лежат цепочкой — с восьми превращающейся в дерево, — по которой ищет `equals`. Амортизированная O(1) держится на хорошем распределении и на удвоении таблицы по мере заполнения.

### Как это работает

#### Индексация: степень двойки, маска, размешивание

`table` — это массив `Node[]`, длина которого всегда степень двойки (по умолчанию 16). Индекс бакета берётся не через деление, а через дешёвую битовую маску:

```
index = (n - 1) & hash
```

Это работает только потому, что `n` — степень двойки: тогда `n - 1` — это маска из младших единичных битов (для 16 это `0b1111`), и `&` оставляет младшие биты хеша. Но здесь и проблема: маска смотрит **только на младшие биты**, а старшие выбрасывает. Если у хешей различаются только старшие биты, они все свалятся в один бакет. Проверим на JDK 21 с хешем `0x7F0000` (младшие 16 бит нулевые):

```
raw hash    = 007f0000
index без размешивания = 0      ← всё в бакет 0
```

Поэтому перед маской `HashMap` **размешивает** хеш, подмешивая старшие биты в младшие (точный исходник JDK 8+):

```java
static int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

```
spread(raw) = 007f007f
index с размешиванием = 15      ← распределение спасено
```

`h >>> 16` — беззнаковый сдвиг, кладущий старшую половину на место младшей; `^` их смешивает. Одна дешёвая операция, а деградация «всё в один бакет» устранена. Кстати, `null`-ключ здесь получает хеш `0` и всегда живёт в бакете 0 — да, `HashMap` разрешает один `null`-ключ (проверено: `get(null)` вернул значение).

#### Коллизии: цепочка, а с восьми — дерево

Коллизии неизбежны — хеш это `int`, а объектов может быть больше. Два ключа с одинаковым индексом бакета складываются в **связный список** (`Node.next`), и поиск внутри бакета идёт перебором, O(n) по длине цепочки. С Java 8 появилась защита от длинных цепочек: если бакет дорастает до 8 элементов **и** таблица не меньше 64, список конвертируется в **красно-чёрное дерево** с поиском O(log n). Проверим рефлексией в реальную таблицу, вставляя ключи с константным `hashCode` (все в один бакет):

```
после 7 вставок : Node     (cap=16)
после 64 вставок: TreeNode (cap=128)
```

Пока таблица маленькая (cap=16), длинный бакет не деревенеет — `HashMap` предпочитает сначала _расширить таблицу_, потому что при малой ёмкости причина длинной цепочки чаще в тесноте, а не во враждебном хеше. И только когда ёмкость дошла до ≥64, а бакет всё ещё ≥8, включается treeification. Вот вся структура целиком:

#### Load factor и resize

За заполненностью следит **load factor** 0.75 (`DEFAULT_LOAD_FACTOR`). Как только `size` превышает `capacity × 0.75`, таблица удваивается, и все элементы раскладываются по новым бакетам. Отсюда и амортизированная O(1): отдельный `put` может влететь в дорогой resize, но эта стоимость размазывается по всем вставкам.

Почему именно 0.75 — компромисс. Низкий load factor (скажем, 0.5) означает много пустых бакетов и меньше коллизий, но лишняя память. Высокий (0.9) экономит память, но удлиняет цепочки и замедляет поиск. 0.75 — эмпирически найденная золотая середина.

### Чем отличается `Dictionary` в C#

Снаружи `HashMap<K,V>` и `Dictionary<TKey,TValue>` — почти одно и то же: положил по ключу, достал по ключу, амортизированно O(1). Внутри — пять существенных расхождений. Начнём с самого фундаментального — как вообще хранится цепочка коллизий:

Пять расхождений, которые стоит держать в голове:

**1. Узел цепочки: объект со ссылкой vs индекс в плоском массиве** (см. диаграмму). В Java каждый элемент бакета — отдельный `Node`-объект в куче, связь строится указателями `node.next`. В C# всё лежит в одном плоском `Entry[]` (где `Entry` — `struct` с полями `hashCode`, `next`, `key`, `value`), а `next` — это `int`-индекс в тот же массив, а не ссылка на объект.

> 💡 **Из C#:** твой `Dictionary` выигрывает cache locality (элементы лежат подряд в памяти) и не аллоцирует объект на каждый узел. Java платит отдельным объектом на узел, но получает узел, живущий независимо от массива. Это тот же trade-off, что «struct в массиве vs объекты по ссылкам» — он тебе знаком.

**2. Treeification — только в Java.** Красно-чёрное дерево при длинном бакете — Java-специфика (защита от hash flooding, JDK-8023463). `Dictionary` так не умеет: цепочка остаётся линейной любой длины.

> 💡 **Из C#:** свою защиту от hash-flooding .NET строит иначе — рандомизацией `string.GetHashCode()`, которая недетерминирована между запусками процесса. Java рандомизацию строк не делает — вместо этого деревенит бакет.

**3. Индексация: степень двойки + маска vs простое число + остаток.** Java берёт размер таблицы степенью двойки и индекс через `(n-1) & hash`, поэтому и нужно размешивание `h ^ (h>>>16)`. C# идёт другим путём: размер таблицы — _простое число_, индекс через `hashCode % size`.

> 💡 **Из C#:** простое число само по себе хорошо перемешивает все биты при взятии остатка, поэтому .NET не нужно отдельное размешивание. Цена — деление дороже битовой маски, но в .NET Core 3.0+ это чинится fast-modulo трюком (предвычисленный множитель вместо деления).

**4. `null`-ключ.** Java `HashMap` разрешает _один_ `null`-ключ (живёт в бакете 0, проверено). C# `Dictionary` на `null`-ключ кидает `ArgumentNullException`. То есть `map.get(null)` в Java — легальная операция, а `dict[null]` в C# — исключение.

**5. Потокобезопасность и Java-7 loop.** Ни `HashMap`, ни `Dictionary` не потокобезопасны. Историческая деталь, которую любят на собесе: в Java 7 `HashMap` при `resize` под конкурентным доступом мог зациклиться навечно (цепочки переставлялись в обратном порядке и образовывали кольцо, поток уходил в 100% CPU). В Java 8 порядок при resize сохраняется, кольцо больше не возникает — но потокобезопасным `HashMap` не стал, для конкурентного доступа есть `ConcurrentHashMap` (глава 24).

### 🔬 Глубже / на синьора

**Почему Java-7 loop исчез в Java 8.** При resize `HashMap` делит каждый старый бакет на два по _одному биту_ хеша: узлы, у которых этот бит `0`, остаются на прежнем индексе (`lo`-список), у которых `1` — уезжают на индекс `+ oldCapacity` (`hi`-список). Хеш каждого узла закеширован в поле `Node.hash`, так что при resize он не пересчитывается, а порядок узлов внутри каждого подсписка сохраняется. Именно сохранение порядка убрало разворот цепочки, из-за которого в Java 7 образовывалось кольцо.

**Как дерево упорядочивает ключи.** Красно-чёрное дерево бакета сортирует узлы по `hash`; при равных хешах, если ключи реализуют `Comparable`, используется их порядок, иначе — тай-брейк по имени класса и identity hash. Поэтому treeification работает даже для ключей без `Comparable`, просто менее эффективно.

**Почему маске нужно размешивание, а остатку — нет.** Взятие остатка по простому числу (C#) вовлекает все биты хеша в результат. Битовая маска (Java) берёт только младшие `log₂(n)` бит, поэтому старшие биты хеша обязаны быть подмешаны заранее — отсюда `h ^ (h >>> 16)`. Это прямое следствие выбора «степень двойки ради дешёвой маски».

### ⚠️ Подводные камни

1. **`HashMap` не потокобезопасен.** Конкурентный `put` во время resize может испортить структуру (в Java 7 — вечный цикл, в Java 8 — потерянные записи). Для конкурентного доступа — `ConcurrentHashMap`, не `Collections.synchronizedMap` на горячем пути.
2. **Мутабельный ключ.** Изменишь поле, входящее в `hashCode`, после вставки — ключ окажется «не в том бакете», и `get` его не найдёт (механика та же, что в уроке 3.9, но теперь ты видишь бакеты). Ключи — immutable.
3. **Плохой `hashCode`.** Константный хеш валит всё в один бакет; с Java 8 treeification спасает до O(log n), но это костыль — нормальный `hashCode` важнее.
4. **Ожидание порядка итерации.** `HashMap` порядок не гарантирует и меняет его при resize. Нужен порядок вставки — `LinkedHashMap`, сортировка — `TreeMap` (глава 13).
5. **Отсутствие `initialCapacity`.** Если знаешь примерный размер — задавай его (`new HashMap<>(expected / 0.75f + 1)`), чтобы избежать череды resize при заполнении.

### На собесе

**1. «Как устроен `HashMap` внутри?»** (очень частый, требует связного рассказа)

RU: `HashMap` — массив бакетов. Индекс считается из `hashCode` ключа: хеш размешивается (`h ^ (h >>> 16)`) и берётся по маске `& (n-1)`, где `n` — размер таблицы, всегда степень двойки. `hashCode` выбирает бакет, а равенство внутри бакета решает `equals` — поэтому они должны быть согласованы. Коллизии складываются в связный список, поиск O(n) по длине бакета; с Java 8 при 8 элементах и таблице ≥64 список превращается в красно-чёрное дерево, O(log n). Load factor 0.75: при превышении таблица удваивается. В среднем `get`/`put` — амортизированно O(1).

EN: `HashMap` is an array of buckets. The index comes from the key's `hashCode`: the hash is spread (`h ^ (h >>> 16)`) and masked with `& (n-1)`, where `n` — the table size — is always a power of two. `hashCode` selects the bucket; equality within the bucket is decided by `equals`, so the two must be consistent. Collisions form a linked list with O(n) lookup per bucket; since Java 8, once a bucket holds 8 elements and the table is at least 64, the list becomes a red-black tree with O(log n). The load factor is 0.75: exceeding it doubles the table. On average `get`/`put` are amortized O(1).

Follow-up-дерево: _«Почему степень двойки?»_ → чтобы индекс считался дешёвой маской `& (n-1)` вместо деления. _«Тогда зачем размешивание?»_ → маска берёт только младшие биты, размешивание подмешивает старшие, иначе хеши, различающиеся старшими битами, коллизируют. _«Когда именно treeify?»_ → 8 в бакете И cap ≥ 64; иначе сначала resize.

**2. «Что такое treeification и когда она срабатывает?»**

RU: Превращение цепочки бакета в красно-чёрное дерево при 8+ элементах и таблице ≥64, дающее O(log n) вместо O(n). Это защита от деградации при плохом или враждебном хеше. При ёмкости <64 вместо деревенения делается resize. Обратно дерево разворачивается в список при ≤6 узлах.

EN: Converting a bucket's chain into a red-black tree when it holds 8+ elements and the table is at least 64, giving O(log n) instead of O(n). It's a defense against degradation from a poor or hostile hash. Below capacity 64 the map resizes instead of treeifying. The tree reverts to a list at 6 or fewer nodes.

**3. «Чем `HashMap` отличается от `Dictionary` в C#?»** (для .NET-бэкграунда)

RU: Четыре ключевых отличия. Узел цепочки — отдельный объект в куче со ссылкой `next` (Java) против плоского `Entry[]` со связью по `int`-индексу (C#, лучше cache locality). Java деревенит длинные бакеты — C# держит линейную цепочку и защищается рандомизацией `GetHashCode`. Индекс: степень двойки + маска + размешивание (Java) против простого числа + остатка (C#). И `null`-ключ: Java разрешает один, C# кидает `ArgumentNullException`.

EN: Four key differences. A chain node is a separate heap object with a `next` reference (Java) versus a flat `Entry[]` linked by `int` index (C#, better cache locality). Java treeifies long buckets — C# keeps a linear chain and defends via randomized `GetHashCode`. Indexing: power-of-two + mask + spread (Java) versus prime size + modulo (C#). And the `null` key: Java allows one, C# throws `ArgumentNullException`.

**4. «Можно ли шарить `HashMap` между потоками?»** (senior add-on)

RU: Нет. `HashMap` не потокобезопасен — конкурентная модификация во время resize портит структуру (в Java 7 — вечный цикл из-за разворота цепочки, в Java 8 — потерянные записи). Для конкурентного доступа — `ConcurrentHashMap`, который лочит по сегментам/бинам, а не целиком.

EN: No. `HashMap` isn't thread-safe — concurrent modification during resize corrupts the structure (an infinite loop from chain reversal in Java 7, lost entries in Java 8). For concurrent access use `ConcurrentHashMap`, which locks per bin rather than the whole map.
