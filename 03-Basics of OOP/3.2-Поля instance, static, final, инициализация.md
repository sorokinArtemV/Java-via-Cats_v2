Поля выглядят самой скучной частью класса — что тут учить после C#. Но в Java у них три собственные механики, каждая из которых стреляет на практике и на собесах. Во-первых, в языке нет properties — поле либо торчит наружу как есть, либо прячется за методами, и от этого зависит дизайн любого API. Во-вторых, у static-полей своя история хранения (PermGen → Metaspace), про которую спрашивают до сих пор. В-третьих, `final` и compile-time константы ведут себя тоньше, чем кажется: `final`-ссылка не защищает объект, а константа может «залипнуть» в чужом байткоде со старым значением. Этот урок разбирает поля как таковые; их инициализация в конструкторе уже разобрана в 3.1, ленивая инициализация классов — впереди в 3.11.

**Ментальная модель.** Instance-поле — своя копия внутри каждого объекта в куче. Static-поле — ровно одно на класс, живёт вместе с классом, а не с объектами. `final` фиксирует содержимое переменной (для ссылки — саму ссылку, не объект за ней). И важное: поля, в отличие от методов, не участвуют в виртуальной диспетчеризации — доступ к полю резолвится в compile-time по типу ссылки.

### Instance vs static

```java
class Counter {
    static int total;          // одно на класс
    final int id;              // своё у каждого объекта
    Counter() { id = ++total; }
}

Counter a = new Counter();
Counter b = new Counter();
Counter c = new Counter();
// a.id=1  b.id=2  c.id=3
// Counter.total=3
```

Три объекта — три независимых `id`, но `total` один: каждый конструктор инкрементировал одно и то же поле. Обращаться к static-полю правильно через имя класса: `Counter.total`.

При этом компилятор _разрешает_ доступ к static через ссылку — и это ловушка читаемости, а не удобство:

```java
a.total = 99;                 // компилируется, но это НЕ поле объекта a
System.out.println(b.total);  // 99 — то же самое одно поле

Counter x = null;
System.out.println(x.total);  // 99 — и никакого NPE!
```

Проверено: `x.total` при `x == null` печатает `99`. Разгадка в ментальной модели: доступ резолвится по _статическому типу_ переменной `x` (это `Counter`), сам объект не нужен и не разыменовывается. Код, который выглядит как чтение поля объекта, объектом даже не интересуется. Вывод: static — только через имя класса; IDE и линтеры не зря подсвечивают обратное.

> [!tip] 💡 Из C#
> концепт идентичен — static-поле как член типа. Но C# строже в синтаксисе: доступ к static через переменную экземпляра (`a.total`) там просто не компилируется (CS0176). Java это разрешает по историческим причинам — отсюда и трюк с `null`, невозможный в C#.

### Где живут поля

Instance-поля — внутри блока объекта в куче, сразу после header (картинка из 3.1). Static-поля живут «с классом», и у этого «с классом» есть версионная история, которую спрашивают постоянно: до Java 8 метаданные классов лежали в PermGen — области фиксированного размера внутри кучи, знаменитой ошибкой `OutOfMemoryError: PermGen space`. **JEP 122** (Java 8) убрал PermGen: метаданные переехали в **Metaspace** — нативную память вне кучи, растущую по умолчанию автоматически.

![[Pasted image 20260703134257.png]]

Формула ответа на mid+: «метаданные класса — в Metaspace с Java 8 (JEP 122), до этого — PermGen внутри кучи». Тонкость, которую придерживают для копающего интервьюера: в HotSpot _значения_ static-полей физически лежат не в Metaspace, а в куче — в объекте-зеркале `java.lang.Class` этого класса (туда они переехали ещё в JDK 7); в Metaspace — именно метаданные: структура класса, байткод методов, constant pool. Для GC это удобно: static-поля — обычные корни достижимости в куче.

> [!tip] 💡 Из C#
> аналог — loader heap CLR, где живут структуры типов; идея «метаданные типа отдельно, экземпляры отдельно» одинаковая. Прямого аналога драмы PermGen→Metaspace в .NET нет — это чисто Java-историческая справка, но её любят как маркер «человек видел OOM в проде».

### `final`-поля: фиксируется ссылка, не объект

`final` на поле означает ровно одно: после инициализации переменной нельзя присвоить другое значение. Для примитива это и есть неизменность значения. Для ссылки — неизменность _ссылки_, объект за ней живёт своей жизнью:

```java
final List<String> tags = new ArrayList<>();
tags.add("java");              // мутация объекта — компилируется и работает
tags.add("jvm");               // [java, jvm]

tags = new ArrayList<>();      // а вот это — нет:
// error: cannot assign a value to final variable tags
```

Поэтому `final`-поле — необходимое, но не достаточное условие иммутабельности класса: если тип поля мутабельный, нужен ещё контроль над тем, кто держит ссылку (защитные копии — глава 3.7).

Разновидности, с которыми надо быть на «ты»:

- **Blank final** — `final`-поле без инициализатора; каждый конструктор обязан присвоить его ровно один раз. Полный разбор с проверенными ошибками компиляции — в 3.1.
- **`static final`** — константа уровня класса. Конвенция имён — `SCREAMING_SNAKE_CASE`: `MAX_RETRIES`, `DEFAULT_TIMEOUT`.
- **Compile-time константа** — особый подвид `static final`: примитив или `String`, инициализированный константным выражением. У неё есть неожиданное поведение с инлайном — это гвоздь 🔬-блока ниже.

> [!tip] 💡 Из C#
> `final`-поле ≈ `readonly` («ссылка, не объект» — та же семантика, тот же вывод про иммутабельность). А пара `static final` vs compile-time константа почти дословно повторяет C#-пару `static readonly` vs `const` — вплоть до одинаковых последствий, см. 🔬.

### Shadowing: параметр затеняет поле

Внутри метода или конструктора локальное имя побеждает имя поля. Классика — параметр конструктора, названный как поле:

```java
class Account {
    String number;
    Account(String number) {
        number = number;      // параметр присвоен сам себе
    }
}

new Account("ACC-1").number   // null
```

Проверено: печатает `null`, без единого предупреждения компилятора — присваивание синтаксически валидно, просто бессмысленно. Поле осталось с дефолтом из зануления (урок 3.1). Лечение — квалификация через `this`:

```java
Account(String number) {
    this.number = number;     // слева поле, справа параметр
}
```

Это и есть главная практическая работа `this` в конструкторах и сеттерах. IDE такой self-assignment подсвечивает, но собес — не IDE.

> [!tip] 💡 Из C#
> правило затенения то же, идиома `this.x = x` та же. Разница только в том, что C#-компилятор на `number = number` выдаёт warning CS1717 из коробки, а `javac` молчит (предупреждение дают только IDE/линтеры).

### Поля не виртуальны: field hiding

В 3.1 мы зафиксировали, что методы в Java виртуальны по умолчанию. Поля — **нет**, и никогда: одноимённое поле в потомке не переопределяет, а _скрывает_ поле родителя, и какой из двух вы читаете — решается по статическому типу ссылки в compile-time:

```java
class Parent {
    String x = "Parent.x";
    String name() { return "Parent.name"; }
}
class Child extends Parent {
    String x = "Child.x";                       // скрывает поле родителя
    @Override String name() { return "Child.name"; }
}

Child c = new Child();
Parent p = c;                  // тот же самый объект

p.x        // "Parent.x"   ← поле: по типу ссылки
c.x        // "Child.x"
p.name()   // "Child.name" ← метод: по реальному объекту
```

Проверено: один объект, но `p.x` и `c.x` — два разных значения, при этом `p.name()` честно уходит в `Child`. Физически в объекте `Child` лежат _оба_ поля — и родительское, и своё; тип ссылки выбирает, к какому смещению обратиться. Практический вывод: одноимённые поля в иерархии — почти всегда ошибка дизайна; поля держат `private` и работают через методы, которые виртуальны.

> [!tip] 💡 Из C#
> поля в C# тоже не виртуальны и тоже скрываются (компилятор потребует `new` для подавления warning'а), так что механика знакомая. Опасность Java-версии в контрасте: методы здесь виртуальны _все и всегда_, и на этом фоне легко по инерции ожидать того же от полей — а они живут по противоположному правилу.

> [!example] 🔬 Глубже / на синьора
> **Compile-time константы инлайнятся — и это видно в байткоде.** Константное выражение из примитивов/`String` компилятор вшивает прямо в место использования (JLS §13.4.9, атрибут `ConstantValue` в class-файле). Реальный `javap` с JDK 21 — контраст двух полей:
> 
> ```java
> public class Limits  { public static final int MAX_USERS = 500; }        // константа
> class Runtime2       { static final int START = (int)(System.nanoTime() % 100); } // нет
> ```
> 
> ```
> static int useConst();          // return Limits.MAX_USERS;
>    0: sipush        500         // ← значение вшито, обращения к Limits НЕТ
>    3: ireturn
> 
> static int useRuntime();        // return Runtime2.START;
>    0: getstatic     #9          // Field Runtime2.START:I  ← честное чтение поля
>    3: ireturn
> ```
> 
> `static final`, инициализированный в рантайме (`nanoTime()`), читается через `getstatic`. Compile-time константа — просто число в чужом байткоде.
> 
> **Следствие №1 — binary compatibility trap.** Прогнал живьём: класс `Client` собран против `MAX_USERS = 100`; меняю константу на 500 и перекомпилирую _только_ `Limits` — `java Client` продолжает печатать `100`. Значение навсегда вшито в `Client.class`, пока его не перекомпилируешь. В момент компиляции JDK 21 свернул даже всю строку целиком: в constant pool клиента лежит готовое `"max = 100, env = PROD"`. Отсюда правило: публичная compile-time константа — это API-контракт «значение не изменится никогда»; всё, что может поменяться (таймауты, лимиты, фичефлаги), — не константа, а конфигурация или хотя бы значение, вычисляемое в рантайме.
> 
> **Следствие №2** ты уже видел в 3.1: чтение такой константы не триггерит инициализацию класса — обращения к классу в байткоде просто нет. Полный разбор ленивой инициализации — в 3.11.
> 
> 💡 **Из C#:** это тот же самый капкан, что у `const` в C# — значение инлайнится в вызывающие сборки, и после замены DLL без перекомпиляции клиентов они видят старое значение; потому и правило «`const` только для математически вечных значений, остальное — `static readonly`». Java-версия правила звучит так же, только имена другие.
> 
> **Зеркало `java.lang.Class`.** Точная формулировка для копающего интервьюера: в HotSpot нативная структура метаданных (`InstanceKlass`) лежит в Metaspace, а значения static-полей — в Java-объекте `java.lang.Class<Counter>` в куче; поля объекта и «поля класса» для GC устроены одинаково — это слоты в heap-объектах. Наружу деталь торчит редко, но объясняет, почему выгрузка класса (и его static-состояния) привязана к сборке его `ClassLoader`'а — тема главы 11.

> [!warning] ⚠️ Подводные камни
> 
> **1. `final` не делает объект неизменяемым.** `final List` защищает только от переприсваивания ссылки; `add`/`remove`/`clear` работают. Иммутабельность = `final`-поля + иммутабельные типы полей или защитные копии (3.7).
> 
> **2. Static через ссылку экземпляра — и через `null`.** `a.total` компилируется, но читает поле класса, а не объекта; `x.total` при `x == null` работает без NPE, потому что объект не разыменовывается. Код лжёт читателю — только `Counter.total`.
> 
> **3. Self-assignment при shadowing.** `number = number` в конструкторе — валидный код, тихо оставляющий поле в дефолте (`null`/`0`). `javac` не предупреждает. Привычка: в конструкторах и сеттерах слева всегда `this.`.
> 
> **4. Field hiding при смене типа ссылки.** Одноимённое поле в потомке не переопределяет родительское: `p.x` и `((Child) p).x` — разные значения одного объекта. Особо коварно вперемешку с методами, которые на том же объекте диспетчеризуются противоположно. Не дублируй имена полей в иерархии; поля — `private`.
> 
> **5. Изменяемое static-поле — глобальная переменная.** Всё, что писали про вред глобального состояния, применимо дословно: скрытая связность, непредсказуемые тесты (состояние течёт между ними), а при многопоточности — гонки, потому что поле одно на всех. Мутабельный static почти всегда просится стать либо instance-полем, либо константой, либо явной зависимостью. Потокобезопасность — главы 23–24.
> 
> **6. «Обновил константу — ничего не изменилось».** Публичная compile-time константа вшита в байткод всех клиентов; замена jar без перекомпиляции зависимых модулей оставляет им старое значение. Симптом на проде выглядит мистикой; причина — JLS §13.4.9.

### На собесе

> [!question]- 1. «Чем instance-поле отличается от static? Где они хранятся?» _(самый частый)_
> 
> _Модельный ответ (RU):_ «Instance-поле — своя копия в каждом объекте, лежит в блоке объекта в куче. Static-поле — одно на класс, живёт с классом: метаданные классов с Java 8 хранятся в Metaspace, нативной памяти вне кучи — JEP 122 убрал PermGen. Обращаться к static — через имя класса.»
> 
> _(EN):_ "An instance field is a per-object copy living inside the object on the heap. A static field exists once per class and lives with the class — class metadata has been stored in Metaspace, native memory outside the heap, since Java 8, when JEP 122 removed PermGen. Static fields should be accessed via the class name."
> 
> _Синьорский добавок:_ «В HotSpot значения static-полей физически лежат в объекте-зеркале `java.lang.Class` в куче; в Metaspace — сами метаданные.»
> 
> _Типичный слабый ответ:_ «static — в Metaspace» без оговорок; если интервьюер копает — стоит различать метаданные и значения полей.

> [!question]- 2. «Что означает `final` на поле? Делает ли оно объект неизменяемым?»
> 
> _Модельный ответ (RU):_ «`final` запрещает переприсваивание после инициализации. Для примитива это неизменное значение; для ссылки — неизменная ссылка, сам объект мутировать можно: `final List` спокойно принимает `add`. Поэтому иммутабельный класс — это `final`-поля _плюс_ иммутабельные типы полей или защитные копии.»
> 
> _(EN):_ "`final` forbids reassignment after initialization. For a primitive that means an immutable value; for a reference it only pins the reference — the object itself can still be mutated: you can `add` to a `final List`. So an immutable class needs `final` fields _plus_ immutable field types or defensive copies."
> 
> _Follow-up:_ «Что такое blank final?» → `final` без инициализатора; каждый конструктор обязан присвоить ровно один раз, иначе ошибка компиляции (разбор — 3.1).

> [!question]- 3. «Поля виртуальны? Что напечатает `p.x`, если поле объявлено и у родителя, и у потомка?»
> 
> _Модельный ответ (RU):_ «Поля не участвуют в виртуальной диспетчеризации — доступ резолвится в compile-time по статическому типу ссылки. Одноимённое поле потомка скрывает родительское (field hiding), в объекте физически живут оба: `p.x` даст родительское, `((Child) p).x` — потомка, при том что `p.method()` на том же объекте уйдёт в потомка. Поэтому поля держат `private`, а полиморфизм строят на методах.»
> 
> _(EN):_ "Fields don't take part in virtual dispatch — access is resolved at compile time by the static type of the reference. A same-named field in a subclass hides the parent's field; the object physically contains both: `p.x` reads the parent's, `((Child) p).x` the child's, while `p.method()` on the same object dispatches to the child. That's why fields stay `private` and polymorphism is built on methods."

> [!question]- 4. «Можно ли обратиться к static-полю через ссылку? Через `null`-ссылку?»
> 
> _Модельный ответ (RU):_ «Компилируется и работает, включая `null`-ссылку — NPE не будет, потому что доступ резолвится по статическому типу переменной, объект не разыменовывается. Но это антипаттерн читаемости: код выглядит как работа с объектом, а трогает состояние класса. Правильно — через имя класса.»
> 
> _(EN):_ "It compiles and works, even through a `null` reference — no NPE, because the access is resolved by the variable's static type and the object is never dereferenced. It's a readability antipattern though: the code looks instance-bound while touching class-level state. Use the class name."

> [!question]- 5. «Чем `static final int X = 100` отличается от `static final int X = compute()`?»
> 
> _Модельный ответ (RU):_ «Первое — compile-time константа: значение инлайнится в байткод всех использующих классов (атрибут `ConstantValue`, JLS §13.4.9), обращения к классу в рантайме нет. Второе — обычное `static final`-поле, читается через `getstatic` при инициализации класса. Два следствия у первой: чтение не триггерит инициализацию класса, и после смены значения нужно перекомпилировать всех клиентов — иначе они видят старое.»
> 
> _(EN):_ "The first is a compile-time constant: its value is inlined into the bytecode of every using class (`ConstantValue` attribute, JLS §13.4.9), so there's no runtime access to the declaring class. The second is a regular `static final` field read via `getstatic`. Two consequences of the first: reading it doesn't trigger class initialization, and changing it requires recompiling all clients — otherwise they keep the stale value."
> 
> _Follow-up-trap:_ «Обновили jar с константой, клиентов не пересобрали — что увидят?» → старое значение, оно вшито в их class-файлы.

> [!question]- 6. «Почему изменяемые static-поля считаются плохой практикой?»
> 
> _Модельный ответ (RU):_ «Это глобальное изменяемое состояние: скрытая связность между модулями, тесты влияют друг на друга через общий класс, а в многопоточном коде поле одно на все потоки — гонки без синхронизации гарантированы. Легитимные static-поля — это константы и stateless-утилиты; изменяемое состояние живёт в объектах и передаётся явно.»
> 
> _(EN):_ "It's global mutable state: hidden coupling between modules, tests leaking state into each other through the shared class, and in concurrent code the single per-class field is shared by all threads — data races unless you synchronize. Legitimate static fields are constants and stateless utilities; mutable state belongs in objects and is passed explicitly."
> 
> _Follow-up:_ «Как сделать static-счётчик потокобезопасным?» → на этом этапе достаточно назвать `AtomicInteger` и отослать к главам 23–24.
