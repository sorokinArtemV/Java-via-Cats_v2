Ты пишешь класс-наследник, переопределяешь метод базового класса — и в C# для этого нужны два ключевых слова: `virtual` на базовом методе и `override` на дочернем. В Java ты садишься сделать то же самое и обнаруживаешь, что никаких ключевых слов не требуется: метод переопределяется просто по факту совпадения сигнатуры, а вызов уходит в версию наследника сам собой. Это не мелкая синтаксическая разница — это противоположная модель по умолчанию, и на ней спотыкается почти каждый переходящий с .NET. Из неё вытекает всё остальное в этом уроке: что `@Override` на самом деле не включает переопределение, почему static-методы ведут себя иначе, и где перегрузка резолвится совсем не там, где ты ждёшь. Разберём методы как таковые; порядок их вызова из конструктора и полная механика наследования — впереди (глава 4 и мастеркласс по инициализации).

**Ментальная модель.** Instance-метод в Java виртуальный по умолчанию: вызов разрешается в рантайме по _фактическому типу объекта_, а не по типу ссылки. Static-метод привязан к классу, не к объекту, `this` у него нет — и потому он не виртуален: одноимённый static в наследнике не переопределяет, а скрывает, и выбор идёт по _типу ссылки_ в compile-time.

### Instance vs static: есть ли `this`

Instance-метод работает с конкретным объектом и имеет доступ к `this` — неявной ссылке на тот объект, на котором метод вызвали. Static-метод привязан к классу; объекта под ним нет, поэтому `this` недоступен физически — не «запрещён стилем», а его неоткуда взять:

```java
class Counter {
    int value;                                   // instance-поле
    void increment() { this.value++; }           // instance: есть this
    static int parse(String s) {                 // static: this нет
        return Integer.parseInt(s);
    }
}
```

`this` в `increment()` — это и есть тот объект, чей `value` инкрементируется. У `parse()` нет «над кем» работать — он получает всё через параметры. Отсюда практическое правило дизайна: метод, которому не нужно состояние объекта, — кандидат в static (утилиты, фабрики, чистые функции над аргументами).

💡 **Из C#:** здесь всё один в один — `static` метод без `this`, instance-метод с ним, та же семантика. Мост прямой, переучиваться нечего. Расхождения начинаются дальше, на виртуальности.

### Virtual by default — отличие №1 от C#

Вот код, который в C# и Java ведёт себя одинаково по результату, но по совершенно разным правилам:

```java
class Animal {
    String sound() { return "..."; }             // никакого ключевого слова
}
class Dog extends Animal {
    @Override String sound() { return "Woof"; }
}

Animal a = new Dog();
a.sound();                                        // "Woof"
```

Проверено на JDK 21: печатается `Woof`. Ссылка имеет тип `Animal`, но вызов ушёл в `Dog.sound()` — диспетчеризация по _реальному типу объекта_, а не по типу переменной. И для этого не понадобилось ни `virtual`, ни `override`: в Java **каждый** non-static, non-final, non-private метод виртуальный автоматически. Это дефолт языка, а не то, что ты включаешь.

Отсюда сразу два следствия, которые надо усвоить как единое целое:

**Чтобы _запретить_ переопределение — ставишь `final`.** В Java ты работаешь не «включателем» виртуальности, а «выключателем»: `final String sound()` означает «этот метод переопределить нельзя», и наследник, попытавшийся это сделать, не скомпилируется.

**`@Override` — не включатель, а проверка.** В отличие от C#-шного `override`, который _делает_ метод переопределяющим, Java-шный `@Override` ничего не включает — переопределение уже произошло по совпадению сигнатуры. Аннотация лишь просит компилятор проверить: «я думаю, что переопределяю метод предка — убедись, что метод с такой сигнатурой там реально есть». И эта проверка спасает от целого класса багов:

```java
class Base { void process(String s) {} }

class Sub extends Base {
    @Override void proces(String s) {}   // опечатка: proces
    // error: method does not override or implement a method from a supertype
}
```

Проверено: с `@Override` опечатка в имени — ошибка компиляции. А вот тот же код _без_ аннотации:

```java
class Sub extends Base {
    void proces(String s) {}   // компилируется молча — это НОВЫЙ метод, не override
}
```

компилируется без единого слова — ты думаешь, что переопределил `process`, а создал новый метод `proces`, и в рантайме будет вызываться родительский `process`. Именно поэтому `@Override` пишут всегда, даже когда «и так работает»: он превращает тихий баг в громкую ошибку компиляции.

💡 **Из C#:** это отличие, на котором палятся чаще всего. В C# метод по умолчанию **невиртуальный**: без `virtual` в базе и `override` в наследнике ты получаешь _method hiding_ (скрытие), и вызов через ссылку базового типа уйдёт в базовую реализацию. В Java — наоборот: виртуальность по умолчанию, `final` чтобы запретить, `@Override` — это не C#-шный `override` (включатель), а C#-шный аналог отсутствует вовсе — ближайшее по духу — это если бы компилятор просто проверял корректность. Держи в голове таблицу соответствий: C# `virtual`+`override` → Java «ничего не пишем»; C# «ничего» (невиртуальный) → Java `final`; C# `override` → Java `@Override` (но с другой ролью — проверка, не включение).

### Method hiding: static не переопределяется

Всё, что сказано про виртуальность, относится к instance-методам. Static-методы **не** виртуальны, и это порождает ловушку, которую обожают на собесах:

```java
class Parent {
    static String who()  { return "Parent.static"; }
    String instWho()     { return "Parent.instance"; }
}
class Child extends Parent {
    static String who()  { return "Child.static"; }        // hiding, НЕ override
    @Override String instWho() { return "Child.instance"; }
}

Parent p = new Child();          // один объект типа Child
p.who()        // "Parent.static"   ← по типу ССЫЛКИ
p.instWho()    // "Child.instance"  ← по типу ОБЪЕКТА
```

Проверено: на одном и том же объекте `p.who()` и `p.instWho()` расходятся. Static-метод выбирается по _статическому типу ссылки_ (`Parent`) в compile-time — это **method hiding** (сокрытие), а не overriding. Instance-метод выбирается по _реальному типу объекта_ (`Child`) в рантайме — это настоящее переопределение. Разгадка в ментальной модели: виртуальная диспетчеризация работает через объект, а у static-метода объекта нет — не по чему диспетчеризовать, остаётся только тип ссылки.

Подтверждение того, что это именно скрытие, а не переопределение: `@Override` на static-методе — ошибка компиляции.

```java
class C extends P {
    @Override static String who() { return "C"; }
    // error: static methods cannot be annotated with @Override
}
```

Компилятор прямым текстом отказывается называть это переопределением. `private`-методы тоже не виртуальны (наследник их не видит, так что переопределить нечего), и `final`-методы переопределить нельзя по определению.

### Перегрузка: разрешается статически, по типу ссылки

Overloading (перегрузка) — несколько методов с одним именем и разными сигнатурами. Ключевое, что надо знать про Java: **какая перегрузка вызовется, решается в compile-time по статическому типу аргумента**, а не по его реальному типу в рантайме. Это прямая противоположность виртуальной диспетчеризации, и на одном объекте два механизма работают в разные стороны:

```java
static void f(Object o) { System.out.println("Object"); }
static void f(String s) { System.out.println("String"); }

Object o = "я строка в рантайме";
f(o);              // "Object"  ← по статическому типу o (Object), не по факту
f("литерал");      // "String"
f((Object) "x");   // "Object"  ← каст меняет статический тип
```

Проверено: `f(o)` печатает `Object`, хотя в `o` лежит `String`. Компилятор видит тип переменной `o` — `Object` — и на этом основании выбирает перегрузку, ещё не зная, что там окажется в рантайме. Overriding смотрит на объект, overloading — на тип ссылки. Практический вывод: не полагайся на перегрузку там, где тип известен только в рантайме, — не сработает так, как кажется.

📝 Терминологический якорь: **overriding** (переопределение) — один метод, разные классы в иерархии, выбор по объекту в рантайме. **Overloading** (перегрузка) — разные методы, один класс/имя, выбор по типам аргументов в compile-time. Их постоянно путают в речи; на собесе чёткое разделение читается как понимание.

💡 **Из C#:** перегрузка резолвится статически и в C# — здесь поведение совпадает. Разница только в контрасте с виртуальностью: в C# ты и так привык, что переопределение требует явных `virtual`/`override`, поэтому «overriding по объекту, overloading по типу» ощущается симметрично. В Java виртуальность — молчаливый дефолт, поэтому легко по инерции ждать, что и перегрузка «умнее», чем есть. Она не умнее.

### varargs — синтаксический сахар над массивом

`varargs` (переменное число аргументов, `int... xs`) позволяет вызвать метод с любым числом аргументов. Под капотом это ровно массив:

```java
static int sum(int... xs) {
    int s = 0;
    for (int x : xs) s += x;
    return s;
}

sum()              // 0  — пустой МАССИВ, не null
sum(1, 2, 3)       // 6
sum(new int[]{4,5})// 9  — можно передать массив напрямую
```

Проверено. Два момента, которые спрашивают. Первый: при вызове без аргументов метод получает _пустой массив_, а не `null` — итерировать безопасно, NPE не будет. Второй — взаимодействие с перегрузкой: точная сигнатура всегда побеждает vararg-версию.

```java
static void f(int a, int b) { System.out.println("exact"); }
static void f(int... xs)    { System.out.println("vararg"); }

f(1, 2)     // "exact"  — точное совпадение приоритетнее
f(1, 2, 3)  // "vararg" — точной нет, идёт vararg
```

Компилятор предпочитает vararg-перегрузку только когда никакая точная не подходит — vararg рассматривается в последнюю очередь при разрешении перегрузки.

💡 **Из C#:** `int... xs` ≈ C# `params int[] xs` — та же идея, тот же массив под капотом, то же правило «точная перегрузка бьёт params». Синтаксис — многоточие после типа вместо ключевого слова `params` перед ним. Семантически эквивалентно.

### Pass-by-value: Java всегда передаёт по значению

Утверждение, которое звучит парадоксально и потому идеально для собеса: **в Java всё передаётся по значению, включая ссылки.** Для объектов по значению копируется _ссылка_ — не объект. Это значит: через параметр можно менять объект (обе ссылки указывают на один), но нельзя заставить исходную переменную указывать на другой объект.

```java
static void reassign(List<String> list) {
    list = new ArrayList<>();     // переприсваиваем ЛОКАЛЬНУЮ копию ссылки
    list.add("NEW");
}
static void mutate(List<String> list) {
    list.add("ADDED");            // мутируем ОБЪЕКТ по ссылке
}

List<String> original = new ArrayList<>(List.of("ORIG"));
reassign(original);   // после: [ORIG]        — не изменился
mutate(original);     // после: [ORIG, ADDED] — изменился
```

Проверено: `reassign` не тронул `original` (`[ORIG]`), `mutate` добавил элемент (`[ORIG, ADDED]`). Объяснение прямо следует из «копируется ссылка по значению»: `reassign` переприсвоил свою локальную копию ссылки — на исходную переменную это не влияет; `mutate` пошёл по копии ссылки к тому же объекту и изменил его. Ссылка — копия, объект — общий.

💡 **Из C#:** по умолчанию C# ведёт себя так же — reference type передаётся копией ссылки, и `reassign` внутри метода не виден снаружи. Разница в том, что у C# есть `ref`/`out`, которые передают саму переменную (тогда `reassign` был бы виден). В Java аналога `ref`/`out` **нет вообще** — передача по значению безальтернативна, «вернуть» изменённую ссылку можно только через `return`. Это частый вопрос на собеседовании: в Java невозможно написать метод, меняющий, на какой объект указывает переданная переменная вызывающего.

### 🔬 Глубже / на синьора

**Байткод диспетчеризации.** Виртуальный вызов компилируется в `invokevirtual`, вызов метода интерфейса — в `invokeinterface`, конструктор и `super`-вызовы — в `invokespecial`. Реальный `javap` с JDK 21:

```
viaVirtual:    invokevirtual  Object.toString
viaInterface:  invokeinterface Speaker.speak
```

`invokevirtual` несёт лишь _символическую_ ссылку на метод; фактический адрес выбирается в рантайме через таблицу методов (vtable-подобную структуру), на которую указывает class pointer в header объекта (см. 3.1). Именно здесь физически живёт «диспетчеризация по реальному типу».

**Тонкость: `invokevirtual` ≠ «всегда виртуально».** Контринтуитивный факт, который проверил на JDK 21: `private`- и `final`-методы тоже компилируются в `invokevirtual` (до Java 11 private шли через `invokespecial` — изменилось с вводом nestmates, JEP 181). Но инструкция здесь не означает динамическую диспетчеризацию: для `final`/`private` целевой метод не может быть переопределён, поэтому JVM резолвит его один раз и статически. Вывод, который стоит проговорить на собесе: «виртуальность метода определяется его _модификаторами_ (`final`/`private`/`static` → невиртуальный), а не только тем, какая байткод-инструкция стоит на месте вызова». JIT вдобавок часто девиртуализирует и инлайнит мономорфные вызовы (когда в точке вызова фактически всегда один тип) — виртуальный вызов может вообще исчезнуть после компиляции.

**Ковариантный возвращаемый тип.** При переопределении разрешено сузить возвращаемый тип до подтипа:

```java
class Producer    { Object  make() { return new Object(); } }
class IntProducer extends Producer {
    @Override Integer make() { return 42; }   // Object -> Integer, ковариантно
}
```

Проверено — компилируется и работает (реальный тип возврата — `Integer`). На уровне байткода это реализовано через _bridge method_: компилятор генерирует синтетический `Object make()`, делегирующий в `Integer make()`, чтобы сохранить сигнатуру родителя для полиморфных вызовов. Та же машинерия bridge-методов всплывёт в генериках (глава 7).

**Правила совместимости при переопределении.** Возвращаемый тип — ковариантен (можно сузить). Модификатор доступа — только _расширять_, сужать нельзя: попытка переопределить `public`-метод как `protected` — ошибка компиляции (`attempting to assign weaker access privileges`), проверено. Проверяемые исключения в `throws` — можно убирать или сужать, но не добавлять новые (детали — глава 6).

### ⚠️ Подводные камни

**1. Забытый `@Override` прячет опечатку.** `void proces(...)` вместо `process` без аннотации компилируется как новый метод; ты уверен, что переопределил, а вызывается родительская версия. Всегда ставь `@Override` — он переводит этот баг в ошибку компиляции.

**2. Static-метод «переопределён».** Одноимённый static в наследнике — это hiding, резолвится по типу ссылки: `Parent p = new Child(); p.who()` даст версию `Parent`. Не полагайся на полиморфизм для static-методов — его там нет.

**3. Перегрузка выбирается по статическому типу.** `f(o)` при `Object o = "str"` уйдёт в `f(Object)`, а не `f(String)` — компилятор смотрит на тип переменной, не на объект. Каст (`(String) o`) меняет выбор перегрузки; будь осторожен с перегрузками по типам одной иерархии.

**4. `null` и разрешение перегрузки.** `f(null)` при наличии `f(String)` и `f(Integer)` не компилируется — «reference to f is ambiguous»: оба одинаково подходят. Лечится явным кастом `f((String) null)`. Особо кусается, когда перегрузок много.

**5. Ожидание `ref`/`out`.** Попытка написать метод, меняющий, на какой объект указывает переменная вызывающего, обречена: Java копирует ссылку по значению. «Вернуть» новую ссылку можно только через `return` или через мутацию поля переданного объекта.

**6. Вызов переопределяемого метода из конструктора.** Уже отмечали в 3.1: из конструктора виртуальный вызов уходит в наследника, чьи поля ещё в дефолтах. С наследованием это даёт знаменитый null-trap — целиком в мастерклассе по инициализации. Правило: из конструктора не зови ничего переопределяемого (Effective Java, Item 19); если нужно — делай метод `private` или `final`.

### На собесе

**1. «Виртуальны ли методы в Java по умолчанию? Как запретить переопределение?»** _(самый частый, разделяющий тех, кто пришёл из C#)_

_Модельный ответ (RU):_ «Да, каждый non-static, non-final, non-private instance-метод виртуален автоматически — вызов диспетчеризуется по реальному типу объекта в рантайме, без всяких ключевых слов. Чтобы запретить переопределение, ставят `final` на метод (или `private`, тогда он не виден наследнику). `@Override` виртуальность не включает — это аннотация-проверка компилятора, что метод с такой сигнатурой действительно есть у предка.»

_(EN):_ "Yes — every non-static, non-final, non-private instance method is virtual by default; the call dispatches on the object's runtime type, no keywords needed. To forbid overriding you mark the method `final` (or `private`, so the subclass doesn't see it). `@Override` doesn't enable anything — it's a compiler-checked annotation asserting that a method with this signature really exists in a supertype."

_Follow-up-trap:_ «Что даёт `@Override`, если и без него работает?» → ловит опечатку сигнатуры/имени на этапе компиляции; без него такой баг тихо создаёт новый метод.

_Типичный слабый ответ:_ «`@Override` делает метод переопределяющим» — калька с C#, неверно: переопределение уже произошло по сигнатуре, аннотация только проверяет.

**2. «Можно ли переопределить static-метод?»**

_Модельный ответ (RU):_ «Нет. Одноимённый static в наследнике — это method hiding (сокрытие), а не overriding. Static не привязан к объекту, у него нет `this`, диспетчеризовать не по чему — выбор идёт по статическому типу ссылки в compile-time. `@Override` на static — ошибка компиляции, компилятор прямо это подтверждает.»

_(EN):_ "No. A same-named static method in a subclass is method hiding, not overriding. A static method isn't bound to an object — no `this`, nothing to dispatch on — so the choice is made by the reference's static type at compile time. `@Override` on a static method is a compile error, which the compiler states outright."

_Follow-up:_ показывают `Parent p = new Child(); p.who()` со static у обоих → версия `Parent`, по типу ссылки. Смени тип ссылки на `Child` — версия `Child`.

**3. «`f(Object)` и `f(String)`; передаём `Object o = "str"`. Что вызовется?»** _(классический трюк на overloading vs overriding)_

_Модельный ответ (RU):_ «`f(Object)`. Перегрузка разрешается статически — по типу переменной, а не по реальному объекту. Компилятор видит тип `o` как `Object` и выбирает соответствующую перегрузку ещё до рантайма. Это противоположность overriding, который смотрит на реальный объект. Чтобы попасть в `f(String)`, нужен статический тип `String` — например, каст.»

_(EN):_ "`f(Object)`. Overload resolution is static — by the variable's declared type, not the runtime object. The compiler sees `o` as `Object` and picks that overload before runtime. It's the opposite of overriding, which looks at the actual object. To hit `f(String)` you need a static type of `String` — e.g. a cast."

_Follow-up-trap:_ «А `f(null)` при `f(String)` и `f(Integer)`?» → не компилируется, ambiguous; лечится кастом.

**4. «Java передаёт аргументы по значению или по ссылке?»** _(любимый парадокс)_

_Модельный ответ (RU):_ «Всегда по значению. Для объектов по значению копируется ссылка — поэтому через параметр можно изменить сам объект (обе ссылки на один), но нельзя изменить, на какой объект указывает переменная вызывающего: переприсваивание параметра внутри метода наружу не видно. Аналога `ref`/`out` из C# в Java нет.»

_(EN):_ "Always by value. For objects it's the reference that's copied by value — so through the parameter you can mutate the object itself (both references point to it), but you can't change which object the caller's variable points to: reassigning the parameter inside the method isn't visible outside. Java has no equivalent of C#'s `ref`/`out`."

_Follow-up:_ «Как тогда "вернуть" новую ссылку?» → только через `return` или мутацию поля переданного объекта.

**5. «Чем overriding отличается от overloading?»** _(проверка терминологии)_

_Модельный ответ (RU):_ «Overriding — переопределение: тот же метод в наследнике, выбор по реальному типу объекта в рантайме, виртуальная диспетчеризация. Overloading — перегрузка: разные методы с одним именем и разными параметрами, выбор по статическим типам аргументов в compile-time. Один смотрит на объект, другой — на типы ссылок; их часто путают, но механизмы противоположны.»

_(EN):_ "Overriding: the same method redefined in a subclass, chosen by the object's runtime type — virtual dispatch. Overloading: different methods sharing a name with different parameters, chosen by the arguments' static types at compile time. One looks at the object, the other at reference types; they're opposite mechanisms though often conflated."

_Синьорский добавок:_ «На уровне байткода overriding живёт в `invokevirtual` + vtable, overloading полностью разрешён компилятором — в байткоде уже стоит конкретная сигнатура.»

**6. «Что такое ковариантный возвращаемый тип?»** _(reach-вопрос, мид+ к синьору)_

_Модельный ответ (RU):_ «При переопределении возвращаемый тип можно сузить до подтипа: родитель возвращает `Object`, наследник — `Integer`. Компилятор для совместимости генерирует bridge-метод — синтетический `Object make()`, делегирующий в `Integer make()`, чтобы полиморфные вызовы через тип родителя продолжали работать.»

_(EN):_ "When overriding, the return type may be narrowed to a subtype: the parent returns `Object`, the child `Integer`. For compatibility the compiler generates a bridge method — a synthetic `Object make()` delegating to `Integer make()` — so polymorphic calls through the parent type keep working."
