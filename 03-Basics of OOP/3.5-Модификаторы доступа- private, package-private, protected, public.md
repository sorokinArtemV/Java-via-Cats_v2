Модификаторы доступа — это граница между тем, что класс обещает наружу, и тем, что оставляет себе. В Java эта граница проведена иначе, чем в C#, в двух местах, где .NET-интуиция подводит систематически. Первое: уровень доступа по умолчанию — когда ты _не написал_ модификатор — в Java не самый закрытый, а промежуточный, видимый целому пакету; забыл написать `private` — и поле открыто половине компонента. Второе: `protected` в Java живёт в двух измерениях сразу (пакет и наследники) и в межпакетном наследовании подчиняется правилу, которое ловит на собесах даже опытных. Разберём все четыре уровня так, чтобы было понятно не «какая клетка в таблице», а _почему_ `protected` капризничает через родительскую ссылку, почему нельзя сузить видимость при переопределении и почему top-level класс не бывает `private`.

**Ментальная модель.** Четыре уровня выстроены в линейную цепочку расширения видимости: `private` (только свой класс) ⊂ package-private (весь пакет) ⊂ `protected` (пакет + подклассы из любых пакетов) ⊂ `public` (отовсюду). Package-private — это отсутствие модификатора, а не отдельное ключевое слово. Проверка доступа целиком компиляционная: в рантайме модификатор — просто флаг в метаданных, ничего не стоит.

### Четыре уровня — карта

Для членов класса (полей, методов, конструкторов, вложенных классов) в Java ровно четыре уровня:

- **`private`** — только внутри своего класса (точнее — внутри своего _nest_, нюанс с вложенными классами в 🔬).
- **package-private** (он же _default_) — класс **плюс весь его пакет**. Ключевого слова нет: это **отсутствие модификатора**.
- **`protected`** — весь пакет (как package-private) **плюс подклассы, включая из других пакетов**.
- **`public`** — отовсюду.

|Откуда обращаемся|`private`|package-private|`protected`|`public`|
|---|---|---|---|---|
|Тот же класс|✓|✓|✓|✓|
|Другой класс, тот же пакет|✗|✓|✓|✓|
|Подкласс в другом пакете|✗|✗|✓ _(с оговоркой)_|✓|
|Неродственный класс, другой пакет|✗|✗|✗|✓|

Оговорка по `protected` (доступ только через ссылку своего типа) — центральная ловушка урока, разберём ниже. По строгости уровни складываются в надмножества: `private` < package-private < `protected` < `public`, каждый следующий видит член из большего числа мест. В спецификации (JLS §6.6.1) package-private называется _default access_; это не «отсутствие уровня», а конкретный уровень — просто без слова.

⚠️ **Главная ловушка для .NET-разработчика — дефолт противоположный.** В C# член без модификатора — `private`, самый закрытый уровень. В Java член без модификатора — **package-private**, видимый всему пакету. Привычка «не написал модификатор → значит инкапсулировал» в Java делает поле доступным всему пакету. Поэтому в Java модификатор полей почти всегда пишут явно.

> [!tip] 💡 Из C#  
> в C# уровней больше, и они завязаны на _сборку_ (assembly): `private`, `protected`, `internal`, `protected internal` (наследник **или** сборка), `private protected` (наследник **и** сборка), `public`. В Java комбинаций «наследник И/ИЛИ единица модульности» нет — нет аналогов `private protected` и `protected internal`. Зато есть package-private, прямого аналога которому в C# нет — и попытка приравнять его к `internal` порождает баги, что разберём сразу.

### package-private — джавоспецифичный уровень, и это не `internal`

C#-инстинкт подсказывает: «package-private — это же `internal`, видимость в пределах единицы сборки». Близко по духу, но именно эта аналогия ломается — единица другая и устроена иначе.

`internal` в C# — видимость в пределах **всей сборки** (одного `.dll`/`.exe`), а сборка может содержать сотни типов из десятков неймспейсов, и `internal`-член виден им всем. package-private в Java — видимость в пределах **одного конкретного пакета**, ровно того, где объявлен класс. И вот что ломает интуицию окончательно, проверено на JDK 21:

java

```java
package com.app.service;
class Helper { static int VALUE = 7; }        // package-private

package com.app.service.internal;             // ПОДпакет
import com.app.service.Helper;
// error: Helper is not public in com.app.service; cannot be accessed from outside package
```

⚠️ **Пакеты в Java не иерархичны по доступу.** `com.app.service.internal` **не** имеет доступа к package-private-членам из `com.app.service`, хотя по имени вложен в него. Вложенность имён (`a.b.c` внутри `a.b`) — про организацию неймспейсов и файловую систему, но **не** про доступ. С точки зрения модификаторов `com.app.service` и `com.app.service.internal` — два независимых, чужих друг другу пакета. Это и есть ключевое расхождение с `internal`: в C# вся сборка — один общий двор; в Java каждый пакет — отдельный закрытый двор, и подпакет тебе не родня.

Зачем package-private нужен на практике:

**Тестовый шов.** Кладёшь тест в тот же пакет (в Maven — тот же пакет, но в `src/test/java`), и он видит package-private-методы и классы без `public`-дыр в API. Идиоматичный Java-приём для тестирования внутренней логики.

**Инкапсуляция пакета как компонента.** Пакет = логический модуль: пара `public`-классов-фасадов, остальное — package-private реализация, которую снаружи не импортировать и не зацепить случайной зависимостью:

java

```java
package com.app.payments;
public class PaymentService {        // фасад — виден всем
    public void charge(long cents) { new RetryPolicy().run(cents); }
}
class RetryPolicy {                  // package-private: снаружи не видно
    void run(long cents) { /* ... */ }
}
```

`RetryPolicy` нельзя импортировать из `com.app.web` — компилятор не видит его как доступный тип. Чисто, без загрязнения `public`.

> [!tip] 💡 Из C#  
> где ты поставил бы `internal` на класс-хелпер, в Java часто ставится package-private (то есть ничего). Но граница разная: `internal`-хелпер виден всей сборке, package-private — только своему пакету. Если «модуль» размазан по нескольким пакетам, package-private его не свяжет — для этого есть JPMS (`module-info.java` с `exports`), добавляющий _модульный_ уровень видимости поверх этих четырёх (см. граничные случаи).

### `protected` — два измерения и любимая ловушка собеса

В C# `protected` простой: член виден классу и наследникам, точка. В Java `protected` даёт доступ по **двум независимым каналам**: (1) весь пакет — как package-private, любой класс пакета видит `protected`-член, даже не наследник; (2) подклассы в любом пакете. И вот во втором канале спрятана оговорка (JLS §6.6.2.1): **подкласс в другом пакете обращается к унаследованному `protected`-члену только через ссылку своего собственного типа (или подтипа), не через ссылку на тип родителя.** Проверенный на JDK 21 разбор всех четырёх форм:

java

```java
// пакет com.app.base
public class Parent { protected int secret = 42; }

// пакет com.app.feature — ДРУГОЙ пакет
public class Child extends Parent {
    void demo(Parent other, Child sibling) {
        this.secret    = 1;   // ✓ свой унаследованный член
        sibling.secret = 2;   // ✓ через ссылку типа Child (свой тип)
        super.secret   = 3;   // ✓ через super — свой канал
        other.secret   = 4;   // ✗ error: secret has protected access in Parent
    }
}
```

Компилятор отверг ровно `other.secret` — доступ через ссылку типа `Parent`. Три остальные формы (`this`, ссылка типа `Child`, `super`) прошли. Один и тот же `protected`-член, одно место в коде — разница только в статическом типе ссылки.

Зачем это ограничение, а не вредность компилятора. `protected` означает «наследник управляет **своим** унаследованным состоянием». Если бы `Child` мог через `Parent`-ссылку лезть в `protected` любого `Parent`-объекта, он трогал бы внутренности **чужих** веток иерархии — например, объекта `OtherChild`, который ему не родственник. Ограничение «только через свой тип» гарантирует: ты копаешься в `protected` лишь у объектов, статически известных как «свои» (твоего типа или подтипа), а не у произвольного родителя.

И тонкость, закрывающая картину: в **том же** пакете этого ограничения нет. Проверено — если `Child` лежит в одном пакете с `Parent`, то `other.secret = 4` компилируется: срабатывает package-канал `protected` (весь пакет видит член напрямую, без оглядки на тип ссылки). Ограничение «только через свой тип» включается ровно при наследовании _между разными пакетами_. Один и тот же код ведёт себя по-разному в зависимости от того, в одном пакете классы или нет.

> [!tip] 💡 Из C#  
> в C# `protected`-член виден наследнику всегда, но есть симметричное правило — обращаться можно только через выражение своего типа или производного (нельзя через ссылку базового типа лезть в `protected` другого объекта). Эта часть логики у языков общая. А вот **пакетного канала** у C# `protected` нет вовсе: C# `protected` ≈ Java `protected` _минус_ «весь пакет». Java-шный `protected` шире.

### Доступ и наследование: сузить нельзя, расширить можно

_Narrowing access_ (сужение видимости) — попытка при переопределении поставить в потомке более строгий модификатор, чем у родителя (`public` → `protected` и т.п.). Компилятор это запрещает, проверено:

java

```java
class Parent { public void run() {} }
class Child extends Parent {
    @Override protected void run() {}
    // error: attempting to assign weaker access privileges; was public
}
```

Расширять можно: package-private-метод родителя потомок вправе сделать `protected` или `public` (проверено — компилируется).

Почему сужать нельзя (JLS §8.4.8.3). Полиморфизм держится на инварианте: если ссылка типа `Parent`, вызов `parent.run()` обязан быть валиден — компилятор проверяет это по типу `Parent`, видит `public run()`, пропускает. В рантайме сработает виртуальная диспетчеризация и вызовется `Child.run()`. Если бы `Child` имел право сделать `run()` приватнее, получилось бы: компилятор разрешил вызов (по типу `Parent` метод `public`), а реальный объект — `Child` с `protected run()`, недоступным из этого места. Дыра в системе типов, нарушение LSP. Поэтому переопределение может только _сохранить или расширить_ видимость.

> [!tip] 💡 Из C#  
> в C# вопрос так остро не стоит — переопределять можно только `virtual`/`abstract`, и модификатор доступа при `override` обязан **совпадать** с базовым, менять его нельзя вообще. В Java все не-`private`/не-`static`/не-`final` методы виртуальны по умолчанию (урок 3.3), и язык разрешает _расширять_ доступ при override, но запрещает _сужать_. Тот же инвариант LSP, достигнут разными средствами.

### Top-level классы: только `public` или package-private

Top-level класс (объявленный прямо в файле, не вложенный) допускает ровно два модификатора, проверено:

java

```java
public class Baz {}       // ✓ имя файла обязано быть Baz.java
class Qux {}              // ✓ package-private top-level — норма
private class Foo {}      // ✗ modifier private not allowed here
protected class Bar {}    // ✗ modifier protected not allowed here
```

Почему `private` и `protected` бессмысленны на top-level. `private` значит «виден только внутри окружающего класса» — но у top-level класса окружающего класса нет, опереться не на что: к такому классу не смог бы обратиться никто. `protected` значит «виден пакету и наследникам» — но для top-level это вырождается ровно в package-private без добавочного смысла. Язык не вводит мёртвые комбинации.

Правило файла: в одном `.java`-файле максимум **один `public` top-level класс**, и имя файла совпадает с его именем; остальные top-level классы в файле — package-private. Для _вложенных_ классов ограничения нет — они члены внешнего класса, поэтому им доступны все четыре модификатора (`private static class`, `protected class` и т.д.) — территория урока 3.12.

### Граничные случаи

**Члены интерфейса неявно `public`.** Проверено на байткоде: в интерфейсе поле — всегда `public static final` (константа), метод без тела — всегда `public abstract`, даже если ты не написал ни одного модификатора:

java

```java
interface Service {
    int MAX = 100;       // javap: ACC_PUBLIC, ACC_STATIC, ACC_FINAL
    void execute();      // javap: ACC_PUBLIC, ACC_ABSTRACT
}
```

Поставить члену интерфейса `protected` или package-private нельзя — компилятор не даст. `private`-методы в интерфейсе появились в Java 9 (часть эволюции `default`-методов) и нужны только для выноса общего кода из `default`/`static`-методов — детали в главе 4.

**Reflection обходит модификаторы.** `field.setAccessible(true)` снимает проверку доступа — можно прочитать/записать `private`-поле. Это тот же механизм, которым Jackson, Spring, JPA лезут в приватные поля. Под модулями (JPMS) это требует «открытия» пакета (`opens`) или флагов JVM, иначе — `InaccessibleObjectException`. Детали рефлексии — глава 12.

**Модули (JPMS).** Поверх четырёх уровней Java 9+ добавляет модульный: даже `public`-класс не виден из другого модуля, если его пакет не `exports`-нут в `module-info.java`. Отдельная тема (глава 8); здесь важно знать, что «`public` ≠ доступен всем», если проект модуляризован.

> [!example] 🔬 Глубже / на синьора  
> **Access flags в байткоде.** Уровень доступа — битовая маска в метаданных члена: `ACC_PUBLIC` (`0x0001`), `ACC_PRIVATE` (`0x0002`), `ACC_PROTECTED` (`0x0004`). Package-private — когда **ни один** из этих битов не выставлен: «отсутствие модификатора» буквально и в байткоде. Смотрится через `javap -p -v`. Отсюда прямое следствие: проверка доступа — компиляционная, в рантайме модификатор ничего не стоит, `private`-метод не «быстрее» `public`.
> 
> **`private` и nestmates — эволюция, релевантная как раз этому уроку.** До Java 11, когда вложенный класс трогал `private`-член внешнего (или наоборот), компилятор тайком генерировал синтетические package-private мостики (`access000‘иподобные)—JVMнеразрешалапрямой‘private‘−доступмеждуразными‘.class‘−файлами,авложенныеклассыкомпилируютсявотдельныефайлы(‘Outer000` и подобные) — JVM не разрешала прямой `private`-доступ между разными `.class`-файлами, а вложенные классы компилируются в отдельные файлы (`Outer 000‘иподобные)—JVMнеразрешалапрямой‘private‘−доступмеждуразными‘.class‘−файлами,авложенныеклассыкомпилируютсявотдельныефайлы(‘OuterInner.class`). С Java 11 (JEP 181, _Nest-Based Access Control_) это убрали: классы одного _nest_ (внешний + все вложенные) перечислены в атрибутах `NestHost`/`NestMembers`, и JVM разрешает прямой `private`-доступ между ними без мостов. Именно поэтому в определении `private` в начале урока стоит «свой nest», а не «свой класс»: с точки зрения JVM единица `private`-доступа — гнездо, а не отдельный class-файл. Увидишь `access$000` в стектрейсе старого кода — теперь знаешь, что это.
> 
> **Рантайм-проверка при линковке.** Основная проверка компиляционная, но при загрузке/линковке JVM проверяет доступ ещё раз — и тут возможен редкий баг раздельной компиляции: класс `A` обращается к `public`-члену `B`, компилируешь вместе (ок), затем подменяешь `B.class` на версию, где член стал package-private, запускаешь со старым `A.class`. Компилятор уже не участвует, а JVM при линковке кинет `IllegalAccessError`. Это `Error` (не `Exception`) — «сломанный мир», несовместимые версии классов. Отличай от `IllegalAccessException` — checked-исключения рефлексии без `setAccessible(true)`, то есть обычной ошибки прикладного кода. Различать `Error` (несовместимость классов) и `Exception` (ошибка кода) любят проверять.

> [!warning] ⚠️ Подводные камни
> 
> **1. Дефолт — package-private, а не `private`.** Забыл модификатор у поля — оно открыто всему пакету, а не инкапсулировано. Прямо противоположно C#. В Java модификатор полей пишут явно почти всегда.
> 
> **2. Подпакет — чужой пакет.** `com.app.core.util` не видит package-private из `com.app.core`. Пакеты не иерархичны по доступу; раскладывать «модуль» по подпакетам в надежде, что вложенность даст доступ, — не сработает.
> 
> **3. `protected` через родительскую ссылку из другого пакета.** `parent.secret` из подкласса в другом пакете — ошибка компиляции; `this`/`super`/ссылка своего типа — ок. В том же пакете ограничение снимается. Один и тот же код компилируется или нет в зависимости от пакета.
> 
> **4. Narrowing при `@Override`.** Сузить видимость при переопределении нельзя (`weaker access privileges`); расширить — можно. Кусается, когда механически «зачищаешь» API до `protected`, не заметив `public`-родителя.
> 
> **5. Члены интерфейса нельзя «приглушить».** `int MAX = 100` в интерфейсе — это `public static final`, а не package-private, как могло бы показаться по отсутствию модификатора. Попытка поставить `protected` члену интерфейса — ошибка компиляции.
> 
> **6. `IllegalAccessError` vs `IllegalAccessException`.** Первое — `Error` при рассинхроне версий раздельно скомпилированных классов (рантайм-линковка). Второе — checked `Exception` при рефлексии без `setAccessible`. Разные вещи, разные корни (`Error` — сломанный мир, `Exception` — ошибка кода).

> _(EN):_ "Four: `private` — own class; no modifier (package-private) — the whole package; `protected` — package plus subclasses from any package; `public` — everywhere. The Java default is package-private, not `private` — the opposite of C#, which is the first gotcha when switching."
> 
> _Follow-up-trap:_ «Значит, поле без модификатора инкапсулировано?» → нет, оно видно всему пакету; в Java модификатор полей пишут явно.

> [!question]- 2. «`internal` в C# и package-private в Java — одно и то же?» _(любят у переходящих с .NET)_  
> _Модельный ответ (RU):_ «Нет. `internal` — видимость в пределах всей сборки; package-private — в пределах одного конкретного пакета. И главное: пакеты в Java не иерархичны по доступу — `com.app.core.util` не видит package-private из `com.app.core`, подпакет это чужой пакет. В сборке C# все типы "свои", в Java каждый пакет — отдельный закрытый двор.»
> 
> _(EN):_ "No. `internal` is assembly-wide visibility; package-private is limited to one specific package. And crucially, Java packages aren't hierarchical for access — `com.app.core.util` can't see package-private members of `com.app.core`; a subpackage is a foreign package. In a C# assembly everything is 'in-house'; in Java each package is a separate closed yard."
> 
> _Follow-up:_ «Как расшарить package-private между пакетами одного модуля?» → напрямую никак: либо один пакет, либо сделать `public` и контролировать видимость через JPMS `exports`.

> [!question]- 3. «Скомпилируется? `Child` в пакете `b`, `Parent` в `a`, поле `protected int x`; внутри `Child`: `void f(Parent p) { p.x = 5; }`» _(коронная ловушка `protected`)_  
> _Модельный ответ (RU):_ «Нет. `protected`-член из подкласса в другом пакете доступен только через ссылку своего типа (`Child`/подтип), а не через ссылку типа `Parent` (JLS §6.6.2.1). Иначе `Child` через `Parent`-ссылку трогал бы `protected` чужих веток иерархии. Через `this`, `super`, `((Child) p).x` — скомпилируется.»
> 
> _(EN):_ "No. A `protected` member accessed from a subclass in a different package is reachable only through a reference of the subclass's own type (`Child` or subtype), not through a `Parent` reference (JLS §6.6.2.1). Otherwise `Child` could touch the `protected` state of unrelated branches of the hierarchy via a `Parent` reference. Through `this`, `super`, `((Child) p).x` — it compiles."
> 
> _Follow-up:_ «А если `Child` и `Parent` в одном пакете?» → скомпилируется, срабатывает package-канал, ограничение по типу не применяется.

> [!question]- 4. «Можно ли при `@Override` сузить видимость?»  
> _Модельный ответ (RU):_ «Нет, сужать запрещено (`weaker access privileges`), расширять — можно (JLS §8.4.8.3). Причина — LSP: полиморфный вызов через тип родителя валидируется по родителю; если бы потомок сделал метод приватнее, в рантайме вызвался бы недоступный метод — дыра в типобезопасности.»
> 
> _(EN):_ "No — narrowing is forbidden (`weaker access privileges`), widening is allowed (JLS §8.4.8.3). The reason is LSP: a polymorphic call through the parent type is validated against the parent; if the subclass made the method more private, the runtime would dispatch to an inaccessible method — a type-safety hole."
> 
> _Follow-up-trap:_ «А return type и checked-исключения?» → return type можно _сузить_ (ковариантный возврат, урок 3.3); список checked-исключений можно только _сузить или убрать_, расширять нельзя (глава 6). Для исключений правило обратное доступу — и по той же логике «вызывающий по типу родителя не должен получить сюрприз».

> [!question]- 5. «Почему top-level класс не может быть `private`?»  
> _Модельный ответ (RU):_ «`private` значит "виден только внутри окружающего класса", а у top-level класса окружающего класса нет — опереться не на что, к нему не смог бы обратиться никто. Поэтому top-level допускает только `public` или package-private; `protected` тоже запрещён — вырождается в package-private без смысла. Вложенный класс — может быть любым из четырёх.»
> 
> _(EN):_ "`private` means 'visible only within the enclosing class', and a top-level class has no enclosing class — nothing to anchor to, no one could reach it. So top-level allows only `public` or package-private; `protected` is also forbidden — it degenerates into package-private with no added meaning. A nested class, however, can take any of the four."
> 
> _Follow-up:_ «Сколько `public`-классов в файле?» → максимум один, имя файла = его имя.

> [!question]- 6. «Влияет ли модификатор на производительность? Возможна ли ошибка доступа в рантайме?» _(reach-вопрос, различение `Error`/`Exception`)_  
> _Модельный ответ (RU):_ «На производительность — нет: проверка компиляционная, в байткоде это флаг, `private`-вызов не быстрее `public`. Но рантайм-ошибка возможна: `IllegalAccessError` — при рассинхроне версий раздельно скомпилированных классов (член был `public`, стал `private`, старый вызывающий не перекомпилирован); это `Error`, сломанный мир. И `IllegalAccessException` — checked-исключение рефлексии без `setAccessible(true)`. Первое — несовместимость классов, второе — ошибка кода.»
> 
> _(EN):_ "Performance — no: the check is at compile time, in bytecode it's just a flag, a `private` call is no faster than a `public` one. But a runtime failure is possible: `IllegalAccessError` — from version skew between separately compiled classes (a member was `public`, became `private`, the old caller wasn't recompiled); that's an `Error`, a broken world. And `IllegalAccessException` — a checked exception from reflection without `setAccessible(true)`. The first is class incompatibility, the second a code error."
