В уроке 1 виртуальность была фактом: «решает реальный тип объекта». Хороший ответ на собесе — но за ним сразу летит «а _как_ JVM это делает?». Здесь мы вскрываем механизм: куда физически уходит вызов. И, что не менее важно, из этого механизма станет видно всё, что виртуальностью **не** является — `super`, поля, `static`, `private`, — потому что каждое из них ломает интуицию ровно там, где ты ждёшь полиморфизма.

> [!info] Ментальная модель  
> У каждого объекта есть служебный указатель на его класс; у класса — таблица методов (`vtable`). Обычный вызов (`invokevirtual`) идёт по таблице **реального объекта**, тип ссылки игнорируется. Виртуальны только instance-методы: **поля и `static` резолвятся по типу ссылки на компиляции** — это сокрытие (hiding), а не переопределение.

### Таблица методов — куда уходит вызов

Прежде чем утверждать на словах, я прогнал байткод через `javap`. Обычный вызов `p.process()` компилируется в инструкцию `invokevirtual` — она и делает «прогулку» по таблице. Механизм такой: у объекта в heap есть слот `class →`, указывающий на его класс. У класса — массив-таблица, где каждому виртуальному методу соответствует слот с адресом реальной реализации. Когда `CardPayment` переопределяет `process()`, он **перезаписывает этот слот** в своей таблице; унаследованные методы (`toString`, `hashCode`) остаются с адресами родителя.

![[Pasted image 20260707112941.png]]

Тонкость из байткода: в строке `invokevirtual #7 // Method Payment.process` стоит `Payment.process` — но это лишь _символическая ссылка_ для проверки типов на компиляции. В рантайме `invokevirtual` всё равно прыгает по таблице реального объекта.

### `invokevirtual` vs `invokespecial` — почему `super` обходит виртуальность

А вот `super.process()` компилируется в **другую** инструкцию — `invokespecial`, которая зовёт **ровно указанную** реализацию, без поиска по таблице. Цель известна на компиляции. Вот что выдал `javap` на реальном коде:

```
// CardPayment.process(), тело = super.process():
1: invokespecial #7    // Method Payment.process:()V   ← цель зафиксирована

// CardPayment.demo(), где Payment p = this; p.process():
3: invokevirtual #7    // Method Payment.process:()V   ← диспатч по vtable
6: invokestatic  #10   // Method audit:()V             ← static, без объекта

// конструктор:
1: invokespecial #1    // Method Payment."<init>":()V   ← super() тоже invokespecial
```

`super.method()` — единственный легальный способ позвать родительскую версию переопределённого метода в обход виртуальности (JVMS §6.5, `invokespecial`). Конструкторы (`<init>`) — тоже `invokespecial`: `super()` в наследнике компилируется в `invokespecial Parent.<init>`. Это пригодится в ctor-ловушке ниже.

> [!tip] Из C#  
> `super` ≈ `base`. Семантика та же — невиртуальный вызов конкретной родительской версии. В C# это опкод `call`, в Java `invokespecial`, но идея идентична: «зови ровно родителя, не диспатчи».

> [!warning] ⚠️ Java 11+ (JEP 181, nestmates)  
> Исторически `private`-методы тоже звались через `invokespecial`. С Java 11 `javac` для вызова `private`-метода внутри класса генерит `invokevirtual` — на моём коде `this.log0()` дал именно `invokevirtual #21 // Method log0`. Семантика не изменилась: `private` по-прежнему **не виртуален** (в таблицу подтипа он не попадает, переопределить нельзя). Поменялся только опкод. Если на собесе спросят «каким опкодом зовётся `private`» — честный ответ: «зависит от версии: до 11 `invokespecial`, с 11 `invokevirtual` из-за nestmates, но полиморфным он всё равно не становится».

### Асимметрия: методы полиморфны, поля и `static` — нет

Виртуальны только instance-методы. Поля и `static`-методы резолвятся по типу ссылки на компиляции. Смотри на **один и тот же объект** через две ссылки разного типа (вывод — с прогона):

```java
Child c = new Child();
Parent p = c;                    // ТА ЖЕ ссылка на ТОТ ЖЕ объект
System.out.println(p.label);     // parent-field   ← по типу ссылки Parent
System.out.println(c.label);     // child-field    ← по типу ссылки Child
System.out.println(p.who());     // Child.who       ← по реальному объекту
p.greet();                       // Parent.greet    ← static по типу ссылки
```

```java
class Parent {
    String label = "parent-field";
    String who() { return "Parent.who"; }
    static void greet() { System.out.println("Parent.greet"); }
}
class Child extends Parent {
    String label = "child-field";          // СКРЫВАЕТ поле, не переопределяет
    @Override String who() { return "Child.who"; }
    static void greet() { System.out.println("Child.greet"); }
}
```

Один объект, одно имя поля `label` — а значение зависит от типа ссылки, через которую смотришь. Обращение к полю компилятор привязывает по статическому типу намертво (`getfield Parent.label` vs `getfield Child.label` — две разные инструкции). Метод `who()` при этом честно едет по реальному объекту. Вот она, формула: **методы смотрят на объект, поля и `static` — на тип ссылки.** Это называется **hiding** (сокрытие), в противовес **overriding**.

> [!tip] Из C#  
> Оба механизма у тебя есть. Сокрытие поля/метода ≈ C#-шный `new` (method hiding) — но в C# ты обязан написать `new`, иначе предупреждение; в Java сокрытие происходит молча. Виртуальный `who()` ≈ твой `virtual`/`override`. Разница в дефолте: чтобы получить C#-шное `new`-поведение для метода в Java — сделай его `private`/`static`/`final` (тогда не виртуален).

### Ковариантные return types

Переопределяющий метод может вернуть **подтип** того, что возвращал родитель — удобно для фабрик и репозиториев, наследник отдаёт конкретику без каста (проверено прогоном — реальный тип `SqlAccount`):

```java
class AccountRepo     { Account find()    { return new Account(); } }
class SqlAccountRepo extends AccountRepo {
    @Override SqlAccount find() { return new SqlAccount(); }   // SqlAccount <: Account
}
```

Появилось в Java 5 (до этого сигнатуры должны были совпадать байт-в-байт). C# умеет ковариантные return с C# 9 — так что для тебя это не новость, просто отметь, что в Java это с 2004 года.

### Правила переопределения — контракт JLS §8.4.8

Когда ты пишешь `@Override`, компилятор проверяет три правила. Их спрашивают, поэтому — наизусть:

|Можно|Нельзя|
|---|---|
|расширить видимость (`protected` → `public`)|сузить видимость (`public` → `protected`)|
|убрать/сузить checked-исключения|добавить/расширить checked-исключения|
|вернуть подтип (ковариантный return)|вернуть несовместимый тип|

Оба «нельзя» я проверил — компилятор валится явно:

```
// public void run()  →  protected void run():
error: run() in Sub cannot override run() in Base
  attempting to assign weaker access privileges; was public

// void run()  →  void run() throws IOException:
error: run() in Sub2 cannot override run() in Base2
  overridden method does not throw IOException
```

Логика обоих запретов — **подстановка Лисков** на уровне языка: код, работавший с базовым типом через базовую ссылку, должен работать и с наследником. Сузишь видимость — вызывающий через `Base`-ссылку внезапно не сможет позвать метод. Расширишь checked-исключения — вызывающий не готов их ловить, хотя компилятор его не обязывал. (Unchecked — `RuntimeException` и наследники — добавлять можно свободно, контракт их не покрывает.)

И финальное, что замыкает механику: `private`, `static`, `final` методы не переопределяются. `private` не наследуется → одноимённый метод в наследнике независим. `static` скрывается. `final` запрещён компилятором. Это не три правила, а три способа сказать «этот метод не попадёт в таблицу как override».

> [!example] 🔬 Глубже / на синьора  
> **Интерфейсы диспатчатся иначе.** Для обычного instance-метода — `invokevirtual` по таблице класса, где слот метода — фиксированный индекс (одиночное наследование классов это гарантирует). Но для вызова через ссылку интерфейса генерится `invokeinterface` и используется отдельная структура (`itable`): один класс реализует много интерфейсов, единого фиксированного индекса слота нет, поиск чуть дороже. Глубоко это — урок про интерфейсы, но знать, что опкод другой, надо.
> 
> **Почему `final int v = 99` печатает `99`, а не `0` (см. ctor-ловушку ниже).** Instance-`final` поле с **константным** инициализатором — это «constant variable» (JLS §4.12.4). Компилятор инлайнит литерал прямо в тело метода: `javap` на `val()` показал `bipush 99; ireturn` — никакого `getfield`. Порядок инициализации перестаёт играть роль, потому что читать поле уже не нужно. Убери `final` или сделай инициализатор не-константным (`final int v = compute()`) — вернётся `getfield` и `0`.

Теперь — граничные случаи, каждый прогнан.

> [!warning] ⚠️ Подводный камень 1 — виртуальный вызов из конструктора видит `null`  
> Если конструктор зовёт метод, переопределённый в наследнике, метод выполнится в версии наследника — но поля наследника ещё в дефолтах (`null`/`0`/`false`), потому что конструктор родителя отрабатывает **раньше** инициализаторов полей наследника (JLS §12.5). Прогон:
> 
> java
> 
> ```java
> class Parent {
>     Parent() { System.out.println(describe()); }   // зовёт виртуальный метод
>     String describe() { return "Parent default"; }
> }
> class Child extends Parent {
>     private String name = "Bob";                    // присвоится ПОСЛЕ super()
>     @Override String describe() { return "name = " + name; }
> }
> new Child();   // печатает: name = null
> ```
> 
> `super()` (конструктор `Parent`) отрабатывает целиком до присвоения `name = "Bob"`. Внутри `Parent()` вызов `describe()` виртуален → уходит в `Child.describe()`, но поле ещё пустое. Метод уже в таблице — поля ещё нет.

Вот этот порядок на таймлайне:

![[Pasted image 20260707112914.png]]

Чинится: не звать переопределяемые методы из конструктора — только `private`/`static`/`final`, они не виртуальны и безопасны.

> [!tip] Из C#  
> Ровно та же ловушка и по той же причине: в C# базовый конструктор тоже зовётся до инициализаторов полей наследника, и виртуальный вызов из конструктора попадёт в `override` с неинициализированными полями. Это кросс-языковой грех ООП. Effective Java, Item 19: из конструктора не зови методы, которые можно переопределить.

> [!warning] ⚠️ Подводный камень 2 — `final int v = 99` в той же ловушке печатает `99`, а не `0`  
> Тот же конструктор, но поле теперь `final` с константой — и вывод переворачивается (прогон подтвердил все три):
> 
> java
> 
> ```java
> class WithFinalConst extends ParentInt {   // ParentInt() зовёт val()
>     final int v = 99;                       // константа → инлайнится
>     @Override int val() { return v; }
> }
> class WithFinalCalc extends ParentInt {
>     final int v = compute();                // final, но НЕ константа
>     @Override int val() { return v; }
>     static int compute() { return 99; }
> }
> // new WithFinalConst() → v = 99   (compile-time константа, bipush 99 в байткоде)
> // new WithFinalCalc()  → v = 0    (getfield, поле ещё не присвоено)
> ```
> 
> Механизм — в 🔬 выше: `final int v = 99` не читается через `getfield`, литерал `99` зашит прямо в тело `val()`, поэтому порядок инициализации не важен. Стоит `compute()` — и это уже обычное поле, читается через `getfield`, видит дефолт `0`.

> [!warning] ⚠️ Подводный камень 3 — «переопределение» `private` это новый независимый метод
> 
> java
> 
> ```java
> class Animal {
>     private String sound() { return "generic"; }     // private → не наследуется
>     void describe() { System.out.println(sound()); }  // зовёт СВОЙ sound()
> }
> class Dog extends Animal {
>     private String sound() { return "woof"; }         // НЕ override — параллельный метод
> }
> new Dog().describe();   // печатает generic, а не woof
> ```
> 
> `Animal.describe()` зовёт `sound()` — и поскольку `private` не виртуален, вызов жёстко привязан к `Animal.sound()`, даже если объект — `Dog`. `Dog.sound()` существует параллельно и из `describe()` недостижим. `@Override` на нём не скомпилировался бы — прямое следствие правила «`private` не попадает в таблицу как override».

> [!warning] ⚠️ Подводный камень 4 — `super.super` не существует
> 
> java
> 
> ```java
> class C extends B {
>     @Override String m() { return super.super.m(); }   // не компилируется
> }
> // error: <identifier> expected  —  super не может стоять после точки
> ```
> 
> `super.m()` уходит ровно в непосредственного родителя (это `invokespecial` с зафиксированной целью). Прыжок через уровень запрещён by design: он позволил бы наследнику ломать инкапсуляцию промежуточного класса, который переопределил `m()` именно чтобы навязать своё поведение. Цепочка `L3 → L2 → L1` (прогон: `L3->L2->L1`) — каждый уровень зовёт строго следующий. Если тебе нужен `super.super`, иерархия спроектирована неверно — рефактори в композицию.

### На собесе

> [!question]- Q1. «Как JVM понимает, какой метод звать для `p.process()`? А если `p` — ссылка интерфейса?»  
> **RU:** Для обычного instance-метода — `invokevirtual`: по объекту находим его класс, в таблице методов класса берём слот → реальная реализация. Тип ссылки не участвует, слот переопределённого метода перезаписан на версию наследника. Для ссылки интерфейса — `invokeinterface` и отдельная структура (`itable`): класс реализует много интерфейсов, единого фиксированного индекса нет, поиск чуть дороже.  
> **EN:** For a normal instance method it's `invokevirtual`: from the object we reach its class, look up the slot in the class's method table → the real implementation. The reference type is irrelevant; the overridden slot points to the subclass version. Through an interface reference it's `invokeinterface` with a separate structure (the `itable`), since a class implements many interfaces and there's no single fixed slot index — lookup costs a bit more.  
> **Follow-up (trap):** «А `super`?» → `invokespecial`, единственный невиртуальный вызов instance-метода, цель зашита на компиляции, обходит таблицу.

> [!question]- Q2. «Ты переопределяешь метод. Что запрещено менять в сигнатуре и почему?»  
> **RU:** Нельзя сузить видимость, нельзя добавить/расширить checked-исключения, нельзя вернуть несовместимый тип (подтип — можно, ковариантно). Причина одна — подстановка Лисков: код, работающий через базовую ссылку, должен работать и с наследником. Сузил `public`→`protected` — вызывающий через `Base` не позовёт метод; расширил checked — вызывающий не ловит то, к чему его не обязали. Режется на компиляции (JLS §8.4.8).  
> **EN:** You can't narrow visibility, can't add or broaden checked exceptions, can't return an incompatible type (a subtype is fine — covariant return). The reason is Liskov substitution: code working through a base reference must keep working with a subclass. Narrowing `public`→`protected` breaks callers through `Base`; broadening checked exceptions forces callers to handle what they weren't required to. The compiler rejects both (JLS §8.4.8).  
> **Follow-up (trap):** «А `RuntimeException` добавить можно?» → Да, без ограничений — unchecked вне контракта компилятора. Именно поэтому многие API оборачивают checked в unchecked при переопределении, чтобы влезть в узкую сигнатуру родителя.

> [!question]- Q3. «Конструктор печатает `null`, хотя поле инициализировано. Объясни и почини.»  
> **RU:** Конструктор зовёт переопределяемый метод; `super()` отрабатывает до инициализаторов полей наследника (JLS §12.5), поэтому виртуальный вызов попадает в `Child`-версию с ещё пустыми полями. Метод в таблице с рождения, поля — нет. Чинится: не звать переопределяемые методы из конструктора (Item 19); если нужно — делать их `private`/`final`.  
> **EN:** The constructor calls an overridable method; `super()` runs before the subclass field initializers (JLS §12.5), so the virtual call lands in the `Child` version with fields still at defaults. The method is in the table from birth, the fields aren't yet. Fix: don't call overridable methods from a constructor (Item 19); if you must, make them `private`/`final`.  
> **Follow-up (trap):** «А если поле `final int v = 99` — что напечатает?» → `99`, не `0`. `final` с константным инициализатором — compile-time константа, компилятор инлайнит литерал прямо в байткод (`bipush 99`), порядок инициализации перестаёт играть роль. Сделай инициализатор не-константным (`final int v = compute()`) — вернётся `0`.

> [!question]- Q4. «Поля полиморфны? `Parent p = new Child(); p.label` — что вернёт?»  
> **RU:** Нет. Полиморфны только instance-методы. Поля и `static` резолвятся по типу ссылки на компиляции — это сокрытие (hiding), не переопределение. `p.label` вернёт `Parent`-поле (по типу ссылки), а `p.who()` — `Child`-версию (по объекту). Формула: методы смотрят на объект, поля и `static` — на ссылку.  
> **EN:** No. Only instance methods are polymorphic. Fields and `static` resolve by the reference type at compile time — that's hiding, not overriding. `p.label` returns the `Parent` field (by reference type), while `p.who()` returns the `Child` version (by object). Rule of thumb: methods look at the object, fields and `static` look at the reference.  
> **Follow-up:** «`super.super.m()` — дотянешься до дедушки?» → Нет, `super.super` не существует (by design). Легально — только рефактор в композицию.

#### На собесе — устная шпаргалка

Механику диспатча проговаривай так: «у объекта — указатель на класс, у класса — таблица методов; `invokevirtual` идёт по таблице **реального объекта**, тип ссылки игнорируется». Слот переопределённого метода перезаписан на версию наследника; унаследованные указывают на родителя.

`super.m()` — это `invokespecial`, невиртуальный вызов конкретного родителя, цель зашита на компиляции; `super.super` не существует. Конструкторы — тоже `invokespecial`.

Асимметрия одной фразой: **методы смотрят на объект, поля и `static` — на тип ссылки.** Отсюда field/static hiding — сокрытие на компиляции, не переопределение.

Три правила переопределения: видимость нельзя сузить, checked нельзя расширить, return ковариантен. Обоснование — Лисков.

Две ловушки держи наготове: (1) виртуальный вызов из конструктора видит дефолтные поля, потому что `super()` раньше инициализаторов — **кроме** `final`-константы, которую инлайнит компилятор (`bipush`); (2) `private`/`static`/`final` не попадают в таблицу как override — «переопределение» `private` даёт независимый параллельный метод.

Если спросят про перформанс — ответ из урока 1: HotSpot девиртуализирует мономорфные call sites через CHA, `final` ради семантики, не скорости.
