Представь, что моделируешь `Money` в fintech-домене. Хочешь, чтобы объект умел сравниваться (для сортировки), сериализоваться, может — печататься по-своему. В C# ты бы влепил `: IComparable, ISerializable` и, возможно, ещё унаследовался от какого-нибудь `ValueObject`. В Java — то же самое, но ты моментально упрёшься в жёсткую асимметрию: после `implements` можно перечислить сколько угодно интерфейсов, а `extends` у класса — ровно один. Эта асимметрия — не каприз синтаксиса, а несущая стена всего блока про интерфейсы: как только увидишь, _почему_ она там, из неё бесплатно вываливается и разница «abstract vs interface», и весь diamond problem.

> **Ментальная модель.** Интерфейс — это контракт способностей («can-do»): список методов, которые тип _обещает_ уметь, без состояния и без конструктора. Именно поэтому реализовать можно сколько угодно интерфейсов, а унаследовать — ровно один класс: класс несёт состояние, а два набора полей в одном объекте неразрешимы.

### Интерфейс — это контракт

Коротко: интерфейс — список способностей, которые тип обязуется предоставить. Никакой реализации (классически) и никакого состояния. Чистый контракт: «кто реализует меня — обязан уметь вот эти методы».

```java
public interface Discountable {
    BigDecimal applyDiscount(BigDecimal price);        // неявно public abstract
    BigDecimal MAX_DISCOUNT = new BigDecimal("0.5");   // неявно public static final
}

public class SeasonalOffer implements Discountable {
    @Override
    public BigDecimal applyDiscount(BigDecimal price) {
        return price.multiply(new BigDecimal("0.8"));
    }
}
```

Два неочевидных момента про **неявные модификаторы**:

- Методы интерфейса (до Java 8) — неявно `public abstract`. Слова можно не писать, компилятор подставит. Поэтому реализация обязана быть `public` — сузить видимость нельзя. Это ровно то правило переопределения из прошлого урока: видимость не сужается. Проверил — попытка объявить метод-реализацию без `public` не компилируется:

> [!example] Прогон на JDK 21 — сужение видимости
> 
> java
> 
> ```java
> interface Discountable { void apply(); }
> class Offer implements Discountable {
>     @Override void apply(){}   // не public → сужение
> }
> ```
> 
> ```
> error: apply() in Offer cannot implement apply() in Discountable
>   attempting to assign weaker access privileges; was public
> ```

- Поля интерфейса — неявно `public static final`, то есть это _константы_, а не состояние объекта. `BigDecimal MAX_DISCOUNT = ...` — публичная константа, разделяемая всеми. Инстансного поля в интерфейсе быть не может в принципе.

> [!tip] Из C#  
> Мапится один-в-один на C#-интерфейс: `:` → `implements`, `@Override` — это аннотация-проверка, не keyword (в C# ты бы писал `override`). Множественная реализация разрешена и там, и там. Главное расхождение — про константы: в C# интерфейс не держит `static final`-полей в таком виде, а в Java `public static final`-константа в интерфейсе — легальный и частый приём (хотя злоупотребление — антипаттерн «constant interface», к нему вернёмся ниже).

### `implements` (много) vs `extends` (один)

Прямой ответ: класс `implements` сколько угодно интерфейсов, но `extends` ровно один класс. При этом сам интерфейс может `extends` _несколько_ интерфейсов.

```java
public final class Money
        implements Comparable<Money>, Serializable {   // много контрактов — ок
    private final long cents;

    public Money(long cents) { this.cents = cents; }

    @Override
    public int compareTo(Money other) {
        return Long.compare(this.cents, other.cents);
    }
}

// интерфейс наследует несколько интерфейсов — тоже ок
public interface ReadWriteRepository
        extends ReadRepository, WriteRepository { }
```

Теперь _почему_ асимметрия. Класс несёт две вещи, которых у классического интерфейса нет: **состояние** (поля) и **конструктор** (порядок инициализации). Представь, что Java разрешила бы `extends A, B`, и оба класса имеют поле `balance` и свой конструктор. Чьё `balance` лежит в объекте? Чей конструктор и в каком порядке вызывается? Это неразрешимо без костылей — то самое «ромбовидное наследование состояния», из-за которого C++ городит виртуальное наследование, а Java и C# просто сказали «нет, один класс».

Интерфейсы же стакаются свободно именно потому, что классически **не несут ни состояния, ни конструктора**. Реализуй хоть десять — конфликтовать нечему: вопроса «чьи это данные» не возникает. Объект может играть много ролей (`Comparable` + `Serializable` + …), но физически состоит из состояния _одной_ линии классов.

> [!tip] Из C#  
> Та же логика: один базовый класс, много интерфейсов, по той же причине (конфликт состояния). C#-интуиция переносится без оговорок. (Забегая вперёд: `default`-методы Java 8 чуть размывают «интерфейс = ноль реализации» и формально создают почву для diamond — но diamond у интерфейсов это конфликт _поведения_, а не _состояния_, и именно поэтому он разрешим. Это урок 4.9.)



> [!info]- 🔬 Глубже / на синьора
> Constant interface — почему это антипаттерн (Effective Java, Item 22)  
> Раз поля интерфейса — публичные константы, возникает соблазн собрать константы в интерфейс и `implements`-ить его, чтобы писать `MAX_DISCOUNT` без квалификации. Это **constant interface antipattern**.
> 
> Проблема в том, что `implements` — часть публичного API типа. Реализовав `Constants`, ты навязываешь эти константы всем наследникам и утекаешь деталь реализации в сигнатуру типа: снаружи видно, что твой класс «is-a Constants», хотя это чистая внутренняя кухня. Убрать константу потом нельзя — сломаешь бинарную совместимость. Bloch прямо советует: константы кладут в `enum`, в `final`-класс с приватным конструктором и `static`-полями, или (с Java 8) в `static`-поля самого интерфейса, которым эти константы принадлежат по смыслу. `implements`-ить интерфейс только ради констант — не делать никогда.
> 
> Связь с ⚠️ ниже: одноимённые константы из двух интерфейсов дают `reference to X is ambiguous` (проверим в 4.9) — ещё один гвоздь в крышку constant interface.

### 

> [!warning]- ⚠️ Подводные камни 
> Edge cases интерфейса-контракта
> 
> - **Поле интерфейса — не инстансное, а одна общая `public static final` константа.** Народ с C#-фоном иногда ждёт, что `MAX_DISCOUNT` будет полем, которое каждый объект носит с собой. Нет — это одна разделяемая константа. Инстансного состояния в интерфейсе нет и быть не может.
> - **Сужение видимости при реализации — ошибка компиляции.** Метод интерфейса неявно `public`; реализация обязана остаться `public`. `void apply()` вместо `public void apply()` → `attempting to assign weaker access privileges` (проверено выше).
> - **Package-private метода в интерфейсе быть не может** — всё `public`. `private`-методы появятся только с Java 9 и только как внутренние хелперы (урок 4.7), не как часть контракта.
> - **`new Interface()` невозможно** — у интерфейса нет конструктора, инстанцировать нечего:
> 
> 
> 
> ```java
> Runme r = new Runme();   // error: Runme is abstract; cannot be instantiated
> ```
> 
> Объект, реализующий интерфейс, создаётся анонимным классом (`new Runme() { public void run(){...} }`) или — если у интерфейса ровно один абстрактный метод (functional interface / SAM) — лямбдой (`Runme r = () -> {...}`). Лямбды — отдельная глава; здесь только мостик: лямбда не магия, а способ создать реализацию SAM-интерфейса на лету.
> 
> - **Интерфейс `extends` несколько интерфейсов — легально** (`ReadWriteRepository extends A, B`), в отличие от класса. Конфликта состояния нет, поэтому нет и ограничения на один родитель.

### На собесе

> [!question]- Почему класс реализует много интерфейсов, но наследует только один класс?  
> **RU:** Потому что класс несёт состояние (поля) и конструктор, а интерфейс классически — нет. Два родительских класса означали бы два набора полей и два конструктора в одном объекте: неразрешимо, чьё поле и чей порядок инициализации — это ромбовидное наследование состояния. Интерфейсы же несут только контракт без состояния, поэтому реализуй сколько угодно — конфликтовать нечему.  
> **EN:** Because a class carries state (fields) and a constructor, while an interface classically doesn't. Two parent classes would mean two sets of fields and two constructors in one object — unresolvable ownership and init order, the "diamond of state". Interfaces carry only a contract with no state, so you can implement as many as you like; there's nothing to conflict.
> 
> _Follow-up: «а интерфейс может наследовать несколько интерфейсов?»_ → Да, `interface C extends A, B` легально — состояния нет, конфликта нет. _Follow-up: «где именно ломается C++ с множественным наследованием?»_ → На состоянии: два базовых класса с одноимённым полем → неоднозначность, лечится виртуальным наследованием. Java/C# просто запретили множественное наследование классов.

> [!question]- Какие неявные модификаторы у членов интерфейса? Можно ли его инстанцировать?  
> **RU:** Методы (до Java 8) — неявно `public abstract`, поля — неявно `public static final` (константы, не состояние). Инстанцировать интерфейс нельзя — нет конструктора; объект создают анонимным классом или, для SAM-интерфейса, лямбдой.  
> **EN:** Methods (pre-Java 8) are implicitly `public abstract`; fields are implicitly `public static final` (constants, not state). You can't instantiate an interface — no constructor; you create an instance via an anonymous class or, for a SAM interface, a lambda.
> 
> _Trap: «значит, реализацию можно сделать package-private, раз это деталь?»_ → Нет: метод интерфейса `public`, сузить видимость нельзя, `attempting to assign weaker access privileges`. _Follow-up: «зачем тогда `private` в интерфейсе с Java 9?»_ → Это внутренние хелперы для декомпозиции `default`-методов, не часть контракта (урок 4.7).
