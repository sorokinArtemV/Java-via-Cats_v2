Это блок, который логично идёт последним в фундаменте, потому что с Java 8 наивный ответ «интерфейс без реализации, abstract с реализацией» **мёртв**: оба умеют методы с телом, оба нельзя инстанцировать напрямую. Значит, разницу надо искать не в этом. И вопрос не академический — на собесе «теперь и то, и то умеет тела методов, в чём разница?» это стандартный follow-up, на котором сыпется middle, отвечающий по учебнику 2010 года.

> **Ментальная модель.** Раз оба умеют тела, разница теперь в четырёх вещах: **состояние, конструктор, единичное vs множественное наследование, моделируемое отношение** (is-a у класса vs can-do у интерфейса). А правильный ответ Middle+ — не выбирать между ними, а сложить: интерфейс несёт контракт, скелетный abstract-класс — общий код.

### Что abstract-класс умеет, чего интерфейс не может

Abstract-класс (`abstract class`, нельзя `new` — `AbstractShape is abstract; cannot be instantiated`) — это обычный класс минус возможность инстанцироваться, плюс abstract-методы. Он умеет всё, что умеет обычный класс:

- **Инстансное состояние** — `private final String name`. У интерфейса инстансных полей нет, только `public static final` константы. Прямое следствие diamond-состояния из 4.6: класс наследуется один, потому что несёт state.
- **Конструктор** — `protected AbstractShape(String name)`. Выполняется при создании наследника (через `super(...)`), проверяет инварианты, инициализирует поля. У интерфейса конструктора нет.
- **Любую видимость членов** — `private`, package-private, `protected`, `public`. У интерфейса публичный контракт (плюс `private`-хелперы с Java 9, но они не для наследников).
- **Частичную реализацию** — реализует `name()`, но `area()` оставляет abstract; наследник дозаполняет «дырки».

> [!tip] Из C#  
> `abstract class` в Java ≈ `abstract class` в C# почти один-в-один: состояние, конструктор, abstract+concrete-члены, единичное наследование, нельзя инстанцировать. C#-интуиция переносится без оговорок. `protected` конструктор у abstract-класса — та же идиома, что в .NET.

### Разница по сути (Java 21)

||`interface`|`abstract class`|
|---|---|---|
|Инстансное состояние|нет (только константы)|да, любые поля|
|Конструктор|нет|да|
|Сколько можно на класс|**много** (`implements A, B, C`)|**один** (`extends`)|
|Видимость членов|`public` (+ `private`-хелперы)|любая|
|Моделирует|**can-do** (способность, роль)|**is-a** (вид сущности)|
|Кого может наследовать тип|_плюс_ к одному классу|_вместо_ других классов|

Последняя строка — практическое ядро. Интерфейс можно «доклеить» к типу, у которого уже есть родитель: `class ArrayList extends AbstractList implements Serializable, Cloneable, RandomAccess` — один родитель + четыре роли. Abstract-класс такого не позволяет: вошёл в иерархию — занял единственный слот наследования.

### Лучший ответ Middle+: не выбирать, а сложить оба (skeletal implementation)

Ключевой инсайт: выбирать не обязательно. Effective Java (Item 20, «Prefer interfaces to abstract classes») предлагает **skeletal implementation** — abstract-класс, который частично реализует интерфейс, давая общий код, а «примитивы» оставляет наследнику. Интерфейс несёт контракт, скелетный abstract-класс — общую механику. Это весь дизайн коллекций JDK:

![[Pasted image 20260707123656.png]]

```java
public interface List<E> { /* контракт: get, size, add, ... */ }

public abstract class AbstractList<E> implements List<E> {
    // общая реализация iterator(), indexOf(), equals(), hashCode() —
    // всё поверх двух "примитивов", которые наследник обязан дать:
    public abstract E get(int index);
    public abstract int size();
}

public class ArrayList<E> extends AbstractList<E> {
    @Override public E get(int index) { /* массив */ return null; }
    @Override public int size()       { /* поле */ return 0; }
    // десятки методов получены даром из AbstractList
}
```

Так устроены `AbstractCollection`, `AbstractList`, `AbstractSet`, `AbstractMap`. Контракт (`List`) можно реализовать напрямую — и тогда ты не привязан к скелету (`MyList` на схеме); а можно унаследовать `AbstractList` и получить кучу методов бесплатно. Гибкость интерфейса + переиспользование abstract-класса в одной связке.

### Критерий выбора (а не зубрёжка)

- Нужно **общее состояние** или **конструктор**, навязывающий инварианты при создании → abstract-класс.
- Тип должен **играть несколько ролей** сразу, или ты хочешь добавить способность **несвязанным** типам (которые уже кого-то наследуют) → интерфейс.
- Моделируешь **контракт/способность** для разнородных типов (`Comparable`, `Closeable`, `Runnable`) → интерфейс.
- Моделируешь **«вид» внутри одной иерархии** с общим кодом и полями → abstract-класс.

Идиома современной Java: по умолчанию — интерфейс (Item 20), потому что abstract-класс навязывает наследование (съедает единственный слот), а интерфейс — нет. Прямой аналог .NET-гайдлайна «program to interfaces».

### 🔬 Глубже / на синьора

> [!info]- Трюк Item 20: скелет без слота наследования, и marker interface  
> **Что если класс уже кого-то наследует и не может `extends AbstractList`?** Тогда он `implements List` напрямую, а внутри держит `private`-вложенный класс, наследующий `AbstractList`, и **делегирует** ему. Получаешь общий код скелета без слота наследования — Bloch называет это «симуляцией множественного наследования». Это ровно та свобода, которую abstract-класс в одиночку не даёт.
> 
> **Marker interface** (Effective Java, Item 41) — пустой интерфейс-метка (`Serializable`, `Cloneable`, `RandomAccess`), который сам ничего не требует, а служит «меткой» типа для рантайма/инструментов. В современной Java для метаданных чаще берут аннотации, но у marker interface есть козырь: он **определяет тип**, и компилятор это проверяет. Проверил — метод, принимающий `Serializable`, не примет несериализуемый объект:
> 
> java
> 
> ```java
> static void needsSer(Serializable s) {}
> needsSer("ok");            // String is Serializable — ок
> needsSer(new Object());    // error: Object cannot be converted to Serializable
> ```
> 
> Аннотация так не умеет — она не создаёт тип, проверяемый на компиляции. Это симметричный edge к «abstract-класс без abstract-методов»: пустой интерфейс = маркер, а пустой abstract-класс = запрет инстанцирования.

### ⚠️ Подводные камни

> [!warning] Edge cases (проверены компилятором)
> 
> - **abstract-класс без единого abstract-метода — легально.** Зачем: запретить `new`, заставить пользоваться наследниками, при этом дать всю реализацию. `new` всё равно нельзя.
> - **`interface` без методов — это marker interface** (`Serializable`, `Cloneable`). Класс `implements Serializable` ничего не реализует — он просто _помечен_, и это проверяет механизм сериализации.
> - **abstract-метод не может быть `private`, `static` или `final`** — он существует, чтобы его переопределили снаружи; `private` не виден, `static` не виртуален, `final` нельзя переопределить.
> - **Конструктор abstract-класса выполняется**, хоть `new` напрямую и нельзя: `Circle` зовёт `super("circle")`, тело конструктора `AbstractShape` отрабатывает. Конструктор нужен не для инстанцирования самого abstract-класса, а для инициализации его части состояния при создании наследника (свяжи с порядком инициализации).
> - **Инстанцировать нельзя ни то, ни другое напрямую** — и `new Shape()` (abstract), и `new Runme()` (interface) дают `is abstract; cannot be instantiated`. Объект создаётся через наследника/анонимный класс/лямбду.

### На собесе

> [!question]- Теперь и интерфейс, и abstract-класс умеют методы с телом — в чём вообще разница?  
> **RU:** Интерфейс не может иметь инстансного состояния и конструктора; его реализуют много штук сразу; он моделирует способность (can-do). Abstract-класс несёт состояние и конструктор; наследуется ровно один; моделирует вид сущности (is-a). По умолчанию беру интерфейс (Item 20), а общий код выношу в скелетный `AbstractXxx`, который его реализует.  
> **EN:** An interface can't have instance state or a constructor; many types implement it at once; it models a capability (can-do). An abstract class carries state and a constructor; exactly one is inherited; it models a kind of entity (is-a). By default I pick the interface (Item 20) and push shared code into a skeletal `AbstractXxx` that implements it.
> 
> _Follow-up: «а если нужно общее состояние для группы реализаций?»_ → abstract-класс или скелетная реализация — состояние требует поля и конструктор, интерфейс их не даёт. _Follow-up: «а если объект должен играть несколько ролей сразу?»_ → только интерфейсы: `implements A, B, C`; abstract-класс съел бы единственный слот наследования.

> [!question]- Что выбрать для `PaymentProcessor` — abstract class или interface?  
> **RU:** По умолчанию интерфейс (Item 20): процессоров много (Stripe, PayPal), они разнородны, могут уже кого-то наследовать. Если у всех общая логика ретраев и поле с конфигом — вынести общее в скелетный `AbstractPaymentProcessor implements PaymentProcessor` с полем-конфигом и реализованными ретраями, оставив abstract один метод `doCharge()`.  
> **EN:** Default to an interface (Item 20): there are many processors (Stripe, PayPal), they're heterogeneous and may already extend something. If they share retry logic and a config field, hoist the shared parts into a skeletal `AbstractPaymentProcessor implements PaymentProcessor` with the config field and implemented retries, leaving one abstract method `doCharge()`.
> 
> _Trap: «а если конкретный процессор уже наследует чужой базовый класс?»_ → Он `implements PaymentProcessor` напрямую и делегирует внутреннему наследнику `AbstractPaymentProcessor` — общий код без слота наследования.
