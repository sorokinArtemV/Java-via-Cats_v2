### 0. Что такое SOLID

Пять принципов ты уже знаешь по C# — здесь важно не «повторить», а увидеть, где Java меняет саму механику: `sealed` + pattern matching переворачивают компромисс OCP, ковариантные возвраты и wildcards ложатся на LSP иначе, а DIP в мире без Spring (фреймворки у нас out of scope) собирается вручную конструктором.

**Mental model.** SOLID — пять эвристик ООП-дизайна (Robert C. Martin; акроним — Michael Feathers), управляющих связностью и направлением зависимостей так, чтобы код можно было менять и расширять, не ломая существующее. Это ставки под изменяемость, а не законы.

- **S** — модуль отвечает перед одним актором (одна причина меняться).
- **O** — расширяй поведение новым кодом, не трогая существующий.
- **L** — подтип подставляется вместо базового без потери корректности.
- **I** — не заставляй клиента зависеть от того, чем он не пользуется.
- **D** — завись от абстракций, не от деталей.

Общая нить: изолировать изменения и развернуть зависимости к стабильным абстракциям.

### 1. SRP — Single Responsibility

Один класс `Employee` считает зарплату, пишет себя в БД и форматирует табель для HR. Через полгода бухгалтерия меняет правило расчёта — и ты правишь файл, в котором заодно живут SQL-схема и вёрстка отчёта; тест на расчёт задевает сериализацию, ревью раздувается. Три стейкхолдера тянут один файл в три стороны.

**Mental model.** У класса одна причина для изменения, потому что он отвечает перед одним актором (стейкхолдером).

**How it works.** «Причина для изменения» — это группа пользователей/стейкхолдеров, чьи требования заставляют код меняться. Если один класс обслуживает двух акторов, их требования столкнутся в одном месте: правка ради одного рискует сломать другого. Разносим код по причинам изменения.

> [!example] Нарушение → фикс
> 
> java
> 
> ```java
> // Нарушение: три актора в одном классе
> class Employee {
>     BigDecimal calculatePay() { ... }   // правила бухгалтерии — актор: Finance
>     void save() { ... }                 // схема БД        — актор: DBA
>     String reportHours() { ... }        // формат табеля   — актор: HR
> }
> 
> // Фикс: один класс — один актор
> class PayCalculator     { BigDecimal calculatePay(Employee e) { ... } }
> class EmployeeRepository { void save(Employee e) { ... } }
> class HourReporter      { String reportHours(Employee e) { ... } }
> ```

Важно: SRP **не** значит «класс делает одну вещь» или «один метод». Класс может иметь много методов — критерий в том, что все они меняются **по одной причине**.

> [!tip] 💡 Из C#  
> Принцип языконезависим — механика та же. Две джава-специфики: (1) единица группировки «по актору» в Java — **пакет** (`com.acme.payroll` / `com.acme.reporting`), тогда как в C# ты мыслил в терминах assembly и `namespace`; (2) в C# `partial class` мог физически размазать один класс по файлам и скрыть нарушение SRP — в Java `partial` нет, класс всегда в одном файле, так что раздувшаяся ответственность видна глазом.

> [!tip] 🔬 Глубже / на синьора  
> Определение эволюционировало: исходно «one reason to change» → уточнено Мартином до «модуль ответственен перед одним и только одним актором». Формулировка через актора операциональна, «одна вещь» — нет.  
> Связь с **cohesion**: SRP — это когезия по _причине изменения_, а не по функциональной близости. Метрика-родственник — LCOM (Lack of Cohesion of Methods): чем меньше методы класса делят между собой поля, тем выше LCOM и тем вероятнее, что в классе живут несколько ответственностей.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Over-splitting.** Дробление до анемичных классов из одного метода превращает навигацию в квест; SRP не требует «класс = функция».
> 2. **«Ответственность» субъективна.** Без привязки к _актору_ спор о границах скатывается в bikeshedding — всегда спрашивай «чьё требование это меняет».
> 3. **DTO + логика в одном классе.** Частый Java-грех: `record`/POJO обрастает бизнес-правилами и заодно аннотациями сериализации — два актора (модель данных и формат обмена).
> 4. **God-object по накоплению.** `*Service` / `*Manager` / `*Util` тихо собирают всё подряд; SRP нарушается не одним решением, а дрейфом.
> 5. **Ложное срабатывание на слоистость.** Методы, меняющиеся вместе по требованию одного актора, разносить _не надо_ — это уже вредит когезии.

> [!question]- На собесе  
> **Вопросы:** «Что такое SRP?» → FU: «что считать responsibility?» → **trap:** «значит, у класса должен быть один метод?»
> 
> **Слабый ответ:** «Класс должен делать одно дело» — нет операционального критерия, не отвечает на «что такое одно дело» и рассыпается на trap.
> 
> **Model answer (RU).** SRP — у модуля одна причина для изменения, то есть он отвечает перед одним актором. Это не про число методов: класс с десятком методов соблюдает SRP, если все они меняются по требованиям одного стейкхолдера. Нарушение — когда сходятся, скажем, правила расчёта (Finance) и формат отчёта (HR): правка для одних ломает других.
> 
> **Model answer (EN).** SRP means a module has a single reason to change — it answers to one actor. It's not about method count: a class with many methods still satisfies SRP if they all change for one stakeholder. The violation is when, say, pay rules (Finance) and report formatting (HR) live in one class, so a change for one risks breaking the other.

### 2. OCP — Open/Closed

Метод `area()` — это `switch` по типу фигуры. Появляется `Triangle` — ты снова лезешь в тот же `switch`, снова его тестируешь, снова рискуешь задеть старые ветки. Каждое расширение требует правки уже проверенного кода — ровно то, чего OCP хочет избежать.

**Mental model.** Модуль открыт для расширения (новое поведение добавляется), но закрыт для модификации (существующий проверенный код не трогаем).

**How it works.** Достигается через **абстракцию**: выделяем ось вариативности в интерфейс, и новые варианты добавляются новыми реализациями, а не правкой старого кода.

> [!example] Нарушение → фикс (полиморфизм / Strategy)
> 
> java
> 
> ```java
> // Нарушение: каждая новая фигура правит существующий switch
> double area(Object shape) {
>     if (shape instanceof Circle c)    return Math.PI * c.r() * c.r();
>     if (shape instanceof Rectangle r) return r.w() * r.h();
>     throw new IllegalArgumentException("unknown shape");  // добавил Triangle — снова сюда
> }
> 
> // OCP: новая фигура = новый класс, калькулятор не трогаем
> interface Shape { double area(); }
> final class Circle   implements Shape { public double area() { return Math.PI * r * r; } }
> final class Triangle implements Shape { public double area() { return 0.5 * b * h; } }  // расширение без модификации
> ```

> [!tip] 💡 Из C#  
> В C# «`switch` по типу» был безусловным анти-паттерном OCP. В Java (как и в свежем C#) картина тоньше: `sealed`-иерархия + pattern matching в `switch` (Java 17 sealed — **JEP 409**; switch-паттерны финализированы в Java 21 — **JEP 441**) делают исчерпывающий `switch` **проверяемым компилятором**. Пропустил вариант — не соберётся:
> 
> java
> 
> ```java
> sealed interface Shape permits Circle, Rect {}
> record Circle(double r) implements Shape {}
> record Rect(double w, double h) implements Shape {}
> 
> double area(Shape s) {
>     return switch (s) {              // без default — компилятор требует полноту
>         case Circle c -> Math.PI * c.r() * c.r();
>         case Rect r   -> r.w() * r.h();
>     };
> }
> ```
> 
> Убрал ветку `Rect` — компилятор: `the switch expression does not cover all possible input values` (проверено на JDK 21). То есть Java даёт _второй_ легитимный способ соблюсти дух OCP — не только «полиморфизм», но и «закрытая иерархия + исчерпывающий разбор».

> [!tip] 🔬 Глубже / на синьора  
> Две исторические трактовки: **Meyer (1988)** — закрытость через наследование; **Martin** — полиморфная OCP через абстракции (по сути опирается на DIP). На интервью ждут вторую.  
> **Expression problem.** Полиморфный OCP (интерфейс + реализации) дёшев к _добавлению нового типа_ (новый класс), но дорог к _добавлению новой операции_ (правишь каждый класс). `sealed` + `switch` — ровно наоборот: дёшево добавить операцию (новый `switch` в одном месте), но добавление нового типа ломает все `switch`… и это **фича**: компилятор перечислит все места, которые надо дописать. Выбор оси — это выбор, что ты ожидаешь варьировать чаще: типы или операции.  
> Нельзя быть открытым ко _всем_ осям сразу. Угадал ось — расширение дёшево; не угадал — всё равно правишь. OCP — ставка, а не бесплатная гарантия.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Спекулятивная абстракция.** Интерфейс с единственной реализацией «на будущее» — это YAGNI-оверхед, а не OCP.
> 2. **Не та ось.** Вынес в абстракцию тип платежа, а меняется способ логирования — правишь всё равно. Абстракция закрывает только выбранную ось.
> 3. **`default`-метод как «расширение» интерфейса** кажется OCP-дружелюбным, но добавление нового `default`-метода с логикой в опубликованный интерфейс — это модификация контракта всех реализаторов.
> 4. **`sealed`-switch с `default`.** Дописал `default` «на всякий» — потерял проверку исчерпывающести; при добавлении варианта компилятор промолчит.
> 5. **Полиморфизм ради полиморфизма.** Если вариант ровно один и не предвидится второго, `switch`/`if` честнее интерфейса.

> [!question]- На собесе  
> **Вопросы:** «Что такое OCP, как закрыть для модификации?» → **trap:** «это же просто наследование?» → FU (Java): «а исчерпывающий `switch` по `sealed` — это нарушение OCP или нет?»
> 
> **Слабый ответ:** «Используй наследование и `override`» — наследование лишь один механизм; суть в выделении абстрактной точки вариативности.
> 
> **Model answer (RU).** OCP — добавлять новое поведение, не меняя существующий код: выносим ось изменения в абстракцию, новые варианты — новыми реализациями. Классика — заменить разрастающийся `switch` по типу на полиморфизм/Strategy. В Java есть и второй путь: `sealed`-иерархия + исчерпывающий `switch`, где полноту проверяет компилятор — тогда «switch по типу» уже не анти-паттерн. Закрытость не абсолютна: ты выбираешь ось.
> 
> **Model answer (EN).** OCP means adding behavior without editing existing code: extract the axis of change into an abstraction and add variants as new implementations — canonically replacing a growing type `switch` with polymorphism/Strategy. Java offers a second route: a `sealed` hierarchy plus an exhaustive `switch` whose completeness the compiler checks — there the "switch on type" is no longer an anti-pattern. Closure isn't absolute — you pick the axis.

### 3. LSP — Liskov Substitution

Ты написал функцию, корректную для `List`. Кто-то передал в неё `List.of(1, 2, 3)` — и на первом `add()` всё падает с `UnsupportedOperationException`, хотя формально это `List`. Тип совпал, а контракт — нет: подстановка сломала корректность.

**Mental model.** Объект подтипа можно подставить везде, где ожидается базовый тип, не нарушив корректность. Наследуется не только сигнатура, но и **контракт** (behavioral subtyping).

**How it works.** Правила подстановки (design by contract):

- предусловия подтипа **не сильнее** базовых (можно слабее — требовать меньше);
- постусловия **не слабее** базовых (можно сильнее — гарантировать больше);
- инварианты базового типа сохраняются;
- history constraint: подтип не допускает изменений состояния, запрещённых базовым.

> [!example] Каноничное нарушение — Square / Rectangle
> 
> java
> 
> ```java
> class Rectangle {
>     protected int w, h;
>     void setWidth(int w)  { this.w = w; }
>     void setHeight(int h) { this.h = h; }
>     int area() { return w * h; }
> }
> class Square extends Rectangle {                 // «квадрат — это прямоугольник» (математически)
>     @Override void setWidth(int w)  { this.w = w; this.h = w; }
>     @Override void setHeight(int h) { this.w = h; this.h = h; }
> }
> 
> int probe(Rectangle r) { r.setWidth(5); r.setHeight(4); return r.area(); } // ждём 20
> // probe(new Rectangle()) -> 20
> // probe(new Square())    -> 16   (проверено на JDK 21)
> ```

Геометрическое _is-a_ ≠ поведенческое _is-a_: `Square` ломает контракт «ширина и высота независимы». Другие запахи: `@Override`, бросающий `UnsupportedOperationException` (аналог `Penguin.fly()`); подтип, усиливающий предусловие. Фикс — не наследовать там, где контракт расходится (общий интерфейс без ловушек-сеттеров, либо immutable-типы).

> [!tip] 💡 Из C#  
> Три моста:
> 
> - **JDK сам нарушает LSP «легально».** `List.of(...)` и `Collections.unmodifiableList(...)` — это `List`, но `add()`/`remove()` бросают `UnsupportedOperationException`; `Arrays.asList(...)` разрешает `set()`, но не `add()` (проверено). В C# ты видел это у `ReadOnlyCollection`/массива-как-`IList`. Урок один: сам факт «реализует интерфейс» не гарантирует соблюдение его контракта.
> - **Ковариантный возврат** есть в обоих, но в Java с 2005 (Java 5), в C# — только с C# 9. По теории он LSP-безопасен (сужение результата не ломает вызывающего).
> - **Вариантность.** В C# ты объявлял её на типе через `in`/`out` (declaration-site). В Java вариантности на типе нет — есть wildcards на месте использования (`? extends` / `? super`, call-site), и они прямо кодируют LSP: параметры контравариантны (`? super`), результаты ковариантны (`? extends`). Детали — глава 7.

> [!tip] 🔬 Глубже / на синьора  
> Идея — Liskov (1987), формализована в Liskov & Wing, _A Behavioral Notion of Subtyping_ (1994).  
> Связь с вариантностью: параметры теоретически **контравариантны**, возвращаемые значения **ковариантны** — поэтому ковариантный возврат безопасен по LSP, а сужение типа параметра — сломало бы. Java (как и C#) не даёт менять тип параметра в `@Override`, так что контравариантность параметров тут остаётся теоретической и реализуется через wildcards в дженериках.  
> `UnsupportedOperationException` / `NotImplementedException` в переопределении — почти всегда явный маркер нарушения LSP: подтип отказывается выполнять операцию, обещанную базовым контрактом.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`@Override` с `throw new UnsupportedOperationException()`** — красный флаг: подтип не is-a.
> 2. **`equals()` в иерархии.** Классический Java-капкан: `equals()` подкласса с новым полем ломает _симметрию/транзитивность_ относительно базового класса (Effective Java, Item 10). Подтип, несовместимый по `equals`, не подставляется корректно — это LSP на уровне `Object`-контракта.
> 3. **Mutable-подтип immutable-базы** нарушает инвариант неизменяемости базы.
> 4. **Усиление предусловия:** база принимала любой `int`, подтип начал требовать «только положительный» и кидать — вызывающий, корректный для базы, падает.
> 5. **Ослабление постусловия:** база гарантировала non-null результат, подтип иногда возвращает `null` — тихая поломка у вызывающего.

> [!question]- На собесе  
> **Вопросы:** «LSP, пример нарушения?» → FU: «что с пред/постусловиями?» → **trap:** «но ведь квадрат — это прямоугольник, почему нельзя наследовать?» → FU (Java): «где LSP нарушает сам JDK?»
> 
> **Слабый ответ:** «Наследник не должен ломать базовый класс» — по сути верно, но без механики (контракт, пред/постусловия, инварианты) не проходит follow-up.
> 
> **Model answer (RU).** LSP — объект подтипа должен подставляться вместо базового без потери корректности: подтип не усиливает предусловия, не ослабляет постусловия и сохраняет инварианты. Square/Rectangle — нарушение, потому что Square ломает контракт независимости сторон, и код, рассчитанный на Rectangle, начинает врать. Живой JDK-пример: `List.of(...)` — это `List`, но `add()` бросает `UnsupportedOperationException`, нарушая контракт мутабельности. Математическое «is-a» ≠ поведенческому.
> 
> **Model answer (EN).** LSP means a subtype is substitutable for its base without breaking correctness: it must not strengthen preconditions, weaken postconditions, or violate invariants. Square/Rectangle breaks it because Square violates the "independent sides" contract, so code written against Rectangle starts giving wrong results. A live JDK example: `List.of(...)` is a `List`, yet `add()` throws `UnsupportedOperationException`, breaking the mutability contract. A mathematical "is-a" isn't a behavioral one.

### 4. ISP — Interface Segregation

Интерфейс `Device` требует `print()`, `scan()`, `fax()`. Старый принтер умеет только печатать — но обязан реализовать `scan()` и `fax()` заглушками, которые бросают исключение. Теперь любое изменение в «факсовой» части интерфейса заставляет перекомпилировать и старый принтер, которому факс не нужен вовсе.

**Mental model.** Клиент не должен зависеть от членов интерфейса, которыми не пользуется; лучше несколько узких ролевых интерфейсов, чем один «толстый».

**How it works.** Толстый интерфейс заставляет реализаторов писать заглушки, а клиентов — зависеть от изменений в ненужных членах. Дробим по ролям клиента.

> [!example] Нарушение → фикс (ролевые интерфейсы)
> 
> java
> 
> ```java
> // Нарушение: старый принтер вынужден реализовывать scan/fax заглушками
> interface MultifunctionDevice { void print(); void scan(); void fax(); }
> class OldPrinter implements MultifunctionDevice {
>     public void print() { ... }
>     public void scan()  { throw new UnsupportedOperationException(); }  // ещё и LSP-запах
>     public void fax()   { throw new UnsupportedOperationException(); }
> }
> 
> // ISP: ролевые интерфейсы
> interface Printer { void print(); }
> interface Scanner { void scan(); }
> class OldPrinter implements Printer { public void print() { ... } }     // зависит только от нужного
> ```

> [!tip] 💡 Из C#  
> Java-доведение ISP до предела — пакет `java.util.function`: `Function`, `Consumer`, `Supplier`, `Predicate` — это _однометодные_ ролевые интерфейсы (SAM), под которые компилятор принимает лямбды. Куда уже интерфейс не сегрегируешь. В C# ту же роль играли делегаты (`Func<>`/`Action<>`) — но в Java это именно интерфейсы с `@FunctionalInterface`, поэтому SAM-интерфейс и ISP-«role interface» здесь буквально одно и то же (глава 9). Обратная сторона: `default`-методы в Java позволяют «раздуть» интерфейс без слома реализаторов — удобно, но легко протащить обратно fat-interface.

> [!tip] 🔬 Глубже / на синьора  
> **ISP vs SRP** — любимый FU. SRP смотрит со стороны _класса_: сколько у него причин меняться. ISP — со стороны _клиента_: от скольких лишних членов он вынужден зависеть. Часто ведут к похожему дроблению, но мотивация разная (производитель vs потребитель). Связано с code smell «fat interface» и понятием role interface (Fowler): интерфейс проектируется под конкретную роль клиента, а не под «всё, что умеет объект».

> [!warning] ⚠️ Подводные камни
> 
> 1. **Over-segmentation.** Интерфейс на каждый метод «ради ISP» — взрыв типов; цель не мельчить, а не навязывать лишнюю зависимость.
> 2. **`default`-заглушки, маскирующие fat-interface.** Дал пустые `default`-реализации `scan()`/`fax()` — реализатор молчит, но зависимость от ненужного никуда не делась.
> 3. **Marker-интерфейсы без методов** (`Serializable`-стиль) — иногда оправданы, но легко превращаются в костыль вместо явной роли.
> 4. **Клиент, зависящий от толстого типа «на всякий случай»** (`Object`-параметр вместо узкой роли) — обратная крайность: недосегрегация.
> 5. **Разделил интерфейс, но клиент всё равно берёт реализацию целиком** — сегрегация только в сигнатурах, а связность осталась.

> [!question]- На собесе  
> **Вопросы:** «Что такое ISP?» → FU: «ISP vs SRP?» → **trap (Java):** «`Function`/`Consumer` — это про ISP?»
> 
> **Слабый ответ:** «Интерфейсы должны быть маленькими» — мельчить ради мельчения не цель; цель — не навязывать клиенту зависимость от лишнего.
> 
> **Model answer (RU).** ISP — не заставлять клиента зависеть от членов интерфейса, которые он не использует. Толстый интерфейс плодит заглушки у реализаторов и лишние зависимости у клиентов; решение — дробить на ролевые интерфейсы под конкретных клиентов. От SRP отличается точкой зрения: ISP про связанность клиента с интерфейсом, SRP про причины изменения класса. Предельный случай ISP в Java — функциональные интерфейсы из `java.util.function`.
> 
> **Model answer (EN).** ISP means clients shouldn't be forced to depend on interface members they don't use. A fat interface breeds stub implementations and needless client coupling; the fix is splitting it into role interfaces shaped by each client. It differs from SRP in viewpoint: ISP is about a client's coupling to an interface, SRP about a class's reasons to change. Java's extreme case of ISP is the single-method interfaces in `java.util.function`.

### 5. DIP — Dependency Inversion

`PasswordReminder` внутри себя вызывает `new MySqlConnection()`. Теперь его нельзя ни протестировать без живой БД, ни переключить на другое хранилище, не переписав сам `PasswordReminder`. Высокоуровневая политика намертво прибита к низкоуровневой детали.

**Mental model.** И высокоуровневые, и низкоуровневые модули зависят от абстракции; саму абстракцию определяет потребитель (политика), а не деталь. «Инверсия» — стрелка `High → Low` разворачивается к абстракции.

**How it works.** Обычно политика напрямую создаёт и вызывает конкретную деталь — high жёстко зависит от low. DIP вводит абстракцию, от которой зависит high; low её реализует. Теперь high не знает о детали, и деталь можно подменить.

> [!example] Нарушение → фикс (constructor injection, без контейнера)
> 
> java
> 
> ```java
> // Нарушение: high-level создаёт и держит конкретную деталь
> class PasswordReminder {
>     private final MySqlConnection db = new MySqlConnection();  // прибито гвоздями
> }
> 
> // DIP: оба зависят от абстракции, деталь подаётся снаружи
> interface MessageStore { void save(String m); }               // абстракция — нужна потребителю
> final class MySqlStore implements MessageStore { public void save(String m) { ... } }
> final class PasswordReminder {
>     private final MessageStore store;
>     PasswordReminder(MessageStore store) { this.store = store; }  // зависимость снаружи
> }
> ```

Ключ: абстракцию формирует high-level слой под свои нужды (как «порт»), а не low-level. Иначе интерфейс просто повторит деталь — и инверсии не будет. На диаграмме — что именно разворачивается:

> [!tip] 💡 Из C#  
> Механика идентична, но контекст курса важен: **DIP не требует фреймворка**. У нас Spring вне области — и это ровно к месту, потому что DIP соблюдается голым конструктором: объявил интерфейс, принял его параметром, подал реализацию извне (в тесте — мок, в проде — `MySqlStore`). DI-контейнер (Spring в Java, `Microsoft.Extensions.DependencyInjection` в .NET) лишь _автоматизирует_ подачу — сам принцип от него не зависит. И наоборот: можно жить с контейнером и нарушать DIP, инжектя конкретные типы.

> [!tip] 🔬 Глубже / на синьора — DIP ≠ DI ≠ IoC  
> Главная ловушка секции:
> 
> - **DIP** — _принцип_ дизайна: завись от абстракций.
> - **DI (Dependency Injection)** — _техника_ подачи зависимости извне (constructor / setter / method injection); один из способов соблюсти DIP. В Java канон — **constructor injection** в `final`-поле.
> - **IoC (Inversion of Control)** — общий _концепт_: поток управления отдаётся фреймворку («Hollywood principle: don't call us, we'll call you»). DI — частный случай IoC применительно к зависимостям.
> 
> Следствия: DIP соблюдается без всякого контейнера — просто передай интерфейс в конструктор руками. И наоборот, можно использовать DI-контейнер и нарушать DIP, если инжектишь конкретные типы. Кто владеет абстракцией: в чистом DIP интерфейс объявлен в слое _потребителя_ (порт), а реализация — снаружи (адаптер). Это же — ядро гексагональной архитектуры (ports & adapters), но детали архитектуры за рамками главы.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Инжектишь конкретный тип.** `Reminder(MySqlStore s)` через контейнер — DI есть, DIP нет: зависимость от детали осталась.
> 2. **Абстракция-калька детали.** Интерфейс, дословно повторяющий один класс (те же методы, те же исключения), не инвертирует ничего — «порт» должен формироваться потребностью политики, а не деталью.
> 3. **Циклы через конструктор.** `A(B b)` + `B(A a)` — при ручном connstructor injection это тупик компиляции/инициализации; сигнал спроектировать общую абстракцию или разорвать цикл.
> 4. **Over-abstraction.** Интерфейс на каждый класс «во имя DIP» — навигационный ад и YAGNI; абстрагируй границы, которые реально пересекают слой или тестируются.
> 5. **`new` внутри high-level, спрятанный в фабрике.** Фабрика, жёстко создающая деталь, просто переносит нарушение на уровень глубже, если сама не абстрагирована.

> [!question]- На собесе  
> **Вопросы:** «Что такое DIP?» → **trap:** «это про DI-контейнер?» → FU: «разница DIP / DI / IoC?»
> 
> **Слабый ответ:** «DIP — это использовать DI-контейнер» — путает принцип с инструментом; не звучит про абстракцию и владение ею.
> 
> **Model answer (RU).** DIP — высокоуровневые модули не зависят от низкоуровневых напрямую; оба зависят от абстракции, которую определяет слой-потребитель. Это не про DI-контейнер: DI — лишь техника подачи зависимости (в Java — конструктор), а IoC — более общий концепт «фреймворк вызывает тебя». Можно соблюдать DIP вручную и нарушать его с контейнером.  
> _(senior add-on)_ Важно, кто владеет абстракцией: интерфейс принадлежит политике, деталь его реализует — иначе инверсии нет.
> 
> **Model answer (EN).** DIP means high-level modules don't depend on low-level ones directly; both depend on an abstraction owned by the consuming layer. It's not about a DI container: DI is just a technique for supplying a dependency (constructor injection in Java), and IoC is the broader "the framework calls you" concept. You can follow DIP by hand and violate it with a container.  
> _(senior add-on)_ What matters is who owns the abstraction: the interface belongs to the policy, the detail implements it — otherwise there's no inversion.

### 6. 🔬 Когда SOLID вредит (на синьора)

> [!tip] 🔬 Полностью bonus-слой  
> SOLID — эвристики под _ожидаемую_ изменяемость, и слепое применение даёт over-engineering: интерфейсы с единственным реализатором «на всякий случай», дробление, после которого по коду тяжелее навигировать, абстракции ради абстракций. Если ось изменения не наступает, гибкость — чистый оверхед (YAGNI).  
> Зрелый ответ на «всегда ли применяешь SOLID»: применяю там, где реально ожидаю изменение или переиспользование, и не плачу за гибкость, которая не понадобится. В Java у этого есть конкретный маркер: если весь «DIP» — это один интерфейс с одной реализацией и одним моком в тесте, честнее спросить, покупает ли абстракция что-то, кроме мокабельности. Критика принципов (расплывчатость формулировок, риск карго-культа) — нормальная часть синьорского разговора, а не ересь.
