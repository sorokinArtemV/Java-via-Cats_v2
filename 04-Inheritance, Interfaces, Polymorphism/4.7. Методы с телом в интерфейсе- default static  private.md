

До Java 8 разделение было простым: интерфейс — только сигнатуры, abstract-класс — тела. Java 8 это сломала: интерфейсам разрешили `default`-методы (с телом) и `static`-методы, а Java 9 добавила `private`-хелперы. Причина не в том, что «джавистам захотелось пихать логику в интерфейсы» — за `default` стоит конкретная инженерная нужда, из-за которой без него не выехала бы вся Streams API. Разберём, что интерфейс теперь умеет держать и, главное, зачем.

> **Ментальная модель.** `default`-метод — это способ _дорастить_ интерфейс новым методом с готовым телом, не заставляя переписывать миллионы существующих реализаций. `static` — фабрика/утилита на самом типе. `private` — внутренний хелпер. Из трёх только `default` наследуется и виртуален; остальные — нет.

### Зачем вообще `default`: эволюция интерфейса

Вот боль, ради которой их придумали. К 2014-му интерфейс `Collection` реализовывали миллионы классов — в JDK, в библиотеках, в чужом коде по всему миру. Java 8 принесла лямбды и `Stream` API, и для них позарез нужно было добавить методы вроде `stream()` и `forEach()` прямо в `Collection`/`Iterable`. Но добавить в интерфейс обычный (абстрактный) метод — значит обязать **каждую** существующую реализацию его написать; не написал — класс перестаёт компилироваться. Это сломало бы всю экосистему за одну ночь.

![[Pasted image 20260707123343.png]]

`default`-метод — метод интерфейса с телом, помеченный `default`, — и есть решение. Ты добавляешь в интерфейс метод сразу с реализацией, и все существующие классы получают его бесплатно, ничего не переписывая. Это механизм эволюции интерфейса без поломки обратной совместимости ([JEP 126](https://openjdk.org/jeps/126), Java 8). То есть `default` родился не из идеи «пихать логику в интерфейсы», а из нужды «как дорастить интерфейс, не убив миллион реализаций» — отсюда вся правильная интуиция о том, когда им пользоваться.

### Синтаксис и механика

```java
public interface Notifier {
    void send(String message);                   // abstract — реализатор обязан написать

    default void sendUrgent(String message) {    // default — есть тело, наследуется даром
        send("[URGENT] " + message);
    }
}

public class EmailNotifier implements Notifier {
    @Override public void send(String message) { System.out.println("email: " + message); }
    // sendUrgent() писать НЕ обязан — унаследован
}
```

Классический паттерн: интерфейс описывает один-два «примитивных» абстрактных метода, а удобные комбинации строит `default`-методами поверх них. Ровно так устроен `Comparator` (`thenComparing`, `reversed` — всё `default`'ы поверх `compare`).

Два свойства `default`-методов надо зафиксировать чётко — я их прогнал кодом:

> [!example] Прогон на JDK 21 — `default` наследуется и виртуален
> 
> 
> 
> ```java
> interface Greeter { default String greet() { return "default greet"; } }
> class Lazy   implements Greeter { }
> class Polite implements Greeter { @Override public String greet() { return "polite greet"; } }
> ```
> 
> Вызовы `new Lazy().greet()` (через ссылку интерфейса и через ссылку класса) и `new Polite().greet()`:
> 
> ```
> Lazy via iface : default greet
> Lazy via class : default greet
> Polite         : polite greet
> ```
> 
> Что доказано: (1) **наследуется** — `Lazy` не писал `greet()`, но он у него есть; (2) **виртуален** — `Polite` переопределил, и выбирается версия по _реальному типу объекта_ (тот самый dynamic dispatch); (3) **зовётся на конкретном типе** — `new Lazy().greet()` через ссылку типа `Lazy` работает, метод реально вошёл в набор методов класса.

Третий пункт — главный водораздел с C#, держи его.

> [!tip] Из C# — `default` methods vs DIM  
> В C# 8 завезли DIM (default interface methods) — почти ровесник Java 8 (2014 vs 2019). Выглядит похоже, но резолюция и доступ устроены принципиально иначе:
> 
> - **Доступ.** В C# DIM-реализация доступна **только через ссылку типа интерфейса**: `IFoo x = new Bar(); x.M();` — работает, а `Bar b = new Bar(); b.M();` — не компилируется, если `Bar` сам не переопределил `M`. DIM-метод физически _не становится_ членом класса. В Java наоборот — `default` входит в набор методов класса и зовётся на конкретном типе (мы это видели). Это, возможно, главная причина, почему DIM в .NET ощущается «инородной» фичей, а в Java — родной.
> - **explicit interface implementation** (`void IFoo.M()`) — в Java аналога нет вообще. Конфликты решаются через `@Override` + `Interface.super`, не через приватные явные реализации.

### `static`-методы интерфейса (Java 8)


```java
interface Validator {
    boolean isValid(String s);
    static Validator nonEmpty() {              // живёт на интерфейсе
        return s -> s != null && !s.isBlank();
    }
}
```

Зовётся **только через имя интерфейса**: `Validator.nonEmpty()`. Контрастируй с `default`: `static` не наследуется реализатором и не виртуален. Проверил — вызов через класс-реализатор не компилируется:

> [!example] Прогон на JDK 21 — `static` не наследуется
> 
> 
> 
> ```java
> class LengthValidator implements Validator { public boolean isValid(String s){ return true; } }
> LengthValidator.nonEmpty();   // ?
> ```
> 
> ```
> error: cannot find symbol
>   symbol: method nonEmpty()
> ```
> 
> `static`-метод интерфейса не входит в набор членов класса-реализатора. И, в отличие от `static`-методов _классов_, тут нет hiding — прятать нечего.

Зачем нужны: фабрики и утилиты, концептуально принадлежащие типу. До Java 8 их складывали в отдельный класс-компаньон — отсюда историческая пара `Collection` (интерфейс) + `Collections` (утилиты), `Path` + `Paths`, `Channel` + `Channels`. Java 8 позволила положить фабрику прямо на интерфейс: `Stream.of()`, `Comparator.comparing()`, `List.of()`, `Map.entry()` — это всё `static`-методы интерфейсов.

> [!tip] Из C#  
> Тот же путь: раньше — статический класс-хелпер (`Enumerable.Where`, `string.Join`). C# тоже разрешил `static`-члены в интерфейсах только с C# 8 (а `static abstract` для generic math — с C# 11). «Фабрика на самом интерфейсе» в обоих языках — относительно свежая возможность; раньше и там, и там жили companion-классы.

### `private`-методы интерфейса (Java 9, JEP 213)

Проблема, которую они закрыли: с Java 8 `default`-методы начали дублировать логику между собой, а спрятать общий кусок было некуда — любой член интерфейса `public`. [JEP 213](https://openjdk.org/jeps/213) добавила `private` и `private static` как внутренние хелперы:

```java
interface AuditLog {
    void record(String event);

    default void recordLogin(String user)  { record(line("LOGIN", user)); }
    default void recordLogout(String user) { record(line("LOGOUT", user)); }

    private String line(String action, String user) {   // хелпер, не часть контракта
        return action + " by " + user;
    }
}
```

`line()` — деталь реализации: не виден реализаторам, не наследуется, не переопределяется, не в публичном API. Чистая декомпозиция, никакой новой семантики.

### 🔬 Глубже / на синьора

> [!info]- Почему в Java `default` — НЕ антипаттерн, но осторожность нужна (Effective Java, Item 21)  
> Твоё чутьё из .NET не пустое — оно просто бьёт мимо. DIM в C# ощущается как антипаттерн, потому что там уже были abstract-классы под shared-поведение, а DIM выглядит как «множественное наследование поведения через заднюю дверь» с неудобным доступом только через интерфейс. В Java `default` — штатный, повсеместный механизм самого JDK: `Collection.stream()`, `Iterable.forEach()`, весь `Comparator`, `Map.getOrDefault()`/`computeIfAbsent()`. Это не костыль, а часть нормального дизайна стандартной библиотеки.
> 
> Но злоупотребление реально, и тут .NET-инстинкт прав. Bloch (Item 21, «Design interfaces for posterity») предупреждает: **нельзя написать `default`-метод, который сохранит инварианты любой мыслимой реализации**. Он добавляется «вслепую» во все классы, реализующие интерфейс, — а они о нём не знали и могут на нём сломаться в рантайме, не на компиляции. Правильное применение: (а) эволюция API и (б) удобные методы поверх абстрактных «примитивов» — не бизнес-логика и не состояние. Граница жёсткая: состояния у интерфейса нет и быть не может (только `public static final` константы), поэтому потянуло хранить состояние или городить иерархию поведения — нужен abstract-класс (урок 4.8).

### ⚠️ Подводные камни

> [!warning] Edge cases состава интерфейса (проверены компилятором)
> 
> - **`default` НЕ может переопределить методы `Object`** (`equals`, `hashCode`, `toString`) — ошибка компиляции, а не тихое игнорирование:
> 
> ```
> error: default method toString in interface BadToString overrides a member of java.lang.Object
> ```
> 
> Причина — «class always wins» (разберём в 4.9): методы `Object` всегда приходят от класса, такой `default` был бы мёртвым кодом, и язык запрещает писать заведомо неработающее ([JLS §9.4.1.3](https://docs.oracle.com/javase/specs/jls/se21/html/jls-9.html#jls-9.4.1.3)). Свяжи с `equals`/`hashCode`-контрактом: реализацию всегда пишут в классе — теперь понятно, почему её _нельзя_ спрятать в интерфейс.
> 
> - **Метод интерфейса нельзя сделать `final`** → `modifier final not allowed here`. Смысл `default` — быть переопределяемым, `final` это запретил бы.
> - **`default` + `static` несовместимы на одном методе** → `illegal combination of modifiers: static and default`. Либо инстансный дефолт, либо статический метод.
> - **`default` без тела невозможен** → `missing method body, or declare abstract`. Либо есть тело (`default`), либо нет (`abstract`).
> - **`static`-метод интерфейса зовётся только как `Interface.method()`**, не через класс-реализатор — вопреки аналогии со `static`-методами классов.
> - **`private`-метод интерфейса обязан иметь тело** → `missing method body`. `private` несовместим с `abstract`: абстрактный существует, чтобы его переопределили снаружи, а `private` снаружи не виден — переопределять некому.

**Полная картина состава интерфейса (Java 21).** Теперь интерфейс держит пять видов членов: `public static final` константы (всегда); `public abstract` методы (всегда — ядро контракта); `public default` методы (Java 8, наследуются и виртуальны); `public static` методы (Java 8, не наследуются, через имя интерфейса); `private`/`private static` методы (Java 9, хелперы). Чего по-прежнему **не может**: инстансного состояния, конструктора, переопределения методов `Object`.

### На собесе

> [!question]- Зачем ввели `default`-методы, что сломалось бы без них?  
> **RU:** Без них добавление метода в `Collection` (например, `stream()`) потребовало бы от каждой существующей реализации, включая чужой код, дописать этот метод — иначе не компилируется. `default` даёт тело сразу, и все реализации наследуют новый метод даром. Это механизм эволюции интерфейса без поломки обратной совместимости (JEP 126).  
> **EN:** Without them, adding a method to `Collection` (say `stream()`) would force every existing implementation, including third-party code, to write it — otherwise it won't compile. A `default` method ships a body, so all implementations inherit it for free. It's the mechanism for evolving an interface without breaking backward compatibility (JEP 126).
> 
> _Follow-up: «не размывает ли это „интерфейс = контракт"?»_ → Философски — да; поэтому Effective Java Item 21 советует применять их для эволюции API и удобных методов поверх примитивов, а не для бизнес-логики или состояния — состояния в интерфейсе всё равно нет.

> [!question]- В чём разница между `default` и `static` методом интерфейса?  
> **RU:** `default` наследуется реализаторами и виртуален (участвует в dynamic dispatch); `static` не наследуется и зовётся через имя интерфейса.  
> **EN:** A `default` method is inherited by implementers and is virtual (takes part in dynamic dispatch); a `static` method is not inherited and is called via the interface name.
> 
> _Follow-up: «зачем `static` в интерфейсе, если есть классы-утилиты?»_ → Чтобы фабрика/утилита жила на самом типе (`Comparator.comparing`, `List.of`), а не в отдельном `XxxUtils` — читаемее и не плодит компаньонов вроде `Collections`. _Trap: «может ли `default` переопределить `toString`?»_ → Нет, ошибка компиляции: `Object`-методы приходят от класса (class always wins), `default` был бы мёртвым.
