
Ты переносишь на Java рабочий кусок C#: базовый `Payment` с методом `process()` и наследник `CardPayment`, который его переопределяет. В C# ты бы пометил метод `virtual` на базе и `override` на наследнике — осознанно, руками. В Java ты убираешь эти ключевые слова (их тут нет), запускаешь вызов через ссылку базового типа — и он уходит в `CardPayment`. Без единого ключевого слова. Из этого одного дефолта растёт половина ловушек наследования, и большая часть собеседований по ООП крутится вокруг него.

> [!info] Ментальная модель  
> В Java каждый обычный метод (`non-static`, `non-private`, `non-final`) виртуален **по умолчанию**: какой код выполнится — решает реальный тип объекта в рантайме, а не тип ссылки. В C# зеркально — метод невиртуален, пока ты явно не написал `virtual`.

### `extends`, единичное наследование и `Object`-корень

Наследование — `extends` вместо `:`. Класс наследует ровно один родительский класс (единичное наследование, как в C#; множественное — только через интерфейсы, это урок 2).

```java
class CardPayment extends Payment { ... }
```

Ключевое, что стоит заякорить с самого начала: **каждый класс в Java неявно наследует `Object`, даже если `extends` нигде не написан.** Именно поэтому у любого объекта — даже вот такого пустого — уже есть методы:

```java
class Cat {}   // никакого extends

Cat c = new Cat();
c.getClass().getName();   // "Cat"          — getClass() от Object
c.toString();             // "Cat@120d6fe6" — дефолтный toString от Object (хэш свой на каждом запуске)
c instanceof Object;      // true           — всё в конечном счёте Object
```

`Object` — корень всей иерархии, как `object` в C#. Оттуда родом `toString()`, `equals()`, `hashCode()`, `getClass()` — те самые методы с контрактом `equals`/`hashCode`, который ты уже разбирал. Они есть у любого объекта, потому что любой объект в конечном счёте — `Object`.

> [!tip] Из C#  
> Один-в-один как `object`: `extends` ≈ `:`, единичное наследование как у тебя, `Object` ≈ `System.Object`. Расхождения начинаются со следующего раздела — и они серьёзные.

### Виртуальность по умолчанию — главный сдвиг из C#

Прямой факт: в Java любой обычный метод виртуален по умолчанию. Что реально вызовется — решает **фактический тип объекта в рантайме**, а не тип ссылки:

```java
Payment p = new CardPayment();   // тип ссылки — Payment, объект — CardPayment
p.process();                     // печатает: CardPayment.process
```

Ссылка объявлена как `Payment`, но объект в куче — `CardPayment`, и вызов уходит в `CardPayment.process()`. JVM смотрит не на тип переменной, а на реальный объект. Этот механизм — выбор реализации по фактическому типу объекта — называется **динамическим диспатчем** (dynamic dispatch). _Как_ именно JVM его делает (через таблицу методов объекта) — разберём в уроке 2; здесь важен сам дефолт: виртуально. Чтобы **запретить** переопределение, метод помечают `final` — это и есть Java-аналог C#-шного невиртуального метода.

> [!tip] Из C#  
> Самый важный контраст урока, держи перед глазами:
> 
> - C#: `void Process()` невиртуален. Нужны `virtual` на базе + `override` на наследнике. Забыл `virtual` → вызов через базовую ссылку уходит в базовый метод.
> - Java: `void process()` уже виртуален. Переопределение работает само. Чтобы **запретить** его — ставишь `final`.
> 
> Дефолты зеркальны: в C# по умолчанию закрыто, открываешь явно (`virtual`); в Java по умолчанию открыто, закрываешь явно (`final`/`private`/`static`).

### Overload vs override — два механизма на разных стадиях

Слова похожи, оба про «несколько методов с одним именем» — но это две принципиально разные вещи, и на собесе их постоянно смешивают в коде, чтобы поймать тебя.

**Overriding (переопределение)** — наследник даёт свою реализацию метода с **той же сигнатурой**. Какой выполнится — решает **реальный тип объекта в рантайме**. Это и есть виртуальность.

**Overloading (перегрузка)** — несколько методов с **одним именем, но разными параметрами**. Какой выберется — решает **компилятор по статическим типам аргументов** на этапе компиляции. В рантайме тут ничего не «выбирается» — выбор уже зашит в байткод.

```java
class Notifier {
    void send(String msg) { ... }
}
class EmailNotifier extends Notifier {
    @Override void send(String msg) { ... }   // override родительского send(String)
    void send(int code)  { ... }              // overload — новая сигнатура
}
```

```java
Notifier n = new EmailNotifier();
n.send("hi");   // OVERRIDE: реальный тип EmailNotifier → EmailNotifier.send(String)

EmailNotifier e = new EmailNotifier();
e.send("hi");   // OVERLOAD: компилятор выбрал send(String) по типу аргумента
e.send(42);     // OVERLOAD: компилятор выбрал send(int)
```

Тонкость, которая ловит: **набор перегрузок определяется статическим типом ссылки.** Напиши `n.send(42)`, где `n` объявлен как `Notifier` (а `send(int)` есть только у `EmailNotifier`) — код **не скомпилируется**. Компилятор видит только `Notifier.send(String)` и пытается впихнуть `int` в `String`:

```
error: incompatible types: int cannot be converted to String
```

То есть перегрузка выбирается из методов, доступных по **типу переменной**, а не по реальному объекту — ровно наоборот тому, как работает override. Вот эти две оси на одной картинке:

Ось, которую надо вбить намертво:

||Overriding|Overloading|
|---|---|---|
|что меняется|реализация, сигнатура та же|сигнатура (параметры) другая|
|кто решает|реальный тип объекта|статические типы аргументов|
|когда решает|в рантайме|на компиляции|

> [!tip] Из C#  
> Перегрузка у тебя такая же — компилятор выбирает по статическим типам. Детали приоритета местами расходятся (об этом ниже, в подводных камнях), но сама идея «overload решается на компиляции, override — в рантайме» переносится один-в-один.

### `@Override` — это проверка, а не выключатель

`@Override` в Java — **аннотация-проверка для компилятора**, а не ключевое слово, которое «включает» переопределение. Переопределение работает и без неё (виртуальность-то по умолчанию). Она лишь говорит компилятору: «я _намереваюсь_ переопределить родительский метод — проверь, что я и правда это делаю».

Это прямо противоположно C#, где `override` обязателен и сам по себе включает переопределение.

Зачем тогда она, если необязательна? Чтобы ловить **опечатки и рассогласование сигнатур** на компиляции:

```java
class Base    { void process() {} }
class Derived extends Base {
    @Override
    void proces() {}   // опечатка: proces
}
```

Компилятор валится сразу:

```
error: method does not override or implement a method from a supertype
    @Override
    ^
```

Без `@Override` это скомпилировалось бы как **новый независимый метод** `proces()` — и ты бы потом полдня искал, почему твоё «переопределение» не вызывается. Правило: **ставь `@Override` всегда, когда намереваешься переопределить.** Бесплатная страховка, которая превращает тихий рантайм-баг в явную ошибку компиляции.

> [!tip] Из C#  
> `@Override` ≈ не `override`, а скорее «компилятор, подтверди мою гипотезу». В C# `override` обязателен и включает переопределение сам. В Java виртуальность включает язык, а `@Override` — необязательный, но крайне рекомендованный детектор ошибок. (На методах, реализующих интерфейс, `@Override` разрешён с Java 6.)

> [!example] 🔬 Глубже / на синьора — виртуальность и перформанс  
> Первое, что спросят про «виртуально по умолчанию»: «это же медленнее?». Короткий ответ — на практике нет, и вот механизм.
> 
> Виртуальный вызов концептуально дороже прямого: нужно сходить в таблицу методов объекта (механику берём в уроке 2). Но HotSpot применяет **Class Hierarchy Analysis (CHA)**: если среди загруженных классов у метода ровно одна реализация — call site **мономорфный**, и JIT инлайнит его как прямой вызов, вставляя guard на случай, если позже подгрузится второй наследник (тогда — деоптимизация и перекомпиляция). То есть «виртуально в байткоде» ≠ «виртуально в машинном коде».
> 
> Отсюда следствие про `final`: прироста он почти не даёт. `final`/`private` делают call site тривиально мономорфным без CHA-анализа, но HotSpot и так справляется. `final` ставят ради **семантики** (запрет переопределения, контракт неизменности), а не ради скорости — иначе это преждевременная оптимизация.
> 
> Ещё глубже: HotSpot инлайнит и **биморфные** call sites (две реализации — через проверку типа), а **мегаморфные** (много реализаций) — уже нет, там полноценный диспатч.

> [!warning] ⚠️ Подводный камень 1 — `equals` перегружен вместо переопределён, коллекции молча теряют элементы  
> Самый дорогой баг этого урока. Сигнатура `Object.equals` — это `equals(Object)`. Если написать `equals(Point)`, это **перегрузка** (новый метод с другой сигнатурой), а не переопределение. Коллекции зовут `equals(Object)` — и попадают в дефолтный `Object.equals`, то есть в сравнение по ссылке.
> 
> java
> 
> ```java
> class Point {
>     final int x, y;
>     Point(int x, int y) { this.x = x; this.y = y; }
>     public boolean equals(Point o) {          // ← Point, не Object: ПЕРЕГРУЗКА
>         return o != null && x == o.x && y == o.y;
>     }
>     @Override public int hashCode() { return 31 * x + y; }
> }
> 
> Set<Point> set = new HashSet<>();
> set.add(new Point(1, 2));
> set.contains(new Point(1, 2));          // false (!) — а ты ждал true
> new Point(1, 2).equals(new Point(1, 2)); // true — прямой вызов работает
> ```
> 
> Самое злое: код компилируется, а прямой юнит-тест `a.equals(b)` **проходит** — там компилятор по статическому типу `Point` выберет твою перегрузку. А `HashSet`/`HashMap` в проде зовут `equals(Object)`, твоя перегрузка для них невидима — элементы теряются молча. `@Override` над `equals` ловит это на компиляции мгновенно (та же ошибка `does not override`). Правило: `equals` — всегда `(Object o)` и всегда с `@Override`.

> [!warning] ⚠️ Подводный камень 2 — overload resolution: `null`, widening → boxing → varargs  
> Перегрузка разрешается по статическим типам, и у компилятора строгий приоритет (JLS §15.12.2).
> 
> java
> 
> ```java
> static void f(Object o) {}
> static void f(String s) {}
> f(null);            // → f(String): самый специфичный применимый тип
> ```
> 
> `null` подходит и к `Object`, и к `String`, но `String` — подтип `Object`, значит «специфичнее», компилятор берёт его. **Но** если оба кандидата несравнимы (`f(String)` и `f(Integer)`) — ни один не специфичнее, и это ошибка компиляции:
> 
> ```
> error: reference to f is ambiguous
>   both method f(String) in ... and method f(Integer) in ... match
> ```
> 
> Разрешается явным кастом: `f((String) null)`.
> 
> Второй классический вопрос — три фазы разрешения:
> 
> java
> 
> ```java
> static void g(long x)      {}
> static void g(Integer x)   {}
> static void g(Object... x) {}
> int i = 5;
> g(i);               // → g(long): widening бьёт boxing бьёт varargs
> ```
> 
> Компилятор идёт по фазам: (1) без бокса/varargs — подходит `g(long)` через widening `int→long`, берёт; до (2) боксинга `int→Integer` и (3) varargs даже не доходит. Мантра: **widening → boxing → varargs**, останавливается на первой фазе, где есть кандидат.
> 
> 💡 Из C#: приоритет похож (точное совпадение → неявное расширение → …), но детали расходятся — особенно `null` между двумя reference-перегрузками: в Java это ошибка компиляции. Не переноси C#-интуицию дословно.

> [!warning] ⚠️ Подводный камень 3 — `@Override`-опечатка → тихий новый метод  
> Тот же класс багов, что и `equals(Point)`: без `@Override` опечатка в имени (`proces` вместо `process`) или лишний параметр даёт независимый метод, который никто не зовёт, а «переопределение» молча не работает. `@Override` превращает это в ошибку компиляции. Единственная причина иногда его _не_ ставить — если ты намеренно объявляешь новый метод, а не переопределяешь.

### На собесе

> [!question]- Q1. «Почему в Java методы виртуальны по умолчанию, а в C# — нет? Это же медленнее?»  
> **RU:** Осознанный дизайн-выбор: Java сделала ставку на полиморфизм как норму, а не исключение. «Медленнее» — миф: HotSpot через Class Hierarchy Analysis девиртуализирует мономорфные call sites и инлайнит их как прямые вызовы, вставляя guard на случай подгрузки второго наследника. «Виртуально в байткоде» ≠ «виртуально в машинном коде».  
> **EN:** It's a deliberate design choice — Java treats polymorphism as the default, not the exception. "Slower" is a myth: HotSpot uses Class Hierarchy Analysis to devirtualize monomorphic call sites and inline them as direct calls, with a guard in case a second subclass loads later. Virtual in bytecode doesn't mean virtual in machine code.  
> **Follow-up (trap):** «Тогда `final`/`private` дают прирост?» → Микроскопический и обычно не нужный: они делают call site тривиально мономорфным, но HotSpot и так справляется. `final` ставят ради семантики, не скорости.

> [!question]- Q2. «Overload или override — как различишь устно и по коду?»  
> **RU:** Override — та же сигнатура, выбор по **реальному типу объекта в рантайме** (это виртуальность). Overload — разные сигнатуры, выбор **компилятором по статическим типам аргументов** на компиляции. Разные оси: одна про реализацию и рантайм, другая про параметры и компиляцию.  
> **EN:** Override — same signature, resolved by the **actual object type at runtime** (that's virtual dispatch). Overload — different signatures, resolved by the **compiler from the static argument types** at compile time. Two different axes: one is about implementation at runtime, the other about parameters at compile time.  
> **Follow-up (trap):** «`Notifier n = new EmailNotifier(); n.send(42)` — что будет, если `send(int)` только у наследника?» → Не скомпилируется: набор перегрузок берётся по статическому типу `Notifier`, у которого только `send(String)`.

> [!question]- Q3. «`equals` переопределён или перегружен? Как отличишь, не запуская, и что сломается?»  
> **RU:** Смотрю на параметр. `equals(Object)` — переопределение. `equals(MyType)` — перегрузка, `Object.equals` не тронут. Коллекции (`HashMap`/`HashSet`/`contains`) зовут `equals(Object)` — если дал только `equals(MyType)`, для них он невидим, срабатывает сравнение по ссылке, и `set.contains(равныйОбъект)` вернёт `false`. Прямой `a.equals(b)` в тесте пройдёт (компилятор по статическому типу выберет перегрузку), а прод молча теряет элементы.  
> **EN:** I look at the parameter. `equals(Object)` overrides; `equals(MyType)` overloads and leaves `Object.equals` untouched. Collections call `equals(Object)`, so an `equals(MyType)`-only class silently falls back to reference equality and `set.contains(equalObject)` returns `false`. A direct `a.equals(b)` unit test passes (the compiler picks the overload by static type), while production quietly loses elements.  
> **Follow-up (trap):** «Как поймать на компиляции?» → `@Override` над `equals`: требует реального переопределения и валится с `does not override`, если сигнатура `equals(MyType)`.

> [!question]- Q4. «`f(null)` при `f(Object)`/`f(String)` — куда уйдёт? А при `f(String)`/`f(Integer)`?»  
> **RU:** В первом случае — в `f(String)`: `String` специфичнее `Object`. Во втором — ошибка компиляции `reference to f is ambiguous`: `String` и `Integer` несравнимы, ни один не специфичнее. Лечится кастом: `f((String) null)`.  
> **EN:** First case → `f(String)`, because `String` is more specific than `Object`. Second case → a compile error, `reference to f is ambiguous`: `String` and `Integer` are unrelated, neither is more specific. Fixed with a cast: `f((String) null)`.  
> **Follow-up:** «А `g(5)` при `g(long)`/`g(Integer)`/`g(Object...)`?» → `g(long)`: widening → boxing → varargs, останавливается на первой подходящей фазе.

#### На собесе — устная шпаргалка

Главный тезис, с которого начинай: **в Java каждый `non-static`, `non-private`, `non-final` метод виртуален по умолчанию.** В C# наоборот — невиртуально, пока не написал `virtual`. Дефолты зеркальны: C# открываешь явно, Java закрываешь явно (`final`/`private`/`static`).

Различай два механизма одной фразой: «overriding — та же сигнатура, выбор по **реальному типу объекта в рантайме**; overloading — разные сигнатуры, выбор компилятором по **статическим типам аргументов** на компиляции». И добавь тонкость, которая отличает middle от junior: набор перегрузок берётся по **типу ссылки**, а диспатч override — по **реальному объекту**. Это ровно противоположные оси.

`@Override` — не выключатель, а проверка: виртуальность работает и без неё, но она ловит опечатки и рассогласование сигнатур на компиляции. Её главный частный случай — `equals(Object)`: `equals(MyType)` тихо ломает коллекции, `@Override` превращает баг в ошибку компиляции.

Если спросят про перформанс виртуальности — «HotSpot девиртуализирует мономорфные call sites через CHA; `final` ради семантики, не скорости». Это снимает почти все follow-up'ы про производительность.
