
Это кульминация блока про интерфейсы. `default`-методы дали интерфейсам тела — а значит, класс может получить один и тот же метод от двух интерфейсов, и встаёт вопрос: чью версию брать? В отличие от множественного наследования классов (запрещённого из-за конфликта _состояния_, урок 4.6), здесь конфликт _поведения_ — и он разрешим. Java решает его тремя правилами резолюции, которые компилятор проверяет по приоритету; если ни одно не выбирает однозначно — он отказывается решать сам и требует, чтобы ты переопределил метод. Механика контринтуитивная, поэтому всё прогнано кодом.

> **Ментальная модель.** Компилятор пробует три правила по приоритету: (1) конкретный метод класса бьёт любой `default` — молча; (2) под-интерфейс бьёт супер-интерфейс; (3) иначе — ambiguous, и ты обязан переопределить и явно выбрать через `Interface.super.method()`.

Почему компилятор не выбирает сам: `Walker.move()` и `Swimmer.move()` равноправны — нет ни одной причины предпочесть одну реализацию другой. В C++ это решали порядком наследования и виртуальными базами (источник боли); Java сказала проще: «не знаю — решай ты». Но прежде чем сдаться, компилятор проверяет три правила по приоритету ([JLS §9.4.1](https://docs.oracle.com/javase/specs/jls/se21/html/jls-9.html#jls-9.4.1)).

### Правило 1: class always wins

Конкретный метод из иерархии **классов** всегда бьёт любой `default` из интерфейса — молча, без ошибки:

```java
interface IHi { default String hi() { return "from interface"; } }
class Base    { public String hi()  { return "from class"; } }
class Child extends Base implements IHi { }

new Child().hi();   // "from class"
```

> [!example] Прогон на JDK 21  
> `new Child().hi()` печатает `from class`. `Child` получил `hi()` и от `Base` (конкретный), и от `IHi` (default) — побеждает класс, компилятор даже не предупреждает.

> [!warning] Самая злая ловушка интерфейсов  
> Ты добавил `IHi` к `Child`, ожидая поведение из интерфейса, а в `Base` (возможно, чужом, в библиотеке) уже есть `hi()` — и твой `default` **никогда не вызовется**. Ни ошибки, ни ворнинга. Логика: интерфейс описывает контракт «снаружи», класс — реальную реализацию; реализация всегда конкретнее обещания. Это та же причина, по которой `default` **не может переопределить методы `Object`**: `Object` — класс, class-wins зашит, поэтому язык запрещает даже писать такой `default` (ошибка из урока 4.7). Как не наступить: ставь `@Override` на `default`'ах, которые ожидаешь увидеть, — компилятор подтвердит, что метод реально оттуда.

### Правило 2: более специфичный интерфейс побеждает

Если один интерфейс наследует другой и оба дают `default` — побеждает **под-интерфейс** (он специфичнее), автоматически:

```java
interface Animal { default String sound() { return "generic"; } }
interface Dog extends Animal { @Override default String sound() { return "woof"; } }
class Puppy implements Animal, Dog { }

new Puppy().sound();   // "woof" — Dog специфичнее Animal
```

> [!example] Прогон на JDK 21  
> `new Puppy().sound()` печатает `woof`. Это не настоящий diamond: `Dog` уже «переопределил» `sound()` относительно `Animal`, и `Puppy` видит более конкретную версию — аналогия с обычным override, только по линии интерфейсов.

### Правило 3: иначе — ambiguous, компилятор требует override

Два **несвязанных** интерфейса (ни один не специфичнее), одинаковая сигнатура `default` — правила 1 и 2 не сработали → конфликт, ошибка компиляции:

> [!example] Прогон на JDK 21 — ambiguous
> 
> java
> 
> ```java
> interface Walker  { default String move() { return "walk"; } }
> interface Swimmer { default String move() { return "swim"; } }
> class Amphibian implements Walker, Swimmer { }
> ```
> 
> ```
> error: types Walker and Swimmer are incompatible;
>   class Amphibian inherits unrelated defaults for move() from types Walker and Swimmer
> ```

Разрулить можно только вручную — переопределить и явно выбрать через `ИмяИнтерфейса.super.method()`:

```java
class Amphibian implements Walker, Swimmer {
    @Override public String move() {
        return Walker.super.move() + "+" + Swimmer.super.move();  // "walk+swim"
    }
}
```

`Walker.super.move()` — новый синтаксис (квалифицированный `super` интерфейса), проверил: печатает `walk+swim`. Обычный `super.move()` тут **не** сработал бы — он адресует суперкласс, а не интерфейс (`error: cannot find symbol`). `super.super` по-прежнему нет — только на один уровень.

> [!tip] Из C#  
> В C# 8 DIM diamond разруливается иначе: реализатор тоже обязан вмешаться при неоднозначности, но `default` доступен только через ссылку типа интерфейса (`((Walker)this).Move()`-стиль), а не через `Interface.super`. Ключевое отличие модели: в Java конфликт ловится, потому что `default`-методы реально _входят_ в класс и сталкиваются там; в C# они живут «при интерфейсе», и столкновение разрешается на уровне приведения к интерфейсу. Твой .NET-навык «приведи к нужному интерфейсу» в Java заменяется на «`@Override` + `Interface.super`».

### 🔬 Глубже / на синьора

> [!info]- Ещё два конфликта + байткод `invokeinterface`  
> **`default` + `abstract` одной сигнатуры из двух интерфейсов** → реализатор **обязан** переопределить. `interface HasDefault { default String m(){...} }` + `interface HasAbstract { String m(); }` + `class Both implements HasDefault, HasAbstract { }`:
> 
> ```
> error: Both is not abstract and does not override abstract method m() in HasAbstract
> ```
> 
> Абстрактный «отравляет» дефолт: наличие `abstract` означает «контракт под вопросом», и компилятор не разрешает молча взять `default`.
> 
> **Одноимённые константы из двух интерфейсов** → голое имя не компилируется. `interface ConfigA { int LIMIT = 10; }` + `interface ConfigB { int LIMIT = 20; }`, внутри реализатора `return LIMIT;`:
> 
> ```
> error: reference to LIMIT is ambiguous
> ```
> 
> Лечится квалификацией: `ConfigA.LIMIT`. Ещё один аргумент против «constant interface» (урок 4.6).
> 
> **Байткод (`invokeinterface` — theater на Middle+, но приятный бонус).** Проверил `javap`: вызов через ссылку _класса_ компилируется в `invokevirtual`, через ссылку _интерфейса_ — в `invokeinterface`. Зачем два опкода: при одиночном наследовании классов у метода фиксированный индекс слота в `vtable`, и `invokevirtual` — быстрый доступ по индексу. А класс реализует _много_ интерфейсов, единого фиксированного индекса для интерфейсного метода нет, поэтому JVM ищет его в отдельной таблице на класс (`itable`) — исторически чуть дороже. Но JIT девиртуализует мономорфные/биморфные interface-call-сайты так же, как обычные (CHA-девиртуализация), так что в горячем коде разница ничтожна. Оптимизировать под это не нужно — знать, что вызовы через интерфейс виртуальны, обязательно; знать про `itable` — бонус, который покажет глубину.

### ⚠️ Подводные камни

> [!warning] Резолюция diamond — что ловит новичка
> 
> - **class-wins срабатывает молча** — самая опасная ловушка: `default` из интерфейса тихо проигрывает методу суперкласса, даже если суперкласс чужой. Ни ошибки, ни ворнинга.
> - **ambiguous ловится на компиляции, а не в рантайме** — `inherits unrelated defaults` не даст собрать проект. Это как раз хорошо: конфликт виден сразу.
> - **`super.move()` не достаёт до интерфейса** — нужен квалифицированный `Walker.super.move()`. Обычный `super` — это суперкласс.
> - **`default` + `abstract` из двух интерфейсов** — `abstract` побеждает в смысле «обязан переопределить», не наследуется `default` молча.
> - **Одноимённые константы** из двух интерфейсов → `reference to X is ambiguous`, лечится квалификацией.

### На собесе

> [!question]- Два интерфейса, одинаковый `default`-метод — что скомпилируется / напечатает?  
> **RU:** Не скомпилируется: `inherits unrelated defaults`. Компилятор не выбирает сам. Разруливаю через `@Override` метода и явный вызов `Walker.super.move()`.  
> **EN:** It won't compile: `inherits unrelated defaults`. The compiler refuses to choose. I resolve it by `@Override`-ing the method and explicitly calling `Walker.super.move()`.
> 
> _Follow-up: «а если один из двух — класс, другой — интерфейс?»_ → Тогда компилируется без ошибки и молча побеждает класс (правило 1, class always wins); `default` интерфейса не вызовется. _Follow-up: «а если под-интерфейс и супер-интерфейс?»_ → Побеждает под-интерфейс (правило 2), тоже автоматически.

> [!question]- Может ли `default`-метод переопределить `toString`? Почему?  
> **RU:** Нет, ошибка компиляции. Потому что class always wins (JLS §9.4.1): методы `Object` приходят от класса и всегда перебьют `default` — такой `default` был бы мёртвым кодом, и язык запрещает его писать.  
> **EN:** No — it's a compile error. Because class always wins (JLS §9.4.1): `Object`'s methods come from the class and always beat a `default`, so such a `default` would be dead code, and the language forbids writing it.
> 
> _Follow-up: «в чём смысл class-wins, не опаснее ли молча игнорировать интерфейс?»_ → Да, это ловушка, но альтернатива хуже: если бы `default` мог незаметно перебить метод суперкласса, добавление метода в интерфейс (а их добавляют ради эволюции, как `Collection.stream`) могло бы тихо сломать существующие классы. Меньшее зло — отдать приоритет реальной реализации класса.
