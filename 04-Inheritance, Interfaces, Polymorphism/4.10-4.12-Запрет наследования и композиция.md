Ты уже прошёл весь механизм наследования — виртуальные вызовы, `super`, диспетчеризацию. Этот урок отвечает на вопрос, который логически идёт следующим и который постоянно всплывает на собесах: а когда наследование _не_ нужно, и как его вообще запретить. Оказывается, самые используемые классы стандартной библиотеки — `String`, обёртки, `LocalDate` — специально закрыты к наследованию, и за этим стоит цельная инженерная позиция. Разберём три инструмента закрытости (`final`, `sealed`, неявный `final` у `record`/`enum`), поймём на запускаемом примере, почему наследование в принципе хрупкое (корректный на вид код молча считает вдвое), и выведем главный дизайн-принцип всего блока.

**Mental model.** Наследование — самая жёсткая связь в ООП: подкласс цепляется не только за _контракт_ базы, но и за _детали её реализации_. Поэтому дефолт зрелого инженера — закрыть класс (`final`) и переиспользовать через композицию, а `extends` открывать только намеренно, задокументированно и при честном `is-a`.

Честно про формат: это **дизайн-урок**. Новой JVM-механики тут почти нет — `final` ты знаешь как «выключатель виртуальности» из урока про полиморфизм, `sealed` это compile-time проверка плюс флажок в class-файле. Глубина здесь не в байткоде, а в дизайн-обосновании: когда запрещать наследование и почему обёртка устойчивее подкласса.

---

#### 1. `final` — запрет наследования

Попробуй унаследоваться от `String`:

```java
class MyString extends String { }   // не компилируется
```

```
error: cannot inherit from final String
```

Это не баг JDK, а сознательное решение. Посмотри на самые ходовые классы стандартной библиотеки — `String`, все обёртки (`Integer`, `Long`, `Boolean`…), `LocalDate`, `LocalDateTime`. Все `final`, закрыты намертво. Почему именно их — разберём в слое 3.

Рамка, которую держи весь урок: **наследование мощно, но не бесплатно.** Первый вопрос дизайнера API — не «как сделать класс расширяемым», а «а надо ли вообще открывать его к наследованию». В Java по умолчанию класс открыт (любой может `extends`), и `final` — способ сказать «этот закрыт».

**`final` class — нельзя `extends`:**

```java
public final class Money {
    private final long cents;
    public Money(long cents) { this.cents = cents; }
    // никто не может extends Money
}
```

**`final` method — нельзя `override`.** Можно запереть не весь класс, а один метод: класс остаётся открытым, но конкретный метод переопределить нельзя.

```java
public class PaymentProcessor {          // класс открыт
    public final void process() {        // но этот метод залочен
        validate();
        charge();
        notifyUser();
    }
    protected void validate()   { /* подкласс может переопределить */ }
    protected void charge()     { /* и это */ }
    protected void notifyUser() { /* и это */ }
}
```

Это классический **template method**: ты фиксируешь _скелет алгоритма_ (`process` — порядок шагов незыблем), но даёшь подклассу переопределить _отдельные шаги_. Подкласс не может сломать порядок, только наполнить шаги. Проверил на JDK 21 — подкласс с `@Override protected charge()` меняет только шаг, `process()` остаётся неизменным.

**`record` (Java 16, JEP 395) и `enum` — неявно `final`.** Ты этого не пишешь, язык делает сам:

```java
public record Point(int x, int y) { }
class Point3D extends Point { }   // error: cannot inherit from final Point
```

Для `enum` компилятор ещё явнее: `error: enum classes are not extensible`. Это другой подход к закрытости — не «дизайнер решил запретить», а «сама конструкция языка не предполагает наследования». Контракт `record` (`equals`/`hashCode`/`toString` по компонентам) держится ровно потому, что нельзя подсунуть подкласс с лишним полем.

> [!tip] 💡 Из C#  
> `final class` — это ровно твой `sealed class`. `public sealed class Money` (C#) = `public final class Money` (Java), один в один.
> 
> **Языковая ловушка:** слово `sealed` в Java значит _совсем другое_ (третье состояние, слой 2). C#-шный `sealed` маппится на Java `final`, а не на Java `sealed` — не дай слову себя запутать.
> 
> `final method` ≈ C# `sealed override` или просто невиртуальный метод. И тут фундаментальный контраст из урока про полиморфизм: в C# метод невиртуален _по умолчанию_, `virtual` пишешь осознанно; в Java наоборот — **всё виртуально по умолчанию**, а `final` нужен, чтобы виртуальность _выключить_. Дефолты противоположны: C# «закрыт-по-умолчанию, открываешь явно», Java «открыт-по-умолчанию, закрываешь явно».

> [!warning] ⚠️ Подводный камень  
> Не ставь `final` на метод _ради скорости_. Да, `final`-метод тривиально девиртуализуется — но, как разбирали в уроке про полиморфизм, JIT делает это и так через CHA (class hierarchy analysis), если в рантайме у метода один реализатор. `final` ставят ради **семантики** («этот шаг менять нельзя»), а не ради перформанса. Перформанс тут — побочный эффект, на который опираться не надо.

#### 2. `sealed` — третье состояние (light)

До сих пор у класса было два состояния: **открыт** (любой может `extends`) или **`final`** (никто). `sealed` (Java 17, JEP 409) добавляет третье: **открыт, но только для перечисленного списка наследников.**

```java
public sealed interface Shape permits Circle, Rectangle, Triangle { }

public record Circle(double radius)                implements Shape { }
public record Rectangle(double width, double height) implements Shape { }
public record Triangle(double base, double height)   implements Shape { }
```

`permits` — белый список. Унаследоваться от `Shape` может _только_ эта тройка; любой четвёртый класс — ошибка компиляции. Не «никому» (как `final`), не «всем» (как открытый), а «вот этим троим».

Зачем — одной фразой: **контролируемая иерархия.** Ты как автор `Shape` гарантируешь, что других вариантов не появится. Это включает мощную штуку — exhaustive `switch` (мостик к уроку про pattern matching): компилятор _знает_, что вариантов ровно три, и проверяет, что ты обработал все, без `default` (проверено на JDK 21, компилируется и печатает `12.00`):

```java
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.width() * r.height();
    case Triangle t  -> 0.5 * t.base() * t.height();
    // default не нужен — компилятор видит, что покрыты все permits
};
```

Добавишь `Square` в `permits`, но не допишешь ветку — `switch` перестанет компилироваться: `error: the switch expression does not cover all possible input values`. Компилятор ловит дыру за тебя (тоже проверено).

> [!tip] 💡 Из C#  
> Прямого аналога в C# **нет**. C#-шный `sealed` — это полный запрет (= Java `final`), а «открыт, но только для списка X, Y, Z» в C# языком не выражается. Одна из вещей, где Java ушла дальше: иерархию можно ограничить, не закрывая полностью.

> [!warning] ⚠️ Подводный камень  
> Каждый класс из `permits` обязан сам объявить свою «закрытость» — быть `final`, `sealed` или `non-sealed`. Компилятор это _требует_: оставишь наследника открытым по умолчанию — `error: sealed, non-sealed or final modifiers expected`. `non-sealed` — это явное «а я снова открываюсь всем», разрыв в цепочке.

Намеренно мелко: полная сила `sealed` раскрывается в связке `sealed` + `record` + pattern matching — это алгебраические типы данных (ADT), функциональный подход к моделированию. Целиком — **глава 17**. Здесь тебе достаточно: `sealed` = «контролируемая иерархия», третье состояние между `final` и открытым. Эволюция для протокола: `final` — с Java 1.0; неявный `final` у `record` — Java 16; `sealed` стандарт — Java 17 (preview в 15/16, JEP 360/397).

#### 3. Почему наследование хрупкое (ядро)

Самый важный слой. Прямой ответ: **наследование нарушает инкапсуляцию.** Подкласс зависит не только от _контракта_ суперкласса (что методы делают), но и от _деталей реализации_ (как они устроены внутри, какой метод зовёт какой). А детали реализации — это то, что автор суперкласса считает своим и вправе менять. Меняет — подкласс ломается молча. Это [Effective Java, Item 18](https://www.oreilly.com/library/view/effective-java/9780134686097/) «Favor composition over inheritance».

Смотри канонический пример. Хотим `Set`, который считает, сколько элементов в него добавляли. Наследуемся от `HashSet`, переопределяем `add` и `addAll`:

```java
class Counter extends HashSet<String> {
    int addCount = 0;

    @Override public boolean add(String e) {
        addCount++;
        return super.add(e);
    }
    @Override public boolean addAll(Collection<? extends String> c) {
        addCount += c.size();
        return super.addAll(c);
    }
}

Counter s = new Counter();
s.addAll(List.of("a", "b", "c"));
System.out.println(s.addCount);   // печатает 6, а не 3
```

Код выглядит безупречно. Добавили 3 элемента через `addAll` — а счётчик показал **6** (запустил, ровно 6). Почему.

Внутри `HashSet` метод `addAll` унаследован от `AbstractCollection` (сам `HashSet` его не переопределяет — проверил через `javap`), а `AbstractCollection.addAll` бежит по коллекции и для каждого элемента зовёт `add(e)`. Вот ключевой момент — это **self-use**, внутренний вызов одного своего метода из другого. И поскольку в Java всё виртуально, этот внутренний `add` резолвится не в `HashSet.add`, а в **твой** `Counter.add`. Каждый из трёх внутренних вызовов щёлкает счётчик.

Итог: твой `addAll` насчитал `+3` (своё), плюс `super.addAll` через self-call дёрнул твой `add` ещё 3 раза (`+3`). Всего 6.

![[Pasted image 20260707133221.png]]

Капкан в том, что **ты не мог этого знать из контракта `HashSet`.** В Javadoc `addAll` нигде не написано «я внутри зову `add`». Это деталь реализации. Завтра авторы JDK перепишут `addAll`, чтобы он не звал `add` (ради скорости) — и твой счётчик начнёт показывать 3. Код ты не трогал, а поведение поплыло. Ты прицепился к тому, _как_ суперкласс работает внутри, а не к тому, _что_ он обещает. Вот почему это называют **fragile base class**.

> [!tip] 💡 Из C#  
> **В C# этого бага просто не было бы.** Методы `HashSet<T>` не `virtual`, поэтому твой «override» был бы на самом деле `new` (сокрытие, hiding), а внутренний вызов из базового `AddRange`/`Add` статически связался бы с методом `HashSet`, не с твоим. Счётчик показал бы 3. Этот пример — буквально иллюстрация «в Java всё виртуально по умолчанию»: тут дефолт бьёт тебя по рукам. (В C# отдельная тонкость — у `HashSet<T>` вообще нет `AddRange`, но суть та же: невиртуальность → никакого сюрприза с self-call.)

#### 4. Композиция + forwarding

Чинится это не патчем наследования, а сменой подхода. Вместо «_быть_ `HashSet`ом» (`extends`) мы будем «_иметь_ `Set` внутри» и оборачивать его. Два термина явно:

- **Wrapper (обёртка)** — класс, который держит другой объект полем и снаружи выглядит как он, но добавляет своё поведение.
- **Forwarding / delegation** — обёртка не делает работу сама, а перекидывает каждый вызов внутреннему объекту (`delegate.add(e)`). Forwarding-метод — однострочник, тупо зовущий тот же метод у компонента.

```java
public class InstrumentedSet<E> implements Set<E> {
    private final Set<E> delegate;    // держим компонент, а не наследуем
    private int addCount = 0;

    public InstrumentedSet(Set<E> delegate) { this.delegate = delegate; }

    @Override public boolean add(E e) {
        addCount++;
        return delegate.add(e);
    }
    @Override public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return delegate.addAll(c);
    }
    public int getAddCount() { return addCount; }

    // size(), contains(), iterator(), remove()… — forwarding в delegate
    @Override public int size() { return delegate.size(); }
    @Override public boolean contains(Object o) { return delegate.contains(o); }
    // …остальные ~13 методов Set форвардятся так же
}
```

Запустил — `addCount == 3` (правильно). И тот же код оборачивает _любой_ `Set`: `new InstrumentedSet<>(new TreeSet<>())` дал 2 из двух добавленных. Вот почему теперь не двоит.

![[Pasted image 20260707133238.png]]

Вся разница — в одной фразе:

- **Наследование:** один объект. `this` внутри кода `HashSet` — это ты. Его внутренний `add` залетает в твой override. Ловушка.
- **Композиция:** два объекта. `this` внутри `delegate` — это сам `delegate`. Он физически не имеет ссылки на тебя, поэтому достать твои методы не может. Стенка.

Ты считаешь ровно на своей границе — в своих `add`/`addAll`, по одному разу. Что `delegate` крутит у себя внутри — не твоё дело. Обёртка зависит **только от публичного контракта `Set`**, а не от кишок `HashSet`. Перепишут `HashSet` в следующем релизе — тебе всё равно. Обрати внимание: явный форвард `delegate.addAll(c)` границу пересекает (это ты его позвал), а неявный `this.add` внутри `delegate` — нет.

> [!tip] 💡 Из C#  
> Паттерн межъязыковой: в .NET ты так же держишь поле `ISet<T>`, пишешь `implements`/`: ISet<T>` и пробрасываешь вызовы. Это **Decorator** (он же делегирование) — мостик к **главе 25**, где `BufferedInputStream` оборачивает любой `InputStream` и добавляет буферизацию. Узнавай его в лицо.

> [!warning] ⚠️ Цена честно  
> Forwarding-методов много — у `Set` штук 15, и каждый надо прокинуть руками. Наследование давало их даром (`HashSet` уже всё реализовал), тут ты пишешь пятнадцать однострочников. Это реальный минус композиции. В проде боль снимают двумя способами: вынести форвардинг в переиспользуемый `ForwardingSet implements Set<E>` (пишешь скучный форвардер один раз, а `InstrumentedSet extends ForwardingSet` добавляет только интересное), либо `Lombok @Delegate` — одной строкой генерит форвардеры (**глава 20**). Детали там, не сейчас.

**Когда наследование всё-таки оправдано (чек-лист Item 19).** Наследование не запрещено — оно требует _работы_. Если осознанно делаешь класс для расширения:

1. **Задокументируй self-use** — в Javadoc прямо напиши, какие методы изнутри зовут какие переопределяемые. Без этого подкласс гадает (как мы гадали про `HashSet.addAll`). Именно поэтому в Javadoc JDK встречается фраза «This implementation…» — это и есть документирование self-use.
2. **`protected`-хуки — только осознанно.** Каждый `protected`-член — публичное обязательство на годы.
3. **Не зови переопределяемые методы из конструктора** (урок про инициализацию: диспатч уже «производный», а поля подкласса ещё в дефолте).
4. **Протестируй, написав подкласс сам** — до релиза. Часто вскрывает, что хук не тот.

А если ресурса на всё это нет — **«or else prohibit it»**: закрой класс (`final` либо приватный конструктор + фабрика). Именно поэтому `String`, обёртки, `LocalDate` — `final`: их авторы выбрали «запретить», а не «документировать навечно».

> [!example] 🔬 Глубже / на синьора  
> **Где физически сидит self-use.** `HashSet` не переопределяет `addAll` — он наследует его от `AbstractCollection`, чей `addAll(c)` буквально `for (E e : c) if (add(e)) modified = true;`. Вызов `add(e)` компилируется в `invokevirtual` и на рантайме идёт через vtable объекта — а объект-то твой `Counter`. Отсюда и двойной счёт. Это не хак JDK, а прямое следствие того, что `AbstractCollection` спроектирован как skeletal-класс _для наследования_ и завязан на собственный `add`. Ирония: `AbstractSet`/`AbstractCollection` наследовать как раз нормально — они это документируют. `ArrayList`/`HashSet` — нет.
> 
> **Почему интерфейсы от этого не страдают.** У интерфейса нет деталей реализации, к которым можно прицепиться, — он отдаёт только контракт. Поэтому `implements` и `interface extends interface` fragile base class не болеют. Это ещё один аргумент за «prefer interfaces».
> 
> **Реальный антипример из JDK — `Stack extends Vector`.** `java.util.Stack` унаследовал `Vector` просто потому, что было удобно переиспользовать массив, — а не потому, что стек _is-a_ вектор. В итоге `Stack` торчит наружу всеми методами `Vector`. Проверил: `stack.add(0, "SNUCK-IN")` на стеке `[a, b, c]` даёт `[SNUCK-IN, a, b, c]` — вставили в «середину стека», `peek()` при этом всё ещё `c`. Ложная `is-a`, которую теперь нельзя убрать из-за обратной совместимости. Через композицию (`Stack` _имеет_ массив) дырки бы не было.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`extends ArrayList`/`HashSet` через границу пакета** — почти всегда мина: завязываешься на недокументированный self-use. Канон — `InstrumentedHashSet` выше.
> 2. **Наследование ради переиспользования кода без `is-a`** — самая частая причина кривых иерархий. Желание переиспользовать реализацию не даёт права на `extends`.
> 3. **`final` на методе ради скорости** — нет, ради семантики (см. слой 1).
> 4. **`sealed`-наследник забыл `final`/`sealed`/`non-sealed`** — не компилируется, компилятор требует явного модификатора.
> 5. **Путать Java `sealed` (ограниченная иерархия) с C# `sealed` (полный запрет = Java `final`)** — разные вещи с одним словом.

И смысловой замок всего блока наследования: **наследование — не дефолт.** Дефолт инженера — `final` + композиция. Открываешь класс к наследованию только намеренно, только задокументированно и только когда `is-a` действительно держится, а не «просто удобно переиспользовать код».

#### На собесе

> [!question]- 1. Почему `String`, `Integer`, `LocalDate` сделаны `final`?  
> Чтобы гарантировать инвариант, на котором держится всё остальное, — для них это **иммутабельность**. Будь `String` наследуемым, кто-то написал бы `MutableString extends String` с изменяемым полем, и весь код, доверяющий «строка не меняется» (ключи `HashMap`, кэш `hashCode`, проверки безопасности), поехал бы.
> 
> **RU:** Их закрыли, чтобы никто не подорвал иммутабельность подклассом с mutable-состоянием. На неизменяемости строки завязаны кэш `hashCode`, работа как ключа в `HashMap` и security-проверки путей/SQL.
> 
> **EN:** They're `final` so no subclass can break immutability by adding mutable state. String immutability underpins `hashCode` caching, its use as a `HashMap` key, and security checks over paths and SQL.
> 
> _Follow:_ «что конкретно сломалось бы?» — контракт `equals`/`hashCode`: мутируемая строка-ключ потеряется в `HashMap`; плюс TOCTOU-дыра, если значение меняют после проверки доступа. _Follow:_ «а `record` почему неявно `final`?» — по той же логике: его `equals`/`hashCode`/`toString` считаются по компонентам, подкласс с лишним полем сломал бы это.

> [!question]- 2. Что не так с `extends ArrayList` / `HashSet`?  
> Ты наследуешь конкретный класс через границу пакета и завязываешься на его недокументированный self-use.
> 
> **RU:** Проблема в скрытом self-use: `HashSet.addAll` внутри сам зовёт `add`, а из-за виртуальности это мой переопределённый `add`. Переопределяю оба со счётчиком, добавляю 3 — счётчик показывает 6. Код выглядит правильным, но хрупок: суперкласс меняет реализацию — я ломаюсь молча.
> 
> **EN:** The trap is hidden self-use: `HashSet.addAll` calls `add` internally, and by virtual dispatch that's my overridden `add`. Overriding both with a counter and adding 3 elements yields 6. The code looks correct but is fragile — if the superclass changes its implementation, I break silently.
> 
> _Follow:_ «как починить?» — композиция: `implements Set`, держу `Set` полем, форвардю; self-call компонента идёт на его собственные методы, счётчик снова 3. _Follow:_ «а если очень надо наследоваться от коллекции?» — ок, если это _твоя_ коллекция в _твоём_ пакете, или есть `AbstractList`/`AbstractSet` — они _спроектированы_ для наследования и документируют, что реализовать.

> [!question]- 3. Композиция или наследование — что выберешь и почему?  
> По умолчанию композиция; наследование — только при железном `is-a` И когда суперкласс спроектирован для расширения.
> 
> **RU:** Беру наследование при истинном `is-a`, когда готов жёстко связать подкласс с реализацией базы; композицию — при `has-a` или когда собираю поведение из независимых частей. Дефолт — композиция: нет fragile base class, легче тестировать, вложенный объект можно подменить в рантайме, тогда как наследование фиксирует связь на этапе компиляции.
> 
> **EN:** Inheritance for a genuine `is-a` when I'm willing to couple the subclass tightly to the base; composition for `has-a` or when assembling behavior from independent parts. I default to composition: no fragile base class, easier testing, the contained object is swappable at runtime, whereas inheritance fixes the link at compile time.
> 
> _Follow:_ «критерий?» — `is-a` vs `has-a`: `Stack` не _is-a_ `Vector` (стек _имеет_ хранилище) — потому `Stack extends Vector` ошибка. _Follow:_ «надо переиспользовать код, но это не `is-a`?» — тогда композиция, всегда. _Follow:_ «цена композиции?» — forwarding-бойлерплейт, снимается `ForwardingX` или `@Delegate`.

> [!question]- 4. Что значит «спроектировать класс для наследования»?  
> Задокументировать self-use, дать `protected`-хуки осознанно, не звать overridable из конструктора, протестировать написанием подкласса (Item 19).
> 
> **RU:** Это значит явно описать в Javadoc, какие методы зовут переопределяемые (self-use), аккуратно выставить `protected`-хуки, не вызывать overridable-методы из конструктора и проверить дизайн, написав подкласс до релиза.
> 
> **EN:** It means documenting self-use in Javadoc (which methods call overridable ones), exposing `protected` hooks deliberately, never calling overridable methods from a constructor, and validating the design by writing a subclass before release.
> 
> _Follow:_ «почему “or else prohibit it”?» — незадокументированный наследуемый класс это обещание, которое ты не давал, но которое тебя свяжет: подклассы прицепятся к текущей реализации, и менять её ты больше не сможешь. Либо берёшь обязательство явно, либо закрываешь класс `final` и остаёшься свободен.

> [!question]- 5. «Наследование нарушает инкапсуляцию» — как именно?  
> Подкласс зависит не от контракта суперкласса, а от деталей реализации — какие методы тот зовёт изнутри. Эти детали суперкласс вправе менять, и тогда подкласс ломается молча.
> 
> **RU:** Инкапсуляция про то, что реализация скрыта и может меняться. Наследование её пробивает: подкласс завязывается на _как_ работает база (её self-use), а не только на _что_ она обещает. Меняется скрытая деталь — рушится подкласс. Пример — `InstrumentedHashSet`.
> 
> **EN:** Encapsulation means the implementation is hidden and free to change. Inheritance pierces it: the subclass depends on _how_ the base works (its self-use), not just _what_ it promises. Change a hidden detail and the subclass breaks — the `InstrumentedHashSet` case.
> 
> _Follow:_ «относится ли это к наследованию интерфейсов?» — нет: у интерфейса нет деталей реализации, только контракт, поэтому `implements` от fragile base class не страдает.

> [!question]- 6. Зачем `sealed`, если есть `final`?  
> Разные задачи. `final` = ноль наследников. `sealed` = наследники только из `permits`-списка: «контролируемая иерархия», а не «закрытая».
> 
> **RU:** `final` запрещает наследование полностью; `sealed` разрешает его строго перечисленным типам. Смысл ограничения — компилятор знает все варианты, и это даёт exhaustive `switch`/pattern matching без `default`: добавил тип в `permits`, не обработал — не компилируется.
> 
> **EN:** `final` forbids inheritance entirely; `sealed` permits it only for a listed set of types. The point of the whitelist is that the compiler knows every case, enabling exhaustive `switch`/pattern matching with no `default`: add a type to `permits` and fail to handle it, and it won't compile.
> 
> _Follow:_ «аналог в C#?» — нет: там `sealed` = полный запрет (= Java `final`), `permits` не выражается. _Follow:_ «полная сила?» — `sealed` + `record` + pattern matching = алгебраические типы, **глава 17**.

**Минутный нарратив (главное, что надо уметь рассказать за минуту).** «Наследование нарушает инкапсуляцию — подкласс цепляется к деталям реализации суперкласса, а не к контракту. Канон: `InstrumentedHashSet extends HashSet`, переопределяю `add` и `addAll` со счётчиком; `HashSet.addAll` (унаследованный от `AbstractCollection`) внутри сам зовёт `add`, а из-за виртуальности это мой переопределённый `add` — добавляю 3 элемента, счётчик показывает 6. Это fragile base class. Чиню композицией: `implements Set`, держу `Set` полем, форвардю вызовы; обёртка зависит только от публичного контракта, поэтому self-call компонента её не задевает — счётчик снова 3. Цена — forwarding-бойлерплейт, снимается `ForwardingSet` или `@Delegate`. Design and document for inheritance, or else prohibit it».

**Оральная шпаргалка:**

- `final` класс — «запрет наследования; в C# это `sealed class`; так закрыты `String`, обёртки, `LocalDate` ради иммутабельности; `record`/`enum` неявно `final`».
- `final` метод — «запрет переопределения; фиксирую инвариантный шаг, классика — template method; ставлю ради семантики, не скорости — JIT девиртуализует и так через CHA».
- `is-a` — «`extends` только при честном is-a; переиспользование кода без is-a — это композиция; `Stack extends Vector` — канонический пример ложной is-a».
- Дизайн-вывод как позиция — «наследование не дефолт; дефолт — `final` плюс композиция, а `extends` открываю только намеренно и задокументированно». Это и отличает мидла, который «умеет `extends`», от мидла, который понимает, _когда не надо_.
