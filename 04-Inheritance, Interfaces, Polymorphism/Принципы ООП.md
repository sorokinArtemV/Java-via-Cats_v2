
**Mental model.** ООП — парадигма, где программа моделируется набором объектов; каждый объект объединяет состояние (данные) и поведение (методы), а взаимодействуют объекты через контракты, а не через прямой доступ к чужим внутренностям.

Четыре принципа — каждый одной фразой для устного ответа:

- **Encapsulation** — данные и операции над ними живут вместе; наружу торчит только необходимое, остальное закрыто.
- **Abstraction** — работаем с тем, что объект умеет, не зная, как он это делает.
- **Inheritance** — новый тип переиспользует и специализирует существующий (отношение is-a).
- **Polymorphism** — один интерфейс, много реализаций; нужная выбирается за тебя в нужный момент.

В Java это собрано из `class` / `interface` / `enum` / `record`, четырёх модификаторов доступа и — важно — из того, что все instance-методы виртуальны по умолчанию (никакого `virtual`). Порядок принципов держим сквозным: E → A → I → P.

> [!tip] 💡 Из C#  
> Список инструментов почти тот же (`class`/`interface`/`record`), но два больших расхождения зашиты уже здесь: в Java нет `struct` как пользовательского value-типа и нет ключевого слова `virtual` — «виртуальность» в Java это дефолт, а не opt-in. Оба всплывут ниже.

### 1. Encapsulation

**Mental model.** Инкапсуляция — объект сам отвечает за свои инварианты: состояние спрятано, изменить его можно только через контролируемый интерфейс.

**How it works.** Два слоя: (1) bundling — состояние и поведение в одном типе; (2) access control — кто и как достаёт до состояния. В Java контроль доступа даёт всего 4 уровня (и один из них — без ключевого слова):

|Модификатор|Тот же класс|Тот же пакет|Подкласс из другого пакета|Все|
|---|---|---|---|---|
|`private`|✓||||
|(package-private, без слова)|✓|✓|||
|`protected`|✓|✓|✓||
|`public`|✓|✓|✓|✓|

Две вещи здесь — чисто джавовые и оба раза ловушка:

- **package-private — это дефолт.** Не написал модификатор — член виден всему своему пакету и больше никому. Отдельного ключевого слова нет.
- **`protected` в Java шире, чем кажется:** это «подкласс ИЛИ тот же пакет», а не только подкласс. То есть `protected`-член доступен любому классу из того же пакета, даже если он вообще не наследник.

Свойств (properties) в Java нет — инкапсуляцию делают руками: private-поля + методы-аксессоры по конвенции JavaBeans (`getX()` / `setX()` / для boolean — `isX()`). Причина та же, что и для property: публичное поле фиксирует представление навсегда, а геттер позволяет позже подложить валидацию/вычисление/логирование, не ломая вызывающий код на уровне исходников.

```java
public class Account {
    private long balanceCents;                 // состояние закрыто

    public long getBalanceCents() { return balanceCents; }

    public void deposit(long cents) {          // инвариант защищён здесь
        if (cents <= 0) throw new IllegalArgumentException("amount must be positive");
        balanceCents += cents;
    }
}
```

> [!tip] 💡 Из C#  
> Маппинг модификаторов НЕ один-к-одному, и это классический провал на собесе:
> 
> - `internal` / `protected internal` / `private protected` в Java нет вообще — потому что нет сборок (assembly). Единица инкапсуляции в Java — пакет, а не сборка.
> - C# `protected` = «только наследник». Java `protected` = «наследник или тот же пакет» — шире.
> - Ближайший аналог `internal` — это package-private, но пакет ≠ assembly: пакет открыт для расширения (любой может объявить класс в твоём пакете и получить package-private доступ), а sealed-инкапсуляцию пакета даёт только JPMS (`module-info`, глава 8).
> - Свойств нет: `public int Age { get; private set; }` превращается в `private int age;` + `getAge();`; «read-only снаружи» делается отсутствием сеттера + `final`.

> [!example] 🔬 Глубже / на синьора  
> **`protected` ещё уже, чем в таблице, при доступе через ссылку.** Из подкласса в другом пакете ты можешь трогать унаследованный `protected`-член только у объектов своего типа (или его подтипа), но не у произвольного объекта базового типа. Правило JLS §6.6.2 существует, чтобы подкласс не «пролез» к `protected`-состоянию чужих наследников базы.
> 
> **Иммутабельность как предельная инкапсуляция.** Полностью неизменяемый класс: все поля `private final`, нет сеттеров, `final` на классе (чтобы наследник не добавил mutable-состояние), а mutable-поля (массивы, `Date`, коллекции) копируются на входе и на выходе (defensive copy). Это тема урока 3.7, но на собесе по инкапсуляции всплывает постоянно.
> 
> **`record` (Java 16, JEP 395)** — синтаксический предел этого паттерна: неизменяемый data-carrier с автогенерацией. Детали — глава 17.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Забытый модификатор ≠ `private`.** Пришедший из C#/C++ рефлекс «нет слова = приватно» ложен: в Java это package-private, то есть виден половине модуля. Утечка инкапсуляции по невнимательности.
> 2. **`protected`-поле открыто всему пакету.** Часто думают «только наследникам» — а его видит любой сосед по пакету.
> 3. **Сеттер, отдающий ссылку на mutable-состояние, пробивает инкапсуляцию:** `getItems()` вернул внутренний `List`, снаружи в него дописали — инвариант сломан. Нужен `List.copyOf(...)` или `Collections.unmodifiableList(...)`.
> 4. **`public` поле в API нельзя «повысить» до аксессора без слома совместимости** — как и в C#, но в Java ломается ещё и бинарная совместимость (доступ к полю — это `getfield`, вызов метода — `invokevirtual`; разный байткод).
> 5. **Рефлексия сносит инкапсуляцию:** `field.setAccessible(true)` добирается до `private`. С JPMS (Java 9+) это уже требует `opens`, иначе `InaccessibleObjectException`.

### На собесе

> [!question]- Что такое инкапсуляция?  
> **Model answer (RU):** Инкапсуляция — объект держит состояние закрытым и сам отвечает за инварианты; снаружи доступен только контролируемый интерфейс (обычно методы-аксессоры). В Java контроль доступа реализуют четыре уровня, и два из них джава-специфичны: дефолт без ключевого слова — это package-private (виден пакету), а `protected` означает «наследник или тот же пакет», то есть шире, чем в C#. Свойств в языке нет, поэтому инкапсуляцию собирают вручную по конвенции JavaBeans.
> 
> **Model answer (EN):** Encapsulation means an object keeps its state private and owns its invariants; the outside world only touches a controlled interface, usually accessor methods. Java enforces access with four levels, two of them Java-specific: the default with no keyword is package-private (visible to the package), and protected means "subclass or same package" — broader than in C#. There are no properties, so encapsulation is assembled by hand following the JavaBeans convention.
> 
> _Follow-up:_ «Чем отличается от abstraction?» _Trap:_ «data hiding — это отдельный принцип?» (нет, это механизм).
> 
> _Слабые ответы:_ «Инкапсуляция — это private-поля» — путает механизм с принципом; ни слова про инварианты и контракт. «Нет модификатора — значит private» — фатально: это package-private. «protected = только подклассам» — забыт пакет.

> [!question]- Перечисли модификаторы доступа в Java  
> `private`, package-private (дефолт, без слова), `protected`, `public`.
> 
> _Trap:_ «что значит их отсутствие?» — package-private. «protected — это только для наследников?» — нет: наследник или пакет.

### 2. Abstraction

**Mental model.** Абстракция — выделяем существенное поведение и прячем несущественную сложность; думаем в терминах «что тип умеет», а не «как он устроен».

**How it works.** Инструменты в Java — `interface` и `abstract class`: они задают, что можно делать, не фиксируя как. `interface List<E>` обещает `add`, `get`, `size` — а `ArrayList` и `LinkedList` реализуют это принципиально по-разному (массив vs узлы). Код, написанный против `List`, не знает и не должен знать, какой именно.

**Abstraction vs Encapsulation** (главный trap-вопрос секции) — это разные уровни, не синонимы:

- **Abstraction** — про дизайн: какой набор операций ты выставляешь, что считаешь существенным. Решает «что».
- **Encapsulation** — про реализацию: как спрятано и защищено состояние за этим интерфейсом. Решает «как спрятано».

`ArrayList` даёт тебе `add` / `size` / `get(i)` — это абстракция «динамический массив»; а то, что внутри `Object[] elementData` и `int size` приватны и сам рост массива (`grow()`) скрыт, — это инкапсуляция. Одно про витрину, другое про замок на складе.

> [!tip] 💡 Из C#  
> Тут расхождений почти нет — `interface` и `abstract class` работают идейно так же. Одна деталь: в Java `interface` исторически (до Java 8) вообще не мог нести реализацию, поэтому «чистая абстракция = интерфейс» здесь была буквальнее, чем в C#. С Java 8 появились default-методы (см. §5), и граница размылась — как DIM в C# 8, только Java пришла к этому на пять лет раньше (2014 vs 2019).

> [!example] 🔬 Глубже / на синьора  
> **Абстракция как контракт** — код зависит от интерфейса, а не от класса; это мост к DIP (глава SOLID) и к тестируемости (мок реализует интерфейс).
> 
> **Leaky abstraction** — когда детали реализации протекают сквозь интерфейс. Каноничный пример из JDK: `Iterator` над `ArrayList` бросает `ConcurrentModificationException`, если список меняли во время обхода, — деталь реализации (`modCount`) торчит наружу через поведение абстракции.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Абстракция без нужды.** Интерфейс с единственной реализацией «на всякий случай» — это не абстракция, а лишний слой. Абстракция оправдана, когда реализаций реально несколько или граница нужна для теста/плагина.
> 2. **Абстракция, повторяющая одну реализацию.** Если интерфейс списан один-в-один с конкретного класса (те же имена, те же исключения), он не абстрагирует, а фиксирует деталь.
> 3. **Путать с инкапсуляцией на собесе** — теряешь пол-очка сразу.

> [!question]- Abstraction vs encapsulation? _(классика, почти всегда)_  
> Слабый ответ: «Абстракция прячет данные, инкапсуляция прячет данные» — стирает разницу; ждут разведения уровней «что» vs «как спрятано».
> 
> **Model answer (RU):** Абстракция — про то, что тип выставляет наружу как существенное поведение (через interface/abstract class), отбрасывая лишнее. Инкапсуляция — про то, как за этим интерфейсом спрятано и защищено состояние. Первое — дизайн контракта, второе — защита реализации; обычно работают вместе.
> 
> **Model answer (EN):** Abstraction is about what a type exposes as its essential behavior — modeling via interfaces or abstract classes and dropping the irrelevant. Encapsulation is about how the state behind that surface is hidden and protected. One designs the contract, the other guards the implementation; they usually go together.

### 3. Inheritance

**Mental model.** Наследование — производный тип переиспользует и специализирует базовый, выражая is-a. Это самая жёсткая связь в ООП, поэтому применяй осознанно и предпочитай композицию, когда отношение не строго is-a.

**How it works.**

- Single inheritance для классов (`extends` — ровно один родитель); multiple — только через интерфейсы (`implements` — сколько угодно).
- Корень всего — `java.lang.Object`: даёт каждому типу `toString()`, `equals()`, `hashCode()`, `getClass()`, а также `clone()`, `wait`/`notify` и (deprecated) `finalize`. (Контракт `equals`/`hashCode` — отдельный урок 3.8.)
- **Конструкторы в иерархии.** Первый оператор любого конструктора — это `super(...)` или `this(...)`; если не написал — компилятор неявно вставляет `super()` (вызов no-arg конструктора базы). Если у базы нет no-arg конструктора, а ты не вызвал `super(args)` явно — ошибка компиляции.

Порядок при `new Derived()` (проверено на JDK 21):

```java
class Base {
    int b = log("Base:    field init");
    { log("Base:    init block"); }
    Base() { log("Base:    ctor body"); }
    static int log(String s) { System.out.println(s); return 0; }
}
class Derived extends Base {
    int d = Base.log("Derived: field init");
    { Base.log("Derived: init block"); }
    Derived() { Base.log("Derived: ctor body"); }
}
// new Derived() печатает:
// Base:    field init
// Base:    init block
// Base:    ctor body
// Derived: field init
// Derived: init block
// Derived: ctor body
```

Ниже — та же последовательность как диаграмма. Обрати внимание на «момент истины» посередине: конструктор базы отрабатывает целиком до того, как инициализируются поля подкласса.

![[Pasted image 20260706123553.png]]

Именно из этого порядка вырастает главная ловушка — виртуальный вызов из конструктора базы. Метод в Java виртуален по умолчанию, поэтому вызов из `Base(...)` уходит в переопределённую версию `Derived`, но поля `Derived` там ещё в дефолте (проверено на JDK 21):

```java
class Base {
    Base() { System.out.println("Base ctor sees: " + describe()); }
    String describe() { return "base"; }
}
class Derived extends Base {
    String msg = "hello";
    @Override String describe() { return "derived, msg=" + msg; }
}
// new Derived()  ->  Base ctor sees: derived, msg=null
```

Диспетчеризация уже «производная» (`Derived.describe`), а поле `msg` ещё `null` — оно инициализируется только на шаге 5 диаграммы, после возврата из `super()`.

**Composition over inheritance.** Наследование жёстко связывает наследника с реализацией базы (fragile base class: изменение базы молча ломает наследников). Если отношение — has-a, а не is-a, бери композицию: вкладывай объект полем и делегируй.

```java
// has-a, не is-a: логгер не «является» кэшем — он его использует
final class LoggingCache {
    private final Cache delegate;                      // композиция
    LoggingCache(Cache delegate) { this.delegate = delegate; }
    Object get(String k) {
        System.out.println("get " + k);
        return delegate.get(k);                        // делегирование
    }
}
```

**LSP (Liskov Substitution):** наследник не должен ломать контракт базового типа — подробно в главе SOLID.

> [!tip] 💡 Из C#  
> Три расхождения, все спрашивают.
> 
> - `base` → `super`. `: BaseClass` → `extends`, реализация интерфейсов → `implements`. Один `extends`, много `implements`.
> - **Порядок инициализации полей отличается.** В C# инициализаторы полей производного класса выполняются до тела базового конструктора. В Java — после полного завершения `super()`. Практическое следствие: в C# при виртуальном вызове из базового ctor поля-инициализаторы наследника уже сработали, а в Java — ещё нет (поэтому `msg=null`, а не `"hello"`). Ловушка та же по духу, но в Java «пустее».
> - `System.Object` → `java.lang.Object`; набор методов близкий, но у Java здесь же живут `wait`/`notify` (мониторы встроены в каждый объект — тема concurrency).

> [!example] 🔬 Глубже / на синьора  
> **Почему нет MI классов — diamond problem.** Если D наследует B и C, оба от A, возникает неоднозначность состояния A (чья копия полей?) и разрешения членов. Java запрещает это для классов. default-методы интерфейсов (Java 8, JEP 126) дают множественное наследование поведения (но не состояния): конфликт разрешается принудительно (см. §5).
> 
> **Statement-before-super()** (Java 21 — превью, JEP 447; в 22/23 дорабатывается). Историческое правило «`super()` строго первый оператор» ослабляется: разрешены операторы до `super()`, если они не трогают `this` (например, валидация аргументов). В Java 21 LTS это ещё превью — в проде полагаться нельзя, но на собесе про траекторию языка спросить могут.
> 
> **`final` на классе/методе** даёт JIT-у право на девиртуализацию — см. §4.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Вызов переопределяемого метода из конструктора** — почти всегда баг (пример выше). Из конструктора зови только private/final/static-методы.
> 2. **Нет no-arg конструктора у базы** — наследник обязан явно звать `super(args)`, иначе не компилируется. Частый сюрприз при наследовании от чужого класса.
> 3. **fragile base class:** база позвала свой публичный метод внутри себя, наследник его переопределил — поведение базы поменялось «сквозь» наследника незаметно.
> 4. **Наследование ради переиспользования кода без is-a** — берёшь лишние методы базы в публичный контракт наследника (`Stack extends Vector` в JDK — канонический антипример).
> 5. **Тень поля при наследовании** (`Derived` объявил поле с тем же именем) — поля не полиморфны, доступ к ним статичен по типу ссылки (см. §4). Источник тихих багов.

### На собесе

> [!question]- Почему Java не поддерживает множественное наследование классов?  
> Слабый ответ: «MI нет, потому что так решили / это плохо» — нет механизма (diamond, неоднозначность состояния).
> 
> **Model answer (RU):** Класс в Java наследует один базовый — это исключает diamond-неоднозначность состояния; множественность достигается через интерфейсы, а с Java 8 интерфейсы несут и реализацию по умолчанию (default-методы) — это множественное наследование поведения, но не состояния (у интерфейса нет instance-полей).
> 
> **Model answer (EN):** A Java class inherits a single base, which rules out the diamond ambiguity over state; you get multiplicity through interfaces, and since Java 8 interfaces can carry default implementations — multiple inheritance of behavior, not of state (interfaces hold no instance fields).
> 
> _Follow-up:_ «как обойти?» _Trap:_ «а default-методы — разве это не MI?»

> [!question]- Inheritance vs composition — что выберешь и почему?  
> Слабый ответ: «Наследование = переиспользование кода» — переиспользование даёт и композиция; не звучит is-a и цена жёсткой связи.
> 
> **Model answer (RU):** Наследование беру при истинном is-a, когда готов жёстко связать наследника с реализацией базы; композицию — при has-a или когда собираю поведение из независимых частей. По умолчанию предпочитаю композицию: нет проблемы fragile base class, легче тестировать, вложенный объект можно подменить в рантайме, тогда как наследование фиксирует связь на этапе компиляции.
> 
> **Model answer (EN):** I use inheritance for a genuine is-a when I'm willing to couple the subclass tightly to the base's implementation; composition for has-a or when assembling behavior from independent parts. I default to composition: no fragile base class problem, easier testing, and the contained object can be swapped at runtime, whereas inheritance fixes the relationship at compile time.

> [!question]- Что напечатает виртуальный вызов из конструктора базы? _(senior-любимый)_  
> См. пример выше: `Base ctor sees: derived, msg=null`. Диспетчеризация уже идёт в `Derived.describe()`, но поля `Derived` ещё не инициализированы — они получают значения только после возврата из `super()`.

### 4. Polymorphism

**Mental model.** Полиморфизм — один интерфейс, много форм поведения; механизм расширяемости: код работает с базовым типом, а конкретная реализация выбирается за него по типу объекта.

**How it works.** Два вида:

- **Compile-time (static / ad-hoc):** method overloading (перегрузка по сигнатуре). Резолвит компилятор по объявленным типам аргументов.
- **Runtime (dynamic / subtype):** overriding — переопределение метода в подклассе. Резолвится по runtime-типу объекта.

Центральное отличие Java, ради которого стоит вся эта секция: все instance-методы виртуальны по умолчанию. Нет `virtual`, нет override-ключевого слова. Чтобы запретить переопределение — ставишь `final`. `@Override` — это аннотация, а не ключевое слово: она ничего не включает, а лишь просит компилятор проверить, что метод и правда что-то переопределяет (спасает от опечаток в сигнатуре).

Теперь — что в Java играет роль C#-шного `new` (hiding). Прямого аналога нет, но hiding в Java есть — для полей и для static-методов, и он резолвится по типу ссылки, а не объекта (проверено на JDK 21):

java

```java
class Base {
    String name = "base-field";
    String instanceMethod() { return "Base.instanceMethod"; }
    static String staticMethod() { return "Base.staticMethod"; }
}
class Derived extends Base {
    String name = "derived-field";
    @Override String instanceMethod() { return "Derived.instanceMethod"; }
    static String staticMethod() { return "Derived.staticMethod"; }
}

Base ref = new Derived();
ref.instanceMethod();     // Derived.instanceMethod  — по типу ОБЪЕКТА (override)
ref.name;                 // base-field              — по типу ССЫЛКИ (поле не полиморфно)
((Derived) ref).name;     // derived-field           — каст меняет тип ссылки
Base.staticMethod();      // Base.staticMethod       — static резолвится статически
```

Ещё две вещи про переопределение, которые компилятор проверяет:

- **Ковариантный возврат** (Java 5+): переопределение может вернуть более узкий тип, чем базовый метод — `Animal reproduce()` → `Cat reproduce()` (проверено).
- **Нельзя сужать видимость при переопределении:** `public`-метод не переопределишь как `protected` — ошибка `attempting to assign weaker access privileges`. Можно только расширять.
- **Нельзя переопределить `final`, `static` и `private`-методы:** `final` запрещён, `static` — прячется (hiding), `private` вообще не виден подклассу.

> [!tip] 💡 Из C#  
> Это самое ценное расхождение всей главы — маппинг НЕ прямой.
> 
> - C#: методы не виртуальны по умолчанию (opt-in через `virtual` + `override`). Java: виртуальны по умолчанию (opt-out через `final`). Полная инверсия.
> - C#-шный `override` → в Java его роль играет сам факт совпадения сигнатуры; `@Override` — лишь проверочная аннотация, а не переключатель.
> - C#-шного `new` (method hiding для instance-методов) в Java нет: instance-метод с той же сигнатурой всегда переопределяет, спрятать его нельзя. Hiding в Java существует только для static-методов и для полей, и он неявный (без ключевого слова), резолвится по типу ссылки — ровно то, что показывает диаграмма.
> - `sealed` в C# на методе → в Java `final` на методе. `sealed` на классе (C#) → `final` на классе (Java). (Не путать с джавовым sealed-классом из Java 17 — это про ограниченную иерархию, глава 17, совсем другое.)
> - Covariant return есть в обоих, но в C# он появился только в C# 9, а в Java — с 2004 года (Java 5).

> [!example] 🔬 Глубже / на синьора  
> **Механизм virtual dispatch.** У каждого объекта в заголовке есть указатель на метаданные его класса (в HotSpot — `klass`, в котором лежит vtable). Вызов instance-метода компилируется в байткод `invokevirtual` (для методов класса) или `invokeinterface` (через интерфейсную ссылку) и на рантайме идёт через слот в таблице методов — поэтому он чуть дороже прямого. Статические вызовы — `invokestatic`, конструкторы/`super`/`private` — `invokespecial`; они связываются без таблицы.
> 
> **Девиртуализация.** Если сайт вызова мономорфный (JIT видит один реальный тип) или метод/класс `final`, C2/JIT заменяет виртуальный вызов на прямой и инлайнит его. Отсюда практический вывод: `final` — это не только про дизайн, но и подсказка оптимизатору.
> 
> **Таксономия полиморфизма:** ad-hoc (overloading) / subtype (overriding — «настоящий» ООП-полиморфизм) / parametric (generics). На FU «overloading — это полиморфизм?» отвечай: формально ad-hoc, в отличие от subtype (многие вообще не считают перегрузку полиморфизмом — это разрешение перегрузки на этапе компиляции).
> 
> **Parametric в Java — с эрзацем.** Дженерики Java стёрты (type erasure): в рантайме `List<String>` и `List<Integer>` — один тип `List`, диспетчеризации по типу-параметру нет вовсе. Это ещё один жёсткий контраст с C#, где дженерики reified и CLR специализирует их под каждый value-тип. (Полный разбор — глава 7.)

> [!warning] ⚠️ Подводные камни
> 
> 1. **Ожидание полиморфизма от полей** — `ref.field` берётся по типу ссылки, всегда. Поля не переопределяются, а затеняются. Классический вопрос-ловушка «что напечатает `ref.field`?».
> 2. **Ожидание полиморфизма от static** — `ref.staticMethod()` тоже резолвится по типу ссылки (и вызывать static через ссылку на объект — плохой стиль, компилятор ворчит). Полиморфны только instance-методы.
> 3. **Перегрузка выбирается по compile-time типу.** `process(Object)` vs `process(String)` при аргументе `Object o = "x"` вызовет `process(Object)` — выбор перегрузки статичен, в отличие от диспетчеризации переопределения.
> 4. **Перегрузка + автобоксинг + varargs** — правила разрешения многоступенчатые (сначала без бокса, потом с боксом, потом varargs); источник неочевидных выборов. Деталь — урок 1.8/1.4.
> 5. **`@Override` забыли** — опечатка в сигнатуре молча создаёт новый метод вместо переопределения; ставь аннотацию всегда.

### На собесе

> [!question]- Compile-time vs runtime polymorphism?  
> Compile-time — overloading, резолвит компилятор по типам аргументов. Runtime — overriding, резолвится по runtime-типу объекта.
> 
> _Follow-up:_ «overloading — это полиморфизм?» — формально ad-hoc, отдельная категория от subtype. _Trap:_ «а generics — какой, и что с ними в рантайме?» — parametric + erasure.

> [!question]- В Java есть virtual?  
> Нет ключевого слова — все instance-методы виртуальны по умолчанию.
> 
> _Trap:_ «как тогда запретить переопределение?» — `final`. «а `@Override` что делает?» — только проверка, ничего не включает.

> [!question]- Что напечатает поле/static через ссылку базового типа?  
> _Trap:_ поле резолвится по типу ссылки → значение базового класса. Instance-метод резолвится по типу объекта → переопределённая версия.

> [!question]- (senior) Как устроен virtual dispatch на уровне байткода?  
> Через `invokevirtual`/`invokeinterface` и слот vtable в метаданных класса объекта. `final`/мономорфность позволяют JIT девиртуализировать и заинлайнить вызов.

> [!question]- Итоговый ответ по теме _(RU/EN)_  
> **RU:** В Java полиморфизм бывает compile-time (overloading, резолвит компилятор по типам аргументов) и runtime (overriding, резолвится по runtime-типу объекта). Ключевое отличие от C#: все instance-методы виртуальны по умолчанию, `virtual` нет, а `final` запрещает переопределение; `@Override` — лишь проверочная аннотация. При этом поля и static-методы не полиморфны — они скрываются и резолвятся по типу ссылки на этапе компиляции, поэтому через ссылку базового типа поле даст базовое значение, а instance-метод — переопределённое. _(senior add-on)_ Под капотом виртуальный вызов идёт через `invokevirtual`/`invokeinterface` и слот vtable; `final`/мономорфность позволяют JIT девиртуализировать и заинлайнить.
> 
> **EN:** Java polymorphism is compile-time (overloading, resolved by the compiler on argument types) or runtime (overriding, resolved by the object's runtime type). The key difference from C#: every instance method is virtual by default — there's no virtual, and final forbids overriding; @Override is only a checking annotation. Fields and static methods are not polymorphic — they're hidden and resolved on the reference type at compile time, so through a base-typed reference a field yields the base value while an instance method yields the overridden one. _(senior add-on)_ Under the hood a virtual call goes through invokevirtual/invokeinterface and a vtable slot; final/monomorphic sites let the JIT devirtualize and inline.
> 
> _Слабые ответы:_ «В Java методы становятся виртуальными через virtual/@Override» — фатально: виртуальны по умолчанию. «Поля тоже переопределяются» — нет, затеняются. «Полиморфизм — это перегрузка» — сводит всё к ad-hoc, теряет subtype.

### 5. abstract class vs interface

**Mental model.** `abstract class` — «частично готовый родитель» с общим состоянием и кодом (is-a, один родитель). `interface` — контракт способностей (can-do, можно реализовать сколько угодно).

**How it works.**

||abstract class|interface|
|---|---|---|
|Instance-поля (состояние)|да|нет (только public static final константы)|
|Конструкторы|да|нет|
|Множественность|нет (1 extends)|да (implements несколько)|
|Реализация по умолчанию|да|нет до Java 8; default-методы (Java 8, JEP 126)|
|static-методы|да|да (Java 8)|
|private-методы|да|да (Java 9, JEP 213)|
|Модификаторы у членов|любые|методы public неявно; private — с Java 9|
|Добавить член без слома реализаторов|да (с телом)|ломает abstract-методом; не ломает default-методом|

```java
abstract class Shape {                          // есть state + общий код
    private final String name;                  // instance-поле — интерфейс так не может
    protected Shape(String name) { this.name = name; }   // конструктор
    abstract double area();                     // обязателен override
    String describe() { return name + ": " + area(); }   // общий код
}

interface Drawable {                            // контракт; реализуемых — сколько угодно
    void draw();
    default void drawTwice() { draw(); draw(); }         // default (Java 8)
    static Drawable noop() { return () -> {}; }          // static (Java 8)
}

final class Circle extends Shape implements Drawable {   // один extends + любое число implements
    private final double r;
    Circle(double r) { super("circle"); this.r = r; }
    @Override double area() { return Math.PI * r * r; }
    @Override public void draw() { /* ... */ }
}
```

Diamond у default-методов — если два интерфейса дают default-метод с одной сигнатурой, компилятор не выбирает за тебя, а требует явного переопределения; вызвать конкретную версию можно синтаксисом `Interface.super.method()` (проверено на JDK 21):

```java
interface A { default String hi() { return "A"; } }
interface B { default String hi() { return "B"; } }

class C implements A, B {
    @Override public String hi() { return A.super.hi() + "+" + B.super.hi(); }  // "A+B"
}
// без переопределения:  error: class D inherits unrelated defaults for hi() from types A and B
```

Когда что: `abstract class` — есть общее состояние + общий код, тесная is-a иерархия, один родитель, хочешь контролировать конструирование. `interface` — контракт для несвязанных типов, нужна множественность, абстракции для DI/плагинов, функциональные интерфейсы (SAM) под лямбды (глава 9).

> [!tip] 💡 Из C#  
> Совпадений тут больше, чем расхождений, но нюансы спрашивают.
> 
> - default-методы ≈ DIM (C# 8), и Java пришла первой (2014 vs 2019). Идея одинаковая: добавить метод с реализацией в опубликованный интерфейс, не сломав реализаторов.
> - Поля интерфейса в Java неявно `public static final` — это только константы, instance-состояния нет (как и в C#). Массовое злоупотребление «constant interface» — антипаттерн (Effective Java, Item 22): интерфейс не место для констант.
> - До Java 9 у интерфейсов не было private-методов, поэтому общий код между default-методами вынести было некуда; JEP 213 это починил.
> - Разрешение diamond через `A.super.hi()` — джава-специфичный синтаксис; в C# конфликт DIM резолвится похоже (явной реализацией), но синтаксис иной.

> [!example] 🔬 Глубже / на синьора  
> **default-метод как инструмент версионирования API.** До Java 8 добавить метод в интерфейс = сломать всех реализаторов. default-метод позволил, например, добавить `Collection.stream()` и `Iterable.forEach()` в Java 8, не переписывая тысячи существующих реализаций. Это и была главная мотивация JEP 126, а не «интерфейсы с логикой ради удобства».
> 
> **Почему default ≠ abstract class:** у интерфейса по-прежнему нет instance-состояния, а разрешение конфликтов идёт по правилу «класс бьёт интерфейс, более конкретный интерфейс бьёт менее конкретный» — и если однозначности нет, компилятор заставляет переопределить. Это не полноценная замена базовому классу.
> 
> **Функциональные интерфейсы** (ровно один абстрактный метод, `@FunctionalInterface`) — частный, но важнейший случай интерфейса: под них компилируются лямбды через `invokedynamic` (глава 9). У `abstract class` такой роли нет.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Diamond у default-методов не компилируется без явного переопределения** — не «побеждает первый», а именно ошибка. Частый сюрприз.
> 2. **Константы в интерфейсе** (`interface Config { int MAX = 10; }`) — работает, но антипаттерн; такие константы «прилипают» к публичному API реализаторов.
> 3. **default-метод не видит состояния** — он не может обратиться к полям (их нет), только к другим методам интерфейса. Попытка «пронести» состояние ломает саму идею.
> 4. **`abstract class` с единственным наследником** часто стоило бы сделать интерфейсом + композицией — иначе тратишь единственный `extends` зря.
> 5. **Ромб из `abstract class` невозможен в принципе** (один `extends`), а из интерфейсов — возможен и разрешается описанным правилом; путать эти два случая на собесе легко.

> [!question]- abstract class vs interface, когда что? _(топ главы)_  
> Слабый ответ: «Интерфейс — контракт, абстрактный класс — почти контракт» — после Java 8 разваливается на FU про default; не назван ключевой водораздел — state.
> 
> **Model answer (RU):** Абстрактный класс несёт общее состояние, конструкторы и реализацию для тесной is-a иерархии с единственным родителем; интерфейс — контракт способностей, реализуемый множеством несвязанных типов. С Java 8 интерфейс умеет default-реализацию (и static-методы, а с Java 9 — private), но всё равно не имеет instance-состояния — поэтому водораздел не «есть ли код», а «нужно ли общее состояние и один ли это родитель». Если два интерфейса дают одинаковый default-метод, компилятор требует явно переопределить его и разрешить конфликт через `Interface.super`.
> 
> **Model answer (EN):** An abstract class carries shared state, constructors and implementation for a tight is-a hierarchy with a single parent; an interface is a capability contract many unrelated types can implement. Since Java 8 an interface can have default implementations (plus static methods, and private ones since Java 9), but it still has no instance state — so the real dividing line isn't "does it have code" but "do you need shared state and is there a single parent." If two interfaces supply the same default method, the compiler forces you to override it and resolve the clash via Interface.super.
> 
> _Follow-up:_ «может ли интерфейс иметь реализацию?» — да, с Java 8. _Trap:_ «тогда зачем вообще abstract class?» — состояние и конструкторы. _Trap 2:_ «два интерфейса дали одинаковый default-метод — что будет?» — ошибка компиляции, требуется явный override.
