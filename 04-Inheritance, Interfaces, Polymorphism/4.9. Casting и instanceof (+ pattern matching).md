Представь: ты достал что-то из `List<Object>`, или метод вернул тебе `Animal`, или прилетел `Object` из десериализации. Ты-то знаешь, что внутри лежит `Dog` — но компилятор видит только тип ссылки, а он более общий, и позвать `bark()` не даёт. Надо как-то сказать «доверься мне, там собака». Ты тянешься к `(Dog) animal` — а выглядит этот синтаксис один в один как `(int) 3.99`, который ты писал сто раз. Это совершенно разные операции, и именно коллизия синтаксиса ловит людей на собесе. Плюс спуск вниз может рвануть в рантайме — и есть способ проверить заранее. С этого и начнём.

> **Ментальная модель.** Объект в куче один и тот же — по лестнице типов «едет» только взгляд на него, тип ссылки. Вверх (к супертипу) компилятор уверен статически → каст бесплатный и неявный. Вниз (к подтипу) уверен только рантайм → каст явный, с проверкой и риском `ClassCastException`.

Иерархия, которая живёт весь урок:

```java
class Animal { void breathe() { System.out.println("breathing"); } }
class Dog extends Animal { void bark() { System.out.println("Woof"); } }
class Cat extends Animal { void meow() { System.out.println("Meow"); } }
```

### Два направления каста ссылки

**Upcast — вверх, к супертипу.** Неявный, синтаксиса не требует, всегда безопасен:

```java
Dog dog = new Dog();
Animal animal = dog;   // upcast — никакого (Animal) не нужно
animal.breathe();      // ок
// animal.bark();      // НЕ компилируется: компилятор видит только Animal
```

Формальное имя — _widening reference conversion_ (расширяющее преобразование ссылки, JLS §5.1.5): тип ссылки становится шире/общее. Объект тот же — поменялся только «вид», то, какие методы разрешает звать компилятор.

**Downcast — вниз, к подтипу.** Явный, проверяется в рантайме (_narrowing reference conversion_, JLS §5.1.6):

```java
Animal animal = new Dog();
Dog dog = (Dog) animal;  // downcast — (Dog) обязателен
dog.bark();              // теперь компилятор разрешает
```

А вот где больно — компилируется, но падает:

```java
Animal animal = new Cat();
Dog dog = (Dog) animal;  // КОМПИЛИРУЕТСЯ, но в рантайме — ClassCastException
```

Компилятор пропустил: `Cat` и `Dog` оба `Animal`, теоретически в `animal` мог лежать `Dog`. А реальный класс — `Cat`, проверка не прошла → `ClassCastException` (проверено ниже, вместе с текстом сообщения).

> [!tip] Из C#  
> `(Dog) animal` ≈ твой C#-каст `(Dog)obj` — оба **бросают исключение** при несовпадении (C# кидает `InvalidCastException`, Java — `ClassCastException`). А вот прямого аналога C#-оператора `as` (возвращает `null` вместо исключения) в Java **нет**: `animal as Dog` написать нельзя. Обходят через `instanceof` (или pattern matching) — ниже.

### `instanceof` — страж перед спуском

```java
Animal animal = getAnimal();   // что вернёт — неизвестно на этапе компиляции
if (animal instanceof Dog) {
    Dog dog = (Dog) animal;    // здесь спуск гарантированно безопасен
    dog.bark();
}
```

`instanceof` — проверка **реального типа объекта в рантайме**, возвращает `boolean`, ничего не бросает. Один сразу важный угол:

```java
Animal animal = null;
boolean isDog = animal instanceof Dog;   // false, без NPE
```

`null instanceof <что угодно>` — всегда `false` (JLS §15.20.2): `null` не ссылается ни на какой объект. Это удобно — `instanceof` уже включает null-чек, писать `x != null && x instanceof Dog` не надо. Запомни эту строчку: на собесе её любят сталкивать с `(String) null`, разберём в edge-слое.

> [!tip] Из C#  
> `instanceof` ≈ C#-оператор `is` в старой булевой форме (`obj is Dog`). Один в один по смыслу.

### Pattern matching for instanceof (Java 16, JEP 394)

До Java 16 проверка и спуск были разнесены, тип писался дважды — тавтология, которая бесила всех: ты уже доказал компилятору, что это `Dog`, а он всё равно заставляет кастовать вручную.

```java
if (animal instanceof Dog) {
    Dog dog = (Dog) animal;   // Dog назвали второй раз, хотя только что проверили
    dog.bark();
}
```

[JEP 394](https://openjdk.org/jeps/394) это убрала:

```java
if (animal instanceof Dog dog) {
    dog.bark();              // dog уже типа Dog, каста нет
}
```

`instanceof Dog dog` — это **type pattern** (типовой паттерн): тип `Dog` + **pattern variable** `dog`. Читается «проверь, что `animal` — это `Dog`, и если да — свяжи его с переменной `dog` типа `Dog`». Проверка и связывание одним движением.

> [!tip] Из C# — здесь C# вёл  
> Один в один C#-шный `if (obj is Dog dog)`. И честный исторический факт: pattern matching для `is` приехал в C# 7 (2017), а в Java финализировался только в 16 (2021; preview в 14/15 — JEP 305 → 375 → 394). Для тебя это не новый концепт — это Java догнала то, что ты умеешь. Семантика совпадает.

**Flow scoping — где живёт pattern variable.** Куда дотягивается `dog` — не «до конца блока» и не «до конца метода». Компилятор вычисляет точную зону, _где паттерн гарантированно совпал_ (**flow scoping**, область видимости по потоку управления), и только там переменная в области видимости. В простом `if` всё интуитивно — `dog` видна внутри ветки «совпало». А вот контринтуитивный случай, который любят на собесе:

> [!example] Прогон на JDK 21 — scope «утекает» через ранний `return`
> 
> java
> 
> ```java
> void handle(Object o) {
>     if (!(o instanceof String s)) { return; }        // не строка — выходим
>     System.out.println(s.length());                  // s ВИДНА здесь, хотя if закрылся
> }
> ```
> 
> Вызов `handle("abc")` печатает `3`. Почему `s` доступна _после_ `if`? Из-за `return` внутри: если мы дошли до кода после `if`, ветка `return` не сработала → `!(o instanceof String s)` было `false` → паттерн совпал → `s` связана. Зона «точно совпало» — весь код после `if` до конца метода.

Ключевое слово — **анализ потока на этапе компиляции**, не рантайм-магия. Поэтому симметричный код без раннего выхода не компилируется:

> [!example] Прогон на JDK 21 — без раннего выхода `s` не видна
> 
> java
> 
> ```java
> if (o instanceof String s) { }
> System.out.println(s.length());
> ```
> 
> ```
> error: cannot find symbol
>   symbol: variable s
> ```
> 
> После обычного `if` путь «не совпало» проваливается дальше → `s` могла остаться несвязанной → компилятор её туда не пускает.

> [!tip] Из C#  
> Definite-assignment-анализ для `is`-паттернов в C# работает по той же идее — переменная видна там, где компилятор доказал, что она присвоена. `if (!(obj is string s)) return;` и потом `s` — ровно тот же flow scoping, просто в C# его так не называют в обиходе.

### 🔬 Глубже / на синьора

> [!info]- Байткод: одна картинка на все четыре операции + почему асимметрия  
> Реальный `javap -c` (Java 21, прогнал прямо сейчас).
> 
> **`upcast` — пусто:**
> 
> ```
> static Animal upcast(Dog);
>     0: aload_0
>     1: areturn
> ```
> 
> Ни одной инструкции каста, никакого `checkcast`. Компилятор статически **доказал**, что `Dog` есть `Animal` — проверять в рантайме нечего. Upcast чисто компайл-таймовый: байткод не генерится, объект не трогается.
> 
> **`downcast` — `checkcast`:**
> 
> ```
> static Dog downcast(Animal);
>     0: aload_0
>     1: checkcast #7   // class Dog
>     4: areturn
> ```
> 
> Опкод `checkcast` — рантайм-проверка: реальный класс совместим с `Dog` → пропускает, нет → `ClassCastException`. Вот буквальный источник CCE из первого слоя.
> 
> **`instanceof` — одноимённый опкод:**
> 
> ```
> static boolean check(Animal);
>     0: aload_0
>     1: instanceof #7  // class Dog
>     4: ireturn
> ```
> 
> Та же рантайм-проверка типа, что и `checkcast`, но **не бросает** — кладёт на стек `0`/`1`. Отсюда null-поведение: для `null` опкод определён вернуть `0`, поэтому `null instanceof X` → `false` без NPE.
> 
> **Pattern matching — десугарится в `instanceof` + `checkcast` + store:**
> 
> ```
> static void pattern(Object o);
>     1: instanceof #7    // проверка
>     4: ifeq 16          // не совпало → мимо тела
>     8: checkcast #7     // спуск
>    11: astore_1         // сохранить в pattern variable
>    13: invokevirtual #9 // Dog.bark
> ```
> 
> Главный вывод: pattern matching — **не новая машинерия**. Это тот же ручной `if (o instanceof Dog) { Dog dog = (Dog) o; }`, который ты писал до Java 16, только boilerplate убран из исходника — байткод идентичный, стоимости сверху ноль (JEP 394).
> 
> **Почему вверх без проверки, вниз с проверкой.** Подтип **доказуемо** IS-A супертип — компилятор видит это статически по дереву `extends`/`implements`, рантайму делать нечего (пустой `upcast`). А супертиповая ссылка может указывать на **любой** подтип — компилятор не знает, на какой, поэтому спуск верифицируется в рантайме (`checkcast`). Асимметрия не каприз — она следует из того, что известно на компиляции, а что только в рантайме. _Это и есть практичный ответ на собесе; точные опкоды — подтверждение, а не заученная мантра._

### ⚠️ Подводные камни

> [!warning] `(String) null` vs `null instanceof` — пара, которая ломает интуицию
> 
> 
> 
> ```java
> (String) null            // -> null   (успех, без CCE)
> null instanceof String   // -> false
> ```
> 
> Каст `null` к **любому** ссылочному типу легален и возвращает `null` — `checkcast` для `null` определён «пропустить» (объекта нет → нечему не совпасть). А `instanceof` для `null` возвращает `false` — «является ли отсутствие объекта строкой? нет». Логика едина: `null` ничем не является (`instanceof` → `false`), но совместим с любой ссылкой (каст → `null`). Один `null`, два разных вопроса.
> 
> Отсюда же: `null instanceof String s` → `false`, ветка не входит, `s` не связывается. **Pattern matching null-safe by design** — `s` внутри тела гарантированно не-`null`, в отличие от голого каста, где `(String) null` тихо вернёт `null`.

> [!warning] `(int) 3.99` и `(Dog) animal` — один синтаксис, разные операции
> 
> - `(int) 3.99` → `3` — _narrowing primitive conversion_ (JLS §5.1.3). Реально **конвертирует значение**, обрезает дробную часть, данные теряются.
> - `(Dog) animal` — _narrowing reference conversion_ (JLS §5.1.6). **Ничего не конвертирует**: объект в куче как лежал, так и лежит; меняется только тип ссылки + рантайм-проверка `checkcast`.
> 
> Формулировка «каст преобразует объект» — ловушка. Каст ссылки **не трогает объект**. Спросят «что делает `(Dog) animal` с объектом» — правильный ответ «ничего, это проверка ссылки», а не «преобразует». Преобразует только примитивный каст. (💡 Из C#: та же коллизия — `(int)3.99` конвертирует значение, `(Dog)obj` проверяет ссылку; интуиция переносится, просто назови явно.)

> [!warning] Каст к «брату» по иерархии → CCE в рантайме
> 
> ```
> Object o = "hello";
> Integer i = (Integer) o;
> // CCE: class java.lang.String cannot be cast to class java.lang.Integer
> ```
> 
> `String` и `Integer` оба `Object`, но друг другу не родственники. Компилятор пропускает каст через общий супертип, рантайм валит на `checkcast`. Классика «компилируется ≠ безопасно». А вот **прямой** каст двух несвязанных классов без `Object`-моста не компилируется: `(Integer) someString` → `incompatible types: String cannot be converted to Integer`. Разница в том, видит ли компилятор иерархию целиком.

> [!warning] `final`-класс к нереализованному интерфейсу → ошибка компиляции (не рантайма)
> 
> 
> 
> ```java
> Runnable r = (Runnable) str;   // str это String (final)
> // error: incompatible types: String cannot be converted to Runnable
> ```
> 
> Если класс **не-`final`**, компилятор разрешает каст к любому интерфейсу — вдруг подкласс его реализует, рантайм разберётся (проверил: не-`final` класс к `Runnable` компилируется). Но `String` — `final`, подклассов быть не может, сам `Runnable` не реализует → **ни один** объект `String` не может быть `Runnable`, компилятор доказывает это статически и режет сразу. JLS называет такие типы _inconvertible_ (§5.5.1); `javac` пишет «incompatible types».

> [!warning] Generics + erasure — отложенный CCE (мостик в главу 7)
> 
> ```
> List raw = new ArrayList(); raw.add(42);         // сырой список с Integer
> List<String> ls = (List<String>) raw;            // unchecked warning, но ПРОХОДИТ в рантайме
> String s = ls.get(0);                            // CCE вылетает ЗДЕСЬ
> // class java.lang.Integer cannot be cast to class java.lang.String
> ```
> 
> В рантайме параметр `<String>` **стёрт** (type erasure): `checkcast` проверяет только сырой `List` — а это и правда `List`, каст проходит. CCE прилетает **позже** — при доступе к элементу, потому что компилятор вставляет `checkcast String` в _точку доступа_ (подтверждено байткодом `getFirst`: `List.get` вернул `Object`, следом `checkcast String`). Отлаживать больно — исключение не там, где косяк. И связанное: `o instanceof List<String>` **не компилируется** вообще (`Object cannot be safely cast to List<String>`) — параметризацию в рантайме не проверить, легальны только `instanceof List<?>` и raw `instanceof List`. Детали стирания — глава 7 (7.4); тут запомни: generic-каст в рантайме проверяет только сырой тип.
> 
> 💡 Из C#: здесь вы разошлись сильно. В C# generics **reified** — `obj is List<string>` компилируется и работает. В Java стёрты, такой проверки нет в принципе. Твой главный контраст «reified vs erased» — вглубь в главе 7.

> [!warning] Pattern variable — обычная локальная переменная  
> `dog` можно переприсвоить (`dog = anotherDog;`) — легально, но бессмысленно, теряется связь с паттерном. И её видимость определяет **компилятор по потоку** (flow scoping), а не фигурные скобки — отсюда «утечка» за `if` через ранний `return`.

### На собесе

> [!question]- 1. Что вернёт `(String) null`? А `null instanceof String`?  
> **RU:** `(String) null` → `null` (успех, без исключения); `null instanceof String` → `false`. Каст `null` к любому ссылочному типу легален (`checkcast` для `null` пропускает), а `instanceof` для `null` возвращает `false`.  
> **EN:** `(String) null` → `null` (succeeds, no exception); `null instanceof String` → `false`. Casting `null` to any reference type is legal (`checkcast` passes null through), while `instanceof` on `null` returns `false`.
> 
> _Follow-up: «как тогда ведёт себя pattern matching с null?»_ → `if (x instanceof String s)` при `x == null` не входит в ветку, `s` не связывается; pattern matching null-safe, внутри тела `s` гарантированно не-`null`. Это аргумент за pattern matching против голого каста.

> [!question]- 2. Почему upcast неявный, а downcast явный?  
> **RU:** Подтип доказуемо IS-A супертип — компилятор видит это статически, проверять нечего. Супертиповая ссылка может указывать на любой подтип — компилятор не знает на какой, поэтому спуск верифицируется в рантайме.  
> **EN:** A subtype provably IS-A supertype — the compiler sees it statically, nothing to check. A supertype reference may point to any subtype — the compiler doesn't know which, so the downcast is verified at runtime.
> 
> _Follow-up: «какой байткод у downcast?»_ → опкод `checkcast`, бросает CCE при несовпадении. _Follow-up: «а upcast что-нибудь генерит?»_ → ничего, ноль инструкций каста; чисто компайл-таймовая смена типа ссылки, объект не трогается.

> [!question]- 3. `(int) 3.99` и `(Dog) animal` — один и тот же оператор?  
> **RU:** Нет, совпал только синтаксис `(Type)`. `(int) 3.99` — сужающее **примитивное** преобразование, реально меняет значение (обрезает до `3`, потеря данных). `(Dog) animal` — проверяемый каст **ссылки**, объект не конвертирует, только проверяет тип в рантайме.  
> **EN:** No, only the `(Type)` syntax coincides. `(int) 3.99` is a narrowing **primitive** conversion — it actually changes the value (truncates to `3`, data loss). `(Dog) animal` is a checked **reference** cast — it doesn't convert the object, only checks the type at runtime.
> 
> _Trap про «преобразование объекта»:_ каст ссылки **ничего не делает с объектом** в куче; меняется только тип ссылки + рантайм-проверка. «Каст преобразует объект» — неверно для ссылок.

> [!question]- 4. Поймает ли `(List<String>) obj` лист интов в рантайме?  
> **RU:** Нет. Из-за type erasure параметр `<String>` стёрт, `checkcast` проверяет только сырой `List` — а это и правда `List`, каст проходит с unchecked warning.  
> **EN:** No. Due to type erasure the `<String>` parameter is erased; `checkcast` verifies only the raw `List` — which it genuinely is, so the cast passes with an unchecked warning.
> 
> _Follow-up: «тогда где вылетит CCE?»_ → отложенно, при первом обращении к элементу: компилятор вставляет `checkcast String` в точку доступа (`String s = list.get(0)`), там и рванёт — отлаживать больно. Плюс `obj instanceof List<String>` вообще не компилируется (нереифицируемый тип), легально только `List<?>` или сырой `List`. Детали — глава 7.

> [!question]- 5. Когда `instanceof` оправдан, а когда это запах?  
> **RU:** Запах — длинная лестница `if (x instanceof A) … else if (x instanceof B)`: это переизобретённый полиморфизм, нарушает Open/Closed (каждая новая фигура = правка метода). Правильно — виртуальный метод на самом типе, диспетч сделает JVM. Отвечать надо через Open/Closed, а не «`instanceof` медленный» — он не медленный, JIT его отлично оптимизирует.  
> **EN:** A smell is a long ladder `if (x instanceof A) … else if (x instanceof B)`: it's reinvented polymorphism, violates Open/Closed (each new type = editing the method). The fix is a virtual method on the type itself; the JVM dispatches. Frame it via Open/Closed, not "`instanceof` is slow" — it isn't, the JIT optimizes it well.
> 
> _Follow-up: «приведи легитимные кейсы»_ → `equals(Object)` (по контракту приходит `Object`); границы с внешним миром (десериализация, рефлексия, нетипизированный `Object` из API); чужие типы, которым нельзя добавить метод; `sealed`-иерархии с pattern matching for switch (глава 17), где разбор по типу с проверкой исчерпывающести — легитимный приём для закрытых ADT, а не запах.

> [!question]- 6. Скомпилируется ли `s.length()` после `if (!(o instanceof String s)) return;`?  
> **RU:** Да. Из-за раннего `return` компилятор знает: если дошли до кода после `if`, паттерн совпал → `s` связана и видна до конца метода.  
> **EN:** Yes. Because of the early `return`, the compiler knows: if execution reaches the code after the `if`, the pattern matched → `s` is bound and in scope until the end of the method.
> 
> _Follow-up: «объясни механизм»_ → это flow scoping: компайл-таймовый анализ потока вычисляет зону, где паттерн гарантированно совпал, и только там pattern variable видна. Без раннего выхода (`if (o instanceof String s) {}` и потом `s` снаружи) — ошибка компиляции: путь «не совпало» проваливается дальше, `s` могла остаться несвязанной.
