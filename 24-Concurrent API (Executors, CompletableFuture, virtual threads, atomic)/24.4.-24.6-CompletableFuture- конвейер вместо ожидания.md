
В главе 2 мы научились отдавать задачи в пул и получать `Future` — ручку, за которую можно потянуть результат. И тут же упёрлись в стену: у `Future` есть ровно один способ получить значение — блокирующий `get()`. Нельзя сказать «когда результат будет готов — преврати его вот так и передай дальше». Как только асинхронных шагов становится два и они зависят друг от друга («загрузи заказ, по нему найди пользователя, пользователю отправь письмо»), приходится либо блокироваться на каждом шаге — и тогда асинхронность превращается в декорацию, — либо городить вручную коллбэки и потоки-ожидатели. `CompletableFuture` (Java 8) — это ответ JDK на эту проблему: способ описать **пайплайн зависимых асинхронных шагов**, не блокируясь ни на одном из них.

**Ментальная модель.** `Future` — пассивная расписка: «результат когда-нибудь будет, приходи с `get()` и жди». `CompletableFuture` — активный узел конвейера: к нему можно _заранее_ прицепить продолжение — «когда будешь готов, сделай вот это». Ты описываешь граф вычислений, а исполнение само течёт по нему по мере готовности данных.

Формально `CompletableFuture<T>` реализует и старый `Future<T>` (так что `get()` никуда не делся), и интерфейс `CompletionStage<T>` — именно в нём живёт вся машинерия `then...`-операторов. Слово «completable» в названии тоже не случайно: такой future можно завершить _вручную_ (`complete(value)` / `completeExceptionally(ex)`), но в этой главе мы сосредоточимся на главном сценарии — построении цепочек.

### Запуск: supplyAsync и runAsync

Две фабрики запускают задачу асинхронно и сразу возвращают узел конвейера:

```java
// Задача с результатом: Supplier<T> -> CompletableFuture<T>
CompletableFuture<Order> orderF =
        CompletableFuture.supplyAsync(() -> orderRepository.load(42L));

// Задача без результата: Runnable -> CompletableFuture<Void>
CompletableFuture<Void> logF =
        CompletableFuture.runAsync(() -> auditLog.write("started"));
```

`supplyAsync` принимает `Supplier<T>` и возвращает `CompletableFuture<T>` — «здесь появится значение». `runAsync` принимает `Runnable` и возвращает `CompletableFuture<Void>` — «здесь появится факт завершения». У обеих фабрик есть перегрузка со вторым параметром `Executor` — на каком пуле выполнять задачу. Если пул не указан, используется общий `ForkJoinPool.commonPool()`; почему это важно и когда опасно — разберём чуть ниже, это отдельная тема.

Обратите внимание: задача стартует **сразу** при вызове `supplyAsync`, а не при первом обращении к результату. Возвращённый объект — это уже бегущее вычисление, к которому мы теперь будем цеплять продолжения.

### Трансформации: thenApply, thenAccept, thenRun

Продолжение — это «когда значение будет готово, сделай с ним X». Три базовых оператора различаются тем, что берут на вход и что отдают на выход:

```java
CompletableFuture<Order> orderF =
        CompletableFuture.supplyAsync(() -> orderRepository.load(42L));

CompletableFuture<Long> userIdF =
        orderF.thenApply(order -> order.userId());        // T -> U: трансформация

CompletableFuture<Void> printed =
        userIdF.thenAccept(id -> System.out.println(id)); // T -> void: потребитель

CompletableFuture<Void> done =
        printed.thenRun(() -> metrics.increment("orders")); // ничего не берёт, ничего не отдаёт
```

`thenApply(Function<T, U>)` — преобразовать значение и передать дальше: из `CompletableFuture<Order>` получается `CompletableFuture<Long>`. Это `map` для асинхронного мира — та же операция, что у `Stream` и `Optional`.

`thenAccept(Consumer<T>)` — значение нужно _употребить_ (напечатать, записать, отправить), дальше по конвейеру передавать нечего. Результат — `CompletableFuture<Void>`.

`thenRun(Runnable)` — предыдущий шаг завершился, а его значение нам даже не нужно: просто выполнить действие. Тоже `CompletableFuture<Void>`.

Каждый оператор возвращает **новый** `CompletableFuture`, не трогая исходный, — поэтому цепочки пишутся fluent-стилем, и от одного узла можно ответвить несколько независимых продолжений.

### thenCompose: когда шаг сам асинхронный

`thenApply` хорош, пока функция-трансформатор _синхронная_: взяла `Order`, вернула `Long`. Но в реальном пайплайне следующий шаг часто сам асинхронный: «по `userId` загрузить пользователя» — это ещё один поход в базу или по сети, и метод честно возвращает `CompletableFuture<User>`:

```java
CompletableFuture<User> loadUser(long id) {
    return CompletableFuture.supplyAsync(() -> userRepository.load(id));
}
```

Попробуем прицепить его через `thenApply` — код скомпилируется, но тип результата вас не обрадует:

```java
CompletableFuture<CompletableFuture<User>> nested =
        CompletableFuture.supplyAsync(() -> 42L)
                .thenApply(id -> loadUser(id));   // функция вернула CF -> он стал ЗНАЧЕНИЕМ
```

`thenApply` не знает и не хочет знать, что вернула функция. Вернула `Long` — будет `CompletableFuture<Long>`. Вернула `CompletableFuture<User>` — будет `CompletableFuture<CompletableFuture<User>>`: future, значением которого является другой future. Матрёшка. Чтобы добраться до `User`, придётся разворачивать её вручную двойным `join()`, а продолжить цепочку нормальными операторами уже не получится.

`thenCompose` создан ровно для этого случая: он принимает функцию `T -> CompletionStage<U>`, дожидается завершения _внутреннего_ future и делает его результат результатом всей стадии:

```java
CompletableFuture<User> user =
        CompletableFuture.supplyAsync(() -> 42L)
                .thenCompose(id -> loadUser(id)); // внутренний CF развёрнут, тип плоский
```

Если вы работали со `Stream` или `Optional`, узнаёте пару: `thenApply` — это `map`, `thenCompose` — это `flatMap`. Правило выбора то же самое и решается по сигнатуре функции: функция возвращает **обычное значение** `U` → `thenApply`; функция возвращает **`CompletableFuture<U>`** → `thenCompose`, иначе получите вложенность.

![[Pasted image 20260704145247.png]]

Проверено на JDK 21: если перепутать и вызвать `thenApply`, компилятор не спасёт — тип честно выведется в `CompletableFuture<CompletableFuture<User>>`, и ошибка вылезет позже, когда попытаетесь этот результат использовать.

### Какой поток всё это выполняет

До сих пор мы делали вид, что стадии «просто выполняются». Пора вскрыть, кем именно, — здесь закопана вторая мина главы.

У каждого оператора есть две формы: обычная (`thenApply`) и `...Async` (`thenApplyAsync`, с опциональным своим `Executor`). Разница принципиальная:

**Не-Async операторы не дают никакой гарантии, на каком потоке выполнится ваша функция.** На практике возможны два случая. Если предыдущая стадия ещё не завершилась к моменту, когда вы прицепили продолжение, — функцию выполнит _тот поток, который завершит предыдущую стадию_ (обычно рабочий поток пула). Если же стадия уже завершилась — функция выполнится прямо _в вызывающем потоке_, синхронно, внутри вызова `thenApply`. Проверено на JDK 21:

```java
CompletableFuture<Integer> done = CompletableFuture.completedFuture(1);
done.thenApply(x -> {
    System.out.println(Thread.currentThread().getName()); // main — выполнил вызывающий!
    return x;
});
```

Для лёгкой трансформации («достать поле из объекта») это нормально и даже эффективно — нет лишнего переключения потоков. Но если в не-Async `thenApply` положить что-то тяжёлое, оно внезапно может выполниться в потоке, который вы совсем не планировали нагружать, — например, в потоке, обслуживающем сетевой запрос.

**`...Async`-варианты гарантируют выполнение на пуле.** Без явного `Executor` — на `ForkJoinPool.commonPool()`, **общем на всю JVM**. Тот же пул по умолчанию используют `supplyAsync`/`runAsync` и, забегая вперёд, parallel streams. Размер этого пула — примерно «число ядер минус один» (`availableProcessors() - 1`).

Отсюда главная эксплуатационная грабля `CompletableFuture`. Общий пул рассчитан на короткие CPU-задачи. Если засунуть в него **блокирующую** работу — I/O, поход по сети, `get()` другого future, — рабочие потоки повиснут в ожидании, и пул кончится. А поскольку он общий, встанет не только ваш пайплайн: застопорится _весь_ параллелизм приложения — чужие `CompletableFuture`, parallel streams, всё, что делит commonPool. На машине с 4 ядрами достаточно трёх повисших задач.

Вывод, который надо унести в прод: **для блокирующих операций всегда передавайте свой `Executor` явной перегрузкой**:

```java
ExecutorService ioPool = Executors.newFixedThreadPool(16);   // отдельный пул под I/O

CompletableFuture<Order> orderF =
        CompletableFuture.supplyAsync(() -> orderRepository.load(42L), ioPool);

CompletableFuture<User> userF =
        orderF.thenApplyAsync(order -> enrich(order), ioPool)  // блокирующий шаг -> свой пул
              .thenCompose(o -> loadUser(o.userId()));
```

Короткое правило выбора: лёгкая синхронная трансформация → не-Async; тяжёлая или блокирующая работа → `...Async` со своим `Executor`; `...Async` без executor'а — только для короткой CPU-работы, и помня, что пул общий.

### Комбинирование: thenCombine, allOf, anyOf

Цепочка — это последовательная зависимость. Но часто шаги _независимы_, их можно запустить параллельно, а потом склеить результаты.

**`thenCombine`** — склеить два независимых future в один результат. Оба вычисления идут параллельно; функция-комбинатор вызывается, когда готовы оба:

```java
CompletableFuture<Integer> priceF    = CompletableFuture.supplyAsync(() -> loadPrice());    // 100
CompletableFuture<Integer> discountF = CompletableFuture.supplyAsync(() -> loadDiscount()); //  15

CompletableFuture<Integer> totalF =
        priceF.thenCombine(discountF, (price, discount) -> price - discount); // 85
```

![[Pasted image 20260704145307.png]]

**`allOf`** — дождаться завершения _всех_ переданных future. И здесь важная деталь, которую надо разжевать: `allOf` возвращает `CompletableFuture<Void>`. Не список результатов — **ничего**. Он умеет только сигнализировать «все завершились» (типы у future могут быть разными, и общий тип результата вывести нельзя). Результаты нужно доставать из **исходных** future после завершения — к этому моменту их `join()` уже не блокирует, значения готовы:

```java
List<CompletableFuture<Integer>> futures = List.of(
        CompletableFuture.supplyAsync(() -> 1),
        CompletableFuture.supplyAsync(() -> 2),
        CompletableFuture.supplyAsync(() -> 3));

CompletableFuture<Void> all =
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]));

CompletableFuture<List<Integer>> results =
        all.thenApply(v -> futures.stream()
                                  .map(CompletableFuture::join)  // уже готовы, не блокирует
                                  .toList());
```

Если хотя бы один из future завершился с ошибкой, `allOf`-future тоже завершится исключительно — но только после того, как отработают _все_ (он не отменяет остальных).

**`anyOf`** — первый ответивший побеждает: результатом становится значение (или исключение) первого завершившегося future. Возвращает `CompletableFuture<Object>` — типы участников могут быть разными, так что кастовать придётся самому. Типичный сценарий — опрос нескольких реплик/зеркал с ответом от самого быстрого.

### Обработка ошибок: exceptionally, handle, whenComplete

То, что новички забывают в первую очередь, а на проде оно решает. Начнём с механики: если внутри любой стадии вылетает исключение, эта стадия завершается _исключительно_, и все последующие трансформации (`thenApply`, `thenCompose`, `thenAccept`…) **пропускаются** — исключение проносится сквозь них дальше по цепочке, пока не встретит обработчик. Это прямой аналог того, как `throw` пролетает мимо оставшихся строк метода до ближайшего `catch`.

![[Pasted image 20260704145324.png]]

Три инструмента, и они не взаимозаменяемы:

**`exceptionally(Function<Throwable, T>)`** — перехватить исключение и подставить запасное значение. Вызывается _только_ при ошибке; при нормальном завершении стадия прозрачно пропускает значение дальше. Это аналог `catch`, который возвращает дефолт:

```java
CompletableFuture<String> nameF =
        CompletableFuture.supplyAsync(() -> loadUserName(42L)) // может бросить
                .thenApply(String::trim)                        // при ошибке пропускается
                .exceptionally(ex -> {
                    log.warn("failed to load user name", ex);
                    return "guest";                             // запасное значение
                });
```

**`handle(BiFunction<T, Throwable, U>)`** — получает _и_ результат, _и_ исключение (заполнено всегда что-то одно, второе — `null`), срабатывает при любом исходе и может преобразовать оба:

```java
CompletableFuture<String> status =
        loadOrder(42L).handle((order, ex) ->
                ex == null ? "OK: " + order.status()
                           : "FAILED: " + ex.getCause().getMessage());
```

**`whenComplete(BiConsumer<T, Throwable>)`** — побочный эффект по завершении (лог, метрика, освобождение ресурса), **не меняющий результат**: и значение, и исключение проходят сквозь него дальше нетронутыми. Проверено на JDK 21: после `whenComplete` над упавшим future `join()` всё так же бросает исходную ошибку.

Правило выбора: подменить ошибку значением → `exceptionally`; свернуть оба исхода в один результат → `handle`; посмотреть и отпустить → `whenComplete`.

И обязательная грабля, о которую бьются все: **исключения внутри цепочки доезжают до обработчиков завёрнутыми в `CompletionException`**. Если стадия бросила `IllegalStateException`, то `exceptionally` получит не его, а `CompletionException`, у которого настоящая причина лежит в `getCause()`. Проверено на JDK 21:

```java
CompletableFuture.supplyAsync(() -> { throw new IllegalStateException("db down"); })
        .exceptionally(ex -> {
            ex instanceof IllegalStateException;            // false!
            ex.getCause() instanceof IllegalStateException; // true
            return fallback;
        });
```

Любой `instanceof`-разбор типа ошибки внутри `exceptionally`/`handle` обязан начинаться с разворачивания причины. (Нюанс: если future завершили вручную через `completeExceptionally(ex)`, обработчик получит `ex` как есть, без обёртки, — поэтому надёжный код проверяет оба варианта.)

> Мостик для интуиции — но именно мостик, а не тождество:
> 
> |Java|C#|
> |---|---|
> |`CompletableFuture<T>`|`Task<T>`|
> |`supplyAsync(...)`|`Task.Run(...)`|
> |`thenApply` / `thenCompose`|`ContinueWith` / связка через `await`|
> |`thenCombine`|комбинация двух задач вручную / `await` обеих|
> |`allOf`|`Task.WhenAll`|
> |`anyOf`|`Task.WhenAny`|
> |`exceptionally` / `handle`|`try/catch` вокруг `await`, `ContinueWith` с проверкой `IsFaulted`|
> |`CompletionException.getCause()`|`AggregateException.InnerException` при `.Result`/`.Wait()`|
> 
> ⚠️ **Ложный друг, самый важный в этой главе: `CompletableFuture` — это НЕ `async/await`.** В Java нет синтаксического сахара и нет «окраски функций» (`async`-модификатора): вы _вручную_ строите цепочку `thenCompose`/`thenApply`, а не пишете линейный с виду код с `await`. Это ровно то место, где в главе 1 мы говорили «Java пошла другим путём»: `async/await` в языке нет, и до виртуальных потоков (им посвящена отдельная глава) `CompletableFuture` — самый близкий к нему инструмент. Но по эргономике он ближе к старому `Task.ContinueWith`, чем к `await`: линейного кода не будет, будет граф из лямбд.
> 
> Ещё два расхождения, на которых спотыкаются после .NET. Первое: типизированный `Task.WhenAll(Task<T>[])` возвращает `Task<T[]>` — массив результатов; `allOf` возвращает `CompletableFuture<Void>`, и результаты вы достаёте сами из исходных future. Второе: у `Task` продолжение по умолчанию планируется через `TaskScheduler`; у не-Async операторов `CompletableFuture` гарантий нет вовсе — вплоть до синхронного выполнения в вызывающем потоке.

> [!example] 🔬 Глубже / на синьора  
> _Опциональный блок — можно пропустить и вернуться._
> 
> **Почему commonPool один на JVM и как его убить.** `ForkJoinPool.commonPool()` — статический общий пул, создаваемый лениво на всю JVM; его параллелизм по умолчанию — `availableProcessors() - 1`. Он спроектирован под короткие неблокирующие CPU-задачи (изначально — для Fork/Join и parallel streams). Каждая блокирующая задача выключает один рабочий поток из оборота; на 8-ядерной машине семь повисших на I/O лямбд означают, что _все_ потребители commonPool в JVM перестают продвигаться. Диагностируется через thread dump: рабочие потоки `ForkJoinPool.commonPool-worker-N` стоят в `WAITING` на сокетах или на `get()`. Отдельный краевой случай (проверено на JDK 21): если параллелизм commonPool меньше двух (одноядерный контейнер, `-Djava.util.concurrent.ForkJoinPool.common.parallelism=0`), `CompletableFuture` вообще не использует общий пул — под каждую async-задачу создаётся новый поток.
> 
> **Кто именно выполняет не-Async стадию.** Внутри это гонка между «потоком, завершающим предыдущую стадию» и «потоком, регистрирующим продолжение». Если продолжение прицепили _до_ завершения — его выполнит завершающий поток. Если _после_ — вызывающий поток, синхронно. Практическое следствие: тяжёлая логика в не-Async операторе может внезапно выполниться в потоке, завершившем future, — например, в потоке I/O-пула, — и задержать его. Поэтому в библиотечном коде тяжёлые продолжения вешают через `...Async`.
> 
> **`CompletionStage`.** Все `then...`/`handle`/`exceptionally` объявлены в интерфейсе `CompletionStage<T>` — это контракт «узла конвейера» без методов ручного завершения и без блокирующего `get()`. Хороший API принимает и возвращает `CompletionStage`, а не `CompletableFuture`: так вы не отдаёте наружу возможность завершить ваш future извне. Метод `toCompletableFuture()` — мост обратно.
> 
> **`join()` vs `get()`.** Оба блокируют до завершения. `get()` — наследие `Future`: бросает _checked_ `ExecutionException` (причина в `getCause()`) и `InterruptedException`. `join()` — собственный метод `CompletableFuture`: бросает _unchecked_ `CompletionException` (причина тоже в `getCause()`) и не бросает `InterruptedException`. Внутри лямбд и стримов `join()` удобнее — не требует `try/catch`; контракт прерывания при этом теряется. Проверено на JDK 21: один и тот же упавший future из `get()` даёт `ExecutionException`, из `join()` — `CompletionException`, причина одна и та же.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Блокирующая работа в commonPool.** `supplyAsync(() -> restClient.call())` без executor'а — мина замедленного действия: под нагрузкой общий пул забивается, и встаёт весь параллелизм JVM, включая чужой код. Блокирующим операциям — свой `Executor`.
> 2. **`thenApply` вместо `thenCompose`.** Функция возвращает `CompletableFuture` — а вы получили `CF<CF<T>>` и разбираете матрёшку руками. Компилятор не предупредит: тип честно выведется, ошибка проявится дальше по коду.
> 3. **`allOf` возвращает `Void`.** Ожидание «сейчас достану список результатов, как из `Task.WhenAll`» разбивается о сигнатуру. Результаты — только из исходных future, поэтому массив, переданный в `allOf`, нужно сохранить в переменную, а не создать инлайн и потерять.
> 4. **`CompletionException` вместо настоящего исключения.** `ex instanceof MyBusinessException` внутри `exceptionally` — `false`, ветка обработки молча не срабатывает. Сначала `getCause()`; помнить, что при ручном `completeExceptionally` обёртки не будет.
> 5. **`exceptionally` не в том месте цепочки.** Он перехватывает только ошибки _выше_ себя. Поставили после второго шага — падение третьего пролетит мимо в необработанном виде. Обработчик ошибок ставится в конец цепочки (или в конец логического участка).
> 6. **Проглоченные исключения.** Если у упавшего future никто не вызвал `get()`/`join()` и не повесил обработчик — исключение исчезает бесследно: ни stack trace, ни лога. Проверено на JDK 21: программа молча завершается. Fire-and-forget цепочки всегда закрывайте хотя бы `whenComplete` с логированием.
> 7. **Тяжёлая логика в не-Async операторе.** Нет гарантии потока: она может выполниться синхронно в вызывающем потоке или в потоке, завершившем предыдущую стадию (например, в чужом I/O-пуле), — и заблокировать его.

### На собесе

> [!question]- Q1. Чем `CompletableFuture` лучше обычного `Future`? _(скрининговый)_  
> **RU:** `Future` — пассивный контейнер: результат можно только блокирующе ждать через `get()`, продолжение прицепить нельзя. `CompletableFuture` реализует `CompletionStage` и позволяет строить неблокирующие цепочки (`thenApply`, `thenCompose`), комбинировать несколько future (`thenCombine`, `allOf`, `anyOf`), декларативно обрабатывать ошибки (`exceptionally`, `handle`) и завершать future вручную (`complete`).
> 
> **EN:** A plain `Future` is passive — the only way to get the result is a blocking `get()`, and you can't attach continuations. `CompletableFuture` implements `CompletionStage`, so you can build non-blocking pipelines, combine futures, handle errors declaratively, and complete it manually.
> 
> _Follow-up:_ «А когда `Future` всё-таки достаточно?» → когда есть одна задача и одна точка синхронного ожидания результата, без пайплайна.

> [!question]- Q2. В чём разница между `thenApply` и `thenCompose`? _(топ-вопрос; ловушка — «оба трансформируют»)_  
> **RU:** `thenApply` — это `map`: принимает функцию `T -> U`, для синхронной трансформации значения. `thenCompose` — это `flatMap`: принимает функцию `T -> CompletableFuture<U>` и разворачивает вложенный future. Если асинхронную функцию передать в `thenApply`, получится `CompletableFuture<CompletableFuture<U>>`.
> 
> **EN:** `thenApply` is `map`: it takes `T -> U` for a synchronous transformation. `thenCompose` is `flatMap`: it takes `T -> CompletableFuture<U>` and flattens the nested future. Passing an async function to `thenApply` gives you `CompletableFuture<CompletableFuture<U>>`.
> 
> _Follow-up:_ «Где ещё в JDK такая же пара?» → `Stream.map`/`flatMap`, `Optional.map`/`flatMap`. Слабый ответ — «thenCompose для асинхронного» без объяснения вложенности.

> [!question]- Q3. Какой поток выполняет `thenApply` по умолчанию и почему это может быть опасно? _(mid+)_  
> **RU:** У не-Async операторов гарантии нет: функцию выполнит либо поток, завершивший предыдущую стадию, либо — если стадия уже завершена — сам вызывающий поток, синхронно. `...Async`-варианты без executor'а идут в `ForkJoinPool.commonPool()` — общий пул JVM размером ~`CPU-1`, рассчитанный на короткие CPU-задачи. Блокирующие задачи (I/O, `get()`) в нём вешают потоки и стопорят весь параллелизм приложения, поэтому для блокирующей работы передают собственный `Executor`.
> 
> **EN:** Non-Async operators give no guarantee: the function runs either on the thread that completed the previous stage, or synchronously on the calling thread if the stage is already complete. `...Async` variants without an executor use `ForkJoinPool.commonPool()` — a JVM-wide pool sized ~`CPU-1`, designed for short CPU-bound tasks. Blocking work in it starves the pool and stalls parallelism across the whole application.
> 
> _Follow-up:_ «Как это диагностировать?» → thread dump: `commonPool-worker`-потоки в `WAITING`/`BLOCKED` на I/O.

> [!question]- Q4. Как обработать исключение в цепочке? Чем отличаются `exceptionally`, `handle`, `whenComplete`? _(ловушка — `CompletionException`)_  
> **RU:** Исключение пропускает все трансформации до первого обработчика. `exceptionally` вызывается только при ошибке и подменяет её запасным значением; `handle` срабатывает всегда, получает `(result, exception)` и может преобразовать оба исхода; `whenComplete` — побочный эффект, результата не меняет. Внутри цепочки исключение приходит завёрнутым в `CompletionException` — настоящую причину достаём через `getCause()`.
> 
> **EN:** An exception skips all transformations until the first handler. `exceptionally` runs only on failure and substitutes a fallback; `handle` always runs, receives `(result, exception)` and can transform either outcome; `whenComplete` is a side effect that doesn't change the result. Inside a chain the exception arrives wrapped in `CompletionException` — unwrap the real cause with `getCause()`.
> 
> _Follow-up:_ «А что бросит `join()` у упавшего future?» → unchecked `CompletionException`; `get()` — checked `ExecutionException`.

> [!question]- Q5. Что возвращает `allOf` и как получить результаты? _(проверка практики)_  
> **RU:** `CompletableFuture<Void>` — только сигнал «все завершились», потому что типы участников могут различаться. Результаты достают из исходных future после завершения `allOf`: например, `all.thenApply(v -> futures.stream().map(CompletableFuture::join).toList())` — `join()` здесь уже не блокирует. Если любой участник упал, `allOf` завершится исключительно (после завершения всех).
> 
> **EN:** It returns `CompletableFuture<Void>` — just a "they're all done" signal, since participants may have different types. You collect results from the original futures once `allOf` completes, e.g. `all.thenApply(v -> futures.stream().map(CompletableFuture::join).toList())` — `join()` doesn't block at that point. If any participant fails, `allOf` completes exceptionally (after all finish).
> 
> _Follow-up:_ «Чем это отличается от `Task.WhenAll` в C#?» → типизированный `WhenAll` возвращает массив результатов, `allOf` — нет.
