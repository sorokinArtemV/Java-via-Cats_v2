Всё, что мы строили в предыдущих главах, стояло на одной невысказанной предпосылке: **поток — дорогой ресурс**. Пулы из главы 2 существуют, чтобы не создавать потоки на каждую задачу. Ограниченные очереди и backpressure из главы 6 — чтобы не захлебнуться, когда задач больше, чем потоков. `CompletableFuture` из главы 3 — чтобы не занимать поток ожиданием, пока идёт I/O. Это не три разные темы — это три способа экономить один и тот же ограниченный ресурс.

В этой главе предпосылка переворачивается. Java 21 (JEP 444) делает поток дешёвым — настолько, что их можно создавать миллионами и выбрасывать после каждой задачи. Половина инженерных ухищрений из прошлых глав при этом становится просто не нужна. Разберёмся, как это устроено, где выигрыш реальный, а где его нет, и куда язык движется дальше — к структурной конкурентности.

### 7.1. Почему платформенный поток дорогой

Обычный `Thread` — тот, с которым мы работали все шесть глав, — теперь официально называется **platform thread**. Это тонкая обёртка над потоком операционной системы, и все его расходы — это расходы потока ОС:

- **Стек.** Каждому потоку ОС выделяется свой стек — по умолчанию порядка 1 МБ (регулируется через `-Xss`). Стек резервируется целиком при создании, независимо от того, сколько поток реально использует. 10 000 потоков — это порядка 10 ГБ виртуальной памяти только на стеки.
- **Создание.** Это системный вызов: ядро заводит структуры, выделяет стек. Дорого настолько, что вся глава 2 была про то, как этого избегать.
- **Планирование.** Кто когда выполняется — решает ядро ОС. Переключение контекста между потоками — это переход в режим ядра, сохранение и восстановление регистров, сброс кешей.

Практический потолок — **тысячи** платформенных потоков на машину, не миллионы. И отсюда весь блокирующий мир Java жил в вилке из двух компромиссов:

1. **Поток на запрос** (thread-per-request). Код простой и линейный: принял запрос — сходил в базу — сходил в соседний сервис — ответил. Но при 10 000 одновременных запросов нужно 10 000 потоков, а их столько нет. Сервер упирается в потолок задолго до того, как кончится CPU или сеть — потоки просто _спят_ в ожидании I/O, занимая по мегабайту каждый.
2. **Пулы + асинхронность.** Небольшой пул потоков, задачи в очереди, ожидание I/O — через колбэки и цепочки `thenCompose`. Масштабируется отлично, но код превращается в то, что мы видели в главе 3: стектрейсы бесполезны, дебаг мучителен, любая блокировка внутри цепочки — саботаж.

💡 **Из C#:** .NET прошёл ровно эту развилку и выбрал второй путь, облагородив его на уровне языка: `async`/`await` превращает цепочку колбэков обратно в линейный на вид код. Цена, о которой мы говорили ещё в главе 1, — **окраска функций**: `async`-метод возвращает `Task`, вызывать его удобно только из другого `async`-метода, и эта «зараза» ползёт по всему стеку вызовов вверх. У тебя два мира — синхронный и асинхронный, и граница между ними болит. Java пошла другим путём — и вот теперь мы до него дошли.

### 7.2. Виртуальный поток: третий путь

Ментальная модель, которую нужно унести с собеса: **виртуальный поток — это обычный на вид `Thread`, но планирует его не ядро ОС, а JVM — поверх небольшого пула несущих (carrier) платформенных потоков.** Это M:N-планирование: M виртуальных потоков (тысячи, миллионы) исполняются на N несущих (по умолчанию — по числу ядер CPU).

Виртуальный поток — это не поток ОС. Это объект JVM: его стек лежит в heap, создание — обычная аллокация без системных вызовов, и ОС о нём вообще не знает.

![[Pasted image 20260704150959.png]]
![[Pasted image 20260704151119.png]]
Итог, который надо произнести прямо: **ты пишешь обычный блокирующий, синхронно выглядящий код — а масштабируется он как асинхронный.** Блокировка перестала быть грехом. Читать из сокета «в лоб», спать в `Thread.sleep()`, ждать на локе — всё это теперь стоит дёшево, потому что блокируется только лёгкий объект в heap, а не поток ОС. Это и есть обещанный в главе 1 «другой путь Java»: та же цель, что у `async`/`await`, но без окраски функций.

#### API: создание и — главное — executor

```java
// Прямое создание — API тот же Thread, что и всегда
Thread vt = Thread.ofVirtual()
        .name("my-virtual-", 0)
        .start(() -> System.out.println(Thread.currentThread()));
vt.join();
// VirtualThread[#13,my-virtual-0]/runnable@ForkJoinPool-1-worker-1
//                                          ^ виден carrier — воркер ForkJoinPool

Thread pt = Thread.ofPlatform().name("my-platform").start(task); // старый добрый
```

Обрати внимание на `toString()`: виртуальный поток честно показывает, на каком несущем потоке он сейчас смонтирован. Проверить природу потока можно через `Thread.currentThread().isVirtual()`.

Но главный рабочий инструмент — не ручное создание, а executor:

```java
import java.time.Duration;
import java.util.concurrent.Executors;
import java.util.stream.IntStream;

public class Scale {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            IntStream.range(0, 100_000).forEach(i -> executor.submit(() -> {
                Thread.sleep(Duration.ofSeconds(1));  // блокирующее ожидание
                return i;
            }));
        } // close() дожидается завершения всех задач
        System.out.println(System.currentTimeMillis() - start + " ms");
    }
}
```

Я запустил это на машине с **одним** ядром CPU (то есть с одним carrier по умолчанию). Сто тысяч задач, каждая блокируется на секунду. Результат: **~3.9 секунды**, пиковое число одновременно живущих задач — все 100 000. Платформенными потоками этот код либо не запустился бы вовсе (100 000 × ~1 МБ стека), либо в пуле из, скажем, 200 потоков занял бы 100 000 / 200 = 500 секунд. Одно ядро. Сто тысяч одновременных блокировок. Вот масштаб эффекта.

### 7.3. Сдвиг мышления: виртуальные потоки не пулуют

Название `newVirtualThreadPerTaskExecutor` — не случайность. Этот executor **не переиспользует потоки**: на каждую задачу он создаёт новый виртуальный поток, а после завершения задачи поток умирает.

Это стоит проговорить, потому что оно ломает рефлекс, который мы шесть глав тренировали. Пул потоков существовал ради одного: переиспользовать дорогой ресурс. Виртуальный поток — дешёвый и одноразовый. Ему не нужен ни warm-up, ни переиспользование: создание — это аллокация объекта, стек начинается с крошечного и растёт в heap по мере надобности. **«Поток на задачу» снова становится правильным паттерном** — тем самым, который в главе 2 мы клеймили как антипаттерн.

Отсюда практическое следствие: пул виртуальных потоков — бессмыслица. Обёртывать их в `newFixedThreadPool` — значит вернуть себе ограничение, ради снятия которого всё затевалось. Если нужно ограничить конкуренцию к конкретному _ресурсу_ (например, к базе пускать не больше 50 одновременных запросов) — это делается `Semaphore` вокруг ресурса, а не искусственным дефицитом потоков.

### 7.4. Когда они помогают — и когда нет

Это ключевой mid+ момент, на котором сыпятся поверхностные ответы. Виртуальные потоки — не «ускоритель всего».

**Помогают: I/O-bound и высококонкурентные нагрузки с ожиданием.** Тысячи одновременных запросов, каждый из которых значительную часть жизни _ждёт_ — сокет, базу, соседний сервис, лок. Раньше эти ожидания сжигали дефицитные потоки ОС; теперь ожидание бесплатно, и throughput сервера ограничен реальными ресурсами (CPU, сеть, база), а не количеством потоков. Пример выше — ровно этот случай: 100 000 «ожиданий» на одном ядре.

**Не помогают: CPU-bound.** Если задача молотит процессор без блокировок, виртуальный поток нечего снимать с carrier — он занимает его от начала до конца. Параллелизм по-прежнему упёрт в число несущих потоков, то есть в число ядер: виртуальные потоки **не создают ядер из воздуха**. Заменить пул на виртуальные потоки в счётной нагрузке — значит не получить ничего (в лучшем случае). Для распараллеливания вычислений есть Fork/Join — это глава 8.

Простая эвристика: посмотри, что делает поток бо́льшую часть времени. Ждёт — виртуальные потоки твой инструмент. Считает — нет.

### 7.5. Грабли №1: pinning

Есть ситуации, когда виртуальный поток **не может** сняться с несущего — он «прибит» (pinned) к carrier, и блокировка внутри него снова стоит столько же, сколько блокировка платформенного потока. Это самая версионно-чувствительная часть главы, поэтому аккуратно разведём «как в 21» и «как стало позже».

#### Как было в Java 21

Классическая причина pinning на Java 21 — **блокирующий вызов внутри `synchronized`-блока или метода**. Механика: JVM исторически записывала владельца монитора в терминах _несущего_ потока ОС. Если бы виртуальный поток размонтировался, держа монитор, на carrier сел бы другой виртуальный поток — и по бухгалтерии JVM «владельцем» монитора выглядел бы он. Мьютекс сломался бы. Поэтому JVM просто запрещала unmount внутри `synchronized`: корректность ценой масштабируемости. Вторая причина — выполнение **нативного кода** (JNI/FFM): чужой стек JVM переносить в heap не умеет.

Это не теоретическая страшилка. Вот замер с той же одноядерной машины (JDK 21): 8 виртуальных потоков, каждый спит 500 мс, держа **свой собственный** лок — то есть contention нет вообще, чистый эффект pinning:

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 8).forEach(i -> executor.submit(() -> {
        Object ownLock = new Object();              // у каждого свой монитор
        synchronized (ownLock) {
            Thread.sleep(Duration.ofMillis(500));   // блокировка, держа монитор
        }
        return i;
    }));
}
```

|Вариант|Время (JDK 21, 1 carrier)|
|---|---|
|`synchronized` + `sleep`|**4016 ms** — pinning, задачи выстроились в очередь за carrier|
|`ReentrantLock` + `sleep`|**522 ms** — unmount работает, все 8 ожиданий наложились|

Разница в 8 раз — ровно потому, что примитивы из `java.util.concurrent.locks` (глава 5) построены на `LockSupport.park()`, который дружит с механизмом unmount, а `synchronized` на Java 21 — нет. Диагностика на 21: флаг `-Djdk.tracePinnedThreads=full` печатает стектрейс с пометкой `reason:MONITOR` и стрелкой `<== monitors:1` на виновном фрейме; в проде — JFR-событие `jdk.VirtualThreadPinned`.

#### Что изменилось после 21

JEP 491 «Synchronize Virtual Threads without Pinning», вошедший в JDK 24, изменил реализацию `synchronized` в JVM так, что виртуальный поток может размонтироваться, находясь внутри `synchronized`-метода или блока, ожидая монитор или в `Object.wait()`. Суть исправления: монитор теперь ассоциируется с самим виртуальным потоком, а не с несущим — та самая бухгалтерия, из-за которой pinning был необходим, переписана. Табличка выше на JDK 24+ схлопывается: оба варианта дают ~520 мс. [Mike my bytes](https://mikemybytes.com/2025/04/09/java24-thread-pinning-revisited/)[GitHub](https://github.com/eclipse-jdt/eclipse.jdt.core/issues/3294)

Следствия для тебя:

- **На Java 21** (наш базовый LTS): блокирующие вызовы на горячих путях внутри `synchronized` — реальная проблема; решение — `ReentrantLock` (глава 5) или вынос блокировки из критической секции.
- **На Java 24+**: совет «меняйте `synchronized` на `ReentrantLock` ради виртуальных потоков» устарел — выбирай примитив по задаче, а не из страха pinning. Флаг `jdk.tracePinnedThreads` при этом из JDK 24 удалён — за оставшимися случаями следят через JFR. [GitHub](https://github.com/eclipse-jdt/eclipse.jdt.core/issues/3294)[GitHub](https://github.com/eclipse-jdt/eclipse.jdt.core/issues/3294)
- **Pinning не исчез полностью**: нативный код (JNI и Foreign Function & Memory API) по-прежнему прибивает виртуальный поток к carrier, плюс редкие случаи вроде блокировки в инициализаторах классов.

На собесе это выигрышная вилка: назвать pinning, объяснить механику через владение монитором — и добавить, что с JDK 24 (JEP 491) `synchronized` больше не пинит. Вторая часть отличает того, кто читал доки в 2023-м, от того, кто в теме сейчас.

### 7.6. Грабли №2: `ThreadLocal` на миллионах потоков

`ThreadLocal` (глава 23 основного курса / глава 1 здесь) проектировался под мир, где потоков сотни и они живут долго. В мире виртуальных потоков обе посылки ложные: потоков миллионы, живут они миллисекунды. Копия «потоколокального» значения на каждый из миллиона потоков — это заметная память; а паттерн «дорогой объект в `ThreadLocal`, чтобы переиспользовать между задачами» вообще теряет смысл — поток одноразовый, переиспользовать не с кем.

Направление, куда движется язык, — `ScopedValue`: неизменяемое значение, привязанное к _области выполнения_, а не к потоку, и дёшево наследуемое дочерними потоками. На Java 21 это preview; финализирован `ScopedValue` в JDK 25 (JEP 506). Детали — в 🔬-блоке. [OpenJDK](https://openjdk.org/jeps/453)

### 7.7. Структурная конкурентность: следующий шаг

Виртуальные потоки сняли ограничение на _количество_ потоков — и тут же обнажили следующую проблему: их _жизненные циклы_. Типовой сценарий: обработчик запроса порождает две подзадачи — сходить в user-service и в order-service — и сливает результаты. С `ExecutorService` или `CompletableFuture` эти подзадачи никак не связаны ни друг с другом, ни с родителем:

- user-service упал через 100 мс — а запрос к order-service продолжает крутиться впустую ещё 10 секунд, жгя ресурсы;
- родителя прервали — подзадачи об этом не узнают и утекут;
- в thread dump эти потоки — просто плоский список, по которому не видно, кто кого породил и зачем.

Отмену и пробрасывание ошибок приходится собирать вручную, и в главе 3 мы видели, насколько это муторно даже для двух задач.

**Структурная конкурентность** переносит на потоки принцип, который в последовательном коде кажется самоочевидным: подзадача не живёт дольше породившего её блока. Ментальная модель: **`try-with-resources`, но для параллельных задач — лексическая область видимости управляет их временем жизни.** Подзадачи рождаются внутри блока, и блок не завершится, пока не завершатся (или не будут отменены) они все.

#### API на Java 21 — и честная пометка про статус

Здесь обязательная оговорка. Structured concurrency прошла инкубацию в JDK 19–20 и с JDK 21 (JEP 453) живёт в статусе preview, переезжая из версии в версию: JEP 462 (22), 480 (23), 499 (24), 505 (25) — уже с серьёзной ломкой API, — и JEP 525 (26); на JDK 27 предложен уже седьмой preview (JEP 533). То есть на середину 2026 года это **всё ещё preview**, и API реально менялся. Учи _идею_ — она стабильна; конкретные сигнатуры сверяй с Javadoc своей версии. [OpenJDK](https://openjdk.org/jeps/453)[OpenJDK](https://openjdk.org/jeps/453)

Код ниже — форма API **Java 21** (JEP 453), скомпилирован и запущен мной с `--enable-preview`:

```java
import java.util.concurrent.StructuredTaskScope;
import java.util.concurrent.StructuredTaskScope.Subtask;

record Profile(User user, Orders orders) {}

Profile fetchProfile(long id) throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        Subtask<User> user = scope.fork(() -> fetchUser(id));      // подзадача 1
        Subtask<Orders> orders = scope.fork(() -> fetchOrders(id)); // подзадача 2

        scope.join()              // ждём обе разом
             .throwIfFailed();    // любая упала → её исключение наружу

        return new Profile(user.get(), orders.get());
    } // close(): гарантированно дожидается/отменяет все подзадачи
}
// компиляция и запуск на 21: javac --enable-preview --release 21 …
//                            java --enable-preview …
```

Каждый `fork` запускает подзадачу в новом виртуальном потоке. Политика `ShutdownOnFailure` — «все или никто»: падение любой подзадачи отменяет (через `interrupt`) остальные, а `throwIfFailed()` пробрасывает причину родителю. Я проверил и сценарий отказа: подзадача A падает через 100 мс, подзадача B «работает» 10 секунд — в реальном запуске B получила interrupt и завершилась на **135-й миллисекунде**, а родитель получил исключение A на 152-й. Никакой ручной оркестрации отмены — она следует из структуры блока.

![[Pasted image 20260704151143.png]]

Бонус, который недооценивают: структура видна инструментам. Thread dump в JSON-формате (`jcmd <pid> Thread.dump_to_file -format=json`) показывает подзадачи как дерево под их scope — впервые дамп отражает то, как программу видит разработчик, а не плоскую кучу потоков.

#### Куда API уехал после 21

В JDK 25 (JEP 505) API заметно перекроили: публичные конструкторы `StructuredTaskScope` заменили статическими фабриками, а политики-подклассы (`ShutdownOnFailure` / `ShutdownOnSuccess`) — отдельной абстракцией `Joiner`. Та же логика в форме 25+ выглядит так (на 21 это **не** скомпилируется):

java

```java
// Форма API JDK 25+ (JEP 505), по-прежнему preview — сверяй со своей версией
try (var scope = StructuredTaskScope.open()) {      // fail-fast по умолчанию
    Subtask<User> user = scope.fork(() -> fetchUser(id));
    Subtask<Orders> orders = scope.fork(() -> fetchOrders(id));
    scope.join();                                   // при ошибке бросает сам
    return new Profile(user.get(), orders.get());
}
// другие политики: StructuredTaskScope.open(Joiner.anySuccessfulResultOrThrow()) и т.п.
```

Идея — рождение и смерть подзадач в одном блоке, централизованные ошибки и отмена — не менялась с первого инкубатора. Меняется обвязка. На собесе честная формулировка: «идея стабильна, API в preview и ещё дорабатывается; на 21 это `ShutdownOnFailure`, с 25 — фабрика `open()` и `Joiner`».

> **Та же проблема, другое решение.** Виртуальные потоки решают ровно ту задачу, что async/await в .NET: масштабирование I/O-bound нагрузки без потока ОС на каждое ожидание. Но .NET решил её на уровне языка и компилятора — async-метод переписывается в стейт-машину, ожидания становятся точками возврата, и появляется два цвета функций. Java решила её на уровне рантайма — JVM научилась дёшево замораживать и размораживать стек потока целиком. Поэтому в Java нет `await`, нет `Task`-обёрток, нет вирусной асинхронной сигнатуры: старый блокирующий метод, написанный в 2010 году, запущенный в виртуальном потоке, масштабируется «сам». Весь существующий синхронный код — JDBC, старые библиотеки — получает выигрыш без переписывания.
> 
> ⚠️ **Ложный друг:** виртуальный поток — это не async-стейт-машина и не корутина уровня языка. Не переноси интуицию async/await один в один: здесь нет точек `await`, нет `ConfigureAwait`, нет «синхронного» и «асинхронного» вариантов метода. Механика другая — mount/unmount стека силами рантайма, — даже если цель совпадает. И наоборот: `Task.Run` в .NET берёт поток из общего пула (дорогой ресурс!), а `Thread.ofVirtual().start()` создаёт одноразовый дешёвый поток — экономические модели противоположны.
> 
> Структурная конкурентность идейно перекликается с `Task.WhenAll` + `CancellationToken`: там ты тоже ждёшь группу задач и можешь отменять их скопом. Разница в том, что в .NET это ручная сборка — токен нужно создать, протащить в каждую задачу, не забыть проверять, а связь «родитель–дети» существует только у тебя в голове. `StructuredTaskScope` делает время жизни группы задач первоклассной конструкцией: границы блока и есть границы жизни подзадач, отмена и пробрасывание ошибок следуют из синтаксической структуры, и эту структуру видят даже дампы потоков.

> [!example] 🔬 Глубже / на синьора  
> **Планировщик — ForkJoinPool в FIFO-режиме.** Несущие потоки виртуальных потоков — это выделенный `ForkJoinPool` (не common pool из глав про параллельные стримы!), работающий в FIFO-режиме, с параллелизмом по умолчанию = `availableProcessors()`. Настраивается системным свойством `jdk.virtualThreadScheduler.parallelism`; верхняя граница числа несущих по умолчанию — 256. Мы видели его прямо в `toString()` потока: `@ForkJoinPool-1-worker-1`.
> 
> **Continuation — механизм заморозки стека.** Под капотом виртуальный поток = `Continuation` (внутренний класс `jdk.internal.vm.Continuation`) + планировщик. Unmount — это `Continuation.yield()`: JVM копирует активные фреймы стека виртуального потока со стека несущего в объекты в heap (stack chunks). Mount — обратная операция: фреймы возвращаются на стек нового carrier (лениво, по мере надобности). Именно поэтому стек виртуального потока «резиновый»: он занимает ровно столько heap, сколько фреймов реально живо, а не мегабайт по резерву. И именно поэтому нативные фреймы пинят: стек, в котором есть кадры C-кода с сырыми указателями, JVM переносить не умеет.
> 
> **Почему synchronized пинил, а Lock — нет.** `ReentrantLock` и прочие примитивы из `java.util.concurrent.locks` при ожидании зовут `LockSupport.park()` — а `park()` для виртуального потока реализован как yield его continuation. Монитор же — конструкция самой VM: до JDK 24 его владение учитывалось в терминах несущего потока, и unmount внутри `synchronized` сломал бы взаимное исключение. JEP 491 (JDK 24) перенёс учёт владения на сам виртуальный поток, с бухгалтерией на каждом mount/unmount — после чего запрет стал не нужен.
> 
> **ScopedValue — замена ThreadLocal для мира миллионов потоков.** Неизменяемое значение, связанное с областью выполнения:
> 
> java
> 
> ```java
> static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();
> 
> ScopedValue.where(REQUEST_ID, "req-42")
>            .run(() -> handle());   // внутри handle(): REQUEST_ID.get()
> ```
> 
> В отличие от `ThreadLocal`: значение нельзя переустановить (нет `set` — нет и проблемы «забыл `remove()` в пуле» из главы 1), привязка автоматически исчезает с выходом из области, а подзадачи `StructuredTaskScope` наследуют привязки родителя дёшево — без копирования, как это делает `InheritableThreadLocal`. Финализирован в JDK 25 (JEP 506); на нашей базовой 21 — ещё preview.
> 
> **CPU-bound и carrier'ы, цифры.** Виртуальных потоков может быть миллион, но исполняются одновременно максимум N — по числу несущих. Важный нюанс: планировщик виртуальных потоков на Java 21 невытесняющий по времени — виртуальный поток отдаёт carrier на блокирующих операциях, а не по тайм-слайсу. Долгая чистая арифметика без блокировок способна надолго оккупировать carrier и ухудшить latency соседей. Для счётных задач ответ прежний: пул размером с число ядер или Fork/Join (глава 8).

> [!warning] ⚠️ Грабли
> 
> 1. **Пуловать виртуальные потоки.** Обёртка `newFixedThreadPool(200, virtualFactory)` возвращает искусственный дефицит, который виртуальные потоки только что устранили. Нужен лимит на ресурс — `Semaphore` вокруг ресурса.
> 2. **Ждать выигрыша на CPU-bound.** «Заменю пул на виртуальные потоки — станет быстрее» — не станет: ядер больше не стало, а невытесняющий планировщик может сделать даже хуже. Считает — Fork/Join, ждёт — виртуальные потоки.
> 3. **Pinning: блокирующий вызов внутри `synchronized`** — на Java 21–23. Диагностика: `-Djdk.tracePinnedThreads=full` (удалён в 24) и JFR `jdk.VirtualThreadPinned`; лечение на 21 — `ReentrantLock` на горячем пути. На 24+ (JEP 491) эта граблина закопана — но нативный код (JNI/FFM) пинит по-прежнему.
> 4. **Тяжёлый `ThreadLocal` на массе потоков** — память × миллион, а паттерн «кеш в `ThreadLocal`» с одноразовыми потоками вовсе не работает. Направление — `ScopedValue`.
> 5. **`Subtask.get()` до `join()`** — в `StructuredTaskScope` это `IllegalStateException`: результаты подзадач читаются только после общего `join()`. API намеренно жёстче, чем `Future.get()`.
> 6. **Считать structured concurrency стабильным API.** Preview с 21-й по 26-ю как минимум; код на `ShutdownOnFailure` с 21 не скомпилируется на 25 без правок. Идею — в голову, сигнатуры — из Javadoc своей версии.

### На собесе

> [!question]- Q1. Чем виртуальный поток отличается от платформенного?  
> Follow-up: «А ОС про него знает?» — ловушка для тех, кто думает, что это «лёгкий поток ОС». Слабый ответ: «он легче и быстрее».
> 
> **RU:** Платформенный поток — обёртка 1:1 над потоком ОС: стек ~1 МБ, создание через syscall, планирует ядро; потолок — тысячи. Виртуальный поток — объект JVM: стек живёт в heap и растёт по надобности, создание — дешёвая аллокация, планирует JVM поверх небольшого пула несущих платформенных потоков (M:N). ОС о виртуальных потоках не знает. Их можно создавать миллионами, API — тот же `Thread`.
> 
> **EN:** A platform thread is a 1:1 wrapper over an OS thread — ~1 MB stack, syscall creation, kernel scheduling, so you get thousands at most. A virtual thread is a JVM object: its stack lives on the heap and grows on demand, creation is a cheap allocation, and the JVM schedules it M:N on a small pool of carrier platform threads. The OS is unaware of virtual threads; you can have millions, and the API is still Thread.

> [!question]- Q2. Что происходит, когда виртуальный поток блокируется?  
> Follow-up: «А несущий поток в это время что делает?»
> 
> **RU:** На блокирующей операции (I/O, sleep, park) JVM размонтирует виртуальный поток с несущего: состояние его стека сохраняется в heap, а на освободившийся carrier монтируется другой готовый виртуальный поток. Когда операция завершается, поток возвращается в очередь планировщика и монтируется заново — возможно, на другой carrier. Несущий поток не блокируется никогда — блокировка стала дешёвой, поэтому обычный блокирующий код масштабируется как асинхронный.
> 
> **EN:** On a blocking operation the JVM unmounts the virtual thread: its stack frames are saved to the heap and the freed carrier immediately mounts another runnable virtual thread. When the operation completes, the thread is rescheduled and remounted, possibly on a different carrier. The carrier itself never blocks — that's why plain blocking code scales like async code.

> [!question]- Q3. Почему виртуальные потоки не нужно пуловать?  
> Follow-up: «А как тогда ограничить нагрузку на базу?»
> 
> **RU:** Пул существует, чтобы переиспользовать дорогой ресурс. Виртуальный поток дешёвый и одноразовый — «поток на задачу» снова корректный паттерн, поэтому стандартный executor так и называется: `newVirtualThreadPerTaskExecutor`, он не переиспользует потоки. Пул виртуальных потоков лишь возвращает искусственное ограничение. Лимитировать доступ к конкретному ресурсу нужно `Semaphore` вокруг ресурса, а не дефицитом потоков.
> 
> **EN:** Pooling exists to reuse an expensive resource; a virtual thread is cheap and disposable, so thread-per-task is the right pattern again — newVirtualThreadPerTaskExecutor creates a fresh thread per task. Pooling them just reintroduces an artificial bottleneck. To limit concurrency against a resource, use a Semaphore around that resource.

> [!question]- Q4. Что такое pinning и когда он возникает?  
> Follow-up: «Это ещё актуально на новых JDK?» — вопрос-водораздел.
> 
> **RU:** Pinning — ситуация, когда виртуальный поток не может размонтироваться и держит carrier во время блокировки, теряя весь выигрыш. На Java 21 главная причина — блокирующий вызов внутри `synchronized`: JVM учитывала владение монитором по несущему потоку, и unmount сломал бы взаимное исключение; лечили заменой на `ReentrantLock`. С JDK 24 (JEP 491) владение монитором привязано к самому виртуальному потоку, и `synchronized` больше не пинит. Остаётся pinning на нативном коде (JNI/FFM). Диагностика — JFR-событие `jdk.VirtualThreadPinned`.
> 
> **EN:** Pinning is when a virtual thread cannot unmount and holds its carrier while blocked, losing the scalability benefit. On Java 21 the main cause was blocking inside synchronized — monitor ownership was tracked per carrier, so unmounting would break mutual exclusion; the workaround was ReentrantLock. Since JDK 24 (JEP 491) monitors are tracked per virtual thread and synchronized no longer pins; native code (JNI/FFM) still does. Diagnose via the jdk.VirtualThreadPinned JFR event.

> [!question]- Q5. Когда виртуальные потоки НЕ помогают?  
> **RU:** На CPU-bound нагрузке. Виртуальные потоки удешевляют ожидание, а не вычисление: если задача не блокируется, её нечего размонтировать, и параллелизм упирается в число несущих потоков — то есть ядер. Плюс планировщик невытесняющий: долгая счётная задача монополизирует carrier. Для вычислений — пул по числу ядер или Fork/Join.
> 
> **EN:** For CPU-bound work. Virtual threads make waiting cheap, not computing: a task that never blocks never unmounts, so parallelism is still capped by carriers, i.e. CPU cores — and the scheduler is non-preemptive, so a long computation monopolizes its carrier. Use a fixed-size pool or Fork/Join for computation.

> [!question]- Q6. Зачем нужна структурная конкурентность, какую проблему она решает?  
> Follow-up: «Это стабильный API?» — проверка на актуальность.
> 
> **RU:** Она связывает время жизни параллельных подзадач с лексическим блоком — как try-with-resources, но для задач. Проблема: подзадачи, запущенные через executor или `CompletableFuture`, никак не связаны с родителем — одна упала, остальные крутятся впустую, отмена и ошибки собираются вручную, а в дампе это плоский список. `StructuredTaskScope`: подзадачи рождаются и умирают в одном блоке, падение одной отменяет остальных (политика вроде `ShutdownOnFailure` на 21), `join()` централизует ошибки, дампы показывают дерево. Статус — preview: с JDK 21 (JEP 453), в JDK 25 API переработан (JEP 505: фабрика `open()` + `Joiner`), финализации пока нет.
> 
> **EN:** It ties the lifetimes of concurrent subtasks to a lexical block — try-with-resources for tasks. Without it, subtasks launched via an executor aren't linked to their parent: one fails, the rest keep burning resources, and cancellation/error handling is manual. With StructuredTaskScope, subtasks are born and die within one block, one failure cancels the siblings, join() centralizes errors, and thread dumps show the tree. Still a preview API: since JDK 21 (JEP 453), significantly reworked in JDK 25 (JEP 505 — open() factory plus Joiner).

> [!question]- Q7. Чем это отличается от async/await в C#?  
> **RU:** Цель одна — дешёвое ожидание I/O, — механика противоположная. C# решает на уровне компилятора: async-метод переписывается в стейт-машину, появляются `Task`-обёртки и два «цвета» функций, асинхронность вирусно ползёт по сигнатурам. Java решает на уровне рантайма: JVM замораживает и размораживает стек целиком (mount/unmount), поэтому код остаётся обычным блокирующим — без `await`, без обёрток, без окраски; существующий синхронный код масштабируется без переписывания. Виртуальный поток — не стейт-машина и не корутина уровня языка.
> 
> **EN:** Same goal — cheap I/O waiting — opposite mechanics. C# solves it in the compiler: async methods become state machines, with Task wrappers and the two-colored-function problem spreading through signatures. Java solves it in the runtime: the JVM freezes and restores whole thread stacks (mount/unmount), so code stays plainly blocking — no await, no wrappers, no coloring — and existing synchronous code scales without rewrites. A virtual thread is not a language-level state machine or coroutine.
