
В главе 1 мы разобрали, что ломается, когда потоки делят изменяемые данные: атомарность, видимость, переупорядочивание. Чинить это мы будем дальше — локами, атомиками, конкурентными коллекциями. Но перед этим нужно закрыть более базовый вопрос: как вообще запускать работу в нескольких потоках так, чтобы приложение это пережило. Создать поток руками несложно — `new Thread(...).start()` — и именно поэтому наивный путь так опасен: он отлично работает в учебном примере и разваливается под первой же реальной нагрузкой. Эта глава — про цивилизованную замену: пулы потоков и абстракцию `ExecutorService`.

### Почему не `new Thread()` на каждую задачу

Представьте обработчик входящих запросов, написанный «в лоб»:

```java
while (true) {
    Request request = acceptRequest();
    new Thread(() -> handle(request)).start(); // поток на каждый запрос
}
```

Компилируется, работает, на демо выглядит убедительно. Проблемы у этого кода три, и все — фундаментальные.

**Поток — дорогой ресурс.** За каждым Java-потоком (до virtual threads, о них — в отдельной главе) стоит поток операционной системы: собственный стек — по умолчанию порядка мегабайта (`-Xss`), — структуры ядра и участие в планировании. Создание и уничтожение потока — это системные вызовы, то есть дорогие операции. Тратить всё это на задачу, которая выполняется миллисекунды, — всё равно что нанимать нового сотрудника на каждое письмо и увольнять после отправки.

**Нет лимита.** В коде выше ничто не ограничивает число потоков. Тысяча запросов в секунду — тысяча живых потоков. Дальше два сценария: либо процессор тонет в переключениях контекста и полезная работа почти останавливается, либо JVM не может выделить очередной стек и падает с `OutOfMemoryError: unable to create native thread`. И заметьте: потоки не переиспользуются — создали, отработал, выбросили, а следующий запрос платит всю цену заново.

**Нет управления.** С «голыми» потоками нет нормального способа получить результат работы (у `Runnable` метод `run()` возвращает `void`), дождаться завершения группы задач, отменить лишнее, ограничить параллелизм. Всё это приходится собирать вручную из `join()` и общих переменных — то есть ровно из того, что в главе 1 мы назвали минным полем.

### `ExecutorService`: отделяем «что» от «как»

Решение, которое предлагает `java.util.concurrent`, — развести два вопроса, которые наивный код склеивает в один. **Ментальная модель главы: задача описывает, _что_ выполнить; executor решает, _как, где и когда_ это выполнить.** Вы отдаёте задачу пулу — а пул сам держит заранее созданные потоки, ставит задачи в очередь и раздаёт их свободным воркерам. Потоки переиспользуются: воркер закончил одну задачу — взял из очереди следующую.

Задача — это либо `Runnable`, либо `Callable<T>`. Разница практическая: `Callable` возвращает результат и может бросить checked-исключение.

```java
Runnable log = () -> System.out.println("side effect, ничего не возвращаю");

Callable<Report> buildReport = () -> {
    Report r = loadAndAggregate();  // может бросить IOException — это легально
    return r;                       // и возвращает результат
};
```

Отдать задачу пулу можно двумя методами. `execute(Runnable)` — «выстрелил и забыл», никакой ручки на задачу вы не получаете. `submit(...)` принимает и `Runnable`, и `Callable` и возвращает `Future` — объект-квитанцию, по которой позже можно забрать результат (о нём — ниже).

```java
ExecutorService pool = Executors.newFixedThreadPool(4);

pool.execute(log);                            // fire-and-forget
Future<Report> future = pool.submit(buildReport); // квитанция на результат
```

Вот как это устроено картинкой: очередь задач посередине, воркеры разбирают её сами.

### Готовые фабрики `Executors` — и почему в проде обычно не они

Класс `Executors` предлагает фабрики на все случаи жизни, и три из них встречаются постоянно:

```java
ExecutorService fixed  = Executors.newFixedThreadPool(8);  // ровно 8 потоков
ExecutorService cached = Executors.newCachedThreadPool();  // потоки «по требованию»
ExecutorService single = Executors.newSingleThreadExecutor(); // один поток, задачи по порядку
```

`newFixedThreadPool(n)` держит ровно `n` потоков; всё, что не влезло, ждёт в очереди. `newCachedThreadPool` создаёт поток под каждую задачу, если свободного нет, и убивает потоки, простоявшие без работы 60 секунд, — годится для множества коротких задач. `newSingleThreadExecutor` — один поток, задачи выполняются строго последовательно; удобно, когда нужен порядок.

Звучит так, будто выбирать пул — дело одной строки. И вот здесь спрятана мина, ради которой этот раздел существует: **фабрики скрывают опасные дефолты**. Обе популярные фабрики — это на самом деле обёртки над одним и тем же классом `ThreadPoolExecutor`, просто с зашитыми параметрами. Вот что реально конструируется (проверено на JDK 21 — параметры вытащены из живых пулов):

```java
// newFixedThreadPool(8) — это:
new ThreadPoolExecutor(8, 8, 0L, TimeUnit.MILLISECONDS,
        new LinkedBlockingQueue<Runnable>());   // очередь БЕЗ лимита (2^31−1)

// newCachedThreadPool() — это:
new ThreadPoolExecutor(0, Integer.MAX_VALUE,    // потоков — БЕЗ лимита
        60L, TimeUnit.SECONDS,
        new SynchronousQueue<Runnable>());      // очередь нулевой ёмкости
```

Теперь читаем это глазами продакшена. У `newFixedThreadPool` жёстко ограничены потоки, но **не ограничена очередь**: если задачи приходят быстрее, чем 8 потоков успевают их разгребать, очередь молча растёт. Час, два, сутки — задачи копятся в heap, пока не прилетит `OutOfMemoryError`. Хуже того: система всё это время выглядит живой и принимает работу, вместо того чтобы притормозить источник и честно сказать «я не успеваю». У `newCachedThreadPool` зеркальная проблема: очередь нулевая, зато **не ограничены потоки** — резкий наплыв задач превращается в тысячи живых потоков, то есть ровно в тот сценарий, от которого мы убегали от `new Thread()`.

Отсюда правило, которое стоит унести из главы: фабрики — для прототипов, тестов и заведомо небольших нагрузок. В проде пул под нагрузку конфигурируют руками через `ThreadPoolExecutor`, осознанно выбирая и лимит потоков, и лимит очереди, и поведение при переполнении.

### `ThreadPoolExecutor` напрямую

Полный конструктор выглядит так:

```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
        4,                                   // corePoolSize — «постоянный штат»
        8,                                   // maximumPoolSize — потолок при перегрузке
        30, TimeUnit.SECONDS,                // keepAliveTime — сколько живёт лишний поток без работы
        new ArrayBlockingQueue<>(100),       // workQueue — ОГРАНИЧЕННАЯ очередь
        runnable -> {                        // threadFactory — именуем потоки для дебага
            Thread t = new Thread(runnable, "order-worker");
            return t;
        },
        new ThreadPoolExecutor.CallerRunsPolicy() // что делать при переполнении
);
```

Смысл параметров: core-потоки — постоянный штат, который пул стремится держать; потоки сверх core (до `maximumPoolSize`) — «подмога», которую пул нанимает при перегрузке и распускает после `keepAliveTime` простоя; `workQueue` — буфер между приёмом задач и их выполнением; `threadFactory` даёт потокам осмысленные имена (в thread dump «order-worker» ищется, «pool-7-thread-3» — нет); `RejectedExecutionHandler` решает судьбу задачи, когда всё заполнено.

А теперь — самое контринтуитивное место всей главы, то, что путают почти все на собеседованиях. Интуиция подсказывает: «сначала пул растит потоки до максимума, потом складывает излишки в очередь». **На самом деле — наоборот.** Порядок принятия решения при поступлении задачи такой:

1. Потоков меньше `corePoolSize` → создать новый core-поток, даже если есть простаивающие.
2. Core заполнен → задача идёт **в очередь**. Не в новый поток — в очередь.
3. Очередь заполнена → и только теперь пул растёт сверх core, до `maximumPoolSize`.
4. И потоки на максимуме, и очередь полна → задача отклоняется, срабатывает `RejectedExecutionHandler`.

![[Pasted image 20260704143605.png]]

Не верьте на слово — вот эксперимент, который это доказывает. Пул: core = 2, max = 4, очередь на 3 задачи. Подаём 9 «висящих» задач и после каждой смотрим состояние:

```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
        2, 4, 30, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(3),
        (task, executor) -> rejected.incrementAndGet());

CountDownLatch hold = new CountDownLatch(1);
for (int i = 1; i <= 9; i++) {
    pool.execute(() -> { try { hold.await(); } catch (InterruptedException e) { Thread.currentThread().interrupt(); } });
    System.out.printf("после задачи %d: потоков=%d, в очереди=%d, отклонено=%d%n",
            i, pool.getPoolSize(), pool.getQueue().size(), rejected.get());
}
```

Реальный вывод на JDK 21:

```text
после задачи 1: потоков=1, в очереди=0, отклонено=0
после задачи 2: потоков=2, в очереди=0, отклонено=0   ← core заполнен
после задачи 3: потоков=2, в очереди=1, отклонено=0   ← идём в очередь,
после задачи 4: потоков=2, в очереди=2, отклонено=0     а НЕ создаём потоки
после задачи 5: потоков=2, в очереди=3, отклонено=0   ← очередь заполнена
после задачи 6: потоков=3, в очереди=3, отклонено=0   ← только теперь пул растёт
после задачи 7: потоков=4, в очереди=3, отклонено=0   ← упёрлись в max
после задачи 8: потоков=4, в очереди=3, отклонено=1   ← reject
после задачи 9: потоков=4, в очереди=3, отклонено=2
```

Отсюда прямое следствие, замыкающее историю про `newFixedThreadPool`: если очередь безлимитная, шаг «очередь заполнена» не наступает **никогда** — а значит, `maximumPoolSize` мёртвый параметр, пул никогда не вырастет сверх core. Именно поэтому у fixed-фабрики core и max честно равны: разница всё равно не сыграла бы.

**Что происходит при reject** — решает `RejectedExecutionHandler`, и это осознанный выбор стратегии деградации:

- `AbortPolicy` (по умолчанию) — бросает `RejectedExecutionException` тому, кто вызвал `submit`/`execute`. Громко и честно: перегрузка становится видимой ошибкой, которую можно обработать.
- `CallerRunsPolicy` — задача выполняется **в потоке вызывающего**. Элегантный побочный эффект: пока вызывающий поток занят чужой работой, он не подаёт новые задачи — источник естественным образом притормаживается. Это простейшая форма backpressure.
- `DiscardPolicy` и `DiscardOldestPolicy` — молча выбросить новую задачу или самую старую из очереди. Молчаливая потеря работы — почти всегда плохой выбор; годится разве что для телеметрии, где свежие данные ценнее старых.

**Остановка пула.** Пул с не-daemon потоками не даст JVM завершиться, пока его не погасили. Корректный паттерн двухступенчатый:

```java
pool.shutdown();                     // новые задачи не принимаем, начатые и очередь — дорабатываем
try {
    if (!pool.awaitTermination(30, TimeUnit.SECONDS)) {
        pool.shutdownNow();          // прерываем работающие, возвращает невыполненные задачи
    }
} catch (InterruptedException e) {
    pool.shutdownNow();
    Thread.currentThread().interrupt();
}
```

Важная пара различий: `shutdown()` — мягкий («доделай, что взял, и очередь тоже»), `shutdownNow()` — жёсткий (шлёт `interrupt()` работающим потокам и возвращает `List<Runnable>` задач, до которых очередь не дошла). И ни один из них не блокирует — ждать фактического завершения нужно отдельно, через `awaitTermination`.

### `Future`: ручка результата

Мы говорили, что `submit` возвращает квитанцию. Формально это `Future<T>` — объект, представляющий результат, которого, возможно, ещё нет:

```java
Future<Integer> future = pool.submit(() -> {
    Thread.sleep(500);          // имитация долгой работы
    return 42;
});

// main свободен и делает свои дела...

Integer result = future.get(); // а вот здесь main ВСТАЁТ и ждёт готовности
```

Ключевое слово — «встаёт». `get()` — блокирующий вызов: вызывающий поток останавливается до завершения задачи (в проверочном прогоне выше main ждал ровно ~500 мс). Кроме него у `Future` есть `get(timeout, unit)` — ждать не дольше указанного и бросить `TimeoutException`; `cancel(mayInterruptIfRunning)` — отменить (ещё не начатая задача просто не запустится, уже работающей при `true` пошлют `interrupt()`); `isDone()` и `isCancelled()` — неблокирующие проверки состояния. Если задача завершилась исключением, `get()` бросит `ExecutionException`, а оригинальная причина лежит в `getCause()`. Внутри пула роль «задача + её Future в одном объекте» играет класс `FutureTask` — это `Runnable` и `Future` одновременно; именно в него `submit` оборачивает вашу задачу.

![[Pasted image 20260704143640.png]]

И сразу честно про ограничение, которое видно на этой картинке: `Future` умеет ждать только блокирующе. Нельзя сказать «когда будет готов отчёт А — построй из него письмо Б, а потом отправь», не заняв поток ожиданием на каждом шаге. Нет композиции, нет продолжений, нет реакции на готовность. Именно эту дыру закрывает `CompletableFuture` — герой следующей главы.

> [!tip]
> **💡 Из C#:** `ExecutorService` по роли — как отдать работу в `ThreadPool` через `Task.Run`; `Callable<T>` — как `Func<T>`, которую вы туда передаёте. Но ⚠️ ложный друг: `Future` ≠ `Task`. У `Task` есть `await` и `ContinueWith` — ожидание без блокировки потока и композиция цепочек; `Future.get()` умеет только встать и ждать. Аналог привычного вам `Task` появится лишь в лице `CompletableFuture`. И второе отличие культур: в .NET пул по умолчанию один и им управляет рантайм, а в Java вы регулярно конфигурируете собственный `ThreadPoolExecutor` под конкретную нагрузку — руками, с числами, которые обосновали.

> [!exmaple]
> **🔬 Глубже / на синьора.** Внутри `ThreadPoolExecutor` состояние пула и число воркеров упакованы в один `AtomicInteger` (`ctl`): старшие 3 бита — состояние жизненного цикла (`RUNNING → SHUTDOWN → STOP → TIDYING → TERMINATED`), остальные 29 — счётчик потоков; это позволяет атомарно, без лока, принимать то самое решение «core / очередь / рост / reject» из диаграммы. `shutdown()` переводит пул в `SHUTDOWN` (очередь дорабатывается), `shutdownNow()` — в `STOP` (воркерам шлётся `interrupt`, очередь дренируется). Тонкие ручки: `prestartAllCoreThreads()` — создать core-потоки заранее, не дожидаясь первых задач (полезно, чтобы не платить за создание потоков на первом же всплеске); `allowCoreThreadTimeOut(true)` — разрешить умирать и core-потокам после `keepAliveTime`, превращая пул в «полностью эластичный». И заметка на будущее: `ForkJoinPool` — принципиально другой зверь: у каждого воркера своя дека задач, и простаивающие потоки _воруют_ работу у занятых (work-stealing); на нём стоят parallel streams. Детально — в главе про Fork/Join.

**⚠️ Подводные камни.**

> [!warning]
> Первый и самый коварный — **`submit` молча проглатывает исключения**. При `execute(Runnable)` необработанное исключение улетает в `UncaughtExceptionHandler` потока и хотя бы печатается в stderr. При `submit` оно упаковывается в `Future` и не проявится **никак**, пока кто-нибудь не вызовет `get()`. Проверено: задача упала, в логах тишина, `isDone() == true`. Забыли про `Future` — потеряли ошибку. Правило: вызвали `submit` — кто-то обязан вызвать `get()` (или в задаче стоит свой try/catch с логированием).
> 
> **Забытый `shutdown()`** — приложение «не выходит». Потоки пула по умолчанию не daemon; `main` завершился, а JVM живёт, потому что воркеры ждут задач. Симптом знакомый каждому: программа отработала и висит.
> 
> **Deadlock через голодание пула (pool starvation).** Задача внутри пула делает `submit` в _тот же_ пул и блокируется в `get()`, ожидая результат. Если все потоки пула заняты такими «родителями», дочерним задачам не на чем выполниться — и все стоят вечно, ожидая друг друга. Классическая ловушка ограниченных пулов; не сабмитьте зависимые подзадачи в тот же пул, где ждёте их результата.
> 
> **`CallerRunsPolicy` не бесплатна.** Задача выполняется в потоке вызывающего — если это поток веб-сервера, его latency внезапно включает чужую работу. И нюанс: если пул уже остановлен (`shutdown`), эта политика задачу молча отбрасывает, а не выполняет.
> 
> **`ThreadLocal` + пул = утечка.** Потоки живут долго и переиспользуются: значение, положенное в `ThreadLocal` одной задачей, достанется следующей задаче на том же потоке — вместе с утечкой памяти и потенциально чужими данными. Положили — очистите (`remove()`) в `finally`. Подробнее — в главе про `ThreadLocal`.

### На собесе

> [!question]- **Чем отличаются `newFixedThreadPool`, `newCachedThreadPool`, `newSingleThreadExecutor` и когда какой?**  
> Эталон: fixed — постоянное число потоков и безлимитная очередь, для стабильной предсказуемой нагрузки; cached — потоки по требованию без верхней границы (простаивающие умирают через 60 с), для множества коротких задач; single — один поток, гарантирует последовательный порядок выполнения. В проде под серьёзную нагрузку обычно не фабрики, а сконфигурированный `ThreadPoolExecutor`, потому что фабрики прячут опасные дефолты: безлимитную очередь или безлимитные потоки.  
> EN: _Fixed keeps N threads with an unbounded queue; cached creates threads on demand with no upper bound; single runs tasks sequentially on one thread. In production we usually configure `ThreadPoolExecutor` manually, because the factories hide dangerous defaults — an unbounded queue or unbounded thread count._

> [!question]- **Что произойдёт с `newFixedThreadPool` при наплыве задач, с которым потоки не справляются?**  
> Эталон: задачи будут копиться в безлимитной `LinkedBlockingQueue`. Потоков больше не станет (core = max), reject не сработает никогда — очередь будет расти, пока не съест heap, и всё закончится `OutOfMemoryError`. Система при этом долго выглядит здоровой, что и делает проблему коварной. Ловушка вопроса — ответить «создаст больше потоков»: не создаст.  
> EN: _Tasks pile up in the unbounded `LinkedBlockingQueue`. The pool never grows (core equals max) and rejection never triggers, so memory keeps growing until an `OutOfMemoryError`. The trap is answering "it creates more threads" — it doesn't._

> [!question]- **Как взаимодействуют `corePoolSize`, `maximumPoolSize` и очередь?**  
> Эталон: порядок контринтуитивный — сначала создаются потоки до core; дальше задачи идут в очередь; только когда очередь заполнена, пул растёт до max; когда исчерпано и то и другое — reject. Следствие: при безлимитной очереди `maximumPoolSize` не задействуется никогда. Follow-up, который почти наверняка прилетит: «то есть при свободной очереди пул не вырастет сверх core, даже если задач много?» — да, именно так.  
> EN: _Threads are created up to core first; then tasks go to the queue — not to new threads; only when the queue is full does the pool grow toward max; only when both are exhausted does rejection kick in. With an unbounded queue, `maximumPoolSize` is effectively dead._

> [!question]- **Чем `Callable` отличается от `Runnable`?**  
> Эталон: `Callable<T>` возвращает результат и может бросать checked-исключения; `Runnable` — `void` и без checked. `submit(Callable)` возвращает `Future<T>` с результатом; исключение задачи достаётся через `ExecutionException.getCause()`.  
> EN: _`Callable<T>` returns a value and may throw checked exceptions; `Runnable` returns nothing and can't. `submit` wraps the outcome in a `Future`; a failure surfaces as `ExecutionException` on `get()`._

> [!question]-  **Как корректно остановить пул?**  
> Эталон: `shutdown()` — перестать принимать новое, доработать очередь; затем `awaitTermination(timeout)`; если не уложились — `shutdownNow()`, который прерывает работающие потоки и возвращает список невыполненных задач. Ни `shutdown`, ни `shutdownNow` сами не ждут — ожидание только через `awaitTermination`.  
> EN: _Call `shutdown()` (no new tasks, queue drains), then `awaitTermination(timeout)`; if it doesn't finish in time, `shutdownNow()`, which interrupts workers and returns the tasks that never ran. Neither shutdown method blocks by itself._

> [!question]-  **Почему `Future.get()` считается ограничением? (мостик на сеньорский follow-up)**  
> Эталон: `get()` блокирует поток до готовности — нельзя навесить продолжение, скомбинировать несколько результатов или отреагировать на завершение без занятого ожиданием потока. Для пайплайнов «когда готово А — сделай Б» нужен `CompletableFuture` с его `thenApply`/`thenCompose`. Бонус-линия на сеньора: блокирующий `get()` внутри задач того же пула — прямой путь к pool starvation deadlock.  
> EN: _`get()` can only block — no continuations, no composition, no non-blocking reaction to completion. That's exactly what `CompletableFuture` fixes. Senior add-on: calling a blocking `get()` from tasks inside the same bounded pool is a classic recipe for pool-starvation deadlock._
