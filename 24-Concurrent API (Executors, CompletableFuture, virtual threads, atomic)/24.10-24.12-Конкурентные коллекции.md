В главах 4 и 5 мы защищали общие данные вручную: `synchronized`, атомики, `ReentrantLock`, `Condition`. Это работает, но у ручного подхода есть цена — каждое обращение к общей коллекции нужно не забыть обернуть в лок, каждую составную операцию — продумать, и одна пропущенная точка синхронизации превращается в гонку, которую ты будешь ловить неделю. Логичный следующий шаг: коллекции, у которых потокобезопасность встроена внутрь — правильно, эффективно и один раз. Именно они и лежат в `java.util.concurrent`. Но прежде чем брать `ConcurrentHashMap`, нужно понять, почему более простой ответ — «оберни всё в `synchronized`» — не работает. Это ключевой момент всей главы, и на собеседовании именно на нём проверяют, понимаешь ты тему или заучил названия классов.

**Ментальная модель главы:** потокобезопасность отдельных методов коллекции ≠ потокобезопасность твоей логики. Конкурентные коллекции ценны не тем, что «не падают под потоками», а тем, что дают _атомарные составные операции_ и _масштабируемость_ — то, чего глобальный лок дать не может в принципе.

### 6.1. Почему «просто обернуть в synchronized» — недостаточный ответ

Наивный путь известен давно. `Hashtable` — ровесник Java 1.0: каждый его публичный метод объявлен `synchronized`, то есть висит на одном intrinsic-локе самого объекта. `Collections.synchronizedMap(map)` делает то же самое поверх любой `Map` — оборачивает каждый вызов в `synchronized (mutex)`. Формально обе конструкции потокобезопасны: ни одна отдельная операция не порвёт внутреннюю структуру карты. Но у этого решения две проблемы, и вторая важнее первой.

**Проблема 1 — производительность.** Один глобальный лок сериализует _все_ обращения. Два потока читают разные ключи и физически никак не мешают друг другу — но всё равно выстраиваются в очередь за одним локом. Под нагрузкой (десятки потоков сервера, горячая карта-кэш) этот лок становится узким местом: чем больше ядер, тем больше времени потоки проводят не в работе, а в ожидании. Это ровно contention из главы 4, только теперь на уровне целой структуры данных.

**Проблема 2 — составные операции всё равно не атомарны.** Вот что действительно ломает программы. `synchronizedMap` делает атомарным _каждый отдельный метод_, но не _последовательность_ методов. Классика — check-then-act. Считаем посещения страницы:

```java
import java.util.*;
import java.util.concurrent.*;

public class BrokenCheckThenAct {
    public static void main(String[] args) throws Exception {
        Map<String, Integer> visits = Collections.synchronizedMap(new HashMap<>());

        int threads = 8;
        int iterations = 10_000;
        try (ExecutorService pool = Executors.newFixedThreadPool(threads)) {
            for (int t = 0; t < threads; t++) {
                pool.submit(() -> {
                    for (int i = 0; i < iterations; i++) {
                        Integer current = visits.get("page"); // атомарно ✓
                        if (current == null) {
                            visits.put("page", 1);            // атомарно ✓
                        } else {
                            visits.put("page", current + 1);  // атомарно ✓ — но пара get+put уже порвалась
                        }
                    }
                });
            }
        }
        System.out.println("Ожидали: " + (threads * iterations));
        System.out.println("Получили: " + visits.get("page"));
    }
}
```

Пять прогонов на JDK 21 — реальные результаты:

```
Ожидали: 80000 → Получили: 33336
Ожидали: 80000 → Получили: 31953
Ожидали: 80000 → Получили: 80000   ← повезло
Ожидали: 80000 → Получили: 42419
Ожидали: 80000 → Получили: 44839
```

Каждый вызов `get` и `put` честно прошёл под локом — и больше половины инкрементов потеряно. Между `get` и `put` в окно влезает другой поток: оба прочитали `41`, оба записали `42`, один инкремент испарился. Это прямое эхо главы 4: **два атомарных действия вместе — не атомарное действие**. Там мы видели это на `count++` и `AtomicInteger`, здесь — то же самое на уровне карты. Оборачивание каждого вызова в лок не помогает, потому что гонка живёт _между_ вызовами, а не внутри них.

Отсюда вывод, который стоит уметь формулировать вслух: `Hashtable` и `synchronizedMap` защищают _внутренности коллекции_, но не _инварианты твоей логики_. Чтобы защитить логику, тебе всё равно пришлось бы брать внешний лок вокруг всей пары `get`+`put` — и тогда зачем вообще synchronized-обёртка?

### 6.2. ConcurrentHashMap — правильный ответ

`ConcurrentHashMap` (CHM) решает обе проблемы сразу, и решает их по-разному.

**Ментальная модель:** вместо одного лока на всю карту — блокировка на уровне отдельного бина (bucket'а). Потоки, работающие с _разными_ ключами, попадают в разные бины и не мешают друг другу вообще; чтения в большинстве случаев идут вовсе без блокировки, опираясь на `volatile`-семантику узлов. Лок берётся коротко и точечно — только на тот бин, который реально меняется.

#### Атомарные составные операции — то, ради чего он нужен

Масштабируемость — приятный бонус, но главная практическая ценность CHM в другом: он даёт составные операции, атомарные _из коробки_. Тот самый check-then-act, который рвался у `synchronizedMap`, здесь решается одним вызовом:

```java
ConcurrentHashMap<String, Integer> visits = new ConcurrentHashMap<>();

// вместо порванной пары get + put — один атомарный вызов:
visits.merge("page", 1, Integer::sum);
```

Тот же тест с 8 потоками × 10 000 итераций через `merge` — на JDK 21 стабильно `80000` в каждом прогоне. Гонки нет, потому что «прочитать-решить-записать» происходит внутри одного вызова, под локом соответствующего бина.

Четыре метода, которые нужно знать:

```java
// положить, только если ключа ещё нет; вернуть то, что лежит (старое или новое)
map.putIfAbsent(key, value);

// если ключа нет — вычислить значение и положить; вернуть текущее.
// Классика: "map списков" без гонки
map.computeIfAbsent("news", k -> new CopyOnWriteArrayList<>()).add("alice");

// пересчитать значение по ключу (функция получает старое, может вернуть null = удалить)
map.compute(key, (k, old) -> old == null ? 1 : old + 1);

// объединить: если ключа нет — положить value; если есть — применить функцию к старому и value
map.merge(key, 1, Integer::sum);
```

`computeIfAbsent` — главный из четырёх. Сравни с тем, как это выглядело бы на `synchronizedMap`:

```java
// ГОНКА: между containsKey и put влезает другой поток,
// и один из двух созданных списков теряется вместе со своим содержимым
if (!syncMap.containsKey("news")) {
    syncMap.put("news", new ArrayList<>());
}
syncMap.get("news").add("alice");
```

В CHM контракт `computeIfAbsent` гарантирует: для данного ключа функция-фабрика вызывается атомарно, и все конкурирующие потоки получат _один и тот же_ созданный объект. Это главный практический вывод главы: если ты пишешь `containsKey` + `put` или `get` + проверку на `null` + `put` на конкурентной карте — ты пишешь гонку, независимо от того, насколько «потокобезопасна» сама карта.

#### null запрещён — и ключ, и значение

```java
map.put(null, 1);   // NullPointerException
map.put("k", null); // NullPointerException
```

Проверено на JDK 21 — оба вызова кидают NPE немедленно. Это не каприз, а следствие конкурентности. В обычном `HashMap` двусмысленность `map.get(k) == null` («ключа нет» или «лежит `null`?») разрешается вызовом `containsKey`. В конкурентной карте так нельзя: между `get` и `containsKey` другой поток может удалить или добавить ключ, и два вызова ответят про _разные состояния карты_. Раз двусмысленность нельзя разрешить без гонки — её запретили на входе. Если тебе нужно «значение отсутствует» как состояние, используй явный объект-маркер или `Optional` внутри значения.

#### Слабо-консистентный итератор

Из темы коллекций ты помнишь fail-fast итератор `HashMap`: изменил карту во время обхода — получи `ConcurrentModificationException`. Итератор CHM устроен иначе — он _weakly consistent_:

```java
// HashMap: модификация во время обхода
for (String k : plainHashMap.keySet()) {
    if (k.equals("b")) plainHashMap.put("d", 4);   // → ConcurrentModificationException
}

// ConcurrentHashMap: тот же трюк
for (String k : chm.keySet()) {
    if (k.equals("b")) chm.put("d", 4);            // работает, исключения нет
}
```

Проверено: `HashMap` бросает CME, CHM спокойно завершает обход. Но важно понимать, _что именно_ гарантирует weakly consistent итератор: он никогда не бросит CME и никогда не вернёт один элемент дважды, но отражает состояние карты «где-то между» началом и концом обхода. Это **не снимок**: изменения, сделанные параллельно, итератор _может_ увидеть, а может и нет. Для мониторинга, логирования, периодической чистки — идеально. Для логики, требующей согласованного среза «всей карты в момент T», — нет, и такого среза CHM не даёт в принципе.

Туда же относится `size()`: под конкурентной записью он приблизителен — это оценка, а не точное число (почему — в 🔬 ниже). Дизайн-философия CHM честная: точные глобальные ответы о постоянно меняющейся структуре стоили бы глобального лока, а глобальный лок — это ровно то, от чего мы ушли.

### 6.3. CopyOnWriteArrayList — список для read-heavy нагрузки

У списков в `java.util.concurrent` другой чемпион, и он затачивался под очень конкретный профиль нагрузки: **много чтений, редкие записи**. Классические примеры — список слушателей событий (подписался один раз, уведомляется тысячи раз) и редко меняющаяся конфигурация.

Механика прямо в имени — copy on write. Внутри лежит массив, и он _никогда не меняется на месте_:

- **Чтение** идёт по текущему массиву вообще без блокировки — это обычный проход по неизменяемому массиву, синхронизировать нечего.
- **Любая запись** (`add`, `set`, `remove`) под локом создаёт полную копию массива с изменением и атомарно подменяет ссылку на внутренний массив.

Итератор захватывает ссылку на массив в момент своего создания и дальше живёт в этом снимке, что бы ни происходило со списком:

```java
CopyOnWriteArrayList<String> cow = new CopyOnWriteArrayList<>(List.of("x", "y"));

Iterator<String> it = cow.iterator(); // снимок: [x, y]
cow.add("z");                          // запись создала новый массив [x, y, z]

// итератор продолжает идти по старому снимку:
while (it.hasNext()) System.out.print(it.next() + " ");  // печатает: x y
System.out.println(cow);                                  // [x, y, z]

it.remove(); // UnsupportedOperationException — снимок неизменяем
```

Проверено на JDK 21: итератор видит `x y`, список к этому моменту уже `[x, y, z]`, а `it.remove()` бросает `UnsupportedOperationException` — модифицировать снимок бессмысленно, его уже никто не «применит» обратно.

![[Pasted image 20260704150625.png]]

Отсюда честный баланс, который нужно называть на собеседовании обеими сторонами. Читатели быстрые, никогда не блокируются и никогда не ловят `ConcurrentModificationException` — обход по определению идёт по неизменяемому снимку. Но каждая запись стоит полной копии массива: `O(n)` по времени и памяти, плюс давление на GC от короткоживущих массивов. Список слушателей на 50 элементов с записью раз в час — идеальный кейс. Список на 100 000 элементов с сотней записей в секунду — катастрофа: ты копируешь мегабайты на каждый `add`. Формула простая: **read-heavy — да, write-heavy — категорически нет**.

### 6.4. BlockingQueue — producer-consumer, упакованный в коллекцию

Здесь всё, что мы делали в предыдущих главах, сходится в одну точку. В главе 5 мы вручную собирали producer-consumer на `ReentrantLock` с двумя `Condition`: ждали «не полно» перед записью, «не пусто» перед чтением, аккуратно вызывали `await` в цикле и `signal` после изменения. Это было полезное упражнение — и вот его пуанта: **`BlockingQueue` — это ровно тот паттерн, уже упакованный в готовую коллекцию**. Вся логика ожидания спрятана за двумя методами:

- `put(e)` — кладёт элемент; если очередь **полна**, блокирует поток, пока не освободится место;
- `take()` — забирает элемент; если очередь **пуста**, блокирует поток, пока элемент не появится.

Обе — с честной обработкой `InterruptedException`, «пробуждением» через те же `Condition`, что мы писали руками, и без единой строчки синхронизации с твоей стороны. Есть и неблокирующие варианты (`offer`/`poll`, в том числе с таймаутом) — для случаев, когда «подождать» не вариант.

#### ArrayBlockingQueue vs LinkedBlockingQueue

Две основные реализации, и разница между ними — не академическая:

**`ArrayBlockingQueue`** — ограниченная всегда: ёмкость задаётся в конструкторе и фиксируется, внутри кольцевой массив. Простая, предсказуемая, без аллокаций на каждый элемент.

**`LinkedBlockingQueue`** — на связанных узлах, и вот грабля: конструктор без аргументов создаёт очередь с ёмкостью `Integer.MAX_VALUE` — **практически неограниченную**. Под перегрузкой, когда производители стабильно быстрее потребителей, такая очередь молча растёт до `OutOfMemoryError`. Это ровно та же ловушка, что и неограниченная очередь пула из главы 2: `Executors.newFixedThreadPool` внутри использует как раз `LinkedBlockingQueue` без ограничения — теперь ты видишь оба конца этой истории. Правило: если очередь стоит между компонентами разной скорости — **всегда задавай ёмкость явно**.

#### Ограниченная очередь = backpressure

Ограниченная ёмкость — не неудобство, а механизм. Когда потребитель не успевает, очередь заполняется, и `put()` естественно притормаживает производителя — система сама выравнивает скорости, вместо того чтобы копить необработанное в памяти. Это и называется backpressure, и это тот самый механизм, который стоит за правильно сконфигурированными пулами из главы 2.

![[Pasted image 20260704150638.png]]
#### Producer-consumer: было и стало

Рабочий пример на JDK 21 — производитель быстрый, потребитель медленный, ёмкость 3:

```java
import java.util.concurrent.*;

public class PipelineDemo {
    public static void main(String[] args) throws Exception {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(3); // ограниченная!

        Thread producer = Thread.ofPlatform().start(() -> {
            try {
                for (int i = 1; i <= 6; i++) {
                    queue.put("task-" + i); // блокируется, если очередь полна
                    System.out.println("произвёл task-" + i + " (в очереди: " + queue.size() + ")");
                }
                queue.put("POISON"); // сигнал завершения
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        Thread consumer = Thread.ofPlatform().start(() -> {
            try {
                while (true) {
                    String task = queue.take(); // блокируется, если пусто
                    if (task.equals("POISON")) break;
                    Thread.sleep(100);          // медленный потребитель
                    System.out.println("обработал " + task);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.join();
        consumer.join();
    }
}
```

Реальный вывод — backpressure виден невооружённым глазом:

```
произвёл task-1 (в очереди: 1)
произвёл task-2 (в очереди: 1)
произвёл task-3 (в очереди: 2)
произвёл task-4 (в очереди: 3)   ← очередь полна
обработал task-1                  ← производитель ЖДАЛ, пока потребитель не освободил место
произвёл task-5 (в очереди: 3)
обработал task-2
произвёл task-6 (в очереди: 3)
обработал task-3
...
```

Производитель выстрелил четыре задачи мгновенно, упёрся в полную очередь — и дальше идёт строго в темпе потребителя. Ни одного лока, ни одного `Condition`, ни одного `while (queue.isFull()) notFull.await()` в нашем коде. Сравни это с версией из главы 5: там на ту же семантику ушло ~40 строк аккуратной работы с `ReentrantLock`, двумя условиями и циклами ожидания — и каждая из этих строк была местом для потенциальной ошибки. Урок глав 5→6 в одном предложении: понимать, как это собирается руками, обязательно; собирать руками в проде — почти никогда.

> [!tip] 💡 Из C#  
> Мостик прямой, но с ложными друзьями.
> 
> `ConcurrentHashMap` ≈ `ConcurrentDictionary<K,V>`; `computeIfAbsent`/`merge` ≈ `GetOrAdd`/`AddOrUpdate`. **Ложный друг №1:** `GetOrAdd(key, valueFactory)` в .NET может вызвать фабрику несколько раз при гонке (выигрывает одно значение, остальные выбрасываются), тогда как `computeIfAbsent` в CHM атомарен для ключа — фабрика вызовется не более одного раза, и все потоки увидят один объект. Не переноси интуицию один в один: если фабрика с побочным эффектом (открывает соединение, регистрирует ресурс), в .NET нужен `Lazy<T>`-трюк, в Java — не нужен.
> 
> **Ложный друг №2:** `ConcurrentDictionary` спокойно живёт с null-значениями (для reference-типов), CHM кидает NPE и на null-ключ, и на null-значение. Дотнетчик, привыкший к nullable, споткнётся здесь первым делом.
> 
> `BlockingQueue` ≈ `BlockingCollection<T>`: `put`/`take` ≈ `Add`/`Take`, ограниченная ёмкость ≈ bounded `BlockingCollection` — семантика backpressure совпадает почти дословно.
> 
> `CopyOnWriteArrayList` прямого аналога в BCL не имеет; идейно это тот же подход, что вручную собирают на `ImmutableList<T>` + атомарная подмена ссылки (`Interlocked.CompareExchange`) под read-heavy нагрузку — только в Java он уже упакован в коллекцию.

> [!example] 🔬 Глубже / на синьора  
> **Эволюция ConcurrentHashMap.** В Java 7 карта делилась на сегменты (`Segment` — по сути мини-`Hashtable` со своим локом, по умолчанию 16 штук): конкурентность ограничивалась числом сегментов. В Java 8 внутренности переписали полностью: сегменты убрали, локом стал первый узел конкретного бина (`synchronized` на нём), вставка в пустой бин идёт вообще без лока — через CAS на слот таблицы (те же механизмы, что в атомиках из главы 4). Плюс из `HashMap` пришла тримификация: длинная цепочка коллизий (от 8 узлов при таблице ≥ 64) превращается в красно-чёрное дерево, срезая worst-case поиска с O(n) до O(log n). Итог: гранулярность лока — один бин из тысяч, а не один сегмент из 16.
> 
> **Почему size() приблизителен.** Точный счётчик означал бы одну горячую переменную, за которую дерутся все пишущие потоки, — глобальная точка contention, от которой весь дизайн уходил. Вместо этого счётчик размазан: `baseCount` плюс массив ячеек `CounterCell`, по которым потоки распределяются при конкуренции — та же идея, что в `LongAdder` из главы 4. `size()` (и правильный для больших карт `mappingCount()`, возвращающий `long`) суммирует ячейки без остановки мира — получая оценку, а не точный срез.
> 
> **Стоимость долгой лямбды в computeIfAbsent.** Атомарность не бесплатна: пока фабрика вычисляется, лок на бине удерживается. Долгая лямбда (поход в базу, HTTP-вызов) сериализует все обращения к ключам этого бина, а лямбда, которая сама лезет в ту же карту, может заблокироваться навсегда — Javadoc прямо требует: короткая, простая, не трогает другие ключи этой карты. Если значение дорого строить — клади в карту дешёвую обёртку (мемоизирующий `Supplier`/`CompletableFuture` из главы 3), а дорогое вычисление выноси из-под лока.
> 
> **Внутренности блокирующих очередей.** `ArrayBlockingQueue` — это буквально наш код из главы 5, доведённый до ума: один `ReentrantLock` и два `Condition` (`notEmpty`, `notFull`); `put` и `take` паркуют потоки через `await` и будят через `signal`. `LinkedBlockingQueue` хитрее: у неё два раздельных лока — `putLock` на хвост и `takeLock` на голову, поэтому производитель и потребитель не конкурируют между собой вовсе (счётчик элементов — `AtomicInteger`). Отсюда её более высокая пропускная способность в сценарии «много и тех и других» — ценой аллокации узла на каждый элемент и давления на GC.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Вера, что `synchronizedMap`/`Hashtable` делает потокобезопасной твою логику.** Атомарны отдельные методы; любая последовательность (`containsKey`+`put`, `get`+проверка+`put`, итерация+модификация) рвётся. Итерация по `synchronizedMap` вообще требует ручного `synchronized (map) { ... }` вокруг цикла — иначе CME даже с «потокобезопасной» обёрткой.
> 2. **null в `ConcurrentHashMap` → NPE.** И ключ, и значение. Всплывает при миграции с `HashMap`, где null жил годами, и у бывших дотнетчиков.
> 3. **`CopyOnWriteArrayList` под частой записью.** Каждый `add` — полная копия массива. На больших списках с активной записью это O(n) на операцию плюс шторм мусора для GC.
> 4. **`LinkedBlockingQueue` без явной ёмкости.** По умолчанию `Integer.MAX_VALUE`: под перегрузкой очередь молча ест память до OOM. Та же грабля зашита в `Executors.newFixedThreadPool` (глава 2).
> 5. **Долгая лямбда в `computeIfAbsent`.** Держит лок бина, сериализует соседей по бину; лямбда, модифицирующая ту же карту, — путь к зависанию.
> 6. **Ожидание снимка от итератора CHM.** Weakly consistent — это «не бросит CME и не вернёт дважды», а не «согласованный срез на момент T». Логика, требующая точного среза, на CHM не строится; `size()` под записью — оценка.

### На собесе

> [!question]- Q1. Чем ConcurrentHashMap отличается от Hashtable и Collections.synchronizedMap?  
> **RU:** У `Hashtable` и `synchronizedMap` один глобальный лок — все операции сериализуются, даже независимые чтения. CHM блокирует на уровне отдельного бина, чтения в основном без лока, поэтому масштабируется по ядрам. Плюс CHM даёт атомарные составные операции (`computeIfAbsent`, `merge`), а у обёрток атомарны только отдельные методы. И итератор: fail-fast у обёрток против weakly consistent у CHM.
> 
> **EN:** Hashtable/synchronizedMap serialize everything on a single global lock, while ConcurrentHashMap locks per-bin with mostly lock-free reads, so it scales. It also provides atomic compound operations like computeIfAbsent, and its iterators are weakly consistent instead of fail-fast.
> 
> _Follow-up-ловушка:_ «Значит, `synchronizedMap` потокобезопасна и её достаточно?» — слабый ответ «да». Сильный: отдельные методы атомарны, но составные операции рвутся, так что для любой логики вида check-then-act она бесполезна без внешнего лока.

> [!question]- Q2. Почему computeIfAbsent лучше связки containsKey + put?  
> **RU:** Пара `containsKey`+`put` — check-then-act: между вызовами влезает другой поток, и оба создают значение, одно теряется. `computeIfAbsent` выполняет проверку и вставку атомарно под локом бина: фабрика вызовется максимум один раз, все потоки получат один и тот же объект.
> 
> **EN:** containsKey+put is a check-then-act race — another thread can slip in between. computeIfAbsent performs the check and insert atomically, guaranteeing the factory runs at most once per key.
> 
> _Follow-up:_ «Какие ограничения у лямбды в `computeIfAbsent`?» — должна быть короткой и не трогать эту же карту: она выполняется под локом бина.

> [!question]- Q3. Почему ConcurrentHashMap запрещает null?  
> **RU:** Из-за неустранимой двусмысленности: `get(k) == null` — «ключа нет» или «лежит null»? В `HashMap` это решает `containsKey`, но в конкурентной карте между двумя вызовами состояние успевает измениться — разрешить двусмысленность без гонки нельзя. Поэтому null запрещён на входе, и ключ, и значение.
> 
> **EN:** Because get() returning null would be ambiguous — "absent" vs "mapped to null" — and in a concurrent map you can't resolve that with a separate containsKey call without a race. So nulls are rejected outright.

> [!question]- Q4. Когда оправдан CopyOnWriteArrayList?  
> **RU:** Профиль «много чтений, редкие записи»: слушатели событий, редко меняющаяся конфигурация. Чтения и итерация — без блокировок и без CME, по неизменяемому снимку. Цена — каждая запись копирует весь массив, поэтому при частой записи или больших размерах он противопоказан.
> 
> **EN:** For read-heavy, rarely-written data like listener lists: reads and iteration are lock-free over an immutable snapshot, but every write copies the whole array, so it's a poor fit for write-heavy workloads.
> 
> _Follow-up:_ «Что увидит итератор, если список изменили во время обхода?» — старый снимок на момент создания итератора; `iterator.remove()` бросит `UnsupportedOperationException`.

> [!question]- Q5. Как устроен producer-consumer на BlockingQueue и зачем ограниченная очередь?  
> **RU:** Производители кладут через `put`, потребители забирают через `take`; `put` блокируется на полной очереди, `take` — на пустой, вся координация внутри (те же лок + условия, что пишутся вручную). Ограниченная ёмкость даёт backpressure: медленный потребитель естественно тормозит производителя, вместо накопления в памяти. Неограниченная (`LinkedBlockingQueue` по умолчанию) под перегрузкой растёт до OOM.
> 
> **EN:** Producers put, consumers take; both block when the queue is full/empty, so all coordination is built in. A bounded queue provides backpressure — a slow consumer naturally throttles producers — whereas an unbounded queue just grows until OOM under sustained overload.
> 
> _Follow-up:_ «`ArrayBlockingQueue` vs `LinkedBlockingQueue`?» — массив с фиксированной ёмкостью и одним локом против связанных узлов с двумя локами (голова/хвост) и ёмкостью по умолчанию `Integer.MAX_VALUE`; вторая быстрее под смешанной нагрузкой, но требует явной ёмкости.

> [!question]- Q6. Fail-fast vs weakly consistent итератор — в чём разница?  
> **RU:** Fail-fast (`HashMap`, `ArrayList`) при обнаружении конкурентной модификации бросает `ConcurrentModificationException` — best-effort детектор багов. Weakly consistent (CHM) не бросает CME, не возвращает элемент дважды и отражает состояние где-то между началом и концом обхода — параллельные изменения может увидеть частично. Снимка не даёт ни тот, ни другой; снимок — это про `CopyOnWriteArrayList`.
> 
> **EN:** Fail-fast iterators throw ConcurrentModificationException on detected concurrent modification; weakly consistent iterators never throw CME, never return an element twice, and reflect some state between iteration start and end — possibly seeing concurrent updates partially. Only copy-on-write collections give a true snapshot.
> 
> _Слабый ответ, который заваливает:_ «итератор CHM возвращает снимок карты» — нет, снимок только у COW-коллекций.
