### 4.1. Recap: монитор, synchronized, volatile

Это быстрый разбор фундамента, на который ляжет всё остальное, — не полный курс по JMM. Если по ходу чтения что-то из главы 1 забылось, вернитесь к разделу про happens-before.

#### Монитор и synchronized

У каждого объекта в Java есть встроенный лок — монитор (intrinsic lock). Ключевое слово `synchronized` захватывает его и даёт две гарантии разом:

- **Взаимное исключение** — в критической секции, защищённой одним и тем же монитором, находится максимум один поток. Остальные блокируются и ждут.
- **Видимость** — выход из synchronized-блока (release) находится в отношении happens-before с последующим входом (acquire) в блок на том же мониторе (JLS §17.4.5). Всё, что поток записал внутри секции, гарантированно увидит следующий поток, вошедший в неё.

Синтаксически есть блок и метод — это одно и то же, различается только выбор монитора:

```java
class Wallet {
    private final Object lock = new Object();
    private long balance;

    // synchronized-метод: монитор — this
    public synchronized void deposit(long amount) {
        balance += amount;
    }

    // synchronized-блок: монитор — явно выбранный объект
    public void withdraw(long amount) {
        synchronized (lock) {
            balance -= amount;
        }
    }

    // static synchronized: монитор — объект Wallet.class
    public static synchronized void auditAll() { /* ... */ }
}
```

Два классических граблевых момента, коротко. `synchronized (this)` и synchronized-метод — это лок на объекте, который виден всем, у кого есть ссылка на него: любой внешний код может сделать `synchronized (wallet) { while(true); }` и заблокировать ваши методы навсегда. То же с локом на любом публично доступном объекте (включая интернированные `String` и закешированные `Integer` — там лок вообще делится между незнакомцами). Поэтому стандартная рекомендация: `private final Object lock = new Object()` — монитор, который контролируете только вы. Подробно об уровнях локов — в главе 5.

#### volatile

`volatile` — это гарантия видимости и порядка, и только их. Запись в volatile-поле happens-before последующего чтения этого поля (JLS §17.4.5): читатель гарантированно видит свежее значение, а компилятор и процессор не переставляют обращения к памяти вокруг него.

Чего `volatile` не даёт — атомарности составных операций. Проверим на счётчике из главы 1, честно пометив поле `volatile`:

```java
import java.util.concurrent.*;

public class BrokenCounter {
    static volatile int counter = 0;

    public static void main(String[] args) {
        try (ExecutorService pool = Executors.newFixedThreadPool(4)) {
            for (int t = 0; t < 4; t++) {
                pool.submit(() -> {
                    for (int i = 0; i < 1_000_000; i++) {
                        counter++;   // read-modify-write: не атомарно
                    }
                });
            }
        } // close() ждёт завершения задач (Java 19+)
        System.out.println("Ожидали 4000000, получили: " + counter);
    }
}
```

Три запуска на JDK 21 у меня дали: 2466329, 4000000, 2038387. Иногда везёт, чаще — нет. Причина: `counter++` — это три действия (прочитать → прибавить → записать), и `volatile` делает каждое из них честным по видимости, но не склеивает их в одно неделимое. Между чтением и записью потока A успевает вклиниться поток B:

|Три беды из главы 1|`synchronized`|`volatile`|
|---|---|---|
|Атомарность|✅ закрывает|❌ не даёт|

|Переупорядочивание|✅ закрывает|✅ закрывает|

`synchronized` лечит все три беды, но платит блокировкой. `volatile` лечит две из трёх — и годится только там, где операция и так неделима: одиночная запись флага, публикация ссылки на готовый объект.

### 4.2. Atomic-классы

#### Мотивация: чинить счётчик локом — стрелять из пушки по воробьям

Счётчик выше можно починить `synchronized` — и это будет корректно. Но посмотрите на цену: ради одной машинной операции сложения мы захватываем монитор, а все конкурирующие потоки блокируются — операционная система приостанавливает их, а потом будит. Для длинной критической секции это оправдано; для `i++` — несоразмерно тяжело.

В `java.util.concurrent.atomic` есть более лёгкий, безблокировочный (lock-free) путь для таких простых случаев:

```java
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    static final AtomicInteger counter = new AtomicInteger(0);

    public static void main(String[] args) {
        try (ExecutorService pool = Executors.newFixedThreadPool(4)) {
            for (int t = 0; t < 4; t++) {
                pool.submit(() -> {
                    for (int i = 0; i < 1_000_000; i++) {
                        counter.incrementAndGet();   // атомарно, без единого лока
                    }
                });
            }
        }
        System.out.println("Ожидали 4000000, получили: " + counter.get());
    }
}
```

Три запуска — три раза ровно 4000000. Ни один поток ни разу не был заблокирован. Как такое возможно, если мы только что видели, что «прочитать-прибавить-записать» разрывается? Ответ — в одной аппаратной операции.

#### CAS — сердце атомиков

CAS (compare-and-swap / compare-and-set) — операция с тремя аргументами: адрес переменной, ожидаемое значение, новое значение. Семантика: «запиши новое значение, но только если там всё ещё лежит ожидаемое; иначе не трогай и скажи, что не вышло». Критически важно: вся эта проверка-и-запись — одна аппаратная инструкция процессора (на x86 — `lock cmpxchg`), между «проверить» и «записать» вклиниться физически невозможно.

На CAS строится стандартный паттерн — CAS-петля (retry loop). Вот как концептуально устроен `incrementAndGet()`: прочитать текущее значение (`current = value`); вычислить новое (`next = current + 1`); `compareAndSet(current, next)` — если в переменной всё ещё `current`, атомарно записать `next` (успех), иначе вернуться к шагу 1 и перечитать.

![[Pasted image 20260704145552.png]]

Обратите внимание: при неудаче поток не блокируется. Он не отдаёт процессор, не встаёт в очередь — просто делает ещё один виток петли. Потерянных обновлений не бывает: проиграть гонку можно, но проигравший это узнает (CAS вернёт `false`) и пересчитает результат от свежего значения.

Ментальная модель — оптимистичный vs пессимистичный подход. Лок пессимистичен: «конфликт возможен, поэтому заранее никого не пущу». CAS оптимистичен: «предположу, что конфликта нет; проверю в момент записи; не угадал — перечитаю и попробую снова». Этот контраст — ключ к пониманию, когда что выбирать: пессимизм окупается, когда конфликты часты и работа под защитой длинная; оптимизм — когда операция короткая, а столкновения редки.

#### AtomicInteger и AtomicLong на практике

Основные операции (все — атомарные, все — на CAS внутри): `get()` / `set(v)` — чтение и запись с семантикой volatile; `incrementAndGet()` / `getAndIncrement()` — атомарный `++i` / `i++` (разница только в том, какое значение возвращается — новое или старое), симметрично `decrementAndGet()`, `addAndGet(delta)`, `getAndAdd(delta)`; `compareAndSet(expected, next)` — тот самый CAS в чистом виде, возвращает `boolean`; `updateAndGet(fn)` / `accumulateAndGet(x, fn)` — «примени функцию атомарно», библиотека сама крутит CAS-петлю вокруг вашей лямбды.

Готовых методов хватает для счётчиков, но CAS-петлю стоит уметь писать руками — на ней строится любая нестандартная атомарная операция. Например, «прибавить, но не выше потолка»:

```java
import java.util.concurrent.atomic.AtomicInteger;

public class CasLoop {
    static final AtomicInteger value = new AtomicInteger(10);

    static int addCapped(int delta, int cap) {
        while (true) {
            int current = value.get();                  // 1. читаем
            int next = Math.min(current + delta, cap);  // 2. вычисляем
            if (value.compareAndSet(current, next)) {   // 3. пишем, если не изменилось
                return next;                            // успех — выходим
            }
            // неудача: кто-то изменил value — виток заново
        }
    }

    public static void main(String[] args) {
        System.out.println(addCapped(5, 100));   // 15
        System.out.println(addCapped(90, 100));  // 100 — обрезано по потолку
    }
}
```

Тот же результат даст `value.updateAndGet(v -> Math.min(v + delta, cap))` — петля внутри. Важное следствие для обоих вариантов: функция может выполниться несколько раз (по разу на каждый виток), поэтому она обязана быть чистой — никаких побочных эффектов вроде записи в лог или инкремента другого счётчика внутри лямбды.

#### AtomicReference: атомарная подмена состояния целиком

CAS работает не только с числами. `AtomicReference<T>` атомарно подменяет ссылку — и это открывает важный паттерн: составное состояние упаковывается в неизменяемый объект, а обновление — это создание нового объекта и атомарный своп ссылки:

```java
import java.util.concurrent.atomic.AtomicReference;

public class RefSwap {
    record Range(int min, int max) {}   // неизменяемое составное состояние

    static final AtomicReference<Range> range =
            new AtomicReference<>(new Range(0, 10));

    static void extend(int newMax) {
        // читается старый Range, строится новый, ссылка свопается CAS-ом
        range.updateAndGet(r -> new Range(r.min(), Math.max(r.max(), newMax)));
    }

    public static void main(String[] args) {
        extend(25);
        extend(15);                       // не уменьшит
        System.out.println(range.get());  // Range[min=0, max=25]
    }
}
```

Здесь `min` и `max` обновляются согласованно без единого лока — потому что снаружи никогда не видно «полусобранного» состояния: ссылка указывает либо на старый целиком консистентный `Range`, либо на новый. Обязательное условие — объект внутри `AtomicReference` неизменяемый: если его можно мутировать по ссылке, вся защита рассыпается.

#### Главное ограничение: атомик защищает ровно одну переменную

Это критический момент уровня mid+, и его нельзя пропустить. Каждая операция атомика неделима — но два независимых атомарных действия, взятых вместе, не атомарны. Классическая ошибка новичка: «навесил `Atomic*` — стало потокобезопасно везде». Проверим:

```java
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class TwoAtomics {
    // Инвариант: total == succeeded + failed
    static final AtomicInteger succeeded = new AtomicInteger();
    static final AtomicInteger failed = new AtomicInteger();
    static final AtomicInteger total = new AtomicInteger();

    public static void main(String[] args) {
        ExecutorService pool = Executors.newFixedThreadPool(2);
        for (int t = 0; t < 2; t++) {
            pool.submit(() -> {
                while (!Thread.currentThread().isInterrupted()) {
                    total.incrementAndGet();       // атомарно
                    succeeded.incrementAndGet();   // атомарно. Но пара — нет
                }
            });
        }
        int violations = 0;
        for (int i = 0; i < 1_000_000; i++) {
            if (total.get() != succeeded.get() + failed.get()) violations++;
        }
        pool.shutdownNow();
        System.out.println("Нарушений инварианта: " + violations);
    }
}
```

Запуск на JDK 21: `Нарушений инварианта: 980394` — из миллиона чтений почти каждое застало систему между двумя инкрементами, в состоянии, где инвариант нарушен. Каждый шаг честно атомарен; составной инвариант — нет.

Правило: если инвариант охватывает больше одной переменной, атомики его не спасут — нужна одна общая критическая секция вокруг всех связанных чтений и записей: лок (`synchronized` или `ReentrantLock` — глава 5). Либо, если состояние получается упаковать в один неизменяемый объект, — трюк с `AtomicReference` из предыдущего раздела: тогда «несколько переменных» снова становятся одной ссылкой.

#### Когда CAS проигрывает локу

Оптимизм хорош, пока конфликты редки. При высокой конкуренции — десятки потоков долбят один атомик — картина портится: в каждый момент побеждает один, остальные жгут CPU на холостых витках петли (busy-spin), причём чем больше проигравших, тем больше повторных попыток, тем выше конкуренция за ту же ячейку памяти. В этом режиме честный лок, который усыпляет ожидающих, может оказаться и дешевле, и предсказуемее.

Для частного, но очень частого случая — горячего счётчика (метрики, статистика) — в JDK есть `LongAdder`: он разносит инкременты по нескольким независимым ячейкам, чтобы потоки не толкались за одну, а сумму собирает при чтении. Механика — в 🔬-блоке ниже.

> [!tip] 💡 Из C#  
> Семейство `Atomic*` — прямой аналог `Interlocked`: `AtomicInteger.incrementAndGet()` ≈ `Interlocked.Increment(ref i)`, `compareAndSet(expected, next)` ≈ `Interlocked.CompareExchange(ref v, next, expected)`. Разница в форме: `Interlocked` — статические методы над обычным полем, атомики — обёртка-объект над значением; и в возврате: `compareAndSet` возвращает `boolean`, а `CompareExchange` — старое значение (точный аналог в Java — `compareAndExchange`, Java 9+).
> 
> ⚠️ **Ложный друг — `volatile`.** В Java его семантика точно определена через happens-before в JMM и сильнее, чем наивное понимание `volatile` в C# (где гарантии — acquire/release-заборы и на практике их часто трактуют неверно). Не переносите интуицию один в один. И главное, что совпадает в обоих языках: `volatile` не даёт атомарности — `volatile int i; i++` сломан и там, и там.

> [!example] 🔬 Глубже / на синьора  
> **Что реально исполняет процессор.** `compareAndSet` компилируется JIT-ом в одну инструкцию: на x86 — `lock cmpxchg`; на ARM — либо пара LL/SC (`ldxr`/`stxr` — «прочитай с меткой / запиши, если метка цела»), либо инструкция `cas` из расширения LSE. А `incrementAndGet()` в HotSpot даже лучше, чем CAS-петля: он делегирует во внутренний `Unsafe.getAndAddInt`, чей Java-фолбэк — действительно `do { v = read(); } while (!weakCAS(v, v + 1))`, но на x86 JIT интринзифицирует его в безусловный `lock xadd` — атомарное сложение вовсе без повторов.
> 
> **Lock-free vs wait-free.** CAS-петля — lock-free: система в целом всегда прогрессирует (чей-то CAS каждый раунд удаётся), но конкретный поток может проигрывать неограниченно долго. Wait-free — более сильная гарантия: каждый поток завершает операцию за конечное число шагов (`lock xadd` для одиночного инкремента по сути её даёт). Полностью wait-free структуры данных сложны и в JDK редки.
> 
> **Проблема ABA.** CAS проверяет значение, а не историю: если между вашим чтением A и записью значение сходило A → B → A, CAS пройдёт, хотя мир успел измениться. Для чисел-счётчиков это безвредно; для алгоритмов над ссылками (классика — lock-free стек, где узел освободили и переиспользовали) — источник тонких багов. Лекарства: `AtomicStampedReference` (ссылка + версия-int, CAS сверяет обе) и `AtomicMarkableReference` (ссылка + boolean-метка). В GC-языке ABA встречается реже, чем в C/C++, но переиспользование логическое — например, пул объектов — воспроизводит её полностью.
> 
> **LongAdder / LongAccumulator.** Под контенцией `LongAdder` расщепляет значение на массив ячеек (`Cell[]`, striping): поток, чей CAS в базовое поле не прошёл, уходит инкрементить «свою» ячейку (выбор — по хэшу потока), и конкуренция за одну строку кэша исчезает. `sum()` просто складывает базу и все ячейки без фиксации снапшота — при параллельных записях результат не является атомарным срезом на момент вызова (это прямо оговорено в Javadoc). Отсюда правило: `LongAdder` — для статистики с частой записью и редким чтением; если нужно точное значение с CAS-семантикой — оставайтесь на `AtomicLong`.
> 
> **Мост в главу 5.** Явные локи (`ReentrantLock` и вся `java.util.concurrent.locks`) внутри построены на `AbstractQueuedSynchronizer` (AQS), а тот захватывает состояние… тем же CAS-ом, и лишь при неудаче ставит поток в очередь и паркует. То есть CAS — фундамент не только «безлоковых» инструментов, но и самих локов. Разберём в следующей главе.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Атомик защищает одну переменную, не составной инвариант.** `total` и `succeeded` по отдельности атомарны — вместе нет (пример выше: 980 394 нарушения на миллион чтений). Инвариант на несколько переменных → лок или упаковка состояния в один объект под `AtomicReference`.
> 2. **`volatile` ≠ атомарность.** `volatile int i; i++` теряет инкременты — проверено запуском. `volatile` — про видимость и порядок, не про неделимость.
> 3. **`compareAndSet` без петли — баг.** Одиночный вызов может просто вернуть `false`, и ваше обновление молча потеряется. Либо петля `while (!cas(...))`, либо готовые `updateAndGet`/`accumulateAndGet`.
> 4. **Лямбда в `updateAndGet` обязана быть чистой.** Библиотечная CAS-петля может вызвать её несколько раз; побочный эффект внутри (лог, инкремент чужого счётчика, I/O) выполнится непредсказуемое число раз.
> 5. **Мутабельный объект в `AtomicReference` — самообман.** Атомарен только своп ссылки; если объект можно изменить по старой ссылке, никакой защиты нет. Внутри — только иммутабельное (`record` подходит идеально).
> 6. **Горячий счётчик на `AtomicLong` под высокой контенцией** — потоки жгут CPU в retry. Для метрик — `LongAdder`.

### На собесе

> [!question]- Почему `i++` не атомарен даже с `volatile`?  
> Потому что `i++` — это три действия: прочитать, прибавить, записать. `volatile` гарантирует видимость и порядок обращений к переменной, но не склеивает read-modify-write в неделимую операцию: два потока читают одно и то же старое значение и перезаписывают инкременты друг друга. Атомарность даёт либо `synchronized`, либо `AtomicInteger`.

> [!question]- Как работает CAS? Что такое оптимистичная блокировка?  
> CAS — атомарная инструкция процессора «запиши новое значение, если там всё ещё лежит ожидаемое; иначе сообщи о неудаче». Вокруг неё строится retry-петля: прочитать → вычислить → CAS; при неудаче — перечитать и повторить. Это оптимистичный подход: не блокируем никого заранее, а конфликт обнаруживаем в момент записи и разрешаем повтором — в противовес пессимистичному локу, который заранее выстраивает потоки в очередь.

> [!question]- Когда атомики, а когда synchronized/lock?  
> Атомики — когда защищается одна переменная (или состояние, упакованное в один неизменяемый объект под `AtomicReference`) и операция короткая: счётчики, флаги, своп ссылки. Лок — когда инвариант охватывает несколько переменных, критическая секция длинная (I/O, сложная логика) или конкуренция настолько высокая, что CAS-петли жгут CPU впустую. (Синьорский добавок: для горячих счётчиков-метрик — `LongAdder`, он снимает контенцию striping-ом.)

> [!question]- Что такое ABA-проблема?  
> CAS сравнивает значение, а не историю: если значение успело измениться A → B → A, CAS посчитает, что «ничего не менялось». Для счётчиков безвредно; в lock-free структурах над переиспользуемыми объектами приводит к порче данных. Решение — CAS по паре «значение + версия»: `AtomicStampedReference`.

> [!question]- Почему два атомарных действия вместе не атомарны?  
> Между ними есть зазор: другой поток может увидеть (или изменить) состояние в момент, когда первое действие уже выполнено, а второе — нет. Атомарность каждого шага не даёт атомарности последовательности; составной инвариант требует общей критической секции.

> [!question]- Чем `volatile` отличается от `synchronized`?  
> `volatile` даёт видимость и запрет переупорядочивания вокруг обращений к полю — и только. `synchronized` даёт то же самое плюс взаимное исключение, то есть атомарность всей критической секции. Из трёх бед многопоточности `volatile` лечит две, `synchronized` — все три; за это он платит блокировкой потоков.
