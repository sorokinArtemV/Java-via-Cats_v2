В главе 4 у нас в руках оказались два инструмента с противоположными характерами. `synchronized` — полноценная взаимная блокировка: охраняет сколь угодно сложный составной инвариант, но негибкая — её нельзя «попробовать взять», нельзя прервать ожидание, нельзя задать таймаут. Атомики — лёгкие и неблокирующие, но их предел мы сформулировали явно: одна переменная. Инвариант из двух полей (`balance` и `history`, `count` и `items`) атомиком не защитить.

Между этими полюсами остаётся пробел: нужна настоящая взаимная блокировка составного состояния, но с бóльшим контролем, чем даёт ключевое слово. Именно этот пробел закрывает пакет `java.util.concurrent.locks` — явные локи, появившиеся в Java 5 вместе со всем `java.util.concurrent` (архитектор — Doug Lea).

### 5.1. ReentrantLock: тот же монитор, но в руках

**Ментальная модель.** `ReentrantLock` — это тот же самый монитор-лок, что и у `synchronized`, но вынутый из неявного ключевого слова в явный объект, которым ты управляешь руками. Семантика памяти идентична: `unlock()` happens-before последующего `lock()` того же лока — все гарантии видимости из главы 1 работают точно так же. За явность платишь дисциплиной, взамен получаешь возможности, которых у `synchronized` нет физически.

Минимальное использование выглядит так:

```java
class Wallet {
    private final ReentrantLock lock = new ReentrantLock();
    private long balance;
    private int operations;   // составной инвариант: два поля меняются вместе

    void deposit(long amount) {
        lock.lock();
        try {
            balance += amount;
            operations++;
        } finally {
            lock.unlock();
        }
    }
}
```

#### Грабля номер один — сразу и жёстко

У `synchronized` лок освобождается сам: на любом выходе из блока — нормальном, через `return`, через исключение. Компилятор генерирует это за тебя. У `ReentrantLock` — нет. Если между `lock()` и `unlock()` вылетело исключение, а `unlock()` не в `finally`, лок останется захваченным навсегда. Не «до таймаута», не «пока GC не приберёт» — навсегда: все потоки, которые придут за этим локом, повиснут, и приложение встанет.

Отсюда правило без исключений, шаблон, который должен впечататься в моторную память:

```java
lock.lock();
try {
    // критическая секция
} finally {
    lock.unlock();
}
```

Обрати внимание: `lock()` стоит до `try`, а не внутри него. Если бы `lock()` был внутри `try` и бросил исключение (например, `lockInterruptibly()` прервали до захвата), `finally` вызвал бы `unlock()` на локе, которым мы не владеем, — и получил бы `IllegalMonitorStateException`, маскирующую исходную проблему. Канонический шаблон именно такой: захват — снаружи, освобождение — в `finally`.

#### Реентерабельность

Имя `ReentrantLock` — от reentrant, «повторновходимый»: поток, уже владеющий локом, может взять его ещё раз, не заблокировав сам себя. Внутри лока живёт счётчик захватов (`getHoldCount()`), и лок реально освобождается, когда счётчик доходит до нуля — сколько было `lock()`, столько должно быть `unlock()`. `synchronized` устроен так же (метод `a()` под монитором может спокойно вызвать `b()` под тем же монитором), просто там это не назвали в имени.

#### Что ReentrantLock даёт сверх synchronized

**1. `tryLock()` — попытаться и не повиснуть.** `synchronized` умеет только одно: ждать, сколько потребуется. `tryLock()` пытается взять лок и немедленно возвращает `boolean`; перегрузка `tryLock(timeout, unit)` ждёт не дольше заданного. Это прямой инструмент против дедлоков. Вспомни классику из главы 1: поток A держит лок X и ждёт Y, поток B держит Y и ждёт X — вечное объятие. С `tryLock` объятие разжимается:

```java
boolean transfer(Account from, Account to, long amount)
        throws InterruptedException {
    while (true) {
        if (from.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
            try {
                if (to.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
                    try {
                        from.balance -= amount;
                        to.balance += amount;
                        return true;
                    } finally { to.lock.unlock(); }
                }
            } finally { from.lock.unlock(); }
        }
        // не достались ОБА лока — всё отпущено, пробуем заново
    }
}
```

Ключевой момент: если второй лок не дался за 50 мс, мы отпускаем первый и начинаем заново. Дедлок невозможен в принципе — никто не держит один лок, вечно ожидая другой. (Я гонял этот код двумя потоками по 1000 встречных переводов: суммарный баланс сходится, зависаний нет.) Цена — возможен livelock при неудачном ритме повторов; в проде к retry добавляют случайную задержку. И заметь вложенность `try`/`finally`: каждый захваченный лок освобождается своим `finally`.

**2. `lockInterruptibly()` — ожидание, откликающееся на прерывание.** Поток, повисший на `synchronized`, глух к `interrupt()` — он выйдет из ожидания только получив лок. `lockInterruptibly()` ждёт лок, но если поток прервали — бросает `InterruptedException` и уходит. Для сервисов с graceful shutdown это разница между «остановились за секунду» и «висим, пока не убьют `kill -9`».

**3. Справедливость (fairness).** `new ReentrantLock(true)` создаёт справедливый лок: он достаётся потокам в порядке очереди ожидания, никто не голодает. Звучит как то, что нужно всегда, — но честная цена такова: справедливый лок заметно медленнее. Каждая передача лока — это обязательное пробуждение конкретного потока из очереди (парковка/распарковка, работа планировщика), тогда как несправедливый лок разрешает «наглость» (barging): только что пришедший и уже работающий на CPU поток может схватить освободившийся лок раньше тех, кто спит в очереди, — и это сокращает простои лока. По умолчанию лок несправедливый, и это правильный дефолт; справедливость включают, когда голодание — реальная, наблюдаемая проблема.

**4. Несколько условий ожидания на одном локе.** У монитора очередь ожидания одна и неявная (`wait`/`notify`). `ReentrantLock` умеет создавать сколько угодно независимых очередей — объектов `Condition`. Это главный сюжет второй половины главы, дойдём в 5.4.

> [!tip] 💡 Из C#  
> `ReentrantLock` ≈ `lock` / `Monitor`, но с явными захватом и освобождением; `tryLock()` ≈ `Monitor.TryEnter`. Дальше в главе: `Condition.await/signal/signalAll` ≈ `Monitor.Wait/Pulse/PulseAll`, `ReadWriteLock` ≈ `ReaderWriterLockSlim`.
> 
> ⚠️ **Ложный друг:** в C# `lock` — синтаксический сахар, который сам генерирует `Monitor.Exit` в `finally`. В Java так же ведёт себя `synchronized`, но не `ReentrantLock`: здесь `unlock()` — только руками. Дотнетчик, привыкший, что «блок сам закроется», получает вечно захваченный лок — это самый частый способ выстрелить себе в ногу при переходе.

#### Так когда ReentrantLock, а когда synchronized?

Правило простое: по умолчанию — `synchronized`. Он короче, его невозможно забыть освободить, современный HotSpot оптимизирует его не хуже (а с приходом виртуальных потоков в JDK 21 разница в поведении при блокировке практически устранена в JDK 24, JEP 491 — но это уже за пределами нашего baseline). `ReentrantLock` оправдан, когда нужна хотя бы одна из его суперспособностей: `tryLock`, прерываемое ожидание, справедливость, несколько `Condition`. Нет ни одной — явный лок только добавляет способ ошибиться.

### 5.2. ReadWriteLock: много читателей или один писатель

Мотивация — из частого прод-сценария. Есть структура, которую читают тысячи раз в секунду, а меняют раз в минуту: кеш конфигурации, справочник курсов валют, роутинг-таблица. Обычный лок (`synchronized` или `ReentrantLock`) сериализует всех — включая читателей, хотя читатели друг другу не мешают вообще: они ничего не меняют. Тысячи потоков выстраиваются в очередь ради защиты от писателя, который приходит раз в минуту. Обидно.

**Ментальная модель.** `ReadWriteLock` — это пара согласованных локов над одними данными: read-лок могут держать много потоков одновременно, write-лок — строго один и эксклюзивно (пока он захвачен, не пройдёт ни один читатель). Формула: **много читателей ИЛИ один писатель**.

```java
class SettingsCache {
    private final ReadWriteLock rw = new ReentrantReadWriteLock();
    private final Map<String, String> settings = new HashMap<>();

    String get(String key) {
        rw.readLock().lock();
        try {
            return settings.get(key);
        } finally {
            rw.readLock().unlock();
        }
    }

    void put(String key, String value) {
        rw.writeLock().lock();
        try {
            settings.put(key, value);
        } finally {
            rw.writeLock().unlock();
        }
    }
}
```

Обычный `HashMap` — не потокобезопасный — здесь корректен, потому что вся дисциплина доступа обеспечена снаружи: читатели видят согласованное состояние (writer эксклюзивен, а release write-лока happens-before последующего захвата read-лока), а конкурентных записей не бывает.

#### Когда выгодно, а когда — нет

Выгодно в read-heavy сценариях: чтений на порядки больше записей, и сама операция чтения не мгновенна (иначе накладные расходы двух локов съедят выигрыш). При частой записи `ReadWriteLock` становится хуже обычного лока: его внутренняя механика дороже (нужно вести учёт читателей), а параллелизм чтений не успевает окупиться — писатели всё равно постоянно всех останавливают. Плюс появляется риск голодания писателя: при непрерывном потоке читателей писатель может ждать неприлично долго (детали механики — в 🔬).

Отсюда следующий эволюционный шаг — `StampedLock` (Java 8): он умеет оптимистичное чтение вообще без захвата лока, с последующей проверкой «а не влез ли писатель». Для baseline-знания достаточно знать, что он есть и зачем; механика — в 🔬-блоке.

### 5.3. Recap: wait / notify — база, без которой не понять Condition

Прежде чем идти к `Condition`, нужно восстановить его прародителя. Это не отдельная тема, а компактная база.

Блокировка отвечает на вопрос «кто сейчас имеет право трогать данные». Но в координации потоков есть второй, отдельный вопрос: «как подождать, пока данные придут в нужное состояние». Потребитель хочет взять элемент из пустого буфера — блокировка ему не поможет: он может эксклюзивно владеть пустым буфером сколько угодно, элемент от этого не появится. Ему нужно отпустить лок и уснуть, пока кто-то не положит элемент и не разбудит его.

Именно это встроено в каждый Java-монитор. У любого объекта, помимо лока, есть одна неявная очередь ожидания, и три метода `Object` работают с ней: `wait()` — «отпусти монитор и усни в очереди этого объекта, пока не разбудят» (вызывать можно только владея монитором, иначе `IllegalMonitorStateException`); `notify()` — «разбуди один произвольный поток из очереди»; `notifyAll()` — «разбуди всех». Проснувшийся поток не продолжает мгновенно: сначала он должен заново захватить монитор — и только потом выполнение идёт дальше.

```java
class Gate {
    private final Object lock = new Object();
    private boolean ready = false;

    void awaitReady() throws InterruptedException {
        synchronized (lock) {
            while (!ready) {   // while, не if — критично
                lock.wait();
            }
        }
    }

    void open() {
        synchronized (lock) {
            ready = true;
            lock.notifyAll();
        }
    }
}
```

#### Правило while, а не if — разжуём до конца

Почему нельзя написать `if (!ready) lock.wait();`? Две независимые причины, каждой достаточно.

**Причина 1 — ложные пробуждения (spurious wakeups).** Спецификация (JLS §17.2.1) прямо разрешает потоку проснуться из `wait()` без всякого `notify` — это плата за то, как реализованы примитивы ожидания на уровне ОС. Редко, но легально. С `if` такой поток пойдёт дальше при `ready == false` — и будет работать с состоянием, которого нет.

**Причина 2 — окно между пробуждением и захватом лока.** `notifyAll()` будит поток, но лок тот получит не сразу — сначала лок освободит будящий, потом его могут перехватить другие потоки. За это окно условие могло снова стать ложным: другой потребитель успел забрать единственный элемент раньше тебя. Проснулся — это не «условие истинно», это всего лишь «есть повод перепроверить».

Обе причины лечатся одним движением: проверка условия — всегда в цикле. Проснулся → перепроверил → условие ложно → уснул обратно. Это не «хорошая практика», это единственный корректный способ; `wait` вне цикла — классический баг и классический вопрос на собеседовании.

У механизма монитора два врождённых ограничения: очередь ожидания одна (все, кто ждёт разного, спят вперемешку — будить приходится всех через `notifyAll`, и большинство проснётся зря), и она намертво привязана к `synchronized`. Оба ограничения снимает `Condition`.

### 5.4. Condition: очереди ожидания под твоим управлением

**Ментальная модель.** `Condition` — это `wait`/`notify`, вынесенные из монитора в явный объект, привязанный к явному локу. Одному `ReentrantLock` можно дать несколько независимых очередей ожидания — по одной на каждое логическое условие — и будить именно ту сторону, которой есть смысл просыпаться.

Соответствие прямое: `lock.newCondition()` создаёт очередь; `await()` ↔ `wait()`, `signal()` ↔ `notify()`, `signalAll()` ↔ `notifyAll()`. Правила те же: вызывать можно только владея локом (иначе `IllegalMonitorStateException` — проверено на JDK 21), `await()` отпускает лок на время сна и заново захватывает при пробуждении, ложные пробуждения разрешены — правило `while` действует без изменений.

Зачем нужны несколько очередей — лучше всего показывает канонический пример, ограниченный буфер (producer-consumer). У буфера два разных условия ожидания: производители ждут «не полно», потребители ждут «не пусто». С монитором обе стороны спали бы в одной куче, и после каждой операции пришлось бы `notifyAll()` — будить всех, включая тех, кому просыпаться бессмысленно. С двумя `Condition` будим адресно:

```java
class BoundedBuffer<T> {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    private final Object[] items;
    private int count, putIndex, takeIndex;

    BoundedBuffer(int capacity) { items = new Object[capacity]; }

    void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) {
                notFull.await();              // буфер полон — производитель спит
            }
            items[putIndex] = item;
            putIndex = (putIndex + 1) % items.length;
            count++;
            notEmpty.signal();                // появился элемент — будим ОДНОГО потребителя
        } finally {
            lock.unlock();
        }
    }

    @SuppressWarnings("unchecked")
    T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notEmpty.await();             // буфер пуст — потребитель спит
            }
            T item = (T) items[takeIndex];
            items[takeIndex] = null;
            takeIndex = (takeIndex + 1) % items.length;
            count--;
            notFull.signal();                 // появилось место — будим ОДНОГО производителя
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

(Код прогнан на JDK 21: producer на 10 элементов при `capacity = 3` и медленный consumer — производитель корректно засыпает на полном буфере и досыпает элементы по мере освобождения места.)

Разбери глазами три вещи. Первое: `await()` — строго внутри `while` по своему предикату. Второе: `put` сигналит `notEmpty` (он изменил условие потребителей), `take` сигналит `notFull` — каждая сторона будит противоположную. Третье: здесь безопасен именно `signal()`, а не `signalAll()`, потому что в каждой очереди все ждут строго одного и того же предиката и любой разбуженный способен обработать событие; одно событие («появился один элемент») — один разбуженный. На пустом ходу никто не просыпается.

![[Pasted image 20260704145920.png]]

Этот пример — мост к следующей главе: `ArrayBlockingQueue` из JDK устроена ровно так — один `ReentrantLock`, два `Condition` (`notEmpty`, `notFull`), можешь открыть её исходники и узнать наш буфер один в один. В проде ты почти никогда не пишешь такой класс руками — берёшь готовую `BlockingQueue`. Но понимать, что упаковано внутри, обязательно: и для собеседования, и для того редкого случая, когда готовая структура не подходит.

> [!example] 🔬 Глубже / на синьора  
> **AQS — общий каркас.** `ReentrantLock`, `ReentrantReadWriteLock`, `Semaphore`, `CountDownLatch` — не независимые изобретения, а тонкие обёртки над одним классом: `AbstractQueuedSynchronizer` (AQS). Внутри AQS два ингредиента: `int state` (что он означает, решает наследник: для `ReentrantLock` — счётчик захватов, для `Semaphore` — число разрешений; у `ReentrantReadWriteLock` один int разделён пополам: старшие 16 бит — число читателей, младшие — глубина захвата писателя) и FIFO-очередь ждущих потоков (вариант CLH-очереди). Захват состояния — через CAS на `state`: вот прямая связка с главой 4 — блокирующие примитивы построены поверх неблокирующего. Быстрый путь (лок свободен, CAS удался) вообще не трогает очередь и не уходит в ядро ОС; парковка потока (`LockSupport.park`) — только когда CAS проиграл и надо ждать.
> 
> **Справедливый vs несправедливый — механика.** Несправедливый `lock()` первым делом пробует CAS «в лоб», игнорируя очередь, — это и есть barging: прибежавший поток может проскочить мимо спящих. Справедливый сначала проверяет `hasQueuedPredecessors()`: есть кто-то раньше меня — встаю в хвост. Отсюда и цена: справедливый режим гарантирует передачу через парковку/распарковку (микросекунды и промах кешей), несправедливый часто отдаёт лок «горячему» потоку без единого системного вызова.
> 
> **Голодание писателя в ReentrantReadWriteLock.** В несправедливом режиме непрерывный поток читателей теоретически может не подпускать писателя бесконечно. Реализация HotSpot смягчает это эвристикой: новый читатель добровольно встаёт в очередь, если в её голове уже ждёт писатель (`apparentlyFirstQueuedIsExclusive`) — но Javadoc честно предупреждает, что при постоянной конкуренции несправедливый режим может неограниченно откладывать как читателей, так и писателей. Гарантию отсутствия голодания даёт только справедливый режим — ценой пропускной способности.
> 
> **StampedLock и оптимистичное чтение.** `tryOptimisticRead()` не берёт лок вообще — возвращает штамп (версию состояния). Читаешь нужные поля в локальные переменные, затем `validate(stamp)`: если за это время не было писателя — данные согласованы, лок так и не был взят, стоимость чтения — почти ноль; если был — откатываешься на честный `readLock()`. Это версионная схема (как optimistic locking в БД), выигрывающая на очень горячем чтении. Цена: `StampedLock` нереентерабелен и не поддерживает `Condition` — это не замена `ReentrantLock`, а специнструмент.
> 
> **Апгрейд read → write невозможен.** Поток, держащий read-лок `ReentrantReadWriteLock`, при попытке взять write-лок дедлочит сам себя: write-лок ждёт ухода всех читателей, включая тебя самого (проверено: `writeLock().tryLock()` при удерживаемом read-локе возвращает `false`). Корректный путь: отпустить read, взять write, перепроверить условие (мир мог измениться в зазоре). Обратное — даунгрейд write → read (взять read, ещё держа write, потом отпустить write) — легален и полезен.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`unlock()` не в `finally` — или `lock()` внутри `try`.** Первое даёт вечно захваченный лок при исключении; второе при неудачном захвате приводит к `unlock()` чужого лока и `IllegalMonitorStateException`, маскирующей настоящую ошибку. Шаблон один: `lock()` → `try` → `finally { unlock() }`.
> 2. **`await()`/`wait()` вне цикла `while`.** Ложные пробуждения плюс окно между пробуждением и повторным захватом лока: проснуться ≠ дождаться. Проверка предиката — только в цикле.
> 3. **`signal()` при смешанной очереди.** Адресный `signal()` корректен, только когда все в очереди ждут один и тот же предикат. Если на одном `Condition` ждут разного (или используешь один `Condition` вместо двух), `signal()` может разбудить поток, которому событие не подходит, — тот перепроверит `while`, уснёт обратно, и сигнал потерян навсегда. Сомневаешься — `signalAll()`: дороже, но не теряет пробуждений.
> 4. **Долгая или блокирующая операция под локом.** I/O, сетевой вызов, `Thread.sleep` внутри критической секции — и весь конкурентный дизайн деградирует до очереди к одному окошку. Под локом — только работа с охраняемым состоянием; тяжёлое — снаружи.
> 5. **Вложенные локи в разном порядке** — дедлок из главы 1 никуда не делся: явные локи подвержены ему так же, как `synchronized`. Лечится либо глобальным порядком захвата, либо `tryLock` с таймаутом и отпусканием всего при неудаче (шаблон из 5.1).
> 6. **`await()`/`signal()` без захваченного лока** — немедленный `IllegalMonitorStateException`. В отличие от пункта 1, эта ошибка хотя бы громкая.
> 7. **Апгрейд read → write в `ReentrantReadWriteLock`** — самодедлок (см. 🔬). Отпусти read, возьми write, перепроверь условие.

### На собесе

> [!question]- Чем ReentrantLock отличается от synchronized и когда он оправдан? _(самый частый вопрос темы)_  
> Follow-up почти всегда: «а что будет, если забыть `unlock()`?» и «так что использовать по умолчанию?». Слабый ответ — «`ReentrantLock` быстрее» (давно неправда как общее утверждение) или перечисление фич без вывода про дефолт.
> 
> **RU:** Это тот же взаимоисключающий реентерабельный лок с той же семантикой памяти, но в виде явного объекта. Сверх `synchronized` он даёт `tryLock()` с таймаутом, прерываемое ожидание `lockInterruptibly()`, опциональную справедливость и несколько `Condition` на одном локе. Цена — освобождение только вручную: `unlock()` обязан быть в `finally`, иначе исключение оставит лок захваченным навсегда. По умолчанию использую `synchronized`; `ReentrantLock` — когда нужна хотя бы одна из его возможностей.
> 
> **EN:** It's the same mutually exclusive, reentrant lock with the same memory semantics, but as an explicit object. On top of synchronized it offers tryLock() with a timeout, interruptible acquisition, optional fairness, and multiple Condition queues per lock. The price is manual release: unlock() must go in finally, otherwise an exception leaves the lock held forever. My default is synchronized; I reach for ReentrantLock only when I need one of those extra capabilities.

> [!question]- Почему wait()/await() вызывают в цикле while, а не в if? _(классика, отсекающая тех, кто копировал код не понимая)_  
> Follow-up: «что такое spurious wakeup?»
> 
> **RU:** Две причины. Спецификация разрешает ложные пробуждения — возврат из `wait()` без сигнала. И даже после честного сигнала между пробуждением и повторным захватом лока есть окно, за которое другой поток мог снова сделать условие ложным. Поэтому пробуждение — лишь повод перепроверить предикат: `while (!condition) wait();`.
> 
> **EN:** Two reasons. The spec permits spurious wakeups — returning from wait() without any signal. And even after a genuine signal there's a window between waking up and reacquiring the lock, during which another thread may have made the condition false again. So waking up only means "recheck the predicate": while (!condition) wait();.

> [!question]- Что даёт ReadWriteLock и когда он вредит?  
> Follow-up-ловушка: «можно ли, держа read-лок, взять write-лок?» — нет, это самодедлок.
> 
> **RU:** Модель «много читателей или один писатель»: read-лок разделяемый, write-лок эксклюзивный. Выгоден при явном перевесе чтений над записями; при частой записи его более тяжёлая механика не окупается, и обычный лок быстрее. Ещё риски: голодание писателя в несправедливом режиме и невозможность апгрейда read→write — нужно отпустить read, взять write и перепроверить состояние.
> 
> **EN:** The model is "many readers or one writer": the read lock is shared, the write lock exclusive. It pays off only in read-heavy workloads; with frequent writes its heavier bookkeeping isn't amortized and a plain lock is faster. Also: possible writer starvation in non-fair mode, and no read→write upgrade — you must release the read lock, acquire the write lock, and recheck the state.

> [!question]- Как tryLock помогает против дедлока?  
> Follow-up: «какая новая проблема появляется взамен?» — livelock, лечится случайной задержкой перед retry.
> 
> **RU:** Дедлок требует, чтобы поток держал один лок и бесконечно ждал другой. `tryLock(timeout)` ломает это условие: не получил второй лок за таймаут — отпускаешь первый и повторяешь попытку. Взамен возможен livelock при синхронном ритме повторов; лечится джиттером.
> 
> **EN:** Deadlock requires holding one lock while waiting indefinitely for another. tryLock(timeout) breaks that: if the second lock isn't acquired within the timeout, you release the first and retry. The trade-off is possible livelock if retries synchronize; adding random backoff fixes it.

> [!question]- Зачем Condition, если есть wait/notify?  
> Follow-up: «когда signal() безопасен вместо signalAll()?»
> 
> **RU:** `Condition` — те же семантика и правила, что у `wait`/`notify`, но на явном локе и, главное, несколько независимых очередей на одном локе. В producer-consumer это отдельные `notFull` и `notEmpty`: будим адресно только нужную сторону вместо `notifyAll` для всех. `signal()` безопасен, когда все в данной очереди ждут один предикат и любой разбуженный может обработать событие; иначе — `signalAll()`.
> 
> **EN:** Condition has the same semantics and rules as wait/notify, but lives on an explicit lock and — crucially — a single lock can have several independent wait queues. In producer-consumer that's separate notFull and notEmpty: you wake exactly the side that can make progress, instead of notifyAll for everyone. signal() is safe when everyone in that queue waits on the same predicate and any woken thread can handle the event; otherwise use signalAll().

> [!question]- Что такое реентерабельность? _(короткий, но частый)_  
> **RU:** Поток, уже владеющий локом, может захватить его повторно, не заблокировав сам себя; внутри ведётся счётчик, и лок освобождается, когда число `unlock()` сравнялось с числом `lock()`. И `synchronized`, и `ReentrantLock` реентерабельны — без этого метод под локом не мог бы вызвать другой метод под тем же локом.
> 
> **EN:** A thread that already owns the lock can acquire it again without blocking itself; a hold count is maintained, and the lock is released when unlock() calls match lock() calls. Both synchronized and ReentrantLock are reentrant — without it a method holding the lock couldn't call another method guarded by the same lock.
