
Глава 7 закончилась на разграничении, которое стоит держать в голове постоянно: виртуальные потоки решают проблему _ожидания_ — тысячи задач, которые в основном спят на I/O. Но они не создают вычислительной мощности из воздуха. Если задача одна, большая и молотит процессор без пауз — пересчитать 50 миллионов строк, отсортировать гигантский массив, обойти дерево, — то хоть миллион виртуальных потоков заведи, быстрее восьми физических ядер она не посчитается. Для счётной (CPU-bound) стороны нужен другой инструмент: взять **одну** большую задачу и разложить её на **все** ядра. Это и есть Fork/Join framework, появившийся ещё в Java 7 — и именно он, как выяснится ниже, живёт под капотом у `parallelStream()`.

**Ментальная модель:** Fork/Join — это пул потоков, заточенный под рекурсивное деление задачи: делим пополам, пока кусок не станет достаточно мелким, считаем куски напрямую и сливаем результаты обратно вверх. `fork` — разветвить, `join` — дождаться и слить. А чтобы ни одно ядро не простаивало, пул использует work-stealing: свободный воркер крадёт задачи у занятого.

### 8.1. Divide-and-conquer: fork вниз, join вверх

Обычный `ThreadPoolExecutor` из главы 2 устроен просто: одна общая очередь, воркеры разбирают из неё независимые задачи. Эта модель ломается, когда задачи _порождают подзадачи и ждут их результатов_: воркер, заблокировавшийся на ожидании подзадачи, занимает поток впустую, и при глубокой рекурсии пул фиксированного размера просто встаёт — все потоки ждут, исполнять подзадачи некому. Fork/Join спроектирован ровно под этот паттерн.

Носители задач — два абстрактных класса:

- `RecursiveTask<V>` — задача, возвращающая результат (`V`);
- `RecursiveAction` — задача без результата (например, отсортировать массив на месте).

Оба требуют переопределить один метод — `compute()`. Внутри него живёт вся логика divide-and-conquer. Вот честный, проверенный на JDK 21 пример — сумма большого массива:

```java
import java.util.concurrent.ForkJoinPool;
import java.util.concurrent.RecursiveTask;

public class ArraySumTask extends RecursiveTask<Long> {

    private static final int THRESHOLD = 10_000;

    private final long[] array;
    private final int from;
    private final int to;

    public ArraySumTask(long[] array, int from, int to) {
        this.array = array;
        this.from = from;
        this.to = to;
    }

    @Override
    protected Long compute() {
        int length = to - from;
        if (length <= THRESHOLD) {          // кусок достаточно мал —
            long sum = 0;                   // считаем напрямую, без деления
            for (int i = from; i < to; i++) {
                sum += array[i];
            }
            return sum;
        }
        int mid = from + length / 2;
        ArraySumTask left = new ArraySumTask(array, from, mid);
        ArraySumTask right = new ArraySumTask(array, mid, to);

        left.fork();                        // левую половину — в очередь пула
        long rightResult = right.compute(); // правую считаем в текущем потоке
        long leftResult = left.join();      // дожидаемся левую

        return leftResult + rightResult;
    }
}
```

Запуск — через `invoke()` на пуле:

```java
long[] data = ...; // 50 млн элементов
long sum = ForkJoinPool.commonPool()
        .invoke(new ArraySumTask(data, 0, data.length));
```

Разбери внимательно идиому в середине `compute()` — она не случайна:

- `left.fork()` **не создаёт поток**. Он кладёт подзадачу в очередь текущего воркера, откуда её заберёт либо он сам, либо сосед (об этом — весь следующий раздел). Это дёшево.
- `right.compute()` — правую половину текущий поток считает **сам, напрямую**. Наивный вариант `left.fork(); right.fork(); left.join() + right.join()` работает, но хуже: ты отдал в очередь обе половины и остался стоять без дела, плюс оплатил лишнюю задачу-обёртку. Идиома «fork одну, compute другую» — стандарт.
- `left.join()` — дождаться результата левой половины. В отличие от `Future.get()` из главы 2, `join()` в Fork/Join-воркере не превращает поток в бревно: пока результата нет, воркер может исполнять другие подзадачи из очереди. Именно поэтому глубокая рекурсия не вешает пул.

#### Порог разбиения — первый вопрос, который отделяет «читал статью» от «понял»

Условие `length <= THRESHOLD` — не техническая деталь, а центральное инженерное решение. У каждой подзадачи есть накладные расходы: создать объект, положить в очередь, забрать, скоординировать `join`. Сама операция «сложить два числа» стоит наносекунды — дешевле, чем вся эта бюрократия вокруг неё.

Отсюда две крайности, и обе плохие:

- **Порог слишком крупный** (например, делим один раз пополам на 16-ядерной машине) — получили 2 куска на 16 ядер, 14 ядер простаивают. Параллелизм не раскрыт.
- **Порог слишком мелкий** — задач стало больше, чем полезной работы. Оверхед на создание и координацию подзадач съедает весь выигрыш; в пределе программа тратит больше времени на менеджмент задач, чем на счёт. Разбивать до единичного элемента — классический антипаттерн из туториалов.

Как о пороге думать: подзадач должно быть заметно больше, чем ядер (чтобы work-stealing'у было что балансировать — куски редко считаются одинаково долго), но каждая листовая задача должна нести ощутимую работу. Javadoc `ForkJoinTask` даёт грубый ориентир: листовая задача — примерно от 100 до 10 000 базовых вычислительных шагов. Для суммы массива это тысячи–десятки тысяч элементов на кусок, не десять. Точное значение — вопрос измерения на твоих данных, не константа из книжки.

### 8.2. Work-stealing: сердце ForkJoinPool

Хорошо, задачу мы нарезали. Теперь второй вопрос: как раздать куски потокам? Наивный ответ — поровну: 16 кусков, 4 воркера, каждому по 4. Проблема в том, что куски почти никогда не считаются одинаково долго: где-то данные «тяжелее», где-то поток вытеснила ОС, где-то дерево рекурсии оказалось кривым (поиск нашёлся в одной половине мгновенно, а в другой пришлось перелопатить всё). При статичной раздаче один воркер закончит через секунду и будет курить, пока сосед пашет ещё десять. Простаивающее ядро — это ровно то, ради чего мы всё затевали, выброшенное в мусор.

`ForkJoinPool` решает это иначе, и это его главное отличие от `ThreadPoolExecutor`:

- У **каждого воркера — своя двусторонняя очередь** (deque), а не одна общая на пул. Когда воркер делает `fork()`, подзадача ложится в _его собственную_ очередь.
- Свои задачи воркер берёт **с одного конца** очереди — с хвоста, куда сам только что положил.
- Когда его очередь пуста, воркер не идёт спать. Он выбирает занятого соседа и **крадёт задачу с другого конца** его очереди — с головы. Это и есть work-stealing.

Результат — почти автоматическая балансировка: пока в системе есть хоть какая-то работа, свободных воркеров нет. Никакого центрального диспетчера, который следит за загрузкой, не существует — балансировка возникает сама из локального правила «нечего делать — укради».

Почему свои — с одного конца, а кража — с другого? Это не эстетика, это механика производительности. Если бы владелец и вор тянулись к одному и тому же концу очереди, каждый доступ требовал бы полноценной синхронизации между ними — и очередь стала бы узким местом. Разнеся их по разным концам, мы получаем ситуацию, где в типичном случае владелец и вор работают с _разными_ задачами в _разных_ концах структуры и почти не конкурируют. Синхронизация нужна только в вырожденном случае — когда в очереди осталась одна задача и оба тянутся к ней одновременно.

![[Pasted image 20260704151642.png]]

Обрати внимание на деталь, которую диаграмма показывает, но легко проглядеть: Worker 3 украл `t₁` — самую _старую_ задачу в очереди, а не самую свежую. В divide-and-conquer старая задача — это ещё не разделённый _крупный_ кусок дерева. Вор уносит к себе большой фронт работы, делит его уже в своей очереди и надолго перестаёт беспокоить соседей. Почему владельцу, наоборот, выгодно брать свежие — разберём в 🔬-блоке.

### 8.3. `parallelStream` под капотом

Теперь практический вывод, ради которого mid+ разработчику вообще стоит знать Fork/Join. Ты почти наверняка не будешь писать `RecursiveTask` руками — но ты будешь писать вот это:

```java
long count = orders.parallelStream()
        .filter(o -> o.total() > 1000)
        .count();
```

И вот три вещи про эту строчку, которые надо знать твёрдо.

**Первое: `parallelStream()` исполняется на `ForkJoinPool.commonPool()`** — общем пуле, одном на всю JVM. Никакого «своего» пула у параллельного стрима нет: сплитератор режет источник на куски (то самое divide-and-conquer, только за тебя), куски превращаются в Fork/Join-задачи и уходят в commonPool. Проверяется в две строки:

```java
List<Integer> data = IntStream.rangeClosed(1, 8).boxed().toList();

data.parallelStream()
    .forEach(i -> System.out.println(i + " -> " + Thread.currentThread().getName()));
```

```
3 -> ForkJoinPool.commonPool-worker-1
4 -> ForkJoinPool.commonPool-worker-1
6 -> main
8 -> ForkJoinPool.commonPool-worker-2
2 -> ForkJoinPool.commonPool-worker-3
...
```

Заметь `main` в выводе: вызывающий поток не стоит и не ждёт — он сам встаёт к станку и обрабатывает часть кусков наравне с воркерами. Это, кстати, объясняет размер commonPool (об этом в 🔬).

Этот пул ты уже встречал дважды. В главе 3 — это дефолтный executor `CompletableFuture.supplyAsync()`, когда ты не передал свой. И в главе 7 всплывал `ForkJoinPool` как механизм планировщика виртуальных потоков — там, правда, это _отдельный_ экземпляр `ForkJoinPool` (не commonPool), но устроен он на том же work-stealing. Итого картина: один и тот же класс несёт `parallelStream`, дефолтные `CompletableFuture` и carrier-потоки виртуальных, причём первые два — буквально **один общий пул на всю JVM**. Что подводит ко второму пункту.

**Второе: блокирующая операция внутри `parallelStream` — та же грабля, что в главе 3, только больнее.** commonPool рассчитан на CPU-bound работу и по размеру примерно равен числу ядер. Положи в него I/O:

```java
// АНТИПАТТЕРН: сетевой вызов в параллельном стриме
List<Profile> profiles = userIds.parallelStream()
        .map(id -> httpClient.fetchProfile(id))   // блокирующий I/O
        .toList();
```

— и на время этих вызовов все воркеры commonPool висят на сокетах. Пул общий, поэтому пострадал не твой стрим — пострадала **вся JVM**: чужие `parallelStream` в других частях приложения и все `CompletableFuture` на дефолтном executor'е выстроились в очередь за пулом, который ничего не считает, а ждёт сеть. Правило из глав 3 и 7 здесь звучит финальным аккордом: `parallelStream` — для CPU-bound. Для I/O — виртуальные потоки или свой executor.

**Третье: `parallelStream` — не бесплатное ускорение, а инструмент под конкретный профиль нагрузки.** У параллельного исполнения есть цена: разрезать источник, раскидать задачи, слить результаты. Выигрыш должен эту цену перекрыть. Когда перекрывает:

- **Данных много.** На коллекции в сотню элементов оверхед разбиения/слияния съест всё; выигрыш начинается на десятках тысяч и больше — в зависимости от тяжести операции.
- **Источник дёшево режется пополам.** Массив и `ArrayList` сплитератор делит арифметикой по индексу — идеально. `LinkedList`, результат `Stream.iterate`, `BufferedReader.lines()` — чтобы найти середину, надо физически пройти элементы; такие источники параллелятся плохо, и `parallelStream` на них часто _медленнее_ последовательного.
- **Операция на элемент достаточно тяжёлая.** Умножить число на два — нет; распарсить, посчитать, провалидировать — уже разговор. Полезный ориентир Брайана Гётца: N × Q (число элементов × стоимость обработки одного) должно быть хотя бы порядка десятков тысяч «единиц работы», иначе игра не стоит свеч.
- **Операции независимы и без общего состояния.** Лямбда, пишущая в общий `ArrayList` или инкрементирующая обычный счётчик, — это data race из глав 1 и 4, теперь размазанный по всем ядрам. Только чистые функции + сбор через `collect`.
- **Слияние дешёвое.** `sum()`, `count()`, `toList()` сливаются дёшево; `collect(toMap(...))` с миллионом ключей — слияние промежуточных map может стоить дороже самой обработки.

Не выполняется хотя бы одно условие — последовательный стрим, скорее всего, быстрее. Вывод, который стоит произносить на собеседовании дословно: `.parallel()` — это гипотеза, которую проверяют бенчмарком (JMH), а не оптимизация по умолчанию.

> [!tip] 💡 Из C#  
> Fork/Join идейно ≈ TPL: планировщик TPL — тоже work-stealing с локальными очередями воркеров. `parallelStream` ≈ PLINQ (`.AsParallel()`) и `Parallel.For`.
> 
> ⚠️ И ложный друг здесь не в различии, а в одинаковости граблей: `parallelStream` — не «магическое ускорение», ровно как `Parallel.For` в .NET. Тот же общий пул на процесс (commonPool ↔ ThreadPool), те же последствия блокирующего I/O внутри параллельного цикла, та же необходимость мерить, а не верить. Если ты обжигался на `Parallel.For` с HTTP-вызовами внутри — картина в Java один в один. Отличие одно, и оно тонкое: у `Task` из TPL есть async/await, снимающий поток на время ожидания; у Fork/Join-задачи такого механизма нет — заблокировался значит заблокировался (частичный костыль — `ManagedBlocker`, см. ниже).

> [!example] 🔬 Глубже / на синьора  
> **LIFO для своих, FIFO для чужих — почему именно так.** Владелец берёт с хвоста самую свежую подзадачу — ту, что сам только что породил. Её данные ещё горячие в кэше процессора (это соседний кусок того же массива, который он только что трогал), и в дереве рекурсии это самый глубокий, мелкий узел — его быстро досчитать и подняться по join. Вор с головы берёт самую старую — верхний, ещё не разделённый узел дерева, то есть максимально крупный кусок: одна кража приносит много работы, и частота краж (а кража дороже локального взятия) падает. Итого: LIFO даёт локальность кэша, FIFO-кража — крупные порции и редкие конфликты.
> 
> **Размер commonPool — `Runtime.availableProcessors() − 1` (минимум 1).** Минус один — потому что вызывающий поток (тот самый `main` из вывода выше) сам участвует в работе: воркеры + вызывающий = число ядер, ровно под CPU-bound без оверсабскрипшена. Крутится системным свойством — проверено:
> 
> ```
> java -Djava.util.concurrent.ForkJoinPool.common.parallelism=4 App.java
> ```
> 
> Это глобальная ручка на всю JVM; менять её ради одного стрима — плохая идея. Известный трюк «свой пул для стрима» — `new ForkJoinPool(8).submit(() -> list.parallelStream()...).get()` — работает (задачи стрима останутся в пуле, из которого запущены), но опирается на недокументированную деталь реализации, а не на контракт API. Знать — стоит, строить на этом прод — с осторожностью.
> 
> **`ManagedBlocker` — способ честно предупредить пул о блокировке.** Если Fork/Join-задаче неизбежно нужно заблокироваться, можно обернуть блокирующий участок в `ForkJoinPool.ManagedBlocker` и вызвать `ForkJoinPool.managedBlock(...)`: пул увидит, что воркер выпадает, и может компенсировать это, временно добавив поток, чтобы целевой параллелизм сохранился. Так, например, `Phaser` из JDK дружит с `ForkJoinPool`. Это костыль для редких случаев, а не индульгенция на I/O в `parallelStream`: компенсирующие потоки не бесплатны, и при массовых блокировках пул всё равно деградирует.
> 
> **Планировщик виртуальных потоков (глава 7)** — это тоже `ForkJoinPool`, но отдельный экземпляр, не commonPool, и сконфигурированный в асинхронном режиме: воркеры разбирают собственные очереди в FIFO-порядке, а не LIFO. Для потока событий/задач, которые не образуют дерево рекурсии, FIFO честнее по латентности (кто раньше пришёл, тот раньше обслужен), тогда как LIFO заточен именно под divide-and-conquer. Один класс — два режима под два профиля нагрузки.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Блокирующий I/O внутри `parallelStream` или Fork/Join-задачи.** Забивает общий на всю JVM commonPool; страдают все параллельные стримы и дефолтные `CompletableFuture` приложения, включая чужой код. Симптом в проде: «параллельные» операции внезапно исполняются последовательно, thread dump показывает воркеров commonPool в `WAITING` на сокетах.
> 2. **`parallelStream` на маленьких данных или дешёвой операции.** Оверхед разбиения и слияния превышает выигрыш — параллельная версия медленнее последовательной. Особенно коварно в горячем коде: замедление на каждом вызове, которое никто не мерил.
> 3. **Плохо сплитуемый источник.** `LinkedList`, `Stream.iterate`, I/O-стримы делятся дорого и криво — куски получаются несбалансированными, часть ядер простаивает. Хочешь параллелить — сначала материализуй в массив или `ArrayList`.
> 4. **Слишком мелкий порог в своём `RecursiveTask`.** Разбиение до единичных элементов: число задач взрывается, координация съедает всё. Ориентир из Javadoc `ForkJoinTask` — 100…10 000 вычислительных шагов на листовую задачу, дальше — измерение.
> 5. **Общее изменяемое состояние в лямбде параллельного стрима.** `forEach(x -> sharedList.add(x))` или инкремент обычного поля — data race (главы 1 и 4): потерянные элементы, `ArrayIndexOutOfBoundsException` из недр `ArrayList`, невоспроизводимые баги. Правильно — `collect(...)`, редукции, в крайнем случае атомики.
> 6. **Вера, что `.parallel()` всегда ускоряет.** Самая частая грабля из всех: параллельность добавляют «на всякий случай», не измерив. Единственный честный ответ на вопрос «ускорит ли здесь `parallelStream`» — бенчмарк на реальных данных.
> 7. **`join()` вне ForkJoinPool-воркера.** Умение `join()` «поработать над другими задачами пока ждёт» действует внутри пула. Обычный поток, вызвавший `task.join()`, просто блокируется, как на `Future.get()`.

### На собесе

> [!question]- 1. Что такое Fork/Join framework и когда его применять? _(частый, mid)_  
> Слабый ответ: «Это пул потоков для параллельных задач» — не отличает его от `ThreadPoolExecutor`, follow-up убьёт.
> 
> **RU:** Fork/Join — пул для CPU-bound задач, которые делятся рекурсивно: задача в `compute()` проверяет порог, ниже порога считает напрямую, выше — делит пополам (`fork()` кладёт подзадачу в очередь воркера), считает одну половину сама и `join()`-ит вторую. Заточен под задачи, порождающие подзадачи: воркер, ждущий `join()`, не простаивает, а исполняет другие задачи. Применение: большие однородные вычисления — сортировка, агрегации по массивам, обход деревьев; в повседневном коде — опосредованно, через `parallelStream` и `Arrays.parallelSort`.
> 
> **EN:** Fork/Join is a thread pool designed for CPU-bound divide-and-conquer work: a task recursively splits itself in half until a threshold, computes small chunks directly, and merges results via fork()/join(). Unlike a regular executor, a worker waiting on join() keeps executing other subtasks instead of blocking, so deep recursive task trees don't starve the pool. In practice you mostly meet it indirectly — it powers parallelStream and Arrays.parallelSort.
> 
> _Follow-up:_ «Чем отличается от `ThreadPoolExecutor`?» → одна общая очередь против per-worker deque + work-stealing; TPE рассчитан на независимые задачи, FJP — на задачи с подзадачами; блокирующий `get()` результата подзадачи в TPE фиксированного размера ведёт к самоблокировке пула.

> [!question]- 2. Что такое work-stealing и почему вор крадёт с другого конца очереди? _(любимый вопрос, mid+)_  
> Слабый ответ: «Свободные потоки забирают задачи у занятых» — без механики deque и без «почему с разных концов» это половина ответа.
> 
> **RU:** У каждого воркера своя двусторонняя очередь. Свои задачи он кладёт и берёт с хвоста (LIFO), а опустев — крадёт с головы чужой очереди (FIFO). Разные концы дают два выигрыша. Первый — минимум конкуренции: владелец и вор в типичном случае трогают разные задачи на разных концах, синхронизация нужна только когда в очереди остался один элемент. Второй — качество выбора: владельцу LIFO даёт свежую подзадачу с горячими в кэше данными, а вору FIFO даёт самую старую, то есть самый крупный неразделённый кусок дерева — одна кража приносит много работы, красть приходится редко.
> 
> **EN:** Each worker owns a deque: it pushes and pops its own tasks at the tail (LIFO), and when empty it steals from the head of a busy neighbor's deque (FIFO). Opposite ends mean the owner and the thief almost never contend — they touch different slots, and synchronization only matters when one task is left. It also optimizes what each side gets: LIFO gives the owner the freshest subtask with cache-hot data, while FIFO gives the thief the oldest, i.e. the largest unsplit chunk — one steal brings a lot of work, so steals stay rare.

> [!question]- 3. Зачем нужен порог разбиения и что будет, если делить до одного элемента? _(частый, mid)_  
> **RU:** Каждая подзадача стоит денег: объект, постановка в очередь, координация join. Порог гарантирует, что листовая задача несёт достаточно полезной работы, чтобы окупить эту бюрократию. Слишком крупный порог — мало кусков, ядра простаивают; слишком мелкий — оверхед координации съедает выигрыш, деление до элемента может сделать «параллельную» версию в разы медленнее цикла. Ориентир из Javadoc: 100–10 000 вычислительных шагов на лист; точнее — только измерением.
> 
> **EN:** Every subtask has overhead — object creation, queueing, join coordination — so the threshold ensures each leaf carries enough real work to pay for it. Too coarse: not enough chunks to keep all cores busy. Too fine: coordination overhead dominates; splitting down to single elements is an antipattern that can make the parallel version slower than a plain loop. The ForkJoinTask javadoc suggests roughly 100–10,000 computational steps per leaf; the exact value comes from benchmarking.

> [!question]- 4. На каком пуле исполняется parallelStream и чем это опасно? _(очень частый, mid+ — проверка на прод-опыт)_  
> Слабый ответ: «На каком-то внутреннем пуле» или «создаёт потоки под задачу». Провал.
> 
> **RU:** На `ForkJoinPool.commonPool()` — общем статическом пуле JVM размером примерно в число ядер (`availableProcessors − 1`, плюс участвует вызывающий поток). Опасность: пул один на всё приложение. Блокирующая операция (HTTP, JDBC, файл) внутри `parallelStream` занимает воркеров ожиданием — и деградируют все параллельные стримы и все `CompletableFuture` на дефолтном executor'е во всей JVM, включая чужой код. Поэтому `parallelStream` — строго для CPU-bound; I/O — в виртуальные потоки или выделенный executor.
> 
> **EN:** On ForkJoinPool.commonPool() — a single JVM-wide pool sized roughly to the number of cores (availableProcessors − 1, with the calling thread pitching in). The danger: it's shared. Blocking calls (HTTP, JDBC) inside a parallel stream park the workers, degrading every parallel stream and every default-executor CompletableFuture across the whole JVM, including unrelated code. So parallel streams are strictly for CPU-bound work; I/O belongs on virtual threads or a dedicated executor.
> 
> _Follow-up-ловушка:_ «А как ограничить `parallelStream` своим пулом?» → трюк с запуском стрима изнутри своего `ForkJoinPool` работает, но это недокументированная деталь реализации, не контракт; честные альтернативы — свой executor + `CompletableFuture`, либо нарезать данные вручную.

> [!question]- 5. Когда parallelStream ускоряет, а когда замедляет? _(частый, mid+)_  
> **RU:** Ускоряет при совпадении условий: большой объём данных; дёшево сплитуемый источник (массив, `ArrayList` — деление по индексу; `LinkedList` и `Stream.iterate` — плохо); достаточно тяжёлая операция на элемент (ориентир Гётца: N×Q от ~10 000); независимые операции без общего состояния; дешёвое слияние (`sum`, `count`, `toList` — да; `toMap` на миллионах ключей — уже вопрос). Замедляет при нарушении любого: на маленьких коллекциях, дешёвых операциях и плохих источниках оверхед разбиения/слияния перевешивает. И это не теория — решение принимается бенчмарком (JMH), не интуицией.
> 
> **EN:** It speeds things up when all of these hold: large data volume; a cheaply splittable source (arrays and ArrayList split by index; LinkedList and Stream.iterate split poorly); non-trivial per-element cost (Goetz's N×Q ≳ 10,000 heuristic); independent, side-effect-free operations; and a cheap merge step. Break any of these and the split/merge overhead wins, making the sequential stream faster. The decision is made with a benchmark (JMH), not intuition.

> [!question]- 6. Чем ForkJoinPool отличается от ThreadPoolExecutor? _(средней частоты, mid+/senior)_  
> **RU:** Три оси. Очереди: у TPE одна общая `BlockingQueue`, у FJP — deque на каждого воркера плюс work-stealing, то есть балансировка встроена и децентрализована. Модель задач: TPE — независимые задачи, FJP — задачи, порождающие подзадачи; `join()` в FJP-воркере позволяет исполнять другую работу во время ожидания, тогда как ожидание результата подзадачи в фиксированном TPE может дедлокнуть пул. Назначение: TPE — универсальный, в т.ч. под I/O с настраиваемой очередью и rejection policy; FJP — специализация под CPU-bound divide-and-conquer, и его commonPool несёт `parallelStream` и дефолтные `CompletableFuture`.
> 
> **EN:** Three axes. Queuing: TPE has one shared BlockingQueue; FJP gives each worker its own deque plus work-stealing, so load balancing is built-in and decentralized. Task model: TPE assumes independent tasks; FJP is built for tasks that spawn subtasks — join() inside a worker lets it execute other tasks while waiting, whereas waiting for a subtask's result in a fixed-size TPE can deadlock the pool. Purpose: TPE is general-purpose (including I/O, with configurable queues and rejection policies); FJP is specialized for CPU-bound divide-and-conquer, and its common pool underlies parallel streams and default CompletableFuture execution.
> 
> _Senior-добавка:_ планировщик виртуальных потоков — тоже `ForkJoinPool`, но отдельный экземпляр в FIFO-режиме (async mode): LIFO оптимален для дерева рекурсии, FIFO — честнее по латентности для потока независимых задач.
