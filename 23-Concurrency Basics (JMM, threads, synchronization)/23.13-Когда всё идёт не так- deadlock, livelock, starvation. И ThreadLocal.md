
В уроках 4–5 мы научились главному: защищать общие данные локами и координировать потоки через `wait`/`notify`. Казалось бы, инструментарий собран — бери и пиши. Но у локов есть обратная сторона: они сами порождают новый класс проблем. Потоки могут честно соблюдать все правила блокировки — и всё равно застрять навсегда, крутиться на месте или годами не получать ресурс. Это не гонки из урока 3, где данные ломаются; здесь данные целы, но _программа перестаёт двигаться_.

В этом уроке мы разберём три патологии блокировок — deadlock, livelock и starvation, — научимся диагностировать их в проде через thread dump, а во второй части посмотрим на инструмент, который стоит особняком: `ThreadLocal`. Он решает проблему разделения данных радикально иначе — не защищает общее, а делает так, чтобы общего не было. И у него есть своя коварная ловушка, которую спрашивают почти на каждом собеседовании уровня mid+.

Это последний урок фундамента. Дальше — блок 24: пулы потоков, атомики, конкурентные коллекции, виртуальные потоки. Всё, что мы разберём сегодня, там будет всплывать постоянно.

### Часть 1. Патологии блокировок

#### 1.1. Deadlock — взаимная блокировка

**Ментальная модель:** deadlock — это ситуация, когда два (или больше) потока держат ресурсы и ждут ресурсы друг друга, образуя замкнутый цикл ожидания. Никто не отпустит своё, пока не получит чужое, — и цикл не разомкнётся никогда.

Начнём с самого частого сценария в реальном коде — перевод денег между счетами. Каждый счёт защищён своим локом, и чтобы перевести деньги, нужно захватить оба:

```java
public class DeadlockDemo {

    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        Thread transfer1 = new Thread(() -> {
            synchronized (lockA) {
                sleep(100); // даём второму потоку время захватить lockB
                synchronized (lockB) {
                    System.out.println("transfer-1: получил оба лока");
                }
            }
        }, "transfer-1");

        Thread transfer2 = new Thread(() -> {
            synchronized (lockB) {
                sleep(100);
                synchronized (lockA) {
                    System.out.println("transfer-2: получил оба лока");
                }
            }
        }, "transfer-2");

        transfer1.start();
        transfer2.start();
    }

    private static void sleep(long ms) {
        try { Thread.sleep(ms); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

Запускаем — и программа молча висит. Ни строчки вывода, ни исключения, ни падения. Разберём, что произошло:

1. `transfer-1` захватил `lockA` и заснул на 100 мс.
2. `transfer-2` за это время захватил `lockB` и тоже заснул.
3. `transfer-1` просыпается и пытается войти в `synchronized (lockB)` — но `lockB` занят. Поток переходит в `BLOCKED` (состояния — урок 2) и ждёт.
4. `transfer-2` просыпается и пытается взять `lockA` — занят. Тоже `BLOCKED`.

Каждый держит то, что нужно другому, и ждёт то, что другой не отдаст. Ключевое свойство `synchronized`: поток, ждущий монитор, **не отпускает уже захваченные** мониторы. Отпустить лок можно, только выйдя из блока, — а выйти нельзя, потому что стоим внутри и ждём. Цикл замкнулся:

##### Четыре условия Коффмана

Дедлок — не невезение, а точная механика. Чтобы система застряла, должны _одновременно_ совпасть четыре вещи. Уберёшь любую одну — дедлок физически невозможен:

1. **Взаимное исключение.** Ресурс не делится: лок в один момент держит ровно один поток. Это сама природа `synchronized` — тут ничего не изменишь.
2. **Удержание с ожиданием.** Поток, держащий один лок, ждёт другой — и не отпускает первый. Именно так ведёт себя вложенный `synchronized`.
3. **Отсутствие вытеснения.** Лок нельзя отобрать силой: его отдаёт только сам владелец, выйдя из блока. JVM никогда не «выдёргивает» монитор у потока.
4. **Круговое ожидание.** Цепочка «кто кого ждёт» замыкается в кольцо: A ждёт B, B ждёт A (или длиннее — A → B → C → A).

Первые три условия — это просто «как устроены локи», они выполняются почти всегда. Поэтому вся практическая борьба с дедлоками сводится к разрушению четвёртого условия — не дать кольцу замкнуться. Или второго — не ждать бесконечно, держа захваченное.

##### Как предотвращать

**Способ 1 — единый глобальный порядок захвата.** Если _все_ потоки берут локи в одном и том же порядке (сначала лок 1, потом лок 2 — всегда), кольцо невозможно математически: чтобы ждать лок 1, держа лок 2, кто-то должен был нарушить порядок. Для счетов естественный порядок даёт `id`:

```java
static void transfer(Account from, Account to, long amount) {
    // Кто бы ни вызвал transfer(a, b) или transfer(b, a) —
    // локи всегда берутся в одном порядке: сначала меньший id
    Account first  = from.id < to.id ? from : to;
    Account second = from.id < to.id ? to : from;

    synchronized (first) {
        synchronized (second) {
            from.balance -= amount;
            to.balance += amount;
        }
    }
}
```

Проверено: два потока гоняют встречные переводы по 10 000 раз — программа детерминированно завершается, балансы сходятся. Тот же код без упорядочивания (каждый берёт «свой» счёт первым) виснет так же, как `DeadlockDemo`.

**Способ 2 — не ждать вечно: `tryLock` с таймаутом.** `synchronized` умеет ждать только бесконечно. Но у `ReentrantLock` из `java.util.concurrent` (подробно — блок 24) есть `tryLock(timeout)`: «попробуй взять лок, а не вышло за 50 мс — вернись с `false`». Это ломает условие «удержание с ожиданием»: не получил второй лок — отпусти первый и попробуй заново:

```java
if (first.tryLock(50, TimeUnit.MILLISECONDS)) {
    try {
        if (second.tryLock(50, TimeUnit.MILLISECONDS)) {
            try {
                // оба лока наши — работаем
            } finally { second.unlock(); }
        }
    } finally { first.unlock(); }
}
// не взяли оба — всё отпущено, внешний цикл повторит попытку
```

Я проверил встречный сценарий (`t1` берёт A→B, `t2` берёт B→A) — оба потока завершаются: тот, кто не успел, отступает и заходит снова. Цена — код заметно многословнее и появляется retry-логика.

**Способ 3 — сокращать вложенность.** Каждый вложенный захват — потенциальное ребро в графе ожидания. Чем меньше мест в коде, где поток держит один лок и берёт другой, тем меньше поверхность для дедлока. Часто вложенность можно убрать вовсе: скопировать данные под первым локом, отпустить его и только потом брать второй.

> [!tip] 💡 Из C#  
> Deadlock — платформенно-нейтральное явление: два вложенных `lock` в разном порядке в .NET виснут ровно так же, и лечится это тем же глобальным порядком захвата. `Monitor.TryEnter(obj, timeout)` — прямой аналог `tryLock`. Разница в диагностике: в .NET ты чаще смотришь дампы через Visual Studio / `dotnet-dump`, а в Java есть штатный thread dump с автодетектом дедлока — про него ниже.

#### 1.2. Livelock — динамическая взаимоблокировка

Deadlock — это тишина: потоки стоят в `BLOCKED` и не тратят CPU. Livelock коварнее: потоки **активны**, состояние `RUNNABLE`, процессор греется — но прогресса нет, потому что потоки бесконечно реагируют друг на друга.

Каноничная картинка, которая точно передаёт механику: двое идут навстречу в узком коридоре. Оба вежливо шагают влево — снова друг напротив друга. Оба вправо — то же самое. Каждый _действует_, каждый _реагирует на другого_ — и именно поэтому никто не проходит.

В коде livelock обычно возникает там, где потоки написаны «слишком вежливо»: обнаружил конфликт → отпустил ресурс → попробовал снова → и второй поток сделал ровно то же самое в тот же момент. Кстати, наш `tryLock`-пример из предыдущего раздела теоретически подвержен этому: если оба потока с идеально совпадающим ритмом берут первый лок, не получают второй, отпускают и повторяют — они могут кружиться долго. На практике это лечится случайной задержкой перед повтором (jitter): рассинхронизировал ритм — коридор разошёлся.

Забегая в блок 24: retry-петля CAS-операций (атомики) под высокой контенцией — родственное явление. Потоки не заблокированы, все крутятся в цикле «прочитал — попытался записать — не вышло, кто-то опередил — повторяю», и конкретному потоку может долго не везти. Система в целом движется (чьи-то CAS проходят), но это та же семья проблем: активность без гарантии прогресса для каждого.

#### 1.3. Starvation — голодание

Третья патология — самая тихая. Поток не заблокирован намертво и не крутится впустую: он просто **годами не получает ресурс, потому что его вечно опережают**. Типичные причины:

- **Несправедливый лок.** `synchronized` не гарантирует очередь: когда монитор освобождается, его получает _кто-то_ из ждущих — не обязательно тот, кто ждёт дольше всех. При высокой конкуренции «свежие» потоки могут раз за разом перехватывать лок у давно ждущего (barging).
- **Жадные соседи.** Потоки, которые захватывают лок надолго и часто, оставляют остальным лишь редкие щели.
- **Приоритеты.** Потоки с низким приоритетом на загруженной машине могут почти не получать процессорное время (в современных ОС это редко бывает решающим фактором, но механика существует).

Забегая вперёд: в блоке 24 мы встретим справедливые локи — `new ReentrantLock(true)`. Fair-режим выстраивает потоки в честную очередь FIFO: кто дольше ждёт, тот следующий. Это прямой ответ на starvation — но ценой пропускной способности: поддержание очереди и принудительная передача лока «по списку» заметно дороже, чем отдать монитор тому, кто уже на процессоре. Поэтому по умолчанию все локи в Java несправедливые, и это осознанный выбор.

#### 1.4. Диагностика: thread dump

Всё вышеописанное объединяет неприятное свойство: **сервис не падает**. Нет исключения, нет stack trace в логах — просто запросы перестают отвечать. Единственный способ увидеть, что происходит внутри, — снять thread dump.

**Thread dump** — это мгновенный снимок всех потоков JVM: имя, состояние (те самые `RUNNABLE` / `BLOCKED` / `WAITING` из урока 2), полный стек вызовов и — критически важно — информация о том, какие мониторы поток держит и какой ждёт.

Как снять с живого процесса:

```bash
jps                    # найти PID Java-процесса
jstack <pid>           # классика; -l добавит информацию о j.u.c-локах
jcmd <pid> Thread.print   # современный универсальный способ
```

Ещё вариант — послать процессу `SIGQUIT` (`kill -3 <pid>` в Linux, `Ctrl+\` в терминале, где запущена программа): дамп напечатается в stdout процесса.

Самое ценное: **JVM сама детектит дедлоки на мониторах и печатает их в конце дампа**. Не нужно вручную сопоставлять стеки — вот реальный фрагмент дампа, снятый мной с зависшего `DeadlockDemo` (JDK 21):

Важная оговорка: детектор находит **только дедлоки**. Livelock и starvation в дампе выглядят как обычные работающие потоки в `RUNNABLE` — почему так и что с этим делать, разберём в блоке «Глубже».

### Часть 2. ThreadLocal — данные без разделения

Сменим угол зрения. Пять уроков подряд мы решали одну задачу: данные общие, доступ конкурентный — как защитить? Локи, `volatile`, happens-before. Но есть и другой путь: **сделать так, чтобы делить было нечего**. Если у каждого потока своя собственная копия данных, то нет разделения — нет гонок — не нужна синхронизация. Вообще. Именно это делает `ThreadLocal`.

**Ментальная модель:** `ThreadLocal<T>` — это переменная, у которой для каждого потока существует своё независимое значение. Один и тот же вызов `get()` вернёт разным потокам разные значения; `set()` меняет только копию текущего потока. Потоки друг друга не видят в принципе.

Механика на пальцах — и она важна для понимания ловушки дальше. Значения хранятся **не внутри объекта `ThreadLocal`**. У каждого объекта `Thread` есть собственная внутренняя карта — `ThreadLocalMap`. Когда поток вызывает `threadLocal.set(value)`, значение кладётся в карту _этого потока_, а ключом служит сам объект `ThreadLocal`. То есть `ThreadLocal` — это не хранилище, а _ключ_, по которому каждый поток находит своё значение в своём личном ящике:

![[Pasted image 20260705081819.png]]

#### 2.1. Зачем это нужно на практике

**Кейс 1 — не-потокобезопасный, но дорогой объект.** Классика жанра — `SimpleDateFormat`: он не потокобезопасен (мутирует внутреннее состояние при `format`/`parse`), но создавать его на каждый вызов дорого. Делить нельзя, создавать каждый раз жалко — компромисс: по экземпляру на поток:

```java
private static final ThreadLocal<SimpleDateFormat> FORMAT =
        ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

// в любом потоке:
String s = FORMAT.get().format(new Date()); // каждый поток форматирует своим экземпляром
```

`withInitial` задаёт фабрику: при первом `get()` в данном потоке значение создаётся лениво. (Для новых дат в современном коде правильный ответ — иммутабельный `DateTimeFormatter` из главы 16, которому `ThreadLocal` не нужен вовсе; но паттерн живёт в тоннах legacy и отлично иллюстрирует идею.)

**Кейс 2 — контекст запроса.** Сервер обрабатывает запрос в одном потоке, и по всей глубине стека вызовов нужно знать: кто пользователь, какой `trace-id`, в какой транзакции мы находимся. Таскать это параметром через каждый метод — мучение. `ThreadLocal` даёт «невидимый параметр»: положил в начале обработки — читаешь на любой глубине. Именно так устроены `MDC` в логировании (глава 21), контекст транзакции в Spring, propagation trace-id в трейсинге.

> [!tip] 💡 Из C#  
> `ThreadLocal<T>` в Java ≈ `ThreadLocal<T>` / `[ThreadStatic]` в .NET по смыслу «своё значение на поток». ⚠️ Но здесь живёт **ложный друг**: в асинхронном .NET для контекста «на логическую операцию» используют `AsyncLocal<T>`, который _протекает через `await`_ — значение следует за логическим потоком исполнения, даже когда продолжение выполняется на другом физическом потоке. В Java такого механизма у `ThreadLocal` нет: он привязан жёстко к **физическому потоку**. Перепрыгнул на другой поток (отдал задачу в пул, продолжил в колбэке `CompletableFuture`) — контекст остался в старом потоке. И вторая асимметрия: проблема утечки в пуле (ниже) в Java стоит острее — пулы потоков и переиспользование здесь повседневная норма, а не деталь рантайма.

#### 2.2. Ловушка: утечка в пуле потоков

Теперь кульминация урока — то, ради чего `ThreadLocal` вообще спрашивают на собеседованиях.

Всё, что мы говорили выше, молчаливо предполагало: поток живёт, пока идёт работа, и умирает вместе с ней — а с ним умирает и его `ThreadLocalMap`. Для потока, который ты создал руками под одну задачу, так и есть. Но в реальных приложениях потоки почти никогда не создаются под задачу — они живут в **пуле** (глава 2 блока 24): пул создаёт N потоков один раз, и они месяцами обслуживают тысячи разных задач по очереди.

И вот что происходит, если задача положила значение в `ThreadLocal` и не убрала за собой. Я воспроизвёл это живьём на пуле из одного потока:

```java
private static final ThreadLocal<String> currentUser = new ThreadLocal<>();

ExecutorService pool = Executors.newFixedThreadPool(1);

// Задача 1: кладёт значение и «забывает» убрать
pool.submit(() -> {
    currentUser.set("alice");
    System.out.println("Задача 1 работает как: " + currentUser.get());
}).get();

// Задача 2: ничего не клала — но что она увидит?
pool.submit(() -> {
    System.out.println("Задача 2 видит: " + currentUser.get());
}).get();
```

Вывод (JDK 21, проверено):

```
Задача 1 работает как: alice
Задача 2 видит: alice
```

Задача 2 ожидала `null` — а получила `alice`. Значение привязано не к задаче, а к **потоку** — а поток тот же самый, он не умер, его `ThreadLocalMap` никто не чистил. Отсюда две беды сразу:

1. **Протекание данных между запросами.** Следующая задача на том же потоке видит чужой контекст. В вебе это выглядит страшно буквально: запрос пользователя B выполняется «от имени» пользователя A, потому что аутентификационный контекст остался от прошлого запроса. Это уже не баг производительности — это инцидент безопасности.
2. **Утечка памяти.** Объект в `ThreadLocalMap` удерживается сильной ссылкой из потока. Поток пула живёт практически вечно → значение не соберётся GC никогда. Если значения тяжёлые (кеши, буферы, распарсенные документы), а `ThreadLocal`-ов много — heap медленно распухает без единого явного «утекающего» места в коде.

![[Pasted image 20260705081840.png]]

Есть и тонкость уровнем глубже, которая часто путает. Ключи в `ThreadLocalMap` — это `WeakReference` на объект `ThreadLocal`. Идея была защитная: если сам `ThreadLocal` больше нигде не используется и собран GC, ключ в карте обнулится. Но **слабый — только ключ; значение держится сильной ссылкой**. Получается запись «null → тяжёлый объект» (stale entry): достучаться до значения уже нельзя, а GC его забрать не может, пока живёт поток. `ThreadLocalMap` подчищает такие записи _попутно_ — при следующих `set`/`get`/`remove` на том же потоке, — но гарантий по времени нет: если поток больше не трогает `ThreadLocal`-ы, мусор висит бессрочно. Так что `WeakReference` смягчает проблему, но не решает её.

Отсюда правило, которое нужно вбить намертво:

> [!warning] Правило  
> **В пуле потоков — всегда `remove()` в `finally` после обработки.** Положил контекст — убери за собой, как бы задача ни завершилась: успехом или исключением.

```java
static String handleRequest(String userId) {
    try {
        currentUser.set(userId);
        return doWork();          // где-то в глубине читают currentUser.get()
    } finally {
        currentUser.remove();     // выполняется всегда — поток вернётся в пул чистым
    }
}
```

`remove()` физически удаляет запись из `ThreadLocalMap` текущего потока: и данные не протекут в следующую задачу, и ссылка на значение освобождена для GC. Именно этот `try/finally`-паттерн ты увидишь в исходниках любого приличного веб-фреймворка вокруг обработки запроса.

И честный взгляд вперёд: в блоке 24 нас ждут виртуальные потоки (Java 21), где потоков могут быть _миллионы_ — и по `ThreadLocalMap` с копиями данных на каждый из них выходит дорого по памяти, а модель «мутируемое значение на поток» плохо сочетается с их короткой жизнью. Язык движется к `ScopedValue` — неизменяемому контексту с явной областью действия (preview в Java 21, JEP 446; стабилизация — в более поздних релизах). Детали там; здесь достаточно знать направление: `ThreadLocal` остаётся рабочей лошадкой для platform-потоков и legacy, но не финальной точкой эволюции.

### 🔬 Глубже / на синьора

> [!example]- 🔬 Опциональный слой: диагностика, философы, устройство ThreadLocalMap  
> **Почему thread dump не видит livelock и starvation.** Детектор дедлоков строит граф «поток ждёт монитор, монитор удерживается потоком» и ищет в нём цикл. Ждущие потоки — это `BLOCKED` (мониторы) и парковка на j.u.c-локах; их отношения формализованы, цикл вычислим. А при livelock и starvation потоки в `RUNNABLE` — формально они _работают_, никакого «ждёт X» в дампе нет, цикла нет, детектировать нечего. Их ловят по симптомам: серия из нескольких дампов подряд (одни и те же потоки в одних и тех же retry-стеках = подозрение на livelock), метрики прогресса (throughput упал, CPU высокий) и профилировщик (JFR / async-profiler, глава 11) — где видно, что потоки жгут циклы в одном и том же узком месте.
> 
> **Dining philosophers.** Каноническая формулировка кругового ожидания (Дейкстра): пять философов за круглым столом, между соседями по одной вилке, есть можно только двумя. Каждый берёт левую вилку и ждёт правую → все держат по одной и ждут соседа → идеальное кольцо из пяти. Решения — те же наши: глобальный порядок (нумеруем вилки, каждый берёт сначала меньшую — один философ окажется «леворуким» и разорвёт цикл) или «не держать, если не можешь взять обе».
> 
> **Иерархия локов как дисциплина.** В большой системе «единый порядок захвата» превращают в архитектурное правило: локам присваивают уровни (например, лок кеша = 10, лок сущности = 20, лок I/O = 30), и захватывать разрешено только в порядке возрастания уровня. Нарушение ловится в тестах assertion-ом или инструментами статического анализа. Это масштабируемая версия трюка с `id` счетов.
> 
> **Устройство `ThreadLocalMap`.** Это не `HashMap`: специализированная хеш-таблица с **открытой адресацией** (linear probing) — при коллизии запись кладётся в следующий свободный слот, никаких списков/деревьев в бакетах. Ключи — `WeakReference<ThreadLocal<?>>`. Stale entry возникает так: GC собрал `ThreadLocal`, ключ в слоте стал `null`, а значение осталось. Методы `set`/`get`/`remove` при проходе по слотам (`expungeStaleEntry`) попутно вычищают такие записи и уплотняют цепочку проб — но только на тех индексах, мимо которых прошли. Отсюда и «может висеть бессрочно»: нет обращений — нет уборки.
> 
> **`InheritableThreadLocal`.** Подвид, который при _создании_ дочернего потока копирует в него значения родителя. Звучит удобно для передачи контекста — но в пуле это мина: потоки пула создаются один раз при старте пула, и «наследуют» они контекст того потока, который пул создавал, а вовсе не того, кто задачу отправил. Плюс копирование происходит ровно однажды — обновления родителя дочерний поток не увидит. В пулах — не использовать.
> 
> **`ScopedValue` (JEP 446, preview в Java 21).** Ответ языка на обе боли `ThreadLocal`: значение **иммутабельно** и живёт строго внутри лексической области — `ScopedValue.where(USER, "alice").run(() -> ...)`; по выходе из `run` значение исчезает автоматически, «забыть remove()» невозможно конструктивно. Спроектирован под миллионы виртуальных потоков и структурированную конкурентность. Пока preview — в проде на 21 не используется, но траекторию задаёт именно он.

### ⚠️ Грабли

> [!warning] Типичные грабли этого урока
> 
> 1. **Вложенные локи в разном порядке в разных местах кода.** Метод `a()` берёт лок X, потом Y; метод `b()` — Y, потом X. По отдельности оба корректны; вместе — дедлок, который выстрелит под нагрузкой. Лечение: единый глобальный порядок захвата, зафиксированный как правило проекта.
> 2. **«На моей машине не дедлочит».** Дедлок тайминг-зависим, как гонки из урока 3: нужен неудачный интерлив, чтобы оба потока успели взять по первому локу. Локально с двумя запросами в минуту его можно не увидеть годами — а под продовой нагрузкой он математически неизбежен, если кольцо возможно. Отсутствие симптомов ≠ отсутствие бага.
> 3. **`ThreadLocal` в пуле без `remove()`.** Двойной урон: утечка памяти (значение живёт, пока жив поток) и протекание данных между задачами (следующий запрос видит чужой контекст — привет, инцидент безопасности). Всегда `try { set } finally { remove }`.
> 4. **`notify` вместо `notifyAll` (урок 5) как ложный дедлок.** Симптом идентичен — всё висит; но в дампе не будет «Found deadlock»: потоки стоят в `WAITING` на `wait()`, а не в `BLOCKED` на мониторе. Проснуться некому — сигнал ушёл не тому. Прежде чем искать кольцо локов, проверь состояния застрявших потоков.
> 5. **`InheritableThreadLocal` в пуле.** Наследование срабатывает при создании потока — а потоки пула созданы давно и не тем, кем ты думаешь. Контекст будет либо чужим, либо устаревшим.
> 6. **Вера, что livelock/starvation видны в дампе как «deadlock».** Нет: там потоки `RUNNABLE`, автодетект молчит. Диагностика — серия дампов, метрики прогресса, профилировщик.

### На собесе

> [!question]- Что такое deadlock и при каких условиях он возможен?  
> **RU:** Deadlock — взаимная блокировка: потоки держат ресурсы и ждут ресурсы друг друга по замкнутому циклу, поэтому никто никогда не продвинется. Возможен при одновременном выполнении четырёх условий Коффмана: взаимное исключение, удержание с ожиданием, отсутствие вытеснения, круговое ожидание. Первые три — свойства самих локов, поэтому на практике борются с четвёртым: не дать циклу ожидания замкнуться.  
> **EN:** A deadlock is a state where threads hold resources while waiting for each other's resources in a closed cycle, so none can ever proceed. It requires all four Coffman conditions simultaneously: mutual exclusion, hold-and-wait, no preemption, and circular wait. The first three are inherent to locks, so in practice prevention targets the fourth — making a wait cycle impossible.

> [!question]- Как предотвратить deadlock?  
> **RU:** Главный приём — единый глобальный порядок захвата локов: если все потоки берут локи в одном порядке (например, по возрастанию id сущности), круговое ожидание невозможно. Второй приём — `tryLock` с таймаутом из `ReentrantLock`: не получил второй лок — отпусти первый и повтори, то есть не «удерживай с ожиданием». Третий — минимизировать вложенные захваты как таковые.  
> **EN:** The primary technique is a single global lock-acquisition order: if every thread takes locks in the same order (e.g., by entity id), a circular wait cannot form. Second, `ReentrantLock.tryLock` with a timeout — if the second lock isn't acquired, release the first and retry, breaking hold-and-wait. Third, minimize nested locking altogether.

> [!question]- Чем livelock отличается от deadlock?  
> **RU:** При deadlock потоки стоят: они в `BLOCKED`/`WAITING` и не тратят CPU. При livelock потоки активны — состояние `RUNNABLE`, CPU расходуется, — но они бесконечно реагируют на действия друг друга и не продвигаются: классика — оба «вежливо» отпускают ресурс и повторяют попытку синхронно. Поэтому livelock не виден автодетектору дедлоков; лечится рассинхронизацией повторов (случайный jitter).  
> **EN:** In a deadlock threads are stopped — `BLOCKED`/`WAITING`, no CPU used. In a livelock threads are active (`RUNNABLE`, burning CPU) but keep reacting to each other without making progress — e.g., both politely back off and retry in lockstep. That's why the JVM deadlock detector can't see it; the fix is desynchronizing retries with random jitter.

> [!question]- Что такое starvation и как с ней борются?  
> **RU:** Starvation — поток долго не получает ресурс, потому что его постоянно опережают: несправедливый лок отдаёт монитор не самому долго ждущему, а «кому-то», и агрессивные соседи перехватывают его снова и снова. Ответ — справедливые локи: `new ReentrantLock(true)` строит FIFO-очередь ожидания, гарантируя продвижение каждому. Цена — заметная потеря пропускной способности, поэтому по умолчанию локи несправедливые.  
> **EN:** Starvation is when a thread can't obtain a resource for a long time because others keep overtaking it: an unfair lock hands the monitor to _some_ waiter, not the longest-waiting one. The remedy is fair locks — `new ReentrantLock(true)` maintains a FIFO wait queue, guaranteeing progress for everyone, at a significant throughput cost, which is why locks are unfair by default.

> [!question]- Как обнаружить deadlock в проде?  
> **RU:** Снять thread dump: `jstack <pid>` (с `-l` для j.u.c-локов), `jcmd <pid> Thread.print` или `kill -3`. JVM сама детектит дедлоки на мониторах и печатает блок `Found one Java-level deadlock` — с указанием, какой поток какой монитор держит и какого ждёт, вплоть до строк исходника. Застрявшие потоки видны в состоянии `BLOCKED`. Livelock/starvation так не находятся — там нужна серия дампов и профилировщик.  
> **EN:** Take a thread dump: `jstack <pid>` (`-l` for j.u.c locks), `jcmd <pid> Thread.print`, or `kill -3`. The JVM auto-detects monitor deadlocks and prints a `Found one Java-level deadlock` section showing exactly which thread holds which monitor and waits for which, down to source lines. Stuck threads show as `BLOCKED`. Livelock/starvation aren't detected this way — you need repeated dumps and a profiler.

> [!question]- Зачем нужен ThreadLocal и как в нём возникает утечка памяти?  
> **RU:** `ThreadLocal` даёт каждому потоку собственное независимое значение — разделяемых данных нет, синхронизация не нужна; типичные применения — не-потокобезопасные дорогие объекты и контекст запроса (user, trace-id). Значения лежат в `ThreadLocalMap` внутри каждого `Thread`, ключ — сам `ThreadLocal`. Утечка возникает в пуле: поток живёт практически вечно, и если задача не вызвала `remove()`, значение удерживается потоком и не собирается GC, а следующая задача на этом потоке видит чужие данные.  
> **EN:** `ThreadLocal` gives each thread its own independent value — no shared state, no synchronization needed; typical uses are expensive non-thread-safe objects and per-request context (user, trace id). Values live in each `Thread`'s internal `ThreadLocalMap`, keyed by the `ThreadLocal` itself. The leak happens in thread pools: pool threads live practically forever, so if a task doesn't call `remove()`, the value stays strongly reachable from the thread (never GC'd), and the next task on that thread sees stale, foreign data.

> [!question]- Почему remove() обязателен именно в finally? И причём тут WeakReference?  
> **RU:** `finally` гарантирует очистку при любом исходе задачи — включая исключение; иначе одна упавшая задача оставляет поток «грязным» навсегда. На `WeakReference` полагаться нельзя: в `ThreadLocalMap` слабый только _ключ_, значение держится сильной ссылкой — если сам `ThreadLocal` собран, остаётся stale entry «null → значение», которую карта чистит лишь попутно при следующих операциях, без гарантий по времени. Явный `remove()` — единственный детерминированный способ.  
> **EN:** `finally` guarantees cleanup on every task outcome, including exceptions — otherwise one failed task leaves the thread "dirty" forever. Relying on `WeakReference` doesn't work: in `ThreadLocalMap` only the _key_ is weak, the value is held by a strong reference — if the `ThreadLocal` itself gets collected, a stale "null → value" entry remains, cleaned only opportunistically on later map operations with no timing guarantee. An explicit `remove()` is the only deterministic cleanup.
