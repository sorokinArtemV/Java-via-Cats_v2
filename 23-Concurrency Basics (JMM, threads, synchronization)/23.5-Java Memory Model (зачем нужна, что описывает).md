В уроках 1–2 мы научились управлять потоками снаружи: создавать, запускать через `start()`, дожидаться через `join()`, корректно прерывать. Потоки при этом делят одну кучу — любой объект, до которого дотягиваются двое, становится общим. И вот здесь начинается самое интересное. Пока поток был один, память вела себя как честный блокнот: записал — прочитал то, что записал. Как только потоков становится два, этот блокнот начинает жить своей жизнью: записи одного потока могут не доходить до другого, а порядок событий — расходиться с порядком строк в коде. Этот урок — про правила, по которым память _на самом деле_ ведёт себя в многопоточном мире, и про единственную гарантию, которая у тебя есть.

Интуитивная часть опасности всем понятна: два потока чередуются, и их действия перемешиваются. Неинтуитивная — в том, что **сама память не обязана показывать одному потоку то, что записал другой**. Начнём с неё.

### Проблема видимости: флажок, который никто не увидел

Классический сценарий: рабочий поток крутит цикл, пока флажок поднят, а `main` в какой-то момент его опускает. Наивное ожидание — worker увидит `false` и выйдет.

```java
public class VisibilityDemo {
    static boolean running = true;   // обычное поле, без volatile

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            long iterations = 0;
            while (running) {        // наивно: «выйдем, когда main поменяет флаг»
                iterations++;
            }
            System.out.println("worker вышел после " + iterations + " итераций");
        });
        worker.start();

        Thread.sleep(1000);          // даём worker'у раскрутиться
        running = false;             // main записал false...
        System.out.println("main: записал running = false");

        worker.join(3000);           // ждём выхода максимум 3 секунды
        System.out.println("worker.isAlive() = " + worker.isAlive());
        System.exit(0);             // зависший worker иначе не даст JVM завершиться
    }
}
```

> [!example] Реальный запуск (JDK 21, HotSpot)
> 
> ```
> main: записал running = false
> worker.isAlive() = true
> ```
> 
> Worker **не вышел**. `main` честно записал `false`, подождал три секунды — а worker так и крутится, и крутился бы вечно. Строка `worker вышел после ...` не напечаталась вообще. Стоит добавить `volatile` к полю — и тот же код завершается мгновенно: `worker вышел после 2091474552 итераций`, `isAlive() = false`.

Поток `main` _записал_ значение. Поток worker его _не увидел_. Чтобы это перестало казаться мистикой, нужно понять две вещи о том, как исполняется твой код на самом деле.

#### Причина 1: у каждого ядра — своя записная книжка

Поток A и поток B почти наверняка работают на разных ядрах процессора. У каждого ядра есть свой быстрый локальный кэш, и работает ядро в первую очередь с ним — ходить в основную память на каждое чтение слишком дорого. Представь двух работников: у каждого — личная записная книжка, а общая доска в коридоре обновляется «когда-нибудь потом». Поток B записал `flag = true` — но записал _к себе в книжку_. Дойдёт ли эта запись до доски, а с доски — до книжки потока A, и когда именно, — без синхронизации не гарантируется ничем.

![[Pasted image 20260704152935.png]]

#### Причина 2: компилятор имеет право переписать твой цикл

Вторая причина ещё коварнее, потому что она не про железо, а про сам код. JIT-компилятор смотрит на горячий цикл:

```java
while (running) {
    iterations++;
}
```

Внутри цикла поле `running` не меняется. С точки зрения _одного потока_ его значение — константа. Значит, компилятор вправе прочитать поле один раз перед циклом и дальше крутить закешированное значение — фактически превратив код в:

```java
if (running) {
    while (true) {   // чтение поля вынесено из цикла
        iterations++;
    }
}
```

Для однопоточной программы такая оптимизация абсолютно незаметна и абсолютно законна. Именно она чаще всего и убивает наш пример: worker после прогрева перестаёт _читать_ поле вообще — и уже неважно, что там в кэшах.

И вот ключевая мысль урока, произнесём её прямо: **это не баг JVM, а разрешённое поведение.** Компилятор, JIT и процессор имеют право кешировать значения и переупорядочивать операции, пока результат неотличим _в пределах одного потока_. Твои ожидания «другой поток же увидит» в этот контракт не входят. Без явной синхронизации у тебя нет гарантии, что один поток увидит записи другого, — есть только везение.

### Зачем нужна Java Memory Model

Если оптимизации законны и полезны (без них Java была бы в разы медленнее), а многопоточный код всё-таки писать надо — нужен договор о границах. Этим договором и является **Java Memory Model (JMM)** — часть спецификации языка (JLS, глава 17).

**Ментальная модель:** JMM — это контракт между твоим кодом и платформой, который отвечает на один вопрос: _при каких условиях запись одного потока гарантированно видна другому_. Внутри этих условий — железные гарантии на любой JVM и любом процессоре; вне их — никаких обещаний.

Важно понять, чем JMM является и чем нет:

- **Описывает** — видимость и порядок операций с общей памятью между потоками. Именно благодаря ей многопоточный код на Java переносим: ты пишешь под контракт, а не под конкретный процессор.
- **Не описывает** — планирование потоков, скорость, справедливость. И главное: она **не обещает никакого порядка там, где ты его явно не установил**. По умолчанию порядка между потоками нет вообще.

Осталось понять, как этот порядок устанавливается. Инструмент один, и он центральный для всего курса.

### Happens-before: единственная гарантия, которая у тебя есть

Начнём с интуиции, формальности оставим спеке. **Happens-before** — это отношение между двумя действиями, которое даёт гарантию видимости: если действие A _happens-before_ действия B, то B гарантированно видит всё, что сделал A, — и не только само A, а всё, что было до A в его потоке. Нет такой связи — нет никаких гарантий: B может увидеть запись A, а может не увидеть, даже если по часам A случилось «давно».

Здесь легче всего ошибиться, поэтому подчеркнём: **happens-before — не про физическое время.** Это не «A выполнилось раньше по секундомеру», а «между A и B установлена гарантия видимости». Поток `main` может записать `flag = false` за три секунды до того, как worker в очередной раз прочитает поле, — и worker всё равно увидит старое значение, потому что _гарантии_ между этими действиями никто не построил. Наш зависший `VisibilityDemo` — ровно это.

Откуда берутся happens-before-связи? Их источники перечислены в JLS §17.4.5, вот главные (пока обзорно — инструменты разберём в следующих уроках):

- **Program order.** Внутри одного потока всё связано happens-before в порядке кода. Поэтому в однопоточном мире проблем и не было.
- **Мониторы.** Освобождение монитора (`synchronized`-блок закончился) happens-before последующего захвата _того же_ монитора другим потоком.
- **`volatile`.** Запись в `volatile`-поле happens-before последующего чтения _этого же_ поля.
- **Жизненный цикл потока.** `Thread.start()` happens-before любого действия запущенного потока. И зеркально: всё, что поток сделал, happens-before возврата из `join()` по нему. Вспомни урок 1 — вот почему после `t.join()` результаты, посчитанные потоком `t`, можно спокойно читать: гарантия уже встроена.
- **Транзитивность.** Если A hb B и B hb C, то A hb C. Скучное на вид правило — но именно оно превращает отдельные гарантии в цепочки, на которых держится всё.

Посмотрим на цепочку в деле. Хотим передать результат из одного потока в другой через флаг готовности:

java

```java
public class Publish {
    static int data = 0;                    // обычное поле, БЕЗ volatile
    static volatile boolean ready = false;  // volatile-флаг публикации

    public static void main(String[] args) throws InterruptedException {
        Thread reader = new Thread(() -> {
            while (!ready) { }                        // (3) ждём volatile-флага
            System.out.println("data = " + data);     // (4) гарантированно 42
        });
        reader.start();

        data = 42;       // (1) обычная запись
        ready = true;    // (2) volatile-запись: публикуем
        reader.join();
    }
}
```

> [!example] Запуск (JDK 21) — стабильно, во всех прогонах
> 
> ```
> data = 42
> ```
> 
> Обрати внимание: `data` — обычное, не-`volatile` поле. Но reader всё равно _гарантированно_ видит `42`, а не `0`. Гарантию строит транзитивная цепочка: `(1) hb (2)` — program order в `main`; `(2) hb (3)` — правило volatile (запись → последующее чтение); `(3) hb (4)` — program order в reader. Итого `(1) hb (4)`: запись `data = 42` видна в точке чтения. Убери `volatile` у `ready` — и рассыпается всё: reader может зависнуть на шаге (3), как в первом примере, а может выйти и напечатать `data = 0`.

### Race condition vs data race: две разные болезни

Оба слова переводятся как «гонка», оба про многопоточность — и всё же это разные проблемы на разных уровнях. Начнём с примера, где присутствуют обе сразу:

```java
public class LostUpdates {
    static int counter = 0;

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 100_000; i++) {
                counter++;               // читай-изменяй-запиши: три шага, не один
            }
        };
        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter);   // ожидание: 200000
    }
}
```

> [!example] Девять запусков подряд (JDK 21)
> 
> ```
> counter = 200000
> counter = 160416
> counter = 200000
> counter = 160416
> counter = 200000
> ...
> ```
> 
> Иногда ровно `200000`, иногда — заметно меньше. Один и тот же код, одни и те же данные, разный результат от запуска к запуску. Это и есть фирменный почерк гонки: не стабильная поломка, а **непредсказуемость**.

Теперь термины.

**Data race (гонка данных)** — понятие низкоуровневое, про память и JMM. Формально: два потока обращаются к одной переменной, хотя бы одно из обращений — запись, и эти обращения **не упорядочены happens-before**. По JLS такое поведение не определено: поток может прочитать устаревшее значение, «половину» значения, что угодно. В `counter++` data race есть: оба потока пишут в `counter` без какой-либо синхронизации. Лечение — установить happens-before: `volatile`, `synchronized`, атомики.

**Race condition (состояние гонки)** — понятие высокоуровневое, про логику программы. Это ситуация, когда **правильность результата зависит от тайминга** — от того, в каком порядке потокам повезло выполниться. В `counter++` она тоже есть: инкремент — это три шага (прочитай, прибавь, запиши), и если два потока прочитали одно и то же старое значение, одно из обновлений теряется. Классическая форма race condition — **check-then-act**: «проверил условие, потом сделал действие», а между проверкой и действием мир успел измениться.

Самое важное: **race condition может жить в коде, где data race полностью устранён.** Вот доказательство:

```java
import java.util.*;

public class CheckThenAct {
    static final List<String> registered =
            Collections.synchronizedList(new ArrayList<>());

    // Каждая операция по отдельности потокобезопасна (лок внутри),
    // но ПОСЛЕДОВАТЕЛЬНОСТЬ contains -> add не атомарна.
    static void registerOnce(String user) {
        if (!registered.contains(user)) {   // (1) проверка
            registered.add(user);           // (2) действие
        }
    }
    // ... два потока одновременно вызывают registerOnce("alice")
}
```

`synchronizedList` берёт лок на каждую операцию — data race нет ни на одном обращении, каждая happens-before-связь на месте. Но между `contains` и `add` лок отпускается. Оба потока успевают выполнить проверку (`alice` ещё нет), оба проходят внутрь — и `alice` регистрируется дважды.

> [!example] Насколько это воспроизводимо  
> Мы прогнали этот сценарий 1000 раз в цикле: в одной серии дубликат появился **1 раз из 1000**, в следующей — **0 раз из 1000**. Вот почему такие баги — самые коварные: тесты зелёные, на твоей машине «всё работает», а в проде под нагрузкой раз в неделю появляется лишняя запись. Отсутствие гарантии — это не немедленная поломка, это непредсказуемость.

Бывает и обратное — data race без race condition: формальное нарушение JMM, при котором логика случайно не страдает (редкий зверь, пример — в диаграмме и в 🔬-блоке ниже). Итог, который стоит произнести на собеседовании дословно: **это пересекающиеся, но разные проблемы, и лечатся они по-разному.** Устранил data race — не значит устранил race condition. Забегая в блок 24: `AtomicInteger` уберёт data race на счётчике, но составной инвариант «проверь и добавь» всё равно потребует лока вокруг _всей_ операции.

На Java-собеседовании отвечать в терминах C#-интуиции про fences — слабый ответ; отвечай в терминах happens-before. Второй: Java-`volatile` даёт полноценную happens-before-гарантию (запись → последующее чтение, со всей транзитивной цепочкой — разберём детально в следующем уроке), тогда как семантику C#-`volatile` в сообществе часто трактуют вольно и с оговорками. Не переноси её сюда один в один — выучи Java-версию как новую.

### 🔬 Глубже / на синьора

Скипабельный блок — на понимание урока не влияет, на собеседовании уровня senior всплывает.

**Формализм.** JMM определена в JLS §17.4. Happens-before строится из _program order_ внутри каждого потока и _synchronization order_ — глобального порядка всех синхронизирующих действий (volatile-доступы, захваты/освобождения мониторов, start/join). Data race формально — конфликтующие обращения, не упорядоченные happens-before, и для программ с data race спека даёт очень слабые обещания.

**Три уровня переупорядочивания.** Порядок твоих операций могут менять: (1) статический компилятор `javac` — почти никогда, он транслирует наивно; (2) JIT — агрессивно: hoisting чтений из циклов, слияние записей, перестановки; (3) процессор — store buffers, out-of-order execution. `volatile` и `synchronized` под капотом компилируются в инструкции с семантикой _барьеров памяти_ (memory fences), которые запрещают конкретные перестановки и форсируют видимость. Какие именно барьеры — зависит от архитектуры: x86 сама по себе даёт довольно строгий порядок (дорогая только `StoreLoad`), ARM — слабый, барьеров нужно больше. Именно поэтому «у меня на ноутбуке работает» ничего не доказывает: тот же код на ARM-сервере ломается чаще.

**Честная поправка к диаграмме с кэшами.** Кэши современных CPU когерентны (протоколы типа MESI): «навсегда потерянная в чужом кэше» запись — упрощение. На практике вечный цикл в `VisibilityDemo` делает в первую очередь JIT-hoisting, а железо добавляет _задержки_ и _переупорядочивание_ через store buffers. Но JMM специально сформулирована так, чтобы разрешать и самое слабое железо: правильная ментальная модель — «считай, что у каждого потока своя копия памяти, пока happens-before не заставит их сверить часы».

**Sequential consistency — то, чего у нас нет.** SC — модель, где все операции всех потоков выстраиваются в один глобальный порядок, согласованный с порядком программ. Интуиция человека по умолчанию ждёт именно SC. Java её _не даёт_, потому что полный SC запретил бы большинство оптимизаций компилятора и процессора. Вместо этого JMM даёт знаменитую сделку **SC-DRF**: если в программе нет data race, она _ведёт себя_ sequentially consistent. Пиши код без гонок — и можешь рассуждать о нём наивно; допустил гонку — все ставки сняты.

**Final и safe publication.** Отдельная часть JMM (JLS §17.5): если объект правильно сконструирован (ссылка `this` не утекла из конструктора), то значения его `final`-полей гарантированно видны любому потоку, получившему ссылку на объект, — даже без синхронизации. На этом стоит безопасная публикация immutable-объектов; подробно вернёмся при разборе `String` и immutability-паттернов.

**Long/double половинками.** JLS §17.7: для не-`volatile` полей типа `long` и `double` JVM формально разрешено выполнять запись как две 32-битные операции — конкурирующий читатель может увидеть «склейку» половин двух разных значений. На современных 64-битных HotSpot этого на практике не происходит, но спека всё ещё это допускает; `volatile` гарантирует атомарность чтения/записи таких полей. Не путай с _word tearing_ (JLS §17.6) — порчей соседних переменных при записи, которую Java как раз категорически запрещает.

**Бенигнные гонки.** Тот самый пример data race без race condition: `String.hashCode()` кеширует хеш в обычном не-`volatile` поле `hash`. Два потока могут посчитать и записать его одновременно — формально data race. Но оба запишут _одно и то же_ значение, запись `int` атомарна, а прочитавший `0` просто пересчитает. Идиома называется _racy single-check_ — она корректна только при очень специфических условиях и требует экспертизы; в прикладном коде так делать не надо.

### ⚠️ Грабли

> [!warning] Подводные камни — коллекция урока
> 
> 1. **`while (!flag) { }` без `volatile`/синхронизации** — возможный вечный цикл. Запись другого потока не обязана стать видимой: JIT вправе вынести чтение из цикла. Проверено запуском выше — worker не вышел никогда.
> 2. **«По времени раньше — значит, увидит».** Нет. Видимость даёт только happens-before-связь, физическое время не даёт ничего. `main` записал `false` за три секунды до очередного чтения — worker всё равно читает старьё.
> 3. **Смешивание data race и race condition в одно слово «гонка».** Это разные уровни: память/JMM против логики/инвариантов. На собеседовании неразличение — маркер поверхностности.
> 4. **«Убрал data race — значит, потокобезопасно».** `synchronizedList` устранил все data race, а `alice` всё равно регистрируется дважды: check-then-act остался. Атомарность отдельных операций не даёт атомарности их последовательности.
> 5. **«На моей машине же работает».** Отсутствие гарантии ≠ немедленная поломка; оно означает непредсказуемость. Наш дубликат воспроизвёлся 1 раз на 1000 прогонов — такой баг переживёт все тесты и выстрелит в проде. И чаще стреляет на ARM, где порядок памяти слабее, чем на x86.
> 6. **Публикация данных через обычное поле-флаг.** Читатель может увидеть флаг, но не увидеть данные, записанные _до_ флага (переупорядочивание). Публикуй через happens-before: `volatile`-флаг, `synchronized`, `join()` — или immutable-объект с `final`-полями.

### На собесе

> [!question]- Зачем нужна Java Memory Model и что она гарантирует? _(спрашивают почти всегда, любой грейд)_  
> **Follow-up-ловушка:** «А без неё что, код бы не работал?» — ждут понимания, что JMM это контракт переносимости: без неё каждый процессор/компилятор оптимизировал бы по-своему, и корректный многопоточный код был бы непереносим.  
> **Слабый ответ:** «JMM описывает, как устроена память JVM — heap, stack…» — это структура памяти (глава про JVM), а не модель памяти. Классическая подмена, интервьюеры её ловят специально.  
> **Эталон (RU):** JMM — часть спецификации языка (JLS §17), определяющая, при каких условиях запись одного потока гарантированно видна другому и какие переупорядочивания допустимы. Компилятор, JIT и процессор вправе кешировать и переставлять операции, пока это незаметно в пределах одного потока; JMM задаёт границы этой свободы через отношение happens-before. Внутри правил — гарантии на любой платформе, вне — поведение не определено.  
> **Model answer (EN):** The JMM is the part of the language spec (JLS §17) that defines when a write made by one thread is guaranteed to be visible to another, and which reorderings are allowed. Compilers, the JIT and the CPU may cache and reorder operations as long as single-threaded semantics are preserved; the JMM bounds that freedom via the happens-before relation. Within its rules you get portable guarantees; outside them, behavior is undefined.

> [!question]- Что такое happens-before и почему это не про физическое время? _(частый, mid+)_  
> **Follow-up-ловушка:** «Если запись случилась на секунду раньше чтения — читатель её увидит?» Правильный ответ — не обязательно: без hb-связи гарантий нет, сколько бы времени ни прошло.  
> **Слабый ответ:** «Это когда одна операция выполняется раньше другой» — ровно та ошибка, которую проверяют.  
> **Эталон (RU):** Happens-before — отношение между действиями, дающее гарантию видимости и порядка: если A hb B, то B видит все эффекты A и всего, что предшествовало A в его потоке. Это про установленную гарантию, а не про часы: действие может физически произойти раньше, но без hb-связи другой поток вправе его не увидеть. Отношение транзитивно, что позволяет строить цепочки гарантий.  
> **Model answer (EN):** Happens-before is a relation between actions that provides visibility and ordering guarantees: if A happens-before B, then B is guaranteed to see all effects of A and everything preceding A in its thread. It's about established guarantees, not wall-clock time — an action may physically occur earlier, yet without a happens-before edge another thread may legally never observe it. The relation is transitive, which lets you chain guarantees.

> [!question]- Какие источники happens-before ты знаешь? _(частый, mid+)_  
> **Follow-up-ловушка:** «Почему после `t.join()` можно читать результаты потока `t` без синхронизации?» — потому что завершение потока hb возврата из `join()`; это встроенная гарантия.  
> **Слабый ответ:** назвать только `volatile` и `synchronized`, забыв program order, start/join и транзитивность — тогда не объяснить, почему в volatile-публикации видно и _не-volatile_ поле `data`.  
> **Эталон (RU):** Program order внутри потока; освобождение монитора hb последующего захвата того же монитора; запись в volatile-поле hb последующего чтения этого поля; `Thread.start()` hb всех действий потока; все действия потока hb возврата из `join()`. Плюс транзитивность — именно она делает видимыми обычные записи, сделанные до volatile-записи («piggyback»).  
> **Model answer (EN):** Program order within a thread; a monitor unlock happens-before every subsequent lock of the same monitor; a volatile write happens-before every subsequent read of that field; `Thread.start()` happens-before every action of the started thread; all actions of a thread happen-before `join()` on it returns. Plus transitivity — which is what makes plain writes performed before a volatile write visible after the matching volatile read (piggybacking).

> [!question]- Чем data race отличается от race condition? Может ли быть одно без другого? _(любимый разводящий вопрос, mid+/senior)_  
> **Follow-up-ловушка:** «Я сделал счётчик `AtomicInteger` — код теперь потокобезопасен?» Data race исчез, но составные операции (check-then-act поверх счётчика) всё ещё могут содержать race condition.  
> **Слабый ответ:** «Это одно и то же, просто термины разные» или определение обоих как «когда потоки одновременно лезут в переменную».  
> **Эталон (RU):** Data race — низкоуровневое понятие из JMM: конфликтующие обращения к одной переменной (хотя бы одно — запись) без happens-before; поведение не определено, лечится синхронизацией. Race condition — высокоуровневое: корректность зависит от порядка выполнения потоков, классика — check-then-act; лечится атомарностью всей составной операции. Одно без другого бывает в обе стороны: check-then-act над `synchronizedList` — race condition без единого data race; кеш `hashCode` в `String` — формальный data race без race condition.  
> **Model answer (EN):** A data race is a low-level JMM concept: conflicting accesses to the same variable (at least one write) not ordered by happens-before; behavior is undefined and the fix is synchronization. A race condition is high-level: correctness depends on thread timing — check-then-act being the classic shape — and the fix is making the whole compound operation atomic. Each can exist without the other: check-then-act over a `synchronizedList` is a race condition with zero data races, while `String`'s hashCode caching is a formal data race with no race condition.

> [!question]- Почему `while (!flag) {}` может крутиться вечно, если другой поток поставил `flag = true`? _(частый практический, mid)_  
> **Follow-up-ловушка:** «Это баг JVM?» — нет, это разрешённое поведение; оптимизации законны, пока незаметны одному потоку.  
> **Слабый ответ:** «Потоки просто медленно синхронизируются, надо подождать» — нет, гарантии увидеть запись не существует вовсе.  
> **Эталон (RU):** Поле не `volatile`, между записью и чтениями нет happens-before. JIT вправе вынести чтение поля из цикла (для одного потока значение не меняется) — цикл превращается в `while (true)`; на уровне железа запись может задерживаться в store buffer/кэше ядра. Это не баг, а контракт JMM: без синхронизации видимость не гарантирована. Лечение — `volatile boolean flag`, создающий hb-связь запись→чтение.  
> **Model answer (EN):** The field isn't `volatile`, so there's no happens-before between the write and the reads. The JIT may legally hoist the field read out of the loop — from a single thread's perspective the value never changes — turning it into `while (true)`; at the hardware level the write may also linger in a store buffer or cache. It's not a JVM bug but the JMM contract: without synchronization, visibility isn't guaranteed. The fix is `volatile boolean flag`, which establishes a write-to-read happens-before edge.
