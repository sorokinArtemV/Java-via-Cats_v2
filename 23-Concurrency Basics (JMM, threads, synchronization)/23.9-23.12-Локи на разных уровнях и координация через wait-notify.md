В уроке 4 мы выяснили главное про `synchronized`: он захватывает монитор _какого-то объекта_, и взаимное исключение работает только между потоками, которые синхронизируются на **одном и том же** мониторе. Из этого следует вещь, которую легко недооценить: выбор объекта-монитора — не косметика, а архитектурное решение с последствиями. Неудачно выбранный монитор открывает твою блокировку чужому коду или тихо разъезжается на два разных объекта. В этом уроке мы сначала разберём три типичных уровня, на которых можно лочиться, и выведем правило «какой лок правильный». А потом добавим второй слой: монитор умеет не только «не пускать», но и **координировать** потоки — усыплять их до наступления условия и будить, когда условие изменилось. Это `wait`/`notify`/`notifyAll` — фундамент, на котором построены все высокоуровневые инструменты блока 24.

**Ментальная модель урока:** лок должен быть приватным и финальным — тогда никто, кроме тебя, не может его захватить; а `wait` — это «отпусти монитор и усни в его комнате ожидания», поэтому после пробуждения условие всегда перепроверяется в цикле `while`.

### Часть 1. На чём синхронизироваться

#### `synchronized(this)`: монитор, который видят все

Когда ты пишешь `synchronized`-метод экземпляра, монитором становится `this` — сам объект. Это удобно и коротко, но у этого монитора есть неприятное свойство: **он публичный**. Любой код, у которого есть ссылка на твой объект, может написать `synchronized(obj)` и захватить _тот же самый_ монитор, на котором работают твои методы. Ты этого не разрешал, не проектировал и никак не можешь запретить.

Посмотри, как это выглядит вживую. Класс `Counter` честно защищает своё состояние `synchronized`-методами — а «чужой» код снаружи парализует его на три секунды:\
```java
class Counter {
    private int value;
    public synchronized void increment() { value++; }  // монитор: this
    public synchronized int get() { return value; }
}

public class HijackDemo {
    public static void main(String[] args) throws InterruptedException {
        Counter counter = new Counter();

        new Thread(() -> {
            synchronized (counter) {              // чужой код захватил ТВОЙ монитор
                System.out.println("Hijacker: держу монитор counter");
                try { Thread.sleep(3000); } catch (InterruptedException e) { }
            }
        }).start();

        Thread.sleep(100);
        System.out.println("Main: пытаюсь вызвать increment()...");
        long start = System.nanoTime();
        counter.increment();                      // висим, пока hijacker не отпустит
        System.out.println("Ждал " + (System.nanoTime() - start) / 1_000_000 + " мс");
    }
}
```

Проверено на JDK 21 — вывод:

```
Hijacker: держу монитор counter
Main: пытаюсь вызвать increment()...
Ждал 2899 мс
```

Никакой магии: `synchronized (counter)` снаружи и `synchronized`-метод внутри борются за один монитор. Внешний код получил контроль над твоей внутренней блокировкой — может задерживать её, может устроить deadlock, о котором ты узнаешь только в проде. Инкапсуляция состояния есть, а инкапсуляции _блокировки_ — нет.

#### `static synchronized`: монитор класса

Второй уровень — статический. `static synchronized`-метод (или явный `synchronized(Counter.class)`) захватывает монитор объекта `Class`, который один на класс во всей JVM (точнее, в пределах одного класслоадера — но об этом в главе 11). Он защищает **статическое** состояние — то, что общее для всех экземпляров.

И здесь важно повторить следствие из урока 4: монитор `this` и монитор `Counter.class` — это **два разных объекта**, а значит, два независимых лока. Инстансная и статическая синхронизация друг другу не мешают вообще:

```java
class Service {
    public synchronized void instanceMethod() {        // монитор: this
        System.out.println("instanceMethod: держу this 2 секунды");
        try { Thread.sleep(2000); } catch (InterruptedException e) { }
    }
    public static synchronized void staticMethod() {   // монитор: Service.class
        System.out.println("staticMethod: вошёл сразу");
    }
}
```

Запускаем: один поток висит внутри `instanceMethod()`, второй в это время вызывает `staticMethod()` — и проходит мгновенно:

```
instanceMethod: держу this 2 секунды
staticMethod: вошёл сразу
staticMethod waited ~0 ms
```

Если статический метод трогает те же данные, что и инстансный, а мониторы у них разные — взаимного исключения между ними **нет**, и это готовая гонка. У `Class`-монитора та же болезнь публичности, что у `this`, только хуже: ссылку на `Service.class` может получить вообще кто угодно.

#### `private final Object lock`: рекомендуемый способ

Решение обеих проблем — завести монитор, который снаружи _недостижим_:

```java
class Counter {
    private final Object lock = new Object();
    private int value;

    public void increment() {
        synchronized (lock) { value++; }
    }
    public int get() {
        synchronized (lock) { return value; }
    }
}
```

Логика простая: чтобы захватить монитор объекта, нужна ссылка на этот объект. На `lock` ссылки за пределами класса не существует — значит, захватить его чужим кодом **физически невозможно**. Ты полностью контролируешь свою блокировку: кто, когда и как долго её держит, определяется только твоим кодом.

Каждое слово в объявлении несёт смысл:

- `private` — ссылка не покидает класс. Никогда не возвращай `lock` из методов и не передавай его наружу — иначе весь смысл теряется.
- `final` — ссылку нельзя переприсвоить. Это не педантизм: если поле `lock` перезаписать, монитор «уедет» — часть потоков будет синхронизироваться на старом объекте, часть на новом, и взаимное исключение молча исчезнет. Это ровно те грабли «разные мониторы = нет синхронизации» из урока 4, только устроенные самому себе.
- `Object` — монитор есть у любого объекта, ничего специальнее не нужно. Пустой `new Object()` — самый дешёвый и честный носитель монитора.

И бонус, который открывается только с приватными локами: их может быть **несколько**. Если у класса две независимые части состояния — скажем, счётчик запросов и кэш конфигурации, — заведи `lockA` и `lockB`. Потоки, работающие с разными данными, перестанут стоять в одной очереди: разные данные — разные локи, меньше конкуренции. С `synchronized(this)` такое невозможно в принципе: монитор один на всё.

#### На чём лочиться нельзя

Правило «лок — приватный, финальный, ничей больше» имеет обратную сторону: список объектов, на которых синхронизироваться **запрещено**, потому что они не только твои.

**`String`-литералы.** Литералы интернируются: `"LOCK"` в твоём классе и `"LOCK"` в чужой библиотеке — это **один и тот же объект** из пула строк. Два «независимых» модуля, залочившиеся на одинаковый литерал, молча делят один монитор. Проверено:

```java
class ModuleA {
    static void doWork() {
        synchronized ("LOCK") {           // литерал интернирован
            try { Thread.sleep(2000); } catch (InterruptedException e) { }
        }
    }
}
class ModuleB {
    static void doWork() {
        synchronized ("LOCK") {           // тот же объект из пула строк!
            System.out.println("ModuleB entered");
        }
    }
}
```

`ModuleB`, который «никак не связан» с `ModuleA`, ждёт почти две секунды: `ModuleB waited ~1900 ms`. В реальном проекте такое ловится неделями — связь между модулями существует только на уровне пула строк.

**Боксированные примитивы.** `Integer` от −128 до 127 кешируется (урок про автобоксинг, 1.4): `Integer lock = 42` в двух разных классах — снова один объект и один общий монитор. Плюс автобоксинг тихо создаёт _новые_ объекты: залочился на `Integer`, потом «увеличил» его — и монитор уехал.

**Не-`final` поля и всё, что может переприсваиваться.** Причина та же, что и выше: сегодня потоки синхронизируются на одном объекте, завтра — на двух разных.

> [!warning] ⚠️ Правило  
> Монитор — это `private final Object lock = new Object()`. Не `this`, не `Class`, не строка, не боксированный примитив, не изменяемое поле. Лок принадлежит только тебе — тогда только ты решаешь, кто и когда его держит.

> [!tip] 💡 Из C#  
> Это дословно идиома `private readonly object _lock = new object();` + `lock (_lock)`. Совет «не лочиться на `this`, на типе (`typeof(...)`) и на строках» в .NET звучит теми же словами и по тем же причинам — здесь экосистемы полностью совпадают. Разница одна: в C# `lock` — единственный массовый низкоуровневый примитив, а в Java рядом с `synchronized` живёт целый слой `java.util.concurrent`, к которому мы придём в блоке 24.

### Часть 2. Координация: `wait` / `notify` / `notifyAll`

#### Новая задача: не «не мешать», а «дождаться»

Всё, что мы делали до сих пор, решало одну задачу: потоки не должны _мешать_ друг другу трогать общие данные. Но в реальных системах есть вторая, совсем другая задача: поток должен _дождаться условия_. Классика — consumer хочет взять элемент из буфера, а буфер пуст. Взять нечего. Что делать?

Наивный ответ — крутиться в цикле:

```java
while (buffer.isEmpty()) { }   // busy-wait: сжигаем ядро CPU впустую
takeItem();
```

Это называется busy-waiting, и это плохо дважды: во-первых, ядро процессора занято на 100% бессмысленной проверкой; во-вторых, между проверкой `isEmpty()` и `takeItem()` нет атомарности — другой consumer мог успеть забрать элемент. Нужен механизм «усни, пока условие не изменится, и пусть тебя разбудят». Этот механизм встроен прямо в монитор — тот самый, который мы весь урок выбираем.

#### Модель: у монитора есть комната ожидания

У каждого монитора, помимо очереди на вход, есть **wait set** — «комната ожидания». Три метода класса `Object` (да, они есть у _любого_ объекта — потому что монитор есть у любого объекта) работают с этой комнатой:

`wait()` — «отпусти монитор и усни в его комнате ожидания». Здесь спрятан важнейший нюанс, и его надо вбить намертво: **`wait` освобождает лок**. Поток засыпает, а монитор становится свободным — иначе никто не смог бы войти и изменить условие, которого мы ждём. Сравни со `sleep` из урока 1: `sleep` просто останавливает поток и **держит** все захваченные локи. Поток, заснувший в `sleep` внутри `synchronized`, блокирует всех; поток, заснувший в `wait`, — никого.

`notify()` — «разбуди один произвольный поток из комнаты ожидания». `notifyAll()` — «разбуди все». И второй нюанс: разбуженный поток **не бежит сразу выполняться**. Проснуться — не значит получить лок. Он выходит из комнаты ожидания и встаёт в общую борьбу за монитор; продолжит выполнение только когда снова его захватит. Между «его разбудили» и «он снова держит лок» может пройти время — и мир за это время может измениться. Это наблюдение станет ключом к главной ловушке через минуту.

Вот полный цикл на таймлайне — обрати внимание на два момента: `wait` освобождает лок (шаг 2), и после пробуждения условие проверяется заново (шаг 6):

#### Жёсткое требование: только владея монитором

`wait`, `notify` и `notifyAll` можно звать **только держа монитор того же объекта** — то есть только внутри `synchronized` на нём. Нарушишь — получишь `IllegalMonitorStateException` в рантайме:

```java
Object lock = new Object();
lock.wait();     // IllegalMonitorStateException: current thread is not owner
lock.notify();   // то же самое
```

Проверено на JDK 21 — оба вызова падают с `current thread is not owner`. Это не бюрократия спецификации, а защита от гонки. Смотри: чтобы решить «мне надо ждать», поток сначала _проверяет условие_ (`buffer.isEmpty()`), а потом _засыпает_ (`wait()`). Если бы это происходило без лока, между проверкой и засыпанием мог вклиниться producer: положить элемент, крикнуть `notify()` — в пустую комнату, где ещё никого нет, — и только потом consumer заснул бы. Навсегда: его сигнал уже прозвучал и пропал. Требование «проверка условия и `wait` — под одним локом» делает пару «проверил — заснул» атомарной относительно пары «изменил — разбудил». Сигнал потеряться не может.

#### Главная ловушка: `wait` всегда в цикле `while`, никогда в `if`

Интуиция подсказывает написать так:

```java
synchronized (lock) {
    if (items.isEmpty()) {      // ❌ скрытый баг
        lock.wait();
    }
    T item = items.removeFirst();   // а если items снова пуст?!
}
```

Логика кажется железной: «меня разбудили — значит, условие выполнилось». Но это неправда, по двум независимым причинам.

**Причина 1: spurious wakeups.** Поток может проснуться из `wait` _сам по себе_, без всякого `notify`. Это не баг JVM — это явно разрешено спецификацией (Javadoc `Object.wait` прямо предписывает вызывать `wait` в цикле именно поэтому; корни — в поведении примитивов ожидания на уровне ОС, futex/condition variables, где ложные пробуждения допустимы). Редко, но случается — и `if`-версия после такого пробуждения полезет в пустой буфер.

**Причина 2 — более важная, потому что срабатывает без всякой экзотики.** Вспомни нюанс из модели: разбуженный поток не продолжает мгновенно — он должен **снова захватить монитор**. Между моментом `notifyAll()` и моментом, когда наш consumer наконец получил лок, монитор мог достаться кому-то другому. Например, второму consumer'у, который не спал, вошёл в `take()` первым и забрал тот единственный элемент. Наш поток просыпается, получает лок — а условие _снова ложно_. `if` этого не заметит и упадёт; `while` спокойно проверит ещё раз и снова уснёт.

Поэтому единственная правильная форма — цикл:

java

```java
synchronized (lock) {
    while (items.isEmpty()) {   // ✅ перепроверяем после каждого пробуждения
        lock.wait();
    }
    T item = items.removeFirst();   // сюда попадаем, только когда условие точно истинно
}
```

Читается это так: «пока условие ложно — спи; проснулся — не верь на слово, проверь». `wait` внутри `while` гарантирует инвариант: код после цикла выполняется строго при истинном условии и под локом. Это классический вопрос на собеседовании, и отвечать на него надо обеими причинами, а не только spurious wakeups.

#### `notify` vs `notifyAll`

`notify()` будит **один произвольный** поток из wait set — какой именно, спецификация не определяет. Это дешевле: проснётся один, остальные продолжат спать, не будет толпы, дерущейся за лок. Но у дешевизны есть цена: если в комнате ожидания потоки ждут **разных условий**, `notify` может разбудить не того. Представь буфер, где в wait set одновременно спят producer'ы (ждут места) и consumer'ы (ждут элементов). Consumer забрал элемент, место освободилось, он зовёт `notify` — а JVM будит другого consumer'а. Тот проверяет `while (items.isEmpty())`, видит пустоту, засыпает обратно. Producer, для которого сигнал предназначался, не проснулся. Система замирает, хотя deadlock'а формально нет.

`notifyAll()` будит **всех**: каждый захватит лок по очереди, перепроверит своё условие в `while`, и те, кому не повезло, снова уснут. Лишние пробуждения — это накладные расходы, но это расходы на корректность.

Правило для этого уровня простое: **сомневаешься — `notifyAll`**. `notify` допустим как оптимизация только когда доказуемо выполняются оба условия: все ждущие потоки ждут _одно и то же_ условие, и одно уведомление способно «обслужить» не более одного из них.

#### Рабочий пример: bounded buffer

Соберём всё в классический producer–consumer с ограниченным буфером. Обрати внимание на все элементы урока сразу: приватный `final` лок, `wait` в `while` с обеих сторон, `notifyAll`:

```java
class BoundedBuffer<T> {
    private final Deque<T> items = new ArrayDeque<>();
    private final int capacity;
    private final Object lock = new Object();

    BoundedBuffer(int capacity) { this.capacity = capacity; }

    public void put(T item) throws InterruptedException {
        synchronized (lock) {
            while (items.size() == capacity) {   // ждём место — в цикле
                lock.wait();
            }
            items.addLast(item);
            lock.notifyAll();                    // будим ждущих consumer'ов
        }
    }

    public T take() throws InterruptedException {
        synchronized (lock) {
            while (items.isEmpty()) {            // ждём элемент — в цикле
                lock.wait();
            }
            T item = items.removeFirst();
            lock.notifyAll();                    // будим ждущих producer'ов
            return item;
        }
    }
}
```

Проверено на JDK 21 с быстрым producer'ом и медленным consumer'ом: буфер ёмкостью 3 заполняется, producer засыпает в `put`, consumer забирает — producer просыпается и докладывает; все 10 элементов проходят в порядке FIFO, программа завершается чисто. Заметь: `wait` бросает `InterruptedException` — это тот самый контракт прерывания из главы 6.10, детально мы разберём его в финальном уроке блока.

И мост вперёд, чтобы ты видел, куда это ведёт. В блоке 24 весь этот механизм уже упакован. `Condition` (в паре с `ReentrantLock`) — это те же «комнаты ожидания», но их можно завести **несколько на одном локе** и именованных: `notFull.await()` / `notEmpty.signal()` — producer будит именно consumer'ов, а не всех подряд, проблема `notify`-не-тому исчезает по построению. А `BlockingQueue` прячет вообще всё: `queue.put(x)` и `queue.take()` — и никаких мониторов в твоём коде. Но внутри у них — ровно та механика, которую ты сейчас разобрал, поэтому фундамент был обязателен.

> [!tip] 💡 Из C#  
> Прямое соответствие: `wait` / `notify` / `notifyAll` ≈ `Monitor.Wait` / `Monitor.Pulse` / `Monitor.PulseAll`, и требование «владеть монитором, чтобы их звать» в .NET такое же (`SynchronizationLockException` вместо `IllegalMonitorStateException`). Правило «`Wait` в цикле с перепроверкой условия» тоже общее для обеих платформ.  
> ⚠️ Ложный друг — не в семантике, а в опыте: разработчик из мира `async`/`Task` этот низкоуровневый слой в .NET почти не трогает — за него всё делают `Channel`, `SemaphoreSlim`, TPL. Поэтому легко решить «ну, это какая-то архаика, пропущу». В Java нельзя: `wait`/`notify` — это фундамент, на котором стоят `Condition` и `BlockingQueue`, и на собеседованиях его спрашивают как маркер реального понимания монитора, а не знакомства с API.

> [!example] 🔬 Глубже / на синьора  
> **Entry set vs wait set.** У монитора две разные очереди: entry set — потоки, заблокированные на _входе_ в `synchronized` (состояние `BLOCKED`), и wait set — потоки, вызвавшие `wait` (состояние `WAITING`/`TIMED_WAITING`; вспомни жизненный цикл из урока 1). `notify` не передаёт лок разбуженному — он лишь перемещает поток из wait set в борьбу за монитор, где тот конкурирует с потоками entry set и с новыми пришедшими. Intrinsic-мониторы **не гарантируют честность (fairness)**: порядок получения лока не определён, поток может обгоняться сколько угодно.
> 
> **Почему `if` ломается даже без spurious wakeups — «украденный сигнал».** Пусть в wait set два consumer'а, producer кладёт один элемент и зовёт `notifyAll`. Оба проснулись, но лок достанется по одному: первый забирает элемент, второй получает лок — элемента нет. С `if` второй consumer упадёт на пустом буфере. Никакой экзотики спецификации — обычная конкуренция за лок между пробуждением и продолжением.
> 
> **Lost wakeup.** Зеркальная патология: сигнал прозвучал _до_ того, как ждущий уснул. `notify` не «запоминается» — если в wait set никого нет, он уходит в никуда. Именно поэтому проверка условия и `wait` обязаны быть под тем же локом, что изменение условия и `notify`: лок делает невозможным чередование «проверил → (producer: изменил, крикнул) → уснул». Состояние (`items`) здесь играет роль «памяти» сигнала — поэтому ждём мы всегда _условие на состоянии_, а не сам факт `notify`.
> 
> **Когда `notifyAll` обязателен для корректности, а не просто «безопаснее».** Как только выполняется любое из: (а) ждущие потоки ждут разных предикатов на одном мониторе; (б) одно изменение состояния может удовлетворить больше одного ждущего (положили пачку элементов — надо будить нескольких consumer'ов). `notify` в этих случаях — не оптимизация, а баг: часть потоков, чьё условие уже истинно, останется спать.
> 
> **Мост к `Condition` (блок 24).** Концептуальная разница: `Object.wait`/`notify` привязаны к intrinsic-монитору — одна комната ожидания на лок, без вариантов. `Lock`/`Condition` разделяют «лок» и «очередь ожидания» как объекты: `lock.newCondition()` можно вызвать многократно и получить несколько независимых очередей (`notFull`, `notEmpty`), плюс появляются честный режим (`new ReentrantLock(true)`), `awaitNanos` с дедлайном и ожидание, прерываемое без захвата лока. Всё это — та же машина, но с раздельными ручками.

> [!warning] ⚠️ Грабли
> 
> - **Лок на `this` / публичном объекте** → любой код со ссылкой на объект может захватить твой монитор извне: задержки, deadlock'и, которые ты не контролируешь.
> - **Лок на `String`-литерале или боксированном `Integer`** → интернирование/кеш делают объект общим — монитор делится с совершенно посторонним кодом.
> - **Не-`final` lock-поле** → переприсваивание «уводит» монитор: потоки синхронизируются на разных объектах, взаимное исключение молча исчезает.
> - **`wait` в `if` вместо `while`** → spurious wakeup или украденный сигнал приводят к работе при ложном условии: `NoSuchElementException`, порча данных.
> - **`wait`/`notify` вне `synchronized` на том же объекте** → `IllegalMonitorStateException` в рантайме; компилятор не спасёт.
> - **`notify` при разных условиях ожидания** → разбужен «не тот» поток, нужный спит вечно; система замирает без формального deadlock'а — тяжело диагностировать.
> - **Путаница `wait` и `sleep`** → `wait` освобождает монитор, `sleep` держит; `sleep` внутри `synchronized` блокирует всех остальных на всё время сна.
> - **Условие и `wait` на разных локах** → проверяешь состояние под одним монитором, ждёшь на другом — атомарность «проверил–уснул» разрушена, возвращается lost wakeup.

### На собесе

> [!question]- Почему рекомендуют `private final Object lock`, а не `synchronized(this)`?  
> **RU:** Монитор `this` публичный: любой код со ссылкой на объект может сделать `synchronized(obj)` и заблокировать мои методы извне — я не контролирую собственную блокировку. Приватный `final` лок недостижим снаружи, значит захватить его может только мой код; `final` защищает от переприсваивания, из-за которого потоки оказались бы на разных мониторах. Бонус: несколько приватных локов позволяют раздельно защищать независимые части состояния и снизить конкуренцию.  
> **EN:** `this` is a publicly reachable monitor — any code holding a reference can lock it and stall my methods from outside. A `private final` lock object is unreachable externally, so I fully own my locking policy; `final` prevents reassignment that would silently split synchronization across two monitors. It also lets me use several locks for independent state.

> [!question]- Блокируют ли друг друга инстансная и статическая синхронизация?  
> **RU:** Нет. `synchronized`-метод экземпляра берёт монитор `this`, `static synchronized` — монитор объекта `Class`. Это два разных объекта и два независимых лока: потоки в них не мешают друг другу. Если статический и инстансный методы трогают одни данные — это готовая гонка.  
> **EN:** No. Instance methods lock `this`; static ones lock the `Class` object — two different monitors, fully independent. If both touch shared data, that's a data race.

> [!question]- Почему нельзя синхронизироваться на строковых литералах?  
> **RU:** Литералы интернируются: одинаковые литералы в разных классах и даже разных библиотеках — один объект из пула строк. Значит, монитор разделяется с чужим кодом: посторонний модуль, залочившийся на такой же литерал, будет конкурировать с моим и может меня заблокировать. То же с боксированными `Integer` из кеша −128..127.  
> **EN:** String literals are interned — identical literals anywhere in the JVM are the same object, so the monitor is shared with unrelated code that happens to lock the same literal. Same issue with cached boxed `Integer`s.

> [!question]- Почему `wait` вызывают в цикле `while`, а не в `if`?  
> **RU:** Две причины. Первая — spurious wakeups: спецификация разрешает потоку проснуться без `notify`. Вторая, более частая: между `notify` и повторным захватом монитора условие могло снова стать ложным — например, другой поток успел забрать ресурс («украденный сигнал»). Поэтому пробуждение — не гарантия условия, а лишь повод его перепроверить; `while` даёт инвариант «после цикла условие истинно и лок наш».  
> **EN:** Two reasons: spurious wakeups are explicitly allowed by the spec, and — more commonly — between `notify` and re-acquiring the monitor another thread may have consumed the resource, making the condition false again. A wakeup is a hint to re-check, not a guarantee; `while` restores the invariant.

> [!question]- Чем `wait` отличается от `sleep` в части удержания лока?  
> **RU:** `wait` освобождает монитор, на котором вызван, и уходит в его wait set — другие потоки могут войти и изменить условие. `sleep` не имеет отношения к мониторам: он просто приостанавливает поток и держит все захваченные локи. `sleep` внутри `synchronized` — блокировка всех остальных на время сна. Плюс `wait` можно звать только владея монитором, `sleep` — где угодно.  
> **EN:** `wait` releases the monitor it's called on and parks in its wait set, letting others change the condition; `sleep` keeps all held locks and just pauses the thread. Also, `wait` requires owning the monitor; `sleep` doesn't.

> [!question]- Разница `notify` и `notifyAll` — и когда второй обязателен?  
> **RU:** `notify` будит один произвольный поток из wait set — дёшево, но опасно: при потоках с разными условиями ожидания можно разбудить не того, и нужный не проснётся никогда. `notifyAll` будит всех; каждый перепроверит своё условие в `while`, лишние снова уснут. `notifyAll` обязателен, когда на мониторе ждут разные условия или когда одно изменение может удовлетворить несколько потоков. Дефолт: сомневаешься — `notifyAll`.  
> **EN:** `notify` wakes one arbitrary waiter — cheap but wrong when waiters wait on different predicates or one change can satisfy several of them; the right thread may never wake. `notifyAll` wakes everyone and the `while` loops sort it out. Default to `notifyAll` unless all waiters wait on the same condition and one signal serves exactly one.

> [!question]- Почему `wait`/`notify` требуют владения монитором?  
> **RU:** Иначе возникает гонка «проверил — уснул»: между проверкой условия и вызовом `wait` другой поток мог изменить условие и крикнуть `notify` в ещё пустой wait set — сигнал теряется, поток спит вечно (lost wakeup). Лок делает пару «проверка + wait» атомарной относительно пары «изменение + notify». Нарушение контракта — `IllegalMonitorStateException`.  
> **EN:** To prevent the lost-wakeup race: without the lock, a notifier could change the condition and signal between my condition check and my `wait`, so the signal lands in an empty wait set. Holding the monitor makes "check + wait" atomic w.r.t. "change + notify". Violations throw `IllegalMonitorStateException`.
