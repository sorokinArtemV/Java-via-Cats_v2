## Глава 23 · Урок 1. Потоки: что это и как их создать

До этого момента любая твоя Java-программа жила простой жизнью: JVM вызывала `main`, инструкции исполнялись одна за другой сверху вниз, и когда `main` заканчивался — программа завершалась. Но настоящие серверные приложения так не работают: пока один запрос ждёт ответа от базы, нужно обслуживать другие; пока считается тяжёлый отчёт, интерфейс должен оставаться живым. Для этого нужно уметь исполнять несколько дел одновременно внутри одной программы — и минимальная единица такого «одновременно» в Java называется потоком (`Thread`). Этот урок — фундамент всего блока про конкурентность: здесь мы разберёмся, что такое поток физически, как его создать и запустить, и получим первые инструменты управления — `sleep()` и `join()`.

### 23.1. Что такое поток

**Ментальная модель:** поток — это независимая линия исполнения кода внутри одной программы. Потоков может быть много, они исполняются параллельно и разделяют общую память процесса.

Начнём с того, что ты уже видел тысячу раз, просто не называл это словом «поток». Когда ты запускаешь программу, JVM исполняет её инструкции строго по одной: вошла в `main`, выполнила первую строку, вторую, зашла в метод, вернулась, пошла дальше. Эта последовательность «где мы сейчас и что исполняем следующим» — и есть линия исполнения. У неё есть свой стек вызовов (кто кого вызвал), свои локальные переменные, своя текущая инструкция.

Так вот: `main` — это не «просто начало программы». Это код, который исполняется в потоке. JVM при старте создаёт поток с именем `main` и отдаёт ему твой метод `main()`. Один поток у тебя был всегда — ты просто не знал, что их может быть больше.

А их может быть больше. Программа может завести вторую, третью, десятую независимую линию исполнения — и все они будут работать _одновременно, внутри одного процесса, с общей памятью_. Представь мастерскую: один общий стол с деталями (это heap — объекты), и за столом может работать один мастер, а может пять. Каждый мастер — поток: у каждого свои руки и свой блокнот с текущей задачей (стек и локальные переменные), но детали на столе — общие. Любой может взять деталь, которую положил другой. В этой общности — вся сила многопоточности (потокам не нужно ничего пересылать друг другу, они видят одни и те же объекты) и все её проблемы (два мастера одновременно тянутся к одной детали — про это будут следующие уроки).

Сразу разведём два слова, которые часто путают:

- **Процесс** — это запущенная программа целиком: своя изолированная память, свои ресурсы. Два процесса (например, твоя JVM и браузер) не видят память друг друга.
- **Поток** — линия исполнения _внутри_ процесса. Потоки одного процесса делят его память: heap один на всех, объекты видны всем потокам.

Эта деталь — «общая память» — кажется сейчас проходной, но именно она станет главным героем урока про модель памяти Java. Запомни картинку: стеки у потоков свои, heap — общий.

### 23.2. Три способа создать поток

Поток в Java состоит из двух вещей, которые полезно разделять в голове: **задача** (что делать — код) и **механизм исполнения** (сам поток ОС, который этот код исполнит). Задачу описывает интерфейс `Runnable` — у него ровно один метод:

```java
@FunctionalInterface
public interface Runnable {
    void run();
}
```

Ничего не принимает, ничего не возвращает, checked-исключений не бросает. Просто «кусок работы». А класс `Thread` — это механизм: он берёт `Runnable` и исполняет его в новой линии исполнения.

#### Способ 1: `implements Runnable` + `new Thread(runnable)` — предпочтительный

```java
public class HelloThread {
    public static void main(String[] args) {
        System.out.println("main работает в потоке: " + Thread.currentThread().getName());

        Thread thread = new Thread(new DownloadTask());
        thread.start();

        System.out.println("main продолжает работу");
    }
}

class DownloadTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Скачиваю... поток: " + Thread.currentThread().getName());
    }
}
```

Вывод (проверено на JDK 21):

```
main работает в потоке: main
main продолжает работу
Скачиваю... поток: Thread-0
```

Обрати внимание на две вещи. Первая: `Thread.currentThread()` возвращает объект того потока, в котором эта строка _сейчас исполняется_, — так код может узнать, «кто я». Вторая: `main продолжает работу` напечаталось _до_ сообщения из `DownloadTask`. После `start()` у программы две линии исполнения, и `main` не ждёт вторую — обе бегут независимо.

Почему этот способ считается правильным? Потому что он разделяет «что делать» и «чем исполнять». `DownloadTask` — это просто задача: её можно отдать потоку, можно позже отдать пулу потоков (глава 24), можно протестировать вызовом `run()` без всякой многопоточности. И класс задачи остаётся свободным: `implements Runnable` не тратит единственное наследование — `DownloadTask` может при необходимости `extends` что-то своё.

#### Способ 2: `extends Thread` — покажу, но так не делают

```java
public class ExtendsThread {
    public static void main(String[] args) {
        Thread thread = new LoggingThread();
        thread.start();
    }
}

class LoggingThread extends Thread {
    @Override
    public void run() {
        System.out.println("Работаю в потоке: " + getName()); // Thread-0
    }
}
```

Работает так же, но у подхода два системных минуса. Во-первых, ты сжигаешь единственное наследование Java на техническую деталь: класс уже не сможет наследовать ничего из твоей доменной иерархии. Во-вторых, задача и механизм склеены в один класс — такую задачу нельзя отдать пулу потоков, нельзя переиспользовать отдельно от `Thread`. Встретишь этот способ в старом коде и в туториалах — узнавай, но сам пиши через `Runnable`.

#### Способ 3: лямбда — то, что ты будешь писать на практике

`Runnable` — функциональный интерфейс (один абстрактный метод), а значит, его можно записать лямбдой (глава 9):

```java
Thread thread = new Thread(() -> {
    System.out.println("Работаю в потоке: " + Thread.currentThread().getName());
});
thread.start();
```

Концептуально это тот же способ 1 — задача отдельно, поток отдельно, — просто без церемонии с отдельным классом. Для коротких задач это стандартная современная форма.

### 23.3. `start()` vs `run()`: сердце урока

Теперь самый важный момент — и классическая ловушка, на которой споткнулось не одно поколение джавистов, и один из самых частых вопросов на собеседовании.

У объекта `Thread` есть и метод `start()`, и метод `run()`. Оба публичные, оба выглядят как «запустить». Разница фундаментальная:

- **`run()`** — это обычный метод. Вызвать его напрямую — значит просто выполнить его код _в текущем потоке_, синхронно, как любой другой вызов метода. Никакого нового потока не появляется.
- **`start()`** — это команда JVM: создай новый поток операционной системы и выполни `run()` _в нём_. Только `start()` рождает параллельность.

Не верь на слово — посмотри глазами. Имя потока нас не обманет:

```java
public class StartVsRun {
    public static void main(String[] args) {
        Runnable task = () ->
            System.out.println("Задача выполняется в потоке: "
                    + Thread.currentThread().getName());

        Thread thread = new Thread(task);

        System.out.println("--- вызываем run() ---");
        thread.run();   // обычный вызов метода, никакого нового потока

        System.out.println("--- вызываем start() ---");
        thread.start(); // вот теперь новый поток
    }
}
```

Вывод (JDK 21):

```
--- вызываем run() ---
Задача выполняется в потоке: main
--- вызываем start() ---
Задача выполняется в потоке: Thread-0
```

При `run()` задача отчиталась: «я в `main`» — мы просто вызвали метод, и он выполнился в той же линии исполнения. При `start()` — «я в `Thread-0`»: появилась вторая линия. Программа при этом _выглядит_ одинаково рабочей в обоих случаях — код-то выполнился! Именно поэтому баг «вызвал `run()` вместо `start()`» такой коварный: ничего не падает, просто вся «многопоточность» тихо исполняется последовательно в одном потоке.

#### Поток одноразовый: повторный `start()` — исключение

Ещё одно свойство, которое стоит впитать сразу: объект `Thread` нельзя запустить дважды. Поток, который отработал, — отработал навсегда; это не «пауза», это финал.

```java
public class DoubleStart {
    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(() -> System.out.println("Работаю"));
        thread.start();
        thread.join();
        thread.start(); // второй раз — исключение
    }
}
```

```
Работаю
Exception in thread "main" java.lang.IllegalThreadStateException
	at java.base/java.lang.Thread.start(Thread.java:1525)
```

Причина в том, что у потока есть жизненный цикл с состояниями: он рождается (`NEW`), запускается, работает и завершается (`TERMINATED`) — и из финального состояния дороги назад нет. `start()` разрешено вызывать только на потоке в состоянии `NEW`. Подробно этот жизненный цикл — тема следующего урока; пока достаточно правила: один объект `Thread` = один запуск. Нужно выполнить задачу ещё раз — создай новый `Thread` (а как не плодить потоки на каждую задачу — расскажет глава 24 про пулы).

### Базовые методы управления

#### `Thread.sleep(ms)` — усыпить текущий поток

`sleep()` приостанавливает поток на заданное время: поток перестаёт исполняться, не ест CPU, и через указанное количество миллисекунд планировщик его будит. Метод бросает checked-исключение `InterruptedException` — почему и что с ним делать, разберём в следующем уроке; пока в примерах будем его честно обрабатывать.

У `sleep()` есть коварная особенность: он **статический** и усыпляет всегда _тот поток, который его вызвал_ — а не тот объект, на котором ты его синтаксически позвал. Java позволяет вызывать статические методы через переменную экземпляра, и это порождает обманчивый код:

```java
public class SleepGotcha {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            for (int i = 0; i < 3; i++) {
                System.out.println("worker: " + i);
            }
        });
        worker.start();

        worker.sleep(500); // выглядит как «усыпить worker», но спит main!
        System.out.println("main: проснулся");
        worker.join();
    }
}
```

Строка `worker.sleep(500)` читается как «усыпи worker», но компилятор разворачивает её в `Thread.sleep(500)` — и спит `main`, вызвавший поток. Именно поэтому по конвенции `sleep` пишут только как `Thread.sleep(...)`, а IDE подсвечивает вызов статики через экземпляр как запах. И ещё одно свойство на будущее, просто зафиксируй: спящий поток **не отпускает мониторы** — если он держал блокировку, он продолжает её держать во сне. Что такое монитор — узнаешь через пару уроков, но эта фраза тогда сэкономит тебе дедлок.

#### `join()` — дождаться завершения другого потока

После `start()` две линии бегут независимо — но часто `main` нужен результат работы второго потока, а значит, нужно уметь _подождать_. Это делает `join()`: вызов `worker.join()` блокирует текущий поток до тех пор, пока `worker` не завершится. Это твой первый инструмент координации потоков.

Сначала посмотрим, что происходит _без_ него:

```java
Thread worker = new Thread(() -> {
    try {
        Thread.sleep(1000); // имитируем долгую работу
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return;
    }
    System.out.println("worker: закончил считать отчёт");
});

worker.start();
System.out.println("main: дошёл до конца и завершился");
```

```
main: дошёл до конца и завершился
worker: закончил считать отчёт
```

`main` убежал вперёд и «закончился» раньше, чем worker досчитал. Если бы `main` попытался в этот момент использовать результат — результата бы ещё не было. Теперь с `join()`:

```java
public class JoinDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return;
            }
            System.out.println("worker: закончил считать отчёт");
        });

        worker.start();
        System.out.println("main: запустил worker, жду его через join()");

        worker.join(); // main блокируется, пока worker не завершится

        System.out.println("main: worker завершён, можно использовать результат");
    }
}
```

```
main: запустил worker, жду его через join()
worker: закончил считать отчёт
main: worker завершён, можно использовать результат
```

Порядок стал детерминированным: `main` доходит до `join()`, засыпает, worker дорабатывает, и только после этого `main` продолжает. Есть и вариант с таймаутом — `join(ms)` ждёт не дольше указанного времени (а завершился ли поток на самом деле, потом можно проверить через `isAlive()`).

#### Остальное — коротко

- `getName()` / `setName(String)` — имя потока. JVM даёт имена автоматически (`main`, `Thread-0`, `Thread-1`…), но осмысленные имена (`report-worker`) — лучший друг при чтении логов и thread dump'ов.
- `isAlive()` — `true`, если поток запущен и ещё не завершился.
- `setDaemon(true)` — делает поток **демоном**: демон-поток не удерживает JVM живой. Обычно JVM работает, пока жив хотя бы один _не_-демон поток — именно поэтому в примере `NoJoinDemo` выше worker успел допечатать своё сообщение уже после конца `main`: JVM ждала его. Демонов JVM не ждёт: когда завершился последний пользовательский поток, JVM выходит, обрывая демонов на полуслове (без `finally`, без закрытия ресурсов). Демоны — для фоновых сервисных задач вроде сборки статистики. Важно: `setDaemon()` вызывается строго _до_ `start()`, иначе `IllegalThreadStateException`.

Ещё у `Thread` есть методы `interrupt()` / `isInterrupted()` — механизм вежливой просьбы «остановись». Это отдельная большая тема со своим контрактом (и с тем самым `InterruptedException`, который мы ловили в примерах) — разберём её детально в следующем уроке вместе с жизненным циклом потока.

> [!tip] 💡 Из C#  
> Мостики для интуиции — с одним большим предупреждением в конце:
> 
> - `Runnable` ≈ `Action` / тело `Task`: «кусок работы» без параметров и результата.
> - `new Thread(runnable).start()` ≈ `new Thread(action).Start()` — здесь API почти дословно совпадают, вплоть до того, что в .NET тоже есть разница между `Start()` и прямым вызовом делегата.
> - `Thread.sleep(1000)` ≈ `Thread.Sleep(1000)` — оба статические, оба усыпляют вызывающий поток.
> - `worker.join()` ≈ `worker.Join()`.
> - Демон-поток ≈ `IsBackground = true`: background-потоки в .NET тоже не держат процесс живым.
> 
> ⚠️ **Ложный друг** — не сам API, а привычки его использования. В современном .NET ты почти никогда не создаёшь `Thread` руками — рядовой код живёт в мире `Task`, async/await и пула потоков, а «сырой» `Thread` считается низкоуровневой экзотикой. В Java расклад другой: прямая работа с `Thread` и `Runnable` — обыденная база, которую спрашивают на каждом собеседовании и на которой построено всё остальное. Task-подобные абстракции в Java есть (`ExecutorService`, `CompletableFuture` — глава 24), но async/await-аналога в языке нет, а до появления virtual threads (Java 21) низкий уровень был виден гораздо чаще, чем в .NET. Так что не жди, что `Thread` можно «пролистать как легаси», — здесь это фундамент.

> [!example] 🔬 Глубже / на синьора  
> **Платформенный поток — тонкая обёртка над потоком ОС.** Когда ты вызываешь `start()`, JVM через системный вызов просит операционную систему создать нативный поток (на Linux — через `pthread_create`), и планирует его исполнение именно ядро ОС, а не JVM. Отсюда две цены. Первая — память: каждому потоку резервируется стек, по умолчанию порядка 1 МБ (управляется флагом `-Xss`), поэтому «создадим по потоку на каждый из 100 000 клиентов» упирается в ресурсы задолго до ста тысяч. Вторая — время: создание потока и переключение контекста между потоками — системные вызовы, на порядки дороже обычного вызова метода. Из этих двух цен вырастает вся глава 24: пулы потоков (создаём дорогие потоки один раз и переиспользуем) и virtual threads из Java 21, JEP 444 (дешёвые потоки, планируемые самой JVM, а не ядром).
> 
> **Условие завершения JVM — формально.** JVM завершается, когда: (а) все не-демон потоки завершились, либо (б) кто-то вызвал `System.exit()`. Именно поэтому `main` может закончиться, а программа — продолжать работать: `main` — обычный не-демон поток, и его завершение ничем не особенное, если живы другие пользовательские потоки.
> 
> **Почему `Runnable.run()` не может бросить checked-исключение.** Посмотри на сигнатуру: `void run()` — без `throws`. Переопределяющий метод не может расширять список checked-исключений (глава 6), значит, твой `run()` обязан разрулить все checked-исключения внутри себя — поэтому в примерах мы оборачивали `Thread.sleep()` в try/catch прямо в лямбде. Дизайн-причина глубже: исключение — это механизм передачи ошибки вверх по стеку вызывающему, а у нового потока стек свой, и «вызывающего» в нём нет: `run()` — дно стека. Некому декларировать `throws`. Что происходит с unchecked-исключением, вылетевшим из `run()` (спойлер: поток умирает, есть `UncaughtExceptionHandler`), и как всё-таки вернуть из потока результат или ошибку — разберём на `Future` в главе 24.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`run()` вместо `start()`.** Код работает, тесты зелёные, но никакой параллельности нет — всё исполняется последовательно в вызывающем потоке. Самый тихий баг урока: ничего не падает, просто программа не та, которой кажется. Проверка — печать `Thread.currentThread().getName()`.
> 2. **Повторный `start()` того же объекта `Thread`** — `IllegalThreadStateException`. Поток одноразовый; на каждый запуск — новый объект (а лучше пул, глава 24).
> 3. **Расчёт на порядок исполнения потоков.** После `start()` никакого гарантированного порядка между потоками нет — кто выполнится раньше, решает планировщик ОС, и от запуска к запуску ответ меняется. Код, который «работает, потому что worker обычно успевает первым», — сломанный код, который пока везёт. Нужен порядок — используй явную координацию (`join()`, дальше — богаче).
> 4. **`worker.sleep(500)` усыпляет не worker.** `sleep` — статический метод, спит всегда вызывающий поток. Вызов статики через переменную экземпляра компилируется, но обманывает читателя. Только `Thread.sleep(...)`.
> 5. **`setDaemon(true)` после `start()`** — `IllegalThreadStateException`. И обратная грабля: сделал рабочий поток демоном — JVM может выйти, оборвав его посреди записи в файл. Демон — только для задач, которые не жалко убить в любой момент.

### На собесе

> [!question]- Q1. Чем start() отличается от run()? _(спрашивают почти всегда)_  
> Слабый ответ: «`start()` запускает поток, а `run()` — это метод потока» — без объяснения, что произойдёт при прямом вызове.
> 
> **Эталон (RU):** `run()` — обычный метод; его прямой вызов выполняет код синхронно в текущем потоке, новый поток не создаётся. `start()` просит JVM создать новый поток ОС и выполнить `run()` в нём — только так возникает параллельное исполнение. Проверяется печатью `Thread.currentThread().getName()`.
> 
> **Model answer (EN):** run() is just a regular method — calling it directly executes the code synchronously in the current thread, no new thread is created. start() asks the JVM to spawn a new OS thread and invoke run() inside it; that's the only way to get actual concurrency. You can verify it by printing Thread.currentThread().getName().
> 
> _Follow-up-ловушка:_ «А что выведет программа, если вызвать `run()`?» — выведет всё то же самое, но с именем вызывающего потока; программа работает, просто последовательно.

> [!question]- Q2. implements Runnable vs extends Thread — что предпочесть и почему?  
> **Эталон (RU):** `Runnable` — потому что он отделяет задачу от механизма исполнения: задачу можно передать пулу потоков, переиспользовать, тестировать без потоков, а класс не тратит единственное наследование. `extends Thread` смешивает «что делать» и «чем исполнять» и встречается в основном в легаси.
> 
> **Model answer (EN):** Prefer Runnable: it separates the task from the execution mechanism, so the same task can be handed to a thread pool, reused, and unit-tested without threads, and the class keeps its single inheritance slot free. extends Thread couples the task to the threading mechanism and is mostly legacy style.

> [!question]- Q3. Что делает join()?  
> **Эталон (RU):** блокирует текущий поток, пока поток, на котором вызван `join()`, не завершится (или пока не истечёт таймаут в перегрузке `join(ms)`). Это базовый способ дождаться результата чужой работы.
> 
> **Model answer (EN):** It blocks the current thread until the thread it's called on terminates (or until the timeout expires in the join(ms) overload). It's the basic way to wait for another thread to finish before using its results.
> 
> _Follow-up:_ «Что будет без `join()`?» — потоки бегут независимо; `main` может завершиться раньше worker'а, но JVM всё равно дождётся не-демон потоков.

> [!question]- Q4. Что такое демон-поток?  
> **Эталон (RU):** поток, который не удерживает JVM живой. JVM завершается, когда завершились все не-демон потоки, — демоны при этом обрываются мгновенно, без `finally` и освобождения ресурсов. Подходят только для фоновых сервисных задач; флаг ставится через `setDaemon(true)` строго до `start()`.
> 
> **Model answer (EN):** A daemon thread doesn't keep the JVM alive — the JVM exits once all non-daemon threads have finished, killing daemons abruptly, with no finally blocks executed. They're suitable only for background housekeeping. The flag must be set via setDaemon(true) before start(), otherwise you get an IllegalThreadStateException.

> [!question]- Q5. Можно ли запустить один Thread дважды?  
> **Эталон (RU):** нет. После завершения поток находится в состоянии `TERMINATED`, повторный `start()` бросает `IllegalThreadStateException`. Для повторного выполнения задачи создают новый `Thread` или используют пул потоков.
> 
> **Model answer (EN):** No. Once a thread has finished, it's in the TERMINATED state, and calling start() again throws IllegalThreadStateException. To run the task again, create a new Thread — or better, submit the task to a thread pool.
