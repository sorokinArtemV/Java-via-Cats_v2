В уроке 3 мы пришли к жёсткому выводу: без установленного happens-before у тебя нет никаких гарантий, что запись одного потока увидит другой. Мы перечислили источники happens-before — но не разобрали ни одного инструмента, который его создаёт. Этот урок — про два самых базовых: `volatile` и `synchronized`. Первый решает ровно одну беду из трёх (видимость), второй закрывает все три сразу (атомарность, видимость, порядок), но берёт за это плату. К концу урока ты будешь точно знать, какой инструмент какую беду закрывает — и где граница между ними.

### 23.7 `volatile`

#### Возвращаемся к зависшему циклу

В уроке 3 мы видели программу, которая может не завершиться никогда: один поток крутит `while (!flag) {}`, другой ставит `flag = true`, но без happens-before между записью и чтением JIT вправе закешировать значение `flag` в регистре — и цикл никогда не увидит обновление.

Починка — одно слово:

```java
public class StopFlag {
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            long iterations = 0;
            while (running) {
                iterations++;
            }
            System.out.println("Остановился после " + iterations + " итераций");
        });
        worker.start();

        Thread.sleep(100);
        running = false;   // запись гарантированно видна worker'у
        worker.join(2000);
        System.out.println("worker жив? " + worker.isAlive());
    }
}
```

Запуск на JDK 21:

```
Остановился после 18966130 итераций
worker жив? false
```

Поток честно остановился. Убери `volatile` — и на серверной JVM цикл с большой вероятностью зависнет навсегда (мы разбирали механику в уроке 3: hoisting чтения из цикла — легальная оптимизация, пока нет happens-before).

#### Что именно даёт `volatile`

Здесь не нужно новой теории — нужно одно правило из уже знакомого нам списка источников happens-before (JLS §17.4.5):

> **Запись в `volatile`-поле happens-before каждого последующего чтения этого же поля.**

Из этого правила вытекают обе гарантии `volatile`:

1. **Видимость.** Если поток B прочитал из `volatile`-поля значение, которое записал поток A, то B видит не только само это значение, а **всё**, что A сделал до записи. Happens-before транзитивен (урок 3): всё, что было «до» записи в программе A, становится «до» для чтения в B.
2. **Запрет переупорядочивания.** Компилятор и процессор не могут переставить обычные операции через обращение к `volatile`-полю так, чтобы сломать это отношение. Запись в `volatile` — это точка, до которой все предыдущие записи «выталкиваются» наружу; чтение — точка, после которой все последующие чтения «свежие».

Именно поэтому `volatile` годится для _публикации_: поток A собрал объект, положил ссылку в `volatile`-поле — поток B, прочитавший ссылку, гарантированно видит объект полностью построенным, а не полусобранным.

#### Чего `volatile` НЕ даёт — главный mid+ момент

А теперь критическая половина темы. В уроке 3 мы разделили видимость и атомарность как две **разные** беды. `volatile` лечит первую и _не трогает_ вторую. Классическая проверка — счётчик:

```java
public class VolatileCounter {
    private static volatile int counter = 0;

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 100_000; i++) {
                counter++; // read-modify-write: не атомарно
            }
        };
        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start(); t2.start();
        t1.join();  t2.join();

        System.out.println("Ожидали: 200000, получили: " + counter);
    }
}
```

Три запуска подряд на JDK 21:

```
Ожидали: 200000, получили: 200000
Ожидали: 200000, получили: 100000
Ожидали: 200000, получили: 200000
```

Во втором запуске потерялась **половина** инкрементов — при том что поле `volatile`. Почему? Потому что `counter++` — это не одна операция, а три: _прочитать_ текущее значение, _прибавить_ единицу, _записать_ обратно. `volatile` делает каждый из этих шагов честным по видимости — каждое чтение видит последнее записанное значение, каждая запись сразу видна всем. Но он **не склеивает три шага в один неделимый**. Два потока успевают прочитать одно и то же значение `5` до того, как кто-то из них запишет `6` — и оба записывают `6`. Один инкремент растворился, и никакая видимость этого не чинит: оба потока всё видели «честно», просто их шаги перемешались.

![[Pasted image 20260704153621.png]]

#### Когда `volatile` уместен

Из механики следует чёткое правило применимости. `volatile` подходит, когда **операция сама по себе атомарна и нужна только видимость**:

- **Флаг остановки / состояния**: `volatile boolean running` — запись одного `boolean` неделима, потоку-читателю нужно только вовремя её увидеть.
- **Однократная публикация ссылки**: поток собрал объект целиком и положил ссылку в `volatile`-поле; читатели видят объект полностью инициализированным.
- **«Последнее записанное значение»**: один поток пишет актуальную метрику/конфиг, остальные только читают.

И где `volatile` **не годится**:

- любой счётчик и вообще любое «прочитать-изменить-записать» (`i++`, `x = x * 2`, `if (cache == null) cache = load()`);
- любой инвариант из нескольких полей (`min <= max`, «баланс и история согласованы») — `volatile` на каждом поле не делает согласованной _пару_ полей.

Как только в игре появляется составная операция или связка полей — нужен инструмент посильнее.

### 23.8 `synchronized`

#### От ограничения — к взаимному исключению

`volatile` не смог спасти счётчик, потому что не умеет главного: сделать так, чтобы **три шага `i++` выполнились как один, без вмешательства других потоков**. Для этого нужна другая гарантия — _взаимное исключение_ (mutual exclusion): в критической секции кода в каждый момент времени находится максимум один поток. Пока он внутри — остальные ждут снаружи. Именно это делает `synchronized`.

#### Intrinsic lock: у каждого объекта есть монитор

Ключевая ментальная модель, на которой держится вся тема:

> **У каждого объекта в Java есть встроенный лок — монитор (intrinsic lock).** `synchronized` захватывает монитор указанного объекта на входе в секцию и отпускает на выходе — в том числе при выходе через `return` или исключение.

Пока поток A держит монитор объекта, любой другой поток, пытающийся войти в секцию **на том же мониторе**, приостанавливается — это то самое состояние `BLOCKED` из урока 2: поток стоит в очереди за монитором и не потребляет CPU. Когда A выходит из секции и отпускает монитор, один из ждущих получает его и заходит.

![[Pasted image 20260704153633.png]]

#### Две гарантии разом

Скажем прямо, потому что это часто упускают: `synchronized` даёт **обе** гарантии, а не только «блокировку».

1. **Взаимное исключение** → атомарность секции. Всё, что внутри, выполняется без чередования с другими потоками на том же мониторе. Составные операции и многополевые инварианты защищены.
2. **Видимость** → потому что у `synchronized` есть своё правило happens-before (JLS §17.4.5): **освобождение монитора happens-before каждого последующего захвата того же монитора**. Поток, вошедший в секцию, гарантированно видит всё, что сделал предыдущий владелец до выхода.

То есть `synchronized` закрывает все три беды из урока 3 — атомарность, видимость, переупорядочивание — но платит блокировкой потоков: ждущие спят, а усыпление и пробуждение стоят времени.

#### Что именно блокируется — самый частый источник путаницы

Блокируется не «метод», не «код» и не «класс». Блокируется **монитор конкретного объекта**. У `synchronized` три формы, и у каждой — свой монитор:

```java
public class Forms {
    private final Object lock = new Object();

    // 1) synchronized-метод экземпляра → монитор this
    public synchronized void a() { /* ... */ }

    // 2) synchronized-блок → монитор указанного объекта
    public void b() {
        synchronized (lock) { /* ... */ }
    }

    // 3) static synchronized-метод → монитор объекта Forms.class (не this!)
    public static synchronized void c() { /* ... */ }
}
```

Отсюда ключевое следствие, которое надо вбить намертво: **взаимное исключение существует только относительно одного и того же монитора.** Два потока, синхронизированные на разных объектах, друг другу вообще не мешают. Проверим оба классических случая на живом коде.

Случай первый: два экземпляра одного класса — два разных `this`, два разных монитора:

```java
public class TwoInstances {
    synchronized void work() throws InterruptedException {
        Thread.sleep(1000);
    }

    public static void main(String[] args) throws InterruptedException {
        TwoInstances a = new TwoInstances();
        TwoInstances b = new TwoInstances();

        // Случай 1: два потока, ДВА разных объекта
        long start = System.nanoTime();
        Thread t1 = new Thread(() -> { try { a.work(); } catch (InterruptedException e) {} });
        Thread t2 = new Thread(() -> { try { b.work(); } catch (InterruptedException e) {} });
        t1.start(); t2.start(); t1.join(); t2.join();
        System.out.println("Разные объекты: ~" + (System.nanoTime() - start) / 1_000_000 + " ms");

        // Случай 2: два потока, ОДИН объект
        start = System.nanoTime();
        Thread t3 = new Thread(() -> { try { a.work(); } catch (InterruptedException e) {} });
        Thread t4 = new Thread(() -> { try { a.work(); } catch (InterruptedException e) {} });
        t3.start(); t4.start(); t3.join(); t4.join();
        System.out.println("Один объект:    ~" + (System.nanoTime() - start) / 1_000_000 + " ms");
    }
}
```

Вывод на JDK 21:

```
Разные объекты: ~1003 ms
Один объект:    ~2001 ms
```

Два потока на разных объектах отработали параллельно (~1 секунда на двоих). Два потока на одном объекте — строго по очереди (~2 секунды). Метод один и тот же; разница только в том, чей монитор захватывается.

Случай второй, ещё коварнее: `synchronized`-метод экземпляра и `static synchronized`-метод **одного класса**. Интуиция подсказывает «они же в одном классе, значит защищают друг друга» — и интуиция врёт: мониторы разные (`this` против `Forms.class`):

```java
public class DifferentMonitors {
    // монитор this
    synchronized void instanceWork() throws InterruptedException {
        Thread.sleep(1000);
    }

    // монитор DifferentMonitors.class
    static synchronized void staticWork() throws InterruptedException {
        Thread.sleep(1000);
    }

    public static void main(String[] args) throws InterruptedException {
        DifferentMonitors obj = new DifferentMonitors();
        long start = System.nanoTime();

        Thread t1 = new Thread(() -> {
            try { obj.instanceWork(); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });
        Thread t2 = new Thread(() -> {
            try { staticWork(); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        t1.start(); t2.start();
        t1.join();  t2.join();

        System.out.println("Общее время: ~" + (System.nanoTime() - start) / 1_000_000 + " ms");
    }
}
```

```
Общее время: ~1008 ms
```

Параллельно. Никакой взаимной защиты между ними нет — и если оба метода трогают одни и те же `static`-данные, это готовый data race при двух честных `synchronized` в коде.

![[Pasted image 20260704153659.png]]

#### Починенный счётчик

Тот же счётчик, который `volatile` не спас — теперь весь `counter++` целиком под монитором:

java

```java
public class SyncCounter {
    private static int counter = 0;

    private static synchronized void increment() {
        counter++; // read-modify-write целиком под монитором SyncCounter.class
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 100_000; i++) increment();
        };
        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start(); t2.start();
        t1.join();  t2.join();
        System.out.println("counter = " + counter);
    }
}
```

```
counter = 200000
```

Всегда `200000`, сколько ни запускай. Заметь: `volatile` на поле больше не нужен — видимость даёт сам `synchronized` через своё правило happens-before. Пока чтения и записи поля происходят под одним монитором, поле обычное.

И честная цена, чтобы не было иллюзий: когда монитор занят, ждущий поток усыпляется средствами ОС и потом пробуждается — это переключения контекста, дорогие относительно самой операции `counter++`, которая занимает наносекунды. Для короткого инкремента блокировка — как башенный кран для гвоздя. Именно поэтому в блоке 24 появятся `AtomicInteger` и семейство CAS-операций (неблокирующая атомарность для простых случаев) и более гибкий `Lock` API — но их фундамент ты уже понимаешь: они решают ту же беду атомарности, только другой ценой.

> [!tip] 💡 Из C#
> 
> - `synchronized (obj) { ... }` ≈ `lock (obj) { ... }` — оба захватывают монитор объекта; C#-`lock` — это сахар над `Monitor.Enter`/`Monitor.Exit` с автоматическим `finally`. В Java `monitorexit` при выходе из блока (включая исключение) тоже гарантирован — компилятор генерирует exception table.
> - `synchronized`-метода как конструкции в C# нет; ближайший идиоматический аналог — `lock` на приватном объекте внутри метода.
> - Идиома `lock (private readonly object _lock)` из C# переносится в Java один в один: лочить на `this` или на объекте класса — плохая практика в обоих языках (почему — в следующем уроке про уровни локов).
> - ⚠️ **Ложный друг — `volatile`.** Ключевое слово есть в обоих языках, но переносить понимание нельзя: в Java семантика `volatile` строго определена через happens-before (JLS §17.4.5) и, как правило, сильнее и предсказуемее наивного C#-понимания (в C# это acquire/release-семантика с более тонкими оговорками про переупорядочивание). Что совпадает намертво в обоих языках: **`volatile` не даёт атомарности** — `i++` ломается и там, и там.

> [!example]- 🔬 Глубже / на синьора  
> **Во что это компилируется.** `synchronized`-блок — это байткод-инструкции `monitorenter`/`monitorexit` (проверено `javap -c` на JDK 21; `monitorexit` встречается дважды — на нормальном выходе и в сгенерированном обработчике исключений, чтобы монитор освобождался всегда). `synchronized`-метод не содержит этих инструкций — вместо этого у метода стоит флаг `ACC_SYNCHRONIZED`, и JVM захватывает/отпускает монитор сама при входе и выходе. `volatile`-поле помечается флагом `ACC_VOLATILE`, а JIT вставляет барьеры памяти (memory fences из урока 3): на x86 volatile-запись обычно компилируется в инструкцию с `lock`-префиксом (эффект StoreLoad-барьера), volatile-чтение на x86 почти бесплатно из-за сильной модели памяти самого железа — на ARM барьеров больше.  
> **Где живёт монитор.** Состояние лока хранится в _mark word_ — первых битах заголовка объекта. JVM оптимистична: сначала пробует **lightweight (thin) locking** — захват монитора одной CAS-операцией над mark word, без участия ОС. Только при реальной конкуренции (второй поток пришёл, пока первый держит) лок «раздувается» (inflation) в полноценный **ObjectMonitor** с очередями ожидающих и парковкой потоков через ОС. То есть неконкурентный `synchronized` дёшев — дорогим его делает contention. CAS здесь — тот же фундамент, на котором построены атомики; подробно разберём в блоке 24.  
> **Biased locking** — историческая оптимизация «привязать лок к первому потоку, чтобы он входил вообще без CAS» — отключён и объявлен deprecated в JDK 15 (JEP 374) и полностью удалён в JDK 18. На собеседовании про него ещё спрашивают, поэтому знать стоит: на Java 21 его уже нет.  
> **Реентерабельность.** Intrinsic lock реентерабелен: поток, уже держащий монитор, может войти в другую секцию на том же мониторе повторно — JVM ведёт счётчик вложенности. Иначе `synchronized`-метод, вызывающий другой `synchronized`-метод того же объекта, мгновенно дедлочил бы сам себя (проверено: рекурсивный вход работает).  
> **Virtual threads.** До недавнего времени `synchronized` _пинил_ виртуальный поток к его carrier-потоку: заблокировавшись на мониторе, виртуальный поток не мог размонтироваться, и carrier простаивал. В JDK 24 это исправлено — JEP 491 «Synchronize Virtual Threads without Pinning» (Java 24, новее нашей базовой Java 21 — на 21 проблема пиннинга ещё актуальна). Подробно — в блоке 24 про виртуальные потоки.

> [!warning] ⚠️ Грабли
> 
> 1. **`volatile` на счётчике / составной операции.** Не спасает: видимость есть, атомарности нет. `volatile int i; i++` теряет инкременты — мы это воспроизвели.
> 2. **Вера, что `synchronized` «блокирует метод».** Блокируется монитор объекта. Два потока на двух экземплярах выполняют один и тот же `synchronized`-метод параллельно.
> 3. **Синхронизация на разных объектах в расчёте на взаимное исключение.** Его нет: `synchronized(a)` и `synchronized(b)` друг другу не мешают, даже если внутри трогаются одни и те же данные, — это полноценный data race.
> 4. **`synchronized`-метод и `static synchronized` в расчёте на взаимную защиту.** Мониторы разные: `this` против `Класс.class`. Работают параллельно — проверено таймингом.
> 5. **Лок на изменяемой/переприсваиваемой ссылке.** `synchronized (config) { config = newConfig; }` — следующий поток захватит монитор уже _другого_ объекта, и два потока окажутся в «одной» секции одновременно. Объект-лок должен быть `private final`.
> 6. **Несимметричная защита поля.** Если запись идёт под `synchronized`, а чтение — без него (и поле не `volatile`), гарантий видимости у читателя нет: happens-before образуют только _пары_ операций на одном мониторе. Защищать поле нужно на каждом обращении.
> 7. **Неверный выбор инструмента в обе стороны.** Тяжёлый `synchronized` там, где хватило бы `volatile` (простой флаг) — лишний contention; и наоборот, `volatile` там, где есть составная операция, — молчаливая потеря данных. Правило: только видимость атомарного значения → `volatile`; составная операция или инвариант из нескольких полей → взаимное исключение.

>> [!question]- Что гарантирует `volatile` и чего не гарантирует?  
> Гарантирует видимость (запись happens-before последующего чтения этого поля) и запрет переупорядочивания вокруг обращения к полю. Не гарантирует атомарность составных операций: `i++` — это read-modify-write из трёх шагов, `volatile` их не склеивает.
> 
> _EN: `volatile` guarantees visibility — a write to a volatile field happens-before every subsequent read of it — and restricts reordering around the access. It does not make compound operations atomic: `i++` is a read-modify-write of three steps that can still interleave._

> [!question]- Почему volatile не спасает счётчик?  
> Каждое чтение и запись честны по видимости, но два потока могут прочитать одно и то же старое значение до того, как кто-то запишет новое — оба запишут одинаковый результат, один инкремент потеряется. Нужна атомарность секции: `synchronized` или `AtomicInteger`.
> 
> _EN: Both threads can read the same stale value before either writes back, so one increment is lost. You need atomicity of the whole read-modify-write — `synchronized` or `AtomicInteger`._

> [!question]- Когда достаточно volatile, а когда нужен synchronized?  
> `volatile` — когда операция сама атомарна и нужна только видимость: флаг остановки, публикация ссылки на готовый объект, «последнее значение». `synchronized` — когда есть составная операция или инвариант из нескольких полей.
> 
> _EN: `volatile` when the operation is inherently atomic and you only need visibility (stop flag, safe publication of a reference). `synchronized` when you have compound actions or multi-field invariants._

> [!question]- Что такое intrinsic lock / монитор?  
> Встроенный лок, который есть у каждого Java-объекта. `synchronized` захватывает монитор указанного объекта на входе и гарантированно отпускает на выходе, в том числе через исключение. Поток, ждущий занятый монитор, находится в состоянии `BLOCKED`.
> 
> _EN: Every Java object has a built-in lock (monitor). `synchronized` acquires it on entry and always releases it on exit, including via exceptions; a thread waiting for a busy monitor is in the `BLOCKED` state._

> [!question]- На каком мониторе синхронизируются метод экземпляра, статический метод, блок?  
> Метод экземпляра — на `this`; `static`-метод — на объекте `Класс.class`; блок — на объекте, указанном в скобках.
> 
> _EN: Instance method — on `this`; static method — on the `Class` object; block — on whatever object you pass in._

> [!question]- Блокируют ли друг друга потоки, синхронизированные на разных объектах?  
> Нет. Взаимное исключение существует только относительно одного и того же монитора. Два экземпляра — два монитора; `synchronized`-метод и `static synchronized` одного класса — тоже разные мониторы (`this` vs `Class`), они выполняются параллельно.
> 
> _EN: No. Mutual exclusion exists only per monitor. Two instances mean two monitors; an instance synchronized method and a static synchronized one of the same class also lock different monitors (`this` vs the `Class` object) and run in parallel._

> [!question]- synchronized даёт только взаимное исключение или ещё и видимость?  
> И то и другое: освобождение монитора happens-before последующего захвата того же монитора, поэтому вошедший поток видит все изменения предыдущего владельца. Помечать поля `volatile` дополнительно не нужно, если все обращения идут под одним монитором.
> 
> _EN: Both. Monitor release happens-before every subsequent acquire of the same monitor, so the next thread sees everything the previous holder wrote. No extra `volatile` is needed if all accesses go through the same monitor._

> [!info]- Follow-up-ловушки  
> «А если писать под `synchronized`, а читать без?» — видимости у читателя нет, happens-before требует пары на одном мониторе.
> 
> «А `synchronized` дорогой?» — неконкурентный дёшев (thin lock через CAS), дорогим его делает contention и парковка потоков через ОС.
> 
> Слабый ответ, который валит кандидата: «`volatile` делает переменную потокобезопасной» — без уточнения, что именно он гарантирует и на каких операциях ломается.
