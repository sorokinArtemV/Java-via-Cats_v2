
### Зачем это вообще

Практически любой реальный бэкенд — конкурентный, даже если вы не написали ни строчки про потоки. Веб-сервер обрабатывает сотни запросов одновременно; каждый запрос ждёт базу, внешний API, диск — и было бы расточительно, если бы всё приложение стояло, пока один запрос ждёт ответа от Postgres. Конкурентность решает три практические задачи: не простаивать на I/O (пока один поток ждёт сеть, другой работает), использовать все ядра процессора (их давно не одно), и держать приложение отзывчивым (долгая задача в фоне не должна замораживать всё остальное).

Плата за это конкретная: как только один и тот же участок памяти трогают несколько потоков, перестают работать вещи, которые в однопоточном коде железобетонны. Весь этот блок глав — про то, что именно ломается и какими инструментами это чинится. Эта глава — фундамент: модель Java, три независимые поломки и одна гарантия, на которую можно опираться.

### Модель Java: потоки и блокирующий код

Java исторически построена вокруг простой модели: поток на задачу, блокирующие вызовы, синхронный на вид код. Поток берёт задачу и выполняет её сверху вниз. Нужно подождать ответа от базы? Поток блокируется — просто стоит на вызове, пока данные не придут. Это не авария и не антипаттерн, это норма языка: почти весь JDK (JDBC, старый I/O, HttpURLConnection) написан в блокирующем стиле, и код читается как обычный последовательный сценарий.

#### Почему здесь нет async/await, как в C#

Это первый вопрос, который возникает у дотнетчика, и на него есть прямой ответ.

Проблема у обеих платформ одна и та же: поток операционной системы — дорогой ресурс (мегабайты стека, дорогое переключение контекста), и держать тысячи ОС-потоков заблокированными на ожидании I/O — расточительно. Дальше пути разошлись.

.NET пошёл через async/await: функция, которая ждёт I/O, становится асинхронной, компилятор разворачивает её в конечный автомат, поток на время ожидания освобождается. Цена — окраска функций: `async` заражает всю цепочку вызовов вверх (async-метод можно нормально ждать только из async-метода), и в кодовой базе возникают два мира — синхронный и асинхронный, с болезненной границей между ними.

Java пошла другим путём. Очень долго ответом были пулы потоков и `Future`: потоков ограниченное число, они переиспользуются, а блокировки просто терпели. А в Java 21 появились виртуальные потоки (JEP 444): дешёвые потоки, которыми управляет не ОС, а JVM. Их можно создавать миллионами, и когда виртуальный поток блокируется на I/O, JVM снимает его с несущего ОС-потока и ставит туда другой. Результат: вы пишете тот же обычный блокирующий, синхронно выглядящий код — а масштабируется он как асинхронный. Без окраски функций, без двух миров.

Поэтому «в Java нет async/await» — не отставание, а другое решение той же задачи: .NET перестроил код под дешёвое ожидание, Java перестроила потоки. Детали виртуальных потоков — в отдельной главе; здесь важна только эта ориентация.

> [!tip] 💡 Из C#  
> Ближайшая интуиция — представьте, что каждый `Task` магически стал дешёвым потоком, и `await` больше не нужен: можно просто блокироваться. `Thread` в Java ≈ `Thread` в .NET; аналога async/await в языке нет намеренно.

### Конкурентность ≠ параллелизм

Два слова, которые постоянно смешивают. Конкурентность — про структуру: программа устроена так, что управляет многими задачами одновременно (они перемежаются, ждут, продолжаются). Параллелизм — про исполнение: несколько задач физически выполняются в один момент времени на разных ядрах. Один официант, жонглирующий десятью столиками, — конкурентность. Десять официантов — параллелизм. Конкурентная программа может работать и на одном ядре (задачи чередуются); параллельная требует нескольких. В Java оба режима строятся на одних и тех же потоках, поэтому дальше мы почти не будем их различать — но на собесе различать надо.

### Что ломается: три независимые поломки

Как только два потока трогают одну и ту же память, ломаются три разные вещи. Это критично: у них разные причины и разные лекарства, и главная ошибка учебников — свалить всё в кучу под вывеской «многопоточность сложная». Разведём жёстко.

#### Поломка №1: атомарность

`counter++` выглядит как одна операция. На самом деле это три: прочитать значение, прибавить единицу, записать обратно (read–modify–write). Между любыми двумя из них планировщик может передать управление другому потоку.

```java
public class LostUpdates {

    static int counter = 0;

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 100_000; i++) {
                counter++;                     // read → modify → write
            }
        };

        Thread first = new Thread(task);
        Thread second = new Thread(task);
        first.start();
        second.start();
        first.join();
        second.join();

        System.out.println(counter);           // ожидаем 200000
    }
}
```

Ожидаем 200000. Реальные результаты нескольких запусков на JDK 21: 200000, 163365, 100000. Иногда всё сходится — и это худшее свойство таких багов: код обычно работает, а падает под нагрузкой в проде. Механика потери:

![[Pasted image 20260704142925.png]]

Оба потока прочитали 0, оба записали 1. Второй инкремент растворился — и никакого исключения, никакого следа в логах. Лекарства от этой поломки — атомарные операции (`AtomicInteger` с CAS) и блокировки (`synchronized`, `Lock`) — впереди, в главах про atomics и локи.

#### Поломка №2: видимость

Эта поломка контринтуитивнее первой, и именно её дотнетчики недооценивают сильнее всего — потому что `lock` и async/await в C# прячут её от вас всю карьеру. Остановимся и разжуём.

Интуиция говорит: переменная — это ячейка памяти; я записал — все прочитали новое значение. В многопоточном мире это неправда. Каждое ядро процессора работает со своим кэшем, а JIT-компилятор вдобавок вправе держать значение в регистре или вообще вынести чтение из цикла. Запись, сделанная одним потоком, физически произошла — но другой поток её не видит: он смотрит в свою «записную книжку», а не в общую память, и никто не обязан ему сообщать, что данные устарели.

```java
public class Visibility {

    static boolean ready = false;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (!ready) {
                // крутимся, ждём сигнала
            }
            System.out.println("Увидел ready = true, выхожу");
        });
        worker.start();

        Thread.sleep(1000);
        ready = true;                       // сигнал отправлен...
        System.out.println("main: записал ready = true");
    }
}
```

На JDK 21 (HotSpot) эта программа печатает `main: записал ready = true` — и зависает навсегда. Проверено запуском: worker так и не увидел новое значение. JIT видит, что внутри цикла `ready` не меняется, и легально оптимизирует `while (!ready)` в `while (true)` — чтение выносится из цикла. Переменная изменилась, но не для всех:

![[Pasted image 20260704142943.png]]

Ключевой сдвиг в голове: без явной синхронизации у записи нет гарантированного «времени доставки» другим потокам — вообще никакого, даже «через секунду». Лекарства — `volatile` и `synchronized` — назовём чуть ниже, разберём в следующей главе.

#### Поломка №3: переупорядочивание

Третья беда бьёт по последней опоре интуиции — по порядку строк в собственном коде. Вы написали «сначала A, потом B», но JIT-компилятор и процессор вправе переставить независимые (с их точки зрения) операции, если это ускоряет исполнение. Внутри одного потока подмену не заметить — результат для этого потока всегда как будто по порядку. А вот наблюдатель из другого потока может увидеть эффекты в перепутанном порядке:

```java
// Общие поля:      int data = 0;   boolean published = false;

// Поток-писатель:
data = 42;
published = true;

// Поток-читатель:
if (published) {
    System.out.println(data);   // может напечатать 0 (!)
}
```

Кажется невозможным: раз `published == true`, значит, `data = 42` уже выполнилось — оно же выше по коду. Но без синхронизации это рассуждение не имеет силы: записи могут быть переставлены писателем (JIT) или увидены в другом порядке читателем (кэши процессора; на x86 модель памяти строгая и такое встречается реже, на ARM — регулярно, но JIT вправе переставить на любой платформе). Спецификация Java прямо разрешает читателю увидеть `published == true` и `data == 0` — это легальный исход для несинхронизированного кода.

### happens-before: единственная гарантия, которая у вас есть

Теперь соберём три поломки в одну рамку. У всех трёх общий корень: между действиями разных потоков по умолчанию нет никакого гарантированного порядка. Не «порядок обычно правильный, но иногда сбоит» — а нет вообще. То, что ваш код сегодня работает, — везение конкретного железа, планировщика и фазы JIT, и полагаться на него нельзя.

Java Memory Model (JMM) вводит для этого одно центральное понятие — отношение happens-before. Читается так: если между действием X и действием Y установлено happens-before, то Y гарантированно видит все эффекты X — записи доехали, порядок соблюдён. Всё. Это единственная официальная стрелка порядка между потоками; любой порядок, не подкреплённый happens-before, — иллюзия.

Откуда берутся эти стрелки? Их создают конкретные конструкции языка: запись в volatile-поле happens-before последующего чтения этого поля; выход из synchronized-блока happens-before входа в него на том же мониторе; `thread.start()` happens-before первого действия потока; последнее действие потока happens-before возврата из `join()`. Полный список и механику разберём в следующей главе — здесь важен сам принцип:

Обратите внимание на последнюю строку под диаграммой — это главное свойство happens-before: стрелка гарантирует видимость не только самой синхронизирующей операции, но и всего, что писатель сделал до неё. Именно на этом стержне держатся все последующие главы: каждый инструмент (`synchronized`, `volatile`, atomics, `ExecutorService`, concurrent-коллекции) полезен ровно настолько, насколько он умеет создавать happens-before.

### Два слова, которые вы уже встречали

Пока — только позиционирование, детали в следующей главе.

`synchronized` — блок или метод, в который в один момент времени может войти только один поток (по данному монитору). Закрывает все три беды сразу для защищённого кода: атомарность (никто не влезет в середину), видимость и порядок (вход/выход создают happens-before).

`volatile` — модификатор поля. Закрывает видимость и порядок (запись happens-before чтения), но НЕ атомарность. `volatile int counter` не спасёт `counter++` — это по-прежнему три операции, и инкременты будут теряться точно так же. Это классическая ловушка на собесе, зафиксируйте прямо сейчас: `volatile` — про «увидеть», не про «не потерять».

> [!tip] 💡 Из C#  
> Все три беды существуют и в .NET — CLR-код исполняется на тех же ядрах с теми же кэшами. Просто вы от них были заслонены: `lock` даёт и атомарность, и барьеры памяти; `Interlocked` — атомарность; async/await вставляет синхронизацию на границах продолжений. Беды не отсутствовали — их прятала инфраструктура.

> [!warning] ⚠️ Ложный друг  
> Java `volatile` ≠ C# `volatile`. Java-версия даёт строгую семантику happens-before по JMM (включая запрет переупорядочивания вокруг неё в обе стороны для последующих чтений); C#-volatile даёт более слабые half-fences с известными тонкостями. Интуицию «один в один» не переносить — учите Java-семантику с нуля.

> [!example] 🔬 Глубже / на синьора  
> **JMM — это контракт, а не описание железа.** Java Memory Model (JLS §17.4, финализирована в JSR-133, Java 5) не рассказывает, как работают кэши конкретного процессора. Она определяет договор между вашим кодом и связкой «JIT + CPU»: какие исходы исполнения легальны, а какие — нет. Всё, что контрактом не запрещено, — разрешено, и оптимизаторы этим пользуются агрессивно. Отсюда важное следствие: «у меня на машине воспроизводится/не воспроизводится» ничего не доказывает — рассуждать о корректности конкурентного кода можно только в терминах контракта, не эксперимента.
> 
> **Барьеры памяти — как happens-before приземляется на железо.** Когда вы пишете в volatile-поле, JIT вставляет в машинный код барьер (memory fence) — инструкцию, запрещающую процессору переносить операции через неё и заставляющую опубликовать/подтянуть значения. `synchronized` делает то же на входе и выходе из блока. То есть happens-before — абстракция уровня языка, а барьеры — её цена на уровне инструкций; поэтому синхронизация не бесплатна, и поэтому её не ставят «на всякий случай на всё».
> 
> **Почему компилятор вправе не показать вашу запись.** В примере `Visibility` JIT превратил `while (!ready)` в `while (true)` — и это не баг, а разрешённая контрактом оптимизация (hoisting): раз между записью `ready` в main и чтением в worker нет ни одной happens-before-связи, JMM позволяет читателю бесконечно видеть старое значение. Однопоточная семантика сохранена (в пределах одного потока всё «как написано» — as-if-serial), а межпоточных обязательств вы не оформили. Компилятор не ломает ваш код — он выполняет ровно тот контракт, который вы подписали, не глядя.

### На собесе

> [!question]- Что такое race condition?  
> Ситуация, когда корректность результата зависит от порядка/тайминга выполнения потоков, который не контролируется. Классика — lost update: два потока делают `counter++`, оба читают старое значение, часть инкрементов теряется. Опасна тем, что проявляется недетерминированно — тесты проходят, прод падает.
> 
> **EN:** A race condition is when the correctness of the result depends on the uncontrolled timing or interleaving of threads — e.g., two threads doing counter++ both read the same old value and one increment is lost.

> [!question]- Чем атомарность отличается от видимости?  
> Это две независимые проблемы. Атомарность — про неделимость операции: `counter++` состоит из read–modify–write, и другой поток может влезть между шагами. Видимость — про доставку: запись потока физически произошла, но другой поток продолжает видеть старое значение из-за кэшей и JIT-оптимизаций. Лекарства разные: атомарность дают atomics/локи, видимость — `volatile`/`synchronized`.
> 
> **EN:** Atomicity is about indivisibility of a compound operation (read-modify-write can be interleaved); visibility is about whether one thread's write is ever observed by another (it may stay in a cache/register). Different problems, different fixes.

> [!question]- Что гарантирует volatile и чего не гарантирует?  
> Гарантирует видимость и порядок: запись в volatile-поле happens-before последующего чтения, плюс тянет за собой видимость всего, что писатель сделал до этой записи. НЕ гарантирует атомарность: `volatile int` не чинит `counter++` — это по-прежнему три операции.
> 
> **EN:** volatile guarantees visibility and ordering (write happens-before subsequent read, publishing everything written before it), but NOT atomicity — counter++ on a volatile field still loses updates.

> [!question]- Что такое happens-before?  
> Отношение из Java Memory Model: если X happens-before Y, то Y гарантированно видит все эффекты X. Это единственная гарантия порядка между потоками; создаётся конкретными конструкциями — `volatile`, `synchronized`, `Thread.start()`/`join()` и т.д. Любой межпоточный порядок без happens-before — совпадение, на которое нельзя полагаться.
> 
> **EN:** A JMM relation: if X happens-before Y, Y is guaranteed to see all effects of X. It's the only cross-thread ordering guarantee; it's established by constructs like volatile, synchronized, start()/join().

> [!question]- Поток vs процесс?  
> Процесс — изолированная единица со своим адресным пространством; процессы не видят память друг друга. Поток — единица исполнения внутри процесса; потоки одного процесса разделяют heap, поэтому дешевле в создании и общении — но именно общая память и порождает все три беды этой главы.
> 
> **EN:** A process has its own isolated address space; threads live inside a process and share its heap — cheaper to create and communicate, but shared memory is exactly what causes races, visibility and reordering issues.

> [!question]- Почему в Java нет async/await, как в C#?  
> Обе платформы решали одну проблему — дорогие ОС-потоки, блокирующиеся на I/O. .NET перестроил код: async/await освобождает поток на ожидании, ценой окраски функций. Java перестроила потоки: исторически — пулы + `Future`, а с Java 21 — виртуальные потоки (JEP 444), которые JVM снимает с несущего ОС-потока при блокировке. Итог тот же (масштабирование), но код остаётся обычным блокирующим, без разделения на sync/async-миры.
> 
> **EN:** Both solved the same problem — expensive OS threads blocked on I/O. .NET restructured the code (async/await, at the cost of function coloring); Java restructured the threads (thread pools + Future historically, and virtual threads since Java 21, JEP 444) so you keep writing plain blocking code that scales like async.
