В прошлом уроке мы научились запускать поток через `start()` и ждать его завершения через `join()`. И мельком заметили странность: поток, который уже отработал, нельзя запустить второй раз — попытка кончается исключением. Объект жив, ссылка на него есть, метод `start()` на месте — а вызвать нельзя. Причина в том, что у потока есть жизненный цикл: маршрут из фиксированных состояний, по которому он движется только вперёд. Из конечной точки этого маршрута пути назад нет.

Этот маршрут — карта, по которой мы будем ориентироваться весь оставшийся курс. Когда в главе про `synchronized` поток «встанет на локе», когда в главе про `wait/notify` он «уснёт в ожидании сигнала», когда в thread dump ты увидишь сотню потоков с пометкой `WAITING` — всё это точки на карте, которую мы сейчас нарисуем. А во второй половине урока разберём вопрос, который из этой карты вырастает сам: если поток нельзя убить, то как его остановить?

### Часть 1. Жизненный цикл потока

**Ментальная модель:** поток — это односторонняя дорога из шести состояний: он рождается (`NEW`), запускается (`RUNNABLE`), может временно останавливаться в трёх видах ожидания (`BLOCKED`, `WAITING`, `TIMED_WAITING`), и умирает навсегда (`TERMINATED`).

Текущее состояние потока в любой момент можно спросить методом `getState()` — он возвращает значение enum'а `Thread.State`. Пройдём по всем шести, но не сухими определениями, а через ситуацию: _в какой момент жизни поток тут оказывается_.

**`NEW`** — объект `Thread` создан, но `start()` ещё не вызван. Это просто Java-объект в куче: никакого потока операционной системы за ним пока нет. Он может сидеть в этом состоянии сколько угодно — хоть вечно, если `start()` так и не позовут.

**`RUNNABLE`** — `start()` вызван, поток живёт и _может_ исполняться. Здесь важный нюанс, на котором спотыкаются почти все: интуитивно ждёшь два отдельных состояния — «исполняется прямо сейчас на ядре CPU» и «готов, но ждёт своей очереди у планировщика ОС». В Java их **нет**. Оба случая — это одно состояние `RUNNABLE`. Отдельного `RUNNING` не существует: у тебя 4 ядра и 200 потоков в состоянии `RUNNABLE` — физически бегут максимум 4, остальные стоят в очереди планировщика, но с точки зрения `Thread.State` все 200 неразличимы. Почему так — разберём в блоке 🔬.

**`BLOCKED`** — поток пытается войти в `synchronized`-секцию, а монитор занят другим потоком. Он стоит у закрытой двери и ждёт, пока её освободят. Ключевое: `BLOCKED` — это _только_ про ожидание монитора, ни про что другое. Сам `synchronized` мы подробно разберём в главе про синхронизацию — сейчас достаточно знать, что такая точка на карте есть.

**`WAITING`** — поток бессрочно ждёт действия _другого потока_. Сюда попадают после `wait()` без таймаута (ждёт, пока кто-то позовёт `notify()`) и после `join()` без таймаута (ждёт, пока другой поток завершится). Разница с `BLOCKED` принципиальна: `BLOCKED` ждёт освобождения _монитора_, `WAITING` ждёт _явного сигнала или события_ от другого потока. На собеседовании это спрашивают именно как пару.

**`TIMED_WAITING`** — то же ожидание, но с будильником: `sleep(ms)`, `wait(ms)`, `join(ms)`. Поток проснётся либо от события, либо по истечении таймаута — смотря что случится раньше.

**`TERMINATED`** — метод `run()` завершился (нормально или с исключением). Поток мёртв навсегда. Объект `Thread` всё ещё существует как обычный объект в куче — можно вызвать `getState()`, прочитать имя, — но нить исполнения ОС за ним уже уничтожена. И вот ответ на загадку из вступления: из `TERMINATED` нет ни одного исходящего перехода, поэтому повторный `start()` невозможен в принципе.

Посмотрим на состояния вживую — код проверен на JDK 21:

```java
public class StatesDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            try {
                Thread.sleep(2000);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        System.out.println("после new:      " + worker.getState()); // NEW
        worker.start();
        System.out.println("после start():  " + worker.getState()); // RUNNABLE
        Thread.sleep(100); // даём worker дойти до sleep
        System.out.println("во время sleep: " + worker.getState()); // TIMED_WAITING
        worker.join();
        System.out.println("после join():   " + worker.getState()); // TERMINATED
    }
}
```

```
после new:      NEW
после start():  RUNNABLE
во время sleep: TIMED_WAITING
после join():   TERMINATED
```

А вот и подтверждение одноразовости — повторный `start()` на завершённом потоке:

```java
Thread t = new Thread(() -> System.out.println("working"));
t.start();
t.join();
System.out.println(t.getState()); // TERMINATED
t.start(); // Exception in thread "main" java.lang.IllegalThreadStateException
```

Строго говоря, `IllegalThreadStateException` прилетит при повторном `start()` из _любого_ состояния, кроме `NEW` — `start()` разрешён ровно один раз за жизнь потока.

Никакой поток не переходит, например, из `WAITING` сразу в `TERMINATED` — сначала он проснётся, снова станет `RUNNABLE`, доисполнит `run()` и лишь тогда умрёт.

> [!tip] 💡 Из C#  
> Состояния `Thread.State` концептуально похожи на `System.Threading.ThreadState`, но набор и семантика различаются. В .NET `ThreadState` — это `[Flags]`-enum с комбинациями (`Running`, `WaitSleepJoin`, `Stopped`, плюс отдельные флаги `Background`, `Suspended`), и все виды ожидания слеплены в один `WaitSleepJoin`. Java-модель чище: шесть взаимоисключающих состояний, и ожидание монитора (`BLOCKED`) отделено от ожидания события (`WAITING`/`TIMED_WAITING`). Это различие не косметика — в thread dump оно позволяет сразу увидеть, потоки _дерутся за лок_ или _ждут работу_.

### Часть 2. Прерывание — как вежливо остановить поток

Теперь главный практический вопрос урока. Поток запущен и что-то долго делает: спит, ждёт данных, крутит вычислительный цикл. Приложение завершается, или задача стала не нужна, или пользователь нажал «Отмена» — как этот поток остановить?

Первое, что стоит вбить намертво: **в Java нет способа насильно убить поток**. Метод `Thread.stop()` существует, но deprecated с незапамятных времён и с Java 20 при вызове просто бросает `UnsupportedOperationException` — он не работает вообще ([JDK-8289610](https://bugs.openjdk.org/browse/JDK-8289610), Java 20). Причина фундаментальная: `stop()` убивал поток _в произвольной точке исполнения_ — посреди записи в структуру данных, посреди обновления двух связанных полей. Объекты оставались в разрушенном, наполовину изменённом состоянии, а мониторы при этом освобождались — и другие потоки тут же видели этот мусор. Починить это невозможно в принципе, поэтому от идеи насильной остановки отказались целиком.

Вместо неё Java предлагает **кооперативный механизм** — прерывание.

**Ментальная модель:** `interrupt()` не останавливает поток — он _просит_ его остановиться, выставляя флаг. Поток сам обязан этот флаг проверять и сам решает, что делать. Это не выключатель, а вежливая просьба, которую можно и проигнорировать.

Вокруг этой мысли построено всё остальное. Разберём механику по частям.

#### Три метода и один флаг

У каждого потока есть внутренний булев флаг — _interrupt status_. С ним работают три метода, и их легко перепутать:

- `t.interrupt()` — выставить флаг у потока `t`. Больше ничего: никакой магии, никакой остановки.
- `t.isInterrupted()` — прочитать флаг, **не трогая его**. Можно спрашивать хоть сто раз подряд — ответ не изменится.
- `Thread.interrupted()` — статический метод: прочитать флаг текущего потока **и сбросить его**. Второй вызов подряд вернёт `false`.

Разница между последними двумя — классический вопрос на собеседовании, и её проще всего увидеть в коде (проверено на JDK 21):

```java
Thread self = Thread.currentThread();
self.interrupt(); // выставили флаг самому себе

System.out.println(self.isInterrupted());   // true
System.out.println(self.isInterrupted());   // true  — чтение флаг не сбрасывает

System.out.println(Thread.interrupted());   // true  — прочитал И СБРОСИЛ
System.out.println(Thread.interrupted());   // false — флага уже нет
System.out.println(self.isInterrupted());   // false
```

Зачем вообще существует сбрасывающий вариант? Он для кода, который _сам обрабатывает_ прерывание: «я увидел просьбу, я на неё отреагирую — считаю её обработанной, флаг снимаю». В повседневном коде тебе почти всегда нужен `isInterrupted()`; случайно написанный `Thread.interrupted()` тихо стирает информацию о прерывании — к этой грабле ещё вернёмся.

#### Поток в вычислительном цикле

Если поток не спит и не ждёт, а честно молотит цикл — просьбу об остановке ему никто в лоб не доставит. Он должен проверять флаг сам, периодически:

```java
Thread worker = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        doChunkOfWork(); // очередной кусок работы
    }
    // увидели флаг — прибираемся и выходим из run() по-человечески
});
```

Обрати внимание на архитектуру: цикл делает _кусок_ работы, потом проверяет флаг, потом следующий кусок. Чем крупнее кусок, тем дольше поток будет реагировать на просьбу. Это и есть кооперативность в чистом виде — отзывчивость к отмене встроена в структуру самого кода, а не навязана снаружи.

#### `InterruptedException` и контракт прерывания

А что, если поток в момент прерывания не крутит цикл, а _спит_ или _ждёт_? Проверять флаг он физически не может — он не исполняется. Для этого случая блокирующие методы — `sleep()`, `join()`, `wait()`, а дальше в блоке 24 и `BlockingQueue.take()`, и `Future.get()` — заключили с прерыванием контракт из двух пунктов:

1. Если поток прерывают, пока он стоит в таком методе (или флаг уже был выставлен на входе), метод **немедленно просыпается и бросает `InterruptedException`**.
2. При этом метод **сбрасывает флаг прерывания**.

Второй пункт — важнейший нюанс всей темы, и он контринтуитивен. Логика контракта такая: исключение — это и _есть_ доставка просьбы. Информация о прерывании не потерялась — она превратилась из флага в `InterruptedException` у тебя в руках, и теперь **ты** отвечаешь за её судьбу. Проверим на живом коде (JDK 21):

```java
Thread worker = new Thread(() -> {
    try {
        Thread.sleep(10_000);
    } catch (InterruptedException e) {
        System.out.println("в catch, isInterrupted(): "
                + Thread.currentThread().isInterrupted()); // false!
    }
});
worker.start();
Thread.sleep(200);   // даём worker дойти до sleep
worker.interrupt();  // просим остановиться
```

```
в catch, isInterrupted(): false
```

Флаг выставили — а внутри `catch` он уже `false`. И вот отсюда растёт **грабля №1 всего Java-многопоточья**.

#### Проглоченное прерывание — главный антипаттерн

Компилятор заставляет обработать checked-исключение `InterruptedException`, и уставший разработчик пишет:

```java
// ТАК НЕЛЬЗЯ — прерывание проглочено
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // ничего, "нам просто нужна пауза"
}
```

Разбери, что здесь произошло, по шагам. Кто-то вызвал `interrupt()` — попросил поток остановиться. `sleep()` честно доставил просьбу исключением и, по контракту, сбросил флаг. Пустой `catch` исключение выбросил, флаг не восстановил. Итог: **информация о том, что поток просили остановиться, стёрта из вселенной**. Флаг — `false`, исключение — проглочено. Цикл уровнем выше проверяет `isInterrupted()`, получает `false` и спокойно молотит дальше. Поток стал неостанавливаемым — и никакое место в коде об этом даже не узнает.

#### Два правильных ответа

Правильных реакции на `InterruptedException` ровно две, и обе сводятся к одному принципу: _просьба о прерывании не должна потеряться_.

**Вариант 1 — пробросить выше.** Если сигнатура позволяет, честно объяви `throws InterruptedException` и отдай решение вызывающему. Он ближе к контексту задачи и лучше знает, что значит «остановиться» на его уровне:

```java
void pollUntilReady() throws InterruptedException {
    while (!ready()) {
        Thread.sleep(200); // прервали — исключение уйдёт вызывающему как есть
    }
}
```

**Вариант 2 — восстановить флаг.** Если пробросить нельзя (сигнатура зафиксирована — `Runnable.run()`, метод интерфейса, callback), поймай исключение и верни флаг на место: `Thread.currentThread().interrupt()`. Тогда код выше по стеку, проверяющий флаг, всё-таки узнает о просьбе. Этот фрагмент нужно знать наизусть — проверен на JDK 21:

```java
Thread worker = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        doChunkOfWork();
        pause(100);
    }
    System.out.println("увидел прерывание, выхожу аккуратно");
});

static void pause(long millis) {
    try {
        Thread.sleep(millis);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt(); // ВОССТАНОВИЛИ флаг — цикл выше его увидит
    }
}
```

Проследи цепочку целиком: `interrupt()` извне → `sleep()` бросает исключение и сбрасывает флаг → `catch` восстанавливает флаг → `while` на следующей итерации видит `isInterrupted() == true` → поток аккуратно завершает `run()` → `TERMINATED`. Каждое звено на своём месте, просьба дошла до адресата.

> [!tip] 💡 Из C#  
> Идея кооперативной отмены тебе знакома по `CancellationToken`: там тоже никто никого не убивает — задача сама опрашивает `IsCancellationRequested` или ловит `OperationCanceledException`. Java-прерывание концептуально о том же. Но здесь ложный друг: `Thread.Interrupt()` в .NET _существует_ (и тоже бросает `ThreadInterruptedException` из `WaitSleepJoin`), однако в реальной практике им почти никогда не пользуются — идиоматика .NET целиком построена на токенах. В Java всё наоборот: **именно `interrupt()`/`InterruptedException` — центральный, повседневный механизм отмены**, на нём стоит весь `java.util.concurrent`. Не переноси рефлекс «у нас для этого токены» — токенов здесь нет, а контракт (сброс флага при исключении, обязательное восстановление) надо знать назубок.

> [!example]- 🔬 Глубже / на синьора  
> **Почему `stop()` и `suspend()` неисправимы.** `stop()` асинхронно вбрасывал `ThreadDeath` в произвольную точку исполнения и при раскрутке стека освобождал все мониторы потока — инварианты, которые эти мониторы охраняли, оставались нарушенными, и повреждённое состояние мгновенно становилось видно остальным потокам. `suspend()` — зеркальная беда: он замораживал поток, _не освобождая_ мониторы, что делало deadlock почти гарантированным (любой поток, которому нужен тот же лок, вставал навсегда). Оба дефекта — свойства самой идеи асинхронной остановки, а не бага в реализации; отсюда кооперативная модель как единственно безопасная. С Java 20 `stop()` окончательно бросает `UnsupportedOperationException` (JDK-8289610), `suspend()`/`resume()` — с Java 21 (JDK-8249627).
> 
> **Почему `RUNNABLE` покрывает и ready, и running.** Java-поток (до виртуальных потоков) отображён 1:1 на поток ОС, и решение «кто прямо сейчас на ядре» принимает планировщик ОС — JVM в это решение не вмешивается и на этом уровне его не видит. Состояние `RUNNABLE` означает «с точки зрения JVM исполнению ничего не мешает»; дальше — юрисдикция операционной системы. Javadoc `Thread.State.RUNNABLE` прямо оговаривает, что поток в этом состоянии может ждать других ресурсов ОС, например процессора.
> 
> **`BLOCKED` не реагирует на `interrupt()`.** Тонкий момент, который стоит увидеть кодом (проверено на JDK 21): поток, вставший на входе в `synchronized`, на прерывание не отзовётся — флаг выставится, но поток продолжит стоять, пока монитор не освободится:
> 
> java
> 
> ```java
> // holder держит LOCK 2 секунды; waiter встал на synchronized(LOCK)
> System.out.println(waiter.getState()); // BLOCKED
> waiter.interrupt();
> Thread.sleep(100);
> System.out.println(waiter.getState()); // всё ещё BLOCKED — просьба не разбудила
> // ...когда holder отпустит лок, waiter войдёт в секцию,
> // а isInterrupted() внутри вернёт true — флаг никуда не делся
> ```
> 
> Прерывание будит только тех, кто ждёт _прерываемым_ способом (`sleep`, `wait`, `join` и блокирующие методы `java.util.concurrent`). Ожидание intrinsic-монитора — непрерываемое. Если нужна возможность отменить ожидание лока — для этого существует `ReentrantLock.lockInterruptibly()`, мы дошли до него в главе 5 серии по конкурентности (в нумерации курса — блок 24).
> 
> **Прерывание и I/O.** Поток, застрявший в классическом блокирующем I/O (`InputStream.read()` на сокете), в общем случае на `interrupt()` тоже не отреагирует — флаг выставится, но `read()` продолжит висеть. Исключение — каналы NIO: `InterruptibleChannel` при прерывании закрывается и бросает `ClosedByInterruptException`. Это ещё одна причина знать пределы механизма: interrupt — не универсальная отмена, а протокол, который работает только там, где код ему следует.

> 1. **Пустой `catch (InterruptedException e) {}`** — проглоченное прерывание: флаг уже сброшен исключением, ты его не восстановил, вышестоящий код никогда не узнает о просьбе. Либо `throws`, либо `Thread.currentThread().interrupt()`.
> 2. **Путаница `isInterrupted()` и `Thread.interrupted()`** — первый только читает флаг, второй читает **и сбрасывает**. Случайный статический вызов в проверке цикла тихо стирает прерывание.
> 3. **Вера, что `interrupt()` немедленно останавливает поток.** Нет: он лишь выставляет флаг и будит прерываемые ожидания. Поток без проверок флага проигнорирует просьбу и доработает до конца.
> 4. **Ожидание состояния `RUNNING`.** Его нет: и «исполняется на ядре», и «готов, ждёт планировщик» — это одно состояние `RUNNABLE`.
> 5. **Попытка прервать поток в `BLOCKED`.** Ожидание входа в `synchronized` непрерываемое: флаг выставится, но поток не сдвинется, пока монитор не освободится. Нужна отменяемость — смотри в сторону `lockInterruptibly()`.
> 6. **Повторный `start()`** на потоке в любом состоянии, кроме `NEW`, — `IllegalThreadStateException`. Поток одноразовый; нужна «переиспользуемость» — это про пулы потоков (блок 24).

### На собесе

> [!question]- Перечисли состояния потока в Java. Что означает каждое?  
> **RU:** Шесть состояний enum `Thread.State`. `NEW` — объект создан, `start()` не вызван. `RUNNABLE` — поток исполняется или готов исполняться; Java не различает «бежит на ядре» и «ждёт очереди у планировщика ОС». `BLOCKED` — ждёт входа в `synchronized`-секцию, занятую другим потоком. `WAITING` — бессрочно ждёт действия другого потока (`wait()`, `join()` без таймаута). `TIMED_WAITING` — то же с таймаутом (`sleep(ms)`, `wait(ms)`, `join(ms)`). `TERMINATED` — `run()` завершился, поток мёртв, повторный `start()` невозможен.  
> **EN:** Six states of the `Thread.State` enum. `NEW` — created, not started. `RUNNABLE` — running or ready to run; Java doesn't distinguish "on a CPU core" from "waiting in the OS scheduler queue". `BLOCKED` — waiting to enter a `synchronized` section held by another thread. `WAITING` — waiting indefinitely for another thread's action (`wait()`, `join()` with no timeout). `TIMED_WAITING` — same but with a timeout. `TERMINATED` — `run()` has finished; the thread is dead and can't be restarted.  
> **Follow-up-ловушка:** «А где состояние RUNNING?» — его нет, это и проверяют. Слабый ответ — начать его выдумывать.

> [!question]- Чем `BLOCKED` отличается от `WAITING`?  
> **RU:** `BLOCKED` — поток ждёт освобождения _монитора_, чтобы войти в `synchronized`; из этого ожидания его выведет только освобождение лока, `interrupt()` его не разбудит. `WAITING` — поток ждёт _события от другого потока_ (`notify()`, завершение потока при `join()`) и стоит в прерываемом ожидании: `interrupt()` разбудит его через `InterruptedException`. В thread dump это различие сразу показывает, дерутся ли потоки за лок или ждут работу.  
> **EN:** `BLOCKED` means waiting for a _monitor_ to enter a `synchronized` block — only the lock being released ends it; `interrupt()` won't wake it. `WAITING` means waiting for an _explicit event from another thread_ (`notify()`, thread completion for `join()`) and is interruptible — `interrupt()` wakes it via `InterruptedException`. In a thread dump this instantly tells you whether threads are contending for a lock or idle waiting for work.

> [!question]- Как правильно остановить поток и почему нельзя насильно?  
> **RU:** Только кооперативно: снаружи вызываем `interrupt()`, а сам поток периодически проверяет `Thread.currentThread().isInterrupted()` в цикле и корректно завершает `run()`; блокирующие вызовы (`sleep`, `wait`, `join`) отзываются на прерывание через `InterruptedException`. Насильно нельзя, потому что `Thread.stop()` убивал поток в произвольной точке, освобождая мониторы и оставляя объекты в наполовину изменённом состоянии, видимом другим потокам; починить это невозможно, и с Java 20 `stop()` просто бросает `UnsupportedOperationException`.  
> **EN:** Only cooperatively: call `interrupt()` from outside, while the thread itself periodically checks `Thread.currentThread().isInterrupted()` and exits `run()` cleanly; blocking calls respond to interruption with `InterruptedException`. Forcible stopping is impossible by design: `Thread.stop()` killed a thread at an arbitrary point, releasing its monitors while leaving objects half-mutated and visible to other threads — unfixable, and since Java 20 `stop()` just throws `UnsupportedOperationException`.  
> **Senior add-on:** ожидание intrinsic-монитора (`BLOCKED`) непрерываемое; отменяемое взятие лока — это `ReentrantLock.lockInterruptibly()`.

> [!question]- Что происходит с флагом прерывания при `InterruptedException` и как правильно обработать это исключение?  
> **RU:** Бросая `InterruptedException`, блокирующий метод **сбрасывает флаг** — в `catch` `isInterrupted()` уже вернёт `false`: информация о прерывании теперь существует только в виде пойманного исключения. Правильных реакций две: пробросить исключение выше (`throws InterruptedException`) или, если сигнатура не позволяет, восстановить флаг — `Thread.currentThread().interrupt()` в `catch`. Пустой `catch` — антипаттерн «проглоченное прерывание»: просьба об остановке теряется навсегда, поток становится неостанавливаемым.  
> **EN:** When a blocking method throws `InterruptedException`, it **clears the interrupt flag** — inside the `catch`, `isInterrupted()` returns `false`; the interruption now exists only as the caught exception. Two correct responses: propagate it (`throws InterruptedException`), or if the signature doesn't allow that, restore the flag with `Thread.currentThread().interrupt()` in the `catch`. An empty `catch` is the "swallowed interrupt" antipattern: the stop request is lost forever.  
> **Follow-up-ловушка:** «Зачем восстанавливать флаг, если исключение уже поймано?» — затем, что о прерывании должен узнать код _выше по стеку_ (цикл, пул потоков), который проверяет флаг, а не ловил это исключение.

> [!question]- В чём разница между `isInterrupted()` и `Thread.interrupted()`?  
> **RU:** `isInterrupted()` — инстанс-метод, читает флаг конкретного потока и **не сбрасывает** его. `Thread.interrupted()` — статический, работает только с _текущим_ потоком, читает флаг **и сбрасывает**: два вызова подряд после прерывания дадут `true`, затем `false`. Для проверки в цикле нужен `isInterrupted()`; сбрасывающий вариант — для кода, который берёт обработку прерывания на себя.  
> **EN:** `isInterrupted()` is an instance method: reads the flag of that thread **without clearing** it. `Thread.interrupted()` is static, applies to the _current_ thread, and reads **and clears** the flag — called twice after an interrupt it returns `true`, then `false`. Use `isInterrupted()` for loop checks; the clearing variant is for code that takes ownership of handling the interruption.  
> **Слабый ответ:** «они одинаковые, просто один статический» — проваливает вопрос, вся суть в сбросе флага.
