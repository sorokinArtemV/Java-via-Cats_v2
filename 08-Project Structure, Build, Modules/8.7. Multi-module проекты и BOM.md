Реальный сервис редко живёт в одном модуле: `service-api` с DTO и интерфейсами, `service-core` с бизнес-логикой, `service-app` с точкой входа. И тут возвращается бытовая боль из прошлого урока в квадрате: пять модулей используют Jackson, версия прописана в пяти `pom.xml`, обновлять — искать по всем пяти, а если один модуль уехал на `2.15`, другой на `2.17`, то при сборке приложения победит одна по nearest wins и кому-то не повезёт. Multi-module решает обе задачи разом: собирает связанные модули одной командой в правильном порядке и держит версии в одном месте. А если ту же идею «версии в одном месте» опубликовать как отдельный артефакт — получится BOM, которым живёт весь Spring Boot.

**Mental model.** Родительский pom даёт две **независимые** вещи: агрегацию (реактор собирает модули в порядке их зависимостей) и наследование (общий конфиг и версии из `dependencyManagement`). BOM — это тот же `dependencyManagement`, но подключаемый композицией, а не наследованием, поэтому их можно импортировать сколько угодно.

### Parent pom

```xml
<groupId>com.example</groupId>
<artifactId>my-service</artifactId>
<version>1.0.0</version>
<packaging>pom</packaging>

<modules>
    <module>my-service-api</module>
    <module>my-service-core</module>
    <module>my-service-app</module>
</modules>
```

`<packaging>pom</packaging>` — родитель не производит jar, он сам по себе только конфигурация. Дочерний модуль ссылается обратно:

```xml
<parent>
    <groupId>com.example</groupId>
    <artifactId>my-service</artifactId>
    <version>1.0.0</version>
</parent>
<artifactId>my-service-core</artifactId>
```

### Два независимых механизма (ключевой водораздел)

Здесь работают **два разных механизма**, которые обычно используют вместе, но путать их нельзя — это прямой собес-вопрос.

**Агрегация** (`<modules>` у родителя) — «собирай нас вместе». Запускаешь `mvn install` в корне — Maven строит **reactor**: читает все модули, выясняет, кто от кого зависит (`core` зависит от `api` → `api` собирается первым), и прогоняет сборку в топологическом порядке. Один заказ — весь проект собран консистентно, без «собрал core против старого api».

**Наследование** (`<parent>` у ребёнка) — «возьми мой конфиг». Ребёнок наследует `groupId`, `version`, properties, конфигурацию плагинов и — ради чего мы сюда шли — **`dependencyManagement`**. Версия Jackson объявляется **один раз** в родителе, все пять модулей пишут зависимость без версии. Обновление Jackson во всём проекте = правка одной строки. Бонусом исчезает целый класс багов «модули разъехались по версиям»: конфликта из прошлого урока просто не возникает, потому что версия физически одна.

![[Pasted image 20260709174008.png]]

> [!tip] 💡 Из C#  
> Агрегация ↔ solution: `.sln` собирает проекты в правильном порядке по project references — прямой аналог реактора. А вот наследования pom в .NET исторически не было; его роль играют `Directory.Build.props` (общий MSBuild-конфиг вниз по дереву) и **Central Package Management** через `Directory.Packages.props` (общие версии пакетов на весь репозиторий). Если ты трогал CPM — parent pom с `dependencyManagement` это ровно оно, только появившееся на ~15 лет раньше.

> [!warning] ⚠️ Подводный камень  
> Агрегация и наследование **разделимы**. Наследование работает без агрегации (корпоративный parent pom — отдельный артефакт из репозитория, у него нет `<modules>` с твоими модулями), и агрегация без наследования (перечислил модули в `<modules>`, но `<parent>` они не указывают). Связь «родитель = агрегатор» — распространённая конвенция, а не требование Maven. На собесе легко перепутать эти два механизма — держи их раздельно.

### BOM: `dependencyManagement` как продукт

Возьми идею «pom, в котором только `dependencyManagement`» и опубликуй как самостоятельный артефакт. Получится **BOM** — Bill of Materials: список согласованных версий целого стека библиотек, проверенных на совместимость друг с другом. Канонический пример — Spring Boot: десятки собственных артефактов плюс сотни третьесторонних, совместимых только в определённых сочетаниях. Подбирать вручную версии `spring-web` + `spring-data` + `jackson` + `micrometer`, которые дружат, безнадёжно. Вместо этого импортируешь их общий BOM:

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.3.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

`<scope>import</scope>` (работает **только** внутри `dependencyManagement`, `<type>pom</type>` обязателен) означает: «вклей содержимое management-секции этого pom'а сюда, как будто я написал её сам». Дальше любая зависимость из стека Spring объявляется без версии — версии гарантированно согласованы, потому что их согласовала команда Spring.

### Parent vs BOM

Разница ровно та же, что между наследованием и композицией. `<parent>` у проекта может быть **один** — как единственный суперкласс в Java, и он тянет не только версии, но и плагины, properties, весь конфиг. А импортировать BOM'ов можно **сколько угодно**: Spring BOM + AWS SDK BOM + Testcontainers BOM в одном проекте, и каждый приносит только версии. BOM — способ получить чужой `dependencyManagement`, не занимая единственный слот родителя.

![[Pasted image 20260709174022.png]]

> [!tip] 💡 Из C#  
> BOM ≈ Central Package Management, но подключаемый как переиспользуемый пакет: вместо своего `Directory.Packages.props` ты импортируешь чужой согласованный набор версий. Прямого «импортируй чужой CPM-файл как артефакт из NuGet» в .NET нет — BOM здесь богаче.

> [!example] 🔬 Глубже / на синьора  
> **`spring-boot-starter-parent` vs импорт `spring-boot-dependencies`** — parent-vs-BOM на живом Spring. Starter-parent — это **parent pom**: тянет и версии, и настроенные плагины (`spring-boot-maven-plugin`, ресурс-фильтрация, `java.version`), но занимает твой единственный слот родителя. `spring-boot-dependencies` — **BOM** с тем же набором версий, но без плагинов; импортируешь через `scope=import`, слот родителя остаётся свободным под свой корпоративный parent. Выбор между ними — ровно про «нужен ли мне ещё и их плагинный конфиг или только версии».  
> **Реактор и частичные сборки.** Порядок реактора — топологическая сортировка графа модулей; при циклической зависимости между модулями Maven падает с ошибкой. Флаги `-pl` (project list) и `-am` (also make) собирают подмножество: `mvn -pl app -am install` соберёт `app` и всё, от чего он зависит, пропустив остальное — экономит время в большом репозитории.  
> **`<relativePath>`** говорит Maven, где искать parent pom. По умолчанию — `../pom.xml` (сосед на файловой системе); для корпоративного parent'а, живущего в репозитории, ставят `<relativePath/>` (пустой) — иначе Maven сначала зря ищет на диске.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`dependencyManagement` в родителе ≠ зависимости у детей.** Он только фиксирует версии; чтобы модуль получил артефакт, тот всё равно объявляется в `<dependencies>` модуля (пусть и без версии). Частый сюрприз: «прописал в родителе, а в модуле класса нет».
> 2. **Приоритет при пересечении:** запись в **твоём собственном** `dependencyManagement` перебивает импортированную из BOM — так и переопределяют версию одной библиотеки из Spring BOM. Если один артефакт управляется двумя BOM'ами — побеждает объявленный **первым**. Снова порядок объявления, снова не «новейшая».
> 3. **`scope=import` без `<type>pom</type>` молча не сработает** так, как ждёшь — оба атрибута обязательны в паре.
> 4. **Один `<parent>` на проект.** Нужен и корпоративный parent, и версии Spring — Spring идёт BOM'ом через import, а не вторым родителем (второго не бывает).
> 5. **Gradle резолвит иначе:** по умолчанию берёт **наибольшую** версию (как NuGet по духу, не как Maven). Если переносишь ментальную модель Maven на Gradle — конфликты разрешатся по-другому. За рамками урока, но знать про различие стоит.

#### На собесе

> [!question]- Зачем multi-module и чем агрегация отличается от наследования? _(топ темы)_  
> **Model answer (RU):** Это два независимых механизма. Агрегация (`<modules>` у родителя) даёт реактор: одной командой модули собираются в топологическом порядке их взаимных зависимостей — `api` раньше `core`. Наследование (`<parent>` у ребёнка) раздаёт общий конфиг и, главное, `dependencyManagement` — версии живут в одном месте, модули объявляют зависимости без версий. Родитель не обязан быть агрегатором: механизмы разделимы.
> 
> **Model answer (EN):** Two independent mechanisms. Aggregation (`<modules>` in the parent) gives the reactor: one command builds the modules in the topological order of their interdependencies — `api` before `core`. Inheritance (`<parent>` in the child) distributes shared config and, crucially, `dependencyManagement` — versions live in one place and modules declare dependencies without versions. The parent need not be the aggregator; the mechanisms are separable.
> 
> _Trap:_ «родитель обязательно собирает детей?» — нет, агрегация и наследование независимы.

> [!question]- Чем BOM отличается от parent pom?  
> **Model answer (RU):** Оба доставляют `dependencyManagement`, но parent — это наследование: слот один (как единственный суперкласс), и тянет ещё плагины, properties, весь конфиг. BOM — композиция через `<scope>import</scope>` (обязателен `<type>pom</type>`): импортировать можно сколько угодно, и приносит он только версии. BOM — способ получить чужой набор согласованных версий, не занимая единственный слот родителя. Spring Boot даёт оба: `spring-boot-starter-parent` (parent) и `spring-boot-dependencies` (BOM).
> 
> **Model answer (EN):** Both deliver `dependencyManagement`, but a parent is inheritance: a single slot (like one superclass), and it also carries plugins, properties, all config. A BOM is composition via `<scope>import</scope>` (requires `<type>pom</type>`): you can import as many as you like, and it brings only versions. A BOM lets you pull in someone's coordinated version set without spending your single parent slot. Spring Boot ships both: `spring-boot-starter-parent` (parent) and `spring-boot-dependencies` (BOM).
> 
> _Follow-up:_ «мой `dependencyManagement` и импортированный BOM управляют одной библиотекой — кто победит?» — твой собственный; между двумя BOM'ами — объявленный первым.

> [!question]- Прописал версию в `dependencyManagement` родителя, а в модуле класса нет. Почему?  
> **RU:** `dependencyManagement` только фиксирует версию — он не добавляет зависимость. Модуль всё равно должен объявить артефакт в своём `<dependencies>` (можно без версии, её подставит родитель). Это тот же трап, что и в 8.6: management ≠ dependency.
