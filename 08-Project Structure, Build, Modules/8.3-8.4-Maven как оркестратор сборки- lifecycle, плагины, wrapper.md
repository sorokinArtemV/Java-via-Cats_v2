Ты склонировал чужой репозиторий, набрал `mvn package` — и оно скомпилировалось, **прогнало тесты** и выдало `jar`, хотя ты просил только «упакуй». Почему сделал больше, чем сказано? Потому что сборка — это конвейер с жёстко зашитым порядком (нельзя паковать непротестированное, нельзя тестировать нескомпилированное), и Maven проходит этот конвейер за тебя. А раз порядок фиксирован и одинаков у всех, то `mvn package` в незнакомом проекте пишется вслепую и значит ровно одно и то же на любой машине.

**Mental model.** Maven-сборка — фиксированный конвейер фаз (**lifecycle**); называешь фазу — Maven исполняет её _и все предыдущие_. Само ядро при этом не умеет ни компилировать, ни тестировать: всю работу делают **плагины**, чьи задачи (**goals**) привязаны к фазам.

#### 1. Default lifecycle и правило «фаза тянет предыдущие»

Рабочие фазы default-цикла по порядку:

`validate → compile → test → package → verify → install → deploy`

И одно правило, на котором всё держится: **вызов фазы исполняет все фазы до неё включительно.** `mvn package` = validate + compile + test + package. Ты не перечисляешь шаги — ты называешь, докуда доехать. Вот и разгадка: путь к `package` физически лежит через `compile` и `test`, поэтому непротестированный код упаковать невозможно.

Схема-лесенка: слева — что накрывает `mvn package`, справа — какие goals плагинов висят на фазах.

Зачем так жёстко: **воспроизводимость.** `mvn package` означает строго одно и то же у тебя, у коллеги и на CI — одна команда, один смысл, для любого проекта на планете.

> [!tip] 💡 Из C#  
> У MSBuild таргеты — свободный граф, и каждый проект сам решает, тянет ли `Pack` за собой тесты. Maven решил за всех раз и навсегда. Размен прямой: минус гибкость, плюс тотальная предсказуемость. Ты называешь фазу, а не собираешь пайплайн из таргетов вручную.

#### 2. install и deploy: `~/.m2` как точка публикации

Хвост конвейера — про то, куда уезжает собранный артефакт.

**`install`** кладёт твой `jar` в `~/.m2/repository` — и вот тут `~/.m2` из урока 8.1 выстреливает второй стороной. Он не только кэш скачанного из Central, но и **локальная точка публикации**: сделал `mvn install` в проекте `payments` — и соседний твой проект указывает `dev.artem:payments:1.0.0` в `<dependencies>` как обычную библиотеку. Maven найдёт её в `~/.m2` так же, как нашёл бы Jackson. Помнишь симметрию «ты — такой же артефакт»? Вот её механика.

**`deploy`** — то же, но в _удалённый_ репозиторий: корпоративный Nexus/Artifactory или Central. `install` — для себя (соседние локальные проекты), `deploy` — для команды и мира.

> [!tip] 💡 Из C#  
> `deploy` ≈ `dotnet nuget push` (публикация в общий фид). А вот у `install` чистого аналога нет: NuGet умеет локальные фиды, но «собери и положи в локальный кэш, чтобы соседний проект на той же машине увидел» — это именно maven'ская идиома единого `~/.m2`.

#### 3. `clean` — отдельный lifecycle

В списке фаз выше `clean` нет — это **отдельный** lifecycle, не часть default. Поэтому канонично пишут `mvn clean package`: два цикла одним вызовом. Зачем вынесли: конвейер _производит_ артефакт, а чистка — операция _над рабочей папкой_, не шаг производства. И это не педантизм: инкрементальность у Maven примитивная (сравнение timestamps), поэтому после переименования или удаления классов в `target/` остаются осиротевшие `.class`, которые могут молча уехать в `jar`. `clean` (снос `target/`) гарантирует сборку с нуля. А в CI, где checkout всегда свежий, `clean` не нужен — и хорошо, что не зашит намертво, иначе все жгли бы время впустую.

#### 4. Плагины: ядро Maven не умеет ничего

Контринтуитивный факт: сам Maven не компилирует, не тестирует и не пакует. Ядро — чистый оркестратор: знает порядок фаз и больше ничего. Всю работу делают **плагины**, а у плагина есть **goals** — конкретные задачи, привязанные к фазам. Наступила фаза → исполнились все goals, висящие на ней. Те подписи справа на схеме — это они: для `jar`-проекта привязки стоят из коробки (`compiler:compile` на `compile`, `surefire:test` на `test`, `jar:jar` на `package`). Вот почему пустой `pom.xml` «просто собирается» — дефолтные привязки уже на месте (откуда именно — см. 🔬 про Super POM).

Архитектура снимает противоречие жёсткого цикла: конвейер зафиксирован, но работа на каждой станции заменяема. Понадобилась кодогенерация — вешаешь плагин на раннюю фазу `generate-sources`, и цикл не меняется, `mvn package` по-прежнему значит то же самое. Расширяемость без потери предсказуемости.

Единственный плагин, который конфигурируют почти всегда — compiler, чтобы зафиксировать версию Java:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.13.0</version>
            <configuration>
                <release>21</release>
            </configuration>
        </plugin>
    </plugins>
</build>
```

(Короткая форма — `<maven.compiler.release>21</maven.compiler.release>` в блоке `<properties>`, плагин подхватит сам.)

Почему `<release>`, а не старая пара `source`/`target`. Пара проверяет синтаксис и версию формата байткода, но **не API**: собираешь на JDK 21 под Java 17 — и можешь случайно вызвать метод, появившийся в 21-м, а поймаешь `NoSuchMethodError` уже в проде. Флаг `--release` (`javac`, JEP 247, Java 9) компилирует против API _ровно целевой версии_ — ошибка прилетит на компиляции. Вот тот же контраст, снятый на JDK 21 (`Math.clamp` добавлен в Java 21):

```
javac --release 21 Clamp.java     → OK compiled
javac --release 17 Clamp.java
    Clamp.java:3: error: cannot find symbol
        long r = Math.clamp(5L, 1L, 10L);
                     ^  symbol: method clamp(long,long,long)
javac -source 17 -target 17 Clamp.java
    warning: [options] system modules path not set ...
    (компилируется! — а на JDK 17 в рантайме упадёт NoSuchMethodError)
```

`--release 17` честно ловит несуществующий в 17-м API на компиляции; старая пара `source`/`target` пропускает вызов, компилируя против API _текущей_ JDK 21, — бомба замедленного действия до первого запуска на 17-м рантайме.

И последнее про плагины: goal можно дёрнуть напрямую, мимо цикла — синтаксис `mvn плагин:goal`. Самый полезный — `mvn dependency:tree` из урока 8.1; ещё `mvn help:effective-pom` (см. 🔬).

#### 5. Maven Wrapper (`mvnw`): пришпилить версию самого Maven

Осталась дыра в «одинаково у всех». `pom.xml` фиксирует версии зависимостей и плагинов, но не версию **самого** `mvn` — а у тебя локально может стоять Maven 3.9, у коллеги 3.6, на CI 4.0, и поведение сборки чуть разъедется. Лечит это **Maven Wrapper**: в репозиторий коммитятся скрипты `mvnw`/`mvnw.cmd` и файл `.mvn/wrapper/maven-wrapper.properties` с точной версией. Дальше все зовут не `mvn`, а `./mvnw package` — wrapper сам скачает нужную версию Maven и запустит именно её. Версия сборщика становится частью репозитория, а не свойством машины.

> [!tip] 💡 Из C#  
> Ближайшая аналогия — `global.json`, которым ты пришпиливаешь версию .NET SDK для репозитория. `mvnw` идёт дальше: не просто требует нужную версию, а сам её доставляет. У Gradle то же самое зовётся `gradlew` (его увидишь в 8.3).

> [!example] 🔬 Глубже / на синьора  
> **Откуда у пустого `pom.xml` дефолтные привязки — Super POM.** Каждый `pom` неявно наследует **Super POM** — ровно как каждый класс неявно наследует `Object`. В нём зашиты: URL Maven Central, стандартная раскладка каталогов (`src/main/java`, `target/`) и дефолтные привязки goals к фазам для `jar`-упаковки. Увидеть итог слияния твоего pom с Super POM и родителями — `mvn help:effective-pom`; это же первый инструмент, когда «откуда взялась эта настройка, я её не писал».
> 
> **`-DskipTests` vs `-Dmaven.test.skip=true` — разный уровень.** `-DskipTests` понимает surefire: тесты **компилируются**, но не бегут. `-Dmaven.test.skip=true` понимает и compiler-plugin: тестовый код даже **не компилируется**. Разница вылезает на сломанном тестовом коде: при `skipTests` сборка всё равно упадёт на `test-compile`, при `maven.test.skip` — пройдёт, замаскировав то, что тесты вообще не собираются.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`clean` не в default-цикле.** Ждёшь, что «Maven пересоберёт изменённое» — но инкрементальность по timestamps примитивна, и осиротевшие `.class` после удаления/переименования уедут в `jar`. Канон — `mvn clean package`.
> 2. **`mvn test` не даёт `jar`.** Фаза `package` идёт _после_ `test`, а вызов фазы тянет только _предыдущие_, не следующие. Ждёшь артефакт от `mvn test` — не получишь; нужен минимум `mvn package`.
> 3. **`source`/`target` вместо `release`** — компиляция против API текущей JDK, `NoSuchMethodError` на целевом рантайме (показано выше). В новом коде — только `<release>`.
> 4. **`-Dmaven.test.skip=true` маскирует нескомпилируемые тесты** — сборка зелёная, а тестовый код давно не собирается. Для «собрать без прогона рабочих тестов» бери `-DskipTests`, а не «выключить тесты совсем».
> 5. **Нет `mvnw` → «у меня собирается».** Версия Maven остаётся свойством машины; тонкие различия в поведении плагинов между версиями сборщика ловятся только на CI. Коммить wrapper.

#### На собесе

> [!question]- Запускаю `mvn package` — тесты прогонятся? _(топ вопрос)_  
> **RU:** Да. Фаза `test` в цикле стоит раньше `package`, а вызов фазы исполняет все предыдущие — значит, `package` обязательно скомпилирует и прогонит тесты. Это и даёт воспроизводимость: непротестированный код упаковать нельзя.  
> **EN:** Yes. `test` comes before `package` in the lifecycle, and invoking a phase runs all preceding ones — so `package` necessarily compiles and runs the tests. That's what gives reproducibility: you can't package untested code.
> 
> _Follow-up:_ «как собрать без тестов?» — `-DskipTests` (компилируются, но не бегут) либо `-Dmaven.test.skip=true` (даже не компилируются). _Trap:_ «в чём разница?» — на сломанном тестовом коде первый флаг всё равно упадёт на компиляции тестов, второй пройдёт.

> [!question]- Объясни lifecycle  
> **RU:** Цепочка фаз: `validate → compile → test → package → verify → install → deploy`. Ключевое — вызов фазы исполняет все предыдущие: `mvn package` пройдёт compile и test. Это даёт воспроизводимость — команда значит одно и то же на любой машине. `clean` — отдельный цикл, отсюда `mvn clean package`.  
> **EN:** A chain of phases: `validate → compile → test → package → verify → install → deploy`. The key point — invoking a phase runs everything before it: `mvn package` goes through compile and test. That yields reproducibility — the command means the same on any machine. `clean` is a separate lifecycle, hence `mvn clean package`.
> 
> _Trap:_ «даст ли `mvn test` jar?» — нет, `package` идёт _после_ `test`; тянутся только предыдущие фазы.

> [!question]- `install` vs `deploy`, и зачем `install` вообще?  
> **RU:** `install` кладёт артефакт в локальный `~/.m2`, `deploy` — в удалённый репозиторий. `install` нужен, чтобы соседний локальный проект взял твой артефакт как обычную зависимость по GAV.  
> **EN:** `install` puts the artifact into the local `~/.m2`; `deploy` publishes it to a remote repository. `install` exists so a neighbouring local project can consume your artifact as a normal GAV dependency.
> 
> _Follow-up:_ «что такое `~/.m2`?» — единое хранилище: кэш скачанного из Central + локально опубликованное; резолв зависимостей всегда стартует оттуда.

> [!question]- Роль плагинов? Почему пустой `pom` вообще собирается?  
> **RU:** Ядро Maven — оркестратор порядка фаз, а работу делают плагины: их goals привязаны к фазам (`compiler:compile` на compile, `surefire:test` на test, `jar:jar` на package). Пустой `pom` собирается, потому что эти привязки приходят из Super POM, который неявно наследует каждый проект — как `Object` для классов.  
> **EN:** Maven's core just orchestrates the phase order; the work is done by plugins whose goals bind to phases (`compiler:compile` on compile, `surefire:test` on test, `jar:jar` on package). An empty `pom` builds because those bindings come from the Super POM that every project implicitly inherits — like `Object` for classes.
> 
> _Follow-up:_ «как зафиксировать версию Java?» — `<release>21</release>` у compiler-plugin: в отличие от `source`/`target` он проверяет ещё и API целевой версии, а не только формат байткода.

> [!question]- Чем `--release` лучше `source`/`target`?  
> `source`/`target` фиксируют версию языка и формат байткода, но компилируют против API _текущей_ JDK — можно вызвать метод, которого нет в целевой версии, и получить `NoSuchMethodError` в рантайме. `--release N` (JEP 247, Java 9) компилирует против API ровно версии N, и такой вызов падает на компиляции.
> 
> _Trap:_ «а если целевая = версия сборки?» — тогда разницы в API нет, но `--release` всё равно предпочтительнее как единый честный флаг.

> [!question]- Что такое `mvnw` и зачем он?  
> Maven Wrapper: закоммиченные в репозиторий скрипты `mvnw` + файл с точной версией Maven. Все зовут `./mvnw`, который сам скачивает и запускает нужную версию — версия сборщика становится частью репозитория, а не свойством машины. Убирает класс «у меня другая версия Maven, потому и собирается иначе». Аналог `global.json` в .NET, только ещё и доставляет сборщик.
