### Как разложить проект по модулям: слои, фичи, монорепо

#### Соответствие единиц

Базовая единица в Maven — **модуль**: это папка с `pom.xml`, которая собирается в один артефакт (`jar`). Вот она и есть аналог `.csproj`.

> [!tip] 💡 Из C#
> 
> - `.csproj` (одна сборка `.dll`) ↔ Maven-модуль / `pom.xml` (один `jar`). Прямое соответствие — компиляционная единица.
> - `.sln` ↔ **aggregator pom** (`<packaging>pom</packaging>` + `<modules>`). Но дельта: `.sln` _только_ агрегирует, а parent pom обычно ещё и наследует конфиг/версии — в .NET за это отвечает отдельный `Directory.Build.props`, в Maven оба механизма в одном файле (про это был прошлый урок).
> - solution folder (виртуальная папка в IDE) ↔ в Maven виртуальных папок **нет**. Папки реальны. Группировка фич, которой нужно быть ещё и единицей сборки, — это промежуточный aggregator pom.
> - `<ProjectReference Include="..."/>` ↔ `<dependency>` на соседний модуль (твой собственный `groupId:artifactId:version`).

> [!tip] 💡 Из C#: имя артефакта ≠ имя пакета  
> В .NET имя сборки и корневой namespace обычно совпадают — `Shop.Orders.Domain.dll` и namespace `Shop.Orders.Domain`. В Java это **два независимых имени**: артефакт (`jar`) называется `orders-domain` (lowercase, через дефис), а пакет внутри — `com.example.shop.orders.domain` (reverse-DNS). Совпадение никто не требует, и обычно его нет.

#### Четыре слоя и куда указывают зависимости

Прежде чем раскладывать по папкам — сама архитектурная инварианта, она одна для всех вариантов раскладки. В схеме `api` / `application` / `infrastructure` / `domain` (это clean/hexagonal) зависимости текут **внутрь**, к `domain`. `domain` не зависит ни от кого; `infrastructure` реализует порты, объявленные внутри, поэтому зависит внутрь, а не наоборот.

Запомни картинку — она объясняет, **зачем** вообще раскладывать слои по модулям: если каждый слой — отдельный `jar`, то `domain` физически не может импортировать `infrastructure`, потому что между ними нет ребра в графе зависимостей. Граница не «по договорённости», а на уровне компилятора. Это и есть единственный сильный аргумент за «модуль на слой».

#### Где живут слои: три варианта

**Вариант A — модуль на слой (буквальный перевод твоего .NET).** Каждый слой каждой фичи — отдельный `jar`:

```
my-shop/                        ← parent pom (packaging=pom), это «решение»
├── pom.xml
├── orders/                     ← aggregator pom фичи (packaging=pom) = solution folder
│   ├── pom.xml
│   ├── orders-domain/          ← jar
│   │   └── src/main/java/com/example/shop/orders/domain/
│   ├── orders-application/     ← jar, зависит на orders-domain
│   ├── orders-api/             ← jar
│   └── orders-infrastructure/  ← jar
├── payments/                   ← вторая фича, та же структура
│   └── ...
└── app/                        ← jar, точка входа, тянет всё, deployable
```

Вот прямой ответ про «solution folder»: в Maven папка-группировка, которой нужно быть ещё и единицей сборки, — это **промежуточный aggregator pom** (`orders/` с `packaging=pom`). Виртуальной папки «просто для красоты в дереве», как в `.sln`, не бывает — папки реальны.

> [!warning] ⚠️ Подводный камень  
> Посчитай модули. 4 слоя × 2 фичи + `app` + два `packaging=pom` = 11 pom-файлов на крошечный проект. Модуль в Maven _дороже_ `csproj` по церемониям (отдельный pom, отдельная строка в реакторе, межмодульные `<dependency>` руками). Поэтому в Java этот вариант берут реже, чем в .NET — и часто приходят к выводу, что границу слоёв дешевле проверять тестом (`ArchUnit`), чем дробить на модули.

**Вариант B — один модуль, слои = пакеты (самый частый дефолт в Java).** Один `pom.xml`, один `jar`, слои — просто пакеты:

```
my-service/
├── pom.xml                     ← один pom, один jar
└── src/main/java/com/example/service/
    ├── api/
    ├── application/
    ├── domain/
    └── infrastructure/
```

> [!tip] 💡 Из C#  
> Вот здесь главное культурное расхождение. Дотнетчик почти всегда дробит сервис на 4 dll; джавист по умолчанию делает один `jar` и слои держит **пакетами**. Дефолтное Spring Boot-приложение — ровно это: один Maven-модуль. Границы между слоями стерегут `package-private` видимость + `ArchUnit` (тест, который падает, если `domain` импортнул `infrastructure`), а не граф сборки.

**Вариант C — модуль на фичу, слои = пакеты внутри (современный модульный монолит).** Золотая середина:

```
my-shop/                        ← parent pom
├── pom.xml
├── orders/                     ← jar: одна фича = один модуль
│   ├── pom.xml
│   └── src/main/java/com/example/shop/orders/
│       ├── api/  application/  domain/  infrastructure/
├── payments/                   ← jar, та же раскладка внутри
│   └── ...
└── app/                        ← jar, deployable, тянет orders + payments
```

Граница модуля = bounded context (фича): `payments` не может импортнуть внутренности `orders`, потому что между ними нет ребра — это держит компилятор. А слои внутри фичи — дешёвые пакеты. Этот подход двигает `Spring Modulith`, и для модульного монолита это сейчас фактический дефолт.

#### Как модули видят друг друга

Никаких `ProjectReference` — ссылка на соседний модуль это обычная `<dependency>` с твоим же `groupId` и `version` проекта:

```xml
<dependency>
    <groupId>com.example.shop</groupId>
    <artifactId>orders-domain</artifactId>
    <version>${project.version}</version>
</dependency>
```

Когда запускаешь сборку из корня, Maven строит **реактор**: читает все модули, видит, что `orders-application` зависит на `orders-domain`, и собирает `domain` первым. Порядок слоёв в реакторе — это ровно стрелки с диаграммы выше, развёрнутые в линию.

#### Монорепо микросервисов

Каждый сервис здесь — **самостоятельная единица деплоя** (свой `jar`/контейнер), а не слой общего приложения. Раскладка:

```
monorepo/
├── pom.xml                     ← корневой parent: общий dependencyManagement / BOM
├── platform-bom/               ← свой BOM с версиями (опц.)
├── shared-kernel/              ← общая библиотека-jar (опц., осторожно!)
├── service-orders/             ← один сервис = свой (мульти)модульный build
│   ├── pom.xml
│   └── src/main/java/...
├── service-payments/
└── service-inventory/
```

Сервисы деплоятся независимо, но шарят корневой parent pom — и это ровно та машинерия из прошлого урока: `dependencyManagement` + BOM выравнивают версии библиотек по всему репо, чтобы `orders` и `payments` не разъехались на разный Jackson.

> [!tip] 💡 Из C#  
> Один-в-один твой монореп с `Directory.Packages.props` в корне (общие версии) + по сервису. Корневой parent pom = `Directory.Packages.props`; `shared-kernel` = общий `Shared.dll`-проект.

> [!warning] ⚠️ Подводный камень  
> `shared-kernel` — заманчиво и опасно. Общий `jar` между сервисами вводит связность: меняешь его — пересобираешь и редеплоишь всех потребителей. В микросервисах это часто анти-цель. Многие держат общим только то, что меняется почти никогда (DTO контрактов, и то спорно).

#### Что бы я выбрал

Один сервис или стартап — **вариант B**, не плоди модули из любви к симметрии. Модульный монолит, где границы между фичами надо держать compile-time — **вариант C** (модуль на фичу), это современный мейнстрим. **Вариант A** (модуль на слой) бери только если команде принципиально запретить «не туда импортнуть» именно графом сборки, а не тестом — цена в виде десятка pom'ов того стоит редко. Для монорепо микросервисов — корневой parent pom с `dependencyManagement`/BOM плюс каждый сервис своим build'ом, а `shared-kernel` минимизируй.

Главный сдвиг в голове: в Java «модуль» (то есть отдельный `jar`) ты заводишь, когда нужна **граница деплоя или граница зависимостей**, а не «потому что это отдельный слой». Слой по умолчанию — это пакет, а не артефакт.
