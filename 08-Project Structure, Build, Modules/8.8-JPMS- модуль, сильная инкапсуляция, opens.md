## 8.8.1. JPMS: модуль, сильная инкапсуляция, `opens`

Представь, что ты выпускаешь библиотеку `shop-greeter`. У неё есть публичное API — пакет `com.shop.api` — и кишки в `com.shop.internal`, которые ты хочешь свободно переписывать между версиями. Проблема в том, что классы из `internal` вынуждены быть `public`: их зовёт `api` из _другого_ пакета, а package-private между пакетами не работает. Но `public` в Java — это «виден всем на classpath». Любой потребитель спокойно пишет `import com.shop.internal.TemplateEngine`, строит на этом код, и твой следующий рефакторинг ломает ему сборку. Слово `internal` в имени пакета — это просьба, а не запрет. Ровно этой болезнью болел сам JDK: полмира импортировало `sun.misc.Unsafe` и прочие внутренности, и Oracle годами не мог их трогать. JPMS (`JEP 261`, проект Jigsaw, Java 9) закрывает дыру между «пакетом» и «всем миром».

**Mental model.** Модуль — именованная группа пакетов с дескриптором `module-info.java`, которая сама объявляет, что ей нужно (`requires`) и что она показывает наружу (`exports`). Доступ к чужому типу = **readability + exports**, обе галочки обязательны, и границу проверяет не code review, а компилятор и JVM.

> [!tip] 💡 Из C#  
> У тебя эта проблема решена с рождения языка: `internal` прячет тип на уровне **сборки** (assembly). В Java до 9 уровня «сборки» просто не существовало — есть класс, есть пакет (причём `com.shop` и `com.shop.internal` друг другу чужие: package-private не тянется на подпакеты), а дальше сразу «весь classpath». Дыру между пакетом и миром JPMS и закрывает — это и есть недостающий у Java уровень «сборки».

### Как это работает

Модуль — это jar (или дерево классов), в корне которого лежит `module-info.java`. Два глагола несут всю базовую семантику:

```java
// модуль библиотеки
module com.shop.greeter {
    exports com.shop.api;   // только этот пакет виден наружу
}                           // com.shop.internal не упомянут → скрыт

// модуль приложения
module com.app {
    requires com.shop.greeter;
}
```

`requires` строит граф зависимостей: `com.app` _читает_ `com.shop.greeter` (в спецификации — readability). `exports` решает доступность: из читаемого модуля тебе видны `public`-типы **только экспортированных** пакетов. Теперь компилятор делает то, что раньше делал только ревьюер. Пробуем из `com.app` импортнуть внутренности:

```java
import com.shop.internal.TemplateEngine;   // в модуле com.app
```

```
error: package com.shop.internal is not visible
  (package com.shop.internal is declared in module com.shop.greeter,
   which does not export it)
```

Это ошибка **компиляции**, не warning (проверено на JDK 21). `public`-класс перестал быть глобально публичным — `public` теперь значит «публичен в пределах того, что модуль решил показать». Вот вся картинка целиком — кто что видит:

![[Pasted image 20260709181925.png]]

`java.base` внизу — не декорация: сам JDK с Java 9 распилен на ~70 модулей (`java.base`, `java.sql`, `java.xml`…), и `java.base` неявно читается всеми. Это была половина смысла Jigsaw — модуляризовать саму платформу.

### Сильная инкапсуляция: даже reflection не пролезает

Вот где JPMS радикальнее всего, что ты видел в C#. Обычная реакция Java-разработчика на закрытую дверь — `Class.forName` плюс `setAccessible(true)`. Пробуем достучаться до `internal` рефлексией:

```java
Class<?> clazz = Class.forName("com.shop.internal.TemplateEngine");
var method = clazz.getMethod("render", String.class);
method.invoke(null, "X");
```

Компилируется (строка — не `import`), но в рантайме падает (проверено на JDK 21):

```
Exception in thread "main" java.lang.IllegalAccessException: class com.app.Main
  (in module com.app) cannot access class com.shop.internal.TemplateEngine
  (in module com.shop.greeter) because module com.shop.greeter does not export
  com.shop.internal to module com.app
```

JVM проверяет границы модулей на **каждом** рефлексивном доступе. Это и называется **strong encapsulation** — инкапсуляция, которую нельзя обойти из кода вообще.

> [!tip] 💡 Из C#  
> Здесь аналогия с `internal` окончательно ломается. В .NET reflection с `BindingFlags.NonPublic` читает что угодно в любой сборке — видимость это compile-time-вежливость, рантайм её не сторожит. В JPMS закрытый пакет закрыт и для рантайма. Прямого аналога в C# нет — не натягивай.

### `opens` — дверь для фреймворков

Но фреймворки-то живут рефлексией: Spring инстанцирует бины через `setAccessible`, Hibernate лезет в `private`-поля сущностей, Jackson — в поля DTO. Если твой модуль не пускает их — ничего не работает. Для этого есть третий глагол:

```java
module com.app {
    requires com.shop.greeter;
    opens com.app.entities;                      // deep reflection в рантайме
    // или адресно: opens com.app.entities to org.hibernate.orm.core;
}
```

Разница, которую спрашивают на собесе: `exports` даёт **прямой доступ к `public` API — на компиляции и в рантайме**. `opens` не даёт компилятору ничего, но разрешает **deep reflection в рантайме**, включая `setAccessible(true)` на `private`-членах. Пакет может быть `opens`, но не `exports`: снаружи на него нельзя написать `import`, а Hibernate через рефлексию — пожалуйста. Можно открыть весь модуль разом: `open module com.app { … }`.

![[Pasted image 20260709181942.png]]

> [!example] 🔬 Глубже / на синьора  
> **Адресные (qualified) директивы.** И `exports`, и `opens` бывают «кому именно»: `exports com.shop.spi to com.app;` открывает пакет ровно одному модулю-получателю, а не всему миру. Это видно в `java --describe-module` как отдельные строки `qualified exports … to …` / `qualified opens … to …` (проверено). Механизм для «дружественных» модулей внутри одной кодовой базы — аналог `[InternalsVisibleTo]` в .NET по духу.
> 
> **`--add-opens` / `--add-exports` как аварийный люк.** Границы можно продавить снаружи, флагом запуска, не трогая `module-info`: `--add-opens java.base/java.lang=ALL-UNNAMED` разрешает всем безымянным модулям рефлексию в `java.lang`. Отсюда этот костыль в стартовых скриптах половины энтерпрайза (см. подводный камень ниже).
> 
> **Reflection всё-таки работает — но только на своё.** Модуль всегда может рефлексировать собственные пакеты без `opens`; `opens` нужен ровно для того, чтобы пустить _чужой_ модуль. Поэтому `setAccessible` внутри одного модуля никогда не падает — обжигаются именно на межмодульной границе.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`exports` ≠ `opens`, и путать их — классическая ошибка в ответе.** `exports`-пакет нельзя рефлексировать с `setAccessible` на `private` — для этого нужен именно `opens`. Экспорт открывает `public` API, а не приватные потроха.
> 2. **Приложение компилируется, но падает в рантайме**, потому что Spring/Hibernate не может открыть твой класс (`InaccessibleObjectException`). Это была главная боль ранних миграций на Java 9–11: `opens` забыли, компилятору всё равно, а рантайм — нет.
> 3. **`--add-opens java.base/…=ALL-UNNAMED` в скрипте запуска** — не «магия для галочки», а признак того, что какая-то библиотека лезет во внутренности JDK. JDK с Java 16 (`JEP 396`) и окончательно с Java 17 (`JEP 403`) закрыл свои internals по умолчанию, и старый код, лазивший туда рефлексией, перестал работать без явного «откройте мне».
> 4. **`open module` открывает вообще всё** — удобно для быстрой миграции, но выкидывает всю сильную инкапсуляцию разом. Точечный `opens … to …` почти всегда правильнее.
> 5. **Рефлексия на _собственные_ приватные поля не требует `opens`** — легко решить, что «reflection в модулях запрещён вообще». Запрещён только межмодульный доступ без разрешения.

### На собесе

> [!question]- Что такое JPMS?  
> **Model answer (RU):** Система модулей уровня языка и JVM с Java 9 (`JEP 261`, Jigsaw). Модуль — группа пакетов с дескриптором `module-info.java`: `requires` объявляет зависимости (readability), `exports` — какие пакеты видны наружу. Неэкспортированный пакет недоступен извне даже через reflection — это strong encapsulation; для рефлексивного доступа есть отдельный глагол `opens`.
> 
> **Model answer (EN):** A language- and JVM-level module system since Java 9 (`JEP 261`, Jigsaw). A module is a group of packages with a `module-info.java` descriptor: `requires` declares dependencies (readability), `exports` declares which packages are visible outside. A non-exported package is inaccessible from outside even via reflection — that's strong encapsulation; reflective access needs the separate `opens` directive.
> 
> _Follow-up:_ «доступ к типу — это одна галочка?» — нет, две: readability (`requires`) И exports целевого пакета. _Trap:_ «`public`-класс всегда публичен?» — нет, только в пределах экспортированного пакета.
> 
> _Слабые ответы:_ «модули — это про Maven/сборку» — путает с multi-module (см. 8.8.2). «`module-info` перечисляет зависимости» — половина правды, забыт `exports` и сильная инкапсуляция.

> [!question]- Зачем `opens`, если есть `exports`?  
> **Model answer (RU):** `exports` — прямой доступ к `public` API на компиляции и в рантайме; `opens` — только рантайм, но deep reflection, включая `setAccessible` на `private`. Entity-пакеты типично `opens` для Hibernate, но не `exports`.
> 
> **Model answer (EN):** `exports` grants direct access to the `public` API at compile time and runtime; `opens` grants runtime-only access but deep reflection, including `setAccessible` on `private` members. Entity packages are typically `opens` for Hibernate but not `exports`.
> 
> _Follow-up:_ «может ли пакет быть `opens`, но не `exports`?» — да, ровно кейс entity: `import` снаружи нельзя, Hibernate рефлексией — можно. _Trap:_ «`exports`-пакет можно рефлексировать на приватные поля?» — нет, для этого нужен `opens`.

> [!question]- Приложение на Java 8 переезжает на 17 и падает с `InaccessibleObjectException` где-то в недрах библиотеки. Что случилось и что делать?  
> **Model answer (RU):** Библиотека рефлексией лезет во внутренности JDK; с `JEP 396` (Java 16) / `JEP 403` (Java 17) они закрыты по умолчанию. Краткосрочно — `--add-opens java.base/…=ALL-UNNAMED` в скрипте запуска, правильно — обновить библиотеку до модуль-совместимой версии.
> 
> **Model answer (EN):** The library reflects into JDK internals; since `JEP 396` (Java 16) / `JEP 403` (Java 17) those are strongly encapsulated by default. Short-term fix — `--add-opens java.base/…=ALL-UNNAMED` at launch; proper fix — upgrade the library.
> 
> _Follow-up:_ «а почему до Java 16 работало?» — был режим предупреждений (`--illegal-access=permit` по умолчанию), потом дефолт ужесточили до deny.

## 8.8.2. JPMS на практике: module path vs classpath, экосистема, `jlink`

Первый урок оставил ощущение, что модули — это уже реальность: компилятор ловит импорты, рантайм ловит рефлексию. Но есть один факт, который переворачивает всю картину для повседневной работы, и именно его любят проверять на собесе. Всё, что ты видел в 8.8.1, включается **только** при одном условии — и в типичном проде это условие не выполнено. Этот урок про то, где проходит рубильник, почему индустрия в основном прошла мимо модулей для прикладного кода, и где JPMS всё-таки выстрелил по-крупному.

**Mental model.** Правила модулей действуют, только если код лежит на **module path**. Тот же самый jar на старом **classpath** — и `module-info` просто игнорируется, всё сваливается в один безымянный **unnamed module**, поведение ровно как до Java 9. «У нас Java 21» и «мы используем модули» — разные утверждения.

### Module path vs classpath — рубильник всей системы

Все правила из 8.8.1 включаются, только если jar лежит на `--module-path`. Положи тот же jar с тем же `module-info.class` на classpath — дескриптор игнорируется. Я прогнал ту самую пару модулей (со скрытым `internal` и рефлексивным вызовом в него) через `-cp`:

```
java -cp out/com.shop.greeter:out/com.app com.app.Main
Привет, X!          // тот самый рефлексивный вызов в internal — теперь работает
```

Весь classpath собирается в один **unnamed module**: безымянный модуль, который читает всё и не прячет ничего.

![[Pasted image 20260709181958.png]]

Вот почему миграция на Java 9+ была относительно безболезненной: никто не заставлял переходить на модули, classpath продолжил работать — и подавляющее большинство приложений в 2026-м так на нём и живёт.

> [!warning] ⚠️ Подводный камень  
> «У нас в проекте Java 21, значит, мы используем модули» — нет. Вы используете модуляризованный _JDK_ (его не отключить), но ваш код почти наверняка на classpath. Это разные вещи, и интервьюер любит проверить, понимаешь ли ты разницу.

Формально видов модулей три, и их полезно назвать явно: **explicit** (есть `module-info` и он на module path), **automatic** (обычный jar без дескриптора, положенный на module path — см. ниже) и **unnamed** (всё, что на classpath).

### Критично: Maven-модуль ≠ JPMS-модуль

Слово «модуль» перегружено, и их путают постоянно — в том числе кандидаты на собесах. Разведём жёстко.

**Maven-модуль** (multi-module проект из урока 8.6) — единица **сборки**: родительский `pom`, подпроекты, порядок компиляции, артефакты. JVM про Maven-модули не знает ничего — на выходе обычные jar'ы на classpath. **JPMS-модуль** — единица **рантайма и языка**: границы доступа, которые проверяют `javac` и JVM, описан `module-info.java`. Оси ортогональны.

![[Pasted image 20260709182007.png]]

Если интервьюер спросит «у вас модульный проект?» — уточни, про какие модули речь. Сам вопрос-уточнение уже очко в твою пользу: бывает Maven-multi-module без единого `module-info` (норма жизни), бывает JPMS в одном Maven-модуле.

### Обзорно: что ещё есть в системе

Глубоко копать не будем, но знать, что существует, надо. Всё ниже видно в `java --describe-module` (проверено на JDK 21).

`requires transitive` — транзитивная зависимость для потребителей. Если API `com.shop.greeter` возвращает типы из `java.sql`, писать каждому потребителю `requires java.sql` руками — издевательство; `requires transitive java.sql` означает «кто читает меня, автоматически читает и его». `requires static` — зависимость нужна на компиляции, в рантайме опциональна (аннотации, optional-интеграции). **Сервисы** (`provides … with …` + `uses …`) — это `ServiceLoader`, встроенный в модули: DI на минималках без фреймворка; в прикладном коде встретишь редко, в самом JDK — постоянно (JDBC-драйверы, провайдеры `Charset`).

**Automatic modules** — мост для мира без `module-info`. Кладёшь обычный jar на module path — он становится «автоматическим»: экспортирует всё, читает всё, а **имя выводится из имени файла**. Проверено: `jackson-databind-2.17.0.jar` → модуль `jackson.databind` (версия `2.17.0` отрезается, дефисы → точки, помечен `automatic`). Хрупкость очевидна: переименовали файл — у всех, кто написал `requires jackson.databind`, сломалась сборка. Лечится манифест-атрибутом `Automatic-Module-Name` — он фиксирует имя (проверено: jar с любым именем и `Automatic-Module-Name: com.acme.plain` в манифесте становится модулем `com.acme.plain`). Приличные библиотеки его проставляют, даже не имея `module-info`.

> [!warning] ⚠️ Split packages  
> Два jar'а на module path с одним и тем же пакетом `com.shop.util` — **ошибка**: пакет может принадлежать только одному именованному модулю. Поймал два варианта на JDK 21: если это ловит компилятор — `error: module com.app reads package com.shop.util from both mod.a and mod.b`; если оба резолвятся в boot layer на запуске — `LayerInstantiationException: Package com.shop.util in both module mod.a and module mod.b`. На classpath то же самое молча работало (выигрывал первый по порядку). Запретили ради однозначности: доступность считается per-package, а пакет в двух модулях делает правила `exports` неразрешимыми.

### Где JPMS всё-таки выстрелил

Честный расклад, без агитации. Для **платформы** JPMS — успех: JDK распилен на модули, внутренности (`sun.misc.*`, приватные поля `java.lang`) закрыты (`JEP 396`/`403`), и появился `jlink` (`JEP 282`) — инструмент, который собирает **кастомный минимальный рантайм** только из нужных модулей. Hello-world-сервису не нужен `java.desktop` со Swing — `jlink` его выкинет.

Прогнал на нашей паре модулей (проверено на JDK 21):

```
jlink --add-modules com.app,com.shop.greeter --module-path "$JAVA_HOME/jmods:out" --output myruntime

myruntime/bin/java --list-modules
  com.app
  com.shop.greeter
  java.base@21.0.11          // и всё — ни java.sql, ни java.xml, ни java.desktop
```

Размеры на этой машине: **57 МБ** кастомный рантайм против **286 МБ** полного JDK. Для Docker-образов и embedded это реальные деньги. Инструменты рядом: `jdeps --print-module-deps app.jar` показывает, от каких модулей ты реально зависишь (на нашем `com.app` выдал `com.shop.greeter,java.base`) — с этого списка и начинается `jlink`; `java --describe-module <name>` печатает разобранный дескриптор.

> [!tip] 💡 Из C#  
> `jlink` по цели похож на self-contained deployment с trimming в .NET, но механика другая: `jlink` режет по границам **модулей** (выкидывает целые модули JDK, которых нет в графе), а .NET-триммер — по IL-анализу достижимости на уровне отдельных типов и методов. Гранулярность у .NET тоньше, зато `jlink` проще и предсказуемее.

### Почему прикладной код прошёл мимо

Для **прикладного** кода баланс другой — три причины плюс фоновая.

**Экосистема не готова.** Чтобы получить выгоду, _весь_ твой граф зависимостей должен играть по правилам. А тысячи библиотек живут без `module-info`, цепляются через хрупкие automatic modules (имя из имени файла). Один не-модульный кирпич — и стройная картина рассыпается.

**Фреймворки дерутся с инкапсуляцией.** Spring, Hibernate, Jackson живут рефлексией в твои `private`-поля. Модули это запрещают по умолчанию → пиши `opens` на каждый пакет → конфигурационный шум, который ничего не даёт, кроме головной боли при миграции.

**Выгода для сервиса мизерная.** Сильная инкапсуляция ценна, когда охраняешь API от чужих рук — это про _библиотеки_. Типичный сервис пишет одна команда и деплоит одним артефактом; границы там держатся code review и пакетной дисциплиной. Платить модулями за то, что и так под контролем, незачем.

Плюс фоновое: **classpath никто не отменял, и он просто работает** — не было пинка переходить. Итог: JDK распилили, а прод оставили на classpath. Осмысленно берут JPMS в основном авторы библиотек и платформ (сам JDK, JavaFX, часть инфраструктурных либ) и те, кому нужен `jlink`.

### На собесе

> [!question]- Кладём модульный jar (с `module-info.class`) на classpath. Что произойдёт с его `exports`/`requires`?  
> **Model answer (RU):** Ничего — дескриптор игнорируется, все классы попадают в unnamed module, доступно всё, как до Java 9. Включение JPMS — это про module path, а не про версию Java.
> 
> **Model answer (EN):** Nothing — the descriptor is ignored, all classes land in the unnamed module, everything is accessible as before Java 9. Enabling JPMS is about the module path, not the Java version.
> 
> _Follow-up:_ «что такое unnamed module и что он читает?» — единый безымянный модуль для всего classpath; читает все остальные модули, экспортирует все свои пакеты, deep reflection в него разрешён.

> [!question]- Maven-модуль и JPMS-модуль — одно и то же?  
> **Model answer (RU):** Нет. Maven-модуль — единица сборки, JVM о ней не знает. JPMS-модуль — единица рантайма и языка, его границы проверяют `javac` и JVM. Multi-module Maven-проект без единого `module-info` — норма.
> 
> **Model answer (EN):** No. A Maven module is a build unit the JVM knows nothing about. A JPMS module is a runtime/language unit whose boundaries `javac` and the JVM enforce. A multi-module Maven project with no `module-info` at all is the norm.
> 
> _Trap:_ «значит, multi-module Maven-проект — это JPMS?» — нет, оси ортогональны.

> [!question]- Почему индустрия почти не использует JPMS для прикладного кода?  
> **Model answer (RU):** Для платформы — успех: модульный JDK, закрытые internals (`JEP 396`/`403`), `jlink` для минимальных рантаймов. Для прикладного кода трение перевешивает: либы без `module-info`, хрупкие automatic modules, `opens` под каждый фреймворк — а выгода для сервиса, который пишет одна команда, небольшая. Classpath работает — на нём и живём.
> 
> **Model answer (EN):** For the platform it's a success: a modular JDK, encapsulated internals (`JEP 396`/`403`), `jlink` for minimal runtimes. For application code the friction wins: libraries without `module-info`, fragile automatic modules, `opens` per framework — while the payoff for a single-team service is small. Classpath just works, so that's where we stay.
> 
> _Follow-up:_ «а когда бы взяли?» — библиотека с публичным API, который надо жёстко охранять, или нужен минимальный рантайм через `jlink`.

> [!question]- Почему automatic modules считаются хрупкими?  
> **Model answer (RU):** Имя модуля выводится из имени jar-файла; переименование файла ломает все `requires` на него. Чинится атрибутом `Automatic-Module-Name` в `MANIFEST.MF`, который фиксирует стабильное имя. Бонус-балл: automatic module читает в том числе весь classpath (unnamed module) — иначе мост бы не работал.
> 
> **Model answer (EN):** The module name is derived from the jar filename; renaming the file breaks every `requires` that referenced it. Fixed by `Automatic-Module-Name` in `MANIFEST.MF`, which pins a stable name. Bonus: an automatic module also reads the whole classpath (the unnamed module) — otherwise the bridge wouldn't work.

> [!question]- Два jar'а на module path содержат пакет `com.shop.util`. Что будет?  
> **Model answer (RU):** Ошибка: split packages запрещены — пакет может принадлежать только одному именованному модулю (`reads package … from both …` на компиляции или `LayerInstantiationException` на запуске). На classpath то же самое молча работало — выигрывал первый по порядку.
> 
> **Model answer (EN):** An error: split packages are forbidden — a package may belong to only one named module (`reads package … from both …` at compile time, or `LayerInstantiationException` at startup). On the classpath the same thing worked silently — first-on-the-path won.
> 
> _Follow-up:_ «почему запретили?» — однозначность: доступность считается per-package, пакет в двух модулях делает правила `exports` неразрешимыми.
