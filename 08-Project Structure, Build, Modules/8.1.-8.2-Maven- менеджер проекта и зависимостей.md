Пока у тебя один `Main.java`, сборка — это две команды: `javac Main.java` компилирует в байткод, `java Main` запускает. Инструменты не нужны. Ломается всё на первой же чужой библиотеке: нужен JSON — берёшь Jackson (аналог `Newtonsoft.Json`), а библиотека в Java — это `jar`-файл, который сам тянет за собой ещё несколько чужих `jar`'ов. Скачать их руками, прописать в classpath, поймать `NoClassDefFoundError` на недостающей транзитивной зависимости — и это на **одной** библиотеке; на реальном проекте с тремя десятками зависимостей (у каждой — свои) счёт идёт на сотни `jar`'ов, часть в конфликтующих версиях. Руками это физически неделаемо. Ровно эту работу — резолвинг зависимостей, вызов компилятора с правильным classpath, прогон тестов, упаковку — берёт на себя отдельный класс программ, **build tools**.

**Mental model.** Maven — ты в файле `pom.xml` декларируешь _координаты проекта_ и _список нужных библиотек_, а Maven по одной команде качает их со всеми транзитивными зависимостями, зовёт компилятор, гоняет тесты и пакует результат. Идея, на которой держится вся экосистема: **твой проект для Maven — такой же артефакт с координатами, как любая чужая библиотека.**

#### 1. Build tools: класс программ, которым ты уже пользовался

Ручной путь для одной библиотеки выглядит так:

```
javac -cp jackson-databind-2.17.1.jar Main.java     # компилируется
java  -cp .:jackson-databind-2.17.1.jar Main         # падает
# Exception ... NoClassDefFoundError: com/fasterxml/jackson/core/JsonProcessingException
```

`jackson-databind` в рантайме зовёт классы из `jackson-core` и `jackson-annotations` — двух `jar`'ов, которые ты не скачивал. Компиляция твоего кода прошла (эти классы при компиляции _твоего_ файла ещё не нужны), а рушится уже при запуске. Вот тот же механизм на голом JDK, без всяких библиотек (проверено на JDK 21):

```java
class Helper { static String greet() { return "hi from Helper"; } }
public class App {
    public static void main(String[] a) { System.out.println(Helper.greet()); }
}
```

```
javac App.java Helper.java   → компилируется (Helper на месте)
java  -cp . App              → hi from Helper
rm Helper.class              → убираем зависимость, App НЕ перекомпилируем
java  -cp . App
    Exception in thread "main" java.lang.NoClassDefFoundError: Helper
        at App.main(App.java:3)
    Caused by: java.lang.ClassNotFoundException: Helper
```

Код скомпилирован против зависимости, но в рантайме её на classpath нет — `NoClassDefFoundError`. Это ровно та стена, в которую упираешься на чужих `jar`'ах, только умноженная на десятки транзитивных связей. Build tool разбирает этот граф за тебя: сказал «мне нужен Jackson» — получил Jackson плюс `core` плюс `annotations`.

> [!tip] 💡 Из C#  
> Ты этим пользовался каждый день, просто не звал по именам. Когда в Rider жал Run или писал `dotnet build`, работали два инструмента: **MSBuild** читал `.csproj` и командовал компиляцией, **NuGet** по строчкам `<PackageReference Include="Newtonsoft.Json" Version="13.0.3"/>` качал пакеты со всеми зависимостями в `~/.nuget/packages`. Ты ни разу не звал `csc` руками и не качал `.dll` с сайтов. **Maven — это MSBuild и NuGet в одном флаконе, для Java.**
> 
> |Ты знаешь из .NET|В Java-мире|
> |---|---|
> |`.csproj` — описание проекта + список пакетов|`pom.xml`|
> |`dotnet build` / кнопка Run|`mvn package`|
> |nuget.org — откуда качаются пакеты|Maven Central|
> |`~/.nuget/packages` — локальный кэш|`~/.m2/repository`|
> |`PackageReference` — строчка вместо ручного скачивания|`<dependency>`|
> 
> `mvn` — консольная утилита (ставится через sdkman/brew), но в IntelliJ Maven встроен: открываешь папку с `pom.xml` — IDEA подхватывает всё сама, как Rider подхватывает `.sln`.

#### 2. Структура проекта: convention over configuration

Первый вопрос: а _как_ ты говоришь Maven'у «вот мой код»? В `.csproj` глоббинг подбирает `*.cs` вокруг файла. Ответ Maven радикальнее — **никак не говоришь**, раскладка папок зашита в сам инструмент:

```
payments/
├── pom.xml
├── src/
│   ├── main/
│   │   ├── java/          ← прод-код
│   │   └── resources/     ← конфиги, поедут внутрь jar
│   └── test/
│       ├── java/          ← тесты
│       └── resources/
└── target/                ← всё, что Maven сгенерил (в git не коммитится)
```

Положил код в `src/main/java` — скомпилируется. Положил в `src/test/java` — скомпилируется отдельно и прогонится как тесты. Никакой настройки. Это и есть **convention over configuration**: вместо «настрой, как у тебя» — «сделай, как у всех, и настраивать нечего». Выигрыш не абстрактный: любой из миллионов Maven-проектов на GitHub устроен одинаково, поэтому «склонировал чужой репозиторий — и сразу ориентируешься». И не только ты: IntelliJ без единого клика метит `src/main/java` как sources root, CI-сервер знает, где искать результаты тестов.

> [!tip] 💡 Из C#  
> В .NET прод-код и тесты — обычно два отдельных `.csproj` (`Payments` и `Payments.Tests`). В Maven это **один** проект, а разделение — по конвенции путей: `main` против `test`. Меньше файлов, та же изоляция; за счёт чего она держится — увидишь в scopes (§5).

#### 3. pom.xml и GAV: ты — такой же артефакт, как Jackson

Минимальный рабочий `pom.xml`:

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>dev.artem</groupId>
    <artifactId>payments</artifactId>
    <version>1.0.0</version>
    <packaging>jar</packaging>
</project>
```

Шапка с `xmlns` — церемониальный шум, копируется не глядя. Содержательны три строки в середине — **GAV**: `groupId:artifactId:version`. Зачем твоему проекту «адрес»? Потому что для Maven **твой проект и чужая библиотека — одно и то же**: и то и другое — артефакт с координатами. У Jackson адрес `com.fasterxml.jackson.core:jackson-databind:2.17.1`, у тебя — `dev.artem:payments:1.0.0`. Сегодня ты потребляешь чужие артефакты, завтра твой `payments` станет зависимостью другого проекта и подключится точно так же — по координатам. Симметрия полная.

Почему частей три, а не две: `artifactId` уникален только внутри `groupId`. `groupId` — namespace издателя в reverse-domain нотации (как имена пакетов), поэтому в мире спокойно живёт десяток библиотек с именем `utils` — у каждой свой издатель. `packaging` (по умолчанию `jar`) говорит, что на выходе: `jar` для обычной либы, `war` для веб-приложения под сервлет-контейнер, `pom` для родительского/агрегирующего модуля (это тема multi-module, урок 8.6).

> [!tip] 💡 Из C#  
> GAV ↔ `PackageId` + `Version`, но отдельной оси `groupId` у NuGet нет: `Newtonsoft.Json` глобально уникален сам по себе, а принадлежность издателю держится на префиксах и prefix reservation. Maven **формализовал** в координатах то, что NuGet держит на соглашениях.

#### 4. Зависимости: ад с `jar`'ами руками — решён

То, за чем ты бегал руками, в Maven — одна декларация по координатам:

```xml
<dependencies>
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
        <version>2.17.1</version>
    </dependency>
</dependencies>
```

Maven скачает её из **Maven Central** (аналог nuget.org), увидит её собственные зависимости и притащит `jackson-core` с `jackson-annotations` **транзитивно** — те самые два `jar`'а, за которыми ты бегал. Всё осядет в локальном кэше `~/.m2/repository` (аналог `~/.nuget/packages`) — второй раз качаться не будет. Этот `~/.m2` запомни, он ещё выстрелит в уроке 8.2 (`install`).

Главный диагностический инструмент, когда «откуда в classpath приехала эта библиотека?» — `mvn dependency:tree`: печатает дерево всех транзитивных зависимостей с их версиями. Когда две твои библиотеки тянут _разные_ версии одной третьей, Maven разрешает конфликт по правилу «ближайшая в дереве побеждает» — но это отдельная большая тема (dependency mediation, `exclusions`, BOM), она в уроке 8.7.

#### 5. Scopes: classpath в проекте не один

Добавим JUnit для тестов — и упрёмся в новую проблему:

```xml
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.2</version>
        <scope>test</scope>
    </dependency>
```

JUnit нужен, чтобы компилировать и гонять тесты, но в прод-`jar` ему делать нечего. В .NET ты решал это отдельным тестовым `.csproj`. В Maven проект один — и нужен другой механизм.

Ключевая мысль, которую и проверяют на собесе: **classpath в проекте не один.** Их несколько — compile classpath (что видит `javac` на `src/main`), test classpath (что видят компилятор и JUnit на `src/test`), runtime classpath (что нужно работающей программе). **Scope отвечает на вопрос «в каких из них живёт зависимость»** — не «насколько она важна», а буквально множество мест, где она существует.

- **`compile`** (дефолт) — везде. Jackson: ты импортируешь `ObjectMapper` в прод-коде, значит он нужен и компилятору, и в рантайме, и уедет в артефакт.
- **`test`** — только в тестовых classpath. JUnit не поедет в прод-артефакт и не достанется транзитивно тем, кто зависит от тебя. Бонус: напишешь `import org.junit...` в `src/main/java` — ошибка компиляции, его там физически нет. Вот она, изоляция «как два `.csproj`», добытая одним словом.
- **`provided`** — есть при компиляции, но в артефакт не кладётся. Классика — `jakarta.servlet-api`: класс `HttpServlet` нужен компилятору, но класть его в `war` нельзя — свою копию даст сервер приложений (Tomcat), а две копии одних классов в рантайме = беда (см. 🔬).
- **`runtime`** — зеркало `provided`: на компиляции нет, в рантайме есть. Классика — JDBC-драйвер: код пишешь против интерфейсов `java.sql` из самой JDK, а конкретный `postgresql`-драйвер нужен, только когда программа реально идёт в базу. `runtime` физически запрещает `import org.postgresql.*` — компилятор драйвер не видит, случайно привязаться к конкретной СУБД невозможно. Архитектурное правило, закодированное в сборке.
- **`import`** — особый зверь для BOM-файлов, живёт в уроке 8.7. Пока просто знай, что есть. (Ещё бывает `system` — как `provided`, но с ручным путём к `jar`; deprecated, в новом коде не использовать.)

> **Почему две копии `servlet-api` — это авария.** В Tomcat твой `war` грузится своим class loader'ом, а серверные классы (включая `HttpServlet`) — родительским. Резолв любой зависимости всегда начинается с `~/.m2`; если там нет — идёт в Central и кладёт туда же. В уроке 8.2 этот же `~/.m2` станет ещё и точкой локальной публикации (`mvn install`).

> [!warning] ⚠️ Подводные камни
> 
> 1. **Забытый `<scope>test</scope>` — тихая ошибка.** Ничего не падает, всё «работает» — просто JUnit молча едет в прод-артефакт и транзитивно навязывается всем, кто зависит от твоей библиотеки.
> 2. **`provided`-зависимость в `compile` scope** → та самая двойная копия классов и `LinkageError`/`ClassCastException` в контейнере. Не «оно и так работает локально» — локально контейнера нет, поэтому и не стреляет до деплоя.
> 3. **`import` `test`-зависимости в `src/main`** не скомпилируется (её нет в compile classpath прод-кода) — и это фича, а не баг: изоляция ловит ошибку на компиляции.
> 4. **`runtime`-зависимость нельзя `import`'ить в `src/main`** — компилятор её не видит; доступ только рефлексивный (`Class.forName("org.postgresql.Driver")`). Так и живут JDBC-драйверы.
> 5. **Отсутствие `packaging`/версии кажется мелочью**, но `SNAPSHOT`-версия (`1.0.0-SNAPSHOT`) резолвится иначе, чем релизная: Maven считает её изменяемой и может перекачивать. Для воспроизводимости в релизах — только фиксированные версии.

#### На собесе

> [!question]- Что такое Maven?  
> **RU:** Инструмент сборки и управления зависимостями. Декларативный `pom.xml` описывает координаты проекта (GAV) и список библиотек, а Maven по одной команде резолвит их со всеми транзитивными зависимостями из Maven Central, компилирует, тестирует и пакует. Работает на принципе convention over configuration: раскладка папок стандартна, поэтому любой проект собирается одной командой.  
> **EN:** A build and dependency-management tool. A declarative `pom.xml` describes the project's coordinates (GAV) and its libraries; Maven then resolves them with all transitive dependencies from Maven Central, compiles, tests and packages — all from one command. It's built on convention over configuration: the layout is standard, so any project builds with the same command.
> 
> _Слабый ответ:_ «Maven качает библиотеки» — сводит инструмент к одной функции NuGet, теряет сборку и конвенции.

> [!question]- Что такое GAV и зачем три части?  
> `groupId:artifactId:version`. `groupId` — namespace издателя (reverse-domain), `artifactId` уникален внутри группы, `version` — версия. Три части, потому что `artifactId` типа `utils` не уникален глобально — уникальность даёт связка с `groupId`.
> 
> _Follow-up:_ «твой проект — тоже артефакт?» — да, `dev.artem:payments:1.0.0`; для Maven свой проект и чужая либа неразличимы, обе адресуются по GAV. _Trap:_ «а `packaging` — часть координат?» — нет, отдельное поле (`jar`/`war`/`pom`), в GAV не входит.

> [!question]- Что такое scope и какие бывают? _(топ вопрос главы)_  
> **RU:** Scope — множество classpath'ов, в которых присутствует зависимость. `compile` (дефолт) — во всех, едет в артефакт; `test` — только в тестовых classpath, в прод не попадает; `provided` — есть на компиляции, но в артефакт не кладётся, класс даст окружение (servlet-api в Tomcat); `runtime` — зеркально: невидим компилятору `src/main`, но есть в рантайме (JDBC-драйвер); `import` — для BOM.  
> **EN:** A scope is the set of classpaths a dependency lives in. `compile` (default) — all of them, ships in the artifact; `test` — test classpaths only, never reaches prod; `provided` — present at compile time but not packaged, the environment supplies it (servlet-api under Tomcat); `runtime` — the mirror: invisible to the `src/main` compiler but present at runtime (a JDBC driver); `import` — for BOMs.
> 
> _Follow-up 1:_ «`provided` vs `runtime`?» — зеркальные: `provided` = есть на компиляции, нет в артефакте; `runtime` = нет на компиляции, есть в рантайме. Перепутавший зубрил список, а не понял classpath'ы.  
> _Follow-up 2:_ «зависимость в `test` scope — можно `import` в `src/main`?» — нет, ошибка компиляции: её нет в compile classpath прод-кода.  
> _Trap:_ «а если положить `servlet-api` в `compile` и задеплоить war в Tomcat?» — две копии классов из разных загрузчиков → `LinkageError`/`ClassCastException`. Нужен `provided`.
> 
> _Слабый ответ:_ «scope — насколько зависимость важная / для какого этапа» — размывает точную модель «множество classpath'ов» и разваливается на follow-up про `provided`/`runtime`.

> [!question]- Что такое транзитивная зависимость и как посмотреть дерево?  
> Зависимость твоей зависимости: подключил `jackson-databind` — Maven сам притащил `jackson-core` и `jackson-annotations`, потому что они перечислены в `.pom` Jackson. Дерево всех транзитивных связей с версиями — `mvn dependency:tree`.
> 
> _Follow-up:_ «две библиотеки тянут разные версии одной третьей — что будет?» — конфликт разрешается медиацией «ближайшая в дереве побеждает» (детально — урок 8.7). _Trap:_ «что такое `~/.m2`?» — единое адресуемое хранилище: кэш скачанного из Central плюс локально опубликованное; резолв всегда стартует оттуда.
