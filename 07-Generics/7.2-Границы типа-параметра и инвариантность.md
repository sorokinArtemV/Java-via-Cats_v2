
В прошлом уроке ты научился объявлять generic-типы и методы, и почти сразу упрёшься в два ограничения этой машинерии. Первое: внутри generic-метода с «голым» `<T>` ты почти ничего не можешь _сделать_ с `T` — компилятор не даст вызвать ни один метод, кроме унаследованных от `Object`. Второе: типы вроде `List<T>` неожиданно жёстко себя ведут при подстановке — `List<Cat>` нельзя передать туда, где ждут `List<Animal>`, хотя кот очевидно животное. Оба ограничения — не капризы, а осознанные решения дизайна, и у обоих есть точечные способы ослабления. В этом уроке разберём первое решение — **границы типа-параметра** (`<T extends ...>`) — и выведем второе явление, **инвариантность**, вместе с его историческим антиподом — ковариантными массивами. Это фундамент, на котором в следующем уроке встанут wildcards и PECS.

**Mental model.** Граница — это контракт «что `T` гарантированно умеет»: внутри метода она _расширяет_ доступный API, снаружи _сужает_ множество допустимых типов. А инвариантность — это правило «`List<Cat>` и `List<Animal>` — несвязанные типы», единственный способ удержать типобезопасность дженериков целиком в компайл-тайме.

### Проблема: голый `<T>` — это `T extends Object`

Боевая задача: универсальный `max`, который находит максимум в любом списке — `Integer`, `String`, `LocalDate`, что угодно. Пишешь честно:

```java
static <T> T maxOf(List<T> list) {
    T best = list.get(0);
    for (T item : list) {
        if (item.compareTo(best) > 0) best = item;   // ← здесь всё умирает
    }
    return best;
}
```

Компилятор (JDK 21):

```
error: cannot find symbol
  symbol:   method compareTo(T)
  location: variable item of type T
  where T is a type-variable:
    T extends Object declared in method <T>maxOf(List<T>)
```

Посмотри на последнюю строку — компилятор сам проговорился: `T extends Object`. Ты написал `<T>`, а он прочитал `<T extends Object>`. Это одно и то же. И раз всё, что он знает про `T` — «это какой-то `Object`», то внутри метода доступны ровно методы `Object`: `equals`, `hashCode`, `toString` и компания. `compareTo` там нет. Generic-метод без границы умеет работать с чем угодно — и именно поэтому не умеет _делать_ почти ничего.

### Верхняя граница: `<T extends ...>`

Лечится одним словом в объявлении:

```java
static <T extends Comparable<T>> T maxOf(List<T> list) {
    T best = list.get(0);
    for (T item : list) {
        if (item.compareTo(best) > 0) best = item;
    }
    return best;
}

maxOf(List.of(3, 7, 5));              // 7
maxOf(List.of("ant", "zebra", "cat")); // zebra
```

Теперь компилятор знает про `T` больше: «любой тип, но обязательно реализующий `Comparable<T>`». Значит, у каждого `item` гарантированно есть `compareTo` — вызов компилируется, оба вызова проверены на JDK 21.

Граница работает в обе стороны. Внутри метода она _расширяет_ то, что можно делать с `T`. Снаружи — _сужает_ то, что можно передать:

```java
static <T extends Number> double sum(List<T> numbers) {
    double total = 0;
    for (T n : numbers) total += n.doubleValue();   // doubleValue() есть у Number
    return total;
}

sum(List.of(1, 2, 3));      // 6.0 — T выведен как Integer
sum(List.of(1.5, 2.5));     // 4.0 — T = Double
sum(List.of("a", "b"));     // ошибка компиляции
```

На последней строке компилятор честно показывает, как рассуждал:

```
reason: inference variable E has incompatible bounds
    upper bounds: Number,Object
    lower bounds: String
```

«Тип должен быть не выше `Number` и при этом `String`» — противоречие, до свидания. Заметь: это всё компайл-тайм, до рантайма такая ошибка не доживает. Запомни эту фразу — через десять минут она станет главным контрастом с массивами.

> [!tip] 💡 Из C#  
> Граница — это `where`-constraint. `static <T extends Number> double sum(...)` ≈ `static double Sum<T>(...) where T : Number`. Два отличия:
> 
> - В Java ограничение живёт прямо в угловых скобках, до возвращаемого типа, а не хвостом после сигнатуры.
> - Аналогов `where T : new()`, `where T : struct`, `where T : unmanaged` в Java нет вообще — и это не случайный пробел, а следствие стирания типов: в рантайме `T` не существует, вызвать его конструктор физически не из чего. Полный разбор — урок 7.4.

#### Несколько границ сразу

Граница может быть составной — через `&`:

```java
static <T extends Number & Comparable<T>> T maxNumber(List<T> list) { ... }

maxNumber(List.of(3, 9, 4));   // 9 — Integer подходит под обе границы
```

«`T` — это `Number`, и он же `Comparable<T>`». Правило порядка: если среди границ есть класс — он строго первым, дальше сколько угодно интерфейсов. Поменяешь местами:

```java
static <T extends Comparable<T> & Number> void f(T t) {}
// error: interface expected here
```

Класс может быть только один — единичное наследование никто не отменял, два класса в границе физически невыполнимы.

> [!tip] 💡 Из C#  
> Ровно то же правило: в `where T : Base, IComparable<T>` базовый класс обязан идти первым в списке. Редкий случай полного совпадения языков.

#### А `<T super Integer>` существует?

Логичный вопрос: есть верхняя граница — должна быть и нижняя? Нет:

```java
static <T super Integer> void f(T t) {}
// error: > expected
```

Это даже не семантическая ошибка — парсер просто не знает такого синтаксиса. И вот почему его не завезли. Спроси себя: что бы нижняя граница _дала_ внутри метода? Верхняя граница `extends Number` дарит методы `Number` — есть что вызвать. А `super Integer` означало бы «`T` — это `Integer`, или `Number`, или `Object`». Какие методы гарантированы у _всех_ этих типов сразу? Только методы `Object` — то есть ровно то, что даёт голый `<T>` без всякой границы. Нижняя граница на параметре типа не добавляет телу метода ни одной возможности — мёртвый синтаксис.

Но сама идея «ограничить снизу» не мёртвая. Она осмысленна в другом вопросе — не «что я могу вызвать у `T`», а «что я могу _положить_ в контейнер». Этот вопрос возникает на месте использования типа, и там `super` живёт и здравствует — это wildcards, урок 7.3. Пока зафиксируй асимметрию: у параметра типа — только `extends`.

#### Рекурсивная граница

Вернись к сигнатуре `maxOf`:

```java
static <T extends Comparable<T>> T maxOf(List<T> list)
```

`T` ограничен через... сам себя. Выглядит как змея, кусающая хвост, но смысл прямой: «`T` умеет сравниваться _с T же_». Проверим, что теряется без рекурсии — граница через raw-тип `Comparable` компилируется, но `compareTo` там принимает `Object`, и компилятор пропустит сравнение несравнимого:

```java
@SuppressWarnings({"unchecked", "rawtypes"})
static <T extends Comparable> T maxRaw(List<T> list) { /* тот же код */ }

maxRaw(List.of((Comparable) 1, (Comparable) "str"));   // компилируется!
// Exception in thread "main" java.lang.ClassCastException:
//   class java.lang.Integer cannot be cast to class java.lang.String
//     at java.base/java.lang.String.compareTo(String.java:142)
```

Взрыв отложился на рантайм — ровно то, от чего дженерики должны защищать. Рекурсивная граница говорит точнее: `Integer implements Comparable<Integer>` — сравним только с себе подобными, и это проверяется при компиляции.

> [!tip] 💡 Из C#  
> `where T : IComparable<T>` — буква в букву тот же паттерн, ты его наверняка видел. В обоих языках это родственник CRTP из C++. (В уроке 7.3 увидишь, что JDK на самом деле пишет мягче — `Comparable<? super T>` — и почему.)

> [!example] 🔬 Глубже / на синьора  
> **Во что граница стирается.** После компиляции `T` заменяется на свою границу, а не на `Object`. Проверено через `javap -s`:
> 
> ```
> static <T extends java.lang.Number> double twice(T);
>   descriptor: (Ljava/lang/Number;)D
> ```
> 
> В байткоде метод принимает `Number`, и `n.doubleValue()` — это обычный `invokevirtual Number.doubleValue`, без каких-либо кастов. Голый `<T>` стирается в `Object`. При нескольких границах `<T extends A & B>` стирается в _первую_ (leftmost) границу — `A`; вызовы методов `B` компилятор обслуживает вставкой `checkcast`. Отсюда практическое следствие для API-дизайна: порядок границ влияет на erased-сигнатуру, а значит на бинарную совместимость — поменял местами `A & B`, и старые скомпилированные клиенты перестанут линковаться. Полная механика стирания — урок 7.4.

### Инвариантность: `List<Cat>` — это не `List<Animal>`

Теперь сцена, на которой держится вся вторая половина темы. Есть иерархия:

```java
class Animal {}
class Cat extends Animal {}
class Dog extends Animal {}
```

`Cat` — это `Animal`, тут вопросов нет: `Animal a = new Cat()` работает. Интуиция тащит дальше: раз кот — животное, то _список котов_ — список животных? Проверяем:

```java
List<Cat> cats = new ArrayList<>();
List<Animal> animals = cats;
// error: incompatible types: List<Cat> cannot be converted to List<Animal>
```

Нельзя. И это не вредность компилятора — давай выведем, что случилось бы, разреши он это присваивание:

```java
List<Cat> cats = new ArrayList<>();
List<Animal> animals = cats;        // представим, что компилируется
animals.add(new Dog());             // в List<Animal> класть Dog — легально!
Cat cat = cats.get(0);              // достаём из списка КОТОВ... собаку
cat.meow();                         // 💥
```

Каждая строка по отдельности безупречна. `animals` имеет тип `List<Animal>` — положить туда `Dog` абсолютно законно. Но `animals` и `cats` — _одна и та же коробка в куче_, две ссылки на один объект. Через ссылку-«животные» ты протащил собаку в список котов, и вся типобезопасность дженериков рассыпалась. Поэтому компилятор рубит на корню саму первую строку: `List<Cat>` и `List<Animal>` — несвязанные типы. Это называется **инвариантность**: отношение подтипов между `Cat` и `Animal` _не переносится_ на `List<Cat>` и `List<Animal>`.

И заметь: инвариантность не делает исключений даже для `Object` — `List<Cat>` не является и `List<Object>`. Вывод тот же: через `List<Object>`-ссылку в список котов можно было бы положить вообще что угодно.

> [!tip] 💡 Из C#  
> `List<Cat>` → `List<Animal>` не компилируется и в C# — классы там тоже инвариантны. Но ты наверняка помнишь, что `IEnumerable<Cat>` в `IEnumerable<Animal>` присваивается спокойно. Запомни это ощущение «но у нас же можно было» — в уроке 7.3 разберём, почему в C# можно, чем за это заплачено, и как Java решает ту же задачу принципиально другим способом.

### Массивы: дыра, прорубленная в 1995-м

А теперь перепиши тот же опасный код, заменив списки на массивы:

```java
Cat[] cats = new Cat[3];
Animal[] animals = cats;       // компилируется!
animals[0] = new Dog();        // и это компилируется!
```

Обе строки компилятор пропускает. Массивы в Java **ковариантны**: `Cat[]` считается подтипом `Animal[]`. Ровно та дыра, которую дженерики закрыли — здесь распахнута настежь. Запускаем (JDK 21):

```
Exception in thread "main" java.lang.ArrayStoreException: Dog
	at Zoo.main(Zoo.java:5)
```

Программа не сломалась молча — JVM поймала запись собаки в кошачий массив. Как? Механика такая: массив, в отличие от `List<T>`, **знает свой тип элементов в рантайме** — массивы reified. У объекта-массива в куче хранится класс элементов, и спецификация (JLS §10.5) требует: при каждой записи _ссылки_ в массив JVM сверяет класс записываемого объекта с фактическим классом элементов массива. Не совпало — `ArrayStoreException`.

Один и тот же баг — две судьбы:

![[Pasted image 20260708174427.png]]

Дженерики ловят баг на компиляции и до рантайма не доносят. Массивы пропускают компиляцию и заставляют JVM проверять _каждую_ запись ссылки в каждый массив программы — это реальная цена ковариантности.

Зачем Java так сделала? Исторически. В 1995-м дженериков не было, а полиморфные утилиты были нужны: один метод `Arrays.fill(Object[] a, Object val)` должен был работать с любым массивом. Без ковариантности `Cat[]` нельзя было бы передать как `Object[]` — пришлось бы писать перегрузку под каждый тип. Ковариантность массивов — костыль эпохи до дженериков, и проверка типа просто переехала из компилятора в рантайм. Дженерики, появившись в Java 5, сделали противоположный выбор: инвариантность + проверка на компиляции. Отсюда правило из _Effective Java_ (3rd ed., Item 28; во 2-м издании — Item 25): предпочитай `List<T>` массивам — список найдёт твой баг на компиляции, массив — в три часа ночи в проде.

> [!example] 🔬 Глубже / на синьора  
> **`aastore` своими глазами.** Запись ссылки в массив компилируется в байткод-инструкцию `aastore` — `javap -c` на класс с `animals[0] = new Dog()` показывает:
> 
> ```
> 12: dup
> 13: invokespecial #11    // Method Dog."<init>":()V
> 16: aastore
> ```
> 
> Именно в `aastore` встроена рантайм-проверка типа (JLS §10.5, JVMS §6.5): класс записываемого объекта сверяется с _фактическим_ компонентным типом массива, который лежит в его заголовке. Записи примитивов (`iastore`, `dastore` и т.д.) проверки не несут — примитиву не во что «мутировать». Это маленький налог на каждую запись ссылки в любой массив; JIT умеет его устранять, когда доказывает тип статически (например, массив создан в том же методе и никуда не утекал — escape analysis), но в общем случае проверка остаётся.
> 
> **Ещё одно следствие reified-массивов:** `new T[10]` внутри generic-кода невозможен — массиву нужен реальный класс элементов в рантайме, а `T` к этому моменту стёрт. Полный список таких запретов — урок 7.4.

> [!tip] 💡 Из C#  
> Массивы ковариантны и там — `Cat[]` присваивается в `Animal[]`, а кривая запись падает с `ArrayTypeMismatchException`. Та же дыра по той же исторической причине: поведение заложено в CLR с первой версии ради совместимости (и, по признанию Эрика Липперта, ради паритета с Java). Здесь языки — честные близнецы, интуиция переносится один в один.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`extends` в границе — и для интерфейсов тоже.** `<T extends Comparable<T>>` — граница-интерфейс, но пишется `extends`, не `implements`. В контексте границ `extends` означает просто «является подтипом», без различия класс/интерфейс. Глаз, привыкший к `class Foo implements Comparable`, спотыкается об это первые пару недель.
> 2. **`<T super X>` не существует.** Синтаксическая ошибка на уровне парсера. `super` легален только у wildcards (урок 7.3) — на собесе любят проверять, различаешь ли ты эти две позиции.
> 3. **Порядок в составной границе:** класс — строго первым, `<T extends Comparable<T> & Number>` не компилируется (`interface expected here`). И класс может быть только один.
> 4. **Забытая рекурсия → raw `Comparable`.** `<T extends Comparable>` компилируется (с warning), но `compareTo` принимает `Object` — сравнение `Integer` со `String` пролезет и взорвётся `ClassCastException` в рантайме. Всегда `Comparable<T>` (а ещё лучше — `Comparable<? super T>`, увидишь в 7.3).
> 5. **Инвариантность без исключений:** `List<Cat>` — не `List<Animal>` и не `List<Object>`. Метод `print(List<Object>)` не примет вообще ни один список, кроме буквально `List<Object>`.
> 6. **Ковариантный массив — мина замедленного действия.** `Object[] arr = new String[2]; arr[0] = 42;` компилируется чисто и падает в рантайме — причём взорваться может далеко от места, где массив «повернули» к базовому типу. В API не принимай `Object[]`, если собираешься туда писать.

Промежуточный итог в одну строку: дженерики инвариантны, потому что инвариантность — единственный способ держать проверку «никто не подсунет собаку котам» в компайл-тайме. Но плата заметная — `feedAll(List<Animal>)` не принимает `List<Cat>`, хотя метод только читает и никого никуда не подсовывает. Слишком строго. Способ ослабить хватку точечно, на месте использования — wildcards `? extends` / `? super` и PECS. Это урок 7.3.

### На собесе

> [!question]- Почему дженерики в Java инвариантны? _(топ темы, спрашивают почти всегда)_  
> **Model answer (RU):** `List<Cat>` нельзя считать `List<Animal>`: обе ссылки указывали бы на один объект, и через ссылку типа `List<Animal>` в список котов легально положили бы `Dog` — каждая строка по отдельности корректна, а вместе они ломают типобезопасность. Вся проверка дженериков живёт в компайл-тайме, рантайм-страховки, как у массивов, нет — поэтому единственный способ гарантировать безопасность — объявить `List<Cat>` и `List<Animal>` несвязанными типами.
> 
> **Model answer (EN):** `List<Cat>` can't be treated as `List<Animal>`: both references would point to the same object, and through the `List<Animal>` reference you could legally add a `Dog` into the list of cats — every line is individually fine, together they break type safety. Generic type checking lives entirely at compile time, there's no runtime safety net like arrays have — so the only way to stay safe is to make `List<Cat>` and `List<Animal>` unrelated types.
> 
> _Follow-up:_ «Но метод, который только читает, страдает зря — как быть?» → wildcards `? extends` (урок 7.3). _Trap:_ «А `List<Cat>` в `List<Object>` можно?» — нет, инвариантность без исключений.
> 
> _Слабые ответы:_ «Так устроен язык» — ноль механизма. «Потому что типы стираются» — путает причину и следствие: инвариантность нужна, потому что проверка _компайл-тайм_; стирание — отдельная тема.

> [!question]- Чем массивы отличаются от дженериков по вариантности?  
> **Model answer (RU):** Массивы ковариантны и reified: `Cat[]` — подтип `Animal[]`, а тип элементов хранится в объекте массива, и JVM проверяет каждую запись ссылки (инструкция `aastore`, JLS §10.5), бросая `ArrayStoreException` при несовпадении. Это наследие 1995 года: до дженериков иначе не было полиморфных утилит вроде `Arrays.fill(Object[], Object)`. Дженерики выбрали противоположное — инвариантность с проверкой на компиляции. Отсюда Effective Java Item 28: предпочитай списки массивам.
> 
> **Model answer (EN):** Arrays are covariant and reified: `Cat[]` is a subtype of `Animal[]`, the element type is stored in the array object, and the JVM checks every reference store (the `aastore` instruction, JLS §10.5), throwing `ArrayStoreException` on mismatch. It's a 1995 legacy: before generics there was no other way to write polymorphic utilities like `Arrays.fill(Object[], Object)`. Generics made the opposite choice — invariance checked at compile time. Hence Effective Java Item 28: prefer lists to arrays.
> 
> _Follow-up:_ «Что выведет `Object[] arr = new String[2]; arr[0] = 42;` и где именно?» — компилируется целиком, падает на второй строке: `42` боксится в `Integer`, `aastore` в `String[]` → `ArrayStoreException: java.lang.Integer`. Точность «на какой строке» отличает того, кто запускал, от того, кто читал. _Follow-up 2:_ «Почему дженерики не могут проверять так же в рантайме?» — тип-аргументы стёрты, `List<Cat>` в рантайме не знает, что он «of Cat» (урок 7.4) — но одной фразой ответить обязан уже сейчас.

> [!question]- Зачем нужны bounded type parameters? Напиши универсальный `max`.  
> **Model answer (RU):** Голый `<T>` — это неявно `<T extends Object>`: внутри метода доступны только методы `Object`, `compareTo` вызвать нельзя. Граница `<T extends Comparable<T>>` даёт компилятору контракт «T умеет сравниваться с T» — внутри метода открывается API границы, снаружи сужается множество допустимых типов, и оба эффекта проверяются на компиляции. Сигнатура: `static <T extends Comparable<T>> T maxOf(List<T> list)`.
> 
> **Model answer (EN):** A bare `<T>` is implicitly `<T extends Object>`: inside the method only `Object`'s methods are available, you can't call `compareTo`. The bound `<T extends Comparable<T>>` gives the compiler the contract "T can compare itself to T" — inside the method it unlocks the bound's API, outside it narrows the set of acceptable types, both checked at compile time. Signature: `static <T extends Comparable<T>> T maxOf(List<T> list)`.
> 
> _Follow-up:_ «Почему граница на самом себе — `Comparable<T>` через `T`?» — рекурсивная граница (родственник CRTP): без неё raw `Comparable` принимает `Object`, и сравнение `Integer` со `String` отложится до `ClassCastException` в рантайме. _(senior add-on)_ В JDK сортировка объявлена мягче — `Comparable<? super T>`, чтобы работали подклассы, унаследовавшие `compareTo` (разбор в 7.3).

> [!question]- Можно ли объявить `<T super Integer>` у метода?  
> **Model answer (RU):** Нет — такого синтаксиса не существует, парсер даёт ошибку. И он был бы бесполезен: нижняя граница означала бы «T — это Integer, Number или Object», а общий гарантированный API всех супертипов вплоть до `Object` — это API самого `Object`, то есть ровно то, что даёт `<T>` без границы. Телу метода она не добавила бы ничего.
> 
> **Model answer (EN):** No — the syntax doesn't exist, it's a parser error. And it would be useless: a lower bound would mean "T is Integer, Number, or Object", and the guaranteed common API of all supertypes up to `Object` is just `Object`'s API — exactly what a bare `<T>` already gives. It would add nothing to the method body.
> 
> _Follow-up:_ «А где `super` тогда легален?» — только у wildcards, где вопрос не «что я могу вызвать у T», а «что я могу положить в контейнер». _Как отвечать:_ это вопрос на отсев зубрёжки — выведи бессмысленность, не ограничивайся «так нельзя».

> [!question]- Сколько границ может быть у типа-параметра и в каком порядке?  
> Сколько угодно, через `&`: `<T extends Number & Comparable<T>>`. Класс — максимум один и строго первым, дальше интерфейсы; нарушение порядка — ошибка компиляции `interface expected here`.
> 
> _Trap:_ «а два класса?» — невозможно, единичное наследование. _(senior)_ «Во что стирается такой T?» — в первую (leftmost) границу; доступ к членам остальных границ компилятор обслуживает `checkcast`-ами, и смена порядка границ ломает бинарную совместимость.

---

#### Практика (опционально)

1. **Сломай `maxOf` всеми способами** (~15 мин, голый JDK): убери границу, замени на raw `Comparable`, поменяй порядок в `Number & Comparable<T>`, попробуй `<T super Number>`. Каждую ошибку компиляции прочитай вслух и переведи на русский — «компилятор знает про T только...».
2. **`aastore` своими глазами** (~15 мин, `javap`): воспроизведи `ArrayStoreException`, потом `javap -c` — найди `aastore` в байткоде записи в массив. Для контраста посмотри, во что компилируется `list.add(...)`.
3. **Стирание границы** (~10 мин): `javap -s` на метод `<T extends Number>` — найди descriptor `(Ljava/lang/Number;)D` и сравни с descriptor'ом метода с голым `<T>`.
