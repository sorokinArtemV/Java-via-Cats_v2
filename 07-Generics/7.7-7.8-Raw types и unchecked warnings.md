### 7.7. Raw types
#### Что это и откуда взялось

Вспомни 2004 год: выходит Java 5 с генериками, а в мире — миллиарды строк на голых `List` и `Map`, потому что других не было. Если бы `List` без аргументов перестал компилироваться, апгрейд на Java 5 означал бы «перепиши всё и сразу». Вместо этого raw оставили легальными: старый код компилируется как есть, новый generic-код зовёт старые библиотеки, старые библиотеки принимают новые коллекции. Это **migration compatibility** — и это та же причина, по которой для генериков выбрали _erasure_, а не reified-типы. Одно проектное решение, два следствия: raw существует именно потому, что генерики стёрты.

> [!tip] 💡 Из C#  
> В C# raw types не существуют как категория. `List` без аргумента — просто не тип, ошибка компиляции. А не-generic коллекции (`ArrayList`, `Hashtable` из `System.Collections`) — это _другие классы_, живущие параллельно с `List<T>`, а не его «сырая» форма. Корневой контраст всего блока про генерики: Java дооснастила генериками _существующие_ типы (отсюда erasure, raw и unchecked-варнинги), C# 2.0 добавил _новые параллельные_ типы рядом со старыми (отсюда ни того, ни другого, ни третьего). Весь этот урок — про цену, которую Java заплатила за свой путь.

#### Что raw делает на самом деле

Интуиция «не указал тип — элементы будут `Object`, читать неудобно, но не страшно» — неверна. Реальность хуже. Код скомпилирован и запущен на JDK 21:

```java
List<String> strings = new ArrayList<>();
strings.add("hello");

List raw = strings;          // легально
raw.add(42);                 // компилируется!

String s = strings.get(1);   // 💥 ClassCastException
```

`raw` — другая _ссылка_ на тот же объект, но через сырой тип. Через неё `raw.add(42)` компилируется, потому что raw выключает generic-проверки: `add(E)` вырождается в `add(Object)`, и `Integer` заходит без сопротивления. В куче теперь `List<String>`, внутри которого живёт `Integer` — это состояние называется **heap pollution** (заражение кучи).

А взрывается оно в другом месте. Рантайм на JDK 21, дословно:

```
Exception in thread "main" java.lang.ClassCastException:
    class java.lang.Integer cannot be cast to class java.lang.String
        at HeapPollution.main(HeapPollution.java:11)
```

Строка 11 — это `strings.get(1)`. **Не** строка `raw.add(42)`, где совершено преступление, а строка, где честный типизированный код читает из списка. Компилятор при стирании вставил в точку чтения синтетический `checkcast String` (ты видел его в `javap` в уроке про erasure) — и рванул именно он. Яд заходит через raw в одном месте, детонирует через типизированную ссылку в другом — возможно, в другом классе, другом модуле, через неделю в проде. Stack trace показывает на невиновного.

Настоящая цена raw — не «неудобно читать», а разрыв между местом ошибки и местом взрыва.

> [!warning] ⚠️ Подводный камень: направление опасно несимметрично  
> Само присваивание `List raw = strings` (типизированный → raw) даже unchecked-варнинга не даёт — только `rawtypes`-варнинг, и то лишь под `-Xlint:rawtypes`. Проверено: под дефолтным `javac` и под `-Xlint:unchecked` эта строка молчит. Опасное с точки зрения компилятора направление — обратное (raw → типизированный присваивание) и любая _запись_ через raw. Сужение проверок несимметрично, и это ловит.

#### Три формы, которые путают: `List<Object>` vs `List<?>` vs `List`

Вопрос, который реально задают на Middle+: «объясни разницу». Внешне все три — «список чего угодно». Выведем различия на коде, всё с того же `List<String> strings` (сообщения `javac` дословные):

```java
List<Object> objects = strings;   // ❌ error: incompatible types:
                                  //    List<String> cannot be converted to List<Object>
List<?> wildcard = strings;       // ✅
List raw = strings;               // ✅ (rawtypes warning под -Xlint)
```

`List<Object>` — конкретный тип: список, куда _гарантированно_ можно класть что угодно. Именно поэтому `List<String>` ему не присваивается — инвариантность из урока про wildcards: иначе через `List<Object>`-ссылку в список строк заехал бы `Integer`. Он полностью типобезопасен, просто его тип-аргумент — `Object`.

`List<?>` — «список чего-то, но я не знаю чего». Читать можно (как `Object`), писать нельзя ничего, кроме `null`:

```java
wildcard.add("x");
// error: incompatible types: String cannot be converted to CAP#1
//   where CAP#1 is a fresh type-variable: CAP#1 extends Object from capture of ?
```

Компилятор рассуждает: вдруг это `List<Integer>`? Не знаю — значит, запрещаю запись. `List<?>` — это типобезопасное _незнание_.

Raw `List` — «мне плевать»: читать как `Object`, писать что угодно, проверки выключены.

Идеологический пуант: `List<?>` и raw `List` внешне почти неразличимы (оба «какой-то список», оба читаются как `Object`), но это **противоположности**. Wildcard: «я не знаю тип — компилятор, защити меня по максимуму». Raw: «я не знаю тип — компилятор, отвернись». Первый запрещает запись, второй разрешает любую. Когда в legacy видишь raw, правильная замена в 90% случаев — именно `List<?>`: семантика «принимаю любой список» сохраняется, дыра закрывается.

![[Pasted image 20260709151847.png]]

Та же тройка как таблица для быстрого перечитывания — что компилируется, а что нет (`strings` — это `List<String>`):

|Операция|`List<Object>`|`List<?>`|raw `List`|
|---|---|---|---|
|`x = strings;` (присвоить `List<String>`)|❌ ошибка (инвариантность)|✅|✅ (`rawtypes` warn)|
|Чтение `x.get(0)`|как `Object`|как `Object`|как `Object`|
|Запись `x.add(...)`|любой объект ✅|только `null`|что угодно ✅ (unchecked)|
|Типобезопасность|полная|полная (через запрет записи)|выключена|

> [!tip] 💡 Из C#  
> Ближайший родственник `List<?>` — `IEnumerable<object>` благодаря ковариантности `out T`, но точного аналога wildcard capture (тот самый `CAP#1` в ошибке) в C# нет, как нет и аналога raw. А `List<object>` в C# — ровно то же, что `List<Object>` в Java: инвариантный конкретный тип, `List<string>` ему не присваивается.

#### Где raw — не лень, а единственная легальная форма

Если унесёшь «raw = всегда зло» — унесёшь неправду. Есть два места, где язык _сам навязывает_ raw, потому что параметризованной формы физически не существует в рантайме.

**Литерал класса.** `List.class` легально, `List<String>.class` — синтаксическая ошибка:

```java
Class<?> c = List<String>.class;   // error: <identifier> expected
```

В рантайме существует ровно один объект `Class` для `List`; класса `List<String>` после стирания просто нет — не на что ссылаться.

**`instanceof`.** Та же логика (проверено, дословно):

```java
obj instanceof List<String>   // error: Object cannot be safely cast to List<String>
obj instanceof List           // ✅ — единственная честная проверка
obj instanceof List<?>        // ✅ — эквивалентна raw-проверке (JLS §15.20.2)
```

JVM в рантайме не может проверить тип-аргумент — его стёрли. Поэтому `instanceof` принимает только raw или эквивалентный unbounded `List<?>`. Это callback к уроку про невозможные операции: там ты видел, _почему_ операции невозможны, здесь — следствие: компилятор в этих двух точках _требует_ raw. На собесе разделение «raw, навязанный языком» vs «raw как неряшливость» звучит как глубина, а не как заученное правило.

#### Raw заражает весь объект, а не только `T`

Неочевидный эффект: обращение через raw-ссылку стирает генерики **у всех членов** объекта, даже тех, что не используют его `T`. Канонический пример — `Map.entrySet()` (проверено):

```java
Map raw = new HashMap<String, Integer>();

for (Map.Entry<String, Integer> e : raw.entrySet()) {   // ❌
    ...
}
// error: incompatible types: Object cannot be converted to Entry<String,Integer>
```

У raw `Map` метод `entrySet()` возвращает не `Set<Map.Entry<K,V>>`, а raw `Set`, элементы которого — голые `Object`. Почему: raw — это взгляд на класс _как будто генериков не существует вообще_. Не «`K` и `V` стали `Object`», а «вся generic-информация сигнатур выключена» — это прописано в JLS §4.8: тип члена raw-типа есть _erasure_ его типа. Один raw в начале цепочки вызовов — и вся цепочка дальше работает с `Object` и кастами. Поэтому raw в сигнатуре публичного API заражает всех, кто им пользуется.

### 7.8. Unchecked warnings и `@SuppressWarnings`

#### Что компилятор пытается тебе сказать

Скомпилируй пример с heap pollution обычным `javac` — увидишь не варнинг, а вот это (дословно):

```
Note: HeapPollution.java uses unchecked or unsafe operations.
Note: Recompile with -Xlint:unchecked for details.
```

По умолчанию `javac` детали _прячет_. Реальная работа начинается с `-Xlint:unchecked`:

```
HeapPollution.java:9: warning: [unchecked] unchecked call to add(E)
    as a member of the raw type List
        raw.add(42);
               ^
```

В Maven это `<compilerArgs><arg>-Xlint:unchecked</arg></compilerArgs>` у compiler-плагина; в `IDEA` инспекции подсвечивают и без флага. Но знать, что дефолтный `javac` молчит о деталях, надо — иначе будешь уверен, что варнингов нет, когда их двадцать.

Теперь — что варнинг _означает_. Два главных триггера:

```java
List<String> a = (List<String>) obj;   // unchecked cast
List<String> b = rawList;              // unchecked conversion
```

В обоих утверждается факт про тип-аргумент — «это список именно строк» — а проверить его JVM не может: в рантайме `<String>` стёрт, `checkcast` подтвердит только, что это `List`. Компилятор честно признаётся: **безопасность здесь держится на информации, которая живёт в стёртом `T`, — я её проверить не могу, отвечаешь ты**. Unchecked warning помечает ровно ту точку, через которую heap pollution может зайти в программу. Это не шум — это карта дыр в типобезопасности.

> [!tip] 💡 Из C#  
> Категории «unchecked, потому что тип стёрт» в C# нет вообще — стирания нет, проверять на веру нечего. Небезопасный каст `(List<string>)obj` в C# либо проходит, либо кидает `InvalidCastException` **прямо в точке каста**: рантайм сверяет настоящий тип-аргумент. В Java та же строка молча «проходит», а взрыв откладывается до первого чтения не того типа. Отложенный взрыв — вот что компенсируют варнинги.

#### `@SuppressWarnings("unchecked")` — дисциплина, не глушилка

Иногда ты _знаешь_, что каст безопасен, а компилятор знать не может. Канон — generic-массив из урока про невозможные операции:

```java
// Безопасно: copyOf создаёт массив рантайм-типа a, T[] по построению
@SuppressWarnings("unchecked")
T[] result = (T[]) Arrays.copyOf(elements, size, a.getClass());
return result;
```

`Arrays.copyOf` с `a.getClass()` создаёт массив _настоящего_ рантайм-типа `a` — каст держится по построению, доказать это компилятору невозможно. Дисциплина — по _Effective Java_, Item 27 «Eliminate unchecked warnings» (3-е изд.; во 2-м — Item 24):

1. **Сначала устрани, не подавляй.** Большинство варнингов чинится без аннотации: добавь тип-аргументы, поставь diamond, замени raw на `List<?>`. Подавление — последний ход.
2. **Подавляй, только доказав безопасность рассуждением.** Не «варнинг мешает», а «я могу объяснить, почему каст не может упасть». Не можешь объяснить — у тебя не варнинг лишний, у тебя код дырявый.
3. **Минимальный scope.** Аннотация — на самое узкое объявление: локальная переменная > метод > _никогда_ класс. `@SuppressWarnings` на классе — это «выключить пожарную сигнализацию во всём здании, потому что подгорел тост»: следующая настоящая дыра пройдёт незамеченной.
4. **Комментарий, почему безопасно.** Иначе ревьюер (и ты сам через год) не отличит «доказано» от «заглушено, чтоб отстало».

Заметь деталь: переменная `result` существует _только ради аннотации_. Написать `return (T[]) Arrays.copyOf(...)` одной строкой нельзя — `@SuppressWarnings` вешается на объявления, не на statement. Проверено:

```java
@SuppressWarnings("unchecked")
return (T[]) Arrays.copyOf(...);   // error: illegal start of type
```

Заводишь локальную переменную ради аннотации — scope подавления сжимается до одной строки. Это стандартная идиома, она в JDK повсюду (загляни в `ArrayList.toArray(T[])`).

Финальный принцип — **гигиена warning-clean**. Если база чиста от варнингов, каждый _новый_ — событие, на него смотрят. Если варнингов триста, триста первый — heap pollution в проде — утонет в шуме.

> [!warning] ⚠️ Подводный камень: `"rawtypes"` ≠ `"unchecked"`  
> У `@SuppressWarnings` есть и другие категории. `"unchecked"` глушит небезопасные касты/конверсии, `"rawtypes"` — варнинги о самих raw-объявлениях. В interop-коде с legacy часто видишь парное `@SuppressWarnings({"unchecked", "rawtypes"})`. Принцип тот же: минимальный scope, обоснование, не на класс.

#### 🔬 Глубже / на синьора

> [!example] 🔬 Что именно детонирует — байткод чтения  
> Почему взрыв на `get`, а не на `add`? Потому что стирание перенесло проверку на сторону чтения. `javap -c` для `String read(List<String> l) { return l.get(0); }`:
> 
> ```
> invokeinterface List.get:(I)Ljava/lang/Object;   // стёрто до Object
> checkcast     #13   // class java/lang/String     ← синтетическая проверка
> areturn
> ```
> 
> `get` возвращает `Object` (erasure), и компилятор сам вставил `checkcast String`. Именно он бросает `ClassCastException`. Через raw-ссылку на _запись_ никаких `checkcast` нет — там проверок нет вовсе. Отсюда следствие для follow-up: **если элемент так и не прочитают как `String`, не упадёт никогда.** Проверено: `strings.size()` и `strings.toString()` на заражённом списке отрабатывают спокойно — `toString` печатает `[hello, 42]`. Заражение живёт незаметно, пока кто-то не прочитает элемент по стёртому типу.
> 
> **JLS §4.8** формализует «raw заражает всё»: у raw-типа тип _каждого_ члена (поля, возврата, параметра) — это _erasure_ его generic-типа, а не точечная замена `T → Object`. `entrySet()` у raw `Map` возвращает raw `Set`, потому что стёрта вся сигнатура, а не только `K`/`V`.
> 
> **`var` ≠ raw** (Java 10, JEP 286) — их регулярно путают. `var list = new ArrayList<>()` даёт **`ArrayList<Object>`**, не raw: diamond без целевого контекста инферится в `Object`. Проверено — компилятор в диагностике прямо пишет `ArrayList<Object>`. `list.add(42)` пройдёт, но `String s = list.get(0)` — ошибка компиляции (`Object cannot be converted to String`): проверки _работают_. Raw `ArrayList` пропустил бы и второе. «var без типа = raw» звучит логично и неверно.
> 
> **Эволюция.** Raw и unchecked — с Java 5 (2004), ровесники генериков, и это навсегда: убрать raw — сломать ту самую совместимость, ради которой их оставили. Diamond `<>` (Java 7, Project Coin) убрал главный _соблазн_ писать raw от лени — `new HashMap<>()` не длиннее `new HashMap()`.

#### ⚠️ Подводные камни

> [!warning] ⚠️ Подводные камни
> 
> 1. **Взрыв не там, где баг.** `ClassCastException` падает не в точке записи через raw, а в точке первого чтения через типизированную ссылку — часто в другом классе. Stack trace показывает на невиновный код.
> 2. **`List raw = strings` молчит.** Присваивание типизированный → raw не даёт unchecked-варнинга, только `rawtypes` под флагом. Опасны обратное присваивание и любая запись через raw.
> 3. **Один raw заражает всю цепочку.** `raw.entrySet()`, `raw.stream()` и прочее возвращают raw-результаты — генерики выключаются у всех членов (JLS §4.8), а не только у `T`.
> 4. **`@SuppressWarnings` на классе/методе глушит и будущие дыры.** Вешай на локальную переменную; на `return` повесить нельзя (`illegal start of type`) — заводи переменную ради аннотации.
> 5. **`var` принимают за raw.** `var x = new ArrayList<>()` — это `ArrayList<Object>` с работающими проверками, не сырой тип.
> 6. **Дефолтный `javac` прячет детали.** «У нас нет варнингов» без `-Xlint:unchecked` (или `-Xlint:all`) ничего не значит.

#### На собесе

> [!question]- Чем `List`, `List<?>` и `List<Object>` отличаются? _(топ вопрос темы)_  
> **Model answer (RU):** `List<Object>` — конкретный безопасный тип: класть можно всё, но `List<String>` ему не присваивается из-за инвариантности. `List<?>` — любой список: чтение как `Object`, запись запрещена (кроме `null`) — типобезопасное «не знаю». Raw `List` — любой список, чтение и запись без проверок. Ключ: wildcard сохраняет типобезопасность через запрет записи, raw выключает её целиком. `List<?>` и raw внешне почти одинаковы, но по смыслу противоположны.
> 
> **Model answer (EN):** `List<Object>` is a concrete, type-safe type — you can put anything in, but `List<String>` isn't assignable to it due to invariance. `List<?>` is any list: reads as `Object`, writes forbidden except `null` — a type-safe "I don't know". Raw `List` is any list with all checks off, read and write. The key: a wildcard keeps type safety by forbidding writes, raw drops it entirely. `List<?>` and raw look almost identical but are opposites.
> 
> _Follow-up:_ «когда какой использовать?» — `List<Object>` при реально гетерогенном списке; `List<?>` в сигнатурах, принимающих любой список; raw — только class-литералы и `instanceof`. _Trap:_ если добавишь «raw нигде, кроме двух навязанных языком мест» — это сильный ответ.
> 
> _Слабые ответы:_ «все три — список чего угодно» — стирает разницу. «`List<?>` = `List<Object>`» — путает инвариантный конкретный тип с wildcard. «raw безопасен, просто элементы `Object`» — упускает heap pollution.

> [!question]- Через raw-ссылку в `List<String>` добавили `Integer`. Где упадёт?  
> **Model answer (RU):** Не в точке `add`, а в точке первого _чтения_ через типизированную ссылку — там компилятор вставил синтетический `checkcast`. Запись через raw не проверяется вообще; стирание перенесло проверку на сторону чтения. Состояние называется heap pollution.
> 
> **Model answer (EN):** Not at the `add`, but at the first _read_ through the typed reference, where the compiler inserted a synthetic `checkcast`. The raw write isn't checked at all; erasure moved the check to the read side. The state is called heap pollution.
> 
> _Follow-up:_ «а если элемент так и не прочитают как `String`?» — не упадёт никогда; `size()` или `toString()` отработают спокойно. Это и страшно: заражение живёт незаметно. _Trap:_ назвать точную строку в stack trace — это строка чтения, а не записи.

> [!question]- Почему `instanceof List<String>` не компилируется, а `instanceof List` — да?  
> **Model answer (RU):** Рантайм-проверка требует информации, существующей в рантайме, а `<String>` стёрт — JVM физически нечем проверить (`Object cannot be safely cast to List<String>`). После стирания `List` — единственное, что осталось, поэтому legal именно raw.
> 
> **Model answer (EN):** A runtime check needs information present at runtime, but `<String>` is erased — the JVM has nothing to check against. After erasure only `List` remains, so raw is the legal form.
> 
> _Follow-up:_ «а `instanceof List<?>`?» — разрешён (JLS §15.20.2), семантически идентичен raw-проверке. «А в C#?» — `is List<string>` работает: reified-генерики хранят тип-аргумент в объекте.

> [!question]- Тимлид требует сборку без варнингов. Как правильно убрать unchecked warning?  
> **Model answer (RU):** По приоритету: сначала _устранить_ (типизировать, diamond, `List<?>` вместо raw). Если каст реально нужен и безопасность доказуема — `@SuppressWarnings("unchecked")` на минимальном scope (локальная переменная) с комментарием-обоснованием. База держится warning-clean, чтобы новый варнинг был событием. Effective Java, Item 27.
> 
> **Model answer (EN):** In order: first _eliminate_ it (add type arguments, diamond, `List<?>` instead of raw). If a cast is genuinely needed and provably safe — `@SuppressWarnings("unchecked")` on the narrowest scope (a local variable) with a justifying comment. Keep the build warning-clean so a new warning is an event. Effective Java, Item 27.
> 
> _Follow-up (ловушка):_ «почему не повесить на класс и забыть?» — подавление на классе глушит и все _будущие_ варнинги, включая настоящие дыры; сигнализация перестаёт работать. _Trap:_ приём «локальная переменная ради аннотации, потому что на `return` её не повесить» отдельно ценится.

> [!question]- Получили raw `Map` из legacy. Почему `for (Map.Entry<K,V> e : raw.entrySet())` не компилируется?  
> **Model answer (RU):** У raw-типа стираются генерики _всех_ членов: `entrySet()` возвращает raw `Set` с элементами `Object`, а не `Set<Map.Entry<K,V>>` (`Object cannot be converted to Entry<...>`). Raw — это взгляд на класс «как до Java 5» целиком, а не точечная замена `T → Object` (JLS §4.8).
> 
> **Model answer (EN):** A raw type erases the generics of _all_ its members: `entrySet()` returns a raw `Set` of `Object`, not `Set<Map.Entry<K,V>>`. Raw is a "pre-Java-5" view of the whole class, not a pointwise `T → Object` swap (JLS §4.8).
> 
> _Follow-up:_ «как чинить?» — на границе с legacy сделать один unchecked-каст в типизированный `Map<K,V>` с подавлением и обоснованием, и дальше работать только с типизированной ссылкой. Дыру локализуют на границе, не размазывают.

> [!question]- `var list = new ArrayList<>()` — это raw?  
> **Model answer (RU):** Нет, это `ArrayList<Object>` — diamond без целевого типа инферится в `Object`. Проверки работают: положить можно всё, но достать `String` без каста нельзя. `var` — это инференс настоящего типа, а raw — отключение генериков.
> 
> **Model answer (EN):** No — it's `ArrayList<Object>`; the diamond with no target type infers `Object`. Checks work: you can put anything in, but you can't pull out a `String` without a cast. `var` infers a real type; raw disables generics.
> 
> _Follow-up:_ «как получить raw через var?» — только явным raw-выражением справа: `var raw = (List) something`. Сам `var` raw не производит.
