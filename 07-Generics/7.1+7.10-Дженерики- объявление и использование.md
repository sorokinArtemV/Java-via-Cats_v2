Рано или поздно понадобится контейнер, который хранит одну вещь: положил — достал. Напишешь его под `String` — а через неделю нужен такой же под `Order`, потом под `User`. Копировать класс целиком ради одного слова внутри — очевидная глупость, а спрятать всё за `Object` — значит вернуть себе ручные касты и `ClassCastException` в рантайме. Дженерики — это способ написать контейнер (или метод) **один раз, для любого типа, не теряя проверок компилятора**. На собесах их спрашивают постоянно, и почти всегда — на стыке с C#, где они устроены принципиально иначе.

**Mental model.** Параметр типа — это **дырка под тип**: место, которое заполнят конкретным типом позже. Ту же дырку можно повесить на класс, на метод или на конструктор. `T` — не значение, а тип-аргумент; компилятор подставляет его и проверяет, а в рантайме от него мало что остаётся (стирание — урок 3).

### 1. Откуда растёт проблема

Заведём коробку под строку:

```java
class StringBox {
    private String value;
    void put(String value) { this.value = value; }
    String get() { return value; }
}
```

Работает, но умеет только строки. Под число, заказ, пользователя — придётся плодить `IntBox`, `OrderBox`, `UserBox`, различающиеся одним словом. Логичная попытка спастись — одна коробка под `Object` (предок всего в Java):

```java
class Box {
    private Object value;
    void put(Object value) { this.value = value; }
    Object get() { return value; }
}
```

Одна коробка на всё — и сразу два несчастья:

```java
Box box = new Box();
box.put(42);                     // компилятор молчит — ему всё равно
String s = (String) box.get();   // компилируется... и падает ClassCastException в рантайме
```

Первое — `get()` возвращает `Object`, тип «забыт», нужен ручной каст `(String)`, где ты за компилятора клянёшься, что там строка. Второе — компилятор не страхует: положить число туда, откуда ждут строку, можно молча, а взрыв случится уже в работающей программе. Баг пойман в рантайме — самый дорогой вариант.

### 2. Generic-тип — дырка под тип на классе

Хочется сказать коробке при создании, что она хранит, и дальше пусть держит обещание сама:

```java
class Box<T> {
    private T value;
    void put(T value) { this.value = value; }
    T get() { return value; }
}
```

Вся новизна — `<T>` после имени класса и `T` внутри вместо конкретного типа. `T` — заглушка: «тут будет какой-то тип, зову его `T`, подставьте один и тот же по всему классу». Дырку заполняют при использовании:

```java
Box<String> box = new Box<>();
box.put("привет");
String s = box.get();   // никакого каста — компилятор знает, что внутри String
box.put(42);            // ОШИБКА КОМПИЛЯЦИИ — сразу, в редакторе, а не в рантайме
```

Оба несчастья исчезли: касты ушли, ошибка типа ловится на этапе компиляции. Ключевая аналогия, ради которой это ложится навсегда — дженерик это **обычный параметр, но этажом выше**:

Ниже — та же мысль картинкой. Обычный параметр `x` — дырка под значение, заполняется при вызове (`square(5)`). Параметр типа `T` — дырка под тип, заполняется при использовании (`new Box<String>()`).

### 3. Дырок может быть несколько

Одна буква — одна дырка. Нужна пара из двух разных типов (ключ-строка и число) — две буквы:

```java
class Pair<A, B> {
    private final A first;
    private final B second;
    Pair(A first, B second) { this.first = first; this.second = second; }
    A first()  { return first; }
    B second() { return second; }
}

Pair<String, Integer> p = new Pair<>("возраст", 30);
String key = p.first();    // A = String
Integer val = p.second();  // B = Integer
```

`A` и `B` — независимые дырки. `Map<K, V>` (твой `Dictionary<TKey, TValue>`) устроен ровно так: `K` под тип ключа, `V` под тип значения. Имена дырок — просто соглашение:

|буква|смысл|
|---|---|
|`T`|type — произвольный тип|
|`E`|element — элемент коллекции (`List<E>`)|
|`K` / `V`|key / value|
|`R`|result — возвращаемый тип|
|`S`, `U`|второй, третий тип, когда `T` занят|

> [!tip] 💡 Из C#  
> В C# принято `TKey`, `TValue` с префиксом `T`. В Java префикса нет — просто `K`, `V`, `T`. Чистая косметика, но глаз, привыкший к `TKey`, спотыкается на голом `K`.

### 4. Generic-метод — дырка на методе, а не на классе

Вот это по-настоящему новое, тут тормозим. До сих пор `T` жила **на классе**: коробка привязана к одному типу на всю жизнь. Но дырка может понадобиться не классу, а **одному методу** — например, утилите, которая берёт список чего угодно и возвращает первый элемент. Метод объявляет **свою** дырку:

```java
static <T> T first(List<T> list) {
    return list.get(0);
}
```

Читаем слева направо: `static` — обычный модификатор, к дженерикам не относится; `<T>` — **здесь объявляется дырка** («ввожу заглушку под тип, зову `T`»), обязательно **перед** типом возврата; второе `T` — тип, который метод **возвращает** (уже использование дырки); `first` — имя; `(List<T> list)` — принимает список элементов `T`. То есть `<T>` встречается дважды не по ошибке: сначала объявили, потом использовали.

Тип обычно не пишут руками — компилятор выводит его из аргумента:

```java
String s = first(List.of("a", "b"));   // T = String, выведен из аргумента; никакого first<String>(...)
```

Разница между двумя видами дырок:

|дырка на классе|дырка на методе|
|---|---|
|`class Box<T>`|`<T> T first(List<T> list)`|
|тип фиксируется при **создании объекта**|тип определяется заново при **каждом вызове**|
|коробка навсегда «коробка строк»|этот вызов — `String`, следующий — `Integer`|

Ты уже пользуешься generic-методами каждый день, просто не замечал: `Collections.emptyList()`, `Optional.of(x)`, `List.of(...)`, `Arrays.asList(...)` — все объявляют собственную `<T>` и выводят её из контекста. Типичный паттерн — статическая фабрика со своей дыркой:

```java
static <T> List<T> repeat(T value, int n) {
    List<T> out = new ArrayList<>();
    for (int i = 0; i < n; i++) out.add(value);
    return out;
}
List<String> aaa = repeat("a", 3);   // T = String
List<Integer> zeros = repeat(0, 5);  // тот же метод, теперь T = Integer
```

> [!tip] 💡 Из C#  
> Три расхождения, и первые два спрашивают.
> 
> - **Позиция параметра типа.** В C# — после имени: `T First<T>(List<T> list)`. В Java — **перед** типом возврата: `<T> T first(List<T> list)`. Рука на автомате напишет `T first<T>(...)` — это не компилируется. Вопрос «где объявляется параметр типа generic-метода?» дёшево отличает того, кто писал Java, от того, кто про неё читал.
> - **Статический метод в Java не видит `T` класса — а в C# видит.** В C# `Holder<int>` и `Holder<string>` — разные сконструированные типы, у каждого свой статический контекст, поэтому статика класса законно оперирует его `T`. В Java класс `Holder` в рантайме **один на все параметризации** (стирание), общей `T` у статики нет — нужна собственная метод-уровневая `<T>`.
> - **Type witness** (явный тип-аргумент) — механика урока B; в C# его пишут часто и естественно, в Java после 8 он почти не нужен.

Проверено на JDK 21, что попытка дать статике классовую `T` — ошибка компиляции:

```java
class Holder<T> {
    static T oops() { return null; }   // non-static type variable T cannot be referenced from a static context
}
```

### 5. Generic-конструктор — экзотика на две минуты

Конструктор тоже может объявить свою дырку, независимую от параметров класса. Синтаксис как у метода — `<T>` перед именем:

```java
class Wrapper {
    <T> Wrapper(T seed) { System.out.println(seed.getClass().getSimpleName()); }
}
new Wrapper(42);              // T = Integer, выведен  -> печатает Integer
new <String>Wrapper("hi");    // явный witness для конструктора — легально (JLS §8.8.4), но так не пишут
```

Проверено на JDK 21: печатает `Integer`, затем `String`. Знай, что это законно, и не залипай — за карьеру встретишь пару раз. На практике то же самое почти всегда чище выражается **статической фабрикой** со своей `<T>` (Bloch, _Effective Java_, Item 1).

> [!example] 🔬 Глубже / на синьора  
> **Почему статике недоступна `T` класса — это тизер стирания.** Классовая `T` концептуально принадлежит _инстансу_ параметризованного типа. В CLR `Foo<int>` и `Foo<string>` — два разных рантайм-типа (reified generics), у каждого свой набор статических членов, привязанных к своей `T`. В JVM всё наоборот: `Foo<int>`... точнее `Foo<Integer>` и `Foo<String>` в байткоде — **один** класс `Foo`, тип-аргументы стёрты. Статический член один на всех, и «чью `T`» он бы имел в виду — неопределимо, поэтому язык это просто запрещает. Полный разбор стирания и его последствий (`new T()`, `T[]`, `instanceof T` — всё невозможно) — урок 3.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Позиция `<T>` у метода** — C#-рефлекс `T first<T>(...)` не компилируется; объявление идёт перед типом возврата.
> 2. **Статический метод + `T` класса** → ошибка компиляции `non-static type variable T ... from a static context`. Лечится собственной `<T>` у метода.
> 3. **Примитив как тип-аргумент запрещён.** `List<int>` не компилируется (`required: reference, found: int`) — только `List<Integer>`. Дженерики работают лишь со ссылочными типами; примитивы протаскиваются через автобоксинг (и через кэш `Integer` для −128..127 — урок 1.4). В C# `List<int>` законен и вдобавок специализируется под значимый тип без боксинга — прямой контраст.
> 4. **Raw type** — `Box b = new Box();` без `<>` компилируется, но **выключает** проверки дженериков и даёт `unchecked`-warning на каждой операции (проверено: `unchecked call to put(T) as a member of the raw type Box`). Существует ради обратной совместимости с кодом до Java 5; писать так нельзя. Детали — урок 5.
> 5. **`new T()`, `new T[]`, `x instanceof T`** внутри дженерика невозможны — тоже следствие стирания (урок 3). Пока просто запомни, что эти три вещи «не разрешены», и не трать время, гадая почему, — причина там.

#### На собесе

> [!question]- Где объявляется параметр типа generic-метода?  
> **RU:** Перед типом возврата: `static <T> T identity(T x)`. Сначала дырку объявляют (`<T>`), потом используют (тип возврата, типы параметров).  
> **EN:** Before the return type: `static <T> T identity(T x)`. The type parameter is declared first (`<T>`), then used in the return and parameter types.
> 
> _Follow-up:_ «а у не-static метода?» → позиция та же, `static` тут ни при чём; generic-методы просто чаще всего статические утилиты. _Trap:_ ждут, что напишешь `T identity<T>(...)` по C#-привычке — это не компилируется.

> [!question]- Почему статический метод не может использовать параметр типа класса? В C# же может.  
> **RU:** Классовая `T` принадлежит инстансу параметризованного типа. В JVM класс один на все параметризации (стирание), общей статической `T` не существует — поэтому статике она недоступна, нужна собственная метод-уровневая `<T>`. В C# иначе: `Holder<int>` и `Holder<string>` — разные reified-типы со своими статическими контекстами, поэтому там это законно.  
> **EN:** A class-level `T` belongs to an instance of the parameterized type. On the JVM there's a single class for all parameterizations (erasure), so there's no shared static `T` — the static method needs its own method-level `<T>`. In C# it's different: `Holder<int>` and `Holder<string>` are distinct reified types with their own static contexts, so it's legal there.
> 
> _Follow-up:_ «почему один класс на все параметризации?» → стирание типов, дженерики — compile-time механизм. _Слабый ответ:_ «потому что static» без объяснения через стирание — не проходит первый же FU.

> [!question]- Можно ли использовать примитив как тип-аргумент — `List<int>`?  
> **RU:** Нет. Тип-аргумент обязан быть ссылочным типом — только `List<Integer>`. Примитивы попадают в дженерики через автобоксинг. В C# `List<int>` не только законен, но и специализируется под значимый тип без упаковки — здесь Java и C# расходятся жёстко.  
> **EN:** No. A type argument must be a reference type — only `List<Integer>`. Primitives enter generics via autoboxing. In C# `List<int>` is not only legal but specialized for the value type without boxing — a hard divergence.
> 
> _Follow-up:_ «а перформанс?» → бокс/анбокс и ловушка кэша `Integer` (урок 1.4); для «примитивных стримов» есть `IntStream`/`LongStream` (глава 10).

> [!question]- Класс-дженерик и метод-дженерик — в чём разница?  
> **RU:** У класса дырка фиксируется при создании объекта — коробка навсегда одного типа. У метода дырка определяется заново на каждом вызове — один и тот же метод возвращает то `String`, то `Integer`. Класс-дженерик описывает параметризованное состояние; generic-метод — параметризованную операцию.  
> **EN:** A generic class fixes its type at object creation — the box is one type forever. A generic method resolves its type per call — the same method yields `String` on one call, `Integer` on the next. A generic class parameterizes state; a generic method parameterizes an operation.
