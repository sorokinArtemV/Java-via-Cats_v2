За год в C# у тебя выработался рефлекс, который ты уже не замечаешь: пишешь `new List<string>()`, и объект **знает**, что он `List<string>` — `list.GetType()` вернёт сконструированный тип, а `typeof(List<int>)` и `typeof(List<string>)` в рантайме различимы без твоей помощи. Тип-аргумент — часть личности объекта, впечатанная в момент `new`.

Пишешь `new Box<String>()` в Java — и интуиция шепчет, что объект где-то у себя запомнил `String`, сделал снимок на `new`. Так вот: снимка нет. Объект про `String` не знает ничего, и из этого одного факта вырастает половина странностей Java-дженериков, которые ждут тебя в следующих уроках (`new T()`, `T[]`, bridge-методы). Сегодня сажаем сам факт — чисто, до байткода, — потому что без него все эти запреты выглядят произволом, а с ним складываются в одно следствие.

**Mental model.** Дженерики в Java — механизм **времени компиляции**. Компилятор проверяет типы и стирает тип-параметры: `T` без границы становится `Object`, с границей — её левым типом. В рантайме `Box<String>` и `Box<Integer>` — один класс, и объект не знает свой тип-аргумент.

#### 1. Один класс на всех

Спросим у двух «разных» списков, кто они:

```java
List<String>  strings  = new ArrayList<>();
List<Integer> integers = new ArrayList<>();
System.out.println(strings.getClass() == integers.getClass());  // true
System.out.println(strings.getClass().getName());               // java.util.ArrayList
```

Не «равны по `equals`» — это буквально **один и тот же объект `Class`**, сравнение по ссылке. В рантайме нет «класса `ArrayList<String>`» и «класса `ArrayList<Integer>`» — есть один `ArrayList`, и оба списка его экземпляры. У объекта негде спросить его `T`: метода `getTypeArgument()` нет, потому что нет самой информации.

> [!tip] 💡 Из C#  
> Ровно наоборот. `new List<string>().GetType() == new List<int>().GetType()` → `false`. CLR хранит каждую параметризацию отдельным сконструированным типом со своим type handle, а для value-типов (`List<int>`, `List<double>`) JIT ещё и генерирует **отдельный машинный код** под каждую — со специализацией под размер. Для ссылочных типов код общий, но идентичность типов всё равно различается. В JVM специализаций ноль: один класс, один код, на всех.
> 
> Отсюда же ответ на вопрос, который обычно откладывают: почему `static`-поле не видит `T` класса. В CLR `Holder<int>` и `Holder<string>` — разные типы, у каждого **своя** копия статики. В JVM класс `Holder` один, статика общая на все параметризации — и «`T` статики» просто не существует.

#### 2. Куда компилятор девает `T`

Если в рантайме `T` нет, значит, его убирает компилятор. Посмотрим, **чем** заменяет. Берём минимальный класс и компилируем — `javap -c -p Box`:

```java
class Box<T> {
    private T value;
    public void put(T value) { this.value = value; }
    public T get() { return value; }
}
```

```
private T value;
   putfield #7   // Field value:Ljava/lang/Object;
public T get();
   getfield #7   // Field value:Ljava/lang/Object;
   areturn
```

Поле `value` в класс-файле имеет дескриптор `Ljava/lang/Object;`. `put` принимает `Object`, `get` возвращает `Object`. `T` без следа заменён на `Object`.

Добавим границу — `T extends Number`:

```java
class NumBox<T extends Number> {
    private T value;
    public double half() { return value.doubleValue() / 2; }
}
```

```
getfield     #7    // Field value:Ljava/lang/Number;
invokevirtual #13  // Method java/lang/Number.doubleValue:()D
```

Теперь `T` стёрся до `Number`. Вот всё правило стирания (JLS §4.6): `T` без границы → `Object`; `T extends Number` → граница `Number`; при нескольких границах `T extends Number & Comparable<T>` → до **левой** (проверил `javap`: дескриптор `Ljava/lang/Number;`). И заметь побочный эффект в `half()`: вызов `doubleValue()` идёт прямо на `Number`, без каста. Граница — это не только ограничение для пользователя, это ещё и тип, с которым реально работает байткод.

#### 3. `checkcast`, которого ты не писал

Стоп. Если `get()` возвращает `Object`, почему это компилируется и работает?

```java
Box<String> box = new Box<>();
box.put("hello");
String s = box.get();          // get() же возвращает Object?!
System.out.println(s.length()); // 5
```

Смотрим байткод вызывающей стороны — `javap -c`:

```
invokevirtual #42  // Method Box.get:()Ljava/lang/Object;
checkcast     #46  // class java/lang/String
```

`checkcast` на `String` — каст, который ты **не писал**. Его дописал компилятор: на этапе компиляции он доказал, что в `Box<String>` попадают только `String`, поэтому после стёртого `get()`, вернувшего `Object`, безопасно вставить рантайм-проверку приведения к `String`.

Складывается якорь урока. Дженерики в Java — **договор времени компиляции**, и компилятор стоит на двух дверях: **на входе** проверяет (в `Box<String>` нельзя положить `Integer` — ошибка компиляции), **на выходе** дописывает `checkcast`. А между дверями живёт голый `Box` с `Object` внутри, и JVM про твой `<String>` не знает ничего. Снимок на `new` не делается, потому что хранить его незачем — все проверки уже случились до запуска.

> [!tip] 💡 Из C#  
> Теперь понятно, почему в C# легальны `new T()` (с `where T : new()`), `typeof(T)`, `default(T)`, `obj is T`, `T[]` — там `T` в рантайме реально существует, JIT знает, что подставлено. В Java всё из этого списка невозможно, но **почему именно** каждая операция ломается и какие есть обходы — это отдельный урок (7.5). Корень у всех один, и он уже у тебя: reified vs erased.

#### 4. Граница «стёрто vs осталось»

Стёрты тип-аргументы у **экземпляров и в исполнении**: объект не знает свой `T`, байткод работает с `Object`/границей. Но generic-сигнатуры **объявлений** — класса, поля, метода — компилятор сохраняет в класс-файле как метаданные, в атрибуте `Signature` (JVMS §4.7.9). Смотри `javap -v -p Box`:

```
private T value;
   descriptor: Ljava/lang/Object;   // стёртая — по ней работает JVM
   Signature:  TT;                  // полная — для компилятора и рефлексии

Signature: <T:Ljava/lang/Object;>Ljava/lang/Object;   // сигнатура класса
```

У каждого члена две жизни: `descriptor` — стёртая, `Signature` — полная (чтобы компилятор мог линковаться с твоим классом из другого кода, и чтобы это читала рефлексия). Проверим рефлексией:

```java
class Holder { private List<String> items; public List<String> first() { return items; } }

Field f = Holder.class.getDeclaredField("items");
f.getType();               // interface java.util.List          ← стёрто
f.getGenericType();        // java.util.List<java.lang.String>  ← из Signature

Holder.class.getDeclaredMethod("first").getGenericReturnType();
                           // java.util.List<java.lang.String>
```

`getType()` отдаёт стёртый `List`, а `getGenericType()` — полный `List<String>`. Рядом живут `getGenericSuperclass()`, `getGenericReturnType()`, `getTypeParameters()` — вся семья читает один и тот же атрибут. Сформулируем границу точно:

- спросить у **объекта** «какой у тебя `T`?» — нельзя, информации нет;
- спросить у **объявления** «с каким `T` ты написано?» — можно, через рефлексию.

`new ArrayList<String>()` и `new ArrayList<Integer>()` неразличимы. Но поле `List<String> items` и поле `List<Integer> items` — различимы прекрасно: текст объявления никуда не делся.

#### 5. Награда: трюк `TypeToken`

Зачем знать, где проходит эта граница? Затем, что на ней стоит приём, которым ты будешь пользоваться с первой Java-работы. Ты из .NET знаешь JSON-сериализацию; в Java это Jackson, и там проблема: как попросить десериализовать в `List<String>`, если `List<String>.class` даже не компилируется (class literal бывает только у стёртого типа — `List.class`)?

Ответ — анонимный **подкласс**:

```java
abstract class TypeReference<T> {
    Type captured() {
        ParameterizedType st = (ParameterizedType) getClass().getGenericSuperclass();
        return st.getActualTypeArguments()[0];
    }
}

new TypeReference<List<String>>() {}.captured();
// java.util.List<java.lang.String>
new TypeReference<Map<String, List<Integer>>>() {}.captured();
// java.util.Map<java.lang.String, java.util.List<java.lang.Integer>>
```

Работает вот почему. `new TypeReference<List<String>>() {}` — это не объект `TypeReference`, это **новый анонимный класс**, в чьём объявлении написано `extends TypeReference<List<String>>`. А объявления сохраняют дженерики в `Signature`. `getGenericSuperclass()` читает оттуда полный `TypeReference<List<String>>` — и `List<String>` пойман. Мы протащили тип-аргумент через рантайм, превратив его из «параметризации экземпляра» (стирается) в «текст объявления» (сохраняется). Ровно так устроены `TypeReference` в Jackson и `TypeToken` в Gson — `mapper.readValue(json, new TypeReference<List<User>>() {})` теперь для тебя не магия.

> [!example] 🔬 Глубже / на синьора  
> **Две жизни члена в класс-файле.** `descriptor` (JVMS §4.3) — то, по чему JVM линкует и вызывает; `Signature` (JVMS §4.7.9) — необязательный атрибут с полной generic-информацией. JVM при исполнении `Signature` игнорирует полностью — он существует только для компилятора (раздельная компиляция: твой `Box` из другого модуля должен знать, что `get()` возвращает `T`) и для Reflection API. Убери атрибут — код исполнится ровно так же, сломается только `getGenericType()`.
> 
> **Почему левая граница, а не любая.** При `T extends Number & Comparable<T>` в дескрипторе поля стоит `Number` — первый тип списка. Правило простое: стирание берёт **самый левый** тип, и по нему строится дескриптор, поэтому порядок границ влияет на байткод (первым ставь тот класс/интерфейс, к которому чаще обращаешься без каста). Вызовы методов второй границы (`compareTo`) пойдут через `checkcast`.
> 
> **Capture у `List<?>`.** Операции над `List<?>` внутри реализуются через приватный generic-хелпер: компилятор даёт анонимному захваченному типу имя (`CAP#1`), чтобы `get`/`set` были согласованы. Полный разбор — в уроке про wildcards, но корень тот же: имя типу нужно только компилятору.
> 
> **Project Valhalla** среди прочего смотрит в сторону специализации дженериков (в т.ч. под примитивы) — но это траектория, а не Java 21. В baseline реальность одна: стирание.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Фигурные скобки в `TypeReference` обязательны.** `new TypeReference<List<String>>()` без `{}` — просто экземпляр (был бы, не будь класс abstract), у которого generic-суперкласса с нужной информацией нет. Именно `{}` создаёт анонимный подкласс с объявлением. Без него `getGenericSuperclass()` вернёт голый `TypeReference`, а `getActualTypeArguments()` упадёт/даст мусор.
> 2. **Перегрузка по тип-аргументу невозможна.** `void f(List<String>)` и `void f(List<Integer>)` в одном классе — ошибка `name clash: ... have the same erasure`: после стирания обе сигнатуры `f(List)`. Ошибка **сохраняется даже при разных return-типах** — на уровне языка перегрузка различается только параметрами, а они после стирания совпадают. (В C# такая перегрузка легальна: в IL это разные сконструированные типы.)
> 3. **`instanceof` с параметризованным типом не компилируется.** `o instanceof List<String>` → ошибка `Object cannot be safely cast to List<String>`: проверить `<String>` в рантайме нечем. Легально только `o instanceof List<?>` (или raw `o instanceof List`) — проверяется лишь стёртый `List`.
> 4. **Heap pollution: `checkcast` может взорваться вдали от настоящей ошибки.** Через raw type в `List<Cat>` можно протащить `Dog`, и `checkcast` щёлкнет не там, где ошибка, а на чтении:
> 
> 
> 
> ```java
> List<Cat> cats = new ArrayList<>();
> List raw = cats;        // мост через raw type — здесь настоящая ошибка
> raw.add(new Dog());     // Dog в List<Cat> — компилятор лишь предупредил (unchecked)
> Cat c = cats.get(0);    // ClassCastException: class Dog cannot be cast to class Cat
> ```
> 
> Дженерики гарантируют типобезопасность только пока **весь** код generic-чистый. Полный разбор raw types — урок 7.7.
> 
> 5. **Каст к generic-типу — unchecked, без рантайм-проверки.** `(List<String>) obj` компилируется с предупреждением, а в байткоде `checkcast` стоит только на raw `List` — «`String`-ность» элементов не проверяется вообще. Отложенный `ClassCastException` — оттуда же.
> 6. **`List<String>.class` не компилируется**, а `static`-член не видит `T` класса — оба следствия того, что отдельного рантайм-типа для параметризации не существует.

#### 6. Почему стирание, а не reified как в C#

Без «зачем» урок не Middle+. Дженерики появились в Java 5 (2004, JSR 14). К тому моменту — почти десятилетие Java-кода: миллионы `.class`, библиотеки с не-generic коллекциями, продакшены на старых JVM. Команда поставила жёсткое требование — **migration compatibility**: новый generic-код обязан работать со старыми библиотеками и класс-файлами, в идеале без новой JVM. Стирание решает это элегантно: generic-`ArrayList<String>` в байткоде — это **тот же самый** `ArrayList`, что и раньше. Старый код видит его как обычный класс, raw types служат мостом между мирами, рантайм менять не пришлось вовсе.

C# стоял перед той же проблемой (`ArrayList`, `Hashtable` из .NET 1.x) и решил её противоположно: C# 2.0 (2005) **расширил сам CLR** — новый IL, рантайм-поддержка сконструированных типов, JIT-специализация. Цена — старые и новые коллекции живут параллельно (`System.Collections` vs `System.Collections.Generic`), миграция была разрывом. Java заплатила выразительностью языка, C# — чистотой разрыва. Одна проблема, два честных противоположных ответа — и из твоего ответа на «почему в Java стирание?» собеседующий мгновенно понимает, знаешь ли ты инженерный контекст.

#### На собесе

> [!question]- Что напечатает `new ArrayList<String>().getClass() == new ArrayList<Integer>().getClass()`?  
> `true` — в рантайме это один класс `ArrayList`, объект `Class` один, сравнение по ссылке истинно.
> 
> **RU:** Оба списка — экземпляры одного стёртого класса `ArrayList`; параметризации в рантайме не существует, поэтому `Class`-объект один и `==` даёт `true`.  
> **EN:** Both lists are instances of the same erased `ArrayList` class; the parameterization doesn't exist at runtime, so there's a single `Class` object and `==` returns `true`.
> 
> _Follow-up:_ «а в C#?» — `false`, там это вообще разные типы: CLR овеществляет каждую параметризацию. _Как усилить:_ добавь про JIT-специализацию value-типов в CLR — это сразу уровень.

> [!question]- Почему `void f(List<String>)` и `void f(List<Integer>)` не компилируются в одном классе?  
> Same erasure: после стирания обе — `f(List)`, в класс-файле это дубликат метода.
> 
> **RU:** После стирания обе сигнатуры сводятся к `f(List)` — компилятор видит два одинаковых метода и выдаёт `name clash: have the same erasure`.  
> **EN:** After erasure both signatures collapse to `f(List)` — the compiler sees two identical methods and reports `name clash: have the same erasure`.
> 
> _Follow-up:_ «а если у них разные возвращаемые типы?» — всё равно ошибка: перегрузка на уровне языка различается только параметрами, а они совпали. _Трап:_ не давай увести себя в «JVM различает по return type в дескрипторе» — это правда про байткод, но `javac` такой исходник не пропустит.
> 
> _Слабый ответ:_ «нельзя, потому что оба `List`» — без слова «erasure» и без понимания, что дело в стёртой сигнатуре.

> [!question]- «В рантайме от дженериков не остаётся ничего». Согласен?  
> Нет.
> 
> **RU:** Стёрты тип-аргументы экземпляров и исполнение, но generic-сигнатуры объявлений (класс, поле, метод) сохранены в атрибуте `Signature` класс-файла и доступны рефлексией — `getGenericType()`, `getGenericSuperclass()`, `getTypeParameters()`. На этом стоит `TypeReference`/`TypeToken`: анонимный подкласс делает тип-аргумент частью объявления.  
> **EN:** Instance type arguments and execution are erased, but the generic signatures of declarations (class, field, method) are kept in the class file's `Signature` attribute and readable via reflection — `getGenericType()`, `getGenericSuperclass()`, `getTypeParameters()`. That's what `TypeReference`/`TypeToken` rely on: an anonymous subclass turns the type argument into part of a declaration.
> 
> _Follow-up:_ «и как этим пользуются?» — приведи сам пример Jackson/Gson, и вопрос закрыт. _Слабый ответ:_ «да, всё стирается» — фатально, ждёт разведения «экземпляр vs объявление».

> [!question]- `String s = list.get(0)` — где здесь типобезопасность в байткоде?  
> `get` возвращает стёртый `Object`, а компилятор дописал `checkcast` на `String` в месте вызова.
> 
> **RU:** Проверка типов случилась в компайл-тайме; `checkcast` — страховочный шов поверх стёртого `Object`, видно в `javap -c`.  
> **EN:** Type checking happened at compile time; the `checkcast` is a safety seam over the erased `Object`, visible in `javap -c`.
> 
> _Follow-up:_ «может ли этот `checkcast` упасть?» — да, если в список протекли чужие элементы через raw types или unchecked-касты; это heap pollution, и `ClassCastException` прилетит далеко от настоящей ошибки. Назвать термин — уровень.

> [!question]- Почему Java выбрала стирание, а не reified-дженерики как C#?  
> Migration compatibility.
> 
> **RU:** 2004, JSR 14: требование работать со старым байткодом и не-generic библиотеками без изменения JVM. Стирание оставило `ArrayList<String>` тем же `ArrayList` в байткоде. C# 2.0 пошёл иначе — изменил CLR и IL, получив reified-дженерики и JIT-специализацию, но ценой параллельных старых и новых коллекций.  
> **EN:** 2004, JSR 14: the requirement was to run against old bytecode and non-generic libraries without changing the JVM. Erasure kept `ArrayList<String>` as the same `ArrayList` in bytecode. C# 2.0 took the opposite route — it changed the CLR and IL, getting reified generics and JIT specialization, at the price of parallel old and new collections.
> 
> _Follow-up:_ «цена решения Java?» — `new T()`, `instanceof T`, `T[]` невозможны, один класс на все параметризации, статика не видит `T`.

> [!question]- Скомпилируется ли `Class<List<String>> c = List<String>.class`?  
> Нет, синтаксическая ошибка: class literal существует только для стёртого типа — `List.class`. Отдельного `Class`-объекта для `List<String>` физически нет.
> 
> _Follow-up:_ «как тогда передать `List<String>` как тип?» — трюк `TypeReference`/`TypeToken` или `ParameterizedType` руками.

> [!question]- Итоговый ответ по теме _(RU/EN)_  
> **RU:** Дженерики в Java — механизм времени компиляции. Компилятор проверяет типы, затем стирает тип-параметры: `T` без границы → `Object`, с границей → её левый тип; на выходе дописывает `checkcast` в местах чтения. В рантайме `Box<String>` и `Box<Integer>` — один класс, объект не знает свой тип-аргумент. Но стирание не тотально: generic-сигнатуры объявлений сохранены в атрибуте `Signature` и читаются рефлексией (`getGenericType`), на чём стоит трюк `TypeReference`/`TypeToken`. Причина стирания — обратная совместимость (Java 5, 2004): работать со старым байткодом без изменения JVM, в отличие от reified-подхода C#. Практические следствия: нельзя перегрузить по тип-аргументу, нет `List<String>.class`, статика не видит `T`, а `new T()`/`instanceof T`/`T[]` невозможны.
> 
> **EN:** Java generics are a compile-time mechanism. The compiler type-checks, then erases type parameters — unbounded `T` → `Object`, bounded → its leftmost type — and inserts a `checkcast` at each read site. At runtime `Box<String>` and `Box<Integer>` are one class and an object doesn't know its type argument. But erasure isn't total: the generic signatures of declarations are kept in the `Signature` attribute and read via reflection (`getGenericType`), which is what the `TypeReference`/`TypeToken` trick exploits. The reason is backward compatibility (Java 5, 2004): run against old bytecode without changing the JVM, unlike C#'s reified approach. Practical consequences: no overloading by type argument, no `List<String>.class`, statics can't see `T`, and `new T()`/`instanceof T`/`T[]` are impossible.
> 
> _Слабые ответы:_ «в рантайме дженериков нет вообще» — забыт `Signature`/рефлексия; «стирание — это чтобы быстрее» — неверный мотив, дело в совместимости; «`checkcast` защищает всегда» — нет, heap pollution ломает гарантию.
