Полное объявление параметризованного типа многословно до абсурда: `Map<String, List<Integer>> m = new HashMap<String, List<Integer>>()` — одно и то же написано дважды. Начиная с Java 7 компилятор научился достраивать тип-аргументы из контекста, и в 99% кода ты их вообще не пишешь. Здесь важно понять **откуда** он их берёт — потому что ровно на этом стоит одна тихая ловушка, на которой пришедшие из C# спотыкаются на код-ревью.

**Mental model.** Компилятор восстанавливает тип-аргументы из контекста: `diamond` тянет их из **целевого типа слева**, вывод для методов — из **аргументов и целевого типа**. Опасность одна: место, где тянуть неоткуда, и он молча берёт `Object`.

### 1. Diamond `<>` — не заставляй писать тип дважды

С Java 7 правую часть сокращают до пустых угловых скобок — это и есть **diamond** (ромб). Компилятор берёт тип-аргументы из левой части — это называется **target typing**, вывод из целевого типа:

```java
Map<String, List<Integer>> scores = new HashMap<>();   // <> = «возьми типы из левой части»
```

С Java 9 (JEP 213) `diamond` разрешён и с анонимными классами — до этого там была ошибка компиляции (проверено: под `--release 8` тот же код даёт `cannot infer type arguments`):

```java
Comparator<String> byLen = new Comparator<>() {        // на Java 8 — ошибка; с Java 9 — ок
    public int compare(String a, String b) { return a.length() - b.length(); }
};
```

> [!tip] 💡 Из C#  
> Ближайший родственник — `target-typed new()` из C# 9: `Dictionary<string,int> d = new();`. Идея та же — достроить тип из левой части. Только `diamond` появился на девять лет раньше и оставляет скобки `<>` на месте (в C# их нет вовсе).

### 2. Вывод типов для generic-методов

Для методов компилятор собирает тип из двух источников: **аргументов вызова** и **целевого типа** (куда присваивается результат). Поэтому `first(List.of("a"))` даёт `T = String` без единого явного указания. А `Collections.emptyList()` выводит `T` из левой части:

```java
List<String> empty = Collections.emptyList();   // T = String выведен из целевого типа слева
```

### 3. Type witness — когда вывод не справился

Изредка выводить неоткуда: ни информативных аргументов, ни понятного целевого типа. Тогда тип-аргумент указывают руками — это **type witness** («свидетель типа»), тип в угловых скобках **перед именем метода, после точки/квалификатора**:

```java
List<String> empty = Collections.<String>emptyList();
```

Синтаксис требует квалификатора — голый witness не парсится (проверено на JDK 21):

```java
this.<String>method();        // ок
ClassName.<String>method();   // ок
<String>method();             // ошибка: illegal start of expression
```

> [!warning] ⚠️ Подводные камни
> 
> 1. **`var` + `diamond` = `ArrayList<Object>`, молча.** Главная ловушка урока — разбор ниже, под диаграммой.
> 2. **`diamond` с анонимным классом — только с Java 9.** На Java 8 — ошибка компиляции. Если проект ещё на 8, `new Comparator<>() {...}` не соберётся.
> 3. **Голый witness не парсится.** `<String>method()` без `this.`/`Класс.` — `illegal start of expression`. Нужен квалификатор.
> 4. **Witness — подсказка, а не каст.** Если указанный тип противоречит аргументам — ошибка **компиляции**, а не `ClassCastException` в рантайме. Это принципиально: witness проверяется, каст — нет.
> 5. **Raw `new HashMap()` vs diamond `new HashMap<>()`** выглядят почти одинаково, а семантика противоположна: `diamond` выводит типы и **сохраняет** проверки; raw их **выключает** и сыплет `unchecked`-warning. Один символ `<>` — разница между типобезопасным и «сырым» кодом.

### 4. Ловушка: `var` + `diamond` несовместимы

Механизмы тянут тип с разных сторон: **diamond — из левой** (целевой тип), **`var` — из правой**. Скрести их — и выводить неоткуда:

```java
var list = new ArrayList<>();   // компилируется! но это ArrayList<Object>, а не то, что ты думал
list.add("hi");
list.add(42);                   // тоже ок — Object всё стерпит
String s = list.get(0);         // а вот здесь ОШИБКА: incompatible types: Object cannot be converted to String
```

Проверено на JDK 21: строка создания компилируется **без ошибки и без предупреждения** — молча `ArrayList<Object>`, — а проблема всплывает тремя строками ниже, на `get(0)`. В C# `var x = new List<string>()` — нормальная идиома, потому что тип написан **справа**, и `var` его оттуда и берёт. В Java так нельзя — с `var` тип-аргумент пиши явно:

```java
var list = new ArrayList<String>();   // правильно: тип-аргумент указан справа, var берёт его оттуда
```

Это задокументированная ловушка — в официальных _Local Variable Type Inference Style Guidelines_ к JEP 286.

> [!example] 🔬 Глубже / на синьора  
> **Как компилятор на самом деле выводит тип** (JLS §18, без формализма) — три шага.
> 
> **Шаг 1 — собрать ограничения.** Два источника: _аргументы вызова_ (передал `List.of("a")` → «`T` = `String`») и _целевой тип_ (куда присваивается результат — это и есть target typing, сильно усиленный в Java 8, поэтому witness почти вымер).
> 
> **Шаг 2 — решить систему.** Один тип из ограничений — он и выводится. Если аргументы дают **разные** типы, берётся их **ближайший общий супертип** (lub, least upper bound):
> 
> java
> 
> ```java
> static <T> List<T> pair(T a, T b) { ... }
> var auto = pair(1, 2.5);            // lub(Integer, Double) — страшный пересечённый тип
> List<Number> nums = pair(1, 2.5);  // а тут target typing: левая часть диктует T = Number
> ```
> 
> Проверено на JDK 21: `pair(1, 2.5)` выводит `T` в пересечённый тип `Number & Comparable<...> & Constable & ConstantDesc` (члены `Constable`/`ConstantDesc` добавлены к боксированным примитивам с JDK 12 — вот почему lub «толще», чем можно ждать по учебникам под Java 8). Вторая строка — target typing вживую: целевой тип слева навязывает `T = Number`, и `pair(1, 2.5)` там спокойно возвращает `[1, 2.5]`.
> 
> **Шаг 3 — не решилось** → ошибка `cannot infer type arguments`, и тогда твой ход: type witness.
> 
> Вся эта машинерия — **целиком compile-time**. В байткоде от выведенных типов остаётся куда меньше, чем ждёшь после C#, — это и есть стирание (урок 3).

#### На собесе

> [!question]- Что выведет `var list = new ArrayList<>()`?  
> **RU:** `ArrayList<Object>`. `diamond` ждёт тип из целевого типа слева, `var` ждёт тип справа — информации нет ни там, ни там, компилятор откатывается на `Object`. Компилируется без предупреждения, а ошибка всплывёт позже, на попытке достать элемент.  
> **EN:** `ArrayList<Object>`. Diamond expects a type from the target on the left, `var` expects it on the right — there's nothing on either side, so the compiler falls back to `Object`. It compiles without a warning; the error surfaces later, when you read an element out.
> 
> _Follow-up:_ «что делать?» → с `var` указывай тип-аргумент явно: `var list = new ArrayList<String>()`. Бонус — упомянуть, что это документированная ловушка в гайдлайнах к JEP 286.

> [!question]- Что такое diamond и что менялось по версиям?  
> **RU:** `<>` при создании объекта — компилятор выводит тип-аргументы из целевого типа. Появился в Java 7; с Java 9 (JEP 213) разрешён и с анонимными классами.  
> **EN:** `<>` at object creation — the compiler infers type arguments from the target type. Introduced in Java 7; since Java 9 (JEP 213) it's allowed with anonymous classes too.
> 
> _Follow-up:_ «`new HashMap<>()` vs raw `new HashMap()`?» → `diamond` выводит типы и сохраняет проверки; raw их выключает и даёт unchecked-warning. Выглядят похоже — семантически противоположны.

> [!question]- Когда нужен type witness?  
> **RU:** Когда вывод не может определить тип — нет ни информативных аргументов, ни целевого типа. Синтаксис: `Collections.<String>emptyList()`, обязательно с квалификатором. Это подсказка компилятору, а не каст: неверный тип даёт ошибку компиляции, не рантайм-исключение.  
> **EN:** When inference can't determine the type — no informative arguments and no target type. Syntax: `Collections.<String>emptyList()`, always with a qualifier. It's a compiler hint, not a cast: a wrong type is a compile error, not a runtime exception.
> 
> _Follow-up:_ «почему в старом коде witness частый, а в новом почти нет?» → Java 8 усилила target typing, и большинство случаев стали выводиться сами.

> [!question]- (senior) Как компилятор выводит `T`?  
> **RU:** Собирает ограничения из аргументов и целевого типа, решает систему; при расходящихся аргументах берёт ближайший общий супертип (lub); если не решается — ошибка `cannot infer type arguments`. Всё это compile-time, в рантайме типов уже нет (стирание).  
> **EN:** It gathers constraints from arguments and the target type, solves the system; for divergent arguments it takes the least upper bound (lub); if unsolvable, `cannot infer type arguments`. All compile-time — at runtime the types are gone (erasure).
