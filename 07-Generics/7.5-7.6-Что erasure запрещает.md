
Ты садишься написать свой `ArrayList`. В C# это пять минут: внутри `T[] _items = new T[capacity]`, рост через `Array.Resize`, и всё. Открываешь IntelliJ, набираешь то же самое на Java — и компилятор упирается на первой же строке: `new T[10]` не компилируется. Лезешь в исходники настоящего `java.util.ArrayList` подсмотреть, как выкрутились авторы, и видишь там `Object[] elementData`. Стандартная библиотека хранит элементы твоего `ArrayList<String>` в массиве `Object`. Не от лени — иначе нельзя.

Это не единичный каприз, а первый пункт целого списка операций, которые в C# ты делал не задумываясь, а Java запрещает на уровне компилятора: `new T()`, `new T[]`, `x instanceof List<String>`, `T.class`, generic-исключения. И весь список — следствие одного факта из прошлого урока про type erasure: **к рантайму `T` не существует**. Компилятор стёр его до `Object` (или до верхней границы), и JVM видит просто класс с полем `Object[]`. Пройдём каждый запрет и выведем его из этого факта — а во второй половине посмотрим на зеркальную ситуацию: место, где стирание ломает **полиморфизм**, и компилятор молча дописывает за тебя методы, которых нет в исходнике (bridge methods).

**Mental model.** Дженерики Java живут только на этапе компиляции: `T` — это указание компилятору проверить типы, а не сущность в рантайме. Всё, что запрещено в generic-коде, запрещено потому, что операция требует настоящий тип в рантайме, а его там нет. Всё, что компилятор дописывает сам, он дописывает, чтобы залатать дыру, которую то же стирание прогрызло в диспетчеризации.

### Часть A (7.5): шесть запретов, выведенных из стирания

#### `new T()` — нельзя, и почему именно

```java
class Pool<T> {
    T createInstance() { return new T(); }   // error: unexpected type; found: type parameter T
}
```

Разверни, что `new` делает в рантайме: JVM нужен конкретный класс — чтобы выделить память правильного размера и вызвать конструктор (байткод `new` + `invokespecial`). А что такое `T` после стирания? Ничего: тело метода превращается в `return new ???()` — класса нет, размера нет, конструктора нет. Проблема даже глубже «информация потерялась»: ещё на этапе компиляции `T` может быть чем угодно, а у, скажем, `LocalDate` вообще нет публичного конструктора без аргументов. Контракт `new T()` нечем гарантировать.

> [!tip] 💡 Из C#  
> Ровно поэтому в C# `new T()` тоже не работает «из коробки» — нужен констрейнт `where T : new()`. Компилятор C# требует доказательство, что безаргументный конструктор есть, и тогда — поскольку дженерики reified и в рантайме `T` это реальный тип — CLR честно создаёт объект. В Java такого констрейнта нет в принципе: даже докажи ты наличие конструктора, в рантайме создавать всё равно нечего.

Обхода два, оба про «протащить знание о типе в рантайм руками»:

```java
import java.util.function.Supplier;

class Pool<T> {
    // Вариант 1: type token — статический метод объявляет СВОЙ T
    static <T> T create(Class<T> type) throws ReflectiveOperationException {
        return type.getDeclaredConstructor().newInstance();
    }
    // Вариант 2: фабрика
    static <T> T create(Supplier<T> factory) { return factory.get(); }
}

StringBuilder a = Pool.create(StringBuilder.class);   // -> StringBuilder
StringBuilder b = Pool.create(StringBuilder::new);    // -> StringBuilder
```

`Class<T>` — тот самый type token из урока 7.4: объект `StringBuilder.class` существует в рантайме и несёт всё, что стёрлось. Вариант с `Supplier<T>` чище — без рефлексии и checked-исключений, и в современном коде предпочтителен.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`type.newInstance()` без `getDeclaredConstructor()` — deprecated с Java 9.** Проверено: `-Xlint:deprecation` даёт «`newInstance()` in Class has been deprecated». Причина не косметическая: старый метод пробрасывает checked-исключения конструктора **мимо** объявления `throws`, обходя систему типов. Правильно — `getDeclaredConstructor().newInstance()`: он честно заворачивает их в `InvocationTargetException`.
> 2. **`T` у статического метода — свой.** `static <T> T create(...)` объявляет `T`, не связанный с `T` класса `Pool`. Легко перепутать, когда оба называются `T`; на собесе спрашивают «какой `T` тут в области видимости».

#### `new T[]` — мост между двумя мирами рушится

```java
T[] data = new T[10];                          // error: generic array creation
List<String>[] lists = new List<String>[10];   // error: generic array creation
```

Здесь сталкиваются два факта, которые ты уже знаешь по отдельности. Из урока про массивы (1.7): массивы **reified** — `String[]` в рантайме знает, что он `String[]`, и **ковариантны** — `Object[] arr = new String[5]` легально, а запись не того типа ловится в рантайме через `ArrayStoreException`. Из урока 7.4: дженерики **стёрты** — `List<String>` в рантайме про `String` не знает. Разреши компилятор `new List<String>[10]` — и через ковариантность типобезопасность тихо протечёт. Вот эта протечка вживую (проверено на JDK 21):

![[Pasted image 20260709145832.png]]

Строка `objs[0] = List.of(42)` — момент истины. Массив в рантайме проверяет: «кладут ли в меня `List`?» Но `<String>` стёрт — рантайм видит просто `List`, проверка проходит, мусор в массиве (это и есть **heap pollution**: переменная типа `List<String>[]` ссылается на данные, ей не соответствующие). А `ClassCastException` рванёт потом, в невинном чтении, где никто ничего не кастовал. Язык, где типобезопасность тихо протекает, компилятор допустить не может — поэтому generic array creation запрещён целиком (JLS §15.10.1: тип создаваемого массива обязан быть reifiable). И отсюда понятно, почему `ArrayList` хранит `Object[]`: это единственный честный массив, который можно создать, не зная `T`.

> [!warning] ⚠️ Три формы, не путай (проверено на JDK 21)
> 
> - `new List<String>[10]` — **нельзя**: параметризованный тип не reifiable.
> - `new List<?>[10]` — **можно**: unbounded wildcard ничего не утверждает про элементы, проверять нечего, тип reifiable.
> - `new List[10]` — **можно**, это raw-массив, но с ним свои проблемы (урок 7.7).

##### Идиома `(T[]) new Object[n]` — и где она кусается

Реальный код всё-таки хочет `T[]`. Стандартный трюк — создать `Object[]` и скастовать:

```java
@SuppressWarnings("unchecked")
T[] data = (T[]) new Object[10];
```

Компилируется (с unchecked warning, который мы давим). Работает? **Внутри класса** — да: читаешь и пишешь через `T`, всё консистентно. Но запомни, что реально лежит в переменной: `Object[]`, не `T[]`. Каст `(T[])` после стирания — пустышка, он ничего не проверил и ничего не превратил. И как только массив утекает наружу — классика, я прогнал её на JDK 21:

```java
class MyList<T> {
    private Object[] data = new Object[10];
    private int size = 0;
    void add(T item) { data[size++] = item; }

    @SuppressWarnings("unchecked")
    T[] toArray() {
        T[] result = (T[]) new Object[size];   // тишина
        System.arraycopy(data, 0, result, 0, size);
        return result;                          // тишина
    }
}

MyList<String> list = new MyList<>();
list.add("hello");
String[] arr = list.toArray();   // <-- ClassCastException ЗДЕСЬ (строка присваивания)
```

```
Exception in thread "main" java.lang.ClassCastException:
  class [Ljava.lang.Object; cannot be cast to class [Ljava.lang.String;
        at ToArrayTrap.main(ToArrayTrap.java:16)
```

Смотри на стектрейс: взрыв не в `toArray`, где жил враньё-каст, а **в коде вызывающего**, на строке `String[] arr = ...`. Внутри `toArray` после стирания кастов нет вообще, а вот на границе компилятор вставил `checkcast [Ljava.lang.String;` — массив-то reified, и рантайм честно говорит: «это `Object[]`, а не `String[]`». Тот же паттерн «наврали в одном месте, рвануло в другом», что и в уроке 7.4, только теперь через массив.

Безопасная версия — создать **настоящий** `T[]`, для чего опять нужен type token (проверено — печатает `String[] -> hello,world`):

```java
import java.lang.reflect.Array;

@SuppressWarnings("unchecked")
T[] toArray(Class<T> type) {
    T[] result = (T[]) Array.newInstance(type, size);  // реально String[]
    System.arraycopy(data, 0, result, 0, size);
    return result;
}
```

`Array.newInstance(String.class, n)` создаёт в рантайме именно `String[]` — каст здесь формальность (сигнатура возвращает `Object`), а не враньё. Ровно поэтому у настоящего `Collection.toArray(T[] a)` параметр — массив: коллекция берёт класс элемента из переданного массива.

> [!tip] 💡 Из C#  
> `new T[n]` в C# — одна строка без оговорок: CLR знает `T`, создаёт настоящий `T[]`. Вся эта секция — про проблему, которой у тебя в .NET просто не было.

#### `instanceof` с параметризацией — проверять нечего

```java
if (x instanceof List<String>) { }   // error: Object cannot be safely cast to List<String>
```

`instanceof` — рантайм-операция: байткод `instanceof` спрашивает у объекта его класс и сверяет. У объекта в куче есть заголовок с указателем на класс — и там написано `ArrayList`. Не `ArrayList<String>`. Аргумент `<String>` испарился при компиляции _того_ кода, который список создал. Вопрос «это `List<String>`?» физически не имеет ответа. Легальны только формы, где вопрос сводится к сырому классу:

```java
x instanceof List<?>    // ок: «это какой-нибудь List?» — проверяется сырой тип
x instanceof List       // ок: raw-форма, то же самое, но стилистически хуже
```

Предпочитай `List<?>`: она говорит читателю «я в курсе про дженерики, мне правда неважен параметр», а raw выглядит как код 2004 года. Pattern matching for `instanceof` (Java 16, JEP 394) тут ничего не меняет — проверено: `x instanceof List<String> ls` так же запрещён, легально только `x instanceof List<?> ls`. Синтаксис новый, физика стирания та же.

> [!tip] 💡 Из C#  
> `x is List<string>` — рабочий код, рантайм-проверка честная, потому что объект в CLR реально помнит, что он `List<string>`.

#### `T.class` — литерал без класса

```java
Class<T> token() { return T.class; }   // error: cannot select from a type variable
```

Литерал `X.class` компилируется в загрузку конкретного `Class`-объекта из constant pool. Для `T` грузить нечего. Обход сквозной для всего урока: `Class<T>` передаётся параметром снаружи, оттуда, где тип ещё конкретен.

> [!tip] 💡 Из C#  
> `typeof(T)` работает прямо в теле generic-метода, плюс есть `default(T)`. В Java аналог `default(T)` невозможен по той же причине — не зная тип, не знаешь даже, ссылочный он или нет. Впрочем, в Java `T` всегда ссылочный, так что `null` — честный «дефолт».

#### Generic-исключения — самый красивый запрет

```java
class AppException<T> extends Exception { }   // error: a generic class may not extend java.lang.Throwable

catch (T e) { }                               // error: unexpected type
```

Выведи сам: что делает JVM, когда летит исключение? Идёт по exception table метода и **сравнивает класс исключения** с классом в каждом catch-блоке — рантайм-сопоставление по типу, по сути тот же `instanceof`. `catch (T e)` потребовал бы сравнить с... ничем. А разреши `class AppException<T> extends Exception` — и `catch (AppException<IOException> e)` против `catch (AppException<SQLException> e)` неразличимы: в рантайме оба просто `AppException`. Поэтому JLS §8.1.2 запрещает generic-классам наследовать `Throwable` целиком.

Тонкость, которую я проверил отдельно: сам _класс исключения_ generic быть не может, но generic-**метод** с `throws T` — легален:

```java
static <T extends Exception> void f(T ex) throws T { throw ex; }   // компилируется
```

Потому что `throws` — это compile-time контракт, а не рантайм-сопоставление по exception table.

#### Два флажка на уроки 7.7–7.8

Коротко, чтобы опознал при встрече. **Каст к `T`** — `(T) obj` компилируется с unchecked warning и в рантайме является no-op; `ClassCastException` всплывает позже, в месте использования. **Generic varargs** — `void f(T... args)` это скрытый `new T[]`, то есть то же heap pollution через чёрный ход; компилятор ругается «possible heap pollution», лечится аннотацией `@SafeVarargs` (Java 7; на private-методах — с Java 9). Полный разбор raw types и стратегии работы с warnings — следующие уроки.

> Одна колонка — один и тот же корень: у C# типы reified, у Java `T` к рантайму испарился.
> 
> |Операция|Java|C#|

### Часть B (7.6): bridge methods — что компилятор дописывает за тебя

Первая половина: стирание било по _тебе_ — ты хочешь операцию, компилятор запрещает. Вторая зеркальная: стирание ломает вещь, которую ты даже не просил, — **переопределение методов**, — и чинит её компилятор сам, тайком дописывая в твой класс методы, которых нет в исходнике.

#### Сломай override мысленно

Сквозной пример до конца урока:

```java
class Node<T> {
    public void set(T value) { System.out.println("Node.set"); }
}
class StringNode extends Node<String> {
    @Override
    public void set(String value) { System.out.println("StringNode.set: " + value.length()); }
}
```

Компилируется, `@Override` доволен. А теперь примени урок 7.4 — сотри типы руками, как это сделает `javac`: `Node<T>` → класс `Node` с методом `set(Object value)`; `StringNode` → класс с методом `set(String value)`. Стоп. `set(Object)` и `set(String)` — **разные сигнатуры**. Из урока про overloading vs overriding (4.2): разные сигнатуры — не переопределение, а перегрузка. И вспомни, как работает `invokevirtual`: JVM ищет метод с **точно той же сигнатурой**, что в инструкции вызова. Скомпилированный вызов через ссылку родителя:

```java
Node<String> node = new StringNode();
node.set("hello");   // в байткоде: invokevirtual Node.set(Object)
```

JVM ищет в `StringNode` метод `set(Object)`... а там по логике стирания только `set(String)`. Не совпало — поднимаемся к `Node`, находим `set(Object)`, печатается `Node.set`. Полиморфизм мёртв, твоё переопределение проигнорировано. **Так должно быть по логике стирания.** Но запусти код — печатается `StringNode.set: 5`. Работает. Значит, в `StringNode` всё-таки **есть** метод `set(Object)`. Откуда?

#### `javap` — доказательство

Прогнал рефлексию на JDK 21:

```java
for (var m : StringNode.class.getDeclaredMethods())
    System.out.println(m + " | bridge=" + m.isBridge() + " synthetic=" + m.isSynthetic());
```

```
public void StringNode.set(java.lang.String) | bridge=false synthetic=false
public void StringNode.set(java.lang.Object) | bridge=true  synthetic=true
```

Два метода. Ты писал один. Смотрим второй через `javap -v StringNode`:

```
public void set(java.lang.Object);
  descriptor: (Ljava/lang/Object;)V
  flags: (0x1041) ACC_PUBLIC, ACC_BRIDGE, ACC_SYNTHETIC
  Code:
     0: aload_0
     1: aload_1
     2: checkcast     #14      // class java/lang/String
     5: invokevirtual #29      // Method set:(Ljava/lang/String;)V
     8: return
  LineNumberTable:
     line 4: 0
```

Вот он — **bridge method**. Компилятор синтезировал в `StringNode` метод со стёртой сигнатурой `set(Object)`, который кастует аргумент к `String` (`checkcast`) и делегирует в твой настоящий `set(String)` (`invokevirtual`). Флаги `ACC_BRIDGE` и `ACC_SYNTHETIC` помечают: «сгенерировано, в исходнике не ищи». Теперь цепочка срастается — вот она целиком:

![[Pasted image 20260709145914.png]]

Родителя переопределяет именно bridge (сигнатуры совпали), а твой код получает управление через делегирование. Отсюда и название — мостик через дыру, которую прогрызло стирание.

#### Bridge — не костыль дженериков, а общий механизм

Дженерики — самый частый, но не единственный заказчик мостов. Второй случай ты знаешь из урока про наследование — **ковариантный возврат** (проверено на JDK 21, дженериков тут нет вовсе):

```java
class Parent { Object foo() { return "parent"; } }
class Child extends Parent { @Override String foo() { return "child"; } }
```

Та же дыра под другим углом: в JVM дескриптор метода включает тип возврата, так что `()Ljava/lang/Object;` и `()Ljava/lang/String;` — разные методы, и `invokevirtual Parent.foo()Object` твой `String foo()` не нашёл бы. Рефлексия в `Child` снова показывает два метода:

```
java.lang.String Child.foo() | bridge=false
java.lang.Object Child.foo() | bridge=true
```

Побочная красота: в `Child` сосуществуют два метода, различающиеся **только типом возврата**. В исходнике такое запрещено, а на уровне байткода — легально, и bridge этим пользуется. Тот же механизм срабатывает у `class Price implements Comparable<Price>` — проверено: твой `compareTo(Price)` получает мост `compareTo(Object)`, без которого не работал бы ни один `Collections.sort`.

> [!tip] 💡 Из C#  
> Мостов нет вообще. `Node<string>` в CLR — овеществлённый тип, у которого `set(string)` — настоящий метод с настоящей сигнатурой; override совпадает напрямую. Bridge — чисто Java-артефакт: цена, которую компилятор платит за стирание, чтобы сохранить тебе полиморфизм.

> [!example] 🔬 Глубже / на синьора  
> **Почему стектрейс из моста указывает в строку объявления класса.** У bridge-метода нет исходного текста, но `LineNumberTable` ему всё равно нужна — и компилятор проставляет строку объявления класса-потомка (в разборе `javap` выше — `line 4: 0`, а строка 4 в файле это `class StringNode extends Node<String> {`). Поэтому фрейм в стектрейсе показывает метод, которого в исходнике нет, на строке, где нет кода метода.
> 
> **`invokevirtual` vs `invokeinterface`.** Мост, реализующий generic-интерфейс (`Comparable<Price>` → `compareTo(Object)`), вызывается через `invokeinterface`, а не `invokevirtual` — разрешение слота идёт иначе, но принцип «стёртая сигнатура + делегирование» тот же.
> 
> **Мост может быть не только для аргументов и возврата.** Компилятор синтезирует мосты и при сужении видимости или при наследовании generic-метода через несколько уровней иерархии — всякий раз, когда стёртая сигнатура наследника расходится с тем, что ожидает `invoke*` по родителю.

> [!warning] ⚠️ Где мост кусается
> 
> 1. **`ClassCastException` из метода, которого нет.** `checkcast` внутри моста — рантайм-проверка, и она проваливается, если через heap pollution в `StringNode` заехал не тот тип. Проверено:
>     
>     java
>     
>     ```java
>     Node<String> node = new StringNode();
>     Node raw = node;   // raw-ссылка, unchecked warning
>     raw.set(42);       // компилируется! летит в bridge set(Object)
>     ```
>     
>     ```
>     java.lang.ClassCastException: class java.lang.Integer cannot be cast to class java.lang.String
>         at StringNode.set(BridgeCCE.java:4)
>         at BridgeCCE.main(BridgeCCE.java:13)
>     ```
>     
>     `StringNode.set(BridgeCCE.java:4)` — строка 4 это `class StringNode extends Node<String> {`, объявление класса. Стектрейс указывает в синтетический метод, которого в исходнике нет. Если не знать про мосты, на дебаге такой фрейм выносит мозг. Симметрия с частью A: наврали в `raw.set(42)` — рвануло внутри сгенерированного кода.
> 2. **Рефлексия видит мосты.** `getDeclaredMethods()` возвращает синтетику — ты видел выше два `set` и два `foo`. Классический баг: код обходит методы по имени, находит **два**, вызывает «не тот» или регистрирует обработчик дважды. Фреймворки поголовно фильтруют через `Method.isBridge()` / `isSynthetic()` — и твой рефлексивный код должен. Вызвать мост рефлексией при этом можно: с точки зрения JVM это обычный публичный метод, просто с флагами.

#### На собесе

> [!question]- Почему `ArrayList` внутри хранит `Object[]`, а не `T[]`?  
> **RU:** `new T[n]` запрещён: массивы reified и требуют рантайм-тип элемента, а `T` стёрт. Разреши это — через ковариантность массивов (`List<String>[]` → `Object[]`) можно положить чужой тип, и `ArrayStoreException` не сработает, потому что проверять нечего: heap pollution. Поэтому тип создаваемого массива обязан быть reifiable (JLS §15.10.1), и единственный честный массив без знания `T` — `Object[]`.
> 
> **EN:** `new T[n]` is forbidden — arrays are reified and need the element's runtime type, but `T` is erased. Allow it and array covariance lets a wrong-typed element in with no `ArrayStoreException` (nothing to check) — heap pollution. So the created array type must be reifiable (JLS §15.10.1), and `Object[]` is the only honest array you can build without knowing `T`.
> 
> _Follow-up:_ «а как тогда работает `toArray(T[] a)`?» → берёт класс элемента из переданного массива и создаёт настоящий типизированный через `Array.newInstance` — type token, протащенный в рантайм.
> 
> _Слабый ответ:_ «чтобы можно было хранить любые объекты» — мимо: дело не в удобстве, а в невозможности создать `T[]`.

> [!question]- Метод возвращает `(T[]) new Object[size]`. Где и почему упадёт?  
> **RU:** Внутри метода — нигде: каст после стирания пустой, no-op. Упадёт `ClassCastException` **у вызывающего**, при присваивании в `String[]`: компилятор вставил там `checkcast`, а массив реально `Object[]`.
> 
> **EN:** Inside the method — nowhere: the cast is a no-op after erasure. It throws `ClassCastException` at the caller, on assignment to `String[]`: the compiler inserted a `checkcast` there, and the array is really `Object[]`.
> 
> _Follow-up:_ «как починить?» → `Array.newInstance(clazz, size)` с переданным `Class<T>` — создаётся настоящий `T[]`. _Второй follow-up:_ «почему внутри класса идиома безопасна?» → массив не утекает наружу под типом `T[]`, все чтения/записи консистентны.

> [!question]- Скомпилируется ли `x instanceof List<String>`? А `x instanceof List<?>`?  
> **RU:** Первое — нет: `instanceof` это рантайм-сверка с классом объекта, а объект помнит только `ArrayList`, без аргумента. Второе — да: wildcard ничего не утверждает, проверяется сырой тип.
> 
> **EN:** The first — no: `instanceof` is a runtime check against the object's class, and the object only remembers `ArrayList`, without the argument. The second — yes: the wildcard asserts nothing, the raw type is checked.
> 
> _Follow-up:_ «pattern matching for `instanceof` (Java 16, JEP 394) это изменил?» → нет, синтаксис новый, физика стирания та же: легально только `List<?> ls`.

> [!question]- Почему generic-класс не может наследовать `Throwable`?  
> **RU:** Потому что `catch` — рантайм-сопоставление по классу исключения (exception table), а параметр стёрт: `catch (AppException<IOException>)` и `catch (AppException<SQLException>)` неразличимы (JLS §8.1.2).
> 
> **EN:** Because `catch` is a runtime match on the exception's class (the exception table), and the parameter is erased: `catch (AppException<IOException>)` and `catch (AppException<SQLException>)` are indistinguishable (JLS §8.1.2).
> 
> _Follow-up-ловушка:_ «а `<T extends Exception> void f() throws T` легален?» → да: `throws` — compile-time контракт, рантайм-сопоставления нет.

> [!question]- Что такое bridge method и зачем он нужен?  
> **RU:** После стирания сигнатура переопределяющего метода (`set(String)`) перестаёт совпадать с родительской (`set(Object)`) — формально это перегрузка, и `invokevirtual` по родительской сигнатуре пролетел бы мимо. Компилятор синтезирует в потомке метод со стёртой сигнатурой, который делает `checkcast` и делегирует в настоящий. Видно в `javap` по флагам `ACC_BRIDGE`/`ACC_SYNTHETIC`.
> 
> **EN:** After erasure the overriding method's signature (`set(String)`) no longer matches the parent's (`set(Object)`) — formally it's an overload, and `invokevirtual` on the parent signature would miss. The compiler synthesizes a method with the erased signature in the subclass that does a `checkcast` and delegates to the real one. Visible in `javap` via `ACC_BRIDGE`/`ACC_SYNTHETIC`.
> 
> _Follow-up:_ «когда мосты появляются без дженериков?» → ковариантный тип возврата: дескриптор в JVM включает возврат. _Ещё follow-up:_ «как мост увидеть?» → `javap -v` или рефлексией `Method.isBridge()`.

> [!question]- Стектрейс показывает `at StringNode.set(File.java:4)`, но на строке 4 — объявление класса, и метода `set(Object)` в исходнике нет. Что происходит?  
> **RU:** `ClassCastException` из `checkcast` внутри bridge-метода: через raw-ссылку в типизированный класс заехал чужой тип, мост поймал его при делегировании. Синтетическому методу компилятор приписывает строку объявления класса.
> 
> **EN:** A `ClassCastException` from the `checkcast` inside the bridge: a foreign type got into the typed class via a raw reference, and the bridge caught it while delegating. The compiler assigns the class-declaration line to the synthetic method.
> 
> _Follow-up:_ «как такое вообще допустили в рантайм?» → raw types отключают проверки компилятора — unchecked warning был, его проигнорировали.

> [!question]- Контраст с C# одной фразой  
> **RU:** В CLR типы овеществлены: `new T()` с констрейнтом, `new T[n]`, `x is List<string>`, `typeof(T)` — всё работает напрямую, мостов нет. Каждый Java-запрет и каждый мост — следствие одного решения: стирание ради совместимости байткода.
> 
> **EN:** In the CLR types are reified: `new T()` with a constraint, `new T[n]`, `x is List<string>`, `typeof(T)` all work directly, and there are no bridges. Every Java prohibition and every bridge follows from one decision: erasure for bytecode compatibility.
