
Представь собес в Klarna. Интервьюер видит в резюме .NET и почти гарантированно спрашивает: «Сравни дженерики Java и C#». Для человека с твоим бэкграундом это вопрос-ловушка. Слабый кандидат начинает перечислять по памяти: «в Java нельзя `new T()`, нельзя `T[]`, нельзя `instanceof List<String>`, есть какие-то raw types, какие-то bridge-методы...» — список из шести пунктов, который надо держать в голове и который на любом follow-up рассыпается.

Сильный кандидат отвечает **одним предложением** и потом разворачивает любой пункт по запросу. Потому что это не шесть фактов. Это один факт с шестью следствиями. Этот урок не добавляет механики — ты всё прошёл в 7.1–7.8. Он собирает пройденное в одну модель, из которой каждое правило _выводится_, а не вспоминается.

**Mental model.** В Java дженерик — это чертёж для компилятора; в C# — деталь в рантайме. Снимок типов делается на компиляции и в рантайм не доезжает. Весь спор двух языков — про один бит: **доедет ли тип-аргумент до рантайма.**

### Корневой бит

Вопрос, который делит два языка: когда ты пишешь `List<String>`, существует ли это `String` после компиляции?

**C# — да.** В .NET 2.0 (2005) Microsoft расширила сам CLR: дженерики — фича _рантайма_. `List<int>` и `List<string>` — два разных типа, которые рантайм знает, различает и под которые JIT генерит код. Тип-аргумент овеществлён — reified.

**Java — нет.** Дженерики (Java 5, 2004) — фича _компилятора_. Он проверяет типы на этапе компиляции, а потом стирает: `List<String>` становится `List`, `T` становится `Object` (или верхней границей, если есть `extends`). В байткоде живёт один-единственный `ArrayList`. Это erasure — стирание (`JLS §4.6`).

Проверяется прямо в дескрипторах метода — `javap -s` на generic-классе:

```java
class Desc<T, U extends Number> {
    T id(T t)   { return t; }   // descriptor: (Ljava/lang/Object;)Ljava/lang/Object;
    U num(U u)  { return u; }   // descriptor: (Ljava/lang/Number;)Ljava/lang/Number;
}
```

`T` без границы стёрт до `Object`, `U extends Number` — до `Number` (верхней границы). Никаких `T`/`U` в рантайме нет. Это весь корень. Теперь смотри, как из него падает дерево — правило игры: я называю ветку, и мы _выводим_ её из корня, а не вспоминаем.

### Регенерируем дерево

**Runtime-идентичность.** Если тип-аргумент стёрт, у `ArrayList<String>` и `ArrayList<Integer>` в рантайме просто нечем различаться (проверено на JDK 21):

```java
var strings = new ArrayList<String>();
var ints    = new ArrayList<Integer>();
System.out.println(strings.getClass() == ints.getClass());  // true
System.out.println(strings.getClass().getName());           // java.util.ArrayList
```

Один объект `Class` на все параметризации. В C# ровно наоборот — `typeof(List<int>) != typeof(List<string>)`, у каждой закрытой конструкции свой `Type`, и `GetType()` объекта знает аргумент.

**Статика.** Один класс в рантайме → одна копия статических полей на все параметризации. И вопрос: если бы Java разрешила `static T cache;` — чей это `T`? От `Box<String>`? От `Box<Integer>`? Поле-то одно. Противоречие — поэтому компилятор запрещает (проверено):

```
Box.java:2: error: non-static type variable T cannot be referenced from a static context
    static T cache;
           ^
```

**`new T()`.** Чтобы вызвать конструктор, рантайм должен знать, _какой класс_ конструировать. К моменту исполнения `T` — уже `Object`, информации нет. В C# с `where T : new()` рантайм знает `T` — JIT подставляет реальный конструктор. Выход в Java: протащить знание руками — `Supplier<T>` или `Class<T>`.

**`new T[]`.** Тоньше. Массивы в JVM — единственная _reified_-конструкция: массив хранит свой компонентный тип и проверяет каждую запись в рантайме (отсюда `ArrayStoreException`). `new T[10]` потребовал бы структуру, которая в рантайме знает `T`, — а `T` стёрт. Конфликт двух систем: reified-массивы против erased-дженериков, компилятор режет на корню (`error: generic array creation`). В C# обе системы reified, конфликта нет.

**`instanceof List<String>`.** `instanceof` — проверка в рантайме, а в объекте лежит только «я `ArrayList`», без аргумента. Проверять нечего — `error: Object cannot be safely cast to List<String>`. Разрешён лишь `instanceof List<?>`. В C# `x is List<string>` — обычный код, информация на месте.

**`T.class`.** Литерал `String.class` — ссылка на рантайм-объект `Class`. Для `T` такого объекта не существует. В C# `typeof(T)` отдаёт его прямо. Java-воркараунд — снова таскать `Class<T>` параметром (type token).

**Bridge methods.** Пишешь `compareTo(MyType)`, реализуя `Comparable<MyType>`. После стирания интерфейс требует `compareTo(Object)` — сигнатуры разъехались, override по правилам JVM не случился бы. Компилятор синтезирует мост. Это не «фича», это _шов_ — место, где стирание сшивают с полиморфизмом вручную.

**Raw types.** Самая историческая ветка. Erasure выбрали ради ретрофита: код 2003 года с голым `List` обязан был компилироваться и _линковаться_ с новым generic-кодом без перекомпиляции. Отсюда raw types и весь зоопарк unchecked-варнингов — это шрам обратной совместимости. C# пошёл другим путём — новый неймспейс `System.Collections.Generic` рядом со старым `System.Collections`; старые коллекции не ретрофитили, поэтому понятия «raw `List<T>`» в C# просто не существует.

Одно последствие raw-мира, которое ты уже трогал руками, — **heap pollution**. Раз в рантайме все `List` одинаковые, проверки нет нигде, кроме точки чтения (проверено на JDK 21):

```java
List<String> names = new ArrayList<>();
Object leaked = names;
List<Integer> numbers = (List<Integer>) leaked;  // unchecked warning — и всё
numbers.add(42);                                 // прошло! рантайму нечего проверять
String first = names.get(0);                     // ClassCastException — здесь
```

Падает не на `add`, а на `get` — потому что единственная рантайм-проверка это `checkcast`, который компилятор вставил _на стороне читателя_. На JDK 21: `add(42)` молча проходит, `ClassCastException: class java.lang.Integer cannot be cast to class java.lang.String` — на последней строке.

А теперь follow-up, который любят интервьюеры: «Если всё стёрто, как тогда рефлексия видит дженерики? `getGenericSuperclass()` же возвращает `String`!» Ответ из 7.4: стирается информация _из инстансов_, но не _из деклараций_. Сигнатуры полей, методов, `extends`-клаузы сохраняются в class-файле как метаданные — атрибут `Signature` (`JVMS §4.7.9`). Проверено:

```java
public class StringList extends ArrayList<String> { }
// StringList.class.getGenericSuperclass()  ->  java.util.ArrayList<java.lang.String>
// javap -v:  Signature: Ljava/util/ArrayList<Ljava/lang/String;>;
```

Объект не знает, что он `List<String>`; но декларация `class StringList extends ArrayList<String>` — знает. На этом стоит super type token и половина Jackson.

> **Как выглядит bridge-метод в байткоде.** Возьми `class MyType implements Comparable<MyType>` с одним `compareTo(MyType)`. В `.class` окажутся _два_ метода `compareTo`. `javap -v` показывает синтетический мост с флагами `ACC_BRIDGE | ACC_SYNTHETIC` (проверено):
> 
> ```
> public int compareTo(java.lang.Object);
>   flags: (0x1041) ACC_PUBLIC, ACC_BRIDGE, ACC_SYNTHETIC
>   Code:
>      0: aload_0
>      1: aload_1
>      2: checkcast     #8    // class MyType
>      5: invokevirtual #19   // Method compareTo:(LMyType;)I
>      8: ireturn
> ```
> 
> Мост принимает `Object` (сигнатура, которую после стирания требует интерфейс), кастит его к `MyType` (`checkcast`) и делегирует в твой реальный `compareTo(MyType)` (`invokevirtual`). Именно этот `checkcast` внутри моста бросит `ClassCastException`, если в `compareTo` прилетит чужой тип через raw-ссылку — то есть мост это ещё и точка проверки типа. В C# мостов нет в принципе: reified-сигнатуры override совпадают сами.
> 
> **Вариантность — соседняя ось, не ветка этого дерева.** Соблазн повесить на корень вообще всё, включая `? extends`/`? super` против `in`/`out`. Не выйдет. Связи с reification у вариантности есть: ковариантность массивов в Java работает _только_ потому, что массивы reified (`ArrayStoreException` бросает рантайм-проверка компонентного типа при записи); а C# ограничивает вариантность ссылочными типами, потому что value-параметризации имеют разное физическое представление. Но сам выбор — _где_ объявлять вариантность — независимое дизайн-решение. Java выбрала use-site (пишущий метод просит `List<? extends Number>`), C# — declaration-site (автор интерфейса помечает `out T`). Оба совместимы и со стиранием, и с reification — Kotlin живёт на той же erased-JVM с declaration-site `in`/`out`. Так что на вопрос «вариантность — следствие erasure?» правильный ответ: нет, отдельная ось; erasure влияет на её _границы_, но не диктует _форму_. Назвать эту границу модели — маркер того, что ты понимаешь, а не заучил.

> 
> **Цена боксинга в цифрах.** `List<int>` в C#: JIT для каждой value-параметризации генерит специализированную нативную копию кода — внутри лежит настоящий `int[]`, инты упакованы инлайн, без единого объекта-обёртки. Для ссылочных типов CLR хитрит: `List<string>` и `List<Uri>` делят одну копию кода (любая ссылка физически одинакова). Цена — каждая value-параметризация это плюс одна сборка кода, code bloat.  
> `ArrayList<Integer>` в Java: кода ровно одна копия, внутри `Object[]`. Примитив туда лечь не может — каждый `int` боксится в объект `Integer`: заголовок объекта ~12 байт на HotSpot 64-bit с compressed oops (mark word 8 + compressed klass 4), плюс 4 байта самого значения = 16 байт на бокс, плюс 4 байта ссылки в массиве. Десять миллионов интов: `int[]` — ~40 МБ сплошным куском; `ArrayList<Integer>` — порядка 200 МБ, разбросанных по куче, с индирекцией и кеш-промахом на каждом чтении. `Integer.valueOf` кеширует `−128..127` (7.1.4) — экономит аллокации для мелких чисел, но индирекцию не убирает.  
> Что Java делает с этой болью на практике: ручная специализация в библиотеке — `IntStream` вместо `Stream<Integer>`, `IntFunction` вместо `Function<Integer, R>`, голые `int[]` в хот-патах. Это не элегантность, это обходной манёвр вокруг erasure, зашитый в JDK.
> 
> **Флажок на будущее (обновлено к июлю 2026).** Project Valhalla — многолетняя попытка OpenJDK принести value-классы и специализацию. Статус сдвинулся: `JEP 401` «Value Classes and Objects» подтверждён к интеграции в mainline с таргетом **JDK 28 как preview** (не GA, выключено по умолчанию; Brian Goetz прогнозирует, что и в следующем LTS — JDK 29 — он останется preview). Важно для этой главы: сами value-классы (`JEP 401`) _не_ убивают боксинг в `ArrayList<Integer>` напрямую — они снимают identity и открывают scalarization/flattening. За специализацию дженериков над value-типами (то, что реально сделало бы `ArrayList<Point>` плоским) отвечает **отдельный** `JEP 218`, ещё дальше по дороге. На собесе это упоминается как «куда движется платформа», не как доступная в LTS фича.

**Почему каждый выбрал своё** (это mid+ core, спрашивают постоянно). Один и тот же вопрос — «что делать с уже написанным кодом?» — два противоположных ответа. Java несла миллиарды строк и принцип «байткод под 1.4 обязан работать на 5». Стирание дало ровно это: JVM менять не пришлось, generic- и pre-generic-код линкуются без перекомпиляции, миграция — файл за файлом. Купили migration compatibility, заплатили рантайм-информацией о типах — и всей правой колонкой таблицы. Microsoft контролировала CLR целиком и была готова его менять: дженерики вживили в рантайм, а легаси решили параллельным неймспейсом. Купили reified-типы и специализацию, заплатили изменением рантайма и тем, что старые и новые коллекции — чужие друг другу типы. Это размен, а не ошибка.

> [!warning] ⚠️ Подводные камни — где .NET-привычка молча даёт другую семантику
> 
> 1. **`static` в generic-классе считает по-разному.** В C# `static int counter;` в `Foo<T>` — свой счётчик у `Foo<int>` и `Foo<string>` (разные рантайм-типы). В Java один класс на все параметризации → счётчик **общий**. Тот же код — другая семантика; и `static T field;` в Java вообще не скомпилируется, хотя в C# легален.
> 2. **`==` на `Integer` из-за кеша.** Боксинг + кеш `−128..127` дают классику: `Integer a=127, b=127; a==b` → `true`, а `128==128` → `false`. Это прямое следствие того, что `Integer` — объект в куче, а не value-тип. В C# `int` сравнивается по значению всегда.
> 3. **Heap pollution падает не там, где грешит.** Ждёшь `ClassCastException` на `add(42)` — а он на `get(0)` у читателя, потому что единственная проверка это `checkcast`, вставленный компилятором на стороне чтения. Хочешь поймать раньше — `Collections.checkedList(list, String.class)` переносит падение на `add` (проверено: `ClassCastException: Attempt to insert ... Integer ... element type ... String`).
> 4. **`instanceof`/`getClass` не различают параметризации.** `list instanceof List<String>` не компилируется; `a.getClass() == b.getClass()` для `ArrayList<String>`/`ArrayList<Integer>` — `true`. Привычка из C# (`x is List<string>`, `typeof(T)`) здесь просто не переносится — таскай `Class<T>` руками.
> 5. **Массивы дженериков.** `new T[10]` и `new List<String>[10]` запрещены (`generic array creation`), потому что reified-массивы конфликтуют с erased-дженериками. В C# и то и другое работает. Обход в Java — `Object[]` + `@SuppressWarnings`, либо `List<List<String>>`.

### На собесе

> [!question]- «Сравни дженерики Java и C#» _(топ-вопрос для .NET-бэкграунда)_  
> **RU:** Разница в одном бите — доедет ли тип-аргумент до рантайма. C# — да, reified: CLR расширили в 2.0, `List<int>` и `List<string>` разные рантайм-типы, JIT их специализирует. Java — нет, erased: дженерики фича компилятора, после проверки типы стираются, в рантайме один `List`, `T` → `Object`. Из этого бита выводится всё: один `Class` на все параметризации, статика общая и не видит `T`, нельзя `new T()`/`new T[]`/`instanceof List<String>`, компилятор синтезирует bridge-методы, существуют raw types. В C# всё зеркально наоборот.
> 
> **EN:** It comes down to one bit — does the type argument survive to runtime. C# — yes, reified: the CLR was extended in 2.0, `List<int>` and `List<string>` are distinct runtime types the JIT specializes. Java — no, erased: generics are a compiler feature; after checking, types are erased, at runtime there's one `List` and `T` becomes `Object`. Everything follows from that bit: one `Class` for all parameterizations, static state shared and blind to `T`, no `new T()`/`new T[]`/`instanceof List<String>`, compiler-synthesized bridge methods, raw types. C# is the mirror image.
> 
> _Follow-up:_ «а в C# что с `getClass`?» → `typeof` даёт разные `Type`. _Trap:_ «а как тогда `getGenericSuperclass()` возвращает `String`?» → стирается инфа из инстансов, декларации хранят её в атрибуте `Signature`; на этом стоит super type token.  
> _Слабый ответ:_ перечисление шести запретов без корня — «список, который надо помнить». Ждут один бит + вывод.

> [!question]- «Почему `static T cache;` не компилируется, ведь в C# я так писал?»  
> **RU:** В Java один класс на все параметризации → одна статика → «чей `T`?» не имеет ответа. В C# `Foo<int>` и `Foo<string>` разные рантайм-типы, у каждого своя статика — поэтому легально.  
> **EN:** In Java there's one class for all parameterizations → one copy of statics → "whose `T`?" has no answer. In C# `Foo<int>` and `Foo<string>` are distinct runtime types, each with its own statics — hence legal.  
> _Follow-up:_ «и что это значит для `static int counter`?» → в Java счётчик общий на все параметризации, в C# раздельный по каждой value-конструкции. Тот же код — разная семантика.

> [!question]- «Где упадёт этот код?» _(показывают heap pollution)_  
> **RU:** Компиляция пройдёт с unchecked warning, `add(42)` пройдёт молча, упадёт `ClassCastException` на `String s = names.get(0)`. Единственная рантайм-проверка — `checkcast`, вставленный компилятором на стороне чтения; сам список не проверяет, ему нечем.  
> **EN:** It compiles with an unchecked warning, `add(42)` passes silently, and it throws `ClassCastException` at `names.get(0)`. The only runtime check is a `checkcast` the compiler inserts at the read site; the list itself checks nothing — it has nothing to check with.  
> _Follow-up:_ «почему не на `add`?» → в рантайме нет `List<String>`, есть голый `List`, запись легальна — это и есть heap pollution. _Trap:_ «как поймать раньше?» → `Collections.checkedList`, падение переезжает на `add`.

> [!question]- «Почему нельзя `new T()`, и как это обходят в реальном коде?»  
> **RU:** Нельзя — к моменту исполнения `T` стёрт до `Object`, конструировать нечего. Обходы: `Supplier<T>` (предпочтительно — type-safe, без рефлексии, дружит с `ArrayList::new`) или `Class<T>` + `getDeclaredConstructor().newInstance()`.  
> **EN:** You can't — by runtime `T` is erased to `Object`, there's nothing to construct. Workarounds: `Supplier<T>` (preferred — type-safe, no reflection, pairs with `ArrayList::new`) or `Class<T>` + `getDeclaredConstructor().newInstance()`.  
> _Follow-up:_ «какой выберете?» → `Supplier`: проверяется компилятором, не падает в рантайме на отсутствующем конструкторе.

> [!question]- «Что быстрее для миллионов чисел — `List<int>` в C# или `ArrayList<Integer>` в Java?»  
> **RU:** C#, и заметно: JIT специализирует value-типы, инты лежат инлайн в `int[]`, без боксинга и индирекции. Java боксит каждый элемент в `Integer` — кратно больше памяти, хуже кеш-локальность. Митигации: `IntStream`/`LongStream`, примитивные массивы, специализированные функциональные интерфейсы; стратегически — Valhalla (`JEP 401` таргетит JDK 28 как preview, специализацию дженериков даст отдельный `JEP 218`), но в текущих LTS его нет.  
> **EN:** C#, noticeably: the JIT specializes value types, ints sit inline in an `int[]`, no boxing or indirection. Java boxes every element into an `Integer` — many times the memory, worse cache locality. Mitigations: `IntStream`/`LongStream`, primitive arrays, specialized functional interfaces; strategically Valhalla (`JEP 401` targets JDK 28 as preview; generic specialization is a separate `JEP 218`), but it's not in a current LTS.  
> _Follow-up:_ «а обратная цена у C#?» → code bloat: копия кода на каждую value-параметризацию. Назвать размен в обе стороны — плюс балл, показывает, что не топишь за один язык.

> [!question]- «Зачем существуют bridge methods, если их никто не пишет?»  
> **RU:** Это шов между erasure и полиморфизмом: после стирания сигнатура переопределяемого метода (`compareTo(MyType)` → интерфейс требует `compareTo(Object)`) перестаёт совпадать, и JVM-овский override сломался бы. Компилятор синтезирует мост, который кастит (`checkcast`) и делегирует. Видно через `javap -v` по флагам `ACC_BRIDGE | ACC_SYNTHETIC`. В C# мостов нет — reified-сигнатуры совпадают сами.  
> **EN:** It's the seam between erasure and polymorphism: after erasure the overriding method's signature (`compareTo(MyType)` → the interface wants `compareTo(Object)`) no longer matches, so JVM overriding would break. The compiler synthesizes a bridge that casts (`checkcast`) and delegates. Visible via `javap -v` by the `ACC_BRIDGE | ACC_SYNTHETIC` flags. C# has no bridges — reified signatures match on their own.

> [!question]- «Вариантность — это следствие erasure?» _(senior-разделитель)_  
> **RU:** Нет, это отдельная ось. Erasure влияет на _границы_ вариантности (ковариантность массивов в Java держится на их reified-природе; C# ограничивает вариантность ссылочными типами из-за разного представления value-типов), но не диктует её _форму_. Где объявлять вариантность — независимое решение: Java выбрала use-site (`? extends`/`? super`), C# — declaration-site (`in`/`out`). Kotlin живёт на erased-JVM с declaration-site — доказательство, что форма от стирания не зависит.  
> **EN:** No — it's a separate axis. Erasure shapes the _limits_ of variance (Java's array covariance rests on arrays being reified; C# restricts variance to reference types because value parameterizations differ physically), but it doesn't dictate its _form_. Where you declare variance is an independent choice: Java picked use-site (`? extends`/`? super`), C# declaration-site (`in`/`out`). Kotlin runs on the erased JVM with declaration-site variance — proof the form is independent of erasure.
