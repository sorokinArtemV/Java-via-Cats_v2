В прошлом уроке инвариантность спасла типобезопасность, но выставила счёт: `feedAll(List<Animal>)` не принимает `List<Cat>`, хотя метод только читает элементы и физически не способен ничего испортить. Писать перегрузку под каждый подтип — абсурд; отказаться от проверки типов — путь raw-типов, от которого дженерики и уходили. Java решает это **wildcards** — механизмом, который включает ковариантность или контравариантность точечно, на месте использования типа, и за каждое послабление берёт понятную плату. Из асимметрии этих плат сам собой вырастет PECS — правило, по которому написана половина сигнатур JDK, — а в конце урока сигнатура `Collections.sort`, месяц назад выглядевшая как шум на линии, прочитается вслух как обычное предложение.

**Mental model.** Wildcard `?` — это честное «не знаю, какой именно тип, но знаю границу»; компилятор разрешает ровно те операции, безопасность которых доказуема при _любом_ реальном типе за ссылкой. Читаешь из коллекции — `? extends` (producer), пишешь в неё — `? super` (consumer): Producer Extends, Consumer Super.

### `List<?>` — честное «не знаю»

Задача на минуту: метод, печатающий любой список. Первая попытка — `printAll(List<Object> list)`. И она уже не работает: ты сам полчаса назад вывел, что `List<String>` — это не `List<Object>`, инвариантность не делает исключений даже для `Object` (проверено: `incompatible types: List<String> cannot be converted to List<Object>`). Нужен тип «список _чего-то_, не знаю чего». Он есть:

```java
static void printAll(List<?> list) {
    for (Object item : list) {        // читать как Object — можно
        System.out.println(item);
    }
}
```

`List<?>` принимает `List<Cat>`, `List<String>`, что угодно. Читать элементы можно — кем бы они ни были, они точно `Object`. А вот класть:

```java
List<Cat> cats = new ArrayList<>(List.of(new Cat()));
List<?> unknown = cats;
unknown.add(new Cat());
```

```
error: incompatible types: Cat cannot be converted to CAP#1
  where CAP#1 is a fresh type-variable:
    CAP#1 extends Object from capture of ?
```

`CAP#1` (в IDE — `capture#1 of ?`) — не баг и не мистика: компилятор взял твой неизвестный тип `?` и дал ему временное имя, чтобы рассуждать о нём. Читается так: «список ожидает элементы какого-то конкретного типа CAP#1, я не знаю какого — и потому не могу доказать, что твой `Cat` это он». Увидишь `capture of` в ошибке — это компилятор честно признаётся в незнании, а не ломается.

И вот где `List<?>` критично отличается от raw-типа `List`, хотя выглядят почти одинаково. Raw `List` тоже принимает любой список — но при этом _выключает_ проверку дженериков целиком:

```java
List<Integer> ints = new ArrayList<>();
List raw = ints;
raw.add("boom");                 // компилируется (unchecked warning)
Integer i = ints.get(0);         // ClassCastException: String cannot be cast to Integer
```

Строка в списке `Integer` — это **heap pollution**: мусор заложен в одном месте, а взрывается в другом, на чтении, где по коду вообще ничего подозрительного. `List<?>` проверку _сохраняет_: «не знаю тип» → «не докажу безопасность записи» → «запрещаю запись». Wildcard — это «не знаю», raw — это «мне плевать». Первое — инструмент, второе — наследие до-Java-5 кода, в новом коде ему не место.

### `? extends` — ковариантность, но на твоих условиях

Возвращаемся к боли из 7.2. Wildcard с верхней границей решает ровно её:

```java
static void feedAll(List<? extends Animal> animals) {
    for (Animal a : animals) {        // читаем как Animal
        feed(a);
    }
}

feedAll(cats);   // List<Cat> — принимается!
feedAll(dogs);   // List<Dog> — тоже
```

`List<? extends Animal>` = «список какого-то типа, про который известно одно: он не выше `Animal`». Реальный список за этой ссылкой может быть `List<Cat>`, `List<Dog>`, `List<Animal>` — неизвестно, какой именно. Теперь выведи из этого незнания, что можно делать с такой ссылкой.

**Чтение.** Каким бы ни был реальный список, каждый его элемент — точно `Animal` (граница гарантирует). `Animal a = animals.get(0)` — безопасно всегда. Дверь наружу открыта.

**Запись.** Пробуем всё подряд:

```java
List<? extends Animal> animals = cats;   // реально за ссылкой — List<Cat>
animals.add(new Dog());      // реальный список — коты, Dog туда нельзя
animals.add(new Animal());   // Animal — не Cat, тоже нельзя
animals.add(new Cat());      // а вдруг реальный список — List<Dog>? тоже нельзя!
```

Третья строка — самая поучительная. Ты-то _видишь_, что за ссылкой коты. Но компилятор проверяет тип ссылки, а тип говорит «какой-то подтип `Animal`, не скажу какой». Для любого кандидата на `add` найдётся возможный реальный список, который он отравит. Поэтому компилятор не перебирает кандидатов — он запрещает `add` целиком. JDK 21 подтверждает: все три строки — `error: ... cannot be converted to CAP#1`, где `CAP#1 extends Animal from capture of ? extends Animal`.

Единственное исключение — `animals.add(null)`: `null` принадлежит любому ссылочному типу, он влезает в любой реальный список (проверено — компилируется). Практической пользы ноль, но как вопрос с подвохом — встречается.

Итого: `? extends` — это ковариантность, которую ты включаешь точечно, и за неё компилятор тут же выставляет счёт — забирает запись. Сравни с массивами из 7.2: там ковариантность бесплатная и всегда включена, а счёт приходит в рантайме `ArrayStoreException`. Здесь — та же сделка, но оплата на компиляции.

### `? super` — а теперь наоборот

Симметричная задача: метод, который _складывает_ котов в переданную коллекцию — сканирует приют и собирает котов на вакцинацию:

```java
static void collectCats(List<? super Cat> sink) {
    sink.add(new Cat());
    sink.add(new Kitten());   // Kitten extends Cat — тоже ок
}
```

`List<? super Cat>` = «список какого-то типа не ниже `Cat`». Реальный список — `List<Cat>`, `List<Animal>` или `List<Object>` (проверено: `collectCats` принимает все три). Выводим возможности из незнания, как в прошлый раз.

**Запись.** Кот — он и в `List<Cat>` кот, и в `List<Animal>` животное, и в `List<Object>` объект. В _любой_ возможный реальный список `Cat` влезает по определению наследования. И любой подтип `Cat` тоже. А вот `Dog` — нет: вдруг реальный список — `List<Cat>`? Проверено: `sink.add(new Dog())` → `Dog cannot be converted to CAP#1`. Дверь внутрь открыта — но только для `Cat` и ниже.

**Чтение.** Что лежит в реальном списке? Если это `List<Object>` — там может лежать вообще что угодно, хоть `String`, положенный кем-то другим. Единственный тип, который компилятор гарантирует для `get`:

```java
Cat c = sink.get(0);      // error: CAP#1 cannot be converted to Cat
Object o = sink.get(0);   // только так
```

### PECS — не правило, а следствие

Смотри, что получилось само собой. `? extends` — читать можно, писать нельзя. `? super` — писать можно, читать только `Object`. Значит, выбор wildcard'а в сигнатуре метода диктуется ролью коллекции.

Коллекция _отдаёт_ тебе данные (ты из неё читаешь) — она producer — бери `? extends`. Коллекция _принимает_ данные (ты в неё пишешь) — она consumer — бери `? super`. **P**roducer **E**xtends, **C**onsumer **S**uper. PECS. Мнемоника от Джоша Блоха (_Effective Java_, 3rd ed., Item 31; во 2-м издании — Item 28; у Нафталина и Уодлера то же самое называется Get and Put Principle) — но ты её только что не запомнил, а вывел, и на собесе сможешь вывести заново, если формулировка вылетит из головы.

Канонический пример, где обе роли в одной сигнатуре:

```java
static <T> void copy(List<? super T> dest, List<? extends T> src) {
    for (T item : src) {       // src отдаёт — producer — extends
        dest.add(item);        // dest принимает — consumer — super
    }
}

List<Cat> cats = new ArrayList<>(List.of(new Cat(), new Cat()));
List<Animal> zoo = new ArrayList<>();
copy(zoo, cats);   // T выводится как Cat: коты текут в зоопарк. Проверено: [Cat, Cat]
```

Это почти дословно сигнатура `Collections.copy` из JDK. Там же живёт `Collection.addAll(Collection<? extends E>)` — добавляемая коллекция только отдаёт элементы, значит producer, значит `extends`. Открой исходники JDK — PECS там на каждой второй сигнатуре.

Два уточнения, когда wildcard _не_ нужен. Если коллекция и producer, и consumer одновременно — ты из неё читаешь `T` и в неё же пишешь `T` (например, `swap(List<T> list, int i, int j)`) — wildcard не подойдёт ни один, нужен честный параметр типа `T`. И если тебе безразлично и то и другое — метод трогает только размер, очищает, печатает — хватает `List<?>`: `printSize(List<?> list)`.

> [!example] 🔬 Глубже / на синьора  
> **Wildcard capture и capture-helper.** Формально при каждом обращении к значению wildcard-типа компилятор выполняет _capture conversion_ (JLS §5.1.10): подставляет вместо `?` свежую типовую переменную `CAP#1` с границами wildcard'а. Проблема `set`/`add` в том, что _каждое_ обращение порождает _новую_ переменную — компилятор не знает, что `CAP#1` из `get(j)` и `CAP#2` из `set(i, ...)` — один тип:
> 
> 
> 
> ```java
> static void swap(List<?> list, int i, int j) {
>     list.set(i, list.get(j));
>     // error: Object cannot be converted to CAP#1
> }
> ```
> 
> Лечится паттерном **capture helper** — приватный generic-метод даёт безымянному типу имя:
> 
> 
> 
> ```java
> static void swap(List<?> list, int i, int j) {   // публичный API — удобный wildcard
>     swapImpl(list, i, j);                        // компилятор выводит T = CAP#1
> }
> private static <T> void swapImpl(List<T> list, int i, int j) {
>     T tmp = list.get(i);
>     list.set(i, list.get(j));                    // теперь всё — один T
>     list.set(j, tmp);
> }
> ```
> 
> Проверено на JDK 21: `swap(["a","b","c"], 0, 2)` → `[c, b, a]`. Именно так устроен `Collections.swap` в JDK (внутри — helper через unchecked cast, идея та же). Это и есть Effective Java Item 31 в senior-версии: wildcard в публичной сигнатуре для гибкости, параметр типа в приватной — для работы.
> 
> **У wildcard — ровно одна граница.** `List<? extends Number & Comparable<?>>` не компилируется (`> or ',' expected`) — intersection-границы (`&`) существуют только у параметров типа. Ещё одна асимметрия двух позиций в копилку к «`super` только у wildcards».

### Награда: читаем сигнатуру из JDK

Открой `Collections.sort` (или `List.sort` — суть та же):

```java
static <T extends Comparable<? super T>> void sort(List<T> list)
```

У тебя есть все детали, кроме одной: почему `Comparable<? super T>`, а не строгое `Comparable<T>`, как мы писали в `maxOf`? Выведем на коде. Зоопарк сравнивает животных по весу:

```java
class Animal implements Comparable<Animal> {
    final int weight;
    Animal(int weight) { this.weight = weight; }
    public int compareTo(Animal other) { return Integer.compare(weight, other.weight); }
}
class Cat extends Animal {
    Cat(int weight) { super(weight); }
}
```

Разумный дизайн: `Cat` наследует `compareTo` от `Animal` — коты сравниваются по весу, как все животные, незачем переопределять. Но посмотри, _что_ реализует `Cat`: унаследованный `Comparable<Animal>`. Не `Comparable<Cat>`. И теперь:

```java
List<Cat> cats = new ArrayList<>(List.of(new Cat(5), new Cat(3)));

// строгая граница: <T extends Comparable<T>>
sortStrict(cats);
// error: inference variable T has incompatible equality constraints Animal,Cat

// граница из JDK: <T extends Comparable<? super T>>
sortJdk(cats);     // компилируется, печатает [Cat(3), Cat(5)]
```

Оба исхода проверены на JDK 21. Строгая версия отвергла совершенно нормальный код: `T = Cat` требует `Comparable<Cat>`, а `Cat` реализует только `Comparable<Animal>`. Версия JDK говорит мягче: «`T` должен уметь сравнивать себя с `T` _или любым его предком_» — `Comparable<Animal>` для `T = Cat` подходит: раз кот сравним с любым животным, то с котом — тем более. А почему именно `super`? Тот же PECS: `Comparable` — потребитель. Он не отдаёт `T`, он _принимает_ `T` в `compareTo` и съедает его для сравнения. Consumer Super — даже внутри границы типа. Прочитай теперь всю сигнатуру вслух: «сортирую список элементов любого типа `T`, лишь бы `T` умел сравнивать себя с собой или кем-то из своих предков». Сигнатура перестала быть шумом — она стала предложением на языке, которым ты владеешь.

Кстати, добавить `implements Comparable<Cat>` к `Cat` и «починить» строгую версию нельзя в принципе: класс не может реализовать `Comparable` дважды с разными аргументами — после стирания это один интерфейс (урок 7.4). Наследование `compareTo` — не лень, а единственный вариант.

> [!tip] 💡 Из C#: два мира вариантности  
> Здесь главное расхождение языков во всей теме, и его любят спрашивать у людей с .NET-бэкграундом.
> 
> В C# вариантность — **declaration-site**: автор типа решает один раз, при объявлении. `IEnumerable<out T>` — `out` говорит «`T` только наружу, тип ковариантен», и с этого момента `IEnumerable<Cat>` → `IEnumerable<Animal>` работает _везде_, у каждого пользователя, без лишних слов. `IComparer<in T>` — контравариантен, тоже раз и навсегда. Потому в C# и нет PECS как практики: за тебя уже подумал автор интерфейса.
> 
> В Java вариантность — **use-site**: тип всегда объявлен инвариантным, а ковариантность/контравариантность ты включаешь сам, в каждом конкретном месте использования, через `? extends` / `? super`. Плата — шум в сигнатурах (видел `Function<? super T, ? extends R>` в Stream API — теперь знаешь, откуда). Выигрыш — гибкость: один и тот же `List` ты в одной сигнатуре поворачиваешь как producer, в другой как consumer, в третьей используешь инвариантно для чтения-и-записи. В C# `List<T>` инвариантен навсегда, и никакой `out` ты к нему на месте не прикрутишь.
> 
> Два уточнения, чтобы не переносить C#-интуицию слишком щедро. Первое: `in`/`out` в C# доступны только интерфейсам и делегатам — классы всегда инвариантны, тут языки совпадают. Второе: вариантность C# работает только для ссылочных типов — `IEnumerable<int>` _не является_ `IEnumerable<object>`, потому что `int` — value-тип и требует боксинга. В Java этот вопрос не возникает по другой причине: в дженериках живут только ссылочные типы, `List<int>` не существует в принципе (привет, автобоксинг из урока 1.4).

> [!warning] ⚠️ Подводные камни
> 
> 1. **В `List<? extends Animal>` нельзя добавить _даже `Animal`_** — частая ловушка на собесах, интуиция шепчет «ну родителя-то можно». Нельзя ничего, кроме `null` — для любого кандидата найдётся возможный реальный список, который он отравит.
> 2. **Wildcard — тип _ссылки_, не объекта.** `new ArrayList<? extends Animal>()` → `error: unexpected type; required: class or interface without bounds`. Логично: при создании ты обязан знать, что создаёшь; wildcard описывает незнание читающего, а не создающего. Корректно: `List<? extends Animal> a = new ArrayList<Cat>();`.
> 3. **Wildcard в возвращаемом типе — антипаттерн.** `static <T> List<? extends T> process(...)` перекладывает capture-головняк на вызывающего: он получит ссылку, в которую ничего не положить, и `capture#1` в ошибках. Effective Java Item 31 явно: «use wildcards on input parameters». Редкое исключение — прокидывание чужого wildcard-типа насквозь (`Class<? extends Annotation>` в reflection API).
> 4. **`List<?>` ≠ raw `List`, хотя внешне похожи.** `<?>` сохраняет типобезопасность (запись запрещена), raw выключает проверки целиком и открывает heap pollution — взрыв `ClassCastException` не там, где нагадили.
> 5. **`? super` не даст прочитать как `T`.** `Cat c = sink.get(0)` из `List<? super Cat>` не компилируется — реальный список может быть `List<Object>` с чем угодно внутри. Только `Object o = sink.get(0)`.
> 6. **Wildcard не «наследуется» вглубь:** `List<? extends Animal>` не имеет отношения к `List<List<? extends Animal>>` — вариантность на каждом уровне вложенности включается отдельно. Отсюда монструозные сигнатуры вроде `Map<String, ? extends List<? super Cat>>` — читай их изнутри наружу.

### На собесе

> [!question]- Скомпилируется ли это? Если нет — какая строка и почему?
> 
> java
> 
> ```java
> List<? extends Number> nums = new ArrayList<Integer>();
> nums.add(42);
> ```
> 
> **Model answer (RU):** Первая строка — да: `ArrayList<Integer>` подходит под «какой-то подтип `Number`». Вторая — нет: реальный список мог быть `List<Double>`, компилятор не может доказать безопасность ни для какого аргумента и запрещает `add` целиком — `Integer cannot be converted to CAP#1`.
> 
> **Model answer (EN):** Line one compiles: `ArrayList<Integer>` fits "some subtype of `Number`". Line two doesn't: the actual list could be a `List<Double>`, the compiler can't prove safety for any argument, so `add` is rejected entirely — `Integer cannot be converted to CAP#1`.
> 
> _Follow-up:_ «А `nums.add(null)`?» — компилируется: `null` принадлежит любому ссылочному типу. _Как отвечать:_ не зубри «в extends нельзя добавлять» — проговори рассуждение «реальный список мог быть другим», это ровно то, что хотят услышать.
> 
> _Слабые ответы:_ «Нельзя, потому что wildcard read-only» — заклинание без механизма; на follow-up про `null` рассыпается.

> [!question]- Что такое PECS? _(топ темы)_  
> **Model answer (RU):** Producer Extends, Consumer Super: коллекция-источник в сигнатуре метода — `? extends T`, коллекция-приёмник — `? super T`. Это не правило для заучивания, а следствие асимметрии: из `? extends` безопасно только читать (любой элемент — точно `T`), в `? super` безопасно только писать `T` и подтипы (они влезают в любой возможный реальный список). Канон — `copy(List<? super T> dest, List<? extends T> src)`, почти дословно `Collections.copy`. Если коллекция и читается, и пишется — wildcard не нужен, нужен честный параметр `T`.
> 
> **Model answer (EN):** Producer Extends, Consumer Super: a source collection in a method signature gets `? extends T`, a sink gets `? super T`. It's not a rule to memorize but a consequence of the asymmetry: from `? extends` you can only safely read (every element is guaranteed a `T`), into `? super` you can only safely write `T` and its subtypes (they fit any possible actual list). The canonical example is `copy(List<? super T> dest, List<? extends T> src)` — essentially `Collections.copy`. If a collection is both read and written, no wildcard fits — use a proper type parameter `T`.
> 
> _Follow-up:_ «Приведи PECS из JDK» — `Collection.addAll(Collection<? extends E>)`, `Stream.map(Function<? super T, ? extends R>)`. _Trap:_ «а если и читаю, и пишу?» — честный `T`, wildcards не подойдут.

> [!question]- В чём разница между `List<?>`, `List<Object>` и raw `List`?  
> **Model answer (RU):** `List<Object>` — конкретный тип: из-за инвариантности принимает только `List<Object>`, зато класть можно что угодно. `List<?>` — принимает любой список, но запись запрещена (кроме `null`): типобезопасность сохранена. Raw `List` — принимает любой и разрешает любую запись: проверки дженериков выключены целиком. Внешне `<?>` и raw похожи, идеологически противоположны: первое — «не знаю тип, поэтому осторожен», второе — «игнорирую типы».
> 
> **Model answer (EN):** `List<Object>` is a concrete type: due to invariance it only accepts a `List<Object>`, but you can put anything in. `List<?>` accepts any list but forbids writes (except `null`) — type safety preserved. Raw `List` accepts any list and allows any write: generic checks are switched off entirely. `<?>` and raw look alike but are ideological opposites: "unknown type, so cautious" versus "types ignored".
> 
> _Follow-up:_ «Что такое heap pollution?» — в `List<Integer>` через raw-ссылку попала `String`; компилируется с warning, взрывается `ClassCastException` позже, на чтении — не там, где нагадили. _Trap:_ «где raw ещё неизбежен?» — `instanceof List` и `.class`-литералы (стирание, урок 7.4).

> [!question]- Почему `Collections.sort` объявлен через `Comparable<? super T>`, а не `Comparable<T>`?  
> **Model answer (RU):** Чтобы сортировались подклассы, наследующие `compareTo` от родителя: `Cat extends Animal implements Comparable<Animal>` реализует `Comparable<Animal>`, не `Comparable<Cat>` — строгая граница отвергла бы `List<Cat>` с ошибкой инференса. `Comparable<? super T>` означает «T сравним с собой или предком» — этого достаточно. И это PECS внутри границы: `Comparable` потребляет `T` в `compareTo`, потому `super`.
> 
> **Model answer (EN):** So that subclasses inheriting `compareTo` still sort: `Cat extends Animal implements Comparable<Animal>` implements `Comparable<Animal>`, not `Comparable<Cat>` — the strict bound would reject `List<Cat>` with an inference error. `Comparable<? super T>` means "T is comparable to itself or an ancestor," which is enough. And it's PECS inside a bound: `Comparable` consumes `T` in `compareTo`, hence `super`.
> 
> _Follow-up:_ «Пример из JDK» — `java.sql.Timestamp extends java.util.Date`: реализует унаследованный `Comparable<Date>` (перегрузка `compareTo(Timestamp)` интерфейса не меняет); без `super` список таймстампов не отсортировать. _Follow-up 2:_ «А почему `Cat` не может реализовать ещё и `Comparable<Cat>`?» — после стирания это один интерфейс, реализовать `Comparable` дважды нельзя. _Как отвечать:_ «PECS внутри границы» переводит ответ из «знаю факт» в «понимаю систему».

> [!question]- Метод объявлен `static <T> List<? extends T> process(...)`. Что не так?  
> **Model answer (RU):** Wildcard в возвращаемом типе — антипаттерн: capture-головняк перекладывается на вызывающего, который получит ссылку, в которую ничего не положить, и `capture#1` в сообщениях об ошибках. Wildcards — для входных параметров (Effective Java Item 31); возвращай точный тип `List<T>`.
> 
> **Model answer (EN):** A wildcard in the return type is an antipattern: it pushes the capture headache onto the caller, who gets a reference nothing can be added to, plus `capture#1` in error messages. Wildcards belong on input parameters (Effective Java Item 31); return the precise `List<T>`.
> 
> _Follow-up:_ «Когда wildcard в return оправдан?» — почти никогда; редкое исключение — прокидывание чужого wildcard-типа насквозь, как `Class<? extends Annotation>` в reflection API.

> [!question]- (senior) Что такое wildcard capture и зачем capture-helper?  
> **Model answer (RU):** Capture conversion (JLS §5.1.10) — компилятор при каждом обращении к wildcard-типу подставляет вместо `?` свежую типовую переменную (`CAP#1`). Каждое обращение — новая переменная, поэтому `list.set(i, list.get(j))` на `List<?>` не компилируется: компилятор не знает, что оба `CAP` — один тип. Паттерн capture helper: публичный метод с `List<?>` делегирует в приватный generic `<T> impl(List<T>)` — инференс связывает `T` с захваченным типом, и внутри всё становится одним `T`. Так устроен `Collections.swap`.
> 
> **Model answer (EN):** Capture conversion (JLS §5.1.10): at each use of a wildcard type the compiler substitutes a fresh type variable (`CAP#1`) for `?`. Every use gets a new variable, so `list.set(i, list.get(j))` on a `List<?>` won't compile — the compiler can't tell both `CAP`s are the same type. The capture-helper pattern: a public method taking `List<?>` delegates to a private generic `<T> impl(List<T>)` — inference binds `T` to the captured type, and inside everything is one `T`. That's how `Collections.swap` works.
> 
> _Follow-up:_ «Почему бы просто не объявить публичный метод с `<T>`?» — можно, но `List<?>` в публичном API честнее сообщает «тип мне безразличен» и проще читается; helper — компромисс гибкости снаружи и работы внутри.

> [!question]- Итоговый ответ по теме _(RU/EN)_  
> **RU:** Дженерики в Java инвариантны, поэтому вариантность включается точечно, на месте использования, через wildcards. `? extends T` — ковариантность для чтения: любой элемент гарантированно `T`, запись запрещена (кроме `null`), потому что реальный список может оказаться любым подтипом. `? super T` — контравариантность для записи: `T` и его подтипы влезают в любой возможный список, чтение усечено до `Object`. Отсюда PECS: producer — extends, consumer — super; если коллекция и то и другое — честный параметр `T`. `List<?>` сохраняет типобезопасность в отличие от raw `List`, который выключает проверки и ведёт к heap pollution. _(senior add-on)_ Внутри границ PECS тоже работает — `Comparable<? super T>` в `sort`; а операции над `List<?>` реализуются через capture-helper, дающий безымянному типу имя.
> 
> **EN:** Java generics are invariant, so variance is enabled point-wise, at the use site, via wildcards. `? extends T` is covariance for reading: every element is guaranteed a `T`, writes are forbidden (except `null`) because the actual list may be any subtype. `? super T` is contravariance for writing: `T` and its subtypes fit any possible list, reads are cut down to `Object`. Hence PECS: producer — extends, consumer — super; if a collection is both, use a proper type parameter `T`. `List<?>` preserves type safety, unlike raw `List`, which disables checks and leads to heap pollution. _(senior add-on)_ PECS works inside bounds too — `Comparable<? super T>` in `sort`; and operations on `List<?>` are implemented via a capture helper that gives the anonymous type a name.
> 
> _Слабые ответы:_ «Wildcard делает список read-only» — не read-only, а недоказуемость записи; `? super` пишется отлично. «PECS — это правило Блоха, надо запомнить» — зубрёжка; жди follow-up «выведи». «`List<?>` и raw `List` — одно и то же» — фатально: противоположная идеология.

#### Практика (опционально)

1. **Сломай `copy` всеми способами** (~20 мин, голый JDK). Напиши `copy(List<? super T> dest, List<? extends T> src)`, затем методично попробуй: читать из `dest` как `T`, писать в `src`, поменять wildcards местами, убрать их совсем. Каждую ошибку компиляции прочитай вслух и объясни через «реальный список мог быть...».
2. **Конвейер приюта** (~30 мин, голый JDK). Иерархия `Animal` ← `Cat` ← `Kitten`, `Animal` ← `Dog`. Напиши `double totalWeight(List<? extends Animal>)`, `void rescueKittens(List<? super Kitten> sink)` и связку, прогоняющую котят из `List<Kitten>` в `List<Animal>` и считающую общий вес. Проверь, что `totalWeight` ест все три списка, а `rescueKittens` принимает `List<Kitten>`, `List<Cat>`, `List<Animal>`, `List<Object>` — и только их.
3. **Расшифровка сигнатур JDK** (~20 мин, исходники JDK в IDEA: Ctrl+Click). Найди `Collections.max`, `Comparator.comparing`, `Stream.map`, `CompletableFuture.thenApply`. Выпиши сигнатуры и для каждого `extends`/`super` ответь: кто здесь producer, кто consumer.
4. **Capture-helper руками** (~15 мин). Напиши `reverse(List<?>)` через приватный generic-хелпер. Потом убери хелпер и попробуй написать напрямую — прочитай capture-ошибку и объясни, почему два `get`/`set` порождают разные `CAP`.
5. **Мост в C#** (~25 мин, если есть .NET под рукой). Перепиши `copy` на C#: сначала через `List<T>` (упрёшься в инвариантность), потом через `IEnumerable<out T>` для источника. Прочувствуй разницу: в C# ты выбираешь _интерфейс с нужной вариантностью_, в Java — _поворачиваешь тип на месте_.
