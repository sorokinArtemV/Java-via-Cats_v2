Промежуточные операции — это словарь конвейера: то, из чего собирается вся середина между источником и терминалом. Их немного, и почти все имеют прямых близнецов в LINQ, так что главная работа этого урока — не выучить список, а откалибровать интуицию там, где Java ведёт себя иначе: `flatMap` как ручной `SelectMany`, `distinct`, молча зависящий от твоего `equals`/`hashCode`, `takeWhile` с его префиксной семантикой и `peek`, который имеет право не выполниться.

> [!abstract] Ментальная модель  
> Промежуточная операция принимает стрим и возвращает новый стрим, ничего не вычисляя, — она лишь достраивает описание конвейера. Все они ленивы; различаются тем, нужна ли им память о других элементах: stateless (`filter`, `map`) обрабатывают элемент в изоляции, stateful (`sorted`, `distinct`) вынуждены видеть больше одного элемента.

### filter и map — рабочие лошадки

```java
record Cat(String name, double weight, List<String> kittens) {}

List<Cat> cats = List.of(
    new Cat("Мурка", 4.2, List.of("Пушок", "Снежок")),
    new Cat("Ласка", 3.8, List.of("Рыжик")),
    new Cat("Багира", 5.1, List.of())
);

cats.stream()
    .filter(c -> c.weight() > 4)   // Stream<Cat> → Stream<Cat>, элементов меньше или столько же
    .map(Cat::name)                // Stream<Cat> → Stream<String>, элементов ровно столько же
    .toList();                     // [Мурка, Багира]
```

`filter` — сито: пропускает элементы, для которых `Predicate` вернул `true`. `map` — преобразователь один-к-одному: каждый элемент заменяется результатом `Function`. Тип стрима после `map` меняется, количество элементов — никогда.

> [!tip] 💡 Из C#  
> `filter` = `Where`, `map` = `Select`, дальше по уроку: `flatMap` = `SelectMany`, `distinct` = `Distinct`, `sorted` = `OrderBy`, `limit` = `Take`, `skip` = `Skip`, `takeWhile`/`dropWhile` = `TakeWhile`/`SkipWhile`. Имена в Java пришли из функционального программирования (map/filter/reduce), а не из SQL-традиции LINQ. Семантика совпадает почти везде — расхождения отмечу по ходу.

### flatMap — разворачивание вложенности

Задача: собрать всех котят приюта в один список. `map(Cat::kittens)` даёт `Stream<List<String>>` — стрим _списков_, а не котят. Нужна операция, которая каждый элемент превращает в стрим и склеивает эти стримы в один:

```java
cats.stream().map(Cat::kittens).toList();
// [[Пушок, Снежок], [Рыжик], []]        — вложенность осталась

cats.stream().flatMap(c -> c.kittens().stream()).toList();
// [Пушок, Снежок, Рыжик]                — плоский стрим котят
```

![[Pasted image 20260716151252.png]]

Три свойства `flatMap`, которые стоит проговорить:

1. **Количество элементов произвольно меняется**: один котёнок у Ласки → один элемент, пустой список у Багиры → ноль элементов. Это делает `flatMap` универсальнее, чем кажется: им можно и фильтровать (вернуть `Stream.empty()`), и размножать.
2. **Маппер обязан вернуть стрим, не `null`.** `null` из маппера — `NullPointerException` при исполнении. Если данные грязные и список котят бывает `null`, здесь раскрывается `Stream.ofNullable` из урока 1 — проверено на JDK 21:

```java
// cat.kittens() может быть null
withNull.stream()
    .flatMap(cat -> Stream.ofNullable(cat.kittens()).flatMap(List::stream))
    .toList(); // null-список тихо дал ноль элементов
```

3. **Каждый внутренний стрим закрывается** после вливания в общий поток — важно, если внутри `flatMap` открываются стримы с ресурсами (`Files.lines`).

> [!tip] 💡 Из C#  
> Это `SelectMany` один в один, вплоть до семантики пустых подпоследовательностей. Разница косметическая, но регулярно спотыкает: в C# маппер возвращает `IEnumerable<T>`, и коллекция подходит сама по себе — `SelectMany(c => c.Kittens)`. В Java маппер обязан вернуть именно `Stream`, поэтому `.flatMap(c -> c.kittens().stream())` — этот лишний `.stream()` забывают чаще всего, благо компилятор поймает.

### distinct — дедупликация на твоём `equals`

`distinct` убирает дубликаты, сравнивая элементы через `equals` (внутри — накопление в `HashSet`-подобную структуру, так что и `hashCode` в деле). Порядок сохраняется, остаётся **первое** вхождение: `Stream.of(3,1,3,2,1).distinct().toList()` → `[3, 1, 2]` (прогон JDK 21).

Тихая зависимость от контракта из главы 3.8 — место, где он ломается на практике:

```java
record CatR(String name, double weight) {}          // equals/hashCode сгенерированы
class CatC { final String name; CatC(String n) { name = n; } }  // унаследованы от Object

Stream.of(new CatR("Мурка", 4.2), new CatR("Мурка", 4.2)).distinct().count(); // 1
Stream.of(new CatC("Мурка"), new CatC("Мурка")).distinct().count();           // 2 (!)
```

Класс без переопределённого `equals` сравнивается по идентичности — `distinct` «не работает», хотя работает ровно по контракту.

### sorted — единственная операция, которой нужно всё

```java
cats.stream().sorted(Comparator.comparingDouble(Cat::weight)).toList();
cats.stream().sorted().toList(); // ClassCastException: Cat не Comparable!
```

Безаргументный `sorted()` требует, чтобы элементы реализовали `Comparable`, — иначе `ClassCastException` **в рантайме при терминальной операции** (проверено: `cannot be cast to class java.lang.Comparable`). Дженерики тут не спасают из-за erasure (глава 7.4): сигнатура `sorted()` не может потребовать `T extends Comparable`.

`sorted` — предельно stateful операция: чтобы выдать первый элемент, нужно увидеть все. Внутри она буферизует весь стрим в массив, сортирует и лишь потом отдаёт элементы дальше. Следствия: на бесконечном стриме зависнет; ленивость конвейера в этой точке «запруживается». Сортировка **стабильная** для упорядоченных стримов — равные элементы сохраняют исходный порядок (проверено на JDK 21), что позволяет каскадировать: отсортировал по кличке, затем по весу — внутри одного веса клички останутся упорядоченными. Впрочем, идиоматичнее `Comparator.comparing(...).thenComparing(...)` — глава 7 это уже разобрала.

> [!tip] 💡 Из C#  
> `OrderBy` тоже стабилен и тоже буферизует. Но заметь: в LINQ каскад делается через отдельный оператор `ThenBy` над `IOrderedEnumerable`, а в Java — комбинацией компараторов до вызова `sorted`. Двойной `sorted().sorted()` в Java — не каскад, а две независимые полные сортировки, вторая просто перезатрёт первую.

### limit и skip — срез конвейера

```java
IntStream.rangeClosed(1, 10).boxed().skip(2).limit(3).toList(); // [3, 4, 5]
IntStream.rangeClosed(1, 10).boxed().limit(3).skip(2).toList(); // [3]
```

Порядок имеет значение, потому что каждая операция применяется к _выходу предыдущей_: `skip(2).limit(3)` — «пропусти два, возьми три следующих», `limit(3).skip(2)` — «возьми три первых, из них пропусти два». `limit` — short-circuiting операция: получив нужное число элементов, она перестаёт запрашивать источник — единственный штатный способ усечь бесконечный стрим (кроме `takeWhile`).

### takeWhile и dropWhile (Java 9)

Пара, которую регулярно путают с `filter`. Проверенный прогон:

```java
List<Integer> nums = List.of(1, 2, 3, 10, 2);

nums.stream().takeWhile(n -> n < 5).toList(); // [1, 2, 3]
nums.stream().dropWhile(n -> n < 5).toList(); // [10, 2]
nums.stream().filter(n -> n < 5).toList();    // [1, 2, 3, 2]
```

`filter` проверяет **каждый** элемент. `takeWhile` берёт **префикс**: элементы до первого провала предиката, после чего обрубает стрим — финальная двойка не имеет шансов, хотя предикату удовлетворяет. `dropWhile` — зеркало: выбрасывает префикс, с первого провала отдаёт всё подряд без проверок (двойка в хвосте прошла). На отсортированных данных `takeWhile` — ещё и оптимизация: не сканировать хвост, который заведомо не подойдёт. `takeWhile` short-circuiting и останавливает бесконечные стримы: `Stream.iterate(1, n -> n * 3).takeWhile(n -> n < 100).toList()` → `[1, 3, 9, 27, 81]`.

> [!tip] 💡 Из C#  
> Семантика `TakeWhile`/`SkipWhile` идентична. Единственное расхождение — имена (`drop` vs `Skip`) и то, что на **неупорядоченном** параллельном стриме в Java поведение `takeWhile` становится недетерминированным — «префикс» без порядка не определён. В LINQ этот вопрос не встаёт: PLINQ с `TakeWhile` принудительно упорядочен. Деталь для урока 6.

### peek — отладочный глазок с подвохом

`peek(Consumer)` пропускает элемент через себя, выполняя побочное действие, — задумано для отладки: `​.peek(System.out::println)` между стадиями. Подвох в том, что `peek` — часть ленивого конвейера, и **выполняется, только если элемент реально течёт через его стадию**. Прогон на JDK 21:

```java
cats.stream().peek(c -> log.add(c.name())).count();
// log ПУСТ, count == 3
```

`count()` увидел, что размер источника известен (`SIZED`-характеристика) и ни одна стадия его не меняет, — и вычислил ответ **вообще без обхода элементов**. `peek` не вызвался ни разу. Добавь `filter` — и обход станет обязательным, `peek` отработает для всех элементов. Javadoc `peek` прямо предупреждает об этом; вывод: `peek` — только для диагностики, любая бизнес-логика в нём (мутация объектов, запись в базу — встречается в живом коде!) может молча не выполниться. Развёрнуто про эту оптимизацию — в уроке 3.

### Stateless vs stateful — карта операций

|Операция|Состояние|Заметки|
|---|---|---|
|`filter`, `map`, `flatMap`, `peek`, `mapMulti`|stateless|элемент обрабатывается в изоляции|
|`takeWhile`, `dropWhile`, `limit`, `skip`|stateful (лёгкие)|помнят счётчик/флаг, short-circuiting у `takeWhile`/`limit`|
|`distinct`|stateful|копит множество увиденных|
|`sorted`|stateful (полный буфер)|не отдаст ничего, пока не увидит всё|

Различие выстрелит дважды: в уроке 3 (stateful-стадии ломают «вертикальное» поэлементное исполнение) и в уроке 6 (stateful — главный тормоз параллельных стримов).

> [!example] 🔬 Глубже / на синьора  
> **Как стадии живут внутри.** Каждая промежуточная операция создаёт объект `ReferencePipeline$StatelessOp` или `StatefulOp`, а при запуске терминала конвейер «схлопывается» в цепочку `Sink`-ов — по сути, вложенных консюмеров: `Sink` фильтра решает, звать ли `Sink` мапа, и так далее. Stateless-стадии сливаются в один проход (operation fusion) — промежуточных коллекций не существует физически. Stateful-стадия (`sorted`, `distinct` в упорядоченном режиме) — барьер: она вынуждена накопить входные элементы, прежде чем что-то отдать следующему `Sink`.
> 
> **`mapMulti` (Java 16)** — низкоуровневая альтернатива `flatMap` для горячих путей: вместо создания стрима на каждый элемент маппер получает `Consumer` и толкает в него ноль или больше значений:
> 
> java
> 
> ```java
> cats.stream()
>     .<String>mapMulti((cat, sink) -> cat.kittens().forEach(sink))
>     .toList(); // [Пушок, Снежок, Рыжик] — прогон JDK 21
> ```
> 
> Выигрыш — когда элементов-контейнеров много, а внутри каждого 0–1 значение: не создаётся объект стрима на элемент. Явный type witness `<String>` обычно обязателен — инференсу не из чего вывести тип выхода. Для читаемости по умолчанию — `flatMap`; `mapMulti` — точечный инструмент.
> 
> **`distinct` и упорядоченность.** В упорядоченном стриме `distinct` обязан оставить первое вхождение — в параллельном исполнении это требует дорогой координации. Если порядок не важен, `stream.unordered().distinct()` разрешает библиотеке дешёвый вариант. Это первый пример общего правила «ORDERED — это контракт, за который платишь», кульминация — в уроке 6.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Бизнес-логика в `peek`.** Может не выполниться вовсе (доказано прогоном: `count()` пропустил стадию целиком) — или, наоборот, выполниться для элементов, которые потом отбросит `limit`. Только диагностика.
> 2. **`null` из маппера `flatMap`** — NPE в момент исполнения, а не создания конвейера, стектрейс укажет вглубь `ReferencePipeline`, не на твою строку. Грязные данные → `Stream.ofNullable` или явная проверка.
> 3. **`sorted()` без компаратора на не-`Comparable`** — компилируется, падает в рантайме с `ClassCastException` при терминальной операции. Erasure не даёт выразить требование в сигнатуре.
> 4. **`takeWhile` вместо `filter` на неотсортированных данных** — молча теряет элементы после первого провала предиката (`[1,2,3,10,2]` → потеряна финальная `2`). Обратное тоже больно: `filter` вместо `takeWhile` на бесконечном стриме не остановит его.
> 5. **`map` с побочными эффектами** («заодно и сохраним в базу») — та же мина, что и `peek`: библиотека вправе не вызывать маппер для элементов, не влияющих на результат, переупорядочить вызовы в parallel, и т.д. Контракт `map` — чистая функция; побочные эффекты живут в терминальных операциях. Подробный разбор — в уроке 6 (10.10).
> 6. **`distinct` на классе без `equals`/`hashCode`** — дедупликация по идентичности ссылок: «одинаковые» объекты остаются. Симптом ошибки не в стриме, а в нарушенном контракте главы 3.8.

### На собесе

> [!question]- В чём разница между `map` и `flatMap`? / What's the difference between `map` and `flatMap`?  
> **RU:** `map` — преобразование один-к-одному: элемент заменяется результатом функции, размер стрима не меняется. `flatMap` — один-ко-многим: функция возвращает стрим для каждого элемента, и все эти стримы склеиваются в один плоский. `map(Cat::kittens)` даёт стрим списков; `flatMap(c -> c.kittens().stream())` — стрим котят. Пустой внутренний стрим даёт ноль элементов, поэтому `flatMap` умеет и фильтровать.  
> **EN:** `map` is a one-to-one transformation: each element is replaced by the function's result, the stream size never changes. `flatMap` is one-to-many: the function returns a stream per element, and all those streams are concatenated into one flat stream. `map(Cat::kittens)` yields a stream of lists; `flatMap(c -> c.kittens().stream())` yields a stream of kittens. An empty inner stream contributes zero elements, so `flatMap` can also filter.  
> **Follow-up:** «Что будет, если маппер вернёт `null`?» → NPE при исполнении конвейера; защита — `Stream.ofNullable` или `Stream.empty()`.

> [!question]- Чем `takeWhile` отличается от `filter`? / How does `takeWhile` differ from `filter`?  
> **RU:** `filter` проверяет каждый элемент стрима. `takeWhile` берёт только префикс — элементы до первого, не прошедшего предикат, — и обрубает стрим, дальнейшие элементы не проверяются, даже подходящие. На `[1,2,3,10,2]` с предикатом `< 5`: `filter` → `[1,2,3,2]`, `takeWhile` → `[1,2,3]`. `takeWhile` — short-circuiting и потому останавливает бесконечные стримы, `filter` — нет.  
> **EN:** `filter` tests every element. `takeWhile` takes only the prefix — elements up to the first predicate failure — then cuts the stream off; later elements aren't even tested. On `[1,2,3,10,2]` with `< 5`: `filter` → `[1,2,3,2]`, `takeWhile` → `[1,2,3]`. `takeWhile` is short-circuiting and can stop an infinite stream; `filter` can't.  
> **Follow-up:** «Когда `takeWhile` предпочтительнее по производительности?» → на отсортированных данных: не сканирует заведомо неподходящий хвост.

> [!question]- Почему `peek` может не выполниться? / Why might `peek` never execute?  
> **RU:** `peek` — ленивая стадия конвейера и вызывается только для элементов, которые реально через неё текут. Если терминальная операция может вычислить результат без обхода — например, `count()` на стриме с известным размером, где ни одна стадия размер не меняет, — обхода не будет и `peek` не вызовется ни разу. Поэтому `peek` годится только для отладки; логика с побочными эффектами в нём ненадёжна по контракту.  
> **EN:** `peek` is a lazy pipeline stage, invoked only for elements that actually flow through it. If the terminal operation can compute its result without traversal — e.g. `count()` on a stream whose size is known and unchanged by any stage — no traversal happens and `peek` never fires. So `peek` is for debugging only; side-effecting logic in it is unreliable by contract.  
> **Ловушка слабого ответа:** «peek выполняется всегда, просто возвращает тот же стрим» — опровергается одной строкой: `list.stream().peek(...).count()`.

> [!question]- Какие промежуточные операции stateful и почему это важно? / Which intermediate operations are stateful and why does it matter?  
> **RU:** Stateless (`filter`, `map`, `flatMap`) обрабатывают элемент, не зная о других, — их можно слить в один проход и дёшево параллелить. Stateful помнят прошлое: `distinct` копит множество увиденных, `skip`/`limit` — счётчик, а `sorted` — крайний случай: буферизует весь стрим, прежде чем отдать первый элемент. Важно потому, что stateful-стадии ломают поэлементное исполнение (на `sorted` конвейер «запруживается»), зависают на бесконечных стримах и дорого стоят в параллельном режиме.  
> **EN:** Stateless ops (`filter`, `map`, `flatMap`) process each element in isolation — they fuse into a single pass and parallelize cheaply. Stateful ops remember the past: `distinct` accumulates seen elements, `skip`/`limit` keep a counter, and `sorted` is the extreme case — it buffers the entire stream before emitting anything. It matters because stateful stages break element-by-element execution (the pipeline "dams up" at `sorted`), hang on infinite streams, and are the main cost driver in parallel streams.

> [!question]- На что опирается `distinct` при сравнении элементов? / What does `distinct` rely on to compare elements?  
> **RU:** На `equals` и `hashCode` элементов. Класс без их переопределения сравнивается по идентичности ссылок — два «одинаковых» объекта дубликатами не считаются. Порядок сохраняется, остаётся первое вхождение. Это не баг стрима, а прямое следствие контракта `equals`/`hashCode`: `distinct` — одно из мест, где его нарушение проявляется молча.  
> **EN:** On the elements' `equals` and `hashCode`. A class that doesn't override them compares by reference identity — two "equal-looking" objects won't be treated as duplicates. Encounter order is preserved, the first occurrence wins. That's not a stream bug but a direct consequence of the `equals`/`hashCode` contract: `distinct` is one of the places where violating it fails silently.

> List<Integer> r = IntStream.rangeClosed(1, 10).boxed().toList();
> System.out.println(r.stream().skip(2).limit(3).toList());
> System.out.println(r.stream().limit(3).skip(2).toList());

> **RU:** `[3, 4, 5]` и `[3]`. Операции применяются к выходу предыдущей стадии: первая цепочка — «пропусти 2, возьми 3 следующих»; вторая — «возьми первые 3 (это `[1,2,3]`), из них пропусти 2». Слабый ответ — «одинаково, порядок limit/skip не важен».

> [!question]- Можно ли вызвать `sorted()` без аргументов на любом стриме? / Can you call no-arg `sorted()` on any stream?  
> **RU:** Скомпилируется на любом, но в рантайме элементы должны реализовывать `Comparable` — иначе `ClassCastException` в момент терминальной операции. Из-за type erasure сигнатура `sorted()` не может потребовать `T extends Comparable<T>` — проверка уезжает в рантайм. Для произвольных типов — `sorted(Comparator)`.  
> **EN:** It compiles on any stream, but at runtime the elements must implement `Comparable` — otherwise you get a `ClassCastException` when the terminal operation runs. Due to type erasure the `sorted()` signature can't require `T extends Comparable<T>`, so the check moves to runtime. For arbitrary types use `sorted(Comparator)`.  
> **Senior add-on:** сортировка стабильна для упорядоченных стримов; каскад делается композицией `Comparator.comparing(...).thenComparing(...)`, а не двумя `sorted`.

> [!question]- Java: аналоги LINQ-операторов и главное расхождение в `SelectMany`? / Java equivalents of LINQ operators and the key `SelectMany` difference?  
> **RU:** `Where`→`filter`, `Select`→`map`, `SelectMany`→`flatMap`, `Distinct`→`distinct`, `OrderBy`→`sorted`, `Take`/`Skip`→`limit`/`skip`, `TakeWhile`/`SkipWhile`→`takeWhile`/`dropWhile`. Расхождение в `flatMap`: LINQ принимает `IEnumerable` (коллекция подходит напрямую), Java требует именно `Stream` — отсюда обязательный `.stream()` внутри маппера. Плюс в Java есть примитивные варианты `flatMapToInt` и низкоуровневый `mapMulti` — у LINQ таких нет из-за reified generics.  
> **EN:** `Where`→`filter`, `Select`→`map`, `SelectMany`→`flatMap`, `Distinct`→`distinct`, `OrderBy`→`sorted`, `Take`/`Skip`→`limit`/`skip`, `TakeWhile`/`SkipWhile`→`takeWhile`/`dropWhile`. The `flatMap` divergence: LINQ accepts an `IEnumerable` (a collection works directly), Java requires a `Stream` — hence the mandatory `.stream()` inside the mapper. Java also has primitive variants (`flatMapToInt`) and the low-level `mapMulti`, which LINQ doesn't need thanks to reified generics.
