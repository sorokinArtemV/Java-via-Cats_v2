До сих пор конвейер был декорацией: мы строили цепочки, но ничего не происходило. Терминальная операция — это спусковой крючок, единственное место, где стрим начинает работать, и одновременно ответ на вопрос «что мне в итоге вернуть»: коллекцию, число, булев факт, один элемент. А заодно пришло время честно разобрать ленивость — понять, в каком порядке _на самом деле_ выполняются стадии, почему `count()` в прошлом уроке проигнорировал `peek` и как конвейер умеет останавливаться, не досмотрев данные.

> [!abstract] Ментальная модель  
> Терминальная операция запускает конвейер и потребляет стрим, производя результат или побочный эффект. Исполнение — **вертикальное**: каждый элемент по одному проходит все стадии до конца, промежуточных коллекций не существует, а short-circuiting операции останавливают запрос элементов, как только ответ известен.

### Ленивость: как конвейер выполняется на самом деле

Главный миф о стримах — «каждая операция обходит всю коллекцию и складывает результат в промежуточный список, как LINQ-цепочка с `.ToList()` после каждого шага». Проверим прогоном (JDK 21):

```java
Stream.of("Барсик", "Мурка", "Пушок")
    .filter(n -> { System.out.println("filter " + n); return n.length() > 5; })
    .map(n ->    { System.out.println("map " + n);    return n.toUpperCase(); })
    .findFirst();
```

```
filter Барсик
map Барсик
```

И всё. Не «filter трижды, потом map» — «Барсик» прошёл фильтр (длина 6 > 5), тут же прошёл `map`, `findFirst` получил элемент и объявил конвейер завершённым. «Мурка» и «Пушок» **не были даже запрошены у источника**. Без short-circuit картина та же, просто до конца:

```java
Stream.of("a", "bb").filter(...).map(...).forEach(...);
// filter a → map a → forEach a → filter bb → map bb → forEach bb
```

Элемент за элементом, каждый — насквозь. Отсюда два следствия:

Следствия вертикальной модели:

1. **Промежуточных коллекций нет** — память не тратится на temp-списки, сколько бы стадий ни было в цепочке.
2. **Short-circuiting работает по-настоящему**: операции, способные завершиться досрочно (`findFirst`, `findAny`, `anyMatch`, `allMatch`, `noneMatch`, `limit`, `takeWhile`), перестают _запрашивать элементы у источника_ — поэтому они и укрощают бесконечные стримы.
3. **Stateful-стадия — плотина**: `sorted` не может отдать первый элемент, не увидев все, поэтому до неё конвейер вынужден прокачать источник целиком, и лишь после неё возобновляется поэлементное течение. `filter.sorted().findFirst()` профильтрует всё, несмотря на `findFirst`.

Теперь разгадка `count()` из урока 2: если источник знает свой размер (характеристика `SIZED` у спплитератора `ArrayList`) и ни одна стадия между ним и `count()` не может изменить количество элементов (`peek`, `map` — не могут; `filter`, `flatMap` — могут), терминал берёт готовый размер и не запускает обход вовсе. Ленивость работает в обе стороны: не только «не раньше, чем нужно», но и «вообще никогда, если ответ известен без работы».

### forEach — самый знакомый и самый переоценённый

```java
cats.stream().filter(Cat::adult).forEach(c -> System.out.println(c.name()));
```

`forEach(Consumer)` выполняет действие для каждого элемента. Две оговорки. Первая: для _простого обхода коллекции_ стрим не нужен — есть `shelter.forEach(...)` прямо у `Iterable`, без накладных расходов на конвейер. Стримовый `forEach` оправдан в конце цепочки преобразований. Вторая: его контракт **явно не гарантирует порядок** — на последовательном стриме элементы фактически идут в encounter order, но спецификация оставляет это на волю реализации, а на параллельном порядок разваливается гарантированно. Для строгого порядка существует `forEachOrdered` — детали в уроке 6.

### reduce — свёртка, которую спрашивают глубже, чем кажется

`reduce` сворачивает стрим в одно значение, многократно применяя бинарную операцию. Три перегрузки, и различия между ними — готовый вопрос собеседования.

**Форма 1: только аккумулятор** — результат `Optional`, потому что пустому стриму нечего вернуть:

```java
List<Integer> nums = List.of(1, 2, 3, 4);
nums.stream().reduce(Integer::sum);           // Optional[10]
Stream.<Integer>empty().reduce(Integer::sum); // Optional.empty
```

**Форма 2: identity + аккумулятор** — есть стартовое значение, результат «голый»:

```java
nums.stream().reduce(0, Integer::sum); // 10; на пустом стриме — 0
```

Слово _identity_ — не синоним «начального значения», это математическое требование: `identity op x == x` для любого `x`. Ноль для сложения, единица для умножения, пустая строка для конкатенации. Нарушение — бомба замедленного действия, прогон на JDK 21:


```java
nums.stream().reduce(10, Integer::sum);         // 20 — «работает»
nums.parallelStream().reduce(10, Integer::sum); // 50 (!)
```

Последовательно `10+1+2+3+4 = 20`, и код проходит ревью. Параллельно стрим режется на куски, **каждый** кусок стартует со своего identity, результаты комбинируются — десятка прибавилась столько раз, сколько было кусков. Контракт `reduce` требует настоящий identity именно затем, чтобы результат не зависел от разбиения. «Хочу стартовать с 10» выражается честно: `10 + stream.reduce(0, Integer::sum)`.

**Форма 3: identity + аккумулятор + комбайнер** — когда тип результата отличается от типа элементов:

```java
double totalWeight = cats.stream()
    .reduce(0.0, (acc, cat) -> acc + cat.weight(), Double::sum);
```

Аккумулятор `(U, T) → U` вливает элемент в результат, комбайнер `(U, U) → U` сливает результаты параллельных кусков. В последовательном исполнении **комбайнер не вызывается вообще** (проверено прогоном: лог пуст) — но передать его обязан ты, и он обязан быть согласован с аккумулятором. Это худшее место для «да сюда никогда не зайдёт»: код будет годами верен, пока кто-то не добавит `.parallel()`. Впрочем, для «свернуть в контейнер» (список, мапу, StringBuilder) правильный инструмент не `reduce` с его требованием немутирующих функций, а `collect` — mutable reduction, урок 4.

> [!tip] 💡 Из C#  
> `reduce` = `Aggregate`, вплоть до трёх перегрузок. Но LINQ-версия не обязана быть parallel-безопасной — обычный `Aggregate` последовательный, и identity там честно называется `seed`: стартовое значение без математических требований. В Java у **того же самого** `reduce` контракт жёстче, потому что один и тот же конвейер обязан давать одинаковый ответ последовательно и параллельно. Привычка из C# «seed — это просто старт» — источник бага с десяткой выше.

### count, min, max

```java
cats.stream().filter(Cat::adult).count();                          // long
cats.stream().max(Comparator.comparingDouble(Cat::weight));       // Optional<Cat>
cats.stream().min(Comparator.comparingDouble(Cat::weight));       // Optional<Cat>
```

`min`/`max` всегда требуют компаратор и возвращают `Optional` — пустой стрим не имеет экстремумов. `count` возвращает `long` (не `int` — источники бывают больше двух миллиардов) и, как мы видели, умеет не обходить стрим.

### Матчеры: anyMatch, allMatch, noneMatch

```java
cats.stream().anyMatch(c -> c.weight() > 7);   // есть ли хоть один тяжелее 7?
cats.stream().allMatch(Cat::adult);            // все ли взрослые?
cats.stream().noneMatch(c -> c.weight() < 1);  // никто не легче килограмма?
```

Все трое short-circuiting: `anyMatch` останавливается на первом совпадении (прогон: на `[1..5]` с предикатом `n == 2` посещены только `1, 2`), `allMatch`/`noneMatch` — на первом контрпримере. Коварство — **пустой стрим** (прогон JDK 21):

```java
Stream.empty().allMatch(x -> false);  // true (!)
Stream.empty().anyMatch(x -> true);   // false
Stream.empty().noneMatch(x -> true);  // true
```

`allMatch` на пустом — `true` по законам логики (vacuous truth: «все элементы пустого множества удовлетворяют чему угодно»), и это регулярно ломает бизнес-проверки вида «все котики привиты → выпускаем» — пустой список котиков радостно проходит валидацию.

> [!tip] 💡 Из C#  
> `Any(pred)` / `All(pred)` ведут себя идентично, включая vacuous truth у `All` на пустой последовательности, — тут интуиция переносится без потерь. `noneMatch` отдельного аналога не имеет (`!Any(pred)`).

### findFirst и findAny

```java
cats.stream().filter(Cat::adult).findFirst(); // Optional<Cat> — первый по encounter order
cats.stream().filter(Cat::adult).findAny();   // Optional<Cat> — любой подходящий
```

На последовательном стриме оба фактически вернут один и тот же первый элемент. Разница — в контракте: `findFirst` **обязан** вернуть первый в encounter order, `findAny` — любой. На параллельном стриме это уже не синонимы, прогон на диапазоне 1..100000 с фильтром «кратно 10»:

```
parallel findFirst: 10
parallel findAny:   50010
```

`findAny` вернул то, что первым нашёл один из потоков, — и в этом его смысл: не заставлять параллельный конвейер координироваться ради порядка. Правило выбора: важен именно первый — `findFirst`; нужен «хоть какой-нибудь» — `findAny`, как явная декларация «порядок мне безразличен», развязывающая руки оптимизатору.

### toList, toArray, collect — материализация

```java
List<String> names = cats.stream().map(Cat::name).toList();          // Java 16
Cat[] arr = cats.stream().toArray(Cat[]::new);
List<String> old = cats.stream().map(Cat::name).collect(Collectors.toList());
```

`toList()` — самый частый терминал в современном коде; чем он отличается от `collect(Collectors.toList())` и почему это любимый вопрос про мутабельность — начало урока 4, где `collect` разобран как механизм.

> [!example] 🔬 Глубже / на синьора  
> **Как терминал запускает конвейер.** Терминальная операция строит из стадий цепочку `Sink`-ов (внутренний интерфейс — `Consumer` с жизненным циклом `begin/accept/end` и каналом отмены `cancellationRequested`). Затем спплитератор источника прокачивается через голову цепочки: `spliterator.tryAdvance(sink)` в цикле для short-circuit сценариев или разовый `forEachRemaining(sink)` для полного обхода. «Вертикальность» — это буквально вложенные вызовы `accept`: `filterSink.accept(барсик)` внутри себя зовёт `mapSink.accept(барсик)`, тот — `findSink.accept(БАРСИК)`.
> 
> **Механика short-circuit.** После каждого `tryAdvance` конвейер опрашивает `cancellationRequested()`: `FindOp` взводит флаг, получив первый элемент, `MatchOp` — первый решающий, `limit` — исчерпав квоту. Флаг распространяется вверх по цепочке синков, и цикл прокачки останавливается — источник больше не трогают.
> 
> **Элизия обхода у `count`/`toArray`.** `count()` реализован как `mapToLong(e -> 1L).sum()` c оптимизацией: если `StreamOpFlag.SIZED` не сброшен ни одной стадией, ответ — `spliterator.getExactSizeIfKnown()`, ноль вызовов пользовательского кода. `filter` и `flatMap` сбрасывают `SIZED`; `map`, `peek`, `sorted` — нет. Отсюда и точный критерий, когда `peek` «исчезает».
> 
> **Почему функции `reduce` обязаны быть ассоциативными и без побочных эффектов.** Параллельный `reduce` вычисляет `combiner(reduce(левая половина), reduce(правая половина))` рекурсивно — результат совпадает с последовательным только если операция ассоциативна, identity нейтрален, а функции не зависят от состояния. Это не «рекомендация», а условие корректности, которое JVM не проверяет: нарушил — получил тихо неверный ответ, как с десяткой выше.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Не-identity в `reduce(identity, op)`.** Последовательно — «работает», параллельно — тихо неверный результат (`20` vs `50` из прогона). Ревью-правило: увидел `reduce` с ненулевым/неединичным стартом — проверь нейтральность.
> 2. **`allMatch` на пустом стриме — `true`.** «Все котики привиты» для нуля котиков — истина. Бизнес-валидации через `allMatch` обязаны отдельно решить, что значит пустой ввод.
> 3. **`get()` на `Optional` от `findFirst`/`min`/`max`** без проверки — `NoSuchElementException` на пустом стриме. Идиоматика: `orElse`, `orElseThrow` с осмысленным исключением, `ifPresent` — детально в уроке 5.
> 4. **Побочные эффекты в предикатах матчеров** — short-circuit посетит неопределённое подмножество элементов: `anyMatch` из прогона тронул только `[1, 2]` из пяти. Считать «сколько проверили» через счётчик в предикате — незаконно.
> 5. **`findFirst` на параллельном стриме там, где хватило бы `findAny`** — оплата координации encounter order без нужды. И зеркально: `findAny` в последовательном коде «работает как findFirst» ровно до момента, когда кто-то добавит `.parallel()`.
> 6. **Ожидание, что `sorted().findFirst()` ленив** — нет: stateful-плотина заставит прокачать весь источник. Для «минимального элемента» — `min(comparator)`: тот же ответ за один проход без буферизации.

### На собесе

> [!question]- Что такое ленивость стрима? Что и когда выполняется? / What is stream laziness? What executes and when?  
> **RU:** Промежуточные операции ничего не вычисляют — они строят описание конвейера. Исполнение запускает только терминальная операция, и идёт оно поэлементно-вертикально: элемент проходит все стадии насквозь, следующий запрашивается у источника, только если результат ещё не готов. Промежуточных коллекций нет; short-circuiting операции прекращают запрос элементов досрочно.  
> **EN:** Intermediate operations compute nothing — they build a pipeline description. Only a terminal operation triggers execution, and it proceeds element-by-element, vertically: an element flows through all stages, and the next one is pulled from the source only if the result isn't ready yet. There are no intermediate collections; short-circuiting operations stop pulling elements early.  
> **Follow-up:** «Докажите кодом» → логирующие `filter`/`map` + `findFirst`: вывод `filter Барсик, map Барсик` и тишина по остальным элементам.

> [!question]- Какие операции short-circuiting и что это значит? / Which operations are short-circuiting and what does that mean?  
> **RU:** Терминальные: `findFirst`, `findAny`, `anyMatch`, `allMatch`, `noneMatch`; промежуточные: `limit`, `takeWhile`. Short-circuiting означает способность завершить конвейер (или сделать его конечным), не обработав все элементы: `anyMatch` останавливается на первом совпадении, `limit` — исчерпав квоту. Именно эти операции делают работоспособными бесконечные стримы.  
> **EN:** Terminal: `findFirst`, `findAny`, `anyMatch`, `allMatch`, `noneMatch`; intermediate: `limit`, `takeWhile`. Short-circuiting means finishing the pipeline (or making it finite) without processing all elements: `anyMatch` stops at the first match, `limit` after its quota. These are exactly the operations that make infinite streams workable.

> [!question]- Расскажите про три формы `reduce`. Зачем комбайнер? / Explain the three `reduce` overloads. Why the combiner?  
> **RU:** (1) `reduce(op)` → `Optional` — пустому стриму нечего вернуть; (2) `reduce(identity, op)` → значение, на пустом стриме — identity; (3) `reduce(identity, accumulator, combiner)` — когда тип результата отличается от типа элементов: аккумулятор вливает элемент в результат, комбайнер сливает частичные результаты параллельных кусков. В последовательном исполнении комбайнер не вызывается, но обязан быть корректным — конвейер должен давать тот же ответ при `.parallel()`.  
> **EN:** (1) `reduce(op)` → `Optional` — an empty stream has nothing to return; (2) `reduce(identity, op)` → a bare value, identity for an empty stream; (3) `reduce(identity, accumulator, combiner)` — for a result type different from the element type: the accumulator folds an element in, the combiner merges partial results from parallel chunks. Sequentially the combiner is never invoked, but it must still be correct — the pipeline must give the same answer under `.parallel()`.  
> **Follow-up-ловушка:** «Почему identity, а не просто seed?» → следующий вопрос.

> [!question]- Что напечатает этот код и почему? / What does this print and why?
> 
> java
> 
> ```java
> List<Integer> nums = List.of(1, 2, 3, 4);
> System.out.println(nums.stream().reduce(10, Integer::sum));
> System.out.println(nums.parallelStream().reduce(10, Integer::sum));
> ```
> 
> **RU:** `20` и недетерминированное число больше 20 (в прогоне — `50`). Контракт требует, чтобы первый аргумент был identity: `identity + x == x`. Десятка им не является, а параллельный `reduce` даёт каждому куску разбиения свой стартовый identity и комбинирует результаты — десятка прибавляется по разу на кусок. Последовательный вариант маскирует ошибку. Правильно: `10 + nums.stream().reduce(0, Integer::sum)`.  
> **EN:** `20`, then a nondeterministic number above 20 (`50` in my run). The contract requires the first argument to be an identity: `identity + x == x`. Ten isn't one, and parallel `reduce` seeds every split chunk with the identity and combines partial results — the ten gets added once per chunk. The sequential run masks the bug. The fix: `10 + nums.stream().reduce(0, Integer::sum)`.

> [!question]- В чём разница между `findFirst` и `findAny`? / What's the difference between `findFirst` and `findAny`?  
> **RU:** `findFirst` обязан вернуть первый элемент в encounter order; `findAny` — любой подходящий. На последовательном стриме результат фактически совпадает; на параллельном `findFirst` платит за координацию порядка между потоками, а `findAny` возвращает первое найденное любым потоком (в моём прогоне на 1..100000: `10` против `50010`). `findAny` — это ещё и декларация «порядок не важен», разрешающая оптимизации.  
> **EN:** `findFirst` must return the first element in encounter order; `findAny` — any matching one. Sequentially they effectively coincide; in parallel, `findFirst` pays for cross-thread ordering coordination, while `findAny` returns whatever any thread finds first (my run on 1..100000: `10` vs `50010`). `findAny` also declares "order doesn't matter," enabling optimizations.

> [!question]- Что вернёт `allMatch` на пустом стриме? / What does `allMatch` return on an empty stream?  
> **RU:** `true`, с любым предикатом — vacuous truth: все элементы пустого множества удовлетворяют любому условию. `anyMatch` — `false`, `noneMatch` — `true`. Практическое следствие: валидация «все элементы корректны → продолжаем» пропускает пустой ввод; пустоту нужно проверять отдельно, если по бизнесу она невалидна.  
> **EN:** `true`, for any predicate — vacuous truth: every element of the empty set satisfies any condition. `anyMatch` gives `false`, `noneMatch` gives `true`. The practical consequence: an "all elements valid → proceed" check happily passes empty input; emptiness must be validated separately if the business considers it invalid.

> [!question]- Почему `stream.sorted().findFirst()` — плохая идея для поиска минимума? / Why is `stream.sorted().findFirst()` a bad way to find the minimum?  
> **RU:** `sorted` — stateful-плотина: чтобы отдать хоть один элемент, буферизует и сортирует весь стрим — O(n log n) времени и O(n) памяти, ленивость `findFirst` не помогает. `min(comparator)` даёт тот же ответ за один проход, O(n) и константную память, и не зависнет соотносимо дольше на больших данных.  
> **EN:** `sorted` is a stateful barrier: to emit even one element it buffers and sorts the entire stream — O(n log n) time and O(n) memory; `findFirst`'s laziness doesn't help. `min(comparator)` yields the same answer in a single O(n) pass with constant memory.  
> **Senior add-on:** общий принцип — short-circuit не пробивает stateful-стадию; ленивость конвейера действует только между плотинами.

> [!question]- Правда ли, что `count()` всегда обходит стрим? / Does `count()` always traverse the stream?  
> **RU:** Нет. Если спплитератор источника знает точный размер (`SIZED`) и ни одна стадия его не меняет (`map`, `peek` не меняют; `filter`, `flatMap` — меняют), `count()` возвращает готовый размер без обхода — пользовательский код, включая `peek`, не вызывается ни разу. Это документировано в Javadoc и означает запрет на побочные эффекты в конвейере, чей результат — только счёт.  
> **EN:** No. If the source spliterator knows its exact size (`SIZED`) and no stage can change it (`map`, `peek` don't; `filter`, `flatMap` do), `count()` returns the known size with no traversal — user code, including `peek`, never runs. This is documented in the Javadoc and effectively bans side effects in a pipeline whose only result is the count.
