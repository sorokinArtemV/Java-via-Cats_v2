`reduce` из прошлого урока честен, но неудобен для главного практического сценария: свернуть стрим не в число, а в коллекцию или мапу. Требование «функции не мутируют аргументы» превращает сборку списка через `reduce` в создание нового списка на каждый элемент — O(n²) копирований. `collect` снимает это ограничение легальным способом: он проектировался под **мутабельные** контейнеры с самого начала. А `Collectors` — библиотека готовых рецептов сборки, от тривиального `toList` до `groupingBy` с вложенными агрегациями, который закрывает добрую половину случаев «а раньше я бы написал GROUP BY».

> [!abstract] Ментальная модель  
> `collect` — это mutable reduction: свёртка стрима в изменяемый контейнер через три функции — создать контейнер (supplier), добавить элемент (accumulator), слить два контейнера (combiner). `Collector` — упакованный набор этих функций; класс `Collectors` — фабрика готовых.

### Механизм: почему collect, а не reduce

Развёрнутая форма показывает всю механику:

```java
record Cat(String name, double weight, String breed) {}

List<String> names = cats.stream().map(Cat::name)
    .collect(
        ArrayList::new,     // supplier: новый пустой контейнер
        ArrayList::add,     // accumulator: (контейнер, элемент) → мутация
        ArrayList::addAll   // combiner: (контейнер, контейнер) → слияние кусков
    );
```

Та же тройка, что у трёхаргументного `reduce`, но контракт противоположный: аккумулятор **обязан мутировать** контейнер, а не возвращать новый. Параллельная безопасность достигается не иммутабельностью, а изоляцией: каждый кусок разбиения получает _свой_ контейнер от supplier, потоки не разделяют состояние, в конце комбайнер сливает контейнеры попарно. Никакой синхронизации внутри аккумулятора не нужно.

На практике развёрнутую форму почти не пишут — передают готовый `Collector`:

```java
List<String> names = cats.stream().map(Cat::name).collect(Collectors.toList());
```

`Collector<T, A, R>` — интерфейс, упаковывающий ту же тройку плюс финишёр (`A → R`, финальное преобразование контейнера) и характеристики-подсказки. Разбор — в 🔬; для использования достаточно знать, что `Collectors.*` возвращают готовые экземпляры, а downstream-коллекторы компонуются как матрёшки.

> [!tip] 💡 Из C#  
> Идейный аналог `collect` — не `Aggregate`, а семейство материализаторов `ToList`/`ToDictionary`/`ToLookup` + `GroupBy`. Разница архитектур: LINQ даёт десяток именованных методов-терминалов, Java — один метод `collect` и открытую алгебру коллекторов, которые вкладываются друг в друга (`groupingBy(..., mapping(..., toList()))`). Композиции коллекторов у LINQ нет — там это выражается продолжением запроса после `GroupBy`. И важное: LINQ-шный `GroupBy` ленив, как всё в LINQ, а `collect(groupingBy(...))` — терминал, материализующий результат немедленно.

### toList, toSet, toCollection — и современный toList()

```java
cats.stream().map(Cat::name).collect(Collectors.toList());          // до Java 16
cats.stream().map(Cat::name).toList();                              // Java 16+
cats.stream().map(Cat::name).collect(Collectors.toSet());
cats.stream().map(Cat::name).collect(Collectors.toCollection(TreeSet::new));
```

`toList()` у `Stream` — не сокращение для `collect(Collectors.toList())`, у них разные контракты. Прогон на JDK 21:

||`Stream.toList()`|`Collectors.toList()`|`Collectors.toUnmodifiableList()`|
|---|---|---|---|
|Фактический класс|`ImmutableCollections$ListN`|`ArrayList`|`ImmutableCollections$ListN`|
|`add()`|`UnsupportedOperationException`|работает|`UnsupportedOperationException`|
|Гарантия по Javadoc|unmodifiable|**никаких гарантий** типа и мутабельности|unmodifiable|
|`null`-элементы|разрешены|разрешены|`NullPointerException`|

Практические следствия. Код, который дособирает список после стрима (`list.add(...)`), сломается при механической замене `collect(toList())` на `.toList()` — это самая частая травма миграции на Java 16+. А полагаться на то, что `Collectors.toList()` вернёт именно `ArrayList`, запрещает сам Javadoc: «no guarantees on the type, mutability, serializability». Нужен гарантированно мутабельный список — `toCollection(ArrayList::new)`, явно и честно. По умолчанию в новом коде — `.toList()`: короче, а неизменяемость результата — фича (глава 13.11).

`toSet()` симметрично не гарантирует ни тип, ни порядок; нужен порядок — `toCollection(LinkedHashSet::new)`.

### toMap — коллектор с двумя минами

```java
Map<String, Double> weightByName = cats.stream()
    .collect(Collectors.toMap(Cat::name, Cat::weight));
```

Две функции: из элемента — ключ, из элемента — значение. Работает, пока данные идеальны. Мина первая — **дубликат ключа**, прогон JDK 21:

```java
cats.stream().collect(toMap(Cat::breed, Cat::name));
// IllegalStateException: Duplicate key siberian
//   (attempted merging values Barsik and Murka)
```

Не «последний победил», как сделал бы `put`, а исключение — и это осознанный дизайн: молчаливая потеря данных хуже падения. Если стратегия слияния есть, её объявляют третьим аргументом:

```java
Map<String, String> namesByBreed = cats.stream()
    .collect(toMap(Cat::breed, Cat::name, (a, b) -> a + ", " + b));
// {angora=Пушок, Снежок, maine-coon=Рыжик, siberian=Барсик, Мурка}
// «последний победил» явно: (a, b) -> b
```

Мина вторая — **`null`-значение**: `toMap` падает с `NullPointerException`, даже с merge-функцией (проверено; внутри — `Map.merge`, запрещающий null). `HashMap` null-значения хранить умеет, а собрать его через `toMap` с ними нельзя — обходной путь через `forEach` + `put` или фильтрацию null заранее. Четвёртый аргумент — supplier мапы, когда нужен конкретный тип: `toMap(k, v, merge, LinkedHashMap::new)`.

> [!tip] 💡 Из C#  
> `ToDictionary` при дубликате тоже бросает (`ArgumentException`), тут интуиция совпадает. Но merge-перегрузки у него нет — конфликт в LINQ решают заранее (`GroupBy` + `ToDictionary(g => g.Key, g => ...)`), а в Java — на месте, третьим аргументом. Про null-значения C#-словарь спокоен — Java-`toMap` нет.

### groupingBy — рабочая лошадь агрегаций

```java
Map<String, List<Cat>> byBreed = cats.stream()
    .collect(Collectors.groupingBy(Cat::breed));
// {angora=[Пушок, Снежок], maine-coon=[Рыжик], siberian=[Барсик, Мурка]}
```

Классификатор → мапа «ключ → список попавших». По умолчанию — `HashMap` с `ArrayList`-значениями (проверено прогоном; но это деталь реализации, контракт снова «no guarantees»). Сила `groupingBy` — второй параметр, **downstream-коллектор**: что делать с элементами внутри группы вместо складывания в список:

```java
// сколько в группе
Map<String, Long> countByBreed = cats.stream()
    .collect(groupingBy(Cat::breed, counting()));
// {angora=2, maine-coon=1, siberian=2}

// не котиков, а их имена: mapping = map внутри группы
Map<String, List<String>> namesByBreed = cats.stream()
    .collect(groupingBy(Cat::breed, mapping(Cat::name, toList())));

// средний вес по породе
Map<String, Double> avgWeight = cats.stream()
    .collect(groupingBy(Cat::breed, averagingDouble(Cat::weight)));
// {angora=3.05, maine-coon=7.8, siberian=5.35}
```

Downstream-коллекторы вкладываются на любую глубину — `groupingBy(breed, groupingBy(adult, counting()))` даёт `Map<String, Map<Boolean, Long>>`. Читаемость кончается на второй вложенности; дальше — выноси внутренний коллектор в переменную.

Тонкая пара: `filtering` (Java 9) как downstream против `filter` до группировки — прогон JDK 21:

```java
// filtering ВНУТРИ группы: породы без тяжёлых котов остаются с пустым списком
cats.stream().collect(groupingBy(Cat::breed,
    filtering(c -> c.weight() > 5, mapping(Cat::name, toList()))));
// {angora=[], maine-coon=[Рыжик], siberian=[Барсик]}

// filter ДО группировки: лёгкие породы исчезают целиком
cats.stream().filter(c -> c.weight() > 5)
    .collect(groupingBy(Cat::breed, mapping(Cat::name, toList())));
// {maine-coon=[Рыжик], siberian=[Барсик]}
```

Первый вариант отвечает на вопрос «для каждой породы — кто из них тяжёлый», второй — «какие породы имеют тяжёлых». Для отчётов, где нужны все категории включая нулевые, — только `filtering`. Симметрично `flatMapping` (Java 9) разворачивает вложенность внутри группы. Третья форма `groupingBy(classifier, mapFactory, downstream)` задаёт тип мапы: `groupingBy(Cat::breed, TreeMap::new, toList())` — группы, отсортированные по ключу.

### partitioningBy — группировка на два лагеря

```java
Map<Boolean, List<Cat>> byAdult = cats.stream()
    .collect(Collectors.partitioningBy(c -> c.weight() > 5));
```

Частный случай группировки по предикату, но с отличием, ради которого он существует: **оба ключа присутствуют всегда**. Прогон на пустом стриме: `{false=[], true=[]}` — тогда как `groupingBy(c -> c.weight() > 5)` для пустого входа дал бы пустую мапу, и `get(true)` вернул бы `null`. Плюс специализированная реализация мапы ровно на две ячейки. Downstream поддерживается так же: `partitioningBy(pred, counting())`.

### joining — сборка строки

```java
cats.stream().map(Cat::name).collect(Collectors.joining(", ", "[", "]"));
// [Барсик, Мурка, Пушок, Рыжик, Снежок]
```

Три формы: без аргументов, с разделителем, с разделителем + префиксом + суффиксом. Внутри — `StringJoiner` поверх `StringBuilder`, так что это и есть правильный способ склеить стрим строк (глава 14.3 — почему не `reduce("", String::concat)`: квадратичная стоимость). Работает только на `Stream<CharSequence>` — не-строки сначала через `map(Object::toString)`.

### teeing — два коллектора за один проход

Классическая задача: из одного стрима нужны _два_ агрегата — скажем, самый лёгкий и самый тяжёлый котик. Два прохода — два стрима (одноразовость!), а `teeing` (Java 12, JDK-8209685) раздваивает поток на два downstream-коллектора и сливает их результаты функцией:

```java
record MinMax(Optional<Cat> min, Optional<Cat> max) {}

MinMax mm = cats.stream().collect(Collectors.teeing(
    minBy(Comparator.comparingDouble(Cat::weight)),
    maxBy(Comparator.comparingDouble(Cat::weight)),
    MinMax::new));
// min=Пушок, max=Рыжик — один проход
```

Каждый элемент попадает в оба коллектора; финишёр собирает пару. С локальными records (глава 17) — идиома «две метрики за проход» без классов-носителей.

> [!example] 🔬 Глубже / на синьора  
> **Интерфейс `Collector<T, A, R>`** — пять методов: `supplier(): A`, `accumulator(): BiConsumer<A,T>`, `combiner(): BinaryOperator<A>`, `finisher(): Function<A,R>`, `characteristics(): Set<Characteristics>`. `A` — тип промежуточного контейнера, часто скрытый (у `joining` это `StringBuilder`, наружу выходит `String`). Характеристики — подсказки движку: `IDENTITY_FINISH` (финишёр — тождественный, каст без вызова — у `toList`), `UNORDERED` (порядок не важен — у `toSet`), `CONCURRENT` (один общий контейнер вместо контейнера-на-кусок; требует потокобезопасный контейнер — у `toConcurrentMap`, `groupingByConcurrent`; на упорядоченном стриме применяется только вместе с `UNORDERED`, иначе движок обязан сохранять encounter order).
> 
> **Свой коллектор** — `Collector.of(...)`:
> 
> 
> 
> ```java
> Collector<Cat, ?, Double> totalWeight = Collector.of(
>     () -> new double[1],
>     (acc, c) -> acc[0] += c.weight(),
>     (a, b) -> { a[0] += b[0]; return a; },
>     acc -> acc[0]);
> // cats.stream().collect(totalWeight) == 24.6 — но для сумм есть summingDouble
> ```
> 
> Реальный повод писать свой — нестандартный контейнер или агрегат, которого нет среди ~40 фабрик `Collectors`; прежде чем писать, стоит поискать композицию из готовых.
> 
> **`groupingByConcurrent`** возвращает `ConcurrentMap` и в параллельном стриме пишет из всех потоков в одну `ConcurrentHashMap` вместо слияния мап-на-поток — дешевле на больших кардинальностях, ценой потери encounter order внутри групп (глава 24.10 про саму CHM). Это единственное семейство коллекторов, где параллельность меняет наблюдаемое поведение (порядок в группах), — легальная сделка, объявленная характеристиками `CONCURRENT + UNORDERED`.
> 
> **Почему `counting()` существует при живом `count()`**: standalone он бессмыслен, его место — downstream-позиция в `groupingBy`/`partitioningBy`/`teeing`, где терминала `count()` нет. Общий паттерн: почти каждому терминалу (`count`, `min`, `reduce`, `toList`) соответствует коллектор-двойник (`counting`, `minBy`, `reducing`, `toList`) именно для композиции.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`toMap` без merge-функции на неуникальном ключе** — `IllegalStateException: Duplicate key ...` в рантайме, на проде, на первых же реальных данных. Ключ из бизнес-поля (email, имя, порода) — почти всегда повод для третьего аргумента.
> 2. **`toMap` с `null`-значениями** — NPE даже с merge-функцией. Маппер значения, способный вернуть null (`Cat::owner` для бездомного), требует предварительного `filter` или другого пути сборки.
> 3. **Замена `collect(toList())` на `.toList()` «для краткости»** — если результат где-то мутируется, получишь `UnsupportedOperationException` вдали от места замены. И обратная зависимость: код, полагающийся на мутабельность `Collectors.toList()`, формально держится на недокументированном поведении — гарантия только у `toCollection(ArrayList::new)`.
> 4. **`groupingBy(...).get(key)` для отсутствующей группы — `null`.** Групп с нулём элементов `groupingBy` не создаёт (в отличие от `partitioningBy` и от `filtering`-downstream). Идиома: `getOrDefault(key, List.of())`.
> 5. **Ожидание порядка от `groupingBy`/`toSet`** — `HashMap`/`HashSet` внутри, порядок ключей произволен. Нужен — задавай фабрику: `groupingBy(k, TreeMap::new, ds)` или `LinkedHashMap::new` для порядка появления.
> 6. **`joining` на стриме не-строк** — не скомпилируется, и это правильно; но «исправление» через `reduce("", (a,b) -> a + b)` компилируется и работает за O(n²). Только `map(Object::toString)` + `joining`.

### На собесе

> [!question]- Чем `collect` отличается от `reduce`? / How does `collect` differ from `reduce`?  
> **RU:** `reduce` — immutable reduction: функции обязаны не мутировать аргументы и возвращать новые значения; сворачивать им список — O(n²) копирований. `collect` — mutable reduction: аккумулятор мутирует контейнер, а параллельная корректность достигается изоляцией (свой контейнер на кусок разбиения от supplier) плюс комбайнером для слияния. Правило: скаляры — `reduce`, контейнеры — `collect`.  
> **EN:** `reduce` is an immutable reduction: its functions must not mutate arguments and must return new values; building a list with it costs O(n²) in copies. `collect` is a mutable reduction: the accumulator mutates a container, and parallel correctness comes from isolation (a fresh container per split from the supplier) plus a combiner to merge. Rule of thumb: scalars → `reduce`, containers → `collect`.

> [!question]- В чём разница между `Stream.toList()` и `collect(Collectors.toList())`? / `Stream.toList()` vs `collect(Collectors.toList())`?  
> **RU:** `Stream.toList()` (Java 16) гарантированно возвращает unmodifiable-список, допускающий null-элементы. `Collectors.toList()` по контракту не гарантирует ничего — ни тип, ни мутабельность; фактически сегодня это `ArrayList`, но опираться на это нельзя. Нужен гарантированно изменяемый список — `Collectors.toCollection(ArrayList::new)`. Механическая замена одного на другое ломает код, дописывающий в результат.  
> **EN:** `Stream.toList()` (Java 16) is guaranteed to return an unmodifiable list that permits null elements. `Collectors.toList()` guarantees nothing by contract — neither type nor mutability; today it happens to be an `ArrayList`, but that's not to be relied on. For a guaranteed-mutable list use `Collectors.toCollection(ArrayList::new)`. Mechanically swapping one for the other breaks code that appends to the result.  
> **Follow-up:** «А `toUnmodifiableList`?» → тоже unmodifiable, но null-hostile: NPE на null-элементе, в отличие от `Stream.toList()`.

> [!question]- Что случится при дубликате ключа в `Collectors.toMap`? / What happens on a duplicate key in `Collectors.toMap`?  
> **RU:** `IllegalStateException: Duplicate key ...` — осознанный дизайн: молча затереть значение опаснее, чем упасть. Стратегия слияния объявляется третьим аргументом — merge-функцией: `(a, b) -> b` для «последний победил», конкатенация, суммирование. Вторая ловушка того же коллектора: null-значение даёт NPE даже с merge-функцией.  
> **EN:** `IllegalStateException: Duplicate key ...` — a deliberate design: silently overwriting a value is worse than failing. A merge strategy is declared as the third argument: `(a, b) -> b` for last-wins, concatenation, summation. The same collector's second trap: a null value throws NPE even with a merge function present.

> [!question]- `groupingBy` против `partitioningBy` — когда что? / `groupingBy` vs `partitioningBy` — when to use which?  
> **RU:** `partitioningBy` — специализация для предиката: ровно два лагеря, и **оба ключа присутствуют всегда**, даже для пустого стрима (`{false=[], true=[]}`). `groupingBy` по булеву классификатору создаст ключ, только если в него что-то попало, — `get(true)` может вернуть null. Деление на да/нет — `partitioningBy`; произвольные ключи — `groupingBy`.  
> **EN:** `partitioningBy` is the predicate specialization: exactly two buckets, and **both keys are always present**, even for an empty stream (`{false=[], true=[]}`). `groupingBy` over a boolean classifier creates a key only if something lands in it — `get(true)` may return null. Yes/no splits → `partitioningBy`; arbitrary keys → `groupingBy`.

> [!question]- Что такое downstream-коллектор? Приведите пример. / What is a downstream collector? Give an example.  
> **RU:** Коллектор, применяемый к элементам внутри каждой группы вместо дефолтного `toList`: `groupingBy(Cat::breed, counting())` — количество по породам, `groupingBy(Cat::breed, mapping(Cat::name, toList()))` — имена вместо объектов, `averagingDouble` — среднее. Коллекторы компонуются на любую глубину — это главное архитектурное отличие от LINQ, где агрегация группы выражается продолжением запроса.  
> **EN:** A collector applied to the elements within each group instead of the default `toList`: `groupingBy(Cat::breed, counting())` for per-breed counts, `groupingBy(Cat::breed, mapping(Cat::name, toList()))` for names instead of objects, `averagingDouble` for means. Collectors compose to any depth — the key architectural difference from LINQ, where group aggregation is expressed by continuing the query.  
> **Follow-up:** «Чем `filtering` как downstream отличается от `filter` до `groupingBy`?» → `filtering` сохраняет группы с пустым результатом, предварительный `filter` удаляет ключи целиком; для отчётов со всеми категориями — только первый.

> [!question]- Как получить из одного стрима сразу два агрегата — например, min и max? / How do you get two aggregates — say, min and max — from one stream?  
> **RU:** Стрим одноразовый, два прохода потребуют пересоздания. Штатное решение — `Collectors.teeing` (Java 12): элементы дублируются в два downstream-коллектора, их результаты сливаются функцией: `teeing(minBy(cmp), maxBy(cmp), MinMax::new)` — оба экстремума за один проход. Для числовой статистики есть готовый `summarizingDouble` — count/sum/min/max/average одним коллектором.  
> **EN:** A stream is single-use, so two passes mean recreating it. The standard tool is `Collectors.teeing` (Java 12): each element is fed to two downstream collectors whose results are merged by a function: `teeing(minBy(cmp), maxBy(cmp), MinMax::new)` — both extremes in one pass. For numeric stats there's `summarizingDouble` — count/sum/min/max/average in a single collector.

> [!question]- Из чего состоит интерфейс `Collector`? / What does the `Collector` interface consist of?  
> **RU:** Пять компонентов: `supplier` — создание промежуточного контейнера, `accumulator` — вливание элемента, `combiner` — слияние контейнеров параллельных кусков, `finisher` — финальное преобразование контейнера в результат (у `joining` — `StringBuilder → String`), `characteristics` — подсказки движку (`IDENTITY_FINISH` — финишёр тождественный, `UNORDERED`, `CONCURRENT` — писать в общий потокобезопасный контейнер вместо слияния). Свой коллектор — `Collector.of(...)`, но сперва стоит поискать композицию готовых.  
> **EN:** Five parts: `supplier` — creates the intermediate container, `accumulator` — folds an element in, `combiner` — merges containers from parallel splits, `finisher` — final container-to-result transform (`StringBuilder → String` for `joining`), and `characteristics` — engine hints (`IDENTITY_FINISH` — the finisher is identity, `UNORDERED`, `CONCURRENT` — write to one thread-safe container instead of merging). Custom collectors come from `Collector.of(...)`, though composing built-ins usually wins.  
> **Senior add-on:** `CONCURRENT` на упорядоченном стриме применяется только в паре с `UNORDERED` — иначе движок обязан сохранить encounter order и откатывается к схеме «контейнер на кусок».
