Представь задачу из реального кода: у тебя список котиков из приюта, и нужно найти имена трёх самых тяжёлых взрослых котов, отсортированных по весу. На циклах это превращается в нагромождение: временный список для отфильтрованных, сортировка, ещё один список для имён, счётчик для лимита. Логика задачи — «отфильтруй, отсортируй, возьми три, извлеки имена» — тонет в механике её исполнения. Streams API переворачивает это: ты описываешь _что_ сделать с данными, а _как_ именно итерироваться — забота библиотеки. Для тебя это не новая идея — это Java-ответ на LINQ, но с достаточным количеством собственных механик и ловушек, чтобы разобрать его с нуля.

> [!abstract] Ментальная модель  
> `Stream` — это **одноразовый ленивый конвейер операций над данными**, а не структура данных. Он не хранит элементы, не модифицирует источник и не выполняет ничего, пока не вызвана терминальная операция.

### 10.1 Зачем Streams

#### Проблема, которую решают стримы

Домен на всю главу:

```java
record Cat(String name, double weight, boolean adult) {}

List<Cat> shelter = List.of(
    new Cat("Барсик", 6.5, true),
    new Cat("Мурка", 4.2, true),
    new Cat("Пушок", 1.1, false),
    new Cat("Рыжик", 7.8, true),
    new Cat("Снежок", 5.0, true)
);
```

Императивно:

```java
List<Cat> adults = new ArrayList<>();
for (Cat c : shelter) {
    if (c.adult()) adults.add(c);
}
adults.sort(Comparator.comparingDouble(Cat::weight).reversed());
List<String> top = new ArrayList<>();
for (int i = 0; i < Math.min(3, adults.size()); i++) {
    top.add(adults.get(i).name());
}
```

Декларативно:

```java
List<String> top = shelter.stream()
    .filter(Cat::adult)
    .sorted(Comparator.comparingDouble(Cat::weight).reversed())
    .limit(3)
    .map(Cat::name)
    .toList();
// [Рыжик, Барсик, Снежок]
```

Выигрыш не в количестве строк, а в том, что второй вариант читается как формулировка задачи. Каждая операция делает ровно одну вещь, промежуточных мутируемых списков нет, а `limit(3)` в связке с ленивостью ещё и способен не обрабатывать лишнее (детали — в уроке 3).

#### Чем стрим НЕ является

Это важнее, чем чем он является, потому что интуиция от коллекций здесь вредит:

1. **Не хранилище.** Стрим не содержит элементов — он тянет их из источника (коллекции, массива, генератора) по требованию.
2. **Не модификатор источника.** `filter` не удаляет котиков из `shelter` — он пропускает через себя те, что проходят предикат. Источник после конвейера не изменён.
3. **Одноразовый.** Один стрим — одна терминальная операция. Повторный запуск — исключение (разбор ниже в ⚠️).
4. **Ленивый.** Цепочка `filter().map()` сама по себе не делает ничего — она лишь строит описание конвейера. Работа начинается с терминальной операции.

> [!tip] 💡 Из C#  
> Прямая параллель — LINQ: `stream()` открывает конвейер как `IEnumerable<T>`-цепочка, `filter` ≈ `Where`, `map` ≈ `Select`, ленивость ≈ deferred execution. Но есть разлом, который кусает именно после C#: **`IEnumerable<T>` можно перечислять сколько угодно раз** (каждый `foreach` заново дёргает `GetEnumerator()`), а **`Stream` — строго один раз**. `Stream` ближе не к `IEnumerable`, а к уже полученному `IEnumerator`: позиция есть, перемотки нет. Хочешь «ещё раз» — бери новый стрим у источника: `shelter.stream()` дёшев, это создание лёгкого объекта-обёртки, а не копирование данных.

### 10.2 Создание стрима

#### Из коллекции — основной путь

```java
Stream<Cat> s = shelter.stream();
Stream<Cat> p = shelter.parallelStream(); // parallel — тема урока 6
```

`stream()` — default-метод интерфейса `Collection`, поэтому он есть у любой коллекции: `List`, `Set`, `Deque`. У `Map` метода `stream()` **нет** — `Map` не является `Collection`. Стрим по мапе берут через её представления: `map.entrySet().stream()`, `map.keySet().stream()`, `map.values().stream()`.

#### Из явных значений и массивов

```java
Stream<String> names = Stream.of("Барсик", "Мурка", "Пушок");
Stream<Cat> one = Stream.of(new Cat("Рыжик", 7.8, true));
Stream<Object> none = Stream.empty(); // для возврата из методов вместо null
```

С массивами — первая настоящая ловушка, проверено на JDK 21:

```java
int[] weights = {1, 2, 3};

Stream.of(weights).count();     // 1  — стрим из ОДНОГО элемента типа int[]
Arrays.stream(weights).sum();   // 6  — IntStream из трёх чисел
```

`Stream.of(T... values)` — varargs-метод. Массив объектов `Cat[]` он разложит на элементы, но `int[]` не может стать `Cat...`-подобным varargs из `int` — примитивы не подставляются в `T`, поэтому весь массив целиком становится единственным элементом `Stream<int[]>`. `Arrays.stream(int[])` возвращает `IntStream` — примитивный стрим без боксинга (подробно — в уроке 5). Правило: **для массивов — всегда `Arrays.stream`**, для перечисления значений — `Stream.of`.

#### Бесконечные и генерируемые стримы

Здесь стримы уходят за пределы того, что умеет коллекция: источник может быть бесконечным, потому что элементы вычисляются по требованию.

```java
// generate: бесконечное повторение работы Supplier
Stream.generate(() -> "мяу").limit(3).toList();  // [мяу, мяу, мяу]

// iterate: seed + функция от предыдущего элемента
Stream.iterate(1, n -> n * 2).limit(5).toList(); // [1, 2, 4, 8, 16]

// iterate с предикатом (Java 9, JEP 269 era) — «стримовый for»
Stream.iterate(1, n -> n < 20, n -> n * 2).toList(); // [1, 2, 4, 8, 16]
```

Трёхаргументный `iterate(seed, hasNext, next)` читается точно как `for (int n = 1; n < 20; n *= 2)` — и это его прямое назначение: конечная последовательность без внешнего `limit`. Двухаргументные `generate`/`iterate` без ограничителя (`limit`, `takeWhile`) при терминальной операции зависнут навсегда — см. ⚠️.

> [!tip] 💡 Из C#  
> `Stream.iterate` — аналога в стандартном LINQ нет (ближайшее — `Enumerable.Range` для чисел или самописный генератор через `yield return`). Вот про `yield` важно: в C# ленивый бесконечный источник ты пишешь сам итератор-методом, в Java ключевого слова `yield` для генераторов нет (есть `yield` в switch expression — не путать), поэтому `generate`/`iterate` — штатная замена этой идиомы.

#### Стрим из возможно-null значения

```java
Stream.ofNullable(null).count();   // 0
Stream.ofNullable("x").count();    // 1
```

`Stream.ofNullable` (Java 9) — микро-утилита: пустой стрим для `null`, одноэлементный иначе. Её сила раскрывается внутри `flatMap`, когда маппер может вернуть `null`, — увидим в уроке 2.

#### Прочие источники, о которых стоит знать

- `IntStream.range(0, 10)` / `rangeClosed(1, 10)` — числовые диапазоны (урок 5);
- `Files.lines(path)` — стрим строк файла; обязателен try-with-resources, это стрим с ресурсом под ним (глава 18);
- `"мяу".chars()` — `IntStream` кодов символов;
- `Pattern.compile(",").splitAsStream(csv)` — ленивый аналог `split`.

> [!example] 🔬 Глубже / на синьора  
> **Что физически создаёт `shelter.stream()`.** Под капотом это `StreamSupport.stream(spliterator(), false)`. `Spliterator` — истинная абстракция источника (аналог `Iterator`, но умеющий `trySplit()` для параллелизма и сообщающий характеристики: `SIZED`, `ORDERED`, `SORTED`, `DISTINCT`). Каждая промежуточная операция создаёт новый объект `ReferencePipeline`-стадии, ссылающийся на предыдущую — конвейер живёт как односвязный список описаний, данные по нему не текут до терминала.
> 
> **Late binding.** Спплитератор коллекции связывается с её содержимым не в момент `stream()`, а в момент терминальной операции. Проверено на JDK 21:
> 
> 
> 
> ```java
> List<String> cats = new ArrayList<>(List.of("Барсик", "Мурка"));
> Stream<String> lazy = cats.stream();
> cats.add("Пушок");
> lazy.count(); // 3 — Пушок попал в стрим, созданный ДО его добавления
> ```
> 
> Следствие: «снимок» данных не делается, стрим видит источник на момент запуска. Модификация _до_ терминала — легальна; модификация _во время_ обхода — interference, см. ⚠️.
> 
> **Почему одноразовость — дизайн-решение, а не лень авторов.** Стрим может держать невоспроизводимый источник (строки файла, сокет, генератор со состоянием) — контракт «один проход» единый для всех источников, чтобы поведение не зависело от того, повторяем ли источник. Плюс это развязывает руки внутренним оптимизациям: стадии могут мутировать своё состояние при исполнении, не заботясь о повторном запуске.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Повторное использование стрима.** Вторая терминальная операция (и даже добавление промежуточной после терминальной) бросает `IllegalStateException: stream has already been operated upon or closed` — сообщение с прогона на JDK 21. Симптом в реальном коде: стрим сохранили в переменную/поле и «переиспользуют». Лекарство — хранить источник или `Supplier<Stream<T>>`, а стрим создавать заново.
> 2. **`Stream.of(int[])`** — стрим из одного элемента-массива, а не из чисел. Компилятор не спасёт: `Stream<int[]>` — легальный тип. Для примитивных массивов только `Arrays.stream`.
> 3. **Модификация источника во время обхода.** `cats.stream().forEach(c -> cats.add("Рыжик"))` на `ArrayList` бросил `ConcurrentModificationException` (прогон JDK 21). Но гарантии нет: тот же тест с `remove` упал с `NullPointerException` внутри лямбды — спплитератор прочитал уже сдвинутый массив. Interference = неопределённое поведение, fail-fast — лишь любезность, как и у итераторов (глава 13).
> 4. **Бесконечный стрим без ограничителя.** `Stream.generate(() -> "мяу").count()` не вернётся никогда — не будет ни исключения, ни OOM-подсказки, просто вечный цикл. Каждому `generate`/двухаргументному `iterate` — `limit`/`takeWhile` до терминала.
> 5. **Стрим как поле или возвращаемый тип «на потом».** Стрим — не значение, а отложенное вычисление с одноразовым зарядом: сохранённый в поле, он либо протухнет после первого использования, либо увидит источник в неожиданном состоянии (late binding). Из методов возвращайте коллекцию; стрим — только если вызывающий по контракту потребляет его немедленно и однократно.

### На собесе

> [!question]- В чём разница между коллекцией и стримом? / What's the difference between a collection and a stream?  
> **RU:** Коллекция — структура данных: хранит элементы в памяти, живёт долго, допускает многократный обход и модификацию. Стрим — одноразовый ленивый конвейер вычислений над источником: не хранит элементы, не модифицирует источник, ничего не выполняет до терминальной операции и потребляется ровно один раз. Коллекция — про данные, стрим — про вычисление над ними.  
> **EN:** A collection is a data structure: it stores elements in memory, can be traversed many times and modified. A stream is a single-use lazy pipeline of computations over a source: it stores nothing, doesn't modify the source, executes nothing until a terminal operation, and is consumed exactly once. Collections are about data; streams are about computation over data.  
> **Follow-up:** «А почему стрим нельзя переиспользовать?» → см. следующий вопрос — это проверка, понимаешь ли ты дизайн, а не зубришь факт.

> [!question]- Что будет, если вызвать терминальную операцию на стриме дважды? Почему так сделано? / What happens if you call a terminal operation twice? Why is it designed that way?  
> **RU:** `IllegalStateException: stream has already been operated upon or closed`. Дизайн-причина: источник стрима может быть невоспроизводимым (строки файла, генератор, сетевые данные), и контракт «один проход» сделан единым для всех источников. Нужен повтор — создаём новый стрим от источника, это дёшево.  
> **EN:** `IllegalStateException` with the message "stream has already been operated upon or closed". The design reason: a stream's source may be non-replayable (file lines, a generator, network data), so the one-pass contract is uniform for all sources. If you need another pass, create a new stream from the source — it's cheap.  
> **Ловушка слабого ответа:** «потому что стрим уже пустой» — нет, дело не в опустошении, а в контракте: исключение бросается даже если терминал не потребил ни одного элемента.

> [!question]- Модифицирует ли `filter` исходную коллекцию? / Does `filter` modify the source collection?  
> **RU:** Нет. Промежуточные операции не трогают источник — они описывают, какие элементы и в каком виде пройдут дальше по конвейеру. Результат материализуется терминальной операцией в новую структуру. Это принцип non-interference: стрим читает источник, но не пишет в него.  
> **EN:** No. Intermediate operations never touch the source — they describe which elements pass down the pipeline and in what form. The result is materialized by a terminal operation into a new structure. This is the non-interference principle: a stream reads its source but never writes to it.  
> **Follow-up:** «А можно ли модифицировать источник самому во время стрима?» → нельзя во время обхода (interference: обычно `ConcurrentModificationException`, но поведение не определено); можно между созданием стрима и терминальной операцией (late binding).

> [!question]- Чем `Stream.of(array)` отличается от `Arrays.stream(array)`? / How does `Stream.of(array)` differ from `Arrays.stream(array)`?  
> **RU:** Для массива объектов — почти ничем: varargs разложит его на элементы. Для примитивного массива `int[]` разница критична: `Stream.of` вернёт `Stream<int[]>` из одного элемента (примитивы не подставляются в generic varargs), а `Arrays.stream` — `IntStream` из чисел без боксинга.  
> **EN:** For an object array — almost nothing: varargs spreads it into elements. For a primitive `int[]` the difference is critical: `Stream.of` returns a single-element `Stream<int[]>` (primitives can't fill a generic varargs), while `Arrays.stream` returns an `IntStream` of the numbers, with no boxing.  
> **Follow-up:** «А зачем вообще `IntStream`, чем плох `Stream<Integer>`?» → боксинг на каждый элемент; развёрнутый ответ — про кэш `Integer` и стоимость аллокаций (глава 1.4, урок 5).

> [!question]- Как создать бесконечный стрим и чем это может кончиться? / How do you create an infinite stream and what can go wrong?  
> **RU:** `Stream.generate(supplier)` — независимые значения, `Stream.iterate(seed, next)` — каждое из предыдущего. Терминальная операция без ограничителя (`limit`, `takeWhile`) уйдёт в вечный цикл. С Java 9 есть `iterate(seed, hasNext, next)` — конечная последовательность в стиле цикла `for`, ограничитель встроен.  
> **EN:** `Stream.generate(supplier)` for independent values, `Stream.iterate(seed, next)` where each element derives from the previous. A terminal operation without a limiter (`limit`, `takeWhile`) loops forever. Since Java 9, `iterate(seed, hasNext, next)` produces a finite, for-loop-style sequence with the limiter built in.

> [!question]- Стрим в Java против `IEnumerable` в C# — в чём главное отличие? / Java streams vs C#'s `IEnumerable` — the key difference?  
> **RU:** Оба ленивы (deferred execution), но `IEnumerable` — воспроизводимая последовательность: каждый обход заново запрашивает энумератор. `Stream` — одноразовый: одна терминальная операция, потом исключение. `Stream` семантически ближе к `IEnumerator`, чем к `IEnumerable`. Плюс у Java отдельные примитивные стримы (`IntStream`) из-за type erasure и боксинга — в C# generics reified, `IEnumerable<int>` работает с примитивом напрямую.  
> **EN:** Both are lazy (deferred execution), but `IEnumerable` is replayable — each traversal requests a fresh enumerator, — while a `Stream` is single-use: one terminal operation, then an exception. A `Stream` is semantically closer to `IEnumerator` than to `IEnumerable`. Also, Java has dedicated primitive streams (`IntStream`) because of type erasure and boxing — C# generics are reified, so `IEnumerable<int>` handles the primitive directly.

> [!question]- Почему у `Map` нет метода `stream()`? / Why doesn't `Map` have a `stream()` method?  
> **RU:** `Map` не расширяет `Collection` — это не последовательность элементов, а соответствие ключ→значение, и неочевидно, что считать «элементом». Поэтому стрим берут у одного из трёх представлений: `entrySet().stream()` (пары), `keySet().stream()`, `values().stream()`.  
> **EN:** `Map` doesn't extend `Collection` — it's a key-to-value mapping, not a sequence, and it's ambiguous what an "element" would be. So you stream one of its three views: `entrySet().stream()` (pairs), `keySet().stream()`, or `values().stream()`.

> 
> java
> 
> ```java
> List<String> cats = new ArrayList<>(List.of("Барсик", "Мурка"));
> Stream<String> s = cats.stream();
> cats.add("Пушок");
> System.out.println(s.count());
> ```
> 
> **RU:** `3`. Спплитератор связывается с содержимым коллекции лениво — в момент терминальной операции (late binding), а не в момент `stream()`. «Пушок», добавленный между созданием стрима и `count()`, виден стриму. Слабый ответ «2, стрим зафиксировал снимок» — неверен: снимка нет. Слабый ответ «`ConcurrentModificationException`» тоже неверен: CME грозит при модификации _во время_ обхода, а здесь она произошла до его начала.  
> **EN:** `3`. The spliterator binds to the collection's contents lazily — at the terminal operation (late binding), not at `stream()` creation. "Пушок", added in between, is visible to the stream. "2, the stream took a snapshot" is wrong — there's no snapshot. "`ConcurrentModificationException`" is also wrong — CME applies to modification _during_ traversal, and this one happened before traversal began.
