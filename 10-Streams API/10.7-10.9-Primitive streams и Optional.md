Этот урок — про два типа-спутника Streams API, которые существуют по одной и той же причине: у Java есть шов между примитивами и объектами, и стримам приходится этот шов обходить. `IntStream` и его братья — обход шва в сторону производительности: дженерики не работают с примитивами, а платить боксингом за каждое число не хочется. `Optional` — обход в сторону семантики: у ссылочных типов нет способа сказать «значения может не быть» громче, чем `null`, который молчит до самого NPE. Для тебя оба сюжета выглядят странно с C#-колокольни — там reified-дженерики и `Nullable<T>` закрывают вопрос иначе, — поэтому разберём, откуда что растёт.

> [!abstract] Ментальная модель  
> `IntStream`/`LongStream`/`DoubleStream` — специализированные стримы поверх примитивов без боксинга, со своими числовыми терминалами (`sum`, `average`, `summaryStatistics`). `Optional<T>` — контейнер «ровно ноль или один элемент», тип возвращаемого значения для «результата может не быть», превращающий забытую проверку на null из NPE в ошибку компиляции.

### 10.7 Primitive streams

#### Проблема: цена Stream<Integer>

`Stream<T>` работает только с объектами — `Stream<int>` не существует, потому что после type erasure (глава 7.4) `T` — это `Object`, а `int` в `Object`-слот не помещается. Значит, каждый `int` в `Stream<Integer>` — это боксинг: аллокация объекта (или попадание в `Integer`-кэш −128..127 — глава 1.4, но рабочие числа в кэш обычно не влезают), указатель вместо значения, промах кэша процессора при чтении. Грубый замер на JDK 21 — сумма 20 млн чисел:

java

```java
IntStream.range(0, n).boxed().reduce(0, Integer::sum); // ~148 мс
IntStream.range(0, n).sum();                           // ~7 мс
```

Порядок разницы — **десятки раз** (это не JMH-бенчмарк, точная цифра зависит от машины и GC, но масштаб эффекта устойчив: аллокация 20 млн объектов против прохода по числам). Ответ библиотеки — три параллельных иерархии: `IntStream`, `LongStream`, `DoubleStream`, у которых элемент — примитив, лямбды — специализированные (`IntPredicate`, `IntUnaryOperator`), и никакого боксинга внутри конвейера. Только три: `byte`/`short`/`char` покрываются `IntStream`, `float` — `DoubleStream`, `boolean`-стрим никому не понадобился.

> [!tip] 💡 Из C#  
> Этой проблемы в C# **нет вообще** — и полезно понимать почему. Дженерики reified: для `IEnumerable<int>` JIT генерирует специализацию, где `int` лежит как значение, без упаковки. Java выбрала erasure ради совместимости (глава 7.11) и платит за это тройным дублированием API стримов: `IntStream` — это рукописная моноспециализация, которую в C# делает компилятор. Отсюда же лишний словарь: `mapToInt`, `boxed`, `mapToObj` — мосты, которых в LINQ просто не существует.

#### Создание и мосты между мирами

java

```java
IntStream.range(1, 5);          // 1, 2, 3, 4      — полуоткрытый, как for
IntStream.rangeClosed(1, 5);    // 1, 2, 3, 4, 5   — включительно
Arrays.stream(new int[]{1, 2}); // из массива — урок 1
"мяу".chars();                  // IntStream кодов символов — глава 14
```

Переходы (направление легко запомнить: `To` — куда конвертируем):

java

```java
record Cat(String name, int age) {}

// объектный → примитивный: mapToInt / mapToLong / mapToDouble
int totalAge = cats.stream().mapToInt(Cat::age).sum();

// примитивный → объектный: mapToObj / boxed
List<String> tags = IntStream.rangeClosed(1, 3).mapToObj(i -> "кот" + i).toList();
List<Integer> ages = cats.stream().mapToInt(Cat::age).boxed().toList();

// расширение между примитивными: asLongStream, asDoubleStream
```

Идиоматический паттерн: перешёл в примитивный мир как можно раньше, посчитал, вернулся через `boxed`/`mapToObj` только если результату нужно быть объектом.

#### Числовые терминалы — то, ради чего всё затевалось

java

```java
int[] weights = {65, 42, 11, 78, 50}; // в сотнях граммов

Arrays.stream(weights).sum();                  // 246
Arrays.stream(weights).average();              // OptionalDouble[49.2]
Arrays.stream(weights).max();                  // OptionalInt[78] — без компаратора!
Arrays.stream(weights).summaryStatistics();
// IntSummaryStatistics{count=5, sum=246, min=11, average=49.2, max=78}
```

У объектного `Stream` ни `sum`, ни `average` нет — «сумма котиков» бессмысленна. У примитивного `max`/`min` не требуют компаратора: порядок чисел естественный. `summaryStatistics` — пять агрегатов за один проход, готовый ответ на «min, max и среднее из одного стрима» (родственник `teeing` из урока 4). Обрати внимание на типы при пустом стриме — прогон JDK 21:

java

```java
IntStream.empty().sum();      // 0                    — сумма нуля слагаемых
IntStream.empty().average();  // OptionalDouble.empty — среднего НЕТ
```

`sum` возвращает голый `int`: ноль — математически честная сумма пустоты. `average` вернуть ноль не может — «средний вес нуля котиков» не ноль, а бессмыслица, — поэтому `OptionalDouble`.

#### Семейство OptionalInt/OptionalLong/OptionalDouble

Тот же erasure-налог: `Optional<Integer>` боксил бы, поэтому у примитивных терминалов свои контейнеры — `OptionalInt` (`getAsInt`, `orElse`, `ifPresent`…). Важная бедность, проверенная рефлексией на JDK 21: **у `OptionalInt` нет `map`, `filter`, `flatMap`** — только извлечение. Нужны трансформации — конвертируй: `optInt.stream().boxed().findFirst()` или просто забирай значение через `orElse`. Это осознанный минимализм API, и на собеседовании «а смапьте OptionalInt» — проверка, работал ли человек с ним руками.

### 10.9 Optional

#### Контракт: для чего он существует

`Optional` придуман как **тип возвращаемого значения** для методов, у которых «не нашлось» — штатный исход. До него сигнатура `Cat findByName(String)` молчала о том, что вернётся null, — и NPE прилетал за три вызова от места ошибки. `Optional<Cat> findByName(String)` делает отсутствие частью типа: игнорировать его не выйдет, компилятор заставит развернуть.

java

```java
Optional<Cat> cat = Optional.of(barsik);          // NPE сразу, если null — «здесь null невозможен»
Optional<Cat> maybe = Optional.ofNullable(found); // null → Optional.empty
Optional<Cat> none = Optional.empty();
```

`of` против `ofNullable` — это заявление о намерениях: `of` падает **в момент создания** (проверено: `Optional.of(null)` → NPE), превращая «null пролез туда, где его быть не должно» в ошибку с честным стектрейсом.

#### Извлечение: лестница от плохого к хорошему

java

```java
Optional<Cat> found = shelter.stream().filter(c -> c.name().equals("Барсик")).findFirst();

found.get();                          // NoSuchElementException, если пусто — почти как NPE
found.orElseThrow();                  // то же, но читается как намерение (Java 10)
found.orElseThrow(CatNotFoundException::new); // своё исключение
found.orElse(DEFAULT_CAT);            // дефолт
found.orElseGet(Cat::createDefault);  // ленивый дефолт
found.ifPresent(c -> feed(c));        // действие, если есть
found.ifPresentOrElse(this::feed, this::logMissing); // обе ветки (Java 9)
```

Антипаттерн, ради искоренения которого всё затевалось:

java

```java
if (found.isPresent()) { feed(found.get()); }  // это тот же if (x != null),
                                               // только дороже и многословнее
```

`isPresent() + get()` легален, но означает, что `Optional` используют как «null с приседаниями». Почти всегда есть прямой метод: `ifPresent`, `map`, `orElse`.

Разница `orElse` и `orElseGet` — не стилистическая. Прогон JDK 21 на **непустом** `Optional`:

java

```java
Optional.of("Барсик").orElse(expensive());        // expensive() ВЫЗВАН
Optional.of("Барсик").orElseGet(() -> expensive()); // не вызван
```

`orElse(x)` — обычный метод: аргумент вычисляется до вызова, всегда, по правилам языка (глава 1.8 — Java strict evaluation). Дорогой дефолт, поход в базу, а хуже того — дефолт с побочным эффектом (`orElse(createAndSaveCat())` — запись в базу на каждый вызов, включая успешные!) — только `orElseGet` с его ленивым `Supplier`. Правило: константа — `orElse`, всё, что _вычисляется_, — `orElseGet`.

#### Трансформации: map, flatMap, filter

`Optional` — «стрим из максимум одного элемента», и у него те же комбинаторы:

java

```java
record Cat(String name, Optional<Cat> mother) {}

String motherName = kitten.mother()        // Optional<Cat>
    .map(Cat::name)                        // Optional<String>; пусто → пусто
    .orElse("неизвестна");
```

Цепочка вместо лесенки `if (x != null && x.getY() != null …)`. Ключевая любезность `map`: если функция вернула null, результат — `Optional.empty`, **не NPE** (проверено) — null-возвращающие legacy-методы встраиваются в цепочку безопасно. `flatMap` — когда функция сама возвращает `Optional` (иначе получишь `Optional<Optional<Cat>>`): `kitten.mother().flatMap(Cat::mother)` — бабушка, если известна. `filter` превращает несоответствие в пустоту: `found.filter(c -> c.weight() > 5)`. `or` (Java 9) — фолбэк на другой `Optional`: `findInShelter(n).or(() -> findInArchive(n))`.

Идиома стыковки со стримами (Java 9) — вытащить значения из стрима опционалов:

java

```java
cats.stream()
    .map(Cat::mother)            // Stream<Optional<Cat>>
    .flatMap(Optional::stream)   // Stream<Cat> — пустые исчезли
    .toList();
```

`Optional.stream()` даёт стрим из 0 или 1 элемента — `flatMap` склеивает, пустые растворяются. До Java 9 писали `filter(Optional::isPresent).map(Optional::get)` — рабочее, но двухходовое.

#### Где Optional запрещён

Контракт из Javadoc и Effective Java (Item 55): `Optional` — для **возвращаемых значений**. Не для полей (не сериализуется — см. 🔬, плюс три состояния поля: значение/empty/null — хуже, чем было), не для параметров (вызывающий обязан заворачивать, а внутри всё равно разворачивать — шум без пользы; перегрузка или `@Nullable` честнее), не для элементов коллекций (`Map<String, Optional<Cat>>` — «может быть, есть, а может, есть пусто»), не для «всего подряд вместо null» — детальные правила в главе 20.6.

> [!tip] 💡 Из C#  
> Соблазнительная параллель `Optional<T>` ≈ `Nullable<T>` — **ложный друг**. `Nullable<T>` работает только со структурами и решает проблему «value type не умеет быть null»; `Optional` — с ссылочными типами, у которых null и так есть, и решает проблему _коммуникации_ отсутствия в сигнатуре. Ближайший честный родственник — nullable reference types (`Cat?`) из C# 8: та же цель «отсутствие видно в типе», но встроенная в компилятор с flow-анализом, а не библиотечная. Отсюда и главное различие практик: в C# аннотируешь тип и пишешь `?.`/`??`, в Java — оборачиваешь возврат и цепляешь `map`/`orElse`. И у `Optional` нет аналога `??=` или паттернов — это просто класс.

> [!example] 🔬 Глубже / на синьора  
> **`Optional` — value-based class.** Javadoc помечает его (как и `Integer`, `Duration`) value-based: идентичность экземпляра — не часть контракта. Практически: `==` между Optional бессмыслен (заводская `Optional.empty()` — синглтон, но полагаться на это нельзя), synchronized на Optional — ошибка, а начиная с JEP 390 (Java 16) JVM вправе выдавать предупреждения за identity-sensitive операции над value-based классами. Это подготовка почвы под Valhalla: value-based классы — кандидаты в value objects, где идентичности не будет физически.
> 
> **Не `Serializable` — осознанно.** Экспертная группа прямо решила не делать `Optional` сериализуемым, чтобы не поощрять его в полях (позиция Brian Goetz, обсуждение в lambda-dev). Поле `Optional<Cat> mother` в `Serializable`-классе даст `NotSerializableException` в рантайме — ещё один довод контракта «только возвращаемые значения».
> 
> **Стоимость.** `Optional` — полноценный объект в куче: заголовок + ссылка, обёртка на каждый вызов метода-возвращателя. В обычном коде это шум на фоне логики; в горячем цикле на миллионы итераций — заметный аллокационный трафик. Escape analysis (глава 11.9) часто скаляризует короткоживущий Optional, но это оптимизация JIT, а не гарантия. Отсюда прагматика: публичные API сервисного слоя — Optional; внутренние горячие пути — null с `@Nullable` и локальной проверкой.
> 
> **Почему у `OptionalInt` нет `map`.** Полная алгебра потребовала бы `map(IntUnaryOperator)`, `mapToObj`, `mapToLong`… — на три класса это комбинаторный взрыв API ради редких сценариев: primitive-опционалы живут одну строку между терминалом стрима и `orElse`/`getAsInt`. Минимализм здесь — цена рукописной моноспециализации, всё тот же erasure-налог.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`orElse` с вычисляемым аргументом** — вычисляется всегда, даже когда значение есть. Особо злая форма — побочный эффект в дефолте: `orElse(repository.createDefault())` пишет в базу на каждом вызове. Всё вычислимое — в `orElseGet`.
> 2. **`Optional.get()`/`orElseThrow()` без уверенности** — `NoSuchElementException: No value present`; тот же NPE, вид сбоку. `get` без предшествующей проверки — маркер на ревью; предпочитай `orElseThrow(своё исключение)` — хоть стектрейс осмысленный.
> 3. **`Optional` в полях, параметрах, коллекциях** — против контракта: поля не сериализуются и получают три состояния (включая null-Optional!), параметры заставляют всех заворачивать, `Optional` в мапе — двусмысленность отсутствия. Возвращаемые значения — единственный дом.
> 4. **`Stream<Integer>` там, где нужен `IntStream`** — боксинг на каждый элемент; на агрегациях больших объёмов — разница в десятки раз (замер выше). Симптом в коде: `.map(x -> x * 2).reduce(0, Integer::sum)` вместо `.mapToInt(...).sum()`.
> 5. **`average().orElse(0.0)` не глядя** — ноль как «среднее» пустого набора искажает статистику (нулевой средний вес котиков ≠ котиков нет). Пустой случай — отдельная ветка через `ifPresentOrElse` или honest `OptionalDouble` наружу.
> 6. **`==` между `Optional`** — value-based класс, идентичность вне контракта: `Optional.of(a) == Optional.of(a)` — false, а `empty() == empty()` — true, но это деталь реализации. Только `equals` (он сравнивает содержимое).

### На собесе

> [!question]- Зачем нужны `IntStream`/`LongStream`/`DoubleStream`? / Why do `IntStream`/`LongStream`/`DoubleStream` exist?  
> **RU:** Из-за type erasure `Stream<int>` невозможен, а `Stream<Integer>` боксит каждый элемент: аллокация, указатели, промахи кэша — на числовых агрегациях это разы и десятки раз. Примитивные стримы держат элементы как примитивы, используют специализированные функциональные интерфейсы (`IntPredicate`) и дают числовые терминалы: `sum`, `average`, `summaryStatistics`. Мосты: `mapToInt`/`mapToObj`/`boxed`.  
> **EN:** Type erasure makes `Stream<int>` impossible, and `Stream<Integer>` boxes every element: allocations, pointer chasing, cache misses — many-fold slowdowns on numeric aggregation. Primitive streams keep elements as primitives, use specialized functional interfaces (`IntPredicate`), and provide numeric terminals: `sum`, `average`, `summaryStatistics`. Bridges: `mapToInt`/`mapToObj`/`boxed`.  
> **Follow-up:** «Почему в C# такого нет?» → reified generics: `IEnumerable<int>` специализируется JIT-ом; `IntStream` — рукописная замена того, что там делает компилятор.

> [!question]- В чём разница между `orElse` и `orElseGet`? / What's the difference between `orElse` and `orElseGet`?  
> **RU:** `orElse(x)` — аргумент вычисляется всегда, до вызова метода, по обычным правилам вычисления аргументов — даже если Optional непуст. `orElseGet(supplier)` вызывает supplier только для пустого. Для константы разницы нет; для вычисляемого дефолта `orElse` — лишняя работа, а с побочным эффектом (создание записи в базе) — баг: эффект срабатывает и при наличии значения.  
> **EN:** With `orElse(x)` the argument is always evaluated, before the method call, by normal argument-evaluation rules — even when the Optional is non-empty. `orElseGet(supplier)` invokes the supplier only when empty. For a constant it doesn't matter; for a computed default `orElse` is wasted work, and with a side effect (creating a DB record) it's a bug: the effect fires even when the value is present.  
> **Ловушка слабого ответа:** «orElse для простых случаев, orElseGet для сложных» — без механизма (strict evaluation аргументов) ответ не засчитают.

> [!question]- Для чего предназначен `Optional` и где его использовать нельзя? / What is `Optional` for, and where must it not be used?  
> **RU:** Это тип возвращаемого значения для методов, где «не найдено» — штатный исход: отсутствие становится частью сигнатуры, и вызывающий обязан его обработать. Не использовать: в полях (не Serializable, три состояния включая null-Optional), в параметрах (шум: заворачивать снаружи, разворачивать внутри), в коллекциях. Внутри метода для локальной логики он тоже избыточен — там достаточно проверки на null.  
> **EN:** It's a return type for methods where "not found" is a normal outcome: absence becomes part of the signature and the caller is forced to handle it. Don't use it: for fields (not Serializable, three states including a null Optional), for parameters (noise: wrap outside, unwrap inside), or in collections. Inside a method for local logic it's also overkill — a null check suffices there.  
> **Follow-up:** «Почему его специально не сделали Serializable?» → чтобы не поощрять в полях; позиция авторов API.

> [!question]- Чем `Optional.of` отличается от `Optional.ofNullable`? / `Optional.of` vs `Optional.ofNullable`?  
> **RU:** `of(null)` бросает NPE немедленно, `ofNullable(null)` возвращает `empty`. Это не дубликаты, а два заявления: `of` — «null здесь невозможен, если пришёл — это баг, падаем в точке создания с честным стектрейсом»; `ofNullable` — «null ожидаем и легально означает отсутствие». Механический выбор `ofNullable` везде прячет баги: null проползает дальше как тихий `empty`.  
> **EN:** `of(null)` throws NPE immediately; `ofNullable(null)` returns `empty`. They're two different statements, not duplicates: `of` says "null is impossible here — if it arrives, that's a bug, fail at creation with an honest stack trace"; `ofNullable` says "null is expected and legitimately means absence." Mechanically using `ofNullable` everywhere hides bugs: nulls slip through as silent `empty`.

> [!question]- Как из `Stream<Optional<T>>` получить `Stream<T>` без пустых значений? / How do you turn a `Stream<Optional<T>>` into a `Stream<T>` without the empties?  
> **RU:** С Java 9 — `flatMap(Optional::stream)`: `Optional.stream()` даёт стрим из 0/1 элемента, пустые растворяются при склейке. До Java 9 — `filter(Optional::isPresent).map(Optional::get)`. Первый вариант — одна операция и без `get`.  
> **EN:** Since Java 9 — `flatMap(Optional::stream)`: `Optional.stream()` yields a 0-or-1-element stream, and empties dissolve in the flattening. Pre-Java 9 — `filter(Optional::isPresent).map(Optional::get)`. The modern form is one operation and avoids `get`.

> [!question]- Что вернут `sum()` и `average()` на пустом `IntStream` и почему по-разному? / What do `sum()` and `average()` return on an empty `IntStream`, and why differently?  
> **RU:** `sum()` — `0`: сумма пустого множества слагаемых определена и равна нулю. `average()` — `OptionalDouble.empty`: среднее нуля элементов не определено (деление на ноль), и вернуть 0.0 значило бы соврать. Общий паттерн API: где у пустоты есть математически честный ответ — голое значение, где нет — Optional (`max`, `min` — тоже `OptionalInt`).  
> **EN:** `sum()` returns `0`: the empty sum is well-defined. `average()` returns `OptionalDouble.empty`: the mean of zero elements is undefined (division by zero), and returning 0.0 would be a lie. The API pattern: where emptiness has a mathematically honest answer — a bare value; where it doesn't — an Optional (`max`, `min` are `OptionalInt` too).

> [!question]- Что напечатает этот код? / What does this print?
> 
> java
> 
> ```java
> Optional<String> name = Optional.of("Барсик");
> System.out.println(name.orElse(fetchDefault()));   // fetchDefault логирует "fetched!"
> System.out.println(name.map(s -> (String) null));
> ```
> 
> **RU:** Сначала в лог упадёт `fetched!` — аргумент `orElse` вычисляется до вызова, непустота Optional не спасает, — затем напечатается `Барсик`. Вторая строка — `Optional.empty`: `map` перехватывает null от функции и превращает в пустоту, NPE не будет. Слабые ответы: «fetchDefault не вызовется — значение же есть» и «вторая строка бросит NPE».  
> **EN:** First, `fetched!` hits the log — `orElse`'s argument is evaluated before the call; the Optional being non-empty doesn't help — then `Барсик` prints. The second line prints `Optional.empty`: `map` intercepts the function's null and turns it into emptiness, no NPE. Weak answers: "fetchDefault won't run — there's a value" and "the second line throws NPE."

> [!question]- Почему `Optional<Integer>` в стриме — плохая идея и что вместо него? / Why is `Optional<Integer>` in a stream pipeline a bad idea, and what instead?  
> **RU:** Двойная обёртка над int: боксинг в `Integer` плюс объект Optional — аллокационный трафик на ровном месте. Примитивные терминалы возвращают специализированные `OptionalInt`/`OptionalLong`/`OptionalDouble` без боксинга. Оговорка: у них нет `map`/`filter`/`flatMap` — только извлечение (`getAsInt`, `orElse`, `ifPresent`); нужны трансформации — конвертируй или забирай значение сразу.  
> **EN:** It double-wraps an int: boxing into `Integer` plus the Optional object — gratuitous allocation traffic. Primitive terminals return specialized `OptionalInt`/`OptionalLong`/`OptionalDouble` with no boxing. The caveat: they have no `map`/`filter`/`flatMap` — extraction only (`getAsInt`, `orElse`, `ifPresent`); if you need transformations, convert or extract immediately.  
> **Senior add-on:** `Optional` и его примитивные братья — value-based classes: `==`, идентичность и синхронизация на них — вне контракта (JEP 390 добавил предупреждения JVM).
