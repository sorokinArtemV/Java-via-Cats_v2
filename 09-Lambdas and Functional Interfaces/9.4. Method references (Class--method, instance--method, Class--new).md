Посмотри на эту лямбду: `cat -> System.out.println(cat)`. Она не делает ничего своего — просто берёт аргумент и передаёт его в уже существующий метод. Параметр `cat` объявлен только чтобы тут же его переслать. Java позволяет убрать этот шум: `System.out::println`. Это **ссылка на метод** (method reference) — лямбда, всё тело которой сводится к вызову одного уже написанного метода. Ты не пишешь тело, ты показываешь пальцем на нужный метод, а компилятор сам достраивает лямбду вокруг него. В C# ты это знаешь как **method group** (`list.ForEach(Console.WriteLine)`).

> [!abstract] Ментальная модель  
> `::` означает «возьми вот этот метод как реализацию функционального интерфейса». Ссылка на метод — это лямбда, тело которой — ровно один вызов существующего метода, без добавлений. Если лямбда только пересылает аргументы в один метод — её почти всегда можно заменить на ссылку.

### Четыре формы

Синтаксически форм четыре, и различаются они тем, _откуда берётся объект-приёмник_ вызова.

![[Pasted image 20260715121353.png]]

Пройдём по формам (весь код проверен на JDK 21):

**1. Статический метод — `Class::staticMethod`.** Самая простая: аргументы лямбды становятся аргументами метода.

```java
Function<String,Integer> parse = Integer::parseInt;   // s -> Integer.parseInt(s)
```

**2. Связанный вызов (bound) — `объект::method`.** Приёмник — конкретный, уже существующий объект. Он «зашит» в ссылку, аргументы лямбды идут в метод.

```java
List<String> log = new ArrayList<>();
Consumer<String> toLog = log::add;        // s -> log.add(s), приёмник = log
System.out.println("Мяу")::length;        // (нельзя так, но идея: System.out::println — приёмник System.out
List.of("a","b").forEach(System.out::println);   // приёмник — System.out
```

**3. Несвязанный вызов (unbound) — `Class::instanceMethod`.** Вот здесь ловушка. Синтаксис как у статического (`Class::method`), но `method` — **экземплярный**. Приёмник не зашит — им становится **первый аргумент лямбды**.

```java
Function<String,String> upper = String::toUpperCase;   // s -> s.toUpperCase()
```

**4. Конструктор — `Class::new`.** Аргументы лямбды становятся аргументами конструктора. Есть и вариант для массивов.

```java
Supplier<List<String>>       listFactory = ArrayList::new;   // () -> new ArrayList<>()
BiFunction<String,Integer,Cat> catFactory = Cat::new;        // (n,a) -> new Cat(n,a)
IntFunction<Cat[]>           arrayFactory = Cat[]::new;      // n  -> new Cat[n]
```

#### Главная ловушка: связанный против несвязанного

Один и тот же метод `toUpperCase` даёт **две разные ссылки** в зависимости от того, зашит приёмник или нет. Это самый частый вопрос на собесе про method references.

![[Pasted image 20260715121412.png]]

Проверено — оба варианта компилируются и работают:

```java
Function<String,String> upper = String::toUpperCase;    // несвязанный → "purr".toUpperCase()
Supplier<String>        shout = "meow"::toUpperCase;     // связанный   → "MEOW"

upper.apply("purr");   // "PURR" — приёмник пришёл аргументом
shout.get();           // "MEOW" — приёмник зашит
```

Несвязанная форма работает и с методами, у которых есть свои аргументы — тогда первый параметр лямбды идёт в приёмник, а остальные — в метод:

```java
BiFunction<String,String,Boolean> starts = String::startsWith;   // (a,b) -> a.startsWith(b)
starts.apply("Мурзик", "Мур");   // true — "Мурзик".startsWith("Мур")
```

> [!tip] 💡 Из C#  
> Ближайший аналог — **method group conversion**: `names.Select(int.Parse)`, `list.ForEach(Console.WriteLine)`. Работает похоже и покрывает статические (`Class::static`) и связанные (`obj::method`) формы.  
> Но есть два отличия:
> 
> - **Несвязанной формы `Class::instanceMethod` в C# в явном виде нет.** В Java `String::toUpperCase` — это `Function<String,String>`, где приёмник «уезжает» в первый аргумент. В C# ты бы написал лямбду `s => s.ToUpperCase()` явно; method group здесь не выручает так же элегантно.
> - **Ссылки на конструктор `Class::new` в C# нет вообще.** Нельзя написать `new Cat` как method group — только лямбда `() => new Cat(...)`. `Cat::new` и `Cat[]::new` — чисто джавовая приятность.

> [!example]- 🔬 Глубже / на синьора  
> **`Class::method` двусмысленно по своей природе.** Одна и та же запись может значить и статический вызов, и несвязанный экземплярный — компилятор выбирает по тому, какой метод существует и подходит по сигнатуре к target-типу. Если в классе есть **и** статический, **и** экземплярный метод с одним именем и оба подходят — ошибка компиляции (проверено на JDK 21):
> 
> java
> 
> ```java
> static class Box {
>     static int size(Box b) { return 1; }   // static: Box -> int
>     int size()            { return 2; }    // instance: (this) -> int
> }
> Function<Box,Integer> f = Box::size;
> // error: reference to size is ambiguous
> //   both method size(Box) in Box and method size() in Box match
> ```
> 
> На практике это редкость, но знать про источник неоднозначности полезно.
> 
> **Как это компилируется.** Как и лямбда (9.3), ссылка на метод разворачивается через `invokedynamic` + `LambdaMetafactory` на этапе связывания, без отдельного `.class`. Детали — в 9.8. Важное следствие уже сейчас: связанная ссылка `obj::method` **захватывает** `obj` в момент создания. Если `obj` — поле, а не локальная переменная, ссылка держит `this` внешнего объекта, что иногда мешает сборке мусора (та же механика захвата — тема 9.5).
> 
> **Момент вычисления приёмника.** В `getExpensive()::run` выражение `getExpensive()` вычисляется **сразу**, при создании ссылки, а не при каждом вызове. Это отличает связанную ссылку от лямбды `() -> getExpensive().run()`, которая звала бы `getExpensive()` каждый раз. Тонко, но на синьорских вопросах всплывает.

> [!warning] ⚠️ Подводные камни
> 
> - **`Class::method` — статический или несвязанный экземплярный?** Зависит от того, какой метод существует. При коллизии имён — ошибка компиляции.
> - **Порядок приёмника в несвязанной форме.** `String::compareTo` — это `(a,b) -> a.compareTo(b)`, а не наоборот. Первый параметр — всегда приёмник.
> - **Связанная ссылка вычисляет приёмник немедленно.** `obj::m` фиксирует `obj` в момент создания; если хотел ленивость — нужна лямбда.
> - **`this::method` и `super::method`** — валидные связанные ссылки на методы текущего/родительского объекта; часто встречаются внутри классов.
> - **`null`-приёмник.** `String s = null; Supplier<Integer> len = s::length;` — `NullPointerException` бросается **при создании ссылки**, не при вызове `.get()`. Приёмник разыменовывается сразу.
> - **Перегрузка целевого метода.** Если метод перегружен, компилятор выбирает перегрузку по target-типу функционального интерфейса; при неоднозначности — ошибка, лечится явным типом.

### На собесе

> [!question]- Назови формы ссылок на метод.  
> **RU:** Четыре: статическая `Class::staticMethod`; связанная `объект::method` (приёмник зафиксирован); несвязанная `Class::instanceMethod` (приёмник — первый аргумент лямбды); конструктор `Class::new` (плюс `Type[]::new` для массивов).  
> **EN:** Four: static `Class::staticMethod`; bound `instance::method` (fixed receiver); unbound `Class::instanceMethod` (receiver is the first lambda argument); constructor `Class::new` (plus `Type[]::new` for arrays).

> [!question]- В чём разница `String::toUpperCase` и `"hi"::toUpperCase`?  
> **RU:** Первая — несвязанная: тип `Function<String,String>`, приёмник приходит первым аргументом (`s -> s.toUpperCase()`). Вторая — связанная: приёмник зафиксирован на `"hi"`, поэтому это `Supplier<String>` без аргументов (`() -> "hi".toUpperCase()`). Один метод, две формы — разница в том, откуда берётся объект.  
> **EN:** The first is unbound: `Function<String,String>`, the receiver arrives as the first argument. The second is bound: the receiver is fixed to `"hi"`, so it's a `Supplier<String>` with no arguments. Same method, two forms — the difference is where the receiver comes from.
> 
> > [!danger] Ловушка-продолжение  
> > _«А `Collections::sort` и `list::sort` — то же самое?»_ → Да, тот же принцип: `list::sort` связанная (приёмник `list`), `List::sort` — несвязанная. Слабый ответ путает, кто приёмник, и подставляет аргументы в обратном порядке.

> [!question]- `Class::method` — это всегда статический метод?  
> **RU:** Нет. `Class::method` может быть и статическим методом, и экземплярным в несвязанной форме — компилятор различает по тому, какой метод есть в классе и подходит по сигнатуре. Если есть оба одноимённых и оба подходят — ошибка неоднозначности.  
> **EN:** No. `Class::method` can be either a static method or an instance method in unbound form — the compiler decides by which method exists and fits the signature. If both same-named methods exist and both match, it's an ambiguity error.

> [!question]- Что такое `Cat::new` и есть ли аналог в C#?  
> **RU:** Это ссылка на конструктор: аргументы функционального интерфейса идут в конструктор. `Cat::new` при target-типе `BiFunction<String,Integer,Cat>` эквивалентно `(n,a) -> new Cat(n,a)`. В C# прямого аналога нет — там пишут лямбду `() => new Cat(...)`.  
> **EN:** A constructor reference: the functional interface's arguments feed the constructor. `Cat::new` with a `BiFunction<String,Integer,Cat>` target equals `(n,a) -> new Cat(n,a)`. C# has no direct equivalent — you write `() => new Cat(...)`.

> [!question]- Когда `System.out::println` вычисляет `System.out` — при создании ссылки или при каждом вызове?  
> **RU:** При создании. Связанная ссылка вычисляет выражение-приёмник немедленно и фиксирует его. Поэтому если приёмник `null`, `NPE` прилетит сразу при создании ссылки, а не при вызове. Для ленивого вычисления нужна лямбда.  
> **EN:** At creation. A bound reference evaluates the receiver expression immediately and captures it. So a `null` receiver throws `NPE` at reference creation, not at invocation. For lazy evaluation, use a lambda.

> [!question]- Когда method reference читается хуже лямбды?  
> **RU:** Когда порядок аргументов или приёмник становится неочевидным. `Class::instanceMethod` с несколькими аргументами (`String::regionMatches`) читается тяжело — лямбда с явными именами понятнее. Ссылка хороша, когда она буквально «имя того, что делаем»: `Cat::name`, `String::trim`, `System.out::println`.  
> **EN:** When argument order or the receiver becomes non-obvious. `Class::instanceMethod` with several arguments reads poorly — an explicit-named lambda is clearer. References shine when they read as "the name of what we do": `Cat::name`, `String::trim`.
