В 9.1 мы увидели: чтобы передать поведение, нужен функциональный интерфейс. Но писать свой `CatAction`, `CatCheck`, `CatFactory` под каждый чих — это ровно та боль, от которой в C# спасают `Func` и `Action`. Java 8 принесла свой набор «канонических делегатов» — пакет `java.util.function`. Там ~43 интерфейса, но выучить надо не список, а **четыре формы** и правило, по которому из них выводится всё остальное. Дальше `Stream`, `Optional`, `CompletableFuture` — всё построено на этих же типах, так что это фундамент половины стандартной библиотеки.

> [!abstract] Ментальная модель  
> Всё сводится к двум вопросам: **сколько значений на входе** и **что на выходе**. Ничего на входе → `Supplier`. Ничего на выходе → `Consumer`. Из одного делаем другое → `Function`. На выходе `boolean` → `Predicate`. Остальные интерфейсы — это те же четыре с приставками.

### Четыре базовые формы

Проще всего думать о них как о маленьких «машинах»: что-то заходит, что-то выходит.

![[Pasted image 20260715121150.png]]

Обрати внимание: у каждого интерфейса **своё имя абстрактного метода** — `get` / `accept` / `apply` / `test`. Это не случайность: раз это разные типы (см. 9.1), метод у каждого зовётся по-своему, и по имени сразу видно, с какой формой работаешь.

```java
Supplier<Cat>       factory  = () -> new Cat("Барсик", 2);   // фабрика / ленивое значение
Consumer<Cat>       print    = c  -> System.out.println(c);   // побочный эффект
Function<Cat,String> nameOf  = c  -> c.name();                // преобразование
Predicate<Cat>      isKitten = c  -> c.age() < 1;             // условие

factory.get();            // -> Cat
print.accept(murzik);     // -> ничего
nameOf.apply(murzik);     // -> "Мурзик"
isKitten.test(murzik);    // -> false
```

> [!tip] 💡 Из C#  
> Соответствие почти один-в-один, кроме имён:
> 
> - `Function<T,R>` ↔ `Func<T,R>` (`apply` вместо `Invoke`)
> - `Consumer<T>` ↔ `Action<T>`
> - `Supplier<T>` ↔ `Func<T>` (в C# отдельного `Supplier` нет — это `Func` без аргументов)
> - `Predicate<T>` ↔ `Predicate<T>` (единственное совпадение и по смыслу, и по имени)
> 
> Ключевое расхождение — в C# один `Func<>`/`Action<>` покрывает **любую** арность через перегрузки по числу параметров (`Func<T1,T2,T3,TResult>`). В Java обобщённого «`Func` на N аргументов» нет: базовые интерфейсы покрывают только 1 и 2 аргумента (`Function`, `BiFunction`). Нужно три входа — пиши свой функциональный интерфейс. И ещё: в Java из-за стирания типов нет специализаций под примитивы «даром» — под них заведены отдельные интерфейсы (об этом ниже), тогда как в C# `Func<int,int>` не боксит сам по себе.

### Как выводится всё остальное: приставки

Четыре формы — это основа. Остальные ~39 интерфейсов получаются добавлением приставок по трём осям. Запоминать их поштучно не надо — надо знать правило сборки имени.

|Ось|Приставка|Что меняет|Пример|
|---|---|---|---|
|Два входа|`Bi‑`|один аргумент → два|`BiFunction<T,U,R>`, `BiConsumer<T,U>`, `BiPredicate<T,U>`|
|Одинаковые типы|`‑Operator`|`Function<T,T>` → короче|`UnaryOperator<T>`, `BinaryOperator<T>`|
|Примитив на входе|`Int‑`/`Long‑`/`Double‑`|`T` → примитив|`IntFunction<R>`, `IntPredicate`, `IntConsumer`|
|Примитив на выходе|`To{Int\|Long\|Double}‑`|`R` → примитив|`ToIntFunction<T>`, `ToDoubleBiFunction<T,U>`|

Отсюда сразу два практических вывода:

**`UnaryOperator<T>` и `BinaryOperator<T>`** — это просто `Function<T,T>` и `BiFunction<T,T,T>`, когда вход и выход одного типа. Причём буквально: `UnaryOperator<T> extends Function<T,T>`, поэтому его можно передавать всюду, где ждут `Function<T,T>` (проверено — присваивание компилируется):

```java
UnaryOperator<String> upper = s -> s.toUpperCase();
Function<String,String> asFunction = upper;   // UnaryOperator IS-A Function<T,T>
BinaryOperator<Integer> sum = (a, b) -> a + b;
```

**`BiSupplier` не существует** — и это логично: у `Supplier` нет входа, а «два входа» относятся ко входу. Приставка `Bi‑` есть только там, где есть что удваивать: `BiFunction`, `BiConsumer`, `BiPredicate`.

> [!example]- 🔬 Глубже / на синьора: зачем нужны примитивные специализации  
> Из-за **стирания типов** (глава 7) дженерики в Java работают только со ссылочными типами. `Function<Cat, Integer>` физически не может вернуть `int` — только `Integer`. А значит, каждый `int` **боксится** в объект `Integer` в куче на выходе, и разбоксивается при чтении. В горячем цикле на миллионы элементов это заметная нагрузка на GC и кэш процессора (механика бокса и `Integer` cache — тема 1.4).
> 
> Ровно ради этого заведены специализации под три «тяжёлых» примитива — `int`, `long`, `double`:
> 
> java
> 
> ```java
> Function<Cat,Integer> ageBoxed = c -> c.age();   // возвращает Integer — БОКСИТ
> ToIntFunction<Cat>    ageRaw   = c -> c.age();   // возвращает int   — без бокса
> 
> IntUnaryOperator doubler = n -> n * 2;   // int -> int, ни одного объекта
> IntPredicate     isAdult = n -> n >= 1;  // int -> boolean
> IntFunction<String> label = n -> "cat#" + n;   // int -> R (ссылочный)
> ```
> 
> Именно поэтому в `Stream` есть `IntStream`/`LongStream`/`DoubleStream` и методы `mapToInt`/`mapToObj` — это та же история про уход от бокса (детали в 10.7). Правило на мид+: **в горячем пути с примитивами бери `IntStream`/`ToIntFunction`, а не `Stream<Integer>`/`Function<T,Integer>`**. В обычном коде разница незаметна — не усложняй без причины.
> 
> Специализаций под `byte`/`short`/`char`/`float` нет намеренно: их расширяют до `int`/`double`, чтобы не плодить сотни интерфейсов.

Комбинирование этих интерфейсов (`andThen`, `compose`, `and`/`or`/`negate`) — отдельная большая тема, это **9.7**. Здесь важно только знать, что такие `default`-методы у них есть, и именно они превращают четыре примитива в конструктор пайплайнов.

> [!warning] ⚠️ Подводные камни
> 
> - **Имя метода зависит от интерфейса, а не от «вызвать функцию».** `apply`, `accept`, `get`, `test`, плюс `applyAsInt`/`getAsInt` у примитивных. Забыл — посмотри форму: у `Predicate` всегда `test`.
> - **`Function<T,Integer>` тихо боксит.** Компилятор не предупредит. Если это горячий путь — бери `ToIntFunction`/`IntStream`.
> - **`BiSupplier`/`TriFunction` нет.** Больше двух входов — пиши свой `@FunctionalInterface`, стандартного нет.
> - **`Consumer` возвращает `void`, но тело-выражение может вернуть значение — оно отбрасывается** (см. 9.3): `Consumer<Cat> c = cat -> list.add(cat)` компилируется, `boolean` от `add` игнорируется.
> - **`Predicate<T>` ≠ `Function<T,Boolean>`.** Первый возвращает примитив `boolean`, второй — боксит в `Boolean`. Для условий всегда `Predicate`.
> - **`null` в `Predicate`/`Function`** — интерфейс не проверяет аргумент за тебя; `NullPointerException` внутри тела твоя забота (или используй `Predicate` с явной проверкой).

### На собесе

> [!question]- Назови базовые функциональные интерфейсы из `java.util.function` и их методы.  
> **RU:** Четыре формы: `Supplier<T>` — `T get()`, ничего на входе; `Consumer<T>` — `void accept(T)`, ничего на выходе; `Function<T,R>` — `R apply(T)`, преобразование; `Predicate<T>` — `boolean test(T)`, условие. Плюс версии на два аргумента с приставкой `Bi‑` (`BiFunction`, `BiConsumer`, `BiPredicate`) и «операторы» для одинаковых типов (`UnaryOperator<T>`, `BinaryOperator<T>`).  
> **EN:** Four shapes: `Supplier<T>` — `T get()`, no input; `Consumer<T>` — `void accept(T)`, no output; `Function<T,R>` — `R apply(T)`, transform; `Predicate<T>` — `boolean test(T)`, condition. Plus two-arg `Bi‑` variants and same-type `Operator` variants.

> [!question]- Зачем `ToIntFunction`, если есть `Function<T,Integer>`?  
> **RU:** Чтобы избежать бокса. Из-за стирания типов `Function<T,Integer>` возвращает объект `Integer`, то есть боксит каждый результат. `ToIntFunction<T>` возвращает примитив `int` — ни одного объекта, ни нагрузки на GC. В горячих циклах и стримах это ощутимо, поэтому существуют `IntStream` и `mapToInt`.  
> **EN:** To avoid boxing. Because of type erasure, `Function<T,Integer>` returns an `Integer` object — it boxes every result. `ToIntFunction<T>` returns a primitive `int` — no objects, no GC pressure. It matters in hot loops and streams, which is why `IntStream` and `mapToInt` exist.
> 
> > [!danger] Ловушка-продолжение  
> > _«А под `char` или `float` такие специализации есть?»_ → **Нет**, только под `int`/`long`/`double`. Остальные примитивы расширяют до них. Слабый ответ — «под все примитивы»; это выдаёт, что человек не заглядывал в пакет.

> [!question]- Чем `Predicate<T>` отличается от `Function<T, Boolean>`?  
> **RU:** `Predicate.test` возвращает примитив `boolean`, а `Function<T,Boolean>` — боксированный `Boolean`. Функционально похоже, но `Predicate` не боксит и даёт `default`-методы `and`/`or`/`negate` для комбинирования. Для условий всегда используют `Predicate`.  
> **EN:** `Predicate.test` returns a primitive `boolean`; `Function<T,Boolean>` returns a boxed `Boolean`. Similar in effect, but `Predicate` avoids boxing and provides `and`/`or`/`negate` for composition. Always use `Predicate` for conditions.

> [!question]- Как связаны `UnaryOperator` и `Function`?  
> **RU:** `UnaryOperator<T> extends Function<T,T>` — это частный случай, когда вход и выход одного типа. Поэтому `UnaryOperator<T>` можно передать всюду, где ждут `Function<T,T>`, но не наоборот. Аналогично `BinaryOperator<T> extends BiFunction<T,T,T>`.  
> **EN:** `UnaryOperator<T> extends Function<T,T>` — a special case where input and output share a type. So a `UnaryOperator<T>` fits anywhere a `Function<T,T>` is expected, but not the reverse. Same for `BinaryOperator<T> extends BiFunction<T,T,T>`.

> [!question]- Почему в Java нет `Func` на любое число аргументов, как в C#?  
> **RU:** Из-за стирания типов и дизайна: в Java нельзя перегружать дженерик по числу type-параметров так, чтобы это давало один тип на все арности, да и общего «типа функции» в языке нет. Поэтому стандартных интерфейсов хватает на 1 и 2 аргумента (`Function`, `BiFunction`); для трёх и более пишут свой `@FunctionalInterface`.  
> **EN:** By design and because of erasure: Java has no single function type spanning all arities, and the standard set only covers 1 and 2 arguments (`Function`, `BiFunction`). For three or more, you declare your own `@FunctionalInterface`.

#### `Supplier` vs `Callable`: одинаковая форма, разное поведение

Как только supplier-образная лямбда полезет делать что-то реальное — прочитать файл, сходить в сеть, — она упрётся в стену. `Supplier<String> s = () -> Files.readString(path)` не скомпилируется, потому что `readString` бросает checked-исключение `IOException`, а `Supplier` их не пускает. И тут выясняется, что в стандартной библиотеке есть второй интерфейс ровно той же формы «ничего на входе, одно значение на выходе» — `Callable<T>` — который checked-исключения как раз пускает. Разница между ними — не про форму, а про исключения, и спрашивают её постоянно.

> [!abstract] Ментальная модель  
> `Supplier<T>` и `Callable<T>` — оба «ноль входов → одно значение». Разница одна: `Supplier.get()` не объявляет `throws`, поэтому checked-исключения в теле запрещены; `Callable.call()` объявлен как `throws Exception`, поэтому checked разрешены. Нужен результат без исключений — `Supplier`. Тело может бросить checked — `Callable`.

`Callable<T>` живёт не в `java.util.function`, а в `java.util.concurrent`, и старше её — появился в Java 5 вместе с `ExecutorService` (тема 24.1), задолго до `Supplier`. Отсюда его метод и подпись:

```java
@FunctionalInterface
public interface Callable<V> {
    V call() throws Exception;   // ← throws Exception в контракте
}
```

Это `throws Exception` и есть вся суть. Сравни, что происходит с одним и тем же телом, которое бросает checked `IOException`:

```java
Supplier<String> s = () -> { throw new IOException("boom"); };   // НЕ компилируется
Callable<String> c = () -> { throw new IOException("boom"); };   // компилируется
```

Первая строка падает на этапе компиляции (проверено, JDK 21):

```
error: unreported exception IOException; must be caught or declared to be thrown
        Supplier<String> s = () -> { throw new IOException("boom"); };
                                     ^
```

Вторая собирается молча — `IOException` укладывается в объявленный `throws Exception` — и исключение прилетает уже в рантайме, при вызове `c.call()`. Механика простая: тело лямбды обязано соответствовать `throws`-клаузе абстрактного метода целевого интерфейса. У `get()` её нет — значит, из тела не должно вылетать ни одного checked. У `call()` она есть — значит, можно.

Практическое следствие: `Callable` — это то, что берёт `ExecutorService.submit`, потому что задача в пуле по определению может упасть с checked-исключением. А `Supplier` берут `Optional.orElseGet`, `CompletableFuture.supplyAsync`, `Stream.generate` — места, где checked неоткуда взяться и не нужно. Когда supplier-образная лямбда должна вызвать что-то checked, а рядом ждут именно `Supplier`, — это ровно та боль, которой посвящена тема 9.6 (лямбды и checked exceptions, `sneaky throws`, обёртки).

> [!warning] ⚠️ Подводные камни
> 
> - **`Callable` и `Supplier` невзаимозаменяемы.** Форма одна, но это разные типы (см. 9.1) — присвоить одно другому нельзя: `Supplier<String> s = someCallable;` даёт `error: incompatible types: Callable<String> cannot be converted to Supplier<String>` (проверено). Только пересоздать лямбдой.
> - **`Runnable` — не «`Callable` без результата» по части исключений.** `Runnable.run()` тоже без `throws`, поэтому checked в него не бросишь — так же, как в `Supplier`. Пара «пускает checked / не пускает» — это `Callable` против всех остальных из `java.util.function` + `Runnable`.
> - **`Callable` бросает `Exception`, а не конкретный тип.** Ловить его придётся широко или разбирать `getCause` после `Future.get`, который заворачивает всё в `ExecutionException` (детали в 24.4).

> [!tip] 💡 Из C#  
> В C# у этого различия аналога нет вовсе — потому что нет checked-исключений. Тип делегата `Func<T>` ничего не знает про то, бросает тело исключение или нет: исключения в C# не часть сигнатуры. То есть `Supplier<T>` и `Callable<T>` оба отображаются на один `Func<T>` — а разница между ними (можно/нельзя checked) в C# просто не выразима на уровне типа. Ближе всего: `Callable<T>` ≈ `Func<T>`, которому «разрешено бросать», но само «разрешено бросать» в C# в типе не кодируется.

> [!question]- В чём разница `Supplier<T>` и `Callable<T>`?  
> RU: Форма одинаковая — ноль входов, одно значение на выходе. Разница в исключениях: `Callable.call()` объявлен `throws Exception`, поэтому тело может бросать checked-исключения; `Supplier.get()` без `throws`, поэтому checked в нём запрещены. Плюс `Callable` живёт в `java.util.concurrent` (с Java 5, для `ExecutorService`), а `Supplier` — в `java.util.function` (с Java 8). Это разные типы, взаимно не присваиваются.  
> EN: Same shape — no input, one output. The difference is exceptions: `Callable.call()` is declared `throws Exception`, so its body may throw checked exceptions; `Supplier.get()` has no `throws`, so checked ones are forbidden. Also `Callable` lives in `java.util.concurrent` (since Java 5, for `ExecutorService`), `Supplier` in `java.util.function` (since Java 8). They're distinct, mutually non-assignable types.

> [!question]- У меня лямбда `() -> Files.readString(path)`, я кладу её в `Supplier<String>` — почему не компилируется?  
> RU: Потому что `Files.readString` бросает checked `IOException`, а `Supplier.get()` не объявляет `throws` — тело `Supplier` не имеет права выпускать checked-исключения. Варианты: обернуть в try/catch внутри лямбды и перебросить unchecked; взять `Callable<String>`, если принимающая сторона его берёт; либо применить приёмы из 9.6. Компилятор скажет `unreported exception IOException; must be caught or declared to be thrown`.  
> EN: Because `Files.readString` throws checked `IOException`, and `Supplier.get()` declares no `throws` — a `Supplier` body may not emit checked exceptions. Options: try/catch inside the lambda and rethrow unchecked; use `Callable<String>` if the caller accepts it; or apply the techniques from 9.6. The compiler says `unreported exception IOException; must be caught or declared to be thrown`.
> 
> > [!danger] Ловушка-продолжение  
> > «А `Runnable` тогда пускает checked, раз он про побочный эффект?» → Нет. `Runnable.run()` тоже без `throws`, checked в него не бросишь — ровно как в `Supplier`. Единственный из ходовых, кто пускает checked, — `Callable`. Слабый ответ — «раз void, то можно бросать что угодно»; путает наличие результата с наличием `throws`.
