У тебя есть маленькие однозадачные функции: одна нормализует строку, другая обрезает, третья валидирует. Хочется собрать из них конвейер, не вкладывая вызовы друг в друга (`validate(trim(normalize(x)))` читается плохо) и не плодя новых лямбд. Ровно для этого у функциональных интерфейсов есть `default`-методы комбинирования — те самые, что мы поминали ещё в 9.1 и 9.2. Они превращают четыре базовые формы (9.2) из «одиночек» в конструктор: из простых функций и предикатов ты складываешь сложные, оставаясь в декларативном стиле.

> [!abstract] Ментальная модель  
> `andThen` — «сначала это, ПОТОМ то» (читается слева направо). `compose` — наоборот, «сначала аргумент, потом this» (математическое `f∘g`, справа налево). Для предикатов — `and`/`or`/`negate`, как обычные `&&`/`||`/`!`, с тем же коротким замыканием.

### Функции: `andThen` против `compose`

Это главная ловушка темы — направление. Возьмём две функции `x+1` и `x*2` и соберём их в обоих порядках. Результаты разные (проверено на JDK 21):

![[Pasted image 20260715121658.png]]

```java
Function<Integer,Integer> plus1  = x -> x + 1;
Function<Integer,Integer> times2 = x -> x * 2;

plus1.andThen(times2).apply(5);   // 12  →  (5+1)*2   : this первым
plus1.compose(times2).apply(5);   // 11  →  (5*2)+1   : аргумент первым
```

Мнемоника: `a.andThen(b)` — «a, а **потом** b», как читается предложение. `a.compose(b)` — как в математике `a∘b`: b внутри, a снаружи. На практике `andThen` используют почти всегда — он совпадает с порядком чтения кода.

Есть ещё `Function.identity()` — функция `x -> x`. Пригождается как «нейтральный элемент» и в `Collectors.toMap`, когда значением должен стать сам элемент (глава 10).

### Предикаты: `and` / `or` / `negate`

Те же комбинаторы, но для условий — и с **коротким замыканием**, как у `&&`/`||` (проверено: при `false` слева `and` не вычисляет правый):

```java
Predicate<String> longName = s -> s.length() > 5;
Predicate<String> startsM   = s -> s.startsWith("М");

longName.and(startsM).test("Мурзик");   // true  — длинное И на "М"
longName.or(startsM).test("Том");        // false — короткое И не на "М"
longName.negate().test("Том");           // true  — НЕ длинное
```

Два полезных `static`-метода: `Predicate.not(p)` (Java 11) отрицает предикат — особенно удобно с method reference, где `.negate()` не напишешь; и `Predicate.isEqual(target)` строит предикат «равно target».

```java
Predicate<String> notLong = Predicate.not(longName);       // Java 11
Predicate<String> isBarsik = Predicate.isEqual("Барсик");
// в стримах: .filter(Predicate.not(String::isBlank))
```

### Consumer: `andThen` для цепочки побочных эффектов

У `Consumer` есть `andThen` — оба потребителя срабатывают, по порядку. `compose` нет: у `Consumer` нет выхода, подавать назад нечего.

```java
Consumer<String> a = log::add;
Consumer<String> b = s -> log.add(s.toUpperCase());
a.andThen(b).accept("meow");   // log == [meow, MEOW]
```

> [!tip] 💡 Из C#  
> В C# нет встроенных `andThen`/`compose` на делегатах — композицию функций пишут вручную (`x => g(f(x))`) или тянут из библиотек. Зато есть `+`/`-` для **multicast-делегатов**: `action1 + action2` — ближайший аналог `Consumer.andThen` (оба вызовутся по порядку). Для предикатов в C# комбинируют обычными `&&`/`||`/`!` внутри лямбды либо строят `Expression`-деревья (в LINQ-провайдерах). Итог: джавовые `default`-комбинаторы — это то, что в C# обычно делаешь руками или через сторонние хелперы.

> [!example]- 🔬 Глубже / на синьора  
> **Типы должны стыковаться.** Сигнатура — `andThen(Function<? super R, ? extends V>)`: выход первой функции (`R`) обязан подходить на вход второй. Wildcards (`? super`/`? extends`, PECS из главы 7) дают гибкость по иерархии, но если промежуточный тип не совпадает — ошибка компиляции. `compose` симметрично требует, чтобы выход аргумента подходил на вход this.
> 
> **Каждый комбинатор — новый объект.** `a.andThen(b)` создаёт новую лямбду-обёртку, которая при вызове дергает обе. Глубокая цепочка из десятков `andThen` — это вложенные вызовы и лишние аллокации; в горячем пути иногда дешевле один явный метод. Но для типичного кода разница незаметна.
> 
> **`null`-аргумент — всегда `NPE`.** `andThen`/`compose`/`and`/`or` документированно бросают `NullPointerException` на `null` (проверено). Это ранняя защита: обёртка сразу требует `Objects.requireNonNull`.
> 
> **`BiFunction.andThen` есть, `compose` — нет.** У `BiFunction` можно постобработать результат (`andThen(Function)`), но «подать назад» два аргумента через `compose` нельзя — поэтому его и не завезли.
> 
> **Порядок и побочные эффекты.** В `p1.and(p2)` при `p1 == false` предикат `p2` **не вызывается** (короткое замыкание). Если в `p2` был важный побочный эффект — он не случится. Держи предикаты чистыми.

> [!warning] ⚠️ Подводные камни
> 
> - **`andThen` и `compose` путают направление.** `a.andThen(b)` = b(a(x)); `a.compose(b)` = a(b(x)). Если сомневаешься — бери `andThen`, он совпадает с порядком чтения.
> - **`negate()` не написать на method reference.** `String::isBlank.negate()` не скомпилируется — нужен `Predicate.not(String::isBlank)`.
> - **`null` в комбинатор → `NPE` сразу при сборке цепочки**, не при вызове.
> - **Короткое замыкание прячет побочные эффекты.** Правый предикат в `and`/`or` может не выполниться — не клади туда логику, которая обязана сработать.
> - **Глубокая цепочка = вложенные обёртки.** Читаемость и производительность страдают после ~5–7 звеньев; выдели именованный метод.
> - **Типы промежуточных шагов должны совпадать** — `Function<String,Integer>.andThen(Function<String,…>)` не соберётся.

### На собесе

> [!question]- В чём разница `andThen` и `compose`?  
> **RU:** `a.andThen(b)` применяет сначала `a`, потом `b` — то есть `b(a(x))`, читается слева направо. `a.compose(b)` наоборот: сначала `b`, потом `a` — `a(b(x))`, как математическое `a∘b`. На одних и тех же функциях `plus1.andThen(times2).apply(5)` даёт 12, а `plus1.compose(times2).apply(5)` — 11.  
> **EN:** `a.andThen(b)` applies `a` first, then `b` — i.e. `b(a(x))`, left-to-right. `a.compose(b)` is the reverse: `b` first, then `a` — `a(b(x))`, like the mathematical `a∘b`. On the same functions, `plus1.andThen(times2).apply(5)` is 12, while `plus1.compose(times2).apply(5)` is 11.

> [!question]- Как отрицать предикат, заданный method reference, например `String::isBlank`?  
> **RU:** `.negate()` тут не написать — это метод экземпляра `Predicate`, а `String::isBlank` ещё не предикат в точке вызова. Нужен `static`-метод `Predicate.not(String::isBlank)` (Java 11). В стриме: `.filter(Predicate.not(String::isBlank))`.  
> **EN:** You can't use `.negate()` — it's an instance method of `Predicate`, and `String::isBlank` isn't a `Predicate` at that point. Use the static `Predicate.not(String::isBlank)` (Java 11). In a stream: `.filter(Predicate.not(String::isBlank))`.
> 
> > [!danger] Ловушка-продолжение  
> > _«А `Predicate.not` и `.negate()` — это одно и то же по результату?»_ → Да, результат идентичен, отличается только применимость: `.negate()` — на готовом объекте `Predicate`, `Predicate.not(...)` — обёртка, принимающая method reference или ссылку. Слабый ответ — «not быстрее»; разницы в скорости нет.

> [!question]- Есть ли короткое замыкание у `Predicate.and`/`or`?  
> **RU:** Да. `and` при `false` слева не вычисляет правый предикат, `or` при `true` слева — тоже. Ровно как `&&`/`||`. Практическое следствие: не клади в правый предикат обязательный побочный эффект — он может не сработать.  
> **EN:** Yes. `and` skips the right predicate when the left is `false`; `or` skips it when the left is `true` — just like `&&`/`||`. Practical consequence: don't put a required side effect in the right predicate; it may not run.

> [!question]- Почему у `Consumer` есть `andThen`, но нет `compose`?  
> **RU:** Потому что `Consumer` ничего не возвращает — «подавать назад» результат в предыдущий шаг нечего. `andThen` просто вызывает оба потребителя по порядку, а `compose` требует выхода первого шага, которого у `Consumer` нет.  
> **EN:** Because `Consumer` returns nothing — there's no output to feed backward into a previous step. `andThen` just invokes both consumers in order, while `compose` needs the first step's output, which `Consumer` lacks.

> [!question]- Что произойдёт при `f.andThen(null)`?  
> **RU:** `NullPointerException`, причём сразу при построении цепочки, а не при вызове `apply`. Комбинаторы документированно проверяют аргумент через `Objects.requireNonNull`. Это ранняя защита от отложенной ошибки.  
> **EN:** A `NullPointerException`, thrown immediately when building the chain, not at `apply` time. The combinators are documented to null-check via `Objects.requireNonNull` — early failure instead of a deferred one.

> [!question]- Зачем нужен `Function.identity()`?  
> **RU:** Это функция `x -> x`. Полезна как нейтральный элемент композиции и в местах, где по API нужна `Function`, а значением должен стать сам элемент — классика `Collectors.toMap(keyFn, Function.identity())`. Использовать `Function.identity()` читаемее, чем писать `x -> x` вручную, и позволяет переиспользовать один экземпляр.  
> **EN:** It's the `x -> x` function. Useful as a composition neutral element and where an API needs a `Function` but the value should be the element itself — classically `Collectors.toMap(keyFn, Function.identity())`. It reads better than `x -> x` and lets one instance be reused.
