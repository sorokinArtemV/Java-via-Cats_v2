Представь, что у тебя есть список котов, и сортировать его нужно по-разному: то по возрасту, то по кличке, то по весу. Ты не хочешь плодить по классу `AgeComparator`, `NameComparator`, `WeightComparator` на каждый случай — ты хочешь **передать поведение** прямо в метод сортировки, как значение. В C# ты бы взял делегат (`Func`, `Comparison<T>`, лямбду). В Java делегатов нет вообще — и вместо них язык переиспользовал то, что уже было: интерфейс. Интерфейс, у которого ровно один абстрактный метод, становится «типом-мишенью», в который можно положить лямбду. Это и есть **функциональный интерфейс**.

> [!abstract] Ментальная модель  
> Функциональный интерфейс — это интерфейс с **ровно одним абстрактным методом** (SAM — Single Abstract Method). Один метод = одно поведение = одна лямбда. Лямбда — это «реализация этого единственного метода», записанная коротко.

### Как это работает

Никакого специального ключевого слова нет. «Функциональность» интерфейса — это просто факт: посчитай абстрактные методы, и если их ровно один, интерфейс годится в мишень для лямбды.

```java
interface CatAction {
    void perform(Cat cat);   // единственный abstract → это функциональный интерфейс
}

CatAction feed = cat -> System.out.println("Кормлю " + cat.name());
feed.perform(new Cat("Мурзик"));   // Кормлю Мурзик
```

Ключевой вопрос — **что считается за «абстрактный метод»**, а что нет. Здесь и живёт вся глубина темы. Считается не всё:

- `default`-методы **не считаются** — у них уже есть тело.
- `static`-методы **не считаются** — они не относятся к экземпляру.
- Методы, унаследованные от `Object` (`equals`, `hashCode`, `toString`), **не считаются** — их реализацию любой объект и так получит от `Object`, так что «функциональным» контрактом они быть не могут.

Классический пример — `Comparator<T>` из стандартной библиотеки. У него куча методов, но абстрактный из них ровно один, поэтому он функциональный:

![[Pasted image 20260715112049.png]]

Именно поэтому `Comparator<T>` можно создать лямбдой — компилятор видит один абстрактный метод `compare` и понимает, что лямбда реализует именно его:

```java
Comparator<Cat> byAge = (a, b) -> Integer.compare(a.age(), b.age());
```

#### `@FunctionalInterface` — это страховка, а не пропуск

Аннотация `@FunctionalInterface` **необязательна**. Лямбда работает с любым интерфейсом, у которого случайно оказался ровно один абстрактный метод — независимо от аннотации. Проверено:

```java
interface Feeder { void feed(String cat); }   // без аннотации
Feeder f = cat -> System.out.println("feeding");   // компилируется, работает
```

Тогда зачем аннотация? Она делает две вещи: **документирует намерение** («этот интерфейс задуман под лямбды») и **включает проверку компилятора**. Если ты пометил интерфейс как `@FunctionalInterface` и потом случайно добавил второй абстрактный метод — код не соберётся:

```java
@FunctionalInterface
interface CatBehavior {
    void meow(String name);
    void purr(String name);   // второй abstract — ошибка компиляции
}
```

Компилятор JDK 21 говорит буквально следующее:

```
error: Unexpected @FunctionalInterface annotation
  CatBehavior is not a functional interface
    multiple non-overriding abstract methods found in interface CatBehavior
```

Без аннотации этот же интерфейс скомпилировался бы молча — просто перестал бы быть пригодным для лямбд, и все места, где ты пытался передать в него лямбду, сломались бы с невнятными ошибками. Аннотация ловит проблему в одной точке — в объявлении, а не по всему коду.

> [!tip] 💡 Из C#  
> В C# ты работаешь с **делегатами**: `Func<...>`, `Action<...>`, `Predicate<T>` или своими `delegate`. У Java делегатов нет — вместо них именованные интерфейсы с одним методом. Отсюда важнейшее следствие: **два функциональных интерфейса с одинаковой сигнатурой — это разные, несовместимые типы.**
> 
> 
> ```java
> interface Feeder { void feed(String cat); }
> interface Petter { void feed(String cat); }
> Feeder f = cat -> {};
> Petter p = f;   // НЕ компилируется: incompatible types: Feeder cannot be converted to Petter
> ```
> 
> В C# делегаты тоже номинальны, но библиотека даёт тебе канонические `Func`/`Action`, которые переиспользуются везде, — поэтому «одинаковую сигнатуру» ты в реальности почти всегда пишешь одним и тем же типом. В Java единого набора нет исторически, но с Java 8 появились стандартные `Function`/`Consumer`/`Supplier`/`Predicate` (это тема 9.2), которые играют ту же роль — «канонических делегатов».

> [!example]- 🔬 Глубже / на синьора  
> **Почему методы `Object` не считаются — по букве JLS.** Правило из JLS §9.8: при подсчёте абстрактных методов игнорируются те, у которых есть _override-эквивалент_ среди `public`-методов `Object`. Логика простая: у любой реализации интерфейса объект **уже** унаследует `equals`/`hashCode`/`toString` от `Object`, поэтому такой метод физически не может остаться «нереализованным» — а значит, не может быть тем самым единственным контрактом. Это не костыль под `Comparator`, а общее правило. Проверено на JDK 21 — интерфейс с `int compare(...)` + `boolean equals(Object)` + `default` + `static` остаётся функциональным, лямбда цепляется к `compare`.
> 
> **Наследование может сломать «функциональность».** SAM считается по **всем** абстрактным методам, включая унаследованные. Интерфейс с нулём собственных abstract-методов, но расширяющий два функциональных, наберёт два абстрактных метода и перестанет быть функциональным. И наоборот: подынтерфейс может «переопределить» унаследованный абстрактный метод дефолтным и снова стать функциональным.
> 
> **Дженерик-нюанс.** Функциональный интерфейс может быть дженериком (`Function<T,R>`). Но метод считается «единственным» после учёта _override-эквивалентности_ с точностью до стирания типов — есть тонкий кейс с generic-методами внутри SAM, который на мид+ почти не встречается; помни только, что «один метод» проверяется по сигнатуре, а не по тексту.

> [!warning] ⚠️ Подводные камни
> 
> - **Добавил второй abstract-метод в «функциональный» интерфейс — сломал всех, кто передавал туда лямбду.** Без `@FunctionalInterface` компилятор промолчит в объявлении и завалит места использования. Ставь аннотацию на всё, что задумано под лямбды.
> - **Пустой интерфейс (0 abstract-методов) не функциональный.** Маркерные интерфейсы вроде `Serializable` лямбдой не создать.
> - **`equals` в интерфейсе выглядит как «лишний метод», но это `Object`-метод и он не в счёт.** `Comparator` объявляет `equals(Object)` намеренно (чтобы задокументировать смысл равенства компараторов) — и всё равно остаётся функциональным.
> - **Два SAM-интерфейса с идентичной сигнатурой не взаимозаменяемы** (см. C#-блок) — присвоить один другому нельзя, только пересоздать лямбдой.
> - **`default`-метод может обращаться к абстрактному** (`reversed()` внутри зовёт `compare`) — это норма и часто именно так строятся комбинаторы (тема 9.7).

### На собесе

> [!question]- Что делает интерфейс функциональным? Обязательна ли `@FunctionalInterface`?  
> **RU:** Функциональный интерфейс — это интерфейс с ровно одним абстрактным методом (SAM). `default`, `static` и унаследованные от `Object` методы не считаются. Аннотация `@FunctionalInterface` необязательна: лямбда работает с любым SAM-интерфейсом. Аннотация нужна для документирования намерения и для проверки компилятором — если случайно добавить второй абстрактный метод, сборка упадёт в точке объявления.  
> **EN:** A functional interface has exactly one abstract method (SAM). `default`, `static`, and `Object`-inherited methods don't count. `@FunctionalInterface` is optional — a lambda targets any SAM interface regardless. The annotation documents intent and enables a compile-time check: adding a second abstract method breaks the build at the declaration site.

> [!question]- Почему `Comparator` — функциональный интерфейс, хотя у него столько методов?  
> **RU:** Потому что абстрактный из них ровно один — `compare(T, T)`. Всё остальное — это `default`-методы (`reversed`, `thenComparing`), `static`-фабрики (`naturalOrder`, `comparing`) и объявленный `equals(Object)`, унаследованный от `Object`. Ни то, ни другое, ни третье в подсчёт SAM не входит.  
> **EN:** Because it has exactly one abstract method — `compare(T, T)`. The rest are `default` methods, `static` factories, and an `equals(Object)` that comes from `Object`. None of those count toward the SAM.
> 
> > [!danger] Ловушка-продолжение  
> > _«А если я добавлю `equals` в свой функциональный интерфейс — он перестанет быть функциональным?»_ → **Нет.** `equals(Object)` — override-эквивалент метода `Object`, он игнорируется при подсчёте. Слабый ответ — «да, ведь методов стало два»; он выдаёт непонимание правила JLS §9.8.

> [!question]- Чем это отличается от делегатов в C#?  
> **RU:** В C# есть первоклассные типы делегатов (`Func`, `Action`) — тип функции существует сам по себе. В Java функции как типа нет: лямбда всегда имеет тип _какого-то интерфейса_. Практическое следствие — два интерфейса с одинаковой сигнатурой в Java несовместимы между собой, тогда как в C# ты переиспользуешь один `Func<...>` повсюду.  
> **EN:** C# has first-class delegate types (`Func`, `Action`) — a function type exists on its own. Java has no function type: a lambda always has _some interface_ type. The practical consequence — two identically-signed interfaces are incompatible in Java, whereas in C# you reuse one `Func<...>` everywhere.

> [!question]- Может ли функциональный интерфейс наследоваться от другого?  
> **RU:** Да, но считаются все абстрактные методы, включая унаследованные. Если после наследования их набирается больше одного — интерфейс перестаёт быть функциональным. Можно, наоборот, «погасить» унаследованный абстрактный метод `default`-реализацией и снова свести к одному.  
> **EN:** Yes, but the count includes inherited abstract methods. If more than one accumulates, it's no longer functional. Conversely, you can override an inherited abstract method with a `default` to bring the count back to one.

> [!question]- Можно ли сделать лямбду из `Runnable`? А из `Serializable`?  
> **RU:** Из `Runnable` — да, у него один абстрактный метод `run()`. Из `Serializable` — нет, это маркерный интерфейс без абстрактных методов, лямбде нечего реализовывать.  
> **EN:** From `Runnable` — yes, it has one abstract method `run()`. From `Serializable` — no, it's a marker interface with zero abstract methods; there's nothing for the lambda to implement.
