Лямбда почти никогда не живёт в вакууме: ей нужен порог для сравнения, значение из конфига, список, куда складывать результат. Когда лямбда обращается к переменной из окружающего метода — она её **захватывает** (capture). Но попробуй потом эту переменную переприсвоить — и компилятор откажется собирать код. Тех, кто пришёл из C#, это ставит в тупик: там замыкания спокойно захватывают и меняют переменные. В Java правило жёсткое: захваченные локальные переменные должны быть `final` или **effectively final**.

> [!abstract] Ментальная модель  
> Лямбда захватывает локальную переменную **по значению — снимком** на момент своего создания. Раз это копия, Java запрещает менять оригинал (иначе копия и оригинал разошлись бы) — отсюда требование «effectively final». А вот **поля** объекта — другое дело: они доступны через `this`, не копируются, и менять их из лямбды можно.

### Что значит «effectively final»

`effectively final` — это переменная, которую **после инициализации ни разу не переприсваивают**, даже если слово `final` не написано. С Java 8 компилятор выводит это сам:

```java
int limit = 3;                                   // final не написан
Predicate<Integer> underLimit = n -> n < limit;   // но limit не меняется → захват ОК
```

Стоит добавить переприсваивание — переменная перестаёт быть effectively final, и захват ломается. JDK 21 говорит прямо:

```java
int count = 1;
Supplier<Integer> s = () -> count;
count = 2;                            // ломает effectively final
// error: local variables referenced from a lambda expression
//        must be final or effectively final
```

Важно: неважно, до или после лямбды идёт переприсваивание — если переменную меняют **где угодно** в методе, она не effectively final.

### Почему по значению — и почему поэтому final

Локальные переменные живут на **стеке** и исчезают, когда метод завершается. А лямбда может пережить метод — её сохранят в поле, передадут в поток, положат в коллекцию. Значит, ссылаться на исчезнувшую стековую переменную нельзя. Java решает это копированием: значение переменной **снимается в момент создания лямбды** и хранится внутри неё.

![[Pasted image 20260715121502.png]]

Теперь становится ясно, зачем нужен запрет на переприсваивание: если бы оригинал на стеке и копия в лямбде могли меняться независимо, было бы совершенно непонятно, кто из них «настоящий». Java не отвечает на этот вопрос — она его запрещает, требуя, чтобы захваченная переменная была неизменной.

### Поля — исключение: они меняются свободно

Правило про «снимок» касается только **локальных переменных**. Поля объекта захватываются иначе: лямбда захватывает `this`, а поля читает и пишет **через него** — вживую, не копией. Поэтому поле можно менять из лямбды (проверено):

```java
private int count = 0;              // ПОЛЕ, не локальная переменная

void demo() {
    Runnable inc = () -> count++;   // компилируется и мутирует поле
    inc.run(); inc.run();
    System.out.println(count);      // 2
}
```

Разница ровно в этом: локальная переменная копируется по значению (и обязана быть effectively final), а поле достаётся через `this` и мутабельно. Если очень нужно «менять переменную из лямбды» — часто ответ в том, чтобы сделать её полем, а не локальной.

### `this` внутри лямбды — это внешний объект

Ещё одно отличие от анонимного класса, которое любят на собесе. Анонимный класс создаёт **свой** `this` (сам анонимный объект). Лямбда — **нет**: `this` внутри лямбды означает окружающий экземпляр, как в обычном коде метода (проверено):

```java
Runnable r1 = () -> System.out.println(this.getClass().getSimpleName());   // FieldsAndThis
Runnable r2 = new Runnable() {
    public void run() { System.out.println(this.getClass().getSimpleName()); }  // "" — сам анон-объект
};
```

Анонимный класс напечатал пустую строку: у анонимных классов нет простого имени — а это как раз доказывает, что его `this` указывает на другой объект (саму реализацию `Runnable`), а не на внешний класс.

### Разница с C#: по значению против по ссылки

Вот главное расхождение. В C# замыкания захватывают переменные **по ссылке**: лямбда и внешний код делят одну и ту же переменную, и изменения видны в обе стороны. Никакого «effectively final» там нет.

![[Pasted image 20260715121514.png]]

Это расхождение объясняет знаменитую ловушку с циклами. В Java счётчик классического `for` переприсваивается на каждой итерации, поэтому он **не** effectively final — и захватить его нельзя (проверено):

```java
for (int i = 0; i < 3; i++) {
    tasks.add(() -> System.out.println(i));   // error: i не effectively final
}
```

А вот переменная `for-each` — **свежая на каждой итерации**, так что каждая лямбда законно ловит своё значение:

```java
for (String name : List.of("Мурзик", "Барсик", "Том")) {
    tasks.add(() -> System.out.print(name));   // каждая лямбда захватила своё name → ОК
}
// печатает: Мурзик Барсик Том
```

В старом C# ровно этот код с `for`/`foreach` захватывал одну общую переменную и все замыкания видели последнее значение — классический баг. Java этот класс ошибок просто исключила запретом на изменяемый захват.

### Обходной приём и почему это запах

Когда всё же нужно «накопить» что-то из лямбды, люди захватывают **изменяемый холдер**: массив из одного элемента или `AtomicInteger`. Ссылка на холдер не меняется (она effectively final), а вот содержимое — да:

```java
int[] counter = {0};
Runnable inc = () -> counter[0]++;   // counter не переприсваивается, а его элемент — можно
inc.run(); inc.run(); inc.run();     // counter[0] == 3
```

Это компилируется, но почти всегда — **сигнал, что выбран не тот инструмент**. Накопление значения из лямбды — обычно работа для `Stream.reduce`, `collect` или `IntStream.sum` (глава 10), а не для ручного счётчика в замыкании. Если холдер нужен для конкуррентного счёта — бери `AtomicInteger`, но и это чаще признак, что логику стоит переписать.

> [!tip] 💡 Из C#
> 
> - **C# — по ссылке, Java — по значению.** В C# захваченную локальную переменную можно менять из лямбды, и снаружи изменение видно. В Java — снимок, и переприсваивание запрещено.
> - **`effectively final` в C# нет** — там компилятор поднимает захваченные переменные в скрытый heap-объект (display class), поэтому изменяемость «бесплатна».
> - **Ловушка цикла.** В C# до версии 5 `foreach` захватывал общую переменную (позже починили для `foreach`, но для `for` — нет). В Java эта ловушка невозможна by design.
> - **`this`.** В обоих языках `this` в лямбде — внешний объект; но в Java это ещё и отличает лямбду от анонимного класса, где `this` — свой.

> [!example]- 🔬 Глубже / на синьора  
> **Как захват выглядит в байткоде.** Захваченные локальные переменные становятся **аргументами** синтетического метода, который генерирует компилятор под лямбду, и передаются в `LambdaMetafactory` при создании экземпляра (детали — 9.8). Незахватывающая лямбда (`() -> 42`) не тянет ничего извне, поэтому JVM может закешировать **один** её экземпляр и переиспользовать — а захватывающая создаёт новый объект на каждый вызов окружающего метода, потому что значения разные. Это иногда важно для аллокаций в горячем пути.
> 
> **Захват ссылки ≠ захват объекта.** `effectively final` касается самой переменной, а не того, на что она указывает. Можно захватить `final List<Cat> cats` и спокойно делать `cats.add(...)` в лямбде — меняется содержимое списка, а не ссылка. Именно на этом стоит приём с холдером.
> 
> **Утечка через захват `this`.** Связанная лямбда, обращающаяся к полю, захватывает **весь** `this`. Если такую лямбду сохранить надолго (в статическом реестре, в長living-слушателе), она удержит внешний объект от сборки мусора — типичная утечка в event-driven коде. Лечится захватом локальной копии нужного значения вместо поля.

> [!warning] ⚠️ Подводные камни
> 
> - **Переприсваивание где угодно ломает захват** — не только после лямбды, но и до неё. Переменная должна быть неизменной на всём протяжении метода.
> - **`for (int i...)` захватить нельзя, `for-each` — можно.** Первый переприсваивает счётчик, второй даёт свежую переменную на итерацию.
> - **Холдер (`int[]`, `AtomicInteger`) обходит правило, но это запах** — обычно нужен `reduce`/`collect`.
> - **Захват `this` может держать объект в памяти** — долгоживущая лямбда с обращением к полю мешает GC.
> - **Снимок фиксирует значение, а не «переменную»** — если захватить ссылку на мутабельный объект, изменения его _содержимого_ лямбда увидит; изменения самой _ссылки_ — нет (её и менять нельзя).
> - **`this` в лямбде ≠ `this` в анонимном классе** — во втором случае это сам анонимный объект.

### На собесе

> [!question]- Что такое effectively final и зачем это нужно?  
> **RU:** Effectively final — локальная переменная, которую после инициализации ни разу не переприсваивают, даже без слова `final`. Требование нужно потому, что лямбда захватывает локальные переменные по значению — снимком. Локальные переменные живут на стеке и исчезают с методом, а лямбда может его пережить, поэтому значение копируется. Чтобы копия и оригинал не расходились, оригинал запрещено менять.  
> **EN:** Effectively final is a local variable never reassigned after initialization, even without `final`. The rule exists because a lambda captures locals by value — a snapshot. Locals live on the stack and vanish with the method, while the lambda may outlive it, so the value is copied. To keep the copy and original from diverging, the original may not change.

> [!question]- Почему поле можно менять из лямбды, а локальную переменную — нет?  
> **RU:** Потому что поле захватывается не по значению, а через `this`: лямбда держит ссылку на объект и обращается к полю вживую. Локальная переменная копируется снимком, поэтому её меняют только до захвата — а вообще не меняют, чтобы копия оставалась согласованной.  
> **EN:** Because a field is captured through `this`, not by value: the lambda holds the object reference and accesses the field live. A local variable is copied as a snapshot, so it can't change at all — that keeps the copy consistent.
> 
> > [!danger] Ловушка-продолжение  
> > _«Тогда как всё-таки накопить значение из лямбды?»_ → Сделать переменную полем, либо (реже) захватить изменяемый холдер `int[]`/`AtomicInteger`. Но правильный ответ на мид+ — обычно это работа для `reduce`/`collect`, а холдер в лямбде — запах.

> [!question]- Скомпилируется ли захват счётчика классического `for`? А `for-each`?  
> **RU:** Классический `for` — нет: `i` переприсваивается каждую итерацию, не effectively final. `for-each` — да: переменная цикла свежая на каждой итерации, каждая лямбда захватывает своё значение. Это тонкое, но частое различие.  
> **EN:** Classic `for` — no: `i` is reassigned each iteration, not effectively final. `for-each` — yes: the loop variable is fresh per iteration, so each lambda captures its own value.

> [!question]- Чем захват в Java отличается от замыканий C#?  
> **RU:** В Java захват локальной переменной — по значению (снимок), поэтому нужна неизменность (effectively final). В C# захват по ссылке: лямбда и внешний код делят одну переменную, изменения видны в обе стороны, никакого effectively final нет. Как следствие, джавовая модель исключает ловушку с захватом переменной цикла.  
> **EN:** In Java, local capture is by value (a snapshot), requiring immutability (effectively final). In C#, capture is by reference: the lambda and outer code share one variable, changes are mutually visible, and there's no effectively-final rule. As a result, Java's model rules out the loop-variable capture bug.

> [!question]- Что напечатает `this.getClass()` внутри лямбды и внутри анонимного класса?  
> **RU:** Внутри лямбды `this` — это внешний объект, поэтому напечатается его класс. Внутри анонимного класса `this` — сам анонимный объект (у него нет простого имени). Лямбда не заводит собственный `this`, анонимный класс — заводит.  
> **EN:** Inside a lambda, `this` is the enclosing object, so its class prints. Inside an anonymous class, `this` is the anonymous object itself (which has no simple name). A lambda introduces no `this` of its own; an anonymous class does.

> [!question]- Можно ли захватить `final List<Cat>` и добавлять в него из лямбды?  
> **RU:** Да. `effectively final` относится к самой ссылке, а не к содержимому. Ссылку менять нельзя, но `list.add(...)` меняет содержимое объекта, а не ссылку — это законно. На этом и построен приём с изменяемым холдером.  
> **EN:** Yes. Effectively final applies to the reference, not the contents. You can't reassign the reference, but `list.add(...)` mutates the object, not the reference — that's legal, and it's what the mutable-holder trick relies on.
