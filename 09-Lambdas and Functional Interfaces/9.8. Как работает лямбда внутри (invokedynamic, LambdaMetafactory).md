Легко подумать, что лямбда — это просто «сахар над анонимным классом»: мол, компилятор разворачивает `() -> {}` в `new Runnable() {...}`. Это неверно, и разница видна невооружённым глазом. Запусти лямбду и анонимный класс, спроси `getClass().getName()` — и получишь `Internals$$Lambda/0x00007f...` против `Internals$1`. Анонимный класс — это настоящий класс, созданный **при компиляции**. Лямбда — класс, который JVM синтезирует **в рантайме**, при первом использовании, и на диск он не пишется. За этим стоит `invokedynamic` — инструкция, ради которой в Java 7 заранее подготовили фундамент, а в Java 8 на нём построили лямбды.

> [!abstract] Ментальная модель  
> Компилятор кладёт тело лямбды в приватный синтетический метод, а в точке использования ставит инструкцию `invokedynamic`. При первом её срабатывании фабрика `LambdaMetafactory` **на лету** создаёт скрытый класс, реализующий функциональный интерфейс. Лямбда — не анонимный класс: у неё нет `.class`-файла, свой `this` она не заводит, а non-capturing вариант кэшируется в один экземпляр.

### Что происходит: две стадии

![[Pasted image 20260715121812.png]]

**Стадия 1 — компиляция.** Компилятор делает две вещи. Тело лямбды (`c.age()`) он выносит в **приватный синтетический метод** самого класса — со всеми захваченными переменными в качестве параметров (глава 9.5). А в точке, где лямбда используется, ставит инструкцию **`invokedynamic`** с указанием «загрузочного» (bootstrap) метода — `LambdaMetafactory.metafactory` из `java.lang.invoke`. Никакого класса под лямбду при компиляции не создаётся.

**Стадия 2 — первый вызов.** Когда `invokedynamic` срабатывает **впервые**, вызывается bootstrap-метод. `LambdaMetafactory` генерирует **скрытый класс** (hidden class), реализующий нужный функциональный интерфейс, чей SAM-метод просто дергает тот самый синтетический метод. Возвращается `CallSite` — «связанная» точка вызова. Для non-capturing лямбды это будет фабрика, отдающая один константный экземпляр.

**Стадия 3 — последующие вызовы.** Точка вызова уже слинкована — повторные срабатывания дешёвы, никакой генерации больше нет.

### Наблюдаемые следствия (проверено на JDK 21)

Механику не видно напрямую без `javap`, но её отпечатки видны в поведении:

```
lambda class:  Internals$$Lambda/0x00007f9e0414c208   ← синтезирован в рантайме
anon class:    Internals$1                            ← настоящий класс с компиляции
non-capturing same instance: true                     ← один кэшированный экземпляр
capturing same instance:     false                    ← новый экземпляр на каждый захват
```

- Имя класса лямбды содержит `$$Lambda/0x...` (hex-адрес скрытого класса) — его нет среди `.class`-файлов на диске. У анонимного класса — обычное имя `Internals$1`, и это реальный файл.
- **Non-capturing** лямбда (`() -> 42`) не тянет ничего извне, поэтому JVM переиспользует **один** экземпляр — при повторных вычислениях `a == b`.
- **Capturing** лямбда (`n -> () -> n`) несёт захваченное значение, поэтому фабрика создаёт **новый** объект на каждый вызов — `c != d`.

### Лямбда против анонимного класса

||Лямбда|Анонимный класс|
|---|---|---|
|`.class`-файл|нет, класс спинится в рантайме|да, отдельный `Outer$1.class` при компиляции|
|Когда создаётся класс|лениво, при первом `invokedynamic`|при компиляции, грузится как обычный|
|`this`|внешний объект|сам анонимный объект|
|Non-capturing экземпляр|кэшируется (один)|всегда новый через `new`|
|Влияние на старт/размер jar|меньше классов, но работа в рантайме|больше классов на диске|

> [!tip] 💡 Из C#  
> В C# делегаты и лямбды тоже компилируются в скрытые методы, а замыкания с захватом — в скрытый **display class** (аналог захвата в объект). Но связывания через `invokedynamic` там нет: C# принимает решение о представлении на этапе компиляции. Джавовый подход отложил это решение в рантайм намеренно — чтобы JVM могла менять стратегию создания лямбд от версии к версии, не требуя перекомпиляции твоего кода. Практический для тебя итог тот же, что в C#: non-capturing лямбды дешёвы и переиспользуются, capturing — аллоцируют.

> [!example]- 🔬 Глубже / на синьора  
> **Зачем вообще `invokedynamic`, а не «сахар над анонимным классом».** Причины из документа Брайана Гоца _Translation of Lambda Expressions_: (1) **гибкость** — стратегию инстанцирования выбирает рантайм, поэтому её можно улучшать (например, ввести кэширование) без перекомпиляции; (2) **отказ от взрыва классов** — один `.class` на каждую лямбду раздул бы jar и замедлил старт; (3) **оптимизация non-capturing** — синглтон-переиспользование возможно именно потому, что решение принимает фабрика в рантайме.
> 
> **Захват = параметры фабрики.** Захваченные переменные передаются в bootstrap как аргументы и становятся полями скрытого класса. Поэтому capturing-лямбда аллоцирует объект при каждом создании, а non-capturing — нет. В горячем пути стоит следить, чтобы часто вызываемая лямбда была non-capturing (не тянула локальные переменные без нужды).
> 
> **Скрытые классы (hidden classes, JEP 371, Java 15).** Начиная с Java 15, `LambdaMetafactory` использует API скрытых классов вместо старого `Unsafe.defineAnonymousClass`. Скрытые классы не видны обычным загрузчикам, не участвуют в разрешении констант и могут выгружаться независимо — это и есть тот `$$Lambda/0x...` в имени. До Java 15 механизм был чуть другим, но идея та же.
> 
> **Method reference — тот же путь.** `Cat::name` (глава 9.4) компилируется через тот же `invokedynamic` + `LambdaMetafactory`, только синтетический метод может и не создаваться, если ссылка указывает прямо на существующий метод. Отдельного `.class` у method reference тоже нет.
> 
> **Первый вызов чуть дороже.** Линковка `invokedynamic` (генерация скрытого класса) происходит один раз и добавляет небольшую задержку на первое срабатывание — это иногда заметно в микробенчмарках холодного старта, но амортизируется на первом же прогоне.

> [!warning] ⚠️ Подводные камни
> 
> - **Лямбда — не анонимный класс.** `getClass().getName()` у них разный; `this` разный; `.class`-файла у лямбды нет.
> - **Capturing-лямбда аллоцирует.** В горячем цикле лишний захват локальной переменной = объект на каждой итерации. Non-capturing переиспользуется.
> - **Нельзя рассчитывать на конкретное имя класса лямбды.** `$$Lambda/0x...` — деталь реализации, менялась между версиями; не парси его и не завязывайся на него.
> - **Сериализация лямбд особая и хрупкая.** Лямбда сериализуема только если её target-интерфейс `Serializable`; механизм (`SerializedLambda`) завязан на имена синтетических методов и легко ломается при рефакторинге. Избегай сериализации лямбд (подробнее — глава 19).
> - **Рефлексия по телу лямбды невозможна.** В отличие от C# expression trees, тело лямбды — исполняемый код, интроспектировать его нельзя (см. 9.3).
> - **Стек-трейсы менее наглядны.** В трейсе появляются кадры вида `lambda$main$0` и `LambdaMetafactory` — знай, что это норма, а не баг.

### На собесе

> [!question]- Лямбда — это синтаксический сахар над анонимным классом?  
> **RU:** Нет. Анонимный класс — реальный класс, созданный при компиляции (`Outer$1.class`). Лямбда компилируется в приватный синтетический метод плюс инструкцию `invokedynamic`; сам класс, реализующий интерфейс, JVM синтезирует в рантайме через `LambdaMetafactory` при первом вызове, и на диск он не пишется. Плюс у лямбды свой `this` не заводится, а non-capturing вариант кэшируется.  
> **EN:** No. An anonymous class is a real compile-time class (`Outer$1.class`). A lambda compiles to a private synthetic method plus an `invokedynamic` instruction; the interface-implementing class is synthesized at runtime by `LambdaMetafactory` on first call, with no `.class` on disk. Also, a lambda introduces no `this` of its own, and non-capturing lambdas are cached.

> [!question]- Что такое `invokedynamic` и при чём тут `LambdaMetafactory`?  
> **RU:** `invokedynamic` — инструкция JVM с отложенным связыванием: она указывает на bootstrap-метод, который вызывается при первом срабатывании и решает, что подставить в точку вызова. Для лямбд bootstrap — это `LambdaMetafactory.metafactory`; он генерирует скрытый класс, реализующий функциональный интерфейс, и возвращает `CallSite`. Дальше точка вызова слинкована и работает дёшево.  
> **EN:** `invokedynamic` is a JVM instruction with deferred linkage: it points to a bootstrap method that runs on first execution and decides what to bind at the call site. For lambdas, the bootstrap is `LambdaMetafactory.metafactory`; it generates a hidden class implementing the functional interface and returns a `CallSite`. Afterwards the site is linked and cheap.

> [!question]- Почему выбрали `invokedynamic`, а не генерацию классов при компиляции?  
> **RU:** Ради гибкости и производительности. Отложив решение в рантайм, JVM может менять стратегию создания лямбд без перекомпиляции кода. Это избегает «взрыва классов» (один `.class` на лямбду раздул бы jar и старт) и позволяет кэшировать non-capturing лямбды в один экземпляр — оптимизация, которую фабрика делает в рантайме.  
> **EN:** For flexibility and performance. Deferring the decision to runtime lets the JVM change lambda instantiation without recompiling code. It avoids class-file explosion (one `.class` per lambda would bloat the jar and startup) and enables caching non-capturing lambdas as a single instance — an optimization the factory does at runtime.
> 
> > [!danger] Ловушка-продолжение  
> > _«Значит, лямбда всегда дешевле анонимного класса?»_ → Не всегда. Non-capturing — да, кэшируется. Capturing аллоцирует объект на каждый захват, как и `new`. А самый первый вызов ещё и линкует `invokedynamic` — крошечная разовая задержка. Слабый ответ — «лямбды всегда быстрее»; правда тоньше.

> [!question]- В чём разница capturing и non-capturing лямбды на уровне экземпляров?  
> **RU:** Non-capturing (`() -> 42`) не тянет ничего извне, поэтому JVM переиспользует один кэшированный экземпляр — повторные вычисления дают тот же объект (`==` истинно). Capturing (`n -> () -> n`) несёт захваченные значения полями скрытого класса, поэтому фабрика создаёт новый объект на каждый захват (`==` ложно). Проверяется сравнением ссылок.  
> **EN:** A non-capturing lambda (`() -> 42`) pulls nothing external, so the JVM reuses one cached instance — repeated evaluations yield the same object (`==` is true). A capturing lambda (`n -> () -> n`) carries captured values as hidden-class fields, so the factory makes a new object per capture (`==` is false). Checkable by reference comparison.

> [!question]- Почему в стек-трейсе видны `lambda$...` и почему у лямбды странное имя класса?  
> **RU:** `lambda$main$0` — это имя синтетического метода, куда компилятор вынес тело лямбды; оно нормально появляется в трейсах. Странное имя класса `Outer$$Lambda/0x...` — это скрытый класс, сгенерированный `LambdaMetafactory` в рантайме (с Java 15 — через API hidden classes). На это имя нельзя завязываться: оно деталь реализации и менялось между версиями.  
> **EN:** `lambda$main$0` is the synthetic method the compiler extracted the lambda body into; it legitimately appears in traces. The odd class name `Outer$$Lambda/0x...` is a hidden class generated by `LambdaMetafactory` at runtime (via the hidden-classes API since Java 15). Don't depend on that name — it's an implementation detail that has changed across versions.
