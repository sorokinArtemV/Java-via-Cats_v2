Ты хочешь прочитать список файлов: `paths.stream().map(Files::readString)`. Компилятор отказывается. Причина не в стриме и не в `readString` — а в том, что `Files.readString` бросает `IOException` (checked), а метод `apply` у `Function` объявлен **без `throws`**. Лямбда не может бросить checked-исключение, которого нет в сигнатуре функционального интерфейса. Это самая частая боль при работе с лямбдами и стримами: любой I/O, JDBC, рефлексия — всё, что бросает checked, — не встаёт в стандартные `Function`/`Consumer`/`Supplier` без танцев. В C# такой проблемы нет вовсе, потому что там checked-исключений не существует.

> [!abstract] Ментальная модель  
> Стандартные функциональные интерфейсы объявлены без `throws`, поэтому лямбда может «выпустить» наружу только **unchecked**-исключение. Checked придётся либо поймать прямо внутри лямбды, либо обернуть, либо (грязно) протащить трюком. Выбор — между честной обёрткой и читаемостью.

### Почему так: SAM без `throws`

Сигнатуры базовых интерфейсов не содержат `throws`:

```java
interface Function<T,R> { R apply(T t); }          // нет throws
interface Consumer<T>   { void accept(T t); }       // нет throws
interface Supplier<T>   { T get(); }                // нет throws
```

Правило Java простое: лямбда может бросать только то, что разрешает абстрактный метод её target-типа. Раз `apply` не объявляет исключений, checked-исключение упирается в стену, а unchecked проходит свободно.

![[Pasted image 20260715121604.png]]

Поэтому `map(Broken::risky)`, где `risky` объявляет `throws Exception`, не собирается. JDK 21:

```java
names.stream().map(Broken::risky)   // risky() throws Exception
// error: incompatible thrown types Exception in functional expression
```

### Три способа выкрутиться

**(а) `try/catch` прямо в лямбде.** Честно, но многословно. Обычно checked оборачивают в unchecked и пробрасывают (паттерн wrap-and-rethrow из главы 6):

```java
names.stream()
     .map(s -> {
         try { return risky(s); }
         catch (Exception e) { throw new RuntimeException(e); }   // или UncheckedIOException
     })
     .collect(toList());
```

Это дефолтный правильный выбор для разового случая. Для `IOException` в JDK есть готовый `UncheckedIOException` — ровно под эту обёртку.

**(б) Свой бросающий интерфейс + адаптер.** Если оборачивать приходится часто, заводят `ThrowingFunction` с `throws` и один раз пишут адаптер, превращающий его в обычный `Function`. Тело лямбды остаётся чистым (проверено):

```java
@FunctionalInterface
interface ThrowingFunction<T, R> { R apply(T t) throws Exception; }

static <T, R> Function<T, R> unchecked(ThrowingFunction<T, R> f) {
    return t -> {
        try { return f.apply(t); }
        catch (Exception e) { throw new RuntimeException(e); }
    };
}

// использование — лямбда снова читаемая:
names.stream().map(unchecked(Workarounds::risky)).collect(toList());
```

**(в) Sneaky throws.** Трюк, который **обманывает компилятор**: через дженерик-каст и стирание типов можно бросить checked-исключение, не объявляя его. Именно это делает Lombok `@SneakyThrows`.

```java
@SuppressWarnings("unchecked")
static <E extends Throwable> void sneakyThrow(Throwable e) throws E {
    throw (E) e;   // E стирается до Throwable — проверки типа в рантайме нет
}
static void risky() {                     // НЕ объявляет throws
    sneakyThrow(new IOException("boom"));  // но реально бросает checked
}
```

Работает — `IOException` реально летит в рантайме. Но у трюка есть **цена**, и она видна сразу: раз `risky()` не _объявляет_ `IOException`, компилятор не верит, что она бросается, и **не даёт её поймать по типу**:

```java
try { risky(); }
catch (IOException e) { ... }
// error: exception IOException is never thrown in body of corresponding try statement
```

Приходится ловить широко (`catch (Exception e)`), а это и есть провал идеи checked-исключений: вызывающий не знает, что может прилететь. Поэтому sneaky throws — инструмент на крайний случай, а не стиль.

|Способ|Плюс|Минус|
|---|---|---|
|`try/catch` в лямбде|явно, ничего не прячет|шумно, повторяется|
|`ThrowingFunction` + адаптер|тело чистое, переиспользуемо|нужен свой интерфейс на каждую форму|
|sneaky throws / Lombok|максимально коротко|ломает контракт checked, нельзя поймать по типу|

> [!tip] 💡 Из C#  
> Этой проблемы у тебя в C# **не было** — там все исключения unchecked, `Func<...>` спокойно «пропускает» любое исключение наружу, никакого `throws` в сигнатурах нет. Поэтому джавовая возня с checked в лямбдах — новый для тебя класс задач, целиком следствие того, что Java делит исключения на checked/unchecked (глава 6). Ближайшая аналогия: sneaky throws превращает Java-модель в C#-модель «всё unchecked» — но нелегально, обманом компилятора.

> [!example]- 🔬 Глубже / на синьора  
> **Почему sneaky throws вообще работает.** Из-за **стирания типов** (глава 7) параметр `E extends Throwable` в рантайме стёрт до `Throwable`, поэтому каст `(E) e` не порождает никакой проверки в байткоде — JVM бросает исходный объект как есть. Компилятор же на этапе компиляции «выводит» `E` из контекста вызова как `RuntimeException` (в месте, где throws не требуется) и потому не заставляет объявлять `throws`. Проверок в двух местах нет, отсюда и дыра.
> 
> **Где в JDK уже решили эту боль.** `Callable<V>` — это, по сути, `Supplier<V>`, которому **можно** бросать: его SAM — `V call() throws Exception` (проверено, лямбда с `IOException` внутри компилируется). Поэтому асинхронные API (`ExecutorService.submit(Callable)`, глава 24) принимают `Callable`, а не `Supplier`. А `CompletableFuture.supplyAsync` берёт `Supplier` — и checked внутри придётся оборачивать вручную. `UncheckedIOException` добавлен в JDK 8 специально как обёртка для `IOException` в потоковом коде.
> 
> **Стрелочка к дизайну.** Существование `Callable` и `UncheckedIOException` — сигнал, что даже авторы JDK признали проблему: единого «бросающего» аналога `Function`/`Consumer` в стандартной библиотеке нет, и это осознанный компромисс, а не забывчивость.

> [!warning] ⚠️ Подводные камни
> 
> - **`map(Files::readString)` не компилируется** — `readString` бросает `IOException`. Нужна обёртка.
> - **`UncheckedIOException` — не то же, что `RuntimeException`.** Для `IOException` бери именно его: он несёт причину и типизирован, ловится точнее.
> - **Sneaky throws ломает `catch` по типу** — вызывающий не сможет поймать незадекларированное исключение иначе как через широкий `catch`.
> - **Обёртка в `RuntimeException` теряет тип** — если наверху нужно различать причины, оборачивай в осмысленный собственный unchecked, а не в голый `RuntimeException`.
> - **Не глотай в лямбде.** `catch (Exception e) {}` внутри стрима — исключение исчезает молча, данные теряются без следа.
> - **`parallelStream` + обёртка** — обёрнутое исключение прилетит из чужого потока; убедись, что оно несёт причину (`new RuntimeException(e)`, не `new RuntimeException(e.getMessage())`).

### На собесе

> [!question]- Почему `list.stream().map(Files::readString)` не компилируется?  
> **RU:** Потому что `Files.readString` бросает `IOException` — checked-исключение, а `Function.apply` объявлен без `throws`. Лямбда может выпустить наружу только то, что разрешает SAM её функционального интерфейса; checked туда не проходит. Нужно поймать внутри лямбды и обернуть в unchecked (например, `UncheckedIOException`).  
> **EN:** Because `Files.readString` throws `IOException` — a checked exception — while `Function.apply` declares no `throws`. A lambda may only propagate what its SAM allows; a checked exception doesn't fit. You catch it inside the lambda and wrap it in an unchecked one (e.g. `UncheckedIOException`).

> [!question]- Как правильно обработать checked-исключение внутри лямбды в стриме?  
> **RU:** Три варианта. Разово — `try/catch` в лямбде с оборачиванием в unchecked (`throw new UncheckedIOException(e)`). Часто — свой `ThrowingFunction` с `throws` плюс адаптер, превращающий его в обычный `Function`. Крайний случай — sneaky throws (как Lombok `@SneakyThrows`), но он ломает контракт checked. Дефолт — обёртка; sneaky только осознанно.  
> **EN:** Three options. One-off — `try/catch` in the lambda, wrapping into an unchecked (`throw new UncheckedIOException(e)`). Frequent — a custom `ThrowingFunction` with `throws` plus an adapter to a plain `Function`. Last resort — sneaky throws (like Lombok's `@SneakyThrows`), which breaks the checked contract. Default to wrapping.

> [!question]- Что такое sneaky throws и почему это опасно?  
> **RU:** Это дженерик-трюк, позволяющий бросить checked-исключение, не объявляя его: `throw (E) e`, где `E` стирается до `Throwable`, так что проверки типа в рантайме нет, а компилятор не требует `throws`. Опасно тем, что вызывающий не знает про исключение и не может поймать его по типу — приходится ловить широко. Это обходит весь смысл checked-исключений.  
> **EN:** A generics trick to throw a checked exception without declaring it: `throw (E) e`, where `E` erases to `Throwable`, so there's no runtime type check and the compiler doesn't demand `throws`. It's dangerous because the caller doesn't know about the exception and can't catch it by type — they must catch broadly. It defeats the purpose of checked exceptions.
> 
> > [!danger] Ловушка-продолжение  
> > _«А почему тогда трюк вообще компилируется?»_ → Из-за стирания типов: `E extends Throwable` в рантайме стёрт, каст `(E) e` не порождает проверки, а на этапе компиляции `E` выводится как unchecked-тип. Слабый ответ — «это баг компилятора»; на деле это следствие erasure.

> [!question]- Почему для асинхронных задач используют `Callable`, а не `Supplier`?  
> **RU:** Потому что `Callable.call()` объявлен как `throws Exception` — в него можно положить лямбду с checked-исключением, а в `Supplier.get()` (без `throws`) — нельзя. Поэтому `ExecutorService.submit` и подобные API принимают `Callable`. `CompletableFuture.supplyAsync`, наоборот, берёт `Supplier` — там checked придётся оборачивать вручную.  
> **EN:** Because `Callable.call()` is declared `throws Exception` — you can put a lambda with a checked exception in it, unlike `Supplier.get()` (no `throws`). That's why `ExecutorService.submit` and similar accept `Callable`. `CompletableFuture.supplyAsync`, conversely, takes a `Supplier`, so checked exceptions there need manual wrapping.

> [!question]- В чём разница `RuntimeException` и `UncheckedIOException` при оборачивании?  
> **RU:** `UncheckedIOException` — специализированная unchecked-обёртка именно для `IOException`, добавленная в JDK 8. Она типизирована (её можно поймать отдельно) и её `getCause()` гарантированно `IOException`. Голый `RuntimeException` теряет эту типизацию — наверху сложнее отличить I/O-ошибку от прочих. Для I/O предпочтителен `UncheckedIOException`.  
> **EN:** `UncheckedIOException` is a specialized unchecked wrapper for `IOException`, added in JDK 8. It's typed (catchable separately) and its `getCause()` is guaranteed to be an `IOException`. A bare `RuntimeException` loses that typing — harder to distinguish I/O failures upstream. Prefer `UncheckedIOException` for I/O.
