В 9.1 мы разобрались, _куда_ кладётся лямбда — в функциональный интерфейс. Теперь про то, _как_ она пишется. До Java 8 «передать поведение» означало анонимный класс: пять строк церемонии ради одной строки логики. Лямбда — это тот же анонимный класс, с которого компилятор снял всё, что и так однозначно выводится из контекста. Остаётся суть: параметры слева, тело справа.

```java
// Было (анонимный класс) — 2016 год
cats.sort(new Comparator<Cat>() {
    @Override public int compare(Cat a, Cat b) {
        return Integer.compare(a.age(), b.age());
    }
});

// Стало (лямбда)
cats.sort((a, b) -> Integer.compare(a.age(), b.age()));
```

> [!abstract] Ментальная модель  
> Лямбда — это «параметры → тело». Стрелка `->` разделяет их. Всё остальное — типы параметров, `return`, фигурные скобки — компилятор достраивает сам, зная тип-мишень (target). Поэтому одну и ту же лямбду можно записать от самой полной формы до самой короткой.

### Как это работает: лестница сокращений

Возьмём функциональный интерфейс `AgeOf { int of(Cat c); }` и запишем одну и ту же лямбду четырьмя способами — все они компилируются и делают ровно одно и то же (проверено на JDK 21):

![[Pasted image 20260715121033.png]]

Разберём, что именно снимается на каждой ступени:

**1. Типы параметров.** В полной форме `(Cat c)` тип указан явно. Но компилятор и так знает: target-тип `AgeOf` объявляет `int of(Cat c)` — значит, единственный параметр обязан быть `Cat`. Поэтому тип можно опустить: `(c) -> ...`. Это и есть **вывод типов (type inference)**: типы параметров лямбды берутся из сигнатуры абстрактного метода функционального интерфейса.

**2. Скобки вокруг параметра.** Если параметр ровно один и его тип не указан явно — скобки не нужны: `c -> ...`. Это чисто синтаксическое сокращение.

**3. Фигурные скобки, `return` и `;`.** Если тело — это одно выражение, значение которого и есть результат, можно писать его напрямую, без блока: `c -> c.age()`. Такое тело называется **тело-выражение** (expression body), и оно не заканчивается точкой с запятой.

Последняя форма — самая идиоматичная, её ты будешь видеть в 95% кода.

#### Крайние формы: ноль и много параметров

```java
Runnable r = () -> System.out.println("мяу");        // 0 параметров — пустые скобки обязательны
BiFunction<Integer,Integer,Integer> sum = (x, y) -> x + y;   // 2+ параметров — скобки обязательны
```

Правило про скобки простое: опустить их можно **только** для единственного параметра без явного типа. Ноль параметров → `()`. Два и больше → всегда `(...)`.

#### Тело-выражение против тела-блока

Тело бывает двух видов, и это не только про краткость:

```java
// тело-выражение: значение выражения = результат, без return, без ;
AgeOf a = c -> c.age();

// тело-блок: обычный код в { }, return обязателен если метод что-то возвращает
AgeOf b = c -> {
    int age = c.age();
    return age * 2;   // без return не скомпилируется — метод возвращает int
};
```

Одна тонкость, которая часто удивляет: тело-выражение может **возвращать значение, которое просто отбрасывается**, если target-метод — `void`. Например, `List.add` возвращает `boolean`, но `Consumer.accept` — `void`, и это законно (проверено):

```java
Consumer<String> collect = name -> cats.add(name);   // add() вернёт boolean, значение отброшено — ОК
```

### У лямбды нет своего типа — она берёт его из target

Это ключевой мид+ момент, на котором спотыкаются. Сама по себе лямбда `() -> {}` **не имеет типа**. Один и тот же текст лямбды может стать разными типами в зависимости от того, куда его кладут:

![[Pasted image 20260715121056.png]]

Отсюда практическое следствие: `var` для лямбды не работает, потому что `var` сам выводит тип из правой части — а у лямбды справа типа нет, получается замкнутый круг. JDK 21 говорит прямо:

```java
var f = c -> c.length();
// error: cannot infer type for local variable f
//   (lambda expression needs an explicit target-type)
```

Target-тип берётся из одного из трёх мест: тип переменной (`Runnable r = ...`), тип параметра метода (`cats.sort(...)`), или тип возвращаемого значения метода (`return c -> ...`). Пока лямбда не «приземлилась» в одно из них — типа у неё нет.

#### `var` в параметрах лямбды (Java 11, JEP 323)

Отдельно не путать с предыдущим: `var` **внутри** списка параметров лямбды — можно, с Java 11:

```java
BiFunction<Integer,Integer,Integer> sum = (var x, var y) -> x + y;   // ОК с Java 11
```

Зачем это нужно, если типы и так выводятся? Единственная реальная причина — навесить **аннотацию** на параметр (например, `(@NonNull var x, @NonNull var y) -> ...`): к голому `x` без типа аннотацию не прицепишь. Главное правило — **всё или ничего**: либо `var` у всех параметров, либо ни у одного; смешивать с явными типами или с голыми именами нельзя (проверено):

```java
(var x, Integer y) -> x + y   // error: cannot mix 'var' and explicitly-typed parameters
(var x, y) -> x + y           // error: cannot mix 'var' and implicitly-typed parameters
```

> [!tip] 💡 Из C#  
> Синтаксис почти зеркальный, отличается стрелка: в C# это `=>`, в Java — `->`. Statement-тело (`c => { return c.Age; }`) и expression-тело (`c => c.Age`) есть в обоих языках.  
> Одно принципиальное расхождение: в C# лямбда может компилироваться **в дерево выражений** (`Expression<Func<...>>`) — структуру данных, которую можно разобрать в рантайме (на этом стоит LINQ-to-SQL, EF). В Java деревьев выражений нет: лямбда всегда компилируется в исполняемый код, интроспектировать её тело нельзя. Ещё нюанс — в C# `var` в параметрах лямбды не разрешён вовсе, а в Java с 11-й версии можно (и с тем же правилом «всё или ничего», что и для обычных `var`).

> [!example]- 🔬 Глубже / на синьора  
> **Target typing и перегрузка.** Раз тип лямбды зависит от контекста, перегрузка метода двумя функциональными интерфейсами создаёт неоднозначность. Если тело-выражение годится и туда, и туда — компилятор не выберет:
> 
> 
> 
> ```java
> interface Handler { void handle(String s); }        // void
> interface Mapper  { String handle(String s); }       // возвращает String
> static void run(Handler h) {}
> static void run(Mapper m) {}
> run(s -> s.toUpperCase());   // error: reference to run is ambiguous
> //   both method run(Handler) and method run(Mapper) match
> ```
> 
> Лечится явным приведением: `run((Mapper) s -> s.toUpperCase())`. На практике — не делай две перегрузки с «похожими» функциональными интерфейсами.
> 
> **Void-compatible vs value-compatible блок.** Блочное тело обязано быть либо целиком «возвращающим значение», либо целиком «void». Нельзя, чтобы одна ветка возвращала значение, а другая — молча проваливалась:
> 
> 
> 
> ```java
> Function<Integer,Integer> f = x -> {
>     if (x > 0) return x;
>     System.out.println("no return");   // другая ветка не возвращает
> };
> // error: incompatible types: bad return type in lambda expression / missing return value
> ```
> 
> **Как это устроено в байткоде** — через `invokedynamic` и `LambdaMetafactory`, и именно поэтому лямбда ≠ анонимный класс (отдельного `.class`-файла не создаётся). Это подробно в 9.8 — здесь важно только, что тип лямбды разрешается на этапе компиляции через target typing, а сам объект создаётся уже в рантайме.
> 
> **Захват переменных** (лямбда обращается к локальной переменной снаружи) требует `effectively final` и ведёт себя не так, как замыкания C#, — это тема 9.5.

> [!warning] ⚠️ Подводные камни
> 
> - **`var f = () -> ...` не компилируется.** Лямбде нужен явный target-тип, `var` его не даёт. Пиши тип интерфейса слева.
> - **Скобки нельзя опускать при явном типе.** `Cat c -> ...` — ошибка; либо `(Cat c) -> ...`, либо `c -> ...`.
> - **`var` в параметрах — всё или ничего.** Нельзя `(var x, y)` или `(var x, Integer y)`.
> - **Перегрузка двумя функциональными интерфейсами → неоднозначность.** Компилятор не угадает, нужен явный каст или отказ от перегрузки.
> - **Тело-блок обязан возвращать значение на всех путях, если метод не `void`.** «Забыл `return` в одной ветке» — ошибка компиляции, а не тихий `null`.
> - **Точка с запятой.** У тела-выражения её нет (`c -> c.age()`), у тела-блока — есть внутри `{ ... ; }`. Лишняя `;` после выражения — ошибка.

### На собесе

> [!question]- Сократи `(Cat c) -> { return c.age(); }` до минимума. Что откуда берётся?  
> **RU:** Минимум — `c -> c.age()`. Тип `Cat` опускается, потому что выводится из target-типа (сигнатуры абстрактного метода функционального интерфейса). Скобки вокруг `c` убираются, так как параметр один и без явного типа. `{ }`, `return` и `;` уходят, потому что тело — одно выражение, чьё значение и есть результат.  
> **EN:** Minimal form is `c -> c.age()`. The `Cat` type is dropped — it's inferred from the target type (the functional interface's abstract method signature). The parentheses go because there's a single implicitly-typed parameter. The braces, `return`, and semicolon go because the body is a single expression whose value is the result.

> [!question]- Почему `var f = () -> {};` не компилируется?  
> **RU:** Потому что у лямбды нет собственного типа — он выводится из target. `var` тоже выводит тип из правой части. Получается, что оба ждут друг друга, и компилятору не от чего оттолкнуться. Нужен явный тип функционального интерфейса: `Runnable f = () -> {};`.  
> **EN:** A lambda has no type of its own — it's inferred from the target. `var` also infers from the right-hand side. So both wait on each other and the compiler has nothing to anchor on. You need an explicit functional interface type: `Runnable f = () -> {};`.
> 
> > [!danger] Ловушка-продолжение  
> > _«А `var` внутри параметров лямбды — тоже нельзя?»_ → **Можно, с Java 11 (JEP 323).** Это разные вещи: `var f = ...` слева от лямбды запрещён, а `(var x, var y) -> ...` в параметрах разрешён — по правилу «всё или ничего». Слабый ответ путает эти два случая.

> [!question]- Что выведет? `Consumer<String> c = s -> list.add(s);` — `add` же возвращает `boolean`.  
> **RU:** Скомпилируется и отработает нормально. `Consumer.accept` — `void`, а тело-выражение может возвращать значение, которое просто отбрасывается. Возвращаемый `boolean` от `add` игнорируется. Ошибки нет.  
> **EN:** It compiles and runs fine. `Consumer.accept` is `void`, and an expression body may return a value that gets discarded. The `boolean` from `add` is simply ignored.

> [!question]- Можно ли записать лямбду без скобок вокруг параметра, если у него указан тип?  
> **RU:** Нет. Опускать скобки можно только для единственного параметра **без** явного типа. `(Cat c) -> ...` — со скобками, `c -> ...` — без типа и без скобок. А `Cat c -> ...` не скомпилируется.  
> **EN:** No. Parentheses can only be dropped for a single parameter with no explicit type. `(Cat c) -> ...` keeps parens; `c -> ...` drops both type and parens. `Cat c -> ...` won't compile.

> [!question]- Почему перегрузка метода двумя функциональными интерфейсами — плохая идея?  
> **RU:** Потому что тип лямбды выводится из контекста, и если тело подходит под обе перегрузки, вызов становится неоднозначным — `reference to ... is ambiguous`. Разрешается явным приведением к нужному интерфейсу, но лучше просто не делать две перегрузки с похожими функциональными типами.  
> **EN:** Because a lambda's type is inferred from context, and if the body fits both overloads, the call is ambiguous — `reference to ... is ambiguous`. You can fix it with an explicit cast to the target interface, but the better move is to avoid overloading with similar functional types.

> [!question]- В чём разница тела-выражения и тела-блока?  
> **RU:** Тело-выражение — одно выражение без `{ }` и без `return`; его значение автоматически становится результатом (`c -> c.age()`). Тело-блок — код в `{ }`, где `return` обязателен, если метод возвращает значение. Блок обязан быть согласованным: либо всюду возвращает значение, либо всюду `void`.  
> **EN:** An expression body is a single expression without braces or `return`; its value becomes the result (`c -> c.age()`). A block body is code in braces where `return` is required if the method returns a value. A block must be consistent — either value-returning on all paths, or `void` throughout.
