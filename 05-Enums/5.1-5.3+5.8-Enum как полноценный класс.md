В Java есть целый класс багов, который в C# ловится только руками, а здесь не возникает в принципе. Твой коллега из C#-мира пишет `OrderStatus status = (OrderStatus) 999;` — компилируется, запускается, в `status` лежит мусор `999`, а через три экрана `switch` по нему молча проваливается в `default`, и заказ уезжает неоплаченным. В Java такой строчки просто **не существует** — она не компилируется, потому что `enum` тут не число с ярлыком, а `final`-класс с фиксированным набором объектов. Каст класса в число невозможен — не во что. Из этого одного факта выпадает почти всё остальное про enum: почему `==` безопасен, почему нельзя `new`, почему потокобезопасно без единого `synchronized`. Разбираемся слоями — сперва осязаемое, механику вскроем на третьем шаге.

**Mental model.** `C# enum` — это именованное целое (тонкая обёртка над `int`). `Java enum` — это `final`-класс, а каждая его константа — единственный на всю JVM объект-синглтон этого класса. Держи эту рамку сквозной: **число с именем vs класс с фиксированным набором объектов.**

### 1. Объявление, константы, `switch`

**How it works.** Базовый синтаксис обманчиво похож на C#:

```java
enum OrderStatus { NEW, PAID, SHIPPED, CANCELLED }
```

`NEW`, `PAID` и компания — это **константы**. Обращаешься через имя типа: `OrderStatus.PAID`. Каждая — отдельное, единственное в своём роде значение этого типа. Из коробки у любой константы есть четыре метода (детально с граблями — в §4):

```java
OrderStatus.PAID.name()        // "PAID"     — строковый идентификатор дословно как в коде
OrderStatus.PAID.ordinal()     // 1          — индекс в порядке объявления (NEW=0, PAID=1…)
OrderStatus.valueOf("SHIPPED") // SHIPPED     — обратный лукап по имени
OrderStatus.values()           // [NEW, PAID, SHIPPED, CANCELLED]
```

Дальше — `switch`. И тут первая тонкость, на которой спотыкаются пришедшие из C#. В классической форме метку принято писать **без квалификатора** — `case NEW`, а не `case OrderStatus.NEW`:

```java
String describe(OrderStatus s) {
    return switch (s) {
        case NEW       -> "ждёт оплаты";
        case PAID      -> "оплачен, собираем";
        case SHIPPED   -> "уехал";
        case CANCELLED -> "отменён";
    };
}
```

Важное уточнение под Java 21, которое многие учебники дают устаревшим. Раньше `case OrderStatus.NEW` был **ошибкой компиляции** — метка обязана была быть простым именем. Начиная с Java 21 (JEP 441) квалифицированные имена констант в `case` **разрешены**, и их можно даже мешать с неквалифицированными в одном `switch` (проверено на JDK 21):

```java
switch (s) {
    case OrderStatus.NEW -> "new";   // квалифицированно — на Java 21 ОК (JEP 441)
    case PAID            -> "paid";  // рядом неквалифицированно — тоже ОК
    ...
}
// компилируется и печатает: new / paid
```

Так что «`case Type.CONST` никогда не компилится» — теперь неверное утверждение и хороший собесный капкан наоборот. Но неквалифицированная форма остаётся общепринятой конвенцией: компилятор и так знает тип из выражения в `switch`.

> [!tip] 💡 Из C#  
> Тут пропасть, из которой растёт вся глава. В C# `enum` — value-тип, тонкая обёртка над целым (`int` по умолчанию). Отсюда всё следствие: кастится в число и обратно, `(OrderStatus)999` валидно по типу, флаги делаются через `[Flags]` + битовые операции, а полей/методов/конструктора у самого enum нет — только extension-методы сбоку. В Java `enum` — это **тип-класс**: в `int` не кастится вообще (`(int) status` не скомпилируется), невалидного значения создать нельзя, а полей и поведения у него может быть сколько угодно (§2). Если в C# тебе нужен «enum с поведением», ты обычно лепишь обычный класс с приватным конструктором и статическими `readonly`-инстансами (smart enum) — и это, забавно, ровно то, во что Java enum разворачивается под капотом (§3).

> [!warning] ⚠️ Подводные камни
> 
> 1. **Рефлекс `case Status.PAID:` со старым синтаксисом.** До Java 21 — ошибка компиляции; с 21 квалификатор легален, но с двоеточием (`:`) это уже classic-форма со своими правилами fall-through. Не путай стрелочную и двоеточную формы.
> 2. **`ordinal()` кажется удобным идентификатором** — это ловушка, разбор в §4.

#### На собесе

> [!question]- Чем Java enum отличается от C# enum? _(топ главы, спрашивают почти всегда)_  
> **RU:** C# enum — именованная целочисленная константа, обёртка над `int`: кастится в число, `(T)999` создаёт валидный по типу мусор, полей и методов у самого enum нет. Java enum — полноценный `final`-класс; каждая константа — объект-синглтон, в `int` не кастится, невалидного значения не бывает, а полей и поведения может быть сколько угодно.
> 
> **EN:** A C# enum is a named integral constant, a wrapper over `int`: it casts to a number, `(T)999` yields a type-valid but meaningless value, and the enum itself carries no fields or methods. A Java enum is a full `final` class; each constant is a singleton object, it doesn't cast to `int`, invalid values can't exist, and it can carry arbitrary fields and behavior.
> 
> _Слабый ответ:_ «то же самое, только синтаксис другой» — теряет главное (класс vs число), после чего валится любой follow-up про `new`, `==`, потокобезопасность.

### 2. Конструктор и поля: enum хранит состояние

**How it works.** Раз enum — класс, у него могут быть конструктор и поля. Классический пример — валюта, у которой есть символ и число знаков после запятой (в банковской логике у йены их 0, у динара Бахрейна — 3):

```java
enum Currency {
    USD("$", 2),
    EUR("€", 2),
    JPY("¥", 0),        // у йены нет «копеек»
    BHD(".د.ب", 3);     // 3 знака после запятой  ← ; обязательна перед членами

    private final String symbol;
    private final int minorUnits;

    Currency(String symbol, int minorUnits) {   // конструктор — неявно private
        this.symbol = symbol;
        this.minorUnits = minorUnits;
    }

    public String symbol()  { return symbol; }
    public int minorUnits() { return minorUnits; }
}
```

Каждая константа теперь не просто «имя» — это объект, в который при создании зашили данные: `USD` знает, что её символ `$` и знаков 2; `JPY` знает, что знаков 0. Это уже не «именованный `int`», а набор готовых, заранее сконфигурированных объектов. Разберём плотный синтаксис по косточкам:

- **Аргументы у константы** — это вызов конструктора. `USD("$", 2)` означает «создай константу `USD`, передав в конструктор `"$"` и `2`».
- **Конструктор неявно `private`.** Вызвать снаружи физически нельзя: `new Currency("£", 2)` не скомпилируется (JLS §8.9.2). Единственные, кто его вызывает, — сами объявления констант, ровно по разу.
- **Поля `private final.**` final` не обязателен синтаксически, но это правильная дисциплина: константа сконфигурирована раз и навсегда (Effective Java, Item 34 — мутабельный enum это запах).
- **Точка с запятой** после списка констант обязательна, когда дальше идут поля/методы/конструктор. Без членов — не нужна. Это единственное место в Java, где `;` ставится после такого списка.

> [!tip] 💡 Из C#  
> В C# `enum Currency { USD, EUR, JPY }` — и всё, привязать к `USD` символ и число знаков нельзя. Данные живут **отдельно**: словарь, атрибуты или extension-метод со `switch` по значению. В Java данные живут **внутри** константы, как поля объекта — прямое следствие того, что константа это полноценный инстанс класса.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Мутабельное поле в enum** (`private int counter;` без `final`, с сеттером). Технически компилируется. Но константа — синглтон на всю JVM (§3), и мутабельное поле превращается в глобальное разделяемое состояние со всеми гонками. Тянет хранить в enum изменяемое — почти всегда нужен обычный класс.
> 2. **Забытая `;`** перед полями — путаная ошибка компиляции, если не знать правила.
> 3. **Тяжёлая работа в конструкторе** (I/O, обращение к другому enum, который ещё инициализируется) — конструкторы отрабатывают в `<clinit>` при загрузке класса; исключение оттуда прилетит как `ExceptionInInitializerError`, а не там, где ты ждёшь.

### 3. Во что enum разворачивается на самом деле — ядро

Ты объявил три строчки — `enum Color { RED, GREEN, BLUE }`, — а `javap -p` показывает вот такой класс (это реальный вывод декомпилятора на JDK 21, не иллюстрация):

```
final class Color extends java.lang.Enum<Color> {
  public static final Color RED;
  public static final Color GREEN;
  public static final Color BLUE;
  private static final Color[] $VALUES;
  public static Color[] values();
  public static Color valueOf(java.lang.String);
  private Color();                 // реальный дескриптор ctor — (String, int)
  static {};                       // <clinit> — здесь создаются все константы
}
```

Разберём, что тут важно:

- **Суперкласс — `java.lang.Enum`.** Твой enum неявно `extends java.lang.Enum<Color>` — компилятор дописал. Поэтому enum нельзя наследовать от другого класса: слот единственного наследования уже занят (`enum Apple extends Fruit` даже не парсится — проверено).
- **Класс `final`.** Никто не может `extends Color`. Набор констант закрыт навсегда — отсюда «фиксированный конечный набор».
- **Конструктор `private` с сюрпризом в сигнатуре.** Я объявил enum без конструктора, а он есть и принимает `(String, int)`. Байткод `<init>` подтверждает: `invokespecial Enum.<init>:(Ljava/lang/String;I)V` — компилятор передаёт **имя константы и её ordinal** в `super(name, ordinal)`. Каждый enum-конструктор втихую получает эти два параметра первыми. Твоя `Currency(String symbol, int minorUnits)` на деле — `Currency(String name, int ordinal, String symbol, int minorUnits)`.
- **Константы — `public static final` поля.** Не числа. Полноценные объекты типа `Color`.
- **Массив `$VALUES`** — `private static final Color[]`. Вот он, источник `values()`: метод отдаёт **клон** этого массива (`getstatic $VALUES; invokevirtual clone(); areturn` в байткоде — вернёмся в §4).

`<clinit>` (статический инициализатор) в байткоде буквально делает `new Color; dup; ldc "RED"; iconst_0; invokespecial <init>; putstatic RED` — и так для каждой константы. То есть твой enum концептуально эквивалентен вот такому рукописному классу:

```java
final class Color extends Enum<Color> {
    public static final Color RED   = new Color("RED", 0);
    public static final Color GREEN = new Color("GREEN", 1);
    public static final Color BLUE  = new Color("BLUE", 2);

    private static final Color[] $VALUES = { RED, GREEN, BLUE };

    private Color(String name, int ordinal) { super(name, ordinal); }
    public static Color[] values() { return $VALUES.clone(); }
}
```

Руками так не напишешь — `extends Enum` компилятор запрещает специально, чтобы никто не подделал enum. Но JVM строит ровно это. Вот та самая картинка, которая заменяет «enum = число»:

![[Pasted image 20260708163622.png]]

Из этой одной картинки выпадают три ответа, которые на собесе спрашивают чаще всего. Все три — следствия, а не отдельные факты.

**Почему нельзя `new Color()`.** Конструктор `private`, и компилятор отдельно запрещает `new` для enum-типов. Причём запрет жёстче, чем просто приватность: `new Color2()` даёт ошибку `enum classes may not be instantiated`, а рефлексия ловится в рантайме (проверено):

```java
Constructor<Color3> c = Color3.class.getDeclaredConstructor(String.class, int.class);
c.setAccessible(true);
c.newInstance("HACK", 99);
// -> java.lang.IllegalArgumentException: Cannot reflectively create enum objects
```

Создать константу может только сам enum в момент объявления. Отсюда — «фиксированный конечный набор». Это не соглашение, а гарантия компилятора и JVM: невалидного значения, как `(OrderStatus)999` в C#, физически не бывает.

**Почему `==` для enum безопасен и предпочтителен.** Каждая константа существует ровно в одном экземпляре на всю JVM. Значит `Color.RED == Color.valueOf("RED")` всегда `true` — это буквально один и тот же объект (проверено: `identityHashCode` совпадает). `Enum.equals()` и `Enum.hashCode()` объявлены `final`, и `equals` просто делегирует в `==` (попытка переопределить любой из них — ошибка компиляции, проверено). Поэтому для enum `==` не только работает, но и лучше `.equals()`: он null-safe (`x == Color.RED` не упадёт на `null`), а опечатка в типе ловится на компиляции (`status == Color.RED`, где `status` другого enum-типа, — ошибка компиляции, тогда как `status.equals(...)` молча вернёт `false`).

**Почему потокобезопасно без единого `synchronized`.** Константы создаются eagerly в `<clinit>`. А инициализация класса в JVM потокобезопасна по спецификации: JVM держит лок на классе во время `<clinit>`, и каждый класс инициализируется ровно один раз (JLS §12.4.2). Все константы гарантированно созданы один раз, до первого обращения, без гонок — без всякой синхронизации в твоём коде. Это, кстати, и причина, почему синглтон через enum считается самым надёжным способом (Effective Java, Item 3) — но это урок 5.7, сейчас только мостик.

> [!tip] 💡 Из C#  
> В C# сравнение enum — это сравнение чисел, и по результату разницы с Java нет. Но в Java `==` несёт ещё и проверку типа на этапе компиляции, потому что enum — это тип, а не `int`: сравнить `Color` с `Size` через `==` компилятор не даст. В C# оба — числа, и `(int)Color.RED == (int)Size.SMALL` спокойно скомпилируется в «правдоподобную» бессмыслицу.

> [!example] 🔬 Глубже / на синьора  
> **`<clinit>` в байткоде** буквально таков (это реальный вывод `javap -c`):
> 
> ```
> new Color; dup; ldc "RED"; iconst_0; invokespecial <init>(String,int); putstatic RED
> new Color; dup; ldc "GREEN"; iconst_1; ...; putstatic GREEN
> ...
> invokestatic $values(); putstatic $VALUES
> ```
> 
> То есть каждая константа — `new` с передачей `(name, ordinal)`, и `<init>` немедленно делает `invokespecial Enum.<init>:(Ljava/lang/String;I)V`. Имя и ordinal хранит родительский `Enum` в приватных полях `name`/`ordinal` — твой enum к ним не имеет прямого доступа, только через `name()`/`ordinal()`.
> 
> **Рекурсивный дженерик-бинд.** Сигнатура базового класса — `Enum<E extends Enum<E>>`. Это F-bounded polymorphism: он нужен, чтобы `compareTo(E)` и `getDeclaringClass()` были типизированы точно своим же enum-типом, а не сырым `Enum`. Отсюда `Color.RED.compareTo(Size.SMALL)` — ошибка компиляции, а не рантайм. Тот же приём ты встретишь в `Comparable<T>`-иерархиях (глава 7).

> [!warning] ⚠️ Подводные камни
> 
> 1. **`.equals()` «на всякий случай».** Раз `==` корректен — не пиши `.equals()` для enum и тем более не сравнивай через `name()`: `a.name().equals(b.name())` работает, но это аллокация/сравнение строк, потеря type-safety и лишняя медлительность там, где `a == b` — одна инструкция сравнения ссылок.
> 2. **Ожидание, что рефлексия «пробьёт» enum.** Не пробьёт: `newInstance` бросит `IllegalArgumentException` даже с `setAccessible(true)`.
> 3. **Инициализация «по кругу».** Если конструктор одного enum обращается к другому enum, который ещё не инициализирован, получишь `null` или `ExceptionInInitializerError` — классическая ловушка с взаимозависимыми enum.

#### На собесе

> [!question]- Почему `new` на enum нельзя, и почему `==` для него безопасен? _(ядро темы)_  
> **RU:** enum разворачивается в `final`-класс, чьи константы — `public static final` синглтоны, созданные один раз в статическом инициализаторе. Конструктор неявно `private`, и `new` для enum-типов запрещён компилятором, а рефлексия бросает `IllegalArgumentException` — поэтому невалидного значения не существует. Раз каждая константа — единственный объект на всю JVM, `==` сравнивает идентичность и всегда корректен; вдобавок он null-safe и ловит несовпадение enum-типов на компиляции, тогда как `equals` (объявленный `final` и делегирующий в `==`) таких плюсов не даёт.
> 
> **EN:** An enum compiles to a `final` class whose constants are `public static final` singletons created once in the static initializer. The constructor is implicitly `private` and `new` on enum types is forbidden by the compiler, while reflection throws `IllegalArgumentException` — so an invalid value can't exist. Since each constant is the single instance in the JVM, `==` compares identity and is always correct; it's also null-safe and catches enum-type mismatches at compile time, whereas `equals` (which is `final` and just delegates to `==`) adds nothing.
> 
> _Follow-up:_ «почему тогда потокобезопасно без `synchronized`?» → инициализация класса под локом JVM, JLS §12.4.2. _Trap:_ «а если очень надо два экземпляра одной константы?» → нельзя, это и есть гарантия.
> 
> _Слабый ответ:_ «enum это синглтоны, потому что так удобно» — не назван механизм (`<clinit>`, `final`-класс, приватный конструктор), после чего валится любой «почему».

### 4. Встроенные методы и где они кусают

**How it works.** Четыре метода из §1, теперь с граблями (всё проверено на JDK 21).

**`values()` клонирует массив на каждый вызов.** `values() == values()` → `false`, но элементы те же синглтоны (`a[0] == b[0]` → `true`). Это защитная копия приватного `$VALUES`: массивы в Java мутабельны, и без копии любой мог бы испортить внутренний массив enum'а. Проверено — испортил локальную копию, `Currency.values()` остался цел. Решение библиотеки правильное, но у него есть цена:

```java
// ПЛОХО: новый Currency[] на каждой итерации
for (int i = 0; i < 1_000_000; i++) process(Currency.values()[i % 3]);

// ХОРОШО: один клон, переиспользуем
private static final Currency[] VALUES = Currency.values();
```

Это ровно то, что делает сам JDK: `EnumSet`/`EnumMap` держат закешированную копию, чтобы не клонировать без нужды.

**`valueOf(String)` бросает, а не возвращает `null`.** Несуществующее имя → `IllegalArgumentException: No enum constant Currency.usd`, и он **чувствителен к регистру** (`valueOf("usd")` тоже бросает — проверено). Для внешнего входа голым его звать нельзя:

```java
public static Optional<Currency> fromCode(String s) {
    if (s == null) return Optional.empty();
    try { return Optional.of(Currency.valueOf(s.trim().toUpperCase())); }
    catch (IllegalArgumentException e) { return Optional.empty(); }
}
```

**`ordinal()` — индекс объявления, персистить нельзя.** Классика собеса: сохранил в БД `ordinal()` валют (`USD`=0, `EUR`=1, `JPY`=2). Через полгода кто-то добавил `GBP` между `USD` и `EUR` по алфавиту — теперь `EUR` стал `ordinal`=2. Все старые записи молча поехали: что было `EUR`, читается как `JPY`. Никакой ошибки — просто тихо неправильные деньги. **Персисти `name()` или своё явное поле `code`, никогда не `ordinal()`** (Effective Java, Item 35). `ordinal()` существует ради `EnumSet`/`EnumMap` — Javadoc прямо пишет, что он не предназначен для прикладного кода.

**`name()` vs `toString()` — частый вопрос.** `name()` объявлен `final` и всегда возвращает идентификатор дословно как в коде — переопределить его нельзя (проверено, ошибка компиляции). `toString()` по умолчанию возвращает то же, что `name()`, но его **можно переопределить** для человекочитаемого вывода:

```java
enum Planet {
    MERCURY, VENUS, EARTH;
    @Override public String toString() { return "Planet:" + name().toLowerCase(); }
}
Planet.EARTH.name()      // "EARTH"        — final, для логики/сериализации
Planet.EARTH.toString()  // "Planet:earth" — переопределён, для отображения
```

Правило: для логики, персистенса и лукапа — `name()`; для показа пользователю — `toString()`. `valueOf` работает по `name()`, не по `toString()`.

**`Comparable` по `ordinal`.** Естественный порядок = порядок объявления (`MERCURY.compareTo(EARTH)` = `-2`). `Enum.compareTo` объявлен `final` — переопределить нельзя (проверено). Удобно для `TreeSet`/`sorted()`, но это та же мина, что и `ordinal`: переставил константы — поменялась сортировка. Нужен устойчивый порядок — задавай `Comparator` явно.

> [!tip] 💡 Из C#  
> В C# `Enum.Parse` тоже бросает, а `Enum.TryParse` возвращает `bool` + `out` — прямого `TryParse` в Java нет, его роль играет твоя обёртка вокруг `valueOf` с `Optional`. И ключевая разница: в C# нужен `Enum.IsDefined`, потому что `(Currency)42` создаёт валидный по типу мусор. В Java мусора не бывает — либо имя совпало и ты получил настоящий синглтон, либо `IllegalArgumentException`. Третьего нет.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`values()` в горячем цикле** — аллокация массива на каждой итерации; кешируй в `static final`.
> 2. **`valueOf` на данных из JSON/БД/фронта** без обёртки — падение `IllegalArgumentException` на невинном `"usd"`.
> 3. **`ordinal()` в БД или в протоколе** — реордер/вставка константы тихо ломает старые данные.
> 4. **Сортировка полагается на порядок объявления** — тот же реордер незаметно меняет `sorted()`/`TreeSet`.
> 5. **`valueOf` по `toString()`** — не работает: `valueOf` знает только `name()`. Переопределил `toString()` для показа — обратный парсинг делай отдельным методом.

#### На собесе

> [!question]- Что вернёт `values()` и есть ли у него цена?  
> **RU:** `values()` возвращает свежий массив на каждый вызов — защитную копию (`clone()`) приватного `$VALUES`, потому что массивы мутабельны и внутренний массив нельзя отдавать наружу. Элементы — те же синглтоны, но сам массив каждый раз новый, поэтому в горячем цикле это лишние аллокации; кешируй в `static final`.
> 
> **EN:** `values()` returns a fresh array each call — a defensive `clone()` of the private `$VALUES`, because arrays are mutable and the internal one must not leak. The elements are the same singletons but the array is new every time, so in a hot loop it's needless allocation; cache it in a `static final`.

> [!question]- Почему нельзя хранить `ordinal()` в базе? _(классика)_  
> **RU:** `ordinal()` — это позиция в текущем порядке объявления, а не стабильный идентификатор. Вставка или перестановка констант сдвигает ordinal, и все ранее сохранённые значения начинают указывать на другие константы — молча, без ошибки. Персистить нужно `name()` или собственный явный код; `ordinal()` предназначен только для `EnumSet`/`EnumMap`.
> 
> **EN:** `ordinal()` is the position in the current declaration order, not a stable id. Inserting or reordering constants shifts ordinals, and every previously stored value silently starts pointing at a different constant — no error. Persist `name()` or an explicit code of your own; `ordinal()` exists only for `EnumSet`/`EnumMap`.
> 
> _Trap:_ «а `name()` безопасно?» — да, если не переименовываешь константы; переименование — это осознанная миграция.

> [!question]- Разница между `name()` и `toString()`?  
> `name()` — `final`, дословный идентификатор из кода, для логики/лукапа/персистенса; `valueOf` работает по нему. `toString()` дефолтит в `name()`, но переопределяем — для человекочитаемого показа. _Trap:_ «`valueOf` разберёт переопределённый `toString()`?» — нет, только `name()`.

### 5. `switch` по enum — под капотом

**How it works.** `switch` по enum на Java 21 умеет три полезные вещи, которых стоит коснуться.

**Exhaustiveness (исчерпываемость).** В `switch`-**expression** над enum, где перечислены все константы, `default` не нужен — компилятор сам видит, что случаи покрыты:

```java
int rate(Currency c) {
    return switch (c) {       // все константы покрыты — default не требуется
        case USD -> 1;
        case EUR -> 2;
        case JPY -> 3;
    };
}
```

**`null` кидает NPE — кроме `case null` (Java 21).** Классический `switch` (и expression-форма без `case null`) на `null`-enum бросает `NullPointerException` ещё до входа в кейсы, потому что под капотом дёргается `ordinal()` на `null` (проверено: и statement-, и expression-форма → NPE). С Java 21 (JEP 441) появился `case null`:

```java
String s = switch (status) {
    case null   -> "было null";   // Java 21 — раньше это был бы NPE
    case OPEN   -> "открыт";
    case CLOSED -> "закрыт";
};
```

**Квалифицированные метки.** Как разобрали в §1 — на Java 21 `case OrderStatus.NEW` легален (JEP 441) и в expression-, и в statement-форме.

> [!example] 🔬 Глубже / на синьора  
> **Как `switch` по enum компилируется (проверено `javap` на JDK 21).** Он идёт **не** напрямую по `ordinal()`. `javac` генерирует синтетический класс-компаньон (`Client$1`) с полем `$SwitchMap$Currency` — массивом `int[]`, который мапит `ordinal()` константы в плотный индекс кейса, и уже по нему делает `tableswitch`/`lookupswitch`. Это верно и для classic-, и для expression-формы.

> **Exhaustiveness — не гарантия навсегда.** Синтетический `default`, который `javac` вставляет в исчерпывающий `switch`-expression, **бросает `MatchException`** (на JDK 21; раньше был `IncompatibleClassChangeError`). Это важно при раздельной компиляции. Я проверил на разнесённых компиляциях: класс со `switch` собран против enum с `{USD, EUR, JPY}`; затем в enum добавили `CHF` и перекомпилировали **только enum**; на известных константах старый `switch` работает, а на новой `CHF` — `java.lang.MatchException` в рантайме. Практический вывод: исчерпывающий enum-`switch` без `default` безопасен только пока оба класса компилируются вместе. В долгоживущем коде против внешнего enum добавляй явный `default` — иначе новая константа даст рантайм-сюрприз, а не «тихий проход».

> [!warning] ⚠️ Подводные камни
> 
> 1. **`null`-enum в `switch`** → `NullPointerException` до входа в кейсы (кроме `case null`, Java 21). Проверяй на `null` заранее или добавляй `case null`.
> 2. **Исчерпывающий `switch` без `default` против внешнего/эволюционирующего enum** → новая константа = `MatchException` в рантайме при раздельной компиляции.
> 3. **Расчёт на конкретный байткод `switch`** (`$SwitchMap`) — это деталь реализации `javac`, JLS её не требует; не строй на ней логику.
> 4. **Смешение `:` и `->` форм** в одном `switch` — не компилируется; выбери одну.

#### На собесе

> [!question]- Что произойдёт при `switch` по `null`-enum?  
> **RU:** Классический `switch` и expression-форма без `case null` бросят `NullPointerException` до входа в кейсы — под капотом вызывается `ordinal()` на `null`. С Java 21 (JEP 441) можно добавить `case null` и обработать его явно; до этого `null` проверяли отдельным `if` перед `switch`.
> 
> **EN:** A classic `switch`, and an expression `switch` without `case null`, throw `NullPointerException` before any case runs — internally `ordinal()` is called on `null`. Since Java 21 (JEP 441) you can add `case null` and handle it explicitly; before that you guarded `null` with an `if` ahead of the `switch`.

> [!question]- Исчерпывающий `switch` по enum без `default` — что будет, если в enum добавят константу? _(senior)_  
> **RU:** Если класс со `switch` перекомпилировали вместе с enum — компилятор снова проверит исчерпываемость и заставит обработать новую константу. Но если перекомпилировали только enum, а класс со `switch` остался старым, на новой константе в рантайме прилетит `MatchException` (JDK 21) — синтетический `default`, который компилятор вставил за тебя. Поэтому против внешнего enum лучше явный `default`.
> 
> **EN:** If the `switch` class is recompiled together with the enum, the compiler re-checks exhaustiveness and forces you to handle the new constant. But if only the enum is recompiled and the `switch` class stays stale, the new constant throws `MatchException` at runtime (JDK 21) — the synthetic `default` the compiler inserted for you. So against an external enum prefer an explicit `default`.

> [!question]- Правда ли, что `case OrderStatus.NEW` не компилируется? _(21-гоча)_  
> Уже нет. До Java 21 метка обязана была быть простым именем (`case NEW`), и квалификатор был ошибкой. С Java 21 (JEP 441) квалифицированные имена констант в `case` разрешены и смешиваются с неквалифицированными. Так что «`case Type.CONST` никогда не компилится» — устаревшее знание.
