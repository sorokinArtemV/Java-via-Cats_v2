Рано или поздно в бэкенде появляется поле «статус»: заказа, платежа, тикета. Соблазн взять `int` или `String` велик — и оба варианта дают дыру, через которую в систему затекает невалидное значение: `-1`, `999`, `"payed"` с опечаткой. Enum закрывает эту дыру на уровне типа: множество допустимых значений становится конечным, известным компилятору и непробиваемым в рантайме. Но чтобы пользоваться этим осознанно (и отвечать на собесе), надо понять одну вещь, которая у пришедших из C# в голове стоит неправильно: **Java `enum` — это не число с ярлыком, это класс.** Из этого факта выпадает вообще всё остальное поведение, так что с него и начнём.

**Mental model.** C# `enum` — это именованная обёртка над `int`. Java `enum` — это `final`-класс, а его константы — заранее созданные синглтон-объекты этого класса. Всё остальное (нельзя `new`, безопасный `==`, потокобезопасность) — следствия отсюда.

#### Слой 1. Синтаксис, константы, switch

Базовое объявление обманчиво похоже на C#:

```java
enum OrderStatus { NEW, PAID, SHIPPED, CANCELLED }
```

`NEW`, `PAID` и остальные — это **константы** enum'а. Обращаешься к ним через имя типа: `OrderStatus.PAID`. Каждая — отдельное, единственное в своём роде значение.

Дальше — `switch`. И тут первое место, где инстинкт из C# спотыкается:

```java
String describe(OrderStatus s) {
    return switch (s) {
        case NEW       -> "ждёт оплаты";
        case PAID      -> "оплачен, собираем";
        case SHIPPED   -> "уехал";
        case CANCELLED -> "отменён";
    };
}
```

В `case` пишется `NEW`, а не `OrderStatus.NEW` — и это **конвенция, а не жёсткое требование** (важная деталь, изменившаяся в Java 21, — ниже в 🔬). Компилятор выводит тип из выражения в `switch`, поэтому неквалифицированное имя достаточно и предпочтительно.

Встроенные методы, которые есть у любой константы из коробки:

- `s.name()` → `"PAID"` — строковый идентификатор дословно как в коде.
- `s.ordinal()` → `1` — индекс в порядке объявления (`NEW`=0, `PAID`=1…).
- `OrderStatus.valueOf("SHIPPED")` → константа `SHIPPED` — обратный лукап по имени.
- `OrderStatus.values()` → массив всех констант.

> [!tip] 💡 Из C#  
> В C# `enum` — это именованная целочисленная константа, тонкая обёртка над `int`. Отсюда всё следует: кастится в число и обратно, `(OrderStatus)999` валидно по типу, флаги делаются через `[Flags]` + битовые операции, а полей/методов/конструктора у самого enum нет — только extension-методы сбоку. В Java `enum` — это **тип-класс**. В `int` не кастится (`(int) status` не скомпилируется), невалидное значение создать нельзя, а полей и поведения у него может быть сколько угодно. Держи рамку: **C# enum = число с именем; Java enum = класс с фиксированным набором объектов.** Дальше всё раскручивается из неё.

#### Слой 2. Конструктор и поля — enum хранит состояние

Раз enum — класс, у его констант могут быть данные. Классический финтех-пример — валюта, у которой есть символ и число знаков после запятой (у йены их 0, у динара Бахрейна — 3, и это важно для банковского округления):

```java
enum Currency {
    USD("$", 2),
    EUR("€", 2),
    JPY("¥", 0),        // у йены нет минорной единицы
    BHD(".د.ب", 3);     // динар Бахрейна — 3 знака

    private final String symbol;
    private final int minorUnits;

    Currency(String symbol, int minorUnits) {   // конструктор
        this.symbol = symbol;
        this.minorUnits = minorUnits;
    }

    public String symbol()  { return symbol; }
    public int minorUnits() { return minorUnits; }
}
```

Разберём плотный синтаксис по косточкам:

- **Аргументы у константы.** `USD("$", 2)` — это вызов конструктора: «создай `USD`, передав `"$"` и `2`». Скобки после имени константы и есть аргументы конструктора.
- **Конструктор.** `Currency(String, int)` — обычный, но **неявно `private`**: вызвать его снаружи физически нельзя (`new Currency("£", 2)` не компилируется). Единственные вызовы — сами объявления констант, по разу на каждую.
- **Поля `private final`** — потому что enum по-хорошему иммутабелен. `final` не обязателен синтаксически, но это правильная дисциплина (Effective Java, Item 34: мутабельный enum — запах).
- **Точка с запятой** после списка констант обязательна, когда дальше идут поля/методы/конструктор. Это единственное место в Java, где `;` ставится после такого списка.

> [!tip] 💡 Из C#  
> Вот тут пропасть. В C# `enum Currency { USD, EUR, JPY }` — и всё, привязать к `USD` символ и число знаков нельзя: данные живут _отдельно_ — словарь, атрибуты или extension-метод со `switch`. В Java данные живут _внутри_ константы, как поля объекта. Если в C# тебе нужен «enum с поведением», ты делаешь обычный класс с приватным конструктором и статическими `readonly`-полями — и это ровно то, во что Java enum компилируется под капотом. Что мы сейчас и вскроем.

#### Слой 3. Во что enum разворачивается на самом деле

Ты пишешь три слова:

```java
enum Color { RED, GREEN, BLUE }
```

А JVM видит совсем другое. Я проверил рефлексией на JDK 21 — вот дословно:

```java
Color.class.getSuperclass()                 // class java.lang.Enum
Modifier.isFinal(Color.class.getModifiers()) // true
// конструктор:  private Color(java.lang.String, int)
// поля:         public static final Color RED / GREEN / BLUE
//               private static final Color[] $VALUES
```

То есть `enum Color { RED, GREEN, BLUE }` концептуально эквивалентен вот такому рукописному классу:

```java
final class Color extends Enum<Color> {
    public static final Color RED   = new Color("RED", 0);
    public static final Color GREEN = new Color("GREEN", 1);
    public static final Color BLUE  = new Color("BLUE", 2);

    private static final Color[] $VALUES = { RED, GREEN, BLUE };

    private Color(String name, int ordinal) { super(name, ordinal); }

    public static Color[] values() { return $VALUES.clone(); }
}
```

Буквально такой код ты не напишешь — `extends Enum` руками запрещён компилятором, именно чтобы никто не подделал enum. Но JVM строит ровно это. Вот та самая структура — всё, что ты видишь, живёт внутри одного класса `Color`:

![[Pasted image 20260707153255.png]]

Из этой одной картинки выпадают три вопроса, которые на собесе спрашивают чаще всего. Все три — **следствия**, а не отдельные факты.

**Почему нельзя `new Color()`.** Конструктор `private`, и компилятор отдельно запрещает `new` для enum-типов (JLS §8.9). Даже через рефлексию не выйдет — я проверил на JDK 21:

```java
Constructor<?> k = Currency.class.getDeclaredConstructors()[0];
k.setAccessible(true);
k.newInstance("FOO", 99);
// -> IllegalArgumentException: Cannot reflectively create enum objects
```

Создать константу может только сам enum в момент объявления. Отсюда «фиксированный конечный набор» — это гарантия компилятора и JVM, а не соглашение. Невалидного значения, как `(OrderStatus)999` в C#, физически не существует.

**Почему `==` для enum безопасен и предпочтителен.** Каждая константа существует ровно в одном экземпляре на всю JVM — синглтон (`identityHashCode(RED)` совпадает при повторных вызовах, проверено). Значит `Color.RED == Color.RED` всегда `true` — это буквально один и тот же объект. `Enum.equals()` и `Enum.hashCode()` объявлены `final`, и `equals` просто делегирует в `==` (тоже проверил рефлексией: оба `final`). Поэтому для enum `==` не только работает, но и лучше `.equals()`: он null-safe (`x == Color.RED` не упадёт на `null`), и опечатка в типе ловится на компиляции — `status == Size.SMALL`, где `status` другого enum-типа, не скомпилируется, а `status.equals(...)` молча вернёт `false`.

> [!tip] 💡 Из C#  
> В C# сравнение enum — это сравнение чисел, по результату разницы с Java нет. Но в Java `==` несёт ещё и проверку типа на компиляции, потому что enum — это тип, а не `int`. Сравнить `Color` с `Size` через `==` компилятор не даст. В C# оба — числа, и `(int)Color.RED == (int)Size.SMALL` спокойно скомпилируется.

**Почему потокобезопасно без единого `synchronized`.** Константы создаются _eagerly_ в статическом инициализаторе класса (`<clinit>`), а инициализация класса в JVM потокобезопасна по спецификации: JVM держит лок на классе во время `<clinit>`, и каждый класс инициализируется ровно один раз (JLS §12.4.2). Все константы гарантированно созданы до первого обращения, без гонок и без синхронизации в твоём коде. Это же — причина, почему синглтон через enum считается самым правильным способом (Effective Java, Item 3; разберём в 5.7).

Вот ядро темы: **Java enum — это `final`-класс, чьи константы суть синглтон-объекты, созданные один раз при инициализации класса.** Всё остальное — следствия.

#### Слой 4. Встроенные методы и где они кусают

**`values()` клонирует массив на каждый вызов.** Проверено: `values() == values()` → `false`, но `values()[0] == values()[0]` → `true` (те же синглтоны). Это защитная копия того самого `$VALUES` из схемы — массивы в Java мутабельны, и без клона любой мог бы испортить внутренний массив enum'а. Решение правильное, но у него есть цена.

**`valueOf(String)` бросает, а не возвращает `null`.** Несуществующее имя → `IllegalArgumentException: No enum constant …`. И он **чувствителен к регистру**: `valueOf("usd")` тоже бросает (оба случая проверены).

**`ordinal()` — индекс в порядке объявления, и его нельзя персистить.** `USD`=0, `EUR`=1, `JPY`=2. Кажется удобным для хранения в БД — и это ловушка (см. ⚠️ ниже).

**`Comparable` по `ordinal`.** Enum реализует `Comparable` автоматически, естественный порядок = порядок объявления (`USD.compareTo(JPY)` = `-2`, проверено). Причём `Enum.compareTo` объявлен `final` — переопределить естественный порядок нельзя, только внешний `Comparator`.

> [!example] 🔬 Глубже / на синьора  
> **`case OrderStatus.NEW` теперь компилируется (Java 21, JEP 441).** Историческое правило «метка `switch` по enum — только простое имя, `case NEW`» изменилось. До Java 21 квалифицированное имя (`case OrderStatus.NEW`) было **ошибкой компиляции** — «an enum switch case label must be the unqualified name». В Java 21 JEP 441 (Pattern Matching for switch) разрешил квалифицированные имена констант и в statement, и в expression. Я проверил на JDK 21: `case OrderStatus.NEW` компилируется и работает в обоих формах. Квалификация нужна прежде всего там, где селектор — **не** enum-тип (например, `sealed`-иерархия, где под одним интерфейсом лежат несколько enum'ов, и компилятор не может вывести тип). Для обычного `switch` по enum неквалифицированное имя по-прежнему идиоматично.
> 
> **`switch` под капотом: `SwitchMap` появляется только при межклассовой компиляции.** Разберём через `javap` — тут тонкость, которую путают даже опытные. > [!warning] ⚠️ Подводные камни
> 
> 1. **`ordinal()` в БД — классика собеса.** Сохранил `ordinal()` валют, через полгода кто-то добавил `GBP` между `USD` и `EUR` (по алфавиту, логично же). Теперь `EUR` стал `ordinal`=2 вместо 1 — все старые записи молча поехали: что было `EUR`, читается как `JPY`. Никакой ошибки, просто тихо неправильные деньги. **Персисти `name()` или своё явное поле `code`, никогда не `ordinal()`.** Javadoc прямо пишет: `ordinal()` «не предназначен для использования прикладным кодом» — он существует ради `EnumSet`/`EnumMap` (быстрый индекс в массив).
> 2. **`valueOf` на внешнем входе.** Прилетел JSON с `"currency": "usd"`, ты делаешь `Currency.valueOf(json.currency)` — приложение падает `IllegalArgumentException` на невинных данных (регистр!). Для внешнего входа оборачивай в свою фабрику:
>     
>     java
>     
>     ```java
>     public static Optional<Currency> fromCode(String s) {
>         if (s == null) return Optional.empty();
>         try { return Optional.of(Currency.valueOf(s.trim().toUpperCase())); }
>         catch (IllegalArgumentException e) { return Optional.empty(); }
>     }
>     ```
>     
> 3. **`values()` в горячем цикле** — аллокация нового массива на каждой итерации (это клон!). Кешируй один раз: `private static final Currency[] VALUES = values();` и работай с `VALUES`. Ровно так делает и сам JDK внутри классов вроде `EnumSet`.
> 4. **`==` vs `name()`.** Не сравнивай константы через `a.name().equals(b.name())` — работает, но это аллокация строк, потеря type-safety и медленнее там, где `a == b` — одна инструкция сравнения ссылок.
> 5. **Естественный порядок = `ordinal`.** `TreeSet<Currency>` и `sorted()` сортируют по порядку объявления. Переставил константы — поехала сортировка. Нужен устойчивый порядок (по коду) — задавай `Comparator` явно, `compareTo` переопределить нельзя (`final`).

#### На собесе

> [!question]- Почему для enum используют `==`, а не `.equals()`?  
> **RU:** Каждая константа — синглтон, единственный экземпляр на JVM, поэтому `==` сравнивает ровно то, что нужно, и делает это null-safe. Вдобавок `==` ловит несовпадение enum-типов на компиляции, а `.equals()` молча вернёт `false`. `Enum.equals` всё равно `final` и делегирует в `==`, так что `.equals()` — это то же сравнение ссылок, только длиннее и без проверки типа.
> 
> **EN:** Each constant is a singleton — one instance per JVM — so `==` compares exactly the right thing and is null-safe. On top of that, `==` catches a mismatch between enum types at compile time, whereas `.equals()` would silently return `false`. `Enum.equals` is `final` and delegates to `==` anyway, so `.equals()` is the same reference check, just longer and without the type check.
> 
> _Follow-up:_ «а `hashCode` у enum какой?» — идентичностный, `Enum.hashCode` тоже `final`. _Trap:_ «можно ли переопределить `equals` для enum?» — нет, он `final`.

> [!question]- Можно ли создать экземпляр enum через `new` или рефлексию?  
> **RU:** Нет. Конструктор enum неявно `private`, и `new` для enum-типа запрещён компилятором (JLS §8.9). Рефлексия тоже не спасает: `Constructor.newInstance` на enum-конструкторе бросает `IllegalArgumentException: Cannot reflectively create enum objects`. Набор констант закрыт — это и есть гарантия конечности.
> 
> **EN:** No. An enum's constructor is implicitly `private`, and `new` on an enum type is forbidden by the compiler (JLS §8.9). Reflection doesn't help either: `Constructor.newInstance` on an enum constructor throws `IllegalArgumentException: Cannot reflectively create enum objects`. The set of constants is sealed — that's the guarantee of finiteness.

> [!question]- Что не так с хранением `ordinal()` в базе? _(частый)_  
> **RU:** `ordinal()` — это позиция в порядке объявления, а порядок — вещь, которую кто угодно поменяет при рефакторинге (вставит константу в середину, отсортирует по алфавиту). Тогда все ранее сохранённые числа начинают указывать на другие константы — тихо, без ошибки. Персистить надо `name()` или собственное стабильное поле-код, а `ordinal()` оставить для внутренних оптимизаций вроде `EnumSet`/`EnumMap`.
> 
> **EN:** `ordinal()` is the position in declaration order, and that order is exactly what someone changes during a refactor — inserting a constant in the middle, sorting alphabetically. Then every previously stored number points at a different constant — silently, no error. Persist `name()` or your own stable code field, and leave `ordinal()` for internal optimizations like `EnumSet`/`EnumMap`.
> 
> _Слабый ответ:_ «`ordinal` быстрее, чем строка» — экономия байтов не стоит тихой порчи данных.

> [!question]- Что напечатает классический `switch` по `null`-константе enum? _(senior-любимый)_  
> **RU:** Классический `switch (status)` при `status == null` бросает `NullPointerException` ещё до входа в ветки — под капотом дёргается `status.ordinal()` на `null`. Проверено на JDK 21: и классический `switch`-statement, и `switch`-expression без `case null` кидают NPE. Лечится это в Java 21 (JEP 441): в `switch`-expression можно написать `case null ->` и обработать его явной веткой — тогда NPE не будет.
> 
> java
> 
> ```java
> return switch (status) {
>     case null -> "нет статуса";   // Java 21, JEP 441
>     case NEW  -> "новый";
>     case PAID -> "оплачен";
> };
> ```
> 
> **EN:** A classic `switch (status)` with `status == null` throws `NullPointerException` before any branch — under the hood it calls `status.ordinal()` on `null`. Verified on JDK 21: both the classic statement and an expression without `case null` throw NPE. Java 21 (JEP 441) fixes it: in a `switch` expression you can write `case null ->` and handle it in its own branch — no NPE then.
> 
> _Follow-up:_ «а квалифицированное имя в `case` — `case OrderStatus.NEW` — скомпилируется?» — до Java 21 нет (ошибка компиляции), с Java 21 да (JEP 441), хотя для enum-селектора идиоматично простое имя.

Дальше — enum как **носитель поведения**: разные действия на каждую константу без единого `switch`, механика анонимных подклассов, лямбда-в-поле и расширяемые enum'ы через интерфейс (Item 38). Скажи «дальше».
