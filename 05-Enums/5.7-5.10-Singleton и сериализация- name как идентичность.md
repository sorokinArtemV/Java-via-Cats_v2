В .NET синглтон ты писал приватным конструктором плюс `static readonly`-полем или `Lazy<T>` — и у этого приёма две дыры, которые приходилось латать руками: рефлексия (`Activator.CreateInstance(..., nonPublic: true)`) обходит приватный конструктор и делает второй объект, а десериализация создаёт новый объект мимо конструктора. В Java можно ровно так же — и так же дырявo. А можно одной строкой: enum с единственной константой. Этот урок про то, _почему_ эта строка непробиваема, и как устроена сериализация enum, которая её и делает непробиваемой. Это пара к прошлому уроку: там `ordinal` работал адресом коллекций, здесь `name` работает идентичностью — при сериализации и в синглтоне.

**Mental model.** Enum-константа — это `public static final` синглтон, созданный JVM ровно один раз. Сериализуется он не состоянием, а именем: в байты уезжает строка `name()`, обратно приходит _та же_ каноническая константа через `valueOf`. Отсюда и неломаемый синглтон, и потеря полей при сериализации.

#### Singleton через enum — поверхность

```java
public enum PaymentGateway {
    INSTANCE;                       // одна константа = ровно один экземпляр

    private final String url = loadUrl();   // полноценные поля объекта
    public void charge(long amountMinor, String currency) { /* ... */ }
}

// использование — без getInstance(), просто константа:
PaymentGateway.INSTANCE.charge(1000, "EUR");
```

Это не «просто маркер»: у enum-синглтона есть поля, состояние, методы, он может реализовывать интерфейсы (`enum X implements Service { INSTANCE; ... }` — часто и нужно для DI/моков). Полноценный объект, которого гарантированно один. Effective Java, Item 3 прямо называет это лучшим способом реализовать синглтон в Java.

#### Три гарантии — все от JVM, не от твоего кода

У классического синглтона единственность ломается тремя способами. Enum закрывает каждый силами самой платформы.

**1. Единственность + потокобезопасность.** Константа — это `public static final` поле, инициализируемое в статическом инициализаторе класса (`<clinit>`). А инициализация класса в JVM потокобезопасна по спецификации и происходит ровно один раз, под внутренней блокировкой, с happens-before на завершение (JLS §12.4.2 — payoff обещания из 5.1). Значит `INSTANCE` создаётся однократно, лениво (при первом обращении к классу), без единого `synchronized`, `volatile` или double-checked locking. То, ради чего в классике городят holder-идиому, здесь идёт из коробки.

**2. Защита от рефлексии — в языке.** В классическом синглтоне приватный конструктор не спасает: `setAccessible(true)` + `newInstance()` создадут второй объект. У enum это невозможно на уровне самой рефлексии — проверено на JDK 21, я честно достал синтетический конструктор, сделал его доступным и позвал:

```java
Constructor<?> k = PaymentGateway.class.getDeclaredConstructors()[0];
k.setAccessible(true);
k.newInstance("HACK", 99);
// -> IllegalArgumentException: Cannot reflectively create enum objects
```

`Constructor.newInstance` специально проверяет флаг `ENUM` у класса и бросает _до_ создания. Лазейки нет.

**3. Защита от сериализации — тоже бесплатно.** Тут классика течёт сильнее всего. Проверено round-trip на JDK 21:

```java
// классический синглтон БЕЗ readResolve:
NoResolve back = roundTrip(NoResolve.INSTANCE);
back == NoResolve.INSTANCE;   // false — КАЖДАЯ десериализация даёт новый объект

// enum:
PaymentGateway back = roundTrip(PaymentGateway.INSTANCE);
back == PaymentGateway.INSTANCE;   // true — та же константа
```

Обычный `Serializable`-синглтон при каждой десериализации создаёт новый объект — чтобы вернуть тот же, нужен boilerplate `readResolve()` (я проверил: с ним `back == INSTANCE` снова `true`). Enum этого метода не требует — _почему_, разберём в следующей секции через by-name-механику. Вот эти два способа рядом:

```java
// Классика — и всё, что приходится защищать руками:
public final class ClassicGateway implements Serializable {
    public static final ClassicGateway INSTANCE = new ClassicGateway();
    private ClassicGateway() {
        if (INSTANCE != null) throw new IllegalStateException("уже создан"); // от рефлексии — хрупко
    }
    private Object readResolve() { return INSTANCE; }                        // от десериализации
}

// Enum — то же самое по гарантиям, и строго сильнее:
public enum NewGateway { INSTANCE; }
```

Причём проверка `if (INSTANCE != null)` рефлексию на самом деле до конца не закрывает, а enum закрывает.

> [!tip] 💡 Из C#  
> Твой ручной синглтон (`static readonly` / `Lazy<T>`) даёт единственность в счастливом случае — потокобезопасную ленивую инициализацию через type initializer .NET (аналог гарантии №1 в Java). Но гарантии №2 и №3 в C# на тебе: рефлексия (`Activator.CreateInstance(..., nonPublic: true)`) обходит приватный конструктор, а при сериализации identity надо стеречь самому. То, что в Java enum закрывает на уровне рантайма и языка, в C# остаётся ручной заботой.

⚠️ **Ограничения синглтона.** За три гарантии платишь двумя: enum **не может `extends` класс** (слот занят `java.lang.Enum` — реф на 5.2), и нет тонкого контроля над ленивостью (инстанс строится при инициализации класса, а не «когда я скажу»). Интерфейсы реализовывать можно сколько угодно. Для подавляющего большинства синглтонов оба ограничения неважны; если синглтон _обязан_ наследовать базовый класс или иметь хитрую отложенную инициализацию — тогда классика с приватным конструктором и `readResolve`, но это редкий случай.

#### Сериализация enum — механика by-name

Гарантия №3 держится на том, как enum сериализуется, — и это самостоятельная тема (5.10). Ключевой факт из спецификации (Java Object Serialization Specification, §1.12): при сериализации константы в поток пишется **только строка `name()`** плюс идентификатор enum-класса — «the serialized form of an enum constant consists solely of its name; field values are not transmitted». При чтении `ObjectInputStream` берёт имя и зовёт `Enum.valueOf(EnumClass, name)` — то есть возвращает уже существующую каноническую константу, а не строит новый объект.

Из механики by-name вытекают четыре следствия — два приятных, два опасных.

**Следствие 1 (приятное): `==` переживает round-trip.** Раз `valueOf` возвращает ту же константу, новому объекту негде взяться — `deserialized == Status.ACTIVE` это `true`. Это и замыкает гарантию №3 синглтона: enum-синглтон serialization-safe бесплатно, без `readResolve`.

**Следствие 2 (опасное): поля константы не сериализуются.** В поток идёт только имя. Проверено на JDK 21 наглядно: я сериализовал константу со счётчиком `calls = 2`, потом инкрементнул живую константу до `3`, десериализовал старые байты — и у восстановленной константы `calls == 3`, не `2`. Значение взялось из _живой_ константы текущей JVM, а не из байтов.

```java
enum Counter { GLOBAL; int calls; void inc() { calls++; } }

Counter.GLOBAL.inc(); Counter.GLOBAL.inc();   // calls = 2
byte[] bytes = serialize(Counter.GLOBAL);      // в байтах только "GLOBAL", двойки там нет
Counter.GLOBAL.inc();                          // calls = 3
Counter back = (Counter) deserialize(bytes);
back.calls;   // 3 — из живой константы, поток поле не нёс
```

Мораль: не держи в enum мутабельное состояние, которое должно пережить сохранение/передачу. Enum для этого не предназначен (и как синглтон он общий для всех потоков — мутабельное поле ещё и станет гонкой, главы 23/24).

**Следствие 3: кастомизировать сериализацию enum нельзя.** Спека (§1.12) жёстко фиксирует: `writeObject`, `readObject`, `readObjectNoData`, `writeReplace`, `readResolve`, объявленные в enum, **игнорируются**; `serialPersistentFields` и `serialVersionUID` тоже игнорируются — у всех enum фиксированный `serialVersionUID = 0L`. Проверено: я навесил на enum свои `writeObject` и `readResolve` с логированием — они **не вызвались**, а round-trip всё равно дал `==`. Механизм by-name зашит намертво.

**Следствие 4 (опасное): имя константы становится контрактом.** Всё держится на строке-имени, поэтому — и это я прогнал раздельной компиляцией (v1 пишет, v2/v3 читают):

- **Переименовал константу → старые данные не читаются.** Сериализовал `ACTIVE`, переименовал в другой версии `ACTIVE → OPEN`, читаю старые байты → `InvalidObjectException: enum constant ACTIVE does not exist in class Status`.
- **Переставил константы местами → безопасно.** Сериализовал `ACTIVE` (был `ordinal 0`), в другой версии переставил так, что `ACTIVE` стал `ordinal 1`, читаю старые байты → корректно вернулся `ACTIVE`. Сериализация смотрит на имя, не на позицию.
- **Добавил новую константу → безопасно** для старых данных (их имена по-прежнему резолвятся).

#### Почему «не персисти `ordinal`» — теперь обосновано

Помнишь правило из 5.1? Вот его доказательство через by-name. Тот прогон с перестановкой — идеальная иллюстрация: `ACTIVE` уехал с `ordinal 0`, а прочитался, когда стал `ordinal 1`, — и всё равно правильно, потому что резолв шёл по имени. А если бы ты руками сохранил `ordinal 0`, после перестановки он бы прочитался как `CLOSED` — тихо, без ошибки.

|Хранишь|Перестановка/вставка константы|Переименование|
|---|---|---|
|`name()` (как делает сериализация)|**безопасно** — имя не изменилось|ломается **заметно** — `valueOf` бросит `InvalidObjectException`|
|`ordinal()`|**ломается тихо** — номер сдвинулся, данные поехали без ошибки|безопасно — номер тот же|

Ключевое слово — _тихо_. Вставка/перестановка констант случаются на порядок чаще переименования (добавить статус в середину enum — обычное дело) и не дают ошибки компиляции: код собирается, а данные уже указывают не туда. Переименование хотя бы падает громко. Поэтому штатная сериализация использует имя, и в БД/на провод храни `name()` или собственный стабильный код-поле — никогда `ordinal()`.

На практике нативный `Serializable` сегодня почти не используют (почему — глава 19, там про RCE через десериализацию и «избегайте `Serializable`»); enum'ы ездят через JSON. Хорошая новость: Jackson по умолчанию тоже сериализует enum **по `name()`**, так что правило «храни имя, не `ordinal`» переносится один в один. Но само Java-имя — плохой внешний контракт: захочешь отрефакторить константу — сломаешь API. Идиоматично **развязать имя и стабильный код** через отдельное поле (детали аннотаций Jackson — глава 19):

```java
public enum OrderStatus {
    ACTIVE("A"), CANCELLED("C"), SHIPPED("S");   // код отдельно от Java-имени
    private final String code;
    OrderStatus(String code) { this.code = code; }
    public String code() { return code; }
    public static OrderStatus fromCode(String c) {
        for (var s : values()) if (s.code.equals(c)) return s;
        throw new IllegalArgumentException("неизвестный код: " + c);
    }
}
```

> [!tip] 💡 Из C#  
> Дефолты **противоположные**, и это классический источник багов при интеропе. C#-enum — число, и `System.Text.Json`/Newtonsoft по умолчанию сериализуют его **числовым значением**. То есть Java по умолчанию по **имени** (ломается на переименовании, устойчива к перестановке), а C# по умолчанию по **числу** (ломается на перестановке/вставке, устойчива к переименованию) — зеркальные хрупкости. Когда Java- и C#-сервис обмениваются данными, формат надо фиксировать явно с обеих сторон: `[JsonConverter(typeof(JsonStringEnumConverter))]` в C#, стабильный код-поле в Java.

> [!example] 🔬 Глубже / на синьора — смысловой замок всей главы 5  
> Три идиомы enum, которые мы разобрали за четыре урока — **быстрые коллекции** (`EnumSet`/`EnumMap`), **надёжный синглтон**, **безопасная сериализация** — это не три отдельные фичи, а три следствия **одного** факта: enum это закрытый конечный набор синглтонов, у каждого под капотом `ordinal` (адрес) и `name` (идентичность). `ordinal` даёт скорость коллекциям (Урок A). Гарантия единственного инстанса от `<clinit>` даёт синглтон. By-name-сериализация связывает identity через провод и делает синглтон неломаемым. Поймёшь эту одну фразу — выведешь любой ответ по теме, а не будешь вспоминать список.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Мутабельное состояние в enum-синглтоне.** Оно (а) не переживёт сериализацию (едет только имя), (б) станет гонкой при конкурентном доступе — синглтон общий для всех потоков. Enum держи иммутабельным: все поля `final`.
> 2. **`ordinal()` в БД/на проводе** — мина из-за тихой поломки при перестановке. Храни `name()` или явный код-поле.
> 3. **Попытка кастомизировать сериализацию enum** через `readResolve`/`writeObject` — молча игнорируется (§1.12). Не полагайся на эти хуки в enum, они мертвы.
> 4. **Классический синглтон без `readResolve`** — при каждой десериализации новый объект, единственность сломана. Если уж классика — не забудь `readResolve` И защиту в конструкторе от рефлексии.
> 5. **enum-синглтон, обязанный `extends` класс** — невозможно (слот занят `Enum`). Тогда только holder-идиома; но `implements` интерфейсов доступен, для DI/моков этого обычно хватает.

#### На собесе

> [!question]- Лучший способ реализовать синглтон в Java — и почему?  
> **RU:** Single-element enum — `enum X { INSTANCE; }` (Effective Java, Item 3). Три гарантии от языка бесплатно: единственный инстанс через инициализацию класса (потокобезопасно и лениво по JLS §12.4.2, без double-checked locking); защита от рефлексии (`Constructor.newInstance` бросает `IllegalArgumentException` на enum); serialization-safe (сериализация by-name возвращает каноническую константу, без `readResolve`). Цена — нельзя наследовать класс и нет ручной ленивой инициализации; для синглтона обычно неважно.
> 
> **EN:** A single-element enum — `enum X { INSTANCE; }` (Effective Java, Item 3). Three guarantees from the language for free: a single instance via class initialization (thread-safe and lazy per JLS §12.4.2, no double-checked locking); reflection protection (`Constructor.newInstance` throws `IllegalArgumentException` on an enum); serialization safety (by-name serialization returns the canonical constant, no `readResolve`). The cost — you can't extend a class and there's no manual lazy init; usually irrelevant for a singleton.
> 
> _Follow-up:_ «почему именно enum, а не приватный конструктор?» → приватный конструктор не закрывает ни рефлексию, ни сериализацию без ручного boilerplate. _Trap:_ «а ленивость?» → есть, но не управляемая: инстанс строится при первом обращении к классу.

> [!question]- Можно ли сломать enum-синглтон рефлексией или сериализацией?  
> **RU:** Нет. Рефлексия: `Constructor.newInstance` проверяет флаг `ENUM` и бросает `IllegalArgumentException: Cannot reflectively create enum objects` до создания. Сериализация: идёт через `Enum.valueOf` к канонической константе, новый объект не создаётся, `==` выживает. Классический синглтон сломать можно и тем, и другим — рефлексией через `setAccessible(true)`, сериализацией (каждый round-trip = новый объект без `readResolve`). Это и есть аргумент в пользу enum.
> 
> **EN:** No. Reflection: `Constructor.newInstance` checks the `ENUM` flag and throws `IllegalArgumentException: Cannot reflectively create enum objects` before creating anything. Serialization: it goes through `Enum.valueOf` to the canonical constant, no new object is made, `==` survives. A classic singleton can be broken by both — reflection via `setAccessible(true)`, and serialization (every round-trip is a new object without `readResolve`). That's the argument for enum.

> [!question]- Как сериализуется enum — by name или by ordinal? _(частый)_  
> **RU:** By name. В поток пишется `name()` константы; на чтении — `Enum.valueOf(type, name)` возвращает каноническую константу. Поэтому `==` переживает round-trip и enum-синглтон serialization-safe. Что безопасно: перестановка/вставка/добавление констант — имя не меняется. Что ломает: переименование — `valueOf` не найдёт имя, `InvalidObjectException`. В БД храни `name()` или свой код, никогда `ordinal()` — иначе рефакторинг тихо разъедет данные.
> 
> **EN:** By name. The constant's `name()` is written to the stream; on read, `Enum.valueOf(type, name)` returns the canonical constant. That's why `==` survives a round-trip and an enum singleton is serialization-safe. Safe: reordering/inserting/adding constants — the name doesn't change. Breaks: renaming — `valueOf` won't find the name, `InvalidObjectException`. In a DB store `name()` or your own code, never `ordinal()` — otherwise a refactor silently corrupts data.
> 
> _Follow-up:_ «можно ли переопределить эту сериализацию?» → нет, `writeObject`/`readResolve` и т.п. в enum игнорируются (§1.12), `serialVersionUID` фиксирован 0L.

> [!question]- Сохранятся ли кастомные поля enum при сериализации?  
> **RU:** Нет. Едет только имя; поля восстановленной константы берутся из её конструктора в текущем коде, не из байтов. Практическое следствие: поменял значение поля в коде и десериализовал старые данные — получишь новое значение поля. Поэтому мутабельное состояние в enum — плохо: оно не переживёт сериализацию и станет гонкой (общий синглтон + конкурентный доступ). Enum держи иммутабельным.
> 
> **EN:** No. Only the name travels; the restored constant's fields come from its constructor in the current code, not from the bytes. Practical consequence: change a field's value in code and deserialize old data — you get the new value. That's why mutable state in an enum is bad: it won't survive serialization and becomes a race (shared singleton + concurrent access). Keep enums immutable.
