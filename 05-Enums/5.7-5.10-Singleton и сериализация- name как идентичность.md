В C# синглтон ты писал так: приватный конструктор + `static readonly` поле или `Lazy<T>`. У этого две дыры, которые ты закрывал руками. Рефлексия может позвать твой приватный конструктор (`Activator.CreateInstance(..., nonPublic: true)`) и сделать второй инстанс. И сериализация при десериализации создаёт **новый** объект мимо конструктора. В Java тоже можно так — и так же дыряво. А можно одной строкой: `enum X { INSTANCE; }`.

Этот урок — вторая половина той же истории, что `EnumSet`/`EnumMap`. Там `ordinal` (адрес) давал скорость коллекциям. Здесь работает второй атрибут константы — `name` (идентичность): он делает синглтон неломаемым и сериализацию безопасной. Оба урока — следствия одного факта: **enum это закрытый конечный набор синглтонов, у каждого под капотом `ordinal` и `name`.**

**Mental model.** Enum-синглтон отдаёт единственность + потокобезопасность + защиту от рефлексии + защиту от сериализации одной строкой, потому что все четыре гарантии встроены в то, как JVM обращается с enum'ами. Цена — нельзя наследовать класс.

### 1. Синглтон — поверхность

```java
public enum DatabaseConfig {
    INSTANCE;                       // одна константа = ровно один экземпляр

    private final String url;       // обычные поля экземпляра — это полноценный объект
    DatabaseConfig() { this.url = loadUrl(); }   // конструктор enum всегда приватный

    public String url() { return url; }
    public synchronized void open() { /* ... */ }
}

DatabaseConfig.INSTANCE.open();     // без getInstance(), просто константа
```

Это не «просто маркер»: у enum-синглтона есть поля, состояние, методы, он может реализовывать интерфейсы (`enum X implements Service { INSTANCE; ... }` — часто и нужно для DI/моков). Полноценный объект, которого гарантированно один.

### 2. Три гарантии

Три способа сломать «единственность» синглтона — enum закрывает каждый силами самой JVM:

![[Pasted image 20260708164148.png]]

Разберу каждый барьер:

**1. Единственность + потокобезопасность — гарантирует JVM.** Константа enum — это `public static final` поле, инициализируемое в статическом инициализаторе класса. Инициализация класса в JVM происходит **ровно один раз**, под внутренней блокировкой, с happens-before на завершение (JLS §12.4.2). Значит `INSTANCE` создаётся однократно, лениво (при первом обращении к классу), и не нужно ни `synchronized`, ни `volatile`, ни double-checked locking. То, ради чего в классике городят holder-идиому, тут из коробки.

**2. Защита от рефлексии — в языке.** Прогнал честно: достал синтетический конструктор enum (сигнатура `(String name, int ordinal)`), сделал `setAccessible(true)`, позвал `newInstance("FAKE", 99)` — и получил:

```
java.lang.IllegalArgumentException: Cannot reflectively create enum objects
```

`Constructor.newInstance` **специально проверяет** флаг `ENUM` у класса и бросает исключение прежде, чем что-либо создать. Лазейки нет на уровне самой рефлексии. Для контраста в том же прогоне обычный класс с приватным конструктором через `setAccessible(true)` + `newInstance()` спокойно дал второй инстанс — `p1 == p2` вернуло `false`. Синглтон сломан.

**3. Защита от сериализации — тоже бесплатно.** Enum сериализуется по `name()`, при десериализации JVM возвращает каноническую существующую константу, а не строит новый объект. Round-trip сохраняет `==`. Это раскрою в части про сериализацию — там же покажу прогоном.

> [!tip] 💡 Из C#  
> Прямого аналога нет. В C# enum — именованные целочисленные константы (value type), без полей, методов и без гарантии единственного экземпляра; «enum-синглтон» там не существует как приём. Ближайшее — `sealed class` + `static readonly`:
> 
> csharp
> 
> ```csharp
> public sealed class DatabaseConfig {
>     public static DatabaseConfig Instance { get; } = new();   // type initializer — потокобезопасен
>     private DatabaseConfig() { }
> }
> ```
> 
> Статический инициализатор типа даёт потокобезопасную однократную инициализацию (аналог п.1). Но п.2 и п.3 в C# на тебе: рефлексия (`Activator.CreateInstance(..., nonPublic: true)`) приватный конструктор обходит, и при сериализации identity стереги сам. То, что Java enum закрывает на уровне рантайма, в C# остаётся ручной заботой.

> [!warning] ⚠️ Ограничения  
> За три гарантии платишь двумя ограничениями. **Первое:** enum не может наследовать класс — он уже наследник `java.lang.Enum` (множественного наследования классов в Java нет). Интерфейсы реализовывать может сколько угодно, так что для DI это обычно не помеха. **Второе:** нет тонкого контроля над ленивой инициализацией — инстанс строится при инициализации класса, а не «когда я скажу». Для подавляющего большинства синглтонов оба ограничения не проблема. Если синглтон _обязан_ extends базовый класс или иметь сложную отложенную инициализацию — тогда классический подход с приватным конструктором и `readResolve`, но это редкий случай.
> 
> Effective Java, Item 3 прямо: **single-element enum — часто лучший способ сделать синглтон.**

### 3. Сериализация enum — механика

Сначала минимум базы (детали `Serializable`, security и «почему индустрия отказывается» — глава 19). Объект живёт в памяти; чтобы **сохранить** его (файл, колонка БД, сообщение) или **передать** на другой сервер, его надо превратить в байты. Это сериализация: `объект → байты`; обратно — десериализация. Вопрос урока один: **когда сериализуешь константу enum — что уезжает в байты и что приедет обратно?**

Обычный объект (`User` с полями `name`, `age`) сериализуется как **копия своих данных** — записываются значения полей, при чтении создаётся **новый** объект и заполняется ими. Enum устроен иначе: это фиксированный закрытый набор именованных объектов, которые **уже существуют** в каждой JVM. Чтобы «передать» `RED`, не нужно описывать содержимое — достаточно ярлыка `"RED"`; принимающая сторона находит свой `RED` по имени.

![[Pasted image 20260708164203.png]]

Технически: `ObjectOutputStream` пишет в поток `name()` константы плюс идентификатор enum-класса; `ObjectInputStream` берёт имя и зовёт `Enum.valueOf(EnumClass, name)` → существующая каноническая константа. Из этого — четыре следствия. Прогнал их разом (JDK 21): `original == restored` → `true`, в байтах есть строка `"EUR"`, но нет значения поля `"euro_label"`, а у восстановленной константы поле на месте.

**Следствие 1 (приятное): `==` переживает round-trip.** Раз `valueOf` возвращает ту же константу, новый объект негде взяться. Это и **замыкает гарантию №3 синглтона**: enum-синглтон serialization-safe бесплатно. Контраст с классикой:

```java
public final class ClassicGateway implements Serializable {
    public static final ClassicGateway INSTANCE = new ClassicGateway();
    private ClassicGateway() {}
    private Object readResolve() { return INSTANCE; }   // без этого каждая десериализация = НОВЫЙ объект
}
```

**Следствие 2 (опасное): поля экземпляра НЕ сериализуются.** В поток идёт только имя. Прогон:

```java
enum Counter {
    GLOBAL;
    private int value;
    void inc() { value++; }
    int value() { return value; }
}

Counter.GLOBAL.inc(); Counter.GLOBAL.inc();   // value = 2
byte[] bytes = serialize(Counter.GLOBAL);      // в байтах только "GLOBAL", двойки там нет
Counter.GLOBAL.inc();                          // value = 3
Counter back = (Counter) deserialize(bytes);
back == Counter.GLOBAL;   // true
back.value();             // 3 — взято из живой константы, поток поле не нёс
```

Мораль: **не держи в enum мутабельное состояние, которое должно переживать сериализацию.** Enum для этого не предназначен.

**Следствие 3: кастомизировать сериализацию enum нельзя.** `writeObject`, `readObject`, `readResolve`, `writeReplace`, `serialPersistentFields`, объявленные в enum, игнорируются или запрещены; `serialVersionUID` фиксирован и игнорируется. Спецификация жёстко требует «только имя».

**Следствие 4 (опасное): имя константы становится контрактом.** Всё держится на строке-имени:

- переименовал константу → старые данные не читаются (`valueOf` бросит `IllegalArgumentException` — проверил на несуществующем имени);
- переставил константы местами → всё ок (важно имя, а не позиция);
- добавил новую константу → старые данные читаются.

### 4. `name` vs `ordinal` для персиста

Здесь стыкуется правило из 5.1–5.3 (Item 35): **никогда не сохраняй `ordinal()` в БД или на провод.**

|Хранишь|Перестановка/вставка константы|Переименование константы|
|---|---|---|
|`name()` (как делает сериализация)|**безопасно** — имя не изменилось|ломается — `valueOf` не найдёт|
|`ordinal()`|**ломается тихо** — номер сдвинулся|безопасно — номер тот же|

Ключевое слово — **тихо**. Перестановка/вставка случаются на порядок чаще переименования (добавил статус в середину enum — обычное дело) и **не дают ошибки компиляции**: код собирается, а данные в БД уже указывают не на ту константу. Переименование хотя бы заметно — `valueOf` кинет исключение, ты сразу увидишь. Поэтому `name()` (или собственный стабильный код-поле) — правильный выбор, а `ordinal()` — мина.

### 5. Практика — JSON, а не нативный `Serializable`

Нативный `Serializable` сегодня почти никто не использует — enum'ы ездят через JSON (Jackson/Gson). По умолчанию там та же логика: по `name()`. Но Java-имя — плохой внешний контракт (захочешь отрефакторить — сломаешь API). Идиома — развязать Java-имя и wire-код:

```java
public enum OrderStatus {
    ACTIVE("A"), CANCELLED("C"), SHIPPED("S");   // код отдельно от Java-имени
    private final String code;
    OrderStatus(String code) { this.code = code; }

    @JsonValue                                    // в JSON уедет "A", не "ACTIVE"
    public String code() { return code; }

    @JsonCreator
    public static OrderStatus fromCode(String code) {
        for (var s : values()) if (s.code.equals(code)) return s;
        throw new IllegalArgumentException("неизвестный код: " + code);
    }
}
// теперь Java можно рефакторить свободно, внешний контракт стабилен
```

> [!tip] 💡 Из C#  
> Дефолты **противоположны** — и это классический источник багов при интеропе. C#-enum — число под капотом, и `System.Text.Json`/Newtonsoft по умолчанию сериализуют его **числовым значением**. Итог: Java по умолчанию по **имени** (ломается на переименовании, устойчива к перестановке), C# по умолчанию по **числу** (ломается на перестановке, устойчив к переименованию).
> 
> csharp
> 
> ```csharp
> public enum OrderStatus { Active, Cancelled, Shipped }        // дефолт: в JSON уедет 0,1,2
> public enum OrderStatus { Active = 1, Cancelled = 2, Shipped = 3 }  // фикс: явные числа
> [JsonConverter(typeof(JsonStringEnumConverter))]              // или строкой, как Java
> public enum OrderStatus { Active, Cancelled, Shipped }
> ```
> 
> Когда Java- и C#-сервис обмениваются данными — фиксируй формат явно с обеих сторон.

> [!tip] 🐱 На котиках  
> Ты не отправляешь кота почтой. Ты отправляешь бирку с его именем — `"МУРКА"`. У получателя дома уже есть эта самая Мурка, он просто берёт её по имени (`valueOf`). Поэтому «пересланный» кот — это тот же самый кот (`==` выживает), а не клон. Но: текущее настроение Мурки (мутабельное поле) на бирке не написано — приедет ярлык, а настроение возьмётся то, что у Мурки прямо сейчас. И если ты переименуешь кошку `МУРКА → МУРЁНА`, старые бирки перестанут находить кота — а вот если просто переставишь миски местами, бирки по-прежнему работают.

### 6. Смысловой замок

Три «фокуса» enum — быстрые коллекции (`EnumSet`/`EnumMap`), надёжный синглтон, безопасная сериализация — это не три отдельные фичи, а три следствия одного факта: **enum это закрытый конечный набор синглтонов, у каждого под капотом `ordinal` (адрес в битовом векторе и в массиве) и `name` (сериализационная идентичность).** `ordinal` даёт скорость коллекциям, гарантия единственного инстанса даёт синглтон, by-name-сериализация связывает их всех. Держишь эту фразу — выводишь любой ответ по теме.

### На собесе

> [!question]- Лучший способ синглтона в Java?  
> **RU:** Single-element enum — `enum X { INSTANCE; }`, Effective Java Item 3. Три гарантии от языка бесплатно: единственный инстанс через инициализацию класса (потокобезопасно, лениво, без DCL/`volatile`); защита от рефлексии (JVM запрещает создавать enum рефлексией); serialization-safe (by-name, без `readResolve`). Цена — нельзя наследовать класс (enum уже наследник `Enum`) и нет ручного контроля ленивой инициализации; для большинства синглтонов неважно.
> 
> **EN:** A single-element enum — `enum X { INSTANCE; }`, Effective Java Item 3. Three language-level guarantees for free: a single instance via class initialization (thread-safe, lazy, no DCL or `volatile`); reflection safety (the JVM forbids creating enums reflectively); serialization safety (by-name, no `readResolve`). The cost — you can't extend a class (an enum already extends `Enum`) and there's no manual control over lazy init; for most singletons that's irrelevant.
> 
> _Follow-up:_ «Почему именно enum, а не приватный конструктор?» → приватный конструктор ломается рефлексией и сериализацией; enum закрывает обе дыры в языке.  
> _Follow-up:_ «Ограничения?» → нет наследования класса, нет ленивой инициализации руками.

> [!question]- Можно ли сломать enum-синглтон рефлексией или сериализацией?  
> **RU:** Нет. Рефлексия: `Constructor.newInstance` проверяет флаг `ENUM` и бросает `IllegalArgumentException: Cannot reflectively create enum objects`. Сериализация: идёт через `Enum.valueOf` к канонической константе, новый объект не создаётся, `==` выживает. Классический синглтон — да, ломается: рефлексией через `setAccessible(true)` + приватный конструктор, и сериализацией (каждый round-trip = новый объект без `readResolve`).
> 
> **EN:** No. Reflection: `Constructor.newInstance` checks the `ENUM` flag and throws `IllegalArgumentException: Cannot reflectively create enum objects`. Serialization: it goes through `Enum.valueOf` to the canonical constant, no new object is created, `==` survives. A classic singleton can be broken — by reflection via `setAccessible(true)` on the private constructor, and by serialization (each round-trip is a new object without `readResolve`).

> [!question]- Как сериализуется enum — by ordinal или by name?  
> **RU:** By name. В поток пишется `name()`, на чтении — `valueOf(type, name)` → каноническая константа. Перестановка/вставка констант безопасна (имя не меняется), переименование ломает. В БД храни `name()` или выделенный стабильный код, **никогда `ordinal()`** — иначе рефакторинг enum тихо разъедет исторические данные без единого исключения.
> 
> **EN:** By name. `name()` is written to the stream; on read `valueOf(type, name)` returns the canonical constant. Reordering or inserting constants is safe (the name is unchanged), renaming breaks it. In a database store `name()` or a dedicated stable code, **never `ordinal()`** — otherwise refactoring the enum silently corrupts historical data with no exception at all.
> 
> _Follow-up:_ «Что безопасно, что ломает?» → перестановка безопасна, переименование ломает; `ordinal` наоборот, но тихо.

> [!question]- Сохранятся ли кастомные поля enum при сериализации?  
> **RU:** Нет. Едет только имя; поля восстановленной константы берутся из её конструктора в текущем коде, а не из байтов. Практический вывод: поменяешь значение поля в коде — десериализация старых данных даст новое значение. Поэтому мутабельное состояние в enum — плохо: оно не переживёт сериализацию и станет гонкой при конкурентном доступе (общий синглтон). Enum держи иммутабельным.
> 
> **EN:** No. Only the name travels; the restored constant's fields come from its constructor in the current code, not from the bytes. Practical consequence: change a field's value in code and deserializing old data yields the new value. So mutable state in an enum is bad — it won't survive serialization and becomes a race under concurrent access (a shared singleton). Keep enums immutable.

### Практика (меню, не контракт)

- **Заменить int-битмаску прав на `EnumSet`** (~30 мин). `int FLAGS = READ | WRITE` → `EnumSet<Permission>`. Прочувствуй контраст с `[Flags]`: что ушло (маски), что добавилось (type safety).
- **`EnumMap` vs `HashMap` для конфига** (~30 мин). Retry-политика на тип ошибки: сначала `HashMap`, потом `EnumMap`. Сравни порядок итерации.
- **Сериализация: перестановка vs ordinal** (~40 мин, самый ценный). Сериализуй enum, восстанови, проверь `==`. Переставь константы, десериализуй старые байты — безопасно. Потом версия с ручным `ordinal()` в «БД» — переставь и поймай, как данные разъехались.
- **Сломать enum рефлексией** (~15 мин). `getDeclaredConstructor(String.class, int.class)`, `setAccessible(true)`, `newInstance` — поймай `IllegalArgumentException`. То же с обычным классом — увидишь разницу.
