В C# ты набор прав моделировал через `[Flags] enum Permission { Read = 1, Write = 2, Delete = 4 }`, а потом руками крутил биты: `granted |= Write`, `(granted & Write) != 0`. Быстро и компактно, но всё вручную — сам расставляешь степени двойки, сам пишешь маски, и ничто не мешает воткнуть `(Permission)999`. В Java для «множества enum-значений» есть `EnumSet`, и первый сюрприз в том, что он ровно настолько же быстр, как твоя ручная маска, — потому что под капотом это она и есть, битовый вектор. Только type-safe и без единого `&` в прикладном коде.

`EnumSet` и `EnumMap` держатся на одном факте: у каждой константы enum есть `ordinal()` — её позиция в объявлении, стабильный номер от нуля. Этот номер работает **адресом**: у `EnumSet` — номером бита, у `EnumMap` — индексом в массиве. Один механизм, два применения; разбираем подряд, потому что порознь они не понимаются.

**Mental model.** Множество enum'ов — это одно число: бит на позиции `ordinal` = «константа присутствует». Map по enum-ключам — это массив: ячейка на позиции `ordinal` = «что лежит по ключу».

### 1. `EnumSet` — поверхность

Это обычный `Set<E>`, специализированный под enum-ключи. Весь API `Set` на месте плюс удобные фабрики:

```java
enum Permission { READ, WRITE, DELETE, ADMIN, AUDIT }

EnumSet<Permission> granted  = EnumSet.of(Permission.READ, Permission.WRITE);
EnumSet<Permission> none     = EnumSet.noneOf(Permission.class);   // пустой
EnumSet<Permission> all      = EnumSet.allOf(Permission.class);    // все константы
EnumSet<Permission> range    = EnumSet.range(Permission.READ, Permission.DELETE); // [READ..DELETE]
EnumSet<Permission> readonly = EnumSet.complementOf(
        EnumSet.of(Permission.WRITE, Permission.DELETE, Permission.ADMIN)); // дополнение

granted.add(Permission.AUDIT);
boolean canWrite = granted.contains(Permission.WRITE);
granted.addAll(all);        // union
granted.retainAll(range);   // intersection
```

Прогон (JDK 21, реально запускал):

```
granted  = [READ, WRITE]
none     = []
all      = [READ, WRITE, DELETE, ADMIN, AUDIT]
range    = [READ, WRITE, DELETE]
readonly = [READ, AUDIT]
class    = java.util.RegularEnumSet
```

Две вещи в выводе: порядок итерации — это **порядок объявления** (он же порядок `ordinal`), не порядок вставки и не хеш-порядок. И класс-реализация — `RegularEnumSet`, а не сам `EnumSet` (он абстрактный).

> [!tip] 💡 Из C#  
> `EnumSet` — это твой `[Flags]`-enum без ручных степеней двойки и без масок в прикладном коде. `of(...)` вместо `Read | Write`, `contains(...)` вместо `(x & Read) != 0`, `addAll` вместо `|=`, `retainAll` вместо `&`. Type safety настоящая: в `EnumSet<Permission>` нельзя положить значение из другого enum или «левое» число — компилятор не даст, а фабрики ещё и в рантайме проверяют класс элемента.

### 2. `EnumSet` — механика

`EnumSet.of(...)` и компания возвращают одну из двух package-private реализаций, выбор — по числу констант в enum:

- **`RegularEnumSet`** хранит **один `long elements`**. Бит на позиции `i` взведён ⟺ в множестве есть константа с `ordinal() == i`. Работает для enum с **≤ 64 константами**.
- **`JumboEnumSet`** хранит **`long[] elements`** для enum с **> 64 константами**. Слово — `ordinal >>> 6`, бит внутри слова — `ordinal & 63`.

Раз всё множество — это число, операции становятся чистой битовой арифметикой. Прогнал реально: `READ.ordinal()==0`, `WRITE.ordinal()==1`, `AUDIT.ordinal()==4`, и `{READ,WRITE}` внутри — это `long` со значением `3`, а `{READ,AUDIT}` — `17` (биты 0 и 4).

Вот как множество превращается в одно число:

Таблица операций (для `RegularEnumSet` — буквально одна машинная инструкция на операцию):

|Операция|Под капотом|
|---|---|
|`add(e)`|`elements \|= (1L << e.ordinal())`|
|`contains(e)`|`(elements & (1L << e.ordinal())) != 0`|
|`addAll(other)` (union)|`elements \|= other.elements`|
|`retainAll(other)` (intersection)|`elements &= other.elements`|
|`removeAll(other)`|`elements &= ~other.elements`|

`union` двух множеств на сотню элементов — это один `|` над двумя `long`. Ноль аллокаций, ноль хеширования, ноль обхода.

> [!example] 🔬 Глубже / на синьора  
> **`RegularEnumSet` vs `JumboEnumSet` — это просто «хватило ли 64 бит».** Прогнал: enum на 65 констант, `EnumSet.of(...)` молча вернул `java.util.JumboEnumSet`, `contains(C64)` работает. API тот же, операции те же — просто побитово по словам массива вместо одного `long`. Индекс слова `ordinal >>> 6` (деление на 64), бит внутри — `ordinal & 63` (остаток).  
> Источник на механику: Javadoc `EnumSet` прямым текстом — множество «represented internally as bit vectors», оттого «extremely compact and efficient». Деление на `Regular`/`Jumbo` видно в исходниках OpenJDK (обе реализации package-private в `java.util`). Effective Java, Item 36 — «use `EnumSet` instead of bit fields» — это ровно про замену C#-шного `[Flags]`.

> [!tip] 🐱 На котиках  
> Представь панель тумблеров на стене — один ряд. Каждое настроение кота (`СПИТ`, `ОХОТИТСЯ`, `ТРЕБУЕТ_ЕДЫ`, `ИГНОРИТ`) — это один тумблер: вкл или выкл. «Кот сейчас охотится?» — глянул на один конкретный тумблер, мгновенно (`contains` = один бит). «Какие вообще настроения активны?» — окинул взглядом всю панель, она целиком в одном `long`. А `HashSet` — это картотека в шкафу: чтобы проверить настроение, посчитай хеш, найди ящик, перебери карточки. Для конечного известного набора тумблеры бьют картотеку всухую.

### 3. Почему быстрее `HashSet` — конкретно

`HashSet` внутри — это `HashMap`, где элементы лежат ключами. На каждый `add`: вычисление `hashCode()`, поиск бакета по маске `hash & (n-1)`, **аллокация объекта-обёртки `Node`**, потенциальная обработка коллизий. Память — массив бакетов плюс узел на каждый элемент. `contains` — снова хеш, снова бакет.

`EnumSet` на тех же данных: один примитивный `long` на всё множество, **ноль аллокаций** per-element, операции — побитовые инструкции процессора. Выигрыш и по скорости, и по памяти — на порядок. Это тот редкий случай, когда специализированная коллекция не преждевременная оптимизация, а правильный дефолт: для enum-ключей `EnumSet` — первый выбор, не `HashSet`.

### 4. `EnumMap` — поверхность

`EnumMap` решает соседнюю боль: значение на каждый enum-ключ — конфиг на статус, обработчик на тип события, цена на тариф. Инстинкт с C#-фоном — `Dictionary<TEnum,V>`, в Java аналог `HashMap<Day,V>`. Работает, но ты опять платишь за хеш-машинерию там, где ключи — конечный известный набор с готовыми адресами.

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

EnumMap<Day, String> plan = new EnumMap<>(Day.class);  // Class enum'а ОБЯЗАТЕЛЕН
plan.put(Day.FRI, "релиз");
plan.put(Day.MON, "планёрка");
plan.put(Day.WED, "review");
plan.get(Day.WED);   // "review"
```

Прогон (JDK 21):

```
EnumMap iter : {MON=планёрка, WED=review, FRI=релиз}
class        : java.util.EnumMap
get(WED)     : review
HashMap iter : {FRI=релиз, WED=review, MON=планёрка}
```

Смотри внимательно: я клал ключи в порядке `FRI → MON → WED`, а `EnumMap` итерируется как `MON → WED → FRI` — **порядок объявления enum**, всегда, гарантированно. `HashMap` на тех же ключах выдал вообще третий порядок (`FRI, WED, MON`) — ни вставки, ни объявления, а как легли хеши; полагаться нельзя. Конструктору **обязательно** передаётся `Day.class` — `EnumMap` не узнаёт тип по ходу, ему нужно знать весь набор констант заранее, чтобы выделить массив под них. Эта деталь прямо следует из механики.

> [!tip] 💡 Из C#  
> Прямого аналога нет. В C# для enum-ключей ты возьмёшь `Dictionary<TEnum,V>` — и заплатишь за `GetHashCode`/`Equals` и бакеты, как с любым ключом. `EnumMap` — специализация, которой в BCL просто нет; там пришлось бы руками катать `V[]`, индексируясь по `(int)enumValue`, теряя весь сахар `Map`. `EnumMap` даёт и то, и другое.

### 5. `EnumMap` — механика

Внутри `EnumMap` — обычный **`Object[] vals`**, проиндексированный `ordinal` ключа, плюс ссылка на массив всех констант (`keyUniverse`, полученный из `Day.class`). `put(key,v)` → `vals[key.ordinal()] = maskNull(v)`, `get(key)` → `vals[key.ordinal()]`. Нет хеширования, нет бакетов, нет коллизий, нет `Node` на пару. Просто массив.

![[Pasted image 20260708164037.png]]

Вот **связь с `EnumSet`**, ради которой они идут подряд: обе коллекции держатся на `ordinal` как на прямом адресе. У `EnumSet` это номер бита, у `EnumMap` — индекс в массиве. И это **единственное легитимное использование `ordinal`** — внутри JDK, как адрес в специализированной структуре. Правило «не персисти `ordinal`, не строй на нём прикладную логику» (Item 35, всплывёт в уроке про сериализацию) — оно про _твой_ код. А здесь, в кишках `EnumSet`/`EnumMap`, `ordinal` ровно для этого и существует. JDK имеет право, ты — нет.

> [!tip] 🐱 На котиках  
> `EnumMap` — это ряд именных мисок вдоль стены, по одной на кота, порядок закреплён навсегда. Хочешь узнать, сколько корма у `МУРКА` — идёшь сразу к её миске по фиксированному месту, не обходя весь ряд и не спрашивая «а которая тут Муркина» (`get` = обращение по индексу, без хеша). `HashMap` заставил бы каждый раз вычислять, в какой шкаф положена миска. Пустая миска (`null`-значение) — это всё ещё Муркина миска, она просто пустая: `containsKey(МУРКА)` честно скажет «есть, да пустая», не спутав с «кота такого нет».

> [!example] 🔬 Глубже / на синьора  
> `get`/`put` у `EnumMap` — разыменование массива по индексу, `O(1)` без констант хеш-лукапа. По памяти: `EnumMap` — один массив фиксированной длины (∝ числу констант, не записей); `HashMap` — массив бакетов плюс `Node` на каждую пару. Коллизий не бывает в принципе — у каждой константы свой уникальный индекс.  
> Источник: Javadoc `EnumMap` — «implemented as arrays… extremely compact and efficient». Effective Java, Item 37 — «use `EnumMap` instead of ordinal indexing»: не делай руками `V[] plan = new V[7]; plan[day.ordinal()] = ...`, а возьми `EnumMap`, который инкапсулирует ровно это и не ломается при рефакторинге enum. Классический антипаттерн `plan[day.ordinal()]` разъезжается тихо, стоит кому-то переставить/вставить константу — ни одной ошибки компиляции.

### 6. Идиомы (меню, не дамп)

**RBAC-права** — каноничный кейс. `disjoint`/`containsAll` — это твои `(flags & mask) != 0` и `(flags & mask) == mask` из C#, только читаемо (прогнал — значения ниже верные):

```java
EnumSet<Permission> alice = EnumSet.of(Permission.READ, Permission.WRITE);
alice.contains(Permission.DELETE);                          // false
alice.addAll(EnumSet.of(Permission.DELETE));               // выдать право = union

EnumSet<Permission> adminLike = EnumSet.of(Permission.DELETE, Permission.ADMIN);
boolean privileged = !Collections.disjoint(alice, adminLike);   // true — пересечение непустое

EnumSet<Permission> required = EnumSet.of(Permission.READ, Permission.WRITE);
boolean ok = alice.containsAll(required);                        // true — есть весь набор
```

**Конечный автомат** — тут видно, зачем `EnumSet` и `EnumMap` ходят парой: ключ — состояние, значение — множество допустимых переходов (`canMove(NEW, SHIPPED)` вернул `false`, `canMove(NEW, PAID)` — `true`):

```java
enum State { NEW, PAID, SHIPPED, DELIVERED, CANCELLED }

Map<State, EnumSet<State>> transitions = new EnumMap<>(State.class);
transitions.put(State.NEW,     EnumSet.of(State.PAID, State.CANCELLED));
transitions.put(State.PAID,    EnumSet.of(State.SHIPPED, State.CANCELLED));
transitions.put(State.SHIPPED, EnumSet.of(State.DELIVERED));
transitions.put(State.DELIVERED, EnumSet.noneOf(State.class));   // терминальное
transitions.put(State.CANCELLED, EnumSet.noneOf(State.class));

boolean canMove(State from, State to) {
    return transitions.get(from).contains(to);   // проверка перехода = один бит
}
```

**`range` / `complementOf`** — когда константы идут подряд. Опираются на порядок объявления, поэтому **порядок констант в enum — часть контракта**: переставишь — сломаешь `range`.

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }
EnumSet<Day> workdays = EnumSet.range(Day.MON, Day.FRI);   // {MON..FRI}
EnumSet<Day> weekend  = EnumSet.complementOf(workdays);    // {SAT, SUN}
```

**Опции метода** (Item 36) — замена «семи `boolean`-параметров». Идиоматичный нюанс: в сигнатуре принимай `Set<E>`, не `EnumSet<E>` — гибкость вызывающему, эффективность по факту:

```java
void render(Image img, Set<RenderOption> opts) {              // Set, не EnumSet
    if (opts.contains(RenderOption.ANTIALIAS)) { /* ... */ }
}
render(img, EnumSet.of(RenderOption.ANTIALIAS, RenderOption.CACHE));
render(img, EnumSet.noneOf(RenderOption.class));              // "без опций" вместо null
```

`EnumSet` **мутабелен** — кладёшь в `record`/поле, делай защитную копию, иначе наружу утечёт ссылка на внутреннее множество:

```java
record User(String name, EnumSet<Permission> perms) {
    User { perms = EnumSet.copyOf(perms); }   // копия на входе (проверил: снова RegularEnumSet)
}
```

### 7. ⚠️ Подводные камни

> [!warning] ⚠️ Edge cases
> 
> 1. **Не потокобезопасны.** То, что внутри один `long`, тут не спасает: `add` — это `elements |= ...`, read-modify-write, три операции, не одна атомарная. Два потока пишут — гонка. Нужна внешняя синхронизация (`Collections.synchronizedSet/Map`, детали в главах 23/24).
> 2. **`null`-ключ → `NullPointerException`** (проверил и для `EnumSet`, и для `EnumMap`). Чтобы найти бит/ячейку, нужен `ordinal`, а у `null` его нет.
> 3. **`null`-значение в `EnumMap` — разрешено** (проверил: `put(SAT, null)` прошёл, `containsKey(SAT)` → `true`, `get(SAT)` → `null`). Внутри — sentinel-объект, отличающий «ячейка пустая» от «в ячейке лежит `null`». В голом массиве так не выйдет — `null` неотличим от незаписанной ячейки.
> 4. **Смешать два разных enum нельзя** — тип параметризован `<E>`, фабрики проверяют класс в рантайме (`ClassCastException`, если просочится через raw type).
> 5. **Память `EnumMap` ∝ числу констант, а не записей.** Положил одну запись в enum на 100 констант — массив всё равно на 100 ссылок. На практике копейки (ссылки, не объекты), но для гигантских разреженных enum держи в уме.
> 6. **`EnumSet.copyOf(Collection)`** на пустой не-`EnumSet`-коллекции бросит `IllegalArgumentException` (не может определить enum-тип); `copyOf(EnumSet)` безопасен всегда.

### На собесе

> [!question]- Почему `EnumSet` быстрее `HashSet`?  
> **RU:** `EnumSet` под капотом — битовый вектор: один `long`, бит на позиции `ordinal`. Операции — побитовые (`|`, `&`), без хеширования и без аллокации узлов `Node`. Множество это одно число, а не массив бакетов с обёртками — отсюда выигрыш и по скорости, и по памяти. Это type-safe замена C#-шного `[Flags]`.
> 
> **EN:** Under the hood `EnumSet` is a bit vector: a single `long`, one bit per `ordinal`. Operations are bitwise (`|`, `&`) — no hashing, no per-element `Node` allocation. The set is one number rather than an array of buckets with wrapper nodes, so it wins on both speed and memory. It's the type-safe replacement for C#'s `[Flags]`.
> 
> _Follow-up:_ «А если констант больше 64?» → переходит на `JumboEnumSet` с `long[]`, прозрачно; та же побитовая логика по словам массива (`ordinal >>> 6` — слово, `ordinal & 63` — бит).  
> _Follow-up:_ «Чем заменяет `[Flags]`?» → ровно им: type-safe множество вместо ручных степеней двойки и масок.  
> _Слабый ответ:_ «ну он специализированный» — не проходит; ждут механизм (битовый вектор).

> [!question]- Почему `EnumMap` лучше `HashMap` для enum-ключей?  
> **RU:** Внутри `EnumMap` — массив `Object[]`, индексированный `ordinal` ключа. `get`/`put` — прямое обращение по индексу, без хеша, без коллизий, без `Node`. Плюс гарантированный порядок итерации — порядок объявления констант, не вставки и не хеш-порядок.
> 
> **EN:** Inside, `EnumMap` is an `Object[]` indexed by the key's `ordinal`. `get`/`put` are direct index access — no hashing, no collisions, no `Node` wrappers. Plus a guaranteed iteration order: declaration order, not insertion or hash order.
> 
> _Follow-up:_ «Где `ordinal` вообще легитимно использовать?» → вот здесь, внутри `EnumSet`/`EnumMap` как адрес. В прикладном коде — нет; ровно для этого `ordinal` и существует.  
> _Follow-up:_ «Почему в конструктор обязателен `Class`?» → чтобы в момент создания выделить массив ровно на `values().length` ячеек; enum конечен, размер известен заранее.
