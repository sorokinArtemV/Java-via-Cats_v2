В бэкенде постоянно всплывают две формы «данные, привязанные к enum-ключу»: либо **значение на каждый ключ** — конфиг на статус заказа, обработчик на тип события, цена на тариф, — либо **множество enum-значений** — набор прав пользователя, множество активных дней. Инстинкт с .NET-фоном: `Dictionary<TEnum,V>` в первом случае, ручной `[Flags]`-enum с битовыми масками во втором. В Java обоим есть специализированный ответ — `EnumMap` и `EnumSet`, — и они на порядок быстрее и компактнее универсальных `HashMap`/`HashSet`. Причина одна: enum-ключи это конечный известный набор, у каждого элемента уже есть готовый «адрес» — его `ordinal`. Этот урок про то, _почему_ они быстрее и когда за ними тянуться.

**Mental model.** Множество enum'ов — это **одно число** (бит на позиции `ordinal`); мапа с enum-ключами — это **массив** (ячейка по индексу `ordinal`). Обе коллекции держатся на одном трюке: `ordinal` как прямой адрес. Помнишь правило из 5.1 «не строй прикладную логику на `ordinal`»? Вот единственное место, где он легитимен — внутри этих двух структур.

#### `EnumSet` — поверхность

Это `Set<E>`, специализированный под enum-ключи. Весь API обычного `Set` на месте плюс удобные фабрики:

```java
enum Permission { READ, WRITE, DELETE, ADMIN, AUDIT }

EnumSet<Permission> granted  = EnumSet.of(Permission.READ, Permission.WRITE);
EnumSet<Permission> none     = EnumSet.noneOf(Permission.class);   // пустой
EnumSet<Permission> all      = EnumSet.allOf(Permission.class);    // все константы
EnumSet<Permission> range    = EnumSet.range(Permission.READ, Permission.DELETE); // [READ..DELETE]
EnumSet<Permission> readonly = EnumSet.complementOf(               // дополнение
        EnumSet.of(Permission.WRITE, Permission.DELETE, Permission.ADMIN));

granted.addAll(range);     // объединение (union)
granted.retainAll(range);  // пересечение (intersection)
```

Прогон на JDK 21:

```
granted  = [READ, WRITE]
none     = []
all      = [READ, WRITE, DELETE, ADMIN, AUDIT]
range    = [READ, WRITE, DELETE]
readonly = [READ, AUDIT]
class    = java.util.RegularEnumSet
```

Две вещи из вывода, на которые смотри внимательно. Во-первых, порядок итерации — это **порядок объявления** (он же порядок `ordinal`), не порядок вставки: я отдельно проверил — вставил `AUDIT, READ, ADMIN`, а итерируется `[READ, ADMIN, AUDIT]`. Javadoc это гарантирует прямым текстом: итератор обходит элементы в их естественном порядке — в порядке объявления констант enum. Во-вторых, класс — `RegularEnumSet`, а не сам `EnumSet` (он абстрактный).

> [!tip] 💡 Из C#  
> `EnumSet` — это твой `[Flags]`-enum, но без ручных степеней двойки и без масок в прикладном коде. Сравни дословно: `EnumSet.of(READ, WRITE)` вместо `Read | Write`; `set.contains(WRITE)` вместо `(flags & Write) != 0`; `addAll` вместо `|=`; `retainAll` вместо `&`; `Collections.disjoint(a,b)` вместо `(a & b) != 0`; `a.containsAll(b)` вместо `(a & b) == b`. И type safety настоящая: в `EnumSet<Permission>` нельзя случайно положить «левое» число или значение другого enum — компилятор не даст, а фабрики ещё и в рантайме проверяют класс элемента.

#### `EnumSet` — механика

`EnumSet` — абстрактный класс (в Java 21 даже `sealed`), фабрики возвращают одну из двух package-private реализаций, и выбор происходит при создании по числу констант в enum. По Javadoc и исходникам OpenJDK:

- **`RegularEnumSet`** — хранит **один `long elements`**. Бит на позиции `i` взведён ⟺ в множестве присутствует константа с `ordinal() == i`. Работает для enum с **≤ 64 константами** (в `long` ровно 64 бита).
- **`JumboEnumSet`** — хранит **`long[] elements`** для enum с **> 64 константами**. Слово берётся как `ordinal >>> 6`, бит внутри слова — `ordinal & 63`.

Раз всё множество — это число, операции становятся чистой побитовой арифметикой (это внутренности `RegularEnumSet` из OpenJDK, не наблюдаемое из твоего кода):

|Операция|Под капотом|
|---|---|
|`add(e)`|`elements \|= (1L << e.ordinal())`|
|`contains(e)`|`(elements & (1L << e.ordinal())) != 0`|
|`addAll(other)` (union)|`elements \|= other.elements`|
|`retainAll(other)` (intersection)|`elements &= other.elements`|
|`removeAll(other)`|`elements &= ~other.elements`|

Объединить два множества хоть на 60 элементов — это **один** `|` над двумя `long`. Вот тот самый «адрес по `ordinal`» — и он же лежит в основе `EnumMap`, поэтому показываю обе структуры одной картинкой:

#### Почему это быстрее `HashSet` — конкретно

`HashSet` внутри — это `HashMap`, где твои элементы лежат ключами (детали устройства — глава 13). Каждый `add` даёт: вычисление `hashCode()`, поиск бакета по маске `hash & (n-1)`, **аллокацию объекта-обёртки** `Node`, потенциальную обработку коллизий. Память — массив бакетов плюс узел на каждый элемент.

`EnumSet` на тех же данных: один примитивный `long` на всё множество, **ноль аллокаций** per-element, операции — побитовые инструкции процессора. Javadoc осторожен в формулировке: все базовые операции выполняются за константное время и «likely (though not guaranteed) to be much faster» — но условие, при котором это держится, понятное: enum-ключи, конечный набор. Для них `EnumSet` — не преждевременная оптимизация, а правильный дефолт: первый выбор, а не `HashSet`. Effective Java, Item 36 так и называется — «use `EnumSet` instead of bit fields».

#### `EnumMap` — поверхность

`EnumMap` решает соседнюю боль: **значение на каждый enum-ключ**. Инстинкт — `HashMap<Day,V>`, работает, но снова платишь за хеш-машинерию там, где ключи — конечный набор с готовыми адресами.

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

EnumMap<Day, String> plan = new EnumMap<>(Day.class);  // Class enum'а ОБЯЗАТЕЛЕН
plan.put(Day.FRI, "релиз");
plan.put(Day.MON, "планёрка");
plan.put(Day.WED, "review");
```

Сразу бросается синтаксическое отличие: конструктору **обязательно передаётся `Day.class`**. `EnumMap` не узнаёт тип из ключей по ходу — ему нужно знать весь набор констант _заранее_, чтобы выделить массив нужной длины. Эта деталь прямо следует из механики.

Прогон на JDK 21 (я вставлял ключи в порядке `FRI → MON → WED`, а рядом заполнил `HashMap` всеми семью днями):

```
EnumMap iter : {MON=планёрка, WED=review, FRI=релиз}
HashMap iter : [THU, SUN, SAT, MON, TUE, WED, FRI]
class        : java.util.EnumMap
```

`EnumMap` итерируется `MON → WED → FRI` — **порядок объявления**, всегда, гарантированно, хотя клал я в другом порядке. А `HashMap` на всех семи днях выдал `THU, SUN, SAT, MON…` — ни порядок вставки, ни объявления, просто как легли хеши, и на другом наборе ключей будет иным. Это важная разница на практике: `EnumMap` даёт стабильный предсказуемый порядок бесплатно — из-за него не «прыгают» логи и вывод между запусками.

#### `EnumMap` — механика

Внутри `EnumMap` — обычный **`Object[] vals`**, проиндексированный `ordinal` ключа, плюс ссылка на массив всех констант (из того самого `Day.class`). Javadoc: «implemented as arrays… extremely compact and efficient».

- `put(key, value)` → `vals[key.ordinal()] = maskNull(value)` — прямая запись по индексу.
- `get(key)` → `unmaskNull(vals[key.ordinal()])` — прямое чтение.
- Нет хеширования, нет бакетов, нет коллизий (у каждой константы уникальный индекс — коллизий не бывает _в принципе_), нет аллокации `Node` на пару. Просто массив.

Вот зачем `Class` в конструкторе: в этот момент выделяется `Object[]` ровно на `Day.values().length` ячеек. Размер известен заранее — enum конечен.

> [!tip] 💡 Из C#  
> Прямого аналога нет. В C# для enum-ключей ты возьмёшь `Dictionary<TEnum,V>` — и заплатишь за `GetHashCode`/`Equals` и бакеты, как с любым ключом (исторически .NET ещё и боксил enum-ключ при сравнении, пока это не починили в рантайме). `EnumMap` — это специализация, которой в BCL просто нет; там пришлось бы руками катать `V[]`, индексируясь по `(int)enumValue`, и терять весь сахар `Map`. `EnumMap` даёт и плоский массив, и полноценный `Map`-интерфейс сразу.

> [!example] 🔬 Глубже / на синьора  
> **Вот единственное легитимное использование `ordinal` — как адреса.** У `EnumSet` это номер бита, у `EnumMap` — индекс в массиве. В 5.1 было правило «не персисти `ordinal`, не строй на нём прикладную логику» — оно про _твой_ код. А здесь, в кишках `EnumSet`/`EnumMap`, `ordinal` ровно для этого и существует: JDK владеет и множеством констант, и их порядком, поэтому имеет право; ты — нет. Именно поэтому Item 37 называется «use `EnumMap` **instead of** ordinal indexing»: классический антипаттерн — `String[] plan = new String[7]; plan[day.ordinal()] = ...` руками. Он работает, пока кто-то не переставит константу — и весь массив тихо разъезжается без ошибки компиляции. `EnumMap` делает то же самое внутри, но привязка инкапсулирована и не ломается при рефакторинге твоего кода.
> 
> **Как `null`-значение уживается с массивом.** В `EnumMap` `null`-значение разрешено (в отличие от `null`-ключа). Но в голом массиве `null` неотличим от незаписанной ячейки. `EnumMap` решает это sentinel-объектом (`maskNull`/`unmaskNull` в OpenJDK): твой `null` хранится как специальный маркер, поэтому `containsKey` честно отвечает `true`, не путая «значение есть и оно `null`» с «ключа нет». Проверено на JDK 21: `put(SAT, null)` → `containsKey(SAT) == true`, `get(SAT) == null`.
> 
> **`values()` не клонируется зря.** Из 5.1 помнишь, что `values()` отдаёт клон на каждый вызов. `EnumSet`/`EnumMap` этого избегают: внутри они берут _разделяемый нескопированный_ массив констант через привилегированный `SharedSecrets.getJavaLangAccess().getEnumConstantsShared(...)` — то самое «JDK кеширует `values()` там, где это важно», о котором я говорил в 5.1.

> [!warning] ⚠️ Подводные камни
> 
> 1. **Обе коллекции не потокобезопасны.** Javadoc прямо велит оборачивать: `Collections.synchronizedSet(EnumSet.noneOf(...))` / `synchronizedMap`. То, что внутри `EnumSet` один `long`, не спасает: `add` — это `elements |= ...`, то есть read-modify-write из трёх операций, не одна атомарная. Два потока пишут одновременно — гонка (детали — главы 23/24).
> 2. **`null`-элемент/ключ → `NullPointerException`.** `EnumSet.add(null)` и `EnumMap.put(null, v)` бросают NPE (проверено): множество/мапа знают только свои константы, у `null` нет `ordinal`. Асимметрия: `null`-**значение** в `EnumMap` при этом разрешено.
> 3. **Два разных enum в одну коллекцию не смешать.** Тип параметризован `<E>`, но через raw type можно просочиться — тогда ловит рантайм: `EnumSet raw = EnumSet.noneOf(Day.class); raw.add(Color.RED);` даёт `ClassCastException` (проверено).
> 4. **`EnumSet` мутабелен — делай защитную копию.** Кладёшь в поле/`record` — копируй на входе, иначе наружу утечёт ссылка на твоё внутреннее множество: `perms = EnumSet.copyOf(perms)`. Нужно неизменяемое — `Collections.unmodifiableSet(EnumSet.of(...))` (внутри остаётся быстрый битовый `EnumSet`, снаружи read-only).
> 5. **Память `EnumMap` пропорциональна числу констант, а не записей.** Массив всегда полной длины: одна запись в enum на 100 констант → массив всё равно на 100 ссылок. На практике копейки (ссылки, не объекты), но для гигантских разреженных enum держи в уме.
> 6. **В сигнатурах принимай интерфейс, конструируй реализацию** (идиома Item 36): параметр — `Set<RenderOption>` / `Map<Day,V>`, а внутри `EnumSet`/`EnumMap`. Вызывающий волен передать любую реализацию, эффективность — по факту.

#### На собесе

> [!question]- Почему `EnumSet` быстрее `HashSet`?  
> **RU:** `EnumSet` под капотом — битовый вектор: один `long`, бит на позиции `ordinal` = константа присутствует. Операции — побитовые (`|`, `&`), без вычисления хеша и без аллокации узлов-обёрток на элемент. `HashSet` на каждый `add` считает `hashCode()`, ищет бакет и аллоцирует `Node`; `EnumSet` — одна инструкция над числом. Отсюда выигрыш и по скорости, и по памяти.
> 
> **EN:** Under the hood `EnumSet` is a bit vector: a single `long`, the bit at position `ordinal` meaning the constant is present. Operations are bitwise (`|`, `&`), with no hash computation and no per-element wrapper-node allocation. `HashSet` on every `add` computes `hashCode()`, finds a bucket, and allocates a `Node`; `EnumSet` is one instruction over an integer. Hence the win in both speed and memory.
> 
> _Follow-up:_ «а если констант больше 64?» → переходит на `JumboEnumSet` с `long[]`, прозрачно; та же побитовая логика, слово = `ordinal >>> 6`, бит = `ordinal & 63`. _Follow-up:_ «чем заменяет `[Flags]` из C#?» → ровно им: type-safe множество вместо ручных степеней двойки и масок. _Слабый ответ:_ «он специализированный» — не проходит, ждут «битовый вектор».

> [!question]- Почему `EnumMap` лучше `HashMap` для enum-ключей? _(частый)_  
> **RU:** Внутри `EnumMap` — массив `Object[]`, индексированный `ordinal` ключа. `get`/`put` — прямое обращение по индексу: без хеша, без бакетов, без коллизий (у каждой константы уникальный индекс), без аллокации `Node`. Массив выделяется один раз при создании (поэтому в конструктор обязателен `Class` enum'а — из него берётся размер). Быстрее и компактнее `HashMap`, плюс детерминированный порядок итерации.
> 
> **EN:** Inside, `EnumMap` is an `Object[]` array indexed by the key's `ordinal`. `get`/`put` are direct index access: no hashing, no buckets, no collisions (each constant has a unique index), no `Node` allocation. The array is allocated once at creation (which is why the constructor requires the enum `Class` — that's where the size comes from). Faster and more compact than `HashMap`, plus a deterministic iteration order.
> 
> _Follow-up:_ «какой порядок итерации?» → гарантированно порядок объявления констант, не вставки и не хеш-порядок. _Follow-up:_ «а `null`?» → `null`-ключ → NPE, но `null`-значение разрешено (внутри sentinel, поэтому `containsKey` не путает его с отсутствием ключа).

> [!question]- Где вообще `ordinal()` можно легитимно использовать?  
> **RU:** Внутри `EnumSet`/`EnumMap` как адрес — номер бита и индекс ячейки соответственно. Это единственное штатное применение: JDK владеет набором констант и их порядком, поэтому имеет право. В прикладном коде `ordinal()` использовать нельзя — ни персистить, ни строить логику: перестановка констант тихо всё сломает (Effective Java Item 37: `EnumMap` вместо ordinal-indexing руками).
> 
> **EN:** Inside `EnumSet`/`EnumMap` as an address — the bit index and the array index respectively. That's the only sanctioned use: the JDK owns the set of constants and their order, so it may rely on it. In application code `ordinal()` must not be used — neither persisted nor used for logic: reordering constants silently breaks everything (Effective Java Item 37: `EnumMap` instead of manual ordinal indexing).
> 
> _Follow-up:_ «а зачем тогда `ordinal()` вообще выставлен наружу?» → в основном ради этих коллекций и совместимости; для твоего кода это ловушка, а не инструмент.
