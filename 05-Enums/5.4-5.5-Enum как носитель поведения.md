В первом уроке главы `enum` у нас _хранил_ данные — константы-синглтоны с полями. Теперь он будет _вести себя_, причём по-разному для каждой константы. Ситуация, где это выстреливает, знакомая: есть набор операций и код, который по операции решает, что посчитать. Пока операций три — терпимо; на пятой `switch` начинает жить своей жизнью и врать в рантайме. Задача урока — научиться вешать поведение прямо на константу так, чтобы забыть про реализацию было физически нельзя, и разобрать, что при этом происходит под капотом. Спойлер: обычный полиморфизм из урока про диспатч, никакой enum-магии.

**Mental model.** `enum` в Java — полноценный класс, поэтому поведение можно привязать к каждой константе. Вызов `op.apply(...)` — это обычный виртуальный диспатч по реальному типу объекта, а не скрытый `switch` внутри.

#### 1. Проблема: `switch`, который молчит

Классический способ навесить поведение на enum — `switch` снаружи:

```java
enum Operation { PLUS, MINUS, TIMES, DIVIDE }

static double apply(Operation op, double x, double y) {
    switch (op) {
        case PLUS:   return x + y;
        case MINUS:  return x - y;
        case TIMES:  return x * y;
        case DIVIDE: return x / y;
        default:     throw new AssertionError("Unknown op: " + op);
    }
}
```

Работает. А потом приходит задача добавить `REMAINDER` (остаток от деления). Ты дописываешь константу в enum — и всё компилируется. Зелёный билд, мёрж, прод. А `switch` молча уходит в `default` и кидает `AssertionError` в рантайме на первом же `%`. Компилятор промолчал, потому что `switch` с `default` синтаксически полон. Логика разъехалась с данными, и поймал это пользователь, а не сборка.

Вопрос, на который отвечает весь урок: можно ли сделать так, чтобы добавить константу _без_ реализации поведения было **невозможно** — чтобы код просто не скомпилировался? В Java можно, и встроено это прямо в enum.

#### 2. Поведение на константе: `abstract`-метод

Раз enum — полноценный класс, у него могут быть методы. А раз могут быть методы, один из них можно объявить `abstract` — и тогда **каждая константа обязана дать своё тело**:

```java
public enum Operation {
    PLUS  { public double apply(double x, double y) { return x + y; } },
    MINUS { public double apply(double x, double y) { return x - y; } },
    TIMES { public double apply(double x, double y) { return x * y; } };

    public abstract double apply(double x, double y);
}
```

Никакого `switch`. Вызов: `Operation.PLUS.apply(6, 2)` → `8.0`, `Operation.TIMES.apply(6, 2)` → `12.0`. Каждая константа несёт свой кусок логики на себе — это называется **per-constant body**: блок `{ ... }` сразу после имени константы, где ты переопределяешь методы.

Ключевое — `apply` объявлен `abstract`, поэтому **компилятор физически не даст добавить константу без реализации**. Допишешь `REMAINDER;` без тела — не скомпилируется (проверено на JDK 21):

```
error: Operation is abstract; cannot be instantiated
        REMAINDER;   // добавил константу, забыл тело
        ^
```

Та самая «exhaustiveness без `switch`»: проверка вариантов переехала из рантайма в компиляцию.

> [!tip] 💡 Из C#  
> C#-овский `enum` так не умеет вообще — это именованная обёртка над числом (`int` по умолчанию), к нему нельзя привязать ни метод, ни тем более разное поведение на значение. Чтобы получить то же самое в C#, есть три обходных пути: `switch` по enum'у (та самая хрупкая штука сверху), `Dictionary<Operation, Func<double,double,double>>` со словарём делегатов, или полноценная иерархия классов руками (паттерн «type-safe enum»). В Java это одна языковая конструкция.

> [!warning] ⚠️ Подводные камни
> 
> 1. **`@Override` на телах констант — не косметика, а страховка.** Опечатаешься в сигнатуре (`apply(double x, int y)` вместо `double y`) — без `@Override` ты не переопределишь абстрактный метод, а _добавишь новый_, и enum останется абстрактным по `apply`. Ошибка будет, но невнятная («PLUS не реализует abstract apply»); с `@Override` — точная и сразу на нужной строке.
> 2. **Не тащи на enum внешние заботы.** Поведение кладут на константу, только когда оно _неотъемлемо от того, чем константа является_. `apply` для `Operation` — да, операция _и есть_ своё действие. А рендеринг, маппинг в DTO, обращение к базе — это внешний контекст, не «чем константа является»; на enum его вешать не надо, иначе он превращается в свалку. Такое поведение — снаружи (`switch`/мапа/сервис).

Это идиоматичный Java-код: ровно этот пример разбирает Джошуа Блох в _Effective Java_, Item 34, и рекомендует именно так. Синтаксис непривычный после C#, но это мейнстрим, а не хак.

#### 3. Механика: тело константы — это анонимный подкласс

Вот ключ ко всему уроку. Когда ты пишешь `abstract double apply(...)`, ты делаешь класс `Operation` **абстрактным по этому методу**. Абстрактный класс нельзя инстанцировать напрямую — а константы существуют как объекты. Как? Компилятор для каждой константы с телом **генерирует анонимный подкласс** `Operation`, переопределяющий `apply` (JLS §8.9.2). То есть

```java
PLUS { public double apply(double x, double y) { return x + y; } }
```

компилятор концептуально разворачивает в

```java
PLUS = new Operation() { public double apply(double x, double y) { return x + y; } };
```

`PLUS` — это не «`Operation` с флажком внутри». Это **экземпляр отдельного безымянного подкласса**, у которого свой `apply`. И когда ты зовёшь `op.apply(6, 2)`, происходит ровно то, что в уроке про диспатч: JVM смотрит на реальный тип объекта в куче, находит его `apply` через таблицу методов и прыгает туда. Не `switch`, не `if` — виртуальный диспатч, тот же механизм, что у любого `@Override`.

![[Pasted image 20260708163807.png]]

Каждое тело константы — **отдельный сгенерированный класс**. Проверено на JDK 21:

```
Operation.PLUS.getClass()                     // class Operation$1   (НЕ Operation!)
Operation.PLUS.getClass() == Operation.class   // false
Operation.PLUS.getDeclaringClass()             // class Operation
Operation.PLUS instanceof Operation            // true
```

`getClass()` возвращает анонимный подкласс `Operation$1`, а не сам `Operation` — потому что `PLUS` буквально _экземпляр подкласса_. А `getDeclaringClass()` всегда вернёт сам enum-тип `Operation` — это надёжный способ спросить «какого enum'а эта константа», когда `getClass()` врёт из-за тел.

> [!example] 🔬 Глубже / на синьора  
> **Байткод.** `op.apply(...)` компилируется в `invokevirtual`: вызов идёт через слот в таблице методов реального класса объекта (в HotSpot указатель на `klass` лежит в заголовке объекта). Это тот же механизм, что и в уроке про диспатч — enum здесь не добавляет ничего своего. JIT на мономорфном или `final`-сайте прекрасно девиртуализирует и инлайнит такой вызов, так что «дороговизна» подкласса на практике нулевая.
> 
> **Имя `Operation$1` — не гарантия.** Так именует **`javac`** анонимные классы top-level enum'а (по порядку появления). У вложенного enum'а имя будет `Enclosing$Operation$1`, а сам суффикс `$1` — деталь компилятора, не спецификация. Опираться в коде на строку имени нельзя; для логики бери `getDeclaringClass()`.
> 
> **Смысловой замок:** константы enum полиморфны, потому что каждая с телом — это подкласс. Никакого скрытого `switch` внутри enum нет.

> [!warning] ⚠️ Подводный камень  
> Хочешь надёжно сравнить «какого типа эта константа» или залогировать — бери `getDeclaringClass()`, не `getClass()`. И осторожно: `PLUS instanceof Operation` → `true` (подкласс _является_ `Operation`), а `PLUS.getClass() == Operation.class` → `false`. На этой паре ловят на собесе.

#### 4. Поведение как данные: лямбда-в-поле

Поведение можно задать не телом константы, а **данными** — положить функцию в поле. Java даёт для этого готовый функциональный интерфейс `DoubleBinaryOperator` (два `double` на вход, `double` на выход):

```java
import java.util.function.DoubleBinaryOperator;

public enum Operation {
    PLUS ((x, y) -> x + y),
    MINUS((x, y) -> x - y),
    TIMES((x, y) -> x * y);

    private final DoubleBinaryOperator op;               // поведение как данные

    Operation(DoubleBinaryOperator op) { this.op = op; }

    public double apply(double x, double y) {
        return op.applyAsDouble(x, y);
    }
}
```

Снаружи всё так же: `Operation.PLUS.apply(6, 2)` → `8.0`. Но внутри принципиально другое — здесь **нет тел констант**, значит **нет анонимных подклассов**. Все константы — экземпляры самого `Operation` (проверено на JDK 21):

```
Operation.PLUS.getClass()                     // class Operation   (сам enum, не $1!)
Operation.PLUS.getClass() == Operation.class   // true
```

Это водораздел двух подходов на уровне механики.

> [!tip] 💡 Из C#  
> Второй способ — это ровно `Dictionary<MyEnum, Func<...>>` из C#, только встроенный в сам тип и без отдельной мапы: поле `op` играет роль значения словаря, константа — ключа. Разница в контракте: в C# словарь живёт _снаружи_ enum'а, и можно забыть добавить запись для новой константы (рантайм-`KeyNotFoundException`). В Java конструктор enum'а **обязывает** дать функцию каждой константе при объявлении — забыть нельзя, не скомпилируется.

#### 5. Тело vs лямбда: как выбирать

Прямой ответ: **лямбда-в-поле — когда поведение это чистая функция от аргументов** (как `apply`: дал два числа — получил число, ничего из enum'а не нужно). Компактнее, классов меньше. **Тело/обычный метод — когда поведению нужен доступ к состоянию самой константы**: к её полям, к `this`.

```java
enum Planet {
    EARTH(5.97e24, 6.37e6),
    MARS (6.42e23, 3.39e6);

    private final double mass, radius;
    Planet(double mass, double radius) { this.mass = mass; this.radius = radius; }

    double gravity() { return 6.67e-11 * mass / (radius * radius); }   // читает this.mass / this.radius
}
// EARTH.gravity() -> 9.81   MARS.gravity() -> 3.73   (проверено на JDK 21)
```

У метода `gravity()` есть `this` — он спокойно читает поля конкретной планеты. А вот лямбда, которую ты передаёшь **в объявлении константы**, живёт в **static-контексте**: константы enum'а — это неявно `static`-поля, и их аргументы вычисляются при статической инициализации, где `this` константы ещё не существует в области видимости. Поэтому лямбда там не может даже _сослаться_ на поля константы — это ошибка компиляции, а не рантайм-сюрприз (проверено):

```java
// не скомпилируется:
PLUS((x, y) -> x + y + factor);   // factor — instance-поле enum'а
// error: non-static variable factor cannot be referenced from a static context
```

Важно не путать причину: дело **не в том**, что «поля ещё не присвоены на момент конструктора» (тело лямбды вообще исполняется позже, при вызове `apply`), а в том, что на месте объявления константы нет `this` — это compile-time scoping, а не runtime timing.

> [!warning] ⚠️ Подводный камень  
> Не пытайся из лямбды-в-объявлении обращаться к `this` или полям константы — не скомпилируется (static-контекст). Нужно состояние константы — бери тело или обычный метод.

Про перформанс — это последнее, по чему стоит выбирать: тело = +1 класс на константу в metaspace (JIT инлайнит виртуальный вызов), лямбда = +1 индирекция через поле. Разница микроскопическая. **Выбирай по читаемости и по тому, нужен ли доступ к состоянию, а не по скорости.**

#### 6. Общее поведение у групп констант: strategy-enum

А если у _групп_ констант поведение общее? Наивно — скопировать одно и то же тело в каждую константу группы, но это дублирование, которое разъедется при первой правке. Канонический пример Блоха (Item 34): расчёт зарплаты, где будни и выходные считают сверхурочные по-разному, но внутри каждой группы — одинаково. Решение: вынести поведение в **вложенный strategy-enum** и делегировать ему:

```java
enum PayrollDay {
    MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY,          // будни
    SATURDAY(PayType.WEEKEND), SUNDAY(PayType.WEEKEND);    // выходные

    private final PayType payType;
    PayrollDay()             { this(PayType.WEEKDAY); }
    PayrollDay(PayType type) { this.payType = type; }

    int pay(int minutes, int rate) { return payType.pay(minutes, rate); }   // делегирование

    private enum PayType {                        // стратегия: логика живёт ЗДЕСЬ, один раз
        WEEKDAY { int overtime(int m, int r) { return m <= SHIFT ? 0 : (m - SHIFT) * r / 2; } },
        WEEKEND { int overtime(int m, int r) { return m * r / 2; } };

        abstract int overtime(int m, int r);
        private static final int SHIFT = 8 * 60;

        int pay(int m, int r) { return m * r + overtime(m, r); }
    }
}
// MONDAY.pay(600, 1)   -> 660   (2 ч сверхурочных ×0.5)
// SATURDAY.pay(480, 1) -> 720   (весь день по правилу выходного)   — проверено на JDK 21
```

Каждый день выбирает свою стратегию оплаты при объявлении, а формула живёт в `PayType` в одном месте. Правило: **однородное поведение внутри групп констант — выноси стратегию во вложенный enum, не размазывай тело по константам.**

#### 7. Enum и интерфейсы (расширяемые enum'ы)

Сначала жёсткий факт, который любят спрашивать: **enum не может `extends` класс.** Слот единичного наследования уже занят — каждый enum неявно `extends java.lang.Enum<E>`, а второго родителя в Java нет. А вот **`implements` интерфейсов — сколько угодно.** Это спроектированная лазейка для одной конкретной проблемы.

Проблема: enum'ы **не расширяемы**. Ты не можешь «дописать» константу в чужой `Operation` снаружи — он закрыт по определению. Обычно это плюс (exhaustiveness!), но иногда нужно, чтобы клиентский код умел работать с операциями из _разных наборов_ — базовых и чьих-то дополнительных. Наследованием enum'а не сделать; интерфейсом — да. Это _Effective Java_, Item 38, «Emulate extensible enums with interfaces».

```java
public interface Operation {
    double apply(double x, double y);

    default String describe(double x, double y) {        // default-метод (Java 8, JEP 126)
        return x + " " + this + " " + y + " = " + apply(x, y);
    }
}

public enum BasicOperation implements Operation {
    PLUS ("+") { public double apply(double x, double y) { return x + y; } },
    TIMES("*") { public double apply(double x, double y) { return x * y; } };

    private final String symbol;
    BasicOperation(String symbol) { this.symbol = symbol; }
    @Override public String toString() { return symbol; }
}
```

Где-то в другом модуле появляется второй enum — `ExtendedOperation` с `EXP` и `REMAINDER`, тоже `implements Operation`. Клиентский код пишется против интерфейса, ему безразлично, откуда операция:

```java
static void runAll(Collection<? extends Operation> ops, double x, double y) {
    for (Operation op : ops) System.out.println(op.describe(x, y));
}
// runAll(List.of(BasicOperation.values()),    4, 2)  ->  4.0 + 2.0 = 6.0,  4.0 * 2.0 = 8.0
// runAll(List.of(ExtendedOperation.values()), 4, 2)  ->  4.0 ^ 2.0 = 16.0, ...   (проверено)
```

Два независимых enum'а разделяют один контракт. Добавить новый набор операций = объявить новый enum, реализующий `Operation`, не трогая существующие.

![[Pasted image 20260708163844.png]]

Про **контракт**: раз enum заявил `implements Operation`, он _обязан_ дать `apply`. Способов три — (1) per-constant телами, как выше; (2) одним обычным методом на весь enum, если поведение общее; (3) `default`-методом в самом интерфейсе, если есть разумная реализация по умолчанию. `describe` — как раз третий случай: оба enum'а получили его даром, переопределять не нужно.

> [!tip] 💡 Из C#  
> В C# реализовать интерфейс enum'ом **нельзя** — точка. Хочешь набор «операций» с общим контрактом, который ещё и расширяется разными группами, — это иерархия классов (`abstract class` + наследники) или интерфейс + классы, плюс ручная фабрика/реестр, чтобы перечислить «все варианты». То, что в Java делает один `enum X implements I`, в C# превращается в россыпь классов.

> [!warning] ⚠️ Подводный камень — честная цена Item 38  
> Расширяемость покупается **потерей exhaustiveness**. Пока был один enum с `abstract apply`, компилятор знал все варианты: забыл реализовать константу — не скомпилировалось; `switch` по enum'у можно было проверить на полноту. Как только операции размазаны по нескольким enum'ам за интерфейсом, компилятор **больше не знает всех реализаций** — никакого exhaustive `switch`, страховочной сетки нет. Прямой размен: гибкость против проверки полноты. Выбираешь осознанно.

> [!example] 🔬 Глубже / на синьора  
> **Bounded intersection type token.** Если клиенту нужны _и_ enum-свойства (перебрать все константы через `getEnumConstants()`), _и_ контракт `Operation`, сигнатуру пишут через пересечение типов (проверено на JDK 21):
> 
> java
> 
> ```java
> static <T extends Enum<T> & Operation> void runAll(Class<T> type, double x, double y) {
>     for (T op : type.getEnumConstants()) System.out.println(op.describe(x, y));
> }
> // runAll(BasicOperation.class, 4, 2);
> ```
> 
> `<T extends Enum<T> & Operation>` требует, чтобы `T` был одновременно enum'ом и реализовывал `Operation`. Болит именно дженерик-сигнатура — но она даёт типобезопасный обход всех констант конкретного набора.
> 
> **Мостик в главу 17.** Enum, реализующий интерфейс, спокойно ложится в `EnumSet`/`EnumMap` (урок 5.6). А если нужна именно _закрытая_ иерархия с проверкой полноты, но богаче enum'а (разные поля у разных вариантов), в Java 21 есть альтернатива — `sealed`-интерфейсы + `record`-ы + pattern matching for switch. Грубо: **enum с поведением — когда варианты однородны и их конечный набор; `sealed` + `record` — когда варианты структурно разные, но тоже закрыты.**

#### На собесе

> [!question]- Как сделать разное поведение на каждую константу без `switch`?  
> **Model answer (RU):** `abstract`-метод в enum: каждая константа даёт своё тело (per-constant body). Компилятор заставляет реализовать метод для каждой константы — добавить константу без реализации не скомпилируется, тогда как `switch` с `default` молча промахнётся в рантайме.
> 
> **Model answer (EN):** Declare an `abstract` method on the enum; each constant supplies its own body (per-constant body). The compiler forces an implementation for every constant — you can't add a constant without one, whereas a `switch` with `default` would silently miss at runtime.
> 
> _Follow-up:_ «Что происходит на уровне типов?» → каждая константа с телом — анонимный подкласс enum'а, переопределяющий метод; `PLUS` это по сути `new Operation(){ ... }`. _Follow-up:_ «`PLUS.getClass()` что вернёт?» → `Operation$1` (подкласс), не `Operation`; enum-тип надёжно берётся через `getDeclaringClass()`.
> 
> _Слабый ответ:_ «сделать `if`/`switch` внутри метода enum'а» — это то же ветвление, просто переехавшее внутрь; теряется главное преимущество (компилятор не заставит реализовать новую константу).

> [!question]- Enum-strategy или `switch` — что выберешь и почему?  
> **Model answer (RU):** Если поведение _принадлежит_ константе (операция и есть своё действие) — enum с per-constant методом: новая константа без реализации не скомпилируется. Если поведение _внешнее_ и не про суть константы (рендеринг, маппинг в чужой формат) — нормально и `switch`, лучше exhaustive switch expression.
> 
> **Model answer (EN):** If the behavior _belongs_ to the constant, use a per-constant method on the enum — a new constant without an implementation won't compile. If the behavior is _external_ and not about what the constant is (rendering, mapping to a foreign format), a `switch` is fine — preferably an exhaustive switch expression.
> 
> _Follow-up:_ «Когда `switch` всё-таки безопасен?» → switch expression без `default` (Java 14+, JEP 361) проверяется компилятором на полноту — добавишь константу, забудешь ветку, не скомпилируется. Хрупок именно _классический statement switch с `default`_.

> [!question]- Может ли enum `extends` класс? `implements` интерфейс?  
> **Model answer (RU):** `extends` — нет: слот единичного наследования занят `java.lang.Enum`. `implements` — да, сколько угодно интерфейсов.
> 
> **Model answer (EN):** `extends` — no: the single-inheritance slot is taken by `java.lang.Enum`. `implements` — yes, any number of interfaces.
> 
> _Follow-up:_ «Зачем enum'у интерфейс?» → расширяемость (Item 38): несколько enum'ов делят один контракт, клиент типизирован интерфейсом. _Follow-up:_ «Что теряешь?» → exhaustiveness: компилятор больше не знает всех реализаций, exhaustive `switch` невозможен.

> [!question]- Лямбда-в-поле vs тело константы — в чём разница?  
> **Model answer (RU):** Тело → анонимный подкласс (`getClass()` вернёт `Operation$1`), внутри доступен `this` константы. Лямбда-в-поле → один общий класс, поведение как данные, без доступа к состоянию константы. Нужен доступ к полям константы — тело; чистая функция от аргументов — лямбда (компактнее).
> 
> **Model answer (EN):** A body gives an anonymous subclass (`getClass()` returns `Operation$1`) and access to the constant's `this`. A lambda-in-field gives one shared class, behavior-as-data, with no access to the constant's state. Need the constant's fields — use a body; a pure function of the arguments — use a lambda.
> 
> _Trap-follow-up:_ «Почему лямбда не видит поля константы?» → потому что объявление константы — это static-контекст, `this` константы там нет в области видимости (ошибка компиляции). Это **не** «поля ещё не присвоены» — тело лямбды исполняется позже, при `apply`; дело в scoping, а не в тайминге.

> [!question]- `Operation.PLUS.getClass() == Operation.class` — `true` или `false`? _(что напечатает)_  
> **Ответ:** зависит от реализации. С телами констант — `false` (`PLUS` это `Operation$1`, подкласс). С лямбдой-в-поле — `true` (подкласса нет, `PLUS` это сам `Operation`).
> 
> _Follow-up:_ «Как тогда надёжно узнать enum-тип?» → `getDeclaringClass()` всегда вернёт `Operation`. И `PLUS instanceof Operation` всегда `true` (подкласс _является_ `Operation`), даже когда `getClass() == Operation.class` — `false`.

> [!question]- Пять констант считают одинаково — дублировать тело в каждой?  
> **Ответ:** нет. Вынести общее поведение групп в отдельную стратегию — вложенный strategy-enum, и делегировать ему (паттерн `PayrollDay` + `PayType`, Item 34). Логика живёт в одном месте, константа лишь выбирает стратегию при объявлении.
> 
> _Trap:_ «а если у каждой константы поведение уникально?» → тогда наоборот, per-constant тела; strategy-enum оправдан именно когда поведение _общее внутри групп_.

> [!question]- Итоговый устный ответ по теме _(RU/EN)_  
> **RU:** В Java enum — полноценный класс, поэтому поведение можно привязать к константам. Способ первый — `abstract`-метод: каждая константа обязана дать тело, и компилятор не даст добавить константу без реализации. Под капотом каждое тело — анонимный подкласс enum'а, а вызов — обычный виртуальный диспатч через vtable, не `switch`; поэтому `getClass()` константы с телом вернёт `Operation$1`, а надёжно взять enum-класс можно через `getDeclaringClass()`. Способ второй — положить функцию (`DoubleBinaryOperator`) в поле: поведение-как-данные, аналог `Dictionary<Enum, Func>`, но встроенный; выбираешь тело, когда нужен доступ к состоянию константы, и лямбду для чистой функции. Enum не может `extends` класс (слот занят `java.lang.Enum`), но реализует любое число интерфейсов — это основа Item 38, расширяемые enum'ы через общий контракт; цена — потеря exhaustiveness.
> 
> **EN:** In Java an enum is a full class, so you can attach behavior to its constants. First way — an `abstract` method: every constant must supply a body, and the compiler won't let you add a constant without one. Under the hood each body is an anonymous subclass of the enum, and the call is ordinary virtual dispatch through the vtable, not a `switch` — that's why a bodied constant's `getClass()` returns `Operation$1`, while `getDeclaringClass()` reliably gives the enum type. Second way — put a function (`DoubleBinaryOperator`) in a field: behavior-as-data, like a built-in `Dictionary<Enum, Func>`; pick a body when you need the constant's state, a lambda for a pure function. An enum can't `extends` a class (the slot is taken by `java.lang.Enum`) but can `implements` any number of interfaces — the basis of Item 38, extensible enums sharing a contract, at the cost of exhaustiveness.
>
