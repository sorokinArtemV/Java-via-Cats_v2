
В бэкенде постоянно возникает связка «enum + логика, зависящая от него»: `switch` по статусу считает что-то по-разному, `switch` по операции применяет разную формулу, `switch` по типу события шлёт разное уведомление. Работает это ровно до того дня, когда кто-то добавляет новую константу и забывает дописать ветку: компилятор молчит (со `switch`-ем всё синтаксически цело), а баг всплывает в рантайме у пользователя. Java позволяет привязать поведение к самой константе — и тогда «забыл ветку» превращается в «не компилируется». Это второй смысловой скачок главы: в прошлом уроке enum _хранил_ данные, теперь он будет _вести себя_, причём по-разному для каждой константы.

**Mental model.** Раз константа — это объект класса (урок 5.1), на неё можно навесить метод. Тогда вместо «спросить `switch`-ем, что это за константа, чтобы решить, что делать» ты просто говоришь `op.apply(...)` — и нужная реализация выбирается сама. Это обычный полиморфизм, не спецфича enum'а.

#### How it works: поведение прямо на константе

Вот типовая боль в исходном виде — калькулятор на `switch`:

```java
enum Operation { PLUS, MINUS, TIMES }

double apply(Operation op, double x, double y) {
    switch (op) {
        case PLUS:  return x + y;
        case MINUS: return x - y;
        case TIMES: return x * y;
        default:    throw new AssertionError("Unknown op: " + op);
    }
}
```

Добавишь `DIVIDE` в enum — всё скомпилируется (`switch` с `default` синтаксически полон), но на первом же делении прилетит `AssertionError` в рантайме. Логика разъехалась с данными, и поймал это пользователь, а не сборка.

Раз enum — полноценный класс, у него может быть `abstract`-метод. А тогда **каждая константа обязана дать своё тело**:

```java
public enum Operation {
    PLUS  { public double apply(double x, double y) { return x + y; } },
    MINUS { public double apply(double x, double y) { return x - y; } },
    TIMES { public double apply(double x, double y) { return x * y; } };

    public abstract double apply(double x, double y);
}
```

Никакого `switch`. Вызов — `Operation.PLUS.apply(6, 2)` → `8.0`, `Operation.TIMES.apply(6, 2)` → `12.0` (проверено на JDK 21). Блок `{ ... }` сразу после имени константы называется **per-constant body** — «тело константы». И ключевое: `apply` объявлен `abstract`, поэтому **компилятор физически не даст добавить константу без реализации**. Допишешь `DIVIDE` без тела — файл не скомпилируется. Та самая exhaustiveness без `switch`: проверка вариантов переехала из рантайма в компиляцию.

> [!tip] 💡 Из C#  
> В C# `enum` так не умеет вообще — это именованная обёртка над `int`, ни метода, ни тем более разного поведения на значение к нему не привяжешь. Чтобы получить то же самое, у тебя три обходных пути: `switch` по enum'у (та самая хрупкая штука), `Dictionary<Operation, Func<double,double,double>>` со словарём делегатов, или полноценная иерархия классов руками (у неё даже имя есть — «type-safe enum»). В Java всё это — одна языковая конструкция.

Теперь — механика, потому что тут есть вопрос уровня Middle+, который любят копать. Когда ты объявляешь `abstract`-метод, сам класс `Operation` становится **абстрактным по этому методу**. Абстрактный класс нельзя инстанцировать — а константы-то существуют как объекты. Как? Компилятор для каждой константы с телом **генерирует анонимный подкласс** `Operation` (JLS §8.9.2). То есть `PLUS { ... }` разворачивается концептуально в `PLUS = new Operation() { @Override apply … }`. Вот что реально построил компилятор:

![[Pasted image 20260707153524.png]]

`PLUS` — это не «`Operation` со флажком внутри», а **экземпляр отдельного безымянного подкласса** `Operation` со своим `apply`. Когда ты зовёшь `op.apply(6, 2)`, происходит ровно то, что в уроке про диспатч: JVM смотрит на реальный тип объекта в куче (`Operation$1`), находит его `apply` через таблицу методов и прыгает туда. Не `switch`, не `if` — виртуальный вызов, тот же механизм, что у любого `@Override`.

> [!example] 🔬 Глубже / на синьора  
> **`getClass()` константы с телом врёт — и это не баг.** Прогнал на JDK 21:
> 
> java
> 
> ```java
> Operation.PLUS.getClass()                    // class Operation$1  (НЕ Operation!)
> Operation.PLUS.getClass() == Operation.class // false
> Operation.PLUS.getDeclaringClass()           // class Operation
> Operation.PLUS instanceof Operation          // true
> ```
> 
> `getClass()` возвращает анонимный подкласс `Operation$1`, потому что `PLUS` буквально _экземпляр подкласса_. Компилятор и файлы кладёт соответственно — рядом с `Operation.class` появляются `Operation$1.class`, `$2`, `$3`. Надёжный способ спросить «какого enum'а эта константа» — `getDeclaringClass()`: он всегда вернёт сам enum-тип. И `instanceof Operation` работает (подкласс _является_ `Operation`), а `getClass() == Operation.class` — нет. Ровно на этой паре ловят на собесе.
> 
> **На уровне байткода** вызов `op.apply(...)` — это `invokevirtual` через слот vtable в метаданных класса объекта. JIT на мономорфном или `final` call-site девиртуализирует и инлайнит его, так что «дороговизна» виртуального вызова тут в проде обычно нулевая (детали механизма — глава про полиморфизм). Смысловой замок: **константы enum полиморфны, потому что каждая с телом — отдельный подкласс.** Никакого скрытого `switch` внутри enum'а нет.

Честная калибровка, потому что «можно» ≠ «всегда нужно». Ровно этот пример (`Operation` с per-constant `apply`) Джошуа Блох разбирает в Effective Java, Item 34, и рекомендует именно так — это идиоматичный, мейнстримный Java-код, а не хак. Но поведение стоит класть на enum, только когда оно **неотъемлемо от того, чем константа является**. `apply` для `Operation` — да, операция _и есть_ своё действие. А рендеринг, сериализация, маппинг в DTO, поход в базу — это не про «чем константа является», это внешние заботы; тащить их в сам тип не надо, иначе enum превращается в свалку. Правило: _внутреннее, определяющее_ поведение — на enum; _внешнее, контекстное_ — снаружи (`switch`, мапа, сервис).

#### Второй способ: поведение как данные (лямбда-в-поле)

Поведение можно задать не телом константы, а **функцией в поле**. Java даёт готовый `DoubleBinaryOperator` (два `double` на вход, `double` на выход):

```java
import java.util.function.DoubleBinaryOperator;

public enum Operation {
    PLUS ((x, y) -> x + y),
    MINUS((x, y) -> x - y),
    TIMES((x, y) -> x * y);

    private final DoubleBinaryOperator op;   // поведение как данные

    Operation(DoubleBinaryOperator op) { this.op = op; }

    public double apply(double x, double y) { return op.applyAsDouble(x, y); }
}
```

Снаружи всё так же: `Operation.PLUS.apply(6, 2)` → `8.0`. Но внутри принципиально иначе — **нет тел констант, значит нет анонимных подклассов**. Все константы — экземпляры самого `Operation`. Проверено на JDK 21:

```java
Operation.PLUS.getClass()                    // class Operation  (сам enum, не $1!)
Operation.PLUS.getClass() == Operation.class // true
```

Сравни с прошлым способом, где было `Operation$1` и `false`. Это и есть водораздел на уровне механики: **тело константы → подкласс + диспатч по vtable; лямбда-в-поле → один общий класс + одна индирекция через поле** (`op.applyAsDouble`, прыжок в лямбду). Поведение одинаковое, реализация разная.

> [!tip] 💡 Из C#  
> Второй способ — это ровно `Dictionary<Operation, Func<...>>` из C#, только встроенный в сам тип, без отдельной мапы: поле `op` играет роль значения словаря, константа — ключа. Разница в том, что в C# словарь живёт _снаружи_ enum'а (и можно забыть добавить запись для новой константы → рантайм-`KeyNotFoundException`), а в Java конструктор **обязывает** дать функцию каждой константе при объявлении — забыть нельзя, не скомпилируется.

#### Когда тело, когда лямбда

Прямой ответ: **лямбда-в-поле — когда поведение это чистая функция от аргументов** (как `apply`: дал два числа — получил число, ничего из самой константы не нужно). Компактнее, классов меньше. **Тело константы — когда поведению нужен доступ к состоянию константы**: к её полям, к `this`.

И вот граничный случай, который любят копнуть. Регулярный метод и тело константы выполняются в **instance-контексте** — у них есть `this`, поля константы доступны. Проверено: у `Planet` с полями `mass`/`radius` обычный метод `gravity()` спокойно читает `this.mass` (`EARTH.gravity()` → `9.81`), и тело константы тоже видит унаследованные поля. А вот аргументы конструктора константы — `EARTH(5.97e24, () -> ...)` — это **static-контекст инициализации**, где `this` не существует в принципе. Поэтому лямбда, которая пытается захватить `this` или поле константы, просто **не компилируется**. Точный текст ошибки на JDK 21:

```java
enum Planet {
    EARTH(5.97e24, () -> this.mass);   // не скомпилируется
    // ...
}
// error: non-static variable this cannot be referenced from a static context
```

То есть дело не в «поля ещё не присвоены в рантайме» — дело в том, что в аргументах константы `this` недоступен уже на этапе компиляции. Отсюда правило: **нужно состояние константы — тело (там есть `this`); чистая функция от аргументов — лямбда**.

Честно про перформанс (последнее, по чему стоит выбирать): тело = +1 сгенерированный класс на константу в metaspace, виртуальный вызов JIT отлично инлайнит; лямбда = +1 индирекция через поле. Разница микроскопическая. **Выбирай по читаемости и по тому, нужен ли доступ к состоянию, а не по скорости.**

#### Enum и интерфейсы: расширяемые enum'ы (Item 38)

Сначала жёсткий факт, который любят спрашивать: **enum не может `extends` класс.** Слот единичного наследования уже занят — каждый enum неявно `extends java.lang.Enum<E>`, второго родителя в Java нет (синтаксически `enum E extends Base` даже не парсится — проверено, компилятор ругается ещё на грамматике). А вот **`implements` интерфейсов — сколько угодно.** Это спроектированная лазейка под одну конкретную проблему.

Проблема: enum'ы **не расширяемы** — дописать константу в чужой `Operation` снаружи нельзя, он закрыт. Обычно это плюс (exhaustiveness!), но иногда клиентскому коду нужно работать с операциями из _разных наборов_ — базовых и чьих-то дополнительных. Наследованием enum'а не сделать, интерфейсом — да (Effective Java, Item 38):

```java
public interface Operation {
    double apply(double x, double y);
    default String describe(double x, double y) {          // default-метод (Java 8)
        return x + " " + this + " " + y + " = " + apply(x, y);
    }
}

public enum BasicOperation implements Operation {
    PLUS("+")  { public double apply(double x, double y) { return x + y; } },
    TIMES("*") { public double apply(double x, double y) { return x * y; } };
    private final String symbol;
    BasicOperation(String symbol) { this.symbol = symbol; }
    @Override public String toString() { return symbol; }
}
```

В другом модуле появляется второй enum — `ExtendedOperation` с `EXP` и `REMAINDER`, тоже `implements Operation`. А клиент типизирован интерфейсом, ему безразлично, откуда операция:

```java
static void runAll(Collection<? extends Operation> ops, double x, double y) {
    for (Operation op : ops) System.out.println(op.describe(x, y));
}
```

Проверено на JDK 21: `runAll` одинаково жуёт и `BasicOperation.values()` (`4.0 + 2.0 = 6.0`, `4.0 * 2.0 = 8.0`), и `ExtendedOperation.values()` (`4.0 ^ 2.0 = 16.0`, `4.0 % 2.0 = 0.0`). Два независимых enum'а делят один контракт. Добавить новый набор операций = объявить новый enum, реализующий `Operation`, не трогая существующие. Контракт можно выполнить тремя способами: (1) per-constant телами, как тут; (2) одним обычным методом на весь enum, если поведение общее; (3) `default`-методом в интерфейсе, если есть разумная реализация по умолчанию (`describe` — как раз третий случай, каждый enum получил его даром).

> [!tip] 💡 Из C#  
> В C# реализовать интерфейс enum'ом **нельзя** — точка. Набор «операций» с общим контрактом, который ещё и расширяется разными группами, — это иерархия классов (`abstract class Operation` + наследники) или интерфейс + классы. То, что в Java делает один `enum X implements I`, в C# превращается в россыпь классов плюс ручной реестр, чтобы перечислить «все варианты».

> [!example] 🔬 Глубже / на синьора  
> Если клиент должен работать не с готовой коллекцией, а с самим enum-типом (например, перебрать его константы), сигнатуру задают через **bounded type token** с пересечением: `<T extends Enum<T> & Operation>`. Это говорит «`T` — это enum И реализует `Operation`», и внутри доступны и `T.getEnumConstants()`, и `apply`. Проверено на JDK 21 — компилируется и работает; типичное место, где болит дженерик-сигнатура, поэтому стоит один раз написать руками.
> 
> **Мостик к главе 17.** Если тебе нужна _закрытая_ иерархия с проверкой полноты, но варианты **структурно разные** (разные поля у разных вариантов, а не однородные константы), в Java 21 есть альтернатива — `sealed`-интерфейсы + `record`-ы + pattern matching for switch (JEP 441, 440). Грубо: **enum с поведением — когда варианты однородны и их конечный набор; sealed + records — когда варианты закрыты, но разнородны.**

> [!warning] ⚠️ Подводные камни
> 
> 1. **`@Override` в теле константы — не косметика, а страховка.** Опечатался в сигнатуре (`apply(double x, int y)` вместо `double y`) — без `@Override` ты не переопределишь абстрактный метод, а _добавишь новый_, и enum останется абстрактным по `apply` → ошибка компиляции с невнятной привязкой. С `@Override` ошибка точная и на нужной строке.
> 2. **`getClass() == Operation.class` для константы с телом → `false`.** Инстинкт «константа же типа `Operation`» ложен: с телом она типа `Operation$1`. Для логирования/сравнения типа бери `getDeclaringClass()`, `instanceof` тоже надёжен.
> 3. **Лямбда в аргументе константы не видит состояние константы** — это static-контекст, `this` там недоступен (не скомпилируется). Хочешь состояние — только тело.
> 4. **Цена расширяемости (Item 38) — потеря exhaustiveness.** Пока был один enum с `abstract apply`, компилятор знал все варианты. Как только операции размазаны по нескольким enum'ам за интерфейсом, компилятор **больше не знает всех реализаций** — exhaustive `switch` невозможен, страховочной сетки нет. Осознанный размен гибкости на проверку полноты.
> 5. **Мутабельное состояние на enum с поведением.** Соблазн держать в константе счётчик/кеш без `final`. Но константа — синглтон на всю JVM, и мутабельное поле становится глобальным разделяемым состоянием со всеми гонками. Тянет хранить изменяемое — почти всегда сигнал, что нужен обычный класс, а не enum.

#### На собесе

> [!question]- Как сделать разное поведение на каждую константу без `switch`?  
> **RU:** `abstract`-метод в enum, каждая константа даёт своё тело (per-constant body). Почему лучше `switch`: компилятор заставляет реализовать метод для каждой константы — забыть нельзя, тогда как `switch` на новой ветке промолчит и упадёт в рантайме.
> 
> **EN:** An `abstract` method on the enum, with each constant providing its own body (per-constant body). Why it beats `switch`: the compiler forces an implementation for every constant — you can't forget one, whereas a `switch` silently misses the new branch and fails at runtime.
> 
> _Follow-up:_ «что происходит на уровне типов?» → каждая константа с телом — анонимный подкласс enum'а, переопределяющий метод; `PLUS` это по сути `new Operation(){…}`. _Follow-up:_ «`PLUS.getClass()` что вернёт?» → `Operation$1`, не `Operation`; надёжно брать тип через `getDeclaringClass()`.

> [!question]- Enum-strategy или `switch` — что выберешь и почему?  
> **RU:** Если поведение _принадлежит_ константе — enum с per-constant методом: новая константа без реализации не скомпилируется. Если поведение _внешнее_ и не принадлежит enum'у (рендеринг, маппинг в чужой формат) — нормально и `switch`, лучше exhaustive switch expression (Java 14+, JEP 361), или мапа снаружи.
> 
> **EN:** If the behavior _belongs to_ the constant — an enum with a per-constant method: a new constant without an implementation won't compile. If the behavior is _external_ and doesn't belong to the enum (rendering, mapping to a foreign format) — a `switch` is fine, preferably an exhaustive switch expression (Java 14+, JEP 361), or a map outside.

> [!question]- Может ли enum `extends` класс? `implements` интерфейс?  
> **RU:** `extends` — нет: слот единичного наследования занят `java.lang.Enum`. `implements` — да, сколько угодно.
> 
> **EN:** `extends` — no: the single-inheritance slot is taken by `java.lang.Enum`. `implements` — yes, as many as you like.
> 
> _Follow-up:_ «зачем enum'у интерфейс?» → расширяемость (Item 38): несколько enum'ов делят контракт, клиент работает с интерфейсом. _Follow-up:_ «что теряешь?» → exhaustiveness: компилятор больше не знает всех реализаций.

> [!question]- Лямбда-в-поле vs тело константы — в чём разница? _(любят копнуть)_  
> **RU:** Тело → анонимный подкласс (`getClass()` вернёт `Operation$1`), доступен `this` константы. Лямбда-в-поле → один общий класс (`getClass() == Operation.class`), поведение как данные, без доступа к состоянию константы. Нужен доступ к полям — тело; чистая функция от аргументов — лямбда.
> 
> **EN:** A body → an anonymous subclass (`getClass()` returns `Operation$1`), the constant's `this` is available. A lambda-in-field → one shared class (`getClass() == Operation.class`), behavior as data, no access to the constant's state. Need the fields — body; a pure function of the arguments — lambda.
> 
> _Follow-up:_ «почему лямбда в аргументе не видит поля константы?» → аргументы константы — static-контекст, `this` там недоступен; попытка захватить его не компилируется (`non-static variable this cannot be referenced from a static context`).

> [!question]- `Operation.PLUS.getClass() == Operation.class` — true или false?  
> **RU:** Зависит от реализации. С телами констант — `false` (`PLUS` это `Operation$1`). С лямбдой-в-поле — `true` (подкласса нет). Надёжно взять enum-тип всегда можно через `getDeclaringClass()`, а `PLUS instanceof Operation` всегда `true`.
> 
> **EN:** It depends on the implementation. With per-constant bodies — `false` (`PLUS` is `Operation$1`). With a lambda-in-field — `true` (no subclass). You can always get the enum type reliably via `getDeclaringClass()`, and `PLUS instanceof Operation` is always `true`.

> [!question]- Как сделать набор операций расширяемым в нескольких enum'ах?  
> **RU:** Общий интерфейс, каждый enum его реализует, клиент типизирован интерфейсом (`Collection<? extends Operation>` или, если нужен сам enum-тип, `<T extends Enum<T> & Operation>`). Цена — нет exhaustive switch: сознательный размен на гибкость (Item 38).

> **EN:** A shared interface, each enum implements it, the client is typed to the interface (`Collection<? extends Operation>`, or `<T extends Enum<T> & Operation>` if you need the enum type itself). The cost — no exhaustive switch: a deliberate trade of completeness for flexibility (Item 38).
